The strongest answer is: **no single system is perfect yet**, but the most human-like direction is a **hybrid of MIRIX + Nemori + Graphiti/Zep + Voyager-style procedural skill memory**.

If I had to pick one existing design as the most human-like **deployable** memory architecture, I would pick **MIRIX**. If I had to pick the most human-like **research idea**, I would pick **Nemori**. If I had to pick the fastest production memory layer, I would pick **Mem0**. If I had to build your OpenQG-style compounding research agent, I would combine all three with a temporal graph and a procedural skill library.

Recent memory surveys frame memory as the key mechanism that lets LLM agents become self-evolving across long interactions, rather than resetting every session. ([Hugging Face][1])

## Ranking rubric

I ranked each system on three axes:

| Axis           | Meaning                                                                                                         |
| -------------- | --------------------------------------------------------------------------------------------------------------- |
| **Human-like** | Has analogues of working, episodic, semantic, procedural, emotional/importance, temporal, or forgetting memory. |
| **Adaptable**  | Can reorganize, consolidate, update, correct, or learn from experience.                                         |
| **Fast**       | Retrieves useful memory with low latency and low token cost.                                                    |

My weighted score is roughly: **40% human-like + 35% adaptable + 25% fast**.

# 1. MIRIX — best overall human-like agent memory

**Why it ranks #1:** MIRIX is the closest thing I found to a full human-like memory stack for AI agents. It explicitly separates memory into **Core, Episodic, Semantic, Procedural, Resource Memory, and Knowledge Vault**, and coordinates them with multiple specialized agents. It also supports multimodal memory, including screen/activity context, which makes it more like embodied human experience than text-only memory. The paper reports 35% higher ScreenshotVQA accuracy than a RAG baseline while reducing storage by 99.9%, and 85.4% on LOCOMO. ([Hugging Face][2])

**Open source / paper:** GitHub: **Mirix-AI/MIRIX**. Paper: **MIRIX: Multi-Agent Memory System for LLM-Based Agents**. ([GitHub][3])

**Theory:** Human memory is not one database. We have identity memory, episodic memory, semantic memory, procedural routines, resources, and private vault-like knowledge. MIRIX copies that pattern. The strongest idea is not “store everything,” but **route each memory to the right subsystem**.

**Build schematic:**

```text
Input stream: text, images, screen, voice, tool traces
        ↓
Meta-memory agent
        ↓ classifies
┌───────────────┬───────────────┬────────────────┬────────────────┐
Core            Episodic         Semantic          Procedural
identity        events           facts/concepts    workflows/skills
└───────────────┴───────────────┴────────────────┴────────────────┘
        ↓
Resource memory + knowledge vault
        ↓
Hybrid retrieval: BM25 + vector + memory-type filters
        ↓
Context pack for reasoning agent
```

**How I would use it:** MIRIX should be the top-level design pattern for an agent that lives with a user or a research project for months. For OpenQG, I would adapt the six-memory split as: Core project memory, Episodic research runs, Semantic physics knowledge, Procedural scripts/tools, Resource memory for papers/datasets/repos, and Knowledge Vault for credentials/private notes.

---

# 2. Nemori — most human-like self-organizing research design

**Why it ranks #2:** Nemori is the most explicitly cognitive-science-inspired system I found. It uses a **Two-Step Alignment Principle** inspired by Event Segmentation Theory to break raw conversation streams into coherent episodes, and a **Predict-Calibrate Principle** inspired by the Free-energy Principle so the agent learns from prediction gaps. In plain language: it tries to remember by forming episodes, predicting what should happen, and reorganizing knowledge when reality disagrees. ([Hugging Face][4])

**Open source / paper:** Paper: **Nemori: Self-Organizing Agent Memory Inspired by Cognitive Science**. I could not verify an official GitHub repo for the paper. ([Hugging Face][4])

**Theory:** Humans do not remember raw token streams. We segment life into events: “the meeting,” “the experiment,” “the failed build,” “the conversation about X.” Then we extract semantic lessons from those episodes. Nemori’s key insight is that **memory granularity must be learned**, not arbitrarily chunked.

**Build schematic:**

```text
Raw interaction stream
        ↓
Boundary detector
        ↓
Episode builder
        ↓
Narrative representation
        ↓
Prediction: what should this episode imply?
        ↓
Calibration: what was surprising / wrong / new?
        ↓
Update episodic memory + semantic memory
```

**How I would use it:** Use Nemori as the **event segmentation and learning layer**. For a research agent, every literature sweep, failed code run, benchmark result, and model critique should become an episode. The system should ask: “What did I expect? What happened? What concept should change?”

---

# 3. Zep / Graphiti — best temporal knowledge-graph memory

**Why it ranks #3:** Zep/Graphiti is one of the most important production-grade designs because it treats memory as a **temporal context graph**. Facts are not just stored as static chunks; they are represented as entities, relationships, and episodes with temporal validity. Graphiti tracks what was true, when it became true, when it was superseded, and where it came from. ([Hugging Face][5])

**Open source / paper:** GitHub: **getzep/graphiti**. Paper: **Zep: A Temporal Knowledge Graph Architecture for Agent Memory**. ([GitHub][6])

**Theory:** Human memory is temporal and relational. We do not only remember “Alice works at X”; we remember “Alice used to work at X, then moved to Y after the March project.” This is critical for agents because stale memory is worse than no memory.

**Build schematic:**

```text
Episode ingestion
        ↓
Entity extraction
        ↓
Fact extraction
        ↓
Temporal edge creation
        ↓
Entity summaries evolve over time

Graph:
Person ──works_at(valid_from, valid_to, source_episode)──> Company
Claim  ──supported_by(time, confidence)──────────────────> Paper
```

**How I would use it:** This should be the **canonical long-term semantic memory** for a serious research system. For OpenQG, every paper, claim, equation, dataset, author, benchmark, and theory becomes a graph node; every relationship gets provenance and time.

---

# 4. A-MEM — best self-organizing Zettelkasten memory

**Why it ranks #4:** A-MEM is very strong for concept compounding. It uses Zettelkasten-like memory: each new memory becomes a structured note with context, keywords, tags, and links to related memories. New memories can also trigger evolution of older memories, so the network reorganizes as experience accumulates. ([Hugging Face][7])

**Open source / paper:** GitHub: **agiresearch/A-mem**. Paper: **A-MEM: Agentic Memory for LLM Agents**. ([GitHub][8])

**Theory:** Zettelkasten is a human knowledge-compounding system: atomic notes, dense links, emergent structure. A-MEM applies that to agents. It is less about “recalling past chat” and more about **growing a concept network**.

**Build schematic:**

```text
New memory
        ↓
Generate structured note:
  - summary
  - context
  - keywords
  - tags
  - entities
  - importance
        ↓
Find similar / related historical notes
        ↓
Create bidirectional links
        ↓
Update older note summaries if new memory changes interpretation
        ↓
Retrieve by semantic + graph link expansion
```

**How I would use it:** Use A-MEM as the **conceptual synthesis layer** on top of raw episodes and graph facts. For OpenQG, ClaimCards and QuestionCards should link like Zettelkasten notes: each claim should point to supporting papers, contradictions, equations, datasets, and open questions.

---

# 5. MemoryOS — best hierarchical memory operating system

**Why it ranks #5:** MemoryOS is an OS-inspired design with **short-term, mid-term, and long-term personal memory** plus four modules: storage, updating, retrieval, and generation. It reports 49.11% F1 and 46.18% BLEU-1 improvements on LoCoMo over baselines, and its repo includes an MCP server and modular tooling. ([GitHub][9])

**Open source / paper:** GitHub: **BAI-LAB/MemoryOS**. Paper: **Memory OS of AI Agent**. ([GitHub][9])

**Theory:** This is analogous to computer memory management and human time-scale memory. Recent interactions stay hot. Sessions consolidate into mid-term memory. Stable personal knowledge migrates into long-term memory.

**Build schematic:**

```text
Short-term memory:
  recent turns, active task state

Mid-term memory:
  session summaries, dialogue chains, working project state

Long-term memory:
  user profile, stable knowledge, persistent preferences

Updater:
  short → mid via FIFO/session chains
  mid → long via segmentation and profile updates

Retriever:
  query all tiers in parallel
  merge and rerank
```

**How I would use it:** MemoryOS is ideal as the **tiering policy** for an agent. In OpenQG, short-term memory is current run state, mid-term is active investigation, long-term is curated scientific memory.

---

# 6. Mem0 — fastest production-ready memory layer

**Why it ranks #6:** Mem0 is less human-like than MIRIX or Nemori, but it may be the strongest “just ship it” memory layer. Its paper reports 26% relative improvement over OpenAI Memory, 91% lower p95 latency, and more than 90% token-cost savings compared with full-context processing. Its 2026 GitHub README describes a newer ADD-only, single-pass retrieval algorithm with semantic, BM25, and entity matching fused in parallel. ([Hugging Face][10])

**Open source / paper:** GitHub: **mem0ai/mem0**. Paper: **Mem0: Building Production-Ready AI Agents with Scalable Long-Term Memory**. ([GitHub][11])

**Theory:** Mem0 optimizes for practical memory extraction and retrieval. It stores salient facts, links entities, retrieves using multiple signals, and minimizes token overhead. It is not the most biologically inspired, but it is extremely useful.

**Build schematic:**

```text
Conversation / agent action
        ↓
Memory extractor
        ↓
Atomic memory facts
        ↓
Entity linking
        ↓
Stores:
  vector index
  keyword/BM25 index
  entity index / graph memory
        ↓
Single-pass multi-signal retrieval
        ↓
Reranked context
```

**How I would use it:** Use Mem0-like retrieval as the **low-latency recall layer**. In a research agent, this is how you quickly answer “what did we already learn about causal dynamical triangulations datasets?” without loading the whole graph.

---

# 7. Cognee — best structured graph-vector knowledge engine

**Why it ranks #7:** Cognee is a graph-vector hybrid memory engine for agents. It combines a graph store, vector store, and relational/provenance store, and supports multiple retrieval modes, including graph completion, RAG completion, graph chain-of-thought traversal, summaries, temporal search, and code-focused retrieval. ([Cognee][12])

**Open source / paper/docs:** GitHub: **topoteretes/cognee**. Documentation/blog: Cognee memory architecture. ([GitHub][13])

**Theory:** Cognee is less “brain-like” than Nemori, but very strong as a **knowledge compiler**. It converts documents, tables, transcripts, and app data into linked graph/vector structures that agents can search and reason over.

**Build schematic:**

```text
Ingest documents / tables / code / conversations
        ↓
Extract → Cognify → Load
        ↓
Relational store:
  documents, chunks, provenance
Graph store:
  entities, relationships, ontology
Vector store:
  embeddings for chunks/nodes/summaries
        ↓
Search modes:
  vector, lexical, graph traversal, temporal, Cypher, summaries
```

**How I would use it:** Cognee is a great model for OpenQG’s **research knowledge compiler**: ingest papers and code, extract concepts, build graph/vector stores, keep citations, and expose search modes to agents.

# 8. HippoRAG — best hippocampus-inspired associative recall

**Why it ranks #8:** HippoRAG is one of the most biologically inspired retrieval designs. It is based on hippocampal indexing theory and combines LLMs, knowledge graphs, and Personalized PageRank. It reports up to 20% improvement over state-of-the-art multi-hop QA methods, while single-step HippoRAG can be 10–30× cheaper and 6–13× faster than iterative retrieval methods like IRCoT. ([Hugging Face][14])

**Open source / paper:** GitHub: **OSU-NLP-Group/HippoRAG**. Paper: **HippoRAG: Neurobiologically Inspired Long-Term Memory for Large Language Models**. ([Hugging Face][14])

**Theory:** The hippocampus does not store every detail directly; it indexes related memories so activation can spread across associated concepts. HippoRAG imitates this using graph traversal and Personalized PageRank.

**Build schematic:**

```text
Corpus / experiences
        ↓
LLM extracts triples/entities
        ↓
Knowledge graph
        ↓
Query maps to seed nodes
        ↓
Personalized PageRank spreads activation
        ↓
Top activated nodes/edges become retrieval context
```

**How I would use it:** Use HippoRAG-style spreading activation for **multi-hop scientific reasoning**. Example: a question about “black-hole entropy corrections” may need paths through AdS/CFT, entanglement entropy, loop quantum gravity, and observational constraints.

---

# 9. Letta / MemGPT — best self-editing memory hierarchy

**Why it ranks #9:** MemGPT introduced an OS-inspired “virtual context management” approach, where agents manage a memory hierarchy beyond the context window. Letta, built by the MemGPT creators, implements self-editing memory blocks, recall memory, archival memory, and agent-managed memory tools. ([Hugging Face][15])

**Open source / paper:** GitHub: **letta-ai/letta**. Paper: **MemGPT: Towards LLMs as Operating Systems**. ([GitHub][16])

**Theory:** This is the “LLM as its own memory manager” design. The agent can decide what belongs in always-visible core memory, what to archive, and when to search old conversations.

**Build schematic:**

```text
Context window:
  system instructions
  core memory blocks
  recent messages

External memory:
  recall memory = conversation history
  archival memory = semantic facts/documents

Agent tools:
  memory_insert
  memory_replace
  memory_rethink
  archival_memory_search
  conversation_search
```

**How I would use it:** Use Letta/MemGPT for **working memory and self-editing project state**. For example, an OpenQG agent should keep a compact “current research stance” memory block and revise it when new evidence appears.

---

# 10. Voyager-style procedural skill memory — best compounding action memory

**Why it ranks #10:** Voyager is not a general memory database, but it is one of the best demonstrations of **experience compounding into reusable procedural skill**. It builds an ever-growing library of executable skills, uses environment feedback and self-verification to improve them, and reuses learned skills in new Minecraft worlds. The paper reports 3.3× more unique items, 2.3× longer travel, and tech-tree milestones up to 15.3× faster than prior systems. ([Hugging Face][17])

**Open source / paper:** GitHub: **MineDojo/Voyager**. Paper: **Voyager: An Open-Ended Embodied Agent with Large Language Models**. ([GitHub][18])

**Theory:** Human expertise is not only facts. It is also procedures: how to run a literature search, how to debug code, how to analyze a dataset, how to reproduce a result. Voyager’s skill library is a blueprint for procedural memory.

**Build schematic:**

```text
Goal proposed by curriculum
        ↓
Generate code/action
        ↓
Execute in environment
        ↓
Observe success/failure
        ↓
Repair through feedback
        ↓
If successful:
  store skill:
    name
    description
    code
    preconditions
    examples
    embeddings
        ↓
Retrieve and compose skills for harder tasks
```

**How I would use it:** OpenQG should absolutely have a Voyager-style skill memory: paper search skills, arXiv harvesters, dataset downloaders, benchmark scripts, equation parsers, citation graph builders, and reproducibility workflows.

## Important honorable mentions

**MemoryBank** is important because it uses Ebbinghaus-style forgetting and reinforcement, user portraits, memory storage/retrieval/updating, and personality adaptation. It is very human-like, but I rank newer systems higher because they are more structurally complete or more deployable. ([AAAI Publications][19])

**Generative Agents** is historically important because it used a memory stream, reflection, and planning to simulate believable human behavior. It is conceptually excellent but less suitable as a modern high-performance research memory backend. ([Hugging Face][20])

**Memory-R1** is promising because it trains a Memory Manager and Answer Agent with RL to learn ADD/UPDATE/DELETE/NOOP and retrieval behavior, reportedly with as few as 152 training QA pairs. It is very interesting for learned memory policy, but I would not use it as the first production base. ([Hugging Face][21])

**MemOS** is a frontier architecture that treats memory as a first-class OS resource across plaintext, activation, and parameter memories through a MemCube abstraction. It is ambitious and probably worth watching closely. ([Hugging Face][22])

# The architecture I would actually build

For your use case—agents that compile scientific knowledge, self-organize concepts, learn from search, and avoid context overload—I would not choose one system. I would build this hybrid:

```text
                  ┌──────────────────────────────┐
                  │        Agent Interface        │
                  └──────────────┬───────────────┘
                                 │
                  ┌──────────────▼───────────────┐
                  │      Working Memory Layer     │
                  │     Letta/MemGPT style        │
                  │  core blocks + active state   │
                  └──────────────┬───────────────┘
                                 │
        ┌────────────────────────┼────────────────────────┐
        │                        │                        │
┌───────▼────────┐     ┌─────────▼─────────┐     ┌────────▼─────────┐
│ Episodic Memory│     │ Semantic Memory   │     │ Procedural Memory│
│ MIRIX/Nemori   │     │ Graphiti/Cognee   │     │ Voyager-style    │
│ events/runs    │     │ claims/entities   │     │ scripts/skills   │
└───────┬────────┘     └─────────┬─────────┘     └────────┬─────────┘
        │                        │                        │
        └────────────────────────┼────────────────────────┘
                                 │
                  ┌──────────────▼───────────────┐
                  │  Self-Organization Layer      │
                  │  A-MEM links + Nemori         │
                  │  predict-calibrate updates    │
                  └──────────────┬───────────────┘
                                 │
                  ┌──────────────▼───────────────┐
                  │ Retrieval / Recall Layer      │
                  │ Mem0 multi-signal retrieval   │
                  │ + HippoRAG graph activation   │
                  └──────────────┬───────────────┘
                                 │
                  ┌──────────────▼───────────────┐
                  │ Context Pack Compiler         │
                  │ small, cited, task-specific   │
                  └──────────────────────────────┘
```

The design principle is:

> **Do not make the LLM remember everything. Make the system organize memory so the LLM receives only the right compressed context at the right time.**

For OpenQG, the memory objects should be:

```text
WorkCard          paper / book / preprint / repo / dataset paper
ClaimCard         atomic scientific claim
EquationCard      equation, variables, units, assumptions
DatasetCard       public dataset manifest + downloader
CodeSkillCard     reusable script/tool/procedure
QuestionCard      open research question
SearchRunCard     query, sources, results, gaps
EpisodeCard       a complete research/search/build event
ConceptCard       synthesized concept with links
ContextPack       task-specific compact memory bundle
```

The best possible memory system for your agent is therefore:

```text
MIRIX memory taxonomy
+ Nemori self-organizing episodes
+ Graphiti temporal graph
+ A-MEM concept links
+ Mem0 fast retrieval
+ HippoRAG spreading activation
+ Voyager procedural skills
+ MemoryOS tiering
+ Letta self-editing working state
```

That is the closest current design to a **human-like, fast, self-organizing, experience-compounding memory system for AI agents**.
