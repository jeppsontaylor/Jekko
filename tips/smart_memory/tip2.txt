## Best answer

Taking **“designed by AI” literally**, the strongest candidate is **OmniMem / Omni-SimpleMem**, because its architecture was discovered through an autonomous AI research pipeline that ran many experiments, diagnosed failures, modified architectures, repaired bugs, and improved prompts. ([arXiv][1])

Taking **“most human-like memory system for AI agents”** more broadly, my top pick is **EverMemOS / EverCore**. It is explicitly modeled around an **engram-like lifecycle**: episodic trace formation, semantic consolidation, and reconstructive recollection. That is closer to how human memory works than a pure vector database, because human memory is not just lookup; it is event capture, abstraction, association, consolidation, forgetting, and reconstruction. ([GitHub][2])

A practical “best stack” today would combine **EverMemOS + MIRIX + A-Mem + Graphiti/Zep + Mem0/SimpleMem**: EverMemOS for the biological lifecycle, MIRIX for typed memory modules, A-Mem for self-organizing concept notes, Graphiti/Zep for temporal knowledge graphs, and Mem0 or SimpleMem for fast low-token recall.

---

## Scoring rubric

These rankings are a synthesis, not a single benchmark leaderboard, because the systems are evaluated on different tasks such as LoCoMo, LongMemEval, ScreenshotVQA, Mem-Gallery, multimodal QA, and multi-hop retrieval.

**Human-like** means: episodic memory, semantic memory, procedural memory, working memory, consolidation, forgetting, temporal context, associative recall, reflection, and reconstruction.

**Adaptable** means: learns quickly from new experience, updates or revises old memories, self-organizes concepts, generalizes across tasks, and improves with repeated use.

**Fast** means: low-latency retrieval, low token cost, efficient indexing, fast reranking, and ability to recall without dumping huge context windows.

# 1. EverMemOS / EverCore

**Best for:** most human-like general memory architecture.

EverMemOS is the most human-like system I found because it treats memory as a lifecycle rather than a database. Its design has three core stages: **Episodic Trace Formation**, where dialogue streams are converted into memory cells; **Semantic Consolidation**, where related cells are organized into thematic memory scenes; and **Reconstructive Recollection**, where the system composes only the necessary context for the current query. ([GitHub][2])

**Theory**

Human memory does not store perfect transcripts. It compresses experiences into traces, links them to themes, updates identity and preferences, and reconstructs context when needed. EverMemOS maps this idea into agent memory:

```text
conversation events
    ↓
MemCells: atomic episodic traces, facts, time, actors, importance
    ↓
MemScenes: clustered themes / situations / relationship contexts
    ↓
user profile + semantic summaries + temporal state
    ↓
query-driven reconstructive recollection
```

**Build schematic**

```text
Input event
  → segment by turn, topic, task, and emotional/goal relevance
  → extract MemCell {
        event_gist,
        atomic_facts,
        entities,
        time,
        speaker,
        salience,
        confidence,
        future_relevance,
        links
     }
  → embed MemCell
  → link to existing cells by entity, topic, time, causality, goal
  → periodically cluster cells into MemScenes
  → update stable profile facts and preferences
  → at query time:
        classify query intent
        retrieve scenes + cells + profile facts
        rerank for necessity, recency, confidence
        compose compact recollection context
```

**Minimal implementation**

Use Postgres for structured memory, pgvector or Qdrant for embeddings, BM25 for keyword recall, and a graph layer for links. Store raw episodes separately from extracted memory cells. Run consolidation after every conversation or every N events. The key is to never retrieve raw history first; retrieve **scenes**, then supporting memory cells.

# 2. MIRIX

**Best for:** complete human-style memory taxonomy.

MIRIX is one of the strongest designs because it explicitly separates memory into **Core Memory, Episodic Memory, Semantic Memory, Procedural Memory, Resource Memory, and Knowledge Vault**. It is also multimodal: the system can observe screen activity and conversations, then form personal memories over time. ([GitHub][3])

**Theory**

This resembles cognitive architecture more than simple RAG. Instead of one memory pile, MIRIX routes new experience into specialized memory systems:

```text
Core memory       → identity, stable user profile, enduring preferences
Episodic memory   → events, conversations, screen observations
Semantic memory   → facts, concepts, relationships
Procedural memory → reusable skills, workflows, habits
Resource memory   → documents, files, tools, references
Knowledge vault   → trusted structured knowledge
```

**Build schematic**

```text
Observation stream: text, screen, image, tool event
  → multimodal parser
  → memory router decides target memory type
  → per-type extractor writes structured memory
  → multi-agent memory manager validates, dedupes, links
  → retrieval planner queries the right memory modules
  → answer composer merges core + episodic + semantic + procedural context
```

**Minimal implementation**

Create six tables or collections, one for each memory type. Add a router LLM that labels each incoming event as core, episodic, semantic, procedural, resource, or vault. For retrieval, classify the user’s query into the same memory types, then retrieve from only the relevant modules. Add a cross-memory graph so that “user likes X,” “did X last week,” and “workflow for X” can point to one another.

# 3. MemVerse

**Best for:** multimodal, compounding memory with consolidation and adaptive forgetting.

MemVerse is powerful because it combines **short-term memory**, **hierarchical knowledge graphs**, **multimodal experience storage**, **continual consolidation**, **adaptive forgetting**, and **parametric distillation**. That last part is important: some memories can be distilled into model parameters or fast internal recall, rather than always retrieved externally. ([GitHub][4])

**Theory**

MemVerse tries to bridge two kinds of memory:

```text
Retrieval memory  → external, editable, inspectable, graph-based
Parametric memory → fast, compressed, model-internal knowledge
```

Human memory works similarly: some knowledge is explicit and recallable, while some becomes automatic intuition or skill.

**Build schematic**

```text
Text / image / audio / video experience
  → modality encoders
  → short-term memory buffer
  → event summarizer
  → hierarchical KG builder:
        entities
        relations
        concepts
        episodes
        modalities
  → consolidation:
        merge duplicates
        strengthen repeated patterns
        decay stale details
        compress stable knowledge
  → optional distillation:
        turn repeated stable knowledge into compact model adapters or prompts
  → retrieval:
        query → modality-aware search → graph traversal → compact context
```

**Minimal implementation**

Use separate encoders for text and images, then normalize everything into an event schema. Build a hierarchical graph with levels such as episode → scene → entity → concept. Add counters for memory frequency, recency, and usefulness. Periodically prune low-value details and promote repeated patterns into stable semantic memory.

# 4. OmniMem / Omni-SimpleMem

**Best for:** strongest literal AI-designed memory system.

OmniMem is especially relevant to your wording because it was discovered by an autonomous AI research pipeline. The pipeline reportedly ran around 50 experiments across benchmarks, diagnosed failures, modified memory architecture, repaired bugs, and improved prompts. The resulting Omni-SimpleMem supports lifelong multimodal memory across text, images, audio, and video. ([arXiv][1])

**Theory**

The core idea is to make memory efficient through selective ingestion, semantic compression, and retrieval planning. Instead of storing everything equally, Omni-SimpleMem keeps what is likely to matter and retrieves progressively.

```text
Experience
  → selective ingestion
  → semantic compression
  → hybrid indexing
  → intent-aware retrieval plan
  → progressive expansion only when needed
```

The SimpleMem repository describes mechanisms such as entropy-driven filtering, hybrid FAISS/BM25 retrieval, pyramid token-budget expansion, and knowledge-graph augmentation. ([GitHub][11])

**Build schematic**

```text
Input event
  → score informativeness / entropy / novelty
  → discard or compress low-value details
  → store compressed semantic memory
  → index by:
        dense vector
        BM25 keywords
        entities
        temporal metadata
        graph links
  → query-time planner:
        infer intent
        start with tiny retrieval budget
        expand only if confidence is low
  → return compact memory pack
```

**Minimal implementation**

Build a memory writer that refuses to store trivial events. For every accepted event, create a compressed fact bundle plus a trace back to the raw source. Retrieval should be staged: first retrieve 3–5 compact memories; only expand to raw episodes if the model says evidence is insufficient.

# 5. A-Mem

**Best for:** self-organizing conceptual memory.

A-Mem is one of the best designs for **self-organizing concepts**. It uses a Zettelkasten-inspired approach: memories become structured notes with descriptions, keywords, tags, contextual links, and evolving attributes. New memories can update old memories, so the network gets richer over time. ([GitHub][5])

**Theory**

Zettelkasten memory is powerful because each note is atomic but heavily linked. The intelligence comes from the network, not from any one note.

```text
atomic memory note
  + context
  + tags
  + keywords
  + links to older notes
  + evolving interpretation
```

This is close to how humans form concepts: a new experience is not just stored; it changes the meaning of older experiences.

**Build schematic**

```text
New memory
  → create structured note {
        content,
        context,
        keywords,
        tags,
        entities,
        timestamp,
        importance
     }
  → retrieve possibly related old notes
  → LLM proposes:
        links
        tag updates
        rewritten descriptions
        higher-level concepts
  → update graph
  → retrieval:
        query → note search → linked-note expansion → synthesis
```

**Minimal implementation**

Use a `memory_notes` table and a `memory_links` table. Every new note should trigger a “linking pass” that searches old notes and asks the LLM: “Which old notes does this change, support, contradict, or generalize?” Store links with types such as `supports`, `contradicts`, `explains`, `example_of`, `caused_by`, and `same_theme`.

# 6. Zep / Graphiti

**Best for:** fast temporal knowledge graph memory in production.

Graphiti builds temporal knowledge graphs for agents. It tracks entities, facts, episodes, provenance, temporal validity, and changing relationships. Zep uses Graphiti-style memory for fast agent context and reports strong latency and memory benchmark results. ([GitHub][6])

**Theory**

Many agent memories fail because they store facts without time. Human memory knows that “Sarah works at Company A” may have been true last year but false now. Graphiti’s key advantage is **temporal fact modeling**.

```text
fact: user_lives_in(Denver)
valid_from: 2024-09-01
valid_to: null
source: conversation episode 492
confidence: 0.91
```

**Build schematic**

```text
Conversation or document
  → extract entities and relations
  → attach temporal metadata:
        valid_from
        valid_to
        observed_at
        source
  → merge with existing graph
  → invalidate or supersede outdated facts
  → retrieval:
        semantic search
        keyword search
        graph traversal
        temporal filtering
  → compose context with current facts first, historical facts when relevant
```

**Minimal implementation**

Use Neo4j, Kuzu, FalkorDB, or another graph database. Store edges as temporal facts, not timeless facts. Every new fact should be checked against existing facts for contradiction or supersession. Use hybrid retrieval: vector search to find likely nodes, then graph traversal to collect related facts.

# 7. HippoRAG 2

**Best for:** hippocampal-style associative recall.

HippoRAG is explicitly inspired by hippocampal indexing theory. It combines LLMs, knowledge graphs, and graph algorithms such as Personalized PageRank to mimic associative retrieval across memory. HippoRAG 2 extends this into a stronger memory system for factual, sense-making, and associative memory tasks. ([GitHub][7])

**Theory**

The hippocampus is good at linking cues to distributed memories. HippoRAG implements a version of this idea:

```text
query cue
  → activate related graph nodes
  → spread activation through memory graph
  → collect associated facts
  → rerank and synthesize
```

This is more human-like than standard vector search because it can find memories connected by relationships, not just textual similarity.

**Build schematic**

```text
Documents / episodes
  → extract triples and entities
  → build knowledge graph
  → embed nodes and passages
  → query:
        map query to seed nodes
        run graph propagation / PageRank
        retrieve top associated nodes and passages
        rerank
        answer with evidence
```

**Minimal implementation**

Build a graph from extracted triples. At query time, identify seed entities and concepts, then run Personalized PageRank over the graph. Retrieve the top nodes and their source passages. This is especially useful for “What connects X and Y?” or “Why did this happen?” questions.

# 8. Letta / MemGPT

**Best for:** foundational agent memory operating-system model.

MemGPT, now associated with Letta, introduced an operating-system-like view of agent memory: the model has a limited context window, but it can page information in and out of different memory tiers. This is analogous to virtual memory in computers and working memory in humans. ([GitHub][8])

**Theory**

The agent controls its own memory operations. It can decide when to write, edit, search, or retrieve memory.

```text
working context
  ↔ short-term conversation memory
  ↔ archival long-term memory
  ↔ external tools / files
```

The important innovation is not just storage; it is giving the agent **memory actions**.

**Build schematic**

```text
Agent loop
  → inspect current context
  → decide:
        answer directly
        search memory
        write memory
        edit memory
        archive memory
  → retrieve relevant archival memory
  → update working memory
  → respond
```

**Minimal implementation**

Give the LLM explicit tools: `memory_search`, `memory_write`, `memory_update`, and `memory_delete`. Add policies that force the model to write durable preferences, facts, unresolved tasks, and relationship information. Keep a small core memory always visible, and put everything else in archival memory.

# 9. Mem0

**Best for:** fastest practical personalization memory.

Mem0 is one of the most practical memory systems for production agents. It focuses on extracting, consolidating, and retrieving salient user memories with low latency and low token cost. Its newer algorithm emphasizes single-pass ADD-only extraction, entity linking, and hybrid retrieval signals such as semantic search, BM25, and entity matching. ([GitHub][9])

**Theory**

Mem0 is less biologically ambitious than EverMemOS or MIRIX, but it is extremely useful because it optimizes the high-impact path:

```text
conversation
  → extract useful memory
  → store compact fact
  → retrieve only relevant facts
  → personalize answer
```

**Build schematic**

```text
New message
  → LLM extracts candidate memories
  → classify:
        preference
        profile fact
        task fact
        relationship
        instruction
  → entity-link to user, project, person, place, object
  → store compact memory
  → retrieve via:
        vector similarity
        keyword match
        entity overlap
        recency
        importance
  → rerank and inject
```

**Minimal implementation**

Start with five memory types: preference, profile, project, relationship, and instruction. Store each memory as one sentence with metadata. Use additive writes first; do not overcomplicate with rewriting until duplicates become a problem. Add contradiction handling later.

# 10. Cognee

**Best for:** graph-vector memory control plane.

Cognee is a strong open-source memory/control-plane approach. It combines graph, vector, and relational storage; ingests many data formats; builds knowledge graphs; supports memory operations such as recall, connect, forget, and improve; and includes mechanisms such as pruning stale nodes and reweighting edges based on usage. ([GitHub][10])

**Theory**

Cognee is useful when memory is not just for one chat agent but for a system of agents or workflows. It treats memory as an evolving data substrate.

```text
data ingestion
  → cognify: extract structured knowledge
  → memify: prune, strengthen, reweight, derive
  → search: graph/vector/relational retrieval
```

**Build schematic**

```text
Raw data
  → chunk / parse / normalize
  → extract entities, facts, claims, relations
  → build graph
  → create embeddings
  → store relational metadata
  → usage feedback:
        strengthen useful edges
        weaken stale edges
        prune irrelevant nodes
        add derived facts
  → retrieve with graph + vector + SQL filters
```

**Minimal implementation**

Use three stores: graph DB, vector DB, and relational DB. Every memory node should have provenance, timestamps, confidence, usage count, and decay score. Retrieval should combine semantic similarity, graph neighborhood, and exact metadata filters.

## The most buildable “human-like” architecture

For an agent that gets smarter with experience, I would build this composite architecture:

```text
                ┌─────────────────────────┐
                │      Working Memory      │
                │ current task, scratchpad │
                └───────────┬─────────────┘
                            │
User / world events ────────▼────────────────
                    Event Segmentation
                            │
                            ▼
              Episodic Memory Writer
     raw event, gist, entities, time, salience
                            │
         ┌──────────────────┼──────────────────┐
         ▼                  ▼                  ▼
 Semantic Graph       Procedural Store     Profile/Core
 facts, concepts,     workflows, skills,   stable identity,
 temporal edges       lessons, habits      preferences
         │                  │                  │
         └──────────────────┼──────────────────┘
                            ▼
                    Consolidation Layer
       cluster, summarize, link, decay, forget, update
                            │
                            ▼
                    Retrieval Planner
       classify query → choose memory types → retrieve
                            │
                            ▼
                    Recollection Composer
       minimal necessary context, evidence, uncertainty
                            │
                            ▼
                         Agent Acts
                            │
                            ▼
                    Feedback / Reconsolidation
       success, failure, corrections, preference updates
```

The key design choice is to separate **memory writing** from **memory recall**. Writing can be slower and smarter: extract facts, link concepts, consolidate scenes, update profiles, and prune noise. Recall must be fast: classify intent, retrieve from the right memory module, rerank, and inject a compact memory pack.

## My final recommendation

For the most human-like memory today, start from **EverMemOS/EverCore**. For the fastest production memory, use **Mem0** or **Zep/Graphiti**. For the most self-organizing conceptual memory, add **A-Mem**. For multimodal long-term memory, look at **MIRIX** or **MemVerse**. For a system literally designed by AI experimentation, study **OmniMem / Omni-SimpleMem**.
