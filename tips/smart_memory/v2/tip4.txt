I unpacked and studied the archive: `CLAUDE_MEMSPEC.md`, `CODEX_MEMSPEC.md`, and all thirteen `tip*.txt` files. The tips converge hard on the same winning pattern: **MIRIX/Hindsight taxonomy + Graphiti temporal truth + A-MEM concept evolution + Mem0/Hindsight fast multi-signal recall + HippoRAG graph activation + Voyager procedural skills + Git/ledger provenance + strict context-pack compilation.** The pasted ANTIGRAVITY CMC spec is directionally excellent, but it overclaims perfection and under-specifies the canonical mutation ledger, retrieval hot path, compression hierarchy, and science/math validation layer. The Claude spec is strongest on Rust schemas and cognitive stores. The Codex spec is strongest on source-controlled memory and practical rollout. My design below fuses them and tightens the parts that matter for speed, scale, and scientific compounding.

The design I would build is:

# MNEMOS-Ω: Rust Cognitive Memory Compiler

**One-line definition:** MNEMOS-Ω is a Rust-first, event-sourced, temporally grounded, self-compressing cognitive memory compiler that turns raw agent experience, literature, code runs, equations, datasets, and research failures into small, cited, task-specific context packs — fast enough for real-time use and strict enough for scientific work.

This is **not a vector database**. It is not just a graph database. It is not just an LLM wiki. It is a **compiler**:

```text
raw experience + sources + tool traces + code runs
  → typed memories
  → evidence-linked claims/equations/skills/concepts
  → temporal belief graph
  → compressed concept kernels
  → hybrid indexes
  → bounded context packs
  → better reasoning
  → feedback + consolidation + new search questions
```

External systems support this direction. MIRIX explicitly uses six memory components — core, episodic, semantic, procedural, resource, and knowledge — and supports BM25 plus embedding-based search across memory types. ([Mirix Documentation][1]) Graphiti/Zep’s core advantage is temporal knowledge graphs that ingest structured and unstructured data and query via time, full-text, semantic, and graph algorithms. ([Zep Documentation][2]) A-MEM contributes the self-organizing Zettelkasten idea: new memories generate structured notes, links, and updates to older context representations. ([NeurIPS Papers][3]) Hindsight’s public architecture is especially valuable because it separates `retain`, `recall`, and `reflect`, extracts structured facts, resolves entities, builds a knowledge graph, and uses multi-strategy retrieval rather than dumping text into vectors. ([Hindsight][4]) Mem0’s newer retrieval design validates the hybrid path: semantic search, BM25 keyword search, and entity matching fused into a single ranking. ([Mem0][5]) Voyager proves why executable skills must be first-class memory: its skill library stores and retrieves code behaviors, composes them, and improves through execution feedback. ([Voyager][6])

For OpenQG specifically, this also matches the repo’s existing direction: durable curated knowledge belongs under `research/knowledge`, while raw provider receipts, daemon output, and run state belong under `target/openqg/research/`.

# 1. The core principle

The agent should **never** ask: “What memories are similar to this query?”

It should ask:

```text
What is the smallest, most truthful, most useful memory program
that should be compiled into my context for this task?
```

That single shift changes everything.

A vector DB returns neighbors.
A graph DB returns relationships.
A wiki returns pages.
**MNEMOS-Ω returns a bounded executable context pack with provenance, contradictions, skills, equations, and next actions.**

# 2. What I would keep, change, and reject from the studied specs

## Keep from `CLAUDE_MEMSPEC.md`

The Claude spec’s 12-store catalog is strong: working, sensory, episodic, semantic, procedural, schema, metacognitive, emotional salience, theorem, equation, citation, and counterexample memory. That is better than the simpler six-lane MIRIX taxonomy for science and math because equations, theorems, citations, and counterexamples deserve first-class schemas.

Keep:

```text
- typed memory envelope
- lifecycle state machine
- hash-chained episodes
- FSRS + Hebbian strengthening
- predict-calibrate reflection
- Memory-R1-like policy
- equation/theorem specialization
- deterministic context pack
- Rust daemon + CLI
```

Change:

```text
- Replace sled as canonical storage with event log + redb tables.
- Keep graph projection rebuildable instead of making petgraph the durable truth.
- Split hot recall from deep recall so the fast path never calls an LLM.
- Make concept kernels explicit compressed artifacts, not just reflection outputs.
- Make belief state a first-class object separate from semantic claims.
```

## Keep from `CODEX_MEMSPEC.md`

The Codex spec’s strongest idea is the exact rule:

```text
Do not make the vector database the memory.
Make structured, source-controlled memory the truth.
Use graphs, vectors, observations, and skills as compiled indexes over it.
```

Keep:

```text
- raw events → observer → episode ledger → typed router
- Git-backed Markdown/YAML curated cards
- source spans and mutation receipts
- concept kernels for strong topics
- topic-strength explainability
- local recall without LLM
- deterministic rebuilds from canonical stores
```

Change:

```text
- Expand storage from SQLite baseline to pure-Rust redb baseline.
- Add dedicated formal/math compiler.
- Add context-pack evaluation memory.
- Add a real compression pyramid.
- Add a stronger multi-agent/daemon API.
```

## Keep from pasted ANTIGRAVITY CMC

The CMC synthesis is conceptually right:

```text
MIRIX for memory shape
Graphiti for temporal truth
A-MEM for concept evolution
Voyager for procedural action
Mastra/HippoRAG/Mem0 for speed and recall
```

Change:

```text
- Do not claim perfect 60/60.
- Do not rely on valid_from/valid_to only on edges; put temporal validity on claims, beliefs, skills, datasets, equations, and resources too.
- Do not discard stale knowledge automatically; mark it stale and exclude it from current packs unless historical mode asks for it.
- Do not use one generic MemoryObject for everything; use a common envelope plus strict payload schemas.
- Do not make LLM reflection part of hot retrieval.
```

# 4. Memory lanes

MNEMOS-Ω uses **13 lanes**. The trick is that they are not independent silos; they are typed views over one canonical event-sourced memory store.

| Lane             | Purpose                                                              |         Hot-path use | Compression target    |
| ---------------- | -------------------------------------------------------------------- | -------------------: | --------------------- |
| `Core`           | Pinned mission, invariant constraints, active identity               |               Always | Tiny immutable blocks |
| `Working`        | Current task state, scratchpad, active plan                          |               Always | Session-local         |
| `Sensory`        | Raw incoming events before extraction                                |                   No | Episode summaries     |
| `Episodic`       | What happened: tool calls, search runs, code failures, conversations |            Sometimes | Observations          |
| `Resource`       | Papers, PDFs, datasets, repos, files, APIs, commits                  |                Often | Resource cards        |
| `SemanticClaim`  | Atomic facts/claims/propositions                                     |           Very often | Claim clusters        |
| `Belief`         | Current stance over claims and evidence                              |   Often for judgment | Belief snapshots      |
| `Concept`        | Self-organizing concept kernels and topic maps                       |                Often | Multi-level kernels   |
| `Equation`       | Equations, units, assumptions, derivations                           |     Often in science | Equation DAG          |
| `Theorem`        | Theorems, proof sketches, formal hooks                               |        Often in math | Dependency closure    |
| `Procedural`     | Executable skills, scripts, workflows, tool recipes                  |      Often in coding | Tested skill cards    |
| `Counterexample` | Mistakes, falsifications, contradictions, failed attempts            |       Always checked | Failure kernels       |
| `Eval`           | Which memories helped or hurt prior answers/actions                  | Background + ranking | Policy training set   |

This expands MIRIX’s human-like taxonomy while keeping its useful separation of core, episodic, semantic, procedural, resource, and knowledge memories. ([Mirix Documentation][7])

# 5. Canonical storage model

The key engineering decision:

```text
Canonical truth = event log + typed records + Git cards.
Indexes = rebuildable projections.
LLM summaries = lossy views, never source of truth.
```

## 5.1 Canonical stack

For a Rust-first local system:

| Need                    | Baseline choice                            | Why                                                                                                                                                                                        |
| ----------------------- | ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Canonical object tables | `redb`                                     | Pure Rust, embedded, ACID, MVCC, crash-safe, copy-on-write B-trees. ([Docs.rs][8])                                                                                                         |
| Mutation log            | append-only binary/JSONL event log         | Replayable and audit-friendly                                                                                                                                                              |
| Full-text search        | Tantivy                                    | Rust Lucene-like full-text engine with inverted indexes and BM25. ([Spice AI][9])                                                                                                          |
| Vector search           | Qdrant local service or local HNSW adapter | Qdrant supports dense/sparse/multivector search and large-scale semantic retrieval. ([Qdrant][10])                                                                                         |
| Graph projection        | custom Rust CSR adjacency + optional Kuzu  | Hot path needs bounded activation; Kuzu is excellent for heavier embedded graph analytics with Cypher, columnar storage, CSR adjacency, ACID, and multicore execution. ([KuzuDB Docs][11]) |
| Local inference         | Candle                                     | Rust-native transformer inference and safetensors support. ([Hugging Face][12])                                                                                                            |
| Curated memory          | Git-backed Markdown/YAML                   | Human review, diffs, branchable science                                                                                                                                                    |
| Large artifacts         | content-addressed object store             | Do not bloat Git or canonical DB                                                                                                                                                           |

This distinction is absolutely central:

```text
Observation ≠ source assertion ≠ extracted claim ≠ agent belief.
```

Without that separation, the system will eventually hallucinate stale or unsupported “facts.”

The system should not treat untested code as a skill. It is a **candidate skill** until it passes tests and has known preconditions.

# 8. Memory lifecycle

```mermaid
stateDiagram-v2
    [*] --> Observed
    Observed --> Proposed: extraction
    Proposed --> Grounded: source span + hash
    Grounded --> Trusted: validation + confidence threshold
    Trusted --> Consolidated: repeated use / reflection
    Consolidated --> Curated: human or policy approval
    Curated --> Archived: low access, high stability

    Proposed --> Rejected: unsupported / duplicate / low confidence
    Grounded --> Contradicted: conflict found
    Trusted --> Contradicted: new evidence
    Consolidated --> Superseded: better version
    Curated --> Superseded: updated consensus

    Contradicted --> HistoricalOnly: valid_to closed
    Superseded --> HistoricalOnly: new memory replaces live use
    HistoricalOnly --> Archived

    Rejected --> [*]
```

No destructive overwrite. Ever.

When a claim changes, the system creates a mutation:

```text
old claim: valid_to = now
new claim: valid_from = now
edge: new_claim supersedes old_claim
counterexample: why old claim failed
belief: revised with reason
```

# 9. Write pipeline

The write path is allowed to be smarter and slower than recall, but it still needs a real-time layer.

```mermaid
sequenceDiagram
    participant Agent
    participant FreshRing
    participant Observer
    participant Ledger
    participant Router
    participant Extractor
    participant Linker
    participant Store
    participant Indexes
    participant Reflector

    Agent->>FreshRing: observe(event)
    FreshRing-->>Agent: immediate receipt, fresh recall visible
    FreshRing->>Observer: segment event
    Observer->>Ledger: append signed episode
    Ledger->>Router: route candidate lanes
    Router->>Extractor: extract claims/equations/entities/skills
    Extractor->>Linker: normalize + dedupe + entity link
    Linker->>Store: commit canonical mutation
    Store->>Indexes: async index updates
    Store->>Reflector: schedule consolidation if surprise/high utility
```

## Stages

```text
1. Accept event
2. Write to fresh ring
3. Segment into observations
4. Append hash-chained episode
5. Extract typed candidates
6. Normalize entities, units, predicates, code refs
7. Ground every candidate to provenance spans
8. Detect duplicates, refinements, contradictions
9. Score novelty, confidence, source quality, surprise
10. Commit canonical mutation
11. Update indexes asynchronously
12. Schedule reflection/consolidation if needed
```

## Fresh-ring trick

For real-time learning, do **not** wait for embeddings. Every new event goes into a tiny in-memory `FreshRing`:

```rust
pub struct FreshRing {
    recent: VecDeque<FreshObservation>,
    max_items: usize,
    max_age: chrono::Duration,
}
```

The hot recall path always checks:

```text
Core blocks
+ Working state
+ FreshRing
+ indexed long-term memory
```

That means the agent can use a fact seconds after observing it, even if entity linking and vector indexing are still running.

# 10. Read pipeline

The hot read path must not call a big LLM. It should compile context deterministically using prebuilt indexes.

```text
query
  → intent classification
  → entity/symbol extraction
  → parallel retrieval
      BM25 / exact
      dense vector
      sparse vector
      entity graph
      temporal graph
      skill precondition search
      counterexample search
  → fusion
  → bounded graph activation
  → rerank
  → contradiction/freshness filter
  → context-pack assembly
  → access/outcome logging
```

Qdrant’s support for dense, sparse, and multivector search is useful here, but the design should use multivectors carefully: Qdrant itself warns that indexing every token-level multivector can create heavy RAM and insert overhead, so use late-interaction multivectors only for reranking selected high-value candidates. ([Qdrant][13])

## Retrieval modes

| Mode       |             Latency target |                    LLM? | Purpose                                   |
| ---------- | -------------------------: | ----------------------: | ----------------------------------------- |
| `hot`      | p50 30–80 ms, p95 < 250 ms |                      No | normal agent step                         |
| `standard` |             p50 150–400 ms | Optional small reranker | coding/research context                   |
| `deep`     |                      1–8 s |                     Yes | multi-hop research, contradiction audit   |
| `sleep`    |              minutes/hours |                     Yes | consolidation, synthesis, search planning |

## Retrieval scoring

```text
score =
  0.20 * bm25_exact
+ 0.18 * dense_similarity
+ 0.12 * sparse_similarity
+ 0.12 * entity_overlap
+ 0.12 * graph_activation
+ 0.08 * temporal_relevance
+ 0.07 * source_quality
+ 0.06 * utility_from_past_success
+ 0.04 * confidence
+ 0.03 * freshness
- 0.10 * unresolved_contradiction_penalty
- 0.06 * stale_without_historical_mode
```

For math/code/symbolic questions, the compiler changes weights:

```text
bm25_exact ↑
symbol_index ↑
dense_similarity ↓
equation_graph ↑
```

For conceptual questions:

```text
concept_kernel ↑
GraphRAG community summary ↑
dense_similarity ↑
claim-level snippets ↓
```

For debugging/code questions:

```text
procedural skills ↑
failed attempts ↑
recent episodes ↑
exact error strings ↑
```

# 12. Knowledge compression pyramid

This is the biggest improvement over naive RAG.

```text
L0 Raw receipts
   PDFs, web pages, tool logs, API responses, code output
   Stored by hash, not normally retrieved

L1 Episodes
   “What happened?” compressed observations with time, actors, entities

L2 Atomic cards
   ClaimCard, EquationCard, TheoremCard, SkillCard, ResourceCard

L3 Concept kernels
   Cited topic summaries with prerequisites, claims, equations, failures, skills

L4 Field maps
   Domain-level maps: theories, datasets, open questions, contradictions

L5 Core principles
   Tiny high-stability axioms/heuristics allowed near core memory
```

The agent normally reads L3/L4 first, then drills down to L2/L1 only when needed.

That is how you handle millions of memories without flooding context.

GraphRAG’s value is exactly this higher-level organization: Microsoft describes GraphRAG as combining text extraction, network analysis, LLM prompting, and summarization to understand text datasets. ([Microsoft][14]) Its global search work also shows why community summaries matter for holistic questions that cannot be answered by local chunk retrieval alone. ([Microsoft][15])

## Compression jobs

```text
compress.topic(topic)
  Input: claims, equations, resources, episodes, failures, skills
  Output: ConceptKernel L3

compress.field(domain)
  Input: concept kernels + citation graph + open questions
  Output: FieldMap L4

compress.core(scope)
  Input: durable, repeatedly useful, low-contradiction kernels
  Output: CorePrinciple L5 candidate
```

## Compression rules

```text
Never summarize away provenance.
Never promote a claim into a concept kernel unless cited.
Never collapse contradictions into “consensus.”
Never delete raw receipts because a summary exists.
Never index every raw receipt equally; index durable structured cards first.
```

# 13. Self-organization and “sleep”

The system gets smarter during idle cycles.

```text
Every N minutes:
  - consolidate duplicate memories
  - update concept kernels
  - strengthen useful links
  - decay unused low-utility memories
  - scan contradictions
  - revise beliefs
  - generate new research questions
  - propose skills from repeated procedures
  - run retrieval eval probes
```

## Sleep-cycle algorithm

```rust
pub async fn sleep_cycle(scope: SleepScope) -> Result<SleepReport> {
    let clusters = cluster_recent_claims(scope).await?;
    let contradictions = scan_contradictions(&clusters).await?;
    let stale = find_stale_beliefs(scope).await?;
    let kernels = update_concept_kernels(&clusters, &contradictions).await?;
    let skills = propose_skills_from_repeated_episodes(scope).await?;
    let questions = generate_frontier_questions(&kernels, &stale).await?;
    let evals = run_retrieval_probes(scope).await?;

    commit_sleep_mutation(SleepMutation {
        kernels,
        belief_revisions: contradictions.into_revisions(),
        proposed_skills: skills,
        frontier_questions: questions,
        eval_results: evals,
    }).await
}
```

## Strengthening

Use three signals:

```text
1. Human-like spaced repetition:
   FSRS-style stability/difficulty/retrievability

2. Hebbian co-activation:
   memories retrieved and used together get stronger links

3. Outcome utility:
   memories that helped successful answers/code/debugging rise
```

FSRS is useful because it models difficulty, stability, and retrievability, and there is a Rust implementation with scheduler and training support. ([GitHub][16])

# 14. Procedural memory

Procedural memory must execute.

```mermaid
flowchart LR
    EP[Repeated episodes] --> CAND[Candidate skill]
    CAND --> TEST[Test generator]
    TEST --> SANDBOX[Sandbox run]
    SANDBOX --> PASS{Passes?}
    PASS -- No --> FAIL[Failure memory + patch proposal]
    FAIL --> CAND
    PASS -- Yes --> SKILL[Promoted SkillCard]
    SKILL --> INDEX[Skill index]
    INDEX --> RECALL[Retrieved by future tasks]
    RECALL --> OUTCOME[Outcome feedback]
    OUTCOME --> SKILL
```

Skill promotion rule:

```text
candidate → proposed skill
  only after repeated similar episodes or explicit human request

proposed → trusted skill
  only after tests pass in sandbox

trusted → curated skill
  only after repeated successful reuse

trusted → deprecated
  if environment, API, assumptions, or tests fail
```

A skill card should be retrieved as:

```text
[Skill: fetch_arxiv_pdf_v4]

Known failures: arXiv rate limit, withdrawn papers
```

Not as 300 lines of pasted code.

# 16. Retrieval compiler internals

```rust
pub async fn compile_context_pack(
    store: &MemoryStore,
    indexes: &IndexLayer,
    query: RecallQuery,
) -> Result<ContextPack> {
    let intent = classify_intent(&query);
    let entities = extract_entities_and_symbols(&query.text);

    let fresh = store.fresh_ring.search(&query).await;
    let core = store.core_blocks(&query).await;

    let (bm25, dense, sparse, entity_hits, graph_hits, time_hits, skill_hits, failures) =
        tokio::join!(
            indexes.bm25.search(&query, &intent),
            indexes.vector.dense_search(&query, &intent),
            indexes.vector.sparse_search(&query, &intent),
            indexes.entities.search(&entities),
            indexes.graph.activate(&entities, &intent),
            indexes.temporal.search(&query.time_scope),
            indexes.skills.search(&query),
            indexes.counterexamples.search(&query),
        );

    let candidates = fuse(FusionInput {
        fresh,
        core,
        bm25: bm25?,
        dense: dense?,
        sparse: sparse?,
        entity_hits: entity_hits?,
        graph_hits: graph_hits?,
        time_hits: time_hits?,
        skill_hits: skill_hits?,
        failures: failures?,
    });

    let expanded = bounded_expand(candidates, indexes.graph.policy_for(&intent)).await?;
    let reranked = rerank(expanded, &query, &intent).await?;
    let filtered = filter_temporal_and_lifecycle(reranked, &query)?;
    let packed = pack_budgeted(filtered, query.budget)?;

    store.log_retrieval(&packed).await?;
    Ok(packed)
}
```

# 18. Math/science specialization

This is where the system becomes more than an agent memory layer.

## Required scientific object separation

| Object               | What it means                             | Common failure prevented       |
| -------------------- | ----------------------------------------- | ------------------------------ |
| `ResourceCard`       | A paper/dataset/repo exists               | Treating a source as a claim   |
| `ClaimCard`          | A proposition extracted from evidence     | Treating belief as fact        |
| `EquationCard`       | Equation with variables/units/assumptions | Symbol hallucination           |
| `TheoremCard`        | Statement/proof/dependencies              | False proof reuse              |
| `BeliefState`        | Current agent stance                      | Stale consensus hallucination  |
| `CounterexampleCard` | Known failure/falsification               | Repeating mistakes             |
| `QuestionCard`       | Open frontier                             | Passive storage without search |
| `SkillCard`          | Executable tested procedure               | Pasting untrusted code         |

## Scientific belief update

```text
Observation:
  Source S reports result R under method M.

Claim:
  R supports proposition P under assumptions A.

Belief:
  The agent currently accepts/rejects/withholds P with confidence C
  because supports {C1..Cn} and contradictors {K1..Km}
  have weights {w1..wn, v1..vm}.
```

The system must answer:

```text
What did we observe?
Who asserted it?
What claim was extracted?
What assumptions apply?
What evidence supports it?
What evidence contradicts it?
What do we currently believe?
When did that belief change?
What test or dataset could reduce uncertainty?
```

# 19. Performance design

## Local single-node target

```text
Scale:
  10M MemoryEnvelope rows
  100M graph edges
  1M concept/claim/equation/resource high-value embeddings
  10M BM25 documents
  100k tested skills

Hot recall:
  p50: 30–80 ms
  p95: < 250 ms
  p99: < 750 ms

Standard recall:
  p50: 150–400 ms
  p95: < 1.5 s

Writes:
  observe immediate receipt: < 5 ms
  canonical commit: < 25 ms
  extraction/indexing: async

Deep research recall:
  1–8 s, allowed to use LLM/reranker/multi-hop search
```

## Why this can be fast

```text
- Hot recall does not call the LLM.
- Core + working + fresh memory are tiny.
- BM25, dense, sparse, graph, and temporal channels run in parallel.
- Graph expansion is bounded by intent, edge type, and activation budget.
- Concept kernels reduce need to retrieve thousands of raw items.
- Multivector reranking is used only after first-pass retrieval.
- Raw receipts are not embedded/indexed equally; structured cards are.
- Indexes are projections and can be aggressively optimized/rebuilt.
```

# 20. Evaluation suite

LoCoMo and LongMemEval are mandatory baselines because they test long-term recall, temporal reasoning, knowledge updates, and abstention. - graph/vector/lexical stores are projections, not truth
```text
- context pack is the central product
```

# 24. The OpenQG-specific version

For OpenQG, I would name the memory layer:

```text
openqg-memory/
  "evidence-gated scientific memory compiler"
```

The core cards:

```text
PaperCard
ClaimCard
EquationCard
TheoryCard
ObservableCard
DatasetCard
CodeRepoCard
SkillCard
BenchmarkCard
QuestionCard
CounterexampleCard
BeliefState
ConceptKernel
FieldMap
SearchRunCard
ContextPack
```

The key graph edges:

```text
Paper --asserts--> Claim
Claim --supports--> Belief
Claim --contradicts--> Claim
Claim --assumes--> Assumption
Theory --predicts--> Observable
Observable --measured_by--> Dataset
Dataset --downloaded_by--> Skill
Equation --formalizes--> Claim
Theorem --depends_on--> Theorem
Skill --implements--> Method
Benchmark --tests--> Prediction
Question --targets_gap_in--> ConceptKernel
Counterexample --falsifies--> Claim
```

The most important OpenQG rule:

```text
No theory candidate gets promoted because the model “likes it.”
It gets promoted only when claims, equations, named parameters,
observables, datasets, and benchmark adapters are all linked.
```

That aligns with the existing OpenQG admission/scoring philosophy: theory manifests need stable IDs, citations, observables, named parameters, and executable adapters, while benchmarks should reward named parameters, observable coverage, reproducible fixtures, and invalid-domain handling.

# 25. The final design decision

The optimal memory system is:

```text
Git-native where humans need review.
Event-sourced where machines need replay.
Graph-shaped where relationships matter.
Vector-assisted where language is fuzzy.
BM25/exact where symbols matter.
Temporal where truth changes.
Procedural where action compounds.
Compressed where context is scarce.
Cited where science demands trust.
```

The biggest mistake would be building “a better RAG database.”

The best system is a **cognitive compiler** that continuously turns experience into:

```text
evidence
→ claims
→ beliefs
→ concepts
→ skills
→ questions
→ better searches
→ better experiments
→ better memory
```

That is how the agent gets smarter without drowning in its own context window.
