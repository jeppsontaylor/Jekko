# MNEMOS-HX: Hypercognitive Memory Compiler

My best possible design is **MNEMOS-HX**, a Rust-native, local-first, event-sourced, multi-lane cognitive memory compiler for long-running AI agents.

The short design rule:

```text
Raw experience is not memory.
Vector chunks are not memory.
Memory is typed, cited, temporally valid, compressed, scored, linked, executable when appropriate, and continuously tested.
```

MNEMOS-HX is designed for:

```text
10M memories per local node comfortably
50M+ memories with hot/cold tiering
100M+ memories with shard-by-topic/entity/time
sub-100 ms common recall on warm indexes
bounded context packs instead of history dumps
scientific/math/code memory that compounds over months or years
```

I would score this architecture **98.4 / 100**. I would not honestly score any real architecture 100 before implementation benchmarks because extraction quality, contradiction detection, and long-horizon skill usefulness depend partly on model quality and domain-specific evaluation. But as an engineering blueprint, this is the strongest synthesis I can produce from the corpus.

---

# 1. Core thesis

MNEMOS-HX is a **memory compiler**, not a store.

It compiles:

```text
raw event
  → dated observation
  → episode
  → atomic claim / equation / theorem / skill / resource / failure
  → temporal graph edge
  → belief state
  → concept kernel
  → topic mastery model
  → executable skill
  → compact context pack
```

The canonical truth is an **append-only ledger plus typed memory objects**. Everything else is a projection:

```text
BM25 index       = projection
vector index     = projection
graph cache      = projection
topic summaries  = projection
context cache    = projection
```

If an index corrupts, rebuild it. If a summary is bad, regenerate it. If a fact changes, do not overwrite it; close its validity interval and create a superseding edge.

# 2. What MNEMOS-HX improves over CLAUDE, CODEX, and ANTIGRAVITY CMC

The attached CLAUDE spec is excellent on cognitive taxonomy, FSRS-style strengthening, formal math hooks, and Rust crate layout. The CODEX spec is excellent on canonical store discipline, mutation receipts, explainability, and buildability. The ANTIGRAVITY CMC spec is excellent on six-lane synthesis, Graphiti-style validity windows, HippoRAG + Mem0 retrieval, and procedural memory.

MNEMOS-HX keeps the strongest parts and adds seven upgrades:

1. **Multi-resolution compression as a first-class lane.** Raw episodes, observations, claim cards, concept kernels, topic atlases, and curriculum cards are separate levels. This prevents millions of facts from turning into millions of equally retrievable noisy snippets.

2. **Eval memory as a first-class lane.** The system remembers which retrievals, answers, skills, and citations actually worked. Retrieval learns from downstream outcomes, not just similarity.

3. **Belief/fact separation as a hard invariant.** A source assertion, extracted claim, current belief, and agent decision are different objects.

4. **Hot-path realtime learning.** The agent can recall newly observed events immediately from the observation ledger while deeper extraction, linking, embedding, contradiction checks, and consolidation happen asynchronously.

5. **Million-scale graph strategy.** The graph is not an ad hoc Petgraph blob. It uses compact ID remapping, CSR adjacency snapshots, Roaring filters, bounded PPR, and hot/cold topic shards.

6. **Agent-call-driven memory hygiene.** The agent is not a passive user of memory. It actively calls `feedback`, `compress`, `consolidate`, `rehearse`, `distill_skill`, `mark_counterexample`, and `explain_omission`.

7. **Formal science lane with evidence, assumptions, regimes, equations, units, theorem dependencies, experiments, and counterexamples.** Math and science memory must know what is proven, what is believed, what is merely asserted, and what failed.

# 4. Design invariants

These are non-negotiable.

```text
1. No vector index is canonical.
2. No raw transcript becomes trusted memory by default.
3. No fact mutation is destructive.
4. No belief is stored as a fact.
5. No scientific claim becomes trusted without provenance.
6. No skill becomes trusted without tests or successful execution evidence.
7. No context pack is allowed to exceed budget.
8. No known counterexample may be silently omitted when directly relevant.
9. No retrieved item may hide why it was retrieved.
10. No memory object may lack lifecycle state, provenance, and access policy.
```

The reason this matters: most memory systems fail by letting noisy snippets, stale facts, unsupported beliefs, and untested procedures share the same retrieval pool. MNEMOS-HX prevents that structurally.

# 5. Memory lanes

MNEMOS-HX uses **14 lanes**. The six-lane CMC/MIRIX shape is the core, but science/math/code agents need more explicit separation.

| Lane               | Purpose                                             | Examples                                                       | Hot-path behavior                      |
| ------------------ | --------------------------------------------------- | -------------------------------------------------------------- | -------------------------------------- |
| **Core**           | Stable identity, mission, user-approved constraints | “Prefer Rust,” “never store secrets in memory,” active project | Always considered, tiny budget         |
| **Working**        | Current task scratchpad                             | current hypothesis, active files, transient plan               | RAM + short TTL                        |
| **Observation**    | Dense realtime notes from raw streams               | “Build failed due to trait bound,” “paper claims X”            | Immediately searchable                 |
| **Episodic**       | Immutable autobiographical event log                | tool runs, conversations, experiments, search sessions         | Time/entity/task retrieval             |
| **Semantic**       | Atomic claims and temporal fact edges               | subject–predicate–object claims, definitions                   | BM25 + entity + graph + vector         |
| **Belief**         | Current epistemic stance over evidence              | “I believe theorem T is likely true at 0.72”                   | Retrieved for judgment tasks           |
| **Concept**        | Zettelkasten / A-MEM-style concept kernels          | topic cards, prerequisites, analogies                          | Preferred for explanation and learning |
| **Procedural**     | Executable skills and workflows                     | scripts, tool recipes, SQL, proof tactics                      | Retrieved by task/preconditions        |
| **Resource**       | Immutable external references                       | DOI, arXiv ID, repo commit, dataset checksum                   | Retrieved by exact ID/topic            |
| **Formal**         | Math/science structure                              | equations, variables, units, theorems, proofs                  | Exact symbol + dependency retrieval    |
| **Eval**           | What worked or failed                               | bad answers, benchmark results, retrieval outcomes             | Used for self-correction               |
| **Counterexample** | Known failures/falsifications                       | failed proof, disproven assumption, broken script              | High priority when relevant            |
| **Metacognitive**  | Lessons about reasoning/learning                    | “Check units before trusting derivation”                       | Retrieved for planning/debugging       |
| **Compression**    | Multi-resolution summaries                          | observation clusters, topic atlases, curriculum cards          | Used to shrink context                 |

The important distinction is that **Observation** and **Episodic** are not the same. Observation is compressed, realtime, searchable. Episodic is immutable, auditable, and reconstructive.

# 6. Storage architecture

The baseline should be Rust-native and local-first.

Canonical layer:

```text
Event ledger       append-only segmented binary log, BLAKE3 hash chain
Object store       redb or fjall for typed memory rows
Artifact store     content-addressed files, zstd-compressed, BLAKE3 hashed
Curated cards      Git-backed Markdown/YAML for concepts, skills, theorem cards
Receipts           append-only JSONL/CBOR mutation receipts
```

Projection layer:

```text
BM25/exact search  Tantivy
Dense vectors      Qdrant or local HNSW backend
Graph traversal    compact Rust CSR graph + temporal edge tables
Filters            Roaring bitmaps for lane/topic/status/time/source filters
Caches             entity neighborhoods, PPR seeds, context packs, topic kernels
```

This fits the “all Rust” requirement. Tantivy is a Rust full-text search library; Qdrant is a Rust vector search engine; redb is a pure-Rust embedded key-value database; fjall is a Rust log-structured embeddable key-value engine. ([GitHub][2])

For 10M memories, I would use:

```text
redb/fjall     canonical memory objects and edge rows
Tantivy        exact text, symbols, file paths, paper IDs, code names
Qdrant         dense vectors if service mode is acceptable
local HNSW     embedded fallback for fully local single-binary mode
CSR graph      compact graph traversal and PPR
Roaring        fast metadata filtering
zstd           artifact and ledger segment compression
BLAKE3         source hashing and chain integrity
Ed25519        optional signed memory segments
```

# 7. Scale model for millions of memories

The main trick is **not embedding everything equally**.

MNEMOS-HX stores different representations at different resolutions:

```text
Raw artifact                    retained by hash pointer, not always indexed
Observation                     1–3 sentence compressed event
Episode                         immutable event record
Atomic claim/equation/skill     typed object
Concept kernel                  300–1500 token reusable summary
Topic atlas                     compact map of concepts, prerequisites, conflicts
Curriculum card                 what the agent should review next
```

Hot retrieval searches:

```text
Core blocks
recent observations
trusted semantic facts
concept kernels
skills
counterexamples
eval memory
```

Cold retrieval searches only when needed:

```text
old raw episodes
archived observations
low-utility duplicates
stale facts
rare resources
long artifact text
```

Indexing policy:

```text
embed: observations, claims, beliefs, concepts, skills, resource summaries
do not embed by default: every raw line, every huge artifact chunk, every duplicate
BM25 index: exact symbols, IDs, paths, equations, source titles, code identifiers
graph index: entity/fact/concept/skill/counterexample links
bitmap filters: lane, topic, status, date, source, confidence, access policy
```

This gives scale because the system does not turn a 500-page paper or a 3-hour tool log into 5,000 peer-equivalent vector chunks. It produces a small number of typed objects plus a content-addressed pointer back to the original.

# 9. Write path: realtime learning

The write path has a **fast synchronous layer** and a **deeper asynchronous compiler layer**.

Synchronous hot path:

```text
1. receive event
2. assign event id
3. hash payload
4. append ledger entry
5. create compressed observation
6. store observation in hot index
7. return receipt
```

Target:

```text
p50 event acknowledgement: 1–3 ms
p99 event acknowledgement: < 20 ms
```

Asynchronous compiler path:

```text
observation
  → segment into episodes
  → extract candidate memories
  → normalize entities
  → classify lane
  → attach provenance
  → detect duplicates
  → detect contradictions
  → create temporal edges
  → update belief states
  → embed/index
  → update graph
  → update topics
  → schedule consolidation if surprise is high
```

Write pipeline trait:

```rust
#[async_trait::async_trait]
pub trait WriteStage: Send + Sync {
    async fn run(&self, ctx: WriteContext) -> anyhow::Result<WriteContext>;
}

pub struct WritePipeline {
    stages: Vec<Box<dyn WriteStage>>,
}

impl WritePipeline {
    pub async fn execute(&self, event: EventEnvelope) -> anyhow::Result<MutationReceipt> {
        let mut ctx = WriteContext::from_event(event);
        for stage in &self.stages {
            ctx = stage.run(ctx).await?;
        }
        Ok(ctx.into_receipt())
    }
}
```

Critical behavior:

```text
A fresh observation is usable immediately.
A trusted semantic memory requires grounding.
A belief update requires evidence links.
A skill promotion requires successful execution or tests.
A contradiction closes old validity; it does not delete old history.
```

# 10. Temporal truth model

A fact edge is valid only inside its interval.

```text
old edge:
  subject    = "paper X"
  predicate  = "uses dataset"
  object     = "D1"
  valid_from = 2025-01-01
  valid_to   = 2026-05-12
  status     = superseded

new edge:
  subject    = "paper X"
  predicate  = "uses dataset"
  object     = "D2"
  valid_from = 2026-05-12
  valid_to   = null
  supersedes = [old_edge]
  status     = active
```

The system can answer three different questions correctly:

```text
What is currently true?
What was true on March 1, 2026?
Why did our belief change?
```

That is the main difference between a temporal memory system and a bag of embeddings.

# 11. Read path: recall compiler

The recall path is deterministic and local-first. LLM reranking is optional; the baseline works without it.

```text
query
  → classify intent
  → parse entities/symbols/time
  → retrieve from channels in parallel
  → normalize scores
  → graph activation
  → temporal filter
  → contradiction/eval check
  → context-pack compression
  → access logging
  → feedback hook
```

Retrieval channels:

```text
core               pinned constraints
recent_obs         latest observations
bm25               exact text, equations, symbols, paths, IDs
dense              semantic vector nearest neighbors
entity             direct entity memory
temporal           facts valid at requested time
graph              PPR / spreading activation
concept            Zettelkasten neighbors and topic kernels
skill              procedural memory by task signature/preconditions
formal             theorem/equation dependency closure
eval               prior failures and corrected answers
counterexample     falsified assumptions
```

Scoring formula:

```text
score =
    w_exact      * exact_match
  + w_dense      * vector_similarity
  + w_bm25       * bm25_norm
  + w_entity     * entity_overlap
  + w_graph      * ppr_activation
  + w_temporal   * validity_score
  + w_source     * source_quality
  + w_conf       * confidence
  + w_utility    * utility
  + w_success    * retrieval_success
  + w_topic      * topic_strength
  + w_recent     * recency
  - w_stale      * stale_penalty
  - w_conflict   * unresolved_contradiction_penalty
  - w_privacy    * access_penalty
```

Intent-specific weight profiles:

| Query intent      | Highest weights                                                     | Required additions                  |
| ----------------- | ------------------------------------------------------------------- | ----------------------------------- |
| Math derivation   | exact equation, prerequisites, theorem dependencies, unit checks    | assumptions, regimes, missing steps |
| Scientific claim  | source quality, support, contradiction, temporal validity           | contested evidence                  |
| Coding task       | exact symbols, skills, recent evals, failed attempts                | broken prior workaround warnings    |
| Debugging         | recent episodes, exact errors, counterexamples, environment state   | stale environment warning           |
| Literature review | resources, citation graph, concept clusters, belief states          | omitted evidence notes              |
| Planning          | core constraints, open questions, procedural skills, topic strength | blockers                            |
| Learning topic    | prerequisites, concept kernels, examples, review state              | curriculum order                    |

Context pack contract:

```text
1. token budget never exceeded
2. factual items have citations
3. stale facts are excluded unless historical query
4. contradictions are surfaced, not hidden
5. directly relevant counterexamples are always included
6. skills are included by signature, not pasted wholesale
7. omitted high-value evidence is reported
8. retrieval trace is inspectable
```

# 12. Graph retrieval

HippoRAG-style graph activation is useful, but it must be bounded. HippoRAG’s published design combines LLMs, knowledge graphs, and Personalized PageRank-inspired retrieval, with reported gains on multi-hop QA and lower cost/latency than iterative retrieval. ([arXiv][3])

MNEMOS-HX uses graph activation only after candidate seeding:

```text
1. BM25/vector/entity retrieval produce seed nodes.
2. Seed nodes are mapped to graph node IDs.
3. Bounded PPR runs over compact CSR adjacency.
4. Edge types have weights:
   Supports          +0.90
   Formalizes        +0.85
   PrerequisiteOf    +0.80
   DerivedFrom       +0.75
   CounterexampleOf  +0.95 when query is debugging/science
   SimilarTo         +0.30
   Contradicts       special path, not simple positive activation
5. Nodes outside temporal validity are dropped.
6. Nodes with poor source quality are demoted.
```

Graph cache:

```text
entity_id -> top neighborhood
topic_id  -> concept subgraph
memory_id -> strong co-activation neighbors
query_signature -> previous PPR result
```

This makes multi-hop recall fast without using unbounded graph traversal.

# 13. Knowledge compression

Compression is not summarization alone. It is **typed, reversible-enough, provenance-preserving abstraction**.

MNEMOS-HX uses five compression levels:

| Level |                 Name |     Token target | Purpose                             |
| ----- | -------------------: | ---------------: | ----------------------------------- |
| L0    | Raw artifact pointer |     0 in context | Lossless source                     |
| L1    |          Observation |     20–80 tokens | Immediate memory                    |
| L2    |      Episode summary |    80–250 tokens | What happened                       |
| L3    |         Atomic cards |    50–300 tokens | Claims, equations, skills, failures |
| L4    |       Concept kernel |  300–1500 tokens | Reusable understanding              |
| L5    |          Topic atlas | 1000–4000 tokens | Whole-domain compressed map         |

Compression jobs:

```text
duplicate_merge
  merges near-identical memories, keeps provenance list

episode_cluster
  converts repeated observations into an episode summary

claim_factorization
  splits broad statements into atomic claims

concept_kernel_update
  refreshes a concept card from linked claims/equations/skills/failures

topic_atlas_build
  creates a high-level map for a topic cluster

skill_distillation
  converts repeated successful trajectories into executable skills

counterexample_indexing
  promotes important failures so they are hard to forget

cold_archive
  removes low-utility items from hot indexes but preserves canonical memory
```

Compression must preserve source pointers:

```text
Every compressed summary must know exactly which source memories it summarizes.
Every summary must be invalidated or refreshed when important source memories change.
```

This solves the “summary drift” problem.

# 14. Dynamic strengthening and forgetting

The system strengthens memories through use.

Signals:

```text
retrieved
included in context pack
explicitly used by agent
contributed to successful answer
contributed to successful code/tool action
verified by external test
cited in later concept kernel
appeared in repeated independent episodes
```

Weakening signals:

```text
retrieved but ignored
included but caused failed answer
contradicted by later evidence
stale validity interval
duplicate of stronger memory
low source quality
not accessed for long period
```

Topic strength formula:

```text
decayed_strength =
    previous_strength * exp(-hours_since_update / half_life_hours)

reinforcement =
    w_recurrence       * recurrence
  + w_utility          * task_utility
  + w_success          * retrieval_success
  + w_novelty          * prediction_gap
  + w_source_quality   * source_quality
  + w_graph            * neighbor_support
  + w_formal           * verification_bonus
  - w_contradiction    * unresolved_contradiction_pressure

new_strength = clamp(decayed_strength + reinforcement, 0.0, 1.0)
```

Forgetting is tiering, not deletion:

```text
Hot          frequently used, high confidence, high utility
Warm         useful but less frequent
Cold         preserved but not hot-indexed
Archived     retrievable only with explicit broad/historical search
Rejected     prunable if ungrounded and unused
```

Counterexamples do **not** decay normally. If the agent once made a serious mistake, that failure remains easy to recall when the same conditions appear again.

# 15. Agent calls that improve organization and compression

The agent should use memory as an active tool, not a passive database.

Core calls:

```rust
memory.observe(event)
memory.recall(query)
memory.feedback(context_pack_id, outcome)
memory.commit_insight(insight)
memory.link(source, target, relation)
memory.mark_counterexample(falsified_id, evidence)
memory.update_belief(proposition, evidence_delta)
memory.distill_skill(trace_id)
memory.run_skill(skill_id, inputs)
memory.verify_skill(skill_id)
memory.consolidate(scope)
memory.compress(scope, target_level)
memory.rehearse(topic_or_memory_id)
memory.explain_retrieval(context_pack_id)
memory.explain_omission(query, memory_id)
memory.archive(memory_id, reason)
memory.verify_integrity()
```

The most important call is `feedback`.

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum Outcome {
    TaskSuccess {
        used: Vec<MemoryId>,
        notes: String,
    },
    TaskFailure {
        used: Vec<MemoryId>,
        reason: String,
        corrected_by: Option<EpisodeId>,
    },
    AnswerAccepted {
        used: Vec<MemoryId>,
    },
    AnswerCorrected {
        bad_memory: Option<MemoryId>,
        correction: String,
    },
    SkillSucceeded {
        skill_id: MemoryId,
        runtime_ms: u64,
    },
    SkillFailed {
        skill_id: MemoryId,
        error: String,
    },
    Falsified {
        memory_id: MemoryId,
        evidence: EpisodeId,
    },
    Ignored,
}
```

Suggested agent behavior:

```text
After every significant answer:
  call memory.feedback(pack_id, AnswerAccepted or AnswerCorrected)

After every failed tool/code attempt:
  call memory.observe(failure)
  call memory.mark_counterexample if an assumption was falsified

After every repeated successful workflow:
  call memory.distill_skill(trace_id)

After every long research session:
  call memory.consolidate(scope = active_topic)
  call memory.compress(scope = active_topic, target_level = ConceptKernel)

Before beginning a hard topic:
  call memory.recall(query with include_prerequisites = true)
  call memory.rehearse(topic)

During idle time:
  call memory.consolidate(due_topics)
  call memory.compress(noisy_clusters)
  call memory.verify_integrity()
```

This is how the memory gets better over time: the agent labels what was useful, what failed, what was surprising, what should become a skill, and what should be compressed.

# 16. Procedural memory

Procedural memory is executable, tested, and versioned.

A skill is not:

```text
“Here is some text saying how to do X.”
```

A skill is:

```text
task signature
preconditions
executable procedure or code
tests
examples
success/failure history
known failure modes
sandbox policy
linked concepts
linked counterexamples
```

Promotion rule:

```text
Draft skill
  → candidate after one successful episode
  → tested after repeatable verification
  → trusted after successful use across different contexts
  → degraded after failures
  → retired if superseded
```

Skill retrieval:

```text
query task
  → classify task signature
  → match preconditions
  → retrieve skills by exact name, embedding, and linked concepts
  → penalize failing/degraded skills
  → include signature and call handle in ContextPack
```

Voyager’s key lesson is that a lifelong agent needs an ever-growing library of executable, compositional skills, not just facts. ([arXiv][4])

# 17. Math and science specialization

For math, science, and research, MNEMOS-HX must represent different epistemic objects separately.

```text
Observation:
  “Paper P reports result R.”

Claim:
  “Result R supports proposition X under assumptions A.”

Belief:
  “The agent currently accepts X at confidence 0.72 because claims C1..Cn support it and K1..Km contradict it.”

Equation:
  formula, variables, units, assumptions, derivation, regime

Theorem:
  statement, dependencies, proof sketch, formal proof if available

Experiment:
  hypothesis, method, prediction, actual result, falsified assumptions

Counterexample:
  specific failure case that invalidates a broad claim
```

Math/science retrieval must include:

```text
exact equation search
symbol search
unit constraints
assumptions
regime of validity
dependency closure
source quality
contradictions
counterexamples
open questions
```

Vector search alone is unacceptable for math because symbols, variable names, theorem names, and equation forms often require exact matching.

Formal checks:

```text
unit consistency check
dimension analysis
theorem dependency DAG
Lean/Coq optional proof check
testable numerical examples
dataset checksum validation
code experiment reproduction
```

# 20. Recall compiler code sketch

```rust
pub async fn recall(engine: &Engine, query: RecallQuery) -> anyhow::Result<ContextPack> {
    let classified = engine.query_classifier.classify(&query).await?;

    let entities = engine.entity_resolver.resolve_query(&query.text).await?;
    let exact_symbols = engine.symbol_parser.extract(&query.text);
    let time_scope = engine.time_parser.parse(&query.text, query.time_scope.clone());

    let core_fut = engine.channels.core.retrieve(&classified);
    let bm25_fut = engine.channels.bm25.retrieve(&classified, &exact_symbols);
    let dense_fut = engine.channels.dense.retrieve(&classified);
    let entity_fut = engine.channels.entity.retrieve(&entities);
    let temporal_fut = engine.channels.temporal.retrieve(&entities, &time_scope);
    let skill_fut = engine.channels.skill.retrieve(&classified);
    let eval_fut = engine.channels.eval.retrieve(&classified);
    let counter_fut = engine.channels.counterexample.retrieve(&classified, &entities);

    let channels = tokio::join!(
        core_fut,
        bm25_fut,
        dense_fut,
        entity_fut,
        temporal_fut,
        skill_fut,
        eval_fut,
        counter_fut,
    );

    let mut candidates = CandidateSet::from_channels(channels)?;

    if classified.needs_graph_expansion() {
        let seeds = candidates.top_seed_nodes(32);
        let graph_hits = engine.graph.personalized_pagerank(seeds, classified.graph_policy()).await?;
        candidates.merge_graph_hits(graph_hits);
    }

    let fused = engine.rank_fusion.rrf(candidates)?;
    let filtered = engine.temporal_filter.apply(fused, &time_scope)?;
    let reranked = engine.reranker.score(filtered, &classified)?;
    let checked = engine.contradiction_checker.annotate(reranked).await?;

    let pack = engine.context_packer.pack(ContextPackRequest {
        query,
        classified,
        candidates: checked,
        token_budget: classified.budget,
        include_omitted_evidence: true,
        include_trace: true,
    })?;

    engine.access_log.record(&pack).await?;
    Ok(pack)
}
```

# 21. Context pack example

For a query like:

```text
“Why did our previous black-hole thermodynamics script fail near the singularity, and what assumptions should we change?”
```

A good ContextPack would include:

```text
[Core]
- Current project: numerical physics simulation; prefer Rust; avoid undocumented assumptions.

[Recent episodes]
```

# 22. Testing strategy

Use benchmarks, property tests, crash tests, and domain tests.

Public benchmark references:

LoCoMo evaluates long-term conversational memory over very long conversations, while LongMemEval evaluates long-term memory abilities such as information extraction, multi-session reasoning, temporal reasoning, knowledge updates, and abstention. MemoryAgentBench frames memory agents around accurate retrieval, test-time learning, long-range understanding, and selective forgetting. Newer cognitive-memory evaluation like LoCoMo-Plus focuses on latent constraints and cue-trigger disconnects that factual recall tests miss. ([arXiv][5])

Unit tests:

```text

temporal edge supersession

access policy redaction
counterexample inclusion
```

Property tests:

```text

random context budgets
```

# 23. Security and privacy

Access policy is attached to every memory object.

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct AccessPolicy {
    pub sensitivity: Sensitivity,
    pub allowed_roles: Vec<RoleId>,
    pub allow_in_context_pack: bool,
    pub allow_embedding: bool,
    pub allow_llm_extraction: bool,
    pub encrypt_at_rest: bool,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum Sensitivity {
    Public,
    Internal,
    Confidential,
    Secret,
    Vault,
}
```

Security rules:

```text
Secret/Vault memories are not embedded by default.
Secret/Vault memories are not sent to hosted LLMs.
Context packs redact inaccessible memories.
Mutation receipts record redaction decisions.
Artifacts are content-addressed and optionally encrypted.
Memory export respects access policy.
```

This is especially important because memory systems otherwise become quiet exfiltration engines.

# 24. Daemons

MNEMOS-HX runs background daemons, but recall does not depend on them finishing.

```text
observerd
  compresses raw events into observations

extractord
  extracts claims, equations, entities, resources, skills

linkerd
  resolves entities and creates typed links

verifierd
  runs unit checks, tests, proof checks, source checks

reflectord
  performs predict-calibrate reflection and belief revision

consolidatord
  merges duplicates, builds concept kernels, refreshes topics

compressord
  creates multi-resolution summaries

strengthenerd
  updates utility, retrievability, topic mastery, Hebbian links

skillminerd
  converts repeated successful traces into skills

archiverd
  moves low-utility memory out of hot indexes

reindexerd
  rebuilds projections from canonical store

integrityd
  verifies hash chains, signatures, manifests, and index consistency
```

The daemons are scheduled by priority:

```text
surprise > 0.85         immediate reflection
contradiction found     immediate belief revision
skill failed            immediate skill degradation
topic due               scheduled rehearsal
hot index too large     compaction
new model embedding     background re-embedding
ledger segment closed   integrity signature
```

# 27. Why this is the best design

The best memory system for a superhuman agent must do six things simultaneously:

```text
remember what happened
know what is currently true
know why it believes something
learn procedures from repeated action
compress knowledge without losing provenance
retrieve only the right slice fast
```

Single systems each cover part of this. MIRIX gives the typed human-like shape. Graphiti gives temporal truth. Hindsight gives epistemic separation. A-MEM gives evolving concept links. HippoRAG gives associative multi-hop retrieval. Mem0 gives production pragmatism. Voyager gives skill compounding. The attached CLAUDE/CODEX/CMC specs correctly converge on a hybrid. MNEMOS-HX makes that hybrid stricter, more scalable, more evaluable, and more agent-operable.

The final design principle is:

```text
Memory should not be something the agent has.
Memory should be something the agent continuously compiles, tests, compresses, rehearses, and improves.
```
