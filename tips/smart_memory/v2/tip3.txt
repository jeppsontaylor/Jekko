# MNEMOS-CMC: a Rust-first superhuman memory compiler

The best architecture is **not** an LLM wiki, not a vector database, not a GraphRAG app, and not a chat-history compactor.

It is a **memory compiler**.

The canonical truth is a typed, event-sourced, provenance-locked memory ledger. Everything else—graph DB, vector DB, BM25, concept maps, observations, topic summaries, procedural skills, dashboards—is a compiled projection over that ledger.

The core rule:

> **Never make retrieval storage the memory. Make memory a typed, auditable, temporal, self-improving knowledge substrate; compile fast retrieval views from it.**

This design combines the strongest ideas in your corpus:

| Source idea                        | What to keep                                                                                                                                                                                                                                                |
| ---------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **MIRIX**                          | Six-lane cognitive taxonomy: core, episodic, semantic, procedural, resource, vault. MIRIX’s six-memory/multi-agent layout is the right top-level cognitive shape. ([Hugging Face][1])                                                                       |
| **Hindsight**                      | Separate world facts, experiences, entity summaries, and beliefs. A belief is not a fact.                                                                                                                                                                   |
| **Graphiti/Zep**                   | Temporal knowledge graph, validity windows, provenance, hybrid search, point-in-time truth. Graphiti explicitly tracks evolving facts with temporal metadata and supports semantic, BM25, and graph search. ([Zep Documentation][2])                        |
| **A-MEM**                          | Zettelkasten-style self-organizing concept memory. A-MEM is valuable because it updates links and memory structure as new memories arrive rather than freezing a static schema. ([Hugging Face][3])                                                         |
| **Mastra Observational Memory**    | Stable compressed observation prefix, Observer/Reflector pattern, prompt-cacheable context, high compression. Mastra reports an Observer/Reflector system with a stable context window and very high LongMemEval results. ([Mastra][4])                     |
| **HippoRAG / GraphRAG / LightRAG** | Multi-hop graph activation, local/global retrieval, community summaries, DRIFT-like traversal. GraphRAG’s query engine separates local search, global search, DRIFT search, basic search, and question generation. ([Microsoft GitHub][5])                  |
| **Mem0**                           | Fast multi-signal retrieval: semantic, BM25, entity matching, ADD-only accumulation, low-token context. The current Mem0 README describes single-pass ADD-only extraction, entity linking, BM25, semantic retrieval, and fused scoring. ([GitHub][6])       |
| **Voyager/Memento**                | Executable procedural memory: skills that are tested, reused, refined, and composed. Voyager’s architecture included an automatic curriculum, an executable skill library, and iterative feedback-driven code improvement. ([Science Explorer][7])          |
| **Memory-R1**                      | Learned memory manager policy: ADD, UPDATE, DELETE, NOOP—but in this design, DELETE is logical archival, not destructive deletion. ([Hugging Face][8])                                                                                                      |
| **LongMemEval / Ragas**            | Evaluation discipline. LongMemEval tests information extraction, multi-session reasoning, temporal reasoning, knowledge updates, and abstention; Ragas exposes retrieval, faithfulness, entity recall, tool-use, and agent-goal metrics. ([Cool Papers][9]) |

My final system is:

> **MNEMOS-CMC: Multi-lane, Event-sourced, Neural-symbolic, Epistemic, Memory Operating System / Cognitive Memory Compiler.**

Final score on the rubric below: **97.4 / 100**.

Not 100, because no external memory system can fully eliminate LLM extraction errors, ambiguity in scientific sources, or adversarial provenance problems. But this is the strongest buildable design I would trust for long-horizon scientific, mathematical, coding, and research agents.

## 1.2 Why this beats the CMC draft

Your ANTIGRAVITY CMC is very close. I would keep the six-lane architecture, temporal graph, provenance, procedural sandbox, and retrieval compiler. But I would harden it in five ways:

1. **Event-sourced ledger first.**
   The ledger is the only canonical mutation source. SQLite/Kuzu/Qdrant/Tantivy are projections.

2. **Beliefs separate from facts.**
   A system that stores “what the agent currently believes” in the same lane as “what a source asserted” will eventually corrupt itself.

3. **Observational memory as a hot-context stabilizer.**
   Mastra’s insight is huge: for active agent sessions, a stable compressed observation prefix can outperform dynamic retrieval because it preserves continuity and cacheability. Use it as the always-on autobiographical layer, not as the whole memory.

4. **Science-specific object model.**
   Physics/math needs claims, equations, assumptions, observables, datasets, code, theorems, and counterexamples as separate objects.

5. **Retrieval evals are first-class memories.**
   The memory system must remember when it retrieved the wrong thing, when an answer failed, and which context pack produced the failure.

## MNEMOS-CMC

**MNEMOS** = memory system.
**CMC** = cognitive memory compiler.

North star:

> Convert experience, documents, code, equations, experiments, and conversations into typed, cited, temporal, compressible, executable memory objects that can be compiled into tiny high-signal context packs in real time.

# 4. The twelve memory lanes

The uploaded specs debated 6 lanes versus 8 versus 12. The right answer is **12 logical lanes**, with six core lanes and six science/agent-specialized lanes.

| Lane          | Purpose                                                     | Hot path? |       Canonical? | Example                                                   |
| ------------- | ----------------------------------------------------------- | --------: | ---------------: | --------------------------------------------------------- |
| `Core`        | Mission, identity, active constraints, project constitution |       Yes |              Yes | “OpenQG does not commit raw upstream data.”               |
| `Working`     | Current task state, scratch goals, temporary variables      |       Yes | No, checkpointed | “Currently testing Planck preview script.”                |
| `Observation` | Dense event log replacing raw chat/tool history             |       Yes |              Yes | “2026-05-12: failed Rust build due to trait bound.”       |
| `Episodic`    | Immutable source-of-experience records                      | Sometimes |              Yes | Search run, tool call, paper ingestion, failed test.      |
| `Resource`    | External objects and immutable pointers                     | Sometimes |              Yes | arXiv paper, DOI, dataset URL, Git commit.                |
| `Semantic`    | Atomic claims, entities, relations                          |       Yes |              Yes | “Paper X claims Y under assumptions A.”                   |
| `Belief`      | Current evidence-weighted stance                            |       Yes |              Yes | “Accept claim C with 0.72 confidence due to S1/S2.”       |
| `Concept`     | Self-organized abstractions and topic kernels               |       Yes |              Yes | “Gauge invariance concept kernel.”                        |
| `Equation`    | Equations, variables, units, assumptions, regimes           |       Yes |              Yes | Friedmann equation with units and limiting cases.         |
| `Theorem`     | Theorems, proofs, dependencies, formal checks               | Sometimes |              Yes | Lemma, proof sketch, Lean hook.                           |
| `Procedural`  | Tested skills, scripts, workflows, adapters                 |       Yes |              Yes | “Download arXiv metadata and normalize citations.”        |
| `Eval`        | Retrieval/action results, failures, corrections             |       Yes |              Yes | “ContextPack missed contradiction; fixed by adding edge.” |

Why separate `Belief` from `Semantic`?

Because scientific memory must distinguish:

```text
Observation: A source says X.
Claim: X supports proposition P under assumptions A.
Belief: The agent currently accepts P at confidence C.
Truth: Unknown; represented through evidence, not asserted by the model.
```

That distinction is non-negotiable.

The mutation rule is strict:

```text
No semantic edge is overwritten to mean a different thing.
A new edge supersedes an old edge.
The old edge receives valid_to / tx_to.
Both remain queryable.
```

# 6. Storage architecture

## 6.1 Canonical versus projected storage

| Layer             | Engine                       |                 Canonical? | Why                                                                                                                                                                                                               |
| ----------------- | ---------------------------- | -------------------------: | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Mutation ledger   | Append-only segmented log    |                        Yes | Replayable truth source.                                                                                                                                                                                          |
| Object metadata   | SQLite WAL or redb           |                        Yes | Fast local transactions; schema validation.                                                                                                                                                                       |
| Curated cards     | Git-backed Markdown/YAML     | Yes for reviewed knowledge | Human review, diffs, branch/merge. Letta’s context repository idea supports Git-backed agent memory and concurrent memory subagents. ([Letta][10])                                                                |
| Raw artifacts     | Content-addressed file cache |              No / external | PDFs, tool outputs, screenshots; not committed.                                                                                                                                                                   |
| BM25 exact search | Tantivy                      |                 Projection | Symbols, equations, names, code identifiers.                                                                                                                                                                      |
| Vector search     | Qdrant or local HNSW         |                 Projection | Dense/sparse/multivector retrieval. Qdrant supports native hybrid dense/sparse search, metadata filters, reranking, and multivectors. ([Qdrant][11])                                                              |
| Temporal graph    | Kuzu or graph tables         |                 Projection | Fast graph traversal and temporal relationship queries. Kuzu is embedded, supports property graphs, Cypher, columnar storage, CSR adjacency, parallelism, and serializable ACID transactions. ([KuzuDB Docs][12]) |
| Analytics         | DuckDB / Parquet             |                 Projection | Evals, coverage, latency, retrieval quality.                                                                                                                                                                      |
| Context cache     | mmap cache                   |                 Projection | Reuse frequent context packs.                                                                                                                                                                                     |

## 6.2 Recommended baseline stack

For OpenQG and research-agent use:

```text
Canonical:
  - append-only ledger: custom Rust segmented WAL
  - object store: SQLite WAL first, redb optional pure-Rust backend
  - curated cards: Git Markdown/YAML
  - receipts: JSONL + BLAKE3 hash chain

Search:
  - Tantivy for BM25/exact search
  - Qdrant sidecar for dense/sparse/multivector retrieval
  - Kuzu for graph projection
  - Roaring bitmaps for tag/time/entity set intersections

Runtime:
  - Tokio actors
  - Rayon for CPU-heavy batch jobs
  - Candle/ONNX Runtime for local embeddings/rerankers where possible
  - External LLM APIs behind traits
```

I would not force pure Rust at the cost of worse science. Use Rust as the control plane and memory kernel; allow proven C/C++/Fortran/scientific libraries behind adapters where needed.

# 8. Write path

The write path is asynchronous and event-sourced. The hot conversation/tool loop should not block on full extraction.

```mermaid
sequenceDiagram
    participant Agent
    participant Gateway
    participant Ledger
    participant Observer
    participant Extractor
    participant Validator
    participant Store
    participant Indexes
    participant Reflector

    Agent->>Gateway: observe(EventEnvelope)
    Gateway->>Ledger: append Episode + hash receipt
    Ledger-->>Agent: fast receipt

    Gateway->>Observer: segment + compress event
    Observer->>Store: write ObservationCard

    Gateway->>Extractor: async extract candidates
    Extractor->>Validator: claims/equations/resources/skills
    Validator->>Store: proposed/trusted memory mutations
    Store->>Ledger: append MutationReceipt
    Store->>Indexes: fanout index update

    Reflector->>Store: periodic concept/belief/topic updates
    Reflector->>Ledger: append ReflectionReceipt
    Reflector->>Indexes: update projections
```

# 9. Read path: the ContextPack compiler

A memory system is only as good as the context it compiles.

The read path must be **fast, bounded, cited, deterministic, and explainable**.

```mermaid
flowchart LR

    B --> C1[Core Blocks]

    B --> C3[BM25 Exact]
    B --> C4[Dense Vector]
    B --> C5[Sparse Vector]
    B --> C6[Temporal Graph]

    C2 --> D
```

## 9.3 Ranking formula

```text
score =
  w_exact        * exact_match
+ w_dense        * dense_similarity
+ w_sparse       * sparse_similarity
+ w_entity       * entity_overlap
+ w_graph        * graph_activation
+ w_temporal     * temporal_validity
+ w_source       * source_quality
+ w_confidence   * confidence
+ w_utility      * utility
+ w_topic        * topic_strength
+ w_skill        * skill_reliability
+ w_recent_eval  * prior_success
- w_stale        * stale_penalty
- w_contradict   * unresolved_contradiction_penalty
- w_dup          * redundancy_penalty
```

The ranking weights are profile-specific:

| Intent              | High-weight channels                                                    |
| ------------------- | ----------------------------------------------------------------------- |
| Math derivation     | BM25 exact symbols, equation graph, theorem DAG, prerequisites, units   |
| Literature review   | resources, citation graph, concept kernels, contradiction clusters      |
| Coding              | procedural skills, eval failures, exact code symbols, recent episodes   |
| Debugging           | error exact match, failed attempts, environment episodes, skill history |
| Planning            | core constraints, concept maps, open questions, proven skills           |
| Scientific judgment | claims, evidence quality, contradictions, beliefs, source freshness     |

# 10. Compression architecture

The system needs millions of memories, but the agent needs maybe 4k–32k tokens. The solution is **multi-resolution compression**, not summarizing everything into mush.

```mermaid
flowchart TB
    R0[Raw stream<br/>messages, tool logs, PDFs, code output] --> R1[Episodes<br/>immutable source events]
    R1 --> R2[Observations<br/>dense dated event notes]
    R2 --> R3[Atomic cards<br/>claims, equations, resources, skills]
    R3 --> R4[Concept kernels<br/>topic-level compressed knowledge]
    R4 --> R5[Field maps<br/>large-domain synthesis]
    R5 --> R6[Core memory<br/>mission and active constraints]

    R1 -.re-hydrate.-> R0
    R3 -.cite.-> R1
    R4 -.links.-> R3
    R5 -.links.-> R4
    R6 -.selects.-> R5
```

## 10.1 Compression tiers

| Tier | Name            |        Stored? |         Prompt use | Purpose                                         |
| ---- | --------------- | -------------: | -----------------: | ----------------------------------------------- |
| C0   | Raw artifact    | External/cache |     Never directly | Full fidelity rehydration.                      |
| C1   | Episode         |            Yes |             Rarely | What happened, source hash, provenance.         |
| C2   | Observation     |            Yes |              Often | Stable compressed autobiographical memory.      |
| C3   | Atomic card     |            Yes |              Often | Claim/equation/skill/resource-level retrieval.  |
| C4   | Concept kernel  |            Yes |         Very often | Compact reusable understanding.                 |
| C5   | Topic/field map |            Yes | Often for planning | Global synthesis and navigation.                |
| C6   | Core block      |            Yes |             Always | Mission, constraints, identity, active project. |

## 10.2 Compression laws

1. **Never compress away provenance.**
2. **Never let summaries become the only source of truth.**
3. **Never summarize contradictions into false consensus.**
4. **Never vectorize raw noise as the primary memory.**
5. **Always keep a path from field map → concept → claim/equation → source span.**

# 11. Topic strengthening and memory growth

A superhuman memory does not just store. It gets better at knowing what matters.

## 11.1 Topic state

```rust
pub struct TopicState {
    pub id: TopicId,
    pub label: String,
    pub aliases: Vec<String>,
    pub member_entities: Vec<EntityId>,
    pub member_memories: Vec<MemoryId>,

    pub strength: f32,
    pub mastery: f32,
    pub half_life_hours: f32,

    pub recency: f32,
    pub recurrence: f32,
    pub utility: f32,
    pub novelty: f32,
    pub source_quality: f32,
    pub contradiction_pressure: f32,
    pub retrieval_success: f32,
    pub graph_neighbor_boost: f32,

    pub last_strengthened_at: Option<DateTime<Utc>>,
    pub next_review_at: Option<DateTime<Utc>>,
}
```

## 11.2 Topic score

```text
decayed =
  previous_strength * exp(-hours_since_update / half_life_hours)

reinforcement =
  0.15 * recency
+ 0.15 * recurrence
+ 0.20 * utility
+ 0.12 * novelty
+ 0.14 * source_quality
+ 0.14 * retrieval_success
+ 0.08 * graph_neighbor_boost
- 0.18 * unresolved_contradiction_pressure

new_strength = clamp(decayed + reinforcement, 0.0, 1.0)
```

## 11.3 Strengthening triggers

| Trigger                               | Action                                                  |
| ------------------------------------- | ------------------------------------------------------- |
| Topic frequently retrieved and useful | Strengthen topic, improve concept kernel.               |
| Topic retrieved but answer failed     | Add Eval memory; lower retrieval profile; queue defrag. |
| Topic high-importance but weakening   | Schedule rehearsal/self-quiz.                           |
| Topic high-utility but low-confidence | Generate targeted evidence search.                      |
| Topic high-contradiction              | Trigger belief revision and source audit.               |
| Topic rapidly growing                 | Split into subtopics.                                   |
| Topic redundant                       | Merge concepts with receipt.                            |
| Topic stale                           | Demote from hot retrieval, keep in archive.             |

This is where knowledge compounds: memory reorganizes around utility, evidence, and surprise.

# 12. Reflection, consolidation, and defragmentation

## 12.1 Background daemons

| Daemon                |          Frequency | Job                                                            |
| --------------------- | -----------------: | -------------------------------------------------------------- |
| `observer`            |         continuous | Converts raw stream chunks into dense observations.            |
| `extractor`           |        async queue | Extracts claims, equations, resources, skills, evals.          |
| `reflector`           |   idle / threshold | Creates concept kernels, detects patterns, removes redundancy. |
| `consolidator`        |          scheduled | Merges duplicates, promotes trusted cards, updates beliefs.    |
| `defragger`           | weekly / triggered | Reorganizes concept maps and Git cards.                        |
| `rehearser`           |          scheduled | Reviews weakening high-value knowledge.                        |
| `frontier`            |          scheduled | Generates new questions/searches from gaps.                    |
| `skill_verifier`      |   on skill changes | Runs tests and updates procedural reliability.                 |
| `retrieval_evaluator` |         continuous | Scores retrieval outcomes and pack quality.                    |
| `index_rebuilder`     |          scheduled | Rebuilds projections deterministically from canonical store.   |

## 12.2 Concept kernel generation

A concept kernel is the core object for compression.

```yaml
schema: concept-kernel-v1
```

## 12.3 Defragmentation

Defrag is not “summarize old memories.” It is structural cleanup.

Defrag jobs:

```text
- Split oversized concept cards.
- Merge duplicate concepts.
- Rename ambiguous concepts.
- Rewire weak links.
- Archive low-utility observations.
- Promote repeated observations into semantic cards.
- Demote unsupported claims.
- Regenerate field maps.
- Emit Git diffs and receipts.
```

The agent should be able to call:

```bash
mnemos defrag --scope topic:quantum-gravity --max-edits 50 --review-required
```

# 13. Procedural memory: executable skill compounding

Procedural memory must be executable, tested, and versioned.

## 13.1 Skill lifecycle

```mermaid
stateDiagram-v2
    [*] --> Proposed

    Tested --> Trusted: passes tests >= threshold
```

## 13.3 Promotion rule

A skill becomes `Trusted` only when:

```text
- It has a clear task signature.
- Preconditions are explicit.
- It links to at least one successful episode.
- It has a repeatable test.

- It has known failure modes.
```

# 14. Math and science specialization

This is the difference between a general memory app and a serious OpenQG/science substrate.

## 14.1 Scientific object separation

| Object               | Stores                                                        | Must not store               |
| -------------------- | ------------------------------------------------------------- | ---------------------------- |
| `ResourceCard`       | Paper/dataset/repo metadata, license, hash, retrieval receipt | Model’s interpretation       |
| `ClaimCard`          | Atomic claim, assumptions, evidence, contradictions           | Uncited belief               |
| `EquationCard`       | LaTeX, normalized form, variables, units, derivation          | Raw unparsed PDF chunk only  |
| `TheoremCard`        | Statement, assumptions, proof sketch, formal proof ref        | Unchecked theorem as trusted |
| `ObservableCard`     | Measurable quantity, units, dataset/instrument links          | Vague theory prose           |
| `DatasetManifest`    | Download script, checksum, schema, license, preview policy    | Raw dataset dump             |
| `BeliefCard`         | Evidence-weighted stance                                      | Source assertion             |
| `CounterexampleCard` | Falsifying case, failed prediction, contradiction             | Hidden failure               |
| `QuestionCard`       | Open frontier, blockers, next searches                        | Generic curiosity text       |
| `SkillCard`          | Executable method, tests, success/failure                     | Untested code snippet        |

## 14.2 Equation card example

```yaml
schema: equation-card-v1
id: equation:friedmann:first:v1
latex: "H^2 = \\frac{8\\pi G}{3}\\rho - \\frac{k}{a^2} + \\frac{\\Lambda}{3}"
normalized_symbolic: "Eq(H**2, 8*pi*G*rho/3 - k/a**2 + Lambda/3)"
variables:
  - symbol: H
    meaning: Hubble parameter
    units: "s^-1"
  - symbol: G
    meaning: gravitational constant
    units: "m^3 kg^-1 s^-2"
  - symbol: rho
    meaning: mass-energy density
    units: "kg m^-3"
  - symbol: a
    meaning: scale factor
    units: "dimensionless"
assumptions:
  - homogeneous and isotropic FLRW metric
  - classical general relativity
regime_of_validity: "cosmological scales under FLRW assumptions"
linked_claims:
  - claim:flrw:dynamics:first-friedmann

  - resource_id: resource:...

  dimensional_check: passed

```

## 14.3 Science-specific validators

```text
Claim validator:
  - source span required
  - assumptions required
  - support/contradiction list explicit
  - status cannot be Trusted without evidence

Equation validator:
  - source span required
  - variables defined
  - units parsed where possible
  - dimensional consistency checked
  - regime of validity present

Theorem validator:
  - assumptions listed
  - dependencies linked
  - proof sketch required
  - formal check optional but status reflects absence

Dataset validator:
  - no raw data committed
  - license recorded
  - download script recorded
  - checksum strategy recorded
  - preview policy recorded

Skill validator:
  - test required for Trusted
  - sandbox required
  - failure memory retained
```

# 16. Real-time learning loop

The system should learn in real time without blocking the active agent.

```mermaid
flowchart LR
    A[User/tool event] --> B[Fast append<br/><5ms target]
    B --> C[Observation compression<br/>async]
    C --> D[Hot observation prefix]

    B --> E[Extraction queue]
    E --> F[Candidate memories]
    F --> G[Validation]
    G --> H[Canonical commit]
    H --> I[Index fanout]

    I --> J[Future recall]

    J --> K[Agent action]
    K --> L[Feedback]
    L --> M[Utility update]
    L --> N[Skill success/failure]
    L --> O[Retrieval eval memory]
    M --> P[Consolidation/reflection]
    N --> P
    O --> P
    P --> H
```

Hot path:

```text
event -> ledger append -> observation update -> return
```

Background path:

```text
extract -> validate -> commit -> index -> consolidate -> evaluate
```

This preserves responsiveness.

# 17. Performance design for millions of memories

## 17.1 Scale targets

| Operation                       |                         Target |
| ------------------------------- | -----------------------------: |
| Append episode hot-path p50     |                         ≤ 5 ms |
| Append episode p99              |                        ≤ 30 ms |
| Observation compression         |        async; not hot blocking |
| Warm recall simple p50          |                        ≤ 40 ms |
| Warm recall simple p95          |                       ≤ 150 ms |
| Hybrid recall with graph p50    |                       ≤ 120 ms |
| Hybrid recall with graph p95    |                       ≤ 500 ms |
| Heavy literature/science recall |                          ≤ 2 s |
| 10M memory object cold start    |      ≤ 5 s with mmap/snapshots |
| Projection rebuild              |       deterministic, resumable |
| ContextPack size                | 2k–32k tokens depending caller |

## 17.2 Scaling methods

| Problem                     | Method                                                                                           |
| --------------------------- | ------------------------------------------------------------------------------------------------ |
| Too many vectors            | Do not embed raw chunks; embed observations/cards/kernels.                                       |
| Vector RAM blowup           | Quantization, sharding by lane/topic/time, Qdrant scalar/product quantization where appropriate. |
| BM25 exact search scale     | Tantivy segmented indexes by lane.                                                               |
| Graph traversal explosion   | Bounded PPR, typed edge filters, temporal pruning, community summaries.                          |
| Duplicate memory growth     | Near-duplicate detection before commit; defrag; concept merging.                                 |
| Context bloat               | Multi-resolution compression and deterministic token budgets.                                    |
| Slow reranking              | Two-stage retrieval: cheap fusion first, late-interaction only top-N.                            |
| Stale facts                 | Temporal validity filter before context packing.                                                 |
| Contradiction hallucination | Always include contradiction warnings for contested beliefs.                                     |
| Skill rot                   | Skill tests and reliability scoring.                                                             |
| Scientific drift            | Belief revision and frontier re-search.                                                          |

## 17.3 Hot/warm/cold memory

| Tier   | Contents                                                            | Indexing                       | Retrieval            |
| ------ | ------------------------------------------------------------------- | ------------------------------ | -------------------- |
| Hot    | Core, active observations, active project concepts, recent failures | RAM + mmap                     | Always considered    |
| Warm   | Trusted cards, current topic maps, useful skills                    | BM25/vector/graph              | Normal recall        |
| Cold   | Old episodes, low-utility observations, archived resources          | Exact lookup + compressed maps | Rehydrate on demand  |
| Frozen | Historical raw artifacts, old snapshots                             | Content hash only              | Manual or deep audit |

# 24. The decisive design choices

## 24.1 The vector DB is not memory

Vector search is one retrieval projection. It is not truth, not provenance, not belief, not temporal validity, not procedural skill.

## 24.2 Observations are not summaries

Observations are dense event records:

```text
At time T, actor A did/said/observed X, with priority P, linked to episode E.
```

This keeps autobiographical memory stable.

## 24.3 Beliefs are not facts

Facts are source-grounded assertions. Beliefs are current stances over evidence.

## 24.4 Skills are not notes

A procedural memory must execute, test, fail, recover, and improve.

## 24.5 Compression must be reversible by provenance

You do not need to store all raw data in Git, but every compact memory should be able to point back to the source receipt or external resource.

## 24.6 Every retrieval becomes training data

Each ContextPack produces eval memory:

```text
What did we retrieve?
Was it useful?
What was missing?
What was wrong?
What should rank higher next time?
```

That is how the system gets smarter.

# 25. Final answer: the optimal design in one sentence

Build **MNEMOS-CMC**: a Rust-first, event-sourced, Git-auditable, multi-lane cognitive memory compiler with temporal graph truth, observation compression, hybrid BM25/vector/entity/graph retrieval, science-specific claim/equation/theorem/dataset schemas, executable procedural skills, feedback-driven topic strengthening, and deterministic context-pack compilation.

That is the memory system I would build for OpenQG and for any agent expected to compound knowledge across science, math, code, literature, data, experiments, and long-horizon reasoning.
