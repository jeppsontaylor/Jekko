# MNEMOS-Ω: the Superhuman Cognitive Memory Compiler

I unpacked and studied the archive: `CLAUDE_MEMSPEC.md`, `CODEX_MEMSPEC.md`, `tip1.txt` through `tip13.txt`, plus the small target state file. I also treated the pasted ANTIGRAVITY / Google CMC spec as the third major design input.

My final answer is: build **a Rust-native cognitive memory compiler**, not a memory database, not a vector store, and not an ever-growing prompt file.

The best design is a hybrid of:

**CLAUDE_MEMSPEC** for the richest typed Rust schema, strengthening loop, formal math/science stores, FSRS, and hash-chained episodes.

**CODEX_MEMSPEC** for the strongest local-first implementation contract: canonical stores, rebuildable indexes, mutation receipts, context-pack compiler, privacy boundaries, and agent-facing CLI/API discipline.

**ANTIGRAVITY CMC** for the clean six-lane compiler paradigm: Core, Episodic, Semantic Temporal Graph, Concept/Zettelkasten, Procedural Skills, and Resource Vault.

The external research baseline agrees with this direction: CoALA frames language agents around modular memory and internal/external actions; LongMemEval evaluates information extraction, multi-session reasoning, temporal reasoning, knowledge updates, and abstention; MemoryAgentBench emphasizes accurate retrieval, test-time learning, long-range understanding, and selective forgetting/conflict handling. ([arXiv][1])

The design below steals the best pieces from current systems: MIRIX’s six memory components and dedicated memory agents, Graphiti’s temporal-validity/provenance graph, Hindsight’s separation of facts/experiences/mental models, A-MEM’s Zettelkasten-style dynamic linking, Mastra Observational Memory’s Observer/Reflector compression, HippoRAG’s Personalized PageRank associative retrieval, Mem0’s production-focused salient memory and latency discipline, Voyager’s executable skill library, Nemori’s event segmentation/predict-calibrate learning, and Memory-R1’s learned ADD/UPDATE/DELETE/NOOP memory manager. ([MIRIX][2])

# 1. Executive decision

## Build: **MNEMOS-Ω**

**MNEMOS-Ω** is a local-first, Rust-native, append-only, typed, temporal, source-backed cognitive memory compiler for long-running AI agents.

Its job is not merely to “remember.” Its job is to continuously transform experience into increasingly useful forms:

```text
raw experience
→ observations
→ episodes
→ claims / facts / equations / skills / resources
→ concept kernels
→ beliefs / counterexamples / lessons
→ reusable procedures
→ task-specific context packs
→ feedback
→ better memory organization
```

The core maxim:

```text
The vector database is not memory.
The prompt window is not memory.
Raw chat history is not memory.

Memory is a typed, temporal, source-backed, evolving knowledge system.
Retrieval is compilation.
Context is the compiled artifact.
```

---

# 2. What makes MNEMOS-Ω better than the source specs

The three major specs are strong, but I would change five things to get the best possible future design.

First, I would make **bi-temporal truth** mandatory. Each fact needs both `valid_time` and `record_time`. This lets the system answer:

```text
What was true in the world on March 1?
What did the agent believe on March 1?
When did the agent learn that the old belief was wrong?
Why did it keep using an old script after the fact changed?
```

Second, I would make **beliefs separate from claims**. A paper claim, an observed event, and the agent’s current belief are not the same thing.

Third, I would add a **hot/warm/cold compression hierarchy**. Millions of memories cannot all live in hot HNSW/vector space. The system must continuously demote, compact, summarize, and preserve source rehydration.

Fourth, I would make **agent memory calls first-class**. The agent should not silently mutate memory. It should call explicit tools: `observe`, `recall`, `feedback`, `reflect`, `consolidate`, `compress`, `promote_skill`, `audit`, and `explain`.

Fifth, I would make **formal science/math memory** a top-level lane, not an afterthought. Equations, proofs, theorem dependencies, assumptions, units, datasets, experiments, and counterexamples need their own schemas.

# 4. The nine cognitive lanes

The cleanest final design is not six lanes and not twelve stores. It is **nine lanes**, with typed payloads inside each lane.

| Lane | Name                        | Purpose                                                                                         | Hot-path role                      |
| ---: | --------------------------- | ----------------------------------------------------------------------------------------------- | ---------------------------------- |
|    1 | **Core / Working**          | Stable mission, user-approved preferences, active task, current constraints, scratch state.     | Always lightly included.           |
|    2 | **Episodic**                | Timestamped episodes: tool runs, research sessions, conversations, failed builds, observations. | Recent continuity and provenance.  |
|    3 | **Semantic Temporal Graph** | Claims, facts, entities, relations, temporal validity, contradiction/supersession.              | Main truth graph.                  |
|    4 | **Formal Knowledge**        | Equations, units, assumptions, theorem DAGs, proof states, datasets, experiments.               | Math/science precision.            |
|    5 | **Concept / Schema**        | Zettelkasten concept kernels, prerequisite maps, analogies, topic summaries.                    | Compression and conceptual recall. |
|    6 | **Procedural**              | Executable skills, workflows, scripts, SQL, shell plans, tests, preconditions.                  | “How to do it.”                    |
|    7 | **Resource / Vault**        | Papers, PDFs, datasets, repos, file hashes, external URIs, source manifests.                    | Rehydration and citation.          |
|    8 | **Belief / Epistemic**      | Current stance over evidence: supported, contested, revised, retired.                           | Judgment and uncertainty.          |
|    9 | **Eval / Metacognitive**    | What worked, what failed, retrieval outcomes, lessons, bad answers, strategy changes.           | Self-improvement and ranking.      |

This is more powerful than MIRIX’s six-lane taxonomy because it splits out formal knowledge, beliefs, and eval/metacognition. It is more buildable than a sprawling twelve-store design because each lane maps to clear Rust enums, storage partitions, and retrieval policies.

# 5. Memory lifecycle

Every memory object has an explicit state. No destructive overwrite is allowed.

```mermaid
stateDiagram-v2
    [*] --> Observed
    Observed --> Proposed: extraction
    Proposed --> Trusted: grounded + validated
    Proposed --> Rejected: low confidence / bad source
    Trusted --> Consolidated: repeated use + stable evidence
    Trusted --> Contested: contradiction found
    Contested --> Revised: belief updated
    Trusted --> Superseded: newer fact closes validity window
    Consolidated --> Archived: low utility / cold tier
    Revised --> Consolidated: stable after review
    Superseded --> Archived
    Rejected --> Pruned: only if no provenance dependency
    Archived --> Rehydrated: explicit recall / source reload
```

Important rule:

```text
Counterexamples never decay.
Known failures remain retrievable forever.
```

That single rule prevents a huge class of repeated agent mistakes.

This separation is essential.

```rust
    pub confidence: f32,

    pub contradicting_claims: Vec<MemoryId>,
    pub counterexamples: Vec<MemoryId>,

    pub rationale: String,

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum BeliefStance {
    Unknown,
    Tentative,
    Supported,
    Contested,
    Revised,
    Retired,
}

#[derive(Debug, Clone, Serialize, Deserialize)]

    Entity(EntityId),
    Text(String),
    Number(f64),
    Bool(bool),

```

Rule:

```text
A claim says: "Source S asserts proposition P."
A fact edge says: "The memory graph records P as valid under conditions C."
A belief says: "The agent currently accepts, contests, or rejects P because of evidence E."
```

## 6.5 Procedural skill payload

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SkillPayload {
    pub name: String,
    pub description: String,
    pub language: SkillLanguage,

    pub code_ref: CodeRef,
    pub entrypoint: String,

    pub preconditions: Vec<Condition>,
    pub postconditions: Vec<Condition>,
    pub examples: Vec<SkillExample>,
    pub tests: Vec<SkillTest>,

    pub failure_modes: Vec<String>,
    pub repair_history: Vec<SkillRepair>,

    pub success_count: u64,
    pub failure_count: u64,
    pub avg_runtime_ms: f32,
    pub last_trace: Option<EpisodeId>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum SkillLanguage {
    Rust,
    Python,
    Shell,
    Sql,
    McpTool,
    Workflow,
}

impl SkillPayload {
    pub fn reliability(&self) -> f32 {
        let total = self.success_count + self.failure_count;
        if total == 0 { 0.5 } else { self.success_count as f32 / total as f32 }
    }
}
```

A skill is not trusted because it was written. It is trusted only after it has:

```text
preconditions
+ tests
+ successful episodes
+ known failure modes
+ versioned code
+ provenance
```

# 8. Canonical storage and index fabric

The canonical rule:

```text
Only canonical stores are truth.
All indexes are projections.
Every projection must be rebuildable.
```

## 8.1 Storage stack

| Concern                     | Recommended Rust-first choice                                | Reason                                                    |
| --------------------------- | ------------------------------------------------------------ | --------------------------------------------------------- |
| Append-only mutation ledger | Custom segmented JSONL/bincode log + BLAKE3 hash chain       | Auditable, replayable, deterministic.                     |
| Canonical object store      | `redb` or `fjall`; optional SQLite adapter                   | Rust-first durable KV; SQLite adapter for inspectability. |
| Full-text / BM25            | Tantivy                                                      | Fast exact retrieval for symbols, names, code, equations. |
| Vector index                | `hnsw_rs`, `usearch` adapter, or Qdrant optional             | Fast semantic retrieval; not source of truth.             |
| Entity alias index          | FST + roaring bitmaps                                        | Fast exact/alias lookups at millions scale.               |
| Temporal graph              | In-memory adjacency + persisted edges; optional Kùzu adapter | Fast valid-at and neighborhood traversal.                 |
| PPR / HippoRAG cache        | Sparse adjacency matrix + topic-local cache                  | Multi-hop recall without iterative LLM calls.             |
| Git memory cards            | Markdown/YAML cards committed by memory daemon               | Human review, diffs, branchable memory.                   |
| Blob/resource store         | Content-addressed files by BLAKE3                            | Rehydrate exact PDFs, code, datasets, logs.               |
| Telemetry                   | Parquet/Arrow optional feature                               | Query traces, evals, failure analysis.                    |

## 8.2 Hot/warm/cold tiering

| Tier | Contents                                                            | Indexed by                          | Purpose                  |
| ---- | ------------------------------------------------------------------- | ----------------------------------- | ------------------------ |
| T0   | Active working memory, last task, last N observations               | In-memory                           | Instant continuity.      |
| T1   | Hot trusted facts, recent episodes, active topics, top skills       | BM25 + HNSW + graph                 | Primary recall.          |
| T2   | Warm concept kernels, older facts, stable skills, resource metadata | BM25 + compressed vectors + graph   | Normal long-term memory. |
| T3   | Cold archived observations, superseded facts, old evals             | BM25-lite + exact ID + source links | Historical recall.       |
| T4   | Raw source blobs                                                    | Content hash only                   | Lossless rehydration.    |

This is how the system scales to millions of memories without stuffing all memory into hot vector space.

# 11. Write path

The write path is split into two phases so the system supports **real-time learning** without blocking on expensive extraction.

## 11.1 Phase A: realtime observe

Target: p50 ≤ 5 ms local commit.

```text
event arrives
→ assign episode id
→ hash payload
→ append ledger entry
→ create compact observation if cheap
→ update recent in-memory cache
→ return receipt
```

The system gives a **read-your-writes guarantee** for recent episodes and observations.

## 11.2 Phase B: async learn

Target: seconds, not milliseconds.

```text
episode
→ segment
→ extract claims/entities/equations/skills/resources
→ normalize entities/predicates/units
→ ground with source spans
→ detect duplicates
→ detect contradictions
→ update graph
→ propose beliefs
→ route to lanes
→ commit mutations
→ update projections
```

## 11.3 Write pipeline

```rust
pub async fn write_pipeline(event: EventEnvelope, engine: &Engine) -> Result<MutationReceipt> {
    let episode = engine.ledger.append_episode(event).await?;

    let observations = engine.observer.segment_and_summarize(&episode).await?;
    engine.store.insert_observations(&observations).await?;

    let candidates = engine.extractor.extract(&episode, &observations).await?;
    let normalized = engine.normalizer.normalize(candidates).await?;
    let grounded = engine.grounder.attach_provenance(normalized).await?;
    let linked = engine.linker.link_existing(grounded).await?;
    let scored = engine.scorer.score(linked).await?;

    let decisions = engine.policy.decide(scored).await?;

    let receipt = engine.committer.commit(decisions).await?;
    engine.indexer.apply_receipt(&receipt).await?;

    Ok(receipt)
}
```

# 12. Temporal contradiction logic

The most important semantic invariant:

```text
A new contradictory fact never overwrites the old fact.
It closes the old fact’s validity window and creates a new edge.
```

```rust
pub async fn supersede_fact(
    graph: &dyn TemporalGraph,
    old_edge: EdgeId,
    new_fact: FactEdgePayload,
    evidence: EvidenceLink,
    now: DateTime<Utc>,
) -> anyhow::Result<EdgeId> {
    let mut old = graph.get_edge(old_edge).await?;

    if old.validity.valid_to.is_none() {
        old.validity.valid_to = Some(now);
    }

    old.validity.recorded_to = Some(now);
    old.superseded_by.push(new_fact.edge_id());

    graph.update_edge(old).await?;

    let mut new_edge = new_fact;
    new_edge.validity.recorded_from = now;
    new_edge.supersedes.push(old_edge);
    new_edge.support.push(evidence);

    let new_edge_id = graph.insert_edge(new_edge).await?;

    graph.insert_link(MemoryLink {
        from: old_edge.into(),
        to: new_edge_id.into(),
        relation: LinkRelation::SupersededBy,
        strength: 1.0,
        created_at: now,
        evidence: vec![],
    }).await?;

    Ok(new_edge_id)
}
```

# 13. Read path: retrieval as compilation

A query does not “search memory.” A query asks the compiler to build a **ContextPack**.

## 13.1 Query planner

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RecallQuery {

    pub intent: RecallIntent,

    pub include_skills: bool,
    pub assume_mastery: bool,
    pub deterministic_seed: Option<u64>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum RecallIntent {
    AnswerFact,
    ExplainConcept,
    MathDerivation,
    LiteratureReview,
    CodeTask,
    Debugging,
    Planning,
    Compare,
    HistoricalTimeline,
    SkillLookup,
    BeliefAudit,
    ContradictionCheck,
    OpenQuestionSearch,
    Mixed,
}
```

## 13.2 Fanout retrieval channels

Every recall runs these channels in parallel:

```text
1. Core memory lookup
2. Recent episodic cache
3. Tantivy BM25 exact search
4. Vector HNSW semantic search
5. Entity alias / exact identifier lookup
6. Temporal valid-at graph query
7. Concept graph neighborhood
8. HippoRAG-style Personalized PageRank
9. Skill precondition/signature lookup
10. Eval memory lookup for prior failures
11. Counterexample lookup
12. Resource/citation lookup
```

## 13.3 Ranking formula

```text
score =
  w_exact      * exact_match
+ w_bm25       * bm25_norm
+ w_vector     * vector_similarity
+ w_entity     * entity_overlap
+ w_temporal   * temporal_validity
+ w_graph      * graph_proximity
+ w_ppr        * ppr_activation
+ w_source     * source_quality
+ w_conf       * confidence
+ w_utility    * utility
+ w_recent     * recency
+ w_skill      * skill_reliability
+ w_topic      * topic_strength
- w_stale      * stale_penalty
- w_contra     * unresolved_contradiction_penalty
```

Use different profiles per intent.

| Intent            | Highest weights                                                            |
| ----------------- | -------------------------------------------------------------------------- |
| Math derivation   | exact symbols, equation schema, prerequisites, units, theorem dependencies |
| Scientific claim  | citations, source quality, contradiction, temporal validity                |
| Coding task       | skill reliability, exact paths/symbols, recent failures, resource metadata |
| Debugging         | recent episodes, exact error text, failed attempts, environment facts      |
| Literature review | resource graph, claim clusters, citation graph, concept kernels            |
| Planning          | core constraints, open questions, skills, eval memory, topic strength      |

## 13.4 ContextPack schema

```rust

pub struct ContextPack {

    pub generated_at: DateTime<Utc>,

    pub items: Vec<ContextItem>,
    pub citations: Vec<Citation>,

    pub warnings: Vec<String>,

```

ContextPack contract:

```text
1. It must fit the token budget.
2. Every factual item must cite source/provenance.
3. Known contradictions must be shown, not hidden.
4. Known counterexamples must be included when directly relevant.
5. Skill bodies are not pasted by default; only callable signatures are included.
6. Order is deterministic for the same memory state and seed.
7. Omitted important evidence is listed.
```

# 14. Knowledge compression

This is the heart of the system.

MNEMOS-Ω keeps a lossless source trail, but aggressively compresses the hot memory surface.

## 14.1 Compression types

| Compression type         | Input                        | Output                      | Lossy?                           | Source preserved? |
| ------------------------ | ---------------------------- | --------------------------- | -------------------------------- | ----------------- |
| Observation compression  | raw messages/tool logs       | compact observations        | yes                              | yes               |
| Atomic extraction        | observations/docs            | claims, equations, entities | partially                        | yes               |
| Deduplication            | similar memories             | canonical memory + aliases  | yes                              | yes               |
| Temporal supersession    | conflicting facts            | old closed + new fact       | no                               | yes               |
| Concept kernel synthesis | claims/equations/resources   | concept card                | yes                              | yes               |
| Belief synthesis         | claims/counterclaims         | belief state                | yes                              | yes               |
| Skill distillation       | repeated successful episodes | executable skill            | yes                              | yes               |
| Archive compaction       | low-utility memories         | cold tier                   | no for source, yes for hot index |                   |
| Community summarization  | graph communities            | topic summaries             | yes                              | yes               |

The compression principle:

```text
Summaries are never the only truth.
Every compressed object points back to source evidence.
```

## 14.2 Consolidation schedule

| Trigger                   | Job                                                           |
| ------------------------- | ------------------------------------------------------------- |
| Every event               | observe, segment, append, cheap observation                   |
| Every few minutes idle    | extract, dedupe, link, update topic strength                  |
| When surprise > threshold | reflect immediately                                           |
| After task completion     | feedback, utility update, eval memory                         |
| Nightly                   | concept kernel evolution, belief revision, skill distillation |
| Weekly                    | topic community recompute, cold archive, index rebuild check  |
| On contradiction          | temporal invalidation + counterexample creation               |
| On failed retrieval       | eval memory + rerank penalty + query expansion lesson         |

## 14.3 Topic strength

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TopicState {
    pub id: TopicId,
    pub label: String,

    pub strength: f32,
    pub mastery: f32,
    pub half_life_hours: f32,

    pub recency: f32,
    pub recurrence: f32,
    pub utility: f32,
    pub novelty: f32,
    pub source_quality: f32,
    pub contradiction_pressure: f32,
    pub retrieval_success: f32,
    pub graph_neighbor_boost: f32,

    pub last_strengthened_at: Option<DateTime<Utc>>,
    pub next_review_at: Option<DateTime<Utc>>,
}
```

Update:

```text
decayed =
  old_strength * exp(-hours_since_update / half_life_hours)

reinforcement =
  recency*w1
+ recurrence*w2
+ utility*w3
+ novelty*w4
+ source_quality*w5
+ retrieval_success*w6
+ graph_neighbor_boost*w7
- contradiction_pressure*w8

new_strength = clamp(decayed + reinforcement, 0, 1)
```

Topic-level memory is critical because humans do not only remember individual facts; they strengthen clusters of understanding.

# 15. Agent tool calls for organization and compression

The agent should not have arbitrary write access to memory. It should have structured calls.

## 15.1 Core calls

```rust
memory.observe(event)
```

Use after every meaningful message, tool run, file ingest, paper read, test result, benchmark, correction, or decision.

```rust
memory.recall(query)
```

Compile a bounded context pack.

```rust
memory.feedback(pack_id, outcome)
```

Tell memory whether the recalled context helped.

```rust
memory.explain(target)
```

Explain why something was stored, retrieved, ranked, omitted, weakened, or strengthened.

## 15.2 Organization calls

```rust
memory.reflect(scope)
```

Generate lessons from a scoped set of episodes, failures, contradictions, or discoveries.

```rust
memory.consolidate(scope)
```

Merge duplicates, build concept kernels, revise beliefs, update topic strength.

```rust
memory.compress(scope)
```

Move low-utility raw observations out of hot indexes; preserve source trail.

```rust
memory.resolve_contradictions(topic_or_entity)
```

Find live contradictions and produce a belief-revision plan.

```rust
memory.rehearse_next(n)
```

Pull the next high-value weakening memories/topics for spaced review.

```rust
memory.promote_skill(candidate_skill)
```

Turn repeated successful actions into a tested skill.

```rust
memory.archive(scope)
```

Move low-value or cold memories out of hot recall.

```rust
memory.rehydrate(memory_or_resource_id)
```

Reload exact source context from cold archive or content-addressed blob.

## 15.3 Safety/audit calls

```rust
memory.verify_hash_chain()
memory.verify_index_consistency()
memory.audit_belief(proposition)
memory.audit_source(source_uri)
memory.redact(scope, policy)
memory.export_snapshot()
memory.rebuild_indexes()
```

The most useful agent behavior pattern is:

```text
before action:
  recall

after action:
  observe result
  feedback pack outcome

during idle:
  consolidate
  compress
  rehearse
  reflect

after failure:
  create counterexample
  update eval memory
  repair skill
  revise belief
```

# 16. Procedural skill compounding

The system should treat procedures as executable memory.

## Skill lifecycle

```mermaid
stateDiagram-v2
    [*] --> Candidate
    Candidate --> Draft: extracted from episode
    Draft --> Tested: test recipe added
```

A skill can become trusted only if:


# 17. Formal science/math memory

For science, math, and research, the memory system must be stricter than ordinary personal memory.

## 17.1 Equations

Every equation should store:

```text
LaTeX
normalized symbolic form
variable definitions
units
assumptions
regime of validity
derivation chain
source span
examples of use
known failures
related claims
related skills
```

Exact equation retrieval must use BM25/symbol search. Vector search alone is unacceptable.

## 17.2 Theorems

Every theorem should store:

```text
natural-language statement
formal statement if available
proof sketch
formal proof if available
assumptions
dependency DAG
verification status
downstream theorems
counterexamples
```

If a dependency is invalidated, downstream theorem verification should be downgraded.

## 17.3 Experiments

Every experiment memory should store:

```text
hypothesis
prediction
method
dataset/code version
result
interpretation
failed assumptions
linked claims/beliefs
```

This lets the agent answer:

```text
Why did I believe this?
Which experiment contradicted it?
Which script generated the result?
Which dataset version was used?
```

# 19. Performance targets

These are design targets, not claimed benchmark results.

| Operation                      |          10M memory target | Notes                                 |
| ------------------------------ | -------------------------: | ------------------------------------- |
| `observe(event)` async receipt |    p50 ≤ 5 ms, p95 ≤ 25 ms | Append ledger + hot observation only. |
| `recall(simple fact)`          |  p50 ≤ 35 ms, p95 ≤ 180 ms | BM25/entity/vector no deep PPR.       |
| `recall(science/multi-hop)`    | p50 ≤ 120 ms, p95 ≤ 500 ms | Includes graph/PPR expansion.         |
| `recall(debug exact error)`    |  p50 ≤ 40 ms, p95 ≤ 200 ms | Exact matching dominates.             |
| `context_pack_compile`         |                p50 ≤ 20 ms | After candidate retrieval.            |
| `nightly consolidation`        |      minutes, not hot path | Batch/idle only.                      |
| index rebuild from canonical   |              deterministic | Time depends on corpus size.          |

Scale tactics:

```text
1. Per-lane vector indexes, not one giant vector space.
2. Topic-sharded HNSW partitions.
3. Roaring bitmaps for entity/topic filters.
4. Tantivy for exact terms, symbols, code paths.
5. Temporal interval indexes for valid-at queries.
6. Cached PPR per hot topic/community.
7. Cold archive removes low-utility items from hot HNSW.
8. Concept kernels reduce long-tail retrieval load.
9. Rehydration from source only when needed.
```

# 21. Privacy and provenance

Rules:

```text
1. Every trusted memory has provenance.
2. Every mutation has a receipt.
3. Every episode is hash-chained.
4. Every source blob is content-addressed.
5. Every context pack can be explained.
6. Vault data is never stored in repo cards.
7. Sensitive data requires explicit access policy.
8. Memory cannot execute code; skills execute only in sandbox.
```

# 27. Final specification summary

The best possible memory system is:

```text
MNEMOS-Ω =
  Rust-native canonical ledger
+ typed cognitive memory lanes
+ bi-temporal semantic graph
+ source-backed claims and beliefs
+ formal equation/theorem/dataset memory
+ A-MEM-style concept kernels
+ Voyager-style tested skill library
+ Mastra/Nemori-style observation compression
+ Mem0-style low-latency hybrid retrieval
+ HippoRAG-style associative graph activation
+ Memory-R1-style learned memory manager
+ FSRS/topic strengthening
+ Git/audit/source rehydration
+ deterministic context-pack compiler
```

The most important design choice is this:

```text
Do not optimize for storing more.
Optimize for compiling the smallest, most truthful, most useful context pack.

Do not trust memory because it is remembered.
Trust memory because it is typed, sourced, temporally
::contentReference[oaicite:2]{index=2}
 valid, contradiction-aware, and useful in successful future work.
```

This is the design I would build.
