# MnemOS-Prime: Superhuman Cognitive Memory Compiler

## 0. One-sentence architecture

**MnemOS-Prime is a Rust-native, append-only, provenance-first cognitive memory compiler that stores raw experience as immutable episodes, compiles durable typed memories into Git/audit ledgers, projects them into fast full-text/vector/graph/temporal indexes, strengthens or decays them from use, and gives agents bounded cited context packs plus callable skills instead of flooding their context window.**

The design aligns cleanly with OpenQG because your repo already separates durable research knowledge from generated receipts and run state: `research/inbox/` holds reviewable candidate cards, `research/knowledge/` holds curated synthesis maps and claim ledgers, and raw provider receipts, daemon ledgers, and run state stay under `target/openqg/research/`.

---

# 1. Why this is the optimal synthesis

The memory systems you had the tips study converge on these facts:

MIRIX is valuable because it separates memory into specialized components such as core, episodic, semantic, procedural, resource, and knowledge-vault memory, with managers routing updates and search across those lanes. MIRIX’s docs also describe BM25, embedding, and string-match search across memory types. ([Mirix Documentation][1])

Graphiti/Zep is valuable because it models facts as temporally aware graph edges with lifecycle timestamps such as `created_at`, `valid_at`, `invalid_at`, and `expired_at`, so stale facts can be invalidated instead of overwritten. ([Zep Documentation][2])

A-MEM is valuable because it uses a Zettelkasten-style memory network: new memories become structured notes with contextual descriptions, keywords, tags, links, and the ability to update older notes as the network evolves. ([Hugging Face][3])

Mastra Observational Memory is valuable because it compresses raw interaction history into dense dated observations and reflections while maintaining a stable, cacheable context window; its published benchmark report claims 94.87% on LongMemEval with `gpt-5-mini`, and the architecture emphasizes no per-turn dynamic retrieval for the stable memory prefix. ([Mastra][4])

MemGPT/Letta is valuable because it frames context as a virtual-memory hierarchy: the limited LLM context window acts like RAM, while external archival memory acts like slower storage managed by tools and control flow. ([arXiv.gg][5])

HippoRAG is valuable because it uses knowledge graphs plus Personalized PageRank-style spreading activation for multi-hop recall; the paper reports lower cost and latency than iterative retrieval while improving multi-hop QA. ([Hugging Face][6])

Voyager is valuable because it turns experience into executable, reusable, compositional skills, using execution feedback and self-verification to improve code-like procedures over time. ([Hugging Face][7])

GraphRAG and RAPTOR are valuable because they solve corpus-scale sensemaking: GraphRAG builds entity graphs and community summaries for global questions, while RAPTOR recursively clusters and summarizes chunks into a retrieval tree. ([Microsoft][8])

LongMemEval is a good benchmark target because it evaluates information extraction, multi-session reasoning, temporal reasoning, knowledge updates, and abstention in long-term interactive memory. ([Hugging Face][9])

The conclusion: **the memory must be typed, temporal, graph-linked, compressed, executable, benchmarked, and provenance-backed.**

---

# 2. The core design rule

Do **not** make the vector database the memory.

Make this the invariant:

```text
Canonical truth = append-only ledger + typed cards + provenance + temporal validity.
Indexes        = rebuildable projections.
LLM context    = compiled output, never the memory itself.
Agent writes   = proposed deltas, not silent truth mutations.
Skills         = executable memories with tests, not unverified snippets.
Summaries      = compressed views, never replacements for source-backed claims.
```

This is the difference between a chatbot with recall and an agent that compounds knowledge.

# 5. Memory lanes

MnemOS-Prime has **10 primary lanes**, not merely six. Six is a good baseline, but science/math agents need more precision.

| Lane         | Purpose                                                      | Canonical examples                                               | Retention                     |
| ------------ | ------------------------------------------------------------ | ---------------------------------------------------------------- | ----------------------------- |
| `Core`       | Always-loaded mission, identity, constraints, current stance | OpenQG mission, boundaries, active phase                         | Durable, tiny                 |
| `Episodic`   | What happened, when, with whom, from what source             | Search runs, tool outputs, failures, paper-read events           | Append-only                   |
| `Resource`   | Pointers to external reality                                 | arXiv paper, DOI, Git commit, dataset URL, file hash             | Durable pointer, raw external |
| `Semantic`   | Atomic source-backed claims                                  | “Paper X claims Y under assumptions A”                           | Temporal                      |
| `Belief`     | Current evidence-weighted stance over claims                 | “We currently accept P at 0.72 because…”                         | Revisable                     |
| `Concept`    | Self-organizing abstractions and topic kernels               | “Asymptotic safety concept kernel”                               | Consolidated                  |
| `Formal`     | Equations, symbols, theorems, derivations, units             | LaTeX, normalized forms, Lean hooks                              | Durable                       |
| `Procedural` | Executable, tested skills                                    | `search_arxiv`, `parse_pdf`, `download_dataset`, `run_benchmark` | Versioned                     |
| `Eval`       | What helped or hurt outcomes                                 | Bad retrievals, failed skills, hallucinated citations            | Durable learning signal       |
| `Meta`       | Memory-management policies and lessons                       | “For equation queries, BM25 beats embedding”                     | Durable, strengthened         |

This is superhuman because it does what humans do poorly: it remembers the source of every belief, tracks time, preserves counterexamples, learns procedural code, and can search millions of memories in parallel.

# 6. Canonical storage model

The canonical model has three layers:

```text
Layer 1: Immutable event truth
  - append-only episode ledger
  - mutation receipts
  - hash chain
  - signed segment manifests

Layer 2: Durable typed memory
  - MemoryObject records
  - ClaimCard / EquationCard / ConceptCard / SkillCard
  - temporal graph edges
  - curated Git cards for reviewable memory

Layer 3: Rebuildable projections
  - BM25 index
  - vector index
  - graph adjacency index
  - temporal interval index
  - concept kernel tree
  - context-pack cache
```

For a Rust-first implementation, I would use:

| Need                  | Default choice                                                                             | Why                                                                                                                                                                                               |
| --------------------- | ------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Embedded canonical KV | `redb` for pure-Rust ACID; `fjall` for high-write LSM profile                              | `redb` is pure Rust, ACID, MVCC, crash-safe; `fjall` is a Rust LSM engine with cross-partition atomic semantics and compression. ([Docs.rs][10])                                                  |
| Exact search          | Tantivy                                                                                    | Rust-native Lucene-like full-text search with BM25, phrase queries, facets, incremental indexing. ([GitHub][11])                                                                                  |
| Vector search         | Local HNSW for embedded mode; Qdrant adapter for larger deployments                        | Qdrant is Rust-powered and supports dense/sparse hybrid search, metadata filters, multivectors, quantization, and RRF-style hybrid queries. ([Qdrant][12])                                        |
| Graph                 | Rust CSR adjacency + `petgraph` for local; optional Kuzu adapter for heavy graph analytics | Kuzu is an embedded graph DB with Cypher, columnar storage, CSR adjacency, parallelism, and ACID transactions, but it is optional because the default should stay Rust-owned. ([KuzuDB Docs][13]) |
| Analytics             | Arrow + DataFusion                                                                         | DataFusion is a Rust query engine using Arrow, with SQL/DataFrame APIs and vectorized, multithreaded execution. ([Apache DataFusion][14])                                                         |
| Large receipts        | Content-addressed file/object store                                                        | Avoids putting raw data/PDFs/blobs in Git.                                                                                                                                                        |
| Human review          | Git-backed Markdown/YAML/Turtle cards                                                      | Diffs, blame, PR review, deterministic curated memory.                                                                                                                                            |
| Spaced reinforcement  | `fsrs-rs`                                                                                  | FSRS has Rust implementations and training/scheduling support. ([GitHub][15])                                                                                                                     |

SQLite WAL remains a valid alternate profile. SQLite’s own docs note WAL improves concurrency because readers do not block writers and writers do not block readers in normal cases, but it is not pure Rust and has host-filesystem caveats. ([SQLite][16])

# 10. Write path

The write path must be append-only, idempotent, and auditable.

```text
observe(event)
  -> segment
  -> append episode
  -> compress observation
  -> extract candidates
  -> normalize entities
  -> validate provenance
  -> detect duplicates
  -> detect contradictions
  -> route lanes
  -> policy decision
  -> commit canonical transaction
  -> append mutation receipt
  -> enqueue projection updates
  -> schedule reflection/consolidation
```

## Write path pseudocode

```rust
pub async fn observe(&self, event: EventEnvelope) -> Result<MutationReceipt> {
    self.privacy_gate.check(&event)?;

    let segments = self.segmenter.segment(event).await?;

    let mut receipt = MutationReceipt::new();

    for segment in segments {
        let episode = self.ledger.append_episode(segment.clone()).await?;
        receipt.episodes.push(episode.id);

        let observations = self.observer.compress(&segment, episode.id).await?;
        let candidates = self.extractor.extract(&segment, &observations).await?;

        for candidate in candidates {
            let grounded = self.grounder.attach_provenance(candidate, &episode)?;
            let normalized = self.normalizer.normalize(grounded).await?;
            let linked = self.linker.link(normalized).await?;

            let validation = self.validator.validate(&linked).await?;
            if !validation.is_promotable() {
                receipt.quarantined.push(linked.preview_id());
                self.store.quarantine(linked, validation).await?;
                continue;
            }

            let decision = self.policy.decide(&linked).await?;

            match decision {
                MemoryDecision::Add => {
                    let id = self.store.insert(linked).await?;
                    receipt.created.push(id);
                }
                MemoryDecision::Supersede { old_id } => {
                    let id = self.store.supersede(old_id, linked).await?;
                    receipt.superseded.push(old_id);
                    receipt.created.push(id);
                }
                MemoryDecision::Link { target_id, relation } => {
                    self.store.link(target_id, linked, relation).await?;
                    receipt.updated.push(target_id);
                }
                MemoryDecision::Noop => {
                    receipt.noops += 1;
                }
                MemoryDecision::Review => {
                    self.store.review_queue(linked).await?;
                }
            }
        }
    }

    self.ledger.append_mutation_receipt(&receipt).await?;
    self.projections.enqueue(receipt.changed_objects()).await?;

    Ok(receipt)
}
```

## Critical write invariants

```text
1. A memory cannot become Trusted without provenance.
2. A fact cannot be overwritten to mean a different fact.
3. A belief cannot exist without support/contradiction links.
4. A skill cannot become Trusted without a test or successful episode.
5. A summary cannot replace its source.
6. A contradiction closes validity or creates contested belief; it does not delete history.
7. Raw external data stays outside Git unless explicitly promoted as a tiny fixture.
8. Every mutation produces a receipt.
```

# 11. Read path

The read path is the heart of the system.

The agent does not ask for “memory.” It asks for a **compiled context pack**.

```text
recall(query)
  -> classify intent
  -> determine budget
  -> load pinned core
  -> parallel retrieval fanout
  -> temporal filtering
  -> rank fusion
  -> graph activation
  -> evidence filtering
  -> compression-level selection
  -> context pack assembly
  -> trace + feedback hook
```

## Retrieval fanout

```text
1. Core memory
   mission, current constraints, active stance

2. BM25 / exact search
   paper IDs, DOI, arXiv IDs, code symbols, equations, named datasets

3. Dense vector search
   concepts, summaries, semantic similarity

4. Sparse vector search
   keyword-heavy hybrid retrieval

5. Entity resolver
   aliases, authors, paper titles, variable names, theory names

6. Temporal graph
   facts valid at requested time

7. Concept graph
   A-MEM links, prerequisite paths, topic kernels

8. HippoRAG graph activation
   bounded Personalized PageRank from seed entities

9. Procedural memory
   relevant tested skills

10. Eval/counterexample memory
   known failures, retrieval regressions, prior mistakes
```

## Retrieval scoring

```text
score =
  0.18 * exact_match
+ 0.14 * bm25_norm
+ 0.13 * dense_sim
+ 0.10 * sparse_sim
+ 0.12 * graph_activation
+ 0.08 * temporal_validity
+ 0.08 * source_quality
+ 0.06 * confidence
+ 0.05 * topic_strength
+ 0.04 * retrieval_success_ema
+ 0.03 * recency_if_relevant
- 0.08 * contradiction_pressure
- 0.04 * staleness_penalty
```

Weights are not global constants. They change by intent.

| Query intent          | Weight shift                                           |
| --------------------- | ------------------------------------------------------ |
| Equation/math         | exact search, symbols, units, derivation DAGs          |
| Literature review     | source quality, citation clusters, concept kernels     |
| Coding/debugging      | skills, eval memory, exact error text, recent episodes |
| Current project state | core, latest episodic, temporal validity               |
| Theory comparison     | concept graph, belief cards, contradictions            |
| Data task             | resource cards, dataset registry, download skills      |
| Planning              | open questions, frontier state, procedural skills      |
| Self-correction       | eval memory, counterexamples, prior failures           |

# 12. Memory compaction and knowledge compression

This is the part that makes it scale.

## Compression ladder

```text
Raw source
  -> Episode
  -> Observation
  -> Atomic memory objects
  -> Concept kernel
  -> Topic map
  -> Community summary
  -> Memory summary
  -> Context pack
```

Each level is reversible downward through provenance.

| Level             | Compression method                     | Used for         |
| ----------------- | -------------------------------------- | ---------------- |
| Raw source        | content hash + object cache            | exact audit      |
| Episode           | event segmentation                     | what happened    |
| Observation       | Mastra-style dense dated notes         | continuity       |
| Atomic cards      | claim/equation/skill/source extraction | grounded memory  |
| Concept kernel    | A-MEM + curated summary                | topic recall     |
| Topic map         | graph clustering                       | navigation       |
| Community summary | GraphRAG-style clusters                | global questions |
| Summary tree      | RAPTOR-style recursive summaries       | long documents   |
| Context pack      | budgeted compiler                      | agent reasoning  |

## Compression rules

```text
1. Never compress away provenance.
2. Never merge contradictory claims into one smooth summary.
3. Always keep counterexamples.
4. Keep source spans for every promoted claim.
5. Use stronger compression for frequent low-risk continuity, weaker compression for high-stakes science/math.
6. Store summaries as views, not truth.
7. Recompile summaries when source clusters change materially.
```

# 13. Self-organization: Reflector, Consolidator, Strengthener

## Observer daemon

The Observer watches raw events and emits dense observations.

```text
Input:
  raw tool log, chat turn, paper text, code output, dataset preview

Output:
  dated observation:
    - what happened
    - what changed
    - what failed
    - what source supports it
    - what should be remembered
```

## Reflector daemon

Runs during idle or after high-surprise events.

```text
Reflector jobs:
  - link related concepts
  - update concept kernels
  - detect analogies
  - produce metacognitive lessons
  - generate next search questions
  - scan contradiction clusters
```

## Consolidator daemon

Turns many observations into durable knowledge.

```text
Consolidator jobs:
  - cluster repeated episodes
  - dedupe claims
  - compress run histories
  - promote repeated workflows into skill candidates
  - rebuild concept summaries
  - archive low-utility noise
```

## Strengthener daemon

Computes what should become easier to retrieve.

```text
Strengthening signals:
  - successful context pack use
  - repeated access
  - explicit user correction
  - source quality
  - task utility
  - graph centrality
  - high novelty
  - high surprise
  - mission relevance
```

## Hebbian co-activation

When memories are repeatedly retrieved together and lead to success, link them.

```text
if memory_a and memory_b appear in successful packs repeatedly:
    coactivation[a,b] += learning_rate * outcome_score

if coactivation[a,b] > threshold:
    create_or_strengthen_link(a, b, relation = "OftenUsefulTogether")
```

This is how the system learns cross-domain association.

# 14. Procedural memory

Procedural memory is where the agent becomes better at doing work, not just answering.

## Skill lifecycle

```mermaid
stateDiagram-v2
    [*] --> Draft
    Draft --> Tested: has runnable code + test
    Tested --> Trusted: repeated success
    Trusted --> Degraded: recent failure or stale precondition
    Degraded --> Tested: patched and re-tested
    Trusted --> Retired: superseded by better skill
    Draft --> Rejected: unsafe or unrepeatable
```

## Example skill

```yaml
id: skill:literature:search_arxiv_openalex_v4
name: Search arXiv/OpenAlex frontier
language: Rust
entrypoint: openqg_search_frontier
description: >
  Given a topic frontier, query arXiv/OpenAlex/Semantic Scholar,
  dedupe papers, update ResourceCards, and emit next-query candidates.
preconditions:
  - network_allowed
  - topic_frontier_exists
postconditions:
  - resource_cards_created
  - frontier_delta_written
tests:
  - fixture: "known asymptotic safety paper should be found"
success_count: 42
failure_count: 3
reliability_score: 0.933
linked_concepts:
  - concept:literature_search
  - concept:openqg_research_frontier
failure_modes:
  - API rate limit
  - query too broad
  - duplicate DOI normalization failure
```

## Skill promotion rule

A skill becomes `Trusted` only if:

```text
1. It has preconditions.
2. It has postconditions.
3. It has a test or reproducible verification.
4. It links to at least one successful episode.
5. It records known failure modes.
6. It can be rolled back or superseded.
```

This is where Voyager-style compounding becomes safe for science and code.

| `memory.promote`            | Move proposed memory to trusted/curated state.                    |
| `memory.supersede`          | Replace stale memory without deleting history.                    |
| `memory.contradiction_scan` | Find live conflicts.                                              |
| `memory.reflect`            | Generate concept links, lessons, and questions.                   |
| `memory.consolidate`        | Compress and organize memories over time.                         |
| `memory.rehearse`           | Strengthen important weakening topics.                            |
| `memory.frontier_next`      | Generate next search queries from gaps.                           |
| `memory.skill_run`          | Execute trusted procedural memory.                                |
| `memory.verify_equation`    | Run units/symbolic/formal checks.                                 |
| `memory.eval_run`           | Benchmark memory accuracy and regressions.                        |

This gives the agent agency without letting it corrupt memory.

# 16. Realtime learning loop

## During a task

```text
1. Agent asks memory.pack(task)
2. Memory returns compact cited context + callable skills
3. Agent acts
4. Agent reports memory.feedback(pack_id, outcome)
5. Successful memories strengthen
6. Harmful or irrelevant memories weaken
7. Failures create EvalMemory
8. Repeated success promotes skill or concept
```

## After a task

```text
1. Observer compresses the run
2. Extractor creates candidate lessons, claims, skills
3. Policy routes candidate memories
4. Reflector updates concept links
5. Consolidator updates topic kernels
6. Strengthener updates FSRS/utility/coactivation
7. Search frontier generator emits next questions
```

## During idle cycles

```text
1. Rebuild stale indexes
2. Run contradiction scans
3. Recompute topic strength
4. Merge duplicate entities
5. Rehearse mission-critical weakening concepts
6. Patch degraded skills
7. Generate open-question frontiers
8. Produce daily/weekly memory deltas
```

This is how it compounds.

# 17. Search frontier memory

For OpenQG-like research, the system must remember not just what it found, but **how it searched**.

## Search frontier card

```yaml
id: frontier:physics:qg:asymptotic-safety
topic: Asymptotic safety quantum gravity
status: active
coverage:
  canonical_reviews: strong
  primary_papers: medium
  mathematical_formalism: medium
  public_datasets: weak
  codebases: weak
  criticisms: medium
  benchmark_relevance: weak
queries_tried:
  - query: "asymptotic safety quantum gravity review"
    source: arxiv
    found: 12
    accepted: 4
    rejected: 8
  - query: "functional renormalization group quantum gravity observables"
    source: openalex
    found: 31
    accepted: 9
gaps:
  - Need better map from theory predictions to low-energy observables.
  - Need reproducible public code survey.
next_queries:
  - "asymptotic safety observable signatures low energy constraints"
  - "FRG quantum gravity reproducible code"
stop_condition:
  marginal_new_canonical_sources_below: 0.03
  contradiction_scan_complete: false
```

## Exhaustion is measurable

A search frontier is “exhausted enough” when:

```text
1. Canonical reviews are mapped.
2. Primary citation clusters are mapped.
3. Major contradictions are represented.
4. Dataset/code availability is classified.
5. Benchmark relevance has a belief card.
6. New queries produce low marginal novelty.
7. Remaining gaps are explicit.
```

This prevents agents from looping forever or prematurely stopping.

# 18. Scientific and mathematical specialization

Science memory must separate:

```text
Observation ≠ Claim ≠ Belief ≠ Equation ≠ Theory ≠ Dataset ≠ Code ≠ Benchmark
```

## Required science objects

| Object               | Stores                                              | Must not store             |
| -------------------- | --------------------------------------------------- | -------------------------- |
| `ResourceCard`       | paper/dataset/repo metadata, hash, source quality   | extracted claims           |
| `ClaimCard`          | atomic assertion, evidence, assumptions             | current belief             |
| `BeliefCard`         | current stance over competing claims                | raw source assertion       |
| `EquationCard`       | LaTeX, variables, units, assumptions, derivation    | hidden assumptions         |
| `TheoremCard`        | statement, dependencies, proof sketch/formal hook   | unsupported interpretation |
| `DatasetCard`        | source, checksum, license, preview, download script | raw data in Git            |
| `CodebaseCard`       | repo, commit, language, tests, relevance            | unpinned branch claim      |
| `BenchmarkCard`      | observable, baseline, metric, dataset, assumptions  | undocumented score         |
| `QuestionCard`       | open gap, blocker, next query                       | resolved fact              |
| `CounterexampleCard` | known failure, contradiction, invalid regime        | generic criticism          |

## Equation handling

Equation search must use exact search. Vector search alone is unacceptable for symbols.

```text
Equation memory must preserve:
  - original source span
  - LaTeX
  - normalized symbolic form
  - variable definitions
  - units and dimensions
  - assumptions
  - limiting cases
  - derivation path
  - linked claims
  - examples
  - contradictions / alternate forms
```

## Belief update rule

```text
Observation:
  Source S reports result R under method M.

Claim:
  R supports proposition P under assumptions A.

Belief:
  The agent currently accepts/rejects/contests P with confidence C
  because claims C1..Cn support it and K1..Km contradict it.
```

The system must always be able to answer:

```text
What did we observe?
Who asserted it?
What claim did we extract?
What evidence supports it?
What evidence contradicts it?
What do we currently believe?
When did that belief change?
Why did it change?
```

# 20. Contradiction and supersession logic

## Temporal fact update

```text
Old edge:
  claim: "Dataset D is available at endpoint E"
  valid_from: 2025-01-01
  valid_to: null

New evidence:
  "Endpoint E was retired on 2026-04-30"

Mutation:
  old.valid_to = 2026-04-30
  new edge: "Dataset D is available at endpoint E2"
  new.valid_from = 2026-05-01
  relation: Supersedes(old)
```

## Scientific contradiction

```text
Claim A:
  Paper X supports proposition P under assumptions A1.

Claim B:
  Paper Y contradicts P under assumptions A2.

Mutation:
  Keep both claims.
  Create Contradicts edge.
  Update BeliefCard:
    stance = Contested
    confidence = recomputed
    unresolved_contradictions += [A, B]
```

## No destructive overwrite

```text
Bad:
  edit old claim text in place

Good:
  create new claim
  supersede old claim if appropriate
  preserve old claim and why it existed
```

# 21. Context-window protection

The system should use progressive disclosure, similar to the pattern in OpenAI’s sandbox memory docs: small summary first, then search/open detailed rollout summaries only if relevant. ([OpenAI GitHub][17])

## Disclosure tiers

```text
Tier 0: Core memory summary
  always loaded; < 2k tokens

Tier 1: Task context pack
  compiled; 4k–32k tokens

Tier 2: Detailed cards
  opened only when needed

Tier 3: Source spans
  opened for verification

Tier 4: Raw artifacts
  opened only for audit or reprocessing
```

## Context budget policy

```yaml
default_budget_tokens: 12000
max_budget_tokens: 64000

allocations:
  core_constraints: 1000
  task_summary: 1000
  concept_kernels: 3000
  claims_and_evidence: 4000
  equations_or_skills: 3000
  contradictions: 1500
  open_questions: 1000
  omissions: 500
```

For small coding tasks, use 4k–8k.
For literature synthesis, use 16k–64k.
For formal math, include fewer claims and more derivation context.

# 22. Performance design

## Scale targets

```text
Single-node baseline:
  10M memory objects
  100M graph edges
  1B source spans possible in cold object store
  1M skills/evals unlikely but supported structurally

Hot recall:
  p50 <= 75 ms
  p95 <= 350 ms
  p99 <= 1200 ms

Deep recall:
  p50 <= 300 ms
  p95 <= 1500 ms
  p99 <= 5000 ms

Write:
  sync ledger append <= 10 ms p50
  async extraction/index update outside hot path
  index lag visible in receipts

Context pack:
  4k pack <= 150 ms after retrieval
  32k pack <= 600 ms after retrieval
```

## Speed techniques

```text
1. UUIDv7/time-sortable IDs.
2. Lane-specific indexes.
3. Compact summaries indexed, raw content external.
4. BM25 for exact strings and equations.
5. Dense vectors for concepts, not every raw chunk.
6. Sparse vectors for hybrid search.
7. Bounded graph activation, not unbounded traversal.
8. Precomputed concept kernels.
9. Query-plan caching.
10. Context-pack caching.
11. Entity alias FST.
12. CSR graph adjacency.
13. mmap where useful.
14. Async projection updates.
15. No LLM call on hot recall path.
16. Optional Qdrant for large vector workloads.
17. Optional local HNSW for pure embedded mode.
18. Token-budget-first pack assembly.
```

## Sharding

```text
Shard key:
  domain_hash + lane + time_bucket

Examples:
  semantic/physics/qg/2026
  episodic/openqg/runs/2026-05
  formal/equations/physics
  skills/data-download
```

Each shard has:

```text
ledger segment
canonical store partition
BM25 segment
vector namespace
graph edge partition
projection manifest
```

Federated recall runs query plans against relevant shards only.

# 23. Evaluation suite

## Required eval categories

```text
1. Exact recall
   Can it retrieve known facts, IDs, equations, symbols?

2. Temporal recall
   Can it answer what was true at time T?

3. Contradiction recall
   Can it surface both support and conflict?

4. Multi-hop recall
   Can it traverse source -> claim -> equation -> dataset -> code?

5. Context economy
   Does it answer with fewer tokens than raw retrieval?

6. Skill recall
```

# 28. The final operating maxim

```text
Do not teach the agent to remember more text.
Teach it to compile experience into source-backed, typed, temporal,
linked, compressed, executable memory — then retrieve only the minimal
evidence-bearing context needed for the current task.
```

That is the optimal path for a research agent that can learn physics, math, code, data, literature, and its own failures over months or years without drowning its context window.
