I unpacked and studied the archive. It contains `CLAUDE_MEMSPEC.md`, `CODEX_MEMSPEC.md`, and `tip1.txt` through `tip13.txt`; there is no standalone Gemini file in the tarball, so I treated the pasted **ANTIGRAVITY CMC** blueprint as the Google/Gemini-style spec.

# MNEMOS-Ω: Rust Cognitive Memory Compiler

**Verdict:** the best architecture is not a vector database, not a graph database, not a chat-history summarizer, and not a single “memory product.” The best design is a **Rust-native cognitive memory compiler**: a canonical, source-grounded memory substrate that continuously compiles raw experience into typed memories, typed memories into concepts, concepts into skills and beliefs, and those into tiny, task-specific context packs.

The source material converges on the same architectural synthesis: MIRIX gives the best typed memory taxonomy, Graphiti/Zep gives temporal truth and provenance, A-MEM gives self-organizing Zettelkasten evolution, Hindsight gives epistemic separation between experience, facts, entity models, and beliefs, HippoRAG gives associative multi-hop graph recall, Mem0 gives production-speed hybrid retrieval, Mastra OM gives stable observation compression, and Voyager/Memento gives executable skill compounding. MIRIX explicitly defines six structured memory types—Core, Episodic, Semantic, Procedural, Resource, and Knowledge Vault—rather than relying on flat chunks. ([arXiv][1]) Graphiti describes context graphs with entities, relationships, facts, validity windows, episodes, and provenance, and emphasizes that changed facts are invalidated rather than deleted. ([GitHub][2]) A-MEM’s key insight is dynamic Zettelkasten-style notes with contextual descriptions, keywords, tags, links, and memory evolution when new memories arrive. ([arXiv][3]) Hindsight’s key insight is that memory should distinguish world facts, agent experiences, synthesized entity summaries, and evolving beliefs through retain/recall/reflect operations. ([arXiv][4]) HippoRAG’s key retrieval insight is to combine LLMs, knowledge graphs, and Personalized PageRank for long-term, multi-hop associative recall. ([arXiv][5]) Mem0’s key production insight is dynamic extraction/consolidation/retrieval of salient memories, with reported reductions in p95 latency and token cost versus full-context approaches. ([arXiv][6])

The design below is my best synthesis.

## 2. Core Principle

```text
Do not make embeddings the memory.
Do not make chat history the memory.
Do not let the LLM freely edit memory.

Make typed, source-controlled, temporal memory the truth.
Make graphs, vectors, BM25, summaries, concept cards, and skills rebuildable projections.
At inference time, compile the smallest correct context pack for the current task.
```

The system is a **compiler** because it transforms raw experience into optimized, typed, auditable, task-shaped memory artifacts.

## 4. Memory Lanes

The final system uses **14 typed lanes**. Six are MIRIX-like, four are epistemic/scientific, two are self-improvement lanes, and two are infrastructure lanes.

| Lane             | Purpose                                                                   |                  Hot? | Why it exists                                                    |
| ---------------- | ------------------------------------------------------------------------- | --------------------: | ---------------------------------------------------------------- |
| `Core`           | Stable mission, user-approved constraints, identity, active project state |                Always | Prevents losing the current stance and goals.                    |
| `Working`        | Short-lived active scratch state, query plans, live task hypotheses       |                Always | Model context is RAM; working memory controls what stays pinned. |
| `Sensory`        | Ring buffer of raw incoming events before extraction                      |                    No | Fast capture without forcing immediate LLM processing.           |
| `Episodic`       | Immutable record of what happened, when, and from where                   |             Sometimes | Autobiographical truth and provenance.                           |
| `Semantic`       | Atomic claims, facts, definitions, relations                              |                   Yes | Scientific and project knowledge.                                |
| `Belief`         | Current agent stance over evidence, separate from facts                   |                   Yes | Prevents “confidence” from pretending to be truth.               |
| `Concept`        | Zettelkasten-style concept cards and topic kernels                        |                   Yes | Self-organizing long-term understanding.                         |
| `Procedural`     | Executable skills, workflows, scripts, playbooks                          |                   Yes | Lets the agent compound actions, not just facts.                 |
| `Resource`       | Papers, repos, datasets, files, APIs, versions, checksums                 |             Sometimes | Rehydrates exact source context.                                 |
| `Formal`         | Equations, theorem DAGs, proof states, units, assumptions                 |                   Yes | Math/science requires structure beyond prose.                    |
| `Counterexample` | Failures, contradictions, falsified beliefs, failed strategies            |  Always when relevant | Prevents repeated mistakes.                                      |
| `Eval`           | Retrieval/answer/skill outcome memory                                     |             Sometimes | Teaches memory what worked.                                      |
| `Metacognitive`  | Lessons about reasoning, searching, debugging, learning                   |             Sometimes | Lets the system improve its own process.                         |
| `Vault`          | Sensitive facts, credentials, private data, secrets                       | Only by access policy | Prevents leakage and keeps secrets out of normal memory.         |

## 5. Storage Model

The strongest design rule from the specs is **canonical truth versus rebuildable projections**.

### Canonical truth

```text
1. Append-only binary WAL
   Every mutation, with hash chain, CRC, timestamp, actor, source, and receipt.

2. redb/fjall-style embedded Rust KV store
   Canonical typed memory objects, entity table, temporal edge rows, topic state.

3. Git-backed curated cards
   Human-reviewable Markdown/YAML for claims, concepts, skills, resources, equations, theorems.

4. Content-addressed artifact store
   PDFs, raw logs, screenshots, repo snapshots, datasets, notebooks, tool outputs.

5. Encrypted vault
   Secrets and sensitive project/user data, never committed to Git, never indexed by default.
```

### Rebuildable projections

```text
1. Tantivy indexes
   BM25 exact search over claims, equations, symbols, paper IDs, paths, tags.

2. Vector indexes
   HNSW / Qdrant-style semantic search over retrieval descriptors, not full raw history.

3. Temporal graph index
   Entity/relation/fact graph with valid_from, valid_to, transaction_time.

4. CSR / compressed adjacency graph
   Fast PPR / HippoRAG-style multi-hop activation.

5. Roaring bitmap filters
   Fast lane, tag, project, privacy, temporal, and status filtering.

6. Skill precondition index
   Maps task signatures and environment preconditions to verified skills.

7. Concept/community cache
   Topic maps, concept kernels, prerequisite chains, contradiction clusters.

8. Context pack cache
   Deterministic cached compiled context for repeated tasks.
```

The canonical store is small, auditable, and exact. The indexes can be deleted and rebuilt.

## 9. Write Path

The write path must be fast enough for real-time learning but strict enough to avoid memory corruption.

```text
observe
→ segment
→ append episode
→ compress observations
→ extract candidates
→ normalize entities / equations / predicates
→ classify lane
→ ground with source spans
→ link to existing graph
→ detect duplicates and contradictions
→ policy decision: ADD / UPDATE / SUPERSEDE / NOOP / REVIEW
→ commit canonical mutation
→ update hot indexes
→ schedule reflection/consolidation
```

### Realtime write strategy

There are two speeds:

```text
Fast commit path:
  raw event → episode → minimal observation → WAL receipt
  target p50: <= 5 ms local

Async cognition path:
  extraction → linking → contradiction check → concept update → skill proposal
  target: seconds to minutes depending on source complexity
```

This prevents the agent from blocking on deep extraction while still making the latest event available immediately as an episode.

### Mutation safety

```rust
pub enum MemoryDecision {
    Add(MemoryObject),
    Noop { reason: String },
    Review { candidate: MemoryObject, reason: String },
    Update { target: MemoryId, patch: MemoryPatch },
    Supersede { old: MemoryId, new: MemoryObject, reason: String },
    Contradict { old: MemoryId, new: MemoryObject, evidence: EvidenceLink },
}

pub trait MemoryPolicy: Send + Sync {
    fn decide(&self, candidate: &MemoryObject, nearest: &[MemoryObject]) -> MemoryDecision;
}

pub fn apply_fact_update(
    old: &mut MemoryObject,
    new_fact: MemoryObject,
    now: DateTime<Utc>,
) -> MemoryDecision {
    if old.lifecycle == LifecycleState::Rejected {
        return MemoryDecision::Add(new_fact);
    }

    old.temporal.valid_to = Some(now);
    old.lifecycle = LifecycleState::Superseded;

    MemoryDecision::Supersede {
        old: old.id,
        new: new_fact,
        reason: "Temporal supersession: new grounded evidence changed the active fact".into(),
    }
}
```

The system never overwrites a fact to mean something else. It closes the old validity window and creates a new fact.

## 10. Recall Path

The recall path is a compiler. It converts a natural-language task into a deterministic retrieval program.

```text
query
→ classify intent
→ resolve entities / symbols / time scope
→ compile retrieval program
→ fan out to BM25, vector, entity, temporal, graph, skill, eval, concept
→ reciprocal rank fusion
→ adaptive rerank
→ temporal validity filter
→ contradiction and evidence filter
→ privacy filter
→ context pack compiler
→ retrieval trace + feedback handle
```

### Query intents

```rust

    ExplainConcept,
    LiteratureReview,
    DeriveMath,
    VerifyClaim,
    DebugCode,
    PlanTask,
    RetrieveSkill,
    CompareTheories,
    FindContradictions,
    HistoricalTimeline,
    RehearseTopic,
    Reflect,
    Mixed,
}
```

### Retrieval channels

| Channel               | Best for                                               | Notes                                    |
| --------------------- | ------------------------------------------------------ | ---------------------------------------- |
| BM25 / Tantivy        | Exact paper IDs, equations, symbols, code paths, names | Mandatory for science/math.              |
| Vector HNSW           | Fuzzy semantic similarity                              | Runs over compact retrieval descriptors. |
| Entity graph          | People, concepts, datasets, repos, theories            | Resolves query mentions first.           |
| Temporal graph        | “What was true when?”                                  | Filters by `valid_from` / `valid_to`.    |
| PPR graph activation  | Multi-hop science questions                            | HippoRAG-style spreading activation.     |
| Concept kernels       | Compressed topic recall                                | Fast and cheap.                          |
| Episodic recency      | Recent runs, failures, corrections                     | Useful for debugging and project work.   |
| Skill index           | Executable workflows                                   | Requires precondition match.             |
| Eval memory           | Prior failures and corrections                         | Prevents repeating bad answers.          |
| Counterexample memory | Known falsifications                                   | Always included when directly relevant.  |

### Rerank formula

```text
score =
  w_exact        * exact_match
+ w_bm25         * bm25_norm
+ w_vector       * cosine_norm
+ w_entity       * entity_overlap
+ w_graph        * ppr_score
+ w_temporal     * temporal_fit
+ w_source       * source_quality
+ w_confidence   * confidence
+ w_utility      * utility
+ w_topic        * topic_strength
+ w_recency      * recency_decay
+ w_eval         * prior_success
- w_contradict   * unresolved_contradiction
- w_stale        * stale_penalty
- w_privacy      * access_penalty
```

The weights are intent-specific. A math query boosts exact symbols, equations, assumptions, and proof dependencies. A debugging query boosts recent episodes, exact errors, failed attempts, and verified skills. A literature query boosts resources, claims, citations, contradictions, and concept clusters.

## 12. Knowledge Compression

This is where the system becomes superhuman. It does not just store memories. It compresses them with provenance-preserving layers.

```text
L0 Raw artifact
   Full PDF, tool log, repo snapshot, dataset, chat, screenshot.
   Stored content-addressed. Not normally put into prompt.

L1 Episode
   What happened, timestamp, actor, source, hash, summary.

L2 Observation
   Dense, dated statement extracted from the episode.

L3 Atomic card
   Claim, equation, theorem, resource, skill, counterexample.

L4 Concept kernel
   Stable abstraction linking claims, equations, resources, skills, misconceptions, open questions.

L5 Topic atlas
   Community-level map of a whole domain: prerequisites, competing theories, open problems.

L6 Retrieval descriptor
   Ultra-short embedding/BM25 target used for fast indexing.

L7 Optional parametric distillation
   A small local model or adapter learns routing/ranking/compression patterns.
   Not canonical truth; only acceleration.
```

### Compression jobs

| Job                       | Trigger                                 | Output                                     |
| ------------------------- | --------------------------------------- | ------------------------------------------ |
| Observation compression   | Every event                             | Dense event summaries.                     |
| Deduplication             | Similarity > threshold                  | Supersession or merge proposal.            |
| Concept kernel generation | Strong topic cluster                    | A-MEM-style concept card.                  |
| Topic atlas generation    | Large corpus/domain                     | GraphRAG-style map.                        |
| Belief synthesis          | Claims around proposition change        | Updated belief state.                      |
| Contradiction resolution  | Live conflicting facts                  | Validity closure + warning.                |
| Skill distillation        | Repeated successful action pattern      | Draft skill.                               |
| Skill degradation         | Repeated failures                       | Degraded/retired skill.                    |
| FSRS review               | Memory retrievability falls             | Rehearsal queue.                           |
| Archive compaction        | Low utility, low importance, low access | Cold store, removed from hot vector index. |

### Compression must be reversible

A summary can never become the only truth. Every compressed layer carries backlinks to raw episodes and source spans.

## 13. Agent-Callable Tools

The agent should not “edit memory.” It should call strict tools that create validated memory mutations.

| Tool call                     | Purpose                                                                 | Effect                                                           |
| ----------------------------- | ----------------------------------------------------------------------- | ---------------------------------------------------------------- |
| `observe(event)`              | Capture raw event, tool result, message, file, paper, dataset, code run | Writes episode and observations.                                 |
| `recall(query)`               | Retrieve task-relevant memory                                           | Returns ContextPack and trace.                                   |
| `remember(candidate)`         | Propose a memory explicitly                                             | Goes through validation and policy gate.                         |
| `feedback(pack_id, outcome)`  | Tell memory whether retrieved context helped                            | Updates utility, FSRS, eval memory, skills.                      |
| `reflect(scope)`              | Ask memory to synthesize lessons                                        | Creates metacognitive lessons, belief revisions, contradictions. |
| `consolidate(scope)`          | Compress and organize                                                   | Generates concept kernels, topic maps, merges, archives.         |
| `rehearse(topic, n)`          | Strengthen weakening knowledge                                          | Returns review cards/questions.                                  |
| `verify_claim(claim_id)`      | Check source, contradiction, formal constraints                         | Updates confidence/status.                                       |
| `verify_equation(eq_id)`      | Run units/symbolic/proof checks                                         | Flags invalid equations.                                         |
| `promote_skill(candidate)`    | Turn repeated workflow into skill                                       | Requires tests and successful episodes.                          |
| `run_skill(skill_id, args)`   | Execute trusted workflow in sandbox                                     | Logs result as episode.                                          |
| `find_counterexamples(topic)` | Surface known failures                                                  | Prevents repeating mistakes.                                     |
| `timeline(entity)`            | Show fact history                                                       | Returns validity windows and supersessions.                      |
| `explain_recall(trace_id)`    | Explain why memory was retrieved or omitted                             | Debugs retrieval.                                                |
| `audit(memory_id)`            | Show provenance, receipts, mutations, citations                         | Ensures trust.                                                   |
| `archive(policy)`             | Move low-value memory out of hot indexes                                | Compresses working set.                                          |
| `rebuild_indexes()`           | Rebuild projections from canonical store                                | Proves durability.                                               |

This is what the agent calls to organize and compress over time: **observe → feedback → reflect → consolidate → rehearse → verify → promote/archive**.

## 14. Topic Strengthening

Topic-level strengthening is more important than isolated fact strengthening.

```rust

    pub member_entities: Vec<EntityId>,

    pub recency: f32,
    pub recurrence: f32,
    pub utility: f32,
    pub novelty: f32,
    pub source_quality: f32,
    pub contradiction_pressure: f32,
    pub retrieval_success: f32,
    pub graph_neighbor_boost: f32,

    pub next_review_at: Option<DateTime<Utc>>,
    pub last_consolidated_at: Option<DateTime<Utc>>,
    pub open_questions: Vec<MemoryId>,
}
```

Score:

```text
decayed =
  previous_strength * exp(-hours_since_update / half_life_hours)

reinforcement =
  recency_weight       * recency
+ recurrence_weight    * recurrence
+ utility_weight       * utility
+ novelty_weight       * novelty
+ source_weight        * source_quality
+ success_weight       * retrieval_success
+ neighbor_weight      * graph_neighbor_boost
- contradiction_weight * unresolved_contradiction_pressure

new_strength = clamp(decayed + reinforcement, 0.0, 1.0)
```

Policies:

```text
High strength + high utility:
  generate or refresh concept kernel.

High strength + high contradiction pressure:
  run belief revision and source audit.

Low strength + mission-critical:
  schedule rehearsal.

High utility + low confidence:
  create targeted research question.

Repeated retrieval failure:
  create Eval memory and lower ranking.

Repeated skill success:
  strengthen linked topic and skill.
```

## 15. Procedural Compounding

Procedural memory is not a prose note saying “do X.” It is an executable, test-gated artifact.

Lifecycle:

```text
draft
→ candidate
→ tested
→ trusted
→ degraded
→ retired
```

Promotion requires:

```text
1. Clear task signature.
2. Preconditions.
3. At least one successful episode.
4. Repeatable test or validation command.
5. Sandboxed execution profile.
6. Source/provenance.
7. Failure handling.
8. Linked eval outcomes.
```

Example skill:

```text
Skill: extract_equations_from_arxiv_pdf

Preconditions:
  - resource.kind == PaperPdf
  - local PDF exists by content hash
  - latex OCR/parser available

Procedure:
  1. Extract text and math regions.
  2. Normalize LaTeX.
  3. Link variables to surrounding definitions.
  4. Run unit/assumption extraction.
  5. Create EquationCards.
  6. Attach source spans.

Tests:
  - fixture PDF produces expected equation count
  - every equation has source span
  - no duplicate normalized equations
```

A failed skill run creates a counterexample and an eval record. A repeated failure degrades the skill automatically.

## 16. Temporal Truth and Belief Revision

The system has three different objects that many memory systems confuse:

```text
Observation:
  "On May 12, the tool returned output O."

Claim:
  "Paper P states that method M improves metric X under assumptions A."

Belief:
  "The agent currently accepts proposition Y at confidence 0.74 because claims C1/C2 support it and K1 contradicts it."
```

Temporal fact update:

```text
old edge:
  subject = project
  predicate = uses_database
  object = sqlite
  valid_from = 2026-02-01
  valid_to = 2026-05-12
  status = superseded

new edge:
  subject = project
  predicate = uses_database
  object = redb
  valid_from = 2026-05-12
  valid_to = null
  supersedes = old_edge
```

Contradiction handling:

```text
1. Do not delete old fact.
2. Close old validity window if new evidence supersedes it.
3. Add Contradicts edge if both claims can coexist as contested evidence.
4. Update BeliefState, not ClaimCard, if the agent’s stance changes.
5. Always include contradiction warning in ContextPack when relevant.
```

## 18. Performance Targets

For a 10M-memory single-node deployment:

| Operation                     |                   Target |
| ----------------------------- | -----------------------: |
| `observe()` fast commit p50   |                   ≤ 5 ms |
| `observe()` fast commit p99   |                  ≤ 50 ms |
| Hot `recall()` p50            |                  ≤ 50 ms |
| Hot `recall()` p95            |                 ≤ 250 ms |
| Graph-expanded `recall()` p95 |                 ≤ 750 ms |
| Context pack assembly         |                  ≤ 50 ms |
| Cold artifact rehydration     |    Seconds, not hot path |
| Index rebuild from canonical  | Deterministic, resumable |
| Daemon overhead               |         ≤ 2 ms local IPC |

The trick is not indexing every raw token. The system indexes compact retrieval descriptors, concept kernels, claims, equations, skills, and entity edges. Raw artifacts stay addressable but cold.

### Scaling strategy

```text
1. Keep raw artifacts outside hot memory.
2. Store compact descriptors per memory.
3. Keep only high-utility memories in hot vector indexes.
4. Use BM25 for exact sparse recall over larger sets.
5. Use graph activation over compressed entity/concept graph, not raw chunks.
6. Use concept kernels to answer common topic queries cheaply.
7. Archive low-utility memories but keep them rehydratable.
8. Rebuild projections from canonical store when indexes drift.
9. Shard by project / domain / entity hash above 50M objects.
10. Keep memory writes append-only and retrieval projections disposable.
```

## 21. Example End-to-End Flow


### Contradiction

```text
New paper contradicts old claim.
System finds same subject/predicate with incompatible object.
Old claim remains stored.
Old active edge gets valid_to = now.
New claim becomes active if source/evidence wins.
BeliefState becomes contested or revised.
ContextPack later warns: "prior claim superseded on date X by source Y."
```

### Recall

```text
Agent asks: "Which approximation did we use last time for this simulation, and was it valid?"

Recall compiler:
  intent = DebugCode + ScientificClaim
  retrieve recent episodes
  retrieve simulation skill
  retrieve equation assumptions
  retrieve counterexamples
  retrieve temporal facts valid at previous run
  retrieve current contradictory evidence

ContextPack:
  - previous approximation
  - source episode
  - equation assumptions
  - failed run warning
  - current belief revision
  - recommended verified skill
```

## 23. Final Architecture Summary

The best possible design is:

```text
MIRIX-style typed cognitive lanes
+ Hindsight-style fact/experience/entity/belief separation
+ Graphiti/Zep-style temporal knowledge graph
+ A-MEM-style evolving Zettelkasten concept memory
+ Mastra-style observation compression
+ Mem0-style hybrid retrieval and low-token context
+ HippoRAG-style Personalized PageRank associative traversal
+ Voyager/Memento-style executable skill library
+ FSRS + Hebbian topic strengthening
+ Memory-R1-style learned memory policy
+ Git/WAL/source-span auditability
+ Formal math/science stores
+ Counterexample-first failure memory
+ Rust-native deterministic canonical store and rebuildable projections
```

The most important operating loop is:

```text
observe
→ ground
→ type
→ link
→ validate
→ commit
→ index
→ recall
→ act
→ feedback
→ reflect
→ consolidate
→ rehearse
→ compress
→ improve policy
```

This design is “human-like” in structure, but superhuman in execution: it has typed episodic/semantic/procedural memory, temporal truth, concept formation, source-grounded beliefs, skill learning, and reflection, while also having perfect audit trails, parallel recall, formal checks, deterministic rebuilds, and machine-speed rehearsal across millions of memories.
