# HYPERMNESIA: Rust Cognitive Memory Compiler v2

**Final design score: 97.0 / 100.**

The missing 3 points are not because the architecture is missing a known component. They are because current LLM extraction, contradiction detection, multimodal grounding, and learned memory-management policies are still imperfect. The system below is the strongest buildable design I can justify: superhuman in provenance, speed, scale, parallel recall, formal hooks, and compounding procedural learning; still dependent on the quality of extractors and evaluators.

## 1. Core thesis

HYPERMNESIA is a **Rust-first, local-first, event-sourced memory compiler** for long-running scientific, coding, and research agents.

It does five things that ordinary RAG does not do:

1. **It separates memory types.** An event, an observation, a claim, a belief, an equation, a skill, a resource, and a concept summary are different objects with different invariants.
2. **It never destroys history.** Contradictions, supersessions, and stale facts close temporal validity windows; they do not overwrite the past.
3. **It compiles memory before recall.** The agent receives a small, cited, task-shaped `ContextPack`, not raw chat history.
4. **It learns how to remember.** Feedback, retrieval success, skill success, spaced repetition, contradiction pressure, and topic utility change future storage and retrieval behavior.
5. **It compresses knowledge without losing provenance.** Summaries are lossy; provenance is lossless. Every compressed memory can be rehydrated to exact source evidence.

The central design rule:

```text
Canonical memory is the event-sourced, typed, cited ledger.
Everything else—vectors, BM25, graph traversal, concept kernels,
skills, summaries, caches, context packs—is a compiled projection.
```

# 2. Memory lanes

HYPERMNESIA uses **14 lanes**, not because more lanes are elegant, but because collapsing these categories creates hallucination and corruption.

| Lane             | Purpose                                                                          | Hot-path role                   |
| ---------------- | -------------------------------------------------------------------------------- | ------------------------------- |
| `Core`           | Stable identity, project constraints, user-approved preferences, current mission | Always loaded, tiny budget      |
| `Working`        | Current task state, scratchpad, open plan, active hypotheses                     | Short-lived, explicit TTL       |
| `Sensory`        | Raw recent input frames: messages, tool calls, screenshots, logs                 | Ring buffer before extraction   |
| `Episodic`       | Immutable “what happened” records                                                | Provenance root                 |
| `Observation`    | Dense dated summaries of episodes                                                | Fast continuity memory          |
| `Semantic`       | Atomic claims, entities, facts, relations                                        | Main knowledge graph            |
| `Belief`         | Current stance over evidence, not raw fact                                       | Scientific judgment             |
| `Concept`        | Zettelkasten-style synthesized kernels                                           | Long-term knowledge compression |
| `Procedural`     | Executable skills, scripts, workflows, tool recipes                              | Capability compounding          |
| `Resource`       | Papers, datasets, repos, files, APIs, commits, DOIs                              | External reality ledger         |
| `Formal`         | Equations, theorems, proofs, units, derivations                                  | Math/science rigor              |
| `Counterexample` | Mistakes, falsified assumptions, failed experiments, bad answers                 | Anti-repetition memory          |
| `Eval`           | Retrieval outcomes, answer quality, skill results                                | Self-improvement signal         |
| `Metacognitive`  | Lessons about how to reason, search, verify, debug, and learn                    | Agent strategy memory           |

This is MIRIX-like in cognitive breadth, Hindsight-like in epistemic separation, Graphiti-like in temporal truth, A-MEM-like in concept linking, Voyager-like in executable procedures, Mastra-like in observation compression, and HippoRAG-like in associative recall.

# 5. Canonical storage design

## 5.1 Storage stack

For “all Rust” I would not make SQLite the default canonical store. SQLite is excellent, but the strongest Rust-native design is:

| Need                      | Default                                     | Why                                                          |
| ------------------------- | ------------------------------------------- | ------------------------------------------------------------ |
| Append-only event log     | Custom binary WAL segments + JSONL receipts | Replayable, inspectable, crash-safe                          |
| Canonical object registry | `redb`                                      | Pure Rust, transactional, embedded                           |
| High-write LSM partitions | `fjall` optional                            | Good for high-volume append/update projections               |
| Text search               | Tantivy                                     | Rust-native BM25/full-text                                   |
| Vector search             | Rust HNSW index, adapter boundary           | Fast local semantic retrieval                                |
| Graph                     | Custom mmap CSR + delta edge log            | Scales better than loading a giant `petgraph` for 10M+ nodes |
| Entity aliases            | FST + normalized alias tables               | Very fast exact/near-exact entity lookup                     |
| Filters                   | Roaring bitmaps                             | Fast lane/time/status/entity filtering                       |
| Curated cards             | Git-backed Markdown/YAML                    | Human review, diffs, code-review workflow                    |
| Raw artifacts             | Content-addressed object store              | Avoid bloating canonical metadata                            |

Rust is appropriate for this system because it is built for memory-efficient, performance-critical services and its ownership/type system gives memory-safety and thread-safety guarantees at compile time. ([Rust][4])

# 8. Write path: realtime learning without blocking

The write path is split into **synchronous commit** and **asynchronous cognition**.

## 8.1 Synchronous hot path

Target: **p50 under 5 ms**, because the agent should not pause to “think about memory.”

```text
observe(event)
  -> validate access/privacy
  -> hash payload
  -> append EpisodeCommitted to WAL
  -> insert sensory/episode shell into canonical store
  -> enqueue extraction jobs
  -> return MutationReceipt
```

The hot path only guarantees: “we captured what happened.”

## 8.2 Asynchronous cognitive path

```text
EpisodeCommitted
  -> segment event
  -> create dense observations
  -> extract candidate claims/equations/resources/skills
  -> normalize entities, aliases, units, predicates
  -> validate provenance and source spans
  -> check duplicates and contradictions
  -> route to lanes
  -> mutate temporal graph
  -> update belief states
  -> update concept kernels
  -> update indexes
  -> emit receipts and traces
```

This gives realtime memory without forcing expensive LLM extraction into the critical path.

## 8.3 Fact update rule

Never destructively update a fact.

```text
Existing edge:
  subject: "model X"
  predicate: "supports"
  object: "function calling"
  valid_from: 2025-01-01
  valid_to: null
  status: trusted

New contradicting evidence arrives at 2026-05-12.

Mutation:
  old.valid_to = 2026-05-12
  old.status = superseded

New edge:
  subject: "model X"
  predicate: "does_not_support"
  object: "function calling"
  valid_from: 2026-05-12
  valid_to: null
  status: trusted
  supersedes: [old_edge_id]
```

## 8.4 Procedural promotion rule

A code snippet is not a skill until it passes gates:

```text
draft snippet
  -> has task signature
  -> has preconditions
  -> linked to successful episode
  -> AST/sandbox validated
  -> test recipe passes
  -> promoted to trusted SkillCard
```

Failures are first-class memories. They are not erased.

## 9.2 Retrieval fanout

The recall system queries all relevant channels in parallel:

```text
1. Core memory lookup
2. Observation log lookup
3. BM25 exact search
4. HNSW semantic search
5. Entity alias lookup
6. Temporal graph valid-at query
7. Concept graph neighbors
8. HippoRAG-style bounded PPR
9. Equation/symbol exact index
10. Code/resource path index
11. Skill precondition index
12. Counterexample and failed-attempt index
13. Eval memory for prior bad answers
```

HippoRAG is especially important for science because its graph plus Personalized PageRank mechanism is designed for deeper multi-hop retrieval and reported major efficiency gains compared with iterative retrieval approaches. ([arXiv][5])

## 9.3 Ranking formula

```text
score =
  w_exact        * exact_match
+ w_semantic     * vector_similarity
+ w_bm25         * bm25_score
+ w_entity       * entity_match
+ w_graph        * ppr_score
+ w_temporal     * temporal_validity
+ w_confidence   * confidence
+ w_source       * source_quality
+ w_utility      * utility
+ w_topic        * topic_strength
+ w_success      * retrieval_success
+ w_recency      * recency_if_intent_requires_it
- w_stale        * stale_penalty
- w_contradict   * unresolved_contradiction_penalty
- w_access       * access_policy_penalty
```

Then the compiler applies **hard filters**:

```text
- never include Rejected
- never include Secret for unauthorized caller
- never include stale facts unless user asks for history
- include known counterexamples when the query matches the same task pattern
- include contradictions for scientific or planning queries
- require citations for factual claims
```

# 10. Knowledge compression strategy

This is the most important part.

The system uses **five compression levels**:

```text
L0 Raw artifact
   Exact messages, files, logs, screenshots, papers, code output.
   Stored by content hash. Not used in hot recall.

L1 Episode
   Immutable record of what happened, with hash chain and metadata.

L2 Observation
   Dense dated note: “what mattered in this episode.”
   Cheap to retrieve. Good for continuity.

L3 Atomic cards
   ClaimCard, EquationCard, SkillCard, ResourceCard, CounterexampleCard.
   Normalized, cited, temporally valid.

L4 Concept kernel
   A compressed, curated, self-organizing topic model:
   definition, prerequisites, claims, equations, examples, failures,
   contradictions, resources, skills, open questions.

L5 Master schema / belief model
   High-level theory, current stance, evidence balance,
   unresolved uncertainty, and action implications.
```

## 10.1 Compression invariant

```text
Lossy summaries may be wrong.
Provenance must be exact.

Therefore every lossy layer must link to:
  - source episode IDs
  - source spans
  - source hashes
  - extraction method
  - confidence
  - contradiction links
```

## 10.2 Concept kernel compaction

A concept kernel is regenerated when:

```text
topic_strength > threshold
OR new contradiction pressure > threshold
OR retrieval failures cluster around the topic
OR source coverage improves materially
OR a skill succeeds/fails repeatedly on the topic
OR scheduled review is due
```

A kernel contains:

```text
- name and aliases
- one-sentence definition
- prerequisite map
- canonical examples
- key claims with confidence
- equations and units
- theorem/proof dependencies
- strongest evidence
- strongest contradictions
- failed attempts and counterexamples
- related skills
- open questions
- source coverage
- last review date
- compression ratio
```

## 10.3 Generational memory compaction

Memory is tiered like an LSM tree:

```text
Gen0: recent raw events, high write rate
Gen1: observations and atomic extractions
Gen2: active concept kernels and trusted claims
Gen3: cold archived episodes and resources
Gen4: external immutable artifacts
```

Hot recall searches Gen1–Gen2 first. Gen0 is used for recency. Gen3/Gen4 are only rehydrated when citations, exact evidence, or deep review are needed.

# 11. Topic strengthening and forgetting

For human-like long-term learning, HYPERMNESIA uses **FSRS-like memory state**, but applied to concepts, claims, and skills. FSRS tracks retrievability, stability, and difficulty, and uses review history to estimate memory state; this maps naturally to agent memory scheduling. ([Anki FAQs][6])

```text
decayed_strength =
    previous_strength * exp(-hours_since_update / half_life_hours)

reinforcement =
    w_usefulness       * downstream_success
  + w_recurrence       * recurrence
  + w_source_quality   * source_quality
  + w_novelty          * novelty
  + w_graph_centrality * graph_neighbor_boost
  + w_skill_success    * linked_skill_reliability
  + w_human_pin        * user_importance
  - w_contradiction    * unresolved_contradiction_pressure
  - w_failure          * retrieval_failure_penalty

new_strength = clamp(decayed_strength + reinforcement, 0, 1)
```

Forgetting does **not** mean deleting important evidence. It means:

```text
- demote from hot index
- reduce retrieval weight
- compress into concept kernel
- archive raw artifact
- keep provenance rehydratable
```

## 12.1 Most important organization/compression calls

| Call                         | What it does                                                    | When agent should call it                                     |
| ---------------------------- | --------------------------------------------------------------- | ------------------------------------------------------------- |
| `observe(event)`             | Captures what happened immediately                              | Every user message, tool result, code run, search, paper read |
| `feedback(pack_id, outcome)` | Tells memory whether retrieved context helped                   | After every answer, plan, tool action, or skill run           |
| `reflect(scope)`             | Finds lessons, contradictions, and belief updates               | After a failed task, surprising result, or research session   |

| `revise_belief(proposition)` | Recomputes current stance from evidence                         | When new evidence contradicts old belief                      |
| `rehearse_next(n)`           | Reviews weak but important topics                               | Idle time                                                     |
| `explain_recall(pack)`       | Shows why context was included or omitted                       | Debugging retrieval failures                                  |

This gives the agent explicit memory hygiene tools instead of letting it silently mutate memory.

# 15. Scale design for millions of memories

## 15.1 Sharding

Memory is partitioned by:

```text
primary: lane
secondary: topic/entity hash
tertiary: time bucket
```

For 10M memories:

```text
semantic claims:      separate HNSW + BM25 + graph partitions
observations:         time-bucketed, compressed, vector-searchable
resources:            exact metadata + citation graph
skills:               small, high-quality, precondition-indexed
concept kernels:      few but high-value, always hot
episodes:             append-only, mostly cold
```

## 15.2 Graph storage

Do not store the whole graph as high-overhead objects in memory.

Use:

```text
- dense integer node IDs
- mmap CSR adjacency arrays
- small delta edge log for recent writes
- periodic compaction into CSR
- temporal edge side table
- roaring bitmap filters for lane/status/time
- bounded PPR on seed neighborhoods
```

## 15.3 Vector storage

Use one vector index per lane and purpose:

```text
semantic_claims.fp16.hnsw
observations.fp16.hnsw
concept_kernels.fp16.hnsw
skills.fp16.hnsw
resources.fp16.hnsw
```

Old low-utility memories can have embeddings quantized or dropped if they are represented by concept kernels and remain rehydratable from source.

## 15.4 Cache hierarchy

```text
L0: pinned core memory
L1: hot concept kernels
L2: entity alias and topic maps
L3: PPR neighborhood cache
L4: context-pack cache
L5: raw source rehydration cache
```

## 15.5 Performance targets

| Operation                 |      1M memories |         10M memories | Notes                              |
| ------------------------- | ---------------: | -------------------: | ---------------------------------- |
| `observe` sync commit p50 |           < 3 ms |               < 5 ms | WAL + shell object only            |
| observation extraction    |            async |                async | not hot path                       |
| warm recall p50           |          < 40 ms |              < 90 ms | no LLM rerank                      |
| warm recall p95           |         < 150 ms |             < 300 ms | bounded graph expansion            |
| warm recall p99           |         < 500 ms |             < 900 ms | excludes source rehydration        |
| context pack compile      |          < 20 ms |              < 40 ms | after candidates                   |
| index rebuild             | hours acceptable | overnight acceptable | canonical store is source of truth |
| skill lookup p50          |          < 20 ms |              < 40 ms | small specialized index            |

The absolute rule: **no LLM call is required on the hot recall path.** LLM reranking can be optional but cannot be necessary for correctness.

# 16. Self-organization daemons

## 16.1 Observer daemon

Inspired by Observational Memory, but stricter:

```text
- watches sensory and episodic streams
- inserts temporal gap markers
- emits dense observations
- anchors every observation to exact episode ranges
- detects prediction gaps and corrections
```

Mastra’s OM explicitly uses Observer and Reflector background agents to maintain dense observations and reduce raw context bloat; HYPERMNESIA adopts that pattern but adds strict provenance and typed extraction. ([Mastra][7])

## 16.2 Reflector daemon

```text
- scans high-surprise episodes
- detects contradictions
- proposes belief revisions
- creates metacognitive lessons
- links concepts bidirectionally
- proposes concept kernel updates
```

## 16.3 Consolidator daemon

```text
- merges duplicates
- compacts episodes into observations
- compacts related claims into concept kernels
- archives cold raw artifacts
- strengthens useful topics
- demotes low-value noise
```

## 16.4 Skill daemon

```text
- detects repeated workflows
- proposes SkillCards
- runs sandbox tests
- tracks reliability
- repairs degraded skills
- stores failures as counterexamples
```

## 16.5 Manager policy daemon

Starts rule-based, later learned.

Inputs:

```text
- query type
- memory age
- source quality
- user feedback
- retrieval success
- contradiction pressure
- topic strength
- skill reliability
- downstream answer quality
```

Actions:

```text
ADD, UPDATE, SUPERSEDE, LINK, COMPRESS, ARCHIVE, REHEARSE, NOOP
```

Memory-R1 supports the direction of using learned policies for structured memory operations rather than relying only on static heuristics. ([arXiv][8])

# 17. State machines

## 17.1 Memory lifecycle

```mermaid
stateDiagram-v2
    [*] --> Proposed
    Proposed --> Trusted: validated provenance
    Proposed --> Rejected: invalid / duplicate / unsupported
    Trusted --> Consolidated: included in kernel
    Trusted --> Contradicted: conflicting evidence
    Trusted --> Superseded: newer valid fact
    Trusted --> Deprecated: low utility or stale method
    Contradicted --> Trusted: contradiction resolved
    Contradicted --> Superseded: replaced by stronger evidence
    Consolidated --> Archived: cold but preserved
    Deprecated --> Archived
    Rejected --> Archived
```

## 17.2 Skill lifecycle

```mermaid
stateDiagram-v2
    [*] --> Draft
    Draft --> Tested: preconditions + test recipe
    Tested --> Trusted: tests pass + successful episode
    Trusted --> Degraded: failures exceed threshold
    Degraded --> Trusted: repaired + tests pass
    Degraded --> Retired: repeated failure
    Trusted --> Retired: obsolete
```

## 17.3 Belief lifecycle

```mermaid
stateDiagram-v2
    [*] --> Unknown
    Unknown --> Tentative: weak evidence
    Tentative --> Supported: strong support
    Supported --> Contested: contradiction arrives
    Contested --> Revised: new evidence wins
    Revised --> Supported: confidence stabilizes
    Revised --> Retired: no longer useful / invalid
```

# 18. Technical invariants

These are non-negotiable.

```text
1. Every trusted memory has provenance.
2. Every claim links to at least one episode or resource.
3. Every factual ContextPack item has a citation.
4. Beliefs are not facts.
5. Contradictions do not overwrite history.
6. Skills are not trusted until tested.
7. Summaries are not source of truth.
8. Indexes are rebuildable from canonical memory.
9. Access policy is enforced before ranking output.
10. Rejected memories never appear in ContextPacks.
11. Temporal queries return facts valid at the requested time.
12. Retrieval traces explain included and omitted evidence.
```

# 19. Benchmark and testing plan

## 19.1 Unit tests

```text
- serde roundtrip for every schema version
- invalid lifecycle transitions rejected
- temporal validity windows work at boundary times
- fact supersession is idempotent
- belief cannot be trusted without evidence
- skill cannot be trusted without passing test
- context pack never exceeds token budget
- access policy redacts secret memories
```

## 19.2 Property tests

```text
- random memory graph -> no rejected item in recall
- random contradictions -> old edges closed, not deleted
- random context budget -> pack size <= budget
- random crash point -> replay yields same canonical state
```

# 21. What this design improves beyond CLAUDE, CODEX, and ANTIGRAVITY

The uploaded specs are already strong. My improvements are mainly about **scale discipline and compression mechanics**:

1. **Custom mmap CSR graph instead of generic in-memory graph.** Better for millions of nodes and tens/hundreds of millions of edges.
2. **Generational memory compaction.** Old raw memory is compressed into observations and concept kernels while remaining rehydratable.
3. **Fourteen lanes, not six or twelve.** Belief, observation, eval, counterexample, and formal memory deserve first-class treatment.
4. **No LLM on hot recall path.** Recall must work deterministically with local indexes.
5. **ContextPack is a compiler artifact.** It has traces, omitted evidence, contradiction notes, and deterministic packing.
6. **Feedback closes the learning loop.** The system learns from whether retrieved memory actually helped.
7. **Procedural memory has a real lifecycle.** Draft snippets are not trusted skills.
8. **Science/math objects are not plain text.** Equations, units, theorems, proofs, experiments, and derivations are structured.
