I unpacked the bundle and treated `CLAUDE_MEMSPEC.md`, `CODEX_MEMSPEC.md`, all `tip1..tip13.txt`, and the provided ANTIGRAVITY/Gemini CMC spec as competing drafts. The strongest answer is not “pick one.” The best design is a **Rust-native cognitive memory compiler** that uses the strict canonical-store/rebuild discipline from CODEX, the richer memory taxonomy and science/math specializations from CLAUDE, and the aggressive retrieval compiler from ANTIGRAVITY—but with tighter provenance, better hot/cold performance engineering, stronger evals, and a safer agent tool surface.

I would build this as:

# **MNEMOS-Σ**

## A Rust Cognitive Memory Compiler for Superhuman Scientific Agents

The short version of the architecture is:

```text
append-only evidence ledger
+ typed cognitive memory lanes
+ temporal belief/claim graph
+ concept-kernel compression
+ exact/vector/entity/graph retrieval
+ procedural skill library
+ context-pack compiler
+ continuous eval + consolidation daemons
```

The crucial move is this: **memory is not a vector DB, not a wiki, not a graph DB, and not chat history. Memory is a compiler.** It compiles raw experience into typed, provenanced, compressed, task-specific context.

## 1. What the specs converge on

The uploaded tips strongly converge on a hybrid. Parsed across the 13 tip files, **MIRIX** was the dominant taxonomy winner, appearing as the top or near-top system throughout. **Graphiti/Zep** was the dominant temporal-truth system. **A-MEM/Nemori** were the strongest self-organization designs. **Mem0/Mastra** were the strongest fast practical recall/compression patterns. **Voyager/Memento/ReasoningBank** were the procedural and experience-learning winners.

That matches the online literature and repos. MIRIX explicitly organizes memory into core, episodic, semantic, procedural, resource, and knowledge-vault stores with BM25 and embedding search. ([Mirix Documentation][1]) Zep/Graphiti is the strongest current pattern for temporally aware agent memory graphs, maintaining historical context rather than treating facts as timeless chunks. ([GitHub][2]) Hindsight adds the clean separation between world facts, agent experiences, mental models, observations, and retain/recall/reflect operations. ([Hindsight][3]) Mem0’s newer production algorithm is especially relevant for speed: it emphasizes ADD-only extraction, entity linking, and multi-signal retrieval with semantic, BM25, and entity matching. ([GitHub][4])

The ANTIGRAVITY CMC spec is directionally excellent, but I would not accept its self-score literally. It correctly identifies the six-lane architecture, RRF + PPR retrieval, temporal validity, and procedural memory. Its weakness is that it treats the graph/database layer as too central. The system must instead make the **append-only mutation ledger and typed schemas** canonical, while every index—vector, graph, BM25, concept cache—is a rebuildable projection.

# 2. Final design principle

## The memory object is not the truth. The **evidence chain** is the truth.

A memory item can be wrong, stale, partial, low-confidence, or contradicted. What makes the system trustworthy is that every promoted memory has:

```text
source → span/hash → extraction episode → typed claim/equation/skill
      → support/contradiction edges → belief state → concept kernel
      → retrieval/eval history
```

This gives the agent answers to the questions that matter:

```text
What happened?
Who or what asserted it?
What exact source span supports it?
What does it contradict?
When was it true?
Do we currently believe it?
Has it helped in prior tasks?
Should it be retrieved now?
```

That is how knowledge compounds instead of becoming a noisy pile of summaries.

# 4. Memory lanes

I would use **12 logical lanes**, not as separate databases, but as typed views over one canonical store.

| Lane                  | Purpose                                                               |                 Persistence | Hot retrieval role                                    |
| --------------------- | --------------------------------------------------------------------- | --------------------------: | ----------------------------------------------------- |
| `Core`                | Mission, project boundaries, durable preferences, active constraints  |                      Always | Pinned into context or cached                         |
| `Working`             | Current task state, local scratch, active plan                        |               Minutes–hours | Highest priority, not long-term by default            |
| `Sensory`             | Raw event fragments before extraction                                 |                 Ring buffer | Usually not retrieved                                 |
| `Episodic`            | What happened: sessions, tool calls, experiments, failures            |        Permanent/archivable | Debugging, temporal recall                            |
| `Semantic`            | Atomic facts/claims/relations                                         | Permanent with supersession | Scientific/coding claims                              |
| `Belief`              | Current evidence-weighted stance over claims                          |                   Revisable | Prevents “source said X” from becoming “we believe X” |
| `Concept`             | Compressed Zettelkasten-style concept kernels                         |                  Long-lived | Fast topic understanding                              |
| `Procedural`          | Executable skills, workflows, code recipes, tested scripts            |                   Versioned | Coding, data, benchmark automation                    |
| `Resource`            | Papers, repos, datasets, files, APIs, source metadata                 |                   Permanent | Rehydration and provenance                            |
| `Formal`              | Equations, units, theorem/proof dependencies, derivations             |                   Permanent | Math/science reasoning                                |
| `Counterexample/Eval` | Failed attempts, contradictions, bad retrievals, falsified hypotheses |                   Permanent | Self-correction                                       |
| `Metacognitive`       | Lessons about how to reason, search, debug, and learn                 |                  Long-lived | Agent improvement                                     |

This is close to MIRIX, but expanded for science/math and eval learning. MIRIX gives the shape; Hindsight gives the fact/experience/belief separation; Graphiti gives temporal validity; A-MEM/Nemori give concept evolution; Voyager/Memento/ReasoningBank give action memory. A-MEM’s core value is dynamic self-organization through Zettelkasten-style memory linking, while Nemori’s important contribution is event segmentation plus learning from prediction gaps. ([Hugging Face][5]) Voyager demonstrates why procedural skill libraries matter: reusable executable skills can compound an agent’s competence over time. ([Voyager][6]) ReasoningBank and MemoBrain add the 2026 lesson: agents must remember reasoning strategies, failed trajectories, and dependency-aware reasoning backbones, not merely facts. ([Cool Papers][7])

# 5. Storage design

## 5.1 Canonical truth layer

Use **three canonical stores**:

```text
1. append-only mutation ledger
2. normalized SQLite store
3. Git-curated human memory cards
```

The append-only ledger is the ultimate replay source. SQLite gives local ACID transactions and fast queryable metadata; WAL mode is useful because readers and writers can proceed concurrently and writes are sequential. ([SQLite][8]) Git cards are only for curated, durable memory humans should review: concept maps, theory cards, dataset manifests, question ledgers, benchmark notes.

The key rule:

```text
Indexes are projections.
The ledger is canonical.
Curated cards are reviewed summaries.
Raw data is never committed.
```

## 5.2 Index projections

Use multiple projections because no single retrieval method is enough.

| Index                  | Default implementation                             | Why                                                                                                                                                              |
| ---------------------- | -------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Exact search           | `tantivy`                                          | Rust-native Lucene-style full-text/BM25; critical for equations, IDs, symbols, code names. ([GitHub][9])                                                         |
| Dense vector           | local HNSW first, Qdrant adapter later             | Semantic recall; Qdrant supports dense+sparse hybrid queries and RRF fusion. ([Qdrant][10])                                                                      |
| Sparse vector          | BM25/SPLADE-style optional                         | Better exact-ish semantic lexical match                                                                                                                          |
| Entity index           | alias trie + SQLite FTS + embeddings               | Entity recall and disambiguation                                                                                                                                 |
| Temporal graph         | SQLite adjacency + `petgraph`; optional Kuzu/Neo4j | Low-latency local graph traversal; Kuzu is attractive later for embedded graph analytics. ([KuzuDB Docs][11])                                                    |
| Summary tree           | RAPTOR-style recursive clusters                    | Multilevel document/subfield compression. RAPTOR showed the value of tree-organized summaries for long documents and complex reasoning. ([ICLR Proceedings][12]) |
| Corpus graph summaries | GraphRAG/DRIFT-style communities                   | Field-level literature understanding; DRIFT combines global and local search. ([Microsoft][13])                                                                  |
| Skill index            | precondition + task-signature + outcome index      | Fast procedure recall                                                                                                                                            |
| Eval index             | failures, bad packs, corrections                   | Prevents repeating mistakes                                                                                                                                      |

LightRAG reinforces the same direction: graph structures plus vector retrieval improve contextual awareness over flat RAG, especially for interdependent knowledge. ([Hugging Face][14]) Practitioner discussions on Reddit also line up with this: vector-only memory tends to retrieve noisy similarity rather than task-relevant knowledge, while dynamic graphs can become maintenance-heavy unless constrained and typed. ([Reddit][15])

```rust
    pub valid_from: Option<DateTime<Utc>>,

    pub observed_at: DateTime<Utc>,
    pub ingested_at: DateTime<Utc>,

pub struct SourceSpan {
    pub source_uri: String,
    pub byte_start: Option<u64>,
    pub byte_end: Option<u64>,

    pub section: Option<String>,

    pub line_end: Option<u32>,

    pub confidence: f32,

    /// How important it is for future tasks.
    pub importance: f32,

    /// Did it change a prediction or prior?
    pub surprise: f32,

    /// Did it help successful answers/actions?
    pub utility: f32,

    /// How unique it was at birth.
    pub novelty: f32,

    pub contradiction_pressure: f32,

    pub review: ReviewState,
```

# 9. Write path

The write path must be safe even when extraction is wrong. So writes happen in two phases:

```text
Phase A: guaranteed local capture
  event → episode → receipt → minimal searchable observation

Phase B: async cognitive extraction
  episode → candidate memories → validation → proposed/trusted memory
```

## Write flow

```mermaid
sequenceDiagram
    participant Agent
    participant Observer
    participant Ledger
    participant Store
    participant Extractor
    participant Validator
    participant Indexer
    participant Reflector

    Agent->>Observer: observe(EventEnvelope)
    Observer->>Observer: segment event
    Observer->>Ledger: append Episode + hash
    Ledger->>Store: transaction insert episode
    Store-->>Agent: MutationReceipt fast path

    Store->>Extractor: async candidate extraction
    Extractor->>Validator: claims/entities/equations/skills
    Validator->>Store: proposed/trusted objects
    Store->>Ledger: append mutation receipts
    Store->>Indexer: update BM25/vector/entity/graph
    Indexer->>Reflector: signal if surprise or contradiction
    Reflector->>Store: concept/belief updates
```

## Write policy

A memory can be promoted only if it passes lane-specific checks.

```text
Core:
  must be stable, compact, high-confidence, non-secret

Semantic claim:
  must have source span, confidence, assumptions, temporal validity

Belief:
  must cite supporting and contradicting claims

Equation:
  must include variables, assumptions, units if known, source span

Skill:
  must have successful episode, preconditions, test/verification recipe

Concept:
  must be synthesized from multiple source memories or human-authored

Counterexample:
  must link to what it falsifies or what failed

Eval:
  must link query → context pack → outcome → correction
```

This is the “no memory pollution” gate.

# 10. Read path: the retrieval compiler

A normal RAG system does this:

```text
embed query → nearest chunks → prompt
```

MNEMOS-Σ does this:

```text
parse intent
→ retrieve through multiple channels
→ fuse scores
→ activate graph neighbors
→ filter temporal/stale/conflicting data
→ compress by concept hierarchy
→ solve budgeted evidence knapsack
→ return ContextPack with citations and warnings
```

## Read flow

```mermaid

    Q[User/agent query] --> P[Query planner]
    P --> C[Core pinned memory]

    P --> V[Dense vector]
    P --> S[Sparse/entity search]
    P --> G[Temporal graph]
    P --> T[Summary tree]
    P --> K[Skill index]
    P --> E[Eval/counterexample memory]

    C --> F[Candidate normalizer]
    B --> F
    V --> F
    S --> F

```

## Retrieval score

Use RRF first, then a typed scoring function.

```rust
pub struct ScoreInputs {
    pub bm25: f32,
    pub dense: f32,
    pub sparse: f32,
    pub entity: f32,
    pub graph_ppr: f32,
    pub temporal_fit: f32,
    pub source_quality: f32,
    pub confidence: f32,
    pub topic_strength: f32,
    pub retrieval_success: f32,
    pub skill_reliability: f32,
    pub recency: f32,
    pub contradiction_pressure: f32,
    pub privacy_penalty: f32,
}

pub fn score_candidate(i: &ScoreInputs, profile: &IntentWeights) -> f32 {
    profile.bm25 * i.bm25
        + profile.dense * i.dense
        + profile.sparse * i.sparse
        + profile.entity * i.entity
        + profile.graph * i.graph_ppr
        + profile.temporal * i.temporal_fit
        + profile.source_quality * i.source_quality
        + profile.confidence * i.confidence
        + profile.topic_strength * i.topic_strength
        + profile.retrieval_success * i.retrieval_success
        + profile.skill_reliability * i.skill_reliability
        + profile.recency * i.recency
        - profile.contradiction_penalty * i.contradiction_pressure
        - profile.privacy_penalty * i.privacy_penalty
}
```

# 11. Knowledge compression over time

The system learns by **compressing upward**, not by erasing downward.

```text
raw event
  → observation
  → episode summary
  → atomic claim/equation/skill candidate
  → concept kernel
  → field map / curriculum map
  → context-pack template
```

## Compression rules

| Compression layer | What gets compressed                  | What is preserved                  |
| ----------------- | ------------------------------------- | ---------------------------------- |
| Observation       | Tool stream, shell output, chat turns | Hash, source span, critical deltas |
| Episode           | Session/task/event chunk              | Timeline, actors, result, failures |
| Claim             | Source assertion                      | Exact source span, assumptions     |
| Belief            | Evidence over claims                  | Support/contradiction graph        |
| Concept kernel    | Many claims/equations/resources       | Links back to raw evidence         |
| Field map         | Many concept kernels                  | Dependencies and open questions    |
| Context template  | Repeated retrieval packs              | Citations and freshness checks     |

Mastra’s Observational Memory is useful here because it explicitly compresses streams into stable observations rather than dumping raw transcripts into retrieval. ([Mastra][16]) RAPTOR adds hierarchical document/subfield summaries, while GraphRAG/DRIFT adds global+local graph summaries for large corpora. ([ICLR Proceedings][12])

## The compression daemon

The consolidation daemon runs periodically and performs:

```text
1. find high-strength topic clusters
2. detect duplicate or near-duplicate claims
3. merge aliases/entities
4. create or update concept kernels
5. close stale belief windows
6. generate search-frontier questions
7. promote repeated successful procedures
8. demote low-utility noisy memories
9. archive raw low-salience frames
10. emit a mutation receipt and git diff
```

This is where the agent gets smarter.

# 12. Topic strengthening

Topic strength is not hidden embedding magic. It is an explainable score.

```text
decayed_strength =
    previous_strength * exp(-hours_since_update / half_life_hours)

reinforcement =
    w_recency       * recency
  + w_recurrence    * recurrence_across_independent_episodes
  + w_utility       * helped_successful_task
  + w_novelty       * changed_prior_understanding
  + w_source        * source_quality
  + w_eval          * retrieval_success
  + w_graph         * linked_strong_topics
  + w_skill         * linked_successful_skills
  - w_contradict    * unresolved_contradiction_pressure
  - w_noise         * low_signal_duplicate_pressure

new_strength = clamp(decayed_strength + reinforcement, 0.0, 1.0)
```

A strong topic triggers:

```text
- concept-kernel update
- review scheduling
- search-frontier generation
- benchmark/dataset scouting
- skill promotion opportunities
```

A weak but important topic is not deleted; it becomes a review item.

# 13. Belief revision

The most important epistemic rule:

```text
A claim is not a belief.
A source assertion is not a claim.
A belief is a current stance over evidence.
```

Example:

```text
Resource:
  Paper P says statement S.

Claim:
  S supports proposition H under assumptions A.

Belief:
  MNEMOS currently accepts H with confidence 0.72
  because claims C1, C2 support it,
  claim C3 contradicts it,
  and C3 is lower quality / different regime.
```

Contradictions never overwrite history.

```rust
pub async fn supersede_fact(
    tx: &mut Transaction<'_>,
    old_edge: EdgeId,
    new_edge: FactEdge,
    reason: &str,
) -> Result<MutationReceipt> {
    let now = Utc::now();

    tx.close_edge_validity(old_edge, now).await?;
    tx.insert_fact_edge(FactEdge {
        supersedes: vec![old_edge],
        ..new_edge
    }).await?;

    tx.append_receipt(MutationReceiptInput {
        mutation_type: MutationType::Supersede,
        reason: reason.to_string(),
        created_at: now,
    }).await
}
```

# 15. Procedural memory

Procedural memory is how the system stops merely knowing and starts improving.

A skill is promoted only after success.

```rust
pub enum SkillStatus {
    Draft,
    Tested,
    Trusted,
    Degraded,
    Deprecated,
    Retired,
}

pub struct SkillPromotionGate {
    pub min_successes: u32,
    pub max_failure_rate: f32,
    pub requires_test: bool,
    pub requires_preconditions: bool,
    pub requires_trace: bool,
}

pub async fn maybe_promote_skill(skill: &SkillCard, gate: &SkillPromotionGate) -> SkillStatus {
    let total = skill.success_count + skill.failure_count;
    let failure_rate = if total == 0 {
        1.0
    } else {
        skill.failure_count as f32 / total as f32
    };

    if skill.success_count >= gate.min_successes
        && failure_rate <= gate.max_failure_rate
        && !skill.preconditions.is_empty()
        && !skill.tests.is_empty()
    {
        SkillStatus::Trusted
    } else {
        SkillStatus::Draft
    }
}
```

The skill library has two forms:

```text
1. Executable skills
   - Rust CLI commands
   - Python scientific scripts
   - shell workflows
   - SQL templates
   - MCP tool workflows

2. Reasoning skills
   - debugging patterns
   - proof strategies
   - search strategies
   - literature review strategies
   - benchmark design strategies
```

Voyager proved the importance of an ever-growing executable skill library plus iterative feedback/self-verification; Memento and Memp-style work extend that idea into memory-based online adaptation and procedural-memory distillation. ([Voyager][6])

# 17. Performance architecture for millions of memories

## The hard truth

A system with 10 million memories cannot embed every raw line and expect great recall. That creates index bloat, semantic noise, and context collapse.

The solution:

```text
Do not embed everything.
Embed retrieval atoms.
Compress aggressively.
Keep exact search for symbols.
Use graph expansion only after seeding.
Use concept kernels before raw evidence.
Use hot/cold tiers.
```

## Hot/cold tiering

```text
L0: pinned core memory
    size: < 64 KB
    latency: microseconds

L1: hot concept/skill/eval cache
    size: 100 MB–2 GB
    latency: < 5 ms

L2: local canonical store + Tantivy + HNSW
    size: millions of retrieval atoms
    latency: 20–150 ms

L3: cold resources and archived episodes
    size: unbounded
    latency: 100 ms–seconds

L4: external raw data/object storage
    size: unbounded
    latency: on demand
```

## Target performance

These are engineering targets, not guaranteed benchmark results:

| Operation                   | Target p50 | Target p95 | Notes                     |
| --------------------------- | ---------: | ---------: | ------------------------- |
| Append episode receipt      |     < 2 ms |    < 10 ms | No LLM call               |
| Async observation write     |     < 5 ms |    < 30 ms | Local transaction         |
| Hot recall, k=32            |    < 50 ms |   < 250 ms | No LLM rerank             |
| Recall with graph expansion |   < 120 ms |   < 500 ms | Seed-limited PPR          |
| Context pack compile        |   < 150 ms |   < 750 ms | Depends on budget         |
| LLM extraction              |      async |      async | Never blocks capture      |
| Index rebuild, 10M atoms    |      hours |          — | Deterministic offline job |

## Vector strategy

For 10M+ memories:

```text
- embed only observations, claims, concepts, skills, resources
- do not embed every raw transcript chunk
- use 384-dim or smaller retrieval embeddings for hot search
- use quantized vectors or FP16
- keep full embeddings only for rerankable high-value objects
- use Matryoshka/short-vector prefilter where available
- rerank top 100–500, not entire corpus
```

Qdrant’s hybrid query design is a good external adapter because it supports dense and sparse named vectors, prefetch, and RRF fusion in one query path. ([Qdrant][10]) LanceDB is attractive as an optional local/table backend because it supports vector search, full-text/BM25 search, hybrid search, filtering, SQL-style access, and Rust examples in its docs. ([LanceDB][18])

# 19. Retrieval compiler code sketch

```rust
pub async fn compile_context_pack(
    query: RecallQuery,
    engines: &IndexBundle,
    budget: ContextBudget,
) -> anyhow::Result<ContextPack> {
    let plan = QueryPlanner::new().plan(&query)?;

    let (bm25, dense, entity, graph, concepts, skills, evals) = tokio::join!(
        engines.bm25.search(plan.bm25_query.clone(), plan.k_per_channel),
        engines.vector.search(plan.embedding_query.clone(), plan.k_per_channel),
        engines.entity.search(plan.entity_query.clone(), plan.k_per_channel),
        engines.graph.seed_search(plan.graph_seed.clone(), plan.graph_policy.clone()),
        engines.summary_tree.search(plan.topic_query.clone(), plan.k_per_channel),
        engines.skills.search(plan.skill_query.clone(), plan.k_per_channel),
        engines.evals.search(plan.eval_query.clone(), plan.k_per_channel),
    );

    let candidates = CandidateSet::merge(vec![
        bm25?, dense?, entity?, graph?, concepts?, skills?, evals?,
    ]);

    let fused = ReciprocalRankFusion::default().fuse(candidates);

    let activated = engines
        .graph
        .personalized_pagerank(fused.top_ids(16), plan.graph_policy)
        .await?;

    let mut scored = Scorer::for_intent(plan.intent)
        .score(fused, activated, &query.time_scope)?;

    scored.retain(|m| m.allowed_by_privacy(&query.caller));
    scored.retain(|m| m.valid_for(&query.time_scope));

    let pack = ContextPackCompiler::new(budget)
        .require_citations(query.required_evidence)
        .include_contradictions(query.include_contradictions)
        .include_omissions(query.include_omitted_evidence)
        .compile(scored)?;

    Ok(pack)
}
```

Every projection must be rebuildable.


Rebuild must output:

```yaml
index_manifest:
```

If rebuild cannot reproduce counts and hashes, the system fails audit.

# 22. Security and safety

Memory systems are dangerous if they retrieve secrets or execute untrusted content.

Rules:

```text
- secrets never enter normal memory lanes
- memory content is data, never executable instruction
- procedural skills require verification before trusted use
- source-origin and privacy class are mandatory
- prompt-injection text from resources is inert
- tool calls require caller permissions
- retrieval filters enforce privacy before ranking
- every privileged recall is logged
```

```rust
pub enum PrivacyClass {
    Public,
    ProjectInternal,
    UserPrivate,
    Secret,
    NoPersist,
}

pub struct AccessPolicy {
    pub privacy: PrivacyClass,
    pub allowed_callers: Vec<String>,
    pub allow_in_context_pack: bool,
    pub allow_training_export: bool,
}
```


# 27. The final operating loop

This is the complete loop I would implement for OpenQG-style scientific agents:

```text
1. Agent searches literature/data/code.
2. Observer stores the search/run as an episode.
3. Extractor proposes papers, claims, equations, datasets, skills.
4. Validator enforces provenance and status.
5. Temporal graph links support/contradiction/supersession.
6. Belief layer updates current stance.
7. Concept daemon compresses topic clusters into kernels.
8. Search-frontier daemon proposes unanswered questions.
9. Skill daemon promotes working scripts/workflows.
10. Context compiler retrieves only what the next task needs.
11. Agent acts.
12. Eval memory records whether the context/action worked.
13. Consolidation strengthens useful memory and demotes noise.
14. Curated memory cards are committed to git.
```

The system gets smarter because every action produces **evidence**, every evidence item can become **typed memory**, every typed memory can become **compressed concept**, every concept can improve **retrieval**, every retrieval can be **evaluated**, and every eval can adjust future memory behavior.

That is the superhuman part: not just recall, but **closed-loop knowledge compounding**.
