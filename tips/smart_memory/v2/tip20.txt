# Final Design: **NOESIS-RS — A Superhuman Cognitive Memory Compiler**

**One-sentence thesis:**
NOESIS-RS is a Rust-native, local-first, append-only, temporally grounded, multi-lane memory compiler that turns raw agent experience into compressed, cited, query-shaped context packs, while preserving exact provenance, learning reusable skills, revising beliefs, strengthening topics, and scaling to tens of millions of memories through hot/warm/cold indexing tiers.

The core rule:

```text
The vector database is not memory.
The chat transcript is not memory.
A summary is not truth.
Memory is an audited, typed, temporal, source-linked compiler pipeline.
```

The system should behave more like a superhuman research mind than a database:

```text
Experience → observations → claims/entities/equations/skills → concept kernels
→ belief revisions → procedural playbooks → context packs → action → feedback
→ stronger topics, better skills, cleaner concepts, fewer repeated mistakes.
```

## 1. Why this beats the prior specs

The CLAUDE spec is strong on cognitive taxonomy, FSRS strengthening, math/science objects, and Rust crates. The CODEX spec is stronger on deterministic local-first buildability, mutation receipts, recall contracts, and handoff discipline. The ANTIGRAVITY CMC spec is strongest as a conceptual synthesis of MIRIX, Graphiti, A-MEM, HippoRAG, Voyager, and Mastra, but it overclaims “perfect” scoring and under-specifies compression, eval, build staging, and million-scale index economics.

NOESIS-RS keeps the best of all three, then adds five missing pieces:

1. **A proof-preserving compression pyramid** so millions of memories can shrink into topic kernels without losing source traceability.
2. **A hot/warm/cold index hierarchy** so recall stays fast even when the canonical store has 10M–100M objects.
3. **An explicit agent-call protocol** so agents know when to observe, recall, reflect, strengthen, compress, verify, and mine skills.
4. **A richer evaluation rubric** tied to LongMemEval, MemoryAgentBench, retrieval latency, contradiction handling, provenance coverage, compression ratio, and skill success. LongMemEval evaluates long-term memory abilities including information extraction, multi-session reasoning, temporal reasoning, knowledge updates, and abstention; MemoryAgentBench evaluates accurate retrieval, test-time learning, long-range understanding, and selective forgetting. ([arXiv][2])
5. **A scale-realistic storage plan** using canonical append-only memory, rebuildable projections, quantized embeddings, graph CSR snapshots, Roaring bitmaps, Tantivy exact search, optional Qdrant, and optional Kuzu-style graph adapters. Tantivy is a Rust full-text search engine, Qdrant is a Rust vector search engine with HNSW and payload filtering, and Kuzu is an embedded graph database with Rust bindings. ([GitHub][3])

# 4. The memory taxonomy

NOESIS-RS should not have dozens of independent “stores.” Too many stores make retrieval brittle and schema maintenance painful. The best design is **eight logical lanes** with typed payloads inside each lane.

| Lane                       | Purpose                                                                                   | Examples                                                                     | Hot recall?                                   |
| -------------------------- | ----------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- | --------------------------------------------- |
| **Core / Working**         | Pinned mission, user-approved constraints, active project state, current task scratchpad. | “Use Rust-first local tools,” current theorem goal, active repo constraints. | Always tiny, always loaded.                   |
| **Episodic**               | What happened, when, with hash-chain provenance.                                          | Tool run, failed proof, code compile, paper read, conversation event.        | Recent hot; old cold.                         |
| **Semantic Temporal KG**   | Atomic claims, entities, relations, facts with validity windows.                          | “Paper X supports claim Y under assumption A.”                               | Hot for facts.                                |
| **Concept / Schema**       | Self-organizing A-MEM/Zettelkasten cards and higher-order mental models.                  | “Gauge invariance kernel,” “Rust borrow-checking failure modes.”             | Hot for planning/explaining.                  |
| **Procedural Skill**       | Executable, tested, reusable workflows and code skills.                                   | `extract_arxiv_equations`, `run_benchmark_smoke_test`.                       | Hot for task execution.                       |
| **Resource / Vault**       | Immutable external pointers, files, datasets, papers, repos, secrets references.          | DOI, arXiv ID, Git commit, dataset hash, file path.                          | Exact lookup.                                 |
| **Belief / Meta / Eval**   | Current stance, lessons, failures, retrieval evals, self-correction records.              | “Belief P contested,” “Prior answer failed because stale source.”            | High priority for avoiding repeated mistakes. |
| **Formal Science Overlay** | Specialized objects for math/science: equations, theorems, units, experiments.            | Lean theorem, unit-checked equation, experiment result.                      | Hot for science/math queries.                 |

Key invariant:

```text
Observation != claim.
Claim != belief.
Belief != truth.
Summary != source.
Skill != trusted until tested.
Old facts are never overwritten; they are superseded.
```

# 5. The proof-preserving compression pyramid

This is the most important upgrade over ordinary RAG.

```mermaid
flowchart BT
    L0[L0 Raw Artifacts<br/>files, PDFs, screenshots, logs, transcripts]
    L1[L1 Episodes<br/>hash-chained event records]
    L2[L2 Observations<br/>dated compressed statements]
    L3[L3 Atomic Cards<br/>claims, equations, entities, skills, resources]
    L4[L4 Concept Kernels<br/>cited topic summaries + links]
    L5[L5 Domain Maps<br/>theory graph, prerequisite map, open questions]
    L6[L6 Context Packs<br/>task-specific retrieval output]

    L0 --> L1 --> L2 --> L3 --> L4 --> L5 --> L6
    L6 -. can drill down .-> L3
    L4 -. cites .-> L2
    L3 -. provenance .-> L1
    L1 -. content hash .-> L0
```

Compression rules:

1. **Raw source is not stuffed into memory.** It is stored by content hash or external pointer.
2. **Episodes are immutable.** They are the audit trail of what happened.
3. **Observations are compressed but dated.** They are the fast autobiographical layer.
4. **Atomic cards are source-linked.** Claims, equations, skills, and resources each cite episodes or spans.
5. **Concept kernels are lossy but proof-preserving.** A kernel may summarize 10,000 events into 1,000 tokens, but every factual line points back to evidence.
6. **Context packs are temporary compiled artifacts.** They are never treated as truth unless explicitly promoted through the write pipeline.
7. **Compression is bidirectional.** The agent can use the kernel for speed, then drill down into claims and raw source when exact evidence matters.

This lets the system store millions of memories while usually recalling a few compressed concept kernels, exact claims, counterexamples, and skills.

# 8. Storage and indexing design

## Canonical truth

The canonical store is:

```text
append-only segmented ledger
+ hash-chained episodes
+ mutation receipts
+ object snapshots
+ rebuildable projections
```

Every mutation produces a receipt:


The projections are rebuildable:

```text
Tantivy indexes
HNSW/PQ vector indexes
temporal interval indexes
graph CSR snapshots
Roaring bitmap filters
skill signature indexes
equation/theorem indexes
concept-neighbor caches
context-pack caches
```

## Hot / warm / cold tiers

| Tier     | Contents                                                                                      | Storage                                             | Recall route           |
| -------- | --------------------------------------------------------------------------------------------- | --------------------------------------------------- | ---------------------- |
| **Hot**  | Core, active project, high-utility concepts, recent episodes, trusted skills, active beliefs. | RAM + mmap + HNSW hot graph.                        | p50 ≤ 60 ms.           |
| **Warm** | Most trusted semantic cards, concept kernels, resources, equations, older useful episodes.    | Local disk, Tantivy, compressed vectors, CSR graph. | p95 ≤ 250–900 ms.      |
| **Cold** | Raw artifacts, archived episodes, low-utility duplicates, superseded facts.                   | Content-addressed files/object store.               | Exact drill-down only. |

The system should embed **concept kernels and atomic cards**, not every raw log line. For 10M memories, dense vectors must be compressed:

```text
768-d f32 vector = ~3 KB
10M raw f32 vectors = ~30 GB before graph/index overhead

Better:
- f16 hot vectors where precision matters
- product quantization for warm vectors
- no vectors for low-value raw episodes
- BM25 + entity + time buckets for cold evidence
- re-embed by priority: importance × utility × access frequency
```

# 9. Write path

```mermaid
sequenceDiagram
    participant Agent
    participant Observe
    participant Ledger
    participant Observer
    participant Extractor
    participant Validator
    participant Store
    participant Index
    participant Daemons

    Agent->>Observe: observe(event)
    Observe->>Ledger: append episode + hash-chain
    Ledger-->>Agent: fast receipt

    Ledger->>Observer: async event segmentation
    Observer->>Extractor: observations
    Extractor->>Validator: candidate memory objects
    Validator->>Validator: provenance, duplicate, contradiction, ACL
    Validator->>Store: commit mutation transaction
    Store->>Index: update BM25/vector/graph/bitmaps
    Store->>Daemons: enqueue consolidation/reflection/skill mining
```

The write path has two speeds:

## 9.1 Synchronous fast path

This happens immediately:

1. Validate privacy class.
2. Append event to ledger.
3. Hash-chain the episode.
4. Add event to recent ring buffer.
5. Extract obvious symbols, file paths, tool names, error messages with rules.
6. Return `ObserveReceipt`.

Target: p50 ≤ 2 ms.

## 9.2 Asynchronous enrichment path

This happens after the agent can continue working:

1. Segment event using topic boundaries, tool boundaries, source boundaries, prediction gaps, and corrections.
2. Compress into observations.
3. Extract candidate claims, equations, resources, entities, skills, failures, evals.
4. Link entities and resolve aliases.
5. Detect contradictions and supersessions.
6. Attach provenance and exact spans.
7. Score novelty, surprise, importance, confidence.
8. Commit memory objects.
9. Update indexes.
10. Trigger reflection or consolidation if surprise or utility is high.

Contradiction rule:

```text
If new claim conflicts with old claim in same entity/predicate/regime/time:
  - do not overwrite old claim
  - set old.valid_to = now
  - mark old as Superseded or Contested
  - add Contradicts / Supersedes edge
  - create CounterexampleCard
  - revise BeliefState if confidence crosses threshold
```

## Retrieval channels

The compiler runs channels in parallel:

1. **Core channel** — pinned active constraints.
2. **BM25 channel** — exact terms, symbols, IDs, paper titles, code paths, equation fragments.
3. **Vector channel** — semantic similarity over summaries, concept kernels, claims, skills.
4. **Entity channel** — resolve mentions to entities and fetch related memories.
5. **Temporal channel** — valid-at facts, supersessions, old/new truth.
6. **Graph activation channel** — PPR-style traversal from seeds for multi-hop reasoning.
7. **Concept channel** — A-MEM/Zettelkasten neighboring kernels and prerequisites.
8. **Procedural channel** — skills matching task signature and preconditions.
9. **Eval/counterexample channel** — prior failures, corrections, bad answers, falsified assumptions.
10. **Resource channel** — papers, datasets, repos, files, external pointers.

HippoRAG’s central insight is worth keeping: seed a graph with query concepts and run Personalized PageRank to retrieve multi-hop evidence in one step rather than repeatedly asking the LLM to search. ([arXiv][4])

## Scoring formula

```rust
pub struct RetrievalWeights {
    pub exact: f32,
    pub semantic: f32,
    pub entity: f32,
    pub graph: f32,
    pub temporal: f32,
    pub source_quality: f32,
    pub confidence: f32,
    pub utility: f32,
    pub topic_strength: f32,
    pub recency: f32,
    pub skill_reliability: f32,
    pub contradiction_penalty: f32,
    pub stale_penalty: f32,
}

pub fn score_candidate(c: &Candidate, w: &RetrievalWeights) -> f32 {
    w.exact * c.bm25_norm
        + w.semantic * c.vector_sim
        + w.entity * c.entity_overlap
        + w.graph * c.ppr_score
        + w.temporal * c.temporal_match
        + w.source_quality * c.source_quality
        + w.confidence * c.confidence
        + w.utility * c.utility
        + w.topic_strength * c.topic_strength
        + w.recency * c.recency_decay
        + w.skill_reliability * c.skill_reliability
        - w.contradiction_penalty * c.unresolved_contradiction
        - w.stale_penalty * c.staleness
}
```

## Context pack contract

A `ContextPack` must include:

```text
- why each item was included
- exact source span or citation for every factual claim
- validity window for temporal facts
- contradiction notes when known
- confidence and freshness summaries
- omitted-evidence notes when important evidence did not fit
- skill call signatures, not giant code blobs
- counterexamples before recommendations
- deterministic ordering for equal scores
```

# 11. The agent-call protocol

This is where most memory systems fail. The agent needs explicit calls, not vague “memory tools.”

## Core calls

| Call                            | Purpose                                      | When the agent should call it                                                 |
| ------------------------------- | -------------------------------------------- | ----------------------------------------------------------------------------- |
| `observe(event)`                | Record what happened.                        | Every meaningful tool result, correction, source read, code run, answer eval. |
| `recall(query)`                 | Compile a context pack.                      | Before answering, planning, coding, proving, debugging, or comparing.         |
| `recall_at(query, time)`        | Historical truth query.                      | “What did we believe last week?” or “What was true in version X?”             |
| `feedback(pack_id, outcome)`    | Tell memory what worked.                     | After task success/failure, answer correction, verified/falsified claim.      |
| `reflect(scope)`                | Generate lessons from surprise/failure.      | After failed proof, failed benchmark, contradiction, repeated error.          |
| `consolidate(scope)`            | Compress and link memories.                  | Task boundaries, idle cycles, end of day, high-volume ingestion.              |
| `build_concept_kernel(topic)`   | Create/update a compact cited topic card.    | Strong recurring topics, scientific domains, project areas.                   |
| `mine_skills(scope)`            | Distill repeated successful workflows.       | Repeated command sequences, code patterns, research workflows.                |
| `verify(memory_id)`             | Check equation/theorem/skill/source.         | Before trusting math, science claims, or executable procedures.               |
| `rehearse(topic_or_n)`          | Strengthen weakening important knowledge.    | Idle cycles or before mission-critical work.                                  |
| `explain(memory_id or pack_id)` | Audit why something exists or was retrieved. | Debugging memory quality.                                                     |
| `archive(policy)`               | Remove low-value memories from hot indexes.  | Maintenance, compaction, memory bloat.                                        |
| `rebuild_indexes()`             | Rebuild projections from canonical truth.    | Corruption, migration, benchmark runs.                                        |

## Agent policy

The agent should follow this loop:

```text
Before acting:
  recall(task, intent, evidence_level, budget)

During acting:
  observe(tool calls/results/errors)
  observe(source spans read)

After acting:
  feedback(pack_id, success/failure/verified/falsified)
  if surprise or failure: reflect(scope)
  if repeated success: mine_skills(scope)
  if topic is recurring: build_concept_kernel(topic)
```

# 12. Consolidation, reflection, and compression daemons

## 12.1 Observer daemon

Turns streams into dense observations:

```text
raw event:
  "cargo test failed with E0277 in trait bound..."

observation:
  "2026-05-12: In repo X, `cargo test` failed because type T lacked trait Bound B in module M. The attempted fix was Y; outcome failed."
```

## 12.2 Concept compiler

Creates concept kernels from clusters:

```text
Trigger:
  - topic strength > threshold
  - repeated retrieval
  - high graph centrality
  - multiple related claims/equations/resources
  - user marks as important

Output:
  - definition
  - prerequisites
  - canonical examples
  - equations
  - claims
  - contradictions
  - resources
  - skills
  - misconceptions
  - open questions
```

A-MEM’s memory evolution idea should be constrained here: update existing concept kernels, but only through mutation receipts and evidence-linked diffs, never silent rewriting. A-MEM’s key idea is dynamic note construction, indexing, linking, and historical memory evolution. ([arXiv][5])

## 12.3 Belief reviser

Belief is a current stance over evidence:

```text
Observation: Source S reports result R.
Claim: R supports P under assumptions A.
Belief: The agent accepts P with confidence C because C1..Cn support it
        and K1..Km contradict it.
```

The reviser computes:

```text
confidence_new =
  calibrated_prior
  + support_strength
  - contradiction_strength
  + source_quality
  + formal_verification_bonus
  - staleness_penalty
```

## 12.4 Strengthener

Use FSRS-like memory variables:

```text
retrievability R = probability memory can be recalled now
stability S = time for retrievability to decay to target
difficulty D = how hard it is to strengthen
```

FSRS uses retrievability, stability, and difficulty as core memory variables, and Anki documents FSRS as learning memory patterns to schedule reviews more efficiently than older approaches. ([Anki FAQs][6])

In NOESIS-RS, FSRS is not just for flashcards. It schedules:

```text
- important concepts that are weakening
- high-utility skills not used recently
- fragile theorem dependencies
- error patterns the agent tends to repeat
- project constraints likely to be forgotten
```

## 12.5 Skill miner

Borrow Voyager’s most important idea: repeated successful behavior becomes executable memory. Voyager’s architecture includes an automatic curriculum, an ever-growing skill library, and iterative prompting with environment feedback and self-verification. ([arXiv][7])

Skill mining pipeline:

```text
episodes with similar task signature
→ sequence alignment over tool calls
→ infer preconditions/postconditions
→ synthesize candidate skill
→ AST/static validation
→ sandbox execution
→ tests
→ trusted skill only if pass
```

## 12.6 Compressor / archiver

Compression actions:

```text
MERGE_DUPLICATE
SUPERSEDE_STALE
CREATE_CAPSULE
REFRESH_KERNEL
ARCHIVE_LOW_UTILITY
KEEP_COUNTEREXAMPLE_HOT
REMOVE_FROM_VECTOR_INDEX
KEEP_IN_CANONICAL_LEDGER
```

Counterexamples never fully decay. They are the “scar tissue” that prevents repeated mistakes.

# 19. 0–100 rubric and score

| Axis                                        | Weight |    Score | Why                                                                                                                                                                               |
| ------------------------------------------- | -----: | -------: | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Taxonomic human-likeness**                |      8 |  **8.0** | Core, working, episodic, semantic, procedural, resource, belief/meta/eval, formal science. Cleaner than six-lane-only designs while avoiding over-fragmentation.                  |
| **Temporal epistemic integrity**            |      9 |  **8.8** | Validity windows, supersession, belief/fact separation, contested state, historical recall. Small deduction because contradiction detection is probabilistic before verification. |
| **Provenance and auditability**             |      7 |  **7.0** | Hash-chained ledger, source spans, receipts, signatures, deterministic replay.                                                                                                    |

# 20. Why this is the best implementable design

NOESIS-RS is the strongest design because it does not confuse memory with retrieval. It separates:

```text
what happened       → episodes
what was observed   → observations
what is claimed     → claims
what is believed    → beliefs
what is useful      → utility / eval memory
what is executable  → skills
what is compressed  → kernels / capsules
what is source      → resources / artifacts
what is true now    → temporal graph
what was true then  → historical validity
```

It is human-like where humans are strong: episodic memory, semantic abstraction, procedural learning, reflection, strengthening, and conceptual linking.

It is superhuman where humans are weak: exact provenance, cryptographic audit, parallel retrieval, formal verification hooks, million-scale review scheduling, temporal truth windows, deterministic replay, and source-preserving compression.

The final architecture is:

```text
MIRIX taxonomy
+ Hindsight epistemic separation
+ Graphiti temporal truth
+ A-MEM concept evolution
+ HippoRAG associative graph activation
+ Mem0/Mastra low-token recall discipline
+ Voyager executable skill compounding
+ FSRS/topic strengthening
+ Rust-native storage, indexing, replay, and eval
+ proof-preserving compression pyramid
```

That is the memory system I would build.
