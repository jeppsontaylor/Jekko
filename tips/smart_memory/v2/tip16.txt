I unpacked the bundle and treated the uploaded **CLAUDE_MEMSPEC**, **CODEX_MEMSPEC**, all 13 tips, and the pasted ANTIGRAVITY/Gemini spec as inputs. The strongest recurring conclusion is clear: **the best system is not a vector database, not a graph database, and not an LLM self-editing notes file. It is a Rust memory compiler whose source of truth is typed, temporal, provenance-backed memory, and whose job is to compile the smallest correct context pack for the agent at inference time.**

External cross-checks support the same direction: MIRIX is valuable for its six-lane memory taxonomy; Graphiti/Zep for temporal context graphs; A-MEM for Zettelkasten-style evolving memory networks; HippoRAG for Personalized PageRank graph activation; Mem0 for low-latency production retrieval; Hindsight for fact/experience/belief separation; Mastra Observational Memory for compressed stable continuity; Voyager for executable skill compounding; and Memory-R1/MemOS/LightMem for adaptive policy, lifecycle, and efficient consolidation. ([arXiv][1])

# MNEMOS‑Σ: Superhuman Rust Cognitive Memory Compiler

## 1. Executive thesis

**MNEMOS‑Σ** is the best architecture I would build for a long-running research, coding, math, science, and personal-agent memory system.

Its core rule:

> **The memory system is not the model’s context window. The memory system is a compiler. It transforms raw experience into typed, cited, temporal, executable, compressed knowledge, then compiles a minimal context pack for the agent at the moment of need.**

This system improves on the CLAUDE, CODEX, and ANTIGRAVITY specs in one important way: it makes **compression, feedback, and organization first-class state transitions**, not background nice-to-haves. It does not merely store memories. It learns which memories are useful, which are stale, which are contradicted, which should become concepts, which should become executable skills, and which should be archived out of hot recall.

I score this design **96.8 / 100** on the full rubric below. I would not honestly score any pre-benchmark architecture 100, because extraction quality, learned policy quality, and real latency under production workloads must be measured. But this is the strongest design I can justify from the evidence and from the uploaded specifications.

# 2. Design principles

## 2.1 Hard rules

1. **Never store raw chat as primary memory.** Raw events go into an immutable ledger. Usable memory is extracted, typed, cited, scored, and linked.

2. **Never let embeddings be the source of truth.** Dense vectors are retrieval hints. Truth lives in typed memory objects, temporal edges, source spans, confidence, and provenance.

3. **Never destructively overwrite facts.** Changing facts create `valid_to`, `supersedes`, and `contradicts` links.

4. **Separate observation, claim, belief, and skill.**
   “A paper said X,” “X is a claim,” “the agent believes X,” and “the agent can execute a method based on X” are different memory objects.

5. **Every promoted scientific or mathematical memory must have evidence.** Claims, equations, theorem statements, experiment results, and datasets require source spans, hashes, and confidence.

6. **Procedural memory must execute.** Untested snippets are not skills. Skills need preconditions, code, tests, success/failure history, and versioning.

7. **Compression must preserve rehydration.** Summaries can be lossy, but every compressed object must point back to original episodes, source spans, artifacts, and receipts.

8. **Recall returns a context pack, not a list of chunks.** The pack is bounded, cited, contradiction-aware, stale-aware, and task-shaped.

9. **The agent must give feedback.** Memory improves when the agent reports what helped, what failed, what was ignored, and what was falsified.

10. **Hot-path recall must be local and deterministic.** LLM calls can help extraction, reflection, and consolidation, but baseline recall must work without online LLM inference.

# 5. Memory lanes

MNEMOS‑Σ uses **12 logical lanes**, grouped into four families.

## 5.1 Runtime lanes

| Lane            | Purpose                                                        | Hot-path behavior                    |
| --------------- | -------------------------------------------------------------- | ------------------------------------ |
| **Working**     | Current task state, goals, scratch, active hypothesis          | Always available, tiny, short-lived  |
| **Core**        | Stable mission, user-approved preferences, project constraints | Always considered, very small budget |
| **Observation** | Compressed dated statements from raw episodes                  | Fast continuity and recent context   |

## 5.2 Knowledge lanes

| Lane              | Purpose                               | Examples                                                         |
| ----------------- | ------------------------------------- | ---------------------------------------------------------------- |
| **Episodic**      | What happened                         | Tool run, failed build, paper read, experiment attempted         |
| **Semantic**      | Atomic claims, definitions, relations | “Paper A proposes method B”; “X depends on Y”                    |
| **Belief**        | Current stance over evidence          | “I currently accept claim C at confidence 0.72”                  |
| **Concept**       | Synthesized topic/kernel notes        | “Gauge invariance,” “Temporal graph memory,” “Bell inequalities” |
| **Resource**      | External anchors                      | arXiv PDF, DOI, repo commit, dataset checksum                    |
| **Source/Social** | Credibility and attribution           | Author, institution, source quality, citation network            |

## 5.3 Action lanes

| Lane              | Purpose                              | Examples                                                            |
| ----------------- | ------------------------------------ | ------------------------------------------------------------------- |
| **Procedural**    | Executable, tested skills            | Search arXiv, extract equations, run benchmark, debug Cargo failure |
| **Eval**          | What worked and failed               | Bad answer record, retrieval miss, skill test result                |
| **Metacognitive** | Lessons about reasoning and learning | “Do not trust dimensionless derivations without unit check”         |

## 5.4 Formal science/math lanes

These are not separate physical stores; they are typed semantic objects with special validators.

| Object                 | Required structure                                                                   |
| ---------------------- | ------------------------------------------------------------------------------------ |
| **EquationCard**       | LaTeX, normalized symbolic form, variables, units, assumptions, derivation links     |
| **TheoremCard**        | Statement, assumptions, proof sketch, formal proof optional, dependency DAG          |
| **ExperimentCard**     | Setup, dataset, code commit, parameters, result, uncertainty, reproducibility status |
| **DatasetCard**        | URI, checksum, schema, license, version, splits, known caveats                       |
| **CounterexampleCard** | Falsified memory, evidence, regime of failure, repair status                         |
| **QuestionCard**       | Open question, why it matters, blockers, next search actions                         |

```rust
impl SkillCard {
    pub fn reliability(&self) -> f32 {
        let n = self.success_count + self.failure_count;
        if n == 0 { return 0.5; }
        self.success_count as f32 / n as f32
    }

        self.tests.iter().any(|t| t.last_status == TestStatus::Passed)
            && self.reliability() >= 0.70
    }
```

## 7.4 Hot/warm/cold tiers

| Tier                        | Contents                                                      | Query latency goal | Storage form                            |
| --------------------------- | ------------------------------------------------------------- | -----------------: | --------------------------------------- |
| **L0 Working**              | active task state                                             |              <1 ms | RAM                                     |
| **L1 Hot**                  | core, recent observations, high-utility claims, active skills |            5–50 ms | RAM/mmap + Tantivy + HNSW               |
| **L2 Warm**                 | most trusted semantic/concept/procedural memory               |          50–250 ms | mmap graph + BM25 + quantized vectors   |
| **L3 Cold**                 | old episodes, low-utility observations, archived duplicates   |     250 ms–seconds | compressed blobs + lexical sparse index |
| **L4 Vault**                | private/sensitive memory                                      |   policy-dependent | encrypted, access-controlled            |
| **L5 Parametric candidate** | stable distilled patterns                                     |       offline only | adapter/training dataset candidate      |

L5 is deliberately **not automatic**. Stable memory can be proposed for distillation into adapters or parameter-level memory, but only after strong evaluation and human approval.

# 8. Write pipeline

The write path can be slower than recall. It should be smart, cautious, and asynchronous.

```text
observe
  → segment
  → append episode
  → extract candidates
  → normalize entities
  → route by lane
  → validate
  → detect duplicates / contradictions
  → score
  → policy decision
  → commit canonical mutation
  → update projections
  → schedule consolidation
```

```text
  "Ran cargo test in sim crate; failure caused by missing Serialize impl for Foo."

observation:
  "At 2026-05-12T20:11Z, `cargo test -p sim` failed because `Foo` lacked `Serialize`."

candidate memories:
  EvalCard: failed test
  ClaimCard: sim crate currently requires Foo: Serialize
  Skill update: when serde trait errors occur, inspect derive macros
```

This follows the same spirit as Mastra’s Observer/Reflector pattern, but it is stricter: every observation is source-linked and can be rehydrated. ([Mastra][5])

## 8.3 Extractor layer

Extractors are composable:

```rust
#[async_trait::async_trait]
pub trait Extractor: Send + Sync {
    async fn extract(&self, ep: &EpisodeCard) -> anyhow::Result<Vec<CandidateMemory>>;
    fn capabilities(&self) -> ExtractorCaps;
}
```

Recommended extractors:

```text
RuleExtractor
  cheap regex/parsers for paths, code errors, equations, DOIs, arXiv IDs

LlmExtractor
  structured output for claims, lessons, concepts, open questions

EquationExtractor
  TeX + symbolic normalization + unit candidates

CodeExtractor
  AST parser for Rust/Python/SQL snippets and reusable workflow traces

PaperExtractor
  sections, claims, equations, datasets, methods, citations

ToolTraceExtractor
  turns repeated successful tool runs into skill candidates
```

## 8.4 Validation gates

A candidate memory can only become `Trusted` if it passes its lane validator.

| Memory type | Required validation                                                           |
| ----------- | ----------------------------------------------------------------------------- |
| Claim       | Source span, subject/predicate/object, confidence, support/contradiction scan |
| Belief      | Evidence links and rationale; cannot be asserted as fact                      |
| Equation    | Source, variables, unit constraints when possible                             |
| Theorem     | Assumptions, dependencies, proof status                                       |
| Skill       | Code reference, preconditions, tests, success episode                         |
| Resource    | URI/path, checksum when possible, version metadata                            |
| Eval        | Outcome, task id, affected memory ids                                         |
| Concept     | Underlying claims/resources/questions; no unsupported facts                   |

## 8.5 Contradiction handling

No overwrites.

```text
old edge:
  subject = user_project
  relation = uses_database
  object = SQLite
  valid_from = 2026-04-01
  valid_to = null

new evidence:
  "We migrated to Postgres on 2026-05-12."

mutation:
  old.valid_to = 2026-05-12
  new edge:
    subject = user_project
    relation = uses_database
    object = Postgres
    valid_from = 2026-05-12
    supersedes = [old_edge]
  contradiction/supersession receipt is appended
```

## 8.6 Memory manager policy

Policy decisions:

```rust
pub enum MemoryDecision {
    Add,
    Noop { reason: String },
    Update { target: MemoryId },
    Link { source: MemoryId, target: MemoryId, relation: Relation },
    Supersede { old: MemoryId },
    Archive { target: MemoryId },
    Reject { reason: String },
}
```

Rule fallback:

```text
if candidate lacks provenance and lane requires provenance:
    Reject or ProposedOnly

elif duplicate similarity > 0.97 and same source:
    Noop

elif duplicate similarity > 0.88:
    Update existing object with new evidence

elif contradicts trusted live fact:
    Supersede old edge + Add new evidence + create Counterexample if needed

elif source_quality low and confidence low:
    ProposedOnly

elif repeated successful trajectory >= 3 and has test:
    Promote skill candidate

else:
    Add
```

Learned policy:

```text
features:
  novelty
  nearest_duplicate_similarity
  source_quality
  confidence
  surprise
  utility_prediction
  recurrence
  contradiction_pressure
  task_importance
  lane
  topic_mastery_gap
  user_feedback
  downstream_success

actions:
  ADD / UPDATE / LINK / SUPERSEDE / NOOP / ARCHIVE / REJECT
```

Memory‑R1 supports the premise that memory management can be learned from outcome rewards rather than hand-written heuristics alone. MNEMOS‑Σ uses that idea only after a safe rule-based policy is already working. ([arXiv][9])

# 9. Read pipeline: the retrieval compiler

The read path must be fast, local, deterministic, and task-shaped.

```text
recall(query)
  → classify query intent
  → parse entities / symbols / time scope
  → retrieve in parallel
  → graph activation
  → temporal filter
  → rank fusion
  → rerank by task weights
  → compile context pack
  → log access
```

## 9.1 Query intent classes

```rust
pub enum RecallIntent {
    AnswerFact,
    ExplainConcept,
    MathDerivation,
    ScientificClaimCheck,
    LiteratureReview,
    Coding,
    Debugging,
    Planning,
    SkillLookup,
    HistoricalTimeline,
    ContradictionAudit,
    BeliefAudit,
    OpenQuestionSearch,
    Mixed,
}
```

## 9.2 Parallel retrieval channels

| Channel            | Finds                                        | Why it matters                |
| ------------------ | -------------------------------------------- | ----------------------------- |
| Core/working       | mission, constraints, active goals           | Always relevant               |
| BM25/Tantivy       | exact terms, paths, symbols, equations, IDs  | Essential for math/code       |
| Vector             | semantic similarity                          | Handles paraphrase            |
| Entity lookup      | known people, papers, datasets, code modules | Prevents embedding drift      |
| Temporal graph     | facts valid now or at time T                 | Prevents stale recall         |
| Concept graph      | higher-level synthesized notes               | Enables knowledge compounding |
| PPR / HippoRAG     | multi-hop associated evidence                | Needed for science/literature |
| Procedural index   | executable skills                            | Turns knowledge into action   |
| Eval/failure index | prior mistakes and bad answers               | Prevents repeated failures    |
| Resource index     | exact papers, files, commits, datasets       | Rehydrate evidence            |

## 9.3 Scoring formula

The final score is not “cosine similarity.” It is a task-specific fusion.

```text
score =
  w_exact      * exact_match
+ w_bm25       * bm25_norm
+ w_vec        * vector_similarity
+ w_entity     * entity_overlap
+ w_graph      * ppr_activation
+ w_temporal   * temporal_validity
+ w_source     * source_quality
+ w_conf       * confidence
+ w_utility    * utility
+ w_success    * retrieval_success
+ w_topic      * topic_strength
+ w_recency    * recency_if_relevant
+ w_skill      * skill_reliability_if_needed
- w_stale      * stale_penalty
- w_conflict   * unresolved_contradiction_penalty
- w_acl        * access_policy_penalty
```

## 9.4 Retrieval profiles

| Intent            | Heavy weights                                                  | Must include warnings                                |
| ----------------- | -------------------------------------------------------------- | ---------------------------------------------------- |
| Math derivation   | exact symbols, equations, prerequisites, proof dependencies    | missing assumptions, unit mismatch, unverified proof |
| Scientific claim  | source quality, support/contradiction, temporal validity       | contested evidence, stale source, low confidence     |
| Coding            | exact paths, skill reliability, recent evals, failed attempts  | stale environment, untested workaround               |
| Debugging         | exact errors, recent episodes, counterexamples, skill failures | repeated failed strategy                             |
| Literature review | resources, citation graph, claim clusters, concept graph       | omitted evidence, contradictory clusters             |
| Planning          | core constraints, open questions, skills, topic gaps           | unresolved blockers                                  |
| Historical        | temporal validity at requested time                            | current fact differs from historical fact            |

## 9.5 Context pack contract

A `ContextPack` is what the LLM receives.

```rust
pub struct ContextPack {
    pub pack_id: Uuid,
    pub query: String,
    pub intent: RecallIntent,
    pub generated_at: DateTime<Utc>,
    pub token_budget: u32,
    pub token_estimate: u32,
    pub items: Vec<PackItem>,
    pub warnings: Vec<PackWarning>,
    pub omitted: Vec<OmittedEvidence>,
    pub callable_skills: Vec<SkillHandle>,
    pub citations: Vec<RenderedCitation>,
    pub trace_id: Uuid,
}

pub struct PackItem {
    pub memory_id: MemoryId,
    pub lane: MemoryLane,
    pub rendered: String,
    pub why_included: String,
    pub score: f32,
    pub confidence: f32,
    pub provenance_refs: Vec<Provenance>,
}
```

Invariants:

```text
1. token_estimate <= token_budget
2. factual items include citations/source spans
3. stale facts are excluded unless historical recall requested
4. contradictions are explicitly labeled
5. known counterexamples are favored over flattering summaries
6. skills are represented by signatures unless code is explicitly needed
7. deterministic ordering for fixed memory state + query seed
8. every pack has a retrieval trace
```

Query:

```text
"Why did our Rust graph traversal benchmark fail last time, and what skill should I use now?"
```

```text

[Skill]
```

# 10. Knowledge compression over time

This is the part that matters most for millions of memories.

MNEMOS‑Σ uses a **compression ladder**, not one summarizer.

## 10.1 Compression ladder

```text
L0 Raw episode
  exact event, file, source, tool trace, hash

L1 Observation
  dense dated statement, 1–3 sentences

L2 Atomic memory
  claim, equation, skill, resource, eval, question

L3 Cluster summary
  repeated episodes grouped by task/topic/entity

L4 Concept kernel
  compact Zettelkasten card with claims, equations, examples, failures, skills

L5 Topic capsule
  high-level compressed map of a topic, with source links and open questions

L6 Curriculum / mastery map
  prerequisites, next reviews, weak areas, high-utility actions

L7 Parametric candidate
  optional offline training/adaptation proposal, never automatic
```

## 10.2 Compression rules

| Rule                                                                | Reason                                                     |
| ------------------------------------------------------------------- | ---------------------------------------------------------- |
| Raw episodes are never destroyed unless explicitly pruned by policy | Prevents false summaries from becoming unrecoverable truth |
| Summaries must point to supporting memories                         | Enables rehydration                                        |
| Concept cards cannot invent unsupported claims                      | Prevents hallucinated synthesis                            |
| Belief summaries must list support and contradiction                | Avoids false certainty                                     |
| Skills cannot be compressed into prose only                         | Procedural memory must remain executable                   |
| Counterexamples do not decay normally                               | Prevents repeated mistakes                                 |
| Compression is evaluated by retrieval tests                         | Summaries must improve recall, not just save space         |

## 10.3 Compression daemons

| Daemon                 |               Cadence | Job                                                           |
| ---------------------- | --------------------: | ------------------------------------------------------------- |
| **Observer**           |             real-time | Compress raw events into observations                         |
| **Deduper**            |               minutes | Merge near-duplicates, preserve evidence links                |
| **ConceptSynth**       |           idle/hourly | Create/update concept kernels                                 |
| **BeliefSynth**        |          hourly/daily | Update evidence-weighted beliefs                              |
| **SkillDistiller**     | after successful runs | Promote repeated workflows to skills                          |
| **TopicCompressor**    |               nightly | Build topic capsules and mastery maps                         |
| **ArchiveManager**     |          daily/weekly | Move low-utility cold memories out of hot indexes             |
| **RehearsalScheduler** |            continuous | Queue weakening important memories                            |
| **PolicyTrainer**      |               offline | Train memory manager from outcomes                            |
| **Verifier**           |             scheduled | Check source links, hashes, skill tests, theorem dependencies |

## 10.4 Topic strengthening formula

```text
decayed_strength =
  previous_strength * exp(-hours_since_update / half_life_hours)

reinforcement =
  0.18 * recurrence
+ 0.20 * task_utility
+ 0.16 * retrieval_success
+ 0.12 * source_quality
+ 0.10 * novelty
+ 0.10 * graph_neighbor_boost
+ 0.08 * user_or_agent_priority
+ 0.06 * successful_skill_links
- 0.18 * unresolved_contradiction_pressure
- 0.08 * stale_source_pressure

new_strength = clamp(decayed_strength + reinforcement, 0.0, 1.0)
```

Topic state drives:

```text
- what becomes a concept card
- what enters the review queue
- what gets archived
- what gets expanded during recall
- what open questions are proposed
- what skills are suggested
```

# 11. Agent-facing calls

This directly answers: **what can the agent call to help organize and compress memory over time?**

The agent should not silently edit memory. It should call audited tools.

## 11.1 Core calls

```rust
#[async_trait::async_trait]
pub trait MemoryEngine {
    async fn observe(&self, event: EventEnvelope) -> Result<MutationReceipt>;

    async fn recall(&self, query: RecallQuery) -> Result<ContextPack>;

    async fn feedback(&self, pack_id: Uuid, outcome: Outcome) -> Result<MutationReceipt>;

    async fn explain(&self, target: ExplainTarget) -> Result<Explanation>;

    async fn consolidate(&self, job: ConsolidationJob) -> Result<ConsolidationReport>;

    async fn verify(&self, scope: VerificationScope) -> Result<VerificationReport>;
}
```

## 11.2 Organization calls

| Call                                        | What it does                                           |
| ------------------------------------------- | ------------------------------------------------------ |
| `memory.link(source, target, relation)`     | Adds a typed edge between memories                     |
| `memory.supersede(old, new, reason)`        | Closes old validity and installs new memory            |
| `memory.mark_contradiction(a, b, evidence)` | Records conflict without deleting either side          |
| `memory.promote(memory_id)`                 | Proposed → Trusted after validation                    |
| `memory.archive(memory_id, reason)`         | Removes from hot recall but preserves canonical object |
| `memory.merge_duplicates(ids)`              | Merges redundant memories while preserving provenance  |
| `memory.create_concept(topic)`              | Forces a ConceptCard synthesis                         |
| `memory.update_belief(proposition)`         | Recomputes evidence-weighted stance                    |
| `memory.create_question(topic, blocker)`    | Stores an unresolved research target                   |
| `memory.register_resource(uri, checksum)`   | Pins external source/version                           |

## 11.3 Compression calls

| Call                                     | What it does                                             |
| ---------------------------------------- | -------------------------------------------------------- |
| `memory.compress_episode_cluster(scope)` | Turns many episodes into a cited cluster summary         |
| `memory.compress_topic(topic_id)`        | Builds or refreshes a topic capsule                      |
| `memory.compact_index(lane)`             | Rebuilds vector/BM25/graph projections                   |
| `memory.archive_cold(scope)`             | Moves low-utility items out of hot recall                |
| `memory.rehydrate(memory_id)`            | Pulls raw supporting episodes/sources back into view     |
| `memory.prune_rejected(scope)`           | Deletes only rejected, unlinked noise after policy delay |
| `memory.rebuild_from_ledger()`           | Reconstructs projections deterministically               |

## 11.4 Learning calls

| Call                                                       | What it does                                            |
| ---------------------------------------------------------- | ------------------------------------------------------- |
| `memory.feedback(pack_id, TaskSuccess { used })`           | Strengthens useful memories and skills                  |
| `memory.feedback(pack_id, TaskFailure { used, reason })`   | Lowers utility, creates EvalCard, schedules reflection  |
| `memory.feedback(pack_id, Falsified { memory, evidence })` | Creates CounterexampleCard and contradiction            |
| `memory.rehearse_next(topic, n)`                           | Retrieves weakening high-value memories for self-review |
| `memory.self_quiz(topic)`                                  | Generates questions, attempts recall, updates mastery   |
| `memory.distill_skill(episode_ids)`                        | Converts repeated success into a skill candidate        |
| `memory.test_skill(skill_id)`                              | Runs skill tests and updates reliability                |
| `memory.repair_skill(skill_id, failure_episode)`           | Produces a new skill version after failure              |
| `memory.train_policy(scope)`                               | Offline Memory‑R1-style policy training                 |

## 11.5 Science/math calls

| Call                                         | What it does                                              |
| -------------------------------------------- | --------------------------------------------------------- |
| `memory.extract_equations(resource_id)`      | Extracts equations, variables, units, source spans        |
| `memory.check_units(equation_id)`            | Performs dimensional analysis                             |
| `memory.verify_theorem(theorem_id)`          | Runs optional Lean/Coq/Isabelle check                     |
| `memory.trace_derivation(equation_id)`       | Retrieves derivation dependency chain                     |
| `memory.compare_claims(claim_ids)`           | Builds support/contradiction matrix                       |
| `memory.find_counterexamples(claim_id)`      | Searches known falsifying evidence                        |
| `memory.replicate_experiment(experiment_id)` | Calls linked skill/resource workflow                      |
| `memory.update_literature(topic)`            | Creates targeted search tasks for stale/high-value topics |

# 14. Engineering performance targets

These are engineering targets, not claims already measured.

| Operation                    | Target at 10M memories/node | Notes                                 |
| ---------------------------- | --------------------------: | ------------------------------------- |
| `observe()` ledger ack       |                  p50 < 5 ms | Just append + receipt                 |
| Candidate extraction         |                       async | May use LLM/rules; not hot path       |
| Hot recall, no graph PPR     |   p50 < 50 ms, p95 < 200 ms | Core + BM25 + vector + filters        |
| Recall with graph activation |  p50 < 120 ms, p95 < 750 ms | PPR/cache bounded                     |
| Context pack compile         |                 p50 < 25 ms | After candidates fetched              |
| Skill lookup                 |                 p50 < 40 ms | Signature + embedding + preconditions |
| Exact equation query         |                 p50 < 75 ms | Tantivy equation tokenizer            |
| Temporal entity timeline     |                p50 < 100 ms | interval index + adjacency            |
| Consolidation cycle          |                     offline | bounded by idle budget                |
| Full index rebuild           |                hours at 10M | deterministic from ledger             |

## 14.1 Scaling tactics

| Problem                     | Solution                                                                               |
| --------------------------- | -------------------------------------------------------------------------------------- |
| 10M embeddings are large    | Use per-lane vectors, int8/f16 quantization, hot/cold ANN, and concept compression     |
| Graph traversal can explode | Limit PPR seeds, edge types, depth, temporal validity, and cache neighborhoods         |
| Memory bloat                | Archive low-utility cold objects; keep source in ledger; remove from hot indexes       |
| Context bloat               | Compile packs with strict token budget and lane-specific renderers                     |
| LLM extraction cost         | Use rules first, batch extraction, async background, small local models where possible |
| Duplicate memories          | Duplicate detector + semantic merge + source evidence aggregation                      |
| Stale facts                 | Temporal validity filter and contradiction pressure                                    |
| Poor retrieval              | Eval memory and feedback update retrieval weights                                      |
| Science/math exactness      | BM25 symbol tokenizer + equation schema + formal hooks                                 |
| Procedural hallucination    | Skills need tests and success episodes                                                 |

# 15. Testing and validation

## 15.1 Unit tests

```text
- serde round-trip for every schema version
- hash chain cannot be tampered with
- MemoryObject invariants per lane
- invalid lifecycle transitions rejected
- facts cannot be overwritten destructively
- temporal `valid_to` closes correctly
- access policies redact secret items
- context pack cannot exceed token budget
- contradiction warnings render correctly
- equation unit checks catch dimension mismatch
- skill cannot become Trusted without tests
```

## 15.2 Property tests

```text
Given random memory objects:
  - Rebuild from ledger yields same object ids and edge ids
  - Rejected memories never appear in normal recall
  - Secret memories never appear without role
  - Stale facts never appear unless historical query
  - ContextPack item order is deterministic
  - Every factual PackItem has provenance
```

## 15.3 Integration tests

```text
1. Ingest a paper.
   Assert: ResourceCard + ClaimCards + EquationCards have source spans.

2. Ingest a contradiction.
   Assert: old fact valid_to is closed; new fact supersedes old; both remain queryable historically.

3. Ingest a failed code run.
   Assert: EvalCard is retrievable for future debugging.

4. Repeat a successful workflow three times.
   Assert: SkillCandidate created; after tests pass, SkillCard becomes Trusted.

5. Query exact equation symbol.
   Assert: BM25/equation tokenizer finds it even if vector search misses.

6. Query a multi-hop scientific question.
   Assert: PPR retrieves connected paper/method/dataset/claim chain.

7. Run consolidation.
   Assert: ConceptCard has links to claims, equations, resources, failures, open questions.

8. Run feedback.
   Assert: used memories strengthen; ignored ones do not; failed ones create eval records.
```


# 16. Example: scientific memory lifecycle

Input:

```text
A paper says: "In the low-temperature regime, model M violates assumption A used by method B."
```

Write output:

```text
EpisodeCard
  source = paper PDF page/span
  summary = read paper section about low-temperature limitation

ClaimCard
  statement = "Model M violates assumption A in low-temperature regime."
  support = paper span
  valid_time = paper publication time / source time
  confidence = 0.82

ConceptCard update
  topic = "Method B validity"
  add known limitation
  link to Model M, assumption A, low-temperature regime

BeliefState update
  proposition = "Method B is reliable for Model M"
  stance = Contested
  confidence decreases
  rationale cites old support and new contradiction

QuestionCard
  "Find alternative methods for Model M in low-temperature regime."

Skill suggestion
  If there is a simulation skill for Method B, mark precondition:
  "Do not use when low-temperature regime violates assumption A."
```

Recall output:

```text
User asks:
"Can we use method B for model M at low temperature?"

ContextPack:
- Claim: Model M violates assumption A in low-temperature regime.
- Belief: Method B is contested for Model M under low-temperature conditions.
- Counterexample: Prior simulation failed under this regime.
- Skill warning: simulation_method_b has precondition excluding this regime.
- Open question: find alternative method.
```

This is where MNEMOS‑Σ becomes superhuman: it does not just remember a sentence. It updates a concept, a belief, a skill precondition, and a future research question.

# 17. Example: procedural compounding lifecycle

Input episodes:

```text
1. Agent searched arXiv for "temporal graph agent memory".
2. Agent filtered papers by date/source.
3. Agent extracted claims/equations.
4. Agent created cited summary.
5. Agent repeated this workflow successfully five times.
```

Distilled skill:

```rust
SkillCard {
    name: "literature_scan_v3",
    task_signature: "literature_scan(topic: String, date_range: DateRange) -> LiteratureReport",
    preconditions: [
        "network access available",
        "topic string is specific",
        "source whitelist configured"
    ],
    postconditions: [
        "report contains ResourceCards",
        "every ClaimCard has provenance",
        "open contradictions listed"
    ],
    tests: [
        "fixture_arxiv_scan_temporal_graph",
        "fixture_no_unsupported_claims",
        "fixture_context_pack_budget"
    ],
    success_count: 5,
    failure_count: 0,
    version: 3,
}
```

When the agent later receives:

```text
"Find the latest memory systems for scientific agents and compare them."
```

The context pack includes:

```text
Callable skill:
  literature_scan_v3(topic, date_range)
Why included:
  high reliability, exact task match, prior success on memory-system literature
```

The agent does not relearn the workflow. It reuses the skill.

# 18. Mermaid state machine

```mermaid
stateDiagram-v2
    [*] --> Proposed
    Proposed --> Trusted: validated + provenance
    Proposed --> Rejected: invalid/noisy/unsafe
    Trusted --> Consolidated: stable + useful + repeated
    Trusted --> Contradicted: new conflicting evidence
    Trusted --> Superseded: newer valid fact
    Trusted --> Archived: low utility + low retrievability
    Consolidated --> Superseded: newer stronger evidence
    Consolidated --> Contradicted: falsified
    Consolidated --> Archived: cold but preserved
    Contradicted --> Archived: resolved/old
    Superseded --> Archived: no longer current
    Rejected --> [*]: hard prune after policy window
    Archived --> Trusted: rehydrated + reused
```

# 19. Security and privacy

## 19.1 Hard guarantees

```text
- Vault memories are encrypted at rest.
- AccessPolicy is checked before rendering ContextPack items.
- Secret memories are never included in ordinary packs.
- Tool outputs are untrusted input.
- Memory content is never executed.
- Skill code executes only in sandboxed test/run contexts.
- Every memory mutation creates a receipt.
- Every promoted memory can be traced to source evidence.
```

```text
mnemos verify --hash-chain
```

| Axis                                             | Weight |    Score | Rationale                                                                                                                                                                                       |
| ------------------------------------------------ | -----: | -------: | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **1. Cognitive taxonomy and human-likeness**     |     10 | **10.0** | Covers working, core, episodic, semantic, procedural, resource, belief, concept, eval, metacognitive, formal science, and source credibility memory.                                            |
| **2. Temporal and epistemic grounding**          |     12 | **11.8** | Facts have validity windows, transaction time, supersession, contradiction links, and belief/fact separation. Slight deduction because automatic contradiction detection will never be perfect. |
| **3. Provenance, audit, and trust**              |      8 |  **7.8** | Hash chains, source spans, mutation receipts, source quality, and access policy. Deduction only for the inherent difficulty of extracting exact spans from all modalities.                      |
| **4. Retrieval quality and multi-hop recall**    |     12 | **11.5** | BM25 + vector + entity + temporal graph + PPR + skill/eval retrieval + context compiler. Deduction because empirical reranking weights require tuning.                                          |
| **5. Latency and scale to millions**             |     12 | **11.2** | Rust hot path, Tantivy, Qdrant/local HNSW, mmap graph, bitmaps, hot/warm/cold tiers, quantization. Deduction until measured at 10M–100M memories.                                               |
| **6. Knowledge compression and context economy** |     10 |  **9.6** | Compression ladder, concept kernels, topic capsules, archive manager, stable observations, rehydration. Deduction because compression quality depends on extractors and eval loops.             |
| **7. Adaptive learning and feedback**            |      9 |  **8.4** | Outcome feedback, utility scoring, topic strength, FSRS, policy learning, eval memory. Deduction because learned memory policies need high-quality training data.                               |
| **8. Procedural skill compounding**              |      8 |  **7.8** | Executable skills, AST validation, tests, repair, preconditions, reliability. Deduction because sandboxing and cross-language skills are complex.                                               |

| **11. Buildability and operational robustness**  |      5 |  **4.5** | Staged rollout, deterministic rebuilds, tests, crash recovery. Deduction because the full architecture is large and must be built in phases.                                                    |

## Why this beats the uploaded specs

The CLAUDE spec is strong on cognitive lanes, FSRS, formal science, and Rust structure. The CODEX spec is strong on canonical stores, context packs, topic strengthening, and buildability. The ANTIGRAVITY spec is strong on the six-lane synthesis, temporal graph, and retrieval compiler. MNEMOS‑Σ keeps those strengths but adds:

```text
1. stricter compression ladder
2. first-class eval/failure memory
3. explicit source/social credibility lane
4. deeper science/math object model
5. hot/warm/cold scaling design
6. concrete agent calls for organization/compression
7. policy gates that separate safe rules from learned memory management
8. context pack invariants as a formal contract
9. deterministic rebuild from ledger as a core requirement
10. skill memory with tests, repair, and reliability scoring
```
