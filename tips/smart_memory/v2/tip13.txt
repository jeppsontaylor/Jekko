I unpacked and studied the uploaded archive. It contains `CLAUDE_MEMSPEC.md`, `CODEX_MEMSPEC.md`, `tip1.txt` through `tip13.txt`, and a small `target/jankurai/update/state.json`. There was no separate `GEMINI_MEMSPEC.md` in the archive, so I treated the pasted **ANTIGRAVITY Cognitive Memory Compiler** as the Google/Gemini-style layout.

The strongest design pattern across all of it is clear:

> **Do not build a memory database. Build a memory compiler.**
> The source of truth is an auditable, typed, temporally grounded memory substrate. Retrieval indexes are only projections. The agent receives compiled context packs, not raw memory dumps.

Below is the best design I would build.

# HYPERMNEMOS: a Rust Cognitive Memory Compiler for superhuman agent memory

**HYPERMNEMOS** is a local-first, Rust-native, multi-lane memory operating system for AI agents that need to learn science, mathematics, code, workflows, and long-running project context over months or years.

It combines the best parts of the systems you studied:

* **MIRIX** for typed cognitive lanes: core, episodic, semantic, procedural, resource, vault. MIRIX explicitly organizes memory into six specialized memory types and coordinates updates/retrieval through memory managers. ([arXiv][1])
* **Graphiti/Zep** for temporal knowledge graphs, where facts evolve over time and include validity windows rather than destructive overwrite. ([GitHub][2])
* **A-MEM** for Zettelkasten-style self-organizing notes, dynamic linking, and memory evolution. ([GitHub][3])
* **Hindsight** for separating facts, experiences, entity summaries, and evolving beliefs, with retain/recall/reflect operations. ([arXiv][4])
* **Nemori** for event segmentation and predict-calibrate learning from prediction gaps instead of blind chunk extraction. ([arXiv][5])
* **Mastra Observational Memory** for stable, compressed observation logs and background reflection, keeping prompt prefixes cacheable and compact. ([Mastra][6])
* **HippoRAG** for associative graph activation using LLM-extracted graphs plus Personalized PageRank, inspired by hippocampal indexing theory. ([arXiv][7])
* **Voyager** for procedural skill memory: executable skills accumulated through exploration and feedback. ([GitHub][8])
* **MemGPT/Letta** for virtual-context management: the context window is RAM, external memory is disk, and the agent uses tools to page memory in and out. ([GitHub][9])
* **MemOS / MemoryOS** for treating memory as an operating-system resource with lifecycle, scheduling, versioning, and governance. ([GitHub][10])

The result is not “human-like” in the weak sense. It is **human-like in structure**, but **superhuman in execution**: cryptographic provenance, exact citations, millions of memories, parallel retrieval, formal validation, temporal truth, and self-improving context compilation.

# 1. Design objective

HYPERMNEMOS must support:

```text
1. Fast recall over millions of memories.
2. Real-time learning from conversations, papers, code runs, experiments, and tool output.
3. Self-organization of concepts over time.
4. Compression without losing provenance.
5. Temporal truth: what was believed, when, why, and what superseded it.
6. Scientific rigor: claims, equations, proofs, assumptions, datasets, scripts, citations.
7. Procedural learning: reusable skills, tests, workflows, patches, failed attempts.
8. Context-window protection: agents receive compiled context packs, not raw memory.
9. Git/audit integration: memory mutations are reviewable, replayable, and reversible.
10. Rust-first implementation, with optional external scientific tools only where justified.
```

The central abstraction is:

```text
raw experience → canonical episode ledger → typed memory objects
→ temporal/entity/concept/skill indexes → context compiler
→ bounded cited context pack → agent action
→ feedback/eval → memory delta → validation → commit
```

# 3. What I would change from the uploaded specs

The CLAUDE, CODEX, and ANTIGRAVITY specs are already strong. I would keep most of their instincts, but make these changes:

| Area                | Keep                                      | Change                                                                                                                                           |
| ------------------- | ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Typed lanes**     | Keep MIRIX/MNEMOS-style typed lanes.      | Add stricter separation between **claim**, **belief**, **concept**, **equation**, **skill**, and **eval** memory.                                |
| **Canonical truth** | Keep append-only receipts and provenance. | Make **MemoryDelta** the only mutation primitive. Agents never directly edit trusted memory.                                                     |
| **Storage**         | Keep Rust-first local stack.              | Use canonical ledger + object store as truth; all indexes are rebuildable projections. Do not put millions of generated cards directly into Git. |
| **Temporal graph**  | Keep Graphiti-style validity windows.     | Add bitemporal semantics: `valid_time` and `transaction_time`.                                                                                   |
| **Compression**     | Keep Observer/Reflector.                  | Add multi-level compression: observation → atomic memory → concept kernel → community summary → curriculum map.                                  |
| **Retrieval**       | Keep BM25 + vector + graph + PPR.         | Make hot recall fully deterministic and LLM-free. LLM reranking is optional slow path.                                                           |
| **Science/math**    | Keep equation/theorem stores.             | Add formal validation states, unit checks, assumption graphs, derivation dependency DAGs, and public-data manifests.                             |
| **Skills**          | Keep Voyager/Memp pattern.                | Require executable tests before promotion to trusted procedural memory.                                                                          |
| **Scoring**         | Keep 0–100 rubric.                        | Do not claim perfect 100. The best buildable design scores around **95.5/100** because extraction and formalization remain imperfect.            |

# 5. Memory lanes

I would use **13 logical lanes**, all sharing one canonical `MemoryObject` envelope.

## 5.1 Core memory

Small, pinned, and intentionally boring.

Contains:

```text
agent identity
project mission
non-negotiable constraints
current research stance
active user-approved preferences
current repo conventions
active safety/privacy rules
```

Core memory is not a giant wiki. It should usually stay under 1,000–2,000 tokens.

## 5.2 Working memory

Current task state.

Contains:

```text
active plan
open subtasks
scratch hypotheses
current files
active experiment
temporary assumptions
```

Working memory is allowed to expire.

## 5.3 Episodic memory

Immutable record of what happened.

Contains:

```text
conversation events
tool calls
tool results
search runs
paper ingestions
code executions
test failures
debugging sessions
human corrections
```

Episodic memory is the autobiography of the agent.

## 5.4 Observational memory

Compressed dense observations derived from episodic memory.

Example:

```text
2026-05-12T03:14Z: The agent tried to parse arXiv source tarballs.
The PDF parser missed equations, so the plan changed to prefer LaTeX source when available.
```

This is the Mastra-style “subconscious” stream: compact, stable, and cheap to retrieve.

## 5.5 Semantic claim memory

Atomic, source-grounded claims.

Example:

```text
Claim:
  "Method X assumes the weak-field approximation."

Support:
  paper:...
  page:...
  span:...
  confidence: 0.78
  status: trusted
```

A claim is not a belief. A claim is an extracted proposition with evidence.

## 5.6 Temporal fact graph

Entity-relation facts with validity windows.

Example:

```text
subject: "preferred parser"
relation: "is"
object: "GROBID"
valid_from: 2026-05-01
valid_to: 2026-05-12

subject: "preferred parser"
relation: "is"
object: "LaTeX-source-first pipeline"
valid_from: 2026-05-12
valid_to: null
supersedes: old_edge_id
```

Old facts are not deleted. They become historical.

## 5.7 Concept memory

Self-organizing Zettelkasten cards.

A concept card links:

```text
definitions
claims
equations
papers
datasets
skills
failed attempts
analogies
open questions
contradictions
```

This is where the agent gets “smarter” over time. It does not just accumulate facts. It builds reusable conceptual structures.

## 5.8 Belief memory

Current epistemic stance.

Example:

```text
Belief:
  "The most promising parser pipeline is LaTeX-source-first, PDF fallback, then human review."

stance: supported
confidence: 0.84
supporting_claims: [...]
contradicting_claims: [...]
changed_by: mutation_receipt_id
rationale: ...
```

Beliefs are allowed to change. Facts and observations remain as evidence.

## 5.9 Procedural memory

Executable skills, scripts, and workflows.

Example:

```text
Skill:
  "extract_equations_from_arxiv_source"
language: Rust
preconditions:
  - arXiv source tarball exists
  - TeX files detected
tests:
  - fixture paper with known equations
success_rate: 0.91
status: trusted
```

No untested script becomes trusted procedural memory.

## 5.10 Resource memory

Pointers to external reality.

Contains:

```text
papers
DOIs
arXiv IDs
datasets
GitHub repos
commit hashes
local files
data manifests
APIs
model cards
licenses
checksums
```

Resource memory is how the system can rehydrate context without storing huge data in Git.

## 5.11 Equation and theorem memory

Specialized formal memory.

Contains:

```text
LaTeX
normalized symbolic form
variables
units
assumptions
regime of validity
derivation graph
proof sketch
formal proof status
dependent claims
```

Science and math need this. Plain text embeddings are not enough.

## 5.12 Eval and failure memory

Records of what worked and what failed.

Contains:

```text
bad retrievals
bad answers
failed code runs
failed proofs
failed data downloads
corrections
benchmark scores
regressions
```

This is crucial. An agent that forgets failure repeats failure.

## 5.13 Topic strength memory

A machine-scale version of spaced repetition.

Each topic has:

```text
strength
retrievability
utility
recency
novelty
source quality
contradiction pressure
review due date
open question count
```

FSRS-style scheduling is useful here because it models memory with difficulty, stability, and retrievability; HYPERMNEMOS adapts that from human flashcards to agent topic mastery. ([GitHub][11])

# 6. Mermaid memory lifecycle

```mermaid
stateDiagram-v2
    [*] --> Observed
    Observed --> Proposed: extractor creates candidate
    Proposed --> Rejected: invalid / duplicate / unsafe
    Proposed --> Trusted: provenance + validation pass
    Trusted --> Consolidated: stable + reinforced
    Trusted --> Contested: contradiction discovered
    Contested --> Revised: belief update or scoped truth
    Revised --> Trusted
    Trusted --> Superseded: newer valid fact replaces old
    Superseded --> Historical: retained for temporal queries
    Consolidated --> Archived: low utility but preserved
    Archived --> Rehearsed: scheduled review restores utility
    Rehearsed --> Trusted
```

# 7. Canonical storage principle

The system has exactly one source of truth:

```text
canonical ledger + canonical object store + mutation receipts
```

Everything else is rebuildable:

```text
BM25 indexes
vector indexes
graph adjacency caches
concept maps
topic summaries
context-pack caches
skill lookup indexes
```

This prevents the core failure mode of agent memory systems: an embedding index or wiki summary becomes treated as truth.

# 9. Storage stack

For the Rust-first baseline:

| Layer                  | Recommended implementation                    | Why                                                                                                                                                     |
| ---------------------- | --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Canonical object store | `redb` or SQLite via `rusqlite`               | `redb` keeps the pure-Rust baseline; SQLite is battle-tested if purity is less important.                                                               |
| Mutation ledger        | append-only JSONL/Zstd + BLAKE3 chain         | Replayable, auditable, portable.                                                                                                                        |
| Lexical retrieval      | Tantivy                                       | Tantivy is a Rust full-text search library similar to Lucene. ([GitHub][12])                                                                            |
| Vector retrieval       | local HNSW or Qdrant adapter                  | Qdrant is an open-source vector search engine written in Rust and supports scalable vector retrieval. ([Qdrant][13])                                    |
| Graph retrieval        | Rust adjacency tables + optional Kuzu adapter | Kuzu is an embedded property graph database with Cypher, vector, and full-text support; use it as optional adapter, not canonical truth. ([GitHub][14]) |
| Embeddings             | Candle local models + external trait fallback | Candle is a Rust ML framework with GPU support and native Transformer model loading. ([GitHub][15])                                                     |
| Git provenance         | `git2`                                        | Human review and commit history for curated artifacts.                                                                                                  |
| Compression            | `zstd`, `roaring`, `rkyv`, `bincode`          | Compact immutable segments and fast ID sets.                                                                                                            |

The important rule:

```text
canonical store must be boring and durable;
indexes can be fast, fancy, and rebuildable.
```

# 12. Write path

```mermaid
sequenceDiagram
    participant Agent
    participant Gateway
    participant Ledger
    participant Router
    participant Validator
    participant Store
    participant Indexes
    participant Sleep

    Agent->>Gateway: observe(event)
    Gateway->>Gateway: segment event
    Gateway->>Ledger: append episode
    Ledger-->>Gateway: episode_id + hash

    Gateway->>Router: extract candidate memories
    Router->>Router: classify lanes
    Router->>Validator: propose MemoryDelta

    Validator->>Validator: schema check
    Validator->>Validator: provenance check
    Validator->>Validator: privacy check
    Validator->>Validator: duplicate check
    Validator->>Validator: contradiction check
    Validator->>Validator: skill test check

    alt accepted
        Validator->>Store: commit canonical objects
        Store->>Ledger: append mutation receipt
        Store->>Indexes: update projections
        Store->>Sleep: enqueue consolidation jobs
    else rejected
        Validator->>Ledger: append rejected/proposed receipt
    end
```

## Write algorithm

```text
observe(event):
  1. Validate privacy class.
  2. Segment by task boundary, source boundary, tool boundary, correction, surprise, elapsed time.
  3. Append immutable Episode first.
  4. Compress Episode into Observations.
  5. Extract candidate claims, entities, resources, equations, skills, beliefs, evals.
  6. Normalize entities and predicates.
  7. Attach source spans and hashes.
  8. Route candidates to lanes.
  9. Build MemoryDelta.
 10. Run validators.
 11. Commit accepted delta.
 12. Update indexes asynchronously.
 13. Schedule consolidation if novelty/surprise/importance is high.
```

The hot write path should not wait for every expensive model call. It should do:

```text
synchronous:
  append episode
  write receipt
  cheap observation
  enqueue extraction

asynchronous:
  claim extraction
  concept linking
  contradiction detection
  skill distillation
  formal validation
  graph community update
```

This gives real-time learning without blocking the agent.

## Reranking score

```text
score(item, query) =
    w_exact       * exact_match
  + w_bm25        * lexical_score
  + w_vector      * semantic_score
  + w_graph       * graph_activation
  + w_entity      * entity_match
  + w_temporal    * temporal_validity
  + w_source      * source_quality
  + w_confidence  * confidence
  + w_utility     * utility
  + w_topic       * topic_strength
```

| Debugging                   | episodic failures, exact error messages, prior patches, environment resources |

| Planning                    | core constraints, active topic state, skills, resource manifests              |
| Historical query            | temporal graph with requested valid time, superseded facts included           |
| “What should I learn next?” | topic strength, open questions, low-confidence beliefs, missing sources       |

# 18. Retrieval performance strategy

For millions of memories, the system must avoid doing expensive work on the hot path.

## Hot path: target behavior

```text
p50 recall: 50–150 ms warm
p95 recall: <750 ms local
p99 recall: <1500 ms local
```

For exact queries, target lower:

```text
exact ID/symbol/file lookup: <20 ms
core memory lookup: <5 ms
BM25 top-k: <50 ms
vector top-k warm: <100 ms
graph neighborhood bounded: <100 ms
```

These are engineering targets, not guaranteed benchmark claims.

## Why it can scale

1. **Not all memory is vectorized.**
   Raw episodes and low-salience observations do not all get expensive embeddings.

2. **Shard by lane and domain.**
   Claims, equations, skills, resources, and evals have separate indexes.

3. **Use exact lookup first.**
   For science and code, exact symbols, IDs, paths, and equations often beat semantic search.

4. **Bound graph traversal.**
   PPR runs only on a seed subgraph, not the whole graph.

5. **Use caches.**
   Cache query embeddings, entity resolutions, graph activation neighborhoods, and recent ContextPacks.

6. **Use tiered storage.**
   Hot memories are memory-mapped or cached; cold memories remain compressed.

7. **Use retrieval profiles.**
   A debugging query should not search all literature-review clusters. A theorem query should not retrieve random conversation notes.

8. **Use deterministic context packing.**
   Avoid slow LLM reranking unless explicitly enabled.

# 19. Knowledge compression policy

The system should compress aggressively, but never confuse summaries with evidence.

```text
Evidence is durable.
Summaries are replaceable.
Indexes are rebuildable.
Beliefs are revisable.
Skills are testable.
```

## Compression operations

| Operation                   | What it does                                                                             |
| --------------------------- | ---------------------------------------------------------------------------------------- |
| **Observation compression** | Converts raw event streams into dense dated observations.                                |
| **Duplicate merge**         | Merges semantically identical memories while preserving all provenance links.            |
| **Concept kernel build**    | Creates compact topic cards from linked claims, equations, papers, skills, and failures. |
| **Belief synthesis**        | Converts evidence clusters into current belief states.                                   |
| **Community rollup**        | Builds higher-level summaries from graph communities.                                    |
| **Skill distillation**      | Converts repeated successful traces into reusable procedures.                            |
| **Failure abstraction**     | Converts repeated failures into counterexample and metacognitive lessons.                |
| **Index compaction**        | Rebuilds and optimizes BM25/vector/graph projections.                                    |
| **Review scheduling**       | Promotes weakening but important topics into review queues.                              |

## Compression invariants

```text
1. Never delete source evidence because a summary exists.
2. Never promote unsupported generated text to trusted memory.
3. Never overwrite a fact with a different fact.
4. Never store a belief as a fact.
5. Never trust a skill without test evidence.
6. Never include stale facts without a freshness warning.
7. Never allow private/vault memories into unprivileged context packs.
```

# 21. Procedural memory design

Procedural memory should be treated like code, not prose.

```rust
pub struct SkillTest {
    pub name: String,
    pub command: Option<String>,
    pub fixture_refs: Vec<String>,
    pub expected_output_hash: Option<String>,
    pub timeout_ms: u64,
}

impl SkillCard {
    pub fn reliability(&self) -> f32 {
        let total = self.success_count + self.failure_count;
        if total == 0 {
            return 0.0;
        }
        self.success_count as f32 / total as f32
    }

    pub fn can_be_trusted(&self) -> bool {
        self.success_count >= 2
            && self.reliability() >= 0.80
            && !self.tests.is_empty()
    }
}
```

Promotion rule:

```text
draft skill
  → tested skill
  → trusted skill
  → degraded skill
  → retired skill
```

A skill can be trusted only if:

```text
it has preconditions
it has at least one successful episode
it has repeatable tests
its failure modes are recorded
it has no unresolved safety/privacy violation
```

# 22. Belief revision

Belief revision is separate from fact storage.

```text
Observation:
  "Test failed because ndarray trait bounds were wrong."

Claim:
  "This Rust code path fails with ndarray version X under condition Y."

Belief:
  "Prefer nalgebra over ndarray for this specific tensor operation in this repo."

Skill:
  "Use nalgebra-based matrix construction for simulation scaffold."

Counterexample:
  "Do not use previous ndarray implementation for sparse tensor case."
```

This separation is the difference between memory that compounds and memory that pollutes itself.

# 25. Evaluation design

Use public memory benchmarks, but do not rely only on them. LoCoMo is designed to evaluate long-term conversational memory across QA, summarization, and multimodal dialogue generation; LongMemEval evaluates long-term memory abilities including information extraction, multi-session reasoning, temporal reasoning, knowledge updates, and abstention. ([Snap Research][17])

Also use RAG-style metrics for retrieval and answer faithfulness; RAGAS defines context precision as whether relevant chunks are ranked above irrelevant ones, and faithfulness as whether answer claims are supported by retrieved context. ([Ragas][18])

## Required eval suites

```text
1. LoCoMo
2. LongMemEval
3. Internal Science-Memory-Bench
4. Internal Math/Equation Recall Bench
5. Internal Code/Skill Replay Bench
6. Contradiction and Supersession Bench
7. ContextPack Faithfulness Bench
8. Privacy Redaction Bench
9. Crash-Recovery Bench
10. 1M / 10M Load Bench
```

# 30. The most important implementation rules

```text
1. Do not make vectors the memory.
2. Do not make summaries the source of truth.
3. Do not let agents directly edit trusted memory.
4. Do not overwrite facts.
5. Do not collapse beliefs into facts.
6. Do not trust untested skills.
7. Do not put raw datasets in Git.
8. Do not require an LLM on the hot recall path.
9. Do not retrieve without provenance.
10. Do not compile context without warnings for staleness and contradictions.
```

# 31. Example: scientific learning loop

```text
User asks agent to study a new physics topic.

1. Agent calls ingest_paper(arxiv_uri).
2. Ledger records resource and parser state.
3. Extractor creates claims, equations, assumptions, datasets, code links.
4. Validator marks weak extractions as Proposed.
5. Temporal graph links claims to papers and equations.
6. Concept daemon updates the topic card.
7. Belief daemon updates current stance.
8. Topic strength sees missing public data and code.
9. Agent recalls topic gaps.
10. Agent searches for datasets and code repos.
11. Dataset manifests and download scripts are added.
12. Skill daemon creates reusable ingestion workflow.
13. Eval daemon tests whether recall improved.
14. Git export commits curated cards.
```

The system gets smarter because:

```text
it remembers sources,
organizes concepts,
tests procedures,
tracks failures,
detects contradictions,
and improves retrieval policy from feedback.
```
