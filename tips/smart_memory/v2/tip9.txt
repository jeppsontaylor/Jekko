I unpacked the uploaded `smart_memory.tar(8).gz` and studied the accessible files: `CLAUDE_MEMSPEC.md`, `CODEX_MEMSPEC.md`, `tip1.txt` through `tip13.txt`, and the pasted ANTIGRAVITY/Google-style CMC spec. I did **not** find a separate `GEMINI_MEMSPEC.md` in the tarball, so I treated the pasted ANTIGRAVITY CMC document as the Gemini/Google-style design input.

The verdict is strong:

# OpenQG Mnemosyne: a Rust Cognitive Memory Compiler

The best possible design is **not** an LLM-wiki, not a vector database, not a plain graph database, and not a raw transcript archive.

It is a **Cognitive Memory Compiler**:

> **Raw experience goes in. Typed, temporal, source-backed, compressed, queryable, executable memory comes out.**

The core rule:

> **Git and the append-only ledger hold truth. Databases are projections. The wiki is a view. The LLM context window receives only a compiled, budgeted, cited ContextPack.**

The uploaded specs converge on the same thesis: MIRIX-style typed cognitive lanes, Hindsight-style separation of experience/fact/belief, Graphiti-style temporal validity, A-MEM-style self-organizing concept notes, Mem0-style fast retrieval, HippoRAG-style associative graph activation, Mastra-style observation compression, Voyager-style executable skills, Memory-R1/ReasoningBank-style learning from outcomes, and a Git/receipt system for auditability.

That synthesis is also consistent with the strongest current public work: MIRIX explicitly separates Core, Episodic, Semantic, Procedural, Resource Memory, and Knowledge Vault memory types; Graphiti/Zep uses temporal context graphs with fact validity windows and provenance; A-MEM uses Zettelkasten-like dynamic linking and evolution; HippoRAG combines LLM extraction, knowledge graphs, and Personalized PageRank for multi-hop retrieval; Hindsight separates facts, experiences, entity summaries, and evolving beliefs; Mem0 focuses on extracting, consolidating, and retrieving salient long-term memory; and Voyager shows why procedural memory must be executable, tested, and reusable rather than merely summarized. ([arXiv][1])

I would build **OpenQG Mnemosyne** as the memory substrate for scientific agents.

# 1. Final design in one sentence

**OpenQG Mnemosyne is a Rust-native, append-only, provenance-first, temporal, typed, multi-index, self-consolidating memory compiler that turns conversations, papers, code runs, searches, equations, datasets, failures, and agent outcomes into durable knowledge objects and compact task-specific ContextPacks.**

It has four absolute invariants:

1. **No fact without provenance.**
2. **No belief pretending to be a fact.**
3. **No context stuffing; only compiled ContextPacks.**
4. **No procedural memory unless it can run, test, and produce receipts.**

---

# 2. Why this beats a wiki, vector DB, or graph-only system

A wiki is good for humans but too ambiguous for agents. A vector DB is good for fuzzy recall but terrible for time, contradictions, symbols, exact paper IDs, equations, and provenance. A graph DB is good for relations but not enough for fast semantic recall, procedural skills, hierarchical summaries, or task outcomes.

The best architecture is a **compiler** with multiple projections:

```text
Canonical truth:
  append-only ledger + typed cards + source receipts

Fast projections:
  BM25 index + vector index + temporal graph + concept summaries + skill registry

Human views:
  generated wiki pages, maps, reviews, dashboards

Agent views:
  small, cited, task-specific ContextPacks
```

GraphRAG and RAPTOR are especially important for corpus-level compression: GraphRAG builds entity graphs and community summaries for local/global corpus questions, while RAPTOR recursively clusters and summarizes chunks into a retrieval tree so queries can retrieve across abstraction levels rather than only raw chunks. ([arXiv][2])

# 4. Memory lanes

I would use **14 lanes**, not the 6-lane CMC exactly and not the 12-lane Claude spec exactly. The 6-lane version is elegant, but for OpenQG-level scientific work, math, source management, counterexamples, and evals deserve first-class treatment.

| Lane                       | Purpose                                                             |               Hot path? | Canonical object examples                 |
| -------------------------- | ------------------------------------------------------------------- | ----------------------: | ----------------------------------------- |
| **Core**                   | Stable mission, user/project constraints, identity, non-negotiables |             Always tiny | `CoreBlock`                               |
| **Working**                | Active task state, scratch hypotheses, current plan                 |             Always tiny | `WorkingFrame`                            |
| **Sensory / Input Buffer** | Raw recent inputs before extraction                                 |                      No | `InputFrame`                              |
| **Episodic**               | What happened: search run, tool run, code failure, reading session  |               Sometimes | `Episode`, `Observation`                  |
| **Semantic**               | Claims, facts, definitions, relationships                           |                     Yes | `ClaimCard`, `FactEdge`                   |
| **Belief**                 | Current stance over evidence, with support/contradiction            |        Yes for judgment | `BeliefState`                             |
| **Concept**                | Zettelkasten-style concept kernels and topic maps                   |                     Yes | `ConceptCard`                             |
| **Formal Math**            | Equations, theorem dependencies, proof sketches, units              |    Yes for science/math | `EquationCard`, `TheoremCard`             |
| **Resource**               | Papers, datasets, repos, APIs, tools, file manifests                |                     Yes | `ResourceCard`, `DatasetCard`, `RepoCard` |
| **Procedural**             | Executable, tested, reusable skills and workflows                   |          Yes for action | `SkillCard`                               |
| **Counterexample**         | Known failures, contradictions, bad assumptions, falsified beliefs  |       Always considered | `CounterexampleCard`                      |
| **Evaluation**             | Retrieval outcomes, answer errors, benchmark results, tests         | Yes for self-correction | `EvalMemory`                              |
| **Metacognitive**          | Lessons about how to reason, search, code, verify                   |               Sometimes | `LessonCard`, `StrategyCard`              |
| **Vault**                  | Secrets/private data; never committed to project memory             |            Permissioned | `VaultPointer`                            |

This goes beyond MIRIX by splitting beliefs, formal math, counterexamples, and evals into dedicated lanes. MIRIX’s six-lane taxonomy is the best base, but OpenQG needs more epistemic machinery because scientific memory is not just personal memory. ([arXiv][1])

# 8. The write path

The write path must be fast enough for realtime work, but strict enough not to poison memory.

```mermaid
sequenceDiagram
    participant Agent
    participant Observer
    participant Ledger
    participant Extractor
    participant Validator
    participant Store
    participant Indexes
    participant Daemons

    Agent->>Observer: observe(event)
    Observer->>Ledger: append Episode + hash receipt
    Ledger-->>Agent: fast ack if async mode

    Observer->>Extractor: segment + extract candidates
    Extractor->>Validator: claims/equations/skills/resources
    Validator->>Validator: provenance, duplicate, contradiction, schema checks
    Validator->>Store: atomic MemoryMutation
    Store->>Ledger: append mutation receipt
    Store->>Indexes: update BM25/vector/graph/temporal projections
    Store->>Daemons: schedule reflection/consolidation if surprise high
```

Write pipeline:

```text
observe
→ segment
→ append episode
→ extract candidate memories
→ normalize entities / units / predicates
→ classify lane
→ validate provenance
→ detect duplicates
→ detect contradictions
→ assign scores
→ apply memory policy
→ commit canonical object
→ update projections
→ emit receipt
→ schedule consolidation
```

## Realtime rule

The first write is fast:

```text
observe(event) must synchronously persist the Episode and return a receipt.
Extraction, linking, embeddings, graph mutation, and consolidation may run async.
```

This prevents losing experience while keeping the agent responsive.

# 9. The read path: ContextPack compiler

The read path is where most memory systems fail. The agent should not “search memory” directly. It should call a compiler that produces a **ContextPack**.

```mermaid
flowchart TD
    Q[Query / Task] --> C[Intent Classifier]
    C --> R1[Core Blocks]
    C --> R2[BM25 Exact Search]
    C --> R3[Dense Vector Search]
    C --> R4[Entity Resolver]
    C --> R5[Temporal Graph PPR]
    C --> R6[Concept Summary Forest]
    C --> R7[Skill Registry]
    C --> R8[Counterexample Search]
    C --> R9[Eval / Failure Search]

    R1 --> F[RRF Fusion]
    R2 --> F
    R3 --> F
    R4 --> F
    R5 --> F
    R6 --> F
    R7 --> F
    R8 --> F
    R9 --> F

    F --> T[Temporal + Status Filter]
    T --> L[Learned Reranker]
    L --> B[Budgeted Context Packing]
    B --> P[ContextPack]
```

Retrieval channels:

```text
1. Core memory lookup
2. BM25 / Tantivy exact lexical search
3. Dense vector HNSW search
4. Entity and alias search
5. Temporal graph traversal
6. Personalized PageRank from seed entities
7. Concept-kernel / summary-tree retrieval
8. Procedural skill matching
9. Counterexample and failure retrieval
10. Evaluation-memory retrieval
```

HippoRAG’s important contribution is associative retrieval: seed the graph from relevant entities and use Personalized PageRank-like spreading activation to recover multi-hop evidence in one retrieval pass. ([arXiv][6])

## Ranking formula

```text
score =
    0.22 * dense_similarity
  + 0.18 * bm25_score
  + 0.15 * entity_overlap
  + 0.13 * graph_activation
  + 0.09 * concept_summary_match
  + 0.07 * temporal_relevance
  + 0.06 * source_quality
  + 0.05 * utility
  + 0.03 * confidence
  + 0.02 * recency
  - 0.20 * contradiction_penalty_if_unresolved
  - 0.15 * stale_fact_penalty
```

Then apply:

```text
status filter:
  reject Rejected
  reject Deprecated unless requested
  reject Superseded unless historical query
  include Contradicted only with warning

mandatory inclusion:
  directly relevant counterexamples
  known failed attempts
  source-quality warnings
  active project constraints
```

## Why `redb` / `fjall`, Tantivy, Qdrant/HNSW

For an all-Rust core, I would use:

| Need                     | Primary choice                 | Why                                                     |
| ------------------------ | ------------------------------ | ------------------------------------------------------- |
| ACID object metadata     | `redb`                         | Pure Rust, embedded, ACID, MVCC, crash-safe             |
| High-volume log/blob KV  | `fjall` optional               | Rust LSM engine for heavy append/write workloads        |
| Full-text / exact search | `tantivy`                      | Rust full-text engine inspired by Lucene                |
| Vector search            | local HNSW + optional Qdrant   | local low-latency, Qdrant for sidecar/distributed scale |
| Graph                    | custom Rust temporal CSR graph | avoids C++/Python dependency and gives exact control    |
| Git                      | `gix` / gitoxide               | pure-Rust git operations where possible                 |
| Hashing/signatures       | `blake3`, `ed25519-dalek`      | fast receipts and audit chain                           |

`redb` is a pure-Rust embedded ACID key-value store; `fjall` is a Rust log-structured embeddable key-value engine; Tantivy is a Rust full-text search library; and Qdrant is a Rust vector search engine. ([GitHub][7])

## Critical design decision

Do **not** make Qdrant, Tantivy, or the graph store canonical.

They are projections:

```text
ledger + object store = truth
indexes = rebuildable acceleration
```

That is how you avoid silent memory corruption.

```mermaid
stateDiagram-v2
    [*] --> Proposed
    Proposed --> Trusted: evidence verified
    Proposed --> Rejected: invalid / duplicate / unsafe
    Trusted --> Consolidated: stable + useful + reviewed
    Trusted --> Contradicted: new evidence conflicts
    Trusted --> Superseded: newer better object replaces it
    Consolidated --> Superseded: theory/version changed

    Superseded --> Archived
    Rejected --> [*]
    Archived --> Trusted: recalled + revalidated
```

# 14. Temporal truth maintenance

The temporal graph update logic is strict:

```rust
pub async fn upsert_fact_edge(&self, new_edge: FactEdge) -> Result<MutationReceipt> {
    let live_edges = self.graph.live_edges_for(
        new_edge.subject,
        new_edge.relation.clone(),
        new_edge.valid_from,
    ).await?;

    let mut mutation = MemoryMutation::new();

    for old in live_edges {
        if old.object != new_edge.object {
            // Do not delete old truth. Close its validity window.
            mutation.close_edge_validity(
                old.id,
                new_edge.valid_from.unwrap_or_else(Utc::now),
                "superseded_by_conflicting_fact",
            );

            mutation.add_edge_relation(
                new_edge.id,
                RelationType::Contradicts,
                old.id,
                new_edge.support.clone(),
            );

            mutation.add_counterexample(CounterexampleCard {
                falsified_memory: old.id.into(),
                new_evidence: new_edge.support.clone(),
                explanation: "New fact conflicts with prior live fact".to_string(),
                severity: CounterexampleSeverity::NeedsReview,
            });
        }
    }

    mutation.insert_edge(new_edge);
    self.apply_mutation(mutation).await
}
```

This is the difference between a durable scientific memory and a hallucination amplifier.

# 15. Compression: how it stays small while learning forever

The system uses **five compression layers**.

## Layer 1: Observation compression

Raw chat/tool streams become timestamped observations.

```text
Raw:
  800 lines of tool output

Observation:
  "At 2026-05-12T03:14Z, dataset preview failed because mirror checksum
  did not match manifest. Retried with alternate mirror. New checksum matched."
```

Mastra’s Observational Memory pattern is useful here: an Observer and Reflector maintain a dense observation log that replaces raw message history as it grows. ([Mastra][9])

## Layer 2: Episode clusters

Recent related observations cluster into session summaries:

```text
"Literature search session on public datasets for X found 18 papers,
4 candidate datasets, 2 dead links, and 1 reproducible script."
```

## Layer 3: Concept kernels

Repeated facts become a structured concept card:

```text
Concept: Public benchmark datasets for topic X
- definition
- source papers
- datasets
- equations/observables
- reproducible scripts
- contradictions
- open questions
- last review
```

A-MEM’s strength is this dynamic note evolution: new memories create structured notes, link to related memories, and can update old memory representations. ([arXiv][10])

## Layer 4: Summary forest

Build RAPTOR-style recursive summaries:

```text
chunk → section summary → paper summary → topic summary → domain map
```

## Layer 5: Community reports

Build GraphRAG-style community reports:

```text
entity graph → communities → topic reports → global map
```

The ContextPack compiler can retrieve at the level needed: raw source span, claim, concept, topic summary, or global map.

# 17. Self-improvement loop

```mermaid
flowchart LR
    A[Agent acts] --> B[Outcome logged]
    B --> C[Attribution<br/>which memories helped or hurt?]
    C --> D[Update utility scores]
    D --> E[Promote useful memories]
    D --> F[Demote noisy memories]
    D --> G[Generate lessons]
    G --> H[ReasoningBank / Strategy Memory]
    H --> I[Future ContextPacks]
    I --> A
```

Memory-R1 is important because it frames memory management as a learned policy over `ADD`, `UPDATE`, `DELETE`, and `NOOP`, while ReasoningBank distills reusable reasoning strategies from successful and failed agent experiences. ([arXiv][11])

The production version should start rule-based, then train a policy.

```rust
#[derive(Clone, Debug)]
pub enum MemoryDecision {
    Add,
    Update { target: MemoryId },
    Supersede { target: MemoryId },
    LinkOnly { target: MemoryId },
    Noop,
    Reject { reason: String },
    QueueForReview,
}
```

Default rule:

```text
if no provenance: Reject or ProposedOnly
elif exact duplicate: Noop
elif same claim but stronger evidence: Update/Supersede
elif contradiction: Add + mark contradiction + review
elif reusable successful trajectory >= 3: ProposeSkill
elif low confidence but high novelty: QueueForReview
else: Add
```

# 25. Performance targets

For a local workstation with warm indexes:

| Operation                             |                            Target |
| ------------------------------------- | --------------------------------: |
| `observe(event)` async ack            |           p50 < 3 ms, p99 < 25 ms |
| `observe(event)` sync full extraction | p50 < 250 ms excluding remote LLM |
| `recall(k=32)` no graph PPR           |         p50 < 50 ms, p99 < 250 ms |
| `recall(k=64)` with graph PPR         |        p50 < 120 ms, p99 < 750 ms |
| `compile_context(4k tokens)`          |      p50 < 150 ms after retrieval |
| 10M memory cold startup               |       < 5 sec metadata, lazy mmap |
| deterministic index rebuild           |             supported from ledger |
| context budget violation              |                 impossible by API |

For 10M+ memories:

```text
Hot tier: core, current project, active topics
Warm tier: trusted semantic/concept/procedural memory
Cold tier: archived episodes, raw observations, old candidates
Projection tier: summaries and community reports
```

The system should never search all memories through one mechanism.

# 32. The most important implementation choices

## Use generated wiki pages, not wiki as source of truth

```text
cards + ledger + graph = truth
wiki = view
```

## Treat summaries as lossy indexes

```text
summary can guide retrieval
summary cannot be final evidence
```

## Always include counterexamples

If a query touches an entity with known failures, include them. This is how the agent stops repeating mistakes.

## Let agents write proposals, not truth

Agents can call:

```text
propose_claim
propose_skill
propose_belief
propose_link
```

Promotion requires validators.

## Keep context small

The system’s job is not to stuff memory into the model. The job is to make the model smarter by giving it exactly the right memory slice.

# 33. Final answer: the optimal design

Build **OpenQG Mnemosyne**:

```text
MIRIX taxonomy
+ Hindsight epistemic separation
+ Graphiti temporal graph
+ A-MEM concept evolution
+ Nemori event segmentation and prediction gaps
+ Mastra observation compression
+ Mem0 hybrid fast retrieval
+ HippoRAG associative graph activation
+ RAPTOR / GraphRAG hierarchical corpus compression
+ Voyager executable skill memory
+ Memory-R1 / ReasoningBank outcome learning
+ Git / ledger provenance
+ Rust-native hot path
+ MCP tools for agents
```

The most important mental model:

```text
The memory is not a database.
The memory is a compiler.

Input:
  experience, papers, code, data, failures, questions

Compilation:
  segment, extract, normalize, validate, link, score, compress, test

Output:
  typed cards, temporal graph edges, source receipts, concept kernels,
  skills, evals, questions, and tiny ContextPacks
```

The agent gets smarter because every cycle produces one of these durable upgrades:

```text
new evidence
new claim
new contradiction
new concept link
new belief revision
new reusable skill
new failed-attempt warning
new topic summary
new open question
new eval result
new search strategy
```

That is the core of superhuman memory: not remembering more text, but continuously compiling experience into **structured, tested, source-backed, compressed, reusable knowledge**.
