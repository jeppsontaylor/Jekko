I unpacked and studied the archive. There is no separate `GEMINI_MEMSPEC.md` in the tarball; I treated the inline **ANTIGRAVITY Cognitive Memory Compiler** as the Google/Gemini design. The strongest synthesis across CLAUDE, CODEX, the 13 tips, and the ANTIGRAVITY spec is not “better RAG.” It is a **Rust-native cognitive memory compiler**: immutable episodes as truth, typed cognitive lanes, temporal epistemic graphs, concept-kernel compression, executable procedural memory, and deterministic low-latency context-pack compilation.

The design below is the best buildable architecture I would put in front of a serious long-running research/coding/science agent.

# MNEMOS-X: Rust Cognitive Memory Compiler

**Score: 98.5 / 100**

MNEMOS-X combines the best ideas from MIRIX, Hindsight, Graphiti/Zep, A-MEM, Mem0, Mastra Observational Memory, HippoRAG, Voyager, MemoryBank/FSRS-style strengthening, GraphRAG/LightRAG corpus maps, and the CLAUDE/CODEX/ANTIGRAVITY specs into one system.

The prior art points are well aligned: MIRIX contributes the six-lane human-like taxonomy; Graphiti/Zep contributes bitemporal fact handling; A-MEM contributes dynamic Zettelkasten-style note evolution; Hindsight contributes the distinction between facts, experiences, entity summaries, and beliefs; Mem0 contributes low-latency salient memory extraction/retrieval; Mastra Observational Memory contributes observer/reflector compression and stable context; HippoRAG contributes graph activation with Personalized PageRank; Voyager contributes executable skill libraries. ([arXiv][1])

The core rule:

```text
Do not make the vector database the memory.

Make source-grounded, typed, temporal, auditable memory the truth.
Treat vectors, graphs, BM25, summaries, neural caches, and context packs as compiled projections.
```

# 1. What MNEMOS-X optimizes for

MNEMOS-X is built for agents that must learn over months or years, especially in domains where errors compound: science, math, codebases, simulations, literature review, theorem proving, research strategy, and long-running software engineering.

It must support:

| Requirement            | Design answer                                                                                                                                 |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| Human-like memory      | Separate working, episodic, semantic, procedural, concept, resource, belief, formal, and counterexample lanes.                                |
| Superhuman recall      | Parallel retrieval over exact search, vector search, entity graph, temporal graph, concept graph, skill index, eval memory, and source cache. |
| Temporal truth         | Facts have valid-time, observed-time, transaction-time, supersession, contradiction, and provenance.                                          |
| Fast hot-path recall   | No LLM required on the hot read path; deterministic compiled context packs.                                                                   |
| Millions of memories   | Sharded lane indexes, hot/warm/cold tiering, compressed concept kernels, mmap projections, per-community graph caches.                        |
| Knowledge compression  | Raw episode → observation → atomic memory → temporal graph → concept kernel → belief state → context pack.                                    |
| Procedural compounding | Skills are executable, tested, versioned, sandboxed, and linked to success/failure episodes.                                                  |
| Science/math readiness | Equation cards, unit checks, assumptions, theorem DAGs, Lean/Coq hooks, datasets, experimental counterexamples.                               |
| Self-improvement       | Feedback, eval memory, topic strengthening, rehearsal queues, learned memory policy.                                                          |
| Auditability           | Append-only mutation ledger, signed provenance, replayable projections, context-pack traces.                                                  |

# 4. The memory lanes

MNEMOS-X uses **12 lanes**. The lanes are logical views over common schemas; they are not separate databases. This matters because a single episode can generate an observation, a claim, a skill candidate, an equation, a counterexample, and a belief update.

| Lane                      | Purpose                                                                            | Stored examples                                                             | Hot-path priority                                  |
| ------------------------- | ---------------------------------------------------------------------------------- | --------------------------------------------------------------------------- | -------------------------------------------------- |
| 1. Core / Working         | Stable mission, active constraints, current hypotheses, user-approved preferences. | Project goals, “never trust uncited claims,” current proof target.          | Always considered; tiny budget.                    |
| 2. Sensory / Observation  | Compressed dated observations from streams before deeper extraction.               | “Build failed because trait bound X was missing.”                           | High for recent tasks.                             |
| 3. Episodic               | Immutable record of what happened.                                                 | Tool runs, conversations, paper reads, experiments, code attempts.          | Retrieved by time, actor, task, source.            |
| 4. Semantic Claim         | Atomic facts and claims.                                                           | “Paper P reports result R under assumptions A.”                             | High for factual questions.                        |
| 5. Belief / Epistemic     | Current agent stance over evidence, never confused with facts.                     | “I currently believe claim C with confidence 0.72 because…”                 | High for judgment, planning, scientific synthesis. |
| 6. Temporal KG            | Entity-relation edges with valid-time windows.                                     | User preference changes, project architecture changes, superseded theories. | High for “what was true when?”                     |
| 7. Concept / Schema       | Higher-order concept kernels and Zettelkasten links.                               | “Gauge invariance,” “temporal RAG failure modes,” “OpenQG strategy.”        | High for learning, explanation, planning.          |
| 8. Procedural Skill       | Executable workflows, code snippets, scripts, tool recipes.                        | “Fetch arXiv PDF,” “run benchmark,” “derive unit constraints.”              | High for coding/actions.                           |
| 9. Resource / Corpus      | Papers, datasets, repos, APIs, files, commits.                                     | DOI, arXiv ID, Git SHA, dataset manifest.                                   | High for citations and rehydration.                |
| 10. Formal Math / Science | Equations, theorem DAGs, assumptions, units, proof states, experiments.            | Equation cards, Lean snippets, unit constraints.                            | High for math/science.                             |
| 11. Eval / Counterexample | Mistakes, failed answers, falsified beliefs, benchmark traces.                     | “Prior answer omitted assumption A,” “script failed in regime R.”           | Deliberately boosted for self-correction.          |
| 12. Vault / Private       | Sensitive or encrypted user/project knowledge.                                     | API keys, private notes, secrets.                                           | Never exposed unless explicit policy allows.       |

The biggest improvement over the existing specs is the **Belief lane as a first-class lane**. Facts are not beliefs. Claims are not beliefs. Observations are not beliefs. The agent’s current stance is a computed, evidence-weighted state over claims, contradictions, source quality, time, and task context.

# 5. Compression ladder

Knowledge compression is the heart of the system. Compression must never destroy evidence; it should only create more useful projections.

```mermaid
flowchart LR
    R0[Raw event / file / tool output] --> R1[Episode<br/>immutable, hash-chained]
    R1 --> R2[Observation<br/>dense dated summary]
    R2 --> R3[Typed atoms<br/>claims, equations, entities, skills]
    R3 --> R4[Temporal graph<br/>valid-time relations]
    R3 --> R5[Concept kernel<br/>compressed topic model]
    R4 --> R6[Belief state<br/>current stance + evidence]
    R5 --> R7[Context pack<br/>task-specific compiled memory]
    R6 --> R7
    R7 --> R8[Outcome feedback]
    R8 --> R9[Strengthen / weaken / revise]
    R9 --> R3
    R9 --> R5
```

## Compression levels

| Level | Name                    |           Lossy? | Used for                              |
| ----- | ----------------------- | ---------------: | ------------------------------------- |
| L0    | Raw artifact            |               No | Rehydration, audit, re-extraction.    |
| L1    | Episode                 |               No | “What happened?” provenance.          |
| L2    | Observation             |           Mildly | Fast autobiographical continuity.     |
| L3    | Atomic memories         |       Structured | Claims, equations, skills, resources. |
| L4    | Temporal graph          |       Structured | Current/historical truth.             |
| L5    | Concept kernel          |       Yes, cited | Compact understanding of a topic.     |
| L6    | Belief state            |    Yes, explicit | Current stance and rationale.         |
| L7    | Context pack            | Yes, task-shaped | What the LLM sees now.                |
| L8    | Neural/activation cache |         Optional | Future acceleration, never canonical. |

The compression policy:

```text
Raw source is never replaced by a summary.
Summary is a cached interpretation with provenance.
Every lossy compression points back to lossless evidence.
Every concept kernel has enough citations to be audited.
Every belief state lists support and contradiction.
Every context pack records why each item was included or omitted.
```

# 6. Memory lifecycle

```mermaid
stateDiagram-v2
    [*] --> Proposed
    Proposed --> Rejected: invalid / duplicate / unsupported
    Proposed --> Trusted: grounded + validated
    Trusted --> Consolidated: stable + useful + repeated
    Consolidated --> Archived: low utility + cold
    Trusted --> Contradicted: falsifying evidence
    Consolidated --> Contradicted: falsifying evidence
    Contradicted --> Revised: new belief / corrected claim
    Trusted --> Superseded: newer valid fact
    Consolidated --> Superseded: newer valid fact
    Superseded --> Archived
    Revised --> Trusted
    Archived --> Trusted: reactivated by query / new evidence
```

No trusted memory is destructively overwritten. A change creates a mutation receipt, a new version, and a supersession/contradiction edge.

# 8. Storage architecture

The canonical store and projections are separated.

## Canonical truth

| Component       | Choice                                         | Reason                                                                             |
| --------------- | ---------------------------------------------- | ---------------------------------------------------------------------------------- |
| Mutation ledger | Append-only binary/JSONL WAL with CRC + BLAKE3 | Replayable, auditable, crash-safe.                                                 |
| Object registry | SQLite WAL or redb                             | Local-first durable metadata. SQLite is operationally proven; redb is Rust-native. |
| Large artifacts | Content-addressed file store                   | Avoid bloating DB; enables exact rehydration.                                      |
| Curated cards   | Git-backed Markdown/YAML                       | Human review, diffs, signed commits.                                               |
| Secrets         | Encrypted vault, separate from project memory  | Prevents leakage into context.                                                     |

## Rebuildable projections

| Projection             | Engine                                                                  |
| ---------------------- | ----------------------------------------------------------------------- |
| Exact/BM25 search      | Tantivy                                                                 |
| Vector ANN             | HNSW local index or Qdrant adapter                                      |
| Graph traversal        | Compressed sparse row adjacency + optional Kùzu/Qdrant/Graphiti adapter |
| Temporal queries       | `(subject, predicate, valid_from, valid_to)` interval indexes           |
| Equation/symbol search | Tantivy fielded symbol index + normalized AST index                     |
| Skill search           | Precondition index + task-signature embeddings                          |
| Topic state            | SQLite/redb materialized table                                          |
| Context-pack cache     | Hash-keyed pack store                                                   |

The key invariant:

```text
Projections may be deleted and rebuilt.
The ledger and canonical object store are the memory.
```

# 9. Write path

The write path must be **fast first, deep later**. On every event, the system commits a lossless episode immediately. More expensive extraction, linking, and reflection can run asynchronously.

```text
observe
  -> privacy gate
  -> event segmentation
  -> immutable episode append
  -> observation compression
  -> candidate extraction
  -> typed routing
  -> entity normalization
  -> grounding/provenance
  -> contradiction scan
  -> memory policy decision
  -> canonical commit
  -> projection index fanout
  -> reflection/consolidation queue
```

## Write stages

| Stage                 | What happens                                               | Hot or async           |
| --------------------- | ---------------------------------------------------------- | ---------------------- |
| 1. Privacy gate       | Reject or encrypt disallowed material.                     | Hot                    |
| 2. Segment            | Split by topic/task/tool/prediction gap/source boundary.   | Hot                    |
| 3. Append episode     | Immutable ledger entry with hash chain.                    | Hot                    |
| 4. Observation        | Dense dated observation summary.                           | Hot/fast               |
| 5. Extract candidates | Claims, equations, skills, resources, questions.           | Async if needed        |
| 6. Normalize          | Entities, aliases, units, predicates, code paths.          | Async                  |
| 7. Ground             | Attach source spans, quotes, hashes, extractor.            | Async                  |
| 8. Link               | Similarity, support, contradiction, prerequisite, analogy. | Async                  |
| 9. Decide             | ADD, UPDATE, SUPERSEDE, MERGE, NOOP, REVIEW.               | Async                  |
| 10. Commit            | Ledger + canonical store + index queue.                    | Hot for final mutation |
| 11. Reflect           | Surprise, lessons, topic updates, skill candidates.        | Background             |

## Fact update rule

```text
Never overwrite a fact to mean something else.

Old edge:
  subject = project_architecture
  predicate = uses_database
  object = sqlite
  valid_from = 2026-04-01
  valid_to = 2026-05-12
  lifecycle = superseded

New edge:
  subject = project_architecture
  predicate = uses_database
  object = redb
  valid_from = 2026-05-12
  valid_to = null
  supersedes = old_edge
```

## Belief update rule

```text
Observation: Source S reported result R under method M.
Claim: R supports proposition P under assumptions A.
Counterclaim: Source T disputes P under condition C.
Belief: Agent currently accepts P at confidence 0.64 because support outweighs contradiction, but only under assumptions A.
```

A belief can change. The evidence remains.

# 10. Read path

The read path is a **context compiler**, not a retrieval dump.

```text
recall(query)
  -> classify query intent
  -> choose recall mode and budgets
  -> retrieve from parallel channels
  -> normalize scores
  -> graph/temporal expansion
  -> reciprocal rank fusion
  -> domain reranking
  -> contradiction/freshness filtering
  -> context-pack compilation
  -> access logging
  -> feedback hook
```

## Query intents

| Intent              | Retrieval emphasis                                                     |
| ------------------- | ---------------------------------------------------------------------- |
| Fact answer         | Claim lane, temporal KG, citations, exact source.                      |
| Historical question | Temporal valid-at query, episode sequence.                             |
| Math derivation     | Equation index, theorem DAG, assumptions, units, prerequisites.        |
| Literature review   | Resource graph, claim clusters, concept kernels, citation communities. |
| Coding task         | Skill index, repo resources, recent failures, exact symbol search.     |
| Debugging           | Recent episodes, error BM25, eval/counterexample lane, known fixes.    |
| Planning            | Core constraints, beliefs, open questions, skills, topic mastery.      |
| Reflection          | Episodes, eval memory, contradictions, low-confidence beliefs.         |

## Retrieval channels

| Channel             | Purpose                                                          |
| ------------------- | ---------------------------------------------------------------- |
| Core pinned         | Always include mission and hard constraints.                     |
| BM25/Tantivy        | Exact symbols, names, equations, file paths, error messages.     |
| Vector ANN          | Semantic similarity over observations, claims, concepts, skills. |
| Entity index        | Alias-resolved entities and linked memories.                     |
| Temporal KG         | Facts valid now or at requested time.                            |
| Concept graph       | Zettelkasten links, prerequisites, analogies.                    |
| Hippo/PPR graph     | Multi-hop associative traversal.                                 |
| Skill search        | Preconditions, task signatures, prior success.                   |
| Eval/counterexample | Past mistakes, failed attempts, falsified beliefs.               |
| Resource/corpus map | Papers, datasets, repos, community summaries.                    |
| Context cache       | Reuse stable packs when memory state unchanged.                  |

## Scoring

```text
score =
  w_exact       * exact_match
+ w_vector      * semantic_similarity
+ w_entity      * entity_overlap
+ w_temporal    * valid_time_match
+ w_graph       * ppr_or_graph_score
+ w_concept     * concept_kernel_relevance
+ w_skill       * skill_reliability
+ w_source      * source_quality
+ w_conf        * confidence
+ w_utility     * utility
+ w_topic       * topic_strength
+ w_recent      * recency_decay
- w_contra      * unresolved_contradiction_pressure
- w_stale       * stale_validity_penalty
- w_budget      * token_cost_penalty
```

Weights are selected by intent. For example, math derivation boosts exact equation/symbol matching; debugging boosts recent failure episodes and exact error messages; scientific judgment boosts source quality, contradiction pressure, and belief state.

## Context-pack contract

Every context pack must include:

1. Bounded token budget.
2. Deterministic item order for fixed state and query.
3. `why_included` per memory.
4. Citations/source spans for factual items.
5. Validity/freshness notes for temporal facts.
6. Contradiction notes if known.
7. Omitted-evidence notes if important evidence was excluded.
8. Retrieval trace for debugging.
9. Feedback hook so the agent can report whether it helped.

# 11. Agent-callable API

The agent should not manually edit memory files. It should call typed operations.

```rust
#[async_trait::async_trait]
pub trait MemoryEngine: Send + Sync {
    async fn observe(&self, event: EventEnvelope) -> Result<MutationReceipt>;
    async fn recall(&self, query: RecallQuery) -> Result<ContextPack>;
    async fn retain(&self, candidate: CandidateMemory) -> Result<MutationReceipt>;
    async fn feedback(&self, outcome: Outcome) -> Result<MutationReceipt>;

    async fn reflect(&self, scope: ReflectionScope) -> Result<ReflectionReport>;
    async fn consolidate(&self, job: ConsolidationJob) -> Result<ConsolidationReport>;
    async fn strengthen_topic(&self, topic: TopicId, reason: StrengthenReason) -> Result<()>;
    async fn rehearse_next(&self, n: usize) -> Result<Vec<MemoryId>>;

    async fn promote_skill(&self, skill: SkillId) -> Result<MutationReceipt>;
    async fn verify_claim(&self, memory: MemoryId, policy: VerificationPolicy) -> Result<VerificationReport>;
    async fn explain(&self, target: ExplainTarget) -> Result<Explanation>;
    async fn rebuild_indexes(&self, scope: RebuildScope) -> Result<RebuildReport>;
}
```

## Calls that organize and compress memory over time

| Agent call                | What it does                                                                 | When to call                                      |
| ------------------------- | ---------------------------------------------------------------------------- | ------------------------------------------------- |
| `observe(event)`          | Commits raw event and creates compressed observations.                       | Every message/tool result/document read.          |
| `recall(query)`           | Returns a bounded, cited context pack.                                       | Before answering, planning, coding, or debugging. |
| `feedback(outcome)`       | Strengthens useful memory, weakens misleading memory, logs failures.         | After task success/failure.                       |
| `reflect(scope)`          | Produces lessons, belief revisions, contradiction notes, open questions.     | After surprising outcomes or idle time.           |
| `consolidate(job)`        | Merges duplicates, creates concept kernels, archives cold noise.             | Idle/sleep cycle.                                 |
| `strengthen_topic(topic)` | Raises review priority for mission-critical concepts.                        | When a topic is important but weak.               |
| `rehearse_next(n)`        | Retrieves weakening memories for review/self-quiz.                           | Idle learning loop.                               |
| `promote_skill(skill)`    | Moves repeated successful workflow into trusted procedural memory.           | After repeatable success + tests.                 |
| `verify_claim(memory)`    | Runs source, unit, formal, or empirical checks.                              | Before using high-stakes knowledge.               |
| `explain(target)`         | Explains why memory was stored, retrieved, omitted, merged, or contradicted. | Debugging trust and retrieval behavior.           |
| `rebuild_indexes(scope)`  | Replays canonical memory into projections.                                   | Migration, corruption, model upgrade.             |

## Example agent loop

```rust
let receipt = mem.observe(EventEnvelope {
    source_kind: SourceKind::ToolResult { tool: "cargo test".into() },
    source_uri: Some("tool://cargo/test/run-2026-05-12".into()),
    observed_at: Utc::now(),
    payload: EventPayload::Text(test_output),
    privacy: PrivacyClass::Project,
    extraction_policy: ExtractionPolicy::FastThenDeep,
}).await?;

let pack = mem.recall(RecallQuery {
    text: "Why did this Rust trait bound fail, and what fix worked before?".into(),
    intent: RecallIntent::Debugging,
    time_scope: TimeScope::RecentFirst,
    required_evidence: EvidenceLevel::SourceSpan,
    budget: ContextBudget { max_tokens: 3500, max_items: 24 },
    include_contradictions: true,
    include_failed_attempts: true,
}).await?;

// Agent uses the pack, fixes code, runs tests.

mem.feedback(Outcome::TaskSuccess {
    context_pack: pack.id,
    used: vec![/* memory ids the agent actually used */],
    note: "Prior failure episode and skill card identified missing lifetime bound.".into(),
}).await?;
```

# 12. Sleep cycle: consolidation and compression

The system’s “sleep” cycle is where compounding happens. It should run during idle time, after a large ingest, or after high-surprise events.

```text
sleep_cycle(scope)
  -> cluster recent episodes
  -> merge duplicate observations
  -> update entity aliases
  -> generate or refresh concept kernels
  -> revise belief states
  -> scan contradictions
  -> promote repeated successful procedures
  -> update topic strength
  -> enqueue rehearsal
  -> archive cold low-utility memories
  -> rebuild affected projections
```

## Consolidation jobs

| Job                      | Output                                                                                                                                                                                                                                       |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Episode clustering       | Coherent event groups, not arbitrary token chunks.                                                                                                                                                                                           |
| Observation compaction   | Dense dated summaries replacing raw chat in hot context.                                                                                                                                                                                     |
| Duplicate merge          | Supersession receipts, not destructive deletes.                                                                                                                                                                                              |
| Concept kernel synthesis | Cited topic cards with prerequisites, equations, claims, skills, open questions.                                                                                                                                                             |
| Belief revision          | Current stance over support/contradiction.                                                                                                                                                                                                   |
| Contradiction scan       | Counterexamples and contested belief notes.                                                                                                                                                                                                  |
| Skill distillation       | Tested workflows from successful repeated episodes.                                                                                                                                                                                          |
| Topic strengthening      | Mastery scores, review queues, next self-study actions.                                                                                                                                                                                      |
| Corpus map update        | GraphRAG/LightRAG-style community summaries for papers/repos. GraphRAG uses graph/community summaries for global/local search, and LightRAG combines graph structures with vector retrieval and incremental updates. ([Microsoft GitHub][4]) |
| Projection refresh       | Tantivy/HNSW/graph/symbol/context-cache updates.                                                                                                                                                                                             |

## Topic strength formula

```text
decayed_base =
  previous_strength * exp(-hours_since_update / half_life_hours)

reinforcement =
  w_recency            * recency
+ w_recurrence         * recurrence
+ w_utility            * utility
+ w_novelty            * prediction_gap
+ w_source_quality     * source_quality
+ w_retrieval_success  * retrieval_success
+ w_skill_success      * linked_skill_success
+ w_graph_neighbor     * linked_topic_strength
- w_contradiction      * unresolved_contradiction_pressure
- w_failure            * retrieval_or_answer_failure

new_strength = clamp(decayed_base + reinforcement, 0.0, 1.0)
```

Topic strength controls:

```text
what gets reviewed
what gets consolidated
what gets promoted to concept kernels
what gets archived
what gets searched for more evidence
what gets included in curriculum-aware context packs
```

# 13. Procedural memory

Procedural memory is not a note saying “try X.” It is executable, testable, versioned skill.

## Skill promotion policy

A skill can become `Trusted` only if:

1. It has clear preconditions.
2. It links to at least one successful episode.
3. It has a repeatable test or verification recipe.
4. It records known failure modes.
5. It runs in a sandbox.
6. It has a success/failure history.
7. It has a source and mutation receipt.

## Skill lifecycle

```text
Draft -> Tested -> Trusted -> Degraded -> Retired
```

A failed skill is not deleted. The failure becomes eval memory. The system may propose a patch, but the patched skill becomes a new version.

## Example skill card

```yaml
id: skill_fetch_arxiv_pdf_v4
name: Fetch arXiv PDF and normalize metadata
task_signature: "fetch_arxiv_pdf(arxiv_id) -> local_pdf_path + metadata"
language: Rust
preconditions:
  - "Network access allowed"
  - "Valid arXiv identifier"
postconditions:
  - "PDF exists in content-addressed artifact store"
  - "ResourceCard has source hash"
tests:
  - "valid id downloads and hashes"
  - "invalid id returns typed error"
success_count: 147
failure_count: 3
known_failure_modes:
  - "arxiv throttling"
  - "invalid version suffix"
linked_concepts:
  - resource_memory
  - citation_discipline
```

# 14. Math and science specialization

For scientific knowledge, the system must distinguish at least these objects:

| Object               | Why it exists                                      |
| -------------------- | -------------------------------------------------- |
| `Observation`        | What was seen or read.                             |
| `ClaimCard`          | What a source asserts.                             |
| `EvidenceLink`       | How a claim is supported or contradicted.          |
| `BeliefState`        | What the agent currently accepts.                  |
| `EquationCard`       | Symbolic relation with units, assumptions, regime. |
| `TheoremCard`        | Formal or informal theorem with dependency DAG.    |
| `ExperimentRecord`   | Hypothesis, method, result, failure/success.       |
| `CounterexampleCard` | First-class falsification memory.                  |
| `ResourceCard`       | Source artifact with checksum and trust profile.   |
| `ConceptKernel`      | Compressed topic model with links and citations.   |

## Equation card requirements

Every trusted equation must store:

```text
original source span
LaTeX
normalized symbolic form
variable definitions
unit constraints
assumptions
regime of validity
derivation steps
linked concepts
linked claims
known counterexamples
example uses
```

Vector search alone is never acceptable for equations. Exact symbol/BM25 and normalized symbolic indexing are mandatory.

## Theorem dependency rule

If theorem `T3` depends on `T2`, and `T2` is downgraded from `FormalCheckPassed` to `Unverified`, then `T3` must be downgraded or marked “verification stale.” This prevents stale formal confidence.

## Experiment log

```text
Hypothesis: H
Prediction: P
Method: M
Actual result: R
Outcome: supports / contradicts / inconclusive
Falsified memories: [...]
Follow-up questions: [...]
```

Counterexamples never decay out of the hot safety path when the same topic recurs.

# 15. Retrieval modes and latency targets

These are engineering targets, not measured claims.

| Mode        | Use                              | LLM required? |                    Target |
| ----------- | -------------------------------- | ------------: | ------------------------: |
| `Hot`       | Every turn recall                |            No | p50 ≤ 35 ms, p95 ≤ 150 ms |
| `Hot+Graph` | Multi-hop but bounded            |            No | p50 ≤ 90 ms, p95 ≤ 350 ms |
| `Deep`      | Literature/math/coding synthesis |      Optional |               p95 ≤ 1.5 s |
| `Research`  | Slow reflective search           |  Yes/optional | minutes allowed, explicit |
| `Sleep`     | Consolidation/compression        |  Yes/optional |           background only |

## Why it scales

| Scaling tactic            | Effect                                                                              |
| ------------------------- | ----------------------------------------------------------------------------------- |
| Lane-specific indexes     | Semantic, skill, equation, episode queries do not compete in one giant index.       |
| Hot/warm/cold tiers       | Top utility memories stay fast; cold memories remain auditable but not hot-indexed. |
| Concept kernels           | Millions of raw facts compress into thousands of topic summaries with links.        |
| Entity-community sharding | Graph traversal is limited to relevant subgraphs.                                   |
| Vector quantization       | Memory footprint decreases for semantic recall.                                     |
| Exact-symbol index        | Avoids expensive semantic search for code/math queries.                             |
| PPR cache                 | Hot entity neighborhoods precomputed.                                               |
| Context-pack cache        | Reuses stable retrieval results when memory state unchanged.                        |
| Append-first writes       | Observations are durable immediately; deep processing can lag safely.               |

# 17. Core API sketch

```rust
pub struct MnemosX {
    ledger: Arc<dyn Ledger>,
    store: Arc<dyn CanonicalStore>,
    indexes: Arc<IndexRegistry>,
    policy: Arc<dyn MemoryPolicy>,
    cognition: Arc<CognitionRuntime>,
}

impl MnemosX {
    pub async fn open(path: impl AsRef<Path>, cfg: Config) -> Result<Self>;

    pub async fn observe(&self, event: EventEnvelope) -> Result<MutationReceipt> {
        let checked = self.privacy_gate(event)?;
        let episodes = self.segmenter.segment(checked).await?;
        let mut receipts = vec![];

        for ep in episodes {
            self.ledger.append_episode(&ep).await?;
            let observations = self.observer.compress(&ep).await?;
            let candidates = self.extractor.extract(&ep, &observations).await?;
            let normalized = self.normalizer.normalize(candidates).await?;
            let decisions = self.policy.decide_batch(&normalized).await?;
            let receipt = self.commit(decisions).await?;
            self.indexes.enqueue_updates(&receipt).await?;
            self.cognition.enqueue_reflection(&receipt).await?;
            receipts.push(receipt);
        }

        Ok(MutationReceipt::merge(receipts))
    }

    pub async fn recall(&self, query: RecallQuery) -> Result<ContextPack> {
        let plan = RecallPlanner::plan(&query, &self.store).await?;
        let candidates = self.indexes.retrieve_parallel(&plan).await?;
        let expanded = self.indexes.expand_graph_temporal(&plan, candidates).await?;
        let fused = reciprocal_rank_fuse(expanded, plan.rrf_k);
        let ranked = self.reranker.rerank(&plan, fused).await?;
        let pack = ContextCompiler::compile(&query, ranked, &self.store).await?;
        self.store.log_context_pack_access(&pack).await?;
        Ok(pack)
    }
}
```

# 18. Testing plan

## Unit tests

| Test                  | Assertion                                                        |
| --------------------- | ---------------------------------------------------------------- |
| Schema roundtrip      | All persisted objects serialize/deserialize with schema version. |
| Lifecycle machine     | Invalid transitions fail.                                        |
| Provenance required   | Trusted claims/equations/theorems cannot lack evidence.          |
| Temporal supersession | Contradictory live fact closes old validity window.              |
| Belief separation     | Belief cannot be stored as fact.                                 |
| Equation units        | Unit mismatch flags or rejects equation.                         |
| Theorem dependency    | Downstream proof status changes when dependency invalidates.     |
| Skill reliability     | Reliability score and promotion gates work.                      |
| Context budget        | Pack never exceeds budget.                                       |
| Vault policy          | Vault memories never auto-retrieve.                              |

## Property tests

```text
Replaying the same ledger twice yields identical canonical state.
Rebuilding indexes from canonical state yields identical manifests.
No Trusted factual memory exists without at least one source span.
No two live temporal edges with same subject/predicate/object conflict silently.
Context packs are deterministic for same query + same state + same seed.
Archiving never deletes provenance.
```

## Integration tests

1. Ingest paper → extract claims/equations/resources → retrieve with citations.
2. Ingest outdated fact → ingest new fact → query current and historical states.
3. Ingest failed code run → later debug query retrieves failure and fix.
4. Repeat workflow three times successfully → skill candidate appears.
5. Run skill test suite → skill becomes trusted.
6. Ingest contradictory experimental result → belief becomes contested.
7. Rebuild every projection from ledger → counts and hashes match.
8. Corrupt index file → `mnx verify` detects and rebuilds projection.
9. Low-confidence source → memory stays Proposed, not Trusted.
10. Prompt asks for stale fact → context pack emits freshness warning.

## Retrieval evals

Use public memory benchmarks plus custom science/code fixtures. LongMemEval evaluates information extraction, multi-session reasoning, temporal reasoning, knowledge updates, and abstention; LoCoMo evaluates very long multi-session conversations; MemoryAgentBench focuses on memory agents with incremental multi-turn interactions. ([arXiv][5])

Custom fixture categories:

```text
exact equation queries
symbol aliases
code error messages
temporally superseded facts
contradictory claims
falsified assumptions
concept prerequisites
paper/resource citations
skill preconditions
repeated failed attempts
```

# 21. Final design decisions

The best architecture is not one existing system. It is this combination:

```text
MIRIX lane taxonomy
+ Hindsight epistemic separation
+ Graphiti/Zep temporal truth graph
+ A-MEM concept evolution
+ Mastra observer/reflector compression
+ Mem0 fast hybrid retrieval
+ HippoRAG associative graph activation
+ Voyager executable skill memory
+ MemoryBank/FSRS strengthening
+ GraphRAG/LightRAG corpus maps
+ Rust immutable ledger and deterministic projections
```

The decisive improvements over a normal memory stack are:

1. **Facts, observations, beliefs, and skills are different objects.**
2. **Memory changes are temporal mutations, not overwrites.**
3. **Every trusted item is source-grounded.**
4. **Raw history is preserved but not repeatedly fed to the LLM.**
5. **Concept kernels compress knowledge without becoming the only truth.**
6. **Counterexamples and failed attempts are first-class memories.**
7. **Executable skills compound ability instead of just improving recall.**
8. **The read path compiles context under budget, with citations and warnings.**
9. **The agent can actively call reflection, consolidation, rehearsal, verification, and explanation tools.**
10. **All indexes are rebuildable from canonical Rust-managed memory.**

That is the strongest buildable design for a superhuman memory system: human-shaped, faster than human, auditable, temporally grounded, capable of learning from experience, and able to scale from a single developer machine to millions of memories without collapsing into vector-chunk noise.
