According to a document from 2026-05-12, the uploaded CMC brief is already aiming at the right target: not вЂњRAG,вЂќ not вЂњchat history,вЂќ not вЂњvector DB memory,вЂќ but a **multi-lane, temporal, self-organizing memory compiler** that tracks facts, experiences, concepts, skills, resources, contradictions, and freshness. Its strongest ideas are the six-lane memory layout, temporal `valid_from`/`valid_to` truth handling, provenance-stamped claims, hybrid BM25/vector/graph retrieval, and bounded context-pack compilation.

Here is the design I would build.

# OpenQG-MNEMOS v2: The Superhuman Memory Compiler

## 0. The verdict

The optimal architecture is:

> **A Rust-first, event-sourced, typed, temporal, self-compressing memory operating system whose durable truth is an append-only ledger plus source-controlled knowledge cards, and whose fast recall comes from rebuildable projections: lexical index, vector index, temporal graph, concept graph, skill registry, equation/theorem index, and context-pack compiler.**

The core rule:

> **The model never receives вЂњthe memory.вЂќ It receives a compiled, cited, task-specific context pack.**

That is the difference between a clever RAG system and a compounding intelligence system.

The uploaded CMC spec correctly rejects flat vector memory and proposes MIRIX-style lanes, Graphiti/Zep-style temporal truth, A-MEM/Zettelkasten concept synthesis, Mastra-style observation compression, HippoRAG-style graph activation, Mem0-style hybrid retrieval, and Voyager-style procedural memory.   My upgrade is to make those ideas **operationally exact**:

1. **No destructive overwrites.** Every memory mutation is an immutable event.
2. **No uncited semantic memory.** Every claim, equation, dataset, code fact, or theorem has provenance.
3. **No vector DB as source of truth.** Vectors are just a projection.
4. **No LLM on the hot retrieval path.** Hot recall is deterministic and fast; LLMs assist extraction, reflection, and compression asynchronously.
5. **No raw transcript bloat.** Conversations, logs, tool calls, papers, and code traces become compressed observations, then typed cards.
6. **No stale fact leakage.** Temporal validity, contradiction state, supersession, and freshness scoring are applied before context packing.
7. **No context-window flooding.** Retrieval emits a compact, ranked, cited, contradiction-aware context pack.

The reason this matters is well supported: long-context models can underuse relevant information when it is buried in long prompts, especially in the middle of the context. That makes **context compilation** more important than simply increasing context length. ([MIT Press Direct][1])

# 1. System name and mission

## Name

**OpenQG-MNEMOS v2**
Full name: **Temporal Zettelkasten Cognitive Memory Compiler**

## Mission

Build a memory substrate that lets agents:

* learn from experience in real time;
* recall millions of memories quickly;
* distinguish facts, beliefs, skills, equations, code, resources, datasets, and questions;
* track what was true, when it was true, why the agent believed it, and what contradicted it;
* compress knowledge over time without losing provenance;
* produce small context packs that let reasoning models operate like they have a long-term scientific mind.

## Target domains

This is not only for chat personalization. It must work for:

* physics;
* mathematics;
* papers and citation graphs;
* equations and derivations;
* codebases;
* public datasets;
* failed experiments;
* simulation notebooks;
* formal proofs;
* scientific claims;
* tool workflows;
* autonomous software engineering.

# 2. The core thesis

The right mental model is not:

```text
query -> vector search -> chunks -> answer
```

The right mental model is:

```text
experience/source -> typed extraction -> temporal truth ledger
                  -> concept synthesis -> executable skill distillation
                  -> hybrid indexes -> context-pack compiler
                  -> reasoning/action -> feedback -> stronger memory
```

The uploaded CMC spec already frames this as a compiler rather than a database, and that is exactly correct. It says the memory system should actively parse, structure, link, refine, and retrieve experience rather than passively store chunks.

My design adds a stronger invariant:

> **Canonical memory is not a database table. Canonical memory is an append-only event log plus typed, source-controlled cards. All databases are projections.**

That means if Tantivy, Qdrant, LanceDB, Kuzu, SQLite, or an in-memory graph corrupts, we rebuild it from the ledger.

# 3. External research synthesis

The design combines the strongest current memory ideas, but each is used only where it is strongest.

| System / idea                      | What we take                                                                        | What we avoid                                                              |
| ---------------------------------- | ----------------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| **Hindsight**                      | Distinguish world facts, agent experiences, entity summaries, and evolving beliefs. | Do not stop at belief/fact modeling; add procedural and scientific memory. |
| **MIRIX**                          | Explicit multi-lane cognitive taxonomy.                                             | Do not rely on flat retrieval across lanes.                                |
| **Graphiti / Zep**                 | Temporal knowledge graph with changing facts, historical validity, and provenance.  | Do not limit memory to semantic facts only.                                |
| **A-MEM**                          | Dynamic Zettelkasten-style note linking and concept evolution.                      | Do not allow noisy links without temporal/provenance constraints.          |
| **Mastra Observational Memory**    | Observer/reflector pattern and dense compressed observations.                       | Do not overcompress away source spans or contradictions.                   |
| **Mem0**                           | Fast hybrid retrieval, salient fact extraction, and memory consolidation.           | Do not let an API abstraction replace a real memory OS.                    |
| **HippoRAG**                       | Associative multi-hop graph activation.                                             | Do not treat it as the whole memory system.                                |
| **Voyager / Memento**              | Executable procedural skills with self-improvement.                                 | Do not confuse executable skill with scientific truth.                     |
| **GraphRAG / LightRAG / RAPTOR**   | Corpus-level summaries, graph communities, hierarchical retrieval.                  | Do not trust summaries without backpointers.                               |
| **MemoryAgentBench / LongMemEval** | Evaluation of retrieval, learning, long-range understanding, and conflict handling. | Do not optimize only anecdotal вЂњfeels smartвЂќ behavior.                     |

Hindsight explicitly models first-class world facts, experiences, summaries, and evolving beliefs; Graphiti emphasizes temporal knowledge graphs and hybrid retrieval over changing relationships; MIRIX exposes specialized memory managers; A-MEM uses Zettelkasten-like memory notes, dynamic linking, and memory evolution. ([Hugging Face][2])

Voyager is the right inspiration for procedural memory because it builds and reuses an executable skill library through feedback and self-verification; GraphRAG, LightRAG, and RAPTOR are the right inspiration for corpus-level synthesis, graph/community summaries, and hierarchical retrieval. ([Hugging Face][3])

Community implementation reports around GraphRAG-style memory repeatedly flag the same production problems: temporal awareness, deduplication, stale facts, vector-only limits, and graph-maintenance cost. Those concerns are exactly why this design uses a ledger, temporal edges, entity-resolution gates, contradiction states, and rebuildable projections. ([Reddit][4])

# 5. The nine memory lanes

The uploaded CMC layout uses six lanes: Core, Episodic, Semantic, Concept, Procedural, and Resource.  I would keep those, but expand to nine lanes because science/math/code need additional first-class structures.

| Lane | Name           | Purpose                                                                                    | Hot-path retrieval?      | Durable? |
| ---: | -------------- | ------------------------------------------------------------------------------------------ | ------------------------ | -------- |
|    1 | **Core**       | Agent identity, project imperatives, active research stance, current repo/task frame.      | Always partially pinned. | Yes      |
|    2 | **Episodic**   | Timestamped observations: attempts, failures, tool traces, user interactions, experiments. | Sometimes.               | Yes      |
|    3 | **Semantic**   | Claims, facts, entities, assumptions, relations.                                           | Yes.                     | Yes      |
|    4 | **Concept**    | Zettelkasten concept kernels, topic maps, abstractions, field summaries.                   | Yes.                     | Yes      |
|    5 | **Procedural** | Executable skills, scripts, workflows, SQL, tool recipes, benchmark harnesses.             | Yes.                     | Yes      |
|    6 | **Resource**   | Source ledger: papers, datasets, repos, URLs, checksums, licenses, local paths.            | Yes.                     | Yes      |
|    7 | **Formal**     | Equations, theorem cards, derivations, units, symbols, proof dependencies.                 | Yes.                     | Yes      |
|    8 | **Belief**     | Hypotheses, uncertainty, competing models, agent confidence, unresolved disputes.          | Yes.                     | Yes      |
|    9 | **Evaluation** | Retrieval tests, answer faithfulness, skill success, benchmark scores, calibration.        | Yes.                     | Yes      |

The new lanes matter:

* **Formal memory** prevents equations and proofs from being flattened into prose.
* **Belief memory** prevents uncertain hypotheses from masquerading as facts.
* **Evaluation memory** lets the system learn which retrieval/compression decisions actually worked.

# 6. Canonical memory vs projections

## 6.1 Canonical memory

Canonical memory consists of:

```text
1. Append-only event ledger
2. Source-controlled typed cards
3. Resource/source ledger
4. Cryptographic content hashes
5. Temporal validity history
6. Mutation receipts
```

## 6.2 Rebuildable projections

Everything else is a projection:

```text
- Tantivy lexical index
- vector index
- temporal graph
- concept graph
- hot in-memory adjacency cache
- skill registry cache
- equation/theorem search index
- topic mastery cache
- context-pack cache
```

This is the single most important engineering choice.

If a projection corrupts, rebuild.

If an embedding model changes, rebuild.

If entity resolution improves, replay the ledger into a new projection.

If a graph edge is invalidated, do not delete history; append a supersession event.

# 7. Storage and performance stack

## 7.1 Rust-first stack

| Layer                  | Recommended implementation                                                               | Why                                                                  |
| ---------------------- | ---------------------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| Event ledger           | Custom Rust append-only journal using `postcard`/`bincode`, `blake3`, `zstd`, CRC footer | Fast, replayable, compact, content-addressed                         |
| Metadata / constraints | SQLite WAL through Rust bindings, or `redb` for pure Rust local mode                     | Durable metadata, constraints, foreign keys, easy inspection         |
| Lexical search         | **Tantivy**                                                                              | Rust-native BM25, phrase queries, incremental/multithreaded indexing |
| Vector search          | Qdrant for high-scale service mode; LanceDB/HNSW for local mode                          | ANN vector recall, metadata filters, hybrid support                  |
| Graph                  | Hot Rust CSR adjacency + optional Kuzu projection                                        | Fast PPR locally; Kuzu for embedded property graph/Cypher            |
| Dataset preview        | **DataFusion**                                                                           | Rust query engine over Arrow/Parquet/CSV/JSON                        |
| Knowledge cards        | Git-backed Markdown/YAML/JSON                                                            | Human review, diffs, PRs, durable wiki                               |
| Skills                 | Rust/Python sandbox with test registry                                                   | Procedural compounding                                               |
| Embeddings             | Pluggable: local Candle/ONNX or remote model                                             | Rebuildable embeddings                                               |
| Telemetry              | OpenTelemetry + structured event logs                                                    | Debuggability                                                        |

Tantivy is a Rust full-text search engine inspired by Lucene and supports BM25, phrase queries, incremental indexing, and multithreaded indexing. ([GitHub][5]) Qdrant is Rust-powered and supports vector search with payload filtering and hybrid search, while LanceDB supports vector search, full-text BM25, and hybrid retrieval. ([Qdrant][6]) Kuzu is an embedded graph database with property graph/Cypher support, columnar disk storage, CSR adjacency, and vectorized query processing; DataFusion is a Rust query engine over Arrow with SQL/DataFrame APIs and support for formats such as CSV, Parquet, JSON, and Avro. ([KuzuDB Docs][7])

## 7.2 Performance targets

For a single high-end local workstation or dev server:

| Operation                            |                               Target |
| ------------------------------------ | -----------------------------------: |
| Append event to ledger               |         p50 < 1 ms async, p95 < 5 ms |
| Insert memory projection             | p50 < 10 ms excluding LLM extraction |
| Recall simple fact from 10M memories |            p50 < 40 ms, p95 < 150 ms |
| Recall with graph activation         |           p50 < 120 ms, p95 < 400 ms |
| Compile 4k-token context pack        |            p50 < 250 ms after recall |
| Rebuild lexical index from 10M cards |                        batch/offline |
| Rebuild vector index                 |                        batch/offline |
| Hot graph PPR over seed neighborhood |        < 100 ms for bounded subgraph |
| LLM extraction                       |                  async, not hot path |
| Reflection/consolidation             |                     async sleep jobs |

The hot path must not call an LLM by default. The hot path is:

```text
query parse -> lexical/vector/entity/graph recall -> rank -> filter -> pack
```

LLMs are used for:

```text
extraction, summarization, contradiction judgment, concept synthesis,
skill repair, proof commentary, question generation, audit reports
```

## 9.2 Temporal validity and provenance

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TemporalValidity {
    /// When the claim/relation is asserted to be valid in the world/model.
    pub valid_from: Option<DateTime<Utc>>,

    /// When the claim/relation stopped being valid.
    /// None means currently valid or currently unresolved.
    pub valid_to: Option<DateTime<Utc>>,

    /// When this system observed or recorded it.
    pub observed_at: DateTime<Utc>,

    /// When the ledger committed it.
    pub committed_at: DateTime<Utc>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SourceSpan {
    pub source_id: Uuid,
    pub uri: String,
    pub title: Option<String>,
    pub locator: Option<String>,        // page, paragraph, line, timestamp
    pub char_start: Option<u64>,
    pub char_end: Option<u64>,
    pub exact_quote_hash: Option<String>,
    pub source_checksum: String,
    pub license: Option<String>,
    pub retrieved_at: DateTime<Utc>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum EvidenceLevel {
    PrimarySource,
    SecondarySource,
    DerivedByAgent,
    Inferred,
    CommunityReport,
    UserProvided,
    Unverified,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Provenance {
    pub spans: Vec<SourceSpan>,
    pub evidence_level: EvidenceLevel,
    pub extractor: Option<String>,
    pub extraction_model: Option<String>,
    pub extraction_prompt_hash: Option<String>,
    pub verifier: Option<String>,
    pub confidence: f32,
}
```

The uploaded CMC specвЂ™s provenance design requires source URI, exact quote, checksum, and confidence; this version makes that provenance multi-span and compatible with papers, code lines, datasets, and tool traces.

# 13. Write path: how memories enter the system

```mermaid
sequenceDiagram
    participant Input as Raw Input
    participant Observer as Observer
    participant Ledger as Event Ledger

    participant Extractor as Extractor

    Input->>Observer: stream/tool/paper/code/log
    Observer->>Observer: segment events + compress observations
    Observer->>Ledger: ObserveEpisode event
    Ledger->>Router: replay new event
    Router->>Extractor: classify lane/kind
    Extractor->>Extractor: extract entities, claims, equations, skills
    Extractor->>Ledger: AddMemory events
    Ledger->>Linker: new memories
    Linker->>Resolver: compare against known facts
    Resolver->>Ledger: AddEdge / Supersede / Contradict
    Ledger->>Cards: write/update typed cards
```

## 13.1 Write algorithm

```text
observe(input):
  1. Segment input into events.
  2. Compress raw stream into observations.
  3. Create episode card.
  4. Append ObserveEpisode ledger event.
  5. Route observations by kind.
  6. Extract typed candidates:
       claims, equations, entities, assumptions, resources,
       skills, datasets, questions, code facts.
  7. Normalize candidates:
       canonical entities, canonical equations, DOI/arXiv/Git hashes,
       normalized predicates, unit signatures.
  8. Ground candidates:
       attach source spans, checksums, quote hashes.
  9. Score candidates:
       confidence, novelty, surprise, source quality, utility.
 10. Link candidates:
       graph edges, concept links, citation links, skill dependencies.
 11. Resolve contradictions:
       compare against same subject/predicate/assumption scope.
 12. Append ledger events.
 13. Update projections.
 14. Schedule consolidation if high novelty/surprise/contradiction.
```

## 13.2 No destructive fact update

```rust
pub async fn assert_fact(
    store: &mut dyn CognitiveStore,
    new_claim: ClaimCard,
) -> anyhow::Result<Uuid> {
    let conflicts = store.find_conflicting_claims(&new_claim).await?;

    let new_id = store.append_memory(MemoryMutation::AddMemory {
        envelope: new_claim.into_envelope(),
    }).await?;

    for old in conflicts {
        store.append_memory(MemoryMutation::CloseValidityWindow {
            id: old.id,
            valid_to: Utc::now(),
            reason: format!("Superseded or contradicted by {}", new_id),
        }).await?;

        store.append_memory(MemoryMutation::MarkContradicted {
            id: old.id,
            by: new_id,
            reason: "Same normalized subject/predicate under incompatible object or assumption scope".into(),
        }).await?;

        store.append_memory(MemoryMutation::AddEdge {
            edge: TemporalEdge {
                id: Uuid::new_v4(),
                source_id: new_id,
                target_id: old.id,
                relation: EdgeRelation::Contradicts,
                temporal: TemporalValidity::now(),
                provenance: new_claim.evidence.clone(),
                weight: 1.0,
                confidence: new_claim.confidence,
                supersedes_edge: None,
            }
        }).await?;
    }

    Ok(new_id)
}
```

# 14. Read path: the context-pack compiler

The uploaded CMC retrieval path proposes dense + BM25, RRF, graph activation, temporal filtering, and bounded context packing.  This design expands it into a full compiler.

## 14.1 Retrieval diagram

```mermaid
flowchart LR
    Q[User/Agent Task] --> P[Query Planner]
    P --> E[Entity + Symbol Extractor]
    P --> T[Temporal Intent]
    P --> B[Budget Planner]

    E --> BM25[Tantivy BM25]
    E --> VEC[Vector Search]
    E --> ENT[Entity Index]
    E --> FORM[Equation/Theorem Index]
    E --> SKILL[Skill Registry]
    T --> GRAPH[Temporal Graph]
    T --> CONCEPT[Concept Graph]

    BM25 --> F[Fusion]
    VEC --> F
    ENT --> F
    FORM --> F
    SKILL --> F
    GRAPH --> F
    CONCEPT --> F

    F --> PPR[Personalized PageRank / Spreading Activation]
    PPR --> RERANK[Deterministic Rerank]
    RERANK --> FILTER[Temporal + Contradiction + Freshness Filter]
    FILTER --> PACK[Context Pack Compiler]
    PACK --> OUT[Bounded Cited Context Pack]
```

## 14.2 Recall channels

For every query, run parallel recall channels:

| Channel          | Best for                                                     |
| ---------------- | ------------------------------------------------------------ |
| Core memory      | active goals, constraints, current repo/project state        |
| BM25             | exact names, formulas, symbols, file paths, DOIs, errors     |
| Vector           | semantic paraphrase                                          |
| Entity           | known papers/authors/datasets/concepts                       |
| Temporal graph   | facts valid at a time; superseded/contradicted claims        |
| Concept graph    | high-level synthesis and field map                           |
| Formal index     | equations, theorem names, units, assumptions                 |
| Procedural index | scripts, skills, workflows, SQL                              |
| Evaluation lane  | what worked before, failed retrievals, benchmark regressions |
| Resource ledger  | source URLs, hashes, licenses, download scripts              |

## 14.3 Scoring formula

```text
score(memory, query) =
  0.18 * bm25_score
+ 0.18 * vector_score
+ 0.14 * entity_match_score
+ 0.12 * graph_activation_score
+ 0.10 * concept_kernel_score
+ 0.08 * temporal_relevance
+ 0.07 * source_quality
+ 0.06 * utility
+ 0.04 * freshness
+ 0.03 * skill_reliability
- 0.10 * contradiction_risk
- 0.08 * staleness_penalty
- 0.05 * redundancy_penalty
```

## 14.4 RRF fusion

```rust
pub fn reciprocal_rank_fusion(
    ranked_lists: &[Vec<ScoredId>],
    k: f32,
) -> Vec<ScoredId> {
    let mut scores: std::collections::HashMap<Uuid, f32> = Default::default();

    for list in ranked_lists {
        for (rank, item) in list.iter().enumerate() {
            let contribution = 1.0 / (k + rank as f32 + 1.0);
            *scores.entry(item.id).or_default() += contribution;
        }
    }

    let mut fused: Vec<_> = scores.into_iter()
        .map(|(id, score)| ScoredId { id, score })
        .collect();

    fused.sort_by(|a, b| b.score.partial_cmp(&a.score).unwrap());
    fused
}
```

## 14.5 Graph activation

```rust
pub fn personalized_pagerank(
    graph: &TemporalGraph,
    seeds: &[(Uuid, f32)],
    now: DateTime<Utc>,
    max_steps: usize,
    damping: f32,
) -> Vec<ScoredId> {
    let mut activation = ActivationMap::from_seeds(seeds);

    for _ in 0..max_steps {
        let mut next = ActivationMap::new();

        for (node, mass) in activation.iter() {
            for edge in graph.valid_out_edges(*node, now) {
                if edge.relation.blocks_context() {
                    continue;
                }

                let rel_weight = edge.relation.activation_weight();
                let temporal_weight = edge.temporal_decay(now);
                let propagated = mass * damping * edge.weight * rel_weight * temporal_weight;

                next.add(edge.target_id, propagated);
            }
        }

        next.add_seed_residual(seeds, 1.0 - damping);
        activation = next.prune_top_k(2_000);
    }

    activation.into_ranked()
}
```

# 15. Compression: how the system gets smarter over time

This is where the design becomes superhuman.

Human memory compresses; it does not retain every raw moment equally. The system should compress too, but with auditability.

## 15.1 The compression pyramid

```mermaid
flowchart TB
    R[Raw Event Stream<br/>chat, logs, tools, papers, code] --> O[Observation<br/>dense timestamped episode summary]
    O --> A[Atomic Memory<br/>claim, equation, entity, skill, resource]
    A --> C[Card<br/>typed durable markdown/yaml card]
    C --> K[Concept Kernel<br/>linked synthesis over many cards]
    K --> M[Field Map<br/>topic-level compressed worldview]
    M --> P[Context Pack<br/>task-specific final prompt memory]

    A -.backpointers.-> R
    C -.source spans.-> R
    K -.citations.-> C
    M -.coverage.-> K
    P -.trace.-> M
```

## 15.2 Compression levels

| Level | Object         | Example                                  | Purpose                  |
| ----: | -------------- | ---------------------------------------- | ------------------------ |
|     0 | Raw source     | PDF, log, tool call                      | Rehydration              |
|     1 | Observation    | вЂњTried build, failed due to trait boundвЂќ | Episodic memory          |
|     2 | Atom           | claim/equation/entity/skill              | Typed retrieval          |
|     3 | Card           | paper card, equation card                | Durable human/agent wiki |
|     4 | Concept kernel | вЂњInformation paradox constraintsвЂќ        | Abstraction              |
|     5 | Field map      | вЂњQuantum gravity dataset landscapeвЂќ      | Strategic navigation     |
|     6 | Context pack   | 4kвЂ“32k task payload                      | Reasoning input          |

## 15.3 Consolidation daemons

| Daemon                     | Role                                                                                                 |
| -------------------------- | ---------------------------------------------------------------------------------------------------- |
| **Observer**               | Compresses raw streams into observations.                                                            |
| **Extractor**              | Converts observations/sources into typed candidates.                                                 |
| **Linker**                 | Adds graph and concept links.                                                                        |
| **Contradiction Resolver** | Detects conflicts, closes validity windows, creates dispute cards.                                   |
| **Entity Deduper**         | Merges duplicate entities using blocking, lexical matching, embeddings, and LLM only when ambiguous. |
| **Reflector**              | Builds concept kernels and field maps.                                                               |
| **Compactor**              | Removes redundant summaries while preserving backpointers.                                           |
| **Rehearsal Daemon**       | Uses spaced repetition/utility to keep important memories sharp.                                     |
| **Skill Distiller**        | Turns repeated successful workflows into executable skills.                                          |
| **Skill Verifier**         | Runs tests, updates reliability, links failures to episodes.                                         |
| **Freshness Auditor**      | Flags stale facts, broken links, old APIs, superseded papers.                                        |
| **Question Generator**     | Creates next searches from gaps, contradictions, and weak evidence.                                  |
| **Eval Runner**            | Tests retrieval and context-pack quality continuously.                                               |

Mastra-style observational memory is useful because it separates observation compression from reflection, and reports large compression gains in tool-heavy contexts; this design uses that pattern but keeps source spans and provenance so compression does not erase evidence. ([Mastra][8])

```rust
pub fn update_utility(memory: &mut MemoryEnvelope, feedback: &FeedbackEvent) {
    let delta = match feedback.outcome {
        FeedbackOutcome::HelpfulAnswer => 0.08,
        FeedbackOutcome::RetrievedButUnused => -0.02,
        FeedbackOutcome::WrongOrStale => -0.15,
        FeedbackOutcome::CitedSuccessfully => 0.05,
        FeedbackOutcome::SkillSucceeded => 0.10,
        FeedbackOutcome::SkillFailed => -0.10,
    };

    memory.scores.utility = (memory.scores.utility + delta).clamp(0.0, 1.0);
    memory.scores.retrieval_success =
        0.9 * memory.scores.retrieval_success + 0.1 * feedback.success_value();
}
```

# 20. Formal memory for math and physics

Scientific memory cannot treat equations as text chunks.

Formal memory should index:

```text
- equation LaTeX;
- canonicalized symbols;
- units and dimensions;
- assumptions;
- derivation steps;
- theorem dependencies;
- counterexamples;
- proof status;
- computational implementation links;
- dataset/experiment links.
```

## Formal retrieval examples

| Query                                                  | Retrieval behavior                            |
| ------------------------------------------------------ | --------------------------------------------- |
| вЂњFind equations using Ricci scalar and entropy boundsвЂќ | Exact symbol + semantic + graph recall        |
| вЂњWhat assumptions make this derivation invalid?вЂќ       | Equation в†’ assumptions в†’ counterexamples      |
| вЂњWhich code implements this equation?вЂќ                 | Equation в†’ Implements edge в†’ Skill/Codebase   |
| вЂњWhat papers contradict this theorem-like claim?вЂќ      | Claim в†’ Contradicts edges в†’ Paper cards       |
| вЂњWhat data would test this?вЂќ                           | Claim в†’ Prediction в†’ Dataset/Experiment cards |

# 21. Procedural memory: executable knowledge

Procedural memory is how the system stops being a passive encyclopedia.

## 21.1 Skill lifecycle

```text
Proposed -> Parsed -> Sandboxed -> Tested -> Trusted -> Reused -> Refined
                                      в†“
                                  Failed -> Repair -> Retest
```

## 21.2 Skill distillation

A skill is created when the system sees repeated successful patterns:

```text
- same workflow repeated в‰Ґ 3 times;
- same code generated and used successfully;
- same dataset download process reused;
- same equation-to-simulation pattern repeated;
- same paper ingestion pipeline repeated.
```

## 21.3 Skill validation

Every skill has:

```text

- code hash;
- AST parse result;
- tests;
- permissions;
- reliability score;
- last successful episode;
- failure traces;
- repair history.
```

VoyagerвЂ™s core insight is that executable skills can compound and be reused, but this system adds provenance, permissions, tests, and links to claims/equations/resources. ([Hugging Face][3])

# 23. Entity resolution at million-memory scale

Entity deduplication is one of the hardest parts.

## 23.1 Four-stage dedupe

```text
1. Blocking
   - normalized name
   - DOI/arXiv/Git hash
   - ORCID
   - dataset DOI
   - symbol + domain

2. Cheap similarity
   - string distance
   - alias table
   - metadata overlap
   - citation/resource overlap

3. Embedding similarity
   - semantic entity description vector
   - co-neighborhood similarity

4. Expensive adjudication
   - LLM or human only for ambiguous cases
```

## 23.2 Merge rule

Never silently merge.

```text
MergeEntities event:
  canonical_entity
  duplicate_entity
  confidence
  evidence
  reversible = true
```

This directly addresses the community pain point that large graph memories often fail through deduplication drift and temporal confusion. ([Reddit][4])

# 25. Security and safety

Memory systems create dangerous failure modes if they silently mutate themselves.

## Rules

```text
- No hidden memory writes.
- No LLM direct database mutation.
- Every mutation goes through typed API.
- Every mutation emits receipt.
- Every claim has provenance or is marked unverified.
- Every skill has sandbox permissions.
- Every source can be quarantined.
- Every projection is rebuildable.
- Every context pack has retrieval trace.
- Every destructive-looking action is actually append-only supersession.
```

## Skill sandbox policies

| Skill type         | Default permission                 |
| ------------------ | ---------------------------------- |
| Rust compute       | local temp dir only                |
| Python scientific  | local temp dir only                |
| Dataset downloader | network allowlist required         |
| Shell              | disabled unless explicitly trusted |
| SQL                | read-only unless granted           |
| Git operations     | branch/PR only unless approved     |
| File writes        | workspace-scoped                   |
| Secret access      | explicit vault permission          |

# 31. Why this is better than the uploaded CMC v1

The uploaded CMC design is excellent, but I would change seven things.

## 31.1 Add an immutable event ledger before every database

The CMC mentions a Git-backed append-only JSONL ledger as Ring 1.  I would make that the non-negotiable root of the system, not a roadmap step.

## 31.2 Split formal memory from semantic memory

Equations, proofs, assumptions, units, and theorem dependencies need specialized representation.

## 31.3 Split beliefs from facts

Agent hypotheses must not become semantic facts.

## 31.4 Split evaluation memory into its own lane

The system must remember which retrievals worked, which skills failed, and which context packs caused bad answers.

## 31.5 Make context compilation the primary product

The memory systemвЂ™s output is not вЂњdocuments.вЂќ It is a context pack.

## 31.6 Make projections rebuildable

Tantivy, vector DB, graph DB, and caches are disposable projections.

## 31.7 Keep LLMs off the hot path

LLMs are for extraction, reflection, and audit; recall should stay deterministic and fast.

# 33. The one-page engineering contract


# 35. Bottom line

The best possible memory system is not a wiki, not a graph DB, not a vector DB, and not a longer context window.

It is a **memory compiler**.

The durable substrate is:

```text
append-only events + typed cards + provenance + temporal graph
```

The fast substrate is:

```text
BM25 + vector + entity index + hot temporal graph + skill registry + formal index
```

The intelligence substrate is:

```text
observer + extractor + linker + contradiction resolver + reflector
+ consolidator + rehearsal + skill distiller + eval runner
```

The agent-facing product is:

```text
small, cited, current-valid, task-specific context packs
```

That is how you get an agent that compounds knowledge over months, learns from failures, recalls fast across millions of memories, tracks science rigorously, writes better code, and does not drown in its own context.
