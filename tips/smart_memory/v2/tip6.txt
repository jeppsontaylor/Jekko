# MNEMOS-Σ: The Superhuman Memory Compiler

**MNEMOS-Σ** is not a vector database, not a wiki, not a graph DB, and not a note-taking system. It is a **Rust-native memory compiler** that turns raw agent experience into durable, typed, temporally grounded, evidence-linked, compressed, searchable, executable knowledge.

The most important correction to the earlier CMC/CLAUDE/CODEX designs is this:

> **The memory system must be built around epistemic separation, not around storage lanes.**

Storage lanes matter, but the deeper principle is: **observation ≠ claim ≠ belief ≠ concept ≠ skill ≠ source**.

That distinction is the difference between an agent that accumulates knowledge and an agent that slowly poisons itself with stale summaries.

The public frontier strongly supports this direction. Hindsight separates world facts, agent experiences, entity/mental-model summaries, and evolving beliefs; Graphiti/Zep track time-valid facts on edges; A-MEM evolves a Zettelkasten-like memory network; Mastra Observational Memory compresses raw history into stable observations; Mem0 emphasizes low-latency extraction/retrieval; Voyager shows executable skill libraries compound behavior; Anthropic’s context-engineering guidance stresses that context is finite and must be curated with compaction, structured notes, and sub-agents rather than stuffed full. ([Hugging Face][1])

The design target:

```text
10M+ memory objects per local node
100M+ compressed memories with sharding
p50 hot recall under 50 ms
p95 normal recall under 250 ms
p99 heavy graph/context recall under 1.5 s
async observe write under 1 ms
source-grounded claims only
no silent destructive overwrites
all indexes rebuildable
all important memories auditable in Git
```

# 1. The Core Philosophy

MNEMOS-Σ is based on six hard rules.

## Rule 1: Git/card memory is canonical; databases are projections

The canonical durable memory is:

```text
append-only mutation ledger
+ content-addressed source/resource ledger
+ Git-backed reviewed cards
+ schema-versioned object registry
```

Vector indexes, BM25 indexes, graph traversals, concept caches, context-pack caches, and topic maps are **compiled artifacts**. They can be deleted and rebuilt.

This is the single most important design principle from your specs. It prevents vendor lock-in, embedding drift, uninspectable hallucinated knowledge, and memory corruption.

## Rule 2: Every memory has an epistemic type

The system must know whether it is storing:

```text
what happened
what a source said
what claim was extracted
what equation was derived
what the agent currently believes
what concept was synthesized
what procedure works
what retrieval failed
what question remains open
```

Most memory systems fail because they store all of these as “text chunks.”

## Rule 3: Time is first-class

A fact is not just true or false. It is true:

```text
during a world-time interval
according to some source
recorded at some transaction time
with some confidence
possibly superseded later
```

Graphiti/Zep are especially important here because their “facts on edges” include `valid_at` and `invalid_at` timestamps, preserving the history of changing facts instead of overwriting them. ([Zep Documentation][2])

## Rule 4: The agent never writes trusted memory directly

The agent can propose memory. The system validates, links, scores, and promotes.

```text
agent writes proposal
validator checks provenance
linker checks duplicates/contradictions
consolidator merges/compresses
review/eval gate promotes
Git commit records trusted card
```

This protects the memory bank from extraction errors, hallucinated facts, prompt injection, and “agent got excited and stored nonsense.”

## Rule 5: Retrieval must be compiled, not searched

A query should not go straight to vector search. It should go through a **context compiler**:

```text
classify intent
retrieve from multiple channels
expand graph neighborhood
apply temporal filters
retrieve contradictions
retrieve failed prior attempts
rerank by task profile
pack into bounded context
return citations + omissions
```

Anthropic’s contextual retrieval work found that embeddings + BM25 outperform embeddings alone, and that contextual embeddings/BM25 plus reranking stack additional retrieval gains. ([Anthropic][3])

## Rule 6: Compression is a hierarchy, not a summary

The memory system should preserve raw evidence but feed agents compressed maps.

```text
Raw source
  → episode
    → observation
      → atomic claim/equation/skill/resource card
        → concept kernel
          → topic map
            → always-loaded memory atlas
```

The agent should not receive 10,000 memories. It should receive a **memory atlas** plus handles to fetch precise evidence.

This directly addresses the “agent cannot search for what it does not know exists” problem that shows up repeatedly in current agent-memory discussions. Community builders on Reddit and X are converging on the same theme: pure vector memory fails for temporal, exact, and relational recall; hybrid retrieval plus small always-loaded maps are increasingly viewed as the durable pattern. ([Reddit][4])

# 2. What I Would Change From CLAUDE, CODEX, and CMC

Your existing specs are strong. The best final architecture combines them but tightens the failure modes.

## CLAUDE_MEMSPEC strengths

`CLAUDE_MEMSPEC.md` is strongest on:

```text
12 memory stores
FSRS-style strengthening
math/science specialization
formal verification hooks
Rust workspace layout
performance targets
hash-chained episodes
```

Its weakness: it still treats many memory types as “stores” rather than as **epistemic roles over a shared ledger**. I would keep the stores but make the ledger/object model the center.

## CODEX_MEMSPEC strengths

`CODEX_MEMSPEC.md` is strongest on:

```text
canonical ownership boundaries
Git-backed cards
append-only JSONL receipts
explicit ClaimCard / EquationCard / ConceptCard / SkillCard / EvalMemory
context-pack contract
topic strengthening
rollout discipline
```

Its weakness: the baseline performance target is too conservative, and the graph/index design is underpowered for 10M+ objects. I would keep its safety model but strengthen the index layer.

## Gemini/CMC strengths

The pasted CMC is strongest on:

```text
compiler metaphor
six-lane architecture
Graphiti-style temporal edges
HippoRAG-style PageRank expansion
Mem0-style fusion
Voyager-style procedural memory
Ring-based implementation roadmap
```

Its weakness: it overclaims “perfect” capability and under-specifies canonical truth, eval gates, privacy, index rebuilds, and historical queries. “Discard stale facts” is wrong for science; stale facts must be **excluded for current truth queries but preserved for historical and reasoning queries**.

## Final synthesis

```text
CLAUDE = broad cognitive + science/math + strengthening
CODEX  = canonical truth + schema rigor + eval/rollout discipline
CMC    = compiler metaphor + graph/PPR + procedural synthesis

MNEMOS-Σ = epistemic ledger + temporal graph + memory atlas + hybrid retrieval compiler + executable skills + continuous eval
```

# 4. The Memory Lanes

MNEMOS-Σ uses **10 lanes**, not because the system needs 10 databases, but because the agent needs 10 epistemic views of memory.

| Lane       | Purpose                                                                   | Hot-path role          | Canonical form                          |
| ---------- | ------------------------------------------------------------------------- | ---------------------- | --------------------------------------- |
| Core       | Mission, durable constraints, active project stance, memory atlas pointer | Always loaded          | `CoreBlock`                             |
| Working    | Current task state, scratch plan, active hypotheses                       | In context / TTL       | `WorkingFrame`                          |
| Episodic   | What happened: searches, tool runs, conversations, builds, failures       | Temporal recall        | `Episode`, `Observation`                |
| Semantic   | Claims, facts, entities, equations, theorems, relationships               | Evidence recall        | `ClaimCard`, `FactEdge`, `EquationCard` |
| Belief     | Current evidence-weighted stance over claims                              | Judgment               | `BeliefState`                           |
| Concept    | Zettelkasten-style topic kernels, maps, prerequisites                     | Compression/navigation | `ConceptKernel`                         |
| Procedural | Executable skills, workflows, tested scripts                              | Action reuse           | `SkillCard`                             |
| Resource   | Papers, datasets, repos, files, APIs, hashes, licenses                    | Source rehydration     | `ResourceCard`                          |
| Eval       | Bad answers, failed retrievals, test results, corrections                 | Self-repair            | `EvalMemory`                            |
| Question   | Open questions, gaps, radar search targets                                | Curiosity/search       | `QuestionCard`                          |

Why include both Semantic and Belief? Because science requires the difference between:

```text
Paper P claims X.
The evidence for X is mixed.
The agent currently believes X is plausible under assumptions A, B, C.
```

That is the core of scientific memory.

# 7. Temporal Truth Model

The graph must be **bitemporal**.

| Time type                 | Meaning                                     |
| ------------------------- | ------------------------------------------- |
| `valid_from` / `valid_to` | When the fact was true in the world/source  |
| `tx_from` / `tx_to`       | When the memory system believed/recorded it |

Example:

```text
2026-01-01: source says method X is SOTA
2026-04-01: new paper supersedes method X
2026-05-12: agent learns about the April paper
```

The old fact is not deleted. It becomes:

```yaml
status: superseded
valid_to: 2026-04-01
tx_to: 2026-05-12
```

The new fact becomes:

```yaml
status: trusted
valid_from: 2026-04-01
tx_from: 2026-05-12
supersedes: [old_edge_id]
```

Current-truth queries exclude old facts. Historical queries include them.

```mermaid
stateDiagram-v2
    [*] --> Proposed
    Proposed --> Trusted: validator accepts
    Proposed --> Rejected: unsupported / duplicate / unsafe
    Trusted --> Consolidated: used + linked + reviewed
    Trusted --> Contradicted: new evidence conflicts
    Trusted --> Superseded: newer valid fact replaces
    Contradicted --> Trusted: conflict resolved
    Contradicted --> Superseded: old claim closed
    Consolidated --> Archived: low utility but preserved
    Superseded --> Archived: cold storage
```

# 8. The Write Path

The write path must be fast on the synchronous path and smart on the asynchronous path.

```text
  hash content
  append ledger
  update object registry
  enqueue extraction/index jobs
  return receipt
```



## Validation gates

A candidate memory is rejected or left as `Proposed` unless it has:

```text
source span or human-authored provenance
schema-valid object
dedupe check
privacy check
temporal check
contradiction check

```

For scientific claims, the validator requires:

```text
claim statement
source span

support links

```

For procedural skills, promotion requires:


## Retrieval channels

| Channel                         | Why it exists                                       |
| ------------------------------- | --------------------------------------------------- |
| Core/Atlas                      | Tells agent what memory exists before searching     |
| BM25/Tantivy                    | Exact equations, IDs, titles, symbols, paths, names |
| Dense vector                    | Semantic paraphrase                                 |
| Sparse vector                   | Lexical-semantic hybrid                             |
| Multi-vector / late interaction | Long technical passages and exact concepts          |
| Temporal graph                  | What was true when                                  |
| Entity graph                    | Who/what relates to what                            |
| Concept kernels                 | Compressed topic-level understanding                |
| Procedural skills               | Reusable action                                     |
| Eval memory                     | Prior failures and corrections                      |
| Resource cards                  | Source rehydration                                  |

This matches the broad direction of current retrieval research and tools. BGE-M3 supports dense, sparse, and multi-vector retrieval in one model family; GraphRAG and LightRAG both show why graph-aware indexing helps beyond flat chunks; Qdrant supports hybrid dense/sparse retrieval; Tantivy gives Rust-native BM25/full-text search. ([Hugging Face][5])

## Score model

Hot path uses deterministic features:

```text
score =
  w_exact        * exact_match
+ w_dense        * dense_similarity
+ w_sparse       * sparse_score
+ w_graph        * graph_activation
+ w_temporal     * temporal_relevance
+ w_source       * source_quality
+ w_confidence   * confidence
+ w_utility      * retrieval_success
+ w_topic        * topic_strength
+ w_recency      * recency
- w_contradict   * unresolved_contradiction_penalty
- w_stale        * stale_penalty
- w_privacy      * access_penalty
```

Then a learned ranker can be trained from eval memory, but the baseline must work without LLM reranking.

## Reciprocal rank fusion

```rust
pub fn rrf_score(rank: usize, k: f32) -> f32 {
    1.0 / (k + rank as f32)
}
```

Default:

```text
k = 60
top_per_channel = 64
fusion_pool = 256
graph_seed_count = 20
final_items = token-budget constrained
```

## Associative graph expansion

Use HippoRAG-like graph activation:

```text
seed nodes = top fused memories
run bounded personalized PageRank
edge weights = relation strength * confidence * freshness * utility
exclude invalid facts unless historical query
return high-activation neighbors
```

HippoRAG’s core insight is that multi-hop recall needs associative graph traversal, not just semantic proximity. ([Hugging Face][6])

# 10. Context Pack Contract

The context pack is the only thing the reasoning model sees.

It must be:

```text
small
deterministic
cited
task-specific
contradiction-aware
freshness-aware
handle-rich
```

## Example context pack shape

```yaml
context_pack:
  query: "Compare theory X and Y under assumption A"
  intent: scientific_compare
  budget_tokens: 6000

  always_loaded:
    - core mission
    - relevant memory atlas slice

  evidence:
    claims:
      - id: claim_...
        statement: ...
        why_included: "highest support for theory X"
        source_spans: [...]
        confidence: 0.82
    equations:
      - id: eq_...
        latex: ...
        assumptions: [...]
    resources:
      - id: paper_...
        title: ...
        source_quality: high

  contradictions:
    - claim_id: claim_...
      contradicts: claim_...
      summary: ...

  concepts:
    - concept_id: concept_...
      compressed_summary: ...

  skills:
    - skill_id: skill_...
      name: "run simulation preview"
      reliability: 0.91

  omitted_evidence:
    - reason: "budget"
      fetch_handle: "memory.get_cluster(cluster_...)"

  tool_handles:
    - memory.expand(claim_...)
    - memory.get_source_span(...)
    - memory.run_skill(skill_...)
```

## Packing policy

```text
10% core/atlas
45% source-grounded evidence
15% concept kernels
10% contradictions/freshness warnings
10% procedural/eval memory
10% handles/omissions/next retrieval routes
```

Long context alone is not the solution. “Lost in the Middle” shows that models often use information best at the beginning/end of context and degrade when relevant information is buried in the middle; Anthropic’s context-engineering guidance similarly treats context as finite and emphasizes compaction, note-taking, and sub-agent summaries. ([MIT Press Direct][7])

# 11. Knowledge Compression System

This is where MNEMOS-Σ becomes “superhuman.”

## Compression hierarchy

```mermaid
flowchart TB
    R[Raw Source / Tool Trace / Paper / Dataset] --> E[Episode]
    E --> O[Dense Observation]
    O --> A[Atomic Cards]
    A --> C[Concept Kernel]
    C --> T[Topic Map]
    T --> M[Memory Atlas]
    M --> P[Context Pack]

    A --> S[Semantic Graph]
    A --> V[Vector Index]
    A --> B[BM25 Index]
    C --> G[Concept Graph]
```

## Memory Atlas

The Memory Atlas is a small, always-loaded compressed map.

Example:

```markdown
# Memory Atlas: OpenQG

## Active Topics
- quantum gravity literature radar → concept:concept_qg_lit
- data-source manifests → concept:concept_data_sources
- Rust scientific workflow tooling → concept:concept_rust_science

## Current High-Value Gaps
- Need stronger coverage of public benchmark datasets.
- Need verified downloader scripts.
- Need equation extraction pipeline comparison.

## Retrieval Handles
- memory.topic("quantum gravity")
- memory.claims(topic="vacuum energy", contradictions=true)
- memory.skills(intent="download_dataset")
- memory.eval_failures(task="paper_summary")
```

The atlas is what prevents the agent from searching blindly.

## Concept kernel generation

A concept kernel is created when topic strength crosses threshold.

Required fields:

```text
name
aliases
short definition
prerequisites
canonical examples
core claims
core equations
key sources
datasets/code
contradictions
failed attempts
open questions
linked skills
retrieval handles
last review date
topic strength explanation
```

## Topic strength formula

```text
decayed_base =
  previous_strength * exp(-hours_since_update / half_life_hours)

reinforcement =
  w_recency             * recency
+ w_recurrence          * recurrence
+ w_utility             * utility
+ w_novelty             * novelty
+ w_source_quality      * source_quality
+ w_retrieval_success   * retrieval_success
+ w_skill_success       * skill_success
+ w_neighbor            * graph_neighbor_boost
- w_contradiction       * unresolved_contradiction_pressure
- w_failure             * repeated_failure_penalty

new_strength = clamp(decayed_base + reinforcement, 0.0, 1.0)
```

## Compression rules

| Situation                        | Action                              |
| -------------------------------- | ----------------------------------- |
| Many repeated observations       | Merge into stable semantic claim    |
| Related claims co-activate often | Link or create concept kernel       |
| Concept kernel too large         | Split into child kernels            |
| Claim contradicted               | Update belief; do not delete source |
| Skill repeatedly succeeds        | Promote and link to topics          |
| Retrieval repeatedly fails       | Create EvalMemory and search gap    |
| Topic weak but important         | Schedule review instead of deleting |
| Old low-utility evidence         | Cold archive, keep hash/source      |

# 12. Procedural Memory and Skill Foundry

Scientific/coding agents need executable memory.

Voyager is the key precedent: it used an automatic curriculum, an ever-growing executable skill library, and iterative prompting with environment feedback/self-verification to compound capabilities. ([Hugging Face][8])

MNEMOS-Σ turns that into a Rust-first skill foundry.

## Skill lifecycle

```mermaid
stateDiagram-v2
    [*] --> Draft
    Draft --> Candidate: generated from successful episode
    Candidate --> Tested: has verification recipe
    Tested --> Trusted: passes tests N times
    Trusted --> Degraded: failures exceed threshold
    Degraded --> Trusted: patched + retested
    Trusted --> Retired: superseded or unsafe
    Degraded --> Retired: repeated failures
```

## Skill card requirements

```text
name
intent signature
preconditions
inputs/outputs
code reference
test command
success episodes
failure episodes
reliability score
linked concepts/resources
safety class
```

## Example skills for OpenQG

```text
extract_arxiv_metadata
download_dataset_manifest
parse_pdf_equations
normalize_latex_equation
run_unit_check
build_citation_neighborhood
generate_literature_radar_queries
preview_public_dataset
run_rust_scientific_benchmark
compare_claim_contradictions
```

## Skill execution policy

```text
Never trust untested generated code.
Never promote a skill without a reproducible test.
Never erase failed executions.
Use failed traces as future debugging memory.
```

# 13. Science and Math Specialization

MNEMOS-Σ must understand science as structured epistemology.

## Required object separation

| Object         | Stores                                     | Must not store                      |
| -------------- | ------------------------------------------ | ----------------------------------- |
| `ResourceCard` | Paper/dataset/repo metadata, hashes, trust | Extracted claims without provenance |
| `ClaimCard`    | Atomic claim, support, contradiction       | Agent belief as fact                |
| `EquationCard` | LaTeX, normalized form, variables, units   | Hidden assumptions                  |
| `TheoremCard`  | Statement, proof, dependencies             | Unverified proof as trusted         |
| `BeliefState`  | Current stance over evidence               | Raw source assertion                |
| `QuestionCard` | Open target, blockers, next actions        | Resolved claims                     |
| `SkillCard`    | Tested procedure                           | One-off hack                        |
| `EvalMemory`   | Failure/correction/test outcome            | Silent correction                   |

## Equation index

Equation retrieval needs exact search and symbolic normalization.

```text
LaTeX string
normalized symbolic form
variable aliases
unit constraints
assumptions
derivation edges
regime of validity
source span
```

Vector search alone is unacceptable for equations.

## Claim graph

```text
Paper → asserts → Claim
Claim → supports → Belief
Claim → contradicts → Claim
Equation → derives → Claim
Dataset → tests → Claim
Skill → reproduces/fails_to_reproduce → Claim
```

## Belief update example

```text
Observation:
  Source S reports result R under method M.

Claim:
  R supports proposition P under assumptions A.

Belief:
  Agent currently accepts P with confidence 0.68 because
  C1, C2 support it and K1 contradicts it.
```

The system must answer:

```text
What did we observe?
Who asserted it?
What claim was extracted?
What evidence supports it?
What evidence contradicts it?
What does the agent currently believe?
When and why did that belief change?
```

# 14. Rust-Native Storage Stack

## Baseline local-first stack

| Need               | Choice                                 | Why                                    |
| ------------------ | -------------------------------------- | -------------------------------------- |
| Append-only ledger | JSONL/CBOR + BLAKE3 hash chain         | Replayable, inspectable                |
| Object registry    | `redb` or `fjall`                      | Rust-native embedded storage           |
| Git cards          | `git2` or shell git                    | Reviewable canonical memory            |
| Full-text search   | Tantivy                                | Rust-native BM25/search                |
| Dense vector       | Qdrant or local HNSW                   | Rust-powered scalable vector retrieval |
| Sparse/multivector | BGE-M3-compatible adapter              | Dense + sparse + multi-vector          |
| Graph              | Custom CSR adjacency + roaring bitmaps | Fast bounded graph traversal           |
| Temporal index     | interval trees + time buckets          | Validity filtering                     |
| Blob store         | content-addressed files                | Raw artifacts out of Git               |
| Token packing      | `tiktoken-rs`/custom tokenizer         | Budget enforcement                     |
| Embeddings         | Candle/ONNX/external trait             | Local-first, provider-agnostic         |

Qdrant is Rust-powered and supports hybrid dense/sparse retrieval patterns; SurrealDB is also Rust-native and can unify document, graph, vector, full-text, and time-series models, but I would not make SurrealDB the canonical baseline because the memory contract should remain independent of one database. Kuzu is excellent for embedded analytical graph workloads, but the Rust-first baseline should use a custom projection and make Kuzu optional. ([Qdrant][9])

## Canonical vs projection

```text
Canonical:
  ledger/
  object registry/
  git cards/
  source vault/

Projection:
  Tantivy
  Qdrant/HNSW
  graph CSR
  topic cache
  context cache
```

Rebuild command:

```bash
mnemos rebuild --from-canonical --all-indexes
```

Expected:

```text
same object count
same card hashes
same graph edge count
same index manifests
receipt-linked warnings
no silent loss
```

## Agent-facing tools

The agent should get powerful but safe tools.

```text
memory.observe(event)
memory.recall(query, budget, evidence_level)
memory.get(id)
memory.expand(id, depth, relation_filter)
memory.timeline(entity, time_scope)
memory.sources(claim_or_resource_id)
memory.propose_claim(...)
memory.propose_question(...)
memory.record_skill_run(...)
memory.search_gap(topic)
memory.reflect(scope)
memory.consolidate_topic(topic)
memory.explain(target)
memory.verify()
```

The agent should **not** get:

```text
trusted_write(...)
delete_fact(...)
overwrite_belief(...)
bypass_provenance(...)
```

## Tools that help organize and compress over time

These are the important “agent call” primitives:

| Tool                                 | What it does                                          |
| ------------------------------------ | ----------------------------------------------------- |
| `memory.reflect(scope)`              | Creates lessons, belief revisions, concept candidates |
| `memory.consolidate_topic(topic)`    | Merges duplicates, creates/updates concept kernel     |
| `memory.create_atlas_slice(scope)`   | Builds compact always-loaded topic map                |
| `memory.find_duplicates(scope)`      | Finds redundant cards for merge proposals             |
| `memory.find_contradictions(scope)`  | Finds unresolved conflicts                            |
| `memory.generate_search_gaps(topic)` | Creates new search queries from weak coverage         |
| `memory.promote_skill(skill)`        | Promotes only after tests pass                        |
| `memory.record_failure(task)`        | Stores failed answers/retrievals as eval memory       |
| `memory.rehearse_due(n)`             | Reviews weak/high-value topics                        |
| `memory.audit_pack(pack_id)`         | Explains why a context pack was built                 |

These are what make memory compound instead of merely accumulate.

# 17. Context-Aware Retrieval Profiles

Different tasks need different scoring.

| Intent             | Top signals                                               | Required warnings                     |
| ------------------ | --------------------------------------------------------- | ------------------------------------- |
| Scientific claim   | source quality, support/contradiction, temporal validity  | stale evidence, contested claims      |
| Math derivation    | exact equation, assumptions, prerequisites, units         | unit mismatch, missing derivation     |
| Coding             | exact symbols, repo paths, skill success, recent failures | stale environment, untested skill     |
| Debugging          | error exact match, recent episodes, failed attempts       | workaround confidence                 |
| Literature review  | resource clusters, citation graph, topic coverage         | omitted sources, weak coverage        |
| Planning           | core constraints, open questions, skills, blockers        | low evidence, unresolved dependencies |
| Historical query   | valid-at time, transaction-time, superseded facts         | current-truth mismatch                |
| Memory self-repair | eval failures, bad retrievals, corrections                | repeated failure pattern              |

| dense vector top-64    |  < 15 ms |  < 50 ms | < 120 ms |

| fused recall k=32      |  < 50 ms | < 180 ms | < 500 ms |
| graph-heavy recall     | < 150 ms | < 600 ms |  < 1.5 s |
| context pack assembly  |  < 20 ms |  < 80 ms | < 200 ms |

## Scale strategy

For 10M+ memories:

```text
partition by tenant/project/domain/lane
time-bucket episodes
store hot concept kernels separately
quantize embeddings
use roaring bitmap filters
use CSR graph adjacency
use mmap indexes
use async index updates
use background compaction
cache atlas and recent context packs
```

## Hot/cold tiers

| Tier     | Contains                                  | Retrieval              |
| -------- | ----------------------------------------- | ---------------------- |
| Core hot | atlas, active topics, current project     | always loaded          |
| Hot      | trusted concepts, recent episodes, skills | fast index             |
| Warm     | claims/resources/equations                | hybrid retrieval       |
| Cold     | raw episodes, old low-utility details     | source rehydration     |
| Archive  | raw PDFs/data/files                       | content-addressed blob |

# 22. Testing Plan

## Unit tests

```text
schema round trips preserve schema_version
invalid status transitions rejected
fact supersession never destructively overwrites
belief cannot exist without evidence
claim requires source span
equation requires normalized form
skill promotion requires passing test
context pack respects token budget
privacy filters apply before retrieval
```

## Property tests

Use `proptest`:

```text
random mutation sequences preserve hash-chain integrity
random status transitions cannot reach invalid states
random context packs never exceed budget
random graph traversals never return inaccessible memories
random supersession chains preserve historical query correctness
```

## Integration tests

```text
ingest paper → extract claims → retrieve cited claim
ingest contradiction → old fact gets valid_to, not deletion
ingest failed code run → debugging query retrieves failure
promote repeated workflow → skill requires tests
rebuild all indexes → same manifests
historical query → returns fact valid at T
current query → excludes superseded fact
```

```text
prior failed answer
```

Targets:



# 26. Why This Beats a Wiki, Graph DB, or Vector DB

## Versus an LLM wiki

A wiki gives human-readable pages but not:

```text
temporal truth
typed claims
source-span enforcement
belief/fact separation
context-pack compilation
skill execution
eval-driven self-repair
```

MNEMOS-Σ can generate a wiki view, but the wiki is not memory.

## Versus a graph database

A graph DB gives relationships but not:

```text
raw episode provenance
context compression
exact equation search
procedural skill tests
source manifests
Git review
retrieval evals
```

MNEMOS-Σ uses a graph projection, but the graph is not memory.

## Versus vector search

Vector search gives semantic similarity but not:

```text
exact symbols
temporal reasoning
contradictions
entity relations
source trust
procedural memory
belief evolution
```

MNEMOS-Σ uses vectors, but vectors are not memory.

## Versus giant context

Giant context still has attention and relevance problems. The right answer is not “load more.” The right answer is **compile the smallest high-signal context needed for the task**. ([MIT Press Direct][7])

# 27. The Single Most Important Feature: The Memory Atlas

The memory atlas is what makes the system feel superhuman.

Without it, an agent searches only for what it already knows to search for.

With it, the agent starts each task with a compressed map:

```text
what topics exist
what sources exist
what contradictions exist
what skills exist
what gaps exist
what handles to call next
```

Example:

```markdown
# Memory Atlas Slice: Dataset Discovery

Known:
- Resource cards exist for arXiv, OpenAlex, Semantic Scholar, Zenodo, NASA ADS.
- Skills exist for arXiv metadata extraction and dataset manifest generation.
- Prior failure: dataset scripts were created without license checks.

Open gaps:
- Need DOI/license normalization.
- Need small preview-only data policy.
- Need checksum enforcement.

Suggested calls:
- memory.skills(intent="dataset_download")
- memory.eval_failures(task="dataset_manifest")
- memory.questions(topic="public physics datasets")
```

This is the difference between search and memory.

MNEMOS-Σ is:

```text
Hindsight-style epistemic networks
+ Graphiti-style bitemporal graph
+ A-MEM-style concept evolution
+ Mastra-style observation compression
+ Nemori-style event segmentation and predict-calibrate learning
+ Mem0-style low-latency hybrid recall
+ HippoRAG-style associative graph traversal
+ Voyager-style executable skills
+ GraphRAG/LightRAG-style corpus maps
+ Git-backed canonical cards
+ Rust-native indexes and daemons

```

The system should always be able to answer:

```text
Why does the agent believe this?
When did it learn it?
What source supports it?
What contradicts it?
What changed?
What skill can act on it?
What failed before?
What should it search next?
```

That is the standard for a compounding scientific memory system.

The epic version in one line:

> **Build a Rust memory compiler whose canonical truth is an auditable source-linked ledger, whose intelligence comes from temporal typed graph + hybrid retrieval + concept compression + executable skills, and whose agent interface is a context-pack compiler that gives the model a tiny, cited, contradiction-aware slice of a massive evolving memory.**
