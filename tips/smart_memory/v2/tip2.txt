# MNEMOS-Ω: the Superhuman Cognitive Memory Compiler

I unpacked and studied the archive, including `CLAUDE_MEMSPEC.md`, `CODEX_MEMSPEC.md`, and all 13 tips. The strongest common conclusion across the whole set is this:

> **The optimal agent memory system is not a vector DB, not chat history, and not a single graph. It is a typed, temporal, source-backed memory compiler whose fast indexes are rebuildable projections over an immutable cognitive ledger.**

The design below is the best version I would build: **MNEMOS-Ω**, a Rust-native, local-first, horizontally extensible memory OS for agents that need real-time learning, fast recall across millions of memories, scientific/math correctness, procedural skill compounding, and long-term knowledge compression.

It synthesizes the strongest ideas from the studied specs and current memory research: MIRIX for typed lanes, Hindsight for separating facts/experiences/beliefs, Graphiti for temporal truth, A-MEM for Zettelkasten evolution, HippoRAG for associative multi-hop activation, Mem0 for fast hybrid retrieval, Voyager/Memp for executable skill memory, Nemori for event segmentation and prediction-gap learning, MemoryBank/Generative Agents for reflection and decay, Memory-R1 for learned memory-management policy, and MemOS for lifecycle/tiering. MIRIX’s six-lane memory taxonomy and benchmark claims make it one of the best complete human-like memory references; Graphiti explicitly tracks changing facts and provenance in temporal context graphs; A-MEM dynamically builds linked Zettelkasten-style notes; HippoRAG combines KGs with Personalized PageRank for efficient multi-hop recall; Mem0 reports major latency/token reductions over full-context approaches; Voyager demonstrates compounding executable skill libraries. ([arXiv][1])

# 1. Executive design thesis

**MNEMOS-Ω is a memory compiler, not a memory database.**

It continuously compiles raw experience into:

1. **Episodic truth**: what happened, when, from what source.
2. **Semantic truth**: what claims, entities, equations, relations, and resources exist.
3. **Belief state**: what the agent currently believes, with evidence and contradiction pressure.
4. **Concept state**: compact, evolving knowledge kernels that summarize entire domains.
5. **Procedural state**: executable, tested, versioned skills that improve through feedback.
6. **Evaluation state**: which memories helped, failed, misled, or were ignored.

The core invariant:

```text
Raw history is never the memory.
Vectors are never the memory.
Summaries are never the source of truth.

The source of truth is:
  immutable episodes
+ typed memory objects
+ temporal/evidential edges
+ mutation receipts
+ source hashes/spans.

Everything else is a fast projection.
```

This is what makes it superhuman: humans remember by lossy reconstruction; MNEMOS-Ω reconstructs too, but with cryptographic provenance, exact temporal validity, formal checks, parallel retrieval, and active self-review.

# 3. The memory lanes

MNEMOS-Ω uses **13 typed lanes**. The first six are the MIRIX/Hindsight/Google-CMC core. The rest are necessary for superhuman research, math, coding, and long-horizon self-improvement.

| Lane | Name                      | Purpose                                                                    | Source of truth                               | Hot-path retrieval?                    |
| ---: | ------------------------- | -------------------------------------------------------------------------- | --------------------------------------------- | -------------------------------------- |
|    1 | **Core**                  | Stable identity, mission, constraints, user-approved long-term preferences | Human-approved cards                          | Always, tiny budget                    |
|    2 | **Working**               | Current task state, hypotheses, scratchpad, active plan                    | Ephemeral + checkpointed                      | Always                                 |
|    3 | **Sensory / Observation** | Recent raw tool/user/file events before extraction                         | Ring buffer + episode ledger                  | Recent only                            |
|    4 | **Episodic**              | What happened: runs, conversations, failures, discoveries, actions         | Immutable episode ledger                      | By time/entity/task                    |
|    5 | **Semantic**              | Atomic facts, claims, relations, definitions                               | Temporal fact graph                           | Yes                                    |
|    6 | **Belief**                | Current stance over evidence: accepted, contested, revised, retired        | Belief state with support/contradiction links | When judgment needed                   |
|    7 | **Concept**               | A-MEM/Zettelkasten concept kernels, prerequisites, analogies               | Concept cards + graph                         | Yes                                    |
|    8 | **Procedural**            | Executable skills, workflows, scripts, tool recipes                        | Versioned code + tests + outcomes             | For action/debug tasks                 |
|    9 | **Resource**              | Papers, repos, datasets, files, APIs, external artifacts                   | Content-addressed refs                        | Exact/citation retrieval               |
|   10 | **Formal**                | Equations, theorem DAGs, unit constraints, proofs, derivations             | Formal cards + optional Lean/SymPy hooks      | Science/math tasks                     |
|   11 | **Counterexample**        | Falsified beliefs, failed assumptions, bad prior answers                   | Never decays                                  | Always considered for matching topics  |
|   12 | **Evaluation**            | Retrieval/answer/skill outcomes, what worked or failed                     | Eval cards + traces                           | Used for reranking and self-correction |
|   13 | **Distillate**            | Compressed topic/community/parametric memory for speed                     | Rebuildable projection, not truth             | Hot acceleration only                  |

**Key difference from ordinary memory stacks:** a claim, a belief, an equation, a failed attempt, a skill, and a paper are not interchangeable “chunks.” They have different schemas, update rules, trust rules, and retrieval rules.

# 4. Core invariants

These are non-negotiable.

```text
I1. No fact is overwritten destructively.
    New evidence creates supersession, contradiction, or belief revision.

I2. Every trusted memory has provenance.
    Source URI, source hash, extractor, span/page/line, timestamp.

I3. Every answer context is compiled.
    The agent receives a bounded ContextPack, not raw search results.

I4. Hot recall requires no LLM call.
    LLM extraction/reflection is write-side or background-side.

I5. Every index is rebuildable.
    Tantivy, HNSW, graph adjacency, concept communities, caches are projections.

I6. Counterexamples never decay out of safety memory.
    A system that forgets falsified assumptions repeats mistakes.

I7. Skills must execute or be downgraded.
    Untested code is not trusted procedural memory.

I8. Compression never destroys auditability.
    Summaries point back to exact atomic memories and sources.

I9. Memory-management policy is learned, but bounded by rules.
    RL can choose ADD/UPDATE/NOOP, but cannot bypass provenance, privacy, or temporal invariants.

I10. Memory is measured by downstream utility, not retrieval similarity alone.
```

# 5. Storage strategy

For the best Rust-first architecture, use **canonical immutable objects + rebuildable projections**.

## 5.1 Canonical local stack

| Need                       | Engine                                                  | Why                                                              |
| -------------------------- | ------------------------------------------------------- | ---------------------------------------------------------------- |
| Immutable mutation ledger  | Custom append-only CBOR/bincode WAL + BLAKE3 hash chain | Fast, audit-safe, replayable                                     |
| Canonical object store     | `redb` typed tables                                     | Pure Rust, transactional, local-first                            |
| Full-text and exact search | `tantivy`                                               | Rust-native BM25; great for equations, symbols, IDs, code paths  |
| Vector search              | Rust HNSW, quantized embeddings, optional adapter       | Fast local semantic recall                                       |
| Graph                      | Custom compressed adjacency + temporal interval indexes | Faster and more controllable than generic graph DB at 10M+ scale |
| Bitmaps                    | `roaring`                                               | Fast filters by entity/tag/lane/time/project                     |
| Embedding inference        | `candle` + cached external adapter                      | Rust-native local embedding path                                 |
| Formal hooks               | optional subprocess/tool adapters                       | Lean/SymPy/Coq are optional verifiers                            |
| Human audit cards          | Git-backed Markdown/YAML cards                          | Reviewable, diffable, source-controlled                          |
| Large artifacts            | content-addressed file store                            | Keeps DB small and backup-friendly                               |

Optional adapters can exist for Qdrant, Postgres/pgvector, Kùzu, Neo4j, LanceDB, or object storage, but **the canonical contract must not depend on them**.

## 7.2 Temporal validity and provenance

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TemporalValidity {
    pub observed_at: DateTime<Utc>,
    pub valid_from: Option<DateTime<Utc>>,
    pub valid_to: Option<DateTime<Utc>>,
    pub transaction_time: DateTime<Utc>,
}

impl TemporalValidity {
    pub fn is_live_at(&self, t: DateTime<Utc>) -> bool {
        let after_start = self.valid_from.map(|v| t >= v).unwrap_or(true);
        let before_end = self.valid_to.map(|v| t < v).unwrap_or(true);
        after_start && before_end
    }
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SourceSpan {
    pub uri: String,
    pub page: Option<u32>,
    pub section: Option<String>,
    pub byte_start: Option<u64>,
    pub byte_end: Option<u64>,
    pub line_start: Option<u32>,
    pub line_end: Option<u32>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Provenance {
    pub sources: Vec<SourceSpan>,
    pub source_hashes: Vec<[u8; 32]>,       // BLAKE3
    pub extractor_id: String,
    pub extractor_version: String,
    pub exact_quote: Option<String>,
    pub confidence: f32,
    pub grounding_status: GroundingStatus,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum GroundingStatus {
    Ungrounded,
    SourceLinked,
    SpanVerified,
    HumanApproved,
    FormallyVerified,
}
```

# 9. Write pipeline

The write path can be slower than recall, but it must be safe, incremental, and never corrupt the memory base.

```text
observe
  → segment
  → append episode
  → extract observations
  → route by lane
  → normalize entities
  → ground provenance
  → detect duplicates
  → detect contradictions
  → score importance/novelty/surprise
  → policy decision: ADD / UPDATE / NOOP / SUPERSEDE / REVIEW
  → commit canonical mutation
  → update projections
  → schedule consolidation
```

## 9.1 Fast write modes

| Mode               | Latency target | What is committed                     |
| ------------------ | -------------: | ------------------------------------- |
| `observe_async`    |     p50 < 1 ms | Episode + raw observation receipt     |
| `observe_indexed`  |    p50 < 10 ms | Episode + BM25 + entity hints         |
| `observe_semantic` | p50 100 ms–2 s | Extracted claims/skills/equations     |
| `observe_trusted`  |       variable | Human/formal/source-verified memories |

This is crucial: **real-time learning does not require blocking on LLM extraction**. The agent can recall the fresh episode immediately, while deeper semantic extraction and consolidation run in background.

## 9.2 Contradiction update rule

```text
Old fact:
  Alice works_at CompanyA
  valid_from = 2026-01-01
  valid_to   = null
  status     = Trusted

New evidence:
  Alice works_at CompanyB
  observed_at = 2026-05-12
  source_span = ...

Mutation:
  old.valid_to = 2026-05-12
  old.status   = Superseded
  new.valid_from = 2026-05-12
  new.status = Trusted
  edge(new, old) = Supersedes
  edge(new, old) = Contradicts if mutually exclusive
  belief(Alice employer) revised with receipt
```

No stale fact silently disappears. Historical queries can still ask what was true at an earlier time.

# 10. Read pipeline

Hot recall must be deterministic, bounded, local-first, and LLM-free.

```text
recall(query)
  → classify intent
  → parse time scope, domain, symbols, entities
  → fetch pinned core + active working state
  → parallel fanout:
       exact/BM25
       vector HNSW
       entity bitmap
       temporal facts
       concept graph
       procedural skills
       eval/counterexample memory
       cached context packs
  → reciprocal rank fusion
  → bounded graph activation
  → temporal and privacy filtering
  → evidence/utility reranking
  → context pack compilation
  → access logging + strengthening
```

## 10.1 Retrieval channels

| Channel             | Purpose                             | Data structure                |
| ------------------- | ----------------------------------- | ----------------------------- |
| Core                | Always-on constraints               | direct key lookup             |
| BM25/exact          | Symbols, IDs, equations, code paths | Tantivy                       |
| Dense semantic      | Similar concepts and paraphrases    | HNSW                          |
| Entity              | People, papers, datasets, variables | Roaring bitmaps               |
| Temporal graph      | Facts valid at time T               | interval index + CSR graph    |
| Concept graph       | Topic kernels and prerequisites     | CSR + community cache         |
| Hippo activation    | Multi-hop science reasoning         | bounded Personalized PageRank |
| Skills              | Executable procedures               | precondition index + HNSW     |
| Eval/counterexample | Prior failures and corrections      | eval index                    |
| Distillate cache    | Fast compressed prior packs         | context cache                 |

## 10.2 Scoring function

```text
score =
  0.18 * exact_match
+ 0.14 * bm25_norm
+ 0.16 * dense_similarity
+ 0.12 * entity_overlap
+ 0.12 * graph_activation
+ 0.08 * temporal_validity
+ 0.07 * source_quality
+ 0.06 * utility_success
+ 0.04 * topic_strength
+ 0.03 * freshness
- 0.10 * contradiction_penalty
- 0.08 * privacy_or_scope_penalty
```

Weights are query-intent dependent. A math derivation boosts exact equation and prerequisite chains. Debugging boosts recent failed attempts. Literature review boosts resources, citation clusters, and contested evidence.

## 10.3 ContextPack contract


A valid ContextPack must satisfy:

```text
- total tokens <= budget

- omitted high-value evidence is reported
- order is deterministic for fixed memory state
```

# 11. Knowledge compression over time

This is the biggest win. To scale to millions of memories, MNEMOS-Ω compresses knowledge into layers without destroying provenance.

## 11.1 Compression pyramid

```text
Layer 0: Raw artifacts
  PDFs, repos, datasets, screenshots, logs.
  Stored by content hash. Not used directly unless needed.

Layer 1: Episodes
  "What happened."
  Immutable, timestamped, hash-chained.

Layer 2: Atomic memories
  Claims, equations, skills, resources, failures, observations.
  Small, typed, source-backed.

Layer 3: Concept cards
  A-MEM-style linked notes.
  Definitions, prerequisites, examples, contradictions, open questions.

Layer 4: Topic kernels / community reports
  GraphRAG-style compressed summaries of whole subgraphs.
  Include top claims, equations, sources, gaps, and skill links.

Layer 5: Distillate cache
  Rebuildable acceleration layer:
    - query templates
    - context pack cache
    - topic embeddings
    - optional small parametric adapter
```

GraphRAG’s global/local/community-summary pattern is useful here: large corpora need both local evidence and higher-level summaries so broad questions do not require reading thousands of chunks. Microsoft describes GraphRAG as building KGs and community summaries from unstructured text, then using those structures for retrieval. ([Microsoft][2])

## 11.2 Consolidation jobs

| Job                   |                           Frequency | Action                                            |
| --------------------- | ----------------------------------: | ------------------------------------------------- |
| Micro-consolidation   |               after important event | create observations, update active topic strength |
| Link evolution        |              every few minutes idle | A-MEM-style link/refine/merge                     |
| Belief revision       | after contradiction or new evidence | recompute stance and confidence                   |
| Concept kernel update |                         hourly/idle | summarize topic clusters                          |
| Duplicate merge       |                               daily | merge near-identical cards                        |
| Archive/decay         |                               daily | move low-utility memories out of hot indices      |
| Self-quiz             |                                idle | generate tests for high-value weakening topics    |
| Skill distillation    |            after repeated successes | promote workflows to skills                       |
| Counterexample sweep  |                               daily | ensure failures attach to concepts/skills         |
| Parametric distillate |                      weekly/offline | train/cache small acceleration model              |

## 11.3 Topic strength

Topic strength is not a mysterious embedding score. It is explainable.

```text
decayed =
  old_strength * exp(-hours_since_update / half_life_hours)

reinforcement =
  w_recency        * recency
+ w_recurrence     * independent_episode_count
+ w_utility        * downstream_success
+ w_novelty        * prediction_gap
+ w_source_quality * evidence_quality
+ w_retrieval      * successful_recall_rate
+ w_graph          * neighbor_strength
- w_conflict       * unresolved_contradiction_pressure
- w_failure        * bad_answer_pressure

new_strength = clamp(decayed + reinforcement, 0, 1)
```

This lets the agent answer:

```text
Why is this topic considered strong?
What evidence supports it?
What contradictions remain?
When should it be reviewed?
Which memories are canonical?
Which skills depend on it?
```

MemoryBank’s Ebbinghaus-inspired forgetting/reinforcement and Generative Agents’ memory/reflection/planning loop are still foundational patterns for this kind of long-running behavior, but MNEMOS-Ω makes them typed, source-backed, and scalable. ([arXiv][3])

# 12. Procedural skill compounding

Procedural memory is not a note. It is an executable, versioned, tested artifact.

A skill can become trusted only if:

```text
- It has clear preconditions.
- It has a task signature.
- It links to successful episodes.
- It has tests or verification.
- It has observed success/failure counts.
- Failures are recorded and retrievable.
- Code is sandboxed and AST-checked.
```

## Example skill lifecycle

```text
Episode 1:
  Agent downloads arXiv paper manually.

Episode 2:
  Agent repeats same workflow.

Episode 3:
  Agent repeats workflow and succeeds.

Consolidator:
  Detects repeated successful action pattern.
  Creates SkillCard: "fetch_arxiv_pdf".
  Generates Rust/Python/shell implementation.
  Adds tests: valid ID, invalid ID, network failure.
  Runs tests in sandbox.

Skill status:
  Draft → Tested → Trusted.

Later:
  Network API changes.
  Skill fails.
  Failure episode is attached.
  Skill becomes Degraded.
  Repair loop patches code.
  Tests pass.
  Skill returns to Trusted.
```

Voyager’s skill-library approach is the key model here: it stores executable code skills, retrieves them later, and improves through environment feedback and self-verification. ([arXiv][4])

# 13. Belief revision and scientific truth

The most dangerous failure mode in AI memory is treating a source assertion as an agent belief.

MNEMOS-Ω separates:

```text
Observation:
  "Paper S reports result R under method M."

Claim:
  "Result R supports proposition P under assumptions A."

Belief:
  "The agent currently accepts P with confidence C because
   claims C1..Cn support it and claims K1..Km contradict it."

ContextPack:
  "For this task, include P, evidence, contradictions, uncertainty,
   and the old superseded state if historically relevant."
```

Hindsight is especially valuable here because it explicitly distinguishes world facts, experiences, synthesized entity summaries, and evolving beliefs, with retain/recall/reflect operations; Memory-R1 adds a learned manager for memory operations like ADD, UPDATE, DELETE, and NOOP rather than relying only on fixed heuristics. ([arXiv][5])

# 14. Performance and scale targets

These are engineering targets for a single high-end local machine with NVMe and 32–64 GB RAM. They are targets, not benchmark results.

| Operation                            | 1M memories target | 10M memories target | Notes                 |
| ------------------------------------ | -----------------: | ------------------: | --------------------- |
| `observe_async`                      |       p50 < 0.5 ms |          p50 < 1 ms | episode only          |
| `observe_indexed`                    |         p50 < 5 ms |         p50 < 10 ms | exact/entity indexing |
| common `recall`                      |        p50 < 20 ms |         p50 < 50 ms | no LLM                |
| recall with bounded graph activation |        p50 < 60 ms |        p50 < 150 ms | PPR subgraph only     |
| p99 common recall                    |           < 150 ms |            < 400 ms | warm indexes          |
| ContextPack compile                  |            < 10 ms |             < 30 ms | after candidates      |
| nightly consolidation                |            < 2 min |            < 20 min | background            |
| full index rebuild                   |           < 20 min |              < 4 hr | offline               |

## 14.1 Why this scales

At 10M memories, the system does not keep everything hot.

```text
Hot tier:
  100k–500k high-utility memories.
  fp16 embeddings, fast graph cache, context pack cache.

Warm tier:
  1M–3M useful memories.
  int8/quantized vectors, BM25, graph adjacency.

Cold tier:
  remaining memories.
  exact search + compressed topic references.
  no expensive dense hot cache.

Frozen/archive:
  old low-utility memory.
  source remains available.
  represented by concept kernels.
```

## 14.2 Vector memory budget

A naive 768-d fp32 embedding for 10M memories costs about 30 GB before index overhead. MNEMOS-Ω avoids that by storing:

```text
- all-memory summary embedding: 384-d int8 ≈ 384 bytes each
- hot-memory detailed embedding: fp16 or fp32 cache
- concept/topic centroid embeddings
- multi-vector only for high-value concepts/resources
```

The important trick: **most retrieval happens over compressed summaries and typed metadata, not raw chunks.**

# 16. Testing strategy

## 16.1 Unit tests

```text
- memory ID round trips
- schema serde compatibility
- temporal validity logic
- invalid lifecycle transitions rejected
- provenance required for Trusted state
- contradiction creates supersession instead of overwrite
- belief cannot be promoted without evidence
- skill cannot become Trusted without tests
- equation unit mismatch creates warning/failure
- ContextPack never exceeds token budget
```

## 16.2 Property tests

```text
- random mutation logs replay to deterministic state
- random fact updates never produce two live mutually exclusive facts
- random context packs preserve citation invariants
- random archive/restore preserves source reachability
- index rebuild equals canonical object counts
```

## 16.3 Integration tests

```text
Test: temporal contradiction
  Ingest "X is true" at T1.
  Ingest "X is false" at T2.
  Query now: returns false + supersession note.
  Query at T1: returns true.

Test: scientific evidence
  Ingest paper, claim, equation, dataset.
  Query claim.
  ContextPack includes source span, confidence, assumptions, contradictions.

Test: coding skill
  Run workflow 3 times successfully.
  Consolidator promotes skill.
  Skill fails later.
  Skill downgraded and repair queued.

Test: self-correction
  Agent gives bad answer.
  Human correction arrives.
  Eval memory created.
  Future similar query retrieves correction and prior failure.

Test: compression
  Ingest 100k topic-related memories.
  Build concept kernel.
  Recall broad topic under 4k token budget.
  Pack includes summary + top evidence + omitted-evidence note.
```

## 16.4 Benchmarks

```text
- LoCoMo-style long conversational memory

- synthetic temporal fact benchmark
- synthetic contradiction benchmark
- Math-Memory-Bench:

    bug recurrence, skill recall, prior failure avoidance
```

# 19. Example: how it behaves

## Scenario: scientific contradiction

```text
Day 1:
  Agent reads Paper A:
    "Method M works under assumption linearity."
  Stores:
    ResourceCard(Paper A)
    ClaimCard(M works under linearity)
    ConceptKernel(Method M)
    BeliefState(M is valid in linear regime)

Day 10:
  Agent runs experiment:
    Method M fails on nonlinear dataset D.
  Stores:
    Episode(experiment failure)
    CounterexampleCard(falsifies naive use of M)
    ClaimCard(M fails under nonlinear D)
    BeliefState revised:
      "M is valid only under linearity; contested outside."

Future query:
  "Use Method M on dataset D."
  ContextPack includes:
    - Method M skill if available
    - assumption warning
    - nonlinear failure counterexample
    - revised belief
    - source spans
```

The system does not simply remember “Method M works.” It remembers where, why, under what assumptions, when that belief changed, and what failed.

## Scenario: coding skill compounding

```text
Agent repeatedly fixes Rust async trait object errors.

After 3 successful episodes:
  Skill detector proposes:
    SkillCard("fix_async_trait_object_send_sync")
    Preconditions:
      - Rust async_trait
      - trait object crossing tokio task boundary
    Steps:
      - add Send + Sync bounds
      - use Arc<dyn Trait + Send + Sync>
      - check lifetime bounds
    Tests:
      - compile fixture crate
      - run clippy
  Skill becomes Trusted.

Future compile error:
  Recall query hits exact error in BM25,
  skill precondition match,
  prior failure eval,
  ContextPack says:
    "You have a trusted skill for this. Apply it."
```

This is how the system becomes better at doing, not merely remembering.

# 20. Final engineering verdict

**MNEMOS-Ω is the best design I can justify from the archive and current research:**

```text
Hindsight epistemic separation
+ MIRIX typed lanes
+ Graphiti temporal truth
+ A-MEM concept evolution
+ HippoRAG graph activation
+ Mem0 hot-path retrieval discipline
+ Voyager/Memp executable skills
+ Nemori event segmentation and surprise learning
+ MemoryBank/FSRS strengthening and decay
+ Memory-R1 learned memory policy
+ GraphRAG-style community compression
+ Rust-native local-first systems engineering
```

The essential breakthrough is **not** adding more retrieval. It is making memory a self-improving compiler:

```text
experience → episode → typed memory → temporal graph → concept kernel
→ skill / belief / counterexample → compressed context pack
→ action → feedback → stronger memory
```

That loop is what gives the agent **real-time learning**, **fast recall**, **knowledge compounding**, **scientific caution**, **procedural mastery**, and **scalable compression across millions of memories**.
