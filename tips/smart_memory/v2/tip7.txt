I unpacked and studied the archive: `CLAUDE_MEMSPEC.md`, `CODEX_MEMSPEC.md`, `tip1` through `tip13`, and the pasted ANTIGRAVITY/CMC design. The strongest pattern is unmistakable: the winning system is **not** a vector DB, not a wiki, not a graph DB, and not a giant context window. It is a **Rust memory compiler** that turns raw experience into typed, temporal, provenance-backed, compressed, queryable, and executable memory.

Below is the design I would build.

# MNEMOS-II: Rust Superhuman Cognitive Memory Compiler

## 0. The core thesis

The optimal future memory system is a **compiler**, not a database.

A database stores things. A compiler transforms one representation into a more useful representation. A superhuman memory system must continuously compile:

```text
raw experience
в†’ observations
в†’ episodes
в†’ claims / equations / entities / skills
в†’ temporal graph facts
в†’ belief states
в†’ concept kernels
в†’ compressed maps
в†’ executable procedures
в†’ bounded context packs
в†’ outcome feedback
в†’ stronger future retrieval
```

Long context windows do not solve this by themselves. вЂњLost in the MiddleвЂќ showed that models can fail to reliably use information buried in long contexts, with performance often best near the beginning or end of the prompt. That is the reason memory must select, rank, compress, and structure evidence before the model sees it. ([MIT Press Direct][1])

The public research and systems landscape points to the same synthesis: MIRIX gives typed cognitive lanes; Graphiti/Zep gives temporal validity and provenance; Hindsight separates facts, experiences, observations, and beliefs; A-MEM gives self-organizing Zettelkasten-style concept evolution; Mastra Observational Memory gives stable low-latency compressed context; HippoRAG gives graph activation and Personalized PageRank-style multi-hop recall; RAPTOR and GraphRAG give hierarchical and community-level compression; Voyager gives executable procedural skill memory. ([Mirix Documentation][2])

The design below is my best synthesis.

---

# 1. The architecture in one sentence

**MNEMOS-II is a Rust-first, event-sourced, multi-lane cognitive memory compiler that stores canonical truth in append-only, provenance-backed records; maintains fast rebuildable projections for BM25, vector, graph, temporal, symbol, and skill retrieval; continuously compresses knowledge into concept kernels and stable observation prefixes; and returns small, cited, contradiction-aware context packs instead of overwhelming the model.**

---

# 2. What I would improve over CLAUDE, CODEX, and CMC

The uploaded specs are already extremely strong. I would keep their core insights and tighten five things.

First, I would make **event sourcing absolutely central**. The append-only mutation ledger is not a phase-one convenience; it is the source of truth. All databases and indexes are projections. This guarantees deterministic rebuilds, reproducibility, auditability, and safe experimentation.

Second, I would separate **fact, belief, observation, and answer** with no exceptions. Hindsight is especially right on this: a fact is not a belief, and an extracted source assertion is not agent knowledge. HindsightвЂ™s public description emphasizes structured memory networks for world facts, agent experiences, synthesized observations, and evolving beliefs, governed by retain, recall, and reflect operations. ([Hugging Face][3])

Third, I would combine **Mastra-style stable observation context** with **dynamic retrieval**. MastraвЂ™s Observational Memory is compelling because it maintains a stable, prompt-cacheable observation prefix while compressing message history through Observer and Reflector agents. That gives speed. But for science, math, and code, stable observations alone are not enough; the system must also retrieve exact equations, citations, counterexamples, source spans, code paths, and graph neighborhoods. ([Mastra][4])

Fourth, I would make **science/math objects first-class**, not generic notes. Equations, theorems, assumptions, regimes of validity, units, datasets, observables, predictions, experiments, and counterexamples need typed schemas and validators.

Fifth, I would make the hot path **zero-LLM by default**. LLMs can extract, reflect, compress, and propose mutations in background jobs. The realtime recall path must run locally from compiled indexes: exact search, vector search, graph activation, temporal filtering, score fusion, and context packing.

# 4. The non-negotiable design rule

```text
Canonical truth is append-only and auditable.
Indexes are disposable.
Summaries are useful but never authoritative.
LLM outputs are proposals until validated.
Every factual memory must be traceable to evidence.
Every belief must be traceable to claims and contradictions.
Every skill must be traceable to tests.
```

This is how you get compounding intelligence without memory corruption.

# 5. Memory lanes

MNEMOS-II uses **12 logical lanes**. They are not necessarily 12 separate databases; they are typed views over a common memory object system.

MIRIXвЂ™s public docs describe six memory components: core, episodic, semantic, procedural, resource, and knowledge, with BM25 and embedding search. MNEMOS-II keeps that shape but adds belief, eval, counterexample, theorem/equation, curriculum, and metacognitive lanes for serious science and engineering work. ([Mirix Documentation][2])

| Lane                 | Purpose                                                                |             Hot-path role | Canonical? |
| -------------------- | ---------------------------------------------------------------------- | ------------------------: | ---------: |
| `Core`               | Tiny always-loaded mission, user/project constraints, current frontier |           Always included |        Yes |
| `Working`            | Current task scratch state, active plan, temporary hypotheses          |                    In RAM |     Partly |
| `Episodic`           | Immutable event log: what happened, when, with which tools             |        Time/entity recall |        Yes |
| `Observation`        | Compressed dated observations from raw episodes                        |    Stable prefix + recall |        Yes |
| `Semantic`           | Atomic claims, facts, definitions, source assertions                   |        Evidence retrieval |        Yes |
| `Belief`             | Current stance over claims, with support and contradiction             |           Judgment recall |        Yes |
| `Concept`            | Self-organizing concept kernels, maps, prerequisites                   | Compression and synthesis |        Yes |
| `Procedural`         | Executable skills, workflows, scripts, tests                           |             Action recall |        Yes |
| `Resource`           | Papers, datasets, repos, APIs, file manifests, licenses                |       Exact source recall |        Yes |
| `EquationTheorem`    | Equations, variables, units, theorem DAGs, proof status                | Math/science exact recall |        Yes |
| `EvalCounterexample` | Failures, bad answers, falsified assumptions, benchmark outcomes       | Prevent repeated mistakes |        Yes |
| `MetaCurriculum`     | Lessons about how to learn/reason; topic mastery and next questions    |          Self-improvement |        Yes |

# 6. Memory compression hierarchy

This is the central trick for millions of memories.

```mermaid
flowchart BT
    R0[Raw tool output / messages / PDFs / code logs] --> R1[Episodes]
    R1 --> R2[Observations]
    R2 --> R3[Atomic claims / equations / skills / resources]
    R3 --> R4[Concept kernels]
    R4 --> R5[Domain maps]
    R5 --> R6[Core frontier memory]

    R3 --> G[Temporal Knowledge Graph]
    R4 --> T[RAPTOR Summary Trees]
    R4 --> C[GraphRAG Community Reports]
    R3 --> S[Procedural Skill Library]

    G --> P[Context Pack]
    T --> P
    C --> P
    S --> P
    R6 --> P
```

Each layer has a different job:

```text
Raw artifacts       = full source, never blindly stuffed into context
Episodes            = immutable event history
Observations        = compressed autobiographical memory
Atomic objects      = searchable facts, equations, claims, skills
Concept kernels     = durable conceptual compression
Domain maps         = navigational memory
Core frontier       = tiny always-visible strategic memory
```

RAPTOR-style recursive clustering and summarization is ideal for document-scale compression because it retrieves over both leaf chunks and higher-level summaries. ([ICLR Proceedings][5]) GraphRAG-style community summaries are ideal for corpus-level вЂњwhat are the main themes?вЂќ questions. ([Microsoft][6])

# 7. Source-of-truth model

## 7.1 Canonical truth

Canonical memory lives in four places:

```text
1. append-only mutation ledger
2. redb object store
3. Git-backed curated cards
4. external resource vault pointers
```

I would use `redb` for the pure-Rust embedded store because it is an ACID, MVCC, crash-safe, copy-on-write B-tree database written in Rust. ([GitHub][7])

## 7.2 Rebuildable projections

These are not truth:

```text
- Tantivy BM25 indexes
- HNSW vector indexes
- temporal graph adjacency cache
- equation/symbol inverted index
- topic-strength cache
- context-pack cache
- RAPTOR trees
- GraphRAG community summaries
```

Tantivy is the Rust Lucene-like full-text search engine I would use for BM25, phrase queries, and exact-symbol retrieval. ([Docs.rs][8]) For local vector search, `hnsw_rs` is a Rust HNSW implementation with insert/search APIs and file dump/reload support. ([Docs.rs][9]) For embeddings, `fastembed` gives a Rust retrieval-embedding interface with supported text embedding models. ([Docs.rs][10])

At larger team scale, optional adapters can target Qdrant, LanceDB, Postgres/pgvector, or Kuzu. Qdrant supports dense and sparse vectors, LanceDB supports vector, full-text, SQL, and hybrid search, and Kuzu is an embedded property graph database with Cypher, disk-backed storage, and Rust API support. ([Qdrant][11])

# 12. Write path

The write path is a compiler pipeline. It should accept raw events quickly, then perform extraction and consolidation asynchronously.

```mermaid
sequenceDiagram

    participant Validator
```

# 13. Read path: the Context Pack Compiler

The model should not retrieve вЂњchunks.вЂќ It should request a **ContextPack**.

```mermaid
flowchart LR
    Q[RecallQuery] --> P[Query Parser]
    P --> I[Intent + Time + Entities + Symbols]
    I --> R1[Core/Stable Prefix]
    I --> R2[BM25 Exact]
    I --> R3[Vector ANN]
    I --> R4[Entity Resolver]
    I --> R5[Temporal Graph]
    I --> R6[Hippo PPR]
    I --> R7[Symbol / Equation]

    I --> R9[Eval / Failures]
    I --> R10[Concept / RAPTOR / Community Summaries]

    R1 --> F[Candidate Fusion]
    R2 --> F
    R3 --> F
    R4 --> F
    R5 --> F
    R6 --> F
    R7 --> F
    R8 --> F
    R9 --> F
```

HippoRAGвЂ™s key retrieval insight is to combine LLM-built knowledge graphs with Personalized PageRank-style activation for efficient multi-hop retrieval. ([Hugging Face][13]) MNEMOS-II uses that as one retrieval channel, not as the whole memory system.

## 13.3 Score fusion

Use Reciprocal Rank Fusion plus calibrated features:

```text
score =
  0.18 * rrf_channel_score
+ 0.15 * exact_match_score
+ 0.13 * semantic_similarity
+ 0.12 * entity_overlap
+ 0.11 * temporal_validity
+ 0.10 * graph_activation
+ 0.08 * source_quality
+ 0.07 * topic_strength
+ 0.06 * retrieval_success_ema
+ 0.05 * utility
+ 0.04 * recency

- 0.08 * stale_penalty
```

# 14. Stable prefix + dynamic retrieval

The system uses two recall modes simultaneously.

## 14.1 Stable prefix

A tiny stable prefix is compiled from:

```text
memory/ROOT.md
memory/ACTIVE_FRONTIER.md
latest high-priority observations
current task state
known hard constraints
```

MastraвЂ™s Observational Memory argues strongly for stable, prompt-cacheable context. Its Observer/Reflector architecture compresses raw history into dated observations and keeps the prompt predictable instead of rebuilding retrieval every turn. ([Mastra][4])

## 14.2 Dynamic ContextPack

For actual hard tasks, dynamic retrieval still happens:

```text
query в†’ exact/vector/entity/graph/symbol/skill/eval retrieval в†’ pack
```

This hybrid is better than either approach alone:

```text
stable prefix = speed and continuity
dynamic pack  = precision and depth
```

## 15.1 Mandatory agent behavior

For a serious autonomous agent, I would enforce this policy:

```text
Before work:
  recall(task)

During work:
  observe(tool calls, errors, discoveries)

After work:
  feedback(success/failure)
  observe(final result)
  propose candidate memories

During idle:
  reflect()
  consolidate()
  rehearse()
  resolve contradictions
  update topic strengths
  promote repeated successful workflows into skills
```

## 15.2 Best agent calls for organization and compression

The most useful calls are:

```text
observe(event)
  Records what happened. Fast, append-only.

compile_observations(scope)
  Compresses raw episodes into dated observations.

extract_candidates(episode)
  Proposes claims, equations, resources, skills, questions.

link_entities(candidates)
  Resolves aliases, merges duplicates, creates graph links.

validate_candidates(candidates)
  Checks source spans, units, status transitions, duplicate risk.

commit(mutations)
  Applies only valid memory mutations.

recall(query)
  Returns a bounded context pack.

reflect(scope)
  Creates concept kernels, belief updates, meta-lessons.

consolidate(topic)
  Compresses many low-level memories into concept/domain maps.

resolve_contradictions(topic)
  Finds live conflicting claims and updates belief states.

promote_skill(workflow)
  Turns repeated successful action traces into tested procedures.

log_outcome(pack_id, outcome)
  Teaches the retriever what helped and what hurt.

rehearse_next(n)
  Selects memories/topics that should be reviewed or strengthened.

  Shows provenance, why it exists, who/what supports it, and what contradicts it.
```

The agent gets smarter only when it closes the loop:

```text
retrieve в†’ act в†’ measure outcome в†’ update utility в†’ consolidate в†’ improve retrieval
```

# 20. Million-memory scale strategy

The target is millions to tens of millions of memory objects on commodity hardware.

## 20.1 Tiering

```text
Tier 0: Core
  Always loaded. Tiny. < 32 KB.

Tier 1: Hot
  Current project, active topics, recent high-utility observations.
  In RAM + redb + indexes.

Tier 2: Warm
  Trusted concept cards, claims, resources, skills.
  Local disk + projections.

Tier 3: Cold
  Raw episodes, old tool logs, low-utility traces.
  zstd-compressed, retrieved only by exact/time/source.

Tier 4: External
  PDFs, datasets, repos, raw data.
  Stored as pointers, manifests, checksums, download scripts.
```

## 20.2 Segmentation

```text
segment key = tenant / project / lane / domain / epoch
```

Example:

```text
project=openqg
lane=semantic
domain=physics.qft
epoch=2026-Q2
```

Each segment has:

```text
- ledger slice
- redb key range
- Tantivy segment
- HNSW shard
- temporal graph delta
- roaring bitmap filters
- checksum manifest
```

## 20.3 Query planning

A query does not search everything.

```text
1. parse intent
2. detect exact IDs/symbols/entities/time
3. choose relevant lanes
4. choose relevant segments
5. run fanout
6. fuse
7. expand graph only from strong seeds
8. pack
```

## 20.4 Compressed indexes

Use:

```text
- Roaring bitmaps for lane/tag/domain/time filters
- zstd for raw episodes
- f16 or int8 vectors for cold embeddings
- full precision vectors for hot/warm
- CSR adjacency for graph traversal
- append-only graph deltas before compaction
```

Roaring bitmaps are a strong fit for fast compressed set filtering because they are designed as compressed bitmaps and can be dramatically faster than conventional compressed bitmap approaches. ([Roaring Bitmap][16])

# 21. Hot-path latency targets

These are engineering targets, not guaranteed claims before benchmarking.

| Operation              | Target p50 |     Target p95 | Notes                 |
| ---------------------- | ---------: | -------------: | --------------------- |
| `observe(raw event)`   |     < 5 ms |        < 25 ms | Append only, no LLM   |
| `recall(simple exact)` |    < 30 ms |       < 120 ms | BM25/entity only      |
| `recall(hybrid)`       |   < 120 ms |       < 750 ms | BM25 + vector + graph |
| `recall(math exact)`   |   < 100 ms |       < 500 ms | Symbol index critical |
| `context_pack compile` |    < 20 ms |        < 80 ms | No LLM                |
| `feedback(outcome)`    |    < 10 ms |        < 50 ms | Updates utility/event |
| `reflect/consolidate`  | background |     background | LLM allowed           |
| `deep research recall` |     1вЂ“10 s | task-dependent | LLM multi-hop allowed |

The key is that only the hot path must be realtime. Reflection, contradiction review, extraction, and summarization can happen asynchronously.

# 24. Contradiction handling

Contradictions are first-class. The system must preserve them.

```mermaid
flowchart TD
    A[New Claim] --> B{Matches existing subject/predicate/scope?}
    B -- No --> C[Add claim]
    B -- Yes --> D{Compatible object/regime?}
    D -- Yes --> E[Add support edge / update confidence]
    D -- No --> F[Create contradiction edge]
    F --> G[Raise contradiction pressure]
    G --> H[Update belief state]
    H --> I{Old claim invalidated?}
    I -- Yes --> J[Close valid_to on old edge]
    I -- No --> K[Keep contested live claims]
```

Scientific domains often contain contested evidence. Automatically closing an old fact is only valid when the system has a clear supersession relation, such as a changed user preference, a replaced API endpoint, or a deprecated dataset version. For science, many contradictions stay live until a human or trusted verifier resolves them.

# 25. Belief update algorithm

```text
for each proposition P:
  collect supporting claims S
  collect contradicting claims C
  weight by:
    source_quality
    recency
    method quality
    replication count
    contradiction pressure
    domain trust policy
  update belief confidence
  record revision receipt
  never delete old stance
```

Pseudo-code:

```rust
pub fn update_belief(input: BeliefUpdateInput) -> BeliefRevision {
    let support = weighted_evidence(&input.supporting_claims);
    let contradiction = weighted_evidence(&input.contradicting_claims);
    let prior = input.current_confidence;

    let delta = sigmoid(support - contradiction) - 0.5;
    let posterior = clamp01(prior + input.learning_rate * delta);

    BeliefRevision {
        at: Utc::now(),
        reason: explain_delta(support, contradiction, posterior),
        confidence_before: prior,
        confidence_after: posterior,
        receipt_id: input.receipt_id,
    }
}
```

# 31. Security and memory poisoning rules

# 38. The final design principle

The memory system should make the agent smarter in four ways:

```text
1. Recall faster than a human.
2. Remember more accurately than a human.
3. Compress knowledge more aggressively than a human.
4. Preserve provenance better than a human.
```

But it should **not** pretend that summaries are truth.

The killer architecture is:

```text
event-sourced canonical memory
+ typed cognitive lanes
+ temporal knowledge graph
+ belief/fact separation
+ stable observation prefix
+ hybrid retrieval compiler
+ concept-kernel compression
+ procedural skill memory
+ science/math validators
+ feedback-driven utility learning
+ Git-readable curated maps
```

That is the system I would build.
