# MNEMOS-Ω: Rust Cognitive Memory Compiler

**Purpose:** a superhuman, local-first, Rust-native memory system for long-running research, coding, mathematics, science, and autonomous agents.

**Core thesis:** do not build “a memory database.” Build a **memory compiler**.

A database stores things. A compiler decides what the agent should see, when, why, how much, with what provenance, and what should happen after the agent acts.

The best design combines:

```text
MIRIX           -> cognitive memory taxonomy
Hindsight       -> observations vs facts vs beliefs vs mental models
Graphiti/Zep    -> temporal knowledge graph and stale-fact prevention
A-MEM           -> Zettelkasten self-organization and concept evolution
Mastra OM       -> low-latency observation compression
Mem0            -> practical fast hybrid retrieval and memory operations
HippoRAG        -> associative graph activation and multi-hop recall
Voyager/Memp    -> executable procedural skill compounding
MemoryBank/FSRS -> decay, strengthening, spaced review
GraphRAG        -> large corpus/topic maps
Claude/Codex specs -> Rust schemas, audit receipts, context packs, formal science support
ANTIGRAVITY CMC -> multi-lane compiler framing and retrieval compilation
```

My final architecture is **MNEMOS-Ω**, a typed, source-grounded, bitemporal, self-organizing, executable memory compiler.

# 1. Executive Design

## One-sentence design

**MNEMOS-Ω is a Rust-native cognitive memory compiler that converts raw experience into typed, source-backed, bitemporal memory objects; indexes them through lexical, vector, entity, temporal, graph, procedural, and topic projections; compresses them into concept kernels over time; and compiles tiny, cited, task-specific context packs for the agent in real time.**

## What makes it better than the prior specs

The Claude, Codex, and ANTIGRAVITY specs are already strong. MNEMOS-Ω keeps their best ideas and adds several missing hardening layers:

| Missing or underdeveloped area        | MNEMOS-Ω addition                                                                                          |
| ------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| Vector/corpus scale beyond hot memory | Two-tier vector system: hot exact HNSW + cold compressed IVF/PQ-like capsules.                             |
| Concept compression over years        | Multi-level compression ladder: raw events → observations → atoms → concept kernels → topic capsules.      |
| Retrieval failure learning            | First-class `EvalMemory` and negative retrieval traces used to improve scoring.                            |
| Agent-call ergonomics                 | Explicit memory tool contract: observe, recall, feedback, consolidate, reflect, rehearse, verify, explain. |
| Pure Rust local-first baseline        | `redb`/`fjall`-style canonical stores, Tantivy, Rust HNSW, mmap graph projections, no mandatory hosted DB. |
| Bitemporal truth                      | Every claim/edge tracks both **valid time** and **transaction time**, not just `valid_from`/`valid_to`.    |
| Million-memory context economy        | Context compiler prefers compressed concept kernels, only expands evidence on demand.                      |
| Skill reliability                     | Procedural memory is sandboxed, tested, versioned, and downgraded by failures.                             |
| Science/math rigor                    | Equation units, theorem DAGs, proof-state tracking, experiment cards, counterexample-first recall.         |
| Compression safety                    | Summaries are never truth. They are lossy projections backed by immutable source receipts.                 |

# 2. Core Principles

1. **Memory is not chat history.**
   Raw transcripts are evidence, not memory. Durable memory is typed, compressed, linked, and source-backed.

2. **Structured memory is truth; indexes are projections.**
   Vector stores, BM25, graph projections, topic maps, and context caches are rebuildable. The canonical truth is the ledger plus typed memory objects.

3. **Observation ≠ claim ≠ belief ≠ skill.**
   The system must know the difference between “we saw this,” “a source asserted this,” “we currently believe this,” and “we can execute this procedure.”

4. **Never destructively overwrite knowledge.**
   Changing facts create supersession, contradiction, and validity-window records.

5. **Compress upward; preserve downward.**
   Old detail should become compact kernels and capsules, but the original evidence must remain retrievable.

6. **Fast recall requires compiled context, not bigger search.**
   The agent receives a bounded `ContextPack`, not a dump of search hits.

7. **The memory system must learn from outcomes.**
   Every retrieval, answer, tool run, failed proof, benchmark, and corrected mistake updates memory utility and retrieval policy.

8. **Procedural memory must execute.**
   A “skill” is not a text note. It is a versioned, tested, sandboxed executable workflow with preconditions and reliability.

9. **Science memory needs epistemic machinery.**
   Claims need evidence, equations need assumptions and units, theorems need dependencies, experiments need reproducibility, beliefs need rationale.

10. **The agent must have memory tools.**
    The agent should be able to call explicit APIs to observe, recall, reflect, consolidate, rehearse, explain, verify, and correct memory.

# 4. Cognitive Memory Lanes

MNEMOS-Ω uses **14 logical lanes**. The first six mirror MIRIX/ANTIGRAVITY. The rest are needed for science, math, evaluation, and self-improvement.

| Lane             | Purpose                                                                  | Examples                                           | Hot-path use                       |
| ---------------- | ------------------------------------------------------------------------ | -------------------------------------------------- | ---------------------------------- |
| `Core`           | Stable identity, project constraints, mission, user-approved preferences | “Prefer local-first Rust,” active project goals    | Always considered; tiny budget     |
| `Working`        | Current task state and scratchpad                                        | active hypothesis, todo, temporary plan            | Always considered; expires quickly |
| `Sensory`        | Raw recent input before extraction                                       | tool output, screen frame, transcript chunk        | Short ring buffer                  |
| `Episodic`       | What happened, when, and where                                           | failed compile, search run, conversation event     | Debugging, temporal recall         |
| `Observation`    | Compressed dated observations                                            | “At 14:03, benchmark X failed due to Y”            | Fast continuity                    |
| `Semantic`       | Atomic claims, facts, definitions, relations                             | “Method X assumes linearity”                       | Normal factual recall              |
| `Belief`         | Current stance over evidence                                             | “I currently believe P with confidence 0.73”       | Judgment, planning, debate         |
| `Concept`        | Zettelkasten concept cards and kernels                                   | gauge invariance, diffusion models, Rust lifetimes | Learning, explanation, compression |
| `Procedural`     | Executable skills, workflows, scripts, playbooks                         | fetch arXiv, run benchmark, derive unit check      | Action selection                   |
| `Resource`       | External source ledger                                                   | paper DOI, repo commit, dataset hash               | Rehydration and provenance         |
| `Formal`         | Equations, theorems, proofs, derivations                                 | theorem DAG, equation units, Lean status           | Math/science reliability           |
| `Experiment`     | Reproducible empirical results                                           | parameters, code commit, data checksum, outcome    | Scientific learning                |
| `Counterexample` | Failures, falsifications, known traps                                    | “This proof strategy fails under condition R”      | Prevent repeated mistakes          |
| `Eval`           | Retrieval/answer/skill performance memory                                | bad answer, missing source, failed tool call       | Self-improvement                   |

The key separation is:

```text
Episode:      what happened
Observation:  compressed account of what happened
Claim:        what was asserted
Evidence:     why a claim is supported or contradicted
Belief:       what the agent currently accepts
Concept:      how claims form a reusable abstraction
Skill:        how to act
Eval:         whether recall/action worked
```

# 5. Memory Lifecycle

```mermaid
stateDiagram-v2
    [*] --> Proposed
    Proposed --> Rejected: invalid / duplicate / unsafe
    Proposed --> Trusted: provenance + validation pass
    Trusted --> Consolidated: repeated support + utility
    Consolidated --> Archived: low utility / cold topic
    Trusted --> Contradicted: strong counterevidence
    Consolidated --> Contradicted: strong counterevidence
    Trusted --> Superseded: newer valid claim
    Consolidated --> Superseded: newer valid claim
    Contradicted --> Revised: belief update
    Revised --> Trusted: evidence restored
    Archived --> Trusted: recalled and useful
    Rejected --> [*]
```

Important invariant:

```text
Superseded, contradicted, archived, and deprecated memories are not deleted.
They are excluded from normal recall unless historical, debugging, or counterexample recall requires them.
```

# 7. Canonical Storage Design

MNEMOS-Ω uses a strict separation:

```text
Canonical truth:
  immutable ledger + typed object store + resource CAS + signed receipts

Rebuildable projections:
  BM25, vector indexes, graph adjacency, topic caches, context-pack caches
```

## Default local-first storage stack

| Concern                       | Default                                              | Reason                                    |
| ----------------------------- | ---------------------------------------------------- | ----------------------------------------- |
| Immutable mutation ledger     | Framed append-only binary log + BLAKE3 Merkle chain  | Fast, replayable, tamper-evident          |
| Canonical objects             | `redb` or `fjall` key-value tables                   | Pure Rust local-first durability          |
| Human-readable curated memory | Git cards through pure-Rust `gix`                    | Diffs, review, branching                  |
| Full-text / BM25              | Tantivy                                              | Fast Rust-native lexical search           |
| Exact symbols/equations       | Tantivy symbol fields + trigram side index           | Vector search fails on symbols            |
| Hot vector search             | Rust HNSW                                            | Fast top-k over recent/high-utility items |
| Cold vector search            | Compressed vector capsules                           | Million-scale low-memory recall           |
| Graph                         | Custom temporal adjacency store + mmap CSR snapshots | Faster and lighter than generic graph DB  |
| Tag/time/entity filters       | Roaring bitmaps                                      | Very fast intersections                   |
| Raw resources                 | Content-addressed file store                         | Keeps binaries out of canonical object DB |
| Embeddings                    | Candle local embedder + adapter trait                | Local-first, no mandatory API             |
| Receipts                      | JSONL + signed binary receipts                       | Human readable + machine verifiable       |

# 8. Bitemporal Temporal Knowledge Graph

The Google/ANTIGRAVITY spec correctly emphasized `valid_from`/`valid_to`. MNEMOS-Ω goes further and uses **bitemporal records**:

| Time                      | Meaning                                                   |
| ------------------------- | --------------------------------------------------------- |
| `valid_from` / `valid_to` | When the proposition was true in the world                |
| `tx_from` / `tx_to`       | When the memory system believed/recorded that proposition |

This matters because an agent often learns late that something was false earlier.

Example:

```text
2026-01-01 world: Claim A becomes valid.
2026-05-12 agent: learns Claim A.
2026-06-01 world: Claim A stops being valid.
2026-06-15 agent: learns Claim A stopped being valid.
```

A simple `valid_to` model cannot answer both:

```text
What was true on 2026-05-20?
What did the agent believe on 2026-05-20?
```

MNEMOS-Ω can answer both.

## Temporal edge schema

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TemporalEdge {
    pub id: EdgeId,

    pub subject: EntityId,
    pub predicate: Predicate,
    pub object: EntityOrLiteral,

    pub validity: BitemporalValidity,

    pub confidence: f32,
    pub support: Vec<EvidenceLink>,
    pub contradiction: Vec<EvidenceLink>,

    pub supersedes: Vec<EdgeId>,
    pub superseded_by: Vec<EdgeId>,

    pub source_memories: Vec<MemoryId>,
    pub provenance: Provenance,
}
```

## Supersession rule

```text
Never mutate old fact semantics.

When new evidence supersedes a live edge:
1. Close old edge.valid_to if world-validity changed.
2. Close old edge.tx_to when memory transaction changes.
3. Insert new edge.
4. Link new edge with Supersedes.
5. Create Counterexample or Contradiction memory if the old edge led to failed action.
6. Update BeliefState, not just Semantic memory.
```

# 9. Realtime Write Path

The write path is built for low-latency acknowledgement and asynchronous intelligence.

```mermaid
flowchart LR
    E[Raw event/tool/doc/code run] --> S[Segment event]
    S --> L[Append episode ledger<br/>fast durable ack]
    L --> O[Observer compression]
    O --> X[Extract candidate memories]
    X --> N[Normalize entities/units/predicates]
    N --> R[Route to lanes]
    R --> V[Validate provenance, duplicates, privacy]
    V --> T[Temporal contradiction check]
    T --> P[Policy: add/update/noop/review]
    P --> C[Commit canonical objects]
    C --> I[Update projection indexes]
    I --> D[Trigger consolidation/reflection if needed]
```

## Staged latency

| Stage                   |         Target | Synchronous?     |
| ----------------------- | -------------: | ---------------- |
| Append raw episode      |         1–5 ms | yes              |
| Return write receipt    |         <10 ms | yes              |
| Observation compression |      20–200 ms | async            |
| LLM/rule extraction     | 100 ms–seconds | async            |
| Index update            |       5–100 ms | async or batched |
| Consolidation           |        seconds | idle/background  |

The agent does not wait for full extraction before continuing. It gets a receipt immediately, then candidate memories mature in the background.

## Candidate write policy

```text
if secret/private and not allowed:
    reject or store only in encrypted vault

if no provenance and lane requires evidence:
    Proposed, not Trusted

if duplicate with similarity > 0.97 and same payload:
    NOOP + strengthen existing

if near-duplicate with similarity > 0.88:
    UPDATE candidate target, preserve old version

if contradiction with live temporal edge:
    close old validity window
    add new edge
    create contradiction/counterexample note
    update belief

if repeated successful workflow:
    propose or strengthen procedural skill

else:
    ADD as Proposed or Trusted depending on validation
```

# 10. Read Path: Retrieval Compiler

MNEMOS-Ω does not “search memory.” It **compiles context**.

```mermaid
flowchart TD
    Q[User/agent query] --> QC[Query classifier<br/>intent, domain, time, evidence level]
    QC --> F[Parallel retrieval fanout]

    F --> CORE[Core/working lookup]
    F --> BM[BM25 + exact symbols]
    F --> VX[Vector hot/cold search]
    F --> ENT[Entity lookup]
    F --> TMP[Temporal valid-at query]
    F --> GR[Graph PPR activation]
    F --> CON[Concept kernels/topic capsules]
    F --> SK[Skill/precondition search]
    F --> EV[Eval/counterexample search]

    CORE --> RRF[Score normalization + RRF]
    BM --> RRF
    VX --> RRF
    ENT --> RRF
    TMP --> RRF
    GR --> RRF
    CON --> RRF
    SK --> RRF
    EV --> RRF

    RRF --> RR[Task-aware reranker]
    RR --> FLT[Validity/privacy/lifecycle filters]
    FLT --> PACK[ContextPack compiler]
    PACK --> LOG[Access log + strengthening]
    PACK --> AGENT[Agent receives compact cited context]
```

## Retrieval channels

| Channel                | Finds                                                  | Why needed                |
| ---------------------- | ------------------------------------------------------ | ------------------------- |
| Core/working           | mission, constraints, active state                     | Always relevant           |
| BM25                   | exact terms, names, paper IDs, code symbols, equations | Embeddings miss exactness |
| Hot vector HNSW        | semantically close recent/high-value memories          | Fast recall               |
| Cold compressed vector | older/cold memories at million scale                   | Memory depth              |
| Entity graph           | objects linked to query entities                       | Precision                 |
| Temporal graph         | facts valid at requested time                          | Prevents stale truth      |
| Personalized PageRank  | multi-hop associated evidence                          | Scientific reasoning      |
| Concept kernels        | compressed topic summaries                             | Token economy             |
| Procedural index       | executable skills                                      | Action                    |
| Eval/counterexample    | past failures and corrections                          | Self-correction           |

## Rerank formula

```text
score =
  w_exact      * exact_match
+ w_semantic   * vector_similarity
+ w_entity     * entity_overlap
+ w_graph      * ppr_activation
+ w_temporal   * temporal_validity
+ w_concept    * concept_kernel_fit
+ w_skill      * skill_reliability
+ w_source     * source_quality
+ w_conf       * confidence
+ w_utility    * utility
+ w_recent     * recency_decay
+ w_topic      * topic_strength
+ w_eval       * prior_retrieval_success
- w_stale      * stale_penalty
- w_contra     * unresolved_contradiction_penalty
- w_privacy    * access_penalty
```

Weights are chosen by query intent.

## Retrieval weight profiles

| Intent                  | Highest weights                                           | Always include                      |
| ----------------------- | --------------------------------------------------------- | ----------------------------------- |
| Math derivation         | exact symbols, prerequisites, equations, theorem DAG      | assumptions, units, counterexamples |
| Scientific claim        | source quality, contradiction pressure, temporal validity | support and contradiction           |
| Coding/debugging        | exact errors, recent episodes, skill reliability          | failed attempts                     |
| Planning                | core goals, open questions, skills, topic strength        | blockers                            |
| Literature review       | resource graph, citation clusters, concept kernels        | omitted evidence notes              |
| Personal/project recall | episodic time/entity, core memory                         | date handling                       |
| Skill selection         | preconditions, success rate, recent failures              | tests and reliability               |

## ContextPack contract

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]

    pub id: ContextPackId,
    pub query: RecallQuery,
    pub generated_at: DateTime<Utc>,

    pub items: Vec<ContextItem>,
    pub citations: Vec<Citation>,
    pub contradictions: Vec<ContradictionNote>,
    pub omitted_evidence: Vec<OmittedEvidenceNote>,
    pub freshness: FreshnessReport,

    pub confidence: f32,
    pub token_estimate: u32,
    pub budget: ContextBudget,

    pub retrieval_trace_id: RetrievalTraceId,
}
```

Each item includes:

```rust
pub struct ContextItem {
    pub memory_id: MemoryId,
    pub lane: MemoryLane,
    pub rendered: String,
    pub why_included: String,
    pub score: f32,
    pub citations: Vec<Citation>,
    pub validity: Option<BitemporalValidity>,
    pub warnings: Vec<String>,
}
```

Guarantees:

```text
1. Token budget is honored.
2. Every factual item has provenance.
3. Contradictions are surfaced, not hidden.
4. Stale facts are excluded unless historical query requests them.
5. Secret/vault memories are redacted unless explicitly authorized.
6. Pack order is deterministic for fixed memory state and query.
7. Access is logged for strengthening and eval.
```

# 11. Knowledge Compression Over Time

This is the most important part for million-memory scale.

MNEMOS-Ω compresses memory in layers:

```text
L0 Raw artifacts
   PDFs, code logs, chat transcripts, datasets, screenshots
   Stored in CAS, never used directly in prompt unless needed.

L1 Episodes
   Immutable records of what happened.

L2 Observations
   Dense dated summaries of episodes.

L3 Atomic memory objects
   Claims, equations, resources, skills, evals, counterexamples.

L4 Concept kernels
   Compact, cited synthesis over many atomic objects.

L5 Topic capsules
   Very compressed “current map” of a topic across concept kernels.

L6 Corpus maps
   GraphRAG-style maps over whole paper/book/repo collections.

L7 Working/core summary
   The tiny always-visible mission and active state.
```

## Compression rule

```text
Lossy summaries are never the source of truth.
They are retrieval accelerators and context budget optimizers.
Every compressed item must point downward to evidence.
```

## Example compression

```text
100,000 raw tokens from papers, searches, code, experiments
    ↓
400 dated observations
    ↓
120 atomic claims/equations/resources/evals
    ↓
12 concept kernels
    ↓
2 topic capsules
    ↓
1 compact ContextPack section
```

The agent can see the capsule first. If it needs proof, it can expand to kernels, then claims, then source spans.

# 12. Consolidation Daemons

MNEMOS-Ω has background daemons. They do not sit in the hot path unless explicitly requested.

| Daemon                | Function                                                |
| --------------------- | ------------------------------------------------------- |
| `observer`            | Compress raw events into dated observations             |
| `extractor`           | Extract claims, equations, skills, resources, questions |
| `linker`              | Add typed edges among memories                          |
| `temporal_reviser`    | Detect contradictions and supersessions                 |
| `concept_synthesizer` | Build and update concept kernels                        |
| `topic_compactor`     | Build topic capsules and corpus maps                    |
| `skill_distiller`     | Promote repeated successful workflows into skills       |
| `eval_learner`        | Learn from retrieval and task outcomes                  |
| `rehearsal_scheduler` | FSRS-style review of important weakening topics         |
| `decay_archiver`      | Move low-utility memories out of hot indexes            |
| `formal_checker`      | Units, proof-state, theorem dependency checks           |
| `index_maintainer`    | Rebuild/compact projections from canonical truth        |

## Concept consolidation algorithm

```text
1. Select candidate memories:
   recent OR high surprise OR high utility OR repeated co-activation.

2. Cluster:
   entity overlap + vector similarity + graph proximity + topic community.

3. Build or update ConceptKernel:
   definition
   prerequisites
   core claims
   equations/theorems
   resources
   skills
   counterexamples
   open questions

4. Validate:
   every factual sentence must link to evidence.
   contradictions must be included.
   unsupported synthesis remains Proposed.

5. Commit:
   insert/update concept memory
   link back to source memories
   emit compression receipt

6. Re-index:
   concept kernel gets its own embedding, BM25 doc, graph node, topic membership.
```

## Topic strengthening

Each topic has an explainable score:

```text
strength =
  decayed_previous_strength
+ recurrence
+ utility
+ retrieval_success
+ source_quality
+ novelty
+ graph_neighbor_boost
+ skill_success
- unresolved_contradiction_pressure
- repeated_retrieval_failure
```

A topic is strengthened when it helps the agent solve tasks. It weakens when it is unused, contradicted, or retrieved but unhelpful.

# 13. Agent-Facing Memory Calls

This directly answers: **what can the agent call to help organize and compress memory over time?**

| Call                            | When agent uses it                                     | Effect                                               |
| ------------------------------- | ------------------------------------------------------ | ---------------------------------------------------- |
| `observe(event)`                | Every user message, tool result, code run, file ingest | Appends immutable episode and schedules extraction   |
| `recall(query)`                 | Before answering or acting                             | Returns bounded cited `ContextPack`                  |
| `remember(candidate)`           | Agent explicitly learned something durable             | Creates proposed memory with provenance              |
| `feedback(pack_id, outcome)`    | After task success/failure                             | Updates utility, retrieval policy, skill reliability |
| `reflect(scope)`                | After surprise, failure, contradiction, or end of task | Produces lessons, belief updates, open questions     |
| `consolidate(scope)`            | Idle or explicit compression request                   | Builds concept kernels and topic capsules            |
| `rehearse(topic, n)`            | Maintain important knowledge                           | Reviews weakening topic memories                     |
| `promote_skill(trace)`          | Repeated successful workflow                           | Creates or updates executable skill                  |
| `verify(memory_id)`             | Check claim/equation/skill/formal object               | Runs provenance/formal/test checks                   |
| `resolve_contradiction(ids)`    | Conflicting evidence detected                          | Updates belief and temporal graph                    |
| `merge_duplicates(ids)`         | Memory bloat detected                                  | Deduplicates without deleting provenance             |
| `archive_low_value(policy)`     | Hot index is noisy                                     | Moves low-value memories to cold tier                |
| `explain(memory_id or pack_id)` | Debug memory behavior                                  | Shows why stored/retrieved/omitted/strengthened      |
| `expand(memory_id, level)`      | Need deeper evidence                                   | Expands capsule → kernel → claim → source            |
| `open_question(topic)`          | Research gap appears                                   | Creates question card and next-action links          |

}

# 14. Procedural Skill Compounding

Procedural memory must compound the agent’s ability to act.

## Skill promotion rule

A workflow can become a trusted skill only when:

```text
1. It appears in at least one successful episode.
2. It has clear preconditions.
3. It has reproducible steps.
4. It has a test, assertion, or verification recipe.
5. It has failure modes recorded.
6. It has source/code provenance.
7. It passes sandbox validation.
```

## Skill lifecycle

```text
Draft -> Tested -> Trusted -> Degraded -> Retired
```

A skill is degraded when:

```text
- recent failures increase
- environment preconditions no longer hold
- dependencies change
- eval memory shows repeated misuse
- a safer/better skill supersedes it
```

## Skill recall example

For a query like:

```text
“Extract equations from this arXiv paper and check unit consistency.”
```

The context compiler should include:

```text
[Skill] extract_arxiv_equations_v4
- Preconditions: PDF path, pdftotext available
- Reliability: 0.91 over 44 uses
- Last failure: OCR failed on scanned PDFs
- Tests: equation_count_nonzero, latex_parse_rate > 0.8
- Related skills: normalize_units, build_equation_cards
```

The agent receives the skill signature and failure warnings, not a giant code dump.

# 15. Science, Math, and Research Specialization

MNEMOS-Ω is designed for scientific knowledge, not just personal assistant memory.

## Required scientific object separation

| Object               | Stores                                        | Must not collapse into |
| -------------------- | --------------------------------------------- | ---------------------- |
| `ResourceCard`       | paper, book, dataset, repo, API, benchmark    | claim                  |
| `ClaimCard`          | atomic proposition with support/contradiction | belief                 |
| `BeliefState`        | current evidence-weighted stance              | fact                   |
| `EquationCard`       | formula, variables, units, assumptions        | plain text note        |
| `TheoremCard`        | statement, proof status, dependency DAG       | claim                  |
| `ExperimentCard`     | method, parameters, data hash, outcome        | anecdote               |
| `CounterexampleCard` | falsification/failure condition               | forgotten error        |
| `QuestionCard`       | unresolved target and blockers                | todo text              |
| `ConceptKernel`      | compressed abstraction over evidence          | source of truth        |
| `SkillCard`          | executable tested workflow                    | untested snippet       |

## Math/science invariants

```text
1. Equations require variables and assumptions.
2. Equations with units must pass dimensional checks to become Trusted.
3. Theorems track dependency closure.
4. Formal checks are optional but status must be explicit.
5. Beliefs require support and contradiction links.
6. Experiments require code/data/source hashes.
7. Claims from papers preserve source spans.
8. Counterexamples are retrieved aggressively.
9. Concept kernels include misconceptions and failure modes.
10. Open questions are first-class memory objects.
```

# 16. Scaling to Millions of Memories

## Target scale

| Target                       |                     Design goal |
| ---------------------------- | ------------------------------: |
| Single-node hot memories     |                     1–3 million |
| Single-node total memories   |                   10–50 million |
| Cold archived memories       |   100+ million via sharding/CAS |
| Retrieval no graph expansion |                    p50 30–80 ms |
| Retrieval with PPR           |                  p50 100–300 ms |
| Cold/deep research recall    |                  p50 300–900 ms |
| Episode append               |                       p50 <5 ms |
| ContextPack budget           |            default 4k–8k tokens |
| Hot index memory             | bounded by utility/topic policy |

## Hot/warm/cold tiering

```text
Hot:
  high utility, recent, mission-critical, active project
  full BM25 + HNSW + graph adjacency + concept cache

Warm:
  useful but not active
  BM25 + compressed vectors + graph summary

Cold:
  low utility or old
  canonical store + resource pointers + compressed topic membership

Archive:
  immutable, not searched unless historical/deep recall
```

## Vector scaling

MNEMOS-Ω does not put 50M full float vectors in RAM.

It uses:

```text
1. Hot HNSW:
   exact or fp16 vectors for top high-value memories.

2. Cold compressed vectors:
   product-quantized or scalar-quantized capsules.

3. Candidate rerank:
   cold search returns approximate hits; exact text/entity/provenance reranks.

4. Concept-first recall:
   for broad queries, search topic capsules and concept kernels before atoms.

5. Utility-driven promotion:
   cold memories that become useful are re-promoted to hot tier.
```

## Graph scaling

The graph is stored as canonical edges plus mmap projections:

```text
Canonical:
  TemporalEdge rows in object store

Projection:
  entity -> outgoing edge IDs
  entity -> incoming edge IDs
  topic -> member memory IDs
  relation -> edge bitmap
  time bucket -> edge bitmap
  CSR adjacency snapshot for PPR
```

Graph traversal is bounded by:

```text
- relation whitelist
- max hops
- activation decay
- temporal validity
- topic/community limits
- token budget pressure
```

# 17. Context Economy

The context compiler should almost never dump raw memory.

Preferred inclusion order:

```text
1. Core constraints
2. Directly relevant concept kernel or topic capsule
3. Exact claim/equation/skill needed for task
4. Counterexamples and known failure modes
5. Minimal evidence citations
6. Omitted-evidence warning if budget is tight
7. Expansion handles for deeper retrieval
```

## ContextPack example shape

```text
# ContextPack: derive diffusion equation from random walk

## Core constraints
- Use local-first Rust/math tooling when possible.

## Concept kernel
- Diffusion equation is the continuum limit of unbiased random walks...
- Prerequisites: Markov chains, Taylor expansion, conservation law.

## Equations
- ∂u/∂t = D ∂²u/∂x²
  Assumptions: continuum limit, independent steps, finite variance.
  Units: D has L²/T.

## Evidence
- Claim C17 from Resource R03 supports continuum-limit derivation.
- Experiment E09 validated finite-difference simulation.

## Counterexamples
- Prior derivation failed when step variance was infinite.
- Do not assume Gaussian limit for heavy-tailed jumps.

## Skills
- `simulate_random_walk_diffusion_v2`, reliability 0.88.
- `check_equation_units`, reliability 0.97.

## Omitted
- 14 lower-ranked derivation notes omitted; expandable by topic id.
```

# 19. Core Code Concepts

## Superseding a temporal fact

```rust
pub async fn supersede_fact(
    store: &dyn CanonicalStore,
    old_edge_id: EdgeId,
    new_edge: TemporalEdge,
    reason: String,
    now: DateTime<Utc>,
) -> Result<MutationReceipt> {
    let mut tx = store.begin_tx().await?;

    let mut old = tx.load_edge(old_edge_id).await?
        .ok_or(Error::MissingEdge(old_edge_id))?;

    // Close transaction-time validity: the memory system no longer treats
    // this edge as the current live edge.
    old.validity.tx_to = Some(now);

    // If the new edge describes a world-validity replacement,
    // close valid_to as well.
    if old.subject == new_edge.subject && old.predicate == new_edge.predicate {
        old.validity.valid_to = new_edge.validity.valid_from.or(Some(now));
    }

    let old_hash = canonical_hash(&old)?;
    tx.update_edge(old.clone()).await?;

    let mut inserted = new_edge;
    inserted.supersedes.push(old_edge_id);
    tx.insert_edge(inserted.clone()).await?;

    tx.insert_memory_link(
        inserted.source_memories[0],
        MemoryLink {
            target: old.source_memories[0],
            relation: Relation::Supersedes,
            strength: 1.0,
            evidence: inserted.support.iter().map(|e| e.source.clone()).collect(),
            created_at: now,
        }
    ).await?;

    tx.append_ledger(LedgerEntry::TemporalEdgeClosed {
        edge_id: old_edge_id,
        valid_to: old.validity.valid_to.unwrap_or(now),
        tx_to: now,
        reason,
    }).await?;

    tx.append_ledger(LedgerEntry::TemporalEdgeInserted(inserted)).await?;

    tx.commit().await
}
```

## Context packing algorithm

```text
1. Reserve budget:
   core: 5%
   direct answers: 35%
   evidence: 25%
   counterexamples: 10%
   skills/actions: 10%
   omissions/warnings: 5%
   slack: 10%

2. Prefer compressed concept kernels for broad questions.

3. Prefer atomic claims/equations for exact questions.

4. Always include counterexamples if directly relevant.

5. Include citations for all factual items.

6. If evidence is omitted, include an omitted-evidence note.

7. Emit expansion handles.
```

# 23. Final Recommended Shape

The final system should be built around this operating loop:

```text
observe
  -> append immutable episode
  -> compress into observation
  -> extract typed candidates
  -> validate provenance/privacy
  -> update temporal graph
  -> route into memory lanes
  -> index as projections
  -> consolidate into concepts/topics
  -> retrieve via compiler
  -> agent acts
  -> feedback updates utility, skills, beliefs, topics
```

The single most important design rule is:

```text
Never let embeddings become the memory.
Embeddings are only recall hints.
The memory is typed, temporal, source-backed, auditable, compressible, and executable.
```

For a superhuman research/coding/science agent, MNEMOS-Ω gives:

```text
human-like structure:
  episodes, concepts, beliefs, skills, forgetting, reflection

superhuman advantages:
  perfect audit trails
  parallel retrieval
  bitemporal truth
  executable skills
  formal checks
  million-memory compression
  counterexample-first recall
  learned memory policy
  deterministic context compilation
```

This is the best architecture I would build from the studied materials.
