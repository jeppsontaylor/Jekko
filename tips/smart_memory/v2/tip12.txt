# ALETHEIA-О©: a Rust Cognitive Memory Compiler for superhuman realtime learning

I studied the archive as a design corpus: `CLAUDE_MEMSPEC.md`, `CODEX_MEMSPEC.md`, `tip1.txt` through `tip13.txt`, plus the Antigravity/Gemini-style CMC blueprint you pasted. The strongest recurring idea is this:

> **Do not build вЂњmemoryвЂќ as a vector database. Build memory as a typed, temporal, source-backed, self-compressing compiler over experience.**

The system below is the design I would actually build.

It keeps the best pieces of the existing specs, but tightens the engineering model around one rule: **canonical truth is append-only and auditable; everything fast is a compiled projection.** That gives you human-like structure, machine-speed recall, realtime updates, and compression that compounds instead of corrupting knowledge.

The design synthesizes the typed six-lane MIRIX model, temporal Graphiti/Zep facts, A-MEM Zettelkasten linking, HippoRAG-style graph activation, Mem0/Mastra-style low-token recall, Voyager/Memento procedural skills, MemGPT/Letta context control, MemOS lifecycle management, and Memory-R1 learned memory policy. MIRIXвЂ™s six memory types and reported storage/accuracy improvements support the typed-memory foundation; Graphiti/Zep supports temporal knowledge graphs; A-MEM supports dynamic Zettelkasten evolution; HippoRAG supports KG plus Personalized PageRank retrieval; Mem0 and Mastra OM support fast compressed recall; Voyager supports executable skill compounding; MemGPT and MemOS support memory-as-operating-system abstractions. ## 2. What the archive converged on

The local files and the pasted CMC spec agree on six non-negotiable requirements:

1. **Typed memory is mandatory.** Core, episodic, semantic, procedural, resource, concept, belief, and evaluation memory cannot be collapsed into one vector table.

2. **Temporal truth beats static truth.** A memory system must know what was believed, when it was believed, what superseded it, and what evidence caused the revision.

3. **Provenance is not optional.** Every promoted claim must be linked to an episode, resource span, code line range, test result, or human-approved note.

4. **Compression must preserve source traceability.** Summaries and concept cards are useful only if they remain compiled views over recoverable evidence.

5. **The hot path must not call an LLM.** LLMs may extract, reflect, consolidate, and verify. They should not be required for normal recall.

6. **The agent must learn procedures, not only facts.** Executable, tested, versioned skills are the main difference between a memory database and an agent that improves.

ALETHEIA-О© makes those requirements hard invariants instead of soft guidelines.

## 3. Design thesis

The optimal memory system is a **two-speed, event-sourced cognitive compiler**:

```text
Fast path:
  observe в†’ append episode в†’ provisional observation в†’ local index delta
  recall  в†’ parallel indexes в†’ temporal graph activation в†’ bounded ContextPack

Slow path:
  extract в†’ ground в†’ link в†’ verify в†’ consolidate в†’ compress в†’ revise beliefs в†’ distill skills
```

The fast path gives realtime continuity.
The slow path gives intelligence.

The key innovation is the **Memory Cell** abstraction:

```text
Memory Cell = immutable atom + bitemporal validity + provenance + typed payload
              + graph links + utility state + compression lineage
```

Every fact, equation, skill, reflection, concept card, belief, counterexample, and resource pointer is a Memory Cell or a compiled capsule over Memory Cells.

## 5. The eight primary memory lanes

ALETHEIA-О© uses eight primary lanes. It does **not** create one database per lane; the lanes are typed views over a shared canonical schema and shared projection indexes.

| Lane                       | Human analogue          | What it stores                                                                                   | Hard rule                                              |
| -------------------------- | ----------------------- | ------------------------------------------------------------------------------------------------ | ------------------------------------------------------ |
| **Core / Working**         | prefrontal task state   | mission, active goals, current constraints, scratchpad, user-approved stable preferences         | always tiny; never grows silently                      |
| **Episodic / Observation** | autobiographical memory | timestamped events, tool runs, code runs, searches, conversations, experiment traces             | append-only, hash-chained                              |
| **Semantic / Claim**       | factual knowledge       | atomic claims, definitions, entities, relations, assumptions, validity windows                   | claim в‰  belief                                         |
| **Belief / Epistemic**     | current stance          | вЂњwhat the agent currently believes,вЂќ confidence, support/contradiction graph                     | beliefs are derived from evidence, not stored as facts |
| **Concept / Schema**       | mental models           | Zettelkasten concept cards, topic kernels, prerequisites, misconceptions, open questions         | summaries must link back to atoms                      |
| **Procedural / Skill**     | habits and skills       | executable workflows, scripts, tool plans, tests, preconditions, success/failure traces          | untested code is not a trusted skill                   |
| **Resource / Vault**       | external ledger         | papers, repos, datasets, local files, DOIs, commit hashes, API descriptors, encrypted vault refs | immutable pointers and checksums                       |
| **Counterexample / Eval**  | mistakes and learning   | failed attempts, contradictions, bad answers, debug lessons, retrieval evals, user corrections   | counterexamples are first-class and bias recall        |

Specialist overlays attach to any lane: math/science formal objects, source trust, privacy class, multimodal descriptors, code-symbol graphs, curriculum state, and review schedules.

## 6. Canonical memory object model

The design has four levels.

```text
L0 Episode
  What happened. Immutable. Hash-chained. Raw or compressed source pointer.

L1 Observation
  Dense dated statement extracted from an episode.
  Example: "At 2026-05-11, cargo test failed because trait Foo lacked Send."

L2 Memory Atom
  Typed, source-backed object: Claim, Equation, Skill, Resource, Belief, Concept, Eval, etc.

L3 Memory Capsule
  A compiled, compressed view over many atoms:
  concept card, literature map, misconception list, skill playbook, topic kernel.

L4 ContextPack
  A short, task-specific prompt payload compiled at inference time.
```

**Summaries are never the source of truth.** Capsules are compressed views, not replacements.

### Temporal contradiction invariant

```text
If new edge E2 has same subject + predicate as active edge E1
and E2.object conflicts with E1.object:

1. Do not overwrite E1.
2. Set E1.valid_to = now.
3. Insert E2.valid_from = now.
4. Add E2 --Supersedes--> E1.
5. Add contradiction/counterexample note.
6. Recompute related BeliefState.
```

That is the single most important anti-hallucination invariant.

## 10. Storage and indexing design

ALETHEIA-О© separates **canonical stores** from **rebuildable projections**.

### Canonical stores

| Store          | Purpose                                   | Recommended baseline                                |
| -------------- | ----------------------------------------- | --------------------------------------------------- |
| Event ledger   | append-only mutation truth                | zstd-compressed JSONL/CBOR segments with hash chain |
| Object catalog | current memory versions, metadata, status | `redb` or SQLite-compatible adapter                 |
| Version store  | all historical versions                   | redb tables or append-only segments                 |
| Resource cache | PDFs, repos, docs, raw text, snapshots    | content-addressed blob store                        |
| Skill repo     | executable skills and tests               | Git-backed directory with signed manifests          |
| Human cards    | curated concept/resource/skill capsules   | Markdown/YAML with generated zones                  |

For Rust-native indexing, Tantivy is a Rust full-text search engine library, Qdrant is a Rust vector search engine option for production, and `redb` is a Rust embedded ACID database; Kuzu has Rust bindings if you want an embedded property-graph adapter for analytics. ([GitHub][2])

### Rebuildable projections

| Projection               | Function                                                       |
| ------------------------ | -------------------------------------------------------------- |
| BM25 index               | exact search, symbols, equations, code identifiers, paper IDs  |
| Dense vector HNSW        | semantic recall over observations, claims, concepts, resources |
| Entity alias index       | canonical entity resolution, fuzzy aliases, abbreviations      |
| Temporal interval index  | вЂњwhat was true at T?вЂќ                                          |
| Graph adjacency          | fast relation traversal and PPR activation                     |
| Co-activation matrix     | Hebbian вЂњthings used togetherвЂќ edges                           |
| Skill precondition index | retrieve executable skills by task state                       |
| Concept capsule cache    | compressed topic summaries for fast context packing            |
| ContextPack cache        | query-signature to compiled pack, invalidated by mutations     |

## 12. Write path: realtime but safe

```mermaid
sequenceDiagram
    participant Agent
    participant API
    participant Observer
    participant Ledger
    participant Extractor
    participant Validator
    participant Store
    participant Indexer
    participant Consolidator

    Agent->>API: observe(EventEnvelope)
    API->>Observer: segment + summarize
    Observer->>Ledger: append Episode event
    Ledger-->>API: EpisodeId + receipt
    API-->>Agent: committed provisional observation

    par async extraction
        Ledger->>Extractor: extract candidates
        Extractor->>Validator: claims/entities/skills/resources
        Validator->>Store: commit MemoryAtoms + edges
        Store->>Indexer: update projection delta
        Store->>Consolidator: schedule reflection/compression
    end
```

### Write pipeline stages

1. **Observe**
   Accept event: chat message, shell output, code run, file ingest, paper, dataset, screenshot, human note.

2. **Segment**
   Detect boundaries: task switch, error, result, decision, user correction, new source, long silence, changed entity set.

3. **Append episode**
   Hash-chain event into ledger before any extraction. This is the source of truth.

4. **Create provisional observation**
   Store a short durable observation immediately, so realtime recall works even before full extraction.

5. **Extract candidates**
   LLM or rules extract claims, entities, equations, skills, resources, observations, evals.

6. **Normalize**
   Canonicalize entities, units, names, predicates, file paths, symbols, dates.

7. **Ground**
   Attach exact source spans, hashes, line ranges, command receipts, test logs.

8. **Validate**
   Check schema, duplicates, contradictions, privacy class, source quality, confidence.

9. **Commit memory mutations**
   Insert immutable versions. No destructive overwrite.

10. **Update projections**
    Async index updates for BM25, vector, entity, graph, temporal, skill, capsule cache.

11. **Schedule consolidation**
    If high surprise, repeated utility, contradiction, or recurring cluster appears, trigger sleep-cycle work.

### Write-time invariants

```text
Invariant 1: No promoted semantic claim without provenance.
Invariant 2: No destructive overwrite of facts or beliefs.
Invariant 3: BeliefState is derived from ClaimCards, never treated as source evidence.
Invariant 4: Skills are untrusted until tests or successful episodes prove them.
Invariant 5: Generated concept capsules must preserve input_memory_ids.
Invariant 6: Secrets are stored as SecretRefOnly, never embedded or summarized.
```

The hot path runs these channels in parallel:

```text
1. pinned core memory
```

### Ranking formula

```text
candidate_score =
    w_exact      * exact_match
  + w_dense      * dense_similarity
  + w_entity     * entity_overlap
  + w_graph      * graph_activation
  + w_temporal   * temporal_validity
  + w_source     * source_quality
  + w_conf       * confidence
  + w_utility    * utility
  + w_topic      * topic_strength
  + w_skill      * skill_reliability
  + w_counter    * counterexample_relevance
  - w_stale      * stale_penalty
  - w_conflict   * unresolved_contradiction_penalty
  - w_privacy    * privacy_penalty
```

The weights are intent-specific. A math query boosts equations, units, theorem dependencies, exact symbols, and counterexamples. A coding query boosts recent failures, code symbols, skill reliability, and local project resources. A literature query boosts source quality, resource graph, contradictory clusters, and citation spans.

## 14. ContextPack contract

A `ContextPack` is not a pile of retrieved chunks. It is a compiled object with guarantees.

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ContextPack {
    pub id: Uuid,

    pub generated_at: DateTime<Utc>,

    pub items: Vec<ContextItem>,
    pub citations: Vec<Citation>,
    pub contradictions: Vec<ContradictionNote>,
    pub freshness: FreshnessReport,
    pub callable_skills: Vec<SkillCallHint>,
    pub omitted_evidence: Vec<OmittedEvidenceNote>,
    pub confidence_summary: ConfidenceSummary,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ContextItem {
    pub memory_id: MemoryId,
    pub kind: MemoryKind,
    pub rendered: String,
    pub why_included: String,
    pub score: f32,
    pub source_spans: Vec<SourceSpan>,
    pub valid_from: Option<DateTime<Utc>>,
    pub valid_to: Option<DateTime<Utc>>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ContextBudget {

```

### ContextPack must include

```text
- why_included for every item
- citations/source spans for factual items
- temporal validity notes
- contradiction warnings
- confidence summary
- callable skills without dumping full code bodies
- omitted-evidence notes when budget excludes important evidence
- deterministic ordering for equal scores
```

## 15. Knowledge compression engine

The compression engine is where the system becomes superhuman.

Humans forget details and keep gist. ALETHEIA-О© keeps both: it compiles gist while preserving raw evidence.

### Compression ladder

```text
L0 Verbatim source
  Raw paper, chat, code output, file, dataset manifest.

L1 Episode summary
  What happened in this event.

L2 Atomic memory
  Claims, equations, observations, skills, resources, evals.

L3 Topic kernel
  Compressed concept with prerequisites, claims, equations, skills, failures.

L4 Domain map
  Community-level graph summary across many topic kernels.

L5 Procedural playbook
  Reusable workflow distilled from repeated successful episodes.
```

### Compression jobs

| Job                     | Trigger                                           | Output                                    |
| ----------------------- | ------------------------------------------------- | ----------------------------------------- |
| duplicate merge         | cosine > 0.95 + same entities + same claim scope  | one trusted atom, old atoms superseded    |
| concept kernel          | topic strength high or repeated retrieval cluster | cited concept card                        |
| contradiction reconcile | active contradiction pressure high                | belief revision + counterexample note     |
| failure distillation    | repeated failed attempts                          | debugging lesson + blocked skill          |
| skill distillation      | repeated successful trajectories                  | executable skill with tests               |
| resource digest         | large paper/repo/dataset ingested                 | resource card + claim/equation extraction |
| topic review            | FSRS due or low retrieval success                 | review queue item                         |
| capsule refresh         | source graph changed                              | new capsule version, old superseded       |
| curriculum synthesis    | topic dependencies stabilize                      | ordered learning graph                    |

### Compression scoring

```text
compression_value =
    utility_gain
  + context_token_savings
  + retrieval_success_gain
  + duplicate_reduction
  + concept_connectivity_gain
  - evidence_loss_risk
  - contradiction_unresolved_penalty
```

A capsule is promoted only if:

```text
coverage_score >= 0.80
loss_risk_score <= 0.25
all factual statements cite MemoryAtoms
no SecretRefOnly content is included
```

## 16. Topic strength and review

Topic strength is an explainable decayed score, not an embedding artifact.

```text
decayed =
  previous_strength * exp(-hours_since_update / half_life_hours)

reinforcement =
  0.20 * recency
+ 0.20 * recurrence
+ 0.25 * utility
+ 0.15 * novelty
+ 0.15 * source_quality
+ 0.15 * retrieval_success
+ 0.10 * graph_neighbor_boost
- 0.20 * unresolved_contradiction_pressure

new_strength = clamp(decayed + reinforcement, 0, 1)
```

Strong topics produce concept capsules. Weak but important topics enter review queues. High-utility, low-confidence topics trigger targeted evidence search. Contradicted topics trigger belief revision.

This is how the agent knows what to organize, what to compress, what to rehearse, and what to investigate.

## 18. Core recall code sketch

```rust
pub async fn recall(engine: &EngineState, query: RecallQuery) -> Result<ContextPack, MemoryError> {
    let intent = engine.intent.classify(&query)?;

    let core_fut = engine.core.load_relevant(&query);
    let bm25_fut = engine.index.bm25.search(&query, intent.bm25_profile());
    let dense_fut = engine.index.vector.search(&query, intent.vector_profile());
    let entity_fut = engine.index.entities.resolve(&query);
    let temporal_fut = engine.index.temporal.lookup(&query.time_scope);
    let skill_fut = engine.index.skills.search(&query, intent.skill_profile());
    let eval_fut = engine.index.eval.search_failures(&query);

    let (core, bm25, dense, entities, temporal, skills, evals) = tokio::try_join!(
        core_fut,
        bm25_fut,
        dense_fut,
        entity_fut,
        temporal_fut,
        skill_fut,
        eval_fut,
    )?;

    let fused = reciprocal_rank_fuse(vec![
        bm25.into_ranked("bm25"),
        dense.into_ranked("dense"),
        entities.into_ranked("entity"),
        temporal.into_ranked("temporal"),
        skills.into_ranked("skill"),
        evals.into_ranked("eval"),
    ], 60.0);

    let seeds = fused.top_n(64);
    let activated = engine.graph.personalized_activation(
        &seeds,
        &query.time_scope,
        intent.graph_profile(),
    )?;

    let candidates = engine.store.load_candidates(fused.merge(activated).top_n(256))?;

    let filtered = temporal_privacy_filter(candidates, &query)?;

    let reranked = rerank(filtered, &query, &intent)?;

    let pack = engine.compiler.compile_context_pack(
        core,
        reranked,
        query.token_budget,
        intent.pack_profile(),
    )?;

    engine.telemetry.log_recall(&pack).await?;
    Ok(pack)
}
```

## 19. Reranking code sketch

```rust
pub fn rerank(
    candidates: Vec<ScoredCandidate>,
    query: &RecallQuery,
    intent: &IntentProfile,
) -> Result<Vec<ScoredCandidate>, MemoryError> {
    let mut out = Vec::with_capacity(candidates.len());

    for mut c in candidates {
        let s = &c.memory.scores;

        let stale_penalty = if c.memory.bitemporal.valid_to.is_some()
            && !query.time_scope.is_historical()
        {
            1.0
        } else {
            0.0
        };

        let score =
            intent.w_exact      * c.features.exact_match
          + intent.w_dense      * c.features.dense_similarity
          + intent.w_entity     * c.features.entity_overlap
          + intent.w_graph      * c.features.graph_activation
          + intent.w_temporal   * c.features.temporal_validity
          + intent.w_source     * s.source_quality
          + intent.w_confidence * s.confidence
          + intent.w_utility    * s.utility
          + intent.w_topic      * c.features.topic_strength
          + intent.w_counter    * c.features.counterexample_relevance
          - intent.w_stale      * stale_penalty
          - intent.w_conflict   * s.contradiction_pressure;

        c.score = score.clamp(0.0, 1.0);
        out.push(c);
    }

    out.sort_by(|a, b| b.score.partial_cmp(&a.score).unwrap_or(std::cmp::Ordering::Equal));
    Ok(out)
}
```

## 20. Supersession code sketch

```rust
pub async fn insert_fact_edge(
    store: &dyn Store,
    graph: &dyn GraphIndex,
    new_edge: MemoryEdge,
) -> Result<MutationReceipt, MemoryError> {
    let conflicts = graph.find_active_conflicts(
        new_edge.source,
        &new_edge.relation,
        new_edge.target,
        new_edge.bitemporal.valid_from,
    ).await?;

    let mut tx = store.begin().await?;

    for old in conflicts {
        let mut closed = old.clone();
        closed.bitemporal.valid_to = Some(Utc::now());
        closed.status = MemoryStatus::Superseded;

        tx.update_edge_version(closed).await?;

        tx.insert_edge(MemoryEdge {
            id: EdgeId(Uuid::new_v4()),
            source: new_edge.id_as_memory_id(),
            target: old.id_as_memory_id(),
            relation: EdgeRelation::Supersedes,
            bitemporal: Bitemporal::now_valid(),
            weight: 1.0,
            confidence: new_edge.confidence,
            provenance: new_edge.provenance.clone(),
            status: MemoryStatus::Trusted,
        }).await?;
    }

    tx.insert_edge(new_edge).await?;
    tx.commit().await
}
```

## 21. Performance architecture

### Two-speed write

```text
Foreground:
  append episode
  write provisional observation
  update tiny delta index
  return receipt

Background:
  extract
  validate
  link
  update heavy indexes
  consolidate
```

This gives realtime usability without polluting trusted memory.

### Segment architecture

Memory atoms are stored in append-only segments:

```text
segment_000001/
  objects.cbor.zst
  edges.cbor.zst
  postings.delta
  vectors.delta
  graph.delta
  manifest.toml
  hash_chain.tail
```

Projection workers periodically seal deltas into optimized snapshots:

```text
hot delta в†’ warm segment в†’ sealed snapshot в†’ compacted archive
```

### Scaling to millions of memories

| Problem                   | Solution                                                                                   |
| ------------------------- | ------------------------------------------------------------------------------------------ |
| huge vector index         | partition by lane + topic + time; int8/PQ compression; HNSW per partition                  |
| graph traversal explosion | seed-limited PPR over active subgraph; relation weights; temporal filters before expansion |
| BM25 noise                | fielded indexes: symbols, code IDs, equations, entities, source titles, claims             |
| context bloat             | ContextPack budget allocator + concept capsules                                            |
| duplicate memories        | high-similarity merge queue + source-preserving supersession                               |
| stale facts               | bitemporal validity filter                                                                 |
| skill clutter             | reliability decay + status transitions                                                     |
| slow cold start           | memory-mapped snapshots + manifest prefetch                                                |
| expensive LLM extraction  | async batching, rule extractors for code/errors, local small-model fallback                |
| privacy leakage           | privacy class filters before embedding and retrieval                                       |

### Storage estimate for 10M atoms

Approximate local node target:

```text
10M memory atom metadata       3вЂ“8 GB
10M compressed payloads        5вЂ“50 GB depending on source size
10M 768-dim int8 embeddings    ~7.7 GB raw, less with PQ
BM25 postings                  5вЂ“20 GB
graph adjacency                2вЂ“10 GB
ledger + versions              workload-dependent
concept capsules               small, usually <2 GB
```

The design is feasible on a workstation SSD. For 100M+, shard by project/user/topic/entity hash and federate recall.

### Skill lifecycle

### Skill reliability

```rust
pub fn reliability(success: u64, failure: u64, recency_boost: f32) -> f32 {
    let alpha = 1.0 + success as f32;
    let beta = 1.0 + failure as f32;
    let mean = alpha / (alpha + beta);
    (mean * 0.9 + recency_boost * 0.1).clamp(0.0, 1.0)
}
```

## 23. Science and math specialization

Generic agent memory fails on math and science because it collapses claims, equations, assumptions, citations, and beliefs. ALETHEIA-О© keeps them separate.

### Required object separation

| Object           | Stores                                | Must not store         |
| ---------------- | ------------------------------------- | ---------------------- |
| `Claim`          | atomic assertion + evidence           | current agent belief   |
| `Belief`         | current stance over claims            | raw source assertion   |
| `Equation`       | symbols, variables, units, derivation | hidden assumptions     |
| `Theorem`        | assumptions, proof, dependencies      | unsupported intuition  |
| `Dataset`        | manifest, checksum, schema, license   | results without method |
| `Experiment`     | method, run config, output, logs      | generalized conclusion |
| `Concept`        | compressed topic synthesis            | uncited fact           |
| `Counterexample` | failure/falsifier/misconception       | buried warning         |

### The system must answer

```text
What did we observe?
Where did it come from?
What claim was extracted?
What evidence supports it?
What evidence contradicts it?
What does the agent currently believe?
When did that belief change?
Which equations/theorems/skills depend on it?
What breaks if it is false?
```

That is the standard for scientific memory.

## 25. Security and privacy

```text
1. SecretRefOnly objects are never embedded.
2. SecretRefOnly objects are never summarized into concept capsules.
```

## 28. Why this beats the existing specs

Compared with the supplied CLAUDE, CODEX, and CMC designs, ALETHEIA-О© improves five things.

### 1. It separates вЂњtruthвЂќ from вЂњprojectionsвЂќ more aggressively

The canonical ledger and memory versions are the only truth. BM25, vector, graph adjacency, capsules, and context caches are all rebuildable projections.

That is the right durability model for long-lived agents.

### 2. It introduces a Memory Cell abstraction

The prior specs have many object types, but the Memory Cell makes every object obey the same invariants:

```text
typed payload
+ provenance
+ bitemporal validity
+ status lifecycle
+ utility state
+ graph links
+ compression lineage
```

That gives uniform indexing, recall, explanation, and rollback.

### 3. It makes compression a first-class subsystem

The key to millions of memories is not bigger retrieval. It is **knowledge compression with lineage**:

```text
episodes в†’ observations в†’ atoms в†’ concept kernels в†’ domain maps в†’ procedural playbooks
```

This is how the memory gets smaller and smarter at the same time.

### 4. It makes eval memory first-class

Most memory systems remember user facts but forget whether remembering helped. ALETHEIA-О© stores retrieval outcomes, answer quality, failures, corrections, and skill success. That lets the system learn what to recall, what to suppress, and what to compress.

### 5. It has a real realtime strategy

It does not wait for extraction before becoming useful. It commits provisional observations immediately, then promotes structured memories asynchronously.

That is the only way to get both realtime responsiveness and high-quality long-term memory.
