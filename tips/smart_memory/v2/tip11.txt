# 1. The core thesis

A superhuman memory system needs **six properties at once**:

1. **Human-like separation of memory types**: working, episodic, semantic, procedural, resource, belief, concept, formal, eval.
2. **Machine-superhuman provenance**: every claim links to source spans, hashes, derivations, tool traces, and validation receipts.
3. **Temporal truth**: it knows what was believed, when, why, what superseded it, and what remains disputed.
4. **Compression without amnesia**: raw evidence is never destroyed; summaries, concept cards, and field maps are compiled projections.
5. **Fast recall**: hot recall must be index-only, parallel, bounded, and usually no-LLM.
6. **Compounding intelligence**: every search, failure, proof attempt, code run, retrieval success, and contradiction changes future retrieval and reasoning.

The context window is **not memory**. The context window is **L1 cache**.

The true memory is:

```text
append-only event ledger
+ typed memory objects
+ temporal graph
+ hybrid search indexes
+ concept cards
+ executable skills
+ evaluation history
+ git-backed human-readable projections
```

# 2. What I would change from the uploaded CMC/MNEMOS specs

The uploaded designs are very strong, but I would make these corrections.

## 2.1 Contradiction is not always supersession

CMC says: if new claim contradicts old claim, close the old `valid_to`. That is correct for facts like “user’s job changed,” but wrong for science.

In science:

```text
Paper A supports hypothesis H under assumptions A1/A2.
Paper B contradicts H under assumptions B1/B2.
Both remain historically and epistemically valid evidence.
```

So MNEMOS-Ω separates:

```text
Temporal validity: when a statement was asserted or applicable.
Epistemic status: current belief about whether it is supported.
Belief state: agent’s current conclusion after weighing evidence.
```

A claim can be old, contradicted, still important, and still retrievable.

## 2.2 Decay should not mean forgetting truth

Ebbinghaus-style decay is useful, but it should decay **retrieval priority**, not erase memory. A rarely used theorem should not become “less true.” Its `utility_score` can decay; its evidence remains.

## 2.3 Observational Memory should be an L1 continuity layer, not the whole memory

Mastra’s Observational Memory is very strong because it keeps a stable, cacheable compressed prefix and avoids dynamic per-turn retrieval, with reported strong LongMemEval results. But for scientific research, you still need exact citations, graph traversal, equation lookup, and source-grounded retrieval. So MNEMOS-Ω uses Observational Memory as the **hot continuity layer**, not as the canonical truth store. ([Mastra][2])

## 2.4 Petgraph alone is not enough for 10M+ memories

`petgraph` is excellent for in-memory prototypes. For millions of memories, use:

```text
typed edge tables
+ compressed sparse row adjacency
+ temporal interval indexes
+ cached PPR neighborhoods
+ optional Qdrant/Graphiti/Kuzu adapters
```

Graphiti’s core idea is crucial: temporal context graphs track evolving facts and provenance, and support hybrid retrieval over time, full text, semantics, and graph structure. ([Zep Documentation][3])

## 2.5 The agent should propose memory patches, not freely mutate truth

Letta/MemGPT-style self-editing memory is powerful, but unrestricted self-editing can corrupt canonical memory. Let the agent update working memory freely; for durable semantic/procedural/science memory, require typed memory patches, validators, receipts, and optional human or policy gates. Letta’s memory-block model is excellent for always-visible working state, but canonical scientific memory needs stricter governance. ([Letta Docs][4])

# 4. Memory lanes

MNEMOS-Ω has **12 lanes**, but they are not 12 unrelated databases. They are typed views over one canonical memory kernel.

| Lane            | Purpose                                             |                                           Examples |              Hot recall? |
| --------------- | --------------------------------------------------- | -------------------------------------------------: | -----------------------: |
| **Core**        | Stable mission, identity, rules, project invariants |  “OpenQG is Rust-first; do not store data in repo” |                   Always |
| **Working**     | Current task state, active hypothesis, current plan |    “Reading spin foam literature; next query is X” |                   Always |
| **Observation** | Dense, dated continuity log                         |       “At 14:02, failed build due to trait bounds” |                     High |
| **Episodic**    | Immutable event history                             | search run, code run, paper read, user instruction |                   Medium |
| **Resource**    | External reality pointers                           | arXiv PDF, DOI, Git commit, dataset DOI, file hash |               High exact |
| **Semantic**    | Atomic claims/facts/definitions                     |                            “Paper P assumes Λ > 0” |                     High |
| **Belief**      | Current weighted conclusions                        |          “Claim C likely false under assumption A” |              Medium/high |
| **Concept**     | Zettelkasten concept cards and topic kernels        |         “Gauge invariance as organizing principle” |       High for synthesis |
| **Formal**      | Equations, theorems, proofs, units, derivations     |                    TeX, SymPy form, Lean statement |            High for math |
| **Procedural**  | Executable skills, scripts, workflows               |      “download dataset X,” “parse arXiv equations” |          High for action |
| **Eval**        | What worked/failed                                  | retrieval failures, bad answers, tests, benchmarks | High for self-correction |
| **Vault**       | Sensitive/private external pointers                 |      credentials, private notes, encrypted secrets |             Policy gated |

This extends MIRIX’s excellent six-memory taxonomy and adds the formal/eval/belief layers needed for serious science and math. MIRIX’s docs describe the six memory components and retrieval across BM25 and embedding search; MNEMOS-Ω keeps that backbone but makes provenance, formal objects, and eval memory first-class. ([Mirix Documentation][5])

# 5. The key architectural object: not “memory,” but “evidence-backed typed state”

A memory object should not just be text plus embedding. It needs identity, time, source, status, confidence, links, utility, and lifecycle.

```rust
use chrono::{DateTime, Utc};
use serde::{Deserialize, Serialize};
use uuid::Uuid;

#[derive(Debug, Clone, Copy, Serialize, Deserialize, PartialEq, Eq, Hash)]
pub struct MemoryId(pub Uuid);

#[derive(Debug, Clone, Copy, Serialize, Deserialize, PartialEq, Eq, Hash)]
pub struct EpisodeId(pub Uuid);

#[derive(Debug, Clone, Copy, Serialize, Deserialize, PartialEq, Eq, Hash)]
pub struct ResourceId(pub Uuid);

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, Hash)]
pub enum MemoryLane {
    Core,
    Working,
    Observation,
    Episodic,
    Resource,
    Semantic,
    Belief,
    Concept,
    Formal,
    Procedural,
    Eval,
    Vault,
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
pub enum Lifecycle {
    Proposed,
    Trusted,
    Consolidated,
    Deprecated,
    Superseded,
    Contradicted,
    Rejected,
    Archived,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TemporalWindow {
    /// When the statement/event/resource became applicable or was asserted.
    pub valid_from: DateTime<Utc>,

    /// When it stopped being applicable.
    /// Important: contradiction does not always close this.
    pub valid_to: Option<DateTime<Utc>>,

    /// When MNEMOS-Ω recorded this object.
    pub transaction_time: DateTime<Utc>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct EvidenceSpan {
    pub resource_id: ResourceId,
    pub uri: String,
    pub byte_start: Option<u64>,
    pub byte_end: Option<u64>,
    pub page: Option<u32>,
    pub section: Option<String>,
    pub line_start: Option<u32>,
    pub line_end: Option<u32>,
    pub quote_hash: String,
    pub quote_excerpt: Option<String>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Provenance {
    pub source_spans: Vec<EvidenceSpan>,
    pub extractor: ExtractorRef,
    pub source_hash: String,      // BLAKE3 or SHA-256
    pub extraction_hash: String,  // hash of prompt/model/config/output
    pub confidence: f32,
    pub verification: VerificationState,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum VerificationState {
    Unverified,
    SourceSpanVerified,
    UnitCheckPassed,
    FormalCheckPassed,
    ReproducedBySkill(MemoryId),
    HumanReviewed,
    Failed(String),
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct MemoryObject {
    pub id: MemoryId,
    pub schema_version: u16,
    pub lane: MemoryLane,
    pub lifecycle: Lifecycle,

    pub title: String,
    pub content: MemoryContent,

    pub temporal: TemporalWindow,
    pub provenance: Provenance,

    pub tags: Vec<String>,
    pub entities: Vec<EntityRef>,
    pub links: Vec<LinkRef>,

    pub utility: UtilityState,
    pub compression: CompressionState,
}
```

The uploaded Claude spec correctly adds equation/theorem/counterexample stores. MNEMOS-Ω goes further: **formal and empirical objects are not generic notes**.

## 6.3 Belief object

This is where MNEMOS-Ω differs from ordinary KGs. A claim is not a belief. A belief is the agent’s current epistemic synthesis over claims.

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Belief {
    pub proposition: String,

    /// Weighted evidence supporting the belief.
    pub positive_evidence: Vec<EvidenceWeight>,

    /// Weighted evidence against the belief.
    pub negative_evidence: Vec<EvidenceWeight>,

    /// Current calibrated estimate.
    pub confidence: f32,

    /// Why the belief changed.
    pub revision_history: Vec<BeliefRevision>,

    /// What would change the belief.
    pub falsification_conditions: Vec<String>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct EvidenceWeight {
    pub memory_id: MemoryId,
    pub direction: EvidenceDirection,
    pub source_quality: f32,
    pub methodological_quality: f32,
    pub recency_weight: f32,
    pub directness: f32,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum EvidenceDirection {
    Supports,
    Challenges,
    NeutralContext,
}
```

This borrows Hindsight’s crucial separation between experience, facts, beliefs, and mental models, but makes it science-grade.

# 7. Temporal epistemic graph

MNEMOS-Ω uses a graph, but not a naive graph.

Each edge has:

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TemporalEdge {
    pub id: EdgeId,
    pub source: MemoryId,
    pub target: MemoryId,
    pub relation: Relation,

    pub temporal: TemporalWindow,
    pub provenance: Provenance,

    /// Strength for spreading activation.
    pub weight: f32,

    /// Current epistemic state of the edge itself.
    pub status: EdgeStatus,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum Relation {
    Supports,
    Challenges,
    Refines,
    Contradicts,
    Supersedes,
    DependsOn,
    DerivedFrom,
    Assumes,
    UsesDataset,
    Implements,
    Cites,
    AuthoredBy,
    SameAs,
    SimilarTo,
    PrerequisiteOf,
    FailedBecause,
    SolvedBy,
    TestedBy,
    Mentions,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum EdgeStatus {
    Current,
    Historical,
    Disputed,
    Deprecated,
    Rejected,
}
```

Graphiti’s temporal context graph idea is central here: facts/relationships should have validity windows and trace back to source episodes, while retrieval combines semantic, keyword, time, and graph traversal signals. ([Zep Documentation][3])

# 8. Storage design

## 8.1 Canonical truth

Canonical truth is event-sourced:

```text
.mnemos/
  ledger/
    mutations-000001.jsonl.zst
    mutations-000002.jsonl.zst
    hash-chain.toml

  objects/
    objects.redb
    resources.redb
    episodes.redb

  projections/
    metadata.parquet
    claims.parquet
    equations.parquet
    edges.parquet
    retrieval_stats.parquet

  indexes/
    tantivy/
    vector/
    graph_csr/
    temporal/
    symbol/
    code_scip/

  cards/
    core/
    concepts/
    papers/
    equations/
    datasets/
    software/
    skills/
    syntheses/

  evals/
    golden_questions.jsonl
    retrieval_failures.jsonl
    contradiction_tests.jsonl
    skill_tests.jsonl
```

The **ledger** is canonical. Everything else is a projection.

This is the most important robustness feature. If indexes corrupt, rebuild. If cards drift, regenerate. If an agent writes garbage, roll back via event log.

## 8.2 Why these substrates

* **Tantivy** for Rust-native BM25, phrase queries, facets, fast fields, incremental indexing, and low startup overhead. Tantivy is a Lucene-inspired full-text search library written in Rust, with BM25 scoring and incremental/multithreaded indexing. ([GitHub][6])
* **Qdrant or local HNSW** for dense/sparse/multivector retrieval. Qdrant is Rust-powered and supports native dense+sparse hybrid search, metadata filtering, multivectors, HNSW, and sparse indexes. ([Qdrant][7])
* **BGE-M3** as the default local embedding model when practical because it emits dense, sparse, and multivector representations and handles up to 8192-token inputs across 100+ languages. ([BGE Model][8])
* **DataFusion + Arrow/Parquet** for analytical projections, dashboards, eval aggregation, and large local memory analytics. DataFusion is a Rust query engine built on Arrow with SQL/DataFrame APIs and vectorized, multithreaded execution. ([Apache DataFusion][9])
* **SCIP** for source-code memory because it provides a language-agnostic code intelligence protocol for go-to-definition, references, implementations, and cross-language indexing, with support from Rust tooling such as rust-analyzer. ([GitHub][10])
* **RO-Crate** metadata for research objects and data/software provenance; RO-Crate uses JSON-LD to describe datasets, files, software, workflows, people, organizations, licenses, and provenance. ([researchobject.org][11])
* **DVC/DataLad-style external data management** for datasets: commit manifests, scripts, hashes, and pipeline definitions, not large data. DVC uses `dvc.yaml` and `dvc.lock` to define reproducible data pipelines versioned with Git. ([DVC][12])

# 10. Write path

```mermaid
sequenceDiagram
    participant A as Agent/Tool
    participant O as Observer
    participant L as Episode Ledger
    participant X as Extractor
    participant V as Validators
    participant P as Patch Queue
    participant K as Memory Kernel
    participant I as Index Projections
    participant C as Consolidators

    A->>O: observe(event)
    O->>O: segment + compress into dated observations
    O->>L: append episode with hash receipt
    L->>X: extract entities/claims/equations/skills/resources
    X->>V: candidate memories
    V->>V: source span, duplicate, temporal, contradiction, schema checks
    V->>P: proposed memory patch
    P->>K: commit if policy allows
    K->>L: append mutation event
    K->>I: update BM25/vector/graph/temporal/code indexes
    K->>C: schedule reflection/consolidation/skill distillation
```

## Write path rules

1. **Every raw event becomes an episode receipt.**
2. **Every scientific claim requires source evidence or stays `Proposed`.**
3. **Contradictions create challenge edges; they do not blindly delete or overwrite.**
4. **Semantic memories and belief memories are separate.**
5. **Skills are not trusted until executed or tested.**
6. **Memory cards are projections; the event ledger is canonical.**
7. **The agent proposes patches; validators commit.**

## Retrieval profiles

A single universal retrieval strategy is wrong. Query type determines the route.

| Query type               | Primary signals                                                           |
| ------------------------ | ------------------------------------------------------------------------- |
| Exact paper/source       | Resource ID, DOI, arXiv, citation graph                                   |
| Equation/math            | symbol index, TeX normalization, BM25, formal dependencies                |
| Broad field synthesis    | concept cards, community summaries, citation graph, representative papers |
| Contradiction check      | temporal graph, belief evidence, counterexamples                          |
| Coding task              | procedural skills, SCIP code symbols, repo cards, prior failures          |
| Dataset task             | dataset manifests, download scripts, license/checksum metadata            |
| “What did I try?”        | episodic ledger, observation memory, eval memory                          |
| Open-question generation | coverage map, weakly supported beliefs, contradiction clusters            |

# 12. Retrieval scoring

Hot recall should be deterministic and fast.

```text
score(memory, query) =
    0.22 * exact_symbol_or_id_match
  + 0.16 * bm25_score
  + 0.16 * dense_similarity
  + 0.10 * sparse_similarity
  + 0.10 * multivector_score
  + 0.12 * graph_activation
  + 0.08 * source_quality
  + 0.06 * utility_score
  + 0.05 * temporal_relevance
  + 0.05 * task_profile_fit
  - 0.12 * contradiction_penalty_unless_requested
  - 0.08 * staleness_penalty_unless_historical
```

Then context packing optimizes:

```text
maximize: usefulness + evidence + diversity + novelty
subject to: token budget, citation requirement, contradiction coverage
```

The context pack is not “top-k chunks.” It is a structured artifact:

```yaml
context_pack:
  id: pack_018f...
  query: "Compare assumptions in paper X and Y"
  profile: "math_contradiction"
  budget_tokens: 12000

  always_visible:
    - core mission
    - current task state

  direct_evidence:
    - claim cards
    - equation cards
    - source spans

  graph_context:
    - assumptions
    - related claims
    - contradictions
    - dependencies

  procedural_context:
    - relevant skills
    - prior failed attempts

  warnings:
    - "Claim C is disputed."
    - "Equation E has no formal verification."
    - "Dataset D has unclear license."

  omitted:
    - memory_id: mem_...
      reason: "lower score; redundant with included source"
```

# 13. Context pack contract

A valid `ContextPack` must satisfy:

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ContextPack {
    pub id: PackId,
    pub query: String,
    pub created_at: DateTime<Utc>,
    pub token_budget: usize,
    pub profile: RetrievalProfile,

    pub sections: Vec<PackSection>,
    pub citations: Vec<PackCitation>,
    pub contradictions: Vec<ContradictionNotice>,
    pub uncertainty: Vec<UncertaintyNotice>,
    pub omitted: Vec<OmittedCandidate>,

    pub retrieval_trace: RetrievalTrace,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum PackSectionKind {
    CoreState,
    TaskState,
    DirectAnswerEvidence,
    SourceSpans,
    EquationContext,
    ConceptContext,
    GraphNeighborhood,
    BeliefState,
    ProceduralSkills,
    PriorFailures,
    OpenQuestions,
    Warnings,
}
```

Hard guarantees:

```text
1. No factual scientific claim without citation or explicit “uncited/proposed” marker.
2. No stale claim marked current unless temporal filter says current.
3. Contradictions appear before synthesis.
4. Counterexamples are preferred over comforting similar memories.
5. Skills include reliability and last failure.
6. Source spans survive summary compression.
7. The pack records why items were omitted.
```

# 15. Knowledge compression hierarchy

MNEMOS-Ω compresses knowledge into **levels**, not one lossy summary.

```text
Level 0: Raw source
  PDF, repo commit, dataset manifest, tool log, conversation

Level 1: Episode / observation
  what happened, when, actor, entities, result

Level 2: Atomic memory
  claim, equation, theorem, resource, skill, failure, question

Level 3: Concept card
  links many atomic memories into a reusable concept

Level 4: Topic kernel
  topic map, key papers, equations, datasets, controversies, skills

Level 5: Field synthesis
  state of field, open problems, best sources, research frontier

Level 6: Core distilled stance
  tiny active guidance loaded into working memory
```

A-MEM is useful for Levels 3–4 because it dynamically generates structured notes, tags, links, and evolving contextual representations. ([Hugging Face][14])

GraphRAG-style community summaries are useful for Level 5, but they must never be the only source of truth. They are maps, not evidence.

# 16. Procedural memory

Procedural memory is not “a note about how to do something.” It is executable.

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Skill {
    pub id: SkillId,
    pub name: String,
    pub description: String,

    pub language: SkillLanguage,
    pub code_ref: CodeRef,

    pub preconditions: Vec<Precondition>,
    pub inputs: Vec<SkillInput>,
    pub outputs: Vec<SkillOutput>,

    pub examples: Vec<SkillExample>,
    pub tests: Vec<SkillTest>,

    pub success_count: u64,
    pub failure_count: u64,
    pub last_success: Option<DateTime<Utc>>,
    pub last_failure: Option<FailureTrace>,

    pub derived_from_episodes: Vec<EpisodeId>,
    pub related_concepts: Vec<MemoryId>,
}

impl Skill {
    pub fn reliability(&self) -> f32 {
        let total = self.success_count + self.failure_count;
        if total == 0 { return 0.5; }
        self.success_count as f32 / total as f32
    }
}
```

Voyager’s key lesson is that agents compound capability by building a skill library and reusing executable skills, not just remembering reflections. ([Hugging Face][15])

For OpenQG-style science work, procedural memory should include skills like:

```text
download_arxiv_pdf
extract_equations_from_latex
query_openalex_citations
search_ads_forward_citations
build_dataset_manifest
preview_dataset_schema
run_lattice_sim_smoke_test
verify_units
summarize_paper_claims
generate_followup_queries
```

# 18. Core engine trait

```rust
#[async_trait::async_trait]
pub trait MemoryEngine: Send + Sync {
    async fn observe(&self, event: ObservationEvent)
        -> anyhow::Result<EpisodeReceipt>;

    async fn propose_patch(&self, patch: MemoryPatch)
        -> anyhow::Result<PatchReceipt>;

    async fn commit_patch(&self, patch_id: PatchId, authority: Authority)
        -> anyhow::Result<CommitReceipt>;

    async fn recall(&self, query: RecallQuery)
        -> anyhow::Result<ContextPack>;

    async fn open(&self, id: MemoryId)
        -> anyhow::Result<MemoryObject>;

    async fn feedback(&self, feedback: PackFeedback)
        -> anyhow::Result<FeedbackReceipt>;

    async fn consolidate(&self, job: ConsolidationJob)
        -> anyhow::Result<ConsolidationReport>;

    async fn explain(&self, target: ExplainTarget)
        -> anyhow::Result<Explanation>;
}
```

The engine is intentionally split:

```text
observe      = record what happened
propose      = suggest what memory should change
commit       = durable mutation with validators
recall       = compile context
feedback     = learn from use
consolidate  = compress and reorganize
explain      = auditability
```

# 20. Real-time performance model

Target scale:

```text
Single-node v1:
  10M memory objects
  100M graph edges
  1B source-span/chunk references if stored as compact projections
  p50 hot recall: <= 35 ms without LLM
  p95 hot recall: <= 150 ms without LLM
  p99 hot recall: <= 500 ms without LLM
  context pack assembly: <= 50 ms after candidates
  synchronous observe receipt: <= 5 ms
  async extraction/indexing: background

Cluster/sharded v2:
  100M+ memories
  sharded by entity/topic/resource hash
  federated context-pack merge
```

The rule:

```text
LLM extraction is allowed on write/consolidation paths.
LLM reranking is optional on cold/high-value recall.
Hot recall must be mostly no-LLM.
```

That is how it stays fast.

# 22. Consolidation algorithms

## 22.1 Observation compression

```text
raw tool/chat/code stream
→ event segmentation
→ dated observations
→ priority tags
→ temporal anchors
→ compact observation block
```

Mastra’s Observer/Reflector pattern is worth stealing because it compresses noisy history into stable observations while preserving temporal anchors. ([Mastra][2])

## 22.2 Concept synthesis

```text
recent memories + highly co-activated memories
→ cluster by embeddings + graph proximity + entities
→ generate concept candidate
→ link to supporting claims/equations/skills
→ create or update ConceptCard
→ validators check citations and duplicate concepts
```

## 22.3 Belief revision

```text
for each disputed proposition:
  collect support/challenge evidence
  weight by source quality, directness, method quality, recency
  update Belief confidence
  emit revision explanation
  schedule contradiction eval
```

## 22.4 Skill distillation

```text
successful repeated episodes
→ action trace mining
→ generate skill candidate
→ sandbox test
→ store only if tests pass
→ link to failures, concepts, resources
```

## 22.5 Retrieval strengthening

After every context pack:

```text
if agent succeeded:
  strengthen included memories
  strengthen co-activation edges
  update retrieval profile weights
else:
  log retrieval failure
  create eval case
  penalize misleading memories
  mine missing evidence
```
