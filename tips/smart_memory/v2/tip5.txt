I unpacked and studied the smart-memory bundle: `CLAUDE_MEMSPEC.md`, `CODEX_MEMSPEC.md`, and `tip1`вЂ“`tip13`, plus your pasted ANTIGRAVITY / Google-style CMC spec. The strongest answer is not вЂњuse a graph DBвЂќ or вЂњbuild an LLM wiki.вЂќ The strongest answer is a **Rust-first Cognitive Memory Compiler**: a local-first, append-only, provenance-backed, temporal, graph/vector/BM25, concept-consolidating, executable-skill memory OS.

The local specs all converge on the same conclusion:

> **Memory must be compiled, not merely stored.**
> Raw events become observations. Observations become episodes. Episodes become claims, equations, resources, skills, failures, concepts, and beliefs. Those are indexed, linked, compressed, tested, recalled, and continuously recompiled into better memory.

I would build the final system as:

# **MNEMOS-О©: Rust Cognitive Memory Compiler**

A superhuman scientific/research memory system for agents that need to learn physics, math, code, literature, datasets, experiments, and workflows over months or years without blowing the context window.

It combines the best ideas from the local specs and the current frontier:

* **MIRIX** for typed human-like memory lanes. MIRIX explicitly provides six memory components: core, episodic, semantic, procedural, resource, and knowledge, with BM25 and embedding-based search. ([Mirix Documentation][1])
* **Graphiti/Zep** for temporal truth, validity windows, provenance, and hybrid graph retrieval. Graphiti tracks changing facts over time, preserves provenance to episodes, supports semantic + keyword + graph traversal, and updates incrementally rather than requiring batch recomputation. ([GitHub][2])
* **A-MEM** for dynamic Zettelkasten-style note evolution, structured note generation, automatic linking, and concept refinement. ([GitHub][3])
* **Mem0** for production-grade multi-signal retrieval: ADD-only extraction, agent-generated facts, entity linking, and fused semantic/BM25/entity retrieval. Its April 2026 results are a useful benchmark target, especially LoCoMo, LongMemEval, and BEAM. ([GitHub][4])
* **HippoRAG** for associative graph activation using knowledge graphs and Personalized PageRank-style multi-hop retrieval. ([Hugging Face][5])
* **GraphRAG/DRIFT** for corpus-level community summaries and global/local search over huge source collections. ([Microsoft GitHub][6])
* **Nemori** for event segmentation and predict-calibrate learning: memory updates should be driven by prediction gaps, not arbitrary token chunks. ([ScienceStack][7])
* **Voyager** for executable procedural memory: skills must be code-tested, versioned, and reusable, not just summarized. ([Hugging Face][8])
* **Tantivy + Qdrant + Candle** as Rust-native or Rust-compatible infrastructure: Tantivy gives Lucene-like BM25/full-text search in Rust, Qdrant is Rust-powered vector/hybrid search at scale, and Candle supports Rust-native model inference. ([GitHub][9])

Reddit/community discussion also strongly reinforces the same practical lesson: the storage layer is not the hard part; lifecycle, relevance policy, conflict resolution, source-of-truth boundaries, and retrieval strategy are the hard parts. Treat that as anecdotal engineering signal, not formal evidence. ([Reddit][10])

# 1. Final thesis

The optimal design is:

> **A Rust-first, append-only, temporal, graph-augmented, hybrid-search, self-consolidating, skill-executing memory compiler with strict provenance and bounded context-pack generation.**

The agent should not вЂњrememberвЂќ by stuffing more text into context.

It should remember by improving these durable structures:

```text
1. Event ledger
2. Observation stream
3. Episode memory
4. Source/resource registry
5. Atomic claim ledger
6. Temporal knowledge graph
7. Concept/Zettelkasten graph
8. Formal math/equation/theorem store
9. Dataset/experiment registry
10. Procedural skill library
11. Counterexample/failure memory
12. Evaluation and retrieval feedback memory
13. Question/search frontier
14. Context-pack compiler
```

The agentвЂ™s context window receives only a small, cited, task-specific **ContextPack** compiled from this memory.

# 3. The core architectural rule

The single most important rule:

> **The agent may propose memory. The compiler validates memory. Only validated memory becomes durable truth.**

Never let the LLM directly mutate trusted memory.

The LLM can call:

```text
propose_claim()
propose_concept_link()
propose_skill_patch()
propose_belief_revision()
```

But promotion requires validators:

```text
schema validation
source-span validation
duplicate detection
contradiction check
temporal validity check
retrieval eval
optional human review
```

This prevents memory corruption over time.

# 5. What the uploaded specs got right

The local specs are very strong. I would preserve most of them, but reorganize them into a more operational, shippable design.

## CLAUDE_MEMSPEC strengths

The CLAUDE spec is strongest on cognitive richness:

```text
Working memory
Sensory buffer
Episodic memory
Semantic memory
Procedural memory
Schema memory
Metacognitive memory
Emotional salience
Theorem store
Equation store
Citation graph
Counterexample store
```

It also correctly includes:

```text
FSRS-style strengthening
Hebbian co-activation
Nemori predict-calibrate reflection
Memory-R1-style policy learning
formal math hooks
units and dimensional analysis
hash-chained episodes
WAL and crash safety
adversarial tests
```

I would keep all of that, but reduce hot-path complexity.

## CODEX_MEMSPEC strengths

The CODEX spec is strongest on engineering discipline:

```text
Git-backed Markdown/YAML cards are curated memory.
SQLite/metadata/indexes are rebuildable projections.
Mutation receipts are append-only.
Context packs are deterministic and cited.
Remote services are optional, not required.
```

I would keep the вЂњcanonical store vs rebuildable projectionвЂќ idea exactly.

## ANTIGRAVITY / Google-style spec strengths

The ANTIGRAVITY spec is strongest on the synthesis:

```text
MIRIX       -> taxonomy
Graphiti    -> temporal truth
A-MEM       -> self-organizing concepts
Voyager     -> executable skills
Mastra OM   -> observation compression
HippoRAG    -> associative retrieval
Mem0        -> fast multi-signal retrieval
```

I agree with this synthesis.

But I would change three things:

1. **Do not make graph traversal sub-millisecond the central promise.** For millions of nodes, retrieval should be fast because it is bounded, cached, and staged, not because the graph DB magically searches everything.
2. **Do not let LLM extraction block the write path.** The synchronous path should append an event and return. Heavy extraction runs async.
3. **Do not use a graph DB as canonical truth.** The canonical truth is the append-only event/mutation ledger plus typed memory objects. Graph/vector/BM25 are projections.

# 6. Final memory lanes

I would use **10 top-level lanes**, with science/math/code subtypes inside them.

```rust
pub enum MemoryLane {
    Core,          // stable mission, identity, project constraints
    Working,       // active task state and temporary scratch memory
    Episodic,      // what happened: runs, searches, conversations, tool outputs
    Semantic,      // atomic claims, definitions, relations
    Formal,        // equations, theorems, proofs, derivations, units
    Concept,       // evolving Zettelkasten concept cards / topic kernels
    Belief,        // current epistemic stance over claims
    Procedural,    // executable skills, scripts, workflows, recipes
    Resource,      // papers, datasets, repos, files, APIs, manifests
    EvalMeta,      // failures, counterexamples, evals, retrieval feedback, lessons
}
```

Why 10?

Because 6 lanes are too coarse for science, and 12вЂ“15 lanes become noisy unless you have strict subtyping. The optimal approach is:

```text
10 lanes at the top
rich content variants below
single universal envelope around all memories
```

# 7. Memory lifecycle

```mermaid
stateDiagram-v2
    [*] --> Observed
    Observed --> Proposed: extractor/router proposes memory
    Proposed --> Rejected: invalid / duplicate / unsafe
    Proposed --> Trusted: validator passes
    Trusted --> Consolidated: recurring use + evidence
    Consolidated --> Archived: low current utility, still valid
    Trusted --> Contradicted: new evidence conflicts
    Consolidated --> Contradicted
    Contradicted --> Superseded: newer belief accepted
    Superseded --> Archived
    Archived --> Reactivated: retrieval/use revives
    Reactivated --> Trusted
```

Critical rule:

```text
Superseded does not mean deleted.
Contradicted does not mean hidden.
Archived does not mean forgotten.
```

The system must answer:

```text
What is true now?
What was believed then?
Why did the agent believe it?
What superseded it?
What source supports it?
What code was written under the old belief?
```

# 8. Canonical storage model

The strongest design is **ledger-first**.

```text
Canonical stores:
  1. append-only event ledger
  2. append-only mutation ledger
  3. typed memory object store
  4. curated Git cards
  5. source/resource manifests

Rebuildable projections:
  1. BM25 index
  2. vector index
  3. graph adjacency index
  4. temporal interval index
  5. entity/symbol index
  6. context-pack cache
  7. concept-neighborhood cache
  8. topic-strength table
```

The canonical store is the source of truth. Indexes are accelerators.

That means if corruption occurs:

```bash
mnemos verify
mnemos rebuild --from-ledger
mnemos diff-projections
```

must reconstruct the system deterministically.

# 11. Write path

The write path must be **fast, durable, and mostly non-LLM**.

```mermaid
sequenceDiagram
    participant Agent
    participant API as mnemos.observe()
    participant Ledger
    participant Router
    participant Queue
    participant Extractor
    participant Validator
    participant Store
    participant Indexes

    Agent->>API: observe(EventEnvelope)
    API->>Ledger: append event + hash chain
    Ledger-->>API: EventReceipt
    API->>Router: cheap route classification
    Router->>Queue: enqueue extraction jobs
    API-->>Agent: MutationReceipt in <5ms target

    Queue->>Extractor: async claim/equation/entity extraction
    Extractor->>Validator: proposed memories
    Validator->>Store: apply valid mutations
    Store->>Indexes: update projections
    Indexes-->>Store: projection receipts
    Store->>Ledger: append mutation receipt
```

The synchronous path does only this:

```text
1. Validate event envelope.
2. Hash and append to ledger.
3. Add cheap metadata.
4. Enqueue async extraction.
5. Return receipt.
```

The async path does:

```text
entity extraction
claim extraction
equation extraction
resource registration
contradiction detection
concept linking
skill candidate detection
index updates
```

This is essential. LLM extraction on every hot write causes latency and cost blowups.

# 12. Read path / recall compiler

The read path is where the system becomes superhuman.

```mermaid
flowchart LR
    Q[User/Agent Query] --> QC[Query Classifier]
    QC --> RP[Recall Plan<br/>intent, lanes, time, budget]

    RP --> B[BM25 / Exact]
    RP --> V[Dense Vector]
    RP --> E[Entity Match]
    RP --> S[Symbol / Equation]
    RP --> T[Temporal Index]
    RP --> SK[Skill Preconditions]

    B --> F[Rank Fusion]
    V --> F
    E --> F
    S --> F
    T --> F
    SK --> F

    F --> PPR[Graph Activation<br/>bounded PPR / spreading activation]
    PPR --> TF[Temporal + Trust Filter]
    TF --> CR[Contradiction Resolver]
    CR --> RR[Reranker]
    RR --> PACK[ContextPack Builder]
    PACK --> OUT[Bounded cited context]
```

The retrieval compiler does not ask: вЂњWhat chunks are similar?вЂќ

It asks:

```text
What kind of question is this?
Which memory lanes matter?
Which time window matters?
Are exact symbols or IDs present?
Are there contradictions?
Are there skills needed?
What does the graph neighborhood imply?
What can fit in the context budget?
What must be cited?
What was omitted?
```

# 13. Retrieval scoring formula

Candidate score:

```text
score(candidate, query) =
    Wdense      * dense_similarity
  + Wbm25       * bm25_score
  + Wentity     * entity_overlap
  + Wsymbol     * exact_symbol_match
  + Wgraph      * graph_activation
  + Wtemporal   * temporal_relevance
  + Wskill      * precondition_match
  + Wutility    * utility_score
  + Wfreshness  * freshness_score
  + Wtrust      * epistemic_score
  - Pstale      * stale_penalty
  - Pcontradict * unresolved_contradiction_penalty
  - Pduplicate  * redundancy_penalty
  - Pprivacy    * access_penalty
```

The weights are not fixed globally. They depend on intent.

Example intent profiles:

```text
Exact equation query:
  BM25/symbol/entity high
  dense medium
  graph medium
  freshness low

Literature survey:
  graph/community/source quality high
  BM25 medium
  dense medium
  freshness high

Debugging failed code:
  episodic/procedural/eval high
  time high
  skill preconditions high

Current belief:
  belief/semantic/contradiction/temporal high
  old superseded facts included as caveats

Open research question:
  concept graph + contradictions + frontier high
  exact source evidence high
```

# 15. Knowledge compression hierarchy

This is where the system compounds knowledge.

```text
Level 0: Raw source pointer
  - PDF, repo commit, dataset DOI, tool log
  - stored outside repo or in cache
  - hash and URI committed

Level 1: Observation
  - dense dated summary of what happened
  - "Agent searched arXiv for X and found Y"

Level 2: Episode
  - immutable event object with actors, source refs, result, timestamp

Level 3: Atomic memory
  - claim, equation, theorem, dataset, skill, failure, question

Level 4: Concept card
  - Zettelkasten node linking claims/equations/skills/resources

Level 5: Topic digest
  - compressed state of a research area
  - what is known, disputed, missing, stale

Level 6: Community report
  - GraphRAG-style cluster summary for large corpora

Level 7: Policy/skill update
  - "When searching this domain, use these sources first"
  - "This workflow reliably extracts equations"
```

The rule:

> **Summaries are compression. Sources are truth.**

Never let a summary become the only memory of a source.


### `reflect`

Use after a batch of work.

```text

```

### `consolidate`

Use during idle cycles.

```text

```

```text
  topic: "differential geometry prerequisites",
  max_items: 20,
  mode: "active_retrieval_test",
})
```

### `verify`

Use before trusting generated knowledge.

```text

  object_id: "eq_...",
  checks: ["units", "source_span", "symbol_consistency"],

```

### `explain`

Use whenever retrieval seems wrong.

```text

  target: "context_pack:...",
  include_omitted: true,
  include_scores: true,

```

This is crucial. A memory system that cannot explain why it retrieved something will eventually rot.

# 20. Temporal truth model

For science and software, truth changes.

A memory edge must have:

```text
valid_from
valid_to
transaction_from
transaction_to
```

This is bitemporal:

```text
valid time       = when the fact is true in the world
transaction time = when the system recorded/believed it
```

Example:

```text
2026-01-01:
  Claim: Dataset X is available at URL A.
  valid_from = 2026-01-01
  valid_to = null

2026-03-01:
  New evidence: URL A is dead, dataset moved to URL B.
  Old claim valid_to = 2026-03-01
  New claim valid_from = 2026-03-01
  Supersedes edge added
```

Now the agent can answer:

```text
Where is the dataset now?
Where did we think it was in January?
Which scripts still use the old URL?
```

# 22. Skill memory

Procedural memory is where most systems are weak.

A skill must not be:

```text
"Here is a summary of how to do it."
```

A skill must be:

```text
tested executable code
with preconditions
with examples
with reliability stats
with failure history
with versioning
```

Example skill card:

```yaml
id: skill_extract_arxiv_equations_v3
lane: Procedural
title: Extract TeX equations from arXiv source tarball
language: Rust
code_ref:
  repo_path: skills/extract_arxiv_equations/src/lib.rs
  commit: abc123
preconditions:
  - arxiv_id is known
  - source tarball is available
postconditions:
  - equations.jsonl emitted
  - each equation has source line span
verification:
  ast_valid: true
  unit_tests_passed: true
  sandbox_passed: true
reliability:
  success_count: 84
  failure_count: 7
failure_modes:
  - source unavailable
  - macro expansion failure
```

When a skill fails:

```text
1. append episode
2. record failure
3. link failure to skill
4. propose patch
5. verify patch
6. promote new version if tests pass
7. supersede old version
```

# 24. Corpus-level memory

For huge collections of papers, use two graph layers.


GraphRAG/DRIFT-style global/local search matters here: global community reports are good for broad вЂњcatch me upвЂќ questions, while local graph traversal is better for precise evidence. ([Microsoft GitHub][6])

# 26. Scaling to millions of memories

The key to million-scale memory is **bounded retrieval**.

Never traverse the full graph.

Use:

```text
1. shard by domain / project / actor / time
2. retrieve seed candidates with BM25 + vector + entity
3. graph-expand only from top seeds
4. cap hops and fanout
5. cache frequent neighborhoods
6. precompute topic/community summaries
7. quantize dense vectors
8. compact stale low-utility records into summaries while retaining source pointers
9. keep exact/symbol indexes separate from dense vectors
10. rebuild projections deterministically
```

Graph activation policy:

```rust
pub struct GraphActivationPolicy {
    pub max_seed_nodes: usize,      // e.g. 32
    pub max_hops: u8,               // e.g. 2 or 3
    pub max_nodes_touched: usize,   // e.g. 2_000
    pub relation_weights: HashMap<RelationKind, f32>,
    pub temporal_cutoff: Option<TimeScope>,
    pub include_superseded: bool,
    pub include_contradictions: bool,
}
```

This prevents graph explosion.

# 37. The single biggest implementation risk

The biggest risk is not storage.

It is **memory pollution**.

Bad memories come from:

```text
LLM extraction errors
ambiguous source text
unreviewed social posts
duplicate facts
outdated facts
over-compression
agent-generated false conclusions
```

Mitigation:

```text
Proposed != Trusted
Summaries != Truth
Beliefs != Facts
Sources != Claims
Contradictions != Deletions
Skills != Trusted until tested
Indexes != Canonical
```

That set of boundaries is what makes the system safe over time.
