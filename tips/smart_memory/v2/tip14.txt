The best answer is not another memory database. It is a **Rust-native Cognitive Memory Compiler** that turns raw experience into typed, temporal, compressed, source-backed, executable, and testable knowledge.

Below is the single best design I would build.

# MNEMOS-Σ: Superhuman Cognitive Memory Compiler

**Design goal:** a real-time, Rust-native memory substrate for agents that can learn from search, code, experiments, papers, proofs, data, failures, user guidance, and tool results; organize that knowledge across millions of memories; recall the right compressed context quickly; and compound intelligence over time.

The core idea:

```text
Do not make the vector DB the memory.
Do not make the LLM remember everything.
Do not let the agent directly corrupt its own memory.

Make an immutable, typed, provenance-backed memory ledger the truth.
Compile fast indexes, concept graphs, skill libraries, and context packs from it.
```

This design synthesizes the strongest ideas from your Claude, Codex, and Gemini memspecs:

```text
MIRIX          → typed cognitive lanes: core, episodic, semantic, procedural, resource, vault
Hindsight      → separate facts, experiences, observations, beliefs; retain/recall/reflect
Graphiti/Zep   → temporal knowledge graph with source provenance and changing facts
A-MEM          → Zettelkasten-style self-organizing concept evolution
Nemori         → event segmentation and predict-calibrate learning from surprise
Mastra OM      → observer/reflector compression and stable cacheable context
Mem0           → fast multi-signal retrieval: semantic + BM25 + entity matching
HippoRAG       → graph spreading activation / Personalized PageRank for multi-hop recall
Voyager/Memp   → executable skill memory that improves through test feedback
Memory-R1      → learned memory manager policy for ADD / UPDATE / DELETE / NOOP
MemoryOS/MemOS → lifecycle, tiering, scheduling, governance
```

MIRIX is the strongest current reference for typed memory because it uses six memory components and dedicated managers; Graphiti/Zep is the strongest reference for temporal context graphs that track how facts change over time with provenance; A-MEM is the clearest self-organizing Zettelkasten-style memory; Mastra Observational Memory shows that observer/reflector compression can maintain a stable cacheable context; Mem0 demonstrates highly practical multi-signal recall; HippoRAG shows why graph activation is powerful for multi-hop recall; and Memory-R1 points toward learned memory-management policies rather than fixed heuristics. ([Mirix Documentation][1])

I would call the system **MNEMOS-Σ**, where `Σ` means the memory is not merely stored; it is **summed, compressed, scored, revised, and compiled**.

# 3. Non-negotiable invariants

These are the rules that prevent long-term memory from turning into hallucinated sludge.

```text
1. The ledger is truth; indexes are projections.
2. Nothing factual becomes trusted without provenance.
3. Contradictions do not overwrite history; they close validity windows.
4. The LLM proposes memory mutations; validators commit them.
5. Skills are not trusted unless they execute and pass tests.
6. Summaries are never the only source of truth.
7. Context packs are compiled, bounded, cited, and explainable.
8. Counterexamples and failures never decay away completely.
9. Search/retrieval has to be fast without an LLM in the hot path.
10. Every recall, mutation, and consolidation emits a trace.
```

# 5. The 11 memory lanes

The biggest upgrade over a normal RAG system is strict type separation.

## Lane 0 — Core / Working Memory

Purpose: tiny, always-visible state.

Stores:

```text
active task
mission
current constraints
current research frontier
active hypotheses
user-approved preferences
agent identity and safety policy
```

This is inspired by MemGPT/Letta’s in-context core memory and archival memory split. Letta’s docs describe core memory as in-context memory blocks and archival memory as out-of-context storage, which is the right pattern for bounded agent state. ([Letta Docs][2])

Target size:

```text
2–16 KB, not millions of items
```

## Lane 1 — Sensory / Observation Memory

Purpose: real-time continuity without storing noisy raw streams.

Stores compressed observations:

```text
2026-05-12T20:03Z: Agent searched arXiv for loop quantum gravity datasets.
2026-05-12T20:05Z: Build failed because ndarray trait bound was missing.
2026-05-12T20:09Z: User clarified that raw datasets must not be committed to repo.
```

Mastra’s Observational Memory uses an Observer and Reflector to compress raw conversation into dense observations, maintaining a stable prompt-cacheable context rather than dynamically injecting different retrieval every turn. ([Mastra][3])

## Lane 2 — Episodic Memory

Purpose: immutable record of what happened.

Stores:

```text
search runs
tool calls
paper ingestion sessions
code execution traces
dataset previews
failed assumptions
human review events
agent decisions
```

Episodic records are never trusted claims by themselves. They are evidence that something occurred.

## Lane 3 — Semantic Temporal Knowledge Graph

Purpose: what is claimed to be true, by whom, when, and under what validity window.

Stores:

```text
entities
claims
relations
definitions
source-backed facts
temporal facts
contradictions
supersession edges
```

Graphiti/Zep is the reference design here: it builds temporal context graphs that track how facts change over time, maintain provenance, and support learned or prescribed ontologies. ([GitHub][4])

## Lane 4 — Belief / Hypothesis Memory

Purpose: separate **what sources say** from **what the agent currently believes**.

Stores:

```text
current confidence in a claim
hypotheses under test
Bayesian evidence aggregation
uncertainty
open disputes
review status
```

This is the key Hindsight lesson. Hindsight separates world facts, experiences, observations/entity summaries, and beliefs/opinions, using retain/recall/reflect operations instead of collapsing all memory into snippets. ([Emergent Mind][5])

Example:

```yaml
belief_id: belief.qg.asymptotic_safety.fixed_point
belief: "There is nontrivial theoretical support for a UV fixed point, but no direct empirical confirmation."
confidence: 0.61
support_claims: [...]
contradicting_claims: [...]
status: disputed
```

## Lane 5 — Concept / Schema Memory

Purpose: self-organizing knowledge compression.

Stores:

```text
concept cards
topic maps
field maps
schema memories
prerequisite graphs
analogy links
community summaries
```

A-MEM is the reference: it creates structured notes and dynamically links memories using Zettelkasten principles, enabling continuous memory evolution. ([Hugging Face][6])

## Lane 6 — Formal Science Memory

Purpose: science/math objects that should not be treated as generic text.

Stores:

```text
equations
symbols
units
assumptions
regimes of validity
theorems
proof sketches
formal proof files
observables
datasets
models
simulation parameters
benchmark manifests
```

This is where MNEMOS-Σ becomes stronger than general-purpose agent memory.

Example object types:

```text
EquationCard
TheoremCard
DatasetCard
ObservableCard
ModelCard
ParameterCard
ExperimentCard
BenchmarkCard
```

## Lane 7 — Procedural Skill Memory

Purpose: executable memory.

Stores:

```text
Rust scripts
Python fallback scripts
SQL queries
data downloaders
paper parsers
proof checkers
benchmark runners
debugging playbooks
tool workflows
```

Voyager showed the importance of an ever-growing executable skill library with environment feedback, self-verification, and compositional reuse. ([Hugging Face][7])

A skill is not trusted because the LLM wrote it. It is trusted because it ran.

## Lane 8 — Resource / Vault Memory

Purpose: pointers to external reality.

Stores:

```text
arXiv IDs
DOIs
Git commit hashes
dataset URLs
checksums
file paths
API schemas
licenses
private credentials
```

Private vault items are never rendered into normal context packs unless the caller has permission.

## Lane 9 — Counterexample / Failure Memory

Purpose: make forgotten mistakes impossible.

Stores:

```text
failed builds
bad assumptions
falsified hypotheses
contradicted claims
known-bad retrievals
unsafe procedures
failed proof attempts
```

Counterexamples never decay to zero. This is the memory lane that prevents the agent from repeating old mistakes.

## Lane 10 — Eval / Metacognitive Memory

Purpose: let the system learn how well its own memory works.

Stores:

```text
retrieval success/failure
answer faithfulness scores
skill pass/fail history
topic mastery
memory usefulness feedback
bad context packs
reflection lessons
manager-policy training examples
```

This lane is what turns memory from storage into compounding intelligence.

# 6. Storage model

There are two profiles.

## Pure Rust local profile

This is the default build.

```text
Canonical truth:
  append-only BLAKE3 hash-chained JSONL ledger
  redb/fjall key-value object store
  Git-backed curated Markdown/YAML cards

Compiled projections:
  Tantivy BM25 index
  hnsw_rs vector index
  custom mmap graph shards
  custom interval index
  entity alias tables
  skill precondition index
  topic mastery tables
```

Tantivy is the right Rust-native exact-search layer because it is a Lucene-inspired full-text search library with BM25 scoring, phrase queries, incremental indexing, and multithreaded indexing. ([GitHub][8])

## Scaled adapter profile

Use when you need team memory or very large graph workloads.

```text
Vector:
  Qdrant, optional

Graph:
  Kuzu, optional

Analytics / data previews:
  DuckDB, optional

Shared metadata:
  PostgreSQL, optional
```

Qdrant supports dense vectors, sparse vectors, hybrid queries, filtering, and reranking; Kuzu is an embedded graph database with property graphs, Cypher, columnar storage, CSR adjacency structures, vectorized processing, and ACID transactions; DuckDB can inspect Parquet schemas and metadata, which is ideal for dataset previews without storing raw data in the repo. ([Qdrant][9])

The key rule: **adapters must never become the canonical memory.** They are rebuildable.

# 10. Write path

```mermaid
sequenceDiagram
    participant Agent
    participant Observer
    participant Ledger
    participant Extractor
    participant Validator
    participant Store
    participant Indexes
    participant Reflector

    Agent->>Observer: observe(event)
    Observer->>Observer: segment + compress
    Observer->>Ledger: append episode receipt
    Ledger-->>Agent: immediate receipt

    Observer->>Extractor: async extract candidates
    Extractor->>Validator: claims / skills / equations / resources
    Validator->>Validator: provenance, dedupe, contradiction, privacy checks
    Validator->>Store: commit trusted/proposed objects
    Store->>Indexes: update BM25/vector/entity/graph/time
    Store->>Reflector: enqueue if surprise/novelty high
    Reflector->>Store: concept links / lessons / beliefs
```

## Write stages

```text
1. Observe
2. Segment
3. Compress
4. Append episode
5. Extract candidate memories
6. Normalize entities, predicates, units, symbols
7. Ground with source spans and hashes
8. Detect duplicates and contradictions
9. Route into lanes
10. Score confidence, importance, novelty, surprise, utility
11. Commit mutation receipt
12. Update indexes
13. Schedule reflection/consolidation
```

## Real-time behavior

There are two commits:

```text
Fast commit:
  append episode + compressed observation
  target p50: < 5 ms local

Full cognitive commit:
  extract claims, equations, skills, concept links
  async, seconds to minutes depending on source
```

This lets the agent learn immediately from what happened, while deeper extraction happens in the background.

# 11. Read path / recall compiler

The read path must be fast and LLM-free until the final reasoning agent receives context.

```mermaid
flowchart LR
    Q[Query] --> C[Intent Classifier]
    C --> P[Retrieval Profile]

    P --> B1[Core Cache]
    P --> B2[Tantivy BM25]
    P --> B3[Vector HNSW]
    P --> B4[Entity/Alias Search]
    P --> B5[Temporal Interval Search]
    P --> B6[Graph PPR]
    P --> B7[Skill Preconditions]
    P --> B8[Counterexample Bias]

    B1 --> F[RRF Fusion]
    B2 --> F
    B3 --> F
    B4 --> F
    B5 --> F
    B6 --> F
    B7 --> F
    B8 --> F

    F --> R[Rerank + Validity Filter]
    R --> K[Context Pack Compiler]
    K --> O[Small cited context pack]
```

## Retrieval channels

```text
core_channel:
  always include relevant pinned state

lexical_channel:
  exact terms, symbols, paths, paper IDs, theorem names, code identifiers

vector_channel:
  semantic similarity over summaries, claims, observations, concepts

entity_channel:
  entities, aliases, authors, datasets, variables, repos, symbols

temporal_channel:
  facts valid at time T, recently changed facts, event timeline

graph_channel:
  HippoRAG-style activation over relations, concepts, claims, equations

skill_channel:
  retrieve executable skills by task, preconditions, language, reliability

counterexample_channel:
  deliberately retrieve known failures/contradictions
```

## Fusion

Use Weighted Reciprocal Rank Fusion:

```rust
pub fn weighted_rrf(rank: usize, weight: f32, k: f32) -> f32 {
    weight / (k + rank as f32)
}
```

Default fusion weights:

```toml
[retrieval.weights]
core = 1.00
lexical = 0.85
vector = 0.75
entity = 0.80
temporal = 0.70
graph = 0.85
skill = 0.65
counterexample = 0.90
```

Counterexamples get high priority because preventing repeated mistakes is more important than maximizing pleasant recall.

## Final scoring

```text
score =
  0.20 * fused_rank
+ 0.15 * lexical_exactness
+ 0.13 * vector_similarity
+ 0.12 * graph_activation
+ 0.10 * entity_overlap
+ 0.08 * temporal_relevance
+ 0.08 * utility
+ 0.07 * importance
+ 0.05 * confidence
+ 0.04 * recency
+ 0.03 * skill_reliability
- 0.10 * stale_penalty
- 0.10 * access_policy_penalty
```

# 13. Self-organization and compression over time

The system gets smarter through consolidation, not through bigger prompts.

## Compression levels

```text
L0 raw event:
  exact tool output, raw message, raw source pointer

L1 observation:
  dense dated note, 5–40x smaller than raw tool streams

L2 atomic memory:
  claim, equation, resource, skill, question, counterexample

L3 concept card:
  synthesis over many atomic memories

L4 community summary:
  high-level map over concept clusters

L5 curriculum map:
  what is known, unknown, disputed, testable, and skill-backed
```

## Consolidation jobs

```text
dedupe:
  merge near-duplicates while preserving all provenance

concept_kernel:
  create or update concept cards from coactivated memories

belief_revision:
  update belief confidence from support/contradiction evidence

counterexample_scan:
  find old assumptions invalidated by new evidence

skill_distillation:
  promote repeated successful trajectories into tested skills

topic_mastery:
  compute mastery/coverage for topic and subtopics

question_generation:
  generate next searches from gaps, contradictions, missing data

community_summary:
  create GraphRAG-style summaries over concept clusters

archive:
  move low-utility memories out of hot indexes while preserving access

rehearsal:
  schedule important low-retrievability memories for review
```

GraphRAG-style community summaries are useful for global questions over a corpus, while HippoRAG-style graph activation is useful for multi-hop local retrieval; they solve different parts of the memory problem. ([Hugging Face][10])

## Topic mastery score

```text
topic_mastery =
  0.20 * source_coverage
+ 0.15 * claim_density
+ 0.15 * contradiction_resolution
+ 0.15 * retrieval_success
+ 0.10 * skill_coverage
+ 0.10 * formal_object_coverage
+ 0.10 * dataset_coverage
+ 0.05 * recent_rehearsal
```

Example:

```yaml
topic: "quantum_gravity/asymptotic_safety"
mastery: 0.64
coverage:
  sources: 0.72
  claims: 0.81
  equations: 0.54
  datasets: 0.22
  skills: 0.18
  contradictions: 0.61
next_actions:
  - "Find public benchmarkable datasets or rule out availability."
  - "Extract beta-function equations into EquationCards."
  - "Create smoke-test script for observable registry."
```

# 14. Strengthening and forgetting

The system should not forget like a broken chatbot. It should decay access priority while preserving auditability.

## Retrieval strengthening

Every recall logs:

```text
query
candidate pool
included memories
omitted memories
final answer outcome
user/tool feedback
whether cited memory was useful
```

The utility update:

```text
utility_new =
  0.90 * utility_old
+ 0.10 * task_reward
```

Reward signals:

```text
+1.0 memory directly helped answer/action
+0.7 memory was cited and supported final answer
+0.5 memory was retrieved but indirectly useful
 0.0 unknown
-0.5 memory was noise
-1.0 memory caused wrong answer or failed action
```

## FSRS-like rehearsal

Important memories should be rehearsed when retrievability drops. FSRS-style schedulers model difficulty, stability, and retrievability; the Rust implementation ecosystem exists, and the DSR model is a strong fit for memory strengthening at scale. ([GitHub][11])

Use it for:

```text
important concepts
core project constraints
dangerous counterexamples
skills with decaying reliability
formal definitions
topic mastery gaps
```

Do not rehearse everything. Rehearse what matters.

## Forgetting policy

```text
Never delete:
  episodes
  trusted claims
  counterexamples
  mutation receipts
  human review records

Archive from hot indexes:
  low-utility, low-importance, low-centrality memories
  duplicates superseded by better cards
  stale observations with no links

Hard prune:
  rejected ungrounded proposals only
```

# 19. Memory manager policy

Start rule-based. Train later.

## Rule-based baseline

```text
ADD if:
  novelty high
  importance high
  grounded source exists
  no near-duplicate trusted memory exists

UPDATE if:
  same entity/predicate with better provenance
  same concept card needs compression refresh
  same skill improves reliability

SUPERSEDE if:
  new evidence invalidates old validity window
  new skill version passes tests and old one fails

NOOP if:
  low novelty
  low importance
  no provenance
  duplicate of trusted memory

REJECT if:
  unsafe
  secret in public lane
  ungrounded claim pretending to be trusted
  failed schema validation
```

## Learned upgrade

Memory-R1 is the reference direction: train a memory manager to perform structured operations such as ADD, UPDATE, DELETE, and NOOP, and train an answer agent to select relevant entries using outcome-driven reinforcement learning. ([Hugging Face][12])

For MNEMOS-Σ, the training examples come naturally from traces:

```yaml
state:
  query: ...
  retrieved: [...]
  memory_candidates: [...]
action:
  op: ADD
  target: claim...
outcome:
  answer_correct: true
  citation_faithful: true
  skill_passed: false
reward: 0.72
```

Over time, the manager learns:

```text
which memories are worth storing
which summaries are too lossy
which retrieval profiles work for which tasks
which skills are reliable
which topics need more search
```

# 32. The final design rule

The system should not ask:

```text
“What chunks are similar to this query?”
```

It should ask:

```text
What is the task?
What kind of memory does it need?
What was true when?
What evidence supports it?
What contradicts it?
What skills can act on it?
What did we already try?
What context is sufficient, not maximal?
What should be compressed, strengthened, archived, or questioned next?
```

That is the difference between a memory database and a cognitive memory compiler.

**Final recommendation:** build **MNEMOS-Σ** as a Rust workspace with an immutable ledger, typed memory lanes, temporal graph truth, A-MEM concept evolution, observer/reflector compression, executable skill memory, science-specific schemas, fast multi-channel retrieval, and a bounded context-pack compiler. This is the best architecture I would trust for an agent that must compile scientific knowledge, learn from experience, avoid context-window overload, and become more capable over time.
