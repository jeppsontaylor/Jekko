## My answer

The **most human-like public memory architecture I would pick today is Hindsight**, because it separates **world facts**, **agent experiences**, **entity/mental-model summaries**, and **evolving beliefs**, then uses explicit **Retain → Recall → Reflect** operations. That is closer to human memory than a flat vector store because it distinguishes “what happened to me,” “what I believe about the world,” “what I know about this entity,” and “how my beliefs changed.” ([Hugging Face][1])

But for the system you actually want—fast recall, quick learning, self-organizing concepts, compounding intelligence—I would **not use one design alone**. I would combine:

```text
Hindsight      = human-like memory anatomy: facts, experiences, beliefs, mental models
A-MEM          = self-organizing Zettelkasten concept graph
Zep/Graphiti   = temporal knowledge graph and provenance
Mem0/Mastra    = fast production recall and observation compression
MIRIX          = full multimodal human-memory taxonomy
```

The best practical architecture is:

```text
event stream
  -> observations
  -> episodic memory
  -> extracted semantic facts
  -> entity/concept graph
  -> procedural skill memory
  -> reflective belief/mental-model layer
  -> compressed context packs
  -> receipts/evals
```

Below is my ranked top 10.

The scores are my architectural judgment, not a single benchmark average, because these systems use different models, datasets, latency constraints, and evaluation methods. I weighted **human-like structure**, **adaptability/self-organization**, and **speed/practical recall**.

# 1. Hindsight — best overall human-like memory

**Why it ranks #1:** Hindsight is the closest I found to a full “agent mind” memory model. It organizes memory into multiple logical networks: world facts, agent experiences, synthesized entity summaries, and evolving beliefs. It exposes three core operations: **Retain**, **Recall**, and **Reflect**. The GitHub project describes “World,” “Experience,” and “Mental Models” memory, with entities, relationships, time series, sparse/dense vectors, and memory banks. ([Hugging Face][1])

**Theory**

Human-like memory is not just retrieval. It needs:

```text
episodic memory   = what happened
semantic memory   = what is true
belief memory     = what the agent currently thinks
entity memory     = what the agent knows about people/things/concepts
reflection        = what past experience means
```

Hindsight’s key idea is that **experiences and beliefs should not be collapsed into the same store**. An agent should be able to say:

```text
I observed X.
Source Y claims Z.
I currently believe W because X and Z support it.
My belief changed after new evidence E.
```

That is critical for scientific agents, because science is full of uncertain, changing, contradictory evidence.

**Schematic**

```text
Raw event / document / conversation
  -> Retain
      -> store experience
      -> extract entities
      -> extract world facts
      -> attach timestamps/provenance
  -> Recall
      -> query by entity, time, vector, keyword, relation
      -> retrieve facts + experiences + beliefs
  -> Reflect
      -> consolidate repeated patterns
      -> update entity summaries
      -> revise beliefs
      -> write mental models
```

**Build version**

Tables:

```text
episodes(id, actor, event, timestamp, source, provenance)
facts(id, subject, predicate, object, confidence, source_episode)
entities(id, name, type, aliases, summary)
beliefs(id, statement, confidence, supporting_facts, contradicting_facts)
reflections(id, trigger, synthesis, affected_beliefs)
memory_edges(src, relation, dst, valid_time)
```

Indexes:

```text
vector index on episode/fact/entity text
BM25/full-text index
entity index
temporal index
graph adjacency index
```

**Best for OpenQG:** Use this as the top-level architecture. It is especially good for keeping **claims**, **evidence**, **beliefs**, and **research-state changes** separate.

# 2. MIRIX — most human-like multimodal memory taxonomy

**Why it ranks #2:** MIRIX explicitly uses six memory types: **Core**, **Episodic**, **Semantic**, **Procedural**, **Resource Memory**, and **Knowledge Vault**. It is built for multimodal memory, including screenshots and screen activity, and the GitHub repo describes local storage, PostgreSQL BM25 + vector similarity, and text/image/voice/screen capture support. ([Hugging Face][2])

**Theory**

MIRIX is powerful because it acknowledges that agents need several kinds of memory:

```text
Core memory        = stable identity, goals, preferences
Episodic memory    = events and experiences
Semantic memory    = facts and concepts
Procedural memory  = skills and workflows
Resource memory    = files, tools, datasets, artifacts
Knowledge vault    = important durable knowledge
```

This is close to human cognition: humans remember events, facts, habits, tools, people, and long-term beliefs differently.

**Schematic**

```text
input event
  -> classify memory type
  -> write to specialized memory module
  -> retrieve through type-aware router
  -> synthesize answer/context
```

Example routing:

```text
"Who is this person?"        -> Core + Semantic + Episodic
"What did I do yesterday?"   -> Episodic
"How do I run this workflow?"-> Procedural
"Where is the dataset?"      -> Resource Memory
"What matters long-term?"    -> Knowledge Vault
```

**Build version**

Create separate memory namespaces:

```text
core_memory/
episodic_memory/
semantic_memory/
procedural_memory/
resource_memory/
knowledge_vault/
```

Each namespace gets different retention rules:

```text
core:        manually reviewed, rarely changes
episodic:    append-only event stream
semantic:    claim/fact graph
procedural:  tested skills and runbooks
resource:    file/data/code references
vault:       high-confidence durable summaries
```

**Best for OpenQG:** Excellent if you want memory to include papers, code, datasets, screenshots, notebooks, and tool-use history.

# 3. A-MEM — best self-organizing concept memory

**Why it ranks #3:** A-MEM is one of the strongest “memory that organizes itself” designs. It is inspired by **Zettelkasten** note-linking: the agent writes structured notes, creates tags, links related memories, updates older memories, and evolves the network as new information arrives. ([Hugging Face][3])

**Theory**

A-MEM’s key move is to treat each memory as a **living note**, not a static chunk.

A memory note contains:

```text
content
context
keywords
tags
links to related memories
reason for link
evolution history
```

When a new memory arrives, the system does not merely store it. It asks:

```text
What older memories does this connect to?
Does it strengthen them?
Does it contradict them?
Does it require updating tags?
Does it reveal a new concept cluster?
```

That is very human-like. Humans learn by connecting new ideas to old ideas.

**Schematic**

```text
new information
  -> generate structured note
  -> assign tags/keywords
  -> retrieve related memories
  -> create bidirectional links
  -> update older memory descriptions
  -> evolve concept clusters
```

**Build version**

Memory object:

```json
{
  "id": "mem_123",
  "content": "...",
  "summary": "...",
  "tags": ["quantum-gravity", "renormalization"],
  "keywords": ["asymptotic safety", "fixed point"],
  "links": [
    {
      "target": "mem_042",
      "relation": "extends",
      "reason": "Both discuss UV fixed points"
    }
  ],
  "evolution": [
    {
      "date": "2026-05-12",
      "change": "Added relation to Weinberg asymptotic safety"
    }
  ]
}
```

**Best for OpenQG:** Use A-MEM for concept maps, topic pages, and “what should we search next?” It is the best design here for **self-organizing science memory**.

# 4. Zep / Graphiti — best temporal knowledge graph memory

**Why it ranks #4:** Graphiti is an open-source temporal knowledge graph for agents. It tracks changing facts over time, preserves provenance back to raw episodes, supports custom/learned ontology, and incrementally updates the graph without full recomputation. ([GitHub][4])

**Theory**

Most memory systems forget that facts change.

For example:

```text
A paper was a preprint in 2023.
It was accepted in 2024.
A dataset URL moved in 2025.
A claim was contradicted in 2026.
```

A temporal graph stores not just:

```text
Paper -> has_status -> accepted
```

but:

```text
Paper -> has_status -> preprint   valid: 2023-01 to 2024-06
Paper -> has_status -> accepted   valid: 2024-06 onward
```

This is crucial for research memory because scientific claims, benchmarks, authorship, datasets, and software status evolve.

**Schematic**

```text
episode
  -> extract entities
  -> extract relationships
  -> attach valid_time + transaction_time
  -> merge into temporal graph
  -> retrieve by current state or historical state
```

**Build version**

Graph tables:

```text
episodes(id, raw_text, source, observed_at)
nodes(id, type, name, aliases)
edges(id, src, predicate, dst, valid_from, valid_to, confidence, episode_id)
communities(id, topic, summary, member_nodes)
```

Retrieval:

```text
query
  -> entity match
  -> temporal filter
  -> graph expansion
  -> vector/BM25 rerank
  -> context pack
```

**Best for OpenQG:** Use this for **paper citation maps, concept dependency maps, contradiction maps, and time-evolving research frontiers**.

# 5. Letta / MemGPT — best classic stateful-agent memory

**Why it ranks #5:** MemGPT/Letta introduced a clean architecture for agents with limited context: small in-context memory, conversation recall, and archival memory outside the context window. The agent can call tools to edit memory and search archival storage. ([Letta Docs][5])

**Theory**

Letta/MemGPT treats the LLM like an operating system process with memory tiers:

```text
core memory        = always in context
conversation memory= searchable interaction history
archival memory    = large external long-term store
```

This is like a computer memory hierarchy:

```text
registers/cache -> RAM -> disk
```

For agents:

```text
prompt context -> active memory -> archival memory
```

**Schematic**

```text
user/task message
  -> agent reads core memory
  -> searches recall/archival memory when needed
  -> updates memory via tools
  -> compacts old conversation into summaries
```

**Build version**

Core memory:

```text
agent_identity
user_profile
current_project
active_goals
important constraints
```

Archival memory:

```text
documents
past tasks
source notes
tool results
summaries
```

Memory tools:

```text
memory_insert()
memory_replace()
memory_search()
conversation_search()
archive_search()
```

**Best for OpenQG:** Good base for agent state management, especially if you want a durable “research assistant identity” with active goals and long-term project state.

# 6. Mastra Observational Memory — fastest compressed conversation memory

**Why it ranks #6:** Mastra’s Observational Memory uses background agents called an **Observer** and **Reflector** to maintain a dense observation log instead of repeatedly retrieving from raw message history. It emphasizes stable, cacheable, low-latency context and claims strong LongMemEval performance in its own research writeup. ([Mastra][6])

**Theory**

Instead of storing every message and searching dynamically every turn, Observational Memory turns conversation into compact observations:

```text
raw messages -> observations -> reflections -> stable context
```

This is powerful because many agent systems waste time retrieving long chat histories that contain tool noise, repeated instructions, and irrelevant fragments.

**Schematic**

```text
conversation turn
  -> Observer extracts durable observations
  -> Reflector consolidates observations
  -> prompt includes compact observation block
  -> raw history remains available but is not primary context
```

**Build version**

Observation record:

```json
{
  "id": "obs_001",
  "subject": "OpenQG",
  "observation": "User prefers Rust-first implementation.",
  "priority": 9,
  "valid_from": "2026-05-12",
  "source_turns": ["turn_12", "turn_13"],
  "tags": ["architecture", "preference", "rust"]
}
```

Reflection record:

```json
{
  "id": "refl_001",
  "summary": "User is building a Rust-first scientific memory compiler.",
  "supporting_observations": ["obs_001", "obs_002"]
}
```

**Best for OpenQG:** Use this for **fast working memory** and project continuity. It is especially useful when the agent is operating over many sessions.

# 7. MemOS / MemoryOS family — memory as an operating system

**Why it ranks #7:** MemOS frames memory as a first-class system resource. It introduces memory objects/cubes, scheduling, lifecycle management, activation, fusion, and storage layers. The related MemoryOS implementation uses short-term, mid-term, and long-term memory modules with updating, retrieval, and generation components. ([Hugging Face][7])

**Theory**

The key idea is that memory should have an operating system:

```text
memory allocation
memory activation
memory scheduling
memory fusion
memory lifecycle
memory garbage collection
```

This is important for agents because different memories have different costs and lifetimes.

Example:

```text
short-term memory: current task
mid-term memory: recent project state
long-term memory: durable facts, preferences, skills
parameter memory: model-level adaptation
activation memory: cached model/internal state
```

**Schematic**

```text
memory API layer
  -> scheduler
      -> decide what to store
      -> decide what to retrieve
      -> decide what to compress
      -> decide what to forget
  -> storage layer
      -> short-term
      -> mid-term
      -> long-term
      -> vector/graph/blob stores
```

**Build version**

Memory lifecycle:

```text
generate -> activate -> retrieve -> consolidate -> decay/archive -> refresh/delete
```

Memory scheduler policy:

```text
importance
recency
frequency
confidence
source quality
task relevance
cost to retrieve
cost to store
```

**Best for OpenQG:** Use this if you want the memory system to become infrastructure, not just a feature.

# 8. HippoRAG / HippoRAG 2 — best hippocampus-inspired associative retrieval

**Why it ranks #8:** HippoRAG is explicitly neurobiologically inspired. It combines LLMs, knowledge graphs, and Personalized PageRank to mimic hippocampal indexing for associative retrieval. The paper reports improvements for multi-hop QA and much lower cost/latency than iterative retrieval baselines; HippoRAG 2 extends the memory framing. ([Hugging Face][8])

**Theory**

The hippocampus helps humans jump between associated memories:

```text
person -> place -> event -> concept -> consequence
```

HippoRAG tries to reproduce that using a graph.

Instead of retrieving only nearest vector chunks, it does:

```text
query entities
  -> seed graph nodes
  -> run graph propagation / Personalized PageRank
  -> retrieve associated evidence
```

This is very good for multi-hop questions.

**Schematic**

```text
documents
  -> extract entities/triples
  -> build knowledge graph
  -> query seeds graph
  -> Personalized PageRank expands associations
  -> return top supporting passages
```

**Build version**

Components:

```text
entity extractor
triple extractor
graph store
PPR scorer
passage retriever
reranker
```

Retrieval formula:

```text
score = vector_similarity + graph_centrality + entity_overlap + passage_quality
```

**Best for OpenQG:** Use this for “connect distant ideas” tasks, such as linking a mathematical formalism to a dataset, codebase, or apparently unrelated theory.

# 9. Mem0 — fastest production memory layer

**Why it ranks #9:** Mem0 is a widely used open-source memory layer for AI agents. Its newer algorithm emphasizes single-pass ADD-only extraction, entity linking, and multi-signal retrieval using semantic, BM25, and entity matching. Its published paper describes a memory-centric architecture with graph-based memory, extraction, consolidation, and retrieval. ([GitHub][9])

**Theory**

Mem0 is less “human-like” than Hindsight or MIRIX, but it is extremely practical:

```text
extract memory facts quickly
link entities
store compact facts
retrieve with hybrid search
inject only relevant memories
```

This is the design you use when latency and simplicity matter.

**Schematic**

```text
conversation/event
  -> extract compact memory facts
  -> link to user/entity/session
  -> store in vector + graph/hybrid index
  -> retrieve top memories for prompt
```

**Build version**

Memory fact:

```json
{
  "user_id": "u1",
  "entity": "OpenQG",
  "fact": "The project prefers Rust-first implementation.",
  "confidence": 0.92,
  "source": "conversation",
  "created_at": "2026-05-12"
}
```

Retrieval:

```text
query
  -> semantic search
  -> BM25 search
  -> entity match
  -> fuse scores
  -> return compact facts
```

**Best for OpenQG:** Use Mem0-style retrieval for fast project preferences, task memory, and “what did we already decide?” queries.

# 10. Generative Agents + MemoryBank — seminal human-like memory stream and forgetting

**Why it ranks #10:** The Generative Agents architecture introduced a famous human-like loop: memory stream, reflection, and planning. Memories are retrieved using recency, importance, and relevance. MemoryBank adds a long-term memory system inspired by the Ebbinghaus forgetting curve, using reinforcement and forgetting over time. ([Hugging Face][10])

**Theory**

Generative Agents showed that believable behavior emerges from:

```text
memory stream
  -> retrieve relevant memories
  -> reflect into higher-level insights
  -> plan future actions
```

Retrieval uses:

```text
recency     = how recent is this memory?
importance  = how significant is it?
relevance   = how related is it to the current situation?
```

MemoryBank adds forgetting/reinforcement:

```text
unused memory decays
important/repeated memory strengthens
```

That is deeply human-like.

**Schematic**

```text
event
  -> append to memory stream
  -> score importance
  -> retrieve by recency + importance + relevance
  -> periodically reflect
  -> generate plans
  -> reinforce or forget memories over time
```

**Build version**

Memory score:

```text
score(memory, query) =
  α * relevance(memory, query)
+ β * importance(memory)
+ γ * recency(memory)
+ δ * reinforcement(memory)
```

Decay:

```text
strength = initial_strength * exp(-time_since_access / half_life)
```

**Best for OpenQG:** Use this as a behavioral layer for research agents: what to revisit, what to forget, what to reflect on weekly, and what to promote into durable memory.

## Runners-up worth studying

**Cognee** is a strong knowledge-engine style memory system. It combines graph, vector, and relational stores, supports session/permanent memory, and exposes multiple search modes. I would study it if your goal is a practical “memory control plane” for agents rather than a purely cognitive model. ([Cognee][11])

**Reflexion** is not a full memory database, but it is essential for procedural self-improvement. It stores verbal reflections from task feedback in episodic memory, allowing agents to improve without weight updates. ([Princeton University][12])

**Voyager** is essential for skill memory. It builds an ever-growing executable skill library, retrieves relevant skills, and composes them for new tasks. ([Robot Perception Lab][13])

## Core memory tables

```text
episodes
  raw events, tool calls, conversations, source-ingest events

observations
  compact durable statements extracted from episodes

claims
  atomic scientific or project claims with citations

entities
  people, papers, theories, datasets, codebases, equations, concepts

relations
  typed graph edges with confidence and validity windows

beliefs
  current agent-level conclusions with supporting and contradicting evidence

reflections
  synthesized lessons, contradictions, strategy updates

skills
  executable procedures, scripts, workflows, proof lanes

resources
  files, datasets, repos, papers, APIs, local cache pointers

memory_mutations
  append-only audit trail of memory changes
```

## Write path

```text
1. Observe
   Convert raw event into compact observations.

2. Extract
   Pull out claims, entities, relations, equations, skills, datasets.

3. Link
   Connect new memories to old memories.

4. Ground
   Attach citations, source spans, tool receipts, or test evidence.

5. Reflect
   Update beliefs, summaries, contradictions, open questions.

6. Validate
   Check schema, duplicates, contradictions, source quality, confidence.

7. Commit
   Write durable memory diff and receipt.
```

## Read path

```text
1. Classify query
   fact, concept, plan, code, paper, dataset, contradiction, skill, history

2. Retrieve candidates
   vector + BM25 + entity graph + temporal graph + recent observations

3. Expand graph
   follow concept/entity/time/provenance links

4. Rerank
   relevance + source quality + recency + confidence + importance

5. Assemble context pack
   include only bounded, cited, task-relevant memory

6. Answer or act
   cite evidence and write new observations from the result
```

## The key design rule

Never let memory be only:

```text
embedding -> nearest chunks -> prompt
```

That is not human-like enough.

Use:

```text
experience -> observation -> claim -> concept link -> belief -> reflection -> skill
```

That is the path to compounding intelligence.

## My final ranking summary

The most human-like **single** system is probably **Hindsight**. The most human-like **taxonomy** is **MIRIX**. The most self-organizing concept system is **A-MEM**. The best temporal world model is **Zep/Graphiti**. The fastest production memory layer is **Mem0**, with **Mastra Observational Memory** also very strong for compressed conversation memory.

For your project, I would build:

```text
Hindsight core
+ A-MEM concept linking
+ Graphiti temporal graph
+ Mem0 fast retrieval
+ Mastra observation compression
+ HippoRAG associative traversal
+ Reflexion/Voyager skill learning
```

That is the closest current path to an AI-agent memory system that is **human-like, fast, adaptive, and capable of getting smarter through accumulated experience**.
