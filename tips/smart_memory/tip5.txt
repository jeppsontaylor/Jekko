The closest answer to “human-like memory for AI agents” is **not one single repo**. The best current pattern is a **hybrid cognitive memory OS**:

```text
working/core memory
+ episodic memory
+ semantic memory
+ procedural/skill memory
+ temporal knowledge graph
+ self-organizing note/claim graph
+ learned memory manager
+ fast hybrid retrieval
+ periodic consolidation / forgetting / contradiction handling
```

If I had to name the **single most human-like current design**, I would pick **MIRIX** for practical agent memory because it explicitly separates **Core, Episodic, Semantic, Procedural, Resource Memory, and Knowledge Vault** and coordinates them with specialized memory agents. For a science-compounding system like OpenQG, I would combine **MIRIX + Graphiti/Zep + A-MEM + Mem0 + Voyager-style procedural memory + a Memory-R1-style learned memory manager**. MIRIX is closest to a whole cognitive memory system; Graphiti/Zep is strongest for temporal factual memory; A-MEM is strongest for self-organizing concepts; Mem0 is strongest for fast production recall; Voyager is strongest for procedural skill compounding. ([Hugging Face][1])

## Ranking heuristic

These are subjective engineering scores, not a standardized benchmark. I weighted:

```text
Human-like = episodic / semantic / procedural / working memory, time, consolidation, forgetting, reflection.
Adaptable = self-organizes, updates memories, learns from feedback, compounds procedures.
Fast = practical low-latency recall, hybrid retrieval, compact context assembly.
```

## 1. MIRIX — strongest whole cognitive memory system

**Why it ranks #1:** MIRIX is the closest current system to a human-like agent memory stack because it separates memory into six explicit systems: **Core, Episodic, Semantic, Procedural, Resource Memory, and Knowledge Vault**, and uses a multi-agent controller to update and retrieve from them. It also handles multimodal memory, including screen activity and images, which matters because humans remember experiences, not just text. The paper reports strong results on ScreenshotVQA and LoCoMo, and the GitHub repo describes local storage, multimodal input, BM25/vector search, and specialized memory components. ([Hugging Face][1])

**Theory:** MIRIX treats memory as a team of specialized subsystems. Core memory stores durable identity/preferences. Episodic memory stores events. Semantic memory stores facts. Procedural memory stores habits/skills. Resource memory tracks external artifacts. Knowledge Vault stores curated durable knowledge.

**Build schematic:**

```text
interaction / screen / tool trace
        ↓
memory router agent
        ↓
┌────────────┬────────────┬─────────────┬──────────────┬──────────────┬──────────────┐
│ core       │ episodic   │ semantic    │ procedural   │ resource     │ knowledge    │
│ identity   │ events     │ facts       │ workflows    │ files/data   │ curated KB   │
└────────────┴────────────┴─────────────┴──────────────┴──────────────┴──────────────┘
        ↓
retrieval coordinator: BM25 + vectors + entity links + recency + memory type
        ↓
context pack for agent
```

**How to build it:** Use one canonical `memory_event` table, then six typed memory tables. Add an LLM extractor that classifies each new observation into one or more memory types. Use BM25 for exact recall, vector search for semantic recall, and entity linking for cross-memory joins. Add a retrieval coordinator that asks: “Is this a fact, an episode, a skill, a resource, or a durable theory?” before retrieving.

For OpenQG, this maps beautifully:

```text
Core          → project goals, epistemic rules, benchmark policy
Episodic      → search runs, failed builds, paper-reading sessions
Semantic      → claims, equations, definitions, theory facts
Procedural    → download scripts, analysis workflows, benchmark recipes
Resource      → datasets, repos, APIs, PDFs, public data portals
Knowledge     → curated field maps, theory cards, evidence ledgers
```

## 2. Zep / Graphiti — best temporal knowledge-graph memory

**Why it ranks #2:** Zep’s memory architecture uses Graphiti, a temporally aware knowledge graph. It is designed for facts that change over time, provenance, relationship tracking, and low-latency retrieval. The Zep paper reports that Zep outperformed MemGPT on Deep Memory Retrieval and improved accuracy/latency on LongMemEval-style enterprise memory tasks. The open-source Graphiti repo says it tracks how facts change over time, maintains provenance, supports learned/prescribed ontology, and is built for agents operating on evolving real-world data. ([Hugging Face][2])

**Theory:** Human memory is temporal. We do not just remember “X is true”; we remember “X was true at a time, from a source, under conditions.” Graphiti models memory as time-stamped episodes, entities, facts, and relationships.

**Build schematic:**

```text
raw episode
  ↓ extract
entities + facts + relations + timestamps + source provenance
  ↓
temporal graph
  ↓
hybrid search:
  keyword + vector + graph-neighborhood + temporal filters + reranker
  ↓
compact context
```

**How to build it:** Store every paper, conversation, search result, and dataset probe as an episode. Extract entities and claims. Represent them as graph edges with validity intervals:

```text
(Paper)-[:ASSERTS {time, source, confidence}]->(Claim)
(Claim)-[:DEPENDS_ON]->(Equation)
(Theory)-[:PREDICTS]->(Observable)
(Observable)-[:MEASURED_BY]->(Dataset)
```

This is the best design for scientific memory where claims are provisional, sources matter, and older facts can be superseded.

## 3. A-MEM — best self-organizing concept memory

**Why it ranks #3:** A-MEM is explicitly built around **Zettelkasten-like self-organization**. When a new memory arrives, it creates a structured note with context, keywords, tags, and links to related historical memories; it can also update older memories as the memory network evolves. Its GitHub repo describes dynamic indexing/linking, comprehensive note generation, interconnected knowledge networks, and continuous memory evolution. ([Hugging Face][3])

**Theory:** A-MEM is closer to how a serious researcher’s notebook evolves. New ideas do not merely get appended; they connect to older ideas, cause old summaries to change, and create concept clusters.

**Build schematic:**

```text
new observation
  ↓
LLM creates structured note:
  - summary
  - context
  - keywords
  - tags
  - entities
  - importance
  ↓
retrieve related notes
  ↓
LLM creates bidirectional links
  ↓
LLM evolves affected old notes
  ↓
concept graph grows
```

**How to build it:** Keep each memory as an atomic note:

```yaml
id: mem_...
text: ...
context: ...
keywords: [...]
tags: [...]
links:
  supports: [...]
  contradicts: [...]
  elaborates: [...]
  analogizes: [...]
evolution_log:
  - changed_at: ...
    reason: ...
```

For OpenQG, this is ideal for concept compounding: a new paper on asymptotic safety can update the concept cards for RG flow, UV fixed points, effective field theory, and observables.

## 4. Mem0 — fastest practical production memory

**Why it ranks #4:** Mem0 is probably the strongest “ship this now” memory layer. Its current open-source repo describes single-pass add-only extraction, entity linking, multi-signal retrieval across semantic/BM25/entity matching, and reported April 2026 benchmark improvements: LoCoMo 91.6, LongMemEval 93.4, p50 latency around 0.88–1.09s, with under 7k tokens per retrieval call. ([GitHub][4])

**Theory:** Mem0 optimizes for production memory economics: extract salient memories once, avoid stuffing full history into context, retrieve a small high-signal memory pack fast.

**Build schematic:**

```text
conversation / action
  ↓
single-pass memory extraction
  ↓
entity linking + fact storage
  ↓
parallel retrieval:
  vector + BM25 + entity match
  ↓
score fusion
  ↓
small memory bundle
```

**How to build it:** Implement an append-only memory ledger. Do not destructively overwrite memories. Store facts, entities, agent actions, timestamps, and provenance. For retrieval, run three searches in parallel:

```text
1. semantic vector search
2. BM25 keyword search
3. entity / relation search
```

Then fuse results. This is the design I would use as the low-latency front door for OpenQG memory.

## 5. Nemori — most cognitively principled self-organizing memory

**Why it ranks #5:** Nemori is one of the most interesting recent human-inspired designs. It uses a **Two-Step Alignment Principle** inspired by Event Segmentation Theory to turn streams into coherent episodes, and a **Predict-Calibrate Principle** inspired by the Free-energy Principle to learn from prediction gaps rather than passively storing memories. It reports strong results on LoCoMo and LongMemEval, especially in longer contexts. I rank it below MIRIX/Graphiti/A-MEM mostly because it appears less mature as an open-source build target. ([Hugging Face][5])

**Theory:** Humans segment experience into events. We also learn when our predictions fail. Nemori operationalizes both ideas: first decide where one “episode” ends and another begins, then use prediction error to decide what knowledge should evolve.

**Build schematic:**

```text
raw conversation stream
  ↓
event boundary detector
  ↓
episode narrative generator
  ↓
predict current/future answer from memory
  ↓
compare prediction vs actual signal
  ↓
calibrate semantic memory
```

**How to build it:** Add a streaming segmenter. Instead of chunking every N tokens, ask the model to detect event boundaries:

```text
new topic?
new goal?
new entity?
new contradiction?
new outcome?
```

Then generate an episode summary and run a prediction test: “What would memory have predicted before seeing this?” If the answer differs from reality, update semantic memory.

This is extremely relevant for agents that need to truly learn from experience rather than simply archive transcripts.

## 6. Memory-R1 / Mem-α — best learned memory-management policy

**Why it ranks #6:** Most memory systems are heuristic: they use hand-written rules for what to store, update, delete, and retrieve. Memory-R1 and Mem-α move toward **training the memory manager itself**. Memory-R1 uses RL-trained specialized agents for structured operations like ADD, UPDATE, DELETE, and NOOP, plus an Answer Agent that selects relevant entries. Mem-α trains agents to construct core, episodic, and semantic memories with reward from downstream QA performance. ([Hugging Face][6])

**Theory:** The most human-like long-term direction is not “LLM plus vector DB”; it is a memory controller that learns which memories matter because downstream performance rewards it.

**Build schematic:**

```text
interaction sequence
  ↓
memory manager policy
  ├─ ADD
  ├─ UPDATE
  ├─ DELETE / DEPRECATE
  └─ NOOP
  ↓
retrieval policy
  ↓
answer / task result
  ↓
reward signal
  ↓
policy improvement
```

**How to build it:** Start rule-based, then collect training examples. For every task, log:

```text
state before memory write
memory operation chosen
retrieved memory
final answer/task score
whether the memory helped
```

Then train a small policy model or preference model to predict memory operations. For OpenQG, the reward can be retrieval accuracy, citation faithfulness, benchmark pass/fail, or “did this memory help propose a better next search?”

## 7. Letta / MemGPT — best OS-style self-editing memory hierarchy

**Why it ranks #7:** MemGPT/Letta introduced a very important idea: treat LLM context like an operating system treats memory. Letta’s docs describe in-context core memory, recall memory for older messages, archival memory for long-term semantic storage, and agents that can self-edit memory through tools. ([Letta Docs][7])

**Theory:** The LLM has a small working memory window. The agent needs tools to page information in and out, edit always-visible memory blocks, search old conversations, and store durable facts externally.

**Build schematic:**

```text
context window
  ├─ system instructions
  ├─ core memory blocks
  └─ recent messages
        ↕
recall memory: searchable conversation history
        ↕
archival memory: long-term semantic store
```

**How to build it:** Give the agent memory tools:

```text
core_memory_append
core_memory_replace
conversation_search
archival_memory_insert
archival_memory_search
memory_compact
```

For OpenQG, core memory should be tiny and policy-like: project goal, current research frontier, active hypotheses, and strict evidence rules. Everything else should be external.

## 8. Generative Agents — best social/reflective episodic memory

**Why it ranks #8:** Generative Agents is still one of the most human-behavior-like designs. It used a memory stream, reflection, and planning to create believable agents that remember past days, form higher-level reflections, and plan future behavior. Its GitHub repo is available, and the paper shows emergent social behavior in a sandbox. ([Hugging Face][8])

**Theory:** Human-like behavior requires not only fact recall but reflective synthesis. Raw memories become higher-level beliefs, which shape future plans.

**Build schematic:**

```text
observation
  ↓
memory stream
  ↓
score memories by:
  relevance + recency + importance
  ↓
periodic reflection:
  "What higher-level insight follows?"
  ↓
daily / task planning
  ↓
action
```

**How to build it:** Store each observation with:

```yaml
text: ...
timestamp: ...
importance: 1-10
embedding: ...
```

When retrieval happens, score:

```text
score = relevance(query, memory) + recency(memory) + importance(memory)
```

Then periodically ask the model to synthesize reflections from the top memories. For scientific agents, reflections become “open questions,” “contradictions,” and “research gaps.”

## 9. HippoRAG — best hippocampus-like associative recall

**Why it ranks #9:** HippoRAG is explicitly inspired by hippocampal indexing theory. It combines LLM extraction, knowledge graphs, and Personalized PageRank to retrieve multi-hop associations. The paper reports up to 20% gains over prior methods on multi-hop QA, and says single-step HippoRAG retrieval can be 10–30x cheaper and 6–13x faster than iterative retrieval approaches. ([Hugging Face][9])

**Theory:** The hippocampus does not store every detail in isolation; it indexes associations so related memories can be reactivated. HippoRAG mimics this by retrieving through a graph rather than only nearest-neighbor chunks.

**Build schematic:**

```text
documents / episodes
  ↓
LLM extracts triples
  ↓
knowledge graph
  ↓
query entities
  ↓
Personalized PageRank over graph
  ↓
top associated nodes + evidence chunks
```

**How to build it:** Extract triples from papers and logs:

```text
(entity_a, relation, entity_b)
```

Build a graph. At query time, identify query entities, seed them, run Personalized PageRank, and retrieve the highest-scoring connected claims and source fragments.

For OpenQG, this is excellent for multi-hop scientific recall: “Which observations constrain theories that modify gravity at infrared scales?”

## 10. Voyager / procedural skill memory — best compounding skill library

**Why it ranks #10:** Voyager is not a general memory database; it is a **lifelong procedural memory** design. It used an automatic curriculum, iterative prompting with execution feedback, and an ever-growing executable skill library in Minecraft. The paper reports much faster exploration and tech-tree progress than prior systems and emphasizes that learned skills are interpretable, compositional, reusable, and reduce catastrophic forgetting. ([Hugging Face][10])

**Theory:** Humans do not only remember facts; we remember how to do things. This is procedural memory. For agents, the best procedural memories are executable, testable, and compositional.

**Build schematic:**

```text
task attempt
  ↓
write code/action plan
  ↓
execute
  ↓
observe error/result
  ↓
self-verify and repair
  ↓
store successful skill
  ↓
retrieve skill for future tasks
```

**How to build it:** Every successful workflow becomes a skill:

```yaml
id: skill.download_hepdata_yaml
description: Download and validate HEPData YAML tables
inputs:
  - record_id
outputs:
  - local_manifest
preconditions:
  - internet_available
code_ref: scripts/hepdata/download.rs
tests:
  - cargo test download_hepdata_yaml
usage_examples: [...]
failure_modes: [...]
```

For OpenQG, this is crucial. The agent should accumulate tested scripts for searching, parsing papers, extracting equations, downloading public data, running benchmarks, validating units, and generating evidence reports.

## Honorable mentions worth stealing from

**MemoryOS** is a practical hierarchical memory-management design inspired by operating systems, with short-term, mid-term, and long-term personal memory plus storage/update/retrieval/generation modules. It is open-sourced and reported large improvements on LoCoMo. Use this if you want a simpler OS-like hierarchy than MIRIX. ([Hugging Face][11])

**MemoryBank** is important because it explicitly uses the **Ebbinghaus forgetting curve** to reinforce or decay memories based on time and importance. It is very relevant if you want memories to strengthen with use and fade when irrelevant. ([AAAI Publications][12])

**EM-LLM** is one of the most human-like episodic-memory designs at the model/context level. It segments long streams into coherent events using Bayesian surprise and graph-theoretic refinement, then retrieves by similarity plus temporal contiguity. It is not a full agent memory system, but its event segmentation should be copied. ([ICLR Proceedings][13])

**GraphRAG / LightRAG** are not “human-like personal memory” systems, but they are extremely useful for corpus-scale knowledge. GraphRAG builds entity/relationship graphs plus community summaries for global questions; LightRAG adds graph indexing, dual-level retrieval, and incremental updates for faster graph-RAG. ([Microsoft GitHub][14])

**Agent KB** is important for cross-agent experience transfer. It stores high-level strategies and detailed execution logs and uses a Reason–Retrieve–Refine loop, improving agent performance across GAIA and SWE-bench-style tasks. ([Hugging Face][15])

**Memp** is a newer procedural-memory design that distills past trajectories into fine-grained instructions and higher-level script abstractions, then updates and deprecates them over time. It is conceptually very aligned with Voyager-style skill memory. ([Hugging Face][16])

## The architecture I would actually build

For a serious science-memory system, I would not pick only one. I would build this:

```text
                    ┌──────────────────────────────┐
                    │        Working Memory         │
                    │ active task, goals, scratch   │
                    └──────────────┬───────────────┘
                                   │
new event / paper / repo / data ───▼───────────────────────────────┐
                    Memory Router / Extractor                      │
                    classify: episode, fact, claim, skill, resource│
                    └──────────────┬───────────────────────────────┘
                                   │
        ┌──────────────────────────┼──────────────────────────┐
        ▼                          ▼                          ▼
  Episodic Store             Semantic Store              Procedural Store
  run logs, events           claims, entities            tested workflows
  timestamps                 equations, relations        scripts, skills
        │                          │                          │
        └──────────────┬───────────┴──────────────┬───────────┘
                       ▼                          ▼
              Temporal Knowledge Graph      Self-organizing Notes
              Graphiti/Zep style            A-MEM/Zettelkasten style
                       │                          │
                       └────────────┬─────────────┘
                                    ▼
                       Hybrid Retrieval Layer
                       BM25 + vector + entity + graph + temporal
                                    ▼
                       Context-Pack Compiler
                       small, cited, task-specific memory pack
                                    ▼
                       Reasoning Agent
                                    ▼
                       Eval / reward / reflection
                                    ▼
                       Memory-R1-style learned memory manager
```

The core memory objects should be:

```text
Episode       = what happened, when, with source/provenance
Claim         = what is asserted, supported by what, contradicted by what
Concept       = stable abstraction linking claims, equations, datasets, code
Procedure     = reusable tested workflow
Resource      = paper, dataset, repo, API, benchmark, script
Question      = unresolved search/research target
Reflection    = distilled lesson from many episodes
```

The write path should be:

```text
observe → extract → type → store → link → evolve → validate → commit
```

The read path should be:

```text
goal → retrieve lexical/vector/graph/temporal → rerank → compile context pack → reason
```

The compounding path should be:

```text
task result → reflection → update concept graph → update skills → update open questions → eval
```

My strongest recommendation: **do not build “a memory database.” Build a cognitive memory compiler.** The database stores memories; the compiler decides what an agent should actually see, when, and why.
