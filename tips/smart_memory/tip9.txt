## Bottom line

The **most human-like buildable AI-agent memory system today is MIRIX**, with **Nemori** as the most cognitively interesting research design. MIRIX wins because it explicitly splits memory into **core, episodic, semantic, procedural, resource, and knowledge-vault memory**, uses **specialized memory-manager agents**, supports **BM25 + embedding retrieval**, and is available as a plug-in memory layer for agents. Nemori may be more elegant cognitively because it self-organizes raw experience into episodes and learns from prediction gaps, but MIRIX currently looks more complete and buildable. ([Hugging Face][1])

For a serious compounding-knowledge agent, I would not pick only one. I would build a **hybrid**:

```text
MIRIX-style typed human memory
+ Graphiti/Zep temporal knowledge graph
+ Mem0-style fast production retrieval
+ A-MEM/Zettelkasten self-linking notes
+ Memory-R1/ReasoningBank self-improvement over task experience
+ GraphRAG/LightRAG-style corpus maps for large scientific libraries
```

That gives the agent **human-like memory shape**, **fast recall**, **concept self-organization**, and **experience-driven improvement**.

---

## Ranking rubric

I scored each design on three axes:

```text
Human-like:
  episodic memory, semantic memory, procedural memory, working/core memory,
  consolidation, forgetting, reflection, temporal reasoning, multimodal experience.

Adaptable:
  can reorganize memories, update old beliefs, learn from mistakes,
  link new concepts, evolve schemas, support multi-agent or cross-task reuse.

Fast:
  retrieval latency, token efficiency, single-pass retrieval, indexed search,
  ability to avoid re-reading full context.
```

Scores are approximate from the published design, repo/docs, and reported benchmarks.

# 1. MIRIX — best human-like buildable memory

**Why it ranks #1:** MIRIX is closest to an explicit artificial analogue of human memory. It separates memory into **six specialized stores**: core, episodic, semantic, procedural, resource, and knowledge vault. Its docs describe eight specialized agents: a meta-agent, six memory managers, and a chat agent, all writing into a shared memory base. It also supports BM25 and embedding search, local-first storage, retention/decay policies, and PostgreSQL or SQLite. ([Mirix Documentation][2])

**Theory:** Human memory is not one database. We have autobiographical events, general knowledge, habits/procedures, working identity, and sensitive private information. MIRIX maps this directly:

```text
Input stream
  → Meta Agent routes content
    → Core Memory        identity, stable user facts, preferences
    → Episodic Memory    time-stamped events and activities
    → Semantic Memory    concepts, people, facts, meanings
    → Procedural Memory  workflows, habits, repeated procedures
    → Resource Memory    files, documents, artifacts
    → Knowledge Vault    sensitive structured facts and credentials
  → Unified retrieval
  → Chat / reasoning agent
```

**How to build it:**

```text
tables/
  core_memory(id, subject, value, confidence, updated_at)
  episodic_memory(id, timestamp, event, actors, location, details, importance)
  semantic_memory(id, concept, definition, relations, evidence)
  procedural_memory(id, task, steps, triggers, success_count, failure_count)
  resource_memory(id, artifact_uri, summary, chunks, checksum)
  vault_memory(id, key, encrypted_value, sensitivity, access_policy)

pipeline/
  classify_input()
  route_to_memory_managers()
  extract_memory_cards()
  deduplicate()
  write_with_provenance()
  retrieve_by_keyword_embedding_and_type()
```

**Best use:** Personal agents, research agents, coding agents, assistants that must remember identity, tasks, projects, procedures, documents, and private facts separately.

**Weakness:** The multi-agent write pipeline costs LLM calls. You need strict memory-write gates or it can over-store.

---

# 2. Nemori — most cognitively elegant self-organizing memory

**Why it ranks #2:** Nemori is explicitly inspired by cognitive science. Its key idea is that memory should not be stored in arbitrary chunks. It uses a **Two-Step Alignment Principle** inspired by Event Segmentation Theory to organize conversations into coherent episodes, then a **Predict-Calibrate Principle** inspired by the Free-energy Principle to learn from prediction gaps. ([Hugging Face][3])

**Theory:** Humans segment experience into meaningful events: “the meeting,” “the bug fix,” “the failed experiment.” Nemori tries to make the memory unit itself emerge from the agent’s understanding instead of from fixed token windows.

```text
Raw interaction stream
  → Boundary detection: where does one event end?
  → Narrative generation: what happened in this episode?
  → Episodic memory
  → Semantic abstraction
  → Prediction: what should happen / be true?
  → Calibration: what prediction failed?
  → Knowledge update
```

**How to build it:**

```text
for each interaction window:
  candidate_boundaries = detect_event_boundaries(stream)
  episodes = generate_episode_narratives(candidate_boundaries)

for each episode:
  semantic_delta = extract_new_facts(episode)
  prediction = agent_predict_next_or_expected_state(context)
  error = compare(prediction, observed_state)
  if error is meaningful:
      create_or_update_semantic_memory(error, evidence=episode)
```

**Best use:** Long-running autonomous agents where experience granularity matters: research assistants, personal assistants, workflow agents, multi-month coding agents.

**Weakness:** Less mature operationally than MIRIX, Mem0, or Graphiti. I found a Nemori AI GitHub organization/MVP, but the paper/repo ecosystem appears less production-developed than the top production memory stacks. ([GitHub][4])

---

# 3. MemOS / MemoryOS — memory as an operating system

There are two closely related but distinct “memory OS” directions.

**MemOS** treats memory as a first-class system resource, with a **MemCube** abstraction that can manage plaintext memory, activation memory, and parameter-level memory. Its paper frames memory management as an OS-like problem: scheduling, lifecycle control, memory fusion, migration, provenance, versioning, and governance. ([Hugging Face][5])

**BAI-LAB MemoryOS** is a personalized-agent memory operating system with **short-term, mid-term, and long-term personal memory**, plus storage, updating, retrieval, and generation modules. The repo reports LoCoMo improvements and has an MCP server and PyPI package. ([Hugging Face][6])

**Theory:** This is the agent equivalent of RAM, disk, cache, filesystem, scheduler, and compactor.

```text
Memory API
  → Memory Scheduler
    → short-term / working memory
    → mid-term session memory
    → long-term personal / semantic memory
    → activation memory
    → parameter / skill memory
  → Lifecycle manager:
      promote, compress, merge, delete, decay, freeze, share
```

**How to build it:**

```text
MemoryCube {
  id
  type: plaintext | activation | parameter | skill | profile | episode
  owner: user | agent | team | project
  content
  embedding
  graph_edges
  provenance
  access_policy
  lifecycle_state: fresh | active | consolidated | archived | deprecated
}

scheduler(query, task):
  candidates = retrieve(memory_cubes)
  rank by relevance + recency + reliability + access_policy
  activate top memories into prompt/context/tools
```

**Best use:** A serious platform where multiple agents, users, projects, and skills need shared but governed memory.

**Weakness:** More complex than most teams need at first. It is a design for a memory platform, not just a retrieval plugin.

---

# 4. Zep / Graphiti — best temporal knowledge graph memory

**Why it ranks high:** Graphiti is one of the strongest open-source designs for dynamic agent memory. It builds **temporal context graphs** with entities, relationships, facts, validity windows, and provenance episodes. Unlike static RAG, it preserves history: what was true, when it became true, and when it was superseded. ([GitHub][7])

**Theory:** Humans remember relationships and time: “Alice worked at X before Y,” “this fact was true last month but changed.” Graphiti models that explicitly.

```text
Episode: raw conversation / document / event
  → entity extraction
  → fact extraction
  → temporal edge creation
  → invalidation / supersession
  → hybrid retrieval:
       semantic vector search
     + BM25 keyword search
     + graph traversal
     + reranking
```

**Graph schema:**

```text
(:Episode)-[:PRODUCED]->(:Fact)
(:Entity)-[:RELATION {valid_from, valid_to, confidence}]->(:Entity)
(:Fact)-[:SUPPORTED_BY]->(:Episode)
(:Entity)-[:HAS_SUMMARY]->(:EntitySummary)
```

**How to build it:**

```text
add_episode(text, timestamp):
  entities = extract_entities(text)
  facts = extract_relationships(text, entities)
  for fact in facts:
      old_conflicting_edges = find_conflicts(fact)
      close_validity_window(old_conflicting_edges)
      insert_temporal_edge(fact, valid_from=timestamp)
  update_entity_summaries()
```

**Best use:** Research memory, enterprise memory, scientific claims, changing user/project facts, contradiction handling.

**Weakness:** Graph extraction quality depends on the model. You still need source receipts and verification gates.

---

# 5. Mem0 — fastest production-ready memory layer

**Why it ranks high:** Mem0 is probably the most practical “drop it into an agent” memory layer. Its open-source repo describes a memory layer with user/session/agent state, entity linking, multi-signal retrieval, and production benchmarks. The repo’s April 2026 update reports single-pass retrieval with p50 latency around 0.88–1.09 seconds on benchmark settings and multi-signal retrieval using semantic, BM25, and entity matching. ([GitHub][8])

**Theory:** Extract compact salient facts from interactions. Store them as durable memories. Retrieve only the relevant few at inference time.

```text
conversation
  → extract salient memories
  → store in vector + keyword + entity index
  → link entities
  → search(query)
  → fuse semantic + BM25 + entity scores
  → inject compact memory context
```

**How to build it:**

```text
memory_add(messages):
  facts = llm_extract_salient_facts(messages)
  entities = extract_entities(facts)
  embeddings = embed(facts)
  store(facts, embeddings, entities, metadata)

memory_search(query):
  dense_hits = vector_search(query)
  sparse_hits = bm25_search(query)
  entity_hits = entity_match(query)
  return rank_fusion(dense_hits, sparse_hits, entity_hits)
```

**Best use:** Production assistants, customer support, coding tools, apps where speed and cost matter.

**Weakness:** Less human-like than MIRIX or Nemori unless you add typed episodic/procedural stores and graph validity.

---

# 6. Letta / MemGPT — best self-editing context manager

**Why it ranks high:** MemGPT’s core insight is still fundamental: the agent should not merely retrieve memory; it should **manage its own memory** through tool calls. Letta’s current docs describe core memory, recall memory, archival memory, memory block editing, conversation search, archival insertion/search, and automatic compaction when context fills. ([Letta Docs][9])

**Theory:** Treat the LLM like a process in an operating system. The context window is RAM. Archival memory is disk. The agent pages information in and out.

```text
context window
  ├── system instructions
  ├── core memory blocks       always visible
  ├── recent messages          working context
  └── retrieved memories       paged in as needed

external memory
  ├── recall memory            old conversation history
  └── archival memory          long-term semantic facts
```

**How to build it:**

```text
tools:
  memory_insert(block, text)
  memory_replace(block, old, new)
  memory_rethink(block)
  conversation_search(query)
  archival_memory_insert(text, tags)
  archival_memory_search(query)

agent_loop:
  inspect task
  decide whether memory is missing
  call search or edit tools
  answer with retrieved context
  update memory if new stable fact appears
```

**Best use:** Stateful agents that need agency over their own context: coding agents, personal assistants, planning agents.

**Weakness:** Self-editing memory can corrupt itself unless every write is audited, reversible, and provenance-backed.

---

# 7. A-MEM — best self-organizing Zettelkasten memory

**Why it ranks high:** A-MEM uses the **Zettelkasten** principle: small atomic notes connected into a knowledge network. New memories create structured notes with descriptions, keywords, and tags; the system links them to related memories and can update older memories as new ones arrive. ([Hugging Face][10])

**Theory:** Human-like knowledge compounding requires links, not just chunks. A-MEM’s core value is dynamic linking and memory evolution.

```text
new memory
  → structured note
      description
      keywords
      tags
      context
  → retrieve related notes
  → create typed links
  → update old notes if new context changes them
  → retrieve by semantic + graph neighborhood
```

**How to build it:**

```text
Note {
  id
  content
  summary
  keywords[]
  tags[]
  links: [{target_id, relation, reason, confidence}]
  last_revised
}

add_memory(text):
  note = generate_note(text)
  neighbors = retrieve_similar_notes(note)
  links = infer_links(note, neighbors)
  evolved_neighbors = update_neighbor_metadata(note, neighbors)
  save(note, links, evolved_neighbors)
```

**Best use:** Research agents, literature review agents, coding knowledge bases, personal PKM systems.

**Weakness:** Link creation costs LLM calls. Without pruning, the graph can become noisy.

---

# 8. Memory-R1 / ReasoningBank — best memory that learns how to remember

**Why it ranks high:** Most systems use hand-written rules: ADD, UPDATE, DELETE, retrieve top-k. Memory-R1 trains a **Memory Manager** and an **Answer Agent** with reinforcement learning so the system learns when to add, update, delete, ignore, and retrieve memories. ReasoningBank is complementary: it distills generalizable reasoning strategies from successes and failures so agents improve over streams of tasks. ([Hugging Face][11])

**Theory:** Humans do not just store facts; we learn better strategies from experience. These systems turn past failures and successes into procedural/reasoning memory.

```text
task attempt
  → trajectory
  → success/failure judgment
  → extract lesson / strategy
  → store reasoning memory
  → retrieve relevant strategy on next task
  → update memory policy by RL
```

**How to build it:**

```text
MemoryManagerPolicy(state):
  action ∈ {ADD, UPDATE, DELETE, NOOP}
  reward = downstream answer quality - memory cost - contradiction penalty

ReasoningMemory {
  situation
  mistake_or_success
  generalized_rule
  evidence_trace
  applicability_conditions
}

after_task():
  judge outcome
  distill lesson
  store lesson
  train/reward memory policy
```

**Best use:** Agents that should become better at web browsing, coding, debugging, research search, tool use, and long-horizon workflows.

**Weakness:** This is more of a training/evaluation framework than a simple memory database. It needs benchmarks and reward design.

# 9. HippoRAG — best hippocampal multi-hop memory

**Why it ranks high:** HippoRAG is explicitly inspired by hippocampal indexing theory. It combines LLM extraction, knowledge graphs, and Personalized PageRank to retrieve multi-hop evidence efficiently. The paper reports that single-step HippoRAG retrieval can be 10–30× cheaper and 6–13× faster than iterative retrieval methods like IRCoT while remaining strong on multi-hop QA. ([Hugging Face][12])

**Theory:** The hippocampus helps index distributed memories so related memories can be reactivated. HippoRAG approximates this with graph paths and PageRank.

```text
corpus
  → OpenIE triples
  → knowledge graph
  → query entity detection
  → Personalized PageRank
  → retrieve high-activation facts/passages
  → answer with evidence
```

**How to build it:**

```text
index_corpus(docs):
  triples = openie_extract(docs)
  graph.add(triples)
  passage_embeddings = embed(docs)

query(q):
  seeds = detect_query_entities(q)
  activated_nodes = personalized_pagerank(graph, seeds)
  passages = map_nodes_to_passages(activated_nodes)
  return rerank(passages, q)
```

**Best use:** Scientific corpora, literature graphs, legal/medical multi-hop retrieval, question answering over connected facts.

**Weakness:** It is not a full personal memory system. It is a powerful long-term relational retrieval substrate.

# 10. Generative Agents + MemoryBank — most important foundational human-like memory

**Why it still matters:** Generative Agents introduced a compelling memory architecture for believable human behavior: **memory stream, reflection, and planning**. Agents remember experiences, synthesize higher-level reflections, and use them to plan actions. MemoryBank added a more explicit long-term companion memory with **Ebbinghaus forgetting-curve-inspired reinforcement and decay**, personality adaptation, and continuous updates. ([Hugging Face][13])

**Theory:** Human-like behavior comes from more than fact recall. It comes from memories being reflected on and converted into plans.

```text
observation
  → memory stream
  → score by recency, importance, relevance
  → retrieve memories
  → reflect into higher-level insights
  → plan daily/task behavior
  → act
```

**How to build it:**

```text
Memory {
  text
  timestamp
  importance
  embedding
}

retrieve(query):
  score = α * relevance(query, memory)
        + β * recency(memory)
        + γ * importance(memory)

periodic_reflection():
  top_memories = retrieve_important_recent_memories()
  insights = llm_summarize_patterns(top_memories)
  store(insights, type="reflection")

plan():
  context = retrieve(current_situation)
  reflections = retrieve("what matters now?")
  generate_plan(context, reflections)
```

**Best use:** Simulated humans, NPCs, companion agents, social agents, human-like planning.

**Weakness:** It is less scalable and less production-grade than newer systems. It does not solve temporal graph consistency, data provenance, or high-speed enterprise retrieval by itself.

---

## Where GraphRAG / LightRAG fit

I would not call GraphRAG or LightRAG the most human-like memory systems, but they are essential for **large-scale knowledge compounding**. Microsoft GraphRAG builds a knowledge graph from documents, detects communities, generates community reports, and supports global/local/DRIFT search. LightRAG combines graph indexing with vector retrieval and incremental updates for faster graph-enhanced RAG. ([GitHub][14])

For a science project like OpenQG, use them as the **collective semantic memory layer**:

```text
papers / books / docs
  → entity + claim extraction
  → citation / concept graph
  → community summaries
  → global maps
  → local evidence retrieval
  → research questions / gaps
```

They should sit beside, not replace, the agent’s human-like memory.

## My final build recommendation

The strongest architecture is not one paper. It is this:

```text
1. MIRIX typed memory
   core / episodic / semantic / procedural / resource / vault

2. Graphiti temporal graph
   entities / claims / relationships / validity windows / provenance

3. Mem0 fast retrieval
   vector + BM25 + entity linking + reranking

4. A-MEM note evolution
   self-linking atomic memories and concept cards

5. Memory-R1 / ReasoningBank learning loop
   learn what to store, retrieve, delete, and generalize from failures

6. GraphRAG / LightRAG corpus memory
   global maps of papers, concepts, theories, datasets, and code
```

The resulting system would look like:

```text
Experience stream
  → Event segmentation
  → Typed memory write
  → Source receipt / provenance
  → Temporal graph update
  → Fast hybrid indexes
  → Periodic consolidation
  → Reflection / reasoning lessons
  → Context compiler
  → Agent action
  → Evaluation feedback
  → Memory policy improvement
```

For your goal—agents that get smarter over months without blowing the context window—the winning principle is:

**Do not store “chat history.” Store typed, linked, temporal, source-backed memories, then train or prompt the agent to manage them like a scientist manages a lab notebook, citation graph, and skill library.**
