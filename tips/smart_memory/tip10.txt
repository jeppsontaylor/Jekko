My best single answer: **MIRIX is currently the most human-like buildable open-source memory system I found**, because it explicitly separates **core, episodic, semantic, procedural, resource, and knowledge-vault memory**, routes updates through specialized memory agents, supports multimodal inputs, and retrieves with BM25 plus embeddings. **Nemori may be the most cognitively elegant research design**, because it self-organizes memory into episodes and learns from prediction gaps, but I did not find an official open-source implementation. ([GitHub][1])

For an OpenQG-style scientific research agent, I would not copy one system exactly. I would build a **hybrid**:

```text
MIRIX memory taxonomy
+ Graphiti/Zep temporal knowledge graph
+ A-MEM self-organizing Zettelkasten links
+ Letta/MemGPT self-editing working memory
+ Voyager/Memento procedural skill + case memory
+ Mem0/HippoRAG fast retrieval
+ Reflexion-style failure lessons
```

That gives you the closest thing to a human-like AI memory: working memory, episodic memory, semantic memory, procedural skill memory, reflection, consolidation, temporal truth tracking, and fast recall.

# 1. MIRIX — most human-like buildable memory system

**Why it ranks #1:** MIRIX is the closest to a full human-like memory stack. It has **core memory** for stable identity and preferences, **episodic memory** for time-based events, **semantic memory** for facts and concepts, **procedural memory** for workflows, **resource memory** for documents/files, and a **knowledge vault** for structured sensitive data. It uses a multi-agent architecture where a meta-agent routes inputs to specialized memory managers, and a chat agent searches across memory types. ([Mirix Documentation][3])

**Build schematic:**

```text
Input stream
  ├─ text
  ├─ screenshots / images
  ├─ voice
  └─ files
        ↓
Meta Agent: classify what kind of memory update is needed
        ↓
Memory managers in parallel
  ├─ Core manager        → identity, preferences, standing facts
  ├─ Episodic manager    → timestamped events and activity summaries
  ├─ Semantic manager    → concepts, facts, relationships
  ├─ Procedural manager  → workflows and repeated processes
  ├─ Resource manager    → documents, files, source material
  └─ Knowledge vault     → sensitive structured data
        ↓
Shared memory DB
  ├─ PostgreSQL / SQLite
  ├─ BM25 text search
  ├─ vector search
  └─ field-specific retrieval
        ↓
Query-time synthesis
```

**What to copy:** Use MIRIX’s **six-memory taxonomy** as the backbone for OpenQG. For physics research, the equivalent would be:

```text
Core        → mission, constraints, coding preferences, active research goals
Episodic    → research runs, searches, failed attempts, discoveries
Semantic    → theories, claims, equations, observables, datasets
Procedural  → reusable research/coding/math/data skills
Resource    → papers, repos, docs, dataset manifests
Vault       → secrets, API keys, credentials, private notes
```

**Open source / paper:** MIRIX has an Apache-2.0 GitHub repo and a 2025 paper; its docs describe the post-0.1.6 direction as a pluggable memory system rather than only a desktop assistant. ([GitHub][1])

# 2. Nemori — best cognitive self-organizing design

**Why it ranks #2:** Nemori is probably the most cognitively interesting design. It is built around two principles: **Two-Step Alignment**, inspired by event segmentation, and **Predict-Calibrate**, inspired by predictive learning / free-energy-style ideas. Instead of storing arbitrary chunks, it segments experience into coherent episodes and updates knowledge when the agent’s prediction fails. ([Hugging Face][4])

**Build schematic:**

```text
Conversation / experience stream
        ↓
Boundary detector
  Detect where one event/episode ends and another begins
        ↓
Episode narrative generator
  Convert raw stream into coherent episode memory
        ↓
Prediction module
  Ask: "Given my memory, what should I expect next?"
        ↓
Calibration module
  Compare prediction vs reality
        ↓
Knowledge evolution
  ├─ strengthen confirmed patterns
  ├─ revise wrong beliefs
  ├─ create new semantic memories
  └─ attach lessons to relevant episodes
```

**What to copy:** Nemori’s key insight is that an agent should not only ask, “What should I remember?” It should ask, **“What did I predict incorrectly, and what concept must change because of that?”** For OpenQG, this is extremely valuable: every failed derivation, bad benchmark, contradicted claim, or dead-end literature path should become a calibration event.

**Open source / paper:** I found a paper record and summaries, but I did **not** find a reliable official GitHub repo for Nemori during this search. ([Hugging Face][4])

---

# 3. Zep / Graphiti — best temporal knowledge graph memory

**Why it ranks #3:** Graphiti is the strongest temporal graph memory design I found. It stores **entities**, **relationships/facts**, and **episodes** with provenance and temporal validity, so the agent can know not only *what* is true, but *when* it was true and whether it was superseded. Graphiti supports hybrid retrieval using semantic search, keyword search, and graph traversal. ([GitHub][5])

**Build schematic:**

```text
Raw episode
  "On May 5, user changed preferred theory baseline to X"
        ↓
Episode node
  id, text, timestamp, source, hash
        ↓
Entity extraction
  User, theory baseline, project, dataset, equation, paper
        ↓
Fact edges
  User --prefers_baseline--> Theory X
  valid_from = May 5
  valid_to   = null
  provenance = episode_id
        ↓
If contradicted later:
  old edge.valid_to = new_time
  new edge.valid_from = new_time
```

**Query flow:**

```text
Question
  ↓
hybrid search
  ├─ vector similarity
  ├─ BM25 keyword match
  ├─ graph neighborhood expansion
  └─ temporal filter: current / historical / as-of-date
  ↓
rerank
  ↓
context packet
```

**What to copy:** For OpenQG, Graphiti-style temporal graphs are essential. Scientific memory changes over time: a dataset is updated, a result is refuted, a theory’s parameterization changes, a paper is superseded. A flat vector store cannot represent that cleanly.

**Open source / paper:** Graphiti is open source on GitHub, and Zep’s paper reports a temporal knowledge graph architecture for agent memory, with benchmark claims against MemGPT and LongMemEval. ([GitHub][5])

---

# 4. Letta / MemGPT — best self-editing memory operating system

**Why it ranks #4:** MemGPT/Letta introduced one of the cleanest memory models: treat the LLM context window like RAM, and use tools to page information between **core memory**, **recall memory**, and **archival memory**. The agent can edit its own memory blocks and search older conversation or archival storage. ([Letta Docs][6])

**Build schematic:**

```text
Prompt context = active RAM
  ├─ system instructions
  ├─ core memory blocks
  │    ├─ persona
  │    ├─ human/user profile
  │    └─ active project state
  └─ recent messages
        ↓
Memory tools
  ├─ memory_insert / memory_replace / memory_rethink
  ├─ conversation_search
  ├─ archival_memory_insert
  └─ archival_memory_search
        ↓
External stores
  ├─ recall DB: old conversation history
  └─ archival DB: long-term semantic memories
```

**What to copy:** Use Letta/MemGPT as the **agent runtime model**: the agent should have a tiny always-loaded kernel plus tools to retrieve, write, and revise durable memory. It should not blindly stuff everything into the context window.

**Open source / paper:** Letta is the production evolution of MemGPT, and its docs explicitly describe self-editing memory, memory hierarchy, core/recall/archival memory, and context window management. ([GitHub][7])

---

# 5. A-MEM — best self-organizing concept memory

**Why it ranks #5:** A-MEM is important because it treats memory less like a database and more like a **Zettelkasten**: each new memory becomes a structured note with context, tags, keywords, and links to related memories. New memories can also update older memory representations, allowing the memory network to evolve. ([Hugging Face][8])

**Build schematic:**

```text
New memory event
        ↓
Generate note
  ├─ summary
  ├─ context
  ├─ tags
  ├─ keywords
  ├─ timestamp
  └─ embedding
        ↓
Find related old notes
  ├─ vector search
  ├─ keyword search
  └─ tag overlap
        ↓
LLM link judge
  "Are these meaningfully related?"
        ↓
Update graph
  ├─ create bidirectional links
  ├─ strengthen useful links
  ├─ update old note context
  └─ evolve tags / abstractions
```

**What to copy:** A-MEM is almost exactly what you want for research compounding. Papers, equations, datasets, and failed hypotheses should become linked notes. The trick is **memory evolution**: old notes should get better as new evidence arrives.

**Open source / paper:** A-MEM has an MIT-licensed GitHub repo and a 2025 paper describing dynamic memory organization, Zettelkasten-inspired linking, intelligent indexing, and continuous memory refinement. ([GitHub][9])

---

# 6. Memento — best experience-replay memory for agents

**Why it ranks #6:** Memento is a modern “learn from experience without fine-tuning the LLM” design. It uses a **case bank** of successful and failed trajectories, a planner-executor architecture, and case-based reasoning. It frames continual learning as memory-based online reinforcement learning rather than weight updates. ([GitHub][10])

**Build schematic:**

```text
Task attempt
  ├─ goal
  ├─ plan
  ├─ retrieved cases
  ├─ tool calls
  ├─ observations
  ├─ result
  └─ reward / evaluation
        ↓
Case Bank write
  Store success and failure trajectories
        ↓
Next task
  Planner retrieves similar cases
        ↓
Executor acts using:
  ├─ current task
  ├─ relevant cases
  ├─ tools
  └─ constraints
        ↓
Outcome gets written back
```

**What to copy:** For OpenQG, every research run should create a case:

```text
Case = question + search strategy + sources found + reasoning path + code run + outcome + lesson
```

That lets the agent improve not only factual recall, but **strategy**.

**Open source / paper:** Memento’s GitHub describes a memory-based continual-learning framework with planner-executor architecture, case-based reasoning, MCP tooling, and memory-augmented learning. ([GitHub][10])

---

# 7. Voyager — best procedural skill memory

**Why it ranks #7:** Voyager is the classic example of experience compounding through **procedural memory**. It learns executable code skills in Minecraft, stores them in a skill library, retrieves similar skills by embedding, and composes them into more complex behaviors. It also uses an automatic curriculum and environment feedback. ([GitHub][11])

**Build schematic:**

```text
State + goal
        ↓
Retrieve top-k similar skills
        ↓
Generate code / action plan
        ↓
Execute in environment
        ↓
Observe errors and feedback
        ↓
Self-verification
  ├─ success → commit skill
  └─ failure → revise and retry
        ↓
Skill library grows
```

**What to copy:** For OpenQG, this becomes:

```text
Research skill library
  ├─ fetch_arxiv_paper()
  ├─ resolve_doi_openalex_semanticscholar()
  ├─ extract_equations_from_tex()
  ├─ build_dataset_manifest()
  ├─ run_smoke_benchmark()
  ├─ compare_theory_to_observable()
  └─ write_claim_card()
```

Each skill should have a description, inputs, outputs, code, tests, examples, failure modes, and provenance.

**Open source / paper:** Voyager’s GitHub and project page describe the automatic curriculum, ever-growing skill library, iterative prompting with feedback, and major benchmark gains in Minecraft exploration. ([GitHub][11])

# 8. MemoryOS — best hierarchical personal memory manager

**Why it ranks #8:** MemoryOS is a memory operating system for personalized agents. It uses short-term, mid-term, and long-term memory, with modules for storage, updating, retrieval, and generation. It is less conceptually radical than Nemori or Graphiti, but it is practical and clean. ([Hugging Face][12])

**Build schematic:**

```text
Conversation pair
        ↓
Short-term memory
  recent interactions, FIFO-like buffer
        ↓
Mid-term memory
  session summaries, dialogue chains, hot topics
        ↓
Long-term memory
  user profile, stable facts, knowledge
        ↓
Retriever
  query all layers in parallel
        ↓
Generator
  produce response using retrieved memory
```

**What to copy:** Use MemoryOS-style **promotion rules**. Not every observation deserves long-term storage. Memory should move upward only when it is repeated, important, surprising, or connected to active goals.

**Open source / paper:** MemoryOS is open source on GitHub, has documentation, and reports LoCoMo improvements in its paper and README. ([GitHub][13])

---

# 9. Mem0 — fastest production-style memory layer

**Why it ranks #9:** Mem0 is not the most human-like, but it is one of the most practical and fast. It extracts salient memories from conversations, stores them for retrieval, and supports graph memory where entities and relationships are persisted alongside embeddings. Its newer algorithm emphasizes single-pass ADD-only extraction, entity linking, and multi-signal retrieval with semantic, BM25, and entity matching. ([GitHub][14])

**Build schematic:**

```text
Conversation / agent event
        ↓
Memory extractor
  extract facts, preferences, entities, timestamps
        ↓
Memory store
  ├─ vector index
  ├─ BM25 / keyword index
  └─ graph edges for entities and relations
        ↓
Query
  ├─ semantic retrieval
  ├─ keyword retrieval
  ├─ entity match
  └─ graph expansion
        ↓
fused ranked memories
```

**What to copy:** Use Mem0-like retrieval for **fast user/project memory**, especially where you need low-latency recall across many sessions. For a scientific system, I would not use it as the whole architecture, but I would copy its multi-signal retrieval layer.

**Open source / paper:** Mem0’s paper reports improvements on LOCOMO and latency/token savings against full-context approaches; the GitHub repo is Apache-2.0 and describes itself as a universal memory layer for AI agents. ([Hugging Face][15])

# 10. HippoRAG — best neuro-inspired fast multi-hop recall

**Why it ranks #10:** HippoRAG is not a complete agent memory system, but it is one of the best recall mechanisms. It is inspired by hippocampal indexing theory and combines LLM extraction, knowledge graphs, and Personalized PageRank. It is especially useful when the answer requires traversing chains of entities across documents. ([Hugging Face][16])

**Build schematic:**

```text
Documents / memories
        ↓
LLM OpenIE extraction
  subject — relation — object triples
        ↓
Knowledge graph
  nodes = entities/concepts
  edges = relations
        ↓
Query
  extract query entities
        ↓
Personalized PageRank
  spread activation through graph
        ↓
Rank passages / memories
  by graph relevance + node specificity
```

**What to copy:** HippoRAG is excellent for scientific literature because many important answers are not keyword matches. You often need to connect “paper A uses method B,” “method B constrains parameter C,” and “dataset D measures observable C.” That is a graph path, not a flat embedding search.

**Open source / paper:** HippoRAG’s paper reports up to 20% improvement over state-of-the-art RAG methods and 10–30× lower cost plus 6–13× faster retrieval than iterative retrieval approaches; the code/data are linked from the paper record. ([Hugging Face][16])

## Important near-misses you should still steal from

**Generative Agents** is the classic human-behavior memory architecture: memory stream, retrieval by recency/importance/relevance, reflection, and planning. It is less modern as infrastructure, but its memory-stream and reflection design remains foundational. ([GitHub][17])

**MemoryBank** is valuable because it adds an Ebbinghaus forgetting-curve-inspired update mechanism, letting memories decay or be reinforced by importance and recurrence. I would copy that as a decay/reinforcement layer on top of MIRIX or Graphiti. ([AAAI Publications][18])

**AriGraph** is strong for agents that must build a world model while acting in an environment. It combines semantic and episodic memory in a graph and outperforms full-history, summarization, and RAG baselines in TextWorld-style tasks. ([IJCAI][19])

**Memory-R1** is important if you want the memory manager itself to be trained. It uses a Memory Manager with ADD, UPDATE, DELETE, and NOOP operations plus an Answer Agent, trained with outcome-driven RL. ([Cool Papers][20])

**GraphRAG, LightRAG, and RAPTOR** are not full agent memories, but they are excellent corpus-memory patterns. GraphRAG builds entity graphs and community summaries for global corpus questions; LightRAG combines graph indexing with vector retrieval and incremental updates; RAPTOR recursively clusters and summarizes text into a retrieval tree. ([GraphRAG][21])

## What I would actually build

For your research-agent system, I would build this:

```text
1. Working memory
   - Letta-style core blocks
   - tiny always-loaded mission, constraints, active questions

2. Episodic memory
   - MIRIX-style timestamped research episodes
   - every search, failed run, benchmark, insight, and correction

3. Semantic memory
   - Graphiti temporal graph
   - papers, theories, claims, equations, observables, datasets, code repos

4. Procedural memory
   - Voyager/Memento-style skills and cases
   - executable research workflows and lessons from past runs

5. Concept organization
   - A-MEM-style Zettelkasten linking
   - notes evolve when new evidence arrives

6. Recall engine
   - Mem0-style multi-signal retrieval
   - HippoRAG graph activation for multi-hop scientific questions
   - BM25 for exact symbols, equations, paper IDs, dataset names

7. Consolidation
   - Nemori-style episode segmentation
   - prediction-gap learning
   - MemoryBank-style decay/reinforcement

8. Governance
   - Git as canonical curated memory
   - graph/vector/SQL as rebuildable projections
   - every promoted memory has provenance and review state
```

The fastest path to a powerful implementation is:

```text
MIRIX schema
+ Graphiti temporal graph
+ Letta-style core memory
+ A-MEM linking
+ Voyager/Memento skill/case bank
+ Mem0/HippoRAG retrieval
```

That combination is more human-like than any single system: it remembers events, abstracts concepts, learns procedures, updates beliefs, tracks time, retrieves quickly, and improves from experience without stuffing everything into the context window.
