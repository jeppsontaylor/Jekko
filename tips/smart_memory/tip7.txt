## My pick: MIRIX is the closest single system to “human-like agent memory”

Forced to choose one, I would pick **MIRIX** as the most human-like memory architecture designed for AI agents so far. It is not just a vector database. It explicitly separates memory into **Core, Episodic, Semantic, Procedural, Resource, and Knowledge Vault** modules, supports multimodal personal context such as screen activity, and reports strong storage compression and benchmark gains. That makes it closest to a human-style memory stack: identity, episodes, facts, skills, resources, and private sensitive knowledge. ([arXiv][1])

For a **scientific research agent**, though, I would not copy only one system. I would combine **MIRIX’s memory taxonomy**, **Graphiti/Zep’s temporal knowledge graph**, **EverMemOS or A-MEM’s self-organization**, **Mem0’s fast production retrieval**, and **Voyager’s executable skill memory**.

## Ranking method

I’m ranking these by a composite of:

| Dimension      | Meaning                                                                                 |
| -------------- | --------------------------------------------------------------------------------------- |
| **Human-like** | Has episodic, semantic, procedural, temporal, reflective, and identity memory.          |
| **Adaptable**  | Learns from experience, reorganizes memory, consolidates, updates, or rewrites beliefs. |
| **Fast**       | Can retrieve compact relevant memory quickly without flooding context.                  |

The scores are judgment calls, not universal truth. They are based on the papers, open-source implementations, benchmark claims, and architectural maturity.

## 1. MIRIX — best overall human-like memory

**Why it ranks first:** MIRIX is the closest to a complete human-inspired memory architecture. It separates stable personal facts, events, concepts, procedures, resources, and private sensitive knowledge. Its paper describes a modular multi-agent memory system with six memory types: **Core, Episodic, Semantic, Procedural, Resource Memory, and Knowledge Vault**. The GitHub implementation describes it as a multi-agent personal assistant with multimodal screen tracking, local storage, BM25 plus vector retrieval, and privacy-first design. ([arXiv][1])

**Core theory:**
Human memory is not one thing. We have relatively stable identity memory, episodic memories, semantic concepts, procedural skills, working context, and private/personal knowledge. MIRIX maps this directly into separate memory agents.

**Build schematic:**

```text
input event
  ├─ text / chat
  ├─ screen frame
  ├─ document
  ├─ action
  └─ observation

        ↓

memory router / classifier
  ├─ Core Memory          stable identity, preferences, mission
  ├─ Episodic Memory      timestamped experiences
  ├─ Semantic Memory      facts, concepts, definitions
  ├─ Procedural Memory    skills, workflows, code patterns
  ├─ Resource Memory      files, papers, datasets, tools
  └─ Knowledge Vault      sensitive/private knowledge

        ↓

hybrid retrieval
  ├─ BM25 exact search
  ├─ vector semantic search
  ├─ memory-type filtering
  ├─ temporal filtering
  └─ reranking

        ↓

context pack for the agent
```

**How to implement:**

Use PostgreSQL with `pgvector` and full-text search, or Rust with Tantivy plus a vector index. Create one table or collection per memory type. Every memory item should have:

```yaml
id:
memory_type: core | episodic | semantic | procedural | resource | vault
content:
summary:
entities:
created_at:
valid_from:
valid_to:
confidence:
source:
embedding:
access_policy:
```

**Best for OpenQG-like work:**
This is the strongest foundation for a scientific memory bank because it naturally separates papers, concepts, equations, data resources, procedures, and private project strategy.

## 2. EverMemOS — best self-organizing cognitive memory

**Why it ranks second:** EverMemOS is explicitly designed as a **self-organizing memory operating system**. It uses an engram-inspired lifecycle with **MemCells**, **MemScenes**, and **Reconstructive Recollection**, and reports state-of-the-art results on long-memory benchmarks. ([arXiv][2])

**Core theory:**
Human memory is reconstructive. We do not replay everything verbatim. We store traces, consolidate related experiences into higher-level scenes, and reconstruct useful memories at query time.

**Build schematic:**

```text
raw experience stream
        ↓
episodic trace extraction
        ↓
MemCells
  small memory atoms:
  - who
  - what
  - when
  - where
  - why
  - source

        ↓
self-organization
  cluster related MemCells
  infer stable themes
  form MemScenes

        ↓
query-time recollection
  retrieve relevant MemScenes
  reconstruct supporting MemCells
  generate compact answer context
```

**Implementation idea:**

```yaml
MemCell:
  id:
  event_text:
  time:
  participants:
  entities:
  emotional_or_priority_weight:
  source:
  embedding:

MemScene:
  id:
  title:
  theme:
  member_cells:
  summary:
  stable_facts:
  open_questions:
  last_updated:
```

Use clustering plus LLM summarization to form scenes. On each new memory, ask:

```text
Does this belong to an existing scene?
Does it revise a scene?
Does it create a new scene?
Does it contradict an older memory?
```

**Best for OpenQG-like work:**
Excellent for turning many paper-reading episodes into conceptual “research scenes,” such as “CMB lensing evidence,” “neutrino-mass constraints,” or “quantum gravity phenomenology.”

## 3. Zep / Graphiti — best temporal knowledge-graph memory and fastest serious recall

**Why it ranks third:** Zep’s Graphiti architecture builds a **temporal knowledge graph** that supports dynamic updates, provenance, hybrid search, and temporally aware facts. The Zep paper reports improved accuracy and much lower latency on long-memory evaluations, while Graphiti’s open-source implementation emphasizes temporal fact validity, provenance, incremental graph construction, custom ontologies, and hybrid retrieval. ([arXiv][3])

**Core theory:**
A lot of memory is not just “what is true?” but **when was it true, according to whom, and what changed?** This is especially important for agents, science, projects, personal assistants, and long-running systems.

**Build schematic:**

```text
episode
  "User said X", "paper claims Y", "dataset version changed"

        ↓

entity extraction
  Paper, Author, Dataset, Theory, Observable, Tool, Person

        ↓

temporal fact extraction
  subject ─ predicate ─ object
  valid_from:
  valid_to:
  source:
  confidence:

        ↓

temporal knowledge graph
  nodes: entities
  edges: facts with time windows and provenance

        ↓

hybrid retrieval
  ├─ vector search
  ├─ keyword search
  ├─ graph traversal
  ├─ temporal filter
  └─ rerank
```

**Implementation idea:**

```yaml
Node:
  id:
  type: paper | theory | dataset | person | concept | codebase
  name:
  aliases:

FactEdge:
  subject:
  predicate:
  object:
  valid_from:
  valid_to:
  observed_at:
  source:
  confidence:
  embedding:
```

Example:

```yaml
subject: dataset:planck-2018
predicate: constrains
object: parameter:sum-neutrino-masses
valid_from: 2018-07-17
source: paper:...
confidence: high
```

**Best for OpenQG-like work:**
This is probably the most important component for scientific memory because scientific knowledge is relational, sourced, temporal, and often contradictory.

## 4. Letta / MemGPT — best operating-system-style memory manager

**Why it ranks fourth:** MemGPT introduced the idea of **virtual context management** inspired by operating systems: the LLM has limited context, so it explicitly pages information between memory tiers. Letta, the open-source successor/framework from the MemGPT creators, supports advanced memory, skills, subagents, continual learning, and model-agnostic agents. ([arXiv][4])

**Core theory:**
Treat the LLM context window like RAM. Treat long-term memory like disk. The agent should decide what belongs in active context, what should be archived, and what should be edited.

**Build schematic:**

```text
LLM active context
  ├─ system instructions
  ├─ core memory blocks
  ├─ recent conversation
  └─ retrieved memory

        ↕ tools

memory manager
  ├─ edit core memory
  ├─ search archival memory
  ├─ insert archival memory
  ├─ summarize conversation
  └─ page memories into context

        ↕

archival store
  vector DB / text DB / graph DB
```

**Implementation idea:**

Create memory tools the agent can call:

```python
core_memory_replace(block_name, old_text, new_text)
archival_memory_insert(text, metadata)
archival_memory_search(query, filters)
conversation_search(query)
memory_reflect()
```

The key is that memory is not passively retrieved. The agent actively manages it.

**Best for OpenQG-like work:**
Useful as the “memory operating system” around your research agent. It tells the agent when to pull papers, when to summarize, when to write durable notes, and when to page old information back in.

## 5. A-MEM — best Zettelkasten-style self-organizing concept memory

**Why it ranks fifth:** A-MEM is built around dynamic memory organization inspired by **Zettelkasten**. New memories become structured notes with context, keywords, tags, links, and evolving relationships. The open-source implementation uses ChromaDB and an agentic memory manager that links new memories to old ones and refines the network over time. ([arXiv][5])

**Core theory:**
Human knowledge compounds when ideas are linked, not merely stored. A-MEM’s strength is that each new memory is inserted into a growing web of concepts.

**Build schematic:**

```text
new memory
        ↓
structured note generation
  ├─ content
  ├─ context
  ├─ keywords
  ├─ tags
  └─ candidate links

        ↓
link discovery
  find related notes
  add bidirectional links
  update affected notes

        ↓
memory evolution
  refine tags
  merge duplicates
  strengthen concepts
  surface contradictions
```

**Implementation idea:**

```yaml
Note:
  id:
  content:
  context:
  keywords:
  tags:
  links:
    - target_note:
      relation:
      strength:
  created_at:
  updated_at:
  embedding:
```

When inserting a note:

```text
1. Generate structured metadata.
2. Search similar notes.
3. Ask model to explain relationships.
4. Add links.
5. Update old notes if the new note changes them.
6. Periodically cluster notes into topic maps.
```

**Best for OpenQG-like work:**
Very strong for concept formation: equations, principles, assumptions, experimental methods, recurring unresolved questions.

## 6. Mem0 — best lightweight production memory

**Why it ranks sixth:** Mem0 is one of the most practical production memory systems. Its paper describes dynamic extraction, consolidation, and retrieval, including a graph variant. It reports strong benchmark performance, lower latency, and large token savings. Its recent open-source benchmark notes an ADD-only extraction approach, entity linking, and multi-signal retrieval combining semantic, BM25, and entity methods. ([arXiv][6])

**Core theory:**
Do not store all conversation history. Extract durable memories, connect them to entities, and retrieve only the relevant compact set.

**Build schematic:**

```text
conversation turn
        ↓
memory extractor
  decide:
  - add memory?
  - update memory?
  - ignore?

        ↓
memory store
  compact facts/preferences/events

        ↓
retrieval
  ├─ semantic search
  ├─ BM25 search
  ├─ entity match
  ├─ recency / importance
  └─ rerank
```

**Implementation idea:**

```yaml
Memory:
  id:
  user_or_agent_id:
  text:
  entities:
  categories:
  created_at:
  updated_at:
  source_message_ids:
  confidence:
```

Fast retrieval formula:

```text
score =
  0.40 * semantic_similarity
+ 0.25 * keyword_score
+ 0.20 * entity_overlap
+ 0.10 * recency
+ 0.05 * importance
```

**Best for OpenQG-like work:**
Use Mem0-like memory for fast project facts, user preferences, lab notes, recurring decisions, and lightweight agent personalization.

## 7. NEMORI — best “what deserves memory?” design

**Why it ranks seventh:** NEMORI is centered on adaptive memory distillation. It uses cognitive ideas such as episodic integration and semantic knowledge distillation, with prediction error helping decide what should be remembered. ([arXiv][7])

**Core theory:**
The hard problem is not storing memory. It is deciding **what deserves memory**. Humans do not remember everything equally. Surprise, usefulness, novelty, contradiction, and prediction error all affect what gets consolidated.

**Build schematic:**

```text
experience
        ↓
prediction
  what did the agent expect?

        ↓
prediction error
  what was surprising, useful, wrong, or new?

        ↓
adaptive distillation
  ├─ keep important episode
  ├─ update semantic belief
  ├─ discard noise
  └─ flag contradiction

        ↓
compressed memory
```

**Implementation idea:**

For each event, compute:

```yaml
novelty_score:
surprise_score:
task_utility_score:
contradiction_score:
recurrence_score:
source_quality_score:
```

Then:

```text
if surprise + utility + source_quality is high:
    store as memory
if repeated across episodes:
    distill into semantic claim
if contradicts previous memory:
    create contradiction record
```

**Best for OpenQG-like work:**
This is critical for large-scale research ingestion. Without a “what deserves memory” gate, the system drowns in facts.

## 8. MemoryOS — best simple hierarchical conversational memory

**Why it ranks eighth:** MemoryOS uses an operating-system-inspired hierarchy for short-, mid-, and long-term memory, with storage, updating, retrieval, and generation modules. Its paper reports large gains on long conversational memory benchmarks. ([arXiv][8])

**Core theory:**
Conversation memory should be hierarchical. Recent raw context goes into short-term memory. Repeated or summarized context moves to mid-term memory. Stable personal facts move into long-term memory.

**Build schematic:**

```text
short-term memory
  recent dialogue, FIFO buffer

        ↓ summarization

mid-term memory
  session summaries, topics, unresolved threads

        ↓ consolidation

long-term memory
  stable user profile
  stable preferences
  important facts
  recurring goals

        ↓ retrieval manager

response generation
```

**Implementation idea:**

```yaml
ShortTerm:
  raw_messages:
  max_tokens:

MidTermPage:
  session_id:
  summary:
  topic:
  unresolved_items:
  source_messages:

LongTermProfile:
  stable_fact:
  evidence:
  confidence:
  last_confirmed:
```

**Best for OpenQG-like work:**
Good for agent conversations and personal research assistants, but less expressive than a full graph/claim-ledger system.

## 9. Agent KB — best collective cross-agent experience memory

**Why it ranks ninth:** Agent KB is less “human autobiographical memory” and more “organizational memory for agents.” It converts agent trajectories into reusable knowledge and exposes APIs for heterogeneous agent frameworks. The paper frames it as universal memory infrastructure for sharing experiences across agents and reports improvements on GAIA and SWE-bench style tasks. ([arXiv][9])

**Core theory:**
One human gets smarter through experience. An organization gets smarter when many people share lessons. Agent KB aims at the second: collective intelligence for agents.

**Build schematic:**

```text
agent trajectory
  ├─ task
  ├─ plan
  ├─ actions
  ├─ tool calls
  ├─ errors
  ├─ fixes
  └─ outcome

        ↓

experience distillation
  ├─ reusable workflow
  ├─ diagnostic rule
  ├─ failure pattern
  ├─ code patch pattern
  └─ tool-use lesson

        ↓

agent knowledge base

        ↓

new agent task retrieves:
  similar tasks
  successful plans
  known pitfalls
  reusable fixes
```

**Implementation idea:**

```yaml
Experience:
  id:
  task_type:
  starting_state:
  plan:
  actions:
  errors:
  corrections:
  final_result:
  reusable_lesson:
  applicable_when:
  avoid_when:
```

**Best for OpenQG-like work:**
Excellent for making research agents better at workflows: paper search, data discovery, benchmark setup, code debugging, failed search recovery, and literature triage.

## 10. Voyager skill memory — best procedural memory

**Why it ranks tenth:** Voyager is an autonomous lifelong-learning agent for Minecraft, but its memory design generalizes well. It uses an automatic curriculum, iterative prompting with environment feedback, self-verification, and a library of executable skills that can be reused compositionally. The paper reports strong compounding exploration results. ([arXiv][10])

**Core theory:**
Humans do not only remember facts. We remember **how to do things**. Voyager’s skill library is procedural memory: reusable, executable knowledge.

**Build schematic:**

```text
task attempt
        ↓
generate code/action plan
        ↓
execute in environment
        ↓
observe success/failure
        ↓
debug/refine
        ↓
if successful:
    save skill
        ├─ name
        ├─ description
        ├─ code
        ├─ preconditions
        ├─ examples
        └─ embedding
```

**Implementation idea:**

```yaml
Skill:
  id:
  name:
  description:
  code:
  language:
  preconditions:
  postconditions:
  examples:
  failure_modes:
  embedding:
  tests:
```

For OpenQG, a skill might be:

```text
download_arxiv_paper
extract_equations_from_latex
query_openalex_citation_neighborhood
generate_dataset_manifest
run_cosmology_benchmark_smoke_test
```

**Best for OpenQG-like work:**
Crucial. A scientific agent should not relearn how to search arXiv, parse PDFs, inspect datasets, or run benchmark scripts every time.

# Important near-misses

## Generative Agents

The **Generative Agents** architecture is historically one of the most human-like memory systems: agents store a natural-language memory stream, retrieve relevant memories, reflect into higher-level insights, and plan behavior. It is highly important conceptually, but I rank it below newer systems for speed, production readiness, and structured memory engineering. ([arXiv][11])

## Reflexion

**Reflexion** is not a full memory OS, but it is extremely important for experience compounding. It stores verbal reflections from task feedback in episodic memory and uses them to improve later decisions. ([arXiv][12])

## MemoryBank

**MemoryBank** is notable because it explicitly uses ideas like long-term memory, user personality modeling, and an Ebbinghaus-style forgetting curve. It is useful, but modern systems have surpassed it architecturally. ([AAAI Publications][13])

# The architecture I would actually build

The best possible human-like AI memory system would be a hybrid, not a single paper implementation:

```text
                 ┌─────────────────────┐
                 │   Working Memory    │
                 │ current task/context │
                 └──────────┬──────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────┐
│                 Memory Router                        │
│ classify: core, episodic, semantic, procedural, etc.  │
└───────┬─────────┬──────────┬──────────┬──────────────┘
        │         │          │          │
        ▼         ▼          ▼          ▼
   Core Memory  Episodic  Semantic  Procedural
   identity     events    concepts  skills/code
        │         │          │          │
        └─────────┴────┬─────┴──────────┘
                       ▼
             Temporal Knowledge Graph
       entities, facts, time windows, provenance
                       │
                       ▼
               Self-Organization Layer
       clustering, contradiction detection, reflection,
       consolidation, forgetting, topic-map generation
                       │
                       ▼
                 Retrieval Engine
       BM25 + vector + graph traversal + time filter
                       │
                       ▼
                  Context Pack
       small, cited, task-specific memory for the LLM
```

## The memory types I would use

```text
1. Core memory
   Stable identity, mission, preferences, values, rules.

2. Episodic memory
   Timestamped events and research episodes.

3. Semantic memory
   Facts, concepts, definitions, equations, theories.

4. Procedural memory
   Skills, scripts, workflows, debugging patterns.

5. Resource memory
   Papers, datasets, repos, files, APIs, tools.

6. Social/source memory
   Who said what, source credibility, author networks.

7. Temporal graph memory
   What was true when, what changed, what supersedes what.

8. Reflective memory
   Lessons learned, failed strategies, search gaps, contradictions.

9. Evaluation memory
   What retrievals worked, what answers were wrong, what tests passed.

10. Private/vault memory
   Sensitive user/project data with strict access rules.
```

## Minimal build plan

For a serious implementation, I would start with this stack:

| Layer                    | Recommendation                                        |
| ------------------------ | ----------------------------------------------------- |
| Durable memory files     | JSONL/YAML/Markdown committed to Git                  |
| Exact retrieval          | Tantivy or PostgreSQL full-text search                |
| Vector retrieval         | pgvector, Qdrant, LanceDB, or Chroma                  |
| Temporal graph           | Graphiti-style graph, Neo4j, Kùzu, or custom JSON-LD  |
| Memory compiler          | Rust-first CLI with Python for ML/PDF/science tooling |
| Reflection/consolidation | Scheduled LLM jobs with schema validation             |
| Skill memory             | Executable scripts with tests and metadata            |
| Context packing          | Deterministic retriever + LLM reranker                |

The most important rule:

> **Never let the LLM’s context window be the memory. The context window should receive only a compiled memory packet.**

The memory system itself should live outside the model, self-organize over time, and produce compact, cited, task-specific context.

## My final recommendation

For your goal — a research agent that gets smarter over months of physics/literature/data/code work — I would build:

```text
MIRIX taxonomy
+ Graphiti temporal knowledge graph
+ A-MEM / EverMemOS self-organization
+ Mem0-style fast retrieval
+ Voyager-style executable skill memory
+ Git-native claim/source ledgers
```

That gives you the most human-like and useful system:

```text
human-like categories
+ fast recall
+ temporal truth
+ concept self-organization
+ procedural skill compounding
+ source-grounded scientific memory
```

The closest existing single answer is **MIRIX**.
The strongest practical architecture is **MIRIX + Graphiti + A-MEM/EverMemOS + Mem0 + Voyager-style skills**.
