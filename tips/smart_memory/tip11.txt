My answer: **the most human-like complete agent-memory design today is MIRIX**, because it explicitly separates **core, episodic, semantic, procedural, resource, and knowledge-vault memory**, then uses specialized memory agents to coordinate update and retrieval. But the **fastest high-performing design** is likely **Mastra Observational Memory**, because it avoids per-turn dynamic retrieval and keeps a stable, cacheable context. The **best production temporal-reasoning substrate** is **Zep/Graphiti**, because it stores changing facts as a temporal knowledge graph instead of static memories.

I would not choose only one. For a serious system like OpenQG, I would combine **MIRIX-style memory taxonomy + Graphiti temporal KG + A-MEM self-organizing notes + Observational Memory compression + Voyager-style procedural skill library**.

## Ranking method

I scored each design on:

| Axis           | Meaning                                                                                                         |
| -------------- | --------------------------------------------------------------------------------------------------------------- |
| **Human-like** | Episodic memory, semantic abstraction, procedural skill, reflection, temporal change, forgetting/consolidation. |
| **Adaptable**  | Learns from new experiences, self-organizes, updates old memories, handles changing facts.                      |
| **Fast**       | Retrieval latency, token efficiency, no huge context stuffing, scalable query path.                             |

LongMemEval is a useful benchmark reference because it evaluates long-term memory across **information extraction, multi-session reasoning, temporal reasoning, knowledge updates, and abstention**. It is not perfect, but those dimensions are close to what agent memory actually needs. ([Hugging Face][1])

# 1. MIRIX — most human-like complete memory architecture

**Why it ranks #1:** MIRIX is the closest to a full cognitive memory stack. It separates memory into **Core, Episodic, Semantic, Procedural, Resource Memory, and Knowledge Vault**, and coordinates them with a multi-agent memory framework. It also supports multimodal memory, including screen activity, which matters because human memory is not text-only. The paper reports strong results on ScreenshotVQA and LOCOMO, and the GitHub repo describes local storage, multi-agent memory components, screen tracking, PostgreSQL BM25/vector search, and multimodal input. ([Hugging Face][2])

**Theory:** human memory is not one database. It is multiple specialized stores: identity/core beliefs, remembered events, abstract knowledge, learned procedures, environmental resources, and durable reference knowledge. MIRIX maps that directly into agent architecture.

**Build schematic:**

```text
User / screen / tools / files
        ↓
Perception + event capture
        ↓
Memory router
        ↓
Core memory        → stable identity, goals, constraints
Episodic memory    → dated events and conversations
Semantic memory    → extracted facts and concepts
Procedural memory  → reusable workflows and skills
Resource memory    → files, apps, screenshots, external resources
Knowledge vault    → durable reference knowledge
        ↓
Specialized update agents
        ↓
Hybrid retrieval: BM25 + vector + structured filters
        ↓
Context pack for main agent
```

**How to build it:** create separate typed stores for each memory kind. Use a router agent to classify incoming events. Use background consolidation agents to promote raw episodes into semantic facts and procedural skills. Keep Core memory small and always visible. Retrieve from all stores with a planner that knows which memory type answers which question.

---

# 2. Zep / Graphiti — best temporal knowledge-graph memory

**Why it ranks #2:** Zep’s core idea is that agent memory should track **what was true when**, not just “facts.” Graphiti, the open-source engine behind Zep’s temporal context graph, stores entities, facts/relationships with validity windows, source episodes, and evolving summaries. It supports hybrid semantic, keyword, and graph traversal retrieval, and it is designed for incremental updates without full graph recomputation. ([GitHub][3])

Zep’s paper reports that its temporal KG architecture outperformed MemGPT on Deep Memory Retrieval and improved LongMemEval performance with major latency reductions versus baseline implementations. ([GraphRAG][4])

**Theory:** human memory is temporal and contextual. If someone says “I only wear Adidas” in September and then “I’m switching to Nike” in October, a good memory should not delete the first fact; it should preserve the history and know which fact is current.

**Build schematic:**

```text
Episode: raw conversation/tool/event
        ↓
Entity extraction
        ↓
Fact extraction: subject → predicate → object
        ↓
Temporal edge:
  valid_from, valid_to, source_episode, confidence
        ↓
Graph update:
  invalidate superseded facts, preserve history
        ↓
Hybrid retrieval:
  semantic + BM25 + graph traversal + time filters
        ↓
Context graph summary for agent
```

**How to build it:** use a temporal property graph. Every fact edge needs `valid_from`, `valid_until`, `source_episode_id`, and `confidence`. Do not overwrite facts; supersede them. Retrieval should answer both “what is true now?” and “what was true then?”

For OpenQG, this is the right substrate for claims like: *paper X supported equation Y in version 1, but later erratum Z changed assumption A.*

---

# 3. MemOS — most ambitious “memory operating system”

**Why it ranks #3:** MemOS treats memory as a first-class OS resource and unifies **plaintext memory, activation memory, and parameter memory** under a `MemCube` abstraction with provenance and versioning. It proposes memory scheduling, memory lifecycle management, and memory evolution across tasks, sessions, and agents. ([Hugging Face][5])

The GitHub project describes a unified memory API, graph-structured editable memory, multimodal memory, and composable memory cubes for multiple knowledge bases. ([GitHub][6])

**Theory:** most memory systems are external note stores. MemOS says a future AI system needs memory at several levels: text you can inspect, latent activation-like state, and learned/parameterized memory. That is closer to a whole cognitive runtime than a RAG layer.

**Build schematic:**

```text
Memory API layer
  add / retrieve / edit / delete / compose

Memory scheduler
  decides what to load, compress, migrate, fuse, retire

Memory cube
  content
  type: plaintext | activation | parameter
  provenance
  version
  permissions
  lifecycle state

Storage layer
  vector store
  graph store
  parameter adapters
  cache
```

**How to build it now:** implement `MemCube` first as plaintext/structured memory. Add provenance, versioning, owner, permissions, embeddings, graph links, and lifecycle state. Later, add LoRA/adapters or fine-tuning artifacts as “parameter memory.” The important idea is not the exact implementation; it is **memory scheduling**.

---

# 4. Mastra Observational Memory — fastest human-inspired design

**Why it ranks #4:** Observational Memory uses two background agents: an **Observer** converts raw conversation into dated dense observations, and a **Reflector** periodically restructures and condenses those observations. The main agent does not do per-turn vector search. It sees a stable observation log plus recent conversation, making the prompt predictable and cacheable. Mastra reports 94.87% on LongMemEval with `gpt-5-mini` and 84.23% with `gpt-4o`, but these are vendor-published results, so I would replicate before treating them as settled. ([Mastra][7])

**Theory:** humans do not remember raw transcripts. We remember compressed observations: “the user changed their mind,” “the project deadline moved,” “that method failed.” OM mimics that.

**Build schematic:**

```text
Recent messages
    ↓ when token threshold exceeded
Observer agent
    ↓
Dated observations:
  - event
  - priority
  - temporal anchor
  - entities
  - current task note
    ↓ when observation log grows
Reflector agent
    ↓
Condensed observation log
    ↓
Stable context prefix + recent messages
```

**How to build it:** keep a text observation log at the front of the context. Append observations in a fixed format. When it grows too large, run a reflector that merges duplicates, marks superseded observations, and preserves temporal anchors. This is extremely simple compared with graph memory and can be very fast.

For OpenQG, this would be excellent for **agent run memory**: “Last run tried provider X; arXiv query Y missed INSPIRE records; equation extraction failed on PDF Z.”

---

# 5. A-MEM — best self-organizing conceptual memory

**Why it ranks #5:** A-MEM explicitly uses **Zettelkasten principles**: each memory becomes a structured note with context, keywords, tags, and links to related memories. New memories can trigger updates to older memories, so the memory network evolves. The GitHub implementation supports ChromaDB indexing, automatic keyword/context/tag generation, semantic connection discovery, and memory evolution. ([Hugging Face][8])

**Theory:** intelligence compounds when memories are not isolated. A Zettelkasten works because notes link into a concept graph. A-MEM applies that idea to agents: the agent’s memory becomes a living network rather than a pile of logs.

**Build schematic:**

```text
New memory
    ↓
Atomic note generator
  content
  context
  keywords
  tags
  timestamp
    ↓
Similarity search over old notes
    ↓
Link generator
  note A ↔ note B
    ↓
Memory evolution
  update old note context/tags/links
    ↓
Self-organizing note graph
```

**How to build it:** store each memory as a typed note. On insert, do semantic search against prior notes, ask an LLM whether a link is meaningful, then update both the new note and older related notes. This is ideal for research knowledge, where concepts, equations, datasets, and theories should cross-link over time.

For OpenQG, A-MEM is one of the best designs for **conceptual self-organization**.

---

# 6. Mem0 — strongest production pragmatic memory layer

**Why it ranks #6:** Mem0 is a production-oriented memory system for persistent personalization and agent memory. The paper describes dynamic extraction, consolidation, and retrieval, with a graph-memory variant for relational structure. It reports accuracy improvements and large latency/token reductions versus full-context approaches; the GitHub project is widely used and open source. ([Hugging Face][9])

Mem0’s 2026 algorithm emphasizes single-pass ADD-only extraction, entity linking, and multi-signal retrieval across semantic, BM25, and entity matching. Its own benchmarks report strong LoCoMo and LongMemEval results with under 7,000 tokens per retrieval call. Again, I would independently replicate vendor benchmark claims. ([Mem0][10])

**Theory:** for production, memory must be cheap, fast, and boring. Mem0 optimizes for extracting useful facts, linking entities, and retrieving with multiple signals rather than doing expensive multi-agent reflection every turn.

**Build schematic:**

```text
Conversation/event
    ↓
Memory extractor
    ↓
Fact memory:
  subject, predicate, object
  entities
  timestamp
  source
    ↓
Entity linker
    ↓
Hybrid retrieval:
  semantic + BM25 + entity match + recency
    ↓
Context snippets
```

**How to build it:** use a fact extractor, entity linker, vector index, BM25 index, and recency/importance scoring. Keep updates append-only where possible. Avoid over-editing old memories unless you have strong temporal/provenance logic.

# 7. HippoRAG — best hippocampal associative recall design

**Why it ranks #7:** HippoRAG is explicitly neurobiologically inspired. It combines LLM extraction, knowledge graphs, and **Personalized PageRank** to mimic hippocampal indexing for long-term memory. The paper reports that single-step HippoRAG retrieval can be 10–30× cheaper and 6–13× faster than iterative retrieval while improving multi-hop QA. ([Hugging Face][11])

**Theory:** the hippocampus does not store every fact as a flat vector. It indexes related experiences and lets activation spread through associations. HippoRAG approximates this using a graph plus PageRank-style spreading activation.

**Build schematic:**

```text
Corpus / episodes
    ↓
Entity + relation extraction
    ↓
Knowledge graph
    ↓
Query entities
    ↓
Personalized PageRank
    ↓
Top related facts / passages
    ↓
Answer context
```

**How to build it:** extract entities/relations from documents, build a graph, map each query to seed nodes, run Personalized PageRank, then retrieve passages attached to top graph nodes. This is excellent for scientific multi-hop questions: “which observations constrain this assumption through which intermediate theory?”

---

# 8. Letta / MemGPT — best OS-style self-editing context memory

**Why it ranks #8:** MemGPT introduced **virtual context management**: the agent treats the context window like fast memory and external storage like slow memory, moving information in and out using tools. Letta is the evolved open-source platform around this idea, with core memory blocks, recall memory, archival memory, memory editing tools, and persisted state. ([Hugging Face][12])

Letta’s current docs describe agents with editable memory blocks, out-of-context recall and archival memory, and automatic compaction/eviction when the context fills. The GitHub repo describes Letta as a platform for stateful agents with advanced memory that can learn and self-improve over time. ([Letta Docs][13])

**Theory:** the context window is RAM. Long-term storage is disk. The agent should learn to page information in and out. That was one of the first strong memory designs for long-running LLM agents.

**Build schematic:**

```text
Context window
  system prompt
  core memory blocks
  recent messages
        ↕ tools
Recall memory
  searchable conversation history
        ↕ tools
Archival memory
  semantic long-term store
        ↕ tools
Agent decides:
  insert / replace / rethink / search / compact
```

**How to build it:** give the agent explicit tools to edit pinned memory blocks, search prior messages, and write archival memories. Add rules for when to compact, when to retrieve, and when to rewrite core memory. This is a strong control-plane pattern, especially for agents that must introspect and manage their own state.

---

# 9. Voyager — best procedural skill memory

**Why it ranks #9:** Voyager is not a general memory database; it is a **lifelong procedural memory system**. It learns executable code skills in Minecraft, stores them in an ever-growing skill library, and uses an automatic curriculum plus iterative prompting with environment feedback. The authors report that learned skills are temporally extended, interpretable, compositional, and reduce catastrophic forgetting. ([GitHub][14])

**Theory:** humans do not only remember facts. We remember how to do things: ride a bike, solve an integral, debug a build, run an experiment. Voyager’s skill library is one of the cleanest examples of procedural memory for agents.

**Build schematic:**

```text
Environment state
    ↓
Automatic curriculum proposes next task
    ↓
Code generation
    ↓
Execute skill
    ↓
Observe errors / success
    ↓
Self-verification and repair
    ↓ if successful
Store reusable skill:
  name
  code
  preconditions
  description
  embeddings
  tests
    ↓
Retrieve and compose skills for harder tasks
```

**How to build it:** every successful workflow becomes a versioned skill with inputs, outputs, preconditions, code, tests, and retrieval metadata. Failed attempts produce reflections. This is critical for OpenQG because the agent should remember **how to query sources, extract equations, validate datasets, run smoke tests, and reproduce benchmark scripts**.

---

# 10. Generative Agents memory stream — most human-behavior-like classic design

**Why it ranks #10:** Generative Agents introduced a highly human-like architecture: a **memory stream**, retrieval by recency/importance/relevance, reflection into higher-level insights, and planning. The paper describes agents that remember, reflect, and plan in a sandbox world; the open-source repo contains the simulation code. ([Hugging Face][15])

It is lower on speed and scalability than modern systems, but it remains one of the clearest cognitive blueprints for believable human-like agents.

**Theory:** behavior emerges from remembered experiences plus reflection plus plans. The agent does not just retrieve a fact; it asks, “What experiences matter now? What higher-level conclusions have I formed? What should I do today?”

**Build schematic:**

```text
Observation
    ↓
Memory stream append:
  event text
  timestamp
  importance score
    ↓
Retrieval score:
  relevance + recency + importance
    ↓
Reflection trigger
    ↓
Higher-level insight memories
    ↓
Daily / task planning
    ↓
Action
```

**How to build it:** maintain an append-only memory stream. Score memories by recency, importance, and semantic relevance. Periodically ask the model to reflect over top memories and produce durable insights. Use those insights to plan future actions.

For OpenQG, this pattern is useful for **research planning**: “What have we learned this week? What hypotheses are recurring? What gaps should drive tomorrow’s search?”

## What I would actually build

For an OpenQG-style scientific memory system, I would build a **hybrid**:

```text
1. MIRIX taxonomy
   Core / Episodic / Semantic / Procedural / Resource / Vault

2. Graphiti temporal KG
   Claims, equations, datasets, papers, authors, contradictions, validity windows

3. A-MEM concept graph
   Zettelkasten notes that self-link across theories, math, datasets, code

4. Observational Memory
   Agent-run observation log for fast continuity and stable context

5. Voyager procedural memory
   Reusable scripts and skills:
   search_arxiv, query_inspire, extract_equations, verify_dataset, run_benchmark

6. MemGPT/Letta control plane
   Core memory blocks + archival search + recall + compaction tools

7. Mem0-style retrieval engine
   Multi-signal search: semantic + BM25 + entities + recency + graph proximity

8. HippoRAG reasoning layer
   Personalized PageRank over the claim/equation/source graph for multi-hop recall
```

The final schematic:

```text
Raw experience
  papers, code, datasets, conversations, agent runs
        ↓
Observer layer
  dense dated observations
        ↓
Typed memory router
  core / episodic / semantic / procedural / resource / vault
        ↓
Canonical stores
  Git cards + temporal graph + vector/BM25 index + skill library
        ↓
Consolidation
  reflection, Zettelkasten linking, contradiction detection, temporal invalidation
        ↓
Retrieval planner
  task → memory types → graph/vector/BM25/PPR → context pack
        ↓
Main reasoning agent
        ↓
Write-back
  new observations, claims, skills, questions, tests
```

The key rule: **never let memory be just embeddings.** Embeddings are recall hints. Real compounding intelligence needs typed memories, temporal provenance, procedural skills, reflection, and self-organizing concept links.
