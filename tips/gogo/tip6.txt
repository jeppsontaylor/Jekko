## My conclusion

The proposed “Daemon Mode” plan is **directionally right**, but it is not strong enough yet. The key correction is this:

**“FOREVER” must not be a prompt instruction. It must be an Jekko runtime mode.**

A model can be instructed to continue, but it can still emit `finish=stop`, lose context after compaction, hit permission blocks, summarize itself into uselessness, or decide the task is complete. A real forever loop has to be enforced by the host application: Jekko should treat the model’s “I’m done” as only one event in a larger daemon state machine.

The right design is:

> **A deliberately armed Jekko Daemon runtime, driven by a structured spec, with host-enforced looping, stop conditions, checkpoint hooks, context-reset policy, durable task state, visible gold TUI mode, and optional multi-agent worker pools.**

I did not find a single universal “standard in-prompt agent scripting language” that Jekko already supports. What exists instead is a set of patterns across current agent systems: protocol standards like MCP, ACP, A2A, and Agent Protocol; orchestration frameworks like LangGraph, AutoGen, CrewAI, Semantic Kernel, and the OpenAI Agents SDK; prompt/programming DSLs like Guidance, LMQL, DSPy, BAML, and Plang; and lifecycle/hook systems like Claude Code hooks. The best Jekko design should synthesize those ideas rather than copy raw XML tags. MCP is an open standard for connecting AI apps to tools/data/workflows, ACP standardizes editor/IDE-to-agent communication, A2A and Agent Protocol address agent-to-agent interoperability, while frameworks such as LangGraph and AutoGen provide explicit state machines, termination conditions, handoffs, and long-running orchestration concepts. ([Model Context Protocol][1])

# 1. What the Jekko ecosystem already shows

Jekko already has many of the pieces needed, but not the runtime semantics that make an agent truly unstoppable until a stop condition is satisfied.

Jekko supports custom commands as prompt templates, including command files under `.jekko/commands/*.md`, front matter, `$ARGUMENTS`, and positional placeholders. That is useful as one frontend for daemon specs, but by itself it is still “prompting,” not host-enforced execution. ([Jekko][2])

Jekko also supports agents and subagents. The docs distinguish primary agents from subagents, allow automatic or `@`-mentioned invocation, and include built-in hidden agents for compaction, title, and summary work. Agents have a `steps` setting where the default behavior is to keep iterating until the model stops or the user interrupts; when a step limit is reached, Jekko prompts the model to summarize instead of continuing tools. This is exactly the current problem: the natural stopping point is still the model’s stopping behavior, not an external daemon contract. ([Jekko][3])

Jekko’s permission model is already expressive enough to be part of the daemon story. Agent permissions cover areas such as `read`, `edit`, `bash`, `task`, `webfetch`, `websearch`, `lsp`, `question`, and `doom_loop`, and markdown agents can define permissions with command patterns and wildcards. That means daemon mode should not invent a new permission system; it should add stricter policy wrappers around the existing one. ([Jekko][3])

Jekko already supports MCP server configuration under an `mcp` section, with unique names, enablement, and server settings. That is enough to make MCP tool availability part of a daemon spec, but Jekko still needs first-class daemon orchestration if you want robust parallel workers, joins, and durable worker state. ([Jekko][4])

The current `jekko run` command supports non-interactive/headless execution and server attachment, which makes it the obvious CLI surface for daemon mode. But the headless runner must not exit just because a session becomes idle if that session is daemon-owned and its stop criteria are not met. ([Jekko][5])

## The community has already described this pain

There is an Jekko feature request asking for native continuous execution loops because agents stop early after compaction, summarization, or intermediate milestones. The requested loop is almost exactly what you described: read instructions, pick the next task, implement, validate, fix failures, update progress, and continue until stop conditions are met. It even proposes config fields like `continuous`, `resume_after_compaction`, and `stop_conditions`. ([GitHub][6])

Another Jekko issue asks for a lifecycle hook before stopping, because plugins currently cannot reliably prevent the prompt loop from ending. The request proposes something like `session.stopping` where a plugin could return `stop: false` and inject a continuation message. That issue also calls out the danger: if `stop: false` is unconditional, the loop continues forever unless the plugin has its own termination condition. ([GitHub][7])

Several Jekko issues show that compaction is part of the problem, not just part of the solution. Users report `jekko run` exiting after compaction, sessions stopping after compaction until manually continued, and unwanted synthetic “Continue” behavior after compaction when the assistant had naturally ended. One issue correctly identifies the missing concept: Jekko needs to distinguish “active task should resume” from “assistant is waiting for user.” ([GitHub][8])

There are also Jekko requests for deterministic hooks, including hooks comparable to Claude Code lifecycle hooks and hooks after a task ends for auto-commit/push/PR workflows. That strongly supports making checkpointing a host hook, not just a sentence in the prompt. ([GitHub][9])

Finally, parallel subagent execution has already been discussed. One issue reported that multiple `Task` tool calls executed sequentially even though regular tool calls were parallelized elsewhere. Whether that specific issue is closed or fixed, daemon mode should explicitly model worker dispatch, isolation, and joins rather than hoping the model calls multiple subagents correctly. ([GitHub][10])

# 2. What broader agent systems teach us

There is no mature universal “put this syntax in a prompt and the model will obey forever” standard. The serious systems are moving in the opposite direction: **the prompt describes intent; the runtime enforces control flow.**

MCP is the relevant tool/context protocol layer. It standardizes how applications expose tools, resources, prompts, sampling, roots, and elicitation. Important for your use case: MCP does not itself define “forever loops”; it provides interoperable tool/context plumbing that a daemon runtime can use. MCP’s sampling design also keeps model access under client control, which is the right security stance for multi-agent systems. ([Model Context Protocol][11])

Agent-to-agent and editor-to-agent standards are also emerging. Agent Protocol presents a framework/language/platform-independent API for agent communication. Google’s A2A is aimed at secure agent communication and capability discovery, including `.well-known/agent-card.json`. ACP standardizes how code editors and IDEs communicate with coding agents; Jekko’s docs also mention ACP support. These are useful for interoperability, but none of them replace an Jekko-native daemon loop. ([agentprotocol.ai][12])

The orchestration frameworks point to the right internal architecture. LangGraph focuses on long-running, stateful agent workflows. AutoGen uses explicit termination conditions such as text mention termination, max message termination, and handoff termination. CrewAI Flows emphasize event-driven workflow, state, and control flow. Semantic Kernel’s agent orchestration supports sequential and handoff patterns. The OpenAI Agents SDK models agents with instructions, tools, handoffs, guardrails, and structured multi-step tool calling. ([GitHub][13])

The prompt-DSL world gives useful syntax ideas, but not the right execution authority. Guidance interleaves generation, prompting, loops, conditionals, tool use, regex constraints, and CFG constraints. LMQL combines prompting with programming and constrained decoding. DSPy argues against brittle strings and represents LM programs as declarative modules or graphs. BAML treats prompts as typed functions. Plang explores prompt text interleaved with control flow. These are valuable references for syntax design, but Jekko should not rely on the model to parse and faithfully execute an infinite loop from prompt text alone. ([GitHub][14])

Claude Code’s hook model is especially relevant. Its hooks run at lifecycle events such as session start, user prompt submit, pre/post tool use, permission request, subagent start/stop, task events, stop, stop failure, and pre-compaction. The docs emphasize that hooks provide deterministic control rather than relying on the LLM, and that hooks can run outside the LLM context. That is exactly the philosophy Jekko daemon mode needs. ([Claude][15])

Context engineering research and discussion also strongly supports your intuition. Context windows are finite, long sessions degrade, compaction can lose important state, and long-horizon software agents suffer from context bloat, cost, latency, and reasoning degradation. The solution is not just “summarize harder”; it is durable state plus controlled hydration into each iteration. ([Anthropic][16])

# 3. Critical review of the AI-proposed plan

The proposed plan has several good instincts:

It correctly identifies the dominant failure mode: the model emits a natural stop and the host loop exits. For daemon mode, that has to be overridden.

It correctly reuses existing Jekko primitives: compaction, session status, TUI theming, `jekko run`, agents, MCP, and permissions.

It correctly recognizes that CLI flags, prompt front matter, and custom commands are separate UX surfaces.

It correctly suggests a visible gold TUI state. A forever loop should be visually impossible to miss.

But it has serious flaws.

## Problem 1: YAML front matter in arbitrary prompts is too easy to trigger accidentally

The plan suggests daemon YAML front matter in a prompt. That is not safe enough. People paste markdown, docs, examples, logs, and specs into coding agents constantly. A daemon spec must require **deliberate arming**, not just parseable syntax.

The parser should require all of the following:

1. A sentinel opening token at column 1.
2. A matching sentinel close token.
3. A schema version.
4. A unique daemon ID.
5. A content hash.
6. A separate arming phrase.
7. Project or CLI enablement.
8. TUI preview or CLI `--arm` confirmation.

Without those gates, the syntax will eventually fire from copied documentation.

## Problem 2: “Outer while prompt()” is too crude

A simple outer loop around `prompt()` may work in trivial cases, but it risks fighting the existing session runner, status updates, cancellation behavior, compaction, and headless `run` lifecycle.

The better design is a **daemon state machine integrated with the session prompt loop**:

```text
assistant stop event
        │
        ▼
daemon stop resolver
        │
        ├── real exit
        ├── pause for permission/human
        ├── retry transient failure
        ├── compact/reset context
        ├── checkpoint
        └── inject next daemon iteration
```

Normal sessions should keep the current behavior. Only daemon-owned sessions should intercept stop events.

## Problem 3: It conflates compaction with continuation

Jekko issues show both sides of this bug: sometimes compaction causes unwanted continuation, and sometimes compaction fails to continue when it should. The daemon runtime must separate these concepts:

```text
compaction = produce smaller context
continuation = daemon policy decision
```

Compaction should never unconditionally inject “continue.” Daemon mode should decide whether to continue based on stop criteria, budgets, errors, and active task state.

## Problem 4: Git checkpointing should be deterministic host behavior

Your `<GIT>` block says “commit and push.” If that is left as a prompt instruction, the model may skip it, do it late, do it with the wrong files, or summarize instead.

Checkpointing needs to be a daemon hook:

```text
BeforeCheckpoint
  git status --short
  validate no protected files
  git add selected paths
  git diff --cached
  commit with structured message
  push
AfterCheckpoint
  record commit SHA
```

The model can propose commit messages or explain changes, but the daemon host should enforce when checkpoint hooks run.

## Problem 5: “Forever” still needs budgets and kill semantics

The user intent is clear: the agent should not stop merely because the model says it is done. But a responsible implementation still needs manual kill, cost budgets, rate limits, retry limits, and protected-path policies.

So the semantics should be:

> **Forever means: the model’s natural stop is not a stop condition. The daemon continues until explicit stop criteria, manual kill, policy pause, or configured safety budget.**

That preserves the “promised infinite loop” while preventing accidental infinite spend or destructive loops.

## Problem 6: Multi-agent work in one dirty worktree is collision-prone

Multiple MCP sessions editing one working tree will corrupt each other’s assumptions. The daemon should use isolated workspaces:

```text
orchestrator session
  ├── worker A in git worktree branch daemon/A
  ├── worker B in git worktree branch daemon/B
  ├── worker C in git worktree branch daemon/C
  └── verifier/merger applies patches back to main worktree
```

The central tracker should not be a markdown file that everyone edits. Markdown can be a human-readable mirror, but the source of truth should be a durable task graph/event log with leases and locks.

# 4. The architecture I would build in Jekko

## Core principle

Jekko should introduce a new runtime concept:

```ts
SessionMode = "normal" | "daemon"
```

A daemon session has:

```ts
DaemonSpec       // what the user asked for
DaemonRunState   // what has happened so far
DaemonEngine      // host-enforced state machine
DaemonStatus      // TUI/headless-visible state
DaemonLedger      // durable event log
DaemonTaskGraph   // durable task queue / DAG
DaemonWorkers     // optional subagent/MCP worker pool
```

The model gets a compact, hydrated view of that state each iteration, but the model does not own the state.

Durable state.

## `context.ts`

This is the key to “context clearing.”

Daemon context should be layered:

```text
Layer 1: immutable daemon spec
Layer 2: durable task graph
Layer 3: last successful checkpoint
Layer 4: current iteration objective
Layer 5: recent failures/blockers
Layer 6: compacted summary of prior work
Layer 7: relevant files/snippets loaded on demand
```

The model should not see the entire history. It should see the **current slice**.

Lifecycle hooks should be deterministic and outside context.

This mirrors the successful pattern in Claude Code, where lifecycle hooks run deterministically instead of depending on model compliance. ([Claude][15])

For multi-agent execution.

Responsibilities:

```text
- assign tasks
- enforce leases
- prevent file collisions
- create git worktrees or isolated sessions
- collect worker results
- join results
- send patches to verifier
```

# 6. Changes to existing Jekko files

## `packages/jekko/src/session/prompt.ts`

This is the most important patch area.

Do **not** simply replace every `break`. Instead add a daemon-aware stop resolver at every exit path.

Conceptual change:

```ts
const decision = yield* DaemonStopResolver.resolve({
  sessionID,
  reason,
  lastAssistant,
  context,
})

switch (decision.type) {
  case "normal-break":
    break

  case "daemon-continue":
    yield* DaemonEngine.continue(decision)
    continue

  case "daemon-compact-then-continue":
    yield* DaemonContext.compact(decision)
    yield* DaemonEngine.continue(decision)
    continue

  case "daemon-exit":
    yield* DaemonEngine.exit(decision)
    break

  case "daemon-pause":
    yield* DaemonEngine.pause(decision)
    break
}
```

Exit paths to intercept:

```text
1. model finish not tool-calls
2. processor returns stop
3. max steps reached
4. compaction fails
5. structured output completes
6. provider/transient errors
7. permission blocked
8. user cancellation
```

In normal mode, current behavior remains unchanged.

In daemon mode:

```text
model_natural_stop       → evaluate exit conditions; otherwise continue
max_agent_steps          → reset step counter or start new iteration
context_overflow         → compact/clear/new session
compaction_failed        → hard clear or new session, preserving durable state
permission_blocked       → pause, ask, or continue depending policy
transient_model_error    → retry/backoff
structured_output        → treat as worker result, not session end
user_cancelled           → real stop
```

Change compaction semantics:

```ts
type CompactionResult = {
```

# 7. The daemon flow

## Normal Jekko flow today

```text
User prompt
   │
   ▼
create user message
   │
   ▼
LLM/tool loop
   │
   ├── tool calls → processor → continue
   │
   ├── compaction → maybe continue
   │
   └── model stop → session idle → done
```

## Proposed daemon flow

```text
DaemonSpec armed
   │
   ▼
DaemonEngine boot
   │
   ▼
Iteration N starts
   │
   ▼
Hydrate bounded context from durable state
   │
   ▼
Run normal Jekko LLM/tool loop
   │
   ▼
Assistant/model stops
   │
   ▼
DaemonStopResolver
   │
   ├── exit conditions satisfied → checkpoint → exit
   ├── budget/policy violation → pause/exit
   ├── context policy says compact → compact → continue
   ├── worker tasks pending → join/spawn → continue
   └── not done → checkpoint if needed → iteration N+1
```

## Multi-agent daemon flow

```text
                     ┌───────────────────────┐
                     │  Orchestrator session  │
                     └───────────┬───────────┘
                                 │
                                 ▼
                     ┌───────────────────────┐
                     │ Durable task graph     │
                     │ events.jsonl           │
                     │ leases/locks           │
                     └───────┬───────┬───────┘
                             │       │
             ┌───────────────┘       └───────────────┐
             ▼                                       ▼
   ┌─────────────────────┐                 ┌─────────────────────┐
   │ Worker A session    │                 │ Worker B session    │
   │ git worktree A      │                 │ git worktree B      │
   └──────────┬──────────┘                 └──────────┬──────────┘
              │                                       │
              ▼                                       ▼
       patch + evidence                         patch + evidence
              │                                       │
              └───────────────┬───────────────────────┘
                              ▼
                    ┌───────────────────┐
                    │ Verifier / merger │
                    └─────────┬─────────┘
                              ▼
                    main worktree commit/push
```

The orchestrator, not workers, owns final commit/push unless the spec explicitly allows worker checkpointing.

# 8. How I would handle context clearing

The user’s `<CLEAR_AGENT_CONTEXT>` idea is correct in spirit, but the implementation must be more precise.

There should be four context policies:

## `compact`

Summarize prior context, preserve durable state, continue same session.

Best for:

```text
- normal long tasks
- debugging
- code review
- worktree cleanup
```

## `clear`

Drop conversation history and rehydrate only daemon state.

Best for:

```text
- repetitive task loops
- audit/fix/audit/fix workflows
- cases where old chat history harms performance
```

## `new_session`

Start a new Jekko session each iteration, linked by daemon run ID.

Best for:

```text
- very long-running daemons
- parallel workers
- avoiding provider-side conversation weirdness
```

## `hybrid`

Compact for a while, then hard reset every N iterations.

Example:

```yaml
context:
  strategy: hybrid
  compact_every_iterations: 2
  clear_every_iterations: 8
  preserve:
    - daemon_spec
    - task_graph
    - current_git_status
    - latest_audit
    - failed_tests
    - active_locks
```

The daemon prompt after context clearing should not say only “continue.” It should render a structured continuation packet:

```text
<daemon-runtime-context>
Run: jankurai-cleanup
Iteration: 8
Mode: FOREVER
Exit condition: jankurai audit reports 0 findings and 0 caps
Current state:
- 12 completed tasks
- 3 pending tasks
- last checkpoint: abc123
- latest audit: 3 findings, 1 cap
- current objective: select one pending task, fix, verify, checkpoint
Rules:
- Do not stop because you completed one item.
- Stop only when daemon runtime reports exit condition satisfied.
- If uncertain, inspect durable task graph and audit output.
</daemon-runtime-context>
```

This is prompt text, but it is generated by the runtime from durable state.

# 10. The syntax: deliberate, armed, impossible to trigger accidentally

Do not use raw `<FOREVER>`. It is too easy to paste by accident and too easy for the model to treat as ordinary XML-ish text.

Use a sentinel that looks like executable infrastructure, not prose.

## Required opening and closing gates

```text
<<<JEKKO_DAEMON_SPEC v1 id="jankurai-cleanup" sha256="...">>>
...
<<<END_JEKKO_DAEMON_SPEC id="jankurai-cleanup">>>
ARM JEKKO DAEMON jankurai-cleanup
```

Execution requires:

```text
1. project config allows daemon specs
2. prompt was submitted through /daemon or CLI --daemon, OR TUI asks for confirmation
3. opening sentinel is at column 1
4. closing sentinel matches ID
5. spec validates against schema
6. sha256 matches canonical spec body
7. arm phrase appears outside the spec
8. user confirms preview, unless CLI --arm was passed
```

If any condition fails, Jekko treats the entire thing as ordinary text.

## Why the hash matters

The hash prevents this:

```text
User copies an old daemon spec from docs.
Model edits the body.
Parser still sees a valid daemon block.
Boom: unexpected behavior.
```

With a hash, any content change invalidates the spec until the user intentionally re-arms it.

# 12. Full syntax library

Below is the syntax library I would design. This should be host-parsed and host-enforced wherever possible. The model may receive a rendered version, but the model should not be trusted as the parser or executor.

## Top-level fields

| Field           | Purpose                                  |
| --------------- | ---------------------------------------- |
| `version`       | Schema version. Required.                |
| `id`            | Stable daemon run/spec ID. Required.     |
| `description`   | Human-readable description.              |
| `owner`         | Optional user/team owner.                |
| `mode`          | Loop semantics.                          |
| `arm`           | Deliberate execution gating.             |
| `prompt`        | Natural-language task body.              |
| `exit`          | Stop conditions.                         |
| `context`       | Compaction/clearing/hydration policy.    |
| `task_state`    | Durable task graph source and sinks.     |
| `selection`     | How next task is chosen.                 |
| `workflow`      | Explicit steps or graph.                 |
| `agents`        | Orchestrator and worker pool.            |
| `mcp`           | MCP server/tool policy.                  |
| `tools`         | Tool policy.                             |
| `permissions`   | Jekko permission overrides.           |
| `verification`  | Required validation before completion.   |
| `checkpoint`    | Commit/push/stash/save policy.           |
| `hooks`         | Lifecycle hooks.                         |
| `budgets`       | Cost/runtime/token/concurrency controls. |
| `on_error`      | Error handling.                          |
| `observability` | TUI/log/metrics settings.                |

## `mode`

```yaml
mode:
  forever: true
  stop_on_model_stop: false
  manual_kill: true
  pause_on_policy_violation: true
  allow_idle_between_iterations: true
```

Supported fields:

| Field                           | Meaning                                              |
| ------------------------------- | ---------------------------------------------------- |
| `forever`                       | Continue until daemon stop criteria, not model stop. |
| `stop_on_model_stop`            | Usually `false` in daemon mode.                      |
| `manual_kill`                   | User can kill/pause. Should always be true.          |
| `max_iterations`                | Optional safety bound.                               |
| `max_runtime_minutes`           | Optional runtime bound.                              |
| `idle_sleep_seconds`            | Delay between iterations.                            |
| `allow_idle_between_iterations` | Lets TUI show daemon-idle without exiting.           |

## `arm`

```yaml

  required_phrase: "ARM JEKKO DAEMON my-run"
  require_hash: true
  require_tui_confirm: true
  require_project_enablement: true
```

Supported fields:

| Field                        | Meaning                                   |
| ---------------------------- | ----------------------------------------- |
| `required_phrase`            | Must appear outside the spec block.       |
| `require_hash`               | Opening sentinel hash must match body.    |
| `require_tui_confirm`        | TUI preview must be confirmed.            |
| `require_project_enablement` | Project config must allow daemon mode.    |
| `allow_cli_arm`              | Permit `--arm` in headless mode.          |
| `nonce`                      | Optional one-time nonce generated by TUI. |

## `exit`

```yaml
exit:
  when:
    all:
      - shell:
          command: "jankurai audit --json"
          expect_json:
            findings: 0
            caps: 0
      - git_clean: true
      - task_queue_empty: true
```

Supported conditions:

| Condition                      | Meaning                                                |
| ------------------------------ | ------------------------------------------------------ |
| `shell`                        | Run command; exit by code, regex, or JSON expectation. |
| `git_clean`                    | Stop only when working tree is clean.                  |
| `task_queue_empty`             | Stop when durable task graph has no pending tasks.     |
| `file_exists` / `file_missing` | Stop based on file presence.                           |
| `json_file`                    | Stop based on JSON value.                              |
| `markdown_no_tasks`            | Stop when markdown checklist has no open items.        |
| `manual`                       | Stop only when user manually exits.                    |
| `all`                          | All nested conditions must pass.                       |
| `any`                          | Any nested condition may pass.                         |
| `not`                          | Invert condition.                                      |
| `stable_for`                   | Condition must be true for N checks.                   |

## `continue_when`

```yaml

    - audit_findings_gt: 0
    - task_queue_not_empty: true
    - git_dirty: true
```

## `workflow`

Sequential workflow:

```yaml

  - inspect:
      commands:
        - "git status --short"
        - "git diff"
  - choose_task:
      from: task_state
      strategy: random_weighted
  - implement: {}
  - verify:
      use: verification.require_before_complete
```

Graph workflow:

```yaml

    inspect:
      next: choose
    choose:
      next: [spawn_workers, implement_single]
    spawn_workers:

    join_workers:

    verify:
```

| Node            | Meaning                        |

| `implement`     | Let model edit/code.           |
| `verify`        | Run validation.                |

| `patches/`                | Worker patches.                  |
| `artifacts/`              | Command outputs, test logs.      |
| `tui.gold_forever_banner` | Visual daemon signal.            |
| `notify`                  | Optional external notifications. |

# 13. How to do this in Jekko today, before native daemon mode

Without code changes, the best approximation is an external supervisor around `jekko run`:

```text
while true:
  run jekko with a prompt
  run jankurai audit
  if 0 findings and 0 caps:
    exit
  run git checkpoint commands
  compact/clear by starting a fresh run
```

This can work because `jekko run` exists for non-interactive usage, custom commands exist for reusable prompts, Jekko supports project instructions via rules/agents, and MCP can be configured. But it is a workaround, not true daemon mode. It cannot fully solve model stop, compaction semantics, permission blocks, TUI gold mode, or robust multi-agent joins inside Jekko. ([Jekko][5])

A “today” setup could be:

```text
.jekko/commands/jankurai-daemon.md
AGENTS.md
JANKURAI_TASKLIST.md
external shell supervisor
```

But I would treat that as a temporary harness only. The proper fix belongs in Jekko’s session runtime.

# 15. Tests I would require

## Parser/gating tests

```text
- daemon block in markdown code fence does not execute
- missing arm phrase does not execute
- bad hash does not execute
- mismatched close ID does not execute
- project daemon disabled does not execute
- valid armed spec executes
```

## Stop resolver tests

```text
- model finish=stop continues if exit condition false
- model finish=stop exits if exit condition true
- max steps continues in daemon
- permission denial pauses or continues according to policy
- transient provider error retries
- user cancellation really stops
```

## Compaction tests

```text
- compact_every_iterations triggers
- compaction does not blindly inject Continue
- daemon continuation survives compaction
- hard clear rehydrates daemon state
- compaction failure falls back to new session if configured
```

## Checkpoint tests

```text
- no staged changes skips commit
- failed tests prevent checkpoint
- protected file changes pause
- commit SHA recorded
- push failure retries or pauses
```

## Multi-agent tests

```text
- workers claim distinct tasks
- leases expire and tasks requeue
- workers cannot edit same locked file
- failed worker patch quarantined
- verified patch merges
- post-merge verification required
```

## Headless/TUI tests

```text
- jekko run --daemon does not exit on daemon idle
- exits when daemon exit criteria pass
- TUI shows gold FOREVER rail
- pause/resume/kill visible and functional
```

# 16. The non-negotiable design rules

1. **Never trust the model to enforce forever.** The runtime enforces forever.
2. **Never let raw pasted XML trigger execution.** Require sentinel, hash, arm phrase, and config enablement.
3. **Never store critical state only in chat context.** Use durable state and hydrate context.
4. **Never conflate compaction with continuation.** Compact is memory management; continuation is policy.
5. **Never let multiple workers freely edit one worktree.** Use worktrees, locks, leases, and a verifier/merger.
6. **Never make commit/push just a prompt instruction.** Use checkpoint hooks with permission policy.
7. **Never hide daemon mode.** Gold TUI rail, iteration count, stop condition, and kill/pause controls should be obvious.
8. **Never let model natural stop be final in daemon mode.** It is just a stop event routed through `DaemonStopResolver`.

# 17. Final recommended product shape

I would ship three user-facing ways to start it.

## CLI

```bash
jekko daemon start jankurai.daemon.yaml --arm
```

or:

```bash
jekko run --daemon --spec jankurai.daemon.yaml --arm
```

## TUI

```text
/daemon jankurai.daemon.yaml
```

The TUI shows a gold preview and asks for the arm phrase.

## Prompt-embedded spec

Allowed only if the project enables it:

```text
<<<JEKKO_DAEMON_SPEC v1 id="..." sha256="...">>>
...
<<<END_JEKKO_DAEMON_SPEC id="...">>>
ARM JEKKO DAEMON ...
```

That gives you the “put it right into the prompt” power, but with enough gates that accidental paste should not run it.

---

## The shortest possible version of the plan

Build **Jekko Daemon Mode** as a first-class runtime:

```text
DaemonSpec
  → DaemonEngine
  → normal Jekko prompt/tool loop
  → DaemonStopResolver intercepts model stop
  → exit/checkpoint/compact/clear/spawn/continue
  → durable task graph + event log
  → gold FOREVER TUI
```

The syntax should be an armed, hashed, schema-validated daemon spec, not XML. The model should see the rendered task and current daemon state, but Jekko should enforce the loop, context clearing, checkpointing, permissions, worker dispatch, joins, and exit criteria. This is the difference between “please keep going forever” and a real forever agent.
