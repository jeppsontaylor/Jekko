## Executive take

I would **not** build this as “the model sees `<FOREVER>` and promises to keep going.” That will always be fragile. The right design is a **host-enforced daemon/workflow runner inside OpenCode** with an optional **deliberate, gated in-prompt RunSpec syntax**. The LLM may say “done,” “continuing,” or “final answer,” but the OpenCode runtime decides whether the daemon continues, compacts, checkpoints, spawns workers, or stops.

The best version is:

> **OpenCode Daemon Mode = durable run controller + strict RunSpec parser + loop interception + context handoff/rotation + worker orchestration + gold TUI status + auditable stop criteria.**

There does **not** appear to be a single universal “standard agent scripting language” today. The ecosystem is converging around adjacent standards: **MCP** for tools/context, **A2A** for agent-to-agent interoperability, **AG-UI** for agent-to-UI event streams, and **ACP** for IDE/editor-to-agent communication. None of those alone gives you a prompt-native “forever loop” language. Workflow frameworks such as LangGraph, Google ADK, AutoGen GraphFlow, Microsoft Agent Framework, Claude Code hooks/subagents, LMQL, Guidance, SGLang, and BAML show the design space, but OpenCode needs its own runtime-level implementation. ([Model Context Protocol][1])

# 1. What exists today in the ecosystem

## 1.1 Protocols are emerging, but not a full “prompt daemon language”

**MCP** is the most relevant existing standard for connecting agents to external tools, prompts, and resources. The official spec describes MCP as a JSON-RPC-based, stateful protocol inspired by LSP, with server features such as resources, prompts, and tools, and client features such as sampling, roots, elicitation, cancellation, progress, logging, and security. That is excellent plumbing for tools and context, but it is not itself a durable workflow language. ([Model Context Protocol][1])

**A2A** is aimed at agent-to-agent interoperability. Google describes it as a protocol for agents across frameworks/vendors to communicate, coordinate, exchange artifacts, and negotiate interaction modalities. It complements MCP rather than replacing it. A2A is valuable for future OpenCode multi-agent collaboration, especially across heterogeneous agents, but it does not define a prompt-embedded “run forever until condition X” syntax. ([Google Developers Blog][2])

**AG-UI** is an event protocol for agent-to-user-interface communication. Its mental model is useful for OpenCode’s TUI: the daemon should emit structured events like `daemon.iteration.started`, `daemon.worker.blocked`, `daemon.stop_condition.failed`, and `daemon.compaction.completed`, rather than relying on text logs. ([docs.ag-ui.com][3])

**ACP**, the Agent Client Protocol, is relevant because OpenCode is a coding agent with editor/TUI/headless surfaces. ACP standardizes communication between coding agents and IDEs/editors, similar to how LSP decouples editors from language servers. It is not a daemon language, but it is a strong hint that OpenCode’s daemon state should be exposed as structured session state, not buried in assistant prose. ([agentclientprotocol.com][4])

## 1.2 Workflow runtimes show the right shape

**LangGraph** is probably the closest conceptual match: it is built for long-running, stateful agents with durable execution, human-in-the-loop support, memory, and debugging/tracing. That is the category OpenCode daemon mode belongs in: not “prompt trick,” but durable orchestration. ([GitHub][5])

**Google ADK’s LoopAgent** explicitly supports deterministic looping over sub-agents and has termination mechanisms. Their docs emphasize that loop agents need explicit termination to avoid unintended infinite loops. For OpenCode, that translates to: “forever” should be possible, but only behind very deliberate gates. ([Adk][6])

**AutoGen GraphFlow** supports sequential, parallel, conditional, and looping behaviors using graph-based teams. Its docs frame loops as graph cycles with safe exit conditions. That maps nicely to OpenCode’s desired worker model: coordinator → workers → validator → merger → checkpoint → stop check → repeat. ([Microsoft GitHub][7])

**Microsoft Agent Framework declarative workflows** are also instructive because they support YAML workflow definitions that get converted into executable graphs. This is exactly the style I would copy for OpenCode’s “in-prompt syntax”: a strict declarative YAML RunSpec, not loose XML tags. ([Microsoft Learn][8])

## 1.3 Claude Code has the most practical coding-agent lessons

Claude Code’s **subagents** are specialized assistants with their own context windows, tools, prompts, and permissions. Anthropic explicitly recommends them for high-volume operations, parallel research, separate context, and agent teams coordinating independent sessions. OpenCode already has agents/subagents, so daemon mode should lean into that instead of trying to make one giant context carry everything forever. ([Claude][9])

Claude Code’s **hooks** are also important. Hooks run at deterministic lifecycle points, including session events, per-turn events, every tool call, and pre/post compaction. Anthropic’s docs also warn that instructions in memory files are not guaranteed; if something must happen at a specific lifecycle point, it should be a hook. That is the key lesson for OpenCode: checkpointing, compaction, stop checks, and worker bookkeeping must be runtime hooks/controllers, not merely instructions in a prompt. ([Claude][10])

Claude’s context compaction guidance matches what OpenCode needs: monitor token usage, summarize older context, clear history, resume from the summary, and continue the task. Anthropic’s API docs further describe compaction as useful for long-running agents, performance, and focus, not only token limits. ([Claude][11])

## 1.4 Prompt/programming DSLs are useful inspiration, not the direct solution

**LMQL** interleaves prompts with Python-like control flow, types, templates, and constraints. It demonstrates that loops and constraints can be expressed in an LLM-specific language. ([LMQL][12])

**Guidance** supports constrained generation and interleaving generation with loops, conditionals, and tool use. Its lesson for OpenCode is that ambiguity must be removed from the runtime boundary: the syntax has to be parsed by the host, not “understood” by the model. ([GitHub][13])

**SGLang** is designed for LM programs with multiple calls, control flow, structured I/O, parallelism, and runtime optimizations. Its parallelism/control-flow ideas are relevant, but OpenCode’s daemon mode lives at the coding-agent/session layer rather than the inference runtime layer. 

**BAML** shows another declarative/typed approach for agents: prompt functions, testing, CI, and generated native functions. The useful lesson is that serious agent workflows need type checking, testing, and repeatability. ([BAML][14])

# 2. OpenCode-specific reality check

OpenCode already has much of the substrate:

OpenCode is an open-source coding agent with a TUI and client/server architecture. Its docs describe agents as specialized assistants with prompts, model selection, and tool access; it has both primary agents and subagents. ([GitHub][15])

OpenCode custom commands can be Markdown files in a `commands/` directory, with frontmatter properties and body templates. That is a natural entry point for `/daemon`, `/run-script`, `/loop`, or `/workers` commands. ([OpenCode][16])

OpenCode MCP support is config-driven: servers live under the `mcp` config key, can be enabled/disabled, and provide external tools/context. The docs warn that too many MCP tools can exceed context limits, which matters a lot for daemon mode and worker spawning. ([OpenCode][17])

OpenCode plugins can hook into events and customize behavior. That suggests a future extensibility point for daemon lifecycle hooks, although the core daemon runner should not depend only on plugins. ([OpenCode][18])

OpenCode’s current `run` command already has headless behavior, agent selection, model selection, file attachment, directory selection, and a `--dangerously-skip-permissions` option. It also auto-rejects tool permissions unless dangerous skip is enabled, and its event loop exits when the target session status becomes idle. ([GitHub][19])

Most importantly, the current prompt loop has a concrete model-stop exit path: when the last assistant message has a finish reason that is not tool calls, there are no pending tool calls, and the assistant is after the last user message, it logs that it is exiting the loop and breaks. That is exactly where daemon mode must intercept “the model says it is done.” ([GitHub][20])

OpenCode status today is basically `idle`, `busy`, or `retry`. `idle` also publishes session-idle events and clears status. Daemon mode needs richer state than that: phase, iteration, stop condition, active workers, checkpoint state, and whether it is waiting, blocked, compacting, or continuing. ([GitHub][21])

OpenCode compaction already exists. It has `isOverflow`, `prune`, `process`, and `create` interfaces, and it can inject a synthetic “Continue if you have next steps…” user message after compaction. That should be reused, but daemon mode needs stronger handoff summaries and deterministic continuation, not just a generic continuation prompt. ([GitHub][22])

## The community is already asking for this

There are OpenCode issues that line up almost exactly with your complaint.

One issue says plugins currently cannot prevent the loop from stopping because the `session.idle` hook fires after the loop has already broken, and `opencode run` may tear down before a plugin can inject a continuation. The proposal in that issue is effectively a `session.stopping` hook before the break. That is very close to what daemon mode needs. ([GitHub][23])

Another issue says OpenCode “won’t run unattended”: on long repetitive tasks, it does some work, announces it is continuing, then stops. ([GitHub][24])

Another issue reports agents stopping unexpectedly during long tasks, often after printing todos, even though the expected behavior is to continue until the task is done, blocked, cancelled, or errored. ([GitHub][25])

Compaction-related stopping is also a real theme. One issue says OpenCode stops after compaction and resumes only when the user says “keep going.” Another reports `opencode run` exiting after compaction when the synthetic continue message is not created. ([GitHub][26])

There are also issues around `/compact` semantics: after compaction, OpenCode needs to know whether there is an active task to continue or whether it should wait for the user. That is exactly why daemon mode needs explicit task state, not vague “continue if needed” language. ([GitHub][27])

# 3. Critical review of the proposed plan

The proposed plan is directionally good, but I would change several important pieces.

## What it gets right

It correctly identifies the main loop-stop problem in `prompt.ts`: the model emits a non-tool-call finish and the runtime exits. That is the core failure mode for “agents always stopping.” ([GitHub][20])

It correctly identifies existing OpenCode components to reuse: compaction, session status, headless `run`, TUI theme/status, agents, MCP, and command parsing.

It correctly recognizes that `opencode run` must not exit just because the session becomes idle when daemon mode is active. Today headless mode breaks on idle status for the target session. ([GitHub][19])

It correctly calls out the need for an obvious TUI state. A “gold forever” banner is not fluff; it is a safety and observability feature.

## What I would not do

I would **not** implement daemon mode as only an outer loop in `run.ts`. That fixes headless mode but leaves the TUI, server sessions, SDK clients, and future integrations inconsistent. The daemon controller belongs in the session/runtime layer.

I would **not** store daemon state casually in a KV blob or only in memory. A forever loop needs durable run state: run ID, iteration, phase, stop-condition history, worker leases, artifacts, commits, compaction summaries, errors, and cancellation state.

I would **not** let checkpoint and exit commands be arbitrary shell strings embedded in an ordinary prompt. That is too easy to paste accidentally and too dangerous for a feature that can commit and push forever.

I would **not** auto-continue after permission denial by default. Permission denials are not “just another stop.” In daemon mode, they should transition to `blocked` or `paused`, with policy-controlled behavior. `--dangerously-skip-permissions` can exist, but it should be visually terrifying in forever mode. Current OpenCode already treats permissions carefully in headless mode. ([GitHub][19])

I would **not** rely on “inject a prompt telling the agent to use MCP parallelism.” That is brittle. The coordinator should have a real worker/session API and a durable queue. MCP can provide tools, but worker orchestration should be host-level.

I would **not** use loose XML tags like `<FOREVER>`. They are too easy to paste accidentally, too hard to validate, and too similar to normal prompt content. Use an exact fenced block plus an explicit command/flag plus a confirmation string.

# 4. Recommended architecture

## 4.1 The core concept

Create a first-class OpenCode runtime feature:

> **DaemonRun**: a durable, host-enforced workflow that repeatedly runs an agent loop, evaluates stop criteria, performs checkpointing, manages context, optionally coordinates workers, and continues until a runtime stop condition is met.

The model can no longer end the daemon by simply saying “final answer.” It can only end an iteration. The daemon controller decides whether to stop the whole run.

## 4.2 State machine

```text
┌─────────────────────┐
│  DaemonRun Created  │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│  Preflight Checks    │
│  config, git, tools  │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│ Iteration Started    │◄──────────────────────────────┐
│ lease work item      │                               │
└──────────┬──────────┘                               │
           │                                          │
           ▼                                          │
┌─────────────────────┐                               │
│ OpenCode Prompt Loop │                               │
│ tools, edits, tests  │                               │
└──────────┬──────────┘                               │
           │ model says stop/final/idle               │
           ▼                                          │
┌─────────────────────┐                               │
│ beforeStop Intercept │                               │
│ daemon owns decision │                               │
└──────────┬──────────┘                               │
           │                                          │
           ▼                                          │
┌─────────────────────┐                               │
│ Verify + Checkpoint  │                               │
│ tests, audit, git    │                               │
└──────────┬──────────┘                               │
           │                                          │
           ▼                                          │
┌─────────────────────┐     pass      ┌───────────────┴──────┐
│ Evaluate Stop Rules  │─────────────►│ Completed / Satisfied │
└──────────┬──────────┘               └──────────────────────┘
           │ fail
           ▼
┌─────────────────────┐
│ Compact or Rotate    │
│ handoff summary      │
└──────────┬──────────┘
           │
           └──────────────► next iteration
```

The crucial thing is the `beforeStop Intercept`. OpenCode’s normal prompt loop can finish, but daemon mode intercepts that transition before the session is treated as truly idle.

## 4.3 The daemon controller owns continuation

The daemon controller should be responsible for:

1. Maintaining the durable run record.
2. Injecting the initial task.
3. Detecting when the inner prompt loop wants to stop.
4. Running post-iteration hooks.
5. Running verification.
6. Running checkpoint rules.
7. Evaluating stop criteria.
8. Triggering compaction or session rotation.
9. Spawning and supervising workers.
10. Updating TUI/headless/SDK status.
11. Continuing the next iteration.

The LLM’s role is to perform work inside an iteration. The runtime’s role is to decide whether the process continues.

# 5. OpenCode implementation plan

## Phase 1 — RunSpec parser and hard safety gates

Add a strict parser for daemon/run scripts.

Likely new files:

```text
packages/opencode/src/session/daemon/runspec.ts
packages/opencode/src/session/daemon/schema.ts
packages/opencode/src/session/daemon/validate.ts
```

The parser should accept only a fenced block with an exact info string, for example:

````markdown
```opencode-run v1
confirm: RUN-FOREVER
mode: daemon
...
```
````

And it should only execute when launched through one of these explicit entry points:

```text
/daemon start
/run-script start
opencode run --daemon --runspec file.yml --confirm RUN-FOREVER
opencode daemon start file.yml --confirm RUN-FOREVER
```

A random pasted block in a normal chat should be treated as inert text unless the user used the explicit command or CLI flag.

Safety validation should reject:

```text
mode: daemon
```

unless all of these are true:

```text
confirm: RUN-FOREVER
lifetime.policy: forever
safety.permissions is explicit
stop_when is present, even if informational
ui.badge: forever
```

Even when the policy is “forever,” the config still needs stop criteria. “Forever until killed” is valid, but it should be represented deliberately:

```yaml
stop_when:
  manual_cancel: true
```

## Phase 2 — Durable daemon state

Likely new files:

```text
packages/opencode/src/session/daemon/events.ts
packages/opencode/src/session/daemon/types.ts
```

Recommended entities:

```text
DaemonRun

- session_id
- status: created | running | paused | blocked | completed | cancelled | failed
- mode: once | loop | daemon
- confirm
- objective_hash
- current_iteration
- current_phase
- created_at
- updated_at
- cancel_requested_at
- last_error
- config_json

DaemonIteration
- id
- run_id
- number

- started_at

- result: continued | stop_satisfied | blocked | failed
- summary

DaemonEvent

```

This should not be just in session messages. A daemon can survive compaction, server restart, TUI reconnect, and headless client teardown only if its state is durable.

## Phase 3 — Modify the prompt loop with a pre-idle interception point

Modify `packages/opencode/src/session/prompt.ts`.

Today, the loop has a break path when the assistant has a non-tool-call finish and no pending tool calls. That is the exact model-stop point daemon mode must intercept. ([GitHub][20])

Do not scatter daemon logic everywhere. Add a clean hook:

```text
DaemonController.beforePromptLoopStop(ctx)
```

Conceptually:

```text
if normal loop wants to break:
  if daemon is not active:
    break
  decision = daemon.beforeStop(...)
  if decision == "continue":
    inject continuation/handoff user message
    continue
  if decision == "pause":
    set daemon blocked/paused
    break
  if decision == "complete":
    set completed
    break
```

This also directly answers the community issue asking for a pre-stop hook that can inject a message and continue before `session.idle` fires. ([GitHub][23])

The daemon should intercept these stop-ish events:

```text
model_stop
model_final_answer
max_steps_iteration_limit
compaction_continue_missing
processor_stop_due_to_transient_error
tool_permission_blocked
structured_output_complete
```

But not all should continue automatically. For example:

```text
permission denied  -> blocked/paused unless policy says skip/continue
fatal model error   -> retry/backoff, then blocked/failed
structured output   -> usually end iteration, not end daemon
max steps reached   -> end iteration if daemon has per-iteration step cap
```

## Phase 4 — Add daemon-aware status

Modify `packages/opencode/src/session/status.ts`.

Current status is essentially `idle | busy | retry`. Daemon mode needs structured state. ([GitHub][21])

Recommended status shape:

```text
type: "daemon"
runID
sessionID
phase:
  created
  preflight
  running
  waiting_for_model
  waiting_for_tool
  verifying
  checkpointing
  evaluating_stop
  compacting
  rotating
  spawning_workers
  merging_workers
  paused
  blocked
  completed
  cancelled
iteration
objective
stopSummary
workers:
  total
  running
  blocked
  completed
lastEvent
unsafePermissions: boolean
```

Important: do **not** make daemon status replace all busy/idle semantics carelessly. Some clients expect `idle`. Better approach:

```text
session.status = busy/idle/retry for compatibility
session.daemon_status = rich daemon state
```

Or add a `daemon` variant while also ensuring old consumers do not interpret daemon-idle as “safe to exit.”

## Phase 5 — Headless CLI support

Modify `packages/opencode/src/cli/cmd/run.ts`.

Today `opencode run` exits when the target session status becomes idle. In daemon mode, that behavior must change. ([GitHub][19])

Add flags:

```text
opencode run --daemon "task..."
opencode run --daemon --runspec daemon.yml --confirm RUN-FOREVER
opencode run --loop --exit-condition "..." --checkpoint "..."
```

Suggested flags:

```text
--daemon
--runspec <file>
--confirm RUN-FOREVER
--exit-condition <command or named condition>
--checkpoint <named policy or command>
--compact-every <n>
--rotate-every <n>
--max-workers <n>
--worker-isolation <same-worktree|git-worktree|branch>
--unsafe-forever-permissions
--dry-run-runspec
```

Headless mode should stream structured daemon events:

```json
{"type":"daemon.iteration.started","iteration":7}
{"type":"daemon.stop_condition.failed","condition":"jankurai audit","summary":"3 findings, 1 cap"}
{"type":"daemon.checkpoint.committed","sha":"..."}
{"type":"daemon.compaction.completed","summary_tokens":1820}
```

## Phase 6 — TUI gold “forever” UX

Add daemon status rendering in the TUI session header/status line.

The user asked for obvious gold, smaller letters at the top, and impossible-to-miss “forever” signaling. I would implement:

```text
╭────────────────────────────────────────────────────────────╮
│  ∞ FOREVER · JANKURAI WORKTREE · iter 007 · workers 3/4     │
│  phase: verifying · stop: jankurai audit == 0 findings/caps │
╰────────────────────────────────────────────────────────────╯
```

Use a gold accent overlay only while daemon mode is active:

```text
primary/accent/borderActive/info = gold/goldenrod family
```

But do not rely on color alone. Include:

```text
∞ FOREVER
iteration number
phase
stop condition summary
worker count
danger badge if unsafe permissions are enabled
```

Also render daemon status in:

```text
session list
bottom status bar
headless log prefix
share/session view
```

OpenCode recently improved v2 session rendering around tool states, compaction summaries, timings, and related session display, so daemon event/status rendering fits that direction. ([GitHub][28])

## Phase 7 — Context clearing: compact, handoff, rotate

Do not think of this as only “clear context.” Think of it as **state-preserving context turnover**.

OpenCode’s existing compaction can already create a summary and continue. ([GitHub][22])

Daemon mode needs three levels:

### Level 1: normal auto-compaction

Use current OpenCode overflow compaction.

### Level 2: forced handoff compaction

After every N iterations, force a daemon-specific summary:

```text
objective
current stop criteria
current iteration
work item selected
files changed
commands run
tests passed/failed
git status
commit sha
open risks
worker results
next recommended action
```

### Level 3: session rotation

After every M iterations, start a fresh OpenCode session with:

```text
original objective
daemon RunSpec
latest handoff summary
ledger path
queue state
stop-condition history
worker state
```

This prevents a single session from becoming spiritually haunted by thousands of tool calls and stale instructions.

Recommended files on disk:

```text
.opencode/daemon/<run_id>/runspec.yml
.opencode/daemon/<run_id>/ledger.jsonl
.opencode/daemon/<run_id>/state.md
.opencode/daemon/<run_id>/queue.json
.opencode/daemon/<run_id>/workers/<worker_id>.md
.opencode/daemon/<run_id>/artifacts/
.opencode/daemon/<run_id>/compactions/
```

The model sees only the compact handoff plus the current objective. The runtime owns the durable truth.

## Phase 8 — Multi-agent coordination

Your “read THIS_TRACKER.md and update it” pattern is common but weak. The better model is:

```text
Coordinator daemon
  ├─ owns durable queue
  ├─ leases work items
  ├─ spawns workers
  ├─ validates worker outputs
  ├─ merges patches
  ├─ runs tests/audits
  ├─ commits/pushes
  └─ updates stop criteria
```

Diagram:

```text
                   ┌────────────────────┐
                   │ Coordinator Daemon  │
                   └─────────┬──────────┘
                             │
                  leases atomic work items
                             │
        ┌────────────────────┼────────────────────┐
        ▼                    ▼                    ▼
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ Worker A     │     │ Worker B     │     │ Worker C     │
│ git worktree │     │ git worktree │     │ git worktree │
└──────┬───────┘     └──────┬───────┘     └──────┬───────┘
       │ reports patch      │ reports patch      │ reports finding
       └────────────┬───────┴────────────┬───────┘
                    ▼                    ▼
              ┌────────────────────────────────┐
              │ Validator / Reviewer / Merger  │
              └──────────────┬─────────────────┘
                             ▼
                  tests → commit → push → loop
```

Worker isolation should be configurable:

```text
same_session_subagent
same_worktree_session
git_worktree_per_worker
branch_per_worker
container_per_worker
remote_agent_via_a2a
```

Default should be:

```text
git_worktree_per_worker
coordinator merges
workers cannot push
workers write artifacts
coordinator validates and commits
```

This avoids multiple agents trampling the same files.

OpenCode’s existing subagent support is a good starting point, and MCP can be used to connect external worker systems. But for robust multi-agent work, the run controller needs a queue, leases, heartbeats, artifacts, and merge policy. ([OpenCode][29])

# 6. The syntax design

## 6.1 Design goals

The syntax must be:

```text
host-parsed, not model-interpreted
deliberately activated
safe against accidental paste
versioned
schema-validated
auditable
durable
compatible with commands, CLI, TUI, and config
expressive enough for loops, workers, context, MCP, git, verification, and stop criteria
```

## 6.2 Recommended format: OpenCode RunSpec v1

Use YAML inside a fenced block:

````markdown
```opencode-run v1
confirm: RUN-FOREVER
mode: daemon
...
```
````

This is better than XML-like tags because:

```text
YAML is easier to validate against a schema.
The opening fence is exact and rare.
The version is explicit.
The confirm string is visible.
It supports nested structures cleanly.
It is inert unless launched through /daemon or --daemon.
```

## 6.3 Activation gates

A RunSpec should execute only if **all** activation gates pass.

Required gates for daemon mode:

````text
Gate 1: explicit command or CLI flag
  /daemon start
  /run-script start
  opencode run --daemon --runspec file.yml

Gate 2: exact fenced block or file type
  ```opencode-run v1

Gate 3: explicit confirmation
  confirm: RUN-FOREVER

Gate 4: mode and lifetime agree
  mode: daemon
  lifetime.policy: forever

Gate 5: safety block present
  safety.permissions: ...

Gate 6: UI forever badge enabled
  ui.badge: forever

Gate 7: parser dry-run succeeds
  schema valid, no unknown dangerous keys
````

If any gate fails, OpenCode should show:

```text
RunSpec detected but NOT executed.
Reason: missing confirm: RUN-FOREVER
Use /daemon start to execute deliberately.
```

That makes accidental pastes harmless.

# 7. Full syntax library

Below is the syntax library I would design for “God mode,” while keeping the runtime safe and deterministic.

## 7.1 Top-level fields

```yaml
api_version: opencode.run/v1
kind: RunSpec
name: string
description: string
mode: plan | once | loop | daemon | dry_run
confirm: RUN-ONCE | RUN-LOOP | RUN-FOREVER
objective: string
instructions: string
labels: [string]
root: path
workspace: path
resume: false | true | run_id
imports: [path]
vars: object
secrets: object
```

Meaning:

```text
api_version
  Version gate for future compatibility.

kind
  Must be RunSpec.

mode
  plan: parse and show plan only.
  once: one controlled execution.
  loop: repeat until stop criteria, but not forever.
  daemon: continue indefinitely until stop criteria/manual cancellation.
  dry_run: validate config without executing.

confirm
  Human-visible confirmation string. Required for loop/daemon.

objective
  The core mission.

instructions
  Additional agent instructions.

root/workspace
  Where commands and edits run.

resume
  Resume a previous durable daemon run.

imports
  Include shared policies, permission profiles, or worker definitions.

vars
  Non-secret variables for templates.

secrets
  References only, not literal secrets.
```

## 7.2 Lifetime and loop control

```yaml
lifetime:
  policy: once | until | count | forever
  max_iterations: number | null
  max_wall_time: duration | null
  sleep_between_iterations: duration
  backoff:
    on_error: exponential | linear | none
    initial: duration
    max: duration
  continue_on:
    model_stop: true
    final_answer: true
    empty_diff: true
    transient_model_error: true
  pause_on:
    permission_denied: true
    test_failure: false
    merge_conflict: true
    repeated_failure_count: 3
```

Meaning:

```text
policy: forever
  Host-enforced daemon. The model cannot end it.

continue_on.model_stop
  Treat model “done” as end-of-iteration, not end-of-run.

pause_on.permission_denied
  Safer default than auto-approval.
```

# 10. Handling the hard problems

## Problem: “The model says it is done”

Solution:

```text
Treat model stop as end-of-iteration only.
Daemon controller checks stop criteria.
If stop criteria fail, inject next iteration message.
```

## Problem: compaction loses critical state

Solution:

```text
Pin the original RunSpec.
Store durable ledger outside the context.
Use daemon-specific handoff schema.
Verify summary contains stop criteria, queue state, git state, and next action.
Rotate sessions periodically.
```

OpenCode compaction issues already show that “continue after compaction” is subtle and can fail if state is ambiguous. ([GitHub][26])

## Problem: permission prompts stop forever mode

Solution:

```text
Explicit permission profile required.
Permission denied transitions to paused/blocked.
Unsafe forever mode requires explicit confirm and gold/red warning.
```

## Problem: multiple agents edit same files

Solution:

```text
Lease work items.
Use git worktree per worker.
Workers produce patches/artifacts.
Coordinator merges.
Conflict pauses or requeues.
```

## Problem: “commit and push forever” is dangerous

Solution:

```text
Protected branch policy.
Diff review required.
No empty commits.
Verification required.
Commit trailers include daemon run ID.
Push policy explicit.
```

## Problem: prompt-injected syntax accidentally executes

Solution:

```text
Exact fenced block.
Explicit command/flag.
confirm: RUN-FOREVER.
Schema validation.
Dry-run preview.
No execution from ordinary pasted text.
```

## Problem: unbounded cost or runaway loops

Solution:

```text
Forever is allowed, but observable.
Budgets can be optional but supported.
Backoff on repeated failures.
Pause on repeated same error.
Gold TUI always visible.
Manual kill always available.
```

# 12. Best implementation order

I would implement in this order:

```text
1. RunSpec schema/parser with deliberate activation gates.
2. Durable daemon store and event model.
3. Prompt-loop before-stop interception.
4. Basic single-agent daemon mode.
5. CLI --daemon and --runspec.
6. Stop criteria evaluator.
7. Forced handoff compaction.
8. Git checkpoint policy.
9. TUI gold forever banner.
10. Worker queue and subagent/session spawning.
11. MCP worker capability integration.
12. Session rotation.
13. Advanced hooks and blackboard UI.
```

The MVP should **not** start with multi-agent spawning. The MVP should first prove the core promise:

```text
The model can stop.
The daemon does not stop.
The runtime evaluates stop criteria.
If stop criteria fail, it continues.
If context grows, it compacts/handoffs.
If checkpoint is configured, it commits safely.
The TUI screams ∞ FOREVER in gold.
```

# 13. The core RFC sentence

The OpenCode RFC should say:

> OpenCode Daemon Mode is a host-enforced, durable run controller. In daemon mode, assistant final messages, provider stop finish reasons, and post-compaction idle states end only the current iteration. The daemon run continues until explicit runtime stop criteria are satisfied, the user cancels it, or a configured safety policy pauses/fails it.

That is the thing that makes this robust. Not `<FOREVER>`. Not “please continue.” Not “don’t stop.” A real controller.

The next best step is to turn this into an OpenCode RFC with the RunSpec schema, state machine, and the minimal single-agent daemon MVP before adding multi-agent orchestration.
