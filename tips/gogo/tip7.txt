## Executive conclusion

The best design is **not** “make the model obey `<FOREVER>` better.” The best design is a native **Jekko Workflow / Daemon runtime** that treats “forever” as an **orchestrator guarantee**, not a prompt suggestion.

The model should do coding, investigation, review, and tool selection. The runtime should own:

```text
looping
stopping
context clearing / compaction
checkpointing
git safety
worker spawning
worker leases
permission policy
TUI state
manual kill / pause / resume
```

That distinction matters because Jekko’s current loop is designed to stop when the assistant finishes normally. In the current `prompt.ts` loop, Jekko exits when the last assistant message has a finish reason that is not `"tool-calls"` and there are no pending tool calls; this is normal interactive behavior, but it is exactly the wrong primitive for “promised forever” mode. ([GitHub][1])

The existing plan you pasted is directionally right, but it is too close to “patch the prompt loop so it doesn’t break.” I would instead implement a **new outer daemon runner** around a refactored single-turn/agent-run core. Normal Jekko sessions should keep their current semantics. Daemon mode should be explicit, persistent, inspectable, gold-colored in the TUI, and impossible to trigger accidentally.

# 1. What the broader ecosystem is converging on

There is **no single accepted standard** today for “in-prompt agent scripting syntax” that reliably gives loops, context clearing, subprocess workers, MCP orchestration, and stop conditions across coding agents.

What is emerging instead is a stack of partially overlapping standards and patterns:

```text
Prompt structure languages
  POML, BAML, XML-ish prompt sections, AGENTS.md

Constrained / programmable prompting languages
  LMQL, Guidance, SGLang-style control, typed prompt functions

Workflow runtimes
  LangGraph, CrewAI Flows, AutoGen termination conditions, OpenAI Agents SDK

Agent/tool protocols
  MCP, A2A, AG-UI, Open Agent Specification

Coding-agent local conventions
  AGENTS.md, commands, subagents, task trackers, worktree isolation, persistent memory
```

The lesson for Jekko: **do not invent only a nicer prompt tag.** Invent a declarative workflow manifest that can be embedded in a prompt, stored in `.jekko/workflows`, launched by CLI, and executed by a runtime.

## 1.1 Prompt markup languages: useful, but insufficient

**POML** is probably the closest emerging “prompt markup” language. Microsoft describes it as a markup language for prompt engineering that provides structure, data integration, styling, templating, and IDE/SDK tooling. ([GitHub][2])

That is valuable, but POML is mostly about **rendering prompts**, not guaranteeing runtime behavior like “loop forever,” “compact every N iterations,” “spawn workers,” or “commit and push after each verified task.”

**BAML** is another important signal. BoundaryML describes it as a language for building agents and moving from prompt engineering toward schema engineering, typed functions, and reliable AI workflows. ([BAML][3])

Again, BAML gives strong structure around prompts and outputs. It does not solve the Jekko-specific problem: an interactive coding agent must continue after the model says “done.”

## 1.2 Programmable prompting languages: powerful, but not the right UX surface

**Guidance** explicitly supports interleaving generation with control structures, including conditionals, loops, tool use, regex/CFG constraints, and structured generation. ([GitHub][4])

**LMQL** describes itself as a programming language for LLMs, combining prompting with types, templates, constraints, and runtime optimization; its language reference frames multi-part reasoning flows as algorithmically controlled programs. ([LMQL][5])

These are excellent prior art, but they are not a drop-in answer for Jekko TUI prompts. They assume a developer-controlled program. Your requirement is different: a user wants to paste or launch a deliberately gated workflow inside Jekko and have the **Jekko runtime** enforce it.

## 1.3 Workflow runtimes: the strongest model for this feature

**LangGraph** is the most relevant architecture pattern: it is a low-level orchestration framework for long-running, stateful agents. ([GitHub][6])

**CrewAI Flows** explicitly model event-driven workflows, state management, branching, loops, and task coordination. ([CrewAI Documentation][7])

**AutoGen** has a mature concept of multi-agent conversations plus explicit termination conditions, which is directly relevant to “do not stop until X.” ([Microsoft GitHub][8])

**OpenAI Agents SDK** also points in the same direction: orchestration, handoffs, guardrails, tracing, and stateful agent runs are runtime concerns rather than mere prompt text. ([OpenAI Developers][9])

The Jekko design should borrow this pattern: **explicit graph/loop state, explicit termination conditions, explicit checkpointing, explicit event stream.**

## 1.4 Agent protocols: MCP is necessary, but not enough

MCP gives agents a standard way to access tools and data. A2A-style protocols give agents a way to communicate with other agents. Recent protocol discussions generally frame MCP as “agent-to-tool” and A2A as “agent-to-agent.” ([Atlan][10])

The Oracle Open Agent Specification is also relevant because it is a declarative, framework-agnostic specification for defining agents and workflows. ([Oracle GitHub Page][11])

GitHub’s agentic workflow direction also matters: it treats agent automation as something that needs sandboxing, permissions, review, and standard workflow control. ([The GitHub Blog][12])

For Jekko, the implication is:

```text
MCP should provide tools.
A2A / message bus should provide agent communication.
Jekko Workflow DSL should provide orchestration.
The model should not be trusted to enforce the loop.
```

# 2. Jekko-specific research findings

I did not find evidence in the official Jekko docs of a native, formal “forever daemon workflow syntax” today. What Jekko does have are the exact primitives needed to build it properly.

## 2.1 Existing Jekko primitives to build on

Jekko already supports custom commands with markdown/frontmatter-style definitions in `.jekko/commands`, and command content is treated as a prompt template. ([Jekko][13])

Jekko already has primary agents, subagents, hidden agents, task permissions, and visual agent colors. ([Jekko][14])

Jekko’s `run` command supports headless execution, while the TUI, desktop app, and local server share the same core agent session architecture. ([Jekko][15])

Jekko plugins can hook into events and extend behavior, which means daemon mode could start as a plugin experiment but should eventually become native because it needs deep control of status, compaction, prompt-loop termination, and session lifecycle. ([Jekko][16])

The codebase confirms these primitives:

* `SessionStatus.Info` currently has only `idle`, `busy`, and `retry`; daemon needs either a new status variant or dedicated daemon events. 
* `SessionRunState` sets the session back to `idle` when the current runner completes; daemon mode cannot let inner idle mean “whole daemon is done.” 
* `jekko run` currently breaks its event loop when it sees `session.status` become `idle`, which is correct today but wrong for daemon mode unless daemon completion has its own event. 
* `Agent.Info` already has `steps?: Finite`; this is a maximum-step brake, not a forever guarantee. 
* Built-in agents include `build`, `plan`, `general`, `explore`, `compaction`, `title`, and `summary`; the `general` subagent is already described as suitable for parallel units of work. 
* The TUI theme system already resolves theme colors and exposes RGBA utilities, so gold daemon UX can be implemented cleanly. 

## 2.2 Existing Jekko ecosystem signals

The official Jekko ecosystem list already includes packages that point in this direction: `@openspoon/subtask2` for orchestration/flow control, `jekko-conductor` for protocol-driven workflows, `jekko-background-agents` for background agents and context persistence, `jekko-scheduler`, `jekko-worktree`, `jekko-workspace`, and `jekko-supermemory`. ([Jekko][17])

There is also a very relevant package called `jekko-agent-hub`. It runs a daemon, starts or connects to an Jekko hub server, discovers sessions, registers agents, routes messages through filesystem-backed JSON message files, and injects messages into Jekko sessions. ([PyPI][18])

That is strong evidence that the community is already feeling the same pain: multiple Jekko sessions need coordination, persistent context, wake-up messages, and a message bus. Native daemon mode should absorb that lesson.

## 2.3 Jekko bug/issue signals that matter

Jekko has had several public issues around compaction, prompt-loop continuation, finish reasons, and accidental infinite continuation. These are not side details; they are the core risk of this feature.

One issue reports that `jekko run` can exit after compaction because the compaction path does not inject the synthetic continuation message expected by the loop. ([GitHub][19])

Another issue reports the opposite problem: an unconditional synthetic “Continue…” after auto-compaction can cause an unwanted infinite loop when the assistant had naturally finished. ([GitHub][20])

There are also issues about compaction config being ignored, large tool outputs causing overflow, and prompt-loop memory/disk growth. ([GitHub][21])

There is also a finish-reason compatibility issue where a provider/AI SDK change caused the prompt loop to stop because finish reason handling did not match expected values. ([GitHub][22])

These issues imply a hard design rule:

> **Daemon mode must not be implemented as a global “always continue after compaction/stop” hack.**
> It must be an explicit, isolated runtime mode with its own lifecycle and stop policy.

# 3. Critique of the proposed plan

The proposed plan is a good first draft, but I would change several things.

## 3.1 Correct: identify the root stop path

The plan correctly identifies that Jekko usually stops because the assistant emits a normal finish reason and no more tool calls are pending. That is the dominant interactive behavior, and it should remain the default. ([GitHub][1])

## 3.2 Incorrect: patching `runLoop()` directly is too dangerous

The plan suggests overriding the normal exit path inside `prompt.ts`. I would not do that as the main design.

Better:

```text
Refactor runLoop into runLoopCore()
  -> returns TerminalReason

Add DaemonRunner
  -> calls runLoopCore repeatedly
  -> owns checkpoint / exit / compaction / restart

Leave normal prompt behavior alone
```

This avoids accidentally turning normal prompts, compaction continuations, or provider finish-reason quirks into runaway loops.

## 3.3 Missing: durable daemon state

The plan says daemon config could be session metadata or KV. That is too weak.

Daemon mode needs durable, queryable state:

```text
run id
spec hash
current iteration
current phase
last terminal reason
last checkpoint
last exit-condition result
worker leases
task ledger
context snapshots
manual pause / stop flags
crash recovery info
```

If Jekko crashes or the TUI closes, the runtime should know whether the daemon is resumable, failed, completed, or intentionally stopped.

## 3.4 Missing: terminal reason taxonomy

Right now, many very different things collapse to “the loop ended.”

Daemon mode needs to distinguish:

```ts
assistant_stop
assistant_max_steps
tool_calls_exhausted
processor_stop
permission_blocked
llm_error
structured_output_done
compaction_requested
compaction_failed
context_overflow
cancelled
runtime_error
```

Only some reasons should cause daemon continuation. Others should checkpoint, pause, escalate, or fail.

## 3.5 Missing: explicit safety boundary for shell/checkpoint commands

Exit conditions and checkpoint commands are shell commands. That is powerful and dangerous.

A daemon that can run forever and push commits must have:

```text
command allowlist / denylist
timeouts
cwd pinning
env redaction
output truncation
non-interactive mode
permission prompts
dangerous-command detection
git branch guard
no-empty-commit behavior
manual kill path
```

Jekko already has a permission system and even a default `doom_loop` permission in the agent defaults, which is highly relevant here. 

## 3.6 Missing: multi-agent worktree isolation

“Spin up multiple MCPs” is probably not the exact primitive. MCP servers expose tools/resources. They are not automatically workers.

For parallel coding, the right primitive is:

```text
Supervisor daemon
  -> task ledger
  -> worker sessions
  -> isolated git worktrees
  -> per-worker MCP tool scopes
  -> merge/review queue
  -> final verifier
```

Without worktree isolation and task leases, multiple agents will clobber the same files, race on the task list, and commit each other’s partial changes.

## 3.7 UI gold overlay should be local and unmistakable

The proposed global theme overlay is okay, but I would prefer a dedicated daemon banner plus local gold border accents. Global theme mutation can make normal UI elements look daemon-related.

The TUI should show:

```text
╔════════════════════════════════════════════════════════════════════╗
║  ∞ FOREVER DAEMON · jankurai-dirty-worktree · iteration 17        ║
║  phase: verify → checkpoint → exit-check · workers: 3 · push:on   ║
╚════════════════════════════════════════════════════════════════════╝
```

Gold should mean one thing: **this process will continue until stopped or exit criteria pass.**

# 4. Best architecture for Jekko

## 4.1 Current flow

```text
User prompt
  ↓
SessionPrompt.prompt()
  ↓
create user message
  ↓
SessionRunState.ensureRunning()
  ↓
prompt.ts runLoop()
  ↓
LLM/tool cycle
  ↓
assistant finish != tool-calls
  ↓
break
  ↓
SessionStatus idle
  ↓
jekko run exits
```

That is correct for normal Jekko.

## 4.2 Proposed daemon flow

```text
User launches explicit workflow
  ↓
Workflow parser validates gated syntax / file / CLI flags
  ↓
DaemonRunner starts durable run
  ↓
Iteration N
  ↓
runLoopCore(sessionID, daemonContext)
  ↓
TerminalReason returned
  ↓
checkpoint policy
  ↓
exit-condition evaluator
  ↓
if satisfied: finish daemon
  ↓
else context policy:
    compact | hard clear | fork | new session | rolling summary
  ↓
inject synthetic daemon restart message
  ↓
Iteration N+1
```

The invariant:

> The model may stop.
> The daemon must not stop unless runtime stop criteria are satisfied.

## 4.3 Multi-agent flow

```text
Daemon supervisor
  ↓
Task ledger / blackboard
  ↓
Lease task A ───────→ Worker session 1 ──→ isolated worktree 1
Lease task B ───────→ Worker session 2 ──→ isolated worktree 2
Lease task C ───────→ Worker session 3 ──→ isolated worktree 3
  ↓
Worker reports
  ↓
Verifier / reviewer agent
  ↓
Merge queue
  ↓
Main worktree checkpoint
  ↓
Exit audit
```

The shared task tracker should not be “everyone edit `THIS_TRACKER.md`.” It should be an atomic ledger with leases.

Daemon loop pseudocode:

```ts
while (!abort.signal.aborted) {
  await setDaemonStatus({ phase: "prompt", iteration })

  const result = await runLoopCore({
    sessionID,
    daemonRunID,
    injectedContext,
  })

  await recordTerminalReason(result.terminalReason)

  if (!policy.shouldContinue(result.terminalReason)) {
    await pauseOrFail(result.terminalReason)
    break
  }

  await setDaemonStatus({ phase: "checkpoint" })
  await checkpointPolicy.run()

  await setDaemonStatus({ phase: "exit-check" })
  const exit = await evaluateExitConditions()

  if (exit.satisfied) {
    await finishDaemon(exit)
    break
  }

  await setDaemonStatus({ phase: "context" })
  await contextPolicy.prepareNextIteration()

  await injectDaemonRestartMessage({
    originalPrompt,
    iteration,
    lastTerminalReason,
    lastExitResult,
    ledgerSummary,
  })

  iteration++
}
```

# 6. The syntax I would implement

I would not use plain XML tags like:

```xml
<FOREVER>
...
</FOREVER>
```

Those are too easy to paste accidentally, too easy for the model to reproduce, and too ambiguous.

Use a deliberately ugly, unmistakable, top-level gate.

## 6.1 The gate

A workflow block is executable only if all are true:

```text
1. It starts at the first non-whitespace character of the human prompt.
2. It uses the exact JEKKO_WORKFLOW sentinel.
3. It has a matching end sentinel with the same id.
4. It includes confirm: RUN_WORKFLOW or confirm: RUN_FOREVER.
5. It is launched through /workflow, /daemon, or jekko run --workflow-block.
6. It is not inside a markdown code fence, quote, pasted transcript, or assistant message.
7. The parsed YAML validates against the WorkflowSpec schema.
8. Dangerous capabilities require explicit permission policy.
```

Canonical block:

```yaml
<<<JEKKO_WORKFLOW:odsl/v1
```

## 6.2 Inert by default

If a user pastes this into a normal chat prompt without `/workflow run` or `/daemon`, Jekko should not run it. It should render:

```text
Workflow block detected but not armed.
Run with /workflow run, /daemon, or jekko run --workflow to execute.
```

For the TUI, the safest UX is:

```text
Detected executable workflow: jankurai-dirty-worktree
Mode: FOREVER daemon
Risk: high
Will run until: jankurai audit => 0 findings, 0 caps
Will run checkpoint: git add/commit/push
Start? [y/N]
```

TUI:

```text
/daemon .jekko/workflows/jankurai.ocw.yaml
```

```text
/workflow run jankurai-dirty-worktree
```

CLI:

```text
jekko run --workflow .jekko/workflows/jankurai.ocw.yaml
```

CLI shorthand:

```text
jekko run \
  --daemon \
  --forever-confirm RUN_FOREVER \
  --exit-condition "jankurai audit --json | jq -e '.findings == 0 and .caps == 0'" \
  --checkpoint "git add -A && git commit -m 'daemon checkpoint' && git push" \
  --compact-every 1 \
```

The shorthand should internally compile to the same WorkflowSpec.

# 7. Full syntax library

Below is the “God mode” library I would support over time. The key is that these are **runtime directives**, not natural-language suggestions.

## 7.1 Top-level fields

| Field     | Purpose                                                                 |
| --------- | ----------------------------------------------------------------------- |
| `version` | Workflow DSL version.                                                   |
| `id`      | Stable workflow identifier.                                             |
| `name`    | Human-readable display name.                                            |
| `mode`    | `once`, `loop`, `daemon`, `supervisor`, `pipeline`.                     |
| `confirm` | Required arming phrase: `RUN_WORKFLOW`, `RUN_DAEMON`, or `RUN_FOREVER`. |
| `risk`    | `low`, `medium`, `high`, `extreme`; drives permission UX.               |
| `agent`   | Primary Jekko agent to use.                                          |
| `model`   | Model override.                                                         |
| `variant` | Reasoning/model variant override.                                       |
| `prompt`  | Main model task prompt.                                                 |
| `vars`    | User-defined variables.                                                 |
| `env`     | Environment variables allowed for runtime commands.                     |
| `labels`  | Metadata labels for UI/filtering.                                       |

## 7.2 Loop controls

| Field                 | Function                                                      |
| --------------------- | ------------------------------------------------------------- |
| `loop.forever`        | If true, no iteration limit. Requires `confirm: RUN_FOREVER`. |
| `loop.until`          | Runtime stop condition.                                       |
| `loop.while`          | Runtime continue condition.                                   |
| `loop.max_iterations` | Safety bound for non-forever loops.                           |
| `loop.min_iterations` | Prevents premature exit.                                      |
| `loop.sleep`          | Delay between iterations.                                     |
| `loop.backoff`        | Backoff after failures.                                       |
| `loop.jitter`         | Adds randomized delay.                                        |
| `loop.watchdog`       | Detects no progress.                                          |
| `loop.heartbeat`      | Emits status every N seconds.                                 |
| `loop.continue_on`    | Terminal reasons that continue.                               |
| `loop.pause_on`       | Terminal reasons that pause.                                  |
| `loop.fail_on`        | Terminal reasons that fail the run.                           |

## 7.3 Exit conditions

| Condition                   | Function                                                              |
| --------------------------- | --------------------------------------------------------------------- |
| `exit_when.shell`           | Run shell command; exit code 0 means true unless otherwise specified. |

# 9. How to do this in Jekko today, before native support

Today, the most reliable way is an **external supervisor loop** around `jekko run`, not a prompt tag.

Use Jekko’s current strengths:

```text
custom commands
headless jekko run
agents/subagents
plugins
git worktrees
external shell loop
external audit command
```

A practical today architecture:

```text
shell supervisor
  ↓
jekko run --continue --command jankurai-daemon-step
  ↓
Jekko does one bounded unit of work
  ↓
shell runs jankurai audit
  ↓
shell moves completed tasks
  ↓
shell commits/pushes if dirty and verified
  ↓
shell compacts or starts new session
  ↓
repeat until audit clean
```

This is not as elegant as native daemon mode, but it gives the “forever” guarantee to the shell/runtime instead of trusting the model.

For multi-agent today, I would combine:

```text
jekko-background-agents / jekko-agent-hub style coordination
jekko-worktree or manual git worktrees
custom Jekko subagents
a shared ledger file or SQLite db
external merge/review script
```

The `jekko-agent-hub` package is particularly relevant because it already uses a daemon to discover Jekko sessions and inject messages, though it is outside core Jekko and has known UI visibility limitations. ([PyPI][23])

# 10. Major pitfalls and solutions

## Pitfall: accidental infinite loops

Solution:

```text
No plain <FOREVER>.
Require JEKKO_WORKFLOW sentinel.
Require RUN_FOREVER confirmation.
Require TUI preview.
Never parse assistant output as workflow control.
```

## Pitfall: compaction causes unwanted continuation

Solution:

```text
Daemon continuation is controlled only by DaemonRunner.
Normal compaction remains normal.
No global synthetic “continue” after every compaction.
```

This directly addresses the class of issues where compaction either stops too early or continues forever accidentally. ([GitHub][19])

## Pitfall: model says “done” even though audit fails

Solution:

```text
Model completion is not daemon completion.
Only runtime exit_when can end daemon.
```

## Pitfall: workers edit same files

Solution:

```text
task leases
file locks
git worktree isolation
merge queue
verifier
```

## Pitfall: daemon commits broken code forever

Solution:

```text
checkpoint requires verification
git commit only if tests/audit pass for task
push is permission-gated
branch guard
no-empty-commit
```

## Pitfall: shell exit condition is flaky

Solution:

```text
structured JSON checks
timeouts
retry policy
captured evidence
show last exit result in TUI
```

## Pitfall: context poisoning

Solution:

```text
workflow directives are parsed only from human-origin launch input
model text cannot create/modify workflow control
runtime context is synthetic and signed/marked
```

## Pitfall: cost runaway

Solution:

```text
forever requires explicit confirmation
always-visible gold banner
manual kill shortcut
optional budget warnings
heartbeats
```

I would still allow true unlimited mode, but it should be visibly armed.

# 12. Final recommendation

Build this as **Jekko Workflow Daemon Mode**, not as prompt magic.

The shortest correct design is:

```text
1. Add a deliberately gated workflow DSL.
2. Parse it outside the model.
3. Store a durable daemon run.
4. Refactor prompt loop to return terminal reasons.
5. Add an outer daemon runner that repeats prompt runs.
6. Evaluate stop conditions in runtime, not by asking the model.
7. Add explicit context clearing strategies.
8. Add gold TUI daemon status.
9. Add task ledger + worker isolation for multi-agent work.
10. Keep normal Jekko behavior unchanged.
```

The most important sentence for the implementation spec:

> **The model is allowed to stop. The daemon is not allowed to stop unless runtime stop criteria pass, the user kills it, or a configured safety policy pauses/fails it.**
