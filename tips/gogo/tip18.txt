# Verdict

Your instinct is right: **this should not be implemented as “please obey these XML tags forever.”** That will never be bulletproof. The only reliable design is:

**A host-parsed OpenCode workflow language + a runtime supervisor.**

The prompt can *contain* the workflow, but OpenCode—not the model—must parse it, validate it, display it, store it, run it, compact context, restart iterations, spawn workers, evaluate exits, and ignore model-level “I’m done” unless the workflow’s stop criteria are satisfied.

The core rule should be:

> In daemon mode, an assistant `finish_reason: stop` is not terminal. It is only a `STOP_CANDIDATE`. The OpenCode runtime decides whether to stop, continue, compact, spawn workers, checkpoint, or re-prompt.

That is the difference between “model, please keep going” and **actually promised forever behavior**.

---

# 1. What I found in OpenCode specifically

OpenCode already has many of the primitives needed, but not the runtime contract you want.

OpenCode supports custom agents with system prompts and permissions, custom commands with markdown frontmatter or JSON config, MCP servers, plugins, project/global AGENTS.md rules, and a permissions system including bash/edit/webfetch/MCP/tool-level permissions. Those are the right building blocks, but none of them currently form a durable workflow VM with formal loops, stop hooks, task leases, context epochs, or multi-agent coordination state. ([OpenCode][1])

The important current OpenCode failure mode is real. The prompt loop checks for a final assistant message whose finish is not `tool-calls`, with no pending tool calls, and then exits the loop. The same loop also exits on compaction failure, processor `"stop"`, structured output completion, and max-step behavior. That means the model’s normal “stop/end turn” can end an autonomous run unless the host explicitly converts that into a continuation decision. ([GitHub][2])

OpenCode’s compaction is already strong enough to build on. Its compaction flow summarizes the conversation, publishes compaction events, and has an auto-continue path that can inject a synthetic continuation message such as “Continue if you have next steps…”. But community reports show that this behavior is fragile: in one issue, `opencode run` exits after compaction because the continuation message is not created in the right path; in another, auto-continuation can create an unwanted loop when the assistant had naturally finished. ([GitHub][3])

There is also an OpenCode issue asking for exactly the missing abstraction: a hook before the session loop stops. The proposed `session.stopping` hook would let a plugin inspect the stop reason and inject a new message or prevent stopping, explicitly noting the risk of infinite loops if used unconditionally. That is almost the same conceptual seam needed for daemon mode, except I would implement it as a first-class runtime feature rather than only a plugin trick. ([GitHub][4])

OpenCode community discussion also points directly at multi-agent coordination as a gap. One proposal describes DB-backed agent team coordination, task state, SQLite WAL, bus events, background execution, cross-session message injection, team management, a task board, and TUI panels. It also states that the current TaskTool execution is synchronous and does not provide true background parallel agent coordination. ([GitHub][5])

There are OpenCode issues showing why a naive “always continue” patch would be dangerous. Users have reported planner/TODO loops, repeated continuations, and agents getting stuck because termination/progress checks were not explicit. These are not arguments against forever mode; they are arguments for **formal daemon semantics, progress detectors, exit predicates, visible UX, and deliberate activation gates.** ([GitHub][6])

There is no evidence that OpenCode currently has a formal, supported in-prompt daemon scripting language. The closest OpenCode-native pieces are custom commands, agents, plugin hooks, MCP configuration, permissions, and compaction. The missing thing is a workflow supervisor that owns continuation and state. ([OpenCode][7])

# 2. What the broader ecosystem is converging on

There is **not** a universal standard for “in-prompt agent scripting syntax” today. Instead, the ecosystem has split into several layers:

| Layer                         | What exists                                  | What it solves                                                               | What it does **not** solve                         |
| ----------------------------- | -------------------------------------------- | ---------------------------------------------------------------------------- | -------------------------------------------------- |
| Tool/context protocol         | MCP                                          | Tools, resources, prompts, sampling, roots, elicitation                      | Local agent loop semantics                         |
| Agent-to-agent protocol       | A2A                                          | Agent discovery, task exchange, multimodal collaboration                     | Local coding-agent workflow language               |
| Workflow runtimes             | LangGraph, CrewAI Flows, AutoGen             | Durable state, loops, handoffs, multi-agent orchestration                    | Inline prompt syntax for OpenCode                  |
| Prompt/programming DSLs       | LMQL, Guidance, SGLang, Outlines, DSPy       | Constrained generation, control flow, structured output, prompt optimization | Coding-agent daemon control over git/files/MCP/TUI |
| Coding-agent harness features | Claude Code hooks, slash commands, subagents | Stop hooks, compaction commands, custom commands, separate subagent contexts | OpenCode-native implementation                     |

MCP is the most important standard around tools. The current MCP spec defines a JSON-RPC based protocol with tools, resources, prompts, roots, sampling, elicitation, and explicit trust/safety guidance. It is a protocol for connecting models/clients to capabilities, not a workflow loop language. ([Model Context Protocol][8])

A2A is the most important emerging standard around agent-to-agent coordination. Google announced A2A in April 2025, later moved it into the Linux Foundation ecosystem, and the spec describes agents discovering capabilities, negotiating interaction modes, managing collaborative tasks, and exchanging information without sharing internal memory or tools. That is highly relevant to your multi-agent direction, but again it is not an inline OpenCode prompt DSL. ([Google Developers Blog][9])

LangGraph is probably the strongest design reference for “real loops that do not lose state.” It is explicitly a framework/runtime for long-running, stateful agents with durable execution, checkpointing, streaming, memory, and human-in-the-loop support. Its big lesson for OpenCode is: **agent loops must be runtime state machines, not just prompts.** ([LangChain Docs][10])

CrewAI Flows are also relevant because they offer event-driven workflows with state, conditional logic, loops, routing, and persistence. The lesson there is similar: the control plane is outside the model, and the model is one actor inside a durable workflow. ([CrewAI Documentation][11])

Claude Code is the closest coding-agent analog. Claude Code has hooks where `Stop` and `SubagentStop` can block stoppage and feed a reason back to the model, custom slash commands with frontmatter, MCP prompts surfaced as slash commands, `/compact`, `/mcp`, and subagents with separate context windows. This is a very useful benchmark for OpenCode: the right pattern is not “trust XML in the prompt,” but **host-level hooks + commands + subagent contexts + compaction controls.** ([Claude][12])

LMQL, Guidance, SGLang, Outlines, and DSPy are useful syntax inspirations, but they are not a direct answer for OpenCode. LMQL is a programming language for LLMs with constraints and loops; Guidance supports interleaving generation with loops, conditionals, tools, regex/CFG constraints; SGLang supports chained generation, control flow, parallelism, and external interactions; DSPy argues for programming language-model systems instead of brittle prompt strings. Their shared lesson: **structured programs beat freeform prompt conventions.** ([LMQL][13])

The community pulse across GitHub, Reddit, LinkedIn, and framework docs is consistent: people are moving away from “one giant clever prompt” toward **harnesses** with durable state, hooks, task boards, traces, guardrails, tool protocols, and multi-agent orchestration. Some community posts say this bluntly: the harness, not only the model, is the real product surface. Treat Reddit and LinkedIn as lower-authority signal, but the pattern aligns with the official framework docs. ([Reddit][14])

# 3. Critique of the proposed AI plan

The prior plan has several good instincts:

It correctly identifies the model-level stop path as the main reason agents stop. It correctly notices compaction can be reused. It correctly proposes a daemon status, CLI flags, TUI gold UX, and an outer loop around the normal model/tool loop. Those are directionally right.

But I would not implement it exactly as written.

## Major problem 1: it treats “daemon” as mostly a prompt-loop patch

Patching `prompt.ts` to override `finish: stop` is necessary, but not sufficient. You need a **workflow runtime** around `SessionPrompt`, not just a special case inside `runLoop`.

The daemon has to own:

* iteration state,
* exit checks,
* checkpoint policy,
* compaction epochs,
* worker task queue,
* MCP profile,
* permissions,
* progress detection,
* event log,
* pause/resume/kill,
* TUI status,
* headless supervision.

Otherwise you will create another brittle auto-continue path like the current compaction problems. ([GitHub][15])

## Major problem 2: automatic checkpoint shell commands are dangerous

The plan proposes a checkpoint command like:

```bash
git add -A && git commit -m checkpoint && git push
```

That should not be the primitive. The primitive should be a **verified checkpoint action** with policy:

```yaml
git:
  checkpoint:
    when: verified
    include: touched_by_iteration
    require_clean_tests: true
    commit_message: generated_from_task
    push: after_commit
```

The runtime may use shell under the hood, but the workflow language should not encourage arbitrary always-run shell blobs between iterations. OpenCode already has a permission model for bash/edit/MCP/tool execution; daemon mode should respect and extend it, not bypass it. ([OpenCode][16])

## Major problem 3: XML-style tags are prompt-injection bait

`<FOREVER>`, `<GIT>`, `<EXIT_IF>`, and `<CLEAR_AGENT_CONTEXT>` are readable, but they are also easy to accidentally paste, quote, nest, or spoof. They look like instructions to the model, not commands to OpenCode.

The workflow block must be:

* parsed by OpenCode before model invocation,
* schema-validated,
* inert unless deliberately activated,
* shown in TUI/CLI preview,
* stored as a workflow run,
* never interpreted merely because the model saw it.

## Major problem 4: “same session or new session?” is the wrong binary

The right answer is:

> same **workflow run**, multiple **session epochs**.

A workflow run should live in durable state. Each iteration can use the same session with compaction until it becomes unhealthy, then fork a new child/epoch session with a handoff capsule. That gives you continuity without depending on a single ever-growing context.

## Major problem 5: multi-agent/MCP is conflated

MCP servers are not agents. They are tool/resource/prompt servers. Robustly “spinning up multiple MCPs” really means two different things:

1. enabling/disabling MCP servers and tools per phase, and
2. spawning multiple OpenCode worker agents/sessions that may use selected MCP tools.

OpenCode docs explicitly warn MCP tools add to context, which can become an issue when many servers/tools are enabled. So daemon mode should use **MCP profiles and lazy tool exposure**, not “turn on everything forever.” ([OpenCode][17])

## Major problem 6: it underestimates the implementation

This is not six files and 400–600 lines if done properly. A real implementation touches:

* workflow schema/parser,
* workflow runtime,
* session prompt loop,
* compaction,
* status/events,
* CLI run command,
* TUI,
* permissions,
* MCP profile handling,
* task/subagent orchestration,
* storage,
* plugin hooks,
* tests.

A minimal MVP can be small. A bulletproof daemon language cannot.

# 4. Recommended architecture

I would build this as:

# **OpenCode Workflow Language + Daemon Runtime**

For naming, use something boring and explicit:

* file extension: `.ocw.yaml`
* internal feature: `workflow`
* mode name: `daemon`
* TUI label: `∞ FOREVER`

Do **not** call the language “prompt syntax” internally. Call it a workflow. It may be embedded in a prompt, but the execution semantics belong to OpenCode.

---

## Core design

```text
User input
   │
   ▼
Workflow Gate Detector
   │
   ├── no valid gate ─────► normal OpenCode prompt
   │
   └── valid gate
          │
          ▼
   Workflow Parser + Schema Validator
          │
          ▼
   Preview / Approval Gate
          │
          ▼
   Durable Workflow Run
          │
          ▼
   Daemon Runtime Supervisor
          │
          ├── calls normal SessionPrompt loop
          ├── intercepts stop candidates
          ├── runs exit checks
          ├── compacts / forks epochs
          ├── checkpoints git
          ├── coordinates workers
          ├── manages MCP profiles
          └── publishes TUI/headless status
```

The runtime state machine:

```text
START
  │
  ▼
PARSE_WORKFLOW
  │
  ▼
REQUIRE_APPROVAL
  │
  ▼
INITIALIZE_RUN
  │
  ▼
ITERATION_START
  │
  ▼
INJECT_RUNTIME_PROMPT
  │
  ▼
MODEL_TOOL_LOOP
  │
  ▼
STOP_CANDIDATE ───────┐
  │                   │ model says stop/end_turn
  ▼                   │
EVALUATE_PROGRESS     │
  │                   │
  ▼                   │
RUN_CHECKPOINTS       │
  │                   │
  ▼                   │
EVALUATE_EXIT
  │
  ├── exit true ─────► FINALIZE_RUN
  │
  └── exit false
          │
          ▼
   COMPACT_OR_FORK_CONTEXT
          │
          ▼
   RECONCILE_WORKERS
          │
          ▼
   NEXT_ITERATION
```

The hard contract:

```text
Normal OpenCode:
  assistant finish=stop -> session stops

Daemon OpenCode:
  assistant finish=stop -> workflow runtime receives STOP_CANDIDATE
  workflow runtime decides:
    continue | compact | checkpoint | spawn | pause | exit
```

# 5. Implementation plan in OpenCode

## Phase 0 — Add the missing seam: stop candidate hook

Before implementing the full language, add a first-class stop decision seam.

### New concept

```ts
type LoopStopReason =
  | "assistant_finished"
  | "processor_stop"
  | "compaction_stop"
  | "max_steps"
  | "structured_output"
  | "cancelled"
  | "error"

type LoopStopDecision =
  | { type: "stop" }
  | { type: "continue"; message?: string; synthetic?: boolean }
  | { type: "compact_then_continue"; message?: string }
  | { type: "pause"; reason: string }
```

### Modify `packages/opencode/src/session/prompt.ts`

Do not bury daemon logic throughout `runLoop`. Instead, make stop paths call a centralized decision point:

```text
current behavior:
  if exit condition in runLoop:
    break

new behavior:
  decision = yield* stopController.decide(...)
  if decision.stop:
    break
  if decision.continue:
    inject synthetic user message
    continue
  if decision.compact_then_continue:
    compact
    inject continuation
    continue
```

In normal mode, the controller always returns `{ type: "stop" }`, preserving existing behavior.

In daemon/workflow mode, it returns continue/compact/pause/exit based on workflow state.

This also aligns with the OpenCode community request for a pre-stop hook, but makes it safer by putting policy in a typed runtime, not arbitrary unconditional plugin logic. ([GitHub][4])

## Phase 3 — Daemon runtime supervisor

### New files

```text
packages/opencode/src/workflow/runtime.ts
packages/opencode/src/workflow/daemon.ts
packages/opencode/src/workflow/evaluator.ts
packages/opencode/src/workflow/checkpoint.ts
packages/opencode/src/workflow/progress.ts
packages/opencode/src/workflow/context.ts
```

### Runtime responsibilities

The daemon runtime should own the outer loop:

```text
while workflow.status === "running":
  publish status: daemon/workflow
  prepare iteration prompt
  run SessionPrompt.loop/prompt
  receive stop candidate
  collect iteration evidence
  run progress detector
  run checkpoint policy
  evaluate exit predicates
  compact/fork context if needed
  reconcile workers
  inject next iteration
```

### Important distinction

`SessionPrompt` should continue to own the **inner model/tool loop**.

`WorkflowRuntime` should own the **outer forever loop**.

That separation prevents daemon mode from contaminating normal interactive behavior.

## Phase 7 — Context clearing / compaction

### Modify

```text
packages/opencode/src/session/compaction.ts
packages/opencode/src/workflow/context.ts
```

OpenCode compaction already summarizes goals, instructions, discoveries, accomplishments, files, and current work, and can inject a continuation message. The daemon system should not replace this. It should add a **workflow handoff capsule** that survives compaction and session epoch changes. ([GitHub][3])

### Context policies

```yaml
context:
  strategy: same_session_with_epochs
  compact:
    every_iterations: 1
    before_token_ratio: 0.70
    after_tool_output_bytes: 200000
    on_worker_join: true
  preserve:
    - workflow.goal
    - workflow.rules
    - exit.conditions
    - task.queue
    - completed.tasks
    - open.blockers
    - dirty.files
    - test.failures
    - git.commits
    - worker.summaries
  drop:
    - raw_tool_outputs_after_summarized
    - duplicate_diffs
    - stale_search_results
```

### Handoff capsule

Every compaction/new epoch should produce a deterministic object:

```yaml
handoff:
  workflow_run_id: wf_...
  iteration: 18
  epoch: 3
  immutable_goal: ...
  current_objective: ...
  exit_conditions:
    - jankurai audit has 0 findings
    - jankurai audit has 0 caps
  completed_since_start:
    - ...
  completed_this_epoch:
    - ...
  active_tasks:
    - ...
  blockers:
    - ...
  dirty_files:
    - ...
  last_known_test_status:
    command: ...
    result: ...
  worker_summaries:
    - worker: verifier-2
      result: ...
  next_recommended_action:
    - ...
```

This is far stronger than `<CLEAR_AGENT_CONTEXT>` because the host decides what must survive.

## Phase 8 — Multi-agent collaboration

The current “read THIS_TRACKER.md and update it” method should become a real task/state layer.

### Add workflow-native team primitives

```text
packages/opencode/src/workflow/team.ts
packages/opencode/src/workflow/task-board.ts
packages/opencode/src/workflow/worker.ts
packages/opencode/src/workflow/artifact.ts
```

### Design

```text
Lead daemon session
  │
  ├── task queue in DB
  ├── artifact store
  ├── event bus
  ├── worker leases
  ├── worker summaries
  └── merge/checkpoint policy

Worker sessions
  ├── scout/read-only workers
  ├── implementer workers
  ├── verifier workers
  ├── reviewer workers
  └── janitor/checkpoint workers
```

### Rule

Workers should not coordinate by sharing one mutable markdown tracker. They should coordinate by:

* claiming tasks with leases,
* writing structured task outputs,
* publishing events,
* attaching artifacts,
* returning handoff summaries,
* optionally writing human-readable mirrors to markdown.

### Worker isolation

For coding tasks, parallel workers need isolation:

```yaml
team:
  isolation:
    strategy: git_worktree_per_worker
    branch_prefix: "wf/jankurai/"
    merge_policy: lead_applies_patches
```

Without separate worktrees or patch isolation, multiple workers touching a dirty worktree will corrupt each other.

### Recommended roles

```yaml
agents:
  lead:
    role: orchestrator
    can_edit: true
  scout:
    role: read_only_code_investigator
    can_edit: false
  implementer:
    role: focused_patch_author
    can_edit: true
  reviewer:
    role: regression_detector
    can_edit: false
  verifier:
    role: test_runner
    can_edit: false
  janitor:
    role: git_checkpoint_and_tasklist_cleanup
    can_edit: true
```

This aligns with A2A’s general model of independent agents exchanging task-level information and with LangGraph/CrewAI’s durable orchestration approach, but implemented natively in OpenCode. ([a2aproject.github.io][18])

## Phase 9 — MCP handling

MCP should be a first-class workflow capability, but not confused with agent spawning.

### Add MCP profiles

```yaml
mcp:
  profile: jankurai
  servers:
    filesystem:
      enabled: true
    github:
      enabled: true
    test_runner:
      enabled: true
  expose:
    mode: phase_scoped
    max_tools_per_iteration: 20
  healthcheck:
    on_start: true
    restart_on_failure: true
    backoff: exponential
```

OpenCode already supports MCP servers in config and documents that many MCP tools add context, which can create context-size issues. A daemon runtime should therefore expose MCP tools phase-by-phase instead of dumping every tool into every model call forever. ([OpenCode][17])

### Add run-level MCP overrides

The OpenCode community has already requested CLI support to enable/disable MCP servers for `opencode run`. That belongs naturally in daemon/workflow mode. ([GitHub][19])

## Phase 10 — Permissions and safety

Daemon mode must not mean “permission bypass mode.”

Add workflow-specific permissions:

```yaml
permissions:
  bash:
    allow:
      - "git status*"
      - "git diff*"
      - "git add*"
      - "git commit*"
      - "git push*"
      - "jankurai audit*"
      - "bun test*"
      - "pnpm test*"
    deny:
      - "rm -rf *"
      - "git push --force*"
      - "sudo *"
  edit:
    allow:
      - "**/*.ts"
      - "**/*.tsx"
      - "JANKURAI_TASKLIST.md"
      - "JANKURAI_TASKLIST.md.completed"
    deny:
      - ".env*"
      - "**/secrets/**"
  webfetch: ask
  mcp:
    github: ask
```

OpenCode already supports permissions for bash, edit, webfetch, tools, and MCP tools, including wildcards and command-level bash patterns. Daemon mode should compile workflow permissions into the existing permission model. ([OpenCode][16])

### Permission denial behavior

In daemon mode:

```text
permission denied + other tasks available:
  mark current task blocked
  continue with another task

permission denied + no other tasks:
  pause workflow visibly
  do not fake progress
  do not bypass permission

permission denied for dangerous action:
  record guardrail event
  require human intervention
```

This preserves “forever” without silently doing unsafe things.

# 6. The deliberate workflow gate

This is the most important syntax decision.

Do **not** use bare XML tags.

Use a rare, host-parsed sentinel that is inert unless activated.

## Recommended gate

```yaml
<<<OPENCODE_WORKFLOW:BEGIN v=1 mode=daemon intent=preview>>>
name: jankurai-daemon
activation:
  require: manual
  command: "/workflow approve jankurai-daemon"
goal: |
  Investigate dirty worktree items, fix regressions, verify, commit, push,
  and repeat until jankurai audit reports 0 findings and 0 caps.
<<<OPENCODE_WORKFLOW:END>>>
```

Default `intent=preview` means: parse, validate, show the workflow, but do not run.

To execute:

```text
/workflow approve jankurai-daemon
```

For headless:

```text
opencode run --workflow .opencode/workflows/jankurai.ocw.yaml
```

## Stronger gate for inline execution

For truly dangerous inline daemon execution:

```yaml
<<<OPENCODE_WORKFLOW:BEGIN v=1 mode=daemon intent=execute nonce="8f8d-jankurai-2026-05-07">>>
activation:
  require:
    - explicit_command
    - nonce_match
  approve_with: "/workflow approve 8f8d-jankurai-2026-05-07"
...
<<<OPENCODE_WORKFLOW:END>>>
```

OpenCode should reject execution unless the user types the approval command with the nonce.

## Parser rules

The parser should only recognize a workflow block when:

1. the begin sentinel appears at the start of the user message, except whitespace;
2. the end sentinel is present;
3. version is supported;
4. body parses as YAML or JSON;
5. schema validation passes;
6. activation is satisfied;
7. TUI/CLI shows a preview before running, unless using a trusted local file with `--workflow`.

If someone pastes an article, issue, README, or adversarial text containing a workflow block, OpenCode should not run it automatically.

# 7. Syntax design principles

The language should be declarative where possible.

Bad:

```yaml
do: "git add -A && git commit && git push"
```

Better:

```yaml
git:
  checkpoint:
    when: verified
    commit: true
    push: true
```

Bad:

```xml
<CLEAR_AGENT_CONTEXT>
```

Better:

```yaml
context:
  compact:
    every_iterations: 1
    strategy: daemon_handoff
```

Bad:

```xml
<EXIST_IF>
If audit says 0, exit.
</EXIST_IF>
```

Better:

```yaml
exit:
  all:
    - command:
        run: "jankurai audit --json"
        expect:
          jsonpath: "$.findings"
          equals: 0
    - command:
        run: "jankurai audit --json"
        expect:
          jsonpath: "$.caps"
          equals: 0
```

# 11. MVP vs full version

## MVP: “real forever mode”

Implement first:

1. workflow parser for gated YAML;
2. daemon runtime single-agent loop;
3. stop candidate override;
4. exit command evaluation;
5. forced compaction with handoff;
6. CLI `--daemon` / `--workflow`;
7. TUI gold badge;
8. basic git checkpoint policy.

That gives you the promised core:

```text
model stops -> runtime checks exit -> not satisfied -> compact/re-prompt -> continue
```

## V2: “team daemon”

Add:

1. DB task board;
2. background worker sessions;
3. worker leases;
4. git worktree isolation;
5. worker artifacts;
6. team panels;
7. consensus/reviewer/verifier workflows.

## V3: “agent operating system”

Add:

1. A2A bridge;
2. remote workers;
3. workflow marketplace;
4. signed workflows;
5. reusable workflow libraries;
6. workflow debugger;
7. visual graph editor.

# 12. The one thing I would not compromise on

The syntax must be **host-executed**, not model-obeyed.

The model can receive a rendered instruction like:

```text
You are running workflow jankurai-daemon, iteration 18.
Your current leased task is X.
You must produce evidence Y.
```

But the model should not be trusted to decide whether:

* the daemon keeps running,
* context gets cleared,
* workers spawn,
* git pushes happen,
* exit criteria are satisfied,
* the workflow block is valid.

That belongs to OpenCode.

# 13. Final recommended product shape

The final product should feel like this:

```text
opencode run --workflow .opencode/workflows/jankurai.ocw.yaml
```

or in the TUI:

```text
/workflow new
[paste workflow]
/workflow approve jankurai-forever-2026-05-07
```

Then the TUI clearly shows:

```text
∞ FOREVER · JANKURAI DAEMON
iteration 42 · epoch 7 · workers 4 · exit: audit clean
last progress: committed jankurai: fix capability audit regression
next: random dirty task
```

And internally:

```text
assistant says stop
  ↓
OpenCode says: exit criteria not met
  ↓
checkpoint if verified
  ↓
compact or fork context
  ↓
spawn/reconcile workers
  ↓
inject next iteration
  ↓
continue
```

That is the architecture that makes “forever” real.
