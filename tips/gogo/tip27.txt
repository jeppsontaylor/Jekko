## Executive verdict

The right design is **not** “teach the model to obey `<FOREVER>`.” The right design is a **deterministic OpenCode control plane** that lets the user declare a loop in a deliberately armed syntax, then OpenCode itself owns the loop, state, checkpointing, compaction, worker orchestration, exit evaluation, retries, and UI warning state.

The model should be responsible for **doing one bounded unit of agentic work**. OpenCode should be responsible for **making sure the bounded unit runs again forever until a real exit predicate passes**.

That distinction is the whole game.

There is no mature universal standard today for “in-prompt infinite agent scripting for coding agents.” What exists is a fast-moving bundle of adjacent standards and patterns:

1. **AGENTS.md / CLAUDE.md / rules files** for persistent repo instructions.
2. **Skills / slash commands / workflows** for reusable prompt procedures.
3. **MCP** for tools, resources, and prompt templates.
4. **A2A / ACP** for external agent-to-agent communication.
5. **Hooks** for deterministic lifecycle enforcement.
6. **Prompt programming languages** like Guidance, LMQL, PDL, SGLang, and APPL for control flow, constraints, tool use, and structured generation.
7. **Compaction / memory / context protocols** as an emerging research area, with ACDL being especially relevant.

Your requested feature should combine those into an OpenCode-native “daemon script” layer. I’d call it **OpenCode Control Plane**, or **OCCTL**, with a user-facing daemon mode named **FOREVER**.

# 1. What I found in the ecosystem

## 1.1 There is no single dominant in-prompt agent scripting standard yet

The closest cross-tool instruction standard is **AGENTS.md**, which describes itself as “a simple, open format for guiding coding agents” and a predictable place for agent-specific context such as setup commands, tests, and conventions. It is useful for persistent repo rules, but it is not a control-flow language and does not solve infinite loops, worker orchestration, checkpointing, or context clearing by itself. ([Agents][1])

The strongest practical pattern is: **put stable instructions in repo files, put reusable procedures in skills/workflows, put external capabilities behind MCP, put lifecycle decisions in hooks, and put long-running loop control in host code**.

That last part is critical. Any “forever” guarantee implemented only as prompt text will eventually fail because the model can decide it is done, forget a constraint after compaction, hit provider output limits, or get derailed by tool errors.

## 1.2 OpenCode already has important pieces, but not the daemon control plane

OpenCode already supports specialized agents with custom prompts, models, tools, and permissions; it also has primary agents, subagents, and an “explore” style subagent that can be used for parallel research or scoped work. ([OpenCode][2]) OpenCode’s docs also identify compaction as a hidden agent that automatically summarizes context when needed. ([OpenCode][2])

OpenCode has MCP support for local and remote servers, and MCP tools are made available alongside built-in tools. The docs also warn that MCP servers can add substantial context/token overhead, even enough to exceed a model’s context window. ([OpenCode][3]) OpenCode also supports custom commands via configuration and Markdown command files. ([OpenCode][4])

So the building blocks are there: **agents, subagents, commands, MCP, compaction, status, TUI, headless run mode, client/server architecture**. The missing thing is a **durable supervisor** that owns the loop and state.

## 1.3 OpenCode community signals match your frustration exactly

There are OpenCode issues asking for things that are basically pieces of your request:

A user asked for hooks to run scripts after a task ends, including automatically committing, pushing, and opening a PR, because asking the AI to do this in prompt text wastes tokens and is inconsistent. That is exactly the argument for deterministic host-side checkpoint hooks. ([GitHub][5])

A custom slash command feature request explicitly compared OpenCode to Claude Code’s command system and proposed reusable Markdown workflow commands with dynamic arguments and shell context injection. ([GitHub][6])

A delayed queue issue describes repeatedly submitting the same request in a loop, capturing learnings, and wanting queued prompts to wait until OpenCode is idle instead of interrupting current work. ([GitHub][7])

There are also several compaction and loop-related issues that matter directly here. Users report OpenCode stopping after compaction, processing stopping every time after compaction, automatic “continue” causing infinite loops, and compaction losing AGENTS.md/CLAUDE.md instruction context. ([GitHub][8]) Another issue says specialized agents can lose their role identity after compaction because the compaction summarizer lacks the source agent’s system prompt and role constraints. ([GitHub][9])

The key lesson from those issues is beautiful and brutal: **OpenCode needs explicit task-state semantics.** “Continue after compaction” is correct only when the agent was actively working. It is wrong when the assistant had naturally finished and was waiting for the user. A daemon mode must therefore carry an explicit state like `daemon.active=true`, not infer intention from the last message.

## 1.4 Claude Code’s newer hooks/skills model is the most useful nearby reference

Claude Code skills are especially relevant. Their docs describe skills as `SKILL.md` files that can be invoked directly, loaded only when used, and used for repeated procedures instead of constantly pasting the same instructions into chat. Claude Code also says its skills follow the Agent Skills open standard and includes bundled skills such as `/loop`. ([Claude API Docs][10])

Claude skills also support invocation controls. For side-effect workflows like deploy or commit, `disable-model-invocation: true` prevents the model from invoking the skill on its own. That is exactly the kind of safety separation OpenCode needs for daemon scripts. ([Claude API Docs][10])

Claude’s hook model is also highly relevant. Hooks fire at lifecycle events and can receive JSON context, block actions, add context, or influence decisions. ([Claude API Docs][11]) The particularly important reference is the `Stop` hook: exit code 2 can prevent the assistant from stopping and continue the conversation, while `PreToolUse` can allow, deny, ask, defer, or modify tool input. ([Claude API Docs][11]) This is almost exactly the behavior OpenCode daemon mode needs, but OpenCode should make it first-class rather than relying on compatibility glue.

## 1.5 Cline workflows show the “Markdown procedure” pattern, but not hard forever guarantees

Cline workflows are Markdown files invoked with slash commands for repetitive or complex tasks; their docs show workflows handling release preparation: version bump, tests, changelog, commit, tag, push. They explicitly allow high-level natural-language steps or precise XML tool syntax/exact commands. ([Cline Documentation][12])

That is useful inspiration for the user syntax, but not sufficient for your requirement. Workflows guide an agent. They do not guarantee a forever loop unless the host application controls the loop.

## 1.6 MCP is for capabilities; A2A/ACP are for agent interoperability

MCP gives tools/resources/prompts to agents. The MCP specification says servers can expose resources, prompts, and tools; tools are executable functions, prompts are reusable workflow templates, and resources are contextual data. ([Model Context Protocol][13])

For multiple agents communicating as peers, A2A and ACP are the more relevant standards. Google’s agent protocol guide frames the current landscape as a set of protocols including MCP, A2A, UCP, AP2, A2UI, and AG-UI, with MCP addressing integration with tools/data and A2A-style protocols addressing agent coordination. ([Google Developers Blog][14]) ACP describes itself as an open protocol for agent interoperability, supporting synchronous and asynchronous communication, streaming, stateful/stateless patterns, discovery, and long-running tasks. ([Agent Communication Protocol][15]) Linux Foundation materials describe A2A as an open protocol for secure agent-to-agent communication and collaboration, and later describe it as an open standard for agents to discover, communicate, and transact across frameworks and vendors. ([Linux Foundation][16])

For OpenCode, I would not make MCP itself the multi-agent coordination layer. I would use:

* **MCP** for tools and external capability servers.
* **OpenCode subagents/workers** for local coding/review/test roles.
* **A2A/ACP compatibility later** for external agent processes.
* **An OpenCode daemon ledger** as the local source of truth.

## 1.7 Prompt programming languages point to the same conclusion

Guidance supports structured generation with constraints and interleaves conditionals, loops, tool use, and generation. ([GitHub][17]) LMQL is a Python-superset language for LLM programming that interweaves conventional code with LLM calls. ([GitHub][18]) IBM’s Prompt Declaration Language is YAML-based, supports model chaining and tool use, and includes type checking for model input/output. ([IBM][19]) SGLang is a structured generation language/runtime designed for complex generation programs involving loops and conditionals. ([NVIDIA Docs][20]) APPL emphasizes async parallelization, tool-calling integration, tracing, failure recovery, and auto-continuation when output limits are hit. ([GitHub][21])

These are not OpenCode-native standards, but they strongly support the architectural principle: **control flow belongs in a real interpreter/runtime, not only in prose.**

## 1.8 ACDL is the most interesting emerging “context syntax” idea

ACDL, a 2026 paper, proposes a language for describing and communicating context structure in agentic LLM systems. It is designed to make context layout clearer for teams, research comparisons, prompt logic documentation, and potentially human-to-AI communication about desired context layout. ([arXiv][22])

That is directly relevant to your “context clearing / compression / reload exact syntax” requirement. I would borrow the idea of explicit context layers from ACDL, but implement the operational semantics inside OpenCode.

# 2. Critique of the proposed AI plan

The plan you received is directionally right, but it is too shallow for a truly bulletproof forever mode.

## What it got right

It correctly identifies that a model can emit a normal stop/end-turn finish and OpenCode will treat that as done. For daemon mode, that must not be terminal unless the daemon’s exit predicates pass.

It correctly suggests an outer loop around the existing prompt loop. That is the right shape. The model should finish a bounded turn; the daemon supervisor should decide whether to run another turn.

It correctly points at compaction, session status, `opencode run`, and the TUI as the right integration surfaces.

It correctly proposes a visual gold indicator. Long-running autonomous agent mode needs an unmistakable UI because it can spend money, modify files, push code, and run indefinitely.

## What is missing or risky

The plan treats the exit condition mostly as a shell command. That is not enough. A bulletproof daemon needs typed predicates: shell, git clean/dirty, test result, audit JSON, queue empty, no-progress, max cost, max wall time, human stop, worker consensus, and ledger state.

It suggests overriding permission denials and continuing. That is dangerous. Permission denial is a safety signal. In daemon mode, denial should go to a policy handler: retry with narrower action, ask user, pause daemon, or mark work item blocked. It should not silently continue forever.

It treats compaction as the state store. That is backwards. Compaction is a **projection** of state into the model context. The real daemon state must live in a persisted ledger that survives compaction, process crashes, and session resumes.

It does not solve multi-agent collisions. Multiple workers touching the same dirty worktree will conflict unless each work item has a lease, file locks, branch/worktree isolation, merge policy, and a central queue.

It does not distinguish “assistant stopped because it is done” from “assistant stopped because output length/rate limit/error/compaction/tool denial.” Daemon mode needs a typed stop reason classifier.

It does not define a safe deliberate syntax. XML-ish tags like `<FOREVER>` are too easy to paste accidentally, too easy for the model to reproduce, and too ambiguous. Executable control syntax should require an explicit command or config path plus a nonce/ack.

It does not provide a crash-resumable controller. A true daemon must recover after terminal close, OpenCode server restart, rate-limit pause, MCP crash, or compaction failure.

It does not address cost/budget observability. Even a “forever” loop should show cost, iteration count, tokens, tool calls, commits, failed attempts, and stop controls.

# 3. The architecture I would build in OpenCode

## 3.1 Core principle

OpenCode needs a **Daemon Supervisor** above the existing agent loop.

Current-ish conceptual flow:

```text
user prompt
  -> prompt loop
      -> model
      -> tools
      -> model
      -> tools
      -> model says stop
  -> session idle
```

Daemon flow:

```text
user explicitly starts daemon
  -> parse + validate daemon spec
  -> persist daemon state + ledger
  -> while daemon.active:
       choose/refresh work item
       run one bounded OpenCode agent turn/session loop
       classify why it stopped
       harvest results
       run verification/checkpoint hooks
       evaluate exit predicates
       compact or roll context if needed
       schedule/reap workers
       either exit/pause or inject next bounded task
```

The model may say “I’m done.” That only means **this bounded inner turn is done**. It does not decide whether the daemon is done.

## 3.2 New modules

I would add a new directory:

```text
packages/opencode/src/session/daemon/
```

With these files:

```text
spec.ts
  Types and schemas:
  DaemonSpec
  DaemonState
  DaemonPhase
  DaemonExitPredicate
  DaemonCheckpointPolicy
  DaemonWorkerSpec
  DaemonContextPolicy
  DaemonPermissionPolicy

parser.ts
  Parses OCCTL fenced blocks and daemon spec files.
  Rejects accidental/freeform prompt usage.
  Validates nonce, confirm phrase, schema version, and unknown keys.

controller.ts
  The deterministic state machine.
  Owns loop iteration, phase transitions, retries, pause/resume/stop, and resume after crash.

ledger.ts
  Durable event log and state store.
  Stores iterations, tool outcomes, work items, leases, commits, test results, audit snapshots, compaction summaries.

queue.ts
  Work item queue.
  Supports discovery, prioritization, random selection, leases, retries, dependency graph, lock scopes.

evaluator.ts
  Evaluates exit predicates and quality gates.
  Shell, git, file, JSON/JQ, regex, test, audit, queue, worker, cost, time predicates.

checkpoint.ts
  Git add/commit/push, branch/worktree strategy, rollback markers.
  Enforces tests-before-push policies.

context.ts
  Builds the synthetic daemon context injected into the model each iteration.
  Separates hot/warm/cold context and restores invariants after compaction.

workers.ts
  Spawns/reaps subagents, external opencode workers, MCP-backed workers, or future A2A workers.
  Manages worker leases and result merging.

watchdog.ts
  Detects no-progress loops, repeated tool failures, MCP failures, rate limits, duplicate diffs, and stuck workers.

events.ts
  Bus event helpers:
  daemon.started
  daemon.iteration.started
  daemon.iteration.completed
  daemon.checkpoint.started
  daemon.exit.evaluated
  daemon.worker.spawned
  daemon.paused
  daemon.stopped
```

## 3.3 Modified modules

### `packages/opencode/src/session/prompt.ts`

Do not hack the existing loop into becoming a daemon. Add a daemon entrypoint that wraps it.

Desired interface shape:

```ts
SessionPrompt.prompt(input)
SessionPrompt.loop(input)
SessionPrompt.daemon(input)
```

`daemon(input)` should call the existing prompt/loop machinery for one bounded agent turn, then return control to `DaemonController`.

The inner prompt loop should expose a structured result:

```ts
type AgentTurnResult =
  | { type: "assistant_stop"; finish: "stop" | "end_turn" | "unknown" }
  | { type: "tool_exhausted" }
  | { type: "output_length" }
  | { type: "context_overflow" }
  | { type: "permission_denied"; permission: PermissionRef }
  | { type: "model_error"; error: ProviderError }
  | { type: "structured_output" }
  | { type: "cancelled" }
```

That is better than making daemon mode inspect message history and guess.

### `packages/opencode/src/session/status.ts`

Add a daemon-aware status. I would not replace `busy`; I would add mode and phase.

```ts
type SessionStatus =
  | { type: "idle" }
  | { type: "busy"; ... }
  | { type: "retry"; ... }
  | {
      type: "daemon"
      active: boolean
      phase:
        | "starting"
        | "discovering_work"
        | "agent_running"
        | "verifying"
        | "checkpointing"
        | "evaluating_exit"
        | "compacting"
        | "spawning_workers"
        | "paused"
        | "exiting"
      specID: string
      iteration: number
      workers: { active: number; max: number }
      exitSummary: string
      lastProgressAt: number
      stopHint: string
    }
```

### `packages/opencode/src/session/compaction.ts`

Daemon mode must support **forced compaction** and **state-preserving compaction**.

The compaction summary should receive the daemon context anchor:

```text
- daemon mission
- current exit predicates
- current work item
- current leases/locks
- last known audit/test status
- uncommitted changes summary
- completed commits
- active worker summaries
- safety/permission invariants
- next required action
```

But the real state remains in the ledger. The summary is only what the model needs now.

Also: auto-continue after compaction must be controlled by explicit task state. The OpenCode issues show both failure modes: sometimes compaction stops work that should continue, and sometimes compaction continues when it should wait. The daemon controller solves that by making `daemon.active` an explicit state, not an inferred behavior. ([GitHub][23])

### `packages/opencode/src/cli/cmd/run.ts`

Add headless daemon support:

```bash
opencode run --daemon-spec .opencode/daemon/jankurai.yaml
opencode run --daemon --exit '...' --checkpoint '...' 'mission...'
```

Headless mode must not exit when the inner agent goes idle if `daemon.active=true`. It should exit only when:

```text
exit predicates pass
manual stop received
fatal policy says stop
max budget/cost/time hit, if configured
```

### Server API

Expose daemon operations over the client/server API:

```text
POST /api/session/:id/daemon/start
POST /api/session/:id/daemon/pause
POST /api/session/:id/daemon/resume
POST /api/session/:id/daemon/stop
GET  /api/session/:id/daemon/status
GET  /api/session/:id/daemon/ledger
POST /api/session/:id/daemon/exit/evaluate
POST /api/session/:id/daemon/queue/update
```

OpenCode’s own repo description emphasizes a client/server architecture where the TUI is only one possible client, so daemon mode should be server-native rather than TUI-only. ([GitHub][24])

### TUI files

Wherever the session header/status pill is rendered, add daemon styling. The exact file should be confirmed in the repo during implementation, but the likely area is under:

```text
packages/opencode/src/cli/cmd/tui/
```

The TUI should not just tint the theme. It should show an unmistakable persistent banner:

```text
∞ FOREVER
DAEMON ACTIVE · iter 42 · workers 3/6 · phase verifying · exit: jankurai_clean=false
Stop: Ctrl-C twice / /daemon stop     Pause: /daemon pause
```

Use a gold overlay on active border, primary accent, progress indicator, and banner label. Do not permanently mutate the user’s theme.

# 5. Context model: hot, warm, cold

A forever loop must not rely on a giant conversation history. It should use layered context.

## Hot context

Injected every iteration:

```text
- exact current work item
- exact current task instructions
- relevant files/diff/audit output
- required verification commands
- allowed tools
- what to do before stopping
```

## Warm context

Injected after compaction or every N iterations:

```text
- mission summary
- exit criteria
- current plan
- current task queue summary
- completed commits
- known blockers
- worker summaries
- safety invariants
```

## Cold context

Never fully injected unless needed:

```text
.opencode/daemon/ledger.sqlite
.opencode/daemon/events.jsonl
.opencode/daemon/work-items.jsonl
.opencode/daemon/audit-history/
.opencode/daemon/worker-results/
.opencode/daemon/context-snapshots/
```

This avoids the “tracker markdown as shared brain” problem. Markdown can be a human-readable mirror, but the source of truth should be a durable structured ledger.

# 6. Multi-agent collaboration design

The orchestrator should not just tell random agents to “read THIS_TRACKER.md.” It should manage a real queue.

## 6.1 Work item object

```yaml
id: dirty-diff-017
title: Fix regression in audit cap parser
source: git_dirty
status: leased
priority: 80
lease:
  owner: worker-reviewer-2
  expires_at: 2026-05-07T18:45:00Z
lock_scope:
  files:
    - packages/opencode/src/session/**
    - packages/opencode/src/config/**
dependencies:
  - daemon-parser-schema
attempts: 2
evidence:
  - git diff -- packages/opencode/src/session
  - jankurai audit --json
definition_of_done:
  - tests pass
  - audit finding resolved
  - commit pushed
```

## 6.2 Worker isolation options

Use one of these strategies:

| Strategy                  |                               Use when | Risk                         |
| ------------------------- | -------------------------------------: | ---------------------------- |
| Same worktree, file locks | Small number of workers, low collision | Conflicts if locks are wrong |
| Git worktree per worker   |       Best default for parallel coding | Merge complexity             |
| Branch per worker         |              Good for external workers | Needs rebase/merge policy    |
| Read-only subagents       |                  Review/research/audit | Cannot directly fix          |
| MCP external tools        |               Specialized capabilities | Token/tool bloat             |

For your dirty-worktree cleanup case, I’d use:

```text
orchestrator in main worktree
review/test/audit workers read-only
fix workers in per-worker git worktrees
orchestrator merges verified patches/commits
```

## 6.3 Collaboration primitives

The daemon ledger needs these primitives:

```text
claim(work_item_id, worker_id, lock_scope, ttl)
heartbeat(worker_id, status)
post_finding(worker_id, work_item_id, finding)
post_patch(worker_id, work_item_id, branch_or_diff)
post_verification(worker_id, command, result)
release(work_item_id)
complete(work_item_id, commit_sha)
block(work_item_id, reason)
```

## 6.4 Consensus flows

For high-risk tasks:

```yaml
review:
  required:
    - role: fixer
      count: 1
    - role: reviewer
      count: 1
    - role: tester
      count: 1
  merge_policy: all_required_pass
```

For random dirty worktree cleanup:

```yaml
review:
  required:
    - role: fixer
      count: 1
    - role: auditor
      count: 1
  merge_policy: tests_and_audit_pass
```

# 7. The syntax: OCCTL / OpenCode Control Plane

Your XML tags are the right instinct but the wrong execution surface. A bulletproof syntax needs to be:

1. **Impossible to trigger by accidental paste.**
2. **Parsed by OpenCode before the model sees it.**
3. **Schema validated.**
4. **Unknown-key rejecting.**
5. **Versioned.**
6. **Explicitly armed by the user.**
7. **Not executable in normal prompts.**
8. **Not triggerable by model output.**

## 7.1 Required gate

A daemon block is executable only when one of these is true:

```text
/daemon run
opencode run --daemon-spec file.yaml
opencode daemon run file.yaml
TUI command palette: “Start Daemon”
```

A raw pasted block in normal chat is treated as text.

## 7.2 Required fence

Use a deliberately ugly but clear fence:

````text
/daemon run
```opencode-daemon v1
occtl: v1
kind: daemon
id: jankurai-cleanup
confirm: FOREVER_UNTIL_EXIT_OR_MANUAL_STOP
...
````

````

The parser rejects:

```text
<FOREVER>
FOREVER:
mode: daemon
```yaml
kind: daemon
````

````

The fence language must be exactly:

```text
opencode-daemon v1
````

## 7.3 Required confirmation phrase

For true infinite loop:

```yaml
confirm: FOREVER_UNTIL_EXIT_OR_MANUAL_STOP
```

For side effects:

```yaml
side_effects_ack:
  git_commit: true
  git_push: true
  shell_commands: true
```

For dangerous permissions:

```yaml
danger_ack: I_UNDERSTAND_THIS_CAN_RUN_INDEFINITELY_AND_CHANGE_MY_REPO
```

That string should be required only for high-permission runs. TUI can also show a one-time modal.

# 8. Syntax library

Below is the OCCTL syntax library I would support.

## 8.1 Top-level structure

```yaml
occtl: v1
kind: daemon
id: string
description: string
confirm: FOREVER_UNTIL_EXIT_OR_MANUAL_STOP

mission: string | block

scope:
  root: "."
  include: []
  exclude: []

loop: {}
work: {}
agents: {}
workers: {}
mcp: {}
context: {}
verification: {}
checkpoint: {}
exit: {}
permissions: {}
safety: {}
observability: {}
hooks: {}
```

Unknown top-level keys should fail validation by default.

## 8.2 `loop`

```yaml
loop:
  strategy: forever | until | while | for_each | schedule | watch
  on_model_stop: continue | evaluate_exit | stop | ask
  on_output_length: continue | compact_then_continue | stop
  on_context_overflow: compact_then_continue | pause | stop
  on_permission_denied: ask | pause | mark_blocked | retry_narrower | stop
  on_model_error: retry | backoff | switch_model | pause | stop
  on_no_progress: rotate_work | compact | spawn_reviewer | pause | stop

  min_iterations: 0
  max_iterations: null
  delay_between_iterations: "0s"
  compact_every_iterations: 3
  compact_at_token_ratio: 0.70

  retry:
    max_attempts_per_item: 3
    backoff: exponential
    initial_delay: "5s"
    max_delay: "5m"

  no_progress:
    enabled: true
    window_iterations: 3
    duplicate_diff_threshold: 0.95
    repeated_command_threshold: 5
    action: spawn_reviewer_then_rotate
```

Important: `strategy: forever` does not mean never exit. It means “run indefinitely until exit predicates pass or user/manual/fatal stop occurs.”

## 8.3 `mission`

```yaml
mission: |
  Investigate uncommitted worktree changes related to JANKURAI_STANDARD.md.
  Pick one dirty worktree item at a time.
  Review git diff, identify regressions, fix, verify, commit, and push.
```

## 8.4 `scope`

```yaml
scope:
  root: "."
  include:
    - "packages/**"
    - "agent/JANKURAI_STANDARD.md"
    - "JANKURAI_TASKLIST.md"
  exclude:
    - "node_modules/**"
    - ".git/**"
    - "dist/**"
    - ".opencode/daemon/**"
```

## 8.5 `work`


# 11. “Never accidentally pasted” rules

I would make these hard rules:

1. A daemon spec is inert unless preceded by `/daemon run`, `opencode daemon run`, or `opencode run --daemon-spec`.
2. Model output containing OCCTL is never executed.
3. OCCTL in a normal user prompt is rendered as text.
4. `kind: daemon` plus `strategy: forever` requires the exact confirmation phrase.
5. Git push, external network calls, destructive shell, and force-push require separate acknowledgements.
6. Unknown keys fail closed.
7. XML-like tags are not executable.
8. A daemon spec cannot modify itself unless `allow_self_modification: true`, and even then it should require user confirmation.
9. Workers cannot start new daemons.
10. Subagents cannot issue `/daemon` commands; only the user, TUI, CLI, or daemon controller can.

That is how you prevent “oops I pasted a prompt from a README and it started a forever process.”

# 12. Better “Stop” semantics

OpenCode daemon mode needs at least these stop classifications:

```text
assistant_natural_stop
assistant_asked_question
output_truncated
tool_calls_complete
tool_failure
permission_denied
model_error_transient
model_error_fatal
context_overflow
compaction_failed
structured_output_complete
manual_cancel
```

Then daemon policy maps stop reason to action:

```yaml
on_stop_reason:
  assistant_natural_stop: evaluate_exit
  assistant_asked_question: continue_with_daemon_instruction
  output_truncated: continue
  context_overflow: compact_then_continue
  permission_denied: pause
  model_error_transient: retry
  model_error_fatal: pause
  compaction_failed: roll_context_or_pause
  manual_cancel: stop
```

This is much stronger than “if finish != tool-calls, continue.”

# 16. The central design sentence

**OpenCode daemon mode should be a durable, deterministic supervisor that repeatedly launches bounded agent work, preserves state outside the model context, deliberately compacts context into a stable anchor, coordinates workers with leases, checkpoints verified progress, and exits only when typed predicates pass.**

That will give you the thing you actually want: not “the model promised to continue,” but **OpenCode itself guarantees the loop continues**.
