## Bottom line

The previous “Daemon Mode” proposal is pointed in the right direction, but it is too model-centric. The truly reliable version should **not** depend on the LLM obeying `<FOREVER>` or emitting `<promise>DONE</promise>`. The syntax can live inside the prompt as the user interface, but OpenCode must parse it **before** the model sees it, compile it into a deterministic runtime plan, and have the runtime enforce looping, context clearing, worker spawning, checkpointing, and exit conditions.

The winning design is:

> **A deliberate, gated OpenCode Orchestration block → parsed by OpenCode → compiled into a DaemonRun → executed by an outer runtime harness around the existing agent loop.**

That gives you the thing you actually want: agents that can stop “mentally” all day long, but the **runtime** keeps restarting them until deterministic stopping criteria pass, the user pauses/stops it, or a configured safety policy fires.

# 1. What I found in the ecosystem

There is no single universal “standard agent scripting language” that has clearly won. There are, however, strong emerging patterns.

The dominant trend is **not** “better XML tags in prompts.” It is:

1. A structured orchestration layer outside the model.
2. Durable state outside the context window.
3. Deterministic loop/exit/checkpoint semantics.
4. LLMs used for reasoning and implementation, not for deciding whether the whole harness keeps running.
5. Context compression, handoff summaries, and subagents treated as first-class infrastructure.

Anthropic’s context-engineering writing is especially aligned with this. They describe long-running agents as systems where the loop constantly generates new data, so context must be actively refined through compaction, structured note-taking, and multi-agent decomposition. They also describe subagents as focused workers that return condensed summaries to a lead agent, and emphasize durable session/event logs outside the model’s context window. ([Anthropic][1])

Frameworks like LangGraph, OpenAI Agents SDK, AutoGen, DSPy, LMQL, Guidance, POML, and Arazzo all point toward the same conclusion: **natural language alone is too ambiguous for reliable orchestration**. LangGraph is explicitly for long-running, stateful agents; OpenAI’s Agents SDK has orchestration, handoffs, guardrails, state, and tracing; AutoGen popularized conversing multi-agent systems; LMQL and Guidance demonstrate that loops, constraints, and control flow belong in a programming/runtime layer; POML shows structured prompt markup; Arazzo shows how workflow dependencies and success/failure actions can be represented declaratively. ([GitHub][2])

For OpenCode specifically, the closest community artifact is **Ralph Loop**. The OpenCode Ralph plugin auto-continues when the assistant becomes idle until it sees `<promise>DONE</promise>` or hits its iteration cap. Another broader Ralph implementation supports multiple CLIs including OpenCode, stores state in `.ralph`, supports custom prompt templates, status dashboards, iteration history, context injection, and agent rotation. ([GitHub][3])

That is very relevant, but it is not enough for your goal. Ralph-style loops are useful, but they are still mostly prompt-loop wrappers. For “God mode” OpenCode, you want something stronger: a **native OpenCode daemon/orchestrator** with hard runtime semantics, not just “continue until the model prints a magic string.”

# 2. OpenCode reality check

OpenCode already has several pieces you can exploit.

OpenCode is a client/server terminal-first coding agent with TUI, IDE, and desktop clients. The docs emphasize multi-session support, shareable links, LSP integration, provider flexibility, and a server architecture. ([OpenCode][4])

It already has custom commands. Commands can live in `.opencode/commands/*.md` or config, are invoked with slash commands, support frontmatter for description/agent/model, support `$ARGUMENTS` and positional variables, can inject shell output using ``!`command` ``, and can reference files with `@path`. Commands can also target agents or subagents. ([OpenCode][5])

It already has primary and subagent concepts. The built-in `build` agent can use all tools, `plan` is restricted, `general` is a subagent that can run multiple units of work in parallel, and `explore` is a read-only subagent. There are also internal compaction/title/summary agents. ([OpenCode][6])

It already has MCP support. OpenCode supports local and remote MCP servers through config, and external tools are made available automatically. The docs also warn that MCP can add substantial context overhead, with the GitHub MCP specifically called out as easy to overfill context. ([OpenCode][7])

It already has plugins. Plugins can hook events and customize behavior, and can be loaded from `.opencode/plugins`, global config, or npm packages. ([OpenCode][8])

Most importantly, OpenCode already has compaction. The current compaction summary template tracks goal, constraints, progress, blocked items, key decisions, next steps, critical context, and relevant files. Compaction can transform messages, prune old compactions, and auto-inject a continuation message when appropriate. ([GitHub][9])

The key weakness is the loop boundary. In the current `prompt.ts` loop, when the last assistant message has a finish reason that is not `"tool-calls"`, has no pending tool calls, and comes after the last user message, the loop logs that it is exiting and breaks. That is exactly why agents stop: the model says it is done, and the runtime believes it. ([GitHub][10])

Headless `opencode run` also currently treats `session.status: idle` as the cue to exit, and permissions in headless mode are rejected unless the dangerous skip-permissions flag is used. ([GitHub][11])

The status model is also too small for this. Today it is basically `idle`, `busy`, and `retry`. There is no first-class status for “daemon active, iteration 12, workers 3, exit condition failing.” ([GitHub][12])

There are already OpenCode issues showing why this needs to be designed carefully. Users have reported auto-compaction infinite loops, `/compact` causing unexpected continuation, compaction-auto settings being ignored on overflow, repeated compaction loops, and TODO-continuation loops. These are not arguments against daemon mode; they are evidence that daemon continuation must be explicit, visible, and separate from normal “assistant waiting for user” behavior. ([GitHub][13])

# 3. Critical review of the proposed plan

The proposed plan has the right instinct:

* Add daemon state.
* Add a daemon status.
* Add CLI flags.
* Reuse compaction.
* Make TUI daemon mode visually obvious.
* Wrap or override the normal loop so model “stop” does not mean process stop.

But I would change several major things.

## 3.1 Do not make `prompt.ts` do everything

The proposal puts too much daemon logic directly inside `prompt.ts`. That file should not become a giant workflow engine. It should expose better structured exit information, but the daemon loop should live in a separate `SessionDaemon` service.

Better:

```text
SessionPrompt.prompt()
  runs one normal OpenCode turn/loop
  returns PromptResult {
    exitReason:
      | "assistant_stop"
      | "tool_loop_complete"
      | "max_steps"
      | "compaction_stop"
      | "processor_error"
      | "permission_blocked"
      | "structured_output"
      | "cancelled"
  }

SessionDaemon.run()
  owns the outer forever loop
  decides whether to continue, compact, checkpoint, spawn workers, or exit
```

That keeps normal OpenCode behavior clean.

## 3.2 Do not rely on model promises as exit conditions

The Ralph plugin’s `<promise>DONE</promise>` mechanism is useful for lightweight loops, but it should be optional only. ([GitHub][3])

For your use case, exit should be deterministic:

```yaml
exit_when:
  all:
    - shell:
        command: "jankurai audit --json"
        success: "exit_code == 0"
        jq: ".findings == 0 and .caps == 0"
    - git:
        worktree: clean
```

The model may say “done,” but the daemon should only stop when the runtime checks pass.

## 3.3 Do not parse arbitrary `<FOREVER>` tags

Your instinct that the XML sketch is dangerous is correct. Any syntax that could accidentally appear in a pasted issue, README, spec, or stack trace is too risky.

The activation must be deliberate.

Bad:

```xml
<FOREVER>
...
</FOREVER>
```

Better:

```text
/daemon <<'OPENCODE_ORCHESTRATION'
<<<OPENCODE_ORCHESTRATION:V1 intent="execute" activation="explicit">>>
version: 1
kind: opencode.orchestration
mode: daemon
...
<<<END_OPENCODE_ORCHESTRATION:V1>>>
OPENCODE_ORCHESTRATION
```

Rules:

1. The block is only active when submitted through `/daemon`, `opencode run --daemon`, or an explicit API endpoint.
2. The opening sentinel must be the first non-whitespace content of the orchestration payload.
3. The closing sentinel must match exactly.
4. The YAML must validate against a schema.
5. Unknown top-level keys should fail closed by default.
6. Blocks found inside files, tool output, quoted text, git diffs, or assistant messages are inert.
7. Dangerous actions require explicit permission profiles.

That makes accidental execution extremely unlikely.

## 3.4 Do not let compaction auto-continue semantics leak into normal chat

The OpenCode issue history already shows confusion between “the agent is mid-task and should continue” and “the assistant answered and is waiting for the user.” ([GitHub][14])

Daemon mode should introduce a new explicit continuation policy:

```ts
continuationPolicy:
  | "interactive_wait"
  | "auto_continue_after_compaction"
  | "daemon_enforced"
```

Normal chat should stay conservative. Daemon mode should be the only place where assistant stop is treated as “end of iteration, begin next iteration.”

## 3.5 Permissions must be preflighted, not ignored

In headless run mode, permissions are currently rejected unless dangerous skip-permissions is set. ([GitHub][11])

Daemon mode should not paper over permission denials by “log and continue.” That can create hot infinite loops with no progress.

Better:

```yaml
permissions:
  profile: daemon-local-dev
  shell:
    allow:
      - "git status"
      - "git diff*"
      - "git add*"
      - "git commit*"
      - "git push*"
      - "jankurai audit*"
      - "npm test*"
    ask:
      - "rm *"
      - "sudo *"
    deny:
      - "curl * | sh"
      - "git push --force*"
  on_denied: pause
```

For fully unattended work, require an explicit flag/config profile.

## 3.6 Multi-agent must use durable coordination, not shared markdown only

A shared `THIS_TRACKER.md` is understandable, but weak. Multiple agents need:

* An append-only event log.
* A task ledger with claims/leases.
* Per-worker workspaces or git worktrees.
* Structured worker result envelopes.
* Coordinator-owned merge/checkpoint.
* Locks around git push and task completion.
* A chatter/event bus for notes and discoveries.

Markdown can be generated as a human view, but it should not be the source of truth.

# 4. The best architecture for OpenCode

## 4.1 Core principle

The LLM should be responsible for local reasoning and code changes.

The daemon runtime should be responsible for:

* Looping forever.
* Context clearing.
* Exit checks.
* Checkpointing.
* Permissions.
* Worker lifecycle.
* Task claims.
* Progress/stall detection.
* TUI status.
* Logs and recovery.

The agent can stop every iteration. That is fine. The daemon restarts it.

# 5. Recommended execution model

## 5.1 Two loops, not one

```text
┌─────────────────────────────────────────────────────────┐
│ OpenCode Daemon Harness                                │
│                                                         │
│  while daemon.active:                                  │
│    load state                                           │
│    evaluate preconditions                              │
│    maybe spawn workers                                 │
│    build iteration prompt                              │
│    call existing SessionPrompt.prompt()                │
│    wait for normal OpenCode loop to become idle         │
│    collect result                                      │
│    run verification                                    │
│    run checkpoint                                      │
│    evaluate exit_when                                  │
│    compact / rollover context                          │
│    update ledger                                       │
│                                                         │
└─────────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│ Existing OpenCode Agent Loop                            │
│                                                         │
│  model → tool calls → processor → model → ...           │
│                                                         │
│  exits when model stops, compaction stops, error, etc.  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

This gives you “promised infinite loop” without fighting the model.

## 5.2 State machine

```text
CREATED
  │
  ▼
PREFLIGHT
  │ validate permissions, repo, MCP profiles, syntax
  ▼
RUNNING_ITERATION
  │ normal OpenCode prompt loop
  ▼
COLLECTING_RESULT
  │ summarize outputs, parse worker envelopes, update ledger
  ▼
VERIFYING
  │ tests, audit, git checks, custom checks
  ├───────────────┐
  │ pass          │ fail
  ▼               ▼
CHECKPOINTING    CONTINUING
  │               │
  ▼               ▼
EVALUATE_EXIT ──► CONTEXT_MAINTENANCE
  │ pass          │ compact / rollover / prune / handoff
  ▼               │
STOPPED ◄─────────┘
```

## 5.3 The daemon owns continuation

Assistant says:

> “I’m done.”

Daemon says:

> “Exit condition failed. Iteration 8 begins. Here is the current ledger, last failure, and next work unit.”

That is the entire trick.

# 6. Files and changes I would make in OpenCode

The exact paths may need small adjustment during implementation, but this is the architecture I would aim for.

## 6.1 New daemon modules

```text
packages/opencode/src/session/daemon/
  schema.ts
  parser.ts
  service.ts
  state.ts
  ledger.ts
  evaluator.ts
  checkpoint.ts
  context.ts
  workers.ts
  permissions.ts
  prompt-builder.ts
  events.ts
```

### `schema.ts`

Defines the validated orchestration language.

Key exported types:

```ts
DaemonSpec
DaemonRun
DaemonIteration
DaemonExitCondition
DaemonCheckpointPolicy
DaemonContextPolicy
DaemonWorkerSpec
DaemonPermissionProfile
DaemonEvent
```

This should be schema-validated with whatever schema system OpenCode already uses internally.

### `parser.ts`

Parses only deliberately activated orchestration blocks.

Responsibilities:

* Verify sentinel.
* Verify activation mode.
* Parse YAML.
* Validate schema.
* Reject unknown/dangerous keys unless explicitly allowed.
* Produce `DaemonSpec`.

### `service.ts`

Main runtime service.

Responsibilities:

* `start(sessionID, spec)`
* `pause(runID)`
* `resume(runID)`
* `stop(runID, reason)`
* `status(runID)`
* `runForever(runID)`

This is the outer loop.

### `state.ts`

Durable run state.

Possible location:

```text
.opencode/daemon/<run-id>/state.json
.opencode/daemon/<run-id>/ledger.jsonl
.opencode/daemon/<run-id>/handoff.md
.opencode/daemon/<run-id>/workers/
.opencode/daemon/<run-id>/snapshots/
```

Do not store the whole thing only in memory. If the terminal dies, daemon state should be resumable.

### `ledger.ts`

Append-only event ledger.

Example events:

```json
{"type":"run.started","run_id":"...","time":"..."}
{"type":"iteration.started","n":7}
{"type":"task.claimed","task_id":"audit-014","worker":"fixer-2"}
{"type":"check.failed","name":"jankurai audit","summary":"3 findings, 1 cap"}
{"type":"context.compacted","iteration":7,"tokens_before":153221}
{"type":"git.committed","sha":"abc123","message":"jankurai: fix audit cap parser"}
```

### `evaluator.ts`

Deterministic exit/check evaluation.

Supports:

* Shell exit code.
* Regex.
* JSON/JQ checks.
* File existence/content.
* Git clean/diff checks.
* Test commands.
* OpenCode tool results.
* Optional model judge, but advisory only unless explicitly allowed.

### `checkpoint.ts`

Git and artifact checkpoint policy.

Should enforce:

* No commit if verification failed unless configured.
* No push if branch policy fails.
* No force push by default.
* Commit message templates.
* Changed-file allow/deny lists.
* Pre-commit checks.
* Push locks.

### `context.ts`

Context strategy.

Supports:

* Normal overflow compaction.
* Forced compaction every N iterations.
* Handoff capsules.
* Hard session rollover.
* Pinned files/instructions.
* Tool-output pruning.
* MCP-profile context trimming.

### `workers.ts`

Multi-agent orchestration.

Supports:

* Subagents inside one session.
* Multiple OpenCode sessions through server API.
* Optional external MCP-based workers.
* Git worktree isolation.
* Worker result envelopes.
* Join/race/quorum strategies.

### `permissions.ts`

Preflight and runtime permission checks.

Daemon mode should refuse to start unattended if the spec requests commands that cannot run without manual approval.

### `prompt-builder.ts`

Turns the spec and current state into the actual prompt sent to the LLM each iteration.

Important: the raw orchestration block should not be blindly pasted into the model forever. The model should get a normalized, minimal, current instruction packet.

## 6.2 Modify `prompt.ts`

Do not stuff all daemon behavior into `prompt.ts`.

Add structured result reporting.

Current behavior internally breaks on assistant stop, compaction stop, processor stop, structured output, etc. ([GitHub][10])

Desired:

```ts
type PromptExitReason =
  | "assistant_stop"
  | "no_tool_calls"
  | "max_steps"
  | "compaction_stop"
  | "processor_stop"
  | "processor_error"
  | "permission_blocked"
  | "structured_output"
  | "cancelled"

interface PromptResult {
  messageID?: string
  exitReason: PromptExitReason
  finishReason?: string
  hadToolCalls: boolean
  steps: number
  error?: SerializedError
}
```

Then daemon mode can react correctly.

## 6.3 Modify `status.ts`

Current status is too limited. Add daemon-aware state. ([GitHub][12])

Option A: add a new status variant.

```ts
{
  type: "daemon",
  runID: string,
  phase:
    | "preflight"
    | "running"
    | "verifying"
    | "checkpointing"
    | "compacting"
    | "waiting"
    | "paused"
    | "stopping",
  iteration: number,
  forever: boolean,
  workers: {
    active: number,
    total: number
  },
  exitSummary?: string,
  lastProgressAt: number
}
```

Option B, better long-term: keep `busy|idle|retry`, but add `mode`.

```ts
{
  type: "busy",
  mode: "interactive" | "daemon",
  daemon?: DaemonStatus
}
```

I prefer Option B because daemon can be busy, retrying, paused, or waiting.

## 6.4 Modify `run.ts`

`opencode run` currently exits when session becomes idle. ([GitHub][11])

Add flags:

```text
opencode run --daemon "task..."
opencode run --daemon-file daemon.yaml
opencode run --daemon-exit "jankurai audit --json"
opencode run --daemon-checkpoint git
opencode run --daemon-compact-every 1
opencode run --daemon-rollover-every 5
opencode run --daemon-state .opencode/daemon/jankurai
opencode run --daemon-resume <run-id>
opencode run --daemon-stop <run-id>
opencode run --daemon-status <run-id>
```

Behavior:

* If `--daemon` is absent, unchanged.
* If `--daemon` is present, `idle` means “iteration ended,” not “process should exit.”
* The daemon service evaluates exit conditions.
* The process exits only when daemon stopped, user cancelled, or configured terminal failure occurs.

## 6.5 Modify command system

OpenCode commands already support slash commands, frontmatter, templates, file references, shell injection, and agent/subagent targeting. ([OpenCode][5])

Add a built-in command:

```text
/daemon
```

Supported forms:

```text
/daemon start <inline task>
/daemon start --file .opencode/daemon/jankurai.yaml
/daemon status
/daemon pause
/daemon resume
/daemon stop
/daemon compact
/daemon inject "new constraint..."
/daemon workers
/daemon tail
```

Custom commands should be able to wrap daemon specs, but arbitrary command markdown should not activate daemon semantics unless the command declares:

```yaml

  enabled: true
```

## 6.6 Modify TUI

You asked for obvious gold “FOREVER” mode. I strongly agree.

When daemon active:

```text
╭────────────────────────────────────────────────────────────╮
│  ∞ FOREVER                                                │
│  DAEMON · jankurai-dirty-worktree · iteration 17           │
│  phase: verifying · workers: 3/4 · exit: audit not clean   │
╰────────────────────────────────────────────────────────────╯
```

Design rules:

* Gold accent overlay.
* Persistent small top banner.
* Do not replace the user’s whole theme.
* Show iteration count.
* Show whether it is truly unbounded.
* Show latest exit-condition failure.
* Show pause/stop hint.
* Show active workers.
* Show last progress time.

This matters because daemon mode can spend money, modify files, commit, push, and run indefinitely.

# 7. The deliberate syntax: OpenCode Orchestration Language

I would not call it “prompt XML.” I would call it something like:

> **OpenCode Orchestration Language v1**, or **OOX v1** / **OAL v1**.

I’ll call it **OAL v1** below.

The key design: it is “in-prompt” from the user’s perspective, but **not interpreted by the LLM**. OpenCode parses it, validates it, and compiles it.

## 7.1 Activation gates

### Safe activation form

```text
/daemon <<'OPENCODE_ORCHESTRATION'
<<<OPENCODE_ORCHESTRATION:V1 intent="execute" activation="explicit">>>
version: 1
kind: opencode.orchestration
mode: daemon

goal: |
  Work through dirty worktree items until jankurai audit is clean.

loop:
  forever: true

exit_when:
  all:
    - shell:
        command: "jankurai audit --json"
        success: "exit_code == 0"
        jq: ".findings == 0 and .caps == 0"

<<<END_OPENCODE_ORCHESTRATION:V1>>>
OPENCODE_ORCHESTRATION
```

### Inert accidental paste

This should **not** activate:

```text
Here is a README that says:

<<<OPENCODE_ORCHESTRATION:V1 intent="execute" activation="explicit">>>
...
<<<END_OPENCODE_ORCHESTRATION:V1>>>
```

Why? Because it was not submitted through `/daemon`, `--daemon`, or the daemon API.

### CLI activation

```bash
opencode run --daemon-file .opencode/daemon/jankurai.yaml
```

### API activation

```text
POST /session/:id/daemon/start
```

with a parsed spec body.

## 7.2 Mandatory top-level fields

```yaml
version: 1
kind: opencode.orchestration
mode: daemon | workflow | oneshot
name: string
activation:
  required: true
  source: slash_command | cli_flag | api
```

For true infinite looping:

```yaml
loop:
  forever: true
  max_iterations: null
```

If `forever: true`, there must be at least one of:

```yaml
exit_when: ...
manual_stop: true
safety: ...
```

Manual stop should always exist, but a deterministic `exit_when` is what makes it useful.

# 8. Example: your JANKURAI daemon spec

This is the style I would actually want OpenCode to support.

```text
/daemon <<'OPENCODE_ORCHESTRATION'
<<<OPENCODE_ORCHESTRATION:V1 intent="execute" activation="explicit">>>
version: 1
kind: opencode.orchestration
mode: daemon
name: jankurai-dirty-worktree

ui:
  banner: forever
  accent: gold
  show:
    - iteration
    - phase
    - workers
    - last_exit_failure
    - last_commit
    - last_progress

goal: |
  Work through the dirty worktree created while implementing
  @agent/JANKURAI_STANDARD.md and @JANKURAI_TASKLIST.md.

instructions: |
  Investigate uncommitted code. Run git diff. Look for mistakes,
  regressions, incomplete work, and test failures. Pick one dirty
  worktree item or task at a time, preferably randomly unless a
  critical blocker is obvious. Fix it, verify it, commit it, and push it.

  Do not stop just because one item is done. Continue until the
  exit conditions pass.

loop:
  forever: true
  on_model_stop: continue
  iteration_prompt: |
    You are in daemon iteration {{iteration.number}}.

    Current objective:
    {{goal}}

    Current task selection:
    - Read the task ledger.
    - Inspect git diff.
    - Claim exactly one work item.
    - Fix it.
    - Verify it.
    - Mark it complete.
    - Return a structured iteration report.

  stall_detection:
    no_file_change_iterations: 2
    same_error_iterations: 3
    on_stall:
      - spawn_reviewer:
          agent: explore
          prompt: "Find why the daemon is stalled and propose the next concrete action."
      - compact_now: true
      - continue: true

context:
  strategy: hybrid
  compact_every_iterations: 1
  rollover_every_iterations: 5
  handoff_file: ".opencode/daemon/jankurai/handoff.md"
  pin:
    files:
      - "agent/JANKURAI_STANDARD.md"
      - "JANKURAI_TASKLIST.md"
      - "JANKURAI_TASKLIST.md.completed"
  summarize:
    include:
      - goal
      - constraints
      - completed_tasks
      - current_findings
      - failing_tests
      - changed_files
      - last_commit
      - next_tasks
      - blockers
  tool_output_policy:
    keep_recent: 20
    summarize_large_outputs: true
    drop_successful_test_logs_after_summary: true

tasks:
  source:
    type: markdown_checklist
    file: "JANKURAI_TASKLIST.md"
  completed_archive: "JANKURAI_TASKLIST.md.completed"
  claim_policy:
    strategy: random_ready
    lease_minutes: 45
    one_task_per_iteration: true
  completion_policy:
    remove_completed_from_source: true
    append_completed_to_archive: true
  discovery:
    audit_findings:
      command: "jankurai audit --json"
      append_new_findings_to: "JANKURAI_TASKLIST.md"

verification:
  required:
    - shell:
        name: "jankurai audit"
        command: "jankurai audit --json"
        success: "exit_code == 0"
        collect_json: true
    - shell:
        name: "project tests"
        command: "npm test"
        success: "exit_code == 0"
        optional_if_missing: true
    - git:
        name: "review diff"
        require_diff_reviewed: true

exit_when:
  all:
    - shell:
        name: "audit clean"
        command: "jankurai audit --json"
        success: "exit_code == 0"
```

# 9. Full syntax library

Below is the library I would design. Not every function needs to ship in v1, but the schema should leave room for these concepts without painting OpenCode into a corner.

## 9.1 Activation and identity

### `version`

Schema version.

```yaml
version: 1
```

### `kind`

Must be exact.

```yaml
kind: opencode.orchestration
```

### `mode`

```yaml
mode: daemon | workflow | oneshot
```

* `daemon`: outer loop continues until exit or stop.
* `workflow`: finite graph/sequence.
* `oneshot`: one compiled prompt with structured policies.

### `name`

Human and log identifier.

```yaml
name: jankurai-dirty-worktree
```

### `activation`

```yaml
activation:
  required: true
  source: slash_command | cli_flag | api
  require_first_payload: true
  allow_from_file: false
```

Prevents accidental execution.

## 9.3 Loop control

### `loop.forever`

```yaml
loop:
  forever: true
  max_iterations: null
```

No implicit max. Stops only when `exit_when`, manual stop, or configured safety pause/stop fires.

### `loop.max_iterations`

```yaml
loop:
  max_iterations: 100
```

Finite loop.

### `loop.until`

```yaml
loop:
  until:
    shell:
      command: "npm test"
      success: "exit_code == 0"
```

Loop until condition passes.

### `loop.while`

```yaml
loop:
  while:
    shell:
      command: "git diff --quiet"
      success: "exit_code != 0"
```

Loop while condition remains true.

### `loop.on_model_stop`

```yaml
loop:
  on_model_stop: continue | evaluate_exit | stop | pause
```

For daemon mode, default should be `evaluate_exit`.

### `loop.iteration_prompt`

```yaml
loop:
  iteration_prompt: |
    Iteration {{iteration.number}}.
    Claim one task, fix it, verify it, report.
```

Prompt template for each iteration.

### `loop.sleep`

```yaml
loop:
  sleep:
    after_iteration_seconds: 5
```

Useful for polling external conditions.

### `loop.retry`

```yaml
loop:
  retry:
    attempts: 3
    backoff_seconds: [5, 30, 120]
```

Retries transient failures.

### `loop.race`

```yaml
loop:
  race:
    - agent: fixer-a
    - agent: fixer-b
  winner: first_verified
```

Multiple workers attempt alternatives; first verified result wins.

### `loop.parallel`

```yaml
loop:
  parallel:
    max_concurrency: 4
    join: all | any | quorum
```

Parallel worker policy.

# 10. Multi-agent collaboration done properly

Your “Read THIS_TRACKER.md and update it” instinct is the common first version. The next version should be a real coordination substrate.

## 10.1 Use a coordinator-worker model

```text
Coordinator
  │
  ├── Finder worker 1: inspect diff
  ├── Finder worker 2: inspect audit findings
  ├── Fixer worker 1: fix claimed task in worktree A
  ├── Fixer worker 2: fix claimed task in worktree B
  └── Reviewer worker: review proposed patches
```

The coordinator alone:

* Owns the main branch.
* Updates the canonical task ledger.
* Runs final verification.
* Commits.
* Pushes.

Workers:

* Claim tasks.
* Work in isolated sessions or git worktrees.
* Return structured results.
* Do not push directly unless explicitly allowed.

## 10.2 Replace markdown tracker as source of truth

Use:

```text
.opencode/daemon/<run-id>/
  tasks.json
  ledger.jsonl
  chat.jsonl
  handoff.md
  workers/
    fixer-1/
      state.json
      result.json
      patch.diff
    fixer-2/
      state.json
      result.json
      patch.diff
  locks/
    git-push.lock
    task-audit-014.lock
```

Markdown files can still exist:

```text
JANKURAI_TASKLIST.md
JANKURAI_TASKLIST.md.completed
```

But the daemon should treat them as imported/exported human-facing artifacts, not the only coordination mechanism.

## 10.3 Worker result contract

Every worker should return this:

```json
{
  "schema": "opencode.worker_result.v1",
  "worker_id": "fixer-1",
  "task_id": "audit-014",
  "status": "done",
  "summary": "Fixed audit cap parsing regression.",
  "files_changed": [
    "src/audit/caps.ts",
    "test/audit/caps.test.ts"
  ],
  "commands_run": [
    {
      "command": "npm test -- caps",
      "exit_code": 0
    }
  ],
  "risks": [],
  "needs_coordinator_action": [
    "Run full test suite",
    "Commit and push"
  ]
}
```

No vague “I think it works.” The coordinator gets data.

## 10.4 Use leases

Task state:

```json
{
  "id": "audit-014",
  "status": "claimed",
  "claimed_by": "fixer-1",
  "lease_expires_at": "2026-05-07T15:30:00-06:00",
  "title": "Fix audit cap parser regression"
}
```

If a worker dies, the task can be requeued.

## 10.5 Git worktree isolation

Multiple writing workers in one repo is a merge-conflict generator.

Use:

```text
.opencode/daemon/<run-id>/worktrees/fixer-1
.opencode/daemon/<run-id>/worktrees/fixer-2
```

Coordinator cherry-picks or applies patches after review.

## 10.6 Central chatter bus

Instead of asking agents to edit the same markdown file, let them append events:

```json
{"from":"finder-1","type":"finding","task_id":"audit-022","message":"Potential regression in parser."}
{"from":"reviewer-1","type":"review","patch":"fixer-2.patch","verdict":"needs_changes"}
```

The TUI can render this as a live “agent room.”

# 11. How this maps to OpenCode today

## 11.1 Current custom commands are a good entry point

Because OpenCode commands already support slash invocation, frontmatter, file refs, shell output injection, and subagent targeting, `/daemon` can be introduced naturally. ([OpenCode][5])

But the daemon spec should not be “just another prompt template.” It should be parsed into runtime behavior.

## 11.2 Current subagents are useful but not enough

OpenCode already has subagents for parallel units of work and read-only exploration. ([OpenCode][6])

Daemon mode should use them, but add:

* Worker IDs.
* Structured result schemas.
* Task leases.
* Join policies.
* Isolation strategy.
* Coordinator-only checkpointing.

## 11.3 Current MCP support needs profile control

OpenCode’s MCP docs warn about context overhead. ([OpenCode][7])

Daemon mode should avoid “all MCP tools always visible.” It should support:

```yaml
mcp:
  phase_profiles:
    inspect: local-readonly
    fix: local-coding
    publish: github-pr
```

This prevents tool bloat and reduces model confusion.

## 11.4 Current compaction is a foundation, not the whole solution

OpenCode already compacts with useful summary categories. ([GitHub][9])

Daemon mode should add:

* Forced compaction by iteration count.
* Handoff capsules.
* Session rollover.
* Explicit “daemon continuation” policy.
* No accidental auto-continue in normal chat.

The existing compaction-loop issues are a warning: do not let daemon behavior leak into non-daemon sessions. ([GitHub][13])

# 12. Implementation phases

## Phase 1: Minimal native daemon

Goal: prove the core forever loop.

Changes:

1. Add `DaemonSpec` schema.
2. Add `/daemon` command or CLI `--daemon`.
3. Add `SessionDaemon` service.
4. Add outer loop around `SessionPrompt.prompt()`.
5. Continue after assistant stop.
6. Evaluate deterministic `exit_when`.
7. Add basic status and gold TUI banner.
8. Add ledger.

Supported v1 syntax:

```yaml
version: 1
kind: opencode.orchestration
mode: daemon
goal: ...
loop:
  forever: true
exit_when:
  shell:
    command: "..."

  shell:
    command: "..."
context:
  compact_every_iterations: 1
```

## Phase 2: Context hardening

Add:

* Forced compaction.
* Handoff file.
* Session rollover.
* Pinned context.
* Tool-output pruning.
* Better compaction prompt for daemon summaries.

## Phase 3: Git/checkpoint hardening

Add:

* First-class git checkpoint policy.
* Commit templates.
* Push locks.
* Branch policy.
* Pre/post verification.
* Dirty-worktree diff review enforcement.

## Phase 4: Multi-agent

Add:

* Worker specs.
* Worker lifecycle.
* Structured worker result schema.
* Task ledger.
* Leases.
* Git worktree isolation.
* Coordinator merge flow.

## Phase 5: MCP profiles and live control

Add:

* Phase-specific MCP profiles.
* Live `/daemon inject`.
* `/daemon pause/resume/status/workers/tail`.
* Plugin hooks.
* Server API endpoints.

# 13. Verification plan

## Parser tests

* Reject orchestration block without `/daemon` or `--daemon`.
* Reject block not first in payload.
* Reject missing closing sentinel.
* Reject unknown dangerous keys.
* Reject `forever: true` without exit/manual stop/safety.
* Accept valid YAML spec.
* Ensure blocks inside tool output or file refs are inert.

## Runtime loop tests

Use a fake model that always returns assistant stop.

Expected:

1. Iteration 1 starts.
2. Model stops.
3. Runtime evaluates exit condition.
4. Exit fails.
5. Iteration 2 starts.
6. Repeats until fake condition passes.

This directly verifies the root requirement.

## Exit condition tests

* Shell exit code pass/fail.
* Regex pass/fail.
* JQ pass/fail.
* Git clean/dirty.
* Tasklist empty/non-empty.
* Combined `all`, `any`, `not`.

## Compaction tests

* `compact_every_iterations: 1` forces compaction.
* Handoff file contains goal, completed tasks, blockers, changed files, next steps.
* Rollover starts a new session with the handoff capsule.
* Normal non-daemon compaction does not unexpectedly continue.

## Permission tests

* Headless daemon refuses to start when required commands are not allowed.
* Permission denial pauses instead of hot-looping.
* Dangerous commands denied by default.

## TUI tests

* Gold banner appears only during daemon mode.
* “FOREVER” label appears for unbounded loop.
* Iteration count updates.
* Pause/stop commands work.
* Worker count updates.

## Multi-agent tests

* Workers claim tasks with leases.
* Expired leases requeue.
* Two workers cannot claim the same task.
* Coordinator alone commits.
* Worker patches are reviewed before merge.

## End-to-end JANKURAI test

Simulate:

1. Dirty worktree.
2. `jankurai audit` returns findings.
3. Daemon claims one task.
4. Fake agent fixes it.
5. Verification passes.
6. Commit occurs.
7. Audit still fails.
8. Loop continues.
9. Audit returns zero findings and zero caps.
10. Worktree clean.
11. Daemon stops.

# 14. Major risks and solutions

## Risk: the model ignores the syntax

Solution: the model does not own the syntax. OpenCode parses and enforces it.

## Risk: accidental pasted orchestration block runs

Solution: require `/daemon`, `--daemon`, or API activation; sentinel must be first payload; loaded files/tool outputs are inert.

## Risk: infinite loop burns money forever

Solution: gold TUI, heartbeat, status, optional budgets, manual stop, pause-on-stall, explicit `forever: true`.

## Risk: permission denial hot loop

Solution: preflight permissions and pause on denial.

## Risk: compaction causes unwanted continuation

Solution: daemon continuation policy is explicit and separate from interactive compaction.

## Risk: multiple workers corrupt git state

Solution: git worktrees, leases, coordinator-only merge/commit/push.

## Risk: MCP tools bloat context

Solution: MCP profiles, lazy loading, phase-based tool visibility.

## Risk: “commit and push” ships broken code

Solution: checkpoint only after verification gates pass; branch policy; no force push; final git status check.

## Risk: daemon loses memory after context clear

Solution: durable state, ledger, task DB, handoff capsule, pinned context.

# 15. The single most important design decision

Do not design this as:

```text
The prompt says FOREVER, so the model should keep going forever.
```

Design it as:

```text
The user deliberately starts a daemon.
OpenCode compiles the daemon spec.
OpenCode owns the forever loop.
The model performs one bounded unit of work per iteration.
The runtime verifies, checkpoints, clears context, and restarts.
```

That is the difference between “agents always stop” and “the process does not stop.”

# 16. What I would build first

I would build the smallest native version with this exact behavior:

```text
/daemon --exit "jankurai audit --json | jq '.findings == 0 and .caps == 0'" --compact-every 1 --checkpoint git
```

Internally:

1. Start daemon state.
2. Show gold FOREVER banner.
3. Send the original task to normal OpenCode prompt loop.
4. When assistant stops, treat it as iteration end.
5. Run exit condition.
6. If exit fails, compact and continue.
7. If verification passed and files changed, commit/push according to policy.
8. Repeat forever.

Then expand into full OAL syntax, task ledger, workers, MCP profiles, and live control.

That gives you immediate value while creating the foundation for the “God mode” orchestration language.
