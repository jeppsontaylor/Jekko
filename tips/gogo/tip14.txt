# Recommendation: build a **deterministic daemon supervisor**, not “magic prompt XML”

The strongest version of this is not “teach the LLM to obey `<FOREVER>`.” The strongest version is:

> **A deliberate, gated Jekko DaemonSpec configures a runtime supervisor. The LLM can plan, code, test, and delegate, but the Jekko runtime owns the loop, stop checks, compaction, worker spawning, checkpoints, permissions, and recovery.**

That distinction matters. Prompt-only looping is exactly why agents stop: the model eventually emits a normal final answer. A daemon runtime should interpret “assistant finished” as only one event in a larger supervisor state machine. In daemon mode, “assistant finished” means: checkpoint, evaluate exit criteria, compact or clear context, enqueue the next turn, and continue unless a trusted stop condition passes or the operator stops it.

I did not find a mature, broadly adopted **in-prompt agent workflow scripting standard** comparable to MCP for tools. The closest standards are **AGENTS.md** for repo instructions and **MCP** for tool/resource integration, while workflow languages remain framework-specific: LangGraph, Dify, CrewAI Flows, OpenAI Agents SDK orchestration, Microsoft Agent Framework orchestrations, Temporal-style durable workflows, and newer research languages like Turn. AGENTS.md is explicitly a simple open Markdown format for coding-agent instructions, used across many coding tools, but it is not a loop/workflow DSL. MCP is an open protocol for tools, resources, prompts, roots, sampling, logging, progress, and cancellation, but it is not itself an agent-control language. ([Agents][1])

# 1. What the broader ecosystem says

## 1.1 There is no real standard yet for “in-prompt agent scripting”

The ecosystem has converged on pieces, not a universal language:

**AGENTS.md** is the closest cross-agent instruction standard. It is intentionally just Markdown: repo-specific build commands, tests, conventions, security notes, and nested precedence. It is powerful as persistent context, but it does not define loops, stop conditions, durable state, worker graphs, or permissions. ([Agents][1])

**MCP** is the closest tool/context standard. It standardizes hosts, clients, servers, tools, resources, prompts, sampling, roots, logging, cancellation, and progress over JSON-RPC. That is exactly the right substrate for “spin up tools and MCP servers robustly,” but MCP does not tell an agent “loop forever until audit is clean.” ([Model Context Protocol][2])

**Framework DSLs exist, but they live outside prompts.** LangGraph describes itself as infrastructure for long-running, stateful agents with durable execution, memory, human-in-the-loop, and observability. Dify has explicit Loop nodes with termination conditions and maximum loop counts. CrewAI Flows provide event-driven workflows with state, branching, loops, and conditional logic. Microsoft Agent Framework has sequential, concurrent, handoff, group chat, and manager-style “Magentic” orchestrations. OpenAI’s Agents SDK frames agents as applications that plan, call tools, collaborate across specialists, and keep state, with orchestration via handoffs or agents-as-tools. ([GitHub][3])

The important conclusion: the industry is putting loops in **runtime control flow**, not trusting the model to maintain a prompt-internal loop indefinitely.

## 1.2 The common production agent loop is simple; the hard part is supervision

The basic agent loop is well understood: call the LLM, run tool calls, feed results back, stop when there are no more tool calls. Steve Kinney’s agent-loop writeup states that major frameworks converge on this pattern, and that the production-hard parts are context compaction, loop detection, budgets, and graceful termination. ([Steve Kinney][4])

That maps directly to Jekko. Normal Jekko behavior should stop when the model gives a normal final response. Daemon behavior should not mutate the normal loop. It should wrap it.

## 1.3 Long-horizon agents need context engineering, not giant prompts

Anthropic’s context-engineering guidance is directly relevant: context is finite, larger context degrades focus, and long-horizon tasks need compaction, structured note-taking, and multi-agent architectures. Anthropic also recommends clear prompt sections but warns against brittle hardcoded logic in prompts. ([Anthropic][5])

This means your syntax should not be an enormous XML prompt that the model must “obey.” It should be a structured, parsed, validated spec that Jekko’s daemon runtime executes. The LLM receives only the current iteration’s distilled context and task.

## 1.4 Context clearing is becoming a normal coding-agent workflow

A recent coding-agent workflow article argues for “one task, one session,” clear context between tasks, and persistent plan/task stores instead of carrying stale conversation history. It describes an emerging pattern: research, plan, split into tasks, then repeatedly run a fresh agent that picks one task, validates it, and exits. ([willness.dev][6])

The Jekko community is already feeling this: a Reddit thread on Jekko context loss describes native compaction as causing “session amnesia,” and proposes plugins that maintain persistent goal/progress blocks, file attention, protected errors/decisions, and todo sync across compaction. ([Reddit][7])

The right daemon should support both strategies:

1. **Compacted continuation** for a single coherent thread.
2. **Fresh-session-per-item** for long queues where stale context is harmful.

## 1.5 Multi-agent work needs bounded manager/worker orchestration

OpenAI’s Agents SDK docs distinguish “handoffs” from “agents as tools”; for this use case, “agents as tools” is usually the better mental model because a manager stays in control and delegates bounded subtasks to specialists. The docs also warn that adding specialists too early creates more prompts, traces, and approval surfaces. ([OpenAI Developers][8])

That is exactly the concern with Jekko subagents. Jekko already has subagents, but the runtime needs recursion guards, worker limits, and isolation. Jekko issues show real runaway recursion risks when subagents can spawn subagents without a depth guard. ([GitHub][9])

# 2. Jekko-specific findings

## 2.1 Jekko already has useful building blocks

Jekko already supports custom commands with Markdown frontmatter, command arguments, positional placeholders, shell-output injection, and file references. That makes YAML/frontmatter a natural user-facing syntax style. ([Jekko][10])

Jekko already has primary agents and subagents. The docs define `build` as the default full-access primary agent, `plan` as a restricted planning agent, `general` as a multi-step subagent that can run units of work in parallel, `explore` as a read-only subagent, and `compaction` as a hidden system agent. ([Jekko][11])

Jekko already supports local MCP servers in config, including local commands, environment, enable/disable, and timeouts. ([Jekko][12])

Jekko already exposes plugin events including `session.status`, `session.idle`, `session.compacted`, `permission.asked`, `tool.execute.before`, `tool.execute.after`, and TUI events. It also has an experimental compaction hook that can inject custom state into compaction or replace the compaction prompt. ([Jekko][13])

Jekko’s SDK exposes session creation, prompting, messages, aborting, summarizing, and `session.prompt({ noReply: true })` for injecting context without triggering an assistant response. ([Jekko][14])

Jekko config already has compaction controls: `auto`, `prune`, and `reserved`. ([Jekko][15])

So the foundation exists. What is missing is a **daemon supervisor** that uses those primitives deterministically.

## 2.2 The core stop behavior is real and community-visible

The most relevant Jekko issue is almost exactly your complaint: “Jekko won’t run unattended for extended periods.” The reporter says Jekko does a dozen or so repetitive items, emits a status like “Continuing research immediately,” then stops; telling it to continue makes it resume briefly, then stop again. ([GitHub][16])

There are also compaction-related stop bugs. One issue describes `jekko run` exiting after auto-compaction because a synthetic “Continue...” message is skipped and the prompt loop’s exit condition fires when the last assistant message has `finish="stop"`. ([GitHub][17])

Another issue reports the inverse failure mode: Jekko continuing after `finish=stop`, causing an invalid assistant-prefill error with Anthropic models, and states the expected behavior: when a step completes with `finish=stop` and not `tool-calls`, the prompt loop should not continue to the next step. ([GitHub][18])

This is crucial: Jekko has bugs on **both sides** of the boundary. Sometimes it stops when users want continuation; sometimes it continues when the provider says the turn is done. That means the solution must not be “hack `runLoop()` to ignore stop.” It must be a separate daemon mode with explicit semantics.

## 2.3 Headless `jekko run` currently exits on idle

The current `run.ts` path listens for `session.status` and breaks when the session becomes `idle`. It also auto-approves permissions only when `dangerously-skip-permissions` is set; otherwise permission requests are auto-rejected. ([GitHub][19])

That means daemon mode must change the headless event loop. In daemon mode, `idle` is not “process complete.” It is “one inner Jekko turn ended; daemon supervisor should checkpoint/evaluate/restart or exit.”

## 2.4 Session status currently cannot express daemon state

`SessionStatus.Info` currently has `idle`, `retry`, and `busy`; setting `idle` publishes `session.idle` and deletes status state. ([GitHub][20])

Daemon mode needs either:

1. A new `daemon.status` event that does not disturb existing `session.status` semantics, or
2. A backward-compatible status extension.

I recommend a separate `daemon.status` event first, because changing the public status union can break consumers.

# 3. Critique of the proposed AI plan

The proposed plan has the right instinct but the wrong center of gravity.

## What it gets right

It correctly identifies that the model’s normal final answer is the main reason agents stop. It correctly notices that compaction already exists and should be reused. It correctly identifies that headless `run.ts`, session status, TUI signaling, and MCP/subagent orchestration all need work.

## What I would change

**Do not override the normal prompt loop globally.** Normal sessions should stop when the model stops. Jekko already has issues around both premature stop and unwanted continuation. The daemon should be an outer runtime supervisor that re-prompts only when a session/run is explicitly daemon-controlled. ([GitHub][17])

**Do not treat permission denial as “continue.”** Permission denial is not a transient tool failure. In daemon mode it should become a policy event: pause, reject-and-continue, or fail, depending on the daemon spec. Headless Jekko currently auto-rejects permission requests unless dangerous skip is enabled, so daemon mode must make this explicit. ([GitHub][19])

**Do not store daemon state only in ephemeral session metadata.** A promised forever loop needs durable run state, heartbeat, iteration log, worker state, and recovery after crash/restart. LangGraph’s durable-execution framing is the right mental model here, even if Jekko implements its own lighter version. ([GitHub][3])

**Do not rely on XML-like tags.** `<FOREVER>` is too easy to paste accidentally, too easy for documentation to trigger, and too easy for malicious text to spoof. Use a deliberately activated, schema-validated fenced block or file-based spec.

**Do not let the LLM own worker spawning limits.** Jekko has active reports of runaway subagent recursion. Daemon mode must enforce `max_parallel`, `max_depth`, `max_children`, `max_open_processes`, and per-worker isolation at the runtime layer. ([GitHub][9])

**Do not use tracker Markdown as the primary coordination system.** Markdown trackers are fine as human-visible reports, but the real coordination layer should be a structured daemon ledger: queue, claims, locks, worker heartbeats, artifacts, verification results, and merge state.

# 4. The best Jekko architecture

## Core design principle

**The daemon runtime owns control flow. The model owns local reasoning.**

That means:

```text
User DaemonSpec
  ↓
Jekko parser validates deliberate activation
  ↓
DaemonRun record is created
  ↓
Daemon supervisor starts iteration 1
  ↓
Normal SessionPrompt.prompt / loop runs
  ↓
Assistant stops, errors, compacts, asks permission, or finishes tools
  ↓
Daemon supervisor handles the event:
    checkpoint → evaluate stop → compact/clear → schedule next prompt
  ↓
Repeat forever unless stop condition passes or operator stops it
```

The inner Jekko loop remains normal. The outer daemon loop interprets normal completion as a restart point.

## 5.3 The daemon runner state machine

Recommended phases:

```text
created
  ↓
starting
  ↓
running_iteration
  ↓
assistant_idle
  ↓
checkpointing
  ↓
evaluating_stop
  ├─ stop passed → completed
  └─ stop failed
       ↓
context_maintenance
  ├─ compact
  ├─ clear_new_session
  └─ keep
       ↓
scheduling_next
       ↓
running_iteration
```

Error branches:

```text
permission_requested → paused_permission
provider_error       → retry_backoff | switch_model | paused_error
no_progress          → replan | split_task | rotate_agent
worker_crash         → requeue | fail_item | pause
compaction_failed    → fresh_session_from_ledger | pause
operator_stop        → stopping → stopped
```

Current `SessionStatus` has `idle`, `retry`, and `busy`. It publishes `session.idle` when a session becomes idle. ([GitHub][20])

```text
daemon.created
```

```text
    runID
    iteration
```

Avoid changing `idle` semantics for normal clients.

## 5.5 Modify headless `run.ts`

Current `run.ts` exits when the session status becomes idle. In daemon mode, it must not break on idle; it must wait for `daemon.completed`, `daemon.stopped`, or fatal error. ([GitHub][19])

Add flags:

```text
jekko run --daemon
jekko run --daemon-spec .jekko/daemons/jankurai.yaml
jekko run --daemon-mode forever
jekko run --exit-condition "jankurai audit --json"
jekko run --checkpoint "git add -A && git commit -m 'daemon checkpoint' && git push"
jekko run --compact-every 3
jekko run --clear-context every-item
jekko run --max-workers 4
jekko run --max-worker-depth 1
jekko run --permission-mode pause
jekko run --daemon-ui gold
```

Behavior changes in daemon mode:

```text
session idle → daemon supervisor continues
permission.asked → obey daemon permission_mode
SIGINT once → pause
SIGINT twice → stop
SIGTERM → stop with cleanup
process restart → resume active runs if autoresume=true
```

## 5.7 Context clearing and compaction

Jekko already has configurable compaction and plugin compaction hooks. ([Jekko][13])

Daemon context policy should support three modes:

```text
context.mode: compact
context.mode: clear_session
context.mode: hybrid
```

### `compact`

Same session, force compaction every N iterations or token threshold. Include daemon ledger state in the compaction prompt:

```text
- original daemon goal
- stop criteria
- current queue summary
- active locks
- completed commits
- blocked items
- decisions
- verification history
- next recommended item
```

### `clear_session`

Start a new Jekko session each item or each iteration. Rehydrate from ledger, AGENTS.md, relevant files, and item-specific context. This is often better for large repetitive queues because stale context becomes noise. ([willness.dev][6])

### `hybrid`

Use fresh worker sessions per item, but keep one orchestrator session compacted.

This is the best default for your case.

# 6. Multi-agent collaboration design

## 6.1 Manager/worker model

Use one daemon orchestrator and bounded workers.

```text
orchestrator
  - owns stop criteria
  - owns ledger
  - owns task queue
  - owns git merge/push/checkpoint
  - spawns workers
  - reviews worker results
  - never edits randomly once workers are active unless needed

worker
  - claims one item
  - works in isolated git worktree or branch
  - runs focused verification
  - reports structured result
  - stops
```

This follows the “manager uses specialists as bounded tools” pattern rather than uncontrolled handoffs. OpenAI’s orchestration docs recommend agents-as-tools when a manager should keep ownership and synthesize outputs. ([OpenAI Developers][8])

## 6.2 Isolation

For coding work, default worker isolation should be:

```text
isolation: git_worktree
branch_template: daemon/${run.name}/${worker.id}/${item.id}
lock_paths: true
```

Workers should not share the same dirty worktree unless explicitly configured.

```text
main worktree
  ├─ daemon orchestrator
  ├─ worktree worker-1 → branch daemon/run/item-a
  ├─ worktree worker-2 → branch daemon/run/item-b
  └─ worktree worker-3 → branch daemon/run/item-c
```

Merge flow:

```text
worker commit
  ↓
orchestrator verifies diff
  ↓
orchestrator rebases/merges
  ↓
global verification
  ↓
push
  ↓
mark item done
```

## 6.3 Worker recursion limits

Default:

```text
workers.max_parallel: 3
workers.max_depth: 1
workers.allow_worker_spawning: false
workers.allow_task_tool_in_subagents: false
```

Jekko has real reports of subagents recursively spawning subagents without depth limits, including one issue reporting 47 sessions and 20 nesting levels. Runtime guards are mandatory. ([GitHub][22])

## 6.4 Central chatter / coordination

Replace “Read THIS_TRACKER.md and update it” with:

```text
Daemon Ledger
  ├─ queue
  ├─ claims
  ├─ locks
  ├─ events
  ├─ decisions
  ├─ worker heartbeats
  ├─ verification results
  ├─ commits
  └─ generated human report
```

Expose ledger tools to the orchestrator and workers:

```text
daemon_queue_claim
daemon_queue_complete
daemon_queue_block
daemon_lock_paths
daemon_unlock_paths
daemon_note
daemon_decision
daemon_worker_report
daemon_artifact_add
daemon_status
```

The model should not edit ledger internals directly. It should call structured tools.

# 7. Syntax design: deliberate, gated, impossible to trigger accidentally

## 7.1 Why not XML tags

This is unsafe:

```xml
<FOREVER>
...
</FOREVER>
```

Reasons:

```text
- Could appear in documentation.
- Could appear in copied examples.
- Could be injected by a malicious file.
- Easy for the model to imitate.
- Hard to validate strictly.
- No schema versioning.
- No activation ceremony.
```

XML tags are good for model-readable sections, not for privileged runtime control.

## 7.2 Recommended activation model

A DaemonSpec should run only when all of these are true:

````text
1. The user invoked a privileged command:
   /daemon start
   or jekko run --daemon-spec
   or SDK daemon.start()

2. The spec is a fenced block with exact language:
   ```jekko-daemon-v1

3. The spec starts with:
   OC_DAEMON_SPEC:

4. The spec contains:
   version: 1
   activate: true

5. The spec includes a required activation phrase:
   activation_phrase: "RUN UNTIL STOP CRITERIA OR MANUAL STOP"

6. The spec ends with:
   END_OC_DAEMON_SPEC

7. Optional but recommended:
   TUI-generated nonce or file-based trusted spec.
````

A normal prompt containing this block should not run it. It should only run via `/daemon start` or `--daemon-spec`.

## 7.3 Preferred forms

### Best for repeatable project use

```text
.jekko/daemons/jankurai-cleanup.yaml
```

Invoke:

```text
/daemon start jankurai-cleanup
```

or:

```text
jekko run --daemon-spec .jekko/daemons/jankurai-cleanup.yaml
```

### Best for one-off TUI use

```text
/daemon draft
```

TUI generates:

```text
nonce: ocd_2026_05_07_x7f9
```

Then the user submits:

````text
/daemon start
```jekko-daemon-v1
OC_DAEMON_SPEC:
  version: 1
  activate: true
  nonce: ocd_2026_05_07_x7f9
  activation_phrase: "RUN UNTIL STOP CRITERIA OR MANUAL STOP"
  run:
    name: jankurai-cleanup
    mode: forever
  work:
    goal: "Clean dirty worktree safely, one item at a time."
END_OC_DAEMON_SPEC
```
````

No nonce, no daemon.

# 8. Proposed DaemonSpec for your exact Jankurai case

````yaml
```jekko-daemon-v1
OC_DAEMON_SPEC:
  version: 1
  activate: true
  activation_phrase: "RUN UNTIL STOP CRITERIA OR MANUAL STOP"

  run:
    name: jankurai-dirty-worktree-cleanup
    mode: forever
    title: "Jankurai cleanup daemon"
    max_iterations: null
    heartbeat_every: "60s"
    autoresume: true

  ui:
    badge: "∞ FOREVER"
    color: gold
    show_iteration: true
    show_workers: true
````

# 9. Full syntax library

This is the proposed “god mode” library. The important design choice is that every function below is parsed by Jekko, validated by schema, and executed by the daemon runtime. The LLM may see a readable summary, but it does not interpret the DSL itself.

## 9.1 Gate and activation fields

| Syntax                   | Function                                                                   |
| ------------------------ | -------------------------------------------------------------------------- |
| `OC_DAEMON_SPEC:`        | Required root key. Without this exact key, parser rejects.                 |
| `version`                | Schema version. Start with `1`.                                            |
| `activate: true`         | Required explicit activation.                                              |
| `activation_phrase`      | Required human phrase proving intent.                                      |
| `nonce`                  | Optional TUI-generated one-time activation token.                          |
| `intent_id`              | Optional stable ID for replay/resume.                                      |
| `END_OC_DAEMON_SPEC`     | Required closing sentinel.                                                 |
| `requires_command: true` | Prevents activation unless launched by `/daemon start` or CLI daemon flag. |
| `spec_hash`              | Runtime-computed fingerprint for audit/resume.                             |

## 9.2 Run control

| Syntax                         | Function                                                               |
| ------------------------------ | ---------------------------------------------------------------------- |
| `run.name`                     | Stable daemon run name.                                                |

| `run.mode: until`              | Run until `stop_when`.                                                 |

# 10. Best path to implement in phases

## Phase 0: Plugin prototype, no core changes

This is the fastest way to validate behavior.

Use existing plugin events:

```text
session.idle → evaluate daemon spec → maybe re-prompt
session.compacted → inject ledger context
permission.asked → pause/reject according policy
tool.execute.after → update ledger
```

Jekko plugins already receive these events, and compaction hooks can inject custom context. ([Jekko][13])

Limitations:

```text
- Less durable.
- Harder to resume after crash.
- Harder to integrate TUI gold UI.
- Harder to enforce worker recursion globally.
- Headless run still needs careful wrapper behavior.
```

This is useful as a proof of concept, not the final architecture.

## Phase 1: Core daemon service

Add daemon schema, parser, DB tables, runner, condition evaluator, checkpoint evaluator, and daemon events.

No TUI gold yet. Use CLI and logs.

Acceptance test:

```text
Fake model always emits finish=stop.
Daemon continues 10 iterations.
Stop command returns false first 9 times, true on 10th.
Daemon exits exactly on 10th.
```

## Phase 2: Headless `jekko run --daemon`

Add CLI flags and event loop behavior.

Acceptance test:

```text
jekko run --daemon --exit-condition "test -f done"
assistant stops
daemon continues
touch done
daemon exits
```

## Phase 3: Context policy

Add forced compaction and fresh-session-per-item.

Acceptance test:

```text
compact_every: 1 produces a daemon-aware summary
clear_each_item starts new worker sessions
ledger state is preserved
```

## Phase 4: TUI gold UX

Add banner, status pill, command palette entries, pause/stop controls.

Acceptance test:

```text
Daemon active → gold banner visible
Normal session → no gold overlay
Nested worker session → shows parent daemon and worker ID
```

## Phase 5: Multi-worker orchestration

Add queue, claims, locks, git worktree isolation, worker heartbeat, merge/push.

Acceptance test:

```text
3 workers claim 3 different items
no file lock conflicts
worker crash releases claim after TTL
orchestrator merges successful worker branch
```

## Phase 6: Safety hardening

Add recursion guard, process cleanup, permission policy, no-progress detector, crash resume.

Acceptance test:

```text
worker tries to spawn worker beyond max_depth → denied by runtime
permission request in headless daemon → pauses, not silently rejects unless configured
provider error → retry/backoff
orphan shell process → cleanup on stop
```

# 12. The plan I would ship

My final recommendation is:

```text
1. Do not implement <FOREVER>.
2. Implement OC_DAEMON_SPEC v1 as a schema-validated, gated runtime spec.
3. Add SessionDaemon as an outer supervisor around normal SessionPrompt.prompt().
4. Keep normal prompt-loop finish semantics unchanged.
5. Use daemon.status events instead of overloading session.status at first.
6. Add jekko run --daemon and /daemon start.
7. Use a gold TUI banner for active forever mode.
8. Add durable ledger, not Markdown tracker files, as the coordination source of truth.
9. Add fresh-session-per-item and hybrid context policies.
10. Add worker isolation through git worktrees.
11. Enforce max worker depth and max parallelism in runtime, not prompt.
12. Make permission handling explicit: pause by default.
13. Treat “forever” as no iteration cap, but still bounded by safety controls.
```

The core idea is simple: **normal Jekko remains an agent loop; daemon Jekko becomes a durable workflow supervisor that repeatedly invokes that agent loop.** That is the architecture most aligned with the broader agent ecosystem, Jekko’s existing primitives, and the specific failure reports in the Jekko community.
