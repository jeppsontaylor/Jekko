## My strongest recommendation

Do **not** build this as “XML in a prompt that the model promises to obey.” That will always fail eventually. The only credible way to make “FOREVER” real is to make OpenCode’s **runtime** own the loop, context clearing, checkpointing, stop-condition evaluation, permissions, worker spawning, retries, and UI state.

The prompt syntax should be a **declarative RunBook**: it declares the job, stop criteria, loop policy, context policy, git policy, and multi-agent topology. OpenCode parses that RunBook into a typed AST, arms it only through a deliberate activation gate, then the daemon scheduler enforces it. The model becomes a worker inside a deterministic harness.

That is the key architectural line:

> **The LLM may decide it is done with an iteration. It must not be allowed to decide that the daemon is done.**

OpenCode already has many of the pieces: agents and subagents, MCP integration, custom commands, skills, AGENTS.md rules, headless `opencode run`, session status events, and compaction. But it does **not** appear to have a formal durable daemon/runbook system today. The official docs describe agents, commands, skills, MCP, and repository rules, not an infinite runtime-owned workflow language. ([OpenCode][1])

# 1. What the broader ecosystem is converging on

There is no single universally accepted “agent scripting language” standard yet. The ecosystem is splitting into several layers:

**Project guidance layer:** `AGENTS.md` is becoming the standard place to put repository-specific instructions for coding agents. It is intentionally simple: a predictable README-like file for agents, not a full workflow runtime. It has broad adoption and is explicitly supported by OpenCode. ([Agents][2])

**Tool/context layer:** MCP is the strongest emerging standard for exposing tools, data, and contextual resources to agents. It standardizes how hosts, clients, and servers exchange tools and context over JSON-RPC, and OpenCode already supports MCP servers through its config. ([Model Context Protocol][3])

**Agent-to-agent layer:** A2A is the most relevant emerging standard for independent agents communicating, discovering capabilities, coordinating tasks, and handling long-running or human-in-the-loop work. It is useful conceptually for OpenCode multi-agent orchestration, even if OpenCode does not currently expose A2A as its native worker protocol. ([A2A Protocol][4])

**Runtime/orchestration layer:** LangGraph, CrewAI Flows, Microsoft Agent Framework, and the OpenAI Agents SDK all point in the same direction: serious long-running agents need durable state, explicit control flow, resumability, tool execution outside the model, and app-owned orchestration. LangGraph emphasizes long-running stateful agents and durable execution; CrewAI Flows expose structured workflows with branching and loops; OpenAI’s Agents SDK is explicitly for cases where the application owns orchestration, approvals, storage, tool execution, and conversation strategy. ([LangChain Docs][5])

**Context engineering layer:** the trend is away from “giant prompt with brittle logic” and toward context selection, compression, isolation, artifact storage, and deterministic orchestration. Practitioner writing on DSLs for agents makes the same point bluntly: LLMs are unreliable routers for loops and transitions, so the harness should own deterministic control flow. ([DEV Community][6])

The conclusion is clear: an OpenCode “forever” feature should look less like `<FOREVER>...</FOREVER>` and more like a typed workflow manifest plus a runtime daemon.

# 2. What I found specific to OpenCode

OpenCode already has the right foundation, but not the final daemon abstraction.

OpenCode’s CLI starts the TUI by default, and `opencode run "..."` supports headless/programmatic execution. That is the natural entry point for daemon mode in CI, servers, and automation. ([OpenCode][7])

OpenCode agents are already differentiated into build, plan, general, explore, and internal utility agents. The docs say the general subagent can run parallel units of work, explore is read-only, and internal agents already exist for compaction, title generation, and summarization. This is exactly the kind of substrate a daemon supervisor should reuse. ([OpenCode][1])

OpenCode commands already support markdown files with YAML front matter, `$ARGUMENTS`, positional arguments, shell interpolation with `!command`, and file references with `@filename`. That is useful for user ergonomics, but it is not enough by itself for safe forever execution because command/frontmatter syntax is too easy to accidentally paste or trigger. ([OpenCode][8])

OpenCode’s MCP docs explicitly warn that MCP tools add to context, and enabling many of them can add up. That matters for daemon mode: a daemon must not blindly expose every MCP tool to every worker forever. It needs lazy MCP capability exposure, per-worker tool scopes, and context budgets. ([OpenCode][9])

The current codebase already has a `SessionPrompt` service with methods such as `prompt`, `loop`, `shell`, `command`, and prompt-part resolution, and that service is wired to Bus, SessionStatus, Session, Agent, Provider, SessionProcessor, SessionCompaction, Command, Config, Permission, MCP, LSP, ToolRegistry, ChildProcessSpawner, SystemPrompt, and LLM services. That means the daemon should not be a separate bolt-on script; it should be a first-class session service built beside the existing prompt/session system. ([GitHub][10])

The prompt pipeline already resolves native tools and MCP tools, and MCP tool execution goes through the permission system. That is important: daemon mode must integrate with permissions rather than bypass them accidentally. ([GitHub][10])

Compaction is already sophisticated. The compaction summary template tracks goal, constraints, progress, decisions, next steps, critical context, and relevant files. The compaction service exposes `isOverflow`, `prune`, `process`, and `create`, with `process` returning `"continue"` or `"stop"`. It also already has auto-continue logic that creates a synthetic user message after compaction. ([GitHub][11])

Headless `run.ts` currently treats `session.status: idle` as the signal to break out and exit. In daemon mode, `idle` must become an **iteration boundary**, not a process exit. ([GitHub][12])

There are also OpenCode issues that cut both ways. Some users have hit unwanted infinite loops caused by finish-reason handling bugs, while others want better auto-continue behavior when generation stops due to length. That means daemon mode must be explicit and visually obvious, not an accidental consequence of loose loop handling. ([GitHub][13])

Permissions are a major risk area. OpenCode headless mode currently auto-approves only with `--dangerously-skip-permissions`; otherwise permission prompts can reject or block. There is also an OpenCode issue about serve/attach sessions hanging when permission prompts are not surfaced correctly. A daemon cannot safely “just continue” through permission failures; it needs explicit allowlists, denylists, block states, and UI/headless reporting. ([GitHub][12])

I did not find an official OpenCode daemon/runbook/infinite-agent scripting standard. The reliable public signal is OpenCode’s existing architecture and GitHub issues around loops, auto-continue, compaction, headless behavior, permissions, and community experiments with agent swarms. ([GitHub][14])

# 3. Critical review of the proposed plan

The proposed plan has the right instinct, but it is too prompt-loop-centric and not durable enough.

## What the proposal gets right

It correctly identifies that normal agents stop because the model emits a terminal finish reason and no more tool calls. For ordinary chat, that is correct. For daemon mode, that should mean “iteration finished,” not “job finished.”

It correctly says OpenCode should leverage existing compaction rather than inventing context clearing from scratch. OpenCode already has compaction, pruning, summary generation, and auto-continue hooks. ([GitHub][11])

It correctly identifies `run.ts` as a required modification point because headless mode currently exits when the session becomes idle. ([GitHub][12])

It correctly proposes a visible TUI state. A forever daemon must be impossible to confuse with a normal chat session.

## Where the proposal is weak

The biggest flaw is putting too much responsibility inside `prompt.ts`. A giant outer `while (daemon.active)` wrapped around `prompt()` will work for demos, but it will be fragile across crashes, retries, permission prompts, tool failures, and duplicated side effects.

Daemon mode needs a **durable job scheduler**, not only a prompt-loop override.

The second flaw is “override processor stop.” Permission denial, blocked state, and LLM errors are not all the same thing. A permission denial should not be silently ignored, and it absolutely should not auto-approve unless the user configured that policy explicitly. Existing OpenCode headless permission behavior already shows why this needs a first-class design. ([GitHub][12])

The third flaw is treating checkpointing as “run this shell command every iteration.” A daemon that commits and pushes must have preconditions: changed files exist, verification passed, no forbidden files changed, audit output is acceptable, commit message is generated from evidence, and push policy is explicit.

The fourth flaw is YAML front matter as the primary syntax. YAML front matter is good for commands, but dangerous for “forever.” The RunBook must require a deliberate gate, nonce, and activation command so accidental pasted text cannot arm an infinite loop.

The fifth flaw is “spin up MCPs through prompt injection.” MCP is a tool/context protocol, not a complete multi-agent scheduler. The daemon should support MCP servers, but worker orchestration should be runtime-owned with explicit worker pools, leases, joins, timeouts, context isolation, and blackboard state.

The sixth flaw is compaction every N iterations as a blunt instrument. A better policy is token-budget based plus phase-based compaction: compact after verification, after checkpoint, before worker fan-out, and before rehydrating the next task. Also, OpenCode has had compaction/tool-result consistency issues in long tool-heavy sessions, so daemon compaction must validate tool-call/tool-result invariants before saving a compacted message history. ([GitHub][15])

# 4. The architecture I would build

## Core model

OpenCode should introduce a first-class concept:

> **Daemon Job**: a durable, inspectable, resumable workflow attached to a session, controlled by a typed RunBook, running iterations until stop criteria pass or the user/process stops it.

The daemon is not the model. The daemon is the supervisor.

A daemon job owns:

* the original objective;
* the parsed RunBook AST;
* loop policy;
* stop criteria;
* context policy;
* git/checkpoint policy;
* permission policy;
* worker topology;
* active leases;
* iteration history;
* verification evidence;
* commit history;
* compaction summaries;
* status events;
* failure state.

The model receives a generated iteration prompt such as:

> “You are worker `fixer-2` in daemon job `jankurai-cleanup`. You own task lease `T-018` until timestamp X. Your allowed tools are Y. Your stop condition is not for you to decide. Complete one unit, produce evidence, and return control.”

The model can stop whenever it wants. The daemon then evaluates deterministic state and either exits or schedules the next iteration.

# 6. Detailed OpenCode implementation plan

## 6.1 New service: `session/daemon.ts`

Add a new daemon service beside the existing session/prompt/status/compaction services.

Conceptual API:

```ts
DaemonService.create(sessionID, runbook): Effect<DaemonJob>
DaemonService.start(jobID): Effect<void>
DaemonService.pause(jobID, reason): Effect<void>
DaemonService.resume(jobID): Effect<void>
DaemonService.stop(jobID, reason): Effect<void>
DaemonService.tick(jobID): Effect<DaemonTickResult>
DaemonService.status(jobID): Effect<DaemonStatus>
DaemonService.appendEvent(jobID, event): Effect<void>
DaemonService.evaluateStop(jobID): Effect<StopResult>
DaemonService.runCheckpoint(jobID, iterationID): Effect<CheckpointResult>
DaemonService.forceCompact(jobID, reason): Effect<void>
```

Do **not** make `prompt.ts` the owner of `while true`. `prompt.ts` should remain the owner of one assistant interaction loop. `DaemonService` should own the larger durable workflow.

## 6.2 Durable storage

Add durable state. Do not store daemon state only in memory or a loose session metadata blob.

Minimum tables/entities:

```text
  worker_policy_json

daemon_iteration

  job_id
  iteration_number
  status

```

This gives you resumability, observability, and idempotency. Without this, crashes can duplicate commits, re-run destructive commands, or lose why the agent made a decision.

## 6.3 RunBook parser

Add:

```text
packages/opencode/src/runbook/schema.ts
packages/opencode/src/runbook/parser.ts
packages/opencode/src/runbook/validate.ts
packages/opencode/src/runbook/digest.ts
packages/opencode/src/runbook/render.ts
```

Use OpenCode’s existing schema style, likely Effect Schema where consistent with the codebase. The parser should produce a typed AST. The daemon should never execute raw YAML directly.

Validation should reject:

* missing activation gate;
* missing nonce;
* mismatched begin/end nonce;
* `forever: true` without explicit authorization;
* shell commands outside permission policy;
* checkpoint without verification preconditions;
* push without explicit push policy;
* worker count above configured maximum;
* MCP server not configured;
* stop condition that is only “assistant says done” unless paired with deterministic evidence;
* context clearing that drops required invariants.

## 6.4 Prompt integration

Modify `packages/opencode/src/session/prompt.ts` minimally.

Recommended changes:

1. Add a daemon-aware continuation policy hook, not a giant embedded daemon loop.
2. Expose a cleaner “prompt one iteration and return final state” internal API if the existing `prompt`/`loop` API does not already make that easy.
3. When the assistant reaches a terminal finish reason in daemon mode, publish an iteration-finished event instead of treating that as daemon completion.
4. Handle `finish_reason: length` separately: auto-continue may be valid when explicitly configured, matching the OpenCode feature request around length stops. ([GitHub][16])
5. Handle `finish_reason: unknown` defensively. There is an OpenCode issue where `unknown` can cause accidental infinite looping when no tools are called. In daemon mode, `unknown` should be classified through explicit policy, not blindly treated as “continue.” ([GitHub][17])

The daemon should inject generated prompts like:

```text
<opencode-daemon-runtime>
Job: jankurai-cleanup
Iteration: 17
Worker: supervisor
Stop condition: runtime-owned; do not claim final completion unless evidence is produced.
Current lease: T-019
Allowed actions: inspect diff, run tests, edit files, update blackboard, request checkpoint.
Required output: evidence summary and next recommended daemon action.
</opencode-daemon-runtime>

User objective:
...
```

This runtime block is not user syntax. It is generated by OpenCode after parsing the RunBook.

## 6.5 Compaction integration

Modify `packages/opencode/src/session/compaction.ts` to support daemon-aware compaction reasons:

```text
overflow
manual
daemon_iteration
daemon_checkpoint
daemon_worker_reset
daemon_before_fanout
```

Extend the compaction summary template with daemon-specific fields:

```text
Daemon Job
- job id
- invariant objective
- stop criteria
- current iteration
- current task lease
- active workers
- completed tasks
- failed/retried tasks
- latest verification evidence
- latest commit sha
- open risks
- permission blocks
- files touched
- commands already run
- exact next daemon action
```

Important: validate the provider message sequence after compaction. Long tool-heavy sessions have had issues where compaction can leave inconsistent tool-use/tool-result state, causing provider errors. ([GitHub][15])

The daemon should support both:

```yaml
context:
  compaction:
    policy: overflow
```

and:

```yaml
context:
  compaction:
    policy: phase
    after:
      - checkpoint
      - every_iteration
      - before_worker_fanout
```

For true forever mode, I would default to compacting after every successful checkpoint and whenever the model reaches 70–80% of the context budget.

## 6.6 Status integration

Extend session status.

Today, headless mode listens for `session.status` and exits when status becomes `idle`. ([GitHub][12])

Daemon mode needs status like:

```ts
type DaemonSessionStatus = {
  type: "daemon"
  state:
    | "armed"
    | "running"
    | "prompting"
    | "waiting"
    | "validating"
    | "checkpointing"
    | "compacting"
    | "blocked"
    | "paused"
    | "satisfied"
    | "cancelled"
    | "failed"
  jobID: string
  iteration: number
  forever: boolean
  activeWorkers: number
  stopSummary: string
  lastCheckpoint?: string
  lastError?: string
}
```

For compatibility, normal sessions should still emit `idle`. Daemon sessions should not emit plain `idle` as a process-level terminal state while the job remains active. They can emit `daemon.state: "waiting"` or `daemon.state: "iteration_idle"`.

## 6.7 Headless CLI changes

Modify `packages/opencode/src/cli/cmd/run.ts`.

Add flags:

```text
--daemon
--daemon-runbook <path>
--daemon-inline
--daemon-name <name>
--exit-condition <command>
--checkpoint <command>
--compact-policy <policy>
--compact-every <n>
--max-iterations <n|inf>
--max-runtime <duration|inf>
--max-parallel-workers <n>
--permission-policy <path>

```

Critical behavioral change:

```text
normal run:
  session.status idle => exit process

daemon run:
  session.status idle => iteration boundary
  daemon.status satisfied/cancelled/failed => exit process
```

Permissions should be explicit. OpenCode already has `--dangerously-skip-permissions`; daemon mode should not silently expand its meaning. For daemon, support:

```yaml
permissions:
  mode: allowlist
  headless_on_ask: block_job
  allow:
    shell:
      - "git status*"
      - "git diff*"
      - "git diff --check*"
      - "pnpm test*"
      - "jankurai audit*"
  ask:
    shell:
      - "git push*"
  deny:
    shell:
      - "rm -rf *"
      - "curl * | sh"
```

In headless mode, `ask` should either block the daemon with a visible status or require `--dangerously-skip-permissions` plus an explicit allow rule. The current headless permission behavior makes this a required design point. ([GitHub][12])

## 6.8 TUI changes: gold “FOREVER” UX

Do not mutate the user’s theme globally. Add daemon-specific theme tokens and apply them only when the active session has daemon status.

TUI behavior:

```text
╭────────────────────────────────────────────────────────────╮
│  FOREVER                                                   │
│  ♾  DAEMON  jankurai-cleanup  iter 17  workers 3/4         │
│  stop: jankurai audit => 0 findings, 0 caps                │

```

Requirements:

* gold border/accent;
* text label “FOREVER” so color is not the only signal;
* visible iteration count;
* visible stop condition;
* visible active workers;
* visible pause/stop affordance;
* warning state for blocked permissions;
* compact activity indicator;
* final “satisfied” state in green/gold rather than silently disappearing.

Use a small persistent header/pill in the session route and daemon details panel. The gold overlay should be status-driven.

Add API endpoints so TUI, CLI, and remote clients can control daemon jobs:

Regenerate SDK types after schema changes.

## 6.10 Multi-agent architecture

Use a **supervisor + worker pool + blackboard** model.

Do not rely on each agent reading and editing one markdown tracker. Markdown can be a human-readable export, not the coordination primitive.

The blackboard should hold:

```text
tasks
claims
leases
workers
findings
decisions
evidence
artifacts
file_locks
commits
verification_runs
messages
risks
```

Workers claim tasks through leases:

```text
claim task T-021 for worker fixer-2
lease ttl: 20 minutes
files locked: src/a.ts, tests/a.test.ts
heartbeat every: 60 seconds
on timeout: release lease and mark stale
```

Join policies:

```yaml
agents:
  topology: supervisor_workers
  supervisor:
    agent: build
  workers:
    - id: explorers
      agent: explore
      count: 3
      readonly: true
      join: all
    - id: fixers
      agent: general
      count: 2
      write_policy: branch_per_worker
      join: quorum
```

OpenCode’s existing general/explore subagents are already close to this: general can run parallel work, explore is read-only, and build/plan have different tool policies. The daemon should formalize this rather than relying on ad hoc tracker files. ([OpenCode][1])

For MCPs:

```yaml
mcp:
  expose:
    - name: github
      to: supervisor
    - name: linear
      to: supervisor
    - name: browser
      to: explorers
  lazy: true
  max_tools_per_worker: 12
```

Because OpenCode warns that MCP tools consume context, daemon mode should expose MCP tools by role and phase, not globally forever. ([OpenCode][9])

# 7. The syntax: OpenCode RunBook v1

I would not use XML tags like `<FOREVER>`. They are too easy to paste accidentally and too easy for the model to imitate.

Use a deliberately gated block format:

```text
/daemon arm
<<<OPENCODE_RUNBOOK_V1:BEGIN nonce="2026-05-07-jankurai-9f31" >>>
...
<<<OPENCODE_RUNBOOK_V1:END nonce="2026-05-07-jankurai-9f31" >>>
```

Then execution requires one of:

```text
/daemon start nonce=2026-05-07-jankurai-9f31
```

or:

```text
opencode run --daemon-runbook ./jankurai.ocrb.yml --yes-forever
```

This gives you three safety gates:

1. the user must invoke `/daemon` or `--daemon`;
2. the block must use the exact reserved RunBook sentinel;
3. begin/end nonces must match, and `forever: true` requires explicit authorization inside the manifest.

Normal pasted text containing RunBook-looking content should be inert.

```yaml
        findings_jsonpath: "$.findings[*]"

  topology: supervisor_workers
  supervisor:
    agent: build
  workers:
    - id: explorers
      agent: explore
      count: 2
      readonly: true
      purpose: "Inspect diffs, identify risks, propose task claims."
    - id: fixers
      agent: general
      count: 2
      write_policy: branch_per_worker
      purpose: "Fix one leased task at a time and produce evidence."

context:
  compaction:
    policy: phase_or_threshold
    threshold_percent: 75
    after:
      - checkpoint
      - every_iteration
    preserve:
      - objective
      - invariants
      - stop
      - current_task
      - active_leases
      - audit_findings
      - completed_commits
      - failed_attempts
      - verification_evidence
      - files_touched

  mode: allowlist
  headless_on_ask: block_job
  shell:
    allow:
      - "git status*"
      - "git diff*"
      - "git diff --check*"

      - "git push*"
```

# 9. Better multi-agent collaboration than tracker markdown

The current “read THIS_TRACKER.md and update it” pattern fails because markdown is not a concurrency primitive. It has no leases, no atomic updates, no heartbeats, no conflict detection, and no structured evidence.

Replace it with this:

```text
.opencode/daemon/<jobID>/state.sqlite   authoritative state
.opencode/daemon/<jobID>/events.jsonl   append-only event stream
.opencode/daemon/<jobID>/artifacts/     diffs, logs, reports
JANKURAI_TASKLIST.md                    rendered human view
JANKURAI_TASKLIST.md.completed          rendered completed view
```

Workers interact with the blackboard through daemon tools:

```text
daemon.claim_task
daemon.release_task
daemon.heartbeat
daemon.lock_files
daemon.append_finding
daemon.append_evidence
daemon.submit_patch
daemon.request_review
daemon.mark_blocked
daemon.mark_complete
```

The markdown files become exported views. Humans can still read them, but the daemon does not depend on fragile text editing for coordination.

Best worker flow:

```text
1. Supervisor refreshes blackboard from git diff and audit.
2. Explorer workers inspect dirty files read-only.
3. Supervisor converts observations into tasks.
4. Fixer workers claim non-overlapping tasks with file locks.
5. Fixers work in isolated branches/worktrees when possible.
6. Review worker or supervisor validates changes.
7. Merge queue applies one verified patch at a time.
8. Checkpoint commits and pushes.
9. Blackboard records commit SHA and evidence.
10. Daemon compacts context and loops.
```

This is much closer to how durable agent frameworks treat long-running work: explicit state, explicit control flow, and deterministic coordination instead of hoping the model remembers the tracker. ([LangChain Docs][5])

# 12. The most important design rules

1. **Runtime owns forever.** The model can end an iteration, not the daemon.

2. **RunBook is declarative.** It describes policy. It is not a free-form prompt the model interprets.

3. **Activation must be deliberate.** `/daemon` plus nonce plus explicit phrase for forever jobs.

4. **Stop criteria must be deterministic.** Shell, git, JSON, file, blackboard, or audit checks. Model judgement can assist, not decide.

5. **Context clearing must preserve invariants.** The daemon objective, stop criteria, leases, evidence, and current task must survive compaction.

6. **Permissions must be first-class.** Infinite loops plus shell plus git push is too dangerous without allow/ask/deny policy.

7. **Checkpoint only verified work.** Commit/push is a phase with preconditions, not a generic command run every loop.

8. **Multi-agent needs leases, not markdown.** Use a blackboard database and render markdown for humans.

9. **MCP exposure must be scoped.** Do not dump every MCP tool into every worker forever.

10. **Gold UI is mandatory for forever.** The user should always know when a daemon is running.

# 13. Final recommended product shape

I would ship this in three stages.

## Stage 1: Safe single-agent daemon

* `/daemon` command.
* RunBook parser with nonce gate.
* Durable daemon job state.
* Headless `--daemon`.
* Runtime loop on idle.
* Stop conditions.
* Forced daemon compaction.
* Git checkpoint policy.
* Gold TUI status.

This solves your immediate “agents always stop” problem.

## Stage 2: Blackboard and worker pools

* Durable blackboard.
* Task leases.
* File locks.
* Worker pools using existing OpenCode agents/subagents.
* Join policies.
* Markdown tracker import/export.

This solves multi-agent collaboration.

## Stage 3: Advanced orchestration

* MCP role/phase scoping.
* Branch/worktree-per-worker.
* Merge queue.
* A2A-compatible worker protocol option.
* Remote daemon control API.
* RunBook registry and signed RunBooks.
* CI/server mode.

This becomes the “God mode” layer: durable infinite workflows, context clearing, multi-agent coordination, MCPs, verification, checkpointing, and visual control.

The central correction to the original plan is this: **do not make OpenCode’s core prompt loop pretend to be a daemon. Add a daemon scheduler above it, make the syntax deliberately gated, and make every loop transition deterministic, durable, inspectable, and permissioned.**
