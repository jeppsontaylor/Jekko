I did **not** code or change anything. I treated this as a research, design, and critique pass.

## Bottom line

The right answer is **not** “teach the model to obey `<FOREVER>`.” The right answer is to add a **deterministic OpenCode runtime control plane** that parses a deliberately gated agent-script block, stores it as durable daemon state, and treats the model’s natural `finish: "stop"` as only an **iteration boundary**, not a stop signal.

A strong OpenCode implementation should split the system into two layers:

1. **Agent Script Runtime**: deterministic, durable, auditable, permission-aware, and responsible for loops, exit checks, budgets, compaction, checkpoints, worker spawning, blackboard state, and TUI status.
2. **LLM prompt layer**: receives only the current task, iteration context, ledger, constraints, and next work instructions. It should never be trusted to enforce “forever.”

That separation is the only way to get something close to your “PROMISED infinite loop” requirement without confusing normal prompts, pasted examples, or accidental XML tags.

# 1. What exists today: standards, community signals, and OpenCode-specific context

## There is no dominant “in-prompt executable agent language” standard yet

The ecosystem has emerging standards, but they are layered, not one unified “agent scripting language”:

| Layer                     | Emerging standard / pattern                                          | What it covers                                                    | What it does **not** cover                   |
| ------------------------- | -------------------------------------------------------------------- | ----------------------------------------------------------------- | -------------------------------------------- |
| Project instructions      | `AGENTS.md`, skills                                                  | Stable project guidance, setup, tests, conventions                | Runtime loops, daemon state, exit conditions |
| Tool/data access          | MCP                                                                  | Standard connection to tools, data, workflows                     | Agent-agent coordination or loop control     |
| Agent-agent communication | A2A                                                                  | Inter-agent tasks, artifacts, long-running coordination           | Local coding-agent runtime semantics         |
| Agent-user UI             | AG-UI                                                                | Streaming, events, interrupts, shared state, frontend interaction | Coding daemon loop execution                 |
| Runtime orchestration     | LangGraph, CrewAI Flows, OpenAI Agents SDK, AutoGen-style frameworks | Deterministic loops, parallelism, state, checkpoints              | Native OpenCode prompt syntax                |
| Prompt structure          | XML/YAML/Markdown delimiters                                         | Helps models parse instructions                                   | Does not enforce behavior                    |

`AGENTS.md` is currently one of the clearest cross-agent “instruction file” standards: it describes itself as a simple open format for guiding coding agents and is used by many projects; it is also explicitly compatible with OpenCode in the listed ecosystem. It is good for **static guidance**, not daemon control. ([Agents][1])

MCP is the strongest standard for tool/data connection. The official MCP docs define it as an open-source standard for connecting AI applications to external systems, including data sources, tools, and workflows. OpenCode already supports local and remote MCP servers, but its own docs warn that enabling many MCP servers can add substantial context and exceed context limits. ([Model Context Protocol][2])

A2A is the closest standard for agent-agent interoperability. Google announced it on April 9, 2025, describing it as an open protocol that lets agents communicate, exchange information, coordinate actions, discover capabilities, manage long-running tasks, and exchange artifacts; Google also explicitly says A2A complements MCP rather than replacing it. ([Google Developers Blog][3])

AG-UI is the strongest emerging event protocol for agent-to-user interaction. Its docs position AG-UI as the agent-user protocol, MCP as agent-tools/data, and A2A as agent-agent. That three-layer split is exactly the mental model OpenCode should adopt. ([AG-UI][4])

OpenAI’s Agents SDK docs make the same architectural point from a runtime perspective: an agent loop continues while tools or handoffs are active and ends when the model produces final output with no tool calls; for more deterministic behavior, the docs recommend orchestrating via code, including while loops, evaluator loops, and parallel agents. ([OpenAI GitHub][5])

LangGraph and CrewAI Flows show where the broader ecosystem is going: graph/state/flow engines with durable checkpoints, event-driven state, human-in-the-loop, and explicit control flow. LangGraph’s docs emphasize durable execution, persistence, checkpoints, human-in-the-loop, and fault-tolerant execution; CrewAI Flows describes structured, event-driven workflows that manage state and control execution. ([LangChain Docs][6])

**Conclusion:** OpenCode should not invent a prompt-only XML dialect and hope models obey it. It should implement an OpenCode-native **Agent Script Language** that lives inside a prompt only as a deliberate activation artifact, then compiles into runtime state.

---

## What the OpenCode community has already discussed

There are strong OpenCode-specific signals that people want exactly this class of feature, but also strong warnings from compaction-loop bugs.

The official docs already show that OpenCode has primary agents, subagents, automatic compaction, `@`-mentioned subagents, custom commands, MCP tools, permissions, and plugin compaction hooks. OpenCode’s built-in `general` subagent is described as useful for parallel work, and the hidden `compaction` agent automatically compacts long context. ([OpenCode][7])

OpenCode’s ecosystem page lists community projects that are already orbiting this problem space: background agents, scheduling, conductor/workflow systems, dynamic context pruning, PTY background processes, workspace/worktree orchestration, and command orchestration with granular flow control. That means people are building “daemon / background / multi-agent / long-running” capability around OpenCode even though core OpenCode does not yet expose a first-class daemon scripting runtime. ([OpenCode][8])

OpenCode issue #5408, opened December 11, 2025, asks for a delayed queue feature so repeated messages can wait until OpenCode is waiting for user input; the author specifically describes looping a request that captures learnings, updates an ADR, and runs overnight or while away. ([GitHub][9])

OpenCode issue #11570, opened February 1, 2026, explicitly proposes “daemon, heartbeat, memory” as OpenCode-ready features, including a daemon to handle heartbeat/auto-wake, memory/hooks for long-term memory, and plugin extensibility. ([GitHub][10])

OpenCode issue #9446, opened January 19, 2026 and closed as not planned, proposed an “opencode-council-swarm” architecture with parallel agent execution, consensus, specialized roles, conflict resolution, a task queue, token budget management, deadlock prevention, and agent spawn/despawn lifecycle. Even though it was not adopted, it is highly relevant design input. ([GitHub][11])

There are also bug reports showing why accidental infinite loops are dangerous. Issue #15533 describes auto-compaction injecting a synthetic “Continue…” message after the assistant had naturally ended its turn, creating an unbounded compaction loop. Issue #19410 reports a compaction loop that kept producing huge logs and could fill disk space quickly. ([GitHub][12])

Outside GitHub, public discussion points in the same direction: Reddit search results show OpenCode users reporting subagents stuck in infinite loops and exhausting usage limits; Hacker News discussions around Oly and Stoneforge describe PTY daemons, dispatch daemons, worktrees, reviewers, and long-running agent sessions; LinkedIn posts discuss `/loop`-style agents that iterate on CI failures, fix conflicts, and push changes. These are not formal standards, but they show the same need: durable autonomous loops plus isolation, status, and review. ([Reddit][13])

**Conclusion:** OpenCode should add a first-class daemon/runbook system, but it must be opt-in, visible, permission-aware, observable, and killable. Otherwise it will look indistinguishable from the infinite-loop bugs users already complain about.

# 2. Critical review of the proposed plan

The proposed plan gets the broad direction right but misses several important architectural details.

## What the proposed plan gets right

The root cause analysis is basically correct. In current OpenCode, the prompt loop exits when the last assistant message has a finish reason that is not `tool-calls`, there are no pending tool calls, and the last user message predates the assistant. That is exactly the normal “model says it is done” path. ([GitHub][14])

The plan is also right that OpenCode already has compaction machinery. Current compaction supports `isOverflow`, `prune`, `process`, and `create`, uses a summary agent, supports plugin customization through `experimental.session.compacting`, and can inject a synthetic “Continue if you have next steps…” after automatic compaction. ([GitHub][15])

It is correct that headless `opencode run` exits on `session.status` becoming `idle`, and that daemon mode needs to change that behavior. The current run command’s event loop breaks on `session.status` with `status.type === "idle"`. ([GitHub][16])

It is correct that OpenCode’s current `SessionStatus` union has only `idle`, `retry`, and `busy`, so the TUI/headless clients currently have no first-class way to distinguish “idle because the session is done” from “idle between daemon iterations.” ([GitHub][17])

It is correct that the TUI should have an unmistakable visual state. A gold “FOREVER” banner is not cosmetic; it is safety UX.

---

## What the proposed plan gets wrong or underspecifies

### 1. Do not simply override `runLoop()` to keep going

The proposed plan focuses on modifying `prompt.ts` so that when the model returns `finish: "stop"`, OpenCode injects another prompt and continues. That is tempting, but it risks corrupting normal session semantics.

Better: leave the normal inner agent loop mostly intact. Add an **outer daemon orchestrator** that observes when a normal prompt run ends, then decides whether to requeue. In other words:

```text
normal runLoop:
  user message -> model/tools loop -> assistant stop -> idle

daemon orchestrator:
  observe idle
  checkpoint
  evaluate exit
  compact if needed
  re-prompt if not done
```

That keeps normal OpenCode behavior stable and makes daemon mode a runtime feature, not a hacked prompt-loop behavior.

### 2. Session daemon state must be durable

The proposed plan suggests storing daemon config in metadata or KV. That is not enough for a process that may run overnight, survive TUI reconnects, attach from another client, or resume after restart.

Daemon state should be persisted as first-class state:

```text
session_daemon
  daemon_id
  session_id
  script_id
  script_hash
  config_json
  status
  phase
  iteration
  active
  paused
  created_at
  updated_at
  last_prompt_message_id
  last_assistant_message_id
  last_checkpoint_result
  last_exit_result
  token_count_total
  cost_total
```

A true “forever” runner should also have an optional `resume_policy: on_server_start`, otherwise a server crash still silently ends the promise.

### 3. Do not replace `busy` with `daemon`; add a daemon overlay

The proposed plan says to add `type: "daemon"` to `SessionStatus`. That is not sufficient because the session can be both “busy” and “daemon.” The current status union is activity state: `idle`, `busy`, or `retry`. Daemon is not just activity; it is **mode/lifecycle overlay**. ([GitHub][17])

Better design:

```text
SessionStatus:
  idle | busy | retry

SessionDaemonStatus:
  inactive | active | paused | stopping | terminal
  phase: working | checkpoint | exit_check | compacting | sleeping | requeueing
  iteration
  script_id
  forever
```

The TUI combines them:

```text
activity: busy
daemon: active / forever / iteration 17 / phase working
```

This avoids breaking existing clients that expect only `idle`, `busy`, and `retry`.

### 4. Permission denials must not be auto-ignored

The proposed plan says blocked permission should log and continue. That is dangerous. Current `opencode run` has a `--dangerously-skip-permissions` flag and otherwise auto-rejects permission requests in headless mode. ([GitHub][16])

Daemon mode needs explicit permission policy:

```yaml
permission_policy:
  default: stop
  on_permission_request: pause_and_notify
  allow:
    bash:
      - "git status*"
      - "git diff*"
      - "jankurai audit*"
  ask:
    bash:
      - "git push*"
  deny:
    bash:
      - "rm -rf*"
```

A forever daemon with automatic `git push` is powerful enough that permissions must be part of the script, not an afterthought.

### 5. “Multiple MCPs” is not the same as “multiple agents”

OpenCode supports MCP tools, and MCP servers become tools available to the LLM. But MCP is a tools/data protocol, not inherently an agent-worker orchestration protocol. OpenCode’s own MCP docs say MCP tools are automatically available alongside built-in tools, but they also warn about context growth. ([OpenCode][18])

For multi-agent work, OpenCode should use:

```text
OpenCode subagents / TaskTool     -> local worker agents
MCP tools                         -> external tools/data/workflows
A2A adapters, future              -> external remote agents
```

A daemon DSL should support all three, but it must not assume “MCP spawning” exists unless a configured MCP server provides that specific tool.

### 6. YAML front matter alone is not safe

The proposed plan’s YAML front matter syntax is too easy to trigger accidentally. A user could paste a blog post, issue body, or docs snippet containing daemon YAML and accidentally activate it.

Daemon activation should require a slash command or CLI flag plus a sentinel block with a matching end marker.

# 3. Best architecture for OpenCode today

## Design principle

**The model may choose actions; the runtime must choose continuation.**

That means:

```text
LLM controls:
  which dirty item to inspect
  what code to read
  what edits to propose
  what tests to run
  when it believes a unit of work is complete

Daemon runtime controls:
  whether the overall process continues
  whether exit criteria are satisfied
  when context is cleared
  when checkpoint commands run
  when workers spawn
  when workers are cancelled
  when permissions/budgets/errors pause or stop the process
```

This turns “forever” from a prompt wish into an actual state machine.

---

## Proposed state machine

```text
┌────────────┐
│  CREATED   │
└─────┬──────┘
      │
      ▼
┌───────────────────┐
│ RUNNING_ITERATION │  normal OpenCode prompt loop
└─────┬─────────────┘
      │ model stops / inner loop idle
      ▼
┌───────────────────┐
│  COLLECT_RESULTS  │  summarize last assistant/tool state
└─────┬─────────────┘
      ▼
┌───────────────────┐
│    CHECKPOINT     │  git/tasklist/ledger hooks
└─────┬─────────────┘
      ▼
┌───────────────────┐
│    EXIT_CHECK     │  deterministic shell/json/jq/file/git checks
└─────┬─────────────┘
      │
      ├── satisfied ───────► SATISFIED
      │
      ▼
┌───────────────────┐
│  COMPACT_CONTEXT  │  forced or token-triggered
└─────┬─────────────┘
      ▼
┌───────────────────┐
│     REQUEUE       │  inject next iteration prompt
└─────┬─────────────┘
      ▼
RUNNING_ITERATION

Terminal states:
  SATISFIED
  MANUAL_STOP
  PAUSED_FOR_PERMISSION
  PAUSED_FOR_BUDGET
  ERROR_STOP
  PROCESS_KILLED
```

For a “PROMISED forever” loop, `finish: "stop"` from the model is not terminal. It is just the transition from `RUNNING_ITERATION` to `COLLECT_RESULTS`.

## Prompt-loop integration: best approach

Current `prompt.ts` has a normal `while (true)` loop and exits when the assistant finishes without pending tool calls. That should remain the normal behavior. ([GitHub][14])

Add daemon orchestration **outside** it:

```text
SessionDaemon.start(script)
  persist daemon config
  set daemon overlay: active, iteration 0
  enqueue first prompt

SessionPrompt.prompt(...)
  existing behavior
  existing runLoop exits normally

SessionDaemon.onSessionIdle(sessionID)
  if no active daemon: do nothing
  if active daemon:
    collect last result
    run checkpoint hooks
    evaluate exit conditions
    if satisfied: terminal SATISFIED
    else maybe compact
    enqueue next synthetic daemon prompt
```

This means the model can stop every iteration and OpenCode still continues.

---

## Daemon prompt injection format

The model should not receive the full executable script every time. It should receive a small, consistent, non-executable runtime context:

```text
<opencode-daemon-runtime>
mode: daemon
script_id: jankurai-worktree-cleanup
iteration: 17
forever: true
runtime_rule: The OpenCode runtime, not you, decides whether the daemon exits.
current_phase: work
exit_condition_summary: jankurai audit must report 0 findings and 0 caps.
checkpoint_summary: commit/push completed work when verified.
ledger_file: .opencode/daemon/jankurai-worktree-cleanup/ledger.jsonl
</opencode-daemon-runtime>

<current-task>
...
</current-task>

<last-iteration-summary>
...
</last-iteration-summary>

<next-action-policy>
Pick one unclaimed dirty-worktree item, verify it, update the ledger, and finish the iteration when that unit is done.
</next-action-policy>
```

The tags are for model clarity, not execution. The executable part is already persisted in daemon state.

---

## Context clearing / compaction plan

OpenCode already supports automatic compaction and a plugin hook that can inject domain-specific compaction context or replace the compaction prompt. The docs explicitly show using `experimental.session.compacting` to preserve task status, decisions, files being worked on, blockers, dependencies, and next steps. ([OpenCode][19])

For daemon mode:

1. Add `compaction.reason = "daemon-forced" | "daemon-token" | "daemon-iteration"`.
2. Add `daemon.compact_every_iterations`.
3. Preserve a durable daemon ledger outside chat context.
4. On compaction, inject daemon state into the compaction prompt:

   * script id
   * original task
   * exit conditions
   * checkpoint policy
   * active workers
   * claimed tasks
   * completed commits
   * current blockers
   * next suggested item
5. After compaction, do **not** rely on generic “Continue if you have next steps.” Instead, daemon mode should inject a deterministic next-iteration user message.

This is also how you avoid recreating the accidental auto-compaction infinite-loop bug described in #15533. The difference is explicit: ordinary auto-compaction should not keep going after a natural stop, but daemon compaction should because daemon mode is active and visible. ([GitHub][12])

---

## Headless `opencode run` changes

Current `run.ts` breaks on idle. In daemon mode, it should not.

Add flags:

```text
opencode run \
  --daemon \
  --script .opencode/daemon/jankurai.agent.yaml

opencode run \
  --daemon \
  --forever \
  --exit-condition "jankurai audit --json" \
  --exit-jq ".findings == 0 and .caps == 0" \
  --checkpoint "git add -A && git commit -m 'daemon checkpoint' && git push" \
  --compact-every 3 \
  "Investigate dirty worktree items..."
```

Headless event behavior:

```text
if session.status == idle:
  if daemon inactive:
    exit like today
  if daemon active:
    continue listening

if daemon.status == terminal:
  exit with mapped code

if permission asked:
  apply daemon permission policy
```

Suggested exit codes:

| Exit code | Meaning                         |
| --------: | ------------------------------- |
|       `0` | Exit condition satisfied.       |
|       `2` | Manual stop.                    |
|       `3` | Permission pause/denial.        |
|       `4` | Budget stop.                    |
|       `5` | Script validation error.        |
|       `6` | Checkpoint failure.             |
|       `7` | Exit-condition command failure. |
|       `8` | Runtime/internal error.         |

The TUI must make daemon mode impossible to miss. The UX should not rely only on text in the transcript.

### Header

```text
╭────────────────────────────────────────────────────────────────────╮
```

### Visual rules

| Element             | Treatment                                               |
| ------------------- | ------------------------------------------------------- |
| Header border       | Gold `#FFD700`.                                         |
| “FOREVER” label     | Small caps, gold foreground, dim black/gold background. |
| Iteration number    | Gold accent.                                            |
| Stop/pause keys     | Always visible.                                         |
| Session list        | Gold infinity icon next to daemon sessions.             |

### TUI safety

When daemon mode is active, `/exit` should ask whether to:

1. detach and leave daemon running,
2. pause daemon,
3. stop daemon.

For headless mode, `Ctrl-C` should map to configurable behavior:

```yaml
interrupt:
  first_ctrl_c: pause
  second_ctrl_c: stop
```

# 4. Multi-agent collaboration: replace `THIS_TRACKER.md` with a real blackboard

Your “Read THIS_TRACKER.md and update it” pattern is the right instinct but the wrong substrate. A Markdown tracker is human-readable, but it is weak for leases, claims, concurrency, conflict resolution, and structured results.

OpenCode already has subagents and child sessions. The docs say subagents can be invoked automatically or manually with `@`, and child sessions can be navigated from the parent. ([OpenCode][7])

The daemon runtime should add a first-class **blackboard**:

```text
daemon_blackboard
  tasks
  claims / leases
  artifacts
  messages
  decisions
  votes
  locks
  metrics
```

## Blackboard task record

```yaml
task:
  id: task-024
  title: "Audit dirty change in packages/foo"
  state: open | claimed | running | blocked | review | done | abandoned
  priority: 50
  claim:
    owner: worker-2
    expires_at: 2026-05-07T18:44:00Z
  files:
    include:
      - "packages/foo/**"
    exclude:
      - "node_modules/**"
  acceptance:
    - "git diff reviewed"
    - "tests pass"
    - "no jankurai findings introduced"
  artifacts:
    - type: patch
      path: ".opencode/daemon/artifacts/task-024.patch"
```

## Worker lifecycle

```text
Coordinator:
  discover tasks
  claim tasks
  spawn workers
  collect artifacts
  assign reviewers
  merge or reject
  checkpoint
  exit-check

Worker:
  read assigned task
  work only inside claimed scope
  produce structured result
  do not push directly unless role allows

Reviewer:
  inspect diff/artifact
  run verification
  vote accept/revise/reject

Merger:
  apply accepted artifact
  resolve conflicts
  run final verification
  commit/push
```

## Multi-agent topologies

| Topology             | Use it when                        | Pattern                                                |
| -------------------- | ---------------------------------- | ------------------------------------------------------ |
| Single daemon worker | Dirty worktree cleanup, small repo | One build agent loops forever.                         |
| Manager + reviewers  | Need confidence before commit      | Build worker → review subagent → merge.                |
| Worktree swarm       | Many independent tasks/files       | One worker per git worktree, merge queue.              |
| Council              | Architecture/design decisions      | Multiple specialists vote; coordinator decides.        |
| Scout/Builder split  | Large unknown repo                 | Read-only explore agents produce map; builder edits.   |
| CI repair loop       | Tests fail repeatedly              | Worker patches, reviewer evaluates, CI feedback loops. |

The key is that agents should not “chat randomly.” They should exchange structured artifacts through the blackboard.

# 5. Recommended daemon implementation plan

## Phase 0: Define semantics before code

Write a short daemon semantics document:

```text
A daemon is active when:
  a script is parsed through an explicit gate,
  validated,
  persisted,
  and started by the daemon service.

A daemon stops only when:
  exit condition is satisfied,
  user/manual stop,
  explicit budget stop,
  permission policy pauses/stops,
  unrecoverable runtime failure,
  process killed without auto-resume.

Model finish does not stop a daemon.
Model refusal/error does not automatically stop a daemon unless on_error says so.
Compaction does not decide daemon continuation.
```

This should become the invariant that tests enforce.

---

## Phase 1: Parser and script schema

Add `agent-script/schema.ts`, `parser.ts`, and `compiler.ts`.

Parser rules:

1. Never parse daemon syntax from ordinary prompts.
2. Only activate via:

   * TUI slash command `/daemon run`, or
   * CLI `opencode run --script`, or
   * API `session.daemon.start`.
3. Require exact begin marker:

   ```text
   ::OPENCODE_AGENT_SCRIPT v1 id="..."
   ```
4. Require exact end marker:

   ```text
   ::END_OPENCODE_AGENT_SCRIPT id="..."::
   ```
5. Require matching id.
6. Require `agent-script.enable: true`.
7. Require `activation.confirm` for unbounded or dangerous scripts.
8. Validate permissions before start.
9. Show a preview in TUI before launching.
10. Treat `<FOREVER>`, `<GIT>`, and other XML as inert text forever.

---

## Phase 2: Daemon service and durable state

Add `SessionDaemon.Service`.

Core methods:

```text
start(sessionID, script)
pause(daemonID, reason)
resume(daemonID)
stop(daemonID, reason)
status(daemonID)
patch(daemonID, patchScript)
onSessionIdle(sessionID)
onPermissionAsked(sessionID, permission)
onServerStart()
```

Durable tables:

```text
session_daemon
daemon_ledger_event
daemon_task
daemon_claim
daemon_artifact
daemon_lock
daemon_worker
```

At minimum, implement `session_daemon` and `daemon_ledger_event`. The richer blackboard can ship later.

---

## Phase 3: Outer loop orchestration

Implement the event-driven loop:

```text
onSessionIdle(sessionID):
  daemon = active daemon for session
  if none: return

  set phase collect_results
  append ledger event from last assistant

  set phase checkpoint
  run checkpoint hooks

  set phase exit_check
  result = evaluate exit conditions
  if result satisfied:
    terminal SATISFIED
    return

  set phase compacting if needed
  maybe force compaction

  set phase requeue

  call SessionPrompt.prompt(noReply=false)
```

# 6. The Agent Script Language: deliberate, gated, bulletproof

I would call this **OASL v1**: OpenCode Agent Script Language.

The important rule: **OASL is not XML prompt text. It is a runtime script embedded in a prompt only behind a deliberate activation gate.**

## Activation forms

### TUI activation

```text
/daemon run
::OPENCODE_AGENT_SCRIPT v1 id="jankurai-worktree-cleanup"
agent-script:
  version: 1
  enable: true
  activation:
    confirm: "RUN-DAEMON jankurai-worktree-cleanup FOREVER"
...
::END_OPENCODE_AGENT_SCRIPT id="jankurai-worktree-cleanup"::
```

### CLI activation

```text
opencode run --script .opencode/daemon/jankurai.agent.yaml
```

### API activation

```text
POST /session/:id/daemon/start
```

## Anti-accidental execution rules

| Rule                                                          | Purpose                                  |
| ------------------------------------------------------------- | ---------------------------------------- |
| First non-whitespace token must be `/daemon run` in TUI.      | Pasted docs cannot activate.             |
| CLI must pass `--script`.                                     | Piped text cannot accidentally activate. |
| Exact begin marker required.                                  | Natural XML/YAML is inert.               |
| Exact end marker required.                                    | Prevent partial parse.                   |
| Begin/end ids must match.                                     | Prevent concatenation errors.            |
| Root key must be `agent-script`.                              | Prevent random YAML front matter.        |
| `enable: true` required.                                      | Prevent examples from executing.         |
| `activation.confirm` required for forever/dangerous actions.  | Prevent accidental unbounded cost.       |
| Dangerous permissions require explicit script policy.         | Prevent silent push/delete/etc.          |
| TUI shows preview before first run unless configured trusted. | Human-visible launch.                    |
| Script hash optional but recommended.                         | Prevent copy/edit drift.                 |

Natural tags like this should **never** execute:

```xml
<FOREVER>
...
</FOREVER>
```

They can be model-readable instructions, but they must not start daemon mode.

## `activation`

Controls deliberate startup.

```yaml
activation:
  confirm: "RUN-DAEMON jankurai-worktree-cleanup FOREVER"
  require_preview: true
  trusted_from:
    - ".opencode/daemon/*.agent.yaml"
  require_clean_parse: true
  require_matching_end_marker: true
```

Functions:

| Function                      | Meaning                                                 |
| ----------------------------- | ------------------------------------------------------- |
| `confirm`                     | Exact phrase required for dangerous/forever scripts.    |
| `require_preview`             | TUI must show launch preview.                           |
| `trusted_from`                | File patterns that may bypass preview if config allows. |
| `require_clean_parse`         | Unknown keys fail validation.                           |
| `require_matching_end_marker` | Enforce sentinel integrity.                             |
| `require_hash`                | Optional body hash validation.                          |
| `require_git_repo`            | Refuse outside git worktree.                            |
| `require_branch`              | Only run on matching branch.                            |

---

## `mode`

```yaml
mode: daemon
```

Values:

| Mode          | Meaning                                                      |
| ------------- | ------------------------------------------------------------ |
| `once`        | Run one prompt normally with script-expanded context.        |
| `daemon`      | Requeue until exit/manual/budget.                            |
| `forever`     | No exit condition required; manual/budget/process stop only. |
| `scheduled`   | Run on cron/interval.                                        |
| `swarm`       | Coordinator with workers.                                    |
| `review_loop` | Build/review/revise cycle.                                   |
| `ci_loop`     | CI feedback loop.                                            |
| `monitor`     | Watch files/events/webhooks and wake agent.                  |

`daemon` plus `loop.kind: forever` is the typical “PROMISED infinite” mode.

## `loop`

```yaml
loop:
  kind: until
  on_model_stop: continue
  compact_every: 3

```

Functions:

| Function                   | Meaning                                            |
| -------------------------- | -------------------------------------------------- |
| `kind: once`               | Run once.                                          |
| `kind: repeat`             | Repeat fixed number of times.                      |
| `kind: while`              | Continue while condition is true.                  |
| `kind: until`              | Continue until condition is true.                  |
| `kind: forever`            | Continue indefinitely.                             |
| `on_model_stop: continue`  | Treat assistant stop as iteration boundary.        |
| `on_model_stop: exit`      | Normal behavior.                                   |
| `compact_every`            | Force compaction every N iterations.               |
| `sleep_between_iterations` | Delay before requeue.                              |
| `backoff`                  | Exponential backoff for errors.                    |

| `max_iterations`           | Hard iteration cap.                                |

| `heartbeat`                | Emit heartbeat event every interval.               |
| `resume_on_start`          | Resume daemon when server restarts.                |

For your “never stop unless exit satisfied or manually killed” requirement:

```yaml

  strict_forever: true

    enforce: false
```

## `checkpoint`

Runs between iterations.


| `commands`                        | Ordered checkpoint shell commands.               |
| `commit.message`                  | Template commit message.                         |
| `commit.when`                     | Commit condition.                                |
| `push.when`                       | Push condition.                                  |
| `require_success`                 | Failure stops/pauses daemon.                     |
| `skip_if_no_diff`                 | Do not commit empty changes.                     |

| `rollback_on_failure`             | Reset to pre-checkpoint snapshot.                |

| `on_no_progress`    | No progress detected.        |
| `on_budget_warn`    | Budget warning.              |
| `on_pause`          | Daemon paused.               |
| `on_resume`         | Daemon resumed.              |
| `on_stop`           | Daemon stopped.              |
| `on_exit_satisfied` | Exit condition satisfied.    |

Hook actions:

| Action       | Meaning                        |
| ------------ | ------------------------------ |
| `shell`      | Run shell command.             |
| `prompt`     | Inject prompt.                 |
| `compact`    | Request compaction.            |
| `blackboard` | Blackboard operation.          |
| `mcp`        | MCP call.                      |
| `notify`     | Desktop/terminal notification. |
| `webhook`    | HTTP webhook.                  |
| `ledger`     | Write ledger event.            |
| `pause`      | Pause daemon.                  |
| `stop`       | Stop daemon.                   |

```text
/daemon patch jankurai-worktree-cleanup

  - op: set

  - op: set
    path: agents.concurrency.max_workers
```

Patch rules:

| Rule                                                        | Meaning                 |
| ----------------------------------------------------------- | ----------------------- |
| Cannot change `id`.                                         | Script identity stable. |

| Can change worker count, sleep, compaction, budgets, hooks. | Live control.           |

# 11. Key pitfalls and fixes

| Pitfall                          | Why it matters                  | Fix                                                                       |
| -------------------------------- | ------------------------------- | ------------------------------------------------------------------------- |
| Model says “done”                | Normal agents stop here.        | Outer daemon loop treats stop as iteration boundary.                      |
| Context compaction loses task    | Long loops forget goals.        | Durable ledger + daemon compaction context.                               |
| Accidental paste triggers daemon | Dangerous/costly.               | Slash command/CLI gate + sentinel + confirmation.                         |
| Infinite compaction bug returns  | Existing community pain.        | Separate ordinary auto-continue from explicit daemon continue.            |
| MCP tool bloat                   | MCP schemas consume context.    | Per-agent MCP allowlist, lazy enable, max advertised tools.               |
| Multiple agents edit same file   | Conflicts/regressions.          | Blackboard leases + file locks + worktree per worker.                     |
| Agent pushes bad code            | Autonomous damage.              | Verification gates + reviewer + permissions + rollback.                   |
| Headless exits on idle           | Current `run` behavior.         | Do not break on idle while daemon active.                                 |
| UI hides autonomy                | User forgets daemon is running. | Gold forever banner, badges, controls.                                    |
| Cost runaway                     | True forever can burn money.    | Explicit unbounded acknowledgement, budget warnings, visible metrics.     |
| Permission prompt blocks forever | Headless stuck.                 | `on_permission` policy: pause/notify/deny/allow.                          |
| Server crash stops daemon        | Not truly durable.              | Persist state and optional resume-on-start.                               |
| No progress loop                 | Burns tokens doing nothing.     | Progress detector warns or pauses unless `strict_forever` says warn only. |

---

# 12. The best implementation sequence

1. **Parser + validation**
   Implement deliberate gated script parsing first. No daemon should start from raw prompt tags.

2. **Daemon durable state**
   Persist config, status, iteration, phase, and ledger.

3. **Outer loop orchestration**
   Observe normal session idle and requeue according to daemon state.

4. **Headless run support**
   Make `opencode run --daemon` not exit on ordinary idle.

5. **Gold TUI overlay**
   Add daemon banner/status/controls early so testing is safe.

6. **Forced daemon compaction**
   Add daemon-specific compaction context and cadence.

7. **Checkpoint and exit evaluators**
   Implement shell/json/git/file conditions.

8. **Permission policy integration**
   Make daemon permissions explicit and test denial paths.

9. **Blackboard MVP**
   Structured tasks, claims, artifacts, ledger mirror.

10. **Worker spawning / swarm**
    Use OpenCode subagents and child sessions first; add A2A/MCP-agent bridges later.

11. **Live patching**
    Add `/daemon patch` after the base daemon is stable.

12. **Full observability**
    JSON events, exports, metrics, logs, cost, tokens.

# 13. The strongest design choice

The strongest version is:

```text
OASL script block
  ↓ parse/validate/confirm
DaemonConfig persisted
  ↓
SessionDaemon state machine
  ↓
Normal OpenCode prompt loop
  ↓
Model stops
  ↓
Runtime checkpoint + exit-check + compact + requeue
  ↓
Repeat forever
```

Not:

```text
Prompt contains <FOREVER>
  ↓
Hope the model obeys forever
```

That distinction is everything.

A good OpenCode daemon mode should make this true:

> The model is allowed to stop. The daemon is not.

And the TUI should make this obvious:

> ♾ FOREVER is active, gold, visible, controllable, auditable, and deliberate.
