## Bottom line

The strongest design is **not** to teach the model that `<FOREVER>` means forever. That will always be unreliable because the model can still “decide it is done,” compaction can change its memory, permissions can block it, and CLI/TUI code can exit when a session becomes idle.

The right design for Jekko is a **host-interpreted agent runbook language**:

> A deliberate, gated, schema-validated control plane that Jekko parses before the prompt reaches the model.
> The runtime, not the LLM, owns loops, stop criteria, context clearing, checkpoints, subagent spawning, ledger state, permissions, and UI state.

I would call it **OACL: Jekko Agent Control Language**. It should be usable from a file, CLI flag, slash command, or intentionally gated in-prompt block. The LLM should never be trusted to honor “forever” by itself. In daemon mode, the model’s `stop` is only an **iteration boundary**, not termination.

A good core invariant:

> **Daemon mode exits only when a validated stop condition succeeds, the user manually stops it, or a configured safety policy pauses/fails it. The model saying “done” is never sufficient.**

# 1. What I found in Jekko today

## 1.1 Jekko already has most primitives, but not the forever runtime

Jekko already has agents, subagents, permissions, compaction, CLI headless execution, plugins, custom commands, MCP support, and session status events. The missing piece is a **first-class runtime loop** that sits above the model loop.

Jekko’s own docs say agents have a `steps` setting, and if `steps` is unset, the agent continues until “the model decides to stop” or the user interrupts. That sentence is basically the root cause of your frustration: model stop is currently a valid terminal condition. ([Jekko][1])

Jekko’s agent system already distinguishes primary agents and subagents. Primary agents handle the main conversation, while subagents can be invoked automatically or by mention. ([Jekko][1]) Hidden subagents can also be invoked programmatically through the Task tool, and `permission.task` controls which agents can invoke which subagents. ([Jekko][1])

That means Jekko already has a good foundation for multi-agent orchestration. It just lacks durable, typed orchestration semantics.

## 1.2 The current loop really can exit when the model stops

The plan you were given is directionally correct about the main stop path. In the current `prompt.ts` loop, Jekko checks whether the last assistant message has a finish reason that is not `tool-calls`, whether there are no pending tool calls, and whether the assistant message came after the last user message; if so, it logs that it is exiting the loop and breaks. ([GitHub][2])

There is also a current Jekko issue describing headless `jekko run` exiting after auto-compaction. The report points to the same kind of loop behavior: after compaction, if the synthetic “continue” message is not created, the loop sees a finished assistant message and exits. ([GitHub][3])

So the problem is not merely prompt engineering. It is runtime semantics.

## 1.3 Jekko already has compaction, but not daemon-safe compaction

Jekko’s config supports automatic compaction, pruning, and reserved token space. ([Jekko][4]) The compaction implementation already summarizes conversation state into sections like goal, constraints, progress, key decisions, next steps, critical context, and relevant files. It also has an experimental compaction hook that plugins can use to inject additional context or override the compaction prompt. ([GitHub][5])

That is useful, but a daemon cannot rely only on chat-history compaction. A forever loop needs a **durable ledger outside the model context**, because compaction is lossy by design. The daemon ledger should survive context clearing, process restarts, subagent handoffs, and accidental model summaries.

## 1.4 Jekko plugins are powerful, but probably not enough for bulletproof forever

Jekko plugins can hook into events, tool execution, TUI prompt append, and compaction. ([Jekko][6]) They can also add custom tools. ([Jekko][6])

However, a community issue argues that plugins currently cannot inject AI-visible messages at the precise moments needed to stop drift or recover after tool calls/compaction. The issue proposes hooks such as `session.stopping` and tool-result injection so plugins can actively steer the model. ([GitHub][7])

That issue strongly supports making daemon mode first-class in core Jekko rather than trying to fake it entirely with a plugin.

## 1.5 The CLI also currently treats idle as terminal

The `jekko run` command supports headless operation and many flags, but there is no daemon flag today. ([Jekko][8]) In the current `run.ts`, the headless event loop exits when it receives `session.status` with `type: "idle"` for the target session. ([GitHub][9])

A daemon cannot use ordinary idle semantics. Either the session must remain in a daemon/busy activity state across loop boundaries, or the CLI must understand that idle inside a daemon run means “iteration ended,” not “process complete.”

## 1.6 Jekko community is already asking for this

There is already a Reddit request asking whether Jekko has a `/loop` or CRON-like feature comparable to Claude Code looping workflows. ([Reddit][10]) There are also reports of agents pausing after compaction and needing a manual “continue.” ([GitHub][11])

So I would treat this as a real product gap:

> Jekko has agent execution. It does not yet have durable agent workflow execution.

# 2. What exists in the broader ecosystem

## 2.1 There is no single dominant “in-prompt forever scripting standard”

I found no broadly adopted, standard, prompt-embedded agent scripting language that solves all of these at once:

* infinite loops
* stop conditions
* context clearing
* durable state
* MCP tool orchestration
* multi-agent blackboard coordination
* checkpointing
* permissions
* UI affordances
* accidental-paste protection

The emerging standards are adjacent, not complete replacements.

## 2.2 AGENTS.md is becoming the project-instruction standard

AGENTS.md is a lightweight open format for giving coding agents project-specific instructions. It is described as a “README for agents,” separate from human-oriented README files. ([Agents][12])

OpenAI, Anthropic, Google, Microsoft, AWS, Block, and others are participating in the Agentic AI Foundation under the Linux Foundation, and OpenAI contributed AGENTS.md there. ([OpenAI][13]) OpenAI says AGENTS.md had been adopted by more than 60,000 open-source projects and supported by tools including Codex, Cursor, Devin, Factory, Gemini CLI, GitHub Copilot, Jules, VS Code, and others. ([OpenAI][13])

But AGENTS.md is **instructions**, not a runtime control language. It is good for “how to work in this repo,” not “loop forever until this JSON predicate passes.”

## 2.3 MCP is the tool/context protocol, not the loop language

MCP standardizes how applications expose tools, resources, prompts, and context to models over a client-server protocol. ([Model Context Protocol][14]) Jekko already supports MCP servers in config, while warning that MCP servers can add a lot of context and should be enabled carefully. ([Jekko][15])

MCP should be treated as the **tool/resource substrate**. It should not be the daemon language itself.

A daemon runbook should be able to say:

```yaml
tools:
  mcp:
    github: enabled
    linear: enabled
    playwright: disabled
```

But MCP is not where `forever`, `stop_when`, `checkpoint`, and `compact_at_boundary` should live.

## 2.4 A2A is for agent-to-agent interoperability

Google’s A2A protocol is aimed at communication between independent agentic applications. The spec includes concepts such as artifacts, streaming, push notifications, long-running/disconnected tasks, context grouping, and extensions. ([A2A Protocol][16])

A2A is relevant if Jekko eventually wants daemon agents to coordinate with external agents, but inside Jekko today, the built-in Task tool and subagent sessions are the more immediate path.

## 2.5 LangGraph has the closest runtime philosophy

LangGraph is not an in-prompt syntax, but it is very relevant architecturally. It emphasizes durable execution, human-in-the-loop, persistence, memory, streaming, and debugging for long-running agents. ([LangChain Docs][17]) Its persistence layer checkpoints state at each step into threads, enabling memory, time travel, fault tolerance, and human intervention. ([LangChain Docs][18])

This is the closest analogy to what Jekko daemon mode should become:

> A durable agent execution graph, not a really long prompt.

## 2.6 OpenAI Agents SDK, AutoGen, CrewAI, and Microsoft Agent Framework show the orchestration patterns

OpenAI’s Agents SDK defines agents as LLMs configured with instructions, tools, lifecycle behavior, handoffs, guardrails, structured outputs, and sessions. ([OpenAI GitHub][19]) It describes two major multi-agent patterns: handoffs, where a specialist takes over, and agents-as-tools, where an orchestrator remains in control. ([OpenAI GitHub][20])

For Jekko, the better default is **agents-as-tools**:

* one daemon orchestrator remains in control
* subagents receive scoped tasks
* subagents return typed artifacts
* orchestrator verifies, merges, and checkpoints

AutoGen popularized multi-agent conversations and group chats. CrewAI emphasizes structured workflows and crews of agents. Microsoft’s newer Agent Framework is positioned as a successor to Semantic Kernel and AutoGen, combining single-agent and multi-agent patterns with session-based state and telemetry. ([Microsoft GitHub][21])

The lesson: multi-agent “chat” is not enough. You need state, roles, task ownership, structured outputs, and arbitration.

## 2.7 Prompt DSLs exist, but they are not enough by themselves

POML, LMQL, SGLang, Guidance-like approaches, and similar languages show that prompts can benefit from structure, templating, constraints, and programmatic composition. Microsoft describes POML as a markup language for structured, maintainable prompt orchestration. ([arXiv][22]) LMQL frames prompting as programming with constraints and typed generation. ([LMQL][23])

These are useful design inspirations, but they do not solve Jekko daemon mode alone.

For this use case, the language must be:

* parsed by Jekko
* validated before execution
* removed from the AI-visible prompt
* connected to permissions
* backed by durable state
* reflected in TUI/CLI
* able to survive context clearing

That is a runtime language, not merely prompt markup.

## 2.8 Research patterns matter: ReAct, Reflexion, Tree-of-Thoughts, Voyager

ReAct established the loop of reasoning plus external actions. ([arXiv][24]) Reflexion adds feedback and self-reflection memory between attempts. ([arXiv][25]) Tree-of-Thoughts explores branching, evaluation, and backtracking. ([NeurIPS Proceedings][26]) Voyager demonstrated continuous exploration with an automatic curriculum, skill library, and iterative prompting with feedback. ([voyager.minedojo.org][27])

The right daemon language should support these patterns explicitly:

* `reflect`
* `evaluate`
* `retry`
* `branch`
* `select`
* `skill.learn`
* `memory.write`
* `queue.next`
* `verify`
* `checkpoint`

# 3. Critical review of the proposed plan

The proposed plan has useful instincts, but it is not bulletproof enough.

## What it gets right

It correctly identifies that the model’s ordinary stop is the main reason the loop ends. Jekko’s own agent docs say unset `steps` means the agent continues until the model chooses to stop or the user interrupts. ([Jekko][1]) The source-level loop behavior supports this. ([GitHub][2])

It correctly notices that compaction already exists and should be reused, not rebuilt. Jekko already has compaction config and plugin hooks. ([Jekko][4])

It correctly notices that `run.ts` exits on idle and that headless daemon mode needs different semantics. ([GitHub][9])

It correctly proposes a visible TUI state. A forever process must be unmistakable.

## What I would change

### 3.1 Do not make XML tags the control plane

`<FOREVER>` is too easy to paste accidentally, too ambiguous, too model-visible, and too likely to be treated as text instead of authority.

Use a **deliberate runbook gate** with version, confirmation, and optionally nonce/hash.

Bad:

```text
<FOREVER>
Do this forever.
</FOREVER>
```

Better:

```yaml
<<<JEKKO_DAEMON_RUNBOOK v1 INTENTIONAL>>>
confirm: "RUN-DAEMON"
mode: daemon
...
<<<END_JEKKO_DAEMON_RUNBOOK>>>
```

Best for serious usage:

```bash
jekko daemon run .jekko/runbooks/jankurai-cleaner.oacl.yaml
```

In-prompt syntax should be supported, but it should be gated and parsed by Jekko, not obeyed by the LLM.

### 3.2 Do not silently continue after permission denial

The proposed plan says daemon mode might log permission denial and continue. That is dangerous.

Permissions are a core Jekko safety boundary. Jekko supports explicit permission rules for tools such as read, edit, bash, task, webfetch, websearch, MCP tools, and others. ([Jekko][1]) Permission evaluation can allow, ask, deny, or reject. ([GitHub][28])

Daemon mode should have explicit policies:

```yaml
on_permission_denied: pause
on_permission_timeout: pause
on_rejected: fail
```

Possible policies:

* `pause`: stop the daemon but keep state resumable
* `skip_item`: release the current work item and continue
* `fail`: terminate with error
* `ask`: ask user in TUI
* `deny_and_continue`: allowed only for explicitly marked noncritical tools

Never default to “just keep going.”

### 3.3 Do not rely on arbitrary shell strings for everything

Shell commands are necessary, but raw shell as the only stop/checkpoint mechanism is brittle and risky.

Instead of only:

```yaml
exit_condition: "jankurai audit | grep '0 findings'"
```

Support typed predicates:

```yaml
stop_when:
  command:
    run: "jankurai audit --json"
    expect:
      json:
        findings: 0
        caps: 0
```

And also support built-ins:

```yaml
stop_when:
  all:
    - git.clean: true
    - audit:
        command: "jankurai audit --json"
        json:
          findings: 0
          caps: 0
```

Shell still exists, but the runbook should encourage typed expectations.

### 3.4 Do not store daemon state only in session status

Status is for UI/event state. It is not enough for durable daemon execution. Jekko’s status union currently has idle, retry, and busy states. ([GitHub][29])

Daemon state should be persisted separately:

* run ID
* session ID
* runbook hash
* iteration
* phase
* current work item
* active subagents
* stop condition history
* checkpoints
* ledger entries
* locks/leases
* last heartbeat
* pause/fail reason

Status should display daemon state, not be the source of truth.

### 3.5 Do not make gold mode a global theme mutation

A gold visual signal is absolutely right, but it should be implemented as a daemon banner/status component, not by mutating the user’s theme globally.

Use a dedicated TUI affordance:

```text
∞ FOREVER · jankurai-worktree-cleaner · iteration 42 · phase VERIFY
stop: jankurai audit findings=3 caps=1
```

The active border/header can be gold. The whole theme should not be transformed.

### 3.6 Do not make multi-agent orchestration just a prompt suggestion

The proposed plan says to inject a prompt telling the model to use `jnoccio_spawn_parallel` or MCP parallelism. That is too weak.

Jekko already has a Task tool that can create subagent sessions and return a `task_id` for resuming work. ([GitHub][30]) Daemon mode should expose first-class runbook operations:

* `spawn`
* `fanout`
* `join`
* `claim`
* `lease`
* `artifact`
* `reduce`
* `review`
* `quorum`

The orchestrator should use a durable blackboard/ledger, not a Markdown tracker as the only coordination mechanism.

# 4. The architecture I would build

## 4.1 Core idea

Add a new Jekko runtime layer:

```text
User Runbook
   ↓
OACL parser / validator / safety gate
   ↓
Daemon Run Store
   ↓
Daemon Engine
   ↓
SessionPrompt.loop / Task tool / MCP tools / shell / git
   ↓
Verifier / Checkpoint / Compaction / Ledger
   ↓
Next iteration or finish
```

The daemon engine owns the lifecycle. The model owns individual problem-solving turns.

## 4.2 Execution semantics

In normal Jekko:

```text
User prompt
 → model/tool loop
 → model says stop
 → session idle
```

In daemon mode:

```text
Runbook starts
 → evaluate stop condition
 → create iteration prompt
 → model/tool loop
 → model says stop
 → daemon treats that as iteration boundary
 → verify
 → checkpoint
 → compact/summarize
 → evaluate stop condition
 → next iteration
```

The model’s “stop” means:

> “I have no more tool calls for this iteration.”

It does not mean:

> “The daemon is done.”

## 4.3 Durable state model

Add a daemon run record. Whether this lives in the existing session storage, a new DB table, or Jekko’s storage KV is an implementation choice, but it must be durable.

Minimum fields:

```yaml
daemon_run:
  id: string
  session_id: string
  parent_session_id: string | null
  runbook_version: string
  runbook_hash: string
  name: string
  mode: daemon | workflow | batch
  active: boolean
  paused: boolean
  created_at: timestamp
  updated_at: timestamp
  last_heartbeat_at: timestamp
  iteration: integer
  phase: idle | stopping_check | planning | executing | verifying | checkpointing | compacting | spawning | joining | paused | failed | complete
  forever: boolean
  stop_reason: string | null
  failure_reason: string | null
  current_work_item_id: string | null
  active_subtasks:
    - task_id
    - agent
    - status
    - lease_until
  permissions_profile: object
  compact_policy: object
  ledger_pointer: string
```

The daemon ledger should be append-only:

```yaml
ledger_entry:
  id: string
  run_id: string
  iteration: integer
  timestamp: timestamp
  type:
    - iteration_started
    - work_item_claimed
    - tool_result
    - finding
    - decision
    - artifact
    - verification_result
    - checkpoint
    - compaction_summary
    - subagent_spawned
    - subagent_result
    - stop_condition_result
    - pause
    - resume
    - error
  payload: object
```

This ledger is the replacement for “everyone edit THIS_TRACKER.md.”

A Markdown tracker can still be generated from the ledger, but it should not be the source of truth.

# 5. Specific Jekko implementation plan

This is an implementation plan, not an implementation.

## 5.1 New module: `packages/jekko/src/session/daemon.ts`

Responsibilities:

* parse validated daemon input
* create daemon run
* persist daemon state
* run the outer loop
* evaluate stop conditions
* create iteration prompts
* call `SessionPrompt.loop` or equivalent
* run verification/checkpoint steps
* call compaction at loop boundaries
* publish daemon events
* manage pause/resume/stop

Primary exported service shape:

```text
Daemon.Service
  start(input)
  stop(runID, reason)
  pause(runID, reason)
  resume(runID)
  status(runID)
  list(sessionID?)
  ledger(runID, query?)
```

Important: the daemon service should live server-side, not just CLI-side. Otherwise TUI, CLI, SDK, and future remote clients will all behave differently.

## 5.2 New module: `packages/jekko/src/runbook/oacl.ts`

Responsibilities:

* detect gated OACL blocks
* parse YAML/JSON
* validate schema
* strip control plane from model-visible prompt
* generate canonical hash
* enforce deliberate activation
* reject accidental syntax

Validation rules:

* block must start with exact opener
* block must end with exact closer
* version must be supported
* `confirm` must match required phrase or nonce
* `mode: daemon` must be explicit
* `loop.forever: true` requires explicit `confirm_forever`
* dangerous tools require explicit permissions
* shell commands require permission scope
* checkpoint push requires explicit git permission
* stop condition required unless `manual_stop_only: true`
* context clearing must specify what state is preserved

## 5.3 Modify `prompt.ts`

Do not shove all daemon logic into `runLoop`. Keep `runLoop` as the model/tool loop.

Minimal changes:

1. Add detection before ordinary prompt handling:

   * if prompt contains valid OACL daemon block, route to `Daemon.start`
   * otherwise behave normally

2. Add an explicit method or internal path:

   * `SessionPrompt.daemon(input)`
   * or `Daemon.Service.start` calls `SessionPrompt.prompt/loop`

3. Avoid changing normal semantics for non-daemon sessions.

4. Optionally add a low-level hook:

   * `session.stopping`
   * daemon engine can intercept model stop cleanly
   * plugins could also use this later

Jekko already has a loop exit path when the assistant finishes without tool calls. ([GitHub][2]) In daemon mode, that path should become an event:

```text
model_iteration_finished
```

not:

```text
session_finished
```

## 5.4 Modify `status.ts`

Current status is idle, retry, busy. ([GitHub][29])

Prefer backward-compatible extension:

```yaml
status:
  type: busy
  activity:
    type: daemon
    run_id: string
    name: string
    iteration: integer
    phase: string
    forever: boolean
    stop_summary: string
```

If the schema cannot safely add optional fields, add a new variant:

```yaml
type: daemon
iteration: integer
phase: string
run_id: string
name: string
forever: boolean
```

But then every status consumer must be audited. The CLI currently checks for `idle`, so a new daemon status must not accidentally break headless execution. ([GitHub][9])

## 5.5 Modify `run-state.ts`

The runner currently sets sessions idle when work completes. For daemon mode, the outer daemon run should be the running unit.

Desired behavior:

```text
daemon active
  status = busy(activity=daemon)
  inner model loop may stop
  status remains daemon/busy while verifier/checkpoint/next iteration runs
daemon complete
  status = idle
```

The idle event should be emitted only when the daemon run has truly completed, paused, failed, or been manually stopped.

## 5.6 Modify `run.ts`

Add headless daemon support.

New commands/flags:

```text
jekko daemon run <runbook>
jekko daemon stop <run-id>
jekko daemon pause <run-id>
jekko daemon resume <run-id>
jekko daemon status <run-id>
jekko daemon ledger <run-id>

jekko run --daemon --runbook <file>
jekko run --daemon --exit-when <condition> --checkpoint <policy> "<task>"
```

The CLI should not exit on ordinary session idle if a daemon run is active. It should exit only when it receives one of:

```text
daemon.completed
daemon.failed
daemon.paused
daemon.stopped
```

Output modes:

```text
--format text
--format json
--format stream-json
```

For automation, `stream-json` should emit every daemon event.

## 5.7 Modify TUI session route

Add a daemon header component.

Example:

```text
╔════════════════════════════════════════════════════════════════════╗
║  ∞ FOREVER  jankurai-worktree-cleaner  iter 17  VERIFYING          ║
║  stop: jankurai audit → findings=2 caps=1                          ║
╚════════════════════════════════════════════════════════════════════╝
```

Behavior:

* gold foreground/accent
* small “FOREVER” label at top
* visible iteration count
* visible phase
* visible stop condition summary
* visible pause/stop key hints
* visible current work item
* visible subagent count

Do not mutate the whole user theme. Use a dedicated daemon color role, falling back to gold.

## 5.8 Modify theme tokens

Add semantic color tokens:

```yaml
daemon:
  foreground: gold
  border: gold
  muted: dark-gold
  danger: amber/red
```

This is better than overriding `primary` or `accent` globally.

## 5.9 Extend plugin hooks

Jekko already supports many plugin hooks, including compaction hooks and tool execution hooks. ([Jekko][6]) For daemon mode, add:

```text
daemon.starting
daemon.iteration.started
daemon.iteration.finished
daemon.stop_condition.evaluated
daemon.work_item.claimed
daemon.checkpoint.before
daemon.checkpoint.after
daemon.compaction.before
daemon.compaction.after
daemon.subagent.spawned
daemon.subagent.finished
daemon.paused
daemon.resumed
daemon.stopping
daemon.stopped
```

Also add a more general hook requested by community:

```text
session.stopping
```

This would allow controlled continuation/injection before a session becomes idle, which aligns with the public plugin limitation request. ([GitHub][7])

## 5.10 Use the existing Task tool for subagents

Jekko’s Task tool already creates subagent sessions and returns a `task_id` for resuming work. ([GitHub][30]) Daemon mode should build on that.

Add daemon-owned subtask management:

```yaml
subtasks:
  - task_id
  - agent
  - prompt_hash
  - status
  - claimed_files
  - lease_until
  - result_artifact
```

Subagents should not coordinate by editing the same Markdown file. They should coordinate through the daemon ledger.

## 5.11 Add a blackboard/ledger API

Add internal daemon tools available only in daemon sessions:

```text
daemon_ledger_append
daemon_ledger_query
daemon_work_claim
daemon_work_release
daemon_artifact_write
daemon_artifact_read
daemon_decision_record
daemon_message_send
daemon_lock_acquire
daemon_lock_release
```

These can be exposed to the orchestrator and optionally to subagents.

This enables:

* real work claiming
* conflict avoidance
* resumability
* structured status
* better compaction
* automatic report generation

## 5.12 Compaction policy

Existing compaction should remain, but daemon mode needs loop-boundary compaction.

Recommended policy:

```yaml
context:
  compact:
    mode: loop_boundary
    also_on_overflow: true
    every_iterations: 3
    preserve_recent_turns: 4
    preserve_ledger_summary: true
    preserve_current_work_item: true
    preserve_open_subtasks: true
```

At each boundary:

1. Write durable ledger summary.
2. Summarize current iteration.
3. Save current work item state.
4. Compact chat if policy says so.
5. Inject a fresh daemon continuation brief.

The model-visible brief should be short and deterministic:

```text
You are in Jekko daemon mode.

Run: jankurai-worktree-cleaner
Iteration: 18
Current phase: EXECUTE
Stop condition: jankurai audit must report findings=0 and caps=0
Current work item: fix dirty worktree item selected from git diff
Important preserved context:
- ...
Rules:
- Do not declare the daemon complete.
- Complete only this iteration’s assigned work.
- The host runtime will decide whether to continue.
```

## 5.13 Git checkpointing

Checkpointing must be typed, not just “run this shell blob.”

Support:

```yaml
checkpoint:
  when: verify.passed
  git:
    add: all
    commit:
      message: "daemon({run.name}): {iteration.summary}"
    push: true
```

Also support raw shell when explicitly allowed:

```yaml
checkpoint:
  steps:
    - run: "git add -A"
    - run: "git commit -m 'daemon checkpoint'"
    - run: "git push"
```

But shell checkpointing should require explicit permissions.

## 5.14 Stop conditions

Stop conditions should run at both start and end of iteration:

```yaml
stop_when:
  timing:
    - before_iteration
    - after_checkpoint
  all:
    - command:
        run: "jankurai audit --json"
        expect:
          json:
            findings: 0
            caps: 0
```

Support `all`, `any`, `not`, `command`, `git`, `file`, `ledger`, `test`, `manual`, `time`, `budget`.

Important distinction:

```yaml
stop_when:
  # successful finish

fail_when:
  # fatal condition

pause_when:
  # human needed
```

## 5.15 Error policy

Daemon mode needs explicit error semantics:

```yaml
errors:
  on_tool_error: retry
  retries: 2
  backoff: exponential
  on_repeated_failure: pause
  on_permission_denied: pause
  on_context_overflow_failure: fail
  on_subagent_timeout: cancel_and_retry
  on_git_conflict: pause
```

Do not let “forever” become “forever repeating the same broken command.”

Jekko already has doom-loop detection for repeated tool calls. ([GitHub][31]) Daemon mode should respect that and escalate to pause/replan rather than suppress it.

# 6. Recommended runtime flow

## 6.1 Single-agent daemon

```text
┌──────────────────────┐
│ OACL runbook input   │
└──────────┬───────────┘
           ↓
┌──────────────────────┐
```

# 7. Syntax design principles

## 7.1 The syntax must be deliberate

Accidental paste protection should be treated as a first-class requirement.

I recommend three activation modes.

### Safest: runbook file

```bash
jekko daemon run .jekko/runbooks/jankurai-cleaner.oacl.yaml
```

This is the default for serious workflows.

### TUI slash command

```text
/daemon load .jekko/runbooks/jankurai-cleaner.oacl.yaml
/daemon start
/daemon pause
/daemon stop
/daemon status
/daemon ledger
```

### In-prompt gated block

Only parsed when all of the following are true:

1. Exact opener present.
2. Exact closer present.
3. `mode: daemon` present.
4. `confirm` present.
5. If `forever: true`, `confirm_forever` present.
6. Optional session nonce matches.
7. Jekko config allows in-prompt runbooks.

Recommended block:

```yaml
<<<JEKKO_DAEMON_RUNBOOK v1 INTENTIONAL>>>
confirm: "RUN-DAEMON"
confirm_forever: "I UNDERSTAND THIS CAN RUN UNTIL STOPPED"
mode: daemon
name: example
...
<<<END_JEKKO_DAEMON_RUNBOOK>>>
```

For maximum safety, TUI can generate a nonce:

```text
/daemon nonce
```

Then require:

```yaml
nonce: "tui-generated-9f31"
confirm: "RUN-DAEMON-tui-generated-9f31"
```

This makes accidental pasted examples inert.

## 7.2 The syntax should be host-interpreted

These fields should never be merely model instructions:

```yaml
loop
stop_when
checkpoint
permissions
context.compact
spawn
parallel
timeout
retry
budget
```

Jekko should parse them, validate them, and enforce them.

The model-visible prompt should receive only a derived task brief.

## 7.3 The syntax should separate control plane from task content

Bad:

```yaml
task: "Do this forever and commit sometimes"
```

Good:

```yaml
goal: "Clean dirty worktree safely."

loop:
  forever: true

checkpoint:
  git:
    commit: true
    push: true
```

Natural language can describe work. Runtime semantics should be typed.

# 8. OACL: full syntax library

Below is the syntax library I would design. This is intentionally expansive. Not everything needs to ship in v1, but the shape should be reserved now so the language does not paint itself into a corner.

## 8.1 Gate and metadata commands

| Function            | Purpose                                          | Calls / runtime behavior                  |
| ------------------- | ------------------------------------------------ | ----------------------------------------- |
| `version`           | OACL schema version                              | Parser rejects unsupported versions       |
| `mode`              | `daemon`, `workflow`, `batch`, `review`, `watch` | Selects runtime engine                    |
| `name`              | Human-readable run name                          | Displayed in TUI/CLI/status               |
| `description`       | Longer run description                           | Stored in ledger                          |
| `confirm`           | Required deliberate activation phrase            | Parser rejects if absent/wrong            |
| `confirm_forever`   | Required when `loop.forever: true`               | Prevents accidental infinite runs         |
| `nonce`             | Optional session-generated activation nonce      | Prevents pasted examples from executing   |
| `owner`             | User/team/process owner                          | Ledger metadata                           |
| `tags`              | Search/filter labels                             | Ledger/index metadata                     |
| `enabled`           | Boolean kill switch                              | Allows checked-in disabled runbooks       |

```yaml
version: 1
```

```yaml
    - "**/*"
  exclude:
    - "node_modules/**"
    - "dist/**"
    - ".git/**"

permissions:
  allow:
    - read
    - edit
    - grep
    - glob
    - task:general
    - bash:git status
    - bash:git diff
    - bash:git add
    - bash:git commit
    - bash:jankurai audit
    - bash:npm test
  ask:
    - bash:git push
  deny:
    - bash:rm -rf
  on_permission_denied: pause

loop:
  forever: true
  order: random
  on_model_stop: verify
  on_empty_queue: rediscover
  heartbeat: "30s"

discover:
  - queue: dirty_worktree_items
    source:
      git:
        diff: true
        group_by: coherent_change
    dedupe_by: path_and_hunk_hash

pick:
  queue: dirty_worktree_items
  strategy: random
  claim:
    lease: "45m"

iteration:
  agent:
    name: general
    prompt: |
      Work only on the claimed dirty-worktree item.
      First inspect git diff and relevant files.
      Identify likely mistakes or regressions.
      Fix the item.
      Run focused verification.
      Produce a structured summary of changed files, tests run,
      remaining risks, and confidence.
    output_schema:
      changed_files: list<string>
      tests_run: list<string>
      summary: string
      confidence: number
      remaining_risks: list<string>

verify:
  steps:
    - run: "git status --short"
      expect:
        exit_code: 0
    - run: "jankurai audit --json"
      expect:
        exit_code: 0
  accept_with_findings: true

checkpoint:
  when: iteration.changed_files_nonempty
  steps:
    - run: "git add -A"
    - run: "git commit -m 'daemon(jankurai): clean dirty worktree item'"
    - run: "git push"

after_iteration:
  steps:
    - run: "jankurai audit --json"
      save_as: latest_audit
    - append_findings:
        from: latest_audit
        to: "JANKURAI_TASKLIST.md"
    - move_completed:
        from: "JANKURAI_TASKLIST.md"
        to: "JANKURAI_TASKLIST.md.completed"

stop_when:
  all:
    - command:
        run: "jankurai audit --json"
        expect:
          json:
            findings: 0
            caps: 0
    - git:
        clean: true

context:
  compact:
    at_loop_boundary: true
    every_iterations: 3
    also_on_overflow: true
  preserve:
    - daemon.ledger
    - latest_audit
    - current_work_item
    - open_subtasks
    - latest_git_status
    - "JANKURAI_TASKLIST.md"
    - "agent/JANKURAI_STANDARD.md"

blackboard:
  path: ".jekko/daemon/{run.id}/ledger.jsonl"
  expose_to_subagents: true
  locks:
    default_lease: "45m"
    on_expired: requeue

ui:
  banner: forever
  color: gold
  show:
    - iteration
    - phase
    - current_work_item
    - stop_condition
    - subagents
<<<END_JEKKO_DAEMON_RUNBOOK>>>
```

Important: the model should not see the whole control block as instructions. Jekko should parse it, enforce it, and derive short iteration prompts from it.

# 10. Multi-agent collaboration design

Your current “read THIS_TRACKER.md and update it” method is understandable, but it has several failure modes:

* concurrent edits conflict
* agents overwrite each other
* no atomic claims
* stale status
* no leases
* no structured artifacts
* compaction loses context
* hard to know which agent owns what
* no clear arbitration when agents disagree

Replace it with a daemon blackboard.

## 10.1 Blackboard entities

```yaml
work_item:
  id
  title
  source
  files
  priority
  status: open | claimed | blocked | done | failed
  claimed_by
  lease_until
  attempts
  parent_id

finding:
  id
  severity
  file
  summary
  evidence
  proposed_fix
  status

decision:
  id
  question
  selected_option
  rationale
  alternatives
  decided_by

artifact:
  id
  type: patch | report | test_result | audit_result | summary
  path
  produced_by
  related_work_item

message:
  from
  to
  topic
  body
  requires_response
```

## 10.2 Agent roles

Recommended default roles:

| Role         | Responsibility                              |
| ------------ | ------------------------------------------- |
| Orchestrator | Owns loop, queue, stop criteria, checkpoint |
| Scout        | Finds issues, classifies dirty changes      |
| Implementer  | Fixes one claimed item                      |
| Reviewer     | Reviews diff for regressions                |
| Verifier     | Runs tests/audits and interprets failures   |
| Reducer      | Merges subagent outputs into one plan       |
| Summarizer   | Writes durable iteration summary            |

Jekko already supports subagents and Task tool invocation, including resumable task IDs. ([GitHub][30]) So this can be built without inventing a whole new subagent mechanism.

## 10.3 Multi-agent runbook example

```yaml
parallelism:
  max_subagents: 5
  isolation:
    mode: git_worktree
    branch_prefix: "daemon/{run.name}/"

fanout:
  queue: dirty_worktree_items
  max_parallel: 3
  worker:
    agent: scout
    prompt: |
      Analyze the claimed dirty item.
      Do not edit.
      Report risk, likely owner, and recommended fix.
    output_schema:
      item_id: string
      risk: low | medium | high
      recommended_fix: string
      files: list<string>

join:
  timeout: "20m"
  min_success: 1

reduce:
  agent: reviewer
  prompt: |
    Merge scout reports into one prioritized implementation plan.
    Detect conflicts and duplicate work.

execute:
  agent: implementer
  max_parallel: 1
  prompt: |
    Implement only the selected item.
    Avoid unrelated cleanup.

review:
  agent: reviewer
  required: true

verify:
  agent: verifier
  required: true
```

For code modification, I would default to **one implementer at a time** unless each worker has an isolated git worktree. Parallel analysis is safe. Parallel editing in the same worktree is not.

# 11. How to do this in Jekko today without core changes

A partial version can be done today with:

* custom commands
* plugins
* compaction hooks
* Task subagents
* AGENTS.md
* external wrapper script

Jekko custom slash commands can be defined in `.jekko/commands/`, where command content becomes the prompt. ([Jekko][32]) Plugins can observe events and customize behavior, including compaction. ([Jekko][6])

But this will not be bulletproof because:

* `jekko run` currently exits when session becomes idle. ([GitHub][9])
* plugins appear limited for injecting AI-visible continuation at exact stop points, based on the community issue requesting such hooks. ([GitHub][7])
* context compaction can still cause pause/exit behavior in current reports. ([GitHub][11])
* an external wrapper cannot fully own TUI state, permissions, subagent ledger, or model-loop boundaries.

So today’s workaround can be useful, but the “promised infinite loop” needs core support.

# 13. Test plan

## Parser safety tests

* Reject `<FOREVER>` XML tags.
* Reject OACL without exact opener/closer.
* Reject OACL without `confirm`.
* Reject `forever: true` without `confirm_forever`.
* Reject stale/missing nonce when nonce mode is enabled.
* Reject unsupported version.
* Reject dangerous shell command without permission.

## Runtime loop tests

* Mock model emits `stop` immediately.
* Verify daemon continues to next iteration.
* Verify iteration count increments.
* Verify stop condition is the only successful exit.
* Verify manual stop exits.
* Verify pause preserves state.
* Verify resume continues from same run ID.

## Compaction tests

* Force compact every iteration.
* Verify daemon ledger survives.
* Verify current work item survives.
* Verify subtask state survives.
* Verify synthetic continuation brief is generated.

## Permission tests

* Denied `git push` pauses if policy says pause.
* Denied edit fails if policy says fail.
* Denied optional read skips if policy says skip.
* No daemon silently ignores permission errors by default.

## CLI tests

* `jekko daemon run` streams iteration events.
* CLI does not exit on inner model stop.
* CLI exits on `daemon.completed`.
* JSON output includes final ledger summary.

## TUI tests

* Gold forever banner appears.
* Iteration/phase/stop condition update live.
* Pause/stop controls work.
* Banner disappears after completion.
* Normal sessions unaffected.

## Multi-agent tests

* Two subagents cannot claim same work item.
* Expired lease requeues work.
* Failed subagent result is recorded.
* Join timeout cancels or pauses according to policy.
* Reducer receives structured artifacts.

## End-to-end test

Fake repo:

1. Create dirty files.
2. Fake `jankurai audit --json` returns findings.
3. Daemon fixes one item per iteration.
4. Commits after verification.
5. Fake audit eventually returns `{ findings: 0, caps: 0 }`.
6. Daemon exits successfully.

# 14. The final recommendation

Implement **Jekko Daemon Runbooks**, not `<FOREVER>` prompt tags.

The core commitments should be:

1. **The runtime owns forever.**
   The model’s `stop` is an iteration boundary.

2. **The syntax is deliberate.**
   Use runbook files first, slash commands second, gated in-prompt blocks third.

3. **State is durable.**
   Store daemon run state and ledger outside chat context.

4. **Context clearing is expected.**
   Compact at loop boundaries, but preserve daemon ledger and current work state.

5. **Multi-agent work uses a blackboard.**
   Replace shared Markdown trackers with structured claims, leases, artifacts, findings, and decisions.

6. **Permissions remain real.**
   Daemon mode must not silently bypass denials.

7. **The UI screams forever.**
   Gold banner, iteration count, phase, stop condition, and manual stop must be obvious.

The most important product sentence:

> **In daemon mode, Jekko must never ask the model whether the daemon is finished. It must ask the runbook.**
