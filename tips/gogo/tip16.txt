## Bottom line

Your instinct is right, but the implementation must **not** be “convince the model harder.” The reliable version is:

> **A host-owned daemon runtime + a deliberately armed runbook syntax.**
> The model may decide it is done; the runtime must not. The runtime owns the loop, context clearing, checkpointing, exit-condition evaluation, worker orchestration, and UI state.

There is **no mature universal standard** today for “paste this syntax into a coding-agent prompt and it will execute forever with guaranteed semantics.” What exists is a set of adjacent standards and patterns:

AGENTS.md is emerging as a project-instructions convention, not an execution language. MCP standardizes agent-to-tool connections, not agent loops. ACP standardizes editor/client-to-agent communication. A2A standardizes agent-to-agent interoperability. PDL, LMQL, Guidance, SGLang, DSPy, AICI, and related systems are closer to “LLM programming,” but they are not a drop-in prompt syntax for OpenCode daemon execution. The best OpenCode-native answer is a new **OpenCode Runbook Language** that is host-parsed, deliberately armed, durable, and treated as configuration for a daemon supervisor, not as ordinary prompt text. ([Agents][1])

OpenCode already has many of the pieces: command templates, agents/subagents, MCP tool integration, plugins, server/client architecture, and compaction. But today those pieces are not a durable “forever agent” runtime. Commands are prompt templates, MCP tools add tool surface/context, and the current `run` command exits when the session becomes idle. ([OpenCode][2])

The most important design correction to the AI proposal you pasted is this:

> Do **not** primarily hack `runLoop()` so “finish: stop” becomes “continue.”
> Build an **outer daemon supervisor** around the normal prompt loop. Let normal OpenCode finish an iteration. Then the daemon runtime checkpoints, evaluates exit conditions, clears/rotates context, and starts the next iteration.

OpenCode community issues already point in exactly this direction: `/loop`, continuous execution, Ultra Mode, background/parallel subagents, and DB-backed coordination have all been requested. The problems people are reporting are not just “the model gave up”; they are missing runtime control flow, fragile compaction behavior, headless permissions, synchronous subagents, and lack of durable coordination. ([GitHub][3])

# 1. What the broader ecosystem says about in-prompt agent scripting

## 1.1 The big lesson: prose is not control flow

The clearest current direction in agent engineering is that reliable agents need deterministic runtime control flow, not stronger prompt wording. One recent piece states the pattern directly: “MANDATORY,” “DO NOT SKIP,” and similar prompt pressure eventually hit a ceiling; robust systems move transitions, validation, and checkpoints into software. ([brian’s thoughts][4])

That maps perfectly to your frustration. The model stops because, from its perspective, it reached a coherent stopping point. A prompt tag like `<FOREVER>` can influence behavior, but it cannot guarantee runtime continuation. The guarantee must come from the host process.

So the syntax should not mean:

```text
Dear model, please loop forever.
```

It should mean:

```text
OpenCode runtime: create durable daemon job.
On session idle: continue.
On compaction: preserve daemon capsule.
On stop condition true: exit.
On manual kill: stop.
```

The model sees a task capsule. The host owns the loop.

---

## 1.2 Existing standards and where they fit

### AGENTS.md

AGENTS.md is an open convention for repository-level agent instructions. It gives agents predictable project guidance, build commands, test rules, style rules, and workflow notes. It is useful for static instructions, but it is not a daemon scripting language. ([Agents][1])

Use it for:

```text
How this repo works.
How tests run.
Coding standards.
Review expectations.
```

Do not use it for:

```text
Loop forever.
Spawn workers.
Commit every iteration.
Exit only when audit passes.
```

That belongs in a runtime runbook.

---

### MCP

MCP standardizes how LLM applications connect to tools and external systems. It defines hosts, clients, servers, and JSON-RPC communication. It is excellent for exposing GitHub, GitLab, databases, browsers, issue trackers, or custom project tools to agents. ([Model Context Protocol][5])

But MCP is mostly **agent-to-tool**, not **agent-to-agent**. OpenCode’s docs also warn that MCP servers add tool definitions to context and can push sessions toward context limits. ([OpenCode][6])

So for your “spin up multiple MCPs robustly” goal, I would phrase the architecture as:

> The daemon can enable multiple MCP servers or MCP profiles, but worker agents should be OpenCode sessions/subagents/remote agents. MCP should provide tools to those workers.

MCP is the toolbox. It is not the team manager.

---

### ACP

The Agent Client Protocol standardizes communication between code editors/IDEs and coding agents, similar in spirit to how LSP standardized editor/language-server communication. This matters for OpenCode because OpenCode already has a server/client architecture and could expose daemon state cleanly to TUI, IDE, desktop, and headless clients. ([Agent Client Protocol][7])

ACP-like thinking says: do not bury daemon state inside prompt text. Expose daemon jobs, iterations, workers, and events as API resources.

---

### A2A

A2A is the more relevant standard for true multi-agent collaboration. It is explicitly about agent-to-agent communication and collaboration, while MCP is agent-to-tool. ([A2A Protocol][8])

For OpenCode, A2A is probably a future-facing interoperability layer. The immediate implementation can be OpenCode-native: DB-backed jobs, worker sessions, a shared ledger, inboxes, artifacts, and events. Later, those worker endpoints could speak A2A.

---

### PDL, LMQL, Guidance, SGLang, DSPy, AICI

These systems are important prior art because they show that agent behavior becomes more reliable when prompts become programs.

PDL is a YAML-like declarative prompt programming language for chaining model calls, tool use, validation, and type checking. ([IBM][9])

Guidance supports control structures, constraints, tool use, and constrained generation. ([GitHub][10])

LMQL, SGLang, DSPy, and AICI each tackle parts of the same problem: programming language structure, runtime control, constraints, parallelism, or token-level steering.

The lesson for OpenCode is not “adopt one of these wholesale.” The lesson is:

> A serious agent DSL must have a parser, schema, runtime semantics, validation, and observability. It cannot be just XML-ish tags sent to the model.

---

### ACDL and context description

A newer research direction is formalizing how agent contexts evolve. ACDL, for example, describes agentic LLM contexts, their structure, and their evolution through loops. It is not a general execution language, but it is relevant because your “clear context and continue forever” problem is really a context-evolution problem. ([arXiv][11])

The right OpenCode design should separate:

```text
Execution language:
  loops, checkpoints, workers, conditions, permissions.

Context language:
  what must survive compaction/hard clear.
  what can be summarized.
  what must be reloaded.
```

---

## 1.3 What strong agent systems do in practice

The best current agent systems tend to use:

```text
1. Host-owned loop.
2. Durable state.
3. Explicit phase machine.
4. Context capsules/summaries.
5. Tool permissions.
6. Checkpoints.
7. Structured worker outputs.
8. Human/manual kill switch.
9. Observability.
```

Anthropic’s context-engineering guidance emphasizes that context is finite and agent loops accumulate data, so systems need to actively refine and manage what stays in the window. ([Anthropic][12])

Manus’s context-engineering writeup emphasizes stable prompts, append-only context, deterministic serialization, and avoiding tool/context explosion. That is directly relevant to OpenCode MCP usage: enabling many tools dynamically can confuse the model and waste context. ([Manus][13])

Martin Fowler’s agent article highlights subtasking with separate context windows and using deterministic scripts where appropriate. That maps cleanly to daemon epochs and worker sessions. ([martinfowler.com][14])

Claude Code hooks are another important comparison: hooks are deterministic shell commands at lifecycle points, explicitly replacing “hope the model remembers” with “the host always runs this.” ([Claude][15])

# 2. OpenCode-specific community signal

This is not an isolated need.

OpenCode issue #18001 asks for a native `/loop` command for automated iterative task execution. The issue says repetitive/time-based tasks currently need long prompts or external scripting and explicitly compares the desired behavior to Claude Code’s `/loop` style workflow. ([GitHub][3])

Issue #18636 asks for continuous execution loops where the agent resumes after compaction, milestones, or partial completion, and only stops when scoped work is complete, an external blocker appears, or missing input is required. It even sketches config like `continuous`, `resume_after_compaction`, and `stop_conditions`. ([GitHub][16])

Issue #23428 proposes an “Ultra Mode” that sounds like a phase-machine version of what you want: plan, implement, verify, iterate, and enforce transitions/tools. ([GitHub][17])

The multi-agent side is also actively discussed. Issue #19999 asks for ephemeral sub-agent teams because current task spawning is sequential, lacks communication, and lacks lead coordination. Issue #20849 proposes plugin-based orchestration to get fire-and-forget and parallel work without core changes. Issue #19215 proposes DB-backed agent team coordination using OpenCode sessions, messages, parts, and Bus events. ([GitHub][18])

Two OpenCode issues are especially important warnings for the daemon design. Issue #15533 describes an auto-compaction infinite loop caused by synthetic “continue” behavior after an assistant had naturally ended. Issue #13946 describes `opencode run` exiting after compaction because the synthetic continuation was skipped and the prompt exit check broke out. Together, they show that naïvely injecting “continue” into the existing prompt loop is fragile. ([GitHub][19])

There is also a headless permission problem: issue #14473 reports a bash permission ask hanging forever in headless/server mode. A daemon runtime cannot allow unresolved permission prompts to become invisible deadlocks. ([GitHub][20])

# 3. Review of the pasted AI plan

The pasted plan is directionally useful, but it is not bulletproof enough.

## What it gets right

It correctly identifies that the normal prompt loop exits when the assistant has a finish reason that is not tool calls, there are no pending tool calls, and the latest user message is older than the assistant message. That is the basic “model said it is done, so host stops” path. ([GitHub][21])

It correctly notices that OpenCode already has compaction, status events, agent step limits, MCP integration, a headless `run` command, and a TUI theme/status layer. OpenCode’s status model currently has `idle`, `retry`, and `busy`, and publishes session status events. ([GitHub][22])

It correctly understands that `opencode run` exits on idle today. The run command’s event loop listens for `session.status` and breaks when the status becomes idle. ([GitHub][23])

It correctly treats the TUI “gold forever” visual state as important. A forever loop must be visually obvious.

## What I would change

### 1. Do not make `runLoop()` itself the daemon

The plan proposes overriding the finish-stop exit path inside `prompt.ts`.

I would avoid that as the core design.

The current `runLoop()` is the inner LLM/tool loop. It should remain responsible for one normal OpenCode turn/session cycle. A daemon should be an **outer supervisor** that notices the inner loop is idle and then decides what to do next.

Why this matters:

```text
Existing runLoop semantics:
  user prompt → model/tool loop → assistant finishes → session idle.

Daemon semantics:
  run iteration → assistant finishes → checkpoint → evaluate exit → maybe compact/rotate → next iteration.
```

If you mutate `runLoop()` directly, you risk breaking normal TUI/chat semantics, structured output, compaction edge cases, and plugin expectations. The compaction-related issues already show how fragile synthetic continuation can be. ([GitHub][19])

Best design:

```text
SessionPrompt.prompt() remains normal.
DaemonSupervisor calls SessionPrompt.prompt() repeatedly.
```

### 2. Durable state cannot be optional

The pasted plan suggests session metadata or Storage KV.

For a promised forever mode, that is too weak.

A daemon needs durable, queryable state:

```text
daemon_job
daemon_iteration
daemon_event
daemon_task
daemon_worker
daemon_message
daemon_artifact
daemon_lock
```

Otherwise you cannot reliably resume after crashes, inspect what happened, prevent duplicate commits, coordinate workers, or render a correct TUI.

### 3. Status should not simply become `type: "daemon"`

Today status is `idle | retry | busy`. Replacing or overloading that with `daemon` can break existing clients. ([GitHub][22])

Better:

```ts
SessionStatus:
  idle | retry | busy

DaemonStatus:
  active | paused | stopping | blocked | completed | failed
```

Then expose daemon state as either:

```text
session.status.busy.detail.daemon
```

or a separate event:

```text
daemon.status
daemon.iteration.started
daemon.iteration.completed
daemon.worker.started
daemon.worker.completed
```

The TUI can combine them.

### 4. Raw shell strings are too dangerous as the main API

The pasted plan proposes:

```bash
--exit-condition "jankurai audit | grep ..."
--checkpoint "git add -A && git commit ... && git push"
```

That is useful as an advanced escape hatch, but it should not be the default runbook model.

The default should be typed commands:

```yaml
exit:
  when:
    all:
      - shell:
          argv: ["jankurai", "audit", "--json"]
          parse: json
          expect:
            "$.findings": 0
            "$.caps": 0

checkpoint:
  after_each_completed_item:
    - git.add: { mode: related_files_only }
    - git.commit:
        message_template: "jankurai: complete ${task.id}"
    - git.push: { remote: origin, branch: current }
```

Typed commands allow:

```text
timeouts
cwd
env filtering
output limits
JSON parsing
idempotency
redaction
permission checks
structured trace logs
```

Raw shell can remain available behind an explicit `dangerously_allow_shell: true`.

### 5. Permission denials should not be silently ignored

The pasted plan says permission denial could be logged and continued.

That is risky. A permission denial usually means the agent tried to do something the runtime did not allow. In a forever loop, silently continuing can create endless thrashing.

Better policy:

```text
TUI daemon:
  permission ask may pause the daemon and show a gold blocking prompt.

Headless daemon:
  ask is forbidden unless an explicit noninteractive policy exists.
  unresolved asks become daemon.blocked, not hidden hangs.

Danger mode:
  allowed only with explicit activation and big UI warning.
```

This is especially important because OpenCode has had headless permission ask freezes. ([GitHub][20])

### 6. Context clearing should be epoch-based, not only compaction-based

OpenCode already has compaction. It summarizes the conversation into sections like goal, constraints, progress, next steps, critical context, and files. It can also inject a continuation message after compaction. ([GitHub][24])

But “forever” needs a stronger model:

```text
Soft compact:
  summarize and continue same session.

Hard clear:
  start a new epoch/session with only a daemon capsule.

Epoch rotation:
  every N iterations, after commit, after context risk, or after no-progress loop.
```

The daemon job, not the chat transcript, must be the source of truth.

### 7. MCP scaling cannot be only prompt injection

The pasted plan says to inject:

```text
Use jnoccio_spawn_parallel for throughput.
```

That is not robust enough.

A real multi-agent daemon needs:

```text
worker sessions
task leases
file/resource locks
worker heartbeats
structured outputs
merge/reconcile policy
artifact publication
shared event bus/inbox
optional separate git worktrees
MCP profiles per worker
```

OpenCode already has a `general` subagent that can run multiple independent tasks in parallel, and subagents can be invoked by the primary agent, but community issues indicate the current model is not enough for parallel fire-and-forget coordinated teams. ([OpenCode][25])

### 8. The syntax must be host-parsed and inert unless armed

The XML-ish syntax in the original prompt is dangerous:

```xml
<FOREVER>
...
</FOREVER>
```

Problems:

```text
Could be accidentally pasted.
Could appear in docs.
Could be copied from an issue.
Could be treated as ordinary prompt text.
Could be partially malformed.
Could be followed inconsistently by the model.
```

The syntax must require:

```text
exact opening delimiter
exact closing delimiter
version
id
nonce
activation phrase
explicit slash command or CLI flag
schema validation
optional spec hash
```

The model should never decide whether a runbook is active. The OpenCode host parser decides.

# 4. Best OpenCode architecture

I would call the feature:

```text
OpenCode Daemon Runtime
OpenCode Runbook Language, or ORL/OXL
```

I will use **OXL** below: **OpenCode Execution Language**.

The key principle:

```text
OXL is not a prompt.
OXL is a host-parsed runbook embedded near a prompt.
```

## 4.1 Layer model

```text
┌─────────────────────────────────────────────────────────────┐
│                    TUI / CLI / IDE / API                    │
│  gold FOREVER badge, job controls, event stream, kill/pause │
└──────────────────────────────┬──────────────────────────────┘
                               │
┌──────────────────────────────▼──────────────────────────────┐
│                  Daemon Supervisor Service                  │
│  loop, state machine, checkpoints, exits, workers, context  │
└───────────────┬──────────────┬──────────────┬───────────────┘
                │              │              │
┌───────────────▼──────┐ ┌─────▼────────┐ ┌───▼──────────────┐
│ Existing Prompt Loop │ │ Daemon Store │ │ Condition Runner │
│ SessionPrompt.prompt │ │ DB + events  │ │ typed checks     │
└───────────────┬──────┘ └─────┬────────┘ └───┬──────────────┘
                │              │              │
┌───────────────▼──────────────▼──────────────▼───────────────┐
│               Tools, MCP servers, subagents, git             │
└──────────────────────────────────────────────────────────────┘
```

The daemon supervisor is the new owner of “forever.”

The existing OpenCode prompt loop remains the owner of “one model/tool iteration.”

---

## 4.2 Runtime state machine

```text
created
  ↓
armed
  ↓
running
  ↓
iteration_start
  ↓
prompting_existing_opencode_loop
  ↓
session_idle_or_error
  ↓
classify_result
  ↓
checkpoint
  ↓
evaluate_exit
  ├── exit true  → completed
  ├── blocked    → blocked
  ├── fatal      → failed
  └── continue   ↓
context_policy
  ↓
worker_reconcile
  ↓
next_iteration
```

Manual controls should be available at any point:

```text
pause
resume
stop_gracefully
kill_now
```

For “FOREVER,” `max_iterations` is absent and `on_model_stop: continue` is set. The runtime continues until the explicit exit predicate passes or the process/job is manually stopped.

# 6. Context-clearing model

The term “clear context” must be made precise.

There are three levels:

## 6.1 Soft compaction

Use OpenCode’s existing compaction, but with daemon-aware preservation.

The daemon compaction prompt must preserve:

```text
objective
non-negotiable constraints
exit criteria
current selected task
open task ledger summary
last audit result
last commit SHA
known blockers
critical decisions
files changed
verification already performed
```

## 6.2 Epoch rotation

After a configured boundary, start a fresh session/epoch:

```text
after_each_commit
every N iterations
after token usage > 80%
after no-progress detector triggers
after worker merge/reconcile
after compaction failure
```

The new epoch receives a deterministic daemon capsule:

```yaml
daemon_capsule:
  job_id: ...
  iteration: ...
  objective: ...
  active_task: ...
  exit_conditions: ...
  permissions: ...
  recent_progress: ...
  last_commit: ...
  open_findings: ...
  next_action: ...
```

The old transcript is no longer required for correctness.

## 6.3 Hard clear

Hard clear means:

```text
Do not include old chat turns.
Do not include irrelevant tool logs.
Do not include stale reasoning.
Only include daemon capsule + selected files + current task.
```

This is the correct meaning of your original `<CLEAR_AGENT_CONTEXT>` idea.

It should never mean “forget the job state.”

# 7. Multi-agent collaboration design

The current “read tracker, update tracker” approach works only because it is simple. It breaks down when workers run in parallel.

The robust design is a lead/worker model.

```text
Lead daemon
  owns global objective
  selects work
  creates leases
  spawns workers
  reviews artifacts
  merges patches
  runs final verification
  commits and pushes

Worker agent
  owns one leased task
  has narrow prompt
  has narrow tool/MCP profile
  works in isolated session/worktree
  publishes structured artifact
  does not directly mutate global state except through daemon APIs
```

## 7.1 Team flow

```text
Lead iteration starts
  ↓
Discover work items
  ↓
Create task records
  ↓
Acquire file/task locks
  ↓
Spawn N workers
  ↓
Workers produce artifacts
  ↓
Lead joins/reconciles
  ↓
Run verification
  ↓
Commit/push
  ↓
Update ledger
  ↓
Evaluate exit
```

## 7.2 Worker isolation

For code changes, each worker should ideally get:

```text
separate OpenCode session
separate git worktree or branch
narrow file lease
narrow tool permission set
limited MCP profile
heartbeat
timeout/backoff policy
structured output schema
```

This avoids worker collisions and makes merge/review possible.

## 7.3 Shared communication

Replace “central chatter file” with:

```yaml
channels:
  - name: lead
  - name: workers
  - name: review
  - name: blockers
  - name: findings
```

Messages are structured:

```yaml
kind: finding
from: worker-3
task: task-17
severity: high
files:
  - src/foo.ts
summary: "Regression in diff parser."
evidence:
  - command: "pnpm test diff-parser"
    result: "failed before patch, passed after patch"
```

Markdown mirrors can be generated:

```text
DAEMON_STATUS.md
DAEMON_TASKS.md
DAEMON_DECISIONS.md
```

But agents should not coordinate by editing those files concurrently.

# 8. The deliberate syntax gate

This is the part I would be strict about.

Do **not** use:

```xml
<FOREVER>
...
</FOREVER>
```

Use an unmistakable runbook block that is inert unless armed.

## 8.1 Required properties

The syntax must have:

```text
exact opening delimiter
exact closing delimiter
version
runbook id
nonce
activation phrase
explicit invocation path
schema validation
optional spec hash
dangerous-mode acknowledgement
paste behavior: inert unless armed
```

That means the same text pasted into a normal chat does nothing special.

## 8.2 Proposed delimiter

```text
<<<OPENCODE.RUNBOOK:v1 id="jankurai-clean-tree" nonce="manual-2026-05-07-jankurai">>>
...
<<<END OPENCODE.RUNBOOK>>>
```

This is better than XML tags because:

```text
it is OpenCode-specific
it is versioned
it carries an id
it carries a nonce
it is unlikely to appear accidentally
it is easy for a host parser to find
it is visibly not normal prose
```

## 8.3 Required activation

The runbook should only activate through one of these:

```bash
opencode daemon start --runbook runbook.oxl --arm "ARM OPENCODE DAEMON jankurai-clean-tree manual-2026-05-07-jankurai"
```

or:

```text
/daemon start jankurai-clean-tree
ARM OPENCODE DAEMON jankurai-clean-tree manual-2026-05-07-jankurai
```

or:

```bash
opencode run --daemon --runbook runbook.oxl --arm "ARM OPENCODE DAEMON ..."
```

A normal paste into chat should produce:

```text
Detected OpenCode runbook block, but it is inert because daemon activation was not requested.
Use /daemon inspect or /daemon start to validate/arm it.
```

This is the “never accidentally pasted and run” requirement.

# 9. Proposed OXL syntax: core example

Here is the syntax I would design for your Jankurai scenario.

```yaml
<<<OPENCODE.RUNBOOK:v1 id="jankurai-clean-tree" nonce="manual-2026-05-07-jankurai">>>
activation:
  requires:
    invocation: "/daemon start jankurai-clean-tree"
    arm_phrase: "ARM OPENCODE DAEMON jankurai-clean-tree manual-2026-05-07-jankurai"
    paste_behavior: inert_unless_armed
    allowed_clients: ["tui", "run"]
    allowed_repo:
      git_remote_contains: "opencode"
    dry_run_first: true

mode:
  kind: daemon
  owner: host_runtime
  loop: forever
  on_model_stop: continue
  manual_stop: allowed

objective: |
  Work through the dirty worktree created while implementing @agent/JANKURAI_STANDARD.md
  and @JANKURAI_TASKLIST.md. Pick one dirty worktree item at a time. Review it,
  fix regressions, verify it, commit it, and push completed work.

sources:
  project_rules:
    - "AGENTS.md"
    - "agent/JANKURAI_STANDARD.md"
    - "JANKURAI_TASKLIST.md"

  work_items:
    - git.dirty_files:
        include_untracked: true
    - file.tasks:
        path: "JANKURAI_TASKLIST.md"
        statuses: ["TODO", "IN_PROGRESS"]

loop:
  select:
    strategy: random
    from: work_items
    lease:
      ttl_minutes: 60
      renew: true

  iteration_prompt: |
    Investigate exactly one selected dirty worktree item.
    Run git diff for the relevant files.
    Look for mistakes, possible regressions, and missing tests.
    Make the smallest correct change.
    Verify the change with focused tests and the Jankurai audit process.
    Do not claim completion until verification evidence exists.

  on_no_progress:
    detector: fingerprint
    threshold: 3
    action: rotate_context_then_continue

verify:
  required:
    - shell:
        id: jankurai_audit
        argv: ["jankurai", "audit", "--json"]
        timeout_seconds: 300
        parse: json
        expect:
          "$.findings": 0
          "$.caps": 0
        on_failure:
          append_findings_to: "JANKURAI_TASKLIST.md"

    - git:
        id: reviewed_diff
        require_reviewed_diff: true

tasklist:
  completed_items:
    move_matching:
      from: "JANKURAI_TASKLIST.md"
      to: "JANKURAI_TASKLIST.md.completed"
      status: "COMPLETED"

checkpoint:
  after_each_completed_item:
    - git.add:
        mode: related_files_only

    - git.commit:
        require_nonempty: true
        message_template: "jankurai: complete ${task.id}"

    - git.push:
        remote: origin
        branch: current
        on_reject: pull_rebase_retry_once

exit:
  when:
    all:
      - condition: "jankurai_audit.findings == 0"
      - condition: "jankurai_audit.caps == 0"
      - condition: "tasklist.open_count == 0"
      - condition: "git.dirty_reviewed_count == 0"
  otherwise: continue_forever

context:
  strategy: epoch

  compact:
    when:
      - "token_usage > 0.80"
      - "after_each_commit"
    preserve:
      - objective
      - exit
      - current_task
      - last_audit
      - last_commit
      - open_findings
      - decisions
      - verification_evidence

  hard_clear:
    every_iterations: 3
    carry: daemon_capsule

permissions:
  headless:
    ask_policy: block_not_hang

  bash:
    allow:
      - ["git", "*"]
      - ["jankurai", "audit", "*"]
      - ["npm", "test", "*"]
      - ["pnpm", "test", "*"]
    deny:
      - ["rm", "-rf", "*"]
    default: ask_in_tui_block_in_headless

  files:
    allow_write:
      - "**"
    deny_write:
      - ".git/**"
      - "**/.env*"

mcp:
  profiles:
    default:
      enable: []
      lazy: true

team:
  enabled: false

ui:
  forever_badge:
    enabled: true
    label: "FOREVER"
    color: gold
    show:
      - iteration
      - phase
      - exit_summary
      - last_commit
<<<END OPENCODE.RUNBOOK>>>
```

If `jankurai audit --json` does not exist, allow text parsing:

```yaml
verify:
  required:
    - shell:
        id: jankurai_audit
        argv: ["jankurai", "audit"]
        parse: regex
        expect_match: "0 findings.*0 caps"
```

The schema should prefer JSON because regex-based audit checks are brittle.

# 15. Specific answers to your original XML concepts

## `<FOREVER>`

Replace with:

```yaml
mode:
  kind: daemon
  loop: forever
  owner: host_runtime
  on_model_stop: continue
```

The word `forever` means:

```text
The daemon supervisor continues after model/session idle.
No iteration limit is applied.
Only explicit exit predicates, fatal configured policy, or manual stop end the job.
```

## `<GIT>`

Replace with typed checkpoints:

```yaml
checkpoint:
  after_each_completed_item:
    - git.add: { mode: related_files_only }
    - git.commit: { message_template: "..." }
    - git.push: { remote: origin, branch: current }
```

## `<EXIT_IF>`

Replace with host-owned exit conditions:

```yaml
exit:
  when:
    all:
      - condition: "jankurai_audit.findings == 0"
      - condition: "jankurai_audit.caps == 0"
      - condition: "tasklist.open_count == 0"
  otherwise: continue_forever
```

The model can advise whether something is complete, but the host evaluates the actual exit condition.

## `<CLEAR_AGENT_CONTEXT>`

Replace with context policy:

```yaml
context:
  strategy: epoch
  hard_clear:
    every_iterations: 3
    carry: daemon_capsule
```

This guarantees context clearing without losing durable state.

# 16. The one design rule I would not compromise on

The runtime should treat the model’s final answer as only an **iteration result**, not as authority to stop the daemon.

In pseudocode terms:

```text
while job.active:
  result = run_one_normal_opencode_prompt_iteration()

  record(result)

  checkpoint_result = run_checkpoints()
  exit_result = evaluate_exit_conditions()

  if manual_stop_requested:
    stop

  if exit_result.true:
    complete

  if fatal_blocker:
    block_or_fail_by_policy

  apply_context_policy()

  continue
```

The model can say:

```text
“All done.”
```

The daemon should reply internally:

```text
“Great. Now I will verify the exit predicate.”
```

If the predicate fails, it continues.

That is the difference between “agent that usually tries” and “daemon that actually keeps going.”

# 17. Recommended v1 scope

For the first serious version, I would not try to ship the entire God-mode syntax. I would ship this:

```text
deliberately armed runbook block
single-agent daemon loop
typed exit conditions
typed git checkpoints
daemon-aware context capsule
hard context rotation
headless permission blocking
gold TUI forever badge
daemon status/events
crash-resumable durable state
```

Then v2:

```text
worker sessions
task ledger
worker inbox/artifacts
file locks
MCP profiles
parallel scatter/gather
```

Then v3:

```text
A2A interoperability
remote workers
scheduler/watch mode
plugin lifecycle ecosystem
advanced UI dashboards
```

That sequence gives you the core promise first:

> The agent does not stop just because the model stopped talking.
