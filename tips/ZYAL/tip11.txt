I’m interpreting **OpenClaw**, **LangChane**, and **Hermis** as **OpenClaw**, **LangChain/LangGraph**, and **Hermes Agent**. I also checked the nearby “wheels” ZYAL should not reinvent: MCP, A2A, AG-UI, A2H, Agent Client Protocol, AGENTS.md, Agent Skills, OpenAI Agents SDK, Microsoft Agent Framework, Temporal-style durable execution, OPA/Rego, W3C PROV, in-toto/SLSA, CycloneDX, SARIF, Letta, Zep, LangMem, and Mem0.

## Overall judgment

ZYAL’s strongest idea is excellent: **the model thinks, but the host governs**. The draft already has the right center of gravity: host-owned loops, preview-before-arming, bounded daemon work, SQLite-backed state, evidence-gated promotion, anti-vibe gates, capability leases, rollback, budgets, repo intelligence, and human approvals. That is meaningfully different from a prompt convention or a normal agent framework. 

The biggest risk is that ZYAL becomes **another private orchestration DSL**. The biggest opportunity is to make ZYAL the **governance/control-plane protocol** that composes with existing agent standards: MCP for tools/context, A2A for agent-to-agent work, AG-UI for frontend event streams, Agent Client Protocol for IDE/coding-agent integration, A2H for auditable human approvals, and existing provenance/security standards for evidence. MCP is already the standard shape for connecting LLM apps to tools/data/context, A2A is designed for independent agents to discover capabilities and collaborate without sharing private internals, and AG-UI standardizes real-time agent/frontend state and event flow. ([Model Context Protocol][1])

OpenClaw and Hermes make the memory/skills problem concrete. OpenClaw is an always-on personal assistant across messaging channels and local devices; Hermes explicitly emphasizes a self-improving learning loop, persistent memory, skill creation, scheduled automations, and cross-session recall. Those are exactly the long-loop capabilities ZYAL should govern rather than duplicate ad hoc. ([GitHub][2])

LangGraph, Microsoft Agent Framework, OpenAI Agents SDK integrations, and Temporal already cover much of durable execution, state, pause/resume, human-in-the-loop, checkpointing, and long-running workflows. ZYAL should not compete with those runtimes; it should define the **contract those runtimes can enforce**. ([LangChain Docs][3])

## The biggest gaps I see

First, **memory is not yet governed enough**. ZYAL says memory exists, but long-loop agents need typed, provenance-bearing, revocable memory. Modern memory systems distinguish thread memory, archival memory, temporal knowledge graphs, prompt/behavior optimization, and hybrid vector/key-value/graph recall; ZYAL should make those explicit instead of treating memory as just persisted notes. ([LangChain Docs][4])

Second, **evidence needs real attestations, not just hashes**. The draft’s `evidence.sign: sha256` is a red flag: SHA-256 is a digest, not a signature or trust chain. ZYAL should use signed provenance/attestation models, borrowing from W3C PROV, in-toto, SLSA, CycloneDX, and SARIF where possible. W3C PROV gives a vocabulary for entities, activities, and responsible agents; in-toto and SLSA focus on supply-chain metadata and tamper resistance; CycloneDX covers BOMs including AI/ML-BOM; SARIF standardizes static-analysis results. ([W3C][5])

Third, **skills are a supply-chain attack surface**. ZYAL’s skill quarantine/promotion idea is important, but it needs to be much harder-edged. Snyk’s ToxicSkills research found critical security issues and prompt-injection/malware patterns across agent skills, and Microsoft/Cisco both frame OpenClaw-style agents around identity, execution, persistence, prompt injection, and skill malware risk. ([Snyk][6])

Fourth, **capabilities need a policy engine**. Regex command floors are useful, but serious deployments will need policy-as-code: ABAC/RBAC, purpose-bound scopes, lease expiry, revocation, risk scoring, and enforcement decoupled from the runtime. OPA/Rego is already built for domain-agnostic policy decisions over structured data. ([Open Policy Agent][7])

Fifth, **human approvals need verifiable consent, not just a pause**. The draft has approval gates, but A2H is directly relevant: it defines a channel-agnostic, auditable surface for agents to inform, collect, authorize, escalate, and report results to humans. ZYAL approvals should emit A2H-compatible authorization requests and store signed approval receipts. ([Twilio][8])

Also, one concrete consistency issue: the draft says state persists in **9 SQLite tables**, but the visible list names only **8**: `daemon_run`, `daemon_iteration`, `daemon_event`, `daemon_task`, `daemon_task_pass`, `daemon_task_memory`, `daemon_worker`, and `daemon_artifact`. That should be fixed because trust in an operating contract starts with exactness. 

## Top 10 best ideas to dramatically upgrade ZYAL

### 1. Add an `interop` block: make ZYAL the control plane over MCP, A2A, AG-UI, A2H, and ACP

ZYAL should declare which external protocol surfaces are active, what they may do, and how they map into ZYAL’s own ledger. Example: MCP tools become governed capabilities; A2A agents become remote workers with leases and evidence requirements; AG-UI events become the frontend event stream; A2H becomes the human approval/consent channel; Agent Client Protocol becomes the IDE bridge. Agent Client Protocol already standardizes editor-to-coding-agent communication over local or remote transports, so ZYAL should plug into it rather than invent an IDE transport. ([Agent Client Protocol][9])

**Spec upgrade:**

```yaml
interop:
  mcp:
    servers:
      - id: github
        profile: readonly_until_approved
        capabilities_from: policy
  a2a:
    expose_agent_card: true
    accepted_remote_agents:
      - trust: signed_org
        max_task_risk: medium
  ag_ui:
    emit_events: [run, step, tool, state_delta, approval, evidence]
  a2h:
    required_for: [credential_use, external_write, payment, production_deploy]
  agent_client_protocol:
    enabled: true
    diff_streaming: true
```

This turns ZYAL into the **governed glue layer** for the agent ecosystem rather than a competing ecosystem.

### 2. Build a true Memory OS: typed, temporal, permissioned, and poison-resistant

ZYAL needs more than “context rotation” and persisted task memory. Add memory strata: working memory, task memory, episodic memory, semantic memory, procedural/skill memory, negative memory, repo memory, user preference memory, and organization policy memory. Every memory should have source, author, confidence, validity interval, retention policy, access policy, poison score, and deletion path.

This is the heart of smarter long loops. Zep’s temporal graph model shows why time-aware relationships matter; Letta’s archival memory shows the value of searchable out-of-context storage; LangMem focuses on learning/adaptation and prompt refinement; Mem0-style systems combine vector, key-value, and graph approaches. ([Zep Documentation][10])

**Spec upgrade:**

```yaml
memory:
  stores:
    working: { scope: run, max_tokens: 24000 }
    task: { scope: task, retention: 90d }
    semantic: { scope: repo, backend: vector_graph, retention: 1y }
    negative: { scope: repo, require_evidence: true }
    procedural: { scope: skill, write_requires: quarantine_review }
  governance:
    require_provenance: true
    allow_model_to_write: proposed_only
    human_review_for: [user_profile, security_policy, skill_instruction]
    poison_detection: { enabled: true, quarantine_on: suspicious_instruction }
    deletion: { supported: true, tombstone: true }
```

The critical design rule: **the model can propose memory, but the host commits memory**.

### 3. Replace “proof bundles” with a cryptographic evidence graph

ZYAL already has evidence-gated promotion. Upgrade it into an append-only, signed evidence graph. Each node should represent an artifact, patch, test result, approval, tool call, model invocation, memory write, sandbox output, or rollback plan. Each edge should represent derivation, dependency, approval, or use.

Borrow the shape from W3C PROV and in-toto/SLSA rather than inventing a new evidence vocabulary. Use CycloneDX for SBOM/AI-BOM outputs and SARIF for static-analysis/security findings. ([W3C][5])

**Spec upgrade:**

```yaml
evidence:
  graph: true
  attestations:
    format: in_toto_statement
    signer: host_kms
    include:
      - patch
      - tests
      - typecheck
      - static_analysis_sarif
      - sbom_cyclonedx
      - approvals
      - rollback
  provenance:
    model: w3c_prov
    hash_algorithm: sha256
    signature_algorithm: ed25519
```

This makes “tests passed” falsifiable, replayable, and audit-ready.

### 4. Add a policy-as-code capability engine

The current `capabilities` and `command_floor` idea is right, but the DSL should not become an entire authorization language. Let ZYAL call a policy backend such as OPA/Rego or Cedar-like ABAC/RBAC. OPA’s model—structured input, declarative policy, decoupled decision-making—fits ZYAL well. ([Open Policy Agent][7])

**Spec upgrade:**

```yaml
policy:
  engine: opa
  bundle: .opencode/policy/zyal.rego
  decision_points:
    - tool_call
    - memory_write
    - skill_install
    - checkpoint
    - remote_agent_delegate
    - approval_auto_grant
  fail_closed: true
```

This dramatically increases enterprise credibility because security teams can inspect policies without reading agent code.

### 5. Make durable execution semantics explicit: replay, idempotency, compensation

ZYAL says SQLite is truth, but long-running autonomy also needs deterministic replay, idempotent tool calls, resumable waits, and compensation for failed side effects. Temporal’s agent integrations emphasize durable execution, retries, state preservation, and long-running human-in-the-loop workflows; OpenAI’s Agents SDK also points to Temporal, Restate, and DBOS for durable long-running agents. ([Temporal Docs][11])

**Spec upgrade:**

```yaml
durability:
  replay:
    deterministic: required
    nondeterministic_calls: activity_only
  idempotency:
    tool_calls: required
    external_writes: required
  compensation:
    required_for: [deploy, migration, external_api_write]
  stores:
    primary: sqlite
    distributed: optional_postgres
```

This would make ZYAL feel less like a loop runner and more like an autonomous workflow kernel.

### 6. Treat skills as packages with SBOMs, signatures, sandbox profiles, and quarantine

Agent Skills and AGENTS.md are valuable because they package reusable agent context and workflows; Anthropic describes Skills as filesystem-based, on-demand modular capabilities, while AGENTS.md is a predictable project-level instruction format for coding agents. ([Claude Platform][12])

But skills are also dangerous. ZYAL should require signed skill manifests, declared permissions, dependency locks, no remote mutable instructions by default, memory-write restrictions, sandbox profiles, and staged promotion. Snyk’s agent-skills research is the clearest warning that skills need supply-chain rigor, not just convenience. ([Snyk][6])

**Spec upgrade:**

```yaml
skills:
  install_policy: signed_or_local_only
  manifest_required: true
  sbom: cyclonedx
  allow_remote_fetch: false
  sandbox_default: no_network
  memory_write: proposed_only
  lifecycle:
    new: quarantine
    promote_requires: [scan_clean, eval_passed, human_review]
```

This turns “self-improving skills” from a scary feature into a governed capability.

### 7. Upgrade approvals into signed human intent receipts

ZYAL’s `approvals` block should support signed approval receipts, role thresholds, multi-party approval, revocation, and exact binding to the patch/action being approved. A2H’s `AUTHORIZE`, `COLLECT`, `ESCALATE`, `INFORM`, and `RESULT` intents are a good external protocol fit. ([Twilio][8])

**Spec upgrade:**

```yaml
approvals:
  protocol: a2h
  receipt:
    bind_to: [run_id, script_hash, diff_hash, risk_score, rollback_hash]
    authn: webauthn
    signature: required
  gates:
    production_deploy:
      required_roles: [tech_lead, service_owner]
      quorum: 2
      expires: 30m
```

This prevents “the human clicked OK somewhere” from becoming an unverifiable claim.

### 8. Create an ZYAL Flight Recorder using OpenTelemetry conventions

ZYAL already has observability, but it should standardize telemetry so traces can flow into existing systems. OpenTelemetry semantic conventions give shared names for traces, metrics, logs, profiles, and resources; OpenAI Agents SDK has built-in tracing for LLM generations, tool calls, handoffs, guardrails, and custom events. ([OpenTelemetry][13])

**Spec upgrade:**

```yaml
observability:
  otel:
    enabled: true
    semantic_namespace: zyal.*
    spans:
      - zyal.run
      - zyal.iteration
      - zyal.model_call
      - zyal.tool_call
      - zyal.memory_retrieval
      - zyal.approval
      - zyal.evidence_attestation
  redaction:
    secrets: required
    pii: policy
  replay_bundle:
    include: [events, prompts_redacted, tool_io_hashes, evidence_graph]
```

This makes ZYAL debuggable across teams and vendors.

### 9. Add an eval-and-regression layer for agents, memories, prompts, and skills

Hypothesis tournaments are powerful, but ZYAL needs a repeatable eval system: golden tasks, historical failure replay, prompt/skill regression tests, model routing benchmarks, cost/performance curves, and “promotion cannot degrade known tasks.” LangGraph/LangSmith, OpenAI tracing/evals, and Microsoft workflow checkpointing show that production agents need inspection and evaluation loops, not just runtime loops. ([LangChain Docs][14])

**Spec upgrade:**

```yaml
evals:
  suites:
    - id: repo_regression
      tasks: .opencode/evals/repo/*.yml
    - id: skill_regression
      tasks: .opencode/evals/skills/*.yml
  run_before:
    - skill_promote
    - model_route_change
    - checkpoint
  metrics:
    - correctness
    - diff_minimality
    - tool_safety
    - memory_precision
    - cost
```

This turns “self-improvement” into measurable improvement.

### 10. Publish ZYAL conformance levels and a reference test harness

To become significant, ZYAL should have **levels**, not just features. Think SLSA-style maturity, but for autonomous agents. SLSA’s value is that it turns supply-chain security into named levels and controls; ZYAL can do the same for agent autonomy. ([SLSA][15])

Possible levels:

| Level       | Meaning                                                                    |
| ----------- | -------------------------------------------------------------------------- |
| **ZYAL-L0** | Valid YAML/runbook preview only                                            |
| **ZYAL-L1** | Host-owned loop, budgets, stop conditions                                  |
| **ZYAL-L2** | Durable state, checkpointing, guardrails, rollback                         |
| **ZYAL-L3** | Signed evidence, policy-as-code, sandbox, memory governance                |
| **ZYAL-L4** | Multi-agent interop, A2A/MCP/AG-UI/A2H adapters, eval gates                |
| **ZYAL-L5** | Federated fleets, cryptographic provenance graph, audited self-improvement |

Add a conformance CLI:

```bash
zyal conformance test --level L3 ./runbook.zyal.yml
zyal evidence verify .opencode/daemon/<runID>/evidence/
zyal replay --from-ledger .opencode/daemon/<runID>/ledger.jsonl
```

This is how ZYAL stops being “a cool OpenCode feature” and becomes a credible protocol.

## Specific fixes I would make immediately

1. **Change `evidence.sign: sha256` to real signing.** Use `hash: sha256` plus `signature: ed25519 | kms | sigstore`, and bind signatures to run ID, repo, branch, commit, diff hash, and approval receipt.

2. **Fix the SQLite table mismatch.** The draft says 9 tables but lists 8. Either add the missing table or correct the claim. 

3. **Make model-derived risk scores non-authoritative.** A model can suggest risk, but the host should compute risk from static facts: touched paths, dependency graph, secret access, generated files, test coverage deltas, security sinks, external writes, and blast radius.

4. **Ban regex-only shell safety.** Regex checks are bypassable. Parse commands structurally where possible, run inside sandbox profiles, and evaluate against policy.

5. **Add memory deletion and correction.** Long-lived agent memory without tombstones, expiry, source tracking, and human correction will eventually poison itself.

6. **Make “negative memory” evidence-bound.** A failed lane should not become permanent negative memory unless the failure was verified and categorized. Otherwise the agent can learn false taboos.

7. **Separate “assistant-visible memory” from “host-only audit memory.”** The model should not see every audit fact, secret, reviewer note, or security finding.

8. **Add distributed fleet semantics.** SQLite is a good local source of truth, but fleets need leases, leader election, remote worker identity, queue semantics, and eventually optional Postgres/event-log backends.

9. **Define generated-code/test-deletion gates formally.** Anti-vibe rules are great, but “assertion weakening” needs language-specific analyzers and SARIF-compatible outputs, not only textual heuristics.

10. **Make rollback executable.** A rollback plan should include `verify`, `apply`, `post_verify`, owner, blast radius, and timeout—not just prose.

## The north-star framing

ZYAL should become:

> **A vendor-neutral, host-enforced autonomy contract for long-running agents, with standard adapters for tools, agents, humans, UIs, memory, evidence, policy, and observability.**

That framing keeps ZYAL from reinventing LangGraph, OpenClaw, Hermes, MCP, A2A, AG-UI, Agent Skills, AGENTS.md, Temporal, OPA, SLSA, or OpenTelemetry. It makes ZYAL the missing layer above them: **the durable constitution for autonomous work.**
