## Bottom line

ZYAL is already unusually strong. It is not merely a prompt format; it is a **host-enforced run contract** with lifecycle control, budgets, evidence gates, approval gates, sandboxing, observability, model routing, anti-vibe quality checks, hypothesis tournaments, rollback, repo intelligence, and governed memory. That is the right architectural center of gravity. The most important upgrade is to make ZYAL the **governance kernel** that composes existing protocols rather than replacing them. The uploaded spec already frames this well: “the model does the thinking; ZYAL does the governing.” 

I’m assuming “OpenClaw, LangChane, and Hermis” refer to **OpenClaw**, **LangChain/LangGraph**, and **Hermes/Hermes Agent-style tool execution**. I also compared ZYAL against MCP, A2A, AG-UI, ACP, OpenAI Agents SDK, AutoGen/AG2, CrewAI, Microsoft Agent Framework, Strands, AgentSkills/SKILL.md, AGENTS.md, Cursor rules, MemGPT/Letta/Zep/Mem0/LlamaIndex memory patterns, and supply-chain attestation standards.

The highest-leverage direction: **turn ZYAL into the durable, replayable, evidence-attested, memory-governed operating contract that wraps MCP/A2A/AG-UI/ACP/skills/rules instead of trying to be all of them.**

## Existing wheels ZYAL should compose, not reinvent

**MCP** should be treated as the default agent↔tool/data substrate. It already standardizes hosts, clients, servers, resources, prompts, tools, roots, sampling, elicitation, progress, cancellation, and explicit security principles around consent, privacy, and tool safety. ZYAL should govern MCP servers and tool calls, not replace MCP. ([Model Context Protocol][1])

**A2A** should be the default agent↔agent delegation protocol. It already defines agent cards, capabilities, skills, task lifecycles, streaming updates, artifacts, and long-running task patterns. ZYAL should use A2A for federated workers while enforcing capability leases, evidence requirements, and memory boundaries. ([A2A Protocol][2])

**AG-UI** should be the default event protocol for agent↔frontend interaction. It already supports lifecycle events, text/tool/state/reasoning events, interrupts, human-in-the-loop steering, shared state, and tool output streaming. ZYAL’s TUI “gold mode” could emit AG-UI-compatible events so any external UI can observe or steer a run. ([AG-UI][3])

**ACP** should be the default editor/IDE bridge for coding agents. It standardizes editor↔agent communication via JSON-RPC, sessions, prompts, cancellation, filesystem and terminal capabilities, diffs, environment control, and extensibility. ZYAL should expose an ACP-compatible surface so editors can run, inspect, pause, approve, and replay ZYAL sessions. ([Agent Client Protocol][4])

**LangGraph/LangChain** already cover durable stateful agents, persistence/checkpointers, memory stores, interrupts, human-in-the-loop execution, and long-running agent workflows. ZYAL’s SQLite truth model is aligned, but it should go further into deterministic replay and idempotent side effects. ([LangChain Docs][5])

**OpenClaw** is especially relevant because it has a local-first gateway, multi-channel inbox, multi-agent routing, skills, live canvas, companion apps, and Markdown-based memory. Its memory docs distinguish context from memory and include background “dreaming,” memory wiki structure, claims/evidence, contradiction tracking, and freshness tracking. However, OpenClaw’s own docs say main-session tools run on the host by default, and recent reporting around malicious OpenClaw skills highlights why ZYAL needs first-class skill supply-chain controls and context-firewalling. ([GitHub][6])

**Hermes-style tool execution** is a major missing power pattern. Hermes Agent lets an agent write code that calls tools programmatically through a local socket, so intermediate tool results do not all flood the context window; only final output returns to the model. That is exactly the kind of mechanism ZYAL needs for long loops over logs, code search, tests, dependency graphs, and issue trackers. ([Hermes Agent][7])

**AgentSkills/SKILL.md, AGENTS.md, Cursor rules, Claude/OpenAI/Microsoft skill formats** should be treated as imported instruction and capability artifacts. The AgentSkills format centers on `SKILL.md` plus optional scripts, references, and assets; AGENTS.md has become a simple open instruction format for coding agents; Cursor has project/team/user rules plus AGENTS.md support. ZYAL should normalize these into governed context, skills, and capability policy. ([Agent Skills][8])

---

## Main gaps and risks I see

The biggest conceptual gap is **interop formalization**. ZYAL includes MCP support and many runtime blocks, but it should explicitly declare itself as the **run governance layer** above MCP, A2A, AG-UI, ACP, AgentSkills, AGENTS.md, and framework runtimes. Otherwise it risks becoming a giant YAML that overlaps everyone.

The second gap is **deterministic replay**. SQLite truth is excellent, but “durable” is not the same as “replayable.” For CI-grade autonomous engineering, every model response, tool result, shell output, file diff, clock/random/network dependency, approval decision, and side effect needs replay semantics. Temporal’s durable execution model records event history so workflows resume from the last recorded event, and its side-effect/idempotency guidance is highly relevant here. ([Temporal Docs][9])

The third gap is **memory as a full operating system**, not just memory stores. ZYAL already has governed memory, provenance, retention, compression, and redaction, but long-loop agents need typed memory layers, temporal validity, contradiction handling, confidence, task-phase retrieval policy, memory-write gates, negative memory, procedural memory promotion, and memory evals. CoALA, MemGPT, Reflexion, Voyager, Zep, Letta, Mem0, LlamaIndex, LangChain, and OpenClaw all provide pieces of this. ([arXiv][10])

The fourth gap is **context security**. ZYAL’s arming block already rejects assistant/tool/web-origin arming, which is excellent. But every context atom should also carry a taint label: trusted user, repo code, tool output, web content, MCP resource, assistant-generated text, memory, skill, test output, etc. MCP itself warns that tool descriptions are untrusted and that implementers must enforce consent and safety; OpenClaw’s skill incident shows why this must be structural, not advisory. ([Model Context Protocol][1])

The fifth gap is **guardrail coverage invariants**. ZYAL should specify that guardrails wrap every action channel, including hosted tools, MCP tools, handoffs, A2A workers, skill scripts, shell, browser, editor operations, and generated code execution. OpenAI’s Agents SDK documentation is a useful warning: tool guardrails apply to function tools but not all other tool/handoff channels, so ZYAL should define universal coverage as a host invariant. ([OpenAI GitHub][11])

One small but telling spec issue: the uploaded ZYAL document says durable state persists in **9 SQLite tables**, but the listed table names appear to enumerate **8**. That kind of mismatch is minor, but before ZYAL becomes a standard, all schema counts, names, block versions, and conformance claims should be mechanically validated.

# Top 10 ideas to dramatically upgrade ZYAL

## 1. Add an explicit `interfaces` block: ZYAL as the governance spine

ZYAL should not define every protocol surface itself. It should declare adapters to existing protocols and define what ZYAL governs for each one: permissions, taint, evidence, approvals, budgets, memory access, and replay.

```yaml
interfaces:
  mcp:
    servers:
      - id: github
        profile: locked_down
        roots: ["repo"]
        allow_tools: ["issues.search", "pulls.create"]
        taint: untrusted_external

  a2a:
    enabled: true
    require_agent_card: true
    delegated_tasks_require:
      - capability_lease
      - evidence_bundle
      - memory_scope

  ag_ui:
    emit_events:
      - run_lifecycle
      - tool_call
      - state_delta
      - approval_request
      - evidence_update

  acp:
    expose_editor_session: true
    permission_bridge: host

  instruction_files:
    read: ["AGENTS.md", "CLAUDE.md", ".cursor/rules/**/*.mdc"]
    normalize_to: zyal_context_policy

  agent_skills:
    import: ["skills/**"]
    require_signed: true
    quarantine_new: true
```

This would make ZYAL more significant because it becomes the **policy and lifecycle layer for the agent ecosystem**, not merely an OpenCode-specific runbook. MCP handles tools, A2A handles agent delegation, AG-UI handles frontend state/events, ACP handles IDE integration, and AgentSkills/AGENTS.md handle portable instructions. ZYAL governs them. ([Model Context Protocol][1])

---

## 2. Add deterministic replay, fork, and audit mode

ZYAL’s “SQLite is truth” principle is very good, but the next leap is **replayable autonomy**. A run should be reproducible, forkable, auditable, and bisectable like a build pipeline.

```yaml
replay:
  mode: record_and_replay
  record:
    - model_response
    - tool_result
    - shell_stdout
    - shell_stderr
    - file_diff
    - env_snapshot
    - random_seed
    - clock_value
    - approval_decision
    - external_side_effect
  side_effects:
    require_idempotency_key: true
    on_duplicate: reuse_recorded_result
  fork:
    allow_from:
      - checkpoint
      - iteration
      - tool_call
      - approval_gate
  audit:
    verify_hash_chain: true
    compare_replay_to_original: true
```

This creates “agent CI”: a failed run can be replayed, a successful run can be independently verified, and a suspicious decision can be forked before the bad tool call. LangGraph already emphasizes durable execution through persisted checkpoints; Temporal’s event-history and idempotency model shows how to handle long-running workflows and side effects safely. ([LangChain Docs][5])

---

## 3. Upgrade `memory` into a full Memory OS

This is probably the single biggest capability upgrade for long loops. ZYAL should define a **memory kernel** with typed layers, provenance, temporal validity, write gates, retrieval budgets, contradiction handling, and memory-quality evals.

```yaml
memory:
  kernel: true

  layers:
    working:
      max_tokens: 24000
      eviction: priority_lru

    episodic:
      store: sqlite
      summarize_after_iterations: 5
      include: [decisions, failed_attempts, tool_results, approvals]

    semantic:
      store: temporal_graph
      validity: { track_valid_from_to: true }
      contradiction_policy: create_objection

    procedural:
      store: skills
      promotion_requires: [human_review, eval_passed, signed_artifact]

    negative:
      capture_failed_hypotheses: true
      prevent_repeat_when_similar: true

    repo_graph:
      indexes: [symbols, dependency_graph, test_graph, ownership]

  write_policy:
    require_evidence_ref: true
    require_source_taint: trusted_or_verified
    confidence_min: 0.65
    redact: [secrets, pii]
    decay_unreferenced_after: 30d

  retrieval_policy:
    budget_tokens: 6000
    require_attribution: true
    rerank_by: [relevance, recency, confidence, task_phase, risk, ownership]
    forbid_taints_for: { shell_command: [web_untrusted, assistant_generated] }

  eval:
    memory_lift_test: true
    hallucination_regression: true
    stale_fact_detection: true
```

The memory model should combine: CoALA’s modular cognitive memory framing, MemGPT’s virtual-context hierarchy, Reflexion’s reflective episodic memory, Voyager’s reusable skill library, Zep’s temporal knowledge graph, Letta’s always-visible memory blocks, Mem0’s factual/episodic/semantic layering, LlamaIndex’s short-term flushing into long-term memory blocks, and OpenClaw’s Markdown memory plus contradiction/freshness tracking. ([arXiv][10])

The key principle: **memory writes should be governed like code writes**. A memory entry should have evidence, scope, confidence, author, validity dates, taint, retention, and revocation. Bad memory is worse than no memory in a long-running agent.

---

## 4. Add a context firewall with taint tracking

ZYAL already has strong arming controls. Extend that idea to every context item and tool path.

```yaml
context_firewall:
  taint_classes:
    - trusted_user
    - repo_code
    - test_output
    - tool_output
    - web_untrusted
    - mcp_resource
    - a2a_worker
    - assistant_generated
    - memory
    - skill

  rules:
    - from: [web_untrusted, mcp_resource, tool_output]
      to: [arming, approvals.auto_approve]
      decision: deny

    - from: [web_untrusted, mcp_resource]
      to: shell_command
      decision: require_human

    - from: [assistant_generated]
      to: memory.procedural
      decision: require_review

    - from: [skill]
      to: secrets
      decision: deny_unless_signed_and_approved

  sanitize:
    detect_prompt_injection: true
    strip_tool_like_instructions_from_untrusted: true
    quarantine_suspicious_context: true
```

This directly addresses prompt injection, malicious tool descriptions, poisoned MCP resources, compromised skills, and cross-agent contamination. MCP’s own security guidance treats tool descriptions as untrusted and emphasizes explicit user consent; OpenClaw’s malicious-skill incident shows that skill supply chain risk is not hypothetical. ([Model Context Protocol][1])

---

## 5. Make capabilities true object-capability leases

ZYAL’s existing `capabilities` block is one of its best ideas. It should become a true **object-capability system** covering tools, paths, commands, network egress, credentials, data classes, memory layers, side effects, and external APIs.

```yaml
capabilities:
  default: deny

  leases:
    - id: github-pr-write
      subject: agent:builder
      tools: [mcp.github.create_pr]
      paths: ["src/**", "tests/**"]
      network:
        allow: ["api.github.com"]
        deny: ["*"]
      data_classes:
        may_read: ["repo_public", "test_output"]
        may_write: ["pull_request"]
        may_not_see: ["secrets", "customer_data"]
      secrets:
        broker: github_app_installation_token
        scope: pull_request_write
        expires: 30m
      renew:
        requires: [progress_evidence, budget_remaining]
      revoke_on:
        - taint_violation
        - no_progress
        - anomalous_egress

  command_floor:
    always_block:
      - "git push --force"
      - "rm -rf /"
      - "sudo "
      - "DROP DATABASE"
```

This is especially important for MCP and ACP. MCP defines roots, tools, and security principles, while ACP makes the client responsible for environment and permissions; ZYAL should be the higher-level policy that makes those permissions auditable, revocable, budgeted, and evidence-aware. ([Model Context Protocol][1])

---

## 6. Add Hermes-style `tool_programs` for token-efficient long loops

Long-running agents die when every grep result, test log, dependency graph, and tool trace gets dumped into the context window. ZYAL should let the host run **audited, sandboxed tool programs** that call tools in loops and return compact, typed summaries.

```yaml
tool_programs:
  enabled: true
  language: python
  sandbox: locked_down
  allowed_tools:
    - repo.search
    - repo.read_file
    - tests.run
    - dependency_graph.query
  context_policy:
    intermediate_results_enter_context: false
    final_output_schema: analysis_summary
    max_final_tokens: 2000
  audit:
    record_tool_calls: true
    record_intermediate_artifacts: true
    attach_to_evidence: true
```

This borrows the best part of Hermes Agent: programmatic tool use through code, where intermediate results do not enter the model context and only final output is returned. For monorepos, CI logs, flaky tests, dependency analysis, and security scans, this is a massive power multiplier. ([Hermes Agent][7])

The safety requirement: these programs must be sandboxed, capability-scoped, replay-recorded, and output-schema-validated. Otherwise they become a bypass around ZYAL’s governance.

---

## 7. Replace flat proof bundles with a signed evidence graph

ZYAL already has typed evidence and SHA-256 signing. Upgrade that into a **content-addressed evidence DAG** with cryptographic attestations.

```yaml
evidence_graph:
  store: merkle_dag
  attestations:
    format: in_toto_statement
    sign: sigstore_keyless
  nodes:
    - type: diff
    - type: test_results
    - type: typecheck_results
    - type: security_scan
    - type: model_decision_summary
    - type: approval_decision
    - type: rollback_plan
    - type: replay_segment
  promotion_paths:
    production_merge:
      require:
        - tests_passed
        - typecheck_passed
        - security_scan_passed
        - rollback_verified
        - no_unresolved_critical_objections
  verify:
    before_checkpoint: true
    before_merge: true
    after_replay: true
```

This would make ZYAL proof bundles legible to CI, security, compliance, and supply-chain tooling. In-toto and SLSA already define attestation/provenance patterns for software supply chains, and Sigstore provides signing/verification infrastructure for artifacts and SBOMs. ([SLSA][12])

The important distinction: a hash proves content identity; an attestation proves **who/what produced it, under what policy, with what inputs, and for what purpose**.

## 8. Add federated worker contracts using A2A agent cards

ZYAL’s multi-agent orchestration should be contract-first. Every delegated worker should advertise its capabilities, trust level, cost model, sandbox, supported evidence types, memory access needs, and data policy. Then the supervisor can assign work without treating external agents as opaque black boxes.

```yaml
federation:
  discovery:
    - local_pool
    - a2a_registry

  require_agent_card: true

  worker_contract:
    must_advertise:
      - skills
      - models
      - cost_model
      - sandbox
      - evidence_types
      - memory_policy
      - data_policy
      - supported_artifacts

  delegation:
    require:
      - capability_lease
      - task_scope
      - evidence_return_schema
      - no_memory_write_without_host_review

  critic_policy:
    must_use_different_provider: true
    blind_review: true
```

A2A already provides agent cards, task lifecycle, streaming updates, artifacts, and long-running task support. ZYAL should add the safety and governance layer: leases, budgets, memory scopes, evidence schemas, and revocation. ([A2A Protocol][2])

This would make ZYAL useful not only for OpenCode, but also for mixed fleets: local agents, SaaS agents, CI agents, security agents, code review agents, and human-in-the-loop reviewers.

---

## 9. Add continuous evals, memory-lift tests, and anti-vibe red-team suites

ZYAL has quality gates, but it should also continuously measure whether the agent system is actually getting smarter. Add an `evals` block that can run before promotion, after memory updates, after skill promotion, and during replay.

```yaml
evals:
  suites:
    - repo_regression
    - prompt_injection
    - memory_recall
    - memory_lift
    - anti_vibe
    - cost_quality
    - tool_safety
    - flaky_test_handling

  acceptance:
    no_new_guardrail_bypass: true
    no_test_weakening: true
    memory_lift_gte: 0.10
    cost_per_verified_fix_lte: baseline_plus_20_percent

  shadow:
    compare_models: [builder, fallback_builder]
    compare_memory_policies: [current, no_memory, graph_memory]

  report:
    emit_opentelemetry: true
    attach_to_evidence_graph: true
```

The killer metric is **memory lift**: does this memory system measurably improve future task success, time-to-fix, fewer repeated mistakes, lower cost, or better security outcomes? Without memory evals, memory systems become folklore. OpenTelemetry’s GenAI semantic conventions are useful here because ZYAL can emit standardized spans, metrics, and events for model calls, tools, costs, guardrails, and workflow phases. ([OpenTelemetry][13])

This also protects against “self-improving” skills that quietly degrade quality. OpenClaw-style skills and Voyager-style skill libraries are powerful, but promotion should require evals, quarantine, provenance, and human review. ([OpenClaw][14])

---

## 10. Add sagas for non-git side effects and make rollback universal

ZYAL’s rollback model is strong for code, but autonomous agents increasingly touch external systems: GitHub issues, PRs, Slack, email, databases, cloud resources, CI settings, feature flags, package registries, billing systems, and deployment tools. Git rollback is not enough.

```yaml
side_effects:
  classify:
    external_write: require_compensation
    irreversible: require_human
    credential_change: require_security_approval
    customer_visible: require_approval

  compensation:
    require_for_tools:
      - github.merge_pr
      - slack.post_message
      - email.send
      - db.migrate
      - cloud.deploy
      - feature_flag.update

    schema:
      must_include:
        - undo_action
        - verification_command
        - owner
        - max_safe_delay
        - manual_escalation

  two_phase:
    enabled_for: [deploy, migration, customer_visible]
    prepare_then_commit: true

  verify_after:
    enabled: true
    on_failure: compensate_or_escalate
```

This makes ZYAL viable beyond code editing. In distributed workflow terms, many external actions need compensation, idempotency keys, and replay-safe side-effect recording. Durable execution systems make this distinction central; ZYAL should too. ([Temporal Docs][9])

## Additional standardization move: define ZYAL conformance levels

This is not a separate feature so much as an adoption requirement. The current ZYAL spec is broad: 40+ top-level blocks, multiple runtime systems, UI, security, memory, approvals, workflow, experiments, and repo intelligence. That is powerful, but it risks overwhelming implementers.

Define levels:

```yaml
conformance:
  level: ZYAL-3
  schema_version: "v2.2"
  profiles:
    - coding
    - secure_enterprise
    - federated_agents

levels:
  ZYAL-0: preview_parse_validate_only
  ZYAL-1: bounded_single_agent_execution
  ZYAL-2: evidence_gated_checkpointing
  ZYAL-3: memory_replay_capabilities_sandbox
  ZYAL-4: federated_a2a_attested_evidence_graph
```

This would let other tools adopt ZYAL incrementally. MCP, A2A, AG-UI, ACP, AgentSkills, and AGENTS.md all benefit from being small enough to implement or adapt. ZYAL should keep its “full power” mode, but standardize smaller profiles so it can spread. ([Model Context Protocol][1])

---

## Highest-priority changes

If I had to pick only four, I would prioritize:

1. **Interop spine**: MCP + A2A + AG-UI + ACP + AgentSkills + AGENTS.md adapters.
2. **Memory OS**: typed temporal memory with provenance, contradiction, confidence, retrieval policy, and evals.
3. **Replay ledger**: deterministic replay, idempotent side effects, forkable runs.
4. **Context firewall**: taint tracking across web, tools, MCP, skills, memory, assistant output, and workers.

Those four turn ZYAL from “a very powerful autonomous coding runbook” into a credible **operating contract standard for long-running autonomous work**.
