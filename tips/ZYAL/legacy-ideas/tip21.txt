I did a broad survey across primary docs/repos, research papers, security advisories, and recent social/forum signals. I would not claim this exhausts every X, LinkedIn, Quora, or forum post on the internet; I weighted primary sources, open-source docs, papers, and security writeups much more heavily than social commentary.

## Executive verdict

ZYAL is **not redundant**. It is aiming at a different layer than OpenHands, LangGraph, Hermes, OpenClaw, Claude Code, Codex, AutoGen, CrewAI, and Google ADK.

The strongest framing is:

> **ZYAL should become the Kubernetes + Temporal + OPA + evidence ledger for autonomous coding agents.**

OpenHands is a coding-agent platform/runtime. LangGraph is a programmable durable workflow framework. Hermes is a persistent self-improving personal/dev agent with memory and skills. OpenClaw is a wide personal automation agent. Claude Code and Codex are coding-agent products with sandboxes, hooks, skills, and cloud sessions. ZYAL can sit **above** all of them as a host-enforced, portable, prompt-embeddable control manifest that says: what may run, for how long, with what memory, in which sandboxes, using which agents, with what evidence, and under what approval gates.

ZYAL already has the correct core philosophy: **host owns the loop, model does not own completion, SQLite is truth, preview is separate from arming, promotion requires evidence, and safety is structural**. That is exactly the direction long-running agent research is converging on: persistent state outside the model context, incremental task selection, clean git states, test gates, and explicit handoff artifacts between sessions. Anthropic’s long-running-agent harness work found that agents need artifacts like a feature list, git history, and progress files to bridge context windows, and that forcing one feature at a time plus clean commits reduced premature “done” behavior. ([Anthropic][1])

The missing piece is that ZYAL v1 is still mostly a **loop-and-guardrail runbook**. To become the default language for the world’s most ambitious coding projects, it needs to become a **typed operating contract for autonomous engineering organizations**: memory, task graph, durable replay, skill lifecycle, codebase indexing, experiment branches, review quorum, merge train, provenance, security labels, model routing, observability, and interoperable adapters.

# 1. What ZYAL already gets right

ZYAL’s best ideas are structurally sound:

1. **Host-owned control**: the model cannot decide when to arm, stop, promote, push, or call a dangerous tool.
2. **Preview-before-arming**: `ZYAL_ARM RUN_FOREVER` is a strong human-intent sentinel.
3. **SQLite as truth**: durable state outside the context window is essential for long-running agents.
4. **Stop conditions and checkpoints**: completion is evaluated by host checks, not assistant self-report.
5. **Incubator passes**: hard tasks are routed through scout, idea, critic, synthesize, prototype, and promotion review.
6. **Write scopes**: scratch versus isolated worktree is a crucial distinction.
7. **Guardrails and constraints**: policies are enforced by runtime, not by “please remember this” prompts.
8. **On-handlers with max-depth 1**: prevents event-handler cascades.
9. **Permission surface**: shell, edit, git commit, workers, MCP, push.
10. **Mirror files**: human-readable state is generated, while SQLite remains authoritative.

This is already better than “vibe coding with a cron loop.” It resembles the practitioner “Ralph loop”: pick unfinished task, prompt the agent, run checks, update progress, and repeat, with persistent task/progress files outside the model context. Addy Osmani’s long-running-agent writeup describes this exact shape and emphasizes that state must live outside the agent context because the agent is effectively amnesiac between sessions. ([addyosmani.com][2])

The upgrade opportunity is to make ZYAL **more typed, more replayable, more interoperable, more secure, and more project-scale**.

# 2. Ecosystem comparison: what ZYAL should absorb, avoid, or interoperate with

## OpenHands

OpenHands has three major things ZYAL should learn from: a typed append-only event model, sandboxed runtimes, and remote workspaces. Its docs describe an immutable, type-safe event framework where events form an append-only log used as both agent memory and an integration point for services. The event system includes action events, observation events, condensation summaries, pause events, and state updates. ([OpenHands Docs][3])

OpenHands also emphasizes Docker sandboxing for safe execution, consistency, resource control, isolation, and reproducibility. ([OpenHands Docs][4]) Its conversation API explicitly includes condensation and pause semantics, but also notes that direct tool execution can bypass the normal agent loop, confirmation policies, and security analyzers, which is exactly the kind of edge ZYAL should guard against at the host layer. ([OpenHands Docs][5])

**ZYAL upgrade:** add a first-class `event_schema`, `action_observation_log`, and `tool_bypass_policy`. ZYAL should be able to run OpenHands as a worker backend, but ZYAL should remain the policy/control plane.

## LangGraph / LangChain

LangGraph’s strongest contribution is durable execution semantics. Its docs define durable execution as saving progress at key points so workflows can pause and resume, including after human-in-the-loop interruptions and failures. It also stresses deterministic/idempotent workflow design and wrapping side effects or nondeterministic operations inside tasks. ([LangChain Docs][6])

LangGraph’s memory model also cleanly separates short-term thread-scoped memory from long-term cross-session memory, with custom namespaces. ([LangChain Docs][7])

**ZYAL upgrade:** add deterministic replay, idempotency keys, side-effect wrappers, and short/long/procedural memory namespaces. ZYAL’s current `context` block is not enough.

## Hermes Agent

Hermes is especially relevant because it has moved aggressively into persistent memory, self-improvement, skills, and durable task boards. Hermes memory is bounded, curated, injected as a frozen snapshot at session start, and split into user/profile and agent/project notes. ([Hermes Agent][8]) Hermes also distinguishes always-in-context memory from SQLite FTS5 session search, and it scans memory entries for injection and exfiltration patterns before accepting them. ([Hermes Agent][8])

Hermes skills are on-demand knowledge documents loaded through progressive disclosure, and Hermes allows agent-managed skills as procedural memory when the agent discovers repeatable workflows. ([Hermes Agent][9]) Its May 2026 “Tenacity” release added a durable multi-agent Kanban board with heartbeat, reclaim, zombie detection, retries, hallucination recovery, `/goal` persistence, checkpoint pruning, gateway auto-resume, and watchdog mode. ([GitHub][10])

**ZYAL upgrade:** add `memory`, `skills`, and `kanban` as first-class typed modules. But ZYAL should be stricter than Hermes: agent-created skills should be quarantined, signed, audited, scored, and promoted only with host approval.

## OpenClaw

OpenClaw’s strength is breadth: messaging channels, browser, canvas, nodes, cron, sessions, Discord/Slack actions, multi-agent routing, onboarding, and skills. Its README describes first-class tools for browser/canvas/nodes/cron/sessions/messaging and a security model where default tools can run on the host for the main session, with non-main sessions sandboxable. ([GitHub][11])

That breadth is also its risk. Cisco’s security blog warns that OpenClaw-style personal agents can run shell commands, read/write files, and execute scripts, creating harmful outcomes if misconfigured or if malicious skills are installed. ([Cisco Blogs][12]) Jamf Threat Labs similarly describes OpenClaw-like agents as broad automation engines with long-term memory and extensive application access, while highlighting risks such as unrestricted system access, lateral movement, data exfiltration, and prompt injection. ([Jamf][13]) Oasis reported a 2026 OpenClaw vulnerability chain allowing website-to-local-agent takeover, later fixed by the project. ([OASIS Security][14])

**ZYAL upgrade:** do not copy OpenClaw’s “wide host access by default.” ZYAL should support wide automation only through capability-scoped sandboxes, explicit network/egress policies, signed skills, taint tracking, and high-impact human approval.

## Claude Code

Claude Code has converged on many features ZYAL should formalize: parallel sessions, long-running browser/cloud tasks, hooks, permissions, checkpointing, MCP, skills, subagents, cost tracking, and OpenTelemetry. Its docs describe long-running cloud tasks and multiple parallel tasks. ([Claude][15]) The Claude Agent SDK exposes permissions, hooks, checkpointing, cost tracking, OpenTelemetry, todo lists, subagents, MCP, and skills as programmable capabilities. ([Claude][16])

Claude’s permissions documentation also highlights a subtle but important point: Bash permission patterns that try to constrain arguments are fragile, and shell/network restrictions need stronger mediation than simple pattern matching. ([Claude][17]) Claude hooks are deterministic lifecycle commands that can enforce rules instead of relying on the model to remember them. ([Claude][18])

**ZYAL upgrade:** replace regex-only command guardrails with shell AST parsing, command classification, taint labels, network egress mediation, and deterministic hooks.

## Codex

Codex is relevant because it shows the direction of hosted coding agents: many independent tasks in isolated cloud sandboxes, each with repository context, tests, terminal logs, and evidence of actions. OpenAI’s Codex launch described a cloud software engineering agent that can work on many tasks in parallel; each task runs in its own isolated environment, can read/edit files and run test harnesses, and provides verifiable evidence from terminal logs and test outputs. ([OpenAI][19]) Codex also reads `AGENTS.md` before work and layers global and project-specific guidance. ([OpenAI Developers][20])

**ZYAL upgrade:** add `evidence_bundles`, `agent_instruction_sources`, and `adapter.codex` so Codex can be used as one worker type under ZYAL policy.

## AutoGen, CrewAI, Google ADK, Pydantic AI

AutoGen is a multi-agent conversation framework where customizable agents combine LLMs, human input, and tools. ([Microsoft][21]) CrewAI focuses on role-playing autonomous agents, “crews,” event-driven “flows,” observability, and enterprise control planes. ([GitHub][22]) Google ADK is an open-source agent framework for building, debugging, deploying, evaluating, and scaling multi-agent systems with predictable pipelines or dynamic routing. ([Google Cloud Documentation][23]) Pydantic AI’s current positioning includes YAML/JSON-defined agents, MCP, A2A, UI event streams, human-in-the-loop approval, durable execution, structured outputs, and graphs. ([Pydantic][24])

**ZYAL upgrade:** do not compete with all of these as a Python framework. Instead, ZYAL should be the declarative host policy and runbook that can compile to, invoke, or wrap these systems.

## MCP, A2A, ACP, AGENTS.md

ZYAL should embrace open standards rather than inventing every interface. MCP defines resources, prompts, and tools, with utilities for progress tracking, cancellation, error reporting, and logging. ([Model Context Protocol][25]) A2A is designed for agents from different vendors/frameworks to communicate, securely exchange information, and coordinate actions. ([Google Developers Blog][26]) ACP standardizes communication between editors/IDEs and coding agents, similar in spirit to how LSP standardized language-server/editor integration. ([Agent Client Protocol][27]) AGENTS.md is now a simple open format for giving coding agents predictable project guidance. ([GitHub][28])

**ZYAL upgrade:** add explicit `interop` support for MCP, A2A, ACP, AGENTS.md, CLAUDE.md, Codex config, OpenHands workers, Hermes skills, and OpenCode sessions.

# 3. The biggest ZYAL gaps

## Gap 1: ZYAL has SQLite truth, but not a typed event ledger

ZYAL says SQLite is truth, but it does not yet define a canonical event schema. That is dangerous because long-running autonomy lives or dies on replayability, observability, and forensic reconstruction.

Add:

```yaml
event_log:
  schema_version: v1
  storage:
    primary: sqlite
    mirror: ".opencode/daemon/{{run_id}}/ledger.jsonl"
  append_only: true
  hash_chain: true
  event_types:
    - run_started
    - run_paused
    - run_resumed
    - run_aborted
    - iteration_started
    - model_call_started
    - model_call_completed
    - tool_call_requested
    - tool_call_allowed
    - tool_call_blocked
    - tool_call_completed
    - file_diff_proposed
    - file_diff_applied
    - test_started
    - test_completed
    - checkpoint_created
    - checkpoint_failed
    - memory_read
    - memory_write_requested
    - memory_write_accepted
    - memory_write_rejected
    - approval_requested
    - approval_granted
    - approval_denied
    - promotion_requested
    - promotion_blocked
    - promotion_completed
  integrity:
    include_prompt_hash: true
    include_tool_args_hash: true
    include_diff_hash: true
    include_artifact_hashes: true
```

Why this matters:

* You can replay what happened.
* You can prove a tool call was approved.
* You can reconstruct which memory was injected.
* You can audit why a promotion happened.
* You can detect tampering in mirror files.
* You can build real dashboards and traces.

## Gap 2: ZYAL needs durable replay and idempotency

A long-running daemon will be interrupted. Machines reboot. Tokens fail. LLM calls time out. Git rebases fail. Tool calls partially complete.

Add:

```yaml
durability:
  replay:
    mode: deterministic
    resume_from: last_committed_event
    require_idempotency_keys: true
  side_effects:
    default: wrapped_task
    idempotency_key: "{{run_id}}/{{task_id}}/{{iteration}}/{{tool_call_id}}"
    on_replay:
      completed_side_effect: reuse_recorded_result
      missing_result: rerun_with_same_idempotency_key
      mismatched_result: pause
  nondeterminism:
    record_model:
      provider: true
      model: true
      temperature: true
      seed: true
      prompt_hash: true
      response_hash: true
    record_environment:
      image_digest: true
      git_sha: true
      dependency_lock_hash: true
```

This is the LangGraph/Temporal lesson: do not just “persist state.” Define what replay means. Side effects must be wrapped, recorded, and idempotent. ([LangChain Docs][6])

## Gap 3: ZYAL’s memory model is too shallow

The existing `context` block handles compaction and clearing. That is not memory. Real autonomous coding needs at least five memory classes:

1. **Run memory**: what happened in this run.
2. **Task memory**: what was learned about this task.
3. **Project memory**: repo conventions, architecture, known traps.
4. **Procedural memory**: repeatable workflows and skills.
5. **User/org memory**: preferences, approval norms, risk tolerance.

Add:

```yaml
memory:
  enabled: true

  stores:
    run:
      type: sqlite_events
      retention: until_run_archived

    task:
      type: sqlite
      namespace: "task/{{task_id}}"
      mirror: ".opencode/daemon/{{run_id}}/tasks/{{task_id}}/MEMORY.md"

    project:
      type: sqlite_fts5
      namespace: "repo/{{repo_fingerprint}}"
      mirror: ".opencode/memory/PROJECT.md"
      max_hot_tokens: 1800

    semantic:
      type: vector
      namespace: "repo/{{repo_fingerprint}}"
      sources:
        - code_symbols
        - docs
        - prior_runs
        - issue_history
      retrieval:
        top_k: 12
        rerank: true
        require_citations: true

    procedural:
      type: skills
      namespace: "org/{{org_id}}/repo/{{repo_id}}"
      allow_agent_create: quarantine_only
      require_promotion_review: true

    user:
      type: profile
      namespace: "user/{{user_id}}"
      max_hot_tokens: 600
      write_requires:
        - explicit_user_request
        - memory_curator_approval

  injection:
    mode: frozen_snapshot
    at:
      - session_start
      - hard_clear
    max_tokens: 3500
    include:
      - project
      - task
      - selected_procedural_skills
      - current_goal
      - last_checkpoint_summary
    exclude:
      - raw_logs
      - secrets
      - untrusted_web_content

  curator:
    schedule: every_10_iterations
    scoring:
      utility: 0.35
      recency: 0.15
      correctness: 0.25
      security_risk: -0.25
    actions:
      - consolidate
      - deduplicate
      - expire
      - promote_to_project_memory
      - demote_to_archive
    on_uncertain: ask

  safety:
    scan_for_prompt_injection: true
    scan_for_secret_exfiltration: true
    block_invisible_unicode: true
    memory_write_quarantine: true
    require_source_event: true
```

Hermes is a good reference here: bounded hot memory, frozen prompt snapshot, session search, capacity management, and memory security scanning are all important. ([Hermes Agent][8])

## Gap 4: ZYAL needs a project/task graph, not just discovered tasks

For 1B+ token/code-scale projects, a flat task list is not enough. You need a DAG, dependencies, ownership, acceptance criteria, risk, test impact, and merge sequencing.

Add:

```yaml
plan:
  source: host
  storage: sqlite
  mirror: ".opencode/daemon/{{run_id}}/PLAN.md"

  task_graph:
    type: dag
    nodes:
      schema:
        id: string
        title: string
        objective: string
        status: enum[todo, active, blocked, incubating, ready, merged, parked]
        dependencies: array
        owner_agent: string
        affected_areas: array
        acceptance_criteria: array
        required_evidence: array
        risk_score: number
        estimated_complexity: number
        max_files_changed: integer
        max_diff_lines: integer
    scheduling:
      strategy: critical_path_then_risk
      max_active_tasks: 8
      block_if_dependencies_unmet: true
      require_smallest_shippable_slice: true

  feature_list:
    path: ".opencode/daemon/{{run_id}}/FEATURES.json"
    format: json
    rule:
      agent_may_update_status_only: true
      forbid_delete_or_rewrite: true
    default_status: failing
```

This directly addresses the Anthropic finding that a comprehensive feature list prevents agents from one-shotting an app or declaring completion too early. ([Anthropic][1])

## Gap 5: ZYAL needs “evidence bundles” as a first-class object

Right now, `checkpoint.verify` can run commands. Good, but not enough. The host should collect and grade a bundle of evidence.

Add:

```yaml
evidence:
  bundles:
    checkpoint:
      collect:
        - git_diff
        - changed_files
        - test_output
        - lint_output
        - typecheck_output
        - coverage_delta
        - affected_tests
        - dependency_changes
        - security_scan
        - performance_benchmark
        - rollback_plan
        - risk_assessment
      require:
        tests:
          status: pass
        diff:
          max_lines: 800
          max_files: 20
        explanation:
          maps_to_acceptance_criteria: true
        rollback:
          command_or_commit: required

    promotion:
      collect:
        - task_capsule
        - prototype_results
        - reviewer_objections
        - resolved_objections
        - final_patch
        - host_metrics
      require:
        no_unresolved_critical_objections: true
        acceptance_criteria_satisfied: all
        model_confidence_ignored_for_final_decision: true
```

Codex’s product direction is instructive: it provides verifiable evidence of actions using terminal logs and test outputs so users can trace what happened. ([OpenAI][19]) ZYAL should generalize that into a portable evidence contract.

## Gap 6: ZYAL needs anti-vibe-coding gates

The key difference between agentic engineering and vibe coding is not “AI versus no AI.” It is **specification, constraints, verification, review, and traceability**.

Recent research on vibe coding safety is worrying: one arXiv benchmark of real-world feature-request tasks reported that while 61% of one agent/model configuration’s outputs were functionally correct, only 10.5% were secure. ([arXiv][29]) Community tools like Jankurai are also emerging around real-time auditing of “vibe coding failure modes,” though those signals are newer and should be treated as ecosystem signals rather than definitive standards. ([Reddit][30])

Add:

```yaml
quality:
  anti_vibe:
    enabled: true
    require_spec_before_code: true
    require_acceptance_criteria: true
    require_tests_for_behavior_change: true
    require_traceability:
      diff_to_task: true
      diff_to_acceptance_criteria: true
      test_to_acceptance_criteria: true

    block_patterns:
      - name: giant-unreviewable-diff
        if_diff_lines_gt: 1200
        action: incubate_or_split
      - name: test-deletion
        if_deleted_tests_gt: 0
        action: require_human_approval
      - name: skipped-tests
        deny_patterns:
          - "\\.skip\\("
          - "it\\.only\\("
          - "describe\\.only\\("
          - "pytest.mark.skip"
        scope: file_diff
        action: block
      - name: fake-green
        deny_patterns:
          - "return true"
          - "TODO: mock this later"
          - "as any"
          - "eslint-disable"
        scope: file_diff
        action: warn_or_block_by_policy

    external_auditors:
      - name: jankurai
        command: "jankurai audit --format json"
        timeout: 30s
        assert:
          json:
            "$.score": { gte: 85 }
        on_fail: block_promotion

    understanding_checks:
      require_codebase_map_before_first_edit: true
      require_affected_symbols_known: true
      require_dependencies_known: true
```

This is where ZYAL can become the anti-vibe-coding standard: every change must be tied to spec, evidence, tests, and host-verified acceptance.

## Gap 7: ZYAL needs stronger security than regex guardrails

Regex guardrails are useful but insufficient. OWASP’s LLM Top 10 includes prompt injection, insecure output handling, supply chain vulnerabilities, insecure plugin design, excessive agency, and overreliance. ([OWASP][31]) OWASP’s “Excessive Agency” guidance specifically calls out excessive functionality, excessive permissions, excessive autonomy, minimizing extensions, avoiding open-ended tools where possible, minimizing permissions, requiring approval, and enforcing mediation in downstream systems. ([OWASP Gen AI Security Project][32])

# 5. Upgrade: workflow graphs, not only loops

`loop.policy: forever` is useful, but large projects need named phases, state machines, DAGs, and transitions.

```yaml
workflow:
  type: state_machine
  start_at: initialize

  states:
    initialize:
      do:
        - index_repo
        - build_project_map
        - create_feature_list
        - capture_baselines
      next: plan

    plan:
      agent: plan
      output_schema: task_graph
      next: select_task

    select_task:
      host_selects:
        from: task_graph
        strategy: highest_value_unblocked
      next: incubate_or_build

    incubate_or_build:
      switch:
        - if: "{{task.risk_score}} >= 0.7"
          next: incubator
        - if: "{{task.estimated_complexity}} >= 8"
          next: experiments
        - default: build

    build:
      agent: build
      max_iterations: 8
      next: verify

    verify:
      do:
        - run_evidence_bundle: checkpoint
      switch:
        - if: "{{evidence.checkpoint.pass}}"
          next: review
        - if: "{{task.attempts}} >= 3"
          next: incubator
        - default: build

    review:
      do:
        - run_review_gate: pre_merge
      switch:
        - if: "{{review.approved}}"
          next: merge_train
        - if: "{{review.needs_changes}}"
          next: build
        - default: park

    merge_train:
      do:
        - enqueue_merge
      next: select_task

    park:
      do:
        - summarize_and_archive
      next: select_task
```

This makes ZYAL more powerful than a forever loop while preserving host enforcement.

# 6. Upgrade: stateful disjoint experiments

The user specifically called out “disjointed experiments, and stateful building on current state.” ZYAL’s incubator gestures toward this, but it needs explicit experiment semantics.

```yaml
experiments:
  enabled: true
  strategy: disjoint_then_synthesize

  route_when:
    any:
      - estimated_complexity_gte: 8
      - repeated_attempts_gte: 2
      - uncertain_architecture: true
      - multiple_viable_designs: true

  baseline:
    source: current_checkpoint
    freeze_git_sha: true
    freeze_dependencies: true

  branches:
    count: 5
    isolation: git_worktree
    naming: "exp/{{task_id}}/{{branch_id}}"
    diversity:
      require_distinct_approaches: true
      dimensions:
        - architecture
        - dependency_choice
        - migration_strategy
        - testing_strategy
    agents:
      - id: conservative
        agent: build
        prompt_bias: "minimal patch, preserve architecture"
      - id: refactor
        agent: build
        prompt_bias: "cleaner architecture, still bounded"
      - id: test_first
        agent: build
        prompt_bias: "write failing tests first"
      - id: performance
        agent: build
        prompt_bias: "optimize hot path and benchmark"
      - id: security
        agent: build
        prompt_bias: "threat-model first"

  scoring:
    metrics:
      tests_pass: 0.25
      acceptance_criteria: 0.25
      diff_smallness: 0.10
      maintainability: 0.15
      security: 0.15
      performance: 0.10
    host_commands:
      - name: tests
        command: "bun test --timeout 60000"
      - name: typecheck
        command: "npx tsgo --noEmit"
      - name: security
        command: "npm audit --json"
      - name: diff_size
        command: "git diff --stat"

  synthesize:
    agent: plan
    input:
      - branch_scores
      - diffs
      - failure_summaries
      - reviewer_notes
    output_schema:
      type: object
      required: [winner, reusable_insights, rejected_approaches, final_plan]
    writes: scratch_only

  apply:
    strategy: best_evidence_patch
    require_human_approval_if:
      - winner_score_lt: 0.75
      - diff_lines_gt: 800
      - touches_security_sensitive_files: true

  memory:
    save:
      - reusable_insights
      - rejected_approaches
      - test_failures
      - architecture_lessons
```

This is the ZYAL-native version of Tree-of-Thoughts and Self-Refine for coding: explore multiple paths, score them with host evidence, synthesize, and save what was learned. Tree of Thoughts is about exploring multiple reasoning paths with self-evaluation and lookahead/backtracking; Self-Refine is about iterative feedback-and-refinement. ZYAL can turn those research patterns into concrete worktree-and-evidence workflows. ([arXiv][33])

# 7. Upgrade: a real skill lifecycle

Hermes is strong here, but ZYAL should be stricter. Skills are dangerous because they are executable procedural memory. They can improve agents, but they can also become a supply-chain and prompt-injection vector.

    project:
      path: ".opencode/skills"
      trust: project
    org:
      path: "oci://skills.company.com/opencode"

      registries:

    progressive: true
    max_index_tokens: 1000
    load_full_skill_only_when_selected: true

  schema:
    required_files:
      - SKILL.md
    required_sections:
      - When to Use
      - Procedure
      - Pitfalls
      - Verification
      - Permissions
      - Failure Modes
      - Rollback

  agent_created:
    allowed: true
    destination: quarantine
    trigger_when:
      - task_successful: true
      - tool_calls_gte: 5
      - user_corrected_agent: true
      - repeated_failure_resolved: true
    promotion:
      require:
        - security_scan_passed
        - human_review
        - replay_on_fixture_passed

# 14. Upgrade: interop layer so ZYAL works inside any chat prompt

# 19. Direct response to “overlap” concerns

## “Is ZYAL just LangGraph in YAML?”

No. LangGraph is a programmable workflow framework. ZYAL should borrow durable replay and idempotency semantics, but ZYAL’s unique value is **prompt-embeddable host-enforced policy for coding daemons**. It should be possible to compile ZYAL to LangGraph, but ZYAL should not require developers to write LangGraph code.

## “Is ZYAL just OpenHands?”

No. OpenHands is a coding agent platform/runtime with event streams and sandboxed execution. ZYAL can use OpenHands as a worker. ZYAL’s role is to define lifecycle, approvals, budgets, evidence, memory policy, merge rules, and safety gates across OpenHands and other agents.

## “Is ZYAL redundant with Hermes?”

No. Hermes is closer to a persistent personal/dev agent with memory, skills, gateway integrations, and self-improvement. ZYAL should learn heavily from Hermes memory and skill lifecycle, but ZYAL should be stricter: agent-created memory and skills need source events, scanning, quarantine, and promotion gates.

## “Is ZYAL redundant with OpenClaw?”

No. OpenClaw is a broad personal automation agent. ZYAL should not become a broad host-access personal assistant. ZYAL should be a safer, narrower, coding-focused control language with strong capability boundaries. OpenClaw’s security controversy is a warning: wide permissions plus long-term memory plus messaging inputs plus skills is powerful but high-risk. ([Cisco Blogs][12])

## “Is ZYAL redundant with Claude Code or Codex?”

No. Claude Code and Codex are agent products/runtimes. ZYAL should orchestrate them. A future ZYAL runner could dispatch a task to Codex, another to Claude Code, another to OpenHands, and still enforce the same evidence, memory, approval, cost, and merge policies.

## “Is ZYAL redundant with AutoGen, CrewAI, or Google ADK?”

No. Those are frameworks for building agents and multi-agent systems. ZYAL should be the declarative operations manifest that says how an autonomous coding run is allowed to proceed.

# 21. The north-star ZYAL design principle

The best possible ZYAL is not “a better prompt.” It is:

> **A declarative, host-enforced operating contract for autonomous engineering work.**

It should guarantee:

* The model cannot arm itself.
* The model cannot decide it is done.
* The model cannot silently escalate permissions.
* The model cannot write durable memory without review/scanning.
* The model cannot promote hard tasks on confidence alone.
* The model cannot hide evidence.
* Every side effect is logged.
* Every checkpoint is reproducible.
* Every merge is traceable.
* Every long-running session can be resumed, audited, or killed.
* Every agent runtime is replaceable.

That is how ZYAL becomes bigger than OpenCode. It becomes the portable control surface for all autonomous coding agents.
