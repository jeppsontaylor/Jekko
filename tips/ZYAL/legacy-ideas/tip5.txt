## Bottom line

ZYAL should **not** try to become another agent framework. That would make it redundant with OpenHands, LangGraph, Claude Code, Codex, AutoGen, CrewAI, Hermes, and OpenClaw. ZYAL’s winning position is stronger:

**ZYAL should become the portable, host-enforced control plane for autonomous coding agents.**

Think of it as the **Kubernetes-style declarative spec for long-running software-engineering agents**: any capable agent runtime can execute, but ZYAL owns policy, lifecycle, budgets, gates, memory permissions, experiment orchestration, evidence, and promotion. That is different from an agent SDK, a chat prompt, or a CLI wrapper.

The current ZYAL draft already has the right philosophical core: host-owned loop, preview before arming, SQLite truth, bounded execution, permissions, guardrails, constraints, checkpoint gating, and incubator passes. The biggest missing pieces are **durable replay semantics, rich task graphs, memory governance, adapter portability, experiment portfolios, evidence-first anti-vibe-coding gates, large-repo indexing, security threat modeling, and enterprise-scale observability**.

I surveyed primary docs, GitHub repos, papers, and selected community/social sources as of **May 8, 2026**. I would treat official docs, papers, and repos as authoritative; X/LinkedIn/Quora-style material is useful for sensing market pain, but it should not drive the spec alone. Quora access was blocked by robots.txt in this environment; LinkedIn/X surfaced current “vibe coding” and Jankurai-style quality-scoring concerns, but not enough detail to treat them as specifications.

# 1. ZYAL’s strongest positioning: not redundant, but above the runtimes

OpenHands is moving toward a production software-agent SDK: event-sourced state, deterministic replay, typed tools, MCP integration, security analysis, local/remote workspaces, REST/WebSocket interfaces, and human inspection/control. Its paper explicitly frames durable state, sandboxing, lifecycle control, multi-LLM routing, and security analysis as production foundations for software agents. ([arXiv][1])

LangGraph is a durable workflow runtime for long-running stateful agents, with persistence, human-in-the-loop, memory, streaming, debugging, and deployment as first-class concerns. ([LangChain Docs][2]) Its durable execution docs emphasize deterministic/idempotent workflow design and wrapping side effects so resuming a run does not repeat dangerous operations. ([LangChain Docs][3])

Hermes Agent is closer to “personal autonomous agent server”: persistent memory, auto-generated skills, scheduler, isolated subagents, MCP, terminal backends, and self-improvement loops. ([GitHub][4]) OpenClaw is closer to a local-first gateway/router for chat channels and coding agents, with a Gateway as session/routing source of truth and isolated sessions per agent/workspace/sender. ([GitHub][5]) Claude Code and Codex expose agentic coding engines with hooks, permissions, subagents, skills, cloud/web/CLI surfaces, MCP, and long-running or parallel tasks. ([Claude][6])

So ZYAL should not compete with these at the “agent loop implementation” layer. It should **compile to them, supervise them, constrain them, and normalize their evidence**.

The clean model is:

```text
Human intent
  ↓
ZYAL runbook: policy, lifecycle, budgets, evidence, safety, memory, promotion
  ↓
Host runtime: SQLite/event log, approvals, adapters, schedulers, checkpoints
  ↓
Agent backend: Jekko, Codex, Claude Code, OpenHands, Hermes, LangGraph, OpenClaw, etc.
  ↓
Tools/workspaces: shell, git, MCP, browser, CI, test runners, indexers, deploy systems
```

That makes ZYAL the thing that can be embedded “inside any chat prompt” while still ensuring the model cannot self-arm, self-promote, silently exceed budget, or claim success without host-verifiable evidence.

---

# 2. What ZYAL already gets very right

The draft is unusually strong in these areas:

**Host-owned lifecycle.** This is the core differentiator. Many agent systems let the model implicitly decide when it is “done.” ZYAL correctly makes stop conditions, promotion, checkpointing, budgets, and guardrails host responsibilities.

**Preview/arm separation.** The `ZYAL_ARM RUN_FOREVER` sentinel is valuable because it makes the user’s transition from “inspect this runbook” to “run this daemon” deliberate. Keep this. Make it even stricter with cryptographic preview hashes.

**SQLite as truth.** Durable state is mandatory for long-running autonomy. OpenHands’ redesign similarly centers a single source of truth and event-sourced state for recovery and replay. ([arXiv][1])

**Incubator passes.** The incubator is the most promising part of ZYAL. It shifts hard work from “one heroic prompt” into structured scouting, ideation, strengthening, critique, synthesis, prototype, review, and compression. That aligns with research showing gains from iterative refinement, test-based flows, deliberative search, and multi-agent debate. ([arXiv][7])

**Guardrails and constraints as runtime rules.** This is exactly the right move. Agentic systems create amplified risks from autonomy, privileges, interconnected tools, and unclear accountability; NSA/CISA-aligned guidance emphasizes governance, monitoring, human oversight, and explicit security design for agentic AI. ([National Security Agency][8])

**Mirror files as generated human inspection artifacts.** This is elegant. The source of truth stays SQLite, while `.jekko/daemon/` remains readable and diffable.

The core foundation is excellent. The upgrade path is to make ZYAL more like a **typed, replayable, multi-runtime, evidence-driven operating system for agent work**.

# 3. Major gap and overlap analysis

## 3.1 OpenHands overlap and gaps

OpenHands overlaps with ZYAL on sandboxed software-agent execution, lifecycle control, security, MCP, persistence, human inspection, and multi-LLM routing. The OpenHands SDK paper says V1 uses event-sourced state management, deterministic replay, immutable configuration, typed tools, MCP integration, local/remote workspace abstraction, REST/WebSocket services, a secret registry, security/confirmation, and continuous QA instrumentation. ([arXiv][1])

**What ZYAL is missing relative to OpenHands:**

1. **Event-sourcing semantics.** ZYAL says SQLite is truth, but not yet that every tool call, approval, retry, memory write, checkpoint, and model output is an immutable event with replay semantics.
2. **Deterministic replay rules.** ZYAL needs explicit side-effect journaling and idempotency keys.
3. **Typed tools and adapter capability negotiation.** ZYAL has broad permission knobs, but not a typed capability model.
4. **Secret registry and credential scopes.** ZYAL can block secret leakage, but it does not yet specify how secrets are mounted, redacted, rotated, or made invisible to model context.
5. **QA harness for the daemon runtime itself.** OpenHands has production reliability/evaluation framing; ZYAL needs `zyal test`, `zyal replay`, `zyal fuzz`, and schema conformance tests.
6. **REST/WebSocket runtime parity.** ZYAL has an API surface, but it should expose live traces, approvals, artifacts, replay, drain, capability diffs, and event subscriptions.

**Recommendation:** ZYAL should adopt OpenHands-like event sourcing as a hard invariant, while remaining a declarative policy language above OpenHands.

---

## 3.2 LangGraph / LangChain overlap and gaps

LangGraph overlaps with ZYAL on durable execution, long-running stateful workflows, persistence, human-in-the-loop, memory, interrupts, and replay. ([LangChain Docs][2]) Its durable execution docs are especially important: workflows must be deterministic and idempotent, and side-effect/non-deterministic operations should be wrapped so they are not repeated after resume. ([LangChain Docs][3]) LangChain’s HITL middleware pauses risky tool calls and supports approve/edit/reject/respond decisions. ([LangChain Docs][9])

**What ZYAL is missing relative to LangGraph:**

1. **Graphs, not just loops.** `loop` plus `on` handlers are useful, but ambitious coding projects require DAGs, routers, subgraphs, joins, interrupts, drains, and resumable checkpoints.
2. **Human decision types.** ZYAL has `ask`, `pause`, and manual promotion, but not typed approval decisions such as approve, edit, reject, respond, delegate, narrow scope, require tests, or require human pairing.
3. **Drain semantics.** A host should gracefully drain a daemon between supersteps, not abruptly interrupt a worker mid-write.
4. **Replay/idempotency controls.** ZYAL needs explicit `idempotency_key`, `side_effect: journaled`, `resume_from`, and `do_not_repeat` semantics.
5. **Hot-path versus background memory writes.** LangGraph distinguishes memory creation during runtime from background memory formation; ZYAL needs this distinction to prevent memory work from slowing or polluting the main coding loop. ([LangChain Docs][10])

**Recommendation:** ZYAL should add a `workflow` or `graph` block while preserving `loop` as a shorthand.

---

## 3.3 Hermes Agent overlap and gaps

Hermes overlaps with ZYAL on long-running autonomy, memory, skills, isolated subagents, schedulers, MCP, terminal backends, and self-improvement. Hermes documents persistent memory, auto-generated skills, FTS5 search, user modeling, scheduler, sandboxed terminal backends, and isolated subagents. ([GitHub][4]) Hermes also has an interesting token-efficiency pattern: the agent can write Python scripts that call Hermes tools through RPC, so intermediate tool results do not all enter model context. ([GitHub][11])

**What ZYAL is missing relative to Hermes:**

1. **Procedural memory / skills lifecycle.** ZYAL has context preservation and incubator scratch, but not a validated “skill creation → test → promote → version → retire” lifecycle.
2. **Cross-session recall.** ZYAL preserves run/task state, but not organization-wide reusable lessons, build commands, known flaky tests, migration patterns, or repo-specific repair recipes.
3. **Scheduler/triggers.** ZYAL runs daemon blocks, but lacks cron, webhook, channel, issue, PR, CI, incident, and scheduled recurring triggers.
4. **Subagent scripts as token-saving tool programs.** ZYAL can run shell, but it should explicitly support bounded generated scripts whose intermediate observations stay outside model context unless summarized.
5. **Self-improvement with safety gates.** Hermes leans into agent growth; ZYAL should allow growth but make every new memory/skill host-reviewed, provenance-tagged, and regression-tested.

**Recommendation:** Add `memory`, `skills`, `schedule`, and `tool_programs` blocks, with strict evidence gates.

---

## 3.4 OpenClaw / OpenClaw coding-agent overlap and gaps

OpenClaw overlaps with ZYAL on background coding agents, channel routing, local-first gateway behavior, session management, process monitoring, and multi-agent routing. OpenClaw’s docs describe a Gateway as the source of truth for sessions, routing, and channel connections, with isolated sessions per agent/workspace/sender. ([GitHub][5]) Its ACP docs describe durable binding of conversations to ACP sessions, and its coding-agent skill delegates to Codex, Claude Code, Jekko, or Pi in background processes while requiring explicit notification routes and process monitoring. ([OpenClaw][12])

**What ZYAL is missing relative to OpenClaw:**

1. **Conversation/channel binding.** If ZYAL is available inside any chat prompt, it needs a durable route back to the initiating surface: Slack thread, Telegram topic, iMessage sender, web session, terminal session, GitHub issue, etc.
2. **Notification contracts.** Background agents need a typed completion/failure/progress message route, not informal prompts.
3. **External harness sessions.** ZYAL should support ACP-style sessions and provider-specific session IDs as first-class runtime resources.
4. **Per-sender/workspace isolation.** OpenClaw’s isolated session model is directly relevant to ZYAL daemons launched from chat.
5. **Current-conversation binding store.** “Resume this daemon” should know exactly which runtime session and channel thread it belongs to.

**Recommendation:** Add `channels`, `bindings`, and `notifications` as host-level blocks.

---

## 3.5 Claude Code and Codex overlap and gaps

Claude Code already exposes hooks, permissions, subagents, skills, MCP, memory, schedules, remote/browser/desktop/IDE surfaces, and cloud sessions. ([Claude][6]) Claude’s permission system evaluates hooks, deny rules, permission modes, allow rules, then runtime callbacks, which is a useful model for ZYAL’s own permission precedence. ([Claude][13]) Claude hooks can block dangerous operations, audit tool calls, transform inputs/outputs, require human approval, and track lifecycle. ([Claude][14]) Codex CLI similarly exposes local terminal execution, code editing, shell execution, MCP, subagents, web search, approval modes, cloud tasks, and scripting. ([OpenAI Developers][15])

**What ZYAL is missing relative to Claude/Codex:**

1. **Provider-specific adapter semantics.** For example, OpenClaw’s coding-agent skill notes different PTY/print-mode conventions for Codex/Jekko/Pi versus Claude Code. ZYAL should model those differences declaratively instead of hiding them in prompts. ([GitHub][16])
2. **Subagent descriptors.** Claude subagents have descriptions, tool restrictions, permission modes, hooks, and skills. ZYAL workers are currently less expressive. ([Claude][17])
3. **Hook precedence.** ZYAL `on` handlers and guardrails should have a deterministic precedence order like Claude’s permission pipeline.
4. **Provider capability discovery.** ZYAL should fail preview if the target runtime cannot support required semantics.

**Recommendation:** Add `runtime.adapters` plus `capability_negotiation`.

---

## 3.6 AutoGen, CrewAI, and MetaGPT overlap and gaps

AutoGen emphasizes customizable, conversable multi-agent collaboration, human/tool/code integration, and conversation patterns. Microsoft’s AutoGen project describes a v0.4 redesign with async event-driven architecture, observability/debugging, scalable/distributed design, extensions, cross-language support, and type support. ([Microsoft][18]) CrewAI emphasizes crews, flows, guardrails, memory, knowledge, observability, stateful event-driven workflows, persistence/resume, and sequential/hierarchical/hybrid task processes. ([CrewAI Documentation][19]) MetaGPT’s key lesson is standardized operating procedures: it encodes SOPs into prompt sequences so role-specialized agents verify intermediate outputs and reduce cascading errors. ([OpenReview][20])

**What ZYAL is missing relative to these:**

1. **Explicit SOP artifacts.** ZYAL has passes, but should define required artifacts per role: requirements, architecture, risk model, tests, diff plan, rollback plan, review notes.
2. **Typed role contracts.** `agent: plan` and `agent: build` are too thin for large projects.
3. **Conversation topology.** Need debate, vote, critic, chair, red-team, architect, implementation, reviewer, and release-manager patterns.
4. **Process visualization.** ZYAL’s mirror should include graph visualizations and live phase state.
5. **Agent-level memory scopes.** CrewAI-style agent/crew/flow memory and Hermes-style skills imply ZYAL should model memory at run, task, repo, org, and role scopes.

**Recommendation:** Add `sop`, `roles`, `artifacts`, and richer `workflow` semantics.

# 4. The largest missing capability: evidence-first engineering

The most important upgrade is **evidence as a first-class object**.

Today ZYAL checks stop conditions and checkpoint verification, but the spec should make evidence explicit, typed, durable, queryable, and promotable.

For serious coding projects, a daemon should never be allowed to say:

> “I fixed it.”

It should have to produce:

```yaml
evidence:
  tests:
    - command: "bun test --timeout 30000"
      exit_code: 0
      log_ref: "event://..."
  static_analysis:
    - command: "npx tsgo --noEmit"
      exit_code: 0
  changed_files:
    - path: "src/auth/session.ts"
      reason: "Fixes token refresh race"
      linked_requirement: "REQ-auth-refresh-003"
  risks:
    - "Refresh cache invalidation may affect multi-tab sessions"
  rollback:
    command: "git revert <commit>"
  human_review:
    status: pending
```

This is how ZYAL beats vibe coding. “Vibe coding” is commonly used for AI-assisted coding where users may accept AI-generated code without thorough review; newer quality tools such as Jankurai are already positioning around real-time audits for “vibe coding failure modes,” although the public source I found was a LinkedIn announcement rather than a formal spec. ([Wikipedia][21])

The anti-vibe-coding rule should be:

**No production change may advance unless ZYAL can point to host-verified evidence.**

That means tests, static analysis, affected-file reasoning, blast-radius analysis, rollback plan, migration safety, performance checks, security checks, and human approval when required.

# 6. Upgrade: runtime adapters and capability negotiation

ZYAL must be able to run inside Jekko, Codex, Claude, OpenHands, Hermes, OpenClaw, or future agents without pretending they are identical.

Add:

```yaml
runtime:
  target: portable

  adapters:
    - id: jekko-local
      kind: jekko
      command: "jekko run"
      workspace: local
      supports:
        - shell
        - edit
        - git
        - mcp
        - background_process
        - structured_output

    - id: claude-code
      kind: claude_code
      command: "claude --print"
      mode:
        pty: false
      supports:
        - edit
        - shell
        - hooks
        - permissions
        - subagents
        - skills
        - mcp

    - id: codex
      kind: codex_cli
      command: "codex exec"
      mode:
        pty: true
      supports:
        - edit
        - shell
        - sandbox
        - approvals
        - mcp
        - subagents

    - id: openhands
      kind: openhands_sdk
      endpoint: "http://localhost:3000"
      supports:
        - event_sourcing
        - deterministic_replay
        - sandbox
        - rest
        - websocket
        - security_analyzer
        - multi_llm_routing

  capability_negotiation:
    require:
      - durable_state
      - path_scoped_edit
      - shell_timeout
      - structured_output
      - git_worktree
      - approval_interrupts
    prefer:
      - event_sourcing
      - deterministic_replay
      - security_analyzer
      - background_process
      - subagents
    on_missing_required: reject_preview
    on_missing_preferred: warn_preview
```

This lets ZYAL answer: “Can this runtime actually enforce the runbook?” If not, preview should fail.

# 7. Upgrade: durable replay, idempotency, and side-effect journaling

ZYAL says SQLite is truth. It should go further:

```yaml
durability:
  state: sqlite
  event_log:
    mode: append_only
    hash_chain: true
    include:
      - user_messages
      - assistant_messages
      - tool_calls
      - tool_results
      - approvals
      - guardrail_decisions
      - memory_writes
      - checkpoints
      - promotions
      - retries
      - errors
      - cost_usage

  replay:
    deterministic: true
    replay_from_event: allow
    side_effects: journaled
    nondeterminism:
      random: require_seed
      time: host_timestamp_only
      network: cache_or_mark_unreplayable
    on_replay_mismatch: pause

  idempotency:
    require_keys_for:
      - git_commit
      - git_push
      - issue_comment
      - pr_create
      - deploy
      - external_api_write
    duplicate_policy: return_recorded_result

  snapshots:
    every_events: 100
    every_duration: 10m
    verify_hash_chain: true

  drain:
    mode: graceful
    boundary: after_tool_or_checkpoint
    max_grace: 60s
    on_timeout: pause
```

This borrows the most important lesson from LangGraph and OpenHands: long-running agents must resume safely, and side effects must not repeat accidentally. ([LangChain Docs][3])

# 8. Upgrade: graph workflows, not only loops

The current `loop` is useful for “fix tests until green.” It is not enough for ambitious projects.

Add:

```yaml
workflow:
  type: graph
  start: discover

  nodes:
    - id: discover
      kind: task_discovery
      agent: plan
      outputs: [task_graph, risk_register]
      next: classify

    - id: classify
      kind: router
      route:
        simple_fix: implement_direct
        hard_task: incubate
        risky_change: require_human_scope
        unclear: ask_human

    - id: implement_direct
      kind: agent_task
      agent: build
      writes: repo
      next: verify

    - id: incubate
      kind: incubator
      ref: incubator
      next: promote_or_park

    - id: require_human_scope
      kind: interrupt
      decision_types: [approve, edit_scope, reject, delegate]
      next:
        approve: incubate
        edit_scope: classify
        reject: park

    - id: verify
      kind: verification
      ref: quality
      next:
        pass: checkpoint
        fail: diagnose

    - id: diagnose
      kind: agent_task
      agent: critic
      writes: scratch_only
      next: classify

    - id: checkpoint
      kind: checkpoint
      next: stop_check

    - id: stop_check
      kind: stop_evaluation
      next:
        done: end
        not_done: discover

  concurrency:
    max_active_nodes: 8
    max_repo_writers: 1
    max_scratch_workers: 12

  drain:
    allowed_at:
      - after_node
      - before_checkpoint
      - before_promotion
```

The language should keep `loop` as syntactic sugar:

```yaml
loop:
  policy: forever
```

could compile into a tiny workflow graph:

```text
iteration → verify/checkpoint → stop_check → iteration
```

# 9. Upgrade: memory that does not become prompt sludge

Memory is where long-running agents either become powerful or become haunted by stale, poisoned, low-quality notes.

The right ZYAL model is **host-mediated memory with typed stores, provenance, trust, retention, and validation**.

Research and production systems point in the same direction: Reflexion stores verbal reflections in episodic memory to improve future trials; Voyager compounds capabilities through an executable skill library and self-verification; LangGraph distinguishes hot-path versus background memory writing; Hermes emphasizes persistent memory and auto-generated skills. ([arXiv][22])

Add:

```yaml
memory:
  policy: host_mediated
  source_of_truth: sqlite

  stores:
    working:
      scope: run
      max_tokens: 12000
      retention: until_run_end
      write: host_only

    episodic:
      scope: task
      description: "Attempts, failures, observations, fixes tried"
      retention: 90d
      write:
        mode: background
        trigger: after_iteration
        require_schema: episodic_memory_v1

    semantic:
      scope: repo
      description: "Stable facts about architecture, APIs, conventions"
      retention: 1y
      write:
        mode: propose_then_review
        require_evidence: true
        min_confidence: 0.8

    procedural:
      scope: repo
      description: "Reusable workflows and skills"
      retention: until_retired
      write:
        mode: skill_promotion
        require_validation: true

    architectural:
      scope: org
      description: "ADRs, invariants, code ownership, platform constraints"
      write:
        mode: human_only

    negative:
      scope: repo
      description: "Known bad approaches and failed hypotheses"
      retention: 180d
      write:
        mode: background
        require_failure_evidence: true

  retrieval:
    strategy: layered
    include:
      - working
      - semantic
      - procedural
      - negative
    max_tokens_per_store:
      working: 12000
      semantic: 6000
      procedural: 4000
      negative: 3000
    rank_by:
      - relevance
      - recency
      - trust
      - evidence_strength
    require_citations_to_events: true

  poisoning_defense:
    reject_if_source_untrusted: true
    scan_tool_outputs: true
    scan_mcp_outputs: true
    require_schema_for_external_tool_memory: true
    quarantine_on_prompt_injection_markers: true
    human_review_for_org_memory: true

  compaction:
    strategy: hierarchical
    levels:
      - iteration_summary
      - task_capsule
      - project_digest
      - org_lesson
    never_store:
      - hidden_chain_of_thought
      - secrets
      - raw_credentials
      - untrusted_tool_instructions
```

The line “never store hidden chain-of-thought” is important. ZYAL memory should store **findings, evidence, summaries, decisions, diffs, hypotheses, failures, and instructions**, not private reasoning traces.

# 10. Upgrade: skills as validated artifacts

Hermes and Claude both make skills/procedural memory central. ZYAL should support this, but with stronger promotion gates.

```yaml
skills:
  enabled: true
  store: sqlite
  mirror: ".jekko/daemon/skills"

  propose_when:
    any:
      - task_repeated_gte: 3
      - same_fix_pattern_gte: 2
      - human_marks_reusable: true
      - successful_run_contains_new_workflow: true

  proposal:
    agent: compress
    output_schema:
      type: object
      required: [name, purpose, trigger, steps, required_tools, validation]
    writes: scratch_only

  validation:
    sandbox: isolated_worktree
    tests:
      - command: "zyal skill test {{skill.name}} --fixture fixtures/{{skill.name}}"
        timeout: 120s
        assert: { exit_code: 0 }
    require:
      - no_secret_leakage
      - no_unscoped_shell
      - deterministic_output
      - human_readable_steps

  promotion:
    require_human: true
    versioning: semver
    on_promote: add_to_repo_memory
    on_fail: archive_with_reason

  retirement:
    retire_when:
      any:
        - validation_failures_gte: 3
        - unused_for: 180d
        - superseded_by: exists
```

This gives ZYAL the benefits of Voyager/Hermes-style skill accumulation while preventing uncontrolled “the agent taught itself a bad habit” drift. ([Voyager][23])

# 11. Upgrade: experiment portfolios for smarter outcomes

The current `incubator` is strong, but it needs a more explicit **experiment portfolio** layer.

Research supports this direction. Tree of Thoughts explores multiple reasoning paths with self-evaluation and backtracking. ([arXiv][24]) Self-Refine improves outputs through iterative feedback and refinement. ([arXiv][7]) Multi-agent debate can improve reasoning and factuality. ([arXiv][25]) AlphaCodium shows that test-based multi-stage code flows can substantially improve code-generation performance on coding tasks. ([arXiv][26])

ZYAL should explicitly distinguish:

**Disjoint experiments:** multiple workers start from the same base and try different approaches.

**Stateful building:** once a candidate wins, later passes build on that exact accepted state.

**Hybrid:** disjoint exploration first, stateful consolidation second.

```yaml
experiments:
  enabled: true
  mode: hybrid_portfolio

  base:
    ref: "HEAD"
    snapshot: true
    restore_before_each_candidate: true

  portfolio:
    max_candidates: 6
    max_parallel: 3

      require_distinct_hypotheses: true

        - algorithm
        - file_scope
        - dependency_strategy
        - migration_strategy
      reject_near_duplicate_plans: true

  candidates:
    - id: minimal_patch
      hypothesis: "Smallest targeted fix with no architecture change"
      agent: build
      budget: { wall_clock: 20m, max_tokens: 60000 }

    - id: refactor_boundary
      hypothesis: "Move boundary logic into dedicated module"
      agent: build
      budget: { wall_clock: 40m, max_tokens: 100000 }

      hypothesis: "Write regression test first, then implement"
      agent: build

        command: "bun test --timeout 30000"
        weight: 0.30
        assert: { exit_code: 0 }

      - id: typecheck
        command: "npx tsgo --noEmit"
        weight: 0.20
        assert: { exit_code: 0 }

      - id: diff_size
        command: "git diff --shortstat"
        weight: 0.10
        prefer: smaller

      - id: maintainability
        agent: critic
        weight: 0.15
        require_structured_output: true

      - id: risk
        agent: security_critic
        weight: 0.15
        prefer: lower

        weight: 0.10

  compare:
    strategy: weighted_score
    require:
      - tests
      - typecheck
      - rollback_known
    break_ties_by:
      - smaller_diff
      - fewer_files
      - simpler_rollback

  merge:
    strategy: best_candidate_then_synthesize
    allow_cherry_pick: true
    require_final_verification: true
```

This is the antidote to “one prompt, one patch, hope.” It turns agent work into a reproducible experimental process.

# 13. Upgrade: security model for autonomous coding

Current guardrails are useful but too shallow for high-autonomy agents.

Agentic AI security guidance highlights privilege risks, design/config risks, behavior risks, structural risks, and accountability risks. ([National Security Agency][8]) MCP tool poisoning is a concrete example: malicious tool outputs can inject instructions into an agent context and cause privileged actions unless responses are validated, privileged tools are isolated, and approvals happen outside the LLM context. ([OWASP][27])

Add:

```yaml
security:
  trust_zones:
    - id: repo
      trust: internal
      paths: ["./"]
    - id: scratch
      trust: low
      paths: [".jekko/daemon/scratch"]
    - id: external_mcp
      trust: untrusted
    - id: secrets
      trust: restricted

  sandbox:
    default: isolated
    filesystem:
      allow_read:
        - "./"
      allow_write:
        - "./src/**"
        - "./tests/**"
        - ".jekko/daemon/**"
      deny_write:
        - ".env"
        - "**/*.pem"
        - "**/secrets/**"
        - "infra/prod/**"
    network:
      default: deny
      allow:
        - host: "registry.npmjs.org"
          reason: "dependency metadata"
      require_approval_for:
        - external_http
        - web_search
        - package_install
    process:
      max_children: 32
      max_runtime: 30m
      kill_on_abort: true

  secrets:
    registry: host
    expose_to_model: false
    mount:
      - id: github_token
        to: env
        env: GITHUB_TOKEN
        scope:
          commands:
            - "gh pr create"
            - "gh pr view"
    redaction:
      scan_tool_input: true
      scan_tool_output: true
      scan_file_diff: true
      scan_commit_message: true

  mcp:
    allowlist:
      - server: "filesystem"
        trust: internal
      - server: "github"
        trust: approved
    deny_unlisted_servers: true
    validate_tool_output_schema: true
    strip_instructions_from_tool_output: true
    isolate_untrusted_mcp_context: true
    require_human_for_cross_zone_actions: true

  approvals:
    out_of_band: true
    decisions:
      - approve
      - reject
      - edit
      - narrow_scope
      - require_more_evidence
    require_for:
      - git_push
      - package_install
      - database_migration
      - network_egress_unlisted
      - secret_access
      - production_deploy
      - deleting_files_over: 5

  provenance:
    sign_events: true
    hash_diffs: true
    include_tool_versions: true
    include_model_versions: true
```

This should be host-enforced. The model may propose a dangerous operation, but the runtime must block it.

# 26. Schema-level details ZYAL should define

To become a world-default language, ZYAL needs exact semantics, not just YAML examples.

## 26.1 Permission precedence

Borrow the clarity of Claude’s permission evaluation order. ([Claude][13]) ZYAL should define:

```text
1. Host emergency deny
2. Security sandbox deny
3. Guardrail deny
4. Path/capability deny
5. Active human approval requirement
6. Runtime adapter permission check
7. Allow rule
8. Tool execution
9. Output guardrail
10. Event log append
```

No ambiguity. No prompt-based enforcement.

## 26.2 Handler precedence

Existing `on` handlers “do not cascade.” Keep that, but define exact ordering:

```text
signal emitted
  → event logged
  → matching handlers selected
  → handlers sorted by priority
  → max one handler group executes
  → handler actions logged
  → no handler-emitted signal triggers another handler
```

Add:

```yaml
on:
  - signal: checkpoint_failed
    priority: 100
    do:
      - incubate_current_task: true
```

# 27. What to remove or avoid

ZYAL should avoid several traps:

**Do not put complex imperative logic in YAML.** Use declarative policies and typed host actions. Arbitrary code belongs in controlled tools or `tool_programs`.

**Do not make model self-confidence central.** Keep `model_confidence_cap`; better yet, make confidence informational only unless backed by evidence.

**Do not let memory become truth.** SQLite events and evidence are truth. Memory is derived, scoped, and revocable.

**Do not let “forever” mean unbounded.** A forever daemon should still have wall-clock budgets, cost budgets, token budgets, retry limits, circuit breakers, leases, and drain points.

**Do not depend on prompt-only safety.** MCP poisoning and agentic security risks make this non-negotiable. ([OWASP][27])

**Do not bake in one backend.** Jekko can be the native home, but ZYAL’s strategic value is portability.

# 29. The essential ZYAL design doctrine

I would rewrite the ZYAL vision as:

> ZYAL is the host-enforced control language for autonomous software engineering. It does not ask models to be safe, honest, bounded, or complete. It makes those properties structural. Models propose actions; the host validates capabilities, budgets, evidence, memory writes, approvals, and promotion. Every durable claim is backed by an event. Every production change is backed by evidence. Every long-running loop is drainable, replayable, observable, and bounded.

That is the path to making ZYAL the default language for the world’s most ambitious coding projects.

The highest-leverage additions are:

1. **Event-sourced replay with idempotency.**
2. **Evidence-first promotion.**
3. **Graph workflows.**
4. **Memory and skill governance.**
5. **Experiment portfolios.**
6. **Runtime adapters and capability negotiation.**
7. **Security trust zones.**
8. **Large-repo indexing and context lenses.**
9. **Channel/session bindings.**
10. **Observability and auditability.**

With those, ZYAL becomes something the existing systems mostly do not provide: a portable, declarative, host-enforced language for turning agentic coding from “vibes” into governed, experimental, evidence-backed engineering.
