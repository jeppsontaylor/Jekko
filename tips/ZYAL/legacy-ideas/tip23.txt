## Scope note

I treated “study all” as a deep public-source review of primary docs, repos, papers, and representative public commentary. I did **not** assume private X/LinkedIn/Quora material is complete or reliable; where primary docs, papers, and repos existed, I weighted those much more heavily. The conclusion is clear: **ZYAL is not redundant if it remains a host-enforced control language.** It becomes redundant only if it tries to be LangGraph, OpenHands, Claude Code, Jekko, Hermes, or OpenClaw. Its strongest future is as a **portable daemon-policy and evidence language** that can compile into those systems.

## Executive verdict

ZYAL already has the right philosophical core: **the model does not own the loop, promotion, stop condition, or safety boundary.** That is the critical distinction. LangGraph gives durable stateful orchestration; OpenHands gives a sandboxed software-agent runtime; Claude Code/Jekko/Codex expose permissions, hooks, skills, and coding-agent ergonomics; Hermes emphasizes self-improving memory and skills; OpenClaw emphasizes a personal always-on gateway with channels and local control. ZYAL should sit above these as a **declarative, auditable run contract** that can be pasted into any chat, previewed, negotiated against host capabilities, armed by a human, and then executed by the host. LangGraph describes itself as a low-level orchestration framework for long-running stateful agents with durable execution, human-in-the-loop controls, memory, and debugging; OpenHands exposes coding agents through Python/REST APIs and local or ephemeral Docker/Kubernetes workspaces; Hermes is explicitly a self-improving agent with memory, skills, and multi-platform operation; OpenClaw is a persistent personal assistant with chat channels, browser/system control, and sandboxing. ZYAL should not duplicate those systems; it should govern them. ([GitHub][1])

The biggest missing idea is **evidence-centered autonomy**. ZYAL has stop conditions, guardrails, checkpoints, and incubator readiness scores, but it does not yet define a complete proof system: traceable requirements, proof lanes, artifact receipts, merge witnesses, ratchets, ownership maps, risk tiers, experiment comparison, rollback evidence, and deployment gates. This matters because modern long-running coding-agent benchmarks are moving away from small isolated bug fixes toward multi-file, long-horizon software evolution tasks; SWE-Bench Pro describes 1,865 human-verified problems across 41 active repositories requiring substantial multi-file changes over hours or days, and SWE-EVO reports long-horizon evolution tasks averaging 21 files and 874 tests where strong agents perform far worse than on SWE-Bench Verified. ([arXiv][2])

The second missing idea is **memory governance**. ZYAL says SQLite is truth, which is excellent, but it does not yet specify what should become memory, what must never become memory, how memory is proven, decayed, contradicted, redacted, retrieved, packed, or promoted. LangChain’s memory docs warn that long conversations overflow context and stale/off-topic content distracts models; they also emphasize persisted state through checkpointers and custom state schemas. Hermes goes further with a closed learning loop, FTS5 recall, user modeling, and automatic skill creation/improvement. ZYAL needs the safety of host-managed state plus the learning capacity of systems like Hermes. ([LangChain Docs][3])

The third missing idea is **portable capability negotiation**. ZYAL says “available inside any chat prompt,” but Codex, Claude Code, Jekko, OpenHands, LangGraph, Hermes, and OpenClaw have different runtimes, permission surfaces, network defaults, memory behavior, skill systems, hooks, and sandbox semantics. Claude Code documents fine-grained allow/ask/deny permissions, hooks that deterministically execute shell commands at lifecycle points, subagents with isolated context, and memory through CLAUDE.md plus auto memory; Jekko similarly exposes permissions, agents, subagents, and agent skills. ZYAL needs a compile target layer: “Here is what this host can actually enforce; here is what will be degraded; here is what must fail closed.” ([Claude][4])

# 1. What ZYAL already gets right

## 1.1 Host ownership is the right primitive

The most important ZYAL principle is that the host, not the model, owns the loop. This is the right line. OpenAI’s Agents SDK guidance frames an agent as planning, tool-calling, collaboration, and state management, but recommends SDK-style ownership when the application needs direct control over orchestration, tools, approvals, state, and storage. ZYAL should be the declarative surface for exactly that host ownership. ([OpenAI Developers][5])

## 1.2 Preview-before-arming is essential

The `ZYAL_ARM RUN_FOREVER` sentinel is a strong idea. For safety, I would make arming even stricter: the host should require an out-of-band user gesture or signed UI action, not merely text that a model can emit. Claude Code’s auto mode exists because users approve many prompts, but its docs also explicitly describe dangerous examples such as deleting remote branches, uploading tokens, or running production migrations; ZYAL should avoid approval fatigue while never letting the assistant self-arm. ([Anthropic][6])

## 1.3 SQLite as truth is the right default

Durable state is not optional for long-running agents. LangGraph emphasizes durable execution for long-running workflows, and LangChain checkpointers persist thread state to databases such as SQLite or Postgres. ZYAL’s “SQLite is truth” principle is strong; it should be expanded into an event-sourced ledger plus normalized tables plus content-addressed artifacts. ([GitHub][1])

## 1.4 Incubator passes are the most distinctive part

The incubator is ZYAL’s best differentiator. It begins to encode what strong human engineering teams do: scout, generate ideas, strengthen, criticize, synthesize, prototype, review, compress. This is aligned with research patterns such as Reflexion-style reflective memory, Tree-of-Thought-style branching and self-evaluation, and lifelong skill acquisition as in Voyager. The upgrade is to make the incubator less scalar and more evidence-driven: readiness should be an evidence graph, not just a score. ([arXiv][7])

# 2. Where ZYAL currently overlaps or risks becoming redundant

## 2.1 LangGraph / LangChain overlap

ZYAL’s `loop`, `on`, `fan_out`, `checkpoint`, `retry`, and `context` sections overlap with LangGraph’s durable agent orchestration. LangGraph already provides long-running stateful execution, human-in-the-loop controls, memory, and deployment/debugging support. LangChain also has checkpointers and custom state schemas. ZYAL should not try to be a general graph runtime. It should define **host policy, safety, evidence, and run intent**, then compile to LangGraph when graph execution is needed. ([GitHub][1])

## 2.2 OpenHands overlap

OpenHands already provides a software-agent runtime where agents interact like human developers through code editing, command line, and browsing in sandboxed environments, with multi-agent coordination and benchmark evaluation. Its SDK supports one-off fixes, routine maintenance, major multi-agent refactors, and local or ephemeral Docker/Kubernetes workspaces. ZYAL should not reimplement that runtime. ZYAL should declare **what OpenHands may do, how long it may do it, what evidence is required, and when promotion is blocked**. ([arXiv][8])

## 2.3 Claude Code / Jekko / Codex overlap

Claude Code and Jekko already expose important mechanics: permissions, hooks, subagents, memories, skills, and specialized agents. Codex CLI emphasizes sandboxing, approvals, and restricted network access. ZYAL overlaps at the surface level, but ZYAL can become stronger by making these features portable and evidence-bound. For example, a Claude Code hook can enforce “run formatter after edit,” while ZYAL can require “formatting proof lane must pass before checkpoint promotion and the receipt must be archived.” ([Claude][4])

## 2.4 Hermes overlap

Hermes is closer to an always-on self-improving assistant. Its docs emphasize a closed learning loop, memory, skill creation/improvement, FTS5 recall, user modeling, six terminal backends, scheduled automations, and multi-platform channels. ZYAL does not yet have those learning and channel capabilities. But Hermes does not appear to be a strict, portable daemon-control schema for coding-project evidence gates. ZYAL should borrow Hermes’s learning loop, but host-govern it: skills and memories should only be promoted after evidence and review. ([Hermes Agent][9])

## 2.5 OpenClaw overlap

OpenClaw focuses on persistent personal-assistant operation across chat channels, local gateway control, browser control, full system access or sandboxing, and pairing/allowlist security. Its security docs explicitly say it is not a hostile multi-tenant boundary and recommend separating boundaries for adversarial users. ZYAL needs an explicit **trust-boundary model** if it is going to live inside arbitrary chat prompts. A daemon run triggered from a local trusted Jekko session is not equivalent to a message from WhatsApp, GitHub comments, Slack, or an untrusted issue body. ([OpenClaw][10])

## 2.6 Jankurai / anti-vibe overlap

Jankurai is highly relevant because it attacks “vibe coding” by requiring auditable merge infrastructure: ownership maps, proof lanes, generated zones, security boundaries, rolling scores, merge witnesses, repair queues, receipts, and ratchets. ZYAL currently has some of this spirit, but not the concrete merge-proof machinery. ZYAL should incorporate Jankurai-like proof lanes and witnesses as first-class promotion gates. ([GitHub][11])

# 3. The largest missing pieces in ZYAL

## 3.1 Capability negotiation across hosts

ZYAL cannot be “available inside any chat prompt” unless it can say which parts are enforceable on the current host. Different tools have different permissions, network controls, memory behaviors, skill systems, and runtimes. MCP is an open protocol for connecting AI apps to tools and data, while AGENTS.md and Skills are emerging as portable instruction/skill surfaces across coding agents; ZYAL needs to bind to those without assuming they enforce the same guarantees. ([Model Context Protocol][12])

Add:

```yaml
runtime:
  target: portable
  capability_negotiation: fail_closed   # fail_closed | degrade_with_notice | best_effort
  adapters:
    jekko:
      agents: true
      skills: true
      permissions: true
      hooks: true
    claude_code:
      subagents: true
      hooks: true
      memory_files: true
      permission_modes: true
    codex:
      sandbox: workspace_write
      network: deny_by_default
      approvals: true
    openhands:
      runtime: docker
      browser: allow
      multi_agent: allow
    langgraph:
      compile_workflow: true
      checkpointer: sqlite
    hermes:
      skills: allow_reviewed
      scheduled_automations: deny
    openclaw:
      channels: deny_by_default
      gateway: local_only

  unknown_feature_policy: reject
  generated_preview:
    include_compiled_plan: true
    include_degraded_features: true
    include_required_human_approvals: true
```

Why this matters: a chat surface can display an ZYAL block, but only the host can say, “I can enforce `git_push: deny`; I cannot enforce browser isolation; this run is rejected.” Without this, ZYAL becomes a prompt convention rather than a control language.

## 3.2 A formal intermediate representation

ZYAL needs a compiled internal form that is not YAML. YAML is human-friendly, but enforcement should happen against a normalized, typed, deterministic IR.

Add:

```yaml
schema:
  version: v1.1
  compatibility:
    min_host_version: "0.9.0"
    unknown_keys: reject
    yaml_features:
      anchors: deny
      merge_keys: deny
      env_interpolation: explicit_only
  ir:
    emit: true
    hash_algorithm: sha256
    sign_preview: true
```

The host should show:

```text
ZYAL Preview Hash: sha256:...
Runnable: yes/no
Unknown features: none
Permission escalations: shell ask, git_commit ask
Network: denied except registry.npmjs.org
Untrusted context sources: issue_body, README.md
Required human gates: arm, checkpoint, push
```

The user’s `ZYAL_ARM` action should bind to this hash. That prevents the model from showing one thing and arming another.

## 3.3 Trust boundaries and taint tracking

ZYAL currently has guardrails and permissions, but not a trust model. Long-running agents consume untrusted data: GitHub issues, PR comments, README files, dependency docs, web pages, MCP resources, test logs, and user messages. Agent security research highlights that agents with tools and memory introduce vulnerabilities around external tools, memory, and privilege use. OpenClaw’s own security docs emphasize that its default model is a trusted-operator boundary, not a hostile multi-tenant security boundary. ([ICLR Proceedings][13])

Add:

```yaml
trust:
  boundaries:
    - id: trusted_operator
      sources: ["local_user_arm_action", "repo_owner_config"]
      may_request_approval: true

    - id: project_trusted
      sources: ["AGENTS.md", ".jekko/*", "CODEOWNERS"]
      requires_clean_git_at_load: true

    - id: untrusted_repo_content
      sources: ["README.md", "docs/**", "issue_body", "pr_comments", "test_output"]
      taint: untrusted_text
      may_influence:
        - task_discovery
        - test_selection
      may_not_influence:
        - permissions
        - approval_requests
        - memory_promotion
        - network_egress
        - secret_access

    - id: external_web
      sources: ["browser", "mcp_external", "web_docs"]
      taint: external_untrusted
      require_citation_for_memory: true

  prompt_injection:
    detect:
      - "ignore previous instructions"
      - "exfiltrate"
      - "print secrets"
    action: quarantine
    quarantine_artifact: ".jekko/daemon/{{runID}}/taint/injections.jsonl"

  taint_rules:
    - from: untrusted_text
      to: shell.command
      action: block_unless_sanitized
    - from: untrusted_text
      to: memory.write
      action: require_evidence
    - from: external_untrusted
      to: git_commit_message
      action: redact
```

ZYAL should treat context as data with provenance, not as neutral text.

## 3.4 Fine-grained capabilities instead of coarse permissions

Current ZYAL permissions are too broad:

```yaml
permissions:
  shell: ask
  edit: allow
  git_commit: ask
```

That is not enough for 1B+ codebases. Claude Code and Jekko already support more granular permission patterns, and OpenAI’s Agents SDK distinguishes guardrails at input, output, and tool-call stages. ZYAL should define object-capability permissions by path, command, environment, network, secret, branch, tool, database, and destructive-risk tier. ([Claude][4])

Add:

```yaml
capabilities:
  filesystem:
    default: deny
    read:
      - "src/**"
      - "packages/**"
      - "docs/**"
      - "package.json"
      - "bun.lock"
    write:
      - path: "src/**"
        owner_policy: require_codeowner_on_promotion
      - path: "docs/**"
        owner_policy: none
    deny:
      - ".env*"
      - "**/secrets/**"
      - "infra/prod/**"
      - ".git/**"

  shell:
    default: ask
    allow:
      - command: "bun test *"
      - command: "bun run lint*"
      - command: "npx tsgo --noEmit*"
      - command: "git status*"
      - command: "git diff*"
    deny:
      - "rm -rf *"
      - "git push --force*"
      - "curl * | sh"
      - "sudo *"
      - "chmod -R 777 *"
    require_approval:
      - "git commit*"
      - "npm install *"
      - "bun add *"

  network:
    default: deny
    egress:
      allow:
        - host: "registry.npmjs.org"
          reason: "dependency metadata"
          methods: ["GET"]
      ask:
        - host: "github.com"
      deny:
        - host: "*"
    log_all_requests: true

  secrets:
    default: deny
    allow:
      - name: "CI_READONLY_TOKEN"
        scope: "ci_status_read"
        expose_to_model: false
        inject_to_tool_env: true
        commands: ["gh run view *"]
    deny_model_visibility: true

  git:
    branch_policy:
      allowed_prefix: "zyal/{{runID}}/"
      protected_branches: ["main", "release/**"]
    commit: ask
    push: ask
    force_push: deny
```

This is where ZYAL becomes more powerful than prompt-level tools: it can enforce least privilege structurally.

## 3.5 Proof lanes and receipts

ZYAL has `checkpoint.verify`, `guardrails`, and `constraints`, but it lacks an explicit proof model. Jankurai’s core insight is that AI-assisted code must produce proof lanes, receipts, ownership maps, generated zones, security boundaries, repair queues, and merge witnesses. OpenAI’s harness-engineering writing also emphasizes strict boundaries, predictable structure, custom lints, and feeding human taste back into the harness. ([GitHub][11])

Add:

```yaml
proof:
  lanes:
    - id: unit
      type: test
      command: "bun test --timeout 30000"
      timeout: 60s
      required: true
      assert: { exit_code: 0 }

    - id: typecheck
      type: static
      command: "npx tsgo --noEmit"
      timeout: 60s
      required: true
      assert: { exit_code: 0 }

    - id: lint
      type: static
      command: "bun run lint"
      timeout: 60s
      required: true
      assert: { exit_code: 0 }

    - id: security
      type: security
      command: "bun run security:audit"
      timeout: 120s
      required_for:
        risk_gte: 0.6

    - id: migration_dry_run
      type: database
      command: "bun run migration:dry-run"
      timeout: 120s
      required_if_changed:
        - "migrations/**"
        - "db/**"

    - id: performance_budget
      type: benchmark
      command: "bun run bench --json > .jekko/daemon/{{runID}}/bench.json"
      timeout: 300s
      required_if_changed:
        - "src/runtime/**"
      assert_json:
        "$.regression_pct": { lte: 3 }

  receipts:
    store: sqlite_and_mirror
    include:
      - command
      - exit_code
      - stdout_hash
      - stderr_hash
      - artifact_paths
      - started_at
      - duration_ms
      - environment_hash
      - git_tree_hash
    redact:
      - secrets
      - access_tokens

  merge_witness:
    require: true
    template:
      - objective
      - changed_files
      - acceptance_criteria_map
      - proof_lane_results
      - unresolved_risks
      - rollback_plan
      - owner_approvals
      - generated_zones_touched
      - security_boundary_changes
```

Then promotion becomes:

```yaml
promotion:
  require_proof_lanes: ["unit", "typecheck", "lint"]
  require_merge_witness: true
  block_on:
    failed_required_proof_lanes_gte: 1
    missing_rollback_plan: true
    owner_approval_missing_for_touched_paths: true
```

This is how ZYAL reduces “vibe coding”: no proof, no promotion.

## 3.6 Requirements traceability

ZYAL’s `job.objective` is too weak for huge projects. The host needs to track acceptance criteria, assumptions, non-goals, risks, and evidence.

Add:

```yaml
spec:
  sources:
    - id: issue
      type: github_issue
      ref: "owner/repo#1234"
      trust: untrusted_repo_content

    - id: maintainer_note
      type: human_instruction
      trust: trusted_operator

  objective: "Replace legacy auth token refresh flow with scoped refresh sessions."

  non_goals:
    - "Do not change OAuth provider configuration."
    - "Do not migrate production data in this run."

  assumptions:
    require_review_before_use: true
    ledger: sqlite

  acceptance:
    - id: A1
      statement: "Expired access tokens refresh through scoped refresh sessions."
      evidence:
        - proof_lane: unit
          test_regex: "refresh.*scoped"
        - file: "src/auth/refresh.ts"

    - id: A2
      statement: "Refresh token reuse is rejected and logged."
      evidence:
        - proof_lane: unit
          test_regex: "reuse.*reject"
        - proof_lane: security

    - id: A3
      statement: "No production migration is executed."
      evidence:
        - proof_lane: migration_dry_run

  ambiguity_policy:
    when_acceptance_conflicts: pause
    when_requirement_missing: incubate
    when_scope_expands: require_human_approval
```

The model can propose assumptions, but the host records and gates them.

## 3.7 Graph workflows, not only loops

ZYAL v1 has loops and event handlers. That is good for “fix until tests pass,” but not enough for complex work. CrewAI separates crews and flows, while AutoGen’s core is event-driven and can express scalable multi-agent systems. LangGraph is explicitly a graph runtime. ZYAL should add a workflow section that can compile to those runtimes while retaining host-enforced policy. ([CrewAI Documentation][14])

Add:

```yaml
workflow:
  type: state_machine
  start: intake

  states:
    intake:
      do:
        - discover_tasks: true
        - build_repo_map: true
      next:
        - when: tasks_found
          goto: triage
        - when: no_tasks_found
          goto: stop

    triage:
      do:
        - classify_tasks:
            dimensions: [risk, scope, ownership, proofability, dependency_depth]
      next:
        - when: high_risk_or_unclear
          goto: incubate
        - when: ready_simple
          goto: implement

    incubate:
      use: incubator
      next:
        - when: promoted
          goto: implement
        - when: exhausted
          goto: park

    implement:
      do:
        - run_agent: build
      next:
        - when: diff_exists
          goto: prove
        - when: no_progress
          goto: incubate

    prove:
      do:
        - run_proof_lanes: required
        - create_merge_witness: true
      next:
        - when: proof_passed
          goto: checkpoint
        - when: proof_failed
          goto: repair

    repair:
      do:
        - classify_failure: true
      next:
        - when: failure_is_flaky
          goto: prove
        - when: failure_is_design
          goto: incubate
        - when: failure_is_local
          goto: implement

    checkpoint:
      do:
        - checkpoint: true
      next:
        - when: stop_conditions_met
          goto: stop
        - when: more_tasks
          goto: triage

    park:
      do:
        - summarize_and_archive: true
      next: triage

    stop:
      terminal: true
```

This lets ZYAL describe multi-stage autonomy without turning every behavior into ad hoc `on` handlers.

## 3.8 Experiment pools and disjointed worktrees

ZYAL’s incubator has idea passes and prototype passes, but it does not yet encode controlled experimentation. For ambitious codebases, the agent should often try several isolated approaches, score them with evidence, synthesize a final plan, and discard the rest. Tree-of-Thought-style approaches formalize branch/search/evaluate patterns, while harness-engineering research suggests gains come from better tools, middleware, and memory rather than just prompting. ([arXiv][15])

Add:

```yaml
experiments:
  enabled: true
  strategy: disjointed_pool_then_synthesize
  isolation: git_worktree
  max_parallel: 4
  max_variants_per_task: 6
  branch_prefix: "zyal-exp/{{runID}}/{{taskID}}/"

  variants:
    - id: minimal_patch
      prompt: "Smallest safe fix preserving public API."
      agent: build
      risk_bias: low

    - id: refactor_first
      prompt: "Refactor the affected module boundaries, then fix."
      agent: build
      risk_bias: medium

    - id: test_first
      prompt: "Write failing tests first, then implement."
      agent: build
      risk_bias: low

    - id: alternative_design
      prompt: "Find a different architecture with lower long-term maintenance cost."
      agent: plan
      writes: scratch_only

  score:
    objective:
      - metric: proof_lanes_passed
        weight: 0.35
      - metric: diff_size_inverse
        weight: 0.10
      - metric: acceptance_coverage
        weight: 0.25
      - metric: maintainability_review
        weight: 0.15
      - metric: rollback_simplicity
        weight: 0.15

  synthesize:
    agent: plan
    input:
      - variant_summaries
      - diffs
      - proof_receipts
      - critic_reports
    output_schema:
      type: object
      required: [winner, merge_strategy, risks, proof_plan]
```

Add promotion gates:

```yaml
promotion:
  require:
    - winning_variant_has_required_proof
    - losing_variants_summarized
    - discarded_worktrees_deleted_or_archived
    - synthesis_reviewed
```

This gives you “disjointed experiments” without unbounded swarm behavior.

## 3.9 Stateful building on current state

The opposite mode is also needed: not disjointed experiments, but stateful incremental construction. ZYAL should distinguish between these modes explicitly.

```yaml
build_strategy:
  mode: stateful_incremental   # stateful_incremental | disjointed_experiments | hybrid
  increments:
    max_files_per_increment: 12
    require_green_between_increments: true
    checkpoint_each_increment: true
    summarize_delta_to_task_memory: true
  drift_control:
    compare_against_original_plan_every: 3
    on_scope_drift: pause
    on_architecture_drift: require_review
```

For 1B+ projects, “keep editing until it works” is too dangerous. The host should require stable intermediate increments.

## 3.10 Memory as governed infrastructure

ZYAL needs memory sections that separate **run memory**, **task memory**, **repo memory**, **human preference memory**, **skill memory**, and **negative memory**. It also needs write filters, provenance, TTL, contradiction handling, and retrieval budgets. Memory research increasingly treats memory as more than short-term vs long-term; modern memory-agent surveys emphasize consolidation, grounded retrieval, reflection, forgetting, privacy, write-path filtering, contradiction handling, and latency budgets. ([arXiv][16])

Add:

```yaml
memory:
  stores:
    run:
      type: sqlite
      purpose: "event log, decisions, receipts"
      retention: "90d"

    task:
      type: sqlite
      purpose: "task capsules, assumptions, failed attempts, accepted plans"
      retention: "1y"

    repo:
      type: fts5_vector
      purpose: "code summaries, dependency graph, APIs, ownership"
      rebuild:
        on_start: incremental
        on_checkpoint: touched_subgraph_only

    procedural:
      type: skills
      purpose: "repeatable workflows"
      promotion: require_skill_tests

    negative:
      type: sqlite
      purpose: "failed approaches, rejected assumptions, flaky tests, unsafe commands"
      retrieval_priority: high_when_similar

  write_policies:
    - name: no_unverified_memory
      from_taint: [untrusted_text, external_untrusted]
      action: require_evidence

    - name: no_secret_memory
      detect: ["secret", "token", "credential"]
      action: redact_and_quarantine

    - name: summarize_after_iteration
      trigger: after_iteration
      writes_to: task
      schema:
        required:
          - what_changed
          - why
          - evidence
          - open_questions
          - failed_attempts

    - name: promote_repo_fact
      trigger: proof_lane_passed
      requires:
        - source_file_hash
        - proof_receipt
        - no_contradiction

  retrieval:
    context_budget_tokens: 24000
    pack_order:
      - current_acceptance
      - relevant_code_summaries
      - negative_memory
      - last_successful_checkpoint
      - recent_failed_attempts
      - style_rules
    stale_memory_policy: downrank
    contradiction_policy: show_both_and_pause_if_material

  compaction:
    every_iterations: 5
    method: evidence_preserving_summary
    preserve:
      - acceptance_criteria
      - decisions
      - proof_failures
      - current_diff_summary
      - rollback_plan
```

This turns “memory” into a governed database, not a bigger prompt.

## 3.11 Skill lifecycle

AGENTS.md and Skills are becoming portable ways to provide project instructions and reusable capabilities. Anthropic describes Skills as folders of instructions/scripts/resources that agents can discover and use, and Jekko supports Claude-compatible and local skill locations. ZYAL should not replace skills; it should govern which skills may be loaded, how they are versioned, and when a daemon may create or modify one. ([GitHub][17])

Add:

```yaml
skills:
  load:
    - source: ".jekko/skills"
      trust: project_trusted
      allow: ["testing", "repo-map", "migration-review"]
    - source: "agentskills.io"
      trust: external_untrusted

    allow: true
    writes: scratch_only

      require:
        - skill_has_tests
        - skill_does_not_request_extra_permissions
```

This becomes the bridge from “agent coding” to “autonomous software evolution.”

## 3.14 Cost, time, and compute budgets

ZYAL has circuit breakers and incubator budgets, but not enough cost governance. Long-running agents need budgets for wall-clock time, tokens, dollars, parallel workers, branches, tool calls, network, and CI minutes.

```yaml
budgets:
  wall_clock: 6h
  tokens:
    total: 8_000_000
    per_iteration: 150_000
    per_worker: 400_000

    max_usd: 75
    on_exceed: pause

    max: 120
    on_exceed: switch_to_targeted_tests
  branches:
    max_open_worktrees: 8
    max_experiment_branches: 12
  tool_calls:
    shell_per_iteration: 40
    browser_per_task: 10

      max_calls: 40

      use_for: [summaries, grep_analysis, simple_repairs]
```

The host should treat budget exhaustion as a normal state, not failure.

## 3.15 Failure taxonomy and semantic retry

`retry.default` is useful, but “retry the same thing” is often wasteful. ZYAL should classify failures.

```yaml
failure_taxonomy:
  classes:
    flaky_test:
      detect:
        - proof_lane_failed_then_passed
      action: rerun_with_seed_capture

    permission_blocked:
      detect:
        - signal: permission_denied
      action: pause_or_request_specific_capability

    design_dead_end:
      detect:
        - no_progress_iterations_gte: 3
        - same_error_repeated_gte: 2
      action: incubate_current_task

    context_corruption:
      detect:
        - contradiction_in_memory
        - model_references_nonexistent_file
      action: hard_clear_and_repack_context

    dependency_breakage:
      detect:
        - lockfile_changed
        - package_install_failed
      action: require_dependency_review

retry:
  semantic:
    flaky_test:
      max_attempts: 3
      vary: [seed, order]
    design_dead_end:
      max_attempts: 1
      then: incubate_current_task
    context_corruption:
      max_attempts: 1
      then: rebuild_context_pack
```

Retries should change the situation or stop.

## 3.16 Release and deployment gates

ZYAL has commits and pushes, but huge projects need PR, review, release, canary, rollback, and production-protection gates.

```yaml
release:
  mode: pull_request
  pr:
    create: ask
    title: "ZYAL {{job.name}}"
    include_merge_witness: true
    require_codeowners: true
    require_ci_green: true
    require_human_review_for:
      - risk_gte: 0.4
      - touches: ["infra/**", "db/**", "auth/**"]

  deploy:
    default: deny
    staging:
      allow: ask
      proof_required:
        - integration
        - migration_dry_run
    production:
      allow: deny

  rollback:
    require_plan: true
    verify_command: "bun run rollback:check"
    on_missing: block_promotion
```

Autonomy should end at a safe PR unless the organization explicitly grants deployment capabilities.

ZYAL v1 is a good minimum viable schema. For the “world’s most ambitious coding projects,” the language needs a bigger but still composable top-level structure:

```yaml

mcp: {}

```

The key is not “more YAML for its own sake.” The key is **separating policy layers**:

1. **Intent layer** — what is the daemon trying to accomplish?
2. **Capability layer** — what can the host enforce?
3. **Trust layer** — what input is trusted?
4. **Execution layer** — what agents/workflows run?
5. **Memory layer** — what state is stored/retrieved/promoted?
6. **Evidence layer** — what proves progress?
7. **Governance layer** — who approves, when, and why?
8. **Release layer** — what may leave the sandbox?

# 9. A 1B+ project autonomous-coding strategy

For a billion-line or billion-token project, ZYAL should not try to stuff project knowledge into context. It should operate as a **control layer over indexes, shards, proofs, and release gates**.

## 9.1 Required architecture

A serious 1B+ project ZYAL host needs:

1. **Repo graph**: packages, services, APIs, ownership, generated files, dependency direction.
2. **Test impact map**: which tests cover which packages, files, symbols, and behavior.
3. **Context packs**: small generated packets per task, package, API, owner boundary, and recent history.
4. **Lease-based task queue**: workers take scoped tasks with leases; expired leases requeue.
5. **Worktree isolation**: every risky task or experiment runs in its own worktree/container.
6. **Proof lanes**: unit, typecheck, lint, integration, security, performance, migration, accessibility, generated-zone regeneration.
7. **Merge witnesses**: every promotion has evidence, risk, rollback, touched owners, changed files, and proof receipts.
8. **Memory governance**: negative memory, repo facts, skill facts, user/org preferences, all with provenance.
9. **Human gates by risk**: low-risk docs can auto-commit; auth, infra, database, and production changes require review.
10. **Release gates**: staging/canary/rollback, not just git commit.

This aligns with the direction of current harness work: strict boundaries, predictable structure, custom lints, decision observability, and tool/memory/harness improvements rather than relying on a stronger prompt. ([OpenAI][19])

## 9.2 Example hyperscale ZYAL fragment

```yaml
repo:
  scale: hyperscale
  index:
    mode: incremental
    engines:
      - ripgrep
      - tree_sitter
      - language_server
      - dependency_graph
      - test_impact_map
      - ownership_map

  shards:
    strategy: package_graph
    max_files_per_context_pack: 80
    max_tokens_per_context_pack: 12000
    include_cross_shard_edges: true

  ownership:
    sources: ["CODEOWNERS", "package.json", ".owners.yml"]
    require_owner_review_for:
      - cross_package_api_change
      - security_boundary_change
      - database_schema_change
      - public_contract_change

  change_classes:
    low_risk:
      examples: ["docs", "formatting", "single failing unit test"]
      auto_checkpoint: true
    medium_risk:
      examples: ["internal API", "dependency minor update"]
      require_merge_witness: true
    high_risk:
      examples: ["auth", "payments", "database migration", "prod infra"]
      require_human_review: true
      require_incubator: true
      require_experiments: true

tasks:
  ledger: sqlite
  queue:
    leasing: true
    lease_timeout: 30m
    priority:
      - production_regression
      - failing_main
      - security
      - flaky_tests
      - cleanup
  discover:
    - command: "bun run tasks:discover --json"
      output_schema_ref: "schemas/tasks.schema.json"

agents:
  supervisor:
    agent: plan
    responsibilities:
      - shard_assignment
      - risk_classification
      - proof_selection
      - conflict_resolution
  workers:
    - id: low_risk_fixer
      count: 8
      agent: build
      isolation: git_worktree
      allowed_change_classes: [low_risk]
    - id: high_risk_incubator
      count: 2
      agent: plan
      isolation: git_worktree
      allowed_change_classes: [medium_risk, high_risk]

proof:
  select:
    strategy: impact_based_plus_required_baseline
    always: [typecheck, lint]
    from_impact_map: true
    require_full_suite_before_promotion_for:
      - high_risk
      - public_api_change
      - dependency_change
```

## 9.3 The key hyperscale rule

For very large projects, ZYAL should enforce this invariant:

```yaml
constraints:
  - name: no-unbounded-context-search
    invariant: "all_code_access_goes_through_repo_index_or_scoped_read"
    on_violation: block

  - name: no-cross-boundary-edit-without-owner
    invariant: "touched_owner_boundaries_lte_approved_boundaries"
    on_violation: block_promotion

  - name: no-proof-no-merge
    invariant: "all_acceptance_criteria_have_evidence"
    on_violation: block_promotion
```

The agent can be ambitious, but the host must be boring.

# 11. The anti-vibe coding doctrine ZYAL should enforce

ZYAL should explicitly define “anti-vibe” invariants. Jankurai’s language around proof lanes, merge witnesses, repair queues, and ratchets is exactly the kind of discipline ZYAL needs to absorb. ([GitHub][11])

Recommended doctrine:

1. **No proof, no promotion.**
2. **No hidden scope expansion.**
3. **No test deletion without replacement evidence.**
4. **No dependency change without license/security/provenance evidence.**
5. **No generated-file edit unless generated by the generator.**
6. **No cross-owner edit without owner gate.**
7. **No memory write from untrusted text without evidence.**
8. **No production capability in default daemon mode.**
9. **No model-owned success condition.**
10. **No unbounded parallelism, retries, or context growth.**

Represented as ZYAL:

```yaml
anti_vibe:
  enabled: true
  invariants:
    - no_proof_no_promotion
    - no_scope_expansion_without_approval
    - no_test_deletion_without_replacement
    - no_dependency_change_without_review
    - no_generated_file_manual_edit
    - no_cross_owner_edit_without_gate
    - no_untrusted_memory_without_evidence
    - no_production_access_by_default
    - no_model_owned_success_signal
    - no_unbounded_parallelism

  ratchet:
    baseline_on_start: true
    metrics:
      - test_count
      - lint_error_count
      - type_error_count
      - security_findings
      - flaky_test_count
    policy:
      must_not_regress: true
      allow_regression_only_with_human_approval: true
```

# 12. What ZYAL should *not* do

ZYAL should avoid several traps.

First, it should not become a general-purpose programming language. The moment it adds arbitrary conditionals, loops, variables, and script execution everywhere, it becomes another automation DSL with all the safety risks of code. Keep the host evaluator deterministic and typed.

Second, it should not store hidden chain-of-thought. Store decisions, summaries, tool traces, evidence, diffs, assumptions, and receipts. Do not store or expose private reasoning. This is compatible with strong observability: you can record “why this decision was made” without dumping private scratchpad.

Third, it should not assume MCP, Skills, AGENTS.md, or memory files are trusted. MCP is useful because it standardizes connections to external tools and data, and AGENTS.md/Skills are useful because they make agent instructions reusable, but ZYAL must still govern provenance, permissions, and promotion. ([Model Context Protocol][12])

Fourth, it should not let arming happen through assistant output alone. The host must require a separate user action bound to a preview hash.

Fifth, it should not optimize for “run forever.” It should optimize for “run durably until the host can prove stop, pause, park, or promote.”

# 14. The strongest positioning for ZYAL

ZYAL should be described as:

> **A host-enforced, evidence-driven run contract for autonomous software work.**

Not:

> A prompt format for agents.

Not:

> A replacement for LangGraph/OpenHands/Claude Code/Jekko/Codex.

Not:

> A swarm launcher.

The right mental model is:

```text
AGENTS.md / Skills / MCP      = reusable instructions and tools
Jekko / Codex / Claude     = coding-agent surfaces
OpenHands                     = software-agent runtime
LangGraph / AutoGen / CrewAI  = orchestration runtimes
Hermes / OpenClaw             = always-on assistant ecosystems
Jankurai                      = anti-vibe proof infrastructure
ZYAL                          = host-enforced daemon contract tying them together
```

ZYAL wins if it becomes the layer that says:

```text
This is the task.
These are the trusted inputs.
These are the allowed capabilities.
These are the budgets.
These are the agents and workflows.
These are the memories that may be read or written.
These are the proof lanes.
These are the promotion gates.
These are the human approvals.
This is the signed preview hash.
Only the host may arm, pause, promote, or stop.
```

That is a real gap. The ecosystem has many powerful agent frameworks, but the world still needs a portable, declarative, host-enforced contract for long-running coding autonomy. ZYAL is very close to that. To become the default language for ambitious autonomous coding, it needs to evolve from **loop control** into **evidence-governed software evolution control**.
