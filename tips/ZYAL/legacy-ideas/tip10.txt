## Executive verdict

ZYAL is **not redundant** with OpenHands, LangGraph, Hermes, OpenClaw, Codex, Claude Code, or LangChain. It is aiming at a different layer: a **host-enforced control language** for autonomous coding runs. That is powerful. Most agent systems give you agents, tools, sandboxes, memories, skills, UI, or orchestration APIs. ZYAL can become the **portable operating contract** that tells any of those systems: what may run, for how long, under which evidence gates, with what memory, what approvals, what rollback plan, what safety policy, and what durable state.

The biggest upgrade is to stop thinking of ZYAL as only a “daemon YAML” and evolve it into a **universal agent-control spec**: something closer to *Temporal + Kubernetes + GitHub Actions + SWE-bench evaluator + policy-as-code + agent memory capsules*, compiled into Jekko, Codex, Claude Code, OpenHands, Hermes, OpenClaw, LangGraph, or future agent harnesses.

I could not literally verify every X, LinkedIn, Quora, blog, repo, and paper on the internet. I weighted official docs, source repositories, papers, and stable specifications highest, then used public social/blog signals only as weak evidence of user pain points.

---

# 1. What ZYAL already gets right

ZYAL’s strongest ideas are exactly the ones most “autonomous agent” systems under-specify:

1. **Host owns lifecycle.**
   The model does not get to decide when long-running work starts, stops, promotes, or commits. This is essential.

2. **Preview before arming.**
   The `ZYAL_ARM RUN_FOREVER` sentinel is a great design. It makes long-running autonomy a separate human action rather than something smuggled inside an assistant response.

3. **SQLite as truth.**
   This is excellent. Durable state should not live in the model context. LangGraph’s durable-execution model similarly persists graph progress and supports pausing/resuming long-running workflows; ZYAL’s SQLite-first approach fits that world well. ([LangChain Docs][1])

4. **Guardrails are runtime-enforced.**
   This is the right answer to “vibe coding” risk. Security guidance around AI-generated code increasingly emphasizes treating generated code as untrusted, validating behavior, governing dependencies, and keeping human oversight in the workflow. ([Checkmarx][2])

5. **Incubator passes are a serious differentiator.**
   The `incubator` block is one of the most promising parts of ZYAL. Multi-pass strengthening, critique, synthesis, and prototype isolation are much closer to how ambitious engineering work should be done than “agent loops until tests pass.”

6. **Write scopes and isolated worktrees.**
   This maps well to how OpenClaw’s code-agent plugin treats delegated background coding: worktree isolation, plan review, lifecycle cleanup, and explicit goal loops. ([GitHub][3])

The weakness is not the concept. The weakness is that ZYAL v1 still looks like a daemon runbook. To become the default language for world-scale coding projects, it needs stronger semantics around **flow, memory, evidence, codebase intelligence, experiments, governance, and harness portability**.

# 3. The most important conceptual upgrade: ZYAL as an agent control plane

ZYAL should define four layers:

1. **Intent layer** — what work should happen.
2. **Control layer** — who may do what, when, with what budget.
3. **Evidence layer** — what must be proven before state changes.
4. **Portability layer** — how this maps to Jekko, Codex, Claude, OpenHands, Hermes, OpenClaw, LangGraph, etc.

Right now ZYAL has strong control primitives but weaker evidence, memory, codebase-intelligence, and portability primitives.

The slogan should be:

> **ZYAL is not a prompt. ZYAL is the contract between humans, hosts, agents, tools, codebases, and evidence.**

# 4. Biggest missing feature: a `compat` / harness adapter layer

You said you want ZYAL available inside **any chat prompt**: Jekko, Codex, Claude, etc. That means ZYAL needs a portability block.

Different systems have different native concepts:

* Jekko has agents, per-agent permissions, LSP, MCP, and many providers. ([Jekko][9])
* Codex has AGENTS.md, skills, sandbox approvals, MCP, and project instructions. ([OpenAI Developers][10])
* Claude Code has hooks, subagents, CLAUDE.md, and lifecycle events. ([Claude][11])
* OpenHands has sandboxed runtime actions, event streams, microagents/skills, and SDK semantics. ([arXiv][4])
* OpenClaw and Hermes can supervise background sessions across agent providers. ([GitHub][3])

ZYAL should explicitly describe how its semantics project into each harness.

**Extension fragment, not armable:**

```yaml
compat:
  schema_version: v1alpha1

      agent_map:

        builder: build

      instructions:
        export_to: AGENTS.md

          - job.objective
          - constraints
          - quality
          - done_definition
      skills:
        export_to: .codex/skills
        from:
          - skills
          - memory.lessons
      sandbox:
        mode: workspace_write
        approval_policy: from_permissions

    claude_code:
      instructions:
        export_to: CLAUDE.md
      hooks:
        export_from:
          - hooks

          - guardrails
      subagents:
        export_from: agents.workers

        export_from: context

    openhands:
      runtime:
        sandbox: docker
```

This block turns ZYAL from “a YAML for Jekko daemon mode” into “the universal spec for agentic coding control.”

# 5. Missing feature: formal execution graph, not only loops

The current `loop`, `fan_out`, `on`, and `incubator` blocks are useful, but they do not fully express complex workflows. For billion-line codebases, you need a DAG/state-machine model:

* discovery
* triage
* blast-radius analysis
* experiment generation
* isolated implementation
* test selection
* security review
* performance review
* merge readiness
* rollback plan
* deploy/canary
* post-merge monitoring

LangGraph is built around graph/state semantics and durable execution; Temporal’s durable workflow model shows how long-running, crash-proof processes can outlive machines and safely continue after failures. ([LangChain Docs][1])

ZYAL should add `flow`.

```yaml
flow:
  start: intake

  nodes:
    intake:
      type: task_intake
      outputs: [task_capsule]
      next: understand

    understand:
      type: analysis
      agent: explorer
      context: repo_sliced
      require_output: understanding_report
      next: plan

    plan:
      type: planning
      agent: planner
      gates:
        - approval: plan_review
      next:
        if:
          risk_score_gte: 0.7
        then: incubate
        else: implement

    incubate:
      type: incubator
      use: incubator.default
      next: implement

    implement:
      type: implementation
      agent: builder
      isolation: git_worktree
      write_scope: bounded_diff
      next: verify

    verify:
      type: verification
      matrix: verification.default
      next:
        if:
          verification_passed: true
        then: review
        else: diagnose

    diagnose:
      type: root_cause
      agent: planner
      max_retries: 3
      next: implement

    review:
      type: review
      agents: [security_reviewer, performance_reviewer, maintainer_reviewer]
      reduce: consensus_or_escalate
      next: checkpoint

    checkpoint:
      type: checkpoint
      next: stop

    stop:
      type: terminal
```

This is the biggest semantic improvement ZYAL can make. A loop is a weak abstraction. A state graph is a control plane.

# 6. Missing feature: explicit task lifecycle state machine

ZYAL already has runs, tasks, passes, iterations, and events. It needs canonical task states. Hermes’ recent release notes are notable here: durable board, heartbeat, reclaim, zombie detection, auto-block on incomplete exit, retries, and hallucination recovery are exactly the operational primitives long-running agents need. ([GitHub][6])

Add this:

```yaml
task_lifecycle:
  states:
    - discovered
    - triaged
    - scoped
    - planned
    - incubating
    - ready
    - implementing
    - verifying
    - reviewing
    - awaiting_human
    - checkpointed
    - promoted
    - merged
    - parked
    - blocked
    - aborted

  transitions:
    - from: discovered
      to: triaged
      require: [task_capsule_created]

    - from: triaged
      to: scoped
      require: [affected_files_known, owner_identified]

    - from: scoped
      to: planned
      require: [test_plan_created, rollback_plan_created]

    - from: planned
      to: implementing
      require: [plan_reviewed]
      approval: optional_for_low_risk

    - from: implementing
      to: verifying
      require: [diff_exists, write_scope_respected]

    - from: verifying
      to: reviewing
      require: [verification_matrix_passed]

    - from: reviewing
      to: promoted
      require: [review_consensus, evidence_bundle_complete]

    - from: any
      to: parked
      when:
        budget_exhausted: true
      require: [parking_summary_created]
```

Task state should be visible in CLI/API/UI and should be the basis for approval decisions.

# 7. Missing feature: real memory architecture

ZYAL’s `context` and SQLite mirror are a good start, but they are not enough.

For ambitious projects, memory must be layered, typed, queryable, compressible, auditable, and portable. Current agent ecosystems are converging on this: Claude’s context-engineering guidance emphasizes just-in-time context retrieval, progressive disclosure, compaction, structured notes, and multi-agent architectures; OpenAI’s Agents SDK memory writes lessons into workspace files across runs; Hermes emphasizes persistent memory and auto-generated skills. ([Anthropic][14])

ZYAL should add a first-class `memory` block.

```yaml
memory:
  stores:
    run:
      type: sqlite
      scope: run
      retention: 90d
      mirror: .jekko/daemon/{{run_id}}/memory/run

    task:
      type: sqlite
      scope: task
      retention: 180d
      mirror: .jekko/daemon/{{run_id}}/tasks/{{task_id}}/MEMORY.md

    repo:
      type: repo_memory
      scope: repository
      files:
        - AGENTS.md
        - CLAUDE.md
        - .jekko/memory/*.md
      update_policy: human_approved

    architecture:
      type: knowledge_graph
      source:
        - docs/**
        - src/**
        - package.json
        - build files
        - service manifests
      refresh: on_checkout_or_changed_files

    lessons:
      type: skill_memory
      scope: organization
      retention: indefinite
      write_policy: after_human_review

    failures:
      type: failure_memory
      scope: repo
      retention: 1y
      index_by:
        - error_signature
        - failing_test
        - touched_files
        - root_cause

    security:
      type: findings_memory
      scope: repo
      redaction: strict
      retention: 1y

  capsules:
    task:
      max_tokens: 3000
      include:
        - objective
        - non_goals
        - constraints
        - affected_files
        - touched_symbols
        - tests
        - last_known_failures
        - decisions
        - unresolved_questions
        - rollback_plan
      exclude:
        - secrets
        - unrelated_chat
        - stale_hypotheses

    handoff:
      max_tokens: 2000
      include:
        - current_state
        - next_action
        - blockers
        - exact_commands_run
        - exact_files_changed
        - verification_status

  retrieval:
    strategy: hybrid
    sources:
      - repo_index
      - task
      - failures
      - architecture
      - lessons
    rank_by:
      - symbol_overlap
      - path_overlap
      - recent_failure_similarity
      - ownership
      - dependency_distance
    require_citations: true

  compaction:
    policy: structured
    preserve:
      - architectural_decisions
      - unresolved_bugs
      - failing_commands
      - changed_files
      - invariants
      - human_instructions
    drop:
      - redundant_tool_output
      - outdated_plans
      - invalidated_hypotheses
```

The host, not the model, should own memory writes. Model-proposed memory should be staged as a patch to memory, then accepted, rejected, or edited.

# 8. Missing feature: memory provenance and contradiction handling

Long-running agents fail when memory becomes stale, wrong, or self-reinforcing. ZYAL needs provenance:

```yaml
memory:
  provenance:
    require_source_for:
      - architectural_claim
      - dependency_claim
      - security_claim
      - production_behavior_claim

    source_types:
      code: highest
      tests: high
      docs: medium
      prior_agent_summary: low
      model_inference: lowest

    contradiction_policy:
      code_over_docs: true
      test_failure_over_prior_success: true
      human_instruction_over_agent_memory: true
      stale_memory_ttl: 30d

    mark_stale_when:
      - file_changed
      - dependency_changed
      - test_removed
      - branch_rebased
      - owner_changed
```

Every memory item should answer:

* Who wrote it?
* When?
* From which evidence?
* Is it still valid?
* Which files/symbols does it concern?
* Which later events contradicted it?

Without this, “memory” becomes a hallucination amplifier.

# 9. Missing feature: skills as portable, inspectable capabilities

Agent Skills are becoming a cross-harness pattern. Codex supports skills through a progressive-disclosure format; OpenHands has skills/microagents; Hermes has auto-generated skills and even skills for delegating to Claude Code, Codex, Jekko, and Hermes itself. ([OpenAI Developers][15])

ZYAL should have a `skills` block that can import, export, version, test, and retire skills.

```yaml
skills:
  registry:
    sources:
      - path: .jekko/skills
      - path: .codex/skills
      - path: .openhands/skills
      - path: .claude/skills
      - url: org://engineering/skills

  activation:
    mode: progressive_disclosure
    max_loaded: 8
    require_match:
      - task_type
      - repo_language
      - touched_files
      - failure_signature

  skill_policy:
    allow_auto_create: true
    require_human_review_before_org_memory: true
    test_before_enable: true
    retire_when:
      unused_for: 180d
      failure_rate_gte: 0.4

  generated_skills:
    from:
      - successful_tasks
      - repeated_failures
      - human_reviews
    include:
      - exact_commands
      - known_pitfalls
      - repo_conventions
      - verification_steps
    exclude:
      - secrets
      - one-off hacks
      - unverified guesses
```

This is crucial for “smarter over time” without letting the agent silently mutate its own operating instructions.

# 10. Missing feature: codebase intelligence fabric for 1B+ LOC projects

For a billion-line codebase, raw grep plus embeddings is not enough. ZYAL needs a `codebase` block that models the repository as a graph of:

* packages
* services
* symbols
* APIs
* tests
* ownership
* deploy targets
* dependency edges
* runtime configuration
* CI shards
* incidents
* security boundaries
* generated code
* documentation

Sourcegraph’s Cody documentation emphasizes codebase-scale context using search, symbols, APIs, and usage patterns across local and remote repositories; GraphRAG approaches extract knowledge graphs and community summaries from large corpora; repo-level code generation research increasingly focuses on retrieval strategies and repository context. ([Sourcegraph][16])

ZYAL should define codebase indexing as a host responsibility.

```yaml
codebase:

  indexes:
    symbols:
      engines: [lsp, tree_sitter, ctags]

    dependency_graph:

        - package_managers
        - build_system
        - import_graph
        - service_manifests

    test_graph:
      map_tests_to:
        - files
        - symbols
        - packages
        - services
      sources:
        - junit
        - coverage
        - historical_ci
        - static_analysis

    ownership:
      sources:
        - CODEOWNERS
        - service_catalog
        - git_history
        - org_directory

    runtime:
      sources:
        - deployment_manifests
        - env_schema
        - feature_flags
        - migration_history

    security:
      sources:
        - sbom
        - dependency_advisories
        - secret_scans
        - sast
        - threat_models

    docs:
      sources:
        - docs/**
        - ADR/**
        - README*
        - AGENTS.md
        - CLAUDE.md

  slicing:
    default:
      strategy: blast_radius
      include:
        - directly_touched_files
        - nearest_symbols
        - callers
        - callees
        - tests
        - owners

  blast_radius:

      - security_boundaries
```

For very large projects, ZYAL should never let the model “just inspect the repo.” It should make the host prepare a **task-specific code slice**.

# 12. Missing feature: verification matrix and test selection

A single `bun test` is not enough. ZYAL needs a verification matrix:

```yaml
verification:
  default:
    strategy: risk_weighted

    preconditions:
      - command: "git diff --check"
        assert: { exit_code: 0 }

    tiers:
      smoke:
        timeout: 60s
        commands:
          - "bun test --timeout 30000 --filter smoke"

      targeted:
        select:
          from: codebase.indexes.test_graph
          related_to: touched_files
          max_tests: 200
        timeout: 10m

      full:
        commands:
          - "bun test --timeout 60000"
          - "bun run lint"
          - "npx tsgo --noEmit"
        timeout: 20m
        required_when:
          risk_score_gte: 0.5

      security:
        commands:
          - "bun audit"
          - "gitleaks detect --source ."
        required_when:
          touches:
            - auth
            - secrets
            - dependencies

      performance:
        commands:
          - "bun run bench:changed"
        required_when:
          touches:
            - hot_path
            - database
            - cache

    flake_policy:
      detect:
        reruns: 3
        inconsistent_results_gte: 1
      action: mark_flaky_and_escalate
      do_not_promote_on_unexplained_flake: true
```

The host should know the difference between:

* relevant tests passed
* unrelated tests failed
* tests were not discovered
* tests were skipped
* tests are flaky
* no tests exist
* verification command itself failed

# 13. Missing feature: anti-vibe / anti-jank quality policy

You explicitly mentioned “vibe coding” and “jankurai.” The term “Jankurai” appears in public social posts as a tool/ruleset for real-time audits of vibe-coding failure modes, but I found social descriptions rather than stable technical documentation, so I would treat it as a useful signal, not a standard. ([LinkedIn][17])

The broader risk is real. The SusVibes paper argues existing coding-agent security benchmarks were inadequate for multi-file, multi-turn, agentic repository work and introduces tasks spanning large repos and many CWE categories. ([arXiv][18])

ZYAL should make anti-vibe policy explicit:

```yaml
quality:
  profile: agentic_engineering

  anti_vibe:
    require_problem_statement_before_code: true
    require_root_cause_before_fix: true
    require_test_plan_before_code: true
    require_diff_explanation: true
    require_negative_tests_when_bugfix: true
    require_rollback_plan_when_risky: true

    block:
      - skipping_tests_without_approval
      - deleting_tests_to_pass
      - broad_refactor_without_scope
      - dependency_addition_without_reason
      - production_config_change_without_owner
      - migration_without_rollback
      - auth_change_without_threat_model
      - secret_in_diff
      - generated_code_without_marker
```

# 14. Missing feature: richer multi-pass experimentation

The incubator currently has pass types, but it does not fully model **parallel hypotheses** and **disjointed experiments**.

For hard work, the daemon should run multiple independent attempts:

* minimal patch
* deeper refactor
* compatibility shim
* test-only reproduction
* instrumentation-first
* alternative architecture
* revert/regression bisection

Then it should compare outcomes with evidence, not vibes. Research patterns like Tree of Thoughts, Self-Refine, Reflexion, Voyager, and Generative Agents all point toward value in exploration, reflection, memory, and reusable skills rather than one-shot linear generation. ([arXiv][19])

Add `experiments`.

```yaml
experiments:
  enabled: true

  hypotheses:
    generate:
      count: 5
      agent: planner
      require:
        - root_cause_theory
        - expected_files
        - expected_tests
        - risk
        - rollback

    diversify_by:
      - minimality
      - architecture
      - dependency_strategy
      - compatibility
      - performance
      - testability

  branches:
    isolation: git_worktree
    naming: "zyal/{{task_id}}/{{hypothesis_id}}"
    max_parallel: 4

  execution:
    per_hypothesis:
      budget:
        max_iterations: 5
        max_wall_time: 30m
        max_diff_lines: 1000
      stop_when:
        any:
          - verification_passed
          - contradiction_found
          - budget_exhausted

  compare:
    strategy: evidence_weighted
    score:
      correctness: 0.4
      minimality: 0.15
      maintainability: 0.15
      test_quality: 0.15
      rollback_safety: 0.1
      performance: 0.05

    require:
      - diff_summary
      - tests_run
      - failures
      - risk_assessment
      - reviewer_objections

  merge:
    mode: best_patch_or_synthesize
    synthesize_when:
      - best_patch_missing_tests
      - two_patches_complementary
      - reviewer_requests_hybrid

  memory:
    record_failed_hypotheses: true
    compress_to_lessons: true
```

This is how you reduce jank. You do not let one agent barrel forward. You force independent alternatives and evidence-based selection.

# 61. Direct response to overlap concerns

## ZYAL vs Hermes

Hermes is closer to a long-lived agent product: persistent memory, skills, scheduled automations, isolated subagents, multiple chat channels, and server deployment. ([GitHub][5])

**Overlap:** daemon behavior, memory, skills, schedules, subagents, task boards.

**ZYAL gap exposed by Hermes:** ZYAL needs stronger fleet operations: heartbeats, reclaim, zombie detection, goal locking, durable task board, skill lifecycle, and automatic recovery. Hermes v0.13.0’s durable multi-agent board and checkpointing improvements are a very strong signal. ([GitHub][6])

**Recommended stance:** Hermes can be a ZYAL runtime. ZYAL should describe policy and lifecycle; Hermes can execute it.

## ZYAL vs OpenHands

OpenHands is a sandboxed agent runtime and platform for software engineering. It emphasizes agents interacting with code, CLI, browser, and sandboxed environments. ([arXiv][4])

**Overlap:** coding-agent execution, sandboxing, event streams, multi-agent coordination, skills.

**ZYAL gap exposed by OpenHands:** ZYAL needs a canonical action/observation event model, sandbox profiles, and runtime compatibility.

**Recommended stance:** ZYAL should compile to OpenHands runtime semantics, especially for sandboxed execution and event streams.

## ZYAL vs OpenClaw

OpenClaw is a local-first gateway with multi-channel inbox, routing, tools, cron, sessions, companions, and background coding-agent control. Its code-agent plugin supports managed background sessions, plan approval, worktree isolation, wake routing, and lifecycle operations. ([GitHub][7])

**Overlap:** background coding sessions, approvals, worktrees, lifecycle.

**ZYAL gap exposed by OpenClaw:** ZYAL needs channel-aware approvals, session suspend/resume/fork/recover, and better UX for long-running background tasks.

**Recommended stance:** OpenClaw is a great UI/gateway for ZYAL-controlled sessions.

## ZYAL vs LangChain / LangGraph

LangGraph is a durable state-graph runtime with persistence, human-in-the-loop, memory, and debugging. ([LangChain Docs][26])

**Overlap:** workflows, state, pause/resume, HITL, retries.

**ZYAL gap exposed by LangGraph:** ZYAL needs DAG/state-machine semantics, not just loops and handlers.

**Recommended stance:** ZYAL should be declarative; LangGraph can be one compiler target.

## ZYAL vs Codex / Claude Code / Jekko

These are coding-agent harnesses with their own project instructions, skills, permissions, hooks, and context systems. Codex has AGENTS.md and skills; Claude Code has hooks/subagents/context patterns; Jekko has agents, permissions, MCP, LSP, and providers. ([OpenAI Developers][10])

**Overlap:** agent config, permissions, project instructions, skills, hooks.

**ZYAL gap exposed by them:** ZYAL must export/project into their native files and behaviors.

**Recommended stance:** ZYAL should become the canonical source of truth that generates harness-specific instructions and policies.

# 63. The central design principle for ZYAL v2

ZYAL should enforce this invariant:

> **No autonomous coding action may become durable unless the host can explain: what changed, why it changed, who or what approved it, what evidence supports it, what risks remain, and how to roll it back.**

That is the difference between an impressive demo and a language capable of managing billion-line software projects.

ZYAL is already pointed in the right direction. The path to making it the most powerful default language for ambitious coding work is to evolve it from a daemon-control YAML into a **typed, portable, evidence-driven, memory-aware control plane for software-engineering agents**.
