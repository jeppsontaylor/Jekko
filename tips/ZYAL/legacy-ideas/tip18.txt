## Bottom line

ZYAL is **not redundant**. It is aiming at a gap that OpenHands, LangGraph, Hermes, OpenClaw, LangChain Deep Agents, CrewAI, AutoGen, Codex, Claude Code, and Jekko do not fully solve as a portable standard: a **host-enforced control language** that can be embedded in any chat surface while keeping lifecycle, permissions, budget, memory, evidence, and promotion under runtime control.

But ZYAL v1 is still too much like a safe “daemon runbook.” To become the default language for the world’s most ambitious coding projects, it needs to evolve into a **durable, evidence-driven, graph-native operating contract for autonomous software engineering**.

The biggest upgrade is this:

> ZYAL should not only say “loop until tests pass.”
> ZYAL should define **what work exists, how truth is stored, how agents branch, how memory is trusted, how claims become evidence, how code is verified, how unsafe behavior is blocked, how learning is promoted, and how humans retain control.**

I would position ZYAL as the **Kubernetes + Temporal + policy engine + research harness for coding agents**, expressed in a prompt-safe YAML language.

I used official docs, papers, GitHub/open-source material, and accessible social sources. Quora was not directly accessible through search because of robots restrictions, and I would treat X/LinkedIn/Reddit-style claims as weak signals rather than authoritative evidence. Official framework docs and papers should drive the spec.

# 1. What the ecosystem says ZYAL must become

## OpenHands: runtime, event sourcing, sandboxing, and replay matter

OpenHands is closest to a general coding-agent runtime. Its SDK emphasizes stateless components, composability, a clear source of truth, workspaces, tool systems, events, security, and context condensers. It supports local and production/sandboxed execution modes, including Docker and Kubernetes-backed workspaces. ([OpenHands Docs][1])

OpenHands’ agent loop is intentionally single-step and event-driven: condense context, query the model, parse actions, request confirmation if needed, execute, observe, and persist events. It also has tool execution modes and security-risk classifications. ([OpenHands Docs][2])

Its runtime docs stress Docker sandboxing for isolation, consistency, resource control, and reproducibility. The runtime executes shell, file, browser, and other actions through a client-server architecture rather than letting arbitrary model text directly mutate the host. ([OpenHands Docs][3])

**ZYAL implication:** ZYAL’s “SQLite is truth” principle is excellent, but incomplete. ZYAL needs an explicit event schema, replay semantics, workspace abstraction, sandbox profile, state migration rules, and deterministic task wrappers. Otherwise, two hosts implementing ZYAL will diverge.

---

## LangGraph: ZYAL needs graph semantics, not only loops

LangGraph’s durable execution saves progress to a persistent store so agents can pause, resume, survive failures, and support human-in-the-loop interactions. It also warns that non-deterministic or side-effecting operations need to be wrapped as tasks for reliable replay. ([LangChain Docs][4])

LangGraph persistence uses threads, checkpoints, super-steps, pending writes, time travel, human-in-the-loop, memory, and fault tolerance. ([LangChain Docs][5]) Dynamic interrupts let a graph pause at a specific point, save state, and resume with human input, which is exactly the kind of primitive ZYAL needs for approvals, review/edit gates, credential gates, and promotion decisions. ([LangChain Docs][6])

**ZYAL implication:** `loop.policy: forever` is too primitive for serious work. ZYAL should add a first-class `workflow` graph with nodes, edges, retries, interrupts, idempotency, checkpoints, and replay rules. The existing `loop` can remain a shorthand.

---

## LangChain Deep Agents: planning, filesystem context, subagents, and memory are table stakes

LangChain’s Deep Agents are explicitly described as an agent harness for complex, multi-step tasks, with built-in planning, filesystem-based context management, subagent spawning, and long-term memory, built on LangGraph for durable execution and human-in-the-loop. ([LangChain Docs][7])

Deep Agents also highlight “context quarantine”: subagents keep the main context clean while working on specialized subtasks. ([LangChain Docs][8])

**ZYAL implication:** ZYAL’s `agents`, `fan_out`, and `incubator` are good starts, but the language needs explicit planning artifacts, virtual filesystem/context artifact rules, context quarantine, subagent leases, and merge policies.

---

## Hermes Agent: ZYAL needs durable Kanban, skills, heartbeats, and learning gates

Hermes is relevant because it pushes toward a self-improving, persistent agent environment. Its README describes a closed learning loop with agent-curated memory, periodic nudges, autonomous skill creation, skill self-improvement, session search, scheduled automations, delegation/parallelization, multiple terminal backends, and research-ready trajectory tooling. ([GitHub][9])

Hermes’ Kanban system is especially important: it uses a durable task board stored in SQLite, with rows claimable by full OS-process workers, and both agents and humans access the same board. ([Hermes Agent][10]) The May 7, 2026 Hermes release added Kanban heartbeat, reclaim, zombie detection, auto-block, retries, hallucination recovery, checkpoints, and auto-resume. ([GitHub][11])

**ZYAL implication:** ZYAL has a `tasks` ledger but not a mature durable workboard. It needs task leases, worker heartbeats, zombie detection, reclaim policies, blocking reasons, handoffs, WIP limits, priorities, dependencies, and evidence-based task state transitions.

---

## OpenClaw: skills, gateway surfaces, and local memory are useful—but security is the warning

OpenClaw appears more like a local-first personal assistant/gateway than a pure coding-agent framework: it connects chat surfaces and local devices, uses skills, and exposes memory files such as long-term memory and daily notes. ([OpenClaw][12]) Third-party reporting and docs describe it as a self-hosted agent runtime/message router connected to chat platforms and a local machine. ([DigitalOcean][13])

But OpenClaw-style skill ecosystems also create a security lesson. Reporting around OpenClaw highlighted concerns about community skills, broad access, and vulnerabilities in agent/plugin surfaces. ([TechRadar][14])

**ZYAL implication:** ZYAL should support skills, channels, and local-first memory, but must treat skill loading as a supply-chain problem. Skills should be versioned, permission-scoped, sandboxed, tested, signed, and revocable. “Community skill marketplace” plus daemon autonomy is dangerous without policy.

---

## MCP, ACP, and A2A: ZYAL should be the control plane across protocols

MCP is an open standard for connecting AI applications to external systems, data sources, tools, and workflows. ([Model Context Protocol][15]) MCP tools have names and schemas; resources expose context; prompts can be templates; sampling can request model calls while preserving client-side control. ([Model Context Protocol][16])

ACP standardizes communication between code editors/IDEs and coding agents over local stdio or remote HTTP/WebSocket. ([Agent Client Protocol][17]) A2A is for agent-to-agent communication, including agent cards, tasks, messages, artifacts, and long-running coordination. ([Google Developers Blog][18])

**ZYAL implication:** ZYAL should not compete with MCP/ACP/A2A. It should orchestrate them:

* **MCP** = tools/resources/prompts.
* **ACP** = editor/client integration.
* **A2A** = agent-to-agent delegation.
* **ZYAL** = lifecycle, policy, evidence, budgets, memory, task graph, promotion, and host control.

---

## Long-horizon research: compaction is not enough

Anthropic’s long-running harness writeup is directly relevant. It found that agents struggle across many context windows, and that naive compaction often leads to failure modes such as one-shotting, undocumented half-implemented features, and premature “done.” Their harness used an initializer, progress files, feature lists, git commits, and restart protocols so later agents could continue safely. ([Anthropic][19])

OpenAI’s Codex long-horizon discussion describes a 25-hour uninterrupted experiment with millions of tokens, tens of thousands of lines of code, and a loop of planning, editing, running tests/build/lint, observing, repairing, updating docs/status, and repeating. It emphasizes real feedback, externalized state, steerability, parallel threads, skills, automations, and git worktrees. ([OpenAI Developers][20])

A memory survey warns that summarization can cause drift, bigger context windows do not solve everything, reflective memory can reinforce errors unless grounded in evidence, and memory systems need detailed operation logs, staleness handling, contradiction handling, forgetting, and governance. ([arXiv][21])

**ZYAL implication:** ZYAL must not rely on “context.strategy: hybrid” alone. It needs a real memory architecture: working, episodic, semantic, procedural, architectural, negative, and task memory, with evidence, provenance, freshness, contradiction handling, and write gates.

---

## Vibe coding / Jankurai / security: functional correctness is not enough

IBM describes vibe coding as using natural-language instructions to have AI generate executable code. ([IBM][22]) Recent research is more concerning: one benchmark of 200 real-world feature-request tasks found that all tested agents performed poorly on security; one configuration reached 61% functional correctness but only 10.5% security correctness. ([arXiv][23])

Community tools like Jankurai are emerging around “anti-vibe coding” audits and claim real-time checks for hundreds of failure modes across tools like Claude, Codex, Cursor, and Jekko; these are social/open-source signals rather than peer-reviewed evidence, but they show the direction of demand. ([Reddit][24]) TheAuditor is a stronger concrete pattern: a database-first code intelligence platform that indexes code into SQLite, supports vulnerability detection, cross-file taint tracking, architecture intelligence, and deterministic query tools to ground agents. ([GitHub][25])

**ZYAL implication:** ZYAL needs an anti-vibe layer: no silent fallbacks, no test weakening, no fake data, no production root access, no hidden retries, no unreviewed security-sensitive changes, no “looks done” promotion without host evidence.

---

# 2. The strongest thing ZYAL already has

ZYAL’s best idea is that **the host owns the loop**.

That is more important than any individual key. It means:

* The model cannot self-promote.
* The model cannot declare done without host-checked stop conditions.
* The model cannot exceed budgets unless the host permits it.
* The model cannot bypass guardrails by “remembering” rules incorrectly.
* The model cannot silently turn a preview into execution.

That is exactly the right philosophy.

The second-best idea is **SQLite as truth**. This aligns with Hermes’ durable Kanban, TheAuditor’s code-fact database idea, and LangGraph’s checkpointed persistence. The mirror folder is useful for humans, but the database must be canonical.

The third-best idea is the **incubator**. This is where ZYAL can differentiate. Most agent systems have “try again,” “delegate,” or “review.” ZYAL’s incubator can become a formal maturation pipeline: scout, ideate, strengthen, criticize, synthesize, prototype, review, compress, promote.

That is powerful. It just needs to become more evidence-based, branch-aware, and memory-aware.

# 3. The biggest architectural gap: ZYAL is loop-first, but ambitious work is graph-first

Current ZYAL:

```yaml
loop:
  policy: forever
  sleep: 5s
```

That is fine for “fix tests.” It is insufficient for:

* multi-repo migrations,
* architecture refactors,
* dependency upgrades,
* security remediations,
* long-running product builds,
* 1B+ token/day agent farms,
* 1B+ LOC codebases,
* multi-week autonomous engineering,
* coordinated PR streams,
* regulated enterprise changes.

ZYAL should keep `loop` as shorthand, but add:

```yaml
workflow:
  mode: graph
  checkpoint: every_node
  replay: deterministic
  nodes:
    - id: select_task
      type: task_queue_claim
    - id: understand
      type: agent_pass
    - id: plan
      type: agent_pass
    - id: implement
      type: agent_pass
    - id: verify
      type: host_verification
    - id: review
      type: multi_review
    - id: checkpoint
      type: git_checkpoint
    - id: promote
      type: human_gate
  edges:
    - from: select_task
      to: understand
    - from: understand
      to: plan
    - from: plan
      to: implement
    - from: implement
      to: verify
    - from: verify
      to: review
      when: verified
    - from: verify
      to: incubate
      when: failed_gte: 2
    - from: review
      to: checkpoint
      when: approved
    - from: checkpoint
      to: promote
```

This should support:

```yaml
workflow:
  interrupts:
    - before: promote
      require: human_approval
    - before: credential_use
      require: security_approval
    - before: destructive_migration
      require: dba_approval

  idempotency:
    shell: require_declared_side_effects
    network: require_cache_key
    model_calls: record_prompt_hash_and_response_id

  replay:
    allow:
      - pure_shell
      - file_reads
      - host_verification
    forbid:
      - deploy
      - send_email
      - rotate_secret
    require_task_wrapper:
      - shell
      - mcp
      - a2a
```

This borrows the right lesson from LangGraph: durable state and replay require more than a transcript. Side effects need structure. ([LangChain Docs][4])

# 4. Critical missing feature: origin-tracked arming

ZYAL says preview is separate from arming, and assistant output must not start or stop the daemon. That principle is correct, but the current textual sentinel is not enough by itself.

A malicious repo file, tool output, prompt injection, or assistant response could contain:

```text
ZYAL_ARM RUN_FOREVER id=...
```

The runtime must not treat every occurrence of that string as trusted.

Add:

```yaml
arming:
  mode: host_origin_required
  require:
    - preview_generated_by_host: true
    - arm_nonce_from_preview: true
    - human_ui_action: true
    - trusted_channel: user_input
  forbid:
    - assistant_output
    - tool_output
    - repository_file
    - mcp_resource
    - web_content
```

Better:

```yaml
arming:
  challenge:
    nonce: host_generated
    expires: 10m
  accept:
    channels: [host_ui_button, signed_cli_command]
  reject_textual_arm_from:
    - assistant
    - tool
    - mcp
    - repo
    - browser
```

This is essential if ZYAL is available “inside any chat prompt.” The host must track **message origin**, not just text content.

# 5. ZYAL needs a formal task/requirements ledger

Current `tasks` is too thin:

```yaml
tasks:
  ledger: sqlite
  discover:
    - command: "cat TODO.md | grep '- \\[ \\]'"
```

That works for toy projects. For serious coding, ZYAL needs an immutable, host-owned requirements ledger.

Anthropic’s harness used a feature list with end-to-end descriptions because agents otherwise forgot requirements, overwrote docs, or prematurely declared completion. ([Anthropic][19]) ZYAL should make that pattern first-class.

Proposed:

```yaml
requirements:
  ledger: sqlite
  mirror: ".jekko/daemon/{{runID}}/requirements.jsonl"

  sources:
    - type: prompt
      trust: human
    - type: github_issues
      query: "label:zyal-run"
      trust: external
    - type: openapi
      path: "openapi.yaml"
      trust: contract
    - type: docs
      paths: ["docs/product/*.md"]
      trust: advisory

  normalize:
    agent: architect
    output_schema:
      type: object
      required: [id, user_value, acceptance, risks, affected_surfaces]
    require_human_review: true

  fields:
    id: string
    status:
      enum: [discovered, scoped, planned, in_progress, verified, blocked, parked, promoted]
    acceptance:
      type: list
    non_goals:
      type: list
    risk:
      enum: [low, medium, high, critical]
    owner:
      type: string
    evidence_required:
      type: list

  mutation_policy:
    model_can_create: true
    model_can_edit_acceptance: propose_only
    model_can_mark_verified: false
    host_marks_verified: true
    human_can_freeze: true

  anti_drift:
    forbid_silent_requirement_deletion: true
    require_diff_for_requirement_change: true
    alert_on_scope_growth: true
```

This is one of the most important upgrades. ZYAL should not let agents “solve” a task by shrinking the task.

# 6. ZYAL needs evidence objects, not just assertions

Current ZYAL has assertions, guardrails, stop conditions, and readiness scores. But a 1B+ project needs evidence as a first-class type.

Add:

```yaml
evidence:
  storage: sqlite
  mirror: true
  require_provenance: true
  types:
    test_result:
      fields: [command, exit_code, stdout_hash, stderr_hash, duration, environment_hash]
    diff_fact:
      fields: [files_changed, hunks, insertions, deletions, semantic_summary]
    code_fact:
      fields: [symbol, file, line_range, query, result_hash]
    security_scan:
      fields: [tool, version, findings, severity, suppressions]
    performance_result:
      fields: [benchmark, baseline, candidate, delta, confidence]
    human_approval:
      fields: [approver, timestamp, scope, decision, comment]
    model_claim:
      fields: [claim, supporting_evidence_ids, confidence_capped]
```

Then every promotion can require evidence:

```yaml
promotion:
  require_evidence:
    - type: test_result
      command_regex: "bun test.*"
      exit_code: 0
      freshness: 30m
    - type: security_scan
      max_severity: medium
    - type: diff_fact
      max_files_changed: 20
    - type: human_approval
      when_risk_gte: high

  reject_if:
    - evidence_missing: true
    - evidence_from_untrusted_worker: true
    - model_claim_without_host_evidence: true
```

This is how ZYAL becomes less “agent says it is ready” and more “host proves it is ready.”

# 7. Memory: ZYAL’s current context block is far too small

Current:

```yaml
context:
  strategy: hybrid
  compact_every: 10
  hard_clear_every: 20
  preserve:
    - "objective"
    - "progress"
```

That is useful, but it treats memory as context-window management. Long-running systems need **memory governance**.

The research warning is clear: summarization can drift, long context can dilute attention, reflection can reinforce errors, and memory needs logs, freshness, contradiction handling, and governance. ([arXiv][21])

ZYAL should add:

```yaml
memory:
  enabled: true

  stores:
    working:
      type: context_window
      policy: recent_relevant
      max_tokens: 60000

    episodic:
      type: sqlite_events
      captures:
        - tool_call
        - observation
        - agent_decision
        - checkpoint
        - failure
        - human_feedback

    semantic:
      type: vector_plus_sqlite
      namespaces:
        - repo_facts
        - architecture
        - requirements
        - decisions
      retrieval:
        strategy: hybrid
        semantic_top_k: 20
        keyword_top_k: 20
        rerank: true

    procedural:
      type: skills
      path: ".jekko/skills"
      promotion: gated

    negative:
      type: sqlite
      description: "Known bad approaches, failed hypotheses, forbidden shortcuts."
      ttl: 90d

    architectural:
      type: sqlite
      captures:
        - ADR
        - API_contract
        - dependency_rule
        - data_model
        - migration_rule

  write_policy:
    model_can_write:
      - episodic
      - proposed_semantic
      - proposed_skill
    host_can_promote:
      - semantic
      - procedural
      - architectural
    require_evidence_for:
      - semantic
      - procedural
      - architectural
    forbid:
      - secrets
      - credentials
      - personal_data_unless_allowed

  retrieval_policy:
    before_task:
      include:
        - requirement_capsule
        - recent_failures
        - relevant_architecture
        - negative_memory
    before_edit:
      include:
        - symbol_graph
        - affected_tests
        - owner_rules
    before_promotion:
      include:
        - acceptance_criteria
        - verification_history
        - unresolved_objections

  contradiction_policy:
    detect: true
    on_conflict: require_review
    prefer:
      - host_evidence
      - human_decision
      - latest_verified_artifact
      - model_summary

  memory_receipts:
    required: true
    fields:
      - memory_ids_read
      - memory_ids_written
      - evidence_ids_used
      - contradictions_found
```

The most important part is **memory receipts**. Every agent pass should record what memory it read, what memory it wrote, and what evidence supported it.

That prevents “vibe memory.”

# 8. ZYAL should add a code intelligence database

For 1B+ coding projects, no model should browse files manually as its main source of truth. ZYAL should make code indexing a runtime primitive.

TheAuditor’s “database-first” model is a strong pattern: index the codebase into SQLite, expose deterministic query tools, and ground the agent in code facts rather than hallucinated architecture. ([GitHub][25])

Add:

```yaml
codebase:
  index:
    enabled: true
    storage: sqlite
    refresh:
      on_start: true
      after_checkpoint: true
      incremental: true

    languages:
      - typescript
      - python
      - go
      - rust
      - java
      - sql
      - terraform

    facts:
      - symbols
      - imports
      - call_graph
      - data_flow
      - taint_flow
      - ownership
      - test_coverage_map
      - dependency_graph
      - api_contracts
      - migrations
      - config_surfaces
      - security_sinks
      - circular_dependencies
      - hotspots

  query_policy:
    before_edit:
      require_queries:
        - affected_symbols
        - affected_tests
        - owner_rules
    before_security_sensitive_change:
      require_queries:
        - taint_paths
        - authz_checks
        - secret_flows
    before_promotion:
      require_queries:
        - changed_symbols
        - impacted_dependents
        - test_coverage_map

  fail_closed:
    if_index_stale: pause
    if_parse_errors_gte: 50
    if_unknown_language: require_human_review
```

This would make ZYAL much stronger than ordinary chat-agent workflows.

# 9. Anti-vibe coding should be a first-class ZYAL module

Current guardrails block patterns like force push, secrets, or `DROP TABLE`. Good, but not enough.

Vibe-coding failures are often not single forbidden strings. They are behavioral patterns:

* deleting tests,
* weakening assertions,
* adding fake fallback data,
* swallowing errors,
* disabling auth,
* ignoring migration safety,
* silently changing public contracts,
* adding retries instead of fixing root cause,
* creating broad `any` types,
* suppressing TypeScript/lint/security warnings,
* producing functional but insecure code.

ZYAL should add:

```yaml
anti_vibe:
  enabled: true
  fail_closed: true

  principles:
    - evidence_over_opinion
    - no_silent_fallbacks
    - no_test_weakening
    - no_scope_shrinking
    - no_fake_success
    - no_unreviewed_security_changes
    - no_hidden_production_side_effects

  checks:
    - name: no-test-deletion
      detect:
        git_diff:
          paths: ["**/*test*", "**/*.spec.*"]
          patterns:
            - "^-.*expect\\("
            - "^-.*assert"
            - "^-.*it\\("
            - "^-.*test\\("
      action: block_promotion
      unless:
        human_approval: qa_owner

    - name: no-assertion-weakening
      detect:
        semantic_diff:
          from:
            - strict_equality
            - throws
            - rejects
            - permission_denied
          to:
            - truthy
            - snapshot_only
            - skipped
            - todo
      action: block

    - name: no-silent-catch
      detect:
        code_patterns:
          - "catch.*\\{\\s*\\}"
          - "catch.*return null"
          - "catch.*return \\[\\]"
      action: require_review

    - name: no-fake-data-fallback
      detect:
        code_patterns:
          - "mock.*production"
          - "fallback.*sample"
          - "dummy.*data"
      action: block_promotion

    - name: no-dangerous-any
      detect:
        code_patterns:
          - ": any"
          - "as any"
          - "@ts-ignore"
          - "@ts-expect-error"
      action: require_justification
      max_new_occurrences: 0

    - name: no-security-regression
      run:
        - "semgrep --config auto"
        - "npm audit --audit-level=high"
        - "bun test:security"
      action: block_promotion
```

Also add a score, but never let score alone promote:

```yaml
anti_vibe:
  scoring:
    enabled: true
    score_key: jank_score
    minimum_for_promotion: 0.85
    score_is_advisory: true
    require_host_evidence: true
```

The research lesson is that functional correctness and security correctness diverge sharply. A daemon that only checks tests will ship insecure code. ([arXiv][23])

# 10. Incubator upgrades: make it branch-aware, adversarial, and evidence-scored

The incubator is ZYAL’s most promising feature. I would expand it from “hard task maturation” into a **bounded research-and-development laboratory**.

Current passes:

* scout
* idea
* strengthen
* critic
* synthesize
* prototype
* promotion_review
* compress

Upgrade to support:

* disjoint hypotheses,
* independent worktrees,
* branch-from-checkpoint,
* adversarial review,
* replay from critical step,
* solution portfolio,
* comparative scoring,
* merge strategy,
* memory learning.

Proposed:

```yaml
incubator:
  enabled: true
  strategy: portfolio_lab

  route_when:
    any:
      - repeated_attempts_gte: 2
      - no_progress_iterations_gte: 3
      - risk_score_gte: 0.6
      - unknown_architecture: true
      - touches_security_boundary: true
      - changes_public_api: true

  lab:
    branch_from: latest_green_checkpoint
    isolation: git_worktree
    max_candidates: 5
    max_parallel: 3
    require_disjointness: true

    candidate_policy:
      diversity_axes:
        - architecture
        - dependency_choice
        - migration_strategy
        - test_strategy
      forbid_duplicate_plans: true
      require_plan_before_code: true

  passes:
    - id: scout
      type: scout
      context: blind
      writes: scratch_only
      output: problem_map

    - id: decompose
      type: decomposition
      context: requirements
      writes: scratch_only
      output: task_graph

    - id: generate_hypotheses
      type: idea
      count: 5
      context: inherit
      writes: scratch_only
      output: candidate_plans

    - id: adversarial_review
      type: critic
      count: 2
      context: critic
      writes: scratch_only
      focus:
        - security
        - migration
        - maintainability
        - hidden_requirement_loss

    - id: prototype_candidates
      type: prototype
      count: 3
      context: candidate
      writes: isolated_worktree
      agent: build

    - id: verify_candidates
      type: host_verification
      writes: scratch_only
      verify:
        - unit
        - integration
        - security
        - affected_tests
        - performance_smoke

    - id: compare
      type: compare
      context: pool
      writes: scratch_only
      scoring:
        - correctness
        - security
        - simplicity
        - diff_size
        - rollbackability
        - test_coverage
        - architecture_fit

    - id: synthesize
      type: synthesize
      context: pool
      writes: isolated_worktree
      allow_merge_from_candidates: true

    - id: promotion_review
      type: promotion_review
      context: promotion
      writes: scratch_only

    - id: compress
      type: compress
      context: ledger_only
      writes: scratch_only
      output:
        - task_memory
        - negative_memory
        - skill_candidate

  promotion:
    promote_at: 0.82
    require_evidence:
      - tests_identified
      - scope_bounded
      - rollback_known
      - affected_files_known
      - security_reviewed
      - candidate_compared
    block_on:
      unresolved_critical_objections_gte: 1
      unverifiable_claims_gte: 1
      stale_code_index: true
```

This also aligns with SWE-Replay-style research: reusing previous trajectories and branching from critical intermediate states can reduce cost and improve outcomes. ([arXiv][26])

# 11. Add experiment management and trajectory replay

ZYAL should formalize experiments. This is different from `fan_out`.

`fan_out` says: split work into parallel items.
`experiments` says: try multiple approaches to the same problem, record trajectories, compare, replay, and promote the best.

```yaml
experiments:
  enabled: true
  storage: sqlite
  mirror: true

  branch:
    from: latest_verified_checkpoint
    isolation: git_worktree
    naming: "exp/{{taskID}}/{{candidateID}}"

  candidates:
    count: 4
    diversity:
      require_distinct_plan: true
      min_plan_distance: 0.35
      axes:
        - minimal_patch
        - refactor
        - dependency_upgrade
        - adapter_layer

  trajectory:
    record:
      - prompts
      - tool_calls
      - file_diffs
      - test_results
      - failures
      - reviewer_objections
    redact:
      - secrets
      - credentials
      - private_user_data

  replay:
    allow_from:
      - before_first_edit
      - before_failed_test
      - before_merge_conflict
      - before_security_objection
    deterministic_where_possible: true

  compare:
    host_scored: true
    metrics:
      correctness:
        weight: 0.35
      security:
        weight: 0.25
      maintainability:
        weight: 0.15
      diff_size:
        weight: 0.10
      performance:
        weight: 0.10
      rollbackability:
        weight: 0.05

  promote:
    require:
      - best_candidate_verified: true
      - loser_summaries_archived: true
      - negative_memory_written: true
```

This would make ZYAL dramatically more powerful than “retry until something works.”

# 12. Task board: ZYAL should absorb the best of Hermes Kanban

Add a first-class durable queue:

```yaml
queue:
  type: kanban
  storage: sqlite
  lanes:
    - backlog
    - ready
    - claimed
    - blocked
    - incubating
    - verifying
    - review
    - done
    - parked

  task_schema:
    required:
      - id
      - title
      - objective
      - acceptance
      - risk
      - lane
      - priority
      - dependencies

  leases:
    enabled: true
    heartbeat_every: 30s
    expires_after: 3m
    on_expire: reclaim

  zombie_detection:
    no_event_for: 10m
    action: reclaim_or_pause
    max_reclaims: 2

  blocking:
    require_reason: true
    allowed_reasons:
      - needs_human_decision
      - missing_dependency
      - flaky_environment
      - unclear_requirement
      - security_risk
      - merge_conflict
      - budget_exhausted

  priorities:
    strategy: weighted
    fields:
      user_impact: 0.35

      effort: -0.10
      age: 0.10

  wip_limits:
    claimed: 10
    incubating: 5
    review: 20
```

This answers a major overlap concern with Hermes: Hermes has a durable Kanban. ZYAL should not merely duplicate it; ZYAL should specify the portable **contract** for such a board so Jekko, Hermes, OpenHands, LangGraph, or a custom runtime can implement it.

# 15. Verification needs to become layered and adversarial

Current `checkpoint.verify` is usually a shell command. Serious ZYAL needs verification classes:

```yaml
verification:
  strategy: layered
  fail_closed: true

  suites:
    unit:
      command: "bun test --unit"
      timeout: 2m
      assert: { exit_code: 0 }

    integration:
      command: "bun test --integration"
      timeout: 10m
      assert: { exit_code: 0 }

    typecheck:
      command: "npx tsgo --noEmit"
      timeout: 2m
      assert: { exit_code: 0 }

    lint:
      command: "bun run lint"
      timeout: 2m
      assert: { exit_code: 0 }

    security:
      commands:
        - "semgrep --config auto"
        - "npm audit --audit-level=high"
      timeout: 10m

    mutation_smoke:
      command: "bun run mutation:smoke"
      timeout: 15m
      required_when:
        - tests_added: true
        - critical_path: true

    e2e:
      command: "bun run test:e2e"
      timeout: 20m
      required_when:
        - touches_ui: true
        - touches_auth: true

  affected_test_selection:
    enabled: true
    source: codebase.index.test_coverage_map
    require_full_suite_when:
      - coverage_unknown: true
      - touches_shared_library: true
      - touches_security_boundary: true

  authenticity:
    forbid:
      - test_skip_added
      - assertion_weakened
      - snapshot_only_for_new_behavior
    require_fail_before_pass:
      when:
        - bugfix
        - security_fix
```

This is how you reduce jank: not by telling the model to be careful, but by making the host require evidence.

# 17. Skills and learning: powerful, but only if gated

Hermes’ closed learning loop is powerful: skills can be created and improved from experience. ([GitHub][9]) But research on SWE skills suggests that indiscriminately loading skills can add large token overhead and often does not improve pass rate; specialized skills can help, but skill systems need evidence and selectivity. ([arXiv][28])

ZYAL should add a `learning` module:

```yaml
learning:
  enabled: true

  skill_candidates:
    create_when:
      - repeated_success_pattern_gte: 3
      - human_marks_reusable: true
      - task_family_recurs: true

    candidate_schema:
      required:
        - name
        - purpose
        - trigger
        - procedure
        - examples
        - contraindications
        - required_tools
        - evidence

  promotion:
    require:
      - validation_tasks_gte: 3
      - no_security_regression: true
      - token_overhead_estimated: true
      - human_approval: true

  loading:
    strategy: selective
    max_skills_per_task: 5
    retrieval:
      by:
        - task_type
        - language
        - framework
        - failure_mode
    forbid_global_skill_dump: true

  retirement:
    stale_after: 90d
    retire_if:
      - success_rate_lt: 0.4
      - causes_regressions_gte: 2
      - incompatible_with_current_stack: true
```

Key rule: the model may propose a skill, but the host promotes it.

# 22. ZYAL needs benchmark/evaluation mode

To become a serious language, ZYAL should include a conformance and benchmark profile.

GitTaskBench found that even strong systems struggle with repository-aware tasks, with failures often caused by environment setup and dependency resolution. ([arXiv][31]) SWE-EVO shows that long-horizon software-evolution tasks remain much harder than isolated SWE-Bench-style tasks. ([arXiv][32])

Add:

```yaml
eval:
  mode: shadow
  datasets:
    - name: internal_regression_tasks
      path: ".jekko/evals/tasks.jsonl"
    - name: swe_bench_style
      adapter: swe_bench
    - name: security_vibe
      adapter: custom

  metrics:
    - task_success
    - security_success
    - tests_passed
    - cost
    - wall_time
    - human_interventions
    - regressions
    - requirement_drift
    - diff_size
    - rollback_success

  compare_against:
    - baseline: human_patch
    - baseline: previous_zyal_version
    - baseline: simple_agent_loop

  release_gate:
    require:
      task_success_gte: 0.65
      security_success_gte: 0.90
      regression_rate_lte: 0.02
```

ZYAL itself should have a public conformance suite:

* parser conformance,
* arming/origin conformance,
* permission conformance,
* replay conformance,
* event-log conformance,
* guardrail conformance,
* memory conformance,
* task-board conformance.

scale:

    forbid_cross_shard_edit_without:
      - dependency_impact_analysis
      - owner_approval

  owners:
    source:
      - CODEOWNERS
      - service_catalog
      - architecture_registry
    require_owner_gate_when:
      - public_api_change
      - migration
      - security_boundary
      - shared_library

  merge_coordinator:
    enabled: true

# 26. Concrete upgraded example: disjoint experiments with stateful synthesis

This addresses your “disjointed experiments, and stateful building on current state” requirement.

```yaml
experiments:
  enabled: true
  objective: Find the safest implementation strategy for the auth migration.

  branch:
    from: latest_green_checkpoint
    isolation: git_worktree

  candidates:
    count: 5
    require_disjointness: true
    axes:
      - minimal_adapter
      - full_refactor
      - compatibility_layer
      - database_first
      - api_gateway_first

  each_candidate:
    passes:
      - understand
      - plan
      - implement
      - verify
      - self_review
      - security_review
    verification:
      - unit
      - integration
      - security
      - affected_e2e

  compare:
    metrics:
      correctness: 0.30
      security: 0.30
      migration_safety: 0.20
      maintainability: 0.10
      diff_size: 0.05
      rollbackability: 0.05
    require_loser_summaries: true
    write_negative_memory: true

  synthesize:
    strategy: best_plus_patches
    allow:
      - cherry_pick_verified_hunks
      - combine_tests
      - import_rollback_plan
    forbid:
      - combining_unverified_code
      - merging_conflicting_architectures

  promote:
    require:
      - winning_candidate_verified
      - synthesis_reverified
      - security_review_passed
      - rollback_plan_present
```

This is much more powerful than “parallel workers merge all.”

# 27. How ZYAL should relate to OpenHands, Hermes, OpenClaw, LangGraph, LangChain, and others

## ZYAL vs OpenHands

**Overlap:** both care about sandboxing, tools, events, runtime control, and agent execution.

**Difference:** OpenHands is a runtime/SDK; ZYAL should be a portable declarative control contract.

**Upgrade path:** ZYAL should be able to compile into an OpenHands execution plan: workspace, tools, events, security policy, and agent loop. OpenHands’ event-sourced, sandboxed architecture is a strong implementation target. ([arXiv][33])

---

## ZYAL vs LangGraph

**Overlap:** durable state, human-in-loop, long-running agents.

**Difference:** LangGraph is a programmable graph runtime; ZYAL should be a YAML control language that can compile to a graph runtime.

**Upgrade path:** add `workflow`, `checkpoint`, `interrupt`, `replay`, and `task_wrapper` semantics so ZYAL can target LangGraph-like execution. ([GitHub][34])

---

## ZYAL vs LangChain Deep Agents

**Overlap:** planning, subagents, memory, filesystem context, long-running work.

**Difference:** Deep Agents are an agent harness; ZYAL should specify lifecycle, policy, evidence, approval, and promotion.

**Upgrade path:** add planning artifacts, virtual context filesystem, subagent context quarantine, and memory receipts. ([LangChain Docs][7])

---

## ZYAL vs Hermes

**Overlap:** SQLite state, durable tasks, multi-agent coordination, memory, skills, scheduled work.

**Difference:** Hermes is an agent environment; ZYAL should be the portable spec that can drive Hermes-like execution while enforcing host policies.

**Upgrade path:** add Kanban leases, heartbeat, zombie detection, skill promotion gates, provider routing, auto-resume, and trajectory archival. Hermes is ahead of ZYAL v1 on durable multi-agent task-board mechanics. ([Hermes Agent][10])

---

## ZYAL vs OpenClaw

**Overlap:** skills, local-first agent surfaces, memory, user/device integrations.

**Difference:** OpenClaw is more gateway/personal-assistant oriented; ZYAL is repo-daemon/control-plane oriented.

**Upgrade path:** learn from OpenClaw’s local-first memory and skill packaging, but do not inherit broad access or unsafe skill loading. ZYAL should sandbox skills and require permission manifests. ([TechRadar][14])

---

## ZYAL vs CrewAI / AutoGen / Microsoft Agent Framework

**Overlap:** multi-agent orchestration, roles, flows, memory, state.

**Difference:** those are programming frameworks; ZYAL should be declarative and prompt-embeddable.

CrewAI emphasizes guardrails, memory, knowledge, and observability; its memory system includes adaptive recall using semantic, recency, and importance signals. ([CrewAI Documentation][35]) AutoGen supports saving and loading state for agents and teams; Microsoft’s newer Agent Framework combines AutoGen-style multi-agent abstractions with enterprise features such as session state, type safety, filters, and telemetry. ([Microsoft GitHub][36])

**Upgrade path:** ZYAL should expose these as portable concepts: `agents`, `memory`, `observability`, `state`, `workflow`, `review`, and `human_gates`.

# 30. The spec should define “done” much more strictly

Current stop conditions are necessary but not sufficient.

Add “done contracts”:

```yaml
done:
  host_owned: true
  require:
    - stop_conditions_met
    - requirements_verified
    - evidence_complete
    - no_unresolved_critical_objections
    - memory_compressed
    - rollback_plan_recorded
    - final_state_clean

  forbid_done_if:
    - model_only_claim: true
    - tests_not_run: true
    - code_index_stale: true
    - requirement_drift_unreviewed: true
    - security_scan_missing_when_required: true
    - pending_human_gate: true
```

This directly attacks premature “done.”

# 34. The philosophical north star

ZYAL should not try to be “the smartest agent.”

It should be the language that makes any agent safe enough, durable enough, observable enough, and evidence-driven enough to work on real software for a long time.

The winning framing:

> **ZYAL is the host-enforced contract for autonomous software engineering.**
> Models can propose. Agents can act. Workers can branch. Skills can evolve.
> But the host owns truth, memory, budgets, permissions, evidence, promotion, and stop.

That is the part the world still needs.
