Absolutely. My strongest recommendation: **do not make ZYAL “another agent framework.”** Make it the **portable host-control specification** that can command *any* agent runtime: Jekko, Codex, Claude Code, OpenHands, Hermes, OpenClaw, LangGraph, Aider, SWE-agent, local shells, cloud sandboxes, CI systems, and MCP tools.

That is the real opening. OpenHands, Hermes, OpenClaw, LangGraph, Codex, and Claude-style agents are converging on long-running agentic coding, but they still mostly expose **runtime behavior, prompt files, tool APIs, or workflow graphs**. ZYAL can become the missing layer: the **auditable, declarative, prompt-embeddable daemon contract** that says who may act, where, for how long, under what evidence gates, with which rollback path, and with what durable memory.

I would evolve ZYAL from:

> “A YAML daemon runbook for Jekko”

into:

> **“The Kubernetes Job / Temporal Workflow / SRE runbook / policy contract for autonomous software engineering agents.”**

## 1. Competitive landscape: what matters from OpenHands, LangGraph, Hermes, OpenClaw, Codex, and research

### OpenHands

OpenHands is closest to ZYAL’s “autonomous coding runtime” problem, but it is currently more of a **runtime and SDK** than a portable prompt-level control language. Its SDK is described as a unified type-safe framework for agents, with applications consuming the SDK, and with local, Docker, and remote workspace options including Kubernetes-style production deployment paths. ([OpenHands Docs][1])

OpenHands has several pieces ZYAL should learn from immediately:

**Event-driven agent steps.** OpenHands frames the agent as a stateless, event-driven loop with LLM calls, tool orchestration, context management, security validation, and a single-step execution model. That is excellent for durability and interruption because a host can checkpoint after every step rather than trusting a monolithic “run.” ([OpenHands Docs][2])

**Runtime isolation and multi-modal tools.** The OpenHands paper describes a runtime with Docker sandboxing plus bash, IPython, and browser interaction through Playwright, and support for arbitrary Docker images, skills, and delegation. ZYAL should not just say `shell: ask`; it needs runtime profiles that define the sandbox, image, egress, file mounts, browser access, and tool risk. 

**Persistence model.** OpenHands persists conversation state as event logs, base state, config, tool outputs, stats, workspace context, activated skills, secrets, and agent state. ZYAL already says “SQLite is truth,” which is excellent, but it needs a typed event schema so different hosts can replay, inspect, and compare runs. ([OpenHands Docs][3])

**Context condenser.** OpenHands has condenser strategies that summarize older context while preserving goals, specs, files, and other key information, with claims of reducing per-turn cost while keeping performance stable. ZYAL’s current `context` block is too shallow; it needs memory layers, compaction receipts, retrieval policies, invalidation, and proof that critical facts survived compaction. ([OpenHands Docs][4])

**Subagents and observability.** OpenHands supports subagents with independent contexts and consolidated observations, and also exposes OpenTelemetry tracing for agent steps, tool calls, LLM calls, browser sessions, and conversation lifecycle. ZYAL’s `agents` and `fan_out` blocks should adopt this level of traceability and define spans, parent-child lineage, and artifact ownership. ([OpenHands Docs][5])

**Security primitives.** OpenHands has action confirmation policies, risk analyzers, and a secret registry that can detect secret references, inject only as needed, and mask outputs. ZYAL’s `permissions` and `guardrails` blocks are a good start, but they should become capability contracts with risk classifications, secret scopes, redaction policies, and per-agent tool entitlements. ([OpenHands Docs][6])

### LangGraph / LangChain

LangGraph is not mainly a coding-agent product; it is a **durable graph orchestration framework** for long-running stateful agents. Its README emphasizes durable execution, human-in-the-loop control, memory, debugging, and deployment. ([GitHub][7])

The big lesson from LangGraph: **ZYAL needs an explicit graph/state-machine layer.** The current ZYAL has `loop`, `on`, `fan_out`, and `incubator`, but those are separate mini-control systems. For ambitious projects, that will become hard to reason about. LangGraph’s durable execution model saves progress so workflows can pause and resume, and its docs explicitly discuss checkpointers, thread IDs, durability modes, and wrapping side effects in tasks. ([LangChain Docs][8])

LangGraph also has explicit interrupts that pause a graph at defined points, save state, wait for external input, then resume with a command. ZYAL should adopt this pattern as first-class `approval_gate`, `interrupt`, and `state_edit` semantics. ([LangChain Docs][9])

LangGraph’s memory model distinguishes short-term thread state from long-term cross-thread memory, and classifies memory into semantic, episodic, and procedural forms. ZYAL should copy that distinction rather than treating “context” as a generic compaction setting. ([LangChain Docs][10])

### Hermes Agent

Hermes is highly relevant because it is not just a coding tool; it is a persistent server agent with memory, skills, channels, automations, subagents, and sandbox backends. Its docs describe persistent memory, auto-generated skills, scheduled automations, isolated subagents, sandbox backends, browser/vision/image/TTS support, and machine-readable docs. ([Hermes Agent][11])

Hermes has several ideas ZYAL should absorb:

**Self-improving skills.** Hermes emphasizes a built-in learning loop that creates skills from experience, improves them during use, nudges persistence of knowledge, and searches past sessions. ([GitHub][12])

**Durable multi-agent operations.** A Hermes release describes durable multi-agent Kanban with heartbeat, reclaim, zombie detection, auto-block, retries, hallucination recovery, checkpoints, pruning, gateway auto-resume, redaction defaults, provider plugins, long-term memory, and post-write lint. ZYAL should absolutely add heartbeat, lease, zombie detection, reclaim, and automatic task parking. ([GitHub][13])

**Memory caps and memory hygiene.** Hermes uses bounded curated memory files, SQLite FTS5 session search, memory tools, capacity/consolidation, duplicate prevention, and scans for prompt injection/secrets. ZYAL needs memory write policy: every memory write should be scoped, attributed, validated, deduplicated, and revocable. ([Hermes Agent][14])

**Progressive skill loading.** Hermes skills use progressive disclosure: a lightweight index first, then deeper skill details only when needed. ZYAL should define `skills` as procedural memory, not just static prompt text. ([Hermes Agent][15])

**Layered security.** Hermes documents user authorization, dangerous command approval, container isolation, MCP credential filtering, context-file scanning, cross-session isolation, input sanitization, and hardline command blocks. ZYAL should include a baseline “dangerous capabilities” policy library rather than forcing every runbook to rediscover the same deny patterns. ([Hermes Agent][16])

### OpenClaw / OpenClaw Code Agent

OpenClaw is important because it treats agents as persistent, gateway-managed background workers. Its docs describe a long-lived Gateway, typed requests/responses/events, JSON Schema validation, idempotency keys for side-effecting methods, pairing/device signatures, and generated schemas. That maps directly to what ZYAL needs for “available inside any chat prompt” while still preventing arbitrary prompt text from arming destructive daemons. ([OpenClaw][17])

OpenClaw’s background task ledger is especially relevant: tasks have statuses such as queued, running, succeeded, failed, timed out, cancelled, and lost; completion is push-driven; tasks are persisted in SQLite; and stale/lost cleanup exists. ZYAL already has SQLite and tasks, but should add task leases, lost detection, reclaim, sticky cancellation, and completion notifications. ([OpenClaw][18])

OpenClaw also has a higher-level Task Flow substrate: durable multi-step flows with state, revision, sync semantics, managed vs mirrored modes, and cancellation that remains sticky across restarts. ZYAL’s current `tasks` block is too thin for huge work. It needs a `workflow` or `flow` layer above tasks. ([OpenClaw][19])

OpenClaw Standing Orders are also directly relevant: permanent operating authority with scope, triggers, approval gates, escalation, and an execute-verify-report pattern. ZYAL should add “standing orders” carefully, but with stronger guardrails and revocation. ([OpenClaw][20])

OpenClaw Code Agent’s plugin model already includes managed background coding sessions from chat, plan approval, worktree isolation, merge/PR follow-through, explicit goal loops, suspend/resume/fork/interrupt/recover, cost/duration tracking, and review-first defaults. ZYAL should make these capabilities portable instead of specific to one runtime. ([GitHub][21])

### Codex, Jekko, AGENTS.md, and Skills

OpenAI’s Codex CLI is a local coding agent that can read, change, and run code in the selected directory, while Codex cloud can work on background tasks in parallel using cloud environments. ([OpenAI Developers][22])

Codex also supports skills: reusable packages of instructions, resources, and optional scripts that build on the open agent-skills standard. ([OpenAI Developers][23])

Jekko itself already supports terminal, desktop, and IDE usage, multiple sessions, model-provider flexibility, and specialized agents/subagents. ([Jekko][24])

AGENTS.md is now a broadly recognized project-instruction file; the AGENTS.md site describes it as a predictable README-like place for coding agents, and Codex reads AGENTS.md before work to layer global and project-specific guidance. ([Agents][25])

So ZYAL should not compete with AGENTS.md or SKILL.md. It should **compose them**:

* `AGENTS.md`: project instructions.
* `SKILL.md`: reusable procedural capabilities.
* `ZYAL`: host-enforced lifecycle, budgets, authority, evidence, memory, workflow, and promotion policy.

### Research signals

Several research lines are directly relevant:

ReAct showed the power of interleaving reasoning and actions for tool-using agents. ([arXiv][26])

Tree of Thoughts showed that branching search over multiple reasoning paths can outperform a single linear chain on planning-heavy tasks. ([arXiv][27])

Reflexion showed that agents can improve by writing verbal feedback into memory instead of updating model weights. ([arXiv][28])

Voyager demonstrated the importance of an automatic curriculum, an ever-growing skill library, and iterative improvement from environment feedback. ([arXiv][29])

SWE-bench evaluates agents on real GitHub issues, and its task format made concrete the need for reproducible issue resolution, patch generation, and host-verifiable tests. ([GitHub][30])

A recent industrial-scale coding-agent paper argues that real-world software engineering needs agents that can operate on massive repositories, sustain long sessions, coordinate complex tools, use advanced context management, and maintain cross-session notes; the paper reports strong SWE-Bench-Pro results using a unified orchestrator, persistent notes, modular tools, and a meta-agent build-test-improve loop. ([arXiv][31])

A 2026 dataset paper on agent-authored pull requests collected 932,791 agentic PRs across 116,211 repositories, which is a strong signal that agentic coding is no longer a toy setting; the next frontier is governance, auditability, and long-horizon correctness. ([arXiv][32])

## 2. The blunt critique: what ZYAL is missing today

ZYAL is already unusually strong on host control, preview-before-arm, SQLite durability, circuit breakers, incubator passes, and promotion gates. But the current syntax is still closer to a **daemon loop config** than a full autonomous engineering operating system.

The biggest gaps:

1. **No formal workflow graph.**
   `loop`, `on`, `fan_out`, and `incubator` are powerful, but they are not a unified state machine. Huge projects need explicit stages, transitions, barriers, retries, compensation, pause points, and manual gates.

2. **No typed event model.**
   SQLite is truth, but what exactly is persisted? A portable ZYAL ecosystem needs canonical event types: `RunStarted`, `TaskClaimed`, `ToolRequested`, `ToolApproved`, `PatchCreated`, `EvidenceRecorded`, `GateBlocked`, `MemoryWritten`, `SkillPromoted`, `RunCompacted`, `BranchMerged`, and so on.

3. **No idempotency model.**
   Long-running agents retry. Retried side effects can duplicate commits, duplicate PRs, rerun migrations, spam notifications, overwrite artifacts, or corrupt state. ZYAL needs `idempotency_key`, `side_effect`, `compensate`, and `exactly_once | at_least_once | best_effort` semantics.

4. **Memory is underpowered.**
   `context` is not enough. You need semantic memory, episodic memory, procedural memory, repo memory, org memory, task memory, failure memory, and invalidation rules.

5. **Permissions are too coarse.**
   `shell: ask`, `edit: allow`, `git_commit: ask` is good for v1, but not enough for 1B+ LOC work. Permissions need to be scoped by phase, agent, file path, command class, network egress, secret class, environment, and risk level.

6. **No repo intelligence layer.**
   Large codebases require dependency graphs, symbol graphs, ownership maps, build graphs, test-impact maps, API compatibility maps, migration maps, generated-code boundaries, and service topology. A giant repo cannot be managed by grep plus vibes.

7. **No first-class evidence objects.**
   `readiness_score` and assertions are useful, but promotion should depend on typed, host-verified evidence receipts. “The model says tests pass” should never count. “Host ran test command X at commit Y with output digest Z” should count.

8. **No experiment tournament.**
   ZYAL has `fan_out` and incubator passes, but it lacks a general system for multiple disjoint hypotheses, candidate branches, scoring, cherry-picking, merging, rollback, and learning from failed attempts.

9. **No branch/worktree lifecycle model.**
   You need `created`, `dirty`, `tested`, `candidate`, `merged`, `abandoned`, `archived`, `deleted`, with lineage and artifact retention.

10. **No standing orders / scheduling / daemon fleet.**
    Long-running work often means “watch CI,” “triage nightly failures,” “keep dependency PRs green,” or “scan regressions every weekday.” ZYAL should support this, but with host-owned authority, expiration, and revocation.

11. **No governance / RBAC / approval identity.**
    Human approval should record who approved, under what role, for what exact diff or command, with expiration and non-replayability.

12. **No observability standard.**
    ZYAL needs spans, traces, metrics, token/cost accounting, queue time, wall time, tool latency, failure clusters, run dashboards, and export to OpenTelemetry.

13. **No compatibility story for other agents.**
    The whole power move is to let ZYAL run Jekko, OpenHands, Codex, Claude Code, Aider, SWE-agent, LangGraph graphs, or custom MCP backends. That means adapters and normalized events.

14. **No anti-prompt-injection model.**
    Guardrails block patterns, but ZYAL needs a deeper distinction between trusted instructions, untrusted repo text, untrusted web pages, untrusted tool output, and privileged host policy.

15. **No economic/resource scheduler.**
    Long projects need budgets for tokens, dollars, wall time, GPU/CPU, worker slots, CI minutes, cloud sandboxes, and human attention.

16. **No explicit “done means behavior preserved” model.**
    For large-scale coding, passing tests is not enough. Need compatibility checks, golden outputs, canaries, snapshot diffing, API contracts, migration safety, performance budgets, and rollback drills.

## 3. ZYAL’s core positioning: the control plane, not the agent

Here is the cleanest mental model:

| Layer                | Existing examples                                                 | ZYAL role                                                  |
| -------------------- | ----------------------------------------------------------------- | ---------------------------------------------------------- |
| Agent runtime        | Jekko, Codex, Claude Code, OpenHands, Hermes, Aider, SWE-agent | ZYAL chooses, constrains, observes, and interrupts         |
| Workflow engine      | LangGraph, Temporal-like systems, OpenClaw Task Flow              | ZYAL declares durable agent workflow policy                |
| Tool protocol        | MCP, shell, browser, GitHub, CI, DB tools                         | ZYAL grants capabilities and validates tool use            |
| Project instructions | AGENTS.md, CLAUDE.md, repo docs                                   | ZYAL imports as untrusted or semi-trusted context          |
| Skills               | SKILL.md, Hermes skills, Codex skills                             | ZYAL selects, audits, promotes, and versions skills        |
| State                | SQLite, event logs, artifacts, traces                             | ZYAL defines the canonical state/event schema              |
| Safety               | approvals, guardrails, constraints, sandboxing                    | ZYAL enforces authority, evidence, and blast-radius limits |

The key product claim should be:

> **ZYAL lets any chat message propose an autonomous software engineering daemon, but only the host can validate, arm, execute, checkpoint, promote, and stop it.**

That is very strong. It separates “prompting” from “authority.”

## 4. Proposed ZYAL v2 architecture

I would split ZYAL v2 into seven planes:

1. **Control plane** — run lifecycle, workflow graph, scheduling, approvals, pause/resume/abort.
2. **Execution plane** — agents, backends, tools, sandboxes, workspaces, shells, browser, MCP.
3. **State plane** — SQLite schema, event log, task ledger, branch ledger, artifact store.
4. **Memory plane** — context, retrieval, compaction, task memory, repo memory, skills, learning.
5. **Evidence plane** — tests, static checks, diffs, coverage, performance, security, code review, receipts.
6. **Policy plane** — permissions, guardrails, constraints, org policy, trust boundaries, secrets.
7. **Observability plane** — traces, metrics, dashboards, costs, notifications, audit trails.

The current ZYAL is mostly control + policy + a little evidence. The upgrade is to make all seven explicit.

## 5. New top-level keys ZYAL should add

### 5.1 `schema` — canonical validation, profiles, and compatibility

ZYAL should have a formal JSON Schema and ideally a CUE-style constraint layer. CUE’s core appeal is that it unifies data, schema, validation, and policy constraints, which is exactly what a daemon language needs. ([CUE][33])

Suggested syntax:

```yaml
schema:
  version: v2
  profile: autonomous_coding
  compatibility:
    accepts: ["v1"]
    rejects_unknown_keys: true
    canonicalize_before_arm: true
  validation:
    static_lints:
      - no_unbounded_loop_without_circuit_breaker
      - no_shell_allow_without_sandbox
      - no_git_push_allow_without_approval_gate
      - no_memory_write_without_scope
      - no_promotion_without_evidence
```

This lets the host reject dangerous or ambiguous ZYAL before preview.

### 5.2 `workflow` — explicit durable graph

This is the biggest missing feature.

Instead of only:

```yaml
loop:
  policy: forever
```

add:

```yaml
workflow:
  type: state_machine
  initial: discover
  states:
    discover:
      agent: plan
      writes: none
      produces: [impact_map, task_graph, risk_register]
      transitions:
        - to: plan
          when: evidence.exists("impact_map")
        - to: incubate
          when: risk.score_gte(0.7)

    plan:
      agent: plan
      writes: scratch_only
      requires:
        - impact_map
        - affected_files_known
      produces: [change_plan, test_plan, rollback_plan]
      gate:
        approval: plan_review
      transitions:
        - to: experiment
          when: approval.granted("plan_review")

    experiment:
      strategy: tournament
      transitions:
        - to: integrate
          when: candidates.best.score_gte(0.82)
        - to: incubate
          when: no_candidate_after: 3

    integrate:
      agent: build
      writes: protected_worktree
      gate:
        evidence:
          - tests.pass("unit")
          - static.pass("typecheck")
          - diff.within_scope
      transitions:
        - to: review

    review:
      reviewers:
        - agent: critic
        - human: code_owner
          required_when: risk.score_gte(0.8)
      transitions:
        - to: checkpoint
          when: review.all_required_pass
        - to: experiment
          when: review.requests_changes

    checkpoint:
      action: git_checkpoint
      transitions:
        - to: stop
          when: stop.all_true
        - to: discover
          when: remaining_tasks.exists
```

This turns ZYAL from “loop with handlers” into a durable workflow spec.

### 5.3 `backend` — run ZYAL on any agent

ZYAL should be backend-neutral.

```yaml
backends:
  default: jekko
  options:
    jekko:
      kind: jekko
      agents:
        plan: plan
        build: build
      session:
        mode: native

    openhands:
      kind: openhands
      workspace: docker
      event_adapter: openhands_event_stream
      tools:
        map:
          shell: bash
          browser: playwright
          python: ipython

    codex:
      kind: codex_cli
      mode: local
      config_profile: safe
      cloud:
        enabled: false

    claude_code:
      kind: external_cli
      command: "claude"
      event_adapter: terminal_jsonl

    langgraph:
      kind: langgraph
      graph_ref: "./agent_graph.py:create_graph"
      checkpoint_adapter: sqlite
```

This is how ZYAL becomes available inside any chat prompt. The YAML is the **authority contract**; the backend is replaceable.

### 5.4 `runtime` and `workspace` — precise sandbox semantics

Current ZYAL has `permissions`, but not enough execution detail.

```yaml
runtime:
  sandbox:
    mode: container
    image: "ghcr.io/acme/agent-node-bun:2026-05"
    network:
      egress: deny_by_default
      allow:
        - registry.npmjs.org
        - api.github.com
    mounts:
      - source: repo
        target: /workspace
        mode: read_write
      - source: host_cache
        target: /cache
        mode: read_only
    secrets:
      exposure: on_demand
      mask_outputs: true
      deny_to_agents: [critic, scout]
    resources:
      cpu: 8
      memory: 24GiB
      disk: 200GiB
      max_processes: 512

workspace:
  strategy: worktree_pool
  base_ref: origin/main
  isolation:
    default: git_worktree
    prototype: ephemeral_worktree
    reviewer: read_only_snapshot
  cleanup:
    on_success: archive_summary_delete
    on_failure: preserve_dirty_worktrees
  locks:
    - path: "packages/payments/**"
      mode: exclusive
    - path: "docs/**"
      mode: shared
```

This lets ZYAL describe blast radius, not just permission toggles.

### 5.5 `repo_intel` — make 1B+ LOC possible

For billion-line or giant monorepo work, the agent cannot “read the repo.” It needs indexed, sharded intelligence.

```yaml
repo_intel:
  indexes:
    - id: symbols
      kind: symbol_graph
      provider: lsp
      refresh: on_start_and_changed_files
    - id: deps
      kind: dependency_graph
      command: "bazel query 'deps(//...)'"
      cache: sqlite
    - id: ownership
      kind: codeowners
      sources: ["CODEOWNERS", ".github/teams.yml"]
    - id: tests
      kind: test_impact
      command: "tools/impacted-tests --json"
    - id: api_contracts
      kind: contract_graph
      sources: ["openapi/**/*.yaml", "proto/**/*.proto"]
    - id: generated
      kind: generated_code_map
      deny_editing: true

  retrieval:
    default_budget_tokens: 18000
    rank_by:
      - changed_symbols
      - dependency_distance
      - recent_failures
      - ownership
      - test_coverage
    require_citations: true

  sharding:
    strategy: ownership_then_dependency
    max_files_per_task: 80
    max_dependency_radius: 2
    forbid_cross_shard_writes_without_gate: true
```

This is one of the most important upgrades. Without repo intelligence, ZYAL will produce “busy autonomous agents.” With repo intelligence, it can produce **bounded, evidence-driven engineering work**.

### 5.6 `memory` — layered, governed, invalidatable memory

ZYAL should explicitly separate memory types.

```yaml
memory:
  store: sqlite
  mirror: true

  layers:
    run:
      scope: current_run
      ttl: until_stop
      writable_by: [supervisor]
      max_tokens: 12000

    task:
      scope: task
      ttl: 180d
      writable_by: [supervisor, critic]
      compaction: evidence_preserving

    repo:
      scope: repository
      ttl: 365d
      writable_by: [supervisor]
      write_requires:
        - evidence.receipt
        - human_approval: maintainer
      invalidation:
        - when_files_change: ["AGENTS.md", "package.json", "build/**"]

    org:
      scope: organization
      writable_by: []
      import_only: true

    episodic:
      description: prior attempts, failures, reviews, and lessons
      retrieval:
        query: objective + failing_tests + touched_symbols
        max_items: 8

    procedural:
      skills:
        format: SKILL.md
        sources:
          - ".jekko/skills"
          - ".agent/skills"
        activation: progressive_disclosure
        promotion_requires:
          - verifier_passed
          - no_secret_leak
          - human_approval: maintainer

  compaction:
    strategy: hierarchical
    preserve:
      - objective
      - constraints
      - open_risks
      - evidence_receipts
      - unresolved_objections
      - changed_files
      - rollback_plan
    verify:
      - name: no-critical-loss
        command: "jekko memory diff-critical --before $BEFORE --after $AFTER"
        assert: { exit_code: 0 }

  writes:
    require_schema: true
    scan_for_secrets: true
    scan_for_prompt_injection: true
    deduplicate: semantic
```

This directly addresses the “vibe coding” problem. Memory becomes audited engineering state, not a vibes journal.

### 5.7 `skills` — integrate with SKILL.md without surrendering control

```yaml
skills:
  sources:
    - path: ".jekko/skills"
      trust: repo
    - path: "~/.config/jekko/skills"
      trust: user
    - registry: "skills.sh"
      trust: remote
      install: ask

  activation:
    mode: host_selected
    progressive_disclosure: true
    max_active: 5

  audit:
    before_use:
      - scan_for_secrets
      - scan_for_prompt_injection
      - validate_frontmatter
    after_use:
      - record_effectiveness
      - compare_to_no_skill_baseline

  creation:
    enabled: true
    from:
      - repeated_successful_task
      - repeated_failure_fixed_by_pattern
      - user_correction
    writes: scratch_only
    promote_requires:
      - deterministic_verifier
      - human_approval: maintainer
```

Skills are essential, but they are dangerous if treated as trusted prompt text. ZYAL should treat skills as **versioned procedural memory with evidence-backed promotion**.

### 5.8 `evidence` — host-verified receipts

This should be central.

```yaml
evidence:
  receipts:
    - id: unit_tests
      kind: command
      command: "bun test --timeout 30000"
      timeout: 60s
      assert: { exit_code: 0 }
      records:
        - stdout_digest
        - stderr_digest
        - exit_code
        - started_at
        - finished_at
        - commit_sha
        - environment_digest

    - id: typecheck
      kind: command
      command: "npx tsgo --noEmit"
      assert: { exit_code: 0 }

    - id: affected_tests
      kind: dynamic
      from_index: tests
      command: "tools/run-impacted-tests --from $BASE --to HEAD --json"
      assert:
        json:
          "$.failed": 0

    - id: diff_scope
      kind: diff
      assert:
        changed_paths_within:
          - "packages/auth/**"
          - "packages/common/**"
        max_files_changed: 40
        max_lines_changed: 2500

    - id: rollback_known
      kind: structured_output
      schema:
        type: object
        required: [rollback_command, risk]
```

Then gates can refer to evidence:

```yaml
promotion:
  require_evidence:
    - unit_tests
    - typecheck
    - affected_tests
    - diff_scope
    - rollback_known
  block_on:
    - unresolved_critical_objections
    - secret_scan_failed
    - generated_code_modified
```

This moves ZYAL from “agent confidence” to **auditable proof**.

### 5.9 `experiments` — disjoint candidate branches and tournament selection

To reduce jank, ZYAL should encourage multiple independent attempts.

```yaml
experiments:
  enabled: true
  strategy: disjoint_tournament
  max_candidates: 5
  max_parallel: 3
  base: clean_snapshot

  candidates:
    - id: minimal_patch
      agent: build
      prompt_strategy: smallest_safe_change
      writes: isolated_worktree
    - id: refactor_first
      agent: build
      prompt_strategy: improve_structure_then_fix
      writes: isolated_worktree
    - id: test_first
      agent: build
      prompt_strategy: reproduce_with_test_then_fix
      writes: isolated_worktree
    - id: dependency_hypothesis
      agent: build
      prompt_strategy: inspect_dependency_boundary
      writes: isolated_worktree

  scoring:
    method: weighted
    metrics:
      tests_pass: 0.30
```

## 6. Anti-jank / anti-vibe-coding upgrades

“Vibe coding” fails when the model optimizes for plausible edits instead of verified engineering outcomes. ZYAL can crush this by making “done” a host-verifiable contract.

### 6.1 Require read-only reconnaissance before writes

```yaml
phase_policy:
  before_first_edit:
    require:
      - repo_intel.impact_map
      - failing_test_reproduced
      - affected_files_known
      - change_plan
      - rollback_plan
    forbid:
      - edit
      - git_commit
```

The first phase should be `scout` or `plan`, with no write permission.

### 6.2 Require a hypothesis ledger

```yaml
hypotheses:
  enabled: true
  require_before_edit: true
  fields:
    - suspected_root_cause
    - evidence_for
    - evidence_against
    - proposed_change
    - test_that_will_fail_before_fix
    - test_that_will_pass_after_fix
  on_failed_hypothesis:
    action: record_and_try_next
```

The agent should not just edit. It should state what it believes and how the host will test it.

### 6.3 Require negative-space analysis

```yaml
review:
  require_negative_space:
    questions:
      - "What nearby behavior could this break?"
      - "What contracts did not have tests?"
      - "What generated files or migrations are implicated?"
      - "What dependency owners should review this?"
      - "What rollback would fail?"
```

This is a practical way to force the critic pass to find what normal implementation misses.

### 6.4 Make “done” a bundle, not a sentence

```yaml
done_contract:
  require:
    - evidence.receipt("unit_tests")
    - evidence.receipt("typecheck")
    - evidence.receipt("affected_tests")
    - evidence.receipt("diff_scope")
    - artifact("change_plan")
    - artifact("rollback_plan")
    - artifact("risk_register")
    - artifact("review_summary")
  forbid:
    - model_only_completion_signal
```

### 6.5 Track behavioral deltas

```yaml
behavior_regression:
  snapshots:
    before:
      command: "tools/capture-golden --output /tmp/before.json"
    after:
      command: "tools/capture-golden --output /tmp/after.json"
    compare:
      command: "tools/compare-golden /tmp/before.json /tmp/after.json"
      allow:
        - "expected_change:auth-token-refresh"
```

For giant projects, test pass/fail is not enough. You need behavior comparison.

## 7. Better multi-pass intelligence

Your incubator is one of ZYAL’s best features. I would generalize it into **pass orchestration** with explicit pass contracts.

Current pass types are good:

* `scout`
* `idea`
* `strengthen`
* `critic`
* `synthesize`
* `prototype`
* `promotion_review`
* `compress`

Add these:

* `reproduce` — prove the bug/failure exists.
* `impact_map` — identify dependencies, owners, tests, contracts.
* `hypothesis` — produce competing root-cause theories.
* `test_design` — design tests before implementation.
* `minimal_patch` — smallest safe change.
* `refactor_patch` — larger structural improvement.
* `compatibility_review` — API/backward compatibility.
* `security_review` — injection, auth, secrets, supply chain.
* `performance_review` — benchmark and complexity risks.
* `migration_review` — DB/data rollout/rollback.
* `merge_synthesis` — combine best candidate patches.
* `postmortem` — summarize lessons and candidate skill updates.

Example:

```yaml
passes:
  - id: reproduce
    type: reproduce
    context: blind
    writes: scratch_only
    must_produce:
      - failing_command
      - failure_digest

  - id: impact
    type: impact_map
    context: repo_indexed
    writes: scratch_only
    must_produce:
      - affected_symbols
      - affected_tests
      - owners
      - dependency_radius

  - id: hypotheses
    type: hypothesis
    count: 5
    context: inherit
    writes: scratch_only
    diversity:
      require_disjoint_root_causes: true

  - id: test_design
    type: test_design
    writes: isolated_worktree
    must_produce:
      - failing_test_before_fix

  - id: prototypes
    type: prototype
    count: 4
    parallel: true
    writes: isolated_worktree
    each_from: hypotheses

  - id: critic
    type: critic
    context: pool
    writes: scratch_only
    reviewers:
      - correctness
      - maintainability
      - security
      - performance

  - id: synthesize
    type: merge_synthesis
    context: pool
    writes: isolated_worktree
    require_clean_replay: true

  - id: promotion
    type: promotion_review
    context: promotion
    writes: scratch_only
    require_evidence:
      - unit_tests
      - affected_tests
      - rollback_known
```

This gives ZYAL an explicit “research → experiment → synthesize → prove” structure.

## 8. Handling overlap and gaps with each ecosystem

### ZYAL vs OpenHands

**Overlap:** autonomous coding runtime, event stream, tools, sandboxing, subagents, persistence.

**Gap in ZYAL:** no explicit runtime backend, no typed event stream, no built-in sandbox profile, no action risk analyzer, no secret registry, no OpenTelemetry spec.

**ZYAL advantage:** OpenHands is a runtime; ZYAL can be the host-enforced run contract that can drive OpenHands or any other runtime.

**Upgrade:** add `backend: openhands`, `runtime`, `events`, `observability`, `secrets`, and `action_confirmation`.

### ZYAL vs LangGraph

**Overlap:** durable long-running control, human gates, retries, state.

**Gap in ZYAL:** no graph, no deterministic replay semantics, no task-level idempotency, no interrupt/resume state editing.

**ZYAL advantage:** LangGraph is code-first; ZYAL can be prompt-native, declarative, host-validated, and portable across agent runtimes.

**Upgrade:** add `workflow`, `interrupts`, `idempotency`, `durability`, and `compensations`.

### ZYAL vs Hermes

**Overlap:** long-running daemon, memory, skills, subagents, sandboxes, scheduled automations.

**Gap in ZYAL:** no bounded curated memory, no skill creation/promotion loop, no heartbeat/reclaim/zombie detection, no channel/gateway model.

**ZYAL advantage:** Hermes is a persistent assistant platform; ZYAL can define the policy contract for any such platform.

**Upgrade:** add `memory.layers`, `skills`, `standing_orders`, `recovery`, and `notifications`.

### ZYAL vs OpenClaw

**Overlap:** persistent background agent work, task ledger, gateway, worktree isolation, plan approval, recover/fork/resume.

**Gap in ZYAL:** no gateway protocol, no idempotency keys, no task flow above tasks, no standing orders, no explicit channel security.

**ZYAL advantage:** OpenClaw is an ecosystem/runtime; ZYAL can be the portable daemon language embedded in any chat.

**Upgrade:** add `gateway`, `task_flow`, `standing_orders`, `idempotency`, and `approval_identity`.

### ZYAL vs Codex / Claude Code / Jekko

**Overlap:** code editing agents, worktrees, skills, AGENTS.md, model-specific execution.

**Gap in ZYAL:** no adapters, no normalized event model, no skill/AGENTS.md import policy.

**ZYAL advantage:** ZYAL can be the vendor-neutral way to specify long-running authority.

**Upgrade:** add `backends`, `skills`, `instructions`, and `capability_mapping`.

### ZYAL vs AutoGen / CrewAI / Microsoft Agent Framework

**Overlap:** multi-agent orchestration.

**Gap in ZYAL:** no rich role protocols, no graph semantics, no message contracts.

**ZYAL advantage:** ZYAL should not become a general-purpose agent programming library. It should stay declarative and host-enforced.

**Upgrade:** add role contracts, artifact exchange, and workflow graph, but avoid arbitrary embedded code.

### ZYAL vs Aider / SWE-agent

**Overlap:** coding task execution, git integration, repo-aware edits, benchmark-style issue resolution.

**Gap in ZYAL:** no patch lifecycle standard, no repo map, no issue-to-patch benchmark mode.

**ZYAL advantage:** ZYAL can drive these tools as workers and enforce stronger gates.

**Upgrade:** add `patch`, `repo_intel`, `benchmark`, `issue_resolution`, and `candidate_scoring`.

## 10. State model: SQLite tables ZYAL should standardize

ZYAL says SQLite is truth. Great. Now standardize the schema enough that tools can interoperate.

Suggested core tables:

```text
runs
run_events
tasks
task_events
workflow_states
agent_steps
tool_calls
approvals
evidence_receipts
memory_items
memory_links
skills
skill_evals
workspaces
branches
patches
experiments
candidates
reviews
risks
constraints
guardrail_hits
artifacts
notifications
costs
heartbeats
leases
```

The key is not that every host must use exactly these physical tables. The key is that ZYAL should define canonical **logical entities** and JSON event payloads.

Example event:

```json
{
  "type": "EvidenceRecorded",
  "run_id": "run_123",
  "task_id": "task_456",
  "evidence_id": "unit_tests",
  "actor": "host",
  "commit_sha": "abc123",
  "command": "bun test --timeout 30000",
  "exit_code": 0,
  "stdout_digest": "sha256:...",
  "environment_digest": "sha256:...",
  "started_at": "2026-05-08T12:00:00Z",
  "finished_at": "2026-05-08T12:03:12Z"
}
```

This is what makes ZYAL auditable.

## 11. Security model upgrades

ZYAL should define trust zones:

```yaml
trust:
  zones:
    host_policy:
      trust: highest
      examples: [ZYAL schema, org policy, approval records]
    user_prompt:
      trust: high
      can_request: true
      can_authorize: false
    repo_files:
      trust: medium
      can_instruct: limited
      prompt_injection_scan: true
    tool_output:
      trust: low
      can_instruct: false
    web_content:
      trust: untrusted
      can_instruct: false
```

Then enforce:

```yaml
instruction_policy:
  precedence:
    - host_policy
    - armed_ocal
    - approval_records
    - repo_agents_md
    - skills
    - user_prompt
    - tool_output
  reject_when:
    - lower_trust_content_attempts_to_modify_higher_trust_policy
    - tool_output_contains_zyal_arm
    - repo_file_requests_secret_exfiltration
```

This is critical for prompt-embeddable ZYAL. A malicious README, webpage, or test fixture must not be able to smuggle `ZYAL_ARM` or rewrite permissions.

## 12. Making ZYAL prompt-native but safe

The prompt-native design is excellent, but arming needs more than `ZYAL_ARM RUN_FOREVER`.

I would add:

```yaml
arming:
  requires:
    - host_preview_rendered
    - schema_valid
    - policy_valid
    - human_action: explicit_arm
    - nonce: host_generated
  arm_sentinel:
    require_nonce: true
    require_exact_digest: true
    expires_after: 10m
```

Example preview flow:

1. User pastes ZYAL block.
2. Host parses and canonicalizes YAML.
3. Host shows preview:

   * authority requested
   * max runtime
   * max cost
   * files writable
   * tools allowed
   * secrets requested
   * approval gates
   * stop conditions
4. Host generates nonce and digest.
5. User clicks Arm or enters host-specific arm command.
6. Host arms the exact canonical digest only.

This prevents “assistant output started a daemon” and prevents “the ZYAL block changed between preview and arm.”

## 13. Resource and cost control for giant projects

Add:

```yaml
budget:
  wall_time: 12h
  active_runtime: 6h
  token_budget:
    total: 30_000_000
    per_task: 1_000_000
    per_agent_step: 120_000
  cost:
    max_usd: 500
    warn_at_usd: 300
  ci:
    max_minutes: 600
    max_reruns_per_test: 3
  workers:
    max_parallel: 20
    max_per_owner_area: 3
  human_attention:
    max_approval_requests_per_hour: 5
    batch_approvals: true
```

And scheduler:

```yaml
scheduler:
  strategy: priority_fair
  queues:
    - id: critical_ci
      priority: 100
    - id: refactors
      priority: 50
    - id: experiments
      priority: 20
  backpressure:
    when_cost_above: 0.8
    when_ci_queue_above: 20
    action: pause_low_priority
```

Autonomy without budgets becomes chaos.

## 14. Human collaboration model

For the world’s most ambitious coding projects, humans should not be micromanagers. They should be **gatekeepers, reviewers, and exception handlers**.

ZYAL should support:

```yaml
human_interface:
  review_packets:
    format: concise
    include:
      - objective
      - current_state
      - proposed_decision
      - evidence
      - risks
      - alternatives
      - recommended_action
    max_words: 600

  approval_batching:
    enabled: true
    group_by:
      - task
      - risk_class
      - code_owner

  escalation:
    - when: blocked_for: 2h
      notify: code_owner
    - when: security_risk_detected
      notify: security
    - when: budget_80_percent
      notify: operator
```

A good daemon should ask fewer, better questions.

## 15. Benchmark and eval mode

ZYAL should include a benchmark mode so people can compare agents and configurations.

```yaml
eval:
  mode: benchmark
  dataset:
    kind: swe_bench
    split: verified
  matrix:
    backends: [jekko, openhands, codex]
    models: [gpt-5.5, claude-sonnet, local-model]
    strategies:
      - single_pass
      - incubator
      - tournament
  metrics:
    - resolved
    - tests_pass
    - patch_size
    - cost_usd
    - wall_time
    - human_interventions
    - regression_rate
  output:
    leaderboard: ".jekko/evals/leaderboard.json"
```

This matters because skills, policies, and incubator passes can make agents better or worse. You need measurement, not anecdotes.

## 16. What ZYAL should avoid

ZYAL should **not** become:

1. A general programming language.
2. A hidden prompt system.
3. A replacement for AGENTS.md.
4. A replacement for SKILL.md.
5. A replacement for MCP.
6. A replacement for OpenHands/LangGraph/Hermes/OpenClaw.
7. A model-owned autonomy loop.
8. A “swarm launcher.”
9. A CI system clone.
10. A place where arbitrary shell scripts silently become policy.

The winning design is declarative, typed, inspectable, host-owned, and backend-neutral.

## 17. Recommended roadmap

### P0: must-have for a credible daemon language

1. Canonical schema and semantic validator.
2. Typed event model.
3. Explicit workflow graph/state machine.
4. Evidence receipts.
5. Runtime/workspace/sandbox model.
6. Capability-scoped permissions.
7. Approval identity and digest binding.
8. Heartbeat, leases, lost-task detection.
9. Backend adapter interface.
10. Prompt-native arming with nonce and canonical digest.

### P1: makes ZYAL dramatically better than normal agent configs

1. Layered memory.
2. Skill import/audit/promotion.
3. Repo intelligence indexes.
4. Experiment tournaments.
5. Branch/worktree lifecycle.
6. OpenTelemetry-style observability.
7. Cost and resource scheduler.
8. Risk scoring.
9. PR/merge/deploy/rollback lifecycle.
10. Trust-zone and prompt-injection policy.

### P2: makes ZYAL world-class

1. Standing orders.
2. Meta-agent config optimization.
3. Benchmark/eval mode.
4. Cross-repo dependency orchestration.
5. Multi-org policy packs.
6. Automated skill synthesis from postmortems.
7. Failure clustering and root-cause memory.
8. Agent marketplace compatibility.
9. Formal replay/simulation mode.
10. “Autonomous engineering SLOs” for reliability, cost, regression rate, and human-intervention rate.

## 18. The north-star ZYAL tagline

I would describe ZYAL as:

> **ZYAL is the open, host-enforced control language for autonomous software engineering. It lets any chat propose long-running agent work, but only the host can arm it, constrain it, observe it, verify it, checkpoint it, promote it, or stop it.**

And the strongest technical claim:

> **ZYAL turns autonomous coding from prompt-driven behavior into evidence-driven, durable, auditable engineering workflow.**

That is the path to making ZYAL the default language for ambitious coding projects.
