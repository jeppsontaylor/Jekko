Here’s the core conclusion: **ZYAL is not redundant — but only if it becomes the portable host-control, evidence, memory, and orchestration layer above existing agent runtimes.** It should not try to replace OpenHands, LangGraph, Hermes, OpenClaw, Codex, MCP, ACP, or LangChain. It should compile to them, constrain them, observe them, and govern them.

The strongest version of ZYAL is not “YAML for starting a coding agent.” It is a **declarative operating contract for long-running autonomous software work**: what may happen, what must be proven, what memory survives, what gets promoted, what gets quarantined, what gets retried, what gets forked, what gets reviewed, and when humans must intervene.

I studied the current docs and research sources for the named systems and adjacent long-running-agent work. I prioritized primary docs, papers, and project repos; social/forum material is useful for pain points, but I would not treat it as authoritative design evidence. The biggest signal from the ecosystem is consistent: long-running agents need **durable state, precise tool interfaces, sandboxing, interrupts, memory, subagents, hooks, skills, replay, and hard evidence gates**. LangGraph emphasizes durable execution, persistence, checkpoints, interrupts, and human-in-the-loop control; OpenHands emphasizes sandboxing, repo customization, hooks, and stop-time verification; Hermes emphasizes persistent memory, skills, scheduled automations, subagents, self-improvement, and multiple execution backends; OpenClaw emphasizes gateway architecture, channel trust, routing, scheduling, and multi-agent identity; Codex emphasizes subagents, skills, repository instructions, compaction, and long-horizon steering; MCP and ACP are protocol surfaces ZYAL should integrate with rather than duplicate. ([LangChain Docs][1])

The research direction also supports ZYAL’s thesis: current agents still struggle with coherent long-horizon work, repeated decision loops, and extended projects. METR’s time-horizon work frames agent difficulty by the length of human tasks agents can reliably complete, and their public results emphasize that long, nontrivially decomposable work remains hard. Vending-Bench shows that even apparently simple long-running business loops can derail into “meltdown” behaviors across millions of tokens, while SWE-EVO highlights that multi-step software evolution is much harder than single-issue repair benchmarks. ([METR][2])

## 1. The high-level verdict

ZYAL v1 already has an unusually strong foundation:

* **Host owns the loop.** This is the most important idea.
* **Preview before arming.** This is essential for prompt-embedded portability.
* **SQLite as truth.** Correct. The mirror should remain generated output.
* **Circuit breakers and budgets.** Correct.
* **Guardrails, constraints, permissions.** Correct.
* **Incubator passes.** This is the most promising differentiator.

But ZYAL v1 is still too much like a daemon runbook. For world-scale autonomous coding, it needs to become a **typed control plane** with first-class concepts for:

1. capability negotiation across hosts;
2. workflow graphs, not only loops;
3. memory lifecycle;
4. codebase indexing;
5. evidence packages;
6. replay and forkable timelines;
7. experiments and ablations;
8. trust boundaries;
9. sandbox resources;
10. skill acquisition;
11. human approval protocols;
12. cost, latency, and model-routing policies;
13. multi-repo and 1B+ LOC sharding;
14. agent identity and delegation contracts;
15. structured “anti-vibe-coding” proof obligations.

The most important architectural rule is this:

> **ZYAL should own policy, lifecycle, memory contracts, evidence, and promotion. Existing systems should own execution details.**

That means ZYAL should be able to run inside Jekko natively, compile into LangGraph, emit OpenHands hooks, invoke Codex subagents, use MCP tools, expose ACP events to editors, borrow Hermes-style memory/skills/scheduling, and adopt OpenClaw-style trust boundaries and gateways.

([Model Context Protocol][8])                                                                                                                                                  | ZYAL `mcp` profiles.                                                     | Keep MCP as the tool/resource protocol. ZYAL should specify **which MCP capabilities are allowed when**, not create a competing tool protocol.                 |
| **ACP**                            | Standard editor/agent protocol, analogous to LSP for coding agents. ACP exists to reduce lock-in between editors and agents. ([Agent Client Protocol][9])                                                                                                                                                                 | ZYAL API surface and UI events.                                          | Expose ZYAL daemon events over ACP so editors can inspect, pause, fork, approve, and replay runs.                                                              |

The conclusion is sharp: **ZYAL is redundant if it becomes another agent framework. ZYAL is valuable if it becomes the portable governance language for agent frameworks.**

## 3. The biggest missing idea: ZYAL needs a compiler model

Right now ZYAL is a syntax. To become universal, it needs to be a **compiled contract**.

An ZYAL host should parse, validate, lint, preview, lower, execute, observe, and record. Different runtimes will support different capabilities. So ZYAL needs explicit capability negotiation.

### Proposed addition: `capabilities`, `adapters`, and `lowering`

```yaml
capabilities:
  required:
    - sqlite_ledger
    - host_stop_evaluation
    - approval_gates
    - sandbox
    - git_worktree_isolation
    - structured_output_validation
  optional:
    - langgraph_checkpointing
    - codex_subagents
    - openhands_hooks
    - hermes_memory
    - mcp_tools
    - acp_events
    - remote_workers
  on_missing:
    required: reject
    optional: degrade_with_warning

adapters:
  jekko:
    mode: native
  codex:
    map_agents_to: codex_subagents
    map_skills_to: codex_skills
    map_context_to: compaction_profile
  openhands:
    emit:
      - ".openhands/hooks.json"
      - ".openhands/setup.sh"
      - ".openhands/finish_checks.sh"
  langgraph:
    compile_to: state_graph
    checkpointer: sqlite
    interrupts: true
  hermes:
    use:
      - persistent_memory
      - skills
      - scheduled_automations
    skill_promotion: host_reviewed
  acp:
    expose:
      - daemon_events
      - approvals
      - diffs
      - checkpoints
      - task_state
```

This makes ZYAL portable without pretending every host can do everything. A Claude chat, Codex session, OpenHands workspace, Jekko daemon, or LangGraph app can all parse the same block, then either execute natively or lower to supported primitives.

### Required compiler phases

ZYAL should define these phases explicitly:

```yaml
compiler:
  parse:
    reject_code_fences: true
    require_matching_sentinels: true
  validate:
    schema: zyal/v2
    unknown_keys: reject
    deprecated_keys: warn
  lint:
    - name: unbounded-loop-without-circuit-breaker
      severity: error
    - name: shell-allow-without-guardrails
      severity: warning
    - name: push-allow-without-approval
      severity: error
    - name: incubator-without-budget
      severity: error
  preview:
    show:
      - permissions
      - write_scopes
      - network_access
      - secrets_access
      - estimated_cost
      - approval_points
      - generated_hooks
      - unsupported_capabilities
  lower:
    targets:
      - native
      - langgraph
      - openhands
      - codex
      - mcp
      - acp
```

This is how ZYAL becomes safe inside **any chat prompt**: the assistant can output the block, but only the host compiler can arm it.

## 4. ZYAL needs workflow graphs, not only loops

The current `loop` plus `on` handlers are good for simple daemons. But ambitious coding projects are not just “repeat until stop.” They are **state machines**:

* discover work;
* classify risk;
* shard;
* assign;
* build;
* test;
* review;
* incubate;
* merge;
* roll out;
* monitor;
* revert if necessary.

LangGraph’s popularity is partly because it gives explicit graph/state execution with persistence and interrupts, not just a loop. ZYAL should absorb that lesson while staying declarative.

### Proposed addition: `workflow`

```yaml
workflow:
  start: discover

  states:
    discover:
      do:
        - run: "zyal task discover"
        - classify_tasks: true
      next:
        - when: "tasks.high_risk.count > 0"
          goto: incubate
        - when: "tasks.ready.count > 0"
          goto: implement
        - otherwise: sleep

    sleep:
      do:
        - wait: 10s
      next: discover

    incubate:
      do:
        - incubate:
            queue: high_risk
            max_parallel: 3
      next:
        - when: "incubator.ready.count > 0"
          goto: implement
        - when: "incubator.exhausted.count > 0"
          goto: human_review
        - otherwise: discover

    implement:
      do:
        - assign:
            queue: ready
            to: workers.builder
            isolation: git_worktree
        - wait_for_workers:
            timeout: 45m
      next: verify

    verify:
      do:
        - run_gate: unit_tests
        - run_gate: typecheck
        - run_gate: lint
        - run_gate: affected_integration_tests
      next:
        - when: "gates.all_passed"
          goto: checkpoint
        - when: "gates.failed.count > 0 && attempts.current < 3"
          goto: repair
        - otherwise: incubate

    repair:
      do:
        - assign:
            to: workers.repair
            context: failing_evidence
      next: verify

    checkpoint:
      do:
        - create_evidence_package: true
        - checkpoint: true
      next:
        - when: "checkpoint.created"
          goto: promote
        - otherwise: human_review

    promote:
      do:
        - approval:
            gate: promote_to_main
            require: maintainer
      next:
        - when: "approval.granted"
          goto: merge
        - otherwise: pause

    merge:
      do:
        - merge_worktree: true
        - run_gate: post_merge_tests
      next:
        - when: "gates.all_passed"
          goto: discover
        - otherwise: rollback

    rollback:
      do:
        - rollback_last_merge: true
        - notify: "Rollback performed after post-merge failure"
      next: human_review

    human_review:
      do:
        - pause: true
        - summarize_run: true
```

This solves a major weakness in v1: `on` handlers are reactive, but large projects need **planned progression**. A state graph also makes replay, fork, and audit easier.

## 5. ZYAL needs first-class evidence, not just stop conditions

The current `stop` and `checkpoint.verify` fields are necessary but not sufficient. For ambitious coding, “tests pass” is often a weak signal. The daemon must prove:

* what changed;
* why it changed;
* which assumptions were tested;
* which tests were relevant;
* what failed before;
* what passed after;
* what was not tested;
* what risks remain;
* what rollback exists;
* who or what approved promotion.

### Proposed addition: `evidence`

```yaml
evidence:
  package:
    format: zyal.evidence/v1
    storage:
      sqlite: true
      mirror: ".jekko/daemon/{{run_id}}/evidence/{{task_id}}"
    include:
      - objective
      - task_capsule
      - changed_files
      - diff_summary
      - dependency_impact
      - commands_run
      - test_results
      - failing_before
      - passing_after
      - coverage_delta
      - benchmark_delta
      - lint_results
      - typecheck_results
      - security_scan_results
      - migration_plan
      - rollback_plan
      - unresolved_questions
      - model_claims
      - host_verified_claims

  claims:
    require:
      - id: root_cause
        type: text
        verifier: reviewer_or_test
      - id: affected_files_known
        type: boolean
        verifier: host
      - id: rollback_known
        type: boolean
        verifier: host
      - id: tests_cover_change
        type: boolean
        verifier: host
      - id: no_skipped_tests
        type: boolean
        verifier: shell

  promotion:
    require_verified_claims:
      - affected_files_known
      - rollback_known
      - tests_cover_change
      - no_skipped_tests
    block_if:
      - unresolved_critical_objections_gte: 1
      - unverified_model_claims_gte: 3
      - test_regressions_gte: 1
```

This is one of the best ways to reduce “vibe coding” or “jankurai.” The model should not be rewarded for sounding confident. It should be rewarded for producing artifacts the host can check.

### Anti-jankurai rule

ZYAL should make this enforceable:

> No non-trivial patch may be promoted unless it includes a bounded hypothesis, changed-file set, verifier set, failure-before/pass-after evidence, blast-radius estimate, and rollback plan.

```yaml
anti_jankurai:
  enabled: true
  require_for:
    - production_code_change
    - migration
    - auth_change
    - security_change
    - dependency_change
    - generated_code_change

  proof_obligations:
    - hypothesis
    - changed_files
    - intended_behavior
    - non_goals
    - tests_added_or_identified
    - failing_signal_before
    - passing_signal_after
    - blast_radius
    - rollback_plan
    - remaining_risks

  block_promotion_if:
    - "proof.hypothesis == null"
    - "proof.tests_added_or_identified.count == 0"
    - "proof.rollback_plan == null"
    - "proof.blast_radius == 'unknown'"
```

This turns “agent said it fixed it” into “host has a proof package.”

## 6. Memory is the largest missing subsystem

ZYAL v1 has `context` and incubator scratch state, but not a full memory model. For long-running autonomous coding, memory must be typed, scoped, budgeted, provenance-tracked, and security-scanned.

Hermes is especially relevant here. Its memory model separates curated persistent memory from session search; memory is bounded, stored in files, injected at session start, and scanned for risky content like prompt injection or secret exfiltration. That is exactly the kind of operational discipline ZYAL should adopt, but with stronger host gating for coding work. ([Hermes Agent][12])

LangChain/Deep Agents also frame long-running agents around context engineering, startup context, runtime context, compression, subagent isolation, long-term memory, and skills/progressive disclosure. ([LangChain Docs][13])

### Proposed addition: `memory`

```yaml
memory:
  truth:
    sqlite: ".jekko/daemon/state.db"
    mirror: ".jekko/daemon/{{run_id}}"

  stores:
    working:
      description: "Short-term iteration memory."
      scope: iteration
      max_tokens: 12000
      eviction: summarize_then_drop

    run:
      description: "Facts and decisions for this daemon run."
      scope: run
      max_tokens: 24000
      write_policy: host_verified_or_model_with_provenance
      include:
        - objectives
        - decisions
        - failed_attempts
        - successful_patterns
        - blocked_paths
        - open_questions

    task:
      description: "Per-task capsule used across passes."
      scope: task
      max_tokens: 8000
      write_policy: host_reviewed_for_promotion
      required_fields:
        - objective
        - current_status
        - attempts
        - known_failures
        - relevant_files
        - tests
        - risks
        - rollback

    repo:
      description: "Repository instructions and architecture memory."
      scope: repository
      sources:
        - "AGENTS.md"
        - "CLAUDE.md"
        - ".jekko/skills/**/SKILL.md"
        - ".codex/skills/**/SKILL.md"
        - "docs/architecture/**"
        - "CODEOWNERS"
      load: progressive
      precedence:
        - nearest_file_instruction
        - repo_root_instruction
        - organization_instruction

    episodic:
      description: "Searchable history of prior sessions/runs."
      sqlite_fts5: true
      summarize_on:
        - session_end
        - checkpoint
        - promotion
      query_policy:
        require_reason: true
        max_results: 8

    semantic_code:
      description: "Code symbol and dependency index."
      index:
        parser: tree_sitter
        chunk_by: symbol
        include_call_graph: true
        include_import_graph: true
        include_test_mapping: true
      embeddings:
        enabled: optional
        provider: host_configured

    procedural:
      description: "Reusable skills and workflows."
      skill_dirs:
        - ".jekko/skills"
        - ".codex/skills"
        - ".agents/skills"
      invocation: explicit_or_matched
      promotion_policy: evidence_reviewed

  compaction:
    triggers:
      - tokens_gte: 0.78
      - iteration_gte: 8
      - before_hard_clear
      - before_worker_handoff
    preserve:
      - objective
      - current_plan
      - decisions
      - failed_attempts
      - evidence
      - active_constraints
      - pending_approvals
      - risk_register
    output_schema:
      type: object
      required:
        - objective
        - state
        - decisions
        - failed_attempts
        - next_actions
        - evidence_refs
        - risks

  security:
    scan_entries_for:
      - prompt_injection
      - secrets
      - credential_exfiltration
      - invisible_unicode
      - tool_override_attempts
      - instruction_hijacking
    taint_untrusted_sources: true
    require_provenance: true
    max_memory_age: 90d
    allow_model_to_delete_memory: false
```

### Memory classes ZYAL should distinguish

| Memory type          | Purpose                   |            Should it be model-writable? | Example                                       |
| -------------------- | ------------------------- | --------------------------------------: | --------------------------------------------- |
| Working memory       | Current iteration only    |                          Yes, ephemeral | “I am fixing auth test failure X.”            |
| Task memory          | Durable per-task capsule  | Yes, but host-reviewed before promotion | Known failures, relevant files, rollback.     |
| Run memory           | Run-level decisions       |                     Yes with provenance | “We chose approach B because A broke test Y.” |
| Repo memory          | Architecture and norms    |                   Mostly human-authored | AGENTS.md, design docs, code ownership.       |
| Episodic memory      | Prior run/session history |                 Append-only, summarized | Previous daemon attempts and outcomes.        |
| Semantic code memory | Symbol/dependency graph   |                          Host-generated | “Function A calls B; tests T cover B.”        |
| Procedural memory    | Skills/playbooks          |           Model-proposed, host-promoted | “How to add a migration safely.”              |
| User/org memory      | Preferences and policies  |                          Human-approved | “Never modify billing code without approval.” |

ZYAL v1’s context section is a start, but it should evolve into this full lifecycle.

## 7. ZYAL needs trust zones and taint tracking

The current `permissions` and `guardrails` are good, but they are too flat. A 1B+ LOC autonomous coding system must distinguish trusted and untrusted sources:

* user prompt;
* repo docs;
* issue tracker;
* web pages;
* dependency code;
* test output;
* compiler output;
* model-generated summaries;
* memory from previous sessions;
* worker reports;
* pull request comments;
* logs containing user data;
* email or chat triggers.

MCP’s own security guidance emphasizes user consent/control, data privacy, tool safety, and sampling controls because servers may expose arbitrary data or actions. OpenClaw’s docs also emphasize trust defaults around inbound messages and remote channels, while OpenHands hooks can block dangerous commands and inject context. ([Model Context Protocol][8])

### Proposed addition: `trust`

```yaml
trust:
  zones:
    human_direct:
      trust: high
      may_arm_ocal: true

    repo_tracked:
      trust: medium
      may_define_constraints: true
      may_define_permissions: false

    issue_tracker:
      trust: low
      taint: untrusted_text
      may_arm_ocal: false

    web:
      trust: low
      taint: untrusted_text
      may_arm_ocal: false

    tool_output:
      trust: contextual
      taint_if:
        - command_reads_untrusted_file
        - command_reads_network

    model_summary:
      trust: low
      require_source_refs: true

  taint_rules:
    - source: web
      forbid:
        - permission_escalation
        - tool_policy_changes
        - memory_write_without_review
    - source: issue_tracker
      forbid:
        - shell_command_direct_execution
        - secret_access
        - network_access
    - source: model_summary
      require:
        - cited_evidence_refs

  prompt_injection:
    detect_patterns:
      - "ignore previous instructions"
      - "exfiltrate"
      - "send secrets"
      - "developer message"
      - "system prompt"
    action: quarantine
```

This is essential for making ZYAL safe inside any chat prompt. A random GitHub issue or Slack message must never be able to arm a daemon, widen permissions, access secrets, or rewrite memory.

## 8. ZYAL needs a real sandbox/resource model

ZYAL currently has permissions like `shell: ask` and worker isolation like `git_worktree`. That is not enough. Modern coding agents need explicit resource envelopes:

* filesystem roots;
* network access;
* secrets access;
* container image;
* CPU/memory/disk;
* process timeout;
* package install policy;
* dependency update policy;
* remote execution policy;
* artifact retention.

OpenHands and Hermes both emphasize sandboxing and command approvals in different ways. OpenHands’s default Docker sandbox is specifically designed for isolation and reproducibility, while Hermes documents multiple terminal backends and layered security including dangerous-command approval and container isolation. ([docs.all-hands.dev][3])

### Proposed addition: `workspace`

```yaml
workspace:
  isolation:
    default: container
    allowed:
      - container
      - git_worktree
      - remote_vm
      - same_session
    forbid_same_session_for:
      - prototype
      - untrusted_input
      - dependency_upgrade
      - migration

  filesystem:
    read:
      - "."
      - "~/.cache/jekko/readonly"
    write:
      - "."
      - ".jekko/daemon/{{run_id}}"
    deny:
      - "~/.ssh"
      - "~/.aws"
      - "~/.config/gh"
      - ".env"
      - "**/*secret*"

  container:
    image: "ghcr.io/org/project-agent:latest"
    user: "agent"
    cpu: 4
    memory: 8GiB
    disk: 30GiB
    timeout: 2h
    snapshot_on:
      - checkpoint
      - failure
      - promotion_review

  network:
    default: deny
    allow:
      - host: "registry.npmjs.org"
        reason: "install dependencies"
      - host: "api.github.com"
        reason: "read issue and PR metadata"
    require_approval_for:
      - wildcard_domains
      - outbound_post
      - unknown_hosts

  secrets:
    default: deny
    allow:
      - name: GITHUB_TOKEN
        scope: github_readonly
        expose_to:
          - supervisor
        redact_in_logs: true
    require_approval_for:
      - write_tokens
      - cloud_credentials
      - production_credentials

  package_managers:
    install: ask
    update_lockfile: ask
    add_dependency: ask
    remove_dependency: ask
```

This would make ZYAL much stronger than a prompt-level “don’t touch secrets” instruction.

## 9. ZYAL should define an Agent-Computer Interface profile

SWE-agent’s ACI lesson is crucial: model performance is heavily shaped by the interface. A daemon language that only controls lifecycle but ignores the tool interface will leave major performance on the table. SWE-agent’s ACI docs emphasize specific environment/tool designs such as concise file viewers, lint-on-edit behavior, and specialized navigation/search/edit tools. ([arXiv][10])

### Proposed addition: `aci`

```yaml
aci:
  file_viewer:
    mode: windowed
    lines: 120
    require_line_numbers: true
    allow_symbol_view: true
    summarize_large_files_over: 2000_lines

  edit:
    mode: patch
    require_exact_context: true
    lint_on_edit: true
    reject_if_lint_fails:
      severity: warning

  search:
    prefer:
      - ripgrep
```

## 10. Multi-agent orchestration needs contracts, not just workers

ZYAL v1 has `agents`, `fan_out`, and incubator passes. That is good. But multi-agent systems fail when agents duplicate work, lose context, overwrite each other, or return unmergeable outputs.

Anthropic’s multi-agent writeup is useful here: orchestrator-worker designs can improve breadth-first work, but they are token-expensive and fragile without careful delegation, evaluation, and coordination. It also notes coding is often less parallelizable than research because tools update shared state and dependencies interact. ([Anthropic][11])

### Proposed addition: `delegation`

```yaml
delegation:
  contracts:
    default_worker_contract:
      must_return:
        - summary
        - files_read
        - files_changed
        - tests_run
        - risks
        - confidence
        - blockers
        - evidence_refs
      must_not:
        - modify_files_outside_scope
        - commit_without_checkpoint
        - spawn_unapproved_workers
      context_policy:
        receive:
          - task_capsule
          - relevant_files
          - constraints
          - prior_failed_attempts
        withhold:
          - other_workers_private_reasoning
          - unrelated_repo_memory

  duplicate_work_detection:
    enabled: true
    compare:
      - task_objective
      - files_touched
      - hypotheses
    action: notify_supervisor

  merge_accountability:
    require_merge_owner: true
    merge_owner: supervisor
    worker_patches_must_be_reviewed: true
    conflict_strategy: synthesize_not_last_writer_wins

  worker_lifecycle:
    max_active: 8
    max_spawn_depth: 1
    idle_timeout: 10m
    kill_on:
      - permission_denied
      - scope_violation
      - budget_exhausted
```

### Better `agents` syntax

```yaml
agents:
  supervisor:
    agent: plan
    responsibilities:
      - task_decomposition
      - scope_control
      - merge_decisions
      - promotion_evidence
    cannot:
      - edit_production_code

  workers:
    - id: explorer
      agent: explore
      count: 3
      isolation: read_only
      purpose: "Find relevant files, risks, and test surfaces."
      writes: scratch_only

    - id: builder
      agent: build
      count: 4
      isolation: git_worktree
      purpose: "Implement bounded changes."
      writes: isolated_worktree

    - id: verifier
      agent: verify
      count: 2
      isolation: git_worktree
      purpose: "Try to falsify the patch."
      writes: scratch_only

    - id: critic
      agent: critic
      count: 2
      isolation: read_only
      purpose: "Find objections and hidden risks."
      writes: scratch_only
```

This makes multi-agent work safer and more productive than a generic swarm.

## 11. The incubator is excellent, but it should become an experiment engine

The incubator is the most original part of ZYAL v1. It should be expanded into a full **hypothesis/experiment/synthesis** system.

The current pass list is linear:

1. scout;
2. ideas;
3. strengthen;
4. critic;
5. synthesize;
6. prototype;
7. review;
8. compress.

That is good, but for ambitious coding projects, you need **disjointed experiments** and **stateful building on current state**:

* run divergent approaches in isolated worktrees;
* prevent idea contamination until reduce time;
* score each branch with common gates;
* keep losing branches as evidence;
* synthesize partial wins;
* run ablations;
* promote only after independent verification.

### Proposed addition: `experiments`

```yaml
experiments:
  enabled: true
  strategy: divergent_then_convergent

  trigger_when:
    any:
      - task.risk_score_gte: 0.6
      - repeated_attempts_gte: 2
      - architectural_uncertainty: true
      - tests_fail_after_repair_gte: 2
      - dependency_impact_unknown: true

  branches:
    isolation: git_worktree
    max_parallel: 6
    max_duration_per_branch: 30m
    forbid_shared_context_until: reduce
    diversity:
      require_distinct:
        - hypothesis
        - primary_files
        - test_strategy
      profiles:
        - id: minimal_patch
          instruction: "Smallest safe change."
        - id: test_first
          instruction: "Write or identify failing test first."
        - id: architectural
          instruction: "Consider deeper design issue."
        - id: rollback_first
          instruction: "Optimize for reversibility."
        - id: performance_first
          instruction: "Preserve or improve performance."
        - id: security_first
          instruction: "Assume adversarial input."

  hypothesis_schema:
    required:
      - hypothesis
      - predicted_failure_mode
      - expected_changed_files
      - validation_command
      - rollback_plan

  score:
    weights:
      tests_pass: 0.30
      typecheck_pass: 0.15
      diff_minimality: 0.10
      maintainability: 0.15
      blast_radius: -0.15
      unresolved_objections: -0.20
      coverage_delta: 0.10
    require:
      - validation_command_run
      - rollback_plan_known

  reduce:
    strategy: evidence_weighted_synthesis
    keep_losing_artifacts: true
    require_ablation: true
    require_critic_review: true
    synthesize_allowed_from:
      - winning_patch
      - partial_successes
      - critic_objections
      - test_evidence

  promotion:
    promote_at: 0.82
    require_independent_verifier: true
    verifier_must_not_be_branch_author: true
```

This is the difference between “try random fixes until tests pass” and “run structured research on the codebase.”

## 12. ZYAL needs replay, fork, and time travel

LangGraph’s persistence/checkpoint model is a major lesson: a long-running system should be able to resume from a checkpoint, interrupt for human input, and inspect or fork prior states. ([LangChain Docs][4])

ZYAL has SQLite truth, which makes this natural, but the syntax does not expose it yet.

### Proposed addition: `replay`

```yaml
replay:
  checkpoints:
    create_on:
      - run_start
      - before_iteration
      - after_iteration
      - before_tool_call
      - after_tool_call
      - before_checkpoint
      - after_checkpoint
      - before_promotion
      - on_error
    include:
      - task_state
      - memory_refs
      - diff_snapshot
      - tool_call_inputs
      - tool_call_outputs
      - approvals
      - random_seed
      - model_id
      - prompt_hash
      - context_hash

  fork:
    allowed_from:
      - checkpoint
      - failed_branch
      - incubator_pass
      - human_review
    require_reason: true
    isolation: git_worktree
    compare_against_parent: true

  deterministic_replay:
    mode: best_effort
    freeze:
      - model_id
      - tool_versions
      - container_image
      - environment
      - prompts
      - file_snapshots
    allow_nondeterminism:
      - llm_sampling

  audit:
    export:
      - sqlite
      - jsonl
      - markdown_summary
      - evidence_packages
```

For 1B+ LOC projects, this becomes mandatory. You need to know not only what the daemon did, but **why a branch was abandoned, what evidence it had, and whether a previous state should be resurrected.**

## 14. ZYAL should include scheduled and event-triggered activation

Hermes and OpenClaw both show that long-running agents are not only manually started. Hermes has scheduled automations; OpenClaw has gateway/channel and task-flow concepts. ([GitHub][5])

ZYAL needs `triggers`, but they must be safe. A Slack message or GitHub issue should be able to request preview, not arm dangerous execution by itself.

### Proposed addition: `triggers`

```yaml
triggers:

    enabled: true
    arm_requires: human_direct

  schedule:

      timezone: "America/Denver"
      action: preview_then_request_arm
      max_runs_per_day: 1

  git:
    - id: on-main-ci-failure
      event: ci_failed
      branch: main
      action: preview
      require_human_arm: true

    - id: on-pr-comment
      event: pull_request_comment
      pattern: "/zyal preview"
      action: preview
      trust_zone: issue_tracker

    - id: external-build-failure
      path: "/webhooks/build-failure"

```

This is how ZYAL becomes available “inside any chat prompt” without making chat itself a dangerous execution authority.

## 15. ZYAL needs stronger stop/progress semantics

Current `stop` is mostly boolean conditions. For long-running agents, the harder problem is **knowing whether useful progress is happening**.

Add measurable progress:

* new failing test identified;
* failing test fixed;
* diff reduced;
* coverage improved;
* blocker resolved;
* number of unknowns reduced;
* risk score reduced;
* evidence completeness increased;
* task readiness increased;
* branch score improved.

### Proposed addition: `progress`

```yaml
progress:
  heartbeat_every: 5m

  metrics:
    - id: tests_passing
      source:
        shell: "bun test --reporter=json"
      extract:
        jsonpath: "$.numPassedTests"
      direction: increase

    - id: type_errors
      source:
        shell: "npx tsgo --noEmit --pretty false 2>&1 | grep -c 'error TS' || true"
      direction: decrease

    - id: unresolved_objections
      source:
        evidence: "critic.unresolved_critical_objections"
      direction: decrease

    - id: evidence_completeness
      source:
        evidence: "package.completeness_score"
      direction: increase

  no_progress:
    window: 4_iterations
    if_none_improved:
      - tests_passing
      - type_errors
      - unresolved_objections
      - evidence_completeness
    action: incubate_or_pause

  regressions:
    - metric: tests_passing
      drop_gte: 1
      action: rollback_branch
    - metric: type_errors
      increase_gte: 5
      action: pause
```

This is much better than asking the model whether it is making progress.

## 20. ZYAL needs negative constraints against fake progress

A strong default language should explicitly forbid common janky moves:

* deleting tests;
* skipping tests;
* weakening assertions;
* changing snapshots blindly;
* hardcoding expected output;
* broad try/catch swallowing errors;
* disabling lint/type checks;
* adding sleeps to hide races;
* silencing warnings;
* force-pushing;
* changing generated files without changing source;
* modifying vendored code;
* creating giant diffs.

### Proposed addition: `quality_gates`

```yaml
quality_gates:
  anti_shortcuts:
    - id: no_test_deletion
      check:
        shell: "git diff --name-only --diff-filter=D | grep -E '(__tests__|test|spec)'"
      assert:
        exit_code: 1
      on_fail: block_promotion

    - id: no_skip_or_only
      check:
        shell: "git diff -U0 | grep -E '\\.(skip|only)\\(|describe\\.only|it\\.only|test\\.only'"

        exit_code: 1
      on_fail: block_promotion

    - id: no_assertion_weakening
      check:
        shell: "git diff -U3 | grep -E 'toBeTruthy\\(|toBeDefined\\(|expect\\(.*\\)\\.not'"
      on_match: require_review

    - id: no_lint_disable_without_reason
      check:
        shell: "git diff -U0 | grep -E 'eslint-disable|ts-ignore|ts-nocheck'"
      on_match: require_review

    - id: no_snapshot_mass_update
      check:
        shell: "git diff --name-only | grep -c '__snapshots__' || true"
      assert:
        max: 3
      on_fail: require_review

    - id: no_generated_only_change
      check:
        shell: "zyal diff generated-source-ratio"
      assert:
        source_changed: true
      on_fail: block_promotion

  diff_limits:
    max_files_changed_without_review: 25
    max_lines_changed_without_review: 1200
    max_binary_files_added: 0

  require_tests_for:
    - production_code_change
    - bug_fix
    - migration
    - parser_change
    - auth_change
```

This is one of the most practical upgrades in the whole spec.

## 23. ZYAL should support “stateful building on current state”

The user specifically mentioned “disjointed experiments, and stateful building on current state.” ZYAL should support both.

### Disjoint experiments

Use isolated branches with context separation:

```yaml
experiments:
  branches:
    isolation: git_worktree
    forbid_shared_context_until: reduce
    count: 5
```

### Stateful building

Use successor branches that inherit from verified partial states:

```yaml
stateful_build:
  enabled: true

  partial_promotions:
    allow: true
    require:
      - tests_for_partial_scope_pass
      - no_regressions_outside_scope
      - rollback_known
    store_as:
      - reusable_checkpoint
      - task_memory_update

  successor_policy:
    inherit_from:
      - last_verified_checkpoint
      - promoted_partial_patch
      - task_capsule
      - unresolved_objections
    do_not_inherit:
      - failed_hypotheses_without_summary
      - stale_context
      - unverified_claims

  build_layers:
    - id: foundation
      objective: "Create minimal safe structure."
      promote_when: "scope_tests_pass && rollback_known"
    - id: integration
      objective: "Integrate with adjacent modules."
      promote_when: "affected_tests_pass"
    - id: hardening
      objective: "Add edge cases, security, performance."
      promote_when: "critic_objections_resolved"
```

This enables an agent to build on verified intermediate states rather than resetting or blindly continuing.

## 27. What ZYAL should explicitly not become

To stay powerful, ZYAL should avoid several traps.

### Do not become a prompt language

ZYAL should not be “instructions for the model.” It should be a host-parsed contract. The model may produce ZYAL, but the host decides whether to preview, reject, or arm.

### Do not become a tool protocol

MCP already exists for tool/resource/prompt integration. ZYAL should govern MCP usage, not replace MCP.

### Do not become an editor protocol

ACP is the right layer for editor-agent communication. ZYAL should emit events over ACP, not duplicate ACP.

### Do not become a swarm launcher

Multi-agent is expensive and fragile when underspecified. ZYAL should require delegation contracts, budgets, isolation, and merge accountability.

### Do not let “memory” become untrusted prompt sludge

Memory must be typed, scoped, provenance-tracked, summarized, and scanned. Model-written memory should be reviewed before becoming durable organizational memory.

### Do not let “self-improvement” bypass review

Hermes-style skills and learning loops are exciting, but for coding projects they need promotion gates, repeated evidence, and human approval before becoming reusable behavior.

---

## 28. Highest-leverage upgrade list

If I had to prioritize ZYAL upgrades, I would do them in this order:

1. **Formal JSON Schema + compiler/linter/preview model.** Without this, portability will be weak.
2. **Capability negotiation and adapters.** This is how ZYAL avoids redundancy with OpenHands, Codex, LangGraph, Hermes, and ACP.
3. **Evidence packages.** This is the anti-vibe-coding backbone.
4. **Memory subsystem.** Typed memory is mandatory for long-running work.
5. **Workflow graph.** Loops are not enough for serious projects.
6. **Trust zones and taint tracking.** Required for prompt-embedded safety.
7. **Sandbox/resource model.** Permissions need concrete resource boundaries.
8. **Experiment engine.** This makes the incubator meaningfully powerful.
9. **Replay/fork/time travel.** Required for debugging autonomous work.
10. **Large-scale repo/index/sharding model.** Required for 1B+ LOC ambitions.
11. **Approval protocols.** Required for teams and enterprises.
12. **Skills with host-reviewed promotion.** Required for durable improvement.
13. **Quality gates against fake progress.** Required to prevent janky fixes.
14. **Cost/model routing.** Required for practical long-horizon runs.
15. **ACP/MCP event integration.** Required for ecosystem adoption.

---

## 29. The strategic positioning

The right positioning is:

> **ZYAL is the portable, host-enforced Agent Operations Language for long-running coding work.**

It should be to coding agents what CI YAML is to build systems, what Kubernetes manifests are to infrastructure, and what policy-as-code is to security — except focused on autonomous software engineering.

The strongest claim ZYAL can make is not “we have agents.” Everyone has agents. The claim should be:

> **ZYAL makes autonomous coding governable.**

That means:

* the host owns lifecycle;
* humans arm and approve;
* memory is durable but controlled;
* experiments are isolated;
* promotion requires evidence;
* every action is replayable;
* every daemon has budgets;
* every worker has scope;
* every risky input is tainted;
* every large project is sharded;
* every patch has rollback;
* every claimed success has proof.

That is how ZYAL can become the default language for ambitious coding projects.
