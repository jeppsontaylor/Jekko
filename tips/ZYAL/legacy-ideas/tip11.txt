Yes — after studying the ZYAL draft against current agent runtimes, coding-agent tools, and agent research, my conclusion is:

**ZYAL is not redundant.** It is aiming at a missing layer: a **portable, host-enforced daemon manifest** for long-running coding agents. OpenHands, LangGraph/DeepAgents, Hermes, OpenClaw, Claude Code, Codex, and SWE-agent-like systems each cover pieces of this, but none quite offer “drop this control surface into any chat prompt, let the host own lifecycle/safety/state, and run ambitious autonomous coding work with evidence-gated promotion.”

The opportunity is huge. But to become the default language for the world’s most ambitious coding projects, ZYAL needs to evolve from a daemon YAML into an **Agentic Software Engineering Operating Contract**: runtime isolation, replay, typed tools, memory discipline, experiment orchestration, evals, observability, human gates, supply-chain policy, and large-repo intelligence.

I did not literally inspect every social post on X, LinkedIn, Quora, and the entire open-source ecosystem. I treated those as weak-signal sources and weighted official docs, GitHub repos, specs, and research papers more heavily. That is the right call here: ZYAL should be designed from durable patterns, not hype.

---

# 1. Executive verdict

ZYAL already has the right philosophical core:

**The host owns the loop. The model proposes. The host disposes.**

That is the most important idea. Most agent systems still let too much lifecycle control leak into the model, the prompt, or an application-specific harness. ZYAL’s strongest differentiators are:

| ZYAL strength                                | Why it matters                                              |
| -------------------------------------------- | ----------------------------------------------------------- |
| Explicit sentinels and arming                | Prevents assistant output from silently becoming execution. |
| Host-enforced stop/checkpoint/permissions    | Makes autonomy bounded and reviewable.                      |
| SQLite source of truth                       | Gives durable local state and human-readable mirrors.       |
| Incubator/promotion model                    | Separates “thinking hard” from touching production code.    |
| Circuit breakers and budgets                 | Prevents unbounded swarms and runaway loops.                |
| Guardrails/constraints as runtime structures | Moves safety out of the prompt and into enforcement.        |

But today ZYAL is still primarily a **loop/control/checkpoint language**. To dominate, it needs these major upgrades:

1. **Runtime and workspace isolation**: Docker/Kubernetes/cloud sandbox/worktree policy, network policy, resource quotas, secret scopes.
2. **Event-sourced replay and time travel**: deterministic replay, fork-from-iteration, idempotency, side-effect ledger.
3. **Graph workflows**: beyond `loop` and `fan_out`, add typed phases, DAGs, interrupts, joins, retries, and state transitions.
4. **First-class memory**: project/task/run/org/user/skill/evidence memory, trust levels, TTLs, compaction, poisoning resistance.
5. **Experiment orchestration**: disjointed parallel attempts, tournament selection, ablations, stateful continuation, merge strategies.
6. **Eval-driven engineering**: test suites, benchmark suites, LLM-as-judge, trace-derived regression datasets, online quality checks.
7. **Observability**: traces, costs, token budgets, tool spans, OpenTelemetry-style propagation, dashboards, anomaly detection.
8. **Typed agent-computer interface**: repo search, symbol graph, patch tool, affected tests, not just shell.
9. **Large-repo intelligence**: code index, dependency graph, ownership, affected-test selection, architectural boundaries.
10. **Security policy packs**: least privilege per tool/file/network/secret/MCP server, prompt-injection/memory-poisoning defenses.
11. **Skill and procedure system**: progressive-disclosure procedural memory with validation, versioning, signatures, and evals.
12. **Universal host adapters**: OpenCode native, OpenHands runtime, LangGraph/DeepAgents runtime, Claude Code/Codex harnesses.
13. **Human gate UX**: preview run cards, risk scoring, approval bundles, merge/open-PR/discard decisions.
14. **Fleet/state backend abstraction**: SQLite for local runs, Postgres/event-store/warehouse for enterprise and 1B+ LOC programs.
15. **Supply-chain and provenance controls**: dependency diffs, SBOMs, lockfile policy, artifact attestations, rollback plans.

The short design slogan I would use:

> **ZYAL should become the Kubernetes manifest for autonomous coding agents — but with Git, memory, evidence, evals, and human approval built in.**

# 2. What the outside systems teach ZYAL

## OpenHands

OpenHands is closest to ZYAL in spirit on runtime and engineering architecture. Its SDK emphasizes stateless agents, event-sourced state, deterministic replay, typed tools, MCP integration, workspace abstraction, REST/WebSocket serving, and interactive workspaces. Its runtime docs also foreground Docker sandboxing for security, reproducibility, resource control, and isolation. Its agent docs describe a stateless event-driven architecture with context, condensers, security analysis, and step-wise tool orchestration. ([arXiv][1])

**What ZYAL should borrow:**

* Event-sourced state model, not just “SQLite tables.”
* Deterministic replay and forkability.
* Workspace abstraction: local repo, Docker, remote sandbox, Git worktree, Kubernetes job, cloud dev environment.
* Typed tools as first-class contracts.
* Runtime server/event stream as a native primitive, not just an API add-on.

**Where ZYAL can be stronger:**

OpenHands is an SDK/runtime. ZYAL can become the **portable policy/runbook language** that configures OpenHands-like runtimes. ZYAL should not compete with OpenHands at the implementation layer; it should be able to compile to or wrap it.

---

## LangGraph, LangChain, DeepAgents, LangSmith

LangGraph’s strongest ideas are durable execution, checkpointed graph state, threads, fault tolerance, human interrupts, memory, replay, and time travel. Its persistence docs explicitly frame checkpointing as a way to inspect state, interrupt for approval, replay or fork prior states, and recover from failures. LangGraph’s broader platform emphasizes durable long-running tasks, stateful agents, HITL, scaling, branching, debugging, and memory/state inspection. ([LangChain Docs][2])

DeepAgents adds practical agent structure: built-in planning, filesystem-backed context, subagent spawning, long-term memory, sandboxed shell execution, declarative filesystem permissions, and human-in-the-loop approval. It also explicitly uses subagents for context quarantine and long-running/parallel work. ([LangChain Docs][3])

LangSmith adds another missing layer: production tracing, eval datasets, online evaluators, regression testing, cost/latency attribution, anomaly monitoring, and turning production traces into test datasets. ([LangChain Docs][4])

**What ZYAL should borrow:**

* Replace simple loops with **graph workflows**.
* Add **thread/checkpoint/fork/time-travel** semantics.
* Add **observability and evals** as first-class top-level concepts.
* Add memory scopes and stores.
* Add human interrupts as typed gates.

**Where ZYAL can be stronger:**

LangGraph is powerful but programmatic. ZYAL can be the **declarative manifest** that non-framework hosts can parse from any chat prompt. It can be the safe, typed, human-readable control surface above LangGraph-like engines.

---

## Hermes Agent

Hermes is important because it pushes persistent personal agents, skills, memory, cron, messaging, multiple terminal backends, self-improving skill loops, and security policy into the product surface. Its README describes a self-improving agent with a learning loop, agent-curated memory, skill creation, skill improvement, search over prior conversations, user modeling, cron scheduling, isolated subagents, and several terminal backends. ([GitHub][5])

Hermes’ security docs are especially relevant: user authorization, dangerous command approval, container isolation, MCP credential filtering, prompt-injection detection in context files, cross-session isolation, hardline blocklists, smart approval modes, Docker flags, resource limits, and credential filtering. ([Hermes Agent][6])

Hermes also has a progressive-disclosure skill system: a small skills list, full skill files loaded on demand, and deeper reference files only loaded when needed. Its memory system is explicitly bounded and curated, with small injected memories, frozen session snapshots, and capacity limits. Its cron system supports natural-language schedules, pause/resume/edit/trigger/remove, skill attachment, delivery targets, and a recursion guard that prevents cron jobs from creating more cron jobs. ([Hermes Agent][7])

**What ZYAL should borrow:**

* Skills as progressive-disclosure procedural memory.
* Bounded curated memory, not infinite vector-store soup.
* Cron/triggers with anti-recursion safeguards.
* Terminal/runtime backend abstraction.
* Hardline security floors even in permissive modes.
* Agent-created skills, but only with validation and review.

**Where ZYAL can be stronger:**

Hermes is an agent product. ZYAL can be a **portable execution contract**. ZYAL’s preview/arm sentinel, host-owned loop, promotion gates, incubator, and durable ledger are more formal than Hermes’ product-level workflows.

---

## OpenClaw / OpenClaw Code Agent

OpenClaw is useful because it emphasizes chat-launched background work, delegated coding sessions, wake routing, worktree isolation, merge/PR follow-through, explicit goal loops, plan-review-execute flow, session lifecycle management, state-driven decision buttons, verifier-driven repair loops, and shared control over Claude/Codex-like coding agents. ([GitHub][8])

**What ZYAL should borrow:**

* “Run from chat, continue from chat” UX.
* Plan → review → execute as a standard approval mode.
* Worktree isolation as a default for delegated coding.
* Session lifecycle: suspend, resume, fork, interrupt, recover.
* Decision UX: merge, open PR, later, discard.
* Verifier-driven repair loops.

**Where ZYAL can be stronger:**

OpenClaw orchestrates background coding agents. ZYAL can define the **formal manifest those sessions obey**. OpenClaw-like products could become ZYAL hosts.

---

## Claude Code

Claude Code’s docs reinforce several ideas ZYAL should adopt: hooks at lifecycle points, subagents with scoped tools/model/memory, aggressive context management, checkpoint rewind/resume, multiple sessions/fan-out, permission modes, and checked-in permission files. Its permission model uses allow/deny/ask ordering and warns that bypass mode should only be used in isolated environments. ([Claude][9])

**What ZYAL should borrow:**

* More lifecycle hook events.
* Subagent-local memory and tool scopes.
* Context compaction/rewind/resume as formal runtime features.
* Project-checked-in policy files.
* Plan mode and auto-edit modes as explicit execution modes.
* “Bypass only in sandbox” as a hard rule.

**Where ZYAL can be stronger:**

Claude Code is tool-specific. ZYAL can be the cross-agent policy that works with OpenCode, Claude Code, Codex, OpenHands, and future tools.

---

## OpenAI Codex and Agents SDK

OpenAI’s Codex CLI runs locally in a selected directory and can read/change/run code; the cloud Codex experience can work on tasks in the background and in parallel, each with its own cloud environment. The Agents SDK exposes concepts such as orchestration, handoffs, guardrails, human review, state, and observability/tracing. ([OpenAI Developers][10])

**What ZYAL should borrow:**

* Cloud/local execution parity.
* Background parallel tasks as a standard pattern.
* Handoffs and guardrails as first-class events.
* Trace spans over model calls, tool calls, handoffs, guardrails, and custom events.

**Where ZYAL can be stronger:**

ZYAL can be the declarative host-enforced task contract that Codex-like tools execute under, rather than another agent runtime.

---

## MCP

MCP is the key external integration protocol. It standardizes how hosts, clients, and servers expose tools, resources, prompts, sampling, and other capabilities over JSON-RPC-style interactions. Its spec also covers prompts, tools, authorization, sampling, and elicitation. ([Model Context Protocol][11])

**What ZYAL should borrow:**

* Full MCP primitive coverage: tools, resources, prompts, sampling, elicitation, roots, auth.
* Server pinning and capability hashing.
* Per-server/per-tool permissions.
* MCP trace propagation and audit events.

**Where ZYAL can be stronger:**

MCP exposes capabilities. ZYAL defines **when, why, and under what constraints** those capabilities may be used.

---

## Security research and standards

OWASP’s LLM and agentic security guidance maps almost directly onto ZYAL’s future roadmap: excessive agency, prompt injection, tool misuse, privilege escalation, data exfiltration, memory poisoning, goal hijacking, cascading failures, malicious configuration, denial-of-wallet, and sensitive information exposure. OWASP also recommends minimizing tools, using least privilege, separating toolsets by trust level, and requiring explicit authorization for sensitive operations. ([OWASP Gen AI Security Project][12])

NIST’s AI Risk Management Framework and GenAI profile are relevant for enterprise adoption because they frame AI risk management as governable, measurable, and documentable rather than purely ad hoc. ([NIST][13])

**What ZYAL should borrow:**

* Least privilege as a schema-level invariant.
* Memory poisoning defenses.
* Denial-of-wallet/cost controls.
* Tool trust zones.
* Explicit identity and authorization.
* Governance logs and risk profiles.

---

## Agent research

SWE-agent demonstrates that the **agent-computer interface** matters: purpose-built coding tools and interaction formats can materially improve repo navigation, editing, and testing. AutoCodeRover emphasizes code-search-guided issue solving and context retrieval before patching. Voyager shows the value of an ever-growing skill library, automatic curriculum, executable skills, and iterative feedback. Reflexion and Self-Refine show the value of feedback/reflection/refinement loops and persistent reflective memory. Plan-and-act work reinforces the benefit of separating planning from execution for long-horizon tasks. ([arXiv][14])

**What ZYAL should borrow:**

* Typed repo tools, not just shell.
* Plan-first and test-first gates.
* Reflection memory, but bounded and evidence-backed.
* Skill creation, but validated and reviewable.
* Retrieval before patching.
* Repeated refinement with hard host checks.

# 3. The biggest thing ZYAL is missing

ZYAL’s current draft says:

> “Host owns the loop.”

The vNext should say:

> **“Host owns the loop, state, runtime, memory, evidence, experiments, promotion, and blast radius.”**

Today ZYAL controls “can the daemon keep going?” It should also control:

* **Where** it runs.
* **What resources** it can consume.
* **What memory** it can read/write.
* **Which code regions** it can touch.
* **Which tools** it can call.
* **Which evidence** counts.
* **Which experiments** are allowed.
* **Which outputs** can promote.
* **Which humans** approve which gates.
* **Which traces/evals** prove progress.
* **Which rollback path** exists.

That is the leap from “daemon language” to “autonomous coding operating contract.”

# 4. ZYAL vNext: proposed top-level expansion

I would keep the existing ZYAL core, but add modular top-level blocks. The design should avoid becoming an unreadable mega-YAML by using policy packs, inheritance, and defaults.

## Proposed vNext top-level keys

| Key             | Purpose                                                                              |
| --------------- | ------------------------------------------------------------------------------------ |
| `runtime`       | Sandbox/workspace/backend/resource/network policy.                                   |
| `state`         | Event log, checkpoints, replay, forking, idempotency.                                |
| `workflow`      | Graph/DAG phases beyond simple loop/fan-out.                                         |
| `planning`      | Plan-first, test-first, approval, todo semantics.                                    |
| `memory`        | Scoped stores, retrieval, compaction, write policy, poisoning defense.               |
| `skills`        | Procedural memory, progressive disclosure, skill validation.                         |
| `experiments`   | Parallel/disjoint/stateful experiments, scoring, merge rules.                        |
| `evals`         | Regression suites, LLM judges, trace-derived tests, quality gates.                   |
| `observability` | Traces, metrics, token/cost spans, logs, dashboards.                                 |
| `security`      | Hardline safety floors, trust zones, prompt-injection and memory-poisoning controls. |
| `secrets`       | Secret mounting, redaction, scoped credentials, leak checks.                         |
| `network`       | Domain/IP allowlists, egress policy, package registry policy.                        |
| `models`        | Model routing, budgets, fallbacks, temperature, thinking mode.                       |
| `tools`         | Typed ACI tools, shell restrictions, MCP capability hashes.                          |
| `approval`      | Human gates, evidence bundles, expiration, default-deny.                             |
| `triggers`      | Cron, GitHub, CI, webhook, chat, manual, issue/PR events.                            |
| `artifacts`     | Patches, PRs, reports, SBOMs, build artifacts, provenance.                           |
| `rollout`       | Merge, canary, deployment, rollback, production gates.                               |
| `governance`    | Policy packs, audit retention, identity, compliance labels.                          |
| `interop`       | Compile/run adapters for OpenCode, OpenHands, LangGraph, Claude, Codex.              |

The current keys are good; they just need to sit inside a larger operating envelope.

# 5. Upgrade 1 — add `runtime`

Current ZYAL has `permissions`, `agents`, `fan_out`, and some worktree isolation, but it does not fully define the execution substrate.

For serious autonomy, the host must know:

* Is this running in local repo, Git worktree, Docker, Kubernetes, remote sandbox, or cloud dev environment?
* What CPU/memory/disk/time limits apply?
* Is the network enabled?
* Which domains are allowed?
* Are package installs allowed?
* Are secrets mounted?
* Are file writes isolated?
* Are dangerous commands blocked even if permission says `allow`?
* Can the daemon spawn nested daemons?
* Can it modify its own ZYAL spec?

Suggested schema:

```yaml
runtime:
  workspace:
    backend: docker              # local | git_worktree | docker | kubernetes | cloud_sandbox | openhands
    image: ghcr.io/acme/dev:sha256-...
    repo_mount: read_write
    worktree:
      strategy: per_task         # per_run | per_task | per_worker | none
      cleanup: after_merge       # after_merge | archive | keep
    snapshots:
      on_start: true
      before_checkpoint: true
      before_tool_risk_gte: 0.8
      retention: 30d

  resources:
    max_wall_time: 12h
    max_cpu: "8"
    max_memory: 16Gi
    max_disk: 50Gi
    max_processes: 256
    max_tool_calls_per_iteration: 50

  network:
    default: deny
    allow_domains:
      - registry.npmjs.org
      - api.github.com
    deny_private_networks: true
    record_requests: true

  shell:
    default_cwd: repo_root
    allowed_interpreters: ["bash", "node", "python", "bun"]
    deny_patterns:
      - "rm -rf /"
      - "git push --force"
      - "curl .*\\| sh"
    require_approval_for:
      - package_install
      - destructive_file_ops
      - external_network

  nested_agents:
    allow: false
    max_depth: 1
```

This is where ZYAL should learn from OpenHands’ sandbox-first runtime posture and Hermes’ defense-in-depth security model. OpenHands explicitly frames Docker runtime as a secure sandbox for isolation, reproducibility, and resource control; Hermes similarly treats terminal backends, dangerous command approval, container isolation, credential filtering, and hardline blocklists as core product features. ([OpenHands Docs][15])

**Design rule:** `permissions.shell: allow` should never mean “unlimited shell.” It should mean “allow shell inside the runtime security envelope.”

# 6. Upgrade 2 — make state event-sourced, replayable, and forkable

ZYAL says SQLite is truth. Good. But “SQLite tables” is not enough.

For long-running autonomous work, you need:

* Immutable event log.
* State snapshots.
* Deterministic replay.
* Tool call input/output records.
* Side-effect ledger.
* Idempotency keys.
* Fork from prior iteration.
* Time-travel debugging.
* Diff between branches.
* “Why did this promote?” evidence trail.

LangGraph’s persistence model uses checkpointed thread state to enable human inspection, interrupts, replay, forking, and fault tolerance; OpenHands’ SDK emphasizes event-sourced state and deterministic replay. ([LangChain Docs][2])

Suggested schema:

```yaml
state:
  backend:
    local: sqlite
    replicate_to: postgres       # optional for fleet/enterprise
  event_sourcing:
    enabled: true
    immutable_log: true
    include:
      - model_requests
      - model_responses
      - tool_calls
      - tool_results
      - file_diffs
      - approvals
      - guardrail_decisions
      - memory_reads
      - memory_writes
      - eval_results
      - cost_events

  snapshots:
    every_iteration: true
    before_checkpoint: true
    before_promotion: true
    before_destructive_action: true

  replay:
    deterministic: required
    tool_result_cache: true
    external_side_effects: record_and_stub
    allow_fork_from:
      - iteration
      - checkpoint
      - experiment
      - promotion_candidate

  idempotency:
    required_for:
      - hooks
      - checkpoint
      - external_api_calls
      - package_installs
    key_template: "{{run_id}}:{{phase}}:{{iteration}}:{{tool_call_hash}}"
```

This one is critical. ZYAL cannot safely run for hours across huge projects unless every meaningful action can be reconstructed.

# 7. Upgrade 3 — add graph workflows, not just loops

The existing `loop`, `fan_out`, and `on` blocks are useful, but ambitious coding projects are not just “loop forever until tests pass.” They are state machines:

1. Discover.
2. Scope.
3. Index.
4. Plan.
5. Review.
6. Prototype.
7. Test.
8. Critique.
9. Harden.
10. Merge.
11. Roll out.
12. Monitor.
13. Learn.

LangGraph’s entire advantage is graph/stateflow execution. ZYAL should add a declarative graph layer while keeping the simpler `loop` syntax as sugar.

Suggested schema:

```yaml
workflow:
  start: discover
  phases:
    discover:
      agent: plan
      do:
        - run_task_discovery: true
        - build_code_index: true
      next: scope

    scope:
      agent: plan
      require_structured_output: scope_report
      gates:
        - approval: scope_review
      next:
        if:
          risk_score_gte: 0.7
        then: incubate
        else: implement

    incubate:
      use: incubator
      next: experiment

    experiment:
      use: experiments.refactor_tournament
      next: select_candidate

    select_candidate:
      reduce:
        strategy: best_evidence
      next: verify

    verify:
      do:
        - run_eval_suite: core_regression
        - run_constraints: all
      next:
        if:
          eval_suite_passed: core_regression
        then: checkpoint
        else: repair

    repair:
      loop:
        max_iterations: 3
        body: implement
      next: verify

    checkpoint:
      use: checkpoint
      next: promote

    promote:
      gates:
        - approval: human_merge
      next: stop
```

This should not replace existing ZYAL. It should complement it:

* `loop` is easy mode.
* `workflow` is ambitious-project mode.

# 8. Upgrade 4 — first-class memory

The current `context` block handles compaction/rotation, but memory is much bigger than context.

A 1B+ LOC coding project needs multiple memory layers:

| Memory type      | Purpose                                               |
| ---------------- | ----------------------------------------------------- |
| Run memory       | What happened in this daemon run.                     |
| Task memory      | Current objective, attempts, blockers, decisions.     |
| Project memory   | Repo conventions, architecture, gotchas.              |
| Component memory | Per-service/package subsystem facts.                  |
| User/org memory  | Preferences, coding standards, risk tolerances.       |
| Evidence memory  | Test results, static analysis, benchmark outcomes.    |
| Skill memory     | Procedures that worked and when.                      |
| Failure memory   | Known bad approaches, flaky tests, prior regressions. |

Hermes’ bounded memory and progressive skills are a better model than naive vector memory: memory should be curated, scoped, capacity-limited, and frozen per session to avoid context churn. ([Hermes Agent][7])

Suggested schema:

```yaml
memory:
  stores:
    - name: run
      scope: run
      backend: sqlite
      max_bytes: 10MiB
      retention: 30d
      writable_by: host

    - name: task
      scope: task
      backend: sqlite
      max_items: 500
      compaction:
        strategy: evidence_summary
        every: 5_iterations

    - name: project
      scope: repo
      backend: sqlite
      max_bytes: 5MiB
      write_policy: approval_required
      trust: reviewed

    - name: component
      scope: path
      backend: code_index
      key_by: package_or_service
      retrieval:
        mode: semantic_plus_symbol_graph
        max_context_tokens: 12000

    - name: skills
      scope: org
      backend: skill_registry
      write_policy: quarantine_then_review
      activation: progressive_disclosure

    - name: failures
      scope: repo
      backend: sqlite
      retention: 180d
      retrieval:
        when:
          - similar_error
          - same_test
          - same_file_cluster

  session:
    freeze_on_start:
      - project
      - user
      - org_policy
    allow_mid_session_refresh:
      - task
      - run
    preserve_across_hard_clear:
      - objective
      - accepted_plan
      - current_diff_summary
      - unresolved_risks

  writes:
    require_evidence_for:
      - project
      - component
      - skills
    deny_model_only_claims: true
    poisoning_checks:
      - prompt_injection_patterns
      - conflicting_policy
      - untrusted_tool_output
      - secret_like_content
```

The most important idea: **memory writes should require evidence.**

Bad:

> “Remember that package X is deprecated.”

Good:

```yaml
memory_write:
  claim: "Package X is deprecated in this repo."
  evidence:
    - file: "docs/architecture.md"
      lines: [120, 132]
    - command: "grep -R \"do not use package-x\" docs/"
      exit_code: 0
  scope: project
  confidence: 0.82
```

For “vibe coding” reduction, this is enormous. The model should not be allowed to create persistent project lore from vibes.

# 9. Upgrade 5 — add a skill system, but make it safer than Hermes

Hermes’ skill model is powerful because it uses progressive disclosure: small skill index, full skill on demand, deeper reference files only when needed. ([Hermes Agent][7])

ZYAL should adopt this, but with stricter validation:

```yaml
skills:
  enabled: true
  registry:
    backend: filesystem
    path: ".opencode/skills"
    source_of_truth: git
  activation:
    mode: progressive_disclosure
    index_budget_tokens: 3000
    load_full_skill_when:
      - task_matches
      - toolset_available
      - host_approved
  creation:
    allow_agent_proposals: true
    write_to: quarantine
    require:
      - example_usage
      - validation_command
      - failure_modes
      - rollback_instructions
      - scope
    promote:
      gates:
        - human_review
        - eval_passed
        - no_secret_leak
  versioning:
    require_semver: true
    signed: true
    changelog_required: true
  execution:
    skills_can_call_tools: false
    skills_are_instructions_only: true
```

ZYAL should separate:

* **Skill as knowledge/procedure.**
* **Tool as executable capability.**
* **Hook as host-executed command.**
* **MCP server as external capability.**

Do not let a skill secretly become a tool.

# 10. Upgrade 6 — make planning/test-first discipline structural

To reduce “vibe coding” and janky output, ZYAL needs a first-class plan contract.

Current ZYAL can run tests and checkpoint, but it does not require the model to prove that it understands:

* What is broken?
* What files are likely affected?
* What test will fail before the fix?
* What is the minimal diff?
* What is the rollback plan?
* What alternatives were rejected?
* What risk does the change carry?
* What would count as success?

Suggested schema:

```yaml
planning:
  mode: plan_then_execute       # direct | plan_then_execute | test_first | investigate_then_plan
  require:
    - problem_statement
    - affected_files_estimate
    - affected_symbols
    - test_plan
    - rollback_plan
    - risk_assessment
    - minimal_diff_strategy
  approval:
    required_when:
      - risk_score_gte: 0.5
      - files_changed_gte: 20
      - touches:
          - "migrations/**"
          - "auth/**"
          - "billing/**"
    approvers:
      - role: maintainer
  anti_jank:
    deny_implementation_without_plan: true
    deny_test_deletion_without_approval: true
    deny_skip_or_only_tests: true
    require_failing_test_before_fix:
      when: bugfix
    max_unexplained_diff_lines: 200
```

This should integrate with `assertions`:

# 11. Upgrade 7 — add disjointed experiments and tournament selection

Your incubator is already strong, but it needs a sibling: **experiments**.

Why? Because hard coding tasks often fail when the agent prematurely commits to one hypothesis. The best results come from multiple isolated attempts:

* One worker tries minimal patch.
* One worker tries architectural cleanup.
* One worker tries test-driven rewrite.
* One worker tries dependency-level fix.
* A critic compares them.
* The host picks based on evidence.

Suggested schema:

```yaml
experiments:
  refactor_tournament:
    strategy: parallel_disjoint       # parallel_disjoint | sequential_stateful | ablation | tournament
    objective: "Find the safest patch that fixes the target tests with minimal architectural disruption."
    max_candidates: 5
    isolation: git_worktree
    inherit:
      - accepted_plan
      - task_memory
      - project_memory
    diversify:
      dimensions:
        - patch_size
        - implementation_strategy
        - dependency_strategy
        - test_strategy
      require_distinct_hypotheses: true

    candidates:
      - id: minimal
        agent: build
        instruction: "Smallest safe change."
      - id: test_first
        agent: build
        instruction: "Write/identify failing test first, then patch."
      - id: architecture
        agent: plan
        instruction: "Consider whether the failure indicates a design boundary issue."
      - id: dependency
        agent: build
        instruction: "Investigate dependency/version/configuration causes."

    score:
      metrics:
        - name: target_tests_pass
          weight: 0.30
          source: shell
          command: "bun test --timeout 30000"
        - name: typecheck_pass
          weight: 0.20
          source: shell
          command: "npx tsgo --noEmit"
        - name: diff_size
          weight: -0.10
          source: git_diff
          measure: changed_lines
        - name: risk_score
          weight: -0.20
          source: critic
        - name: maintainability
          weight: 0.20
          source: reviewer
          judge: llm
          rubric: maintainability_v1

    reduce:
      strategy: best_evidence
      require:
        - tests_passed
        - no_guardrail_violations
        - rollback_known
      on_tie: prefer_smaller_diff
      merge: cherry_pick_selected

    memory:
      record_failed_hypotheses: true
      summarize_to_task_memory: true
```

This is one of the most important upgrades. The agent world often over-indexes on “one agent thinking longer.” ZYAL should make **multiple disjoint attempts** cheap, bounded, and evidence-scored.

# 12. Upgrade 8 — distinguish stateful building from disjoint experiments

You specifically called out “disjointed experiments, and stateful building on current state.” ZYAL should support both because they solve different problems.

| Mode                 | Best for                                                     | Risk                                 |
| -------------------- | ------------------------------------------------------------ | ------------------------------------ |
| Disjoint experiments | Unclear solution, high uncertainty, many possible approaches | Merge complexity                     |
| Stateful building    | Known direction, many incremental steps                      | Local optimum / compounding mistakes |

Suggested schema:

```yaml
strategy:
  mode: hybrid
  phases:
    - name: explore
      experiment: refactor_tournament
      stateful: false
    - name: commit_direction
      approval: maintainer
    - name: build
      stateful: true
      loop:
        max_iterations: 10
        checkpoint_every_verified_change: true
    - name: harden
      experiment: critic_sweep
      stateful: false
```

This prevents the common failure pattern:

> Agent makes a questionable early choice, then spends 100 iterations rationalizing it.

ZYAL should deliberately alternate between:

* **Exploration**: isolated, divergent, parallel.
* **Exploitation**: stateful, incremental, verified.
* **Critique**: external review, adversarial.
* **Compression**: durable summary, memory update.

```yaml
tools:
```

```yaml
      enabled: true
      engine: ripgrep
```

# 31. Specific issues in the current ZYAL draft

These are not philosophical concerns; they are spec-hardening items.

## 31.1 Optional key count mismatch

The draft says “Optional Keys — 21 total top-level keys,” but the visible optional list appears shorter than that. The spec should avoid prose counts and instead publish:

* JSON Schema.
* Canonical normalized config.
* Versioned schema URL.
* Strict unknown-key behavior.
* Extension namespace rule.

Example:

```yaml
schema:
  id: "https://opencode.ai/zyal/v1/schema.json"
  unknown_keys: reject
  extensions:
    allow_namespaced: true
    prefix_required: "x-"
```

## 31.2 `stop.all` and `stop.any` semantics need precision

If both are present, does the daemon stop when:

* `all(all)` is true AND `any(any)` is true?
* `all(all)` is true OR `any(any)` is true?
* `all` is mandatory and `any` is additional?

Define this formally.

Suggested:

```yaml
stop:
  mode: all_and_any   # all_only | any_only | all_and_any | expression
```

Or better:

```yaml
stop:
  expression:
    and:
      - condition: tests_pass
      - or:
          - condition: git_clean
          - condition: no_open_tasks
```

## 31.3 Duration grammar

Define `5s`, `10m`, `2h`, ISO-8601 durations, or one canonical format. Do not leave duration parsing to each host.

## 31.4 Shell assertions need schemas

Current `assert: { json }` is underspecified. Add:

```yaml
assert:
  json:
    path: "$.status"
    equals: "ok"
```

or:

```yaml
assert:
  json_schema_ref: zyal.schemas.TestSummary
```

## 31.5 Guardrail scope default

When `scope` is omitted, does a guardrail apply to tool input, output, file diff, commit message, or all? This must be explicit.

## 31.6 Commit message defaults

Examples omit `commit_message` while the reference shows it. Decide if it is required or defaulted.

## 31.7 Agent names and capabilities

`agent: plan` and `agent: build` need a registry:

```yaml
agents:
  registry:
    plan:
      model_role: planner
      tools: [read, search]
    build:
      model_role: builder
      tools: [read, search, edit, shell]
```

## 31.8 Handler cascade rule should be in schema

The safety invariant says on-handlers do not cascade. Great. Encode it:

```yaml
on:
  cascade: deny
  max_depth: 1
```

## 31.9 Preview should produce a canonical Run Card

Preview should not merely show parsed YAML. It should show:

* Objective.
* Effective permissions.
* Runtime.
* Network.
* Secrets.
* Tool access.
* Budgets.
* Approval gates.
* Stop conditions.
* Checkpoint commands.
* Guardrail summary.
* Risk score.
* Host capability mismatches.
* Unsupported keys.
* Exact hash of canonical manifest.

Suggested:

```yaml
preview:
  run_card:
    include:
      - effective_permissions
      - risk_score
      - budget
      - runtime
      - approvals
      - guardrails
      - unsupported_features
      - manifest_hash
```

# 32. How ZYAL should answer “is this redundant with X?”

## Is ZYAL redundant with OpenHands?

No. OpenHands is a runtime/SDK for agents. ZYAL should be a **declarative runbook and policy manifest** that can target an OpenHands-like runtime.

OpenHands has event-sourced state, deterministic replay, typed tools, MCP, and workspace abstraction. ZYAL should absorb those concepts, but it can stay host/language level. ([arXiv][1])

**Best integration:** ZYAL → OpenHands adapter.

---

## Is ZYAL redundant with LangGraph?

No. LangGraph is a graph execution framework. ZYAL can compile `workflow` into LangGraph or use LangGraph-style semantics. LangGraph’s checkpointing, interrupts, replay, memory, and durable execution are features ZYAL should model declaratively. ([LangChain Docs][2])

**Best integration:** ZYAL `workflow` → graph runtime.

---

## Is ZYAL redundant with DeepAgents?

No. DeepAgents is an agent factory/runtime pattern with planning, subagents, filesystem context, sandboxed shell, memory, and HITL. ZYAL can configure when and how such agents run. ([LangChain Docs][3])

**Best integration:** ZYAL `agents`, `memory`, `skills`, and `runtime` map to DeepAgents constructs.

---

## Is ZYAL redundant with Hermes?

No. Hermes is a personal/product agent with memory, skills, cron, messaging, subagents, terminal backends, and a learning loop. ZYAL should borrow those primitives but remain a host-enforced execution contract. ([GitHub][5])

**Main ZYAL gap vs Hermes:** skills, cron, progressive memory, backends, and hardline security floors.

**Main ZYAL advantage:** formal preview/arm, promotion gates, daemon ledger, incubator, declarative stop/checkpoint semantics.

---

## Is ZYAL redundant with OpenClaw?

No. OpenClaw coordinates background coding sessions from chat and emphasizes worktrees, plan review, lifecycle, wake routing, verifier loops, and merge/PR decisions. ZYAL should define the formal policy those sessions obey. ([GitHub][17])

**Main ZYAL gap vs OpenClaw:** conversational lifecycle UX and merge/PR decision flow.

**Main ZYAL advantage:** portable, typed, host-enforced manifest.

---

## Is ZYAL redundant with Claude Code?

No. Claude Code is an agentic coding tool with hooks, subagents, permissions, checkpoints, and context management. ZYAL can become the cross-tool equivalent of a checked-in, host-enforced agent policy. ([Claude][9])

**Best integration:** ZYAL host invokes Claude Code as one backend while enforcing ZYAL permissions externally.

---

## Is ZYAL redundant with Codex?

No. Codex provides local/cloud coding agent execution. ZYAL can define bounds, evals, memory, and approvals around Codex tasks. ([OpenAI Developers][10])

**Best integration:** ZYAL creates/verifies Codex tasks and records evidence.

---

# 33. The killer feature: evidence-gated autonomy

The most powerful ZYAL idea should be:

> A coding agent cannot promote a change because it sounds good. It promotes only because evidence satisfies a host-checked contract.

That means every promotion should have an evidence bundle:

```yaml
evidence:
  required_for_promotion:
    - name: accepted_plan
      source: approval
    - name: affected_files_known
      source: repo_intelligence
    - name: tests_identified
      source: planning
    - name: targeted_tests_passed
      source: evals
    - name: typecheck_passed
      source: evals
    - name: no_guardrail_violations
      source: security
    - name: rollback_plan
      source: planning
    - name: unresolved_critical_objections
      source: critic
      assert:
        equals: 0
```

This generalizes your incubator readiness model across the entire language.

The host should maintain an evidence graph:

```yaml
evidence_graph:
  nodes:
    - claim
    - test_result
    - diff
    - approval
    - memory_write
    - eval_score
    - artifact
  edges:
    - supports
    - contradicts
    - supersedes
    - derived_from
```

For high-stakes changes, ZYAL should be able to say:

> “This change is blocked because the affected-tests evidence is missing.”

That is how you beat vague agent optimism.

# 35. ZYAL for “any chat prompt”

The sentinel idea is excellent. To make it universal, define a strict host protocol:

## Universal ZYAL extraction protocol

1. Host scans assistant/user messages for `<<<ZYAL ...>>>`.
2. Host validates sentinel IDs and schema.
3. Host rejects code-fenced ZYAL.
4. Host canonicalizes YAML to normalized manifest.
5. Host computes manifest hash.
6. Host renders Run Card preview.
7. Host waits for explicit separate arm action.
8. Host verifies `ZYAL_ARM` matches manifest ID/hash.
9. Host creates run token.
10. Host executes under host policy.

Add hash binding:

```text
<<<ZYAL v1:daemon id=fix-tests sha256=preview-only>>>
...
<<<END_ZYAL id=fix-tests>>>
```

After preview, host displays:

```text
ZYAL_ARM RUN_FOREVER id=fix-tests sha256=abc123...
```

This prevents a malicious/mistaken assistant from editing the YAML after preview while preserving the same ID.

Even stronger:

```yaml
arming:
  require_host_generated_token: true
  token_ttl: 10m
  bind_to:
    - manifest_hash
    - user_id
    - repo
    - base_sha
```

Then the actual arm sentinel becomes:

```text
ZYAL_ARM RUN_FOREVER id=fix-tests token=host-issued-token
```

That is more secure than relying only on ID.

# 36. What ZYAL should not do

To stay powerful and safe, ZYAL should avoid these traps:

1. **Do not become a prompt library.** It is a host-enforced contract.
2. **Do not let models self-arm.** Ever.
3. **Do not let skills become executable code without review.**
4. **Do not let memory overwrite policy.**
5. **Do not make shell the primary coding interface.** Add typed tools.
6. **Do not treat “tests pass” as sufficient for promotion.**
7. **Do not allow unbounded fan-out.**
8. **Do not make SQLite global enterprise truth for all runs.** Use SQLite per-run, replication for fleet.
9. **Do not hide unsupported features.** Preview must show host capability gaps.
10. **Do not let MCP tools float unpinned.** Pin capability hashes.
11. **Do not allow recursive scheduled autonomy.**
12. **Do not let model confidence count as evidence.**

Your `model_confidence_cap` is exactly right. I would go further:

```yaml
evidence:
  model_claims:
    default_weight: 0
    may_support:
      - summaries
      - hypotheses
    may_not_satisfy:
      - promotion_requirements
      - safety_requirements
      - eval_requirements
```

Model claims can suggest. Host evidence decides.

# 40. Final recommendation

ZYAL should position itself as:

> **The host-enforced control language for autonomous software engineering.**

Not “an agent framework.”
Not “a prompt format.”
Not “a YAML wrapper around Claude/Codex/OpenCode.”
Not “a swarm launcher.”

The winning design is:

1. **Portable** across chat prompts and agent hosts.
2. **Host-enforced** rather than model-enforced.
3. **Replayable** through event-sourced state.
4. **Evidence-gated** rather than confidence-gated.
5. **Memory-disciplined** rather than vector-store-chaotic.
6. **Experiment-friendly** rather than single-trajectory.
7. **Eval-driven** rather than test-only.
8. **Sandboxed** rather than permission-vague.
9. **Observable** rather than magical.
10. **Human-governed** at risk boundaries.

ZYAL’s current draft is a very strong start. The core insight is correct. The expansion path is to turn it into a full **AgentOps + Software Engineering + Safety manifest** that can run inside OpenCode first, then compile to OpenHands, LangGraph/DeepAgents, Claude Code, Codex, Hermes/OpenClaw-style hosts, and future agent systems.

The strongest single upgrade: **make every autonomous coding decision evidence-backed, replayable, and promotable only through host-checked gates.**

That is how ZYAL becomes the default language for ambitious autonomous coding.
