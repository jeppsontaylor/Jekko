Absolutely — ZYAL is **not redundant** if it stays focused on the thing most agent frameworks still do poorly: **a host-enforced, portable, declarative control plane for long-running autonomous software work**.

My strongest take: **ZYAL should not become another agent framework.** It should become the **Autonomous Software Delivery Contract** that can sit above Jekko, Codex, Claude Code, OpenHands, LangGraph, Hermes, OpenClaw, local shells, MCP, A2A, Temporal, Kubernetes, and CI. In other words: ZYAL should be the language that tells the host what the agent is allowed to attempt, how evidence must be gathered, how memory is governed, how experiments are isolated, how promotions happen, and how autonomy is bounded.

I did a broad online study across primary docs, open-source repos, research papers, and public posts. I would not honestly claim to have read every X, LinkedIn, Quora, Reddit, blog, repo, and paper ever written on the topic; the highest-confidence recommendations below are anchored in primary sources and representative public discussions.

# 1. Executive verdict

ZYAL v1 already has several excellent instincts:

1. **Host owns lifecycle.**
2. **Preview and arming are separate.**
3. **State is durable.**
4. **Work is bounded.**
5. **Promotion requires evidence.**
6. **Guardrails are structural, not prompt-only.**
7. **Incubation separates thinking from production writes.**
8. **SQLite mirror gives humans inspectability.**

That is a great start.

But ZYAL v1 is still mostly a **daemon loop language**. For the world’s most ambitious coding projects, it needs to become a **full autonomous engineering governance language**.

The biggest missing categories are:

| Missing area                                | Why it matters                                                                                                                                                                               |
| ------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Workflow durability beyond local SQLite** | Billion-scale engineering needs leases, replay, snapshots, sharding, queues, idempotency, retries, and distributed workers.                                                                  |
| **Typed memory governance**                 | Current `context` is too shallow. Agents need active memory, episodic memory, archival memory, task memory, decision memory, invalidation, provenance, and promotion rules.                  |
| **Code intelligence layer**                 | Shell commands are not enough. ZYAL needs LSP, AST, symbol graph, build graph, dependency graph, ownership, coverage, blame, test selection, runtime traces.                                 |
| **Experiment orchestration**                | Incubator is good, but needs patch tournaments, branch search, disjoint hypotheses, control groups, rollback proofs, and merge arbitration.                                                  |
| **Evidence bundles**                        | “Tests pass” is necessary but insufficient. Promotion needs structured evidence: changed files, affected graph, invariants, generated tests, risk deltas, rollout plan.                      |
| **Skill and micro-agent system**            | OpenHands microagents, Codex Skills, Claude subagents, Hermes skills, and OpenClaw skills all point toward progressive disclosure and reusable agent capabilities. ZYAL should govern these. |
| **Interop with MCP and A2A**                | ZYAL should treat MCP tools and A2A agent handoffs as policy-governed capabilities, not as opaque tool calls.                                                                                |
| **Fine-grained permissions**                | `shell: ask` is not enough. You need permissions by phase, agent, path, network target, secret scope, package manager, tool class, repo, and risk score.                                     |
| **Anti-vibe-coding controls**               | ZYAL needs explicit requirements for spec-first, test-first, review, semantic diff, traceability, and rollback.                                                                              |
| **Observability and audit**                 | Event logs are good; production-scale autonomy needs traces, spans, metrics, prompt/tool provenance, replay, redaction, and cost telemetry.                                                  |
| **Federated repository management**         | 1B+ line projects require shardable task graphs, monorepo build graph integration, repo registries, ownership routing, and dependency-aware scheduling.                                      |

The winning strategy is not “ZYAL vs OpenHands” or “ZYAL vs LangGraph.” It is:

> **ZYAL as the host-enforced policy and runbook layer that can compile to or drive Jekko, OpenHands, Codex, Claude Code, LangGraph, Temporal, CI, Kubernetes, and MCP/A2A tools.**

# 2. What the outside ecosystem teaches us

## 2.1 OpenHands

OpenHands is highly relevant because it is already thinking about software-agent infrastructure, not just prompts. The OpenHands SDK paper emphasizes production concerns such as sandboxed execution, lifecycle control, model-agnostic routing, REST/WebSocket interfaces, and event-sourced state with deterministic replay. It also highlights local-to-remote portability and integrations with VS Code, VNC, browser, and API interfaces. ([arXiv][1])

OpenHands also exposes a practical product direction: extensible agents, micro-agents, model-agnostic execution, GitHub/GitLab/Slack/API integrations, sandboxed Docker/Kubernetes runtimes, and cloud agents that can scale parallel work. ([OpenHands][2])

**What ZYAL should absorb from OpenHands:**

1. **Event-sourced replay**, not just “state in SQLite.”
2. **Remote execution and sandbox abstraction.**
3. **Workspace observability**, including browser, terminal, VNC, logs, and artifacts.
4. **Micro-agent triggers**: small, reusable, context-specific instructions and tools.
5. **API-first daemon control**, not just CLI.

**Where ZYAL can be stronger than OpenHands:**

OpenHands is an agent platform. ZYAL can be the **portable governance language** that works even outside OpenHands. ZYAL should define the run contract; OpenHands can be one executor.

---

## 2.2 LangChain / LangGraph

LangGraph’s core advantages are durable execution, human-in-the-loop control, memory, inspectable state, and deployability. Its docs explicitly describe durable execution, human inspection/modification of state, short-term and long-term memory, and debugging/observability through LangSmith. ([LangChain Docs][3])

LangGraph memory docs distinguish short-term memory stored in thread state from long-term memory across conversations, and they show production checkpointing backed by a database. ([LangChain Docs][4]) Its human-in-the-loop middleware can pause tool calls, persist state, and let humans approve, edit, reject, or respond. ([LangChain Docs][5])

**What ZYAL should absorb from LangGraph:**

1. **Explicit graph/workflow semantics**, not only loop semantics.
2. **State inspection and state editing by humans.**
3. **Human decisions beyond ask/allow/deny**: approve, edit, reject, respond, redirect.
4. **Memory as a first-class runtime component.**
5. **Checkpointing as a formal concept.**

**Where ZYAL can be stronger:**

LangGraph is a programming framework. ZYAL can be a **non-Turing-complete declarative contract** that is safer to embed in any chat prompt.

---

## 2.3 Hermes Agent

Assuming you mean the NousResearch Hermes Agent, Hermes is interesting because it leans hard into persistent agents: memory, skills, self-improvement, scheduled automations, communication channels, isolated subagents, and multiple sandbox backends. Its docs describe a self-improving agent that creates skills from experience, persists knowledge, searches past conversations, and can run through Telegram, Discord, Slack, WhatsApp, email, CLI, local/Docker/SSH/Singularity/Modal environments, and subagents. ([GitHub][6])

Hermes also has an environment framework for reinforcement learning, benchmarks, scored rollouts, and SWE-style reward functions, plus self-evolution using DSPy and GEPA to evolve skills, tool descriptions, prompts, and code. ([Hermes Agent][7])

**What ZYAL should absorb from Hermes:**

1. **Skill creation from experience.**
2. **Persistent personal/project memory.**
3. **Scheduled automations and background jobs.**
4. **Subagent concurrency controls.**
5. **Environment/evaluation loops that generate scored trajectories.**
6. **Self-improvement, but only through gated evaluation.**

**Where ZYAL should be stricter than Hermes:**

Hermes’s strength is adaptive agency. ZYAL’s strength should be **governed autonomy**. If ZYAL adds self-improving skills, it must require evaluation, provenance, rollback, and human approval before new skills affect production code.

---

## 2.4 OpenClaw

OpenClaw is relevant because it looks like a multi-interface agent workspace: routing, voice/talk mode, live canvas, tools, browser, cron, sessions, Discord/Slack, companion apps, onboarding, and skills. Its security docs also make an important point: main-session tools can run on the host with broad access, while sandboxed mode changes the risk posture. ([GitHub][8])

OpenClaw also has a coding-agent skill that delegates work to Codex, Claude Code, Jekko, and other coding agents via background processes. ([GitHub][9])

**What ZYAL should absorb from OpenClaw:**

1. **Channel adapters**: chat, Slack, Discord, CLI, web UI.
2. **Cron/scheduled autonomous work.**
3. **Canvas/workspace views for humans.**
4. **Delegation to multiple coding agents.**

**Where ZYAL should be stricter:**

ZYAL should not merely “launch Codex/Claude/Jekko in the background.” It should wrap each delegated agent with a **capability envelope**: path scope, allowed tools, max tokens, max time, test requirements, artifact requirements, and promotion gates.

---

## 2.5 Codex and Claude Code

Codex CLI and Codex cloud show the local/cloud split ZYAL needs to support. Codex can run locally, read/change/run code, use sandboxing and approval modes, perform cloud tasks in parallel, and use AGENTS.md for layered instructions. Codex Skills also use progressive disclosure: the model initially sees only skill name/description/path, and reads full skill instructions only when needed. ([OpenAI Developers][10])

Claude’s Agent SDK exposes tools, hooks, subagents, MCP, permissions, and sessions. Claude subagents can have custom prompts, tool restrictions, permission modes, hooks, and skills; hooks can run shell commands, HTTP endpoints, or LLM prompts at lifecycle points. ([Claude][11])

**What ZYAL should absorb:**

1. **AGENTS.md / project instruction layering.**
2. **Skill packages with progressive disclosure.**
3. **Subagents with restricted tools and permissions.**
4. **Lifecycle hooks with structured JSON context.**
5. **Local/cloud execution modes.**
6. **Sandboxing and approval as separate concepts.**

**Where ZYAL should be stronger:**

Codex and Claude have their own runtime semantics. ZYAL can become the **cross-runtime declarative layer** that says: “regardless of whether this is Codex, Claude, Jekko, or OpenHands, here is the lifecycle, policy, memory, and evidence contract.”

---

## 2.6 SWE-agent, AutoCodeRover, RepairAgent, R2E, Agentless

The research trend is clear: better coding agents are not just “chat harder.” They improve by giving the model better interfaces, better code search, better validation, and better repair loops.

SWE-agent’s paper argues that Agent-Computer Interfaces matter: LM-centric commands, feedback formats, linting on edit, and file viewers improve coding-agent performance. ([arXiv][12])

AutoCodeRover uses program-structure-aware search, AST-based retrieval, class/method APIs, and fault localization. Its paper reports improved SWE-bench performance through iterative context retrieval and localization. ([arXiv][13])

RepairAgent emphasizes a finite-state machine for interleaving information gathering, repair, and validation, and reports repairing 164 bugs, including 39 not previously fixed by prior tools. ([arXiv][14])

R2E argues that static benchmarks are insufficient because real codebases involve dependencies, execution environments, and runtime traces; it turns GitHub repos into executable programming-agent environments with harnesses, tests, sliced context, coverage, and traces. ([r2e.dev][15])

Agentless is a useful counterexample: it achieved strong SWE-bench Lite results with a simpler, interpretable approach, reminding us that orchestration complexity must be justified by evidence. ([arXiv][16])

**What ZYAL should absorb:**

1. **Code-search APIs, not just shell.**
2. **AST, LSP, symbol, and dependency graph awareness.**
3. **Fault localization.**
4. **Finite-state repair loops.**
5. **Executable environments and evaluation harnesses.**
6. **Simplicity bias when possible.**

**The upgrade implication:**

ZYAL needs a `code_intel` block and a `repair_loop` / `experiments` block. Without those, it will remain a safe daemon runner, not a world-class autonomous engineering language.

# 3. ZYAL’s strongest positioning

ZYAL should be positioned as:

> **A portable, declarative, host-enforced control language for autonomous software delivery.**

Not:

* a prompt format,
* a swarm launcher,
* a workflow framework,
* a memory database,
* an IDE,
* an agent model,
* a CI system,
* a benchmark harness.

Instead, ZYAL should be the thing that **binds those systems together safely**.

A good mental model:

| Layer             | Examples                                      | ZYAL role                                    |
| ----------------- | --------------------------------------------- | -------------------------------------------- |
| Model             | GPT, Claude, Gemini, Qwen, local models       | Select, route, constrain, compare            |
| Coding agent      | Jekko, Codex, Claude Code, OpenHands       | Govern lifecycle and tools                   |
| Workflow engine   | SQLite loop, LangGraph, Temporal, DBOS        | Declare durable execution policy             |
| Tool protocol     | MCP, A2A, shell, HTTP                         | Gate capabilities and evidence               |
| Code intelligence | LSP, AST, Bazel, Nx, Pants, coverage          | Require scoped context and affected analysis |
| Verification      | tests, static analysis, fuzzing, mutation, CI | Define promotion gates                       |
| Human governance  | approvals, reviewers, owners, audit           | Enforce critical gates                       |
| Delivery          | branch, PR, release, rollout, rollback        | Govern promotion and deployment              |

# 4. Major missing block: `runtime`

Current ZYAL says SQLite is truth. That is excellent for local runs and inspectability, but not enough for 1B+ line projects or thousands of concurrent daemons.

Temporal’s docs describe workflow executions as durable, reliable, scalable, and capable of running from seconds to years; state is recovered through event history and replay. ([Temporal Docs][17])

ZYAL should keep SQLite as a default local ledger, but allow pluggable runtime engines.

### Proposed syntax

```yaml
runtime:
  engine: sqlite_local        # sqlite_local | postgres | temporal | dbos | k8s_queue
  event_model: event_sourced
  replay: deterministic

  leases:
    enabled: true
    heartbeat: 30s
    expire_after: 2m
    steal_on_expire: true

  idempotency:
    required: true
    key_template: "{{run.id}}:{{step.id}}:{{input.hash}}"
    reuse_completed_steps: true

  snapshots:
    every_events: 500
    compact_after_events: 50000
    include:
      - task_graph
      - memory_index_refs
      - open_worktrees
      - approvals
      - evidence_bundles

  concurrency:
    max_active_tasks: 50
    max_active_workers: 200
    max_repo_locks: 20
    backpressure:
      when_queue_depth_gte: 1000
      action: pause_low_priority

  mirror:
    filesystem: ".jekko/daemon/{{run.id}}"
    regenerate_from_state: true
    include_debug_artifacts: ask
```

### Why this matters

For small repos, SQLite is perfect. For billion-line projects, you need:

* durable replay,
* worker leases,
* crash recovery,
* queue backpressure,
* idempotent steps,
* partial replay,
* event compaction,
* state snapshots,
* distributed scheduling,
* resumable tasks.

ZYAL should preserve the principle:

> **The host owns the loop.**

But the host should be allowed to be a local Jekko process, a Kubernetes runner, a Temporal workflow, or a CI-integrated daemon cluster.

# 5. Major missing block: `workspace`

ZYAL currently has tasks, checkpoints, agents, and permissions, but not a rich workspace model.

OpenHands emphasizes local-to-remote execution, sandboxed environments, Docker/Kubernetes, WebSockets, VNC monitoring, browser/API interfaces, and isolated execution. ([arXiv][1])

ZYAL needs to describe where work happens.

### Proposed syntax

```yaml
workspace:
  root: repo
  mode: local_or_remote       # local | remote | local_or_remote

  environments:
    default:
      type: docker            # host | docker | devcontainer | kubernetes | ssh | modal
      image: "ghcr.io/org/project-dev:latest"
      network:
        egress: restricted
        allow:
          - "registry.npmjs.org"
          - "api.github.com"
      resources:
        cpu: 8
        memory: 16Gi
        disk: 100Gi
      secrets:
        mount: ask
        allow:
          - GITHUB_TOKEN_READONLY
        deny:
          - PROD_DATABASE_URL

    prototype:
      type: git_worktree
      base: "origin/main"
      cleanup: summarize_and_delete

  repo:
    base_ref: "origin/main"
    branch_prefix: "zyal/{{run.id}}/"
    require_clean_start: true

  terminals:
    allow_background_processes: false
    capture_stdout: true
    capture_stderr: true
    redact_secrets: true

  browser:
    enabled: ask
    record_session: true

  artifacts:
    path: ".jekko/daemon/{{run.id}}/artifacts"
    max_size: 2Gi
```

### Key upgrade

Separate:

* **where code is edited**,
* **where commands run**,
* **where prototypes run**,
* **where tests run**,
* **where secrets may be mounted**,
* **where artifacts are stored**.

That avoids the common failure mode where an agent tests in one environment but commits code intended for another.

# 6. Major missing block: `memory`

This is probably the most important upgrade.

Current ZYAL has:

```yaml
context:
  strategy: hybrid
  compact_every: 10
  hard_clear_every: 20
  preserve:
    - objective
    - progress
```

That is not enough.

Modern agent memory systems distinguish between pinned/core memory, conversation state, episodic memory, semantic archival memory, and structured knowledge. Letta describes memory blocks as structured, persistent, always-visible context sections that can be shared or marked read-only, and archival memory as a semantically searchable database queried on demand rather than pinned in context. ([Letta Docs][18]) Letta’s memory taxonomy also distinguishes message buffer, core memory, recall memory, and archival memory. ([Letta][19])

Zep’s Graphiti-backed memory focuses on temporal knowledge graphs, historical relationships, and context assembly. ([arXiv][20]) Mem0 similarly positions itself as a self-improving memory layer for agents. ([Mem0][21])

ZYAL should not pick one memory vendor. It should define the **memory contract**.

### Proposed syntax

```yaml
memory:
  enabled: true

  active:
    max_tokens: 12000
    blocks:
      - id: objective
        source: job.objective
        visibility: always
        write: read_only
        max_tokens: 600

      - id: current_plan
        visibility: always
        write: host_review
        max_tokens: 1500
        update_when:
          - plan_changed
          - checkpoint_accepted

      - id: risk_register
        visibility: always
        write: host_review
        max_tokens: 1200
        schema:
          type: array
          items:
            type: object
            required: [risk, severity, mitigation]

      - id: repo_map
        visibility: retrieve_on_need
        write: code_intel_only
        max_tokens: 3000

      - id: decisions
        visibility: always
        write: append_only
        max_tokens: 2000

  episodic:
    store: sqlite
    captures:
      - failed_attempts
      - test_failures
      - reviewer_feedback
      - tool_errors
      - reversions
      - successful_patterns
    summarize:
      every_events: 50
      preserve_raw: true
      include_links_to_events: true

  archival:
    store: local_graph_vector       # local_graph_vector | zep | letta | mem0 | custom_mcp
    retrieval:
      max_results: 12
      max_tokens: 5000
      require_provenance: true
      rerank: true
      staleness:
        revalidate_if_older_than: 14d
        invalidate_on:
          - file_deleted
          - symbol_removed
          - dependency_major_upgrade

  task_memory:
    capsule_template:
      include:
        - objective
        - affected_symbols
        - prior_attempts
        - known_bad_paths
        - accepted_constraints
        - tests_to_run
        - rollback_plan
    refresh:
      before_iteration: true
      before_promotion: true

  promotion:
    memory_write_requires:
      - provenance
      - evidence_link
      - confidence_lte: 0.8
    block_if:
      - contradicts_read_only_memory
      - missing_source_event
      - stale_reference
```

### Why this matters

Without memory governance, long-running agents create several failure modes:

1. **Context rot** — summaries slowly corrupt the real task.
2. **Stale facts** — old repo structure is treated as current.
3. **Memory poisoning** — failed guesses get preserved as facts.
4. **Decision loss** — the agent reopens decisions already made.
5. **Hidden assumption drift** — the agent forgets constraints.
6. **Repeated failures** — the same bad patch gets retried.

ZYAL should make memory **typed, scoped, provenance-backed, and invalidatable**.

# 7. Major missing block: `code_intel`

For ambitious coding projects, shell commands are too blunt. ZYAL needs a first-class code intelligence layer.

Bazel, Nx, and Pants show why build graphs matter: Bazel is built for large, multi-language workspaces and dependency graphs; Nx can run only affected tasks; Pants infers dependencies at file level for invalidation, caching, and concurrency. ([Bazel][22])

Research coding agents also point in the same direction: AutoCodeRover uses AST-aware search and fault localization; R2E emphasizes execution traces, coverage, harnesses, and sliced context. ([arXiv][13])

### Proposed syntax

```yaml
code_intel:
  enabled: true

  indexes:
    - id: lsp
      type: language_server
      languages: auto
      refresh: on_file_change

    - id: ast
      type: tree_sitter
      languages: auto
      include:
        - symbols
        - imports
        - call_graph
        - class_hierarchy

    - id: build_graph
      type: build_system
      providers:
        - bazel
        - nx
        - pants
        - package_json
        - cargo
        - go_mod
        - maven
        - gradle
      fallback: infer

    - id: ownership
      type: codeowners
      sources:
        - CODEOWNERS
        - git_blame
        - repo_metadata

    - id: runtime_traces
      type: traces
      sources:
        - coverage
        - logs
        - failing_test_stacktraces

    - id: dependency_risk
      type: dependency_graph
      include:
        - licenses
        - security_advisories
        - transitive_dependencies

  retrieval:
    strategy: affected_first
    max_context_tokens: 20000
    include:
      - direct_symbols
      - callers
      - callees
      - tests
      - configs
      - recent_changes
    exclude:
      - vendored
      - generated
      - build_artifacts

  affected:
    compute:
      - changed_files
      - imported_symbols
      - reverse_dependencies
      - test_coverage
      - ownership
    max_blast_radius:
      files: 200
      packages: 20
    on_exceed: require_plan_review

  fault_localization:
    enabled: true
    inputs:
      - failing_tests
      - stack_traces
      - recent_commits
      - coverage
    output_required:
      - suspected_files
      - suspected_symbols
      - confidence
      - alternative_hypotheses
```

### Key principle

ZYAL should require the agent to work from a **bounded affected graph**, not from vibes.

For example:

```yaml
constraints:
  - name: affected-scope-known
    check:
      code_intel:
        require:
          - affected_files
          - affected_tests
          - suspected_symbols
    invariant: non_empty
    on_violation: block
```

This is one of the strongest anti-jank upgrades.

# 8. Major missing block: `evidence`

Right now, checkpoint verification is mostly command-based. That is necessary but insufficient.

ZYAL should introduce a formal **evidence bundle** object. Every promotion, checkpoint, merge, release, or memory update should point to evidence.

### Proposed syntax

```yaml
evidence:
  bundles:
    checkpoint:
      require:
        - changed_files
        - diff_summary
        - affected_graph
        - tests_run
        - test_results
        - static_analysis
        - risk_assessment
        - rollback_plan
      optional:
        - coverage_delta
        - benchmark_delta
        - reviewer_notes
        - screenshots
        - trace_links

    promotion:
      require:
        - checkpoint
        - acceptance_criteria
        - invariant_results
        - unresolved_objections
        - owner_approval_if_required
        - dependency_impact
        - release_notes

  provenance:
    include:
      - model
      - agent
      - prompt_hash
      - tool_calls
      - command_outputs
      - environment_hash
      - source_refs
      - memory_refs

  redaction:
    secrets: true
    pii: ask
```

### Promotion should become evidence-driven

```yaml
promotion:
  promote_at: 0.78
  require_evidence:
    - tests_identified
    - tests_passed
    - scope_bounded
    - rollback_known
    - affected_files_known
    - no_unresolved_critical_objections
  block_on:
    - missing_evidence
    - stale_memory
    - affected_scope_exceeds_limit
    - owner_approval_missing
```

This is how ZYAL stops “the model says it is done” from becoming “the work is done.”

# 9. Major missing block: `experiments`

The incubator is one of ZYAL’s best ideas. But it should go further.

The best long-running agents will not do one linear attempt. They will run **disjoint experiments**, compare them, merge the best, and learn from failures.

Research supports this direction. Tree of Thoughts showed the value of exploring and evaluating multiple reasoning paths instead of one path. ([arXiv][23]) Self-Refine showed iterative feedback/refinement can improve outputs without extra training. ([arXiv][24]) Reflexion showed verbal feedback stored in episodic memory can improve future decisions. ([arXiv][25]) AlphaCodium improved code-generation results through a multi-stage, test-based, code-oriented iterative flow. ([arXiv][26])

ZYAL’s incubator should become a general **experiment portfolio** system.

### Proposed syntax

```yaml
experiments:
  enabled: true
  strategy: portfolio_search       # portfolio_search | branch_and_bound | tournament | ablation

  trigger_when:
    any:
      - risk_score_gte: 0.6
      - failing_attempts_gte: 2
      - affected_files_gte: 20
      - architecture_change: true
      - human_requests_alternatives: true

  variants:
    count: 5
    diversity:
      - minimal_patch
      - test_first
      - architectural
      - dependency_free
      - rollback_safe
    require_unique_hypothesis: true
    forbid_shared_scratch: true

  isolation:
    mode: git_worktree
    base_ref: "{{checkpoint.base_ref}}"
    cleanup: archive_best_and_summarize_rest

  scoring:
    normalize: true
    criteria:
      - id: tests
        weight: 0.30
        command: "bun test --timeout 30000"
        assert: { exit_code: 0 }

      - id: typecheck
        weight: 0.15
        command: "npx tsgo --noEmit"
        assert: { exit_code: 0 }

      - id: minimality
        weight: 0.15
        metric: diff_size_inverse

      - id: affected_scope
        weight: 0.15
        metric: affected_graph_size_inverse

      - id: maintainability
        weight: 0.15
        reviewer: critic_agent

      - id: rollback_safety
        weight: 0.10
        require: rollback_plan

  reduce:
    strategy: best_score_with_critic_review
    require_margin_gte: 0.08
    if_no_clear_winner: synthesize_hybrid

  memory:
    record_losers: true
    record_failure_modes: true
    promote_success_pattern: host_review
```

### Why this is powerful

For hard coding tasks, the first plausible patch is often not the best patch. ZYAL should let the host say:

* “Try five materially different solutions.”
* “Keep them isolated.”
* “Score them on tests, minimality, blast radius, maintainability, and rollback.”
* “Require a critic review.”
* “Merge only the best or synthesize.”
* “Record why the losers failed.”

That is how ZYAL becomes less vibe-driven and more science-driven.

# 10. Major missing block: `skills`

ZYAL should not stuff every instruction into context. Codex Skills use progressive disclosure; the agent initially sees just the skill metadata and reads the full instructions only when selected. ([OpenAI Developers][27]) OpenHands has micro-agent concepts with triggers and instructions. ([GitHub][28]) Hermes creates and persists skills from experience. ([GitHub][6]) Claude subagents and skills also point toward reusable, scoped capabilities. ([Claude][29])

ZYAL should add governed skills.

### Proposed syntax


# 11. Major missing block: `agent_graph`

Current `agents` is useful but too simple. Long-running software work needs agent topology: planner, builder, critic, test author, security reviewer, migration expert, release manager, documentation agent, etc.

But unconstrained swarms are dangerous. ZYAL should support **bounded agent graphs**.

### Proposed syntax

```yaml
agent_graph:
  max_depth: 3
  max_total_agents: 20
  max_parallel_agents: 6

  nodes:
    planner:
      agent: plan
      model: reasoning_large
      tools: [read, code_intel, memory]
      writes: scratch_only

    builder:
      agent: build
      model: coding_large
      tools: [read, edit, shell, code_intel]
      writes: isolated_worktree

    critic:
      agent: critic
      model: reasoning_large
      tools: [read, code_intel, test_results]
      writes: scratch_only

    test_author:
      agent: build
      model: coding_medium
      tools: [read, edit, shell]
      writes: isolated_worktree

    security:
      agent: security_review
      model: reasoning_large
      tools: [read, code_intel, dependency_risk]
      writes: scratch_only

  edges:
    - from: planner
      to: builder
      when: plan_approved

    - from: builder
      to: test_author
      when: tests_missing

    - from: builder
      to: critic
      when: patch_ready

    - from: critic
      to: builder
      when: fixable_objections
      max_cycles: 2

    - from: critic
      to: promotion
      when: no_critical_objections

  delegation:
    require_task_contract: true
    child_timeout: 20m
    child_budget:
      tokens: 200000
      shell_commands: 80
      file_writes: 40
```

### Why this is better than swarm syntax

This is not “spawn agents until something works.” It is:

* typed,
* bounded,
* replayable,
* budgeted,
* inspectable,
* phase-aware,
* evidence-driven.

# 14. Major missing block: `permissions` v2

Current permissions are useful:

```yaml
permissions:
  shell: ask
  edit: allow
  git_commit: ask
  git_push: deny
```

But ambitious projects need fine-grained capability control.

Also, Jekko has already seen concerns that MCP tools can bypass normal permission filtering if not handled carefully. ([GitHub][30]) OpenAI Agents SDK docs also show that guardrails can have scoped limitations around tools and handoffs. ([OpenAI GitHub][31])

ZYAL should introduce a universal **capability gateway** that applies to:

* shell,
* file edits,
* MCP tools,
* A2A handoffs,
* browser actions,
* network,
* secrets,
* package installs,
* subagents,
* background processes,
* git operations,
* cloud operations.

### Proposed syntax

```yaml
permissions:
  default: deny

  capabilities:
    read_repo:
      allow: true

    edit_source:
      allow: true
      paths:
        allow:
          - "src/**"
          - "packages/**"
        deny:
          - "infra/prod/**"
          - ".github/workflows/deploy.yml"

    shell:
      mode: ask
      allow_patterns:
        - "bun test*"
        - "bun run lint*"
        - "npx tsgo --noEmit*"
        - "git diff*"
        - "git status*"
      deny_patterns:
        - "rm -rf *"
        - "curl * | sh"
        - "git push --force*"

    network:
      mode: restricted
      allow_hosts:
        - "registry.npmjs.org"
        - "api.github.com"
      deny_hosts:
        - "*"

    secrets:
      mode: ask
      allow:
        - GITHUB_TOKEN_READONLY
      deny:
        - PROD_DATABASE_URL
        - STRIPE_SECRET_KEY

    mcp:
      mode: gateway
      servers:
        filesystem:
          tools:
            read_file: allow
            write_file: ask
        github:
          tools:
            create_pr: ask
            merge_pr: deny

    a2a:
      mode: gateway
      handoffs:
        allow:
          - security-review-agent
          - docs-agent
        deny:
          - external-untrusted-agent

  by_phase:
    scout:
      edit_source: deny
      shell: allow_readonly
    prototype:
      edit_source: allow
      shell: ask
    promotion:
      edit_source: deny
      git_commit: ask
      git_push: deny

  by_agent:
    critic:
      edit_source: deny
    builder:
      edit_source: allow
    test_author:
      edit_source:
        paths:
          allow: ["test/**", "src/**/*.test.*"]
```

### Important principle

All tools should pass through the same policy gateway.

No “MCP exception.”
No “subagent exception.”
No “handoff exception.”
No “background shell exception.”

# 15. Major missing block: `evals`

ZYAL has `checkpoint.verify` and `guardrails.output`, but it needs a richer evaluation harness.

SWE-bench has expanded into Verified, Multilingual, Lite, Multimodal, and live/contamination-resistant variants. ([SWE-bench][32]) SWE-Bench Pro emphasizes long-horizon, enterprise, multi-file, multi-language tasks across many repos and languages. ([OpenReview][33])

ZYAL should support project-local evals, live evals, regression evals, generated tests, mutation tests, fuzzing, benchmark deltas, and domain-specific oracles.

### Proposed syntax

```yaml
evals:
  suites:
    unit:
      command: "bun test --timeout 30000"
      timeout: 60s
      required: true

    typecheck:
      command: "npx tsgo --noEmit"
      timeout: 60s
      required: true

    lint:
      command: "bun run lint"
      timeout: 60s
      required: true

    affected:
      command: "npx nx affected -t test --base={{base_ref}} --head=HEAD"
      timeout: 10m
      required: true

    generated_tests:
      enabled: true
      require_human_review: false
      store_under: "test/generated/zyal"
      mutation_check:
        enabled: ask
        min_kill_rate: 0.60

    fuzz:
      enabled: ask
      duration: 5m
      targets:
        - "src/parser/**"

    performance:
      enabled: ask
      command: "bun run bench"
      compare_to: "{{base_ref}}"
      max_regression_percent: 5

  promotion:
    require:
      - unit
      - typecheck
      - lint
      - affected
    allow_flaky:
      max_retries: 2
      require_flake_label: true

  failure_triage:
    classify:
      - test_regression
      - environment_failure
      - flaky_test
      - unrelated_failure
      - insufficient_harness
    unrelated_failure_policy: require_human_review
```

### Add “oracle quality”

For 1B+ projects, the agent must not only pass tests. It must know whether tests are meaningful.

```yaml
evals:
  oracle_quality:
    require:
      - failing_test_reproduced_before_fix
      - new_test_fails_on_base_when_bugfix
      - new_test_passes_on_patch
      - no_snapshot_only_assertions_without_review
    on_violation: block_promotion
```

This reduces the classic coding-agent failure mode: adding a weak test that always passes.

# 22. Anti-vibe-coding / anti-jankurai controls

Recent public posts around Jankurai describe realtime auditing of hundreds of vibe-coding failure modes across tools like Claude, Codex, Cursor-like systems, and Jekko. ([Reddit][37]) More broadly, vibe-coding discussions describe the risk of accepting AI-generated code without deep review, and recent research characterizes recurring issues around specification, reliability, debugging, latency, review burden, collaboration, stochasticity, and trust. ([Wikipedia][38])

ZYAL should make “anti-vibe” structural.

### Proposed block

```yaml
anti_jank:
  enabled: true

  require_before_write:
    - problem_statement
    - acceptance_criteria
    - affected_scope
    - test_strategy

  forbid:
    - unreviewed_large_diff
    - test_deletion_without_approval
    - snapshot_update_without_reason
    - dependency_upgrade_without_impact
    - touching_unrelated_files
    - silent_api_behavior_change
    - generated_code_without_marker
    - config_change_without_owner_review

  realtime_audits:
    - name: unrelated-change-detector
      check: semantic_diff
      on_fail: ask

    - name: weak-test-detector
      check:
        require_new_tests_fail_on_base: true
      on_fail: block_promotion

    - name: hallucinated-api-detector
      check:
        code_intel_require_symbol_exists: true
      on_fail: block

    - name: migration-danger
      deny_patterns:
        - "DROP TABLE"
        - "DROP COLUMN"
        - "DELETE FROM"
      scope: file_diff
      on_fail: require_named_reviewer

    - name: rollback-missing
      check:
        require: rollback_plan
      on_fail: block_promotion

  scoring:
    output:
      jank_score:
        scale: 0_to_100
        components:
          - unbounded_scope
          - weak_evidence
          - unrelated_diff
          - missing_tests
          - stale_memory
          - risky_dependency_change
          - unexplained_behavior_change
      max_allowed_for_checkpoint: 35
      max_allowed_for_promotion: 20
```

### The key anti-vibe rule

ZYAL should require:

```text
No production write without scope.
No checkpoint without evidence.
No promotion without tests.
No memory update without provenance.
No deployment without rollback.
```

# 23. Upgrade the incubator into a full maturation system

Your incubator is the most distinctive part of ZYAL. Keep it. Expand it.

Current incubator stages:

* scout,
* idea,
* strengthen,
* critic,
* synthesize,
* prototype,
* review,
* compress.

That is good. I would add:

| New pass         | Purpose                                                    |
| ---------------- | ---------------------------------------------------------- |
| `reproduce`      | Reproduce bug or failing behavior before patch.            |

| `postmortem`     | Record what worked/failed after task completion.           |

```yaml
        - suspected_files
```

# 24. Add `delivery`

ZYAL currently stops around checkpoint/commit. Big projects need PRs, merge queues, releases, rollout, canaries, and rollback.

### Proposed syntax

```yaml
delivery:

  pull_request:
```

# 31. Direct response to overlap concerns

## “Is ZYAL redundant with OpenHands?”

No — unless ZYAL tries to become OpenHands.

OpenHands is an agent platform and SDK with sandboxed execution, event-sourced state, remote runtimes, and integrations. ZYAL should become the **declarative control contract** that can use OpenHands as an executor. ([arXiv][1])

**Upgrade needed:** add `interop.adapters.openhands`, `runtime`, `workspace`, `microagents`, and event-sourced replay semantics.

---

## “Is ZYAL redundant with LangChain/LangGraph?”

No — unless ZYAL becomes a Python workflow framework.

LangGraph gives durable graph execution, human-in-the-loop, memory, and state inspection. ZYAL should describe those behaviors declaratively and compile to LangGraph when desired. ([LangChain Docs][3])

**Upgrade needed:** add `agent_graph`, `runtime.engine: langgraph`, state inspection, resumable steps, and typed human decisions.

---

## “Is ZYAL redundant with Hermes?”

No. Hermes is closer to a persistent self-improving agent/productivity system. ZYAL should absorb Hermes-like skills, persistent memory, scheduled jobs, and self-evolution, but govern them with evidence and approval. ([GitHub][6])

**Upgrade needed:** add `skills.creation`, `memory`, `scheduler`, `evals`, and skill promotion gates.

---

## “Is ZYAL redundant with OpenClaw?”

No. OpenClaw appears closer to an agent workspace with tools, canvas, voice, cron, sessions, channels, and coding-agent delegation. ZYAL should be the policy envelope around those capabilities. ([GitHub][8])

**Upgrade needed:** add `channels`, `scheduler`, `delegation`, and strict capability boundaries for background coding agents.

---

## “Is ZYAL redundant with Codex / Claude Code?”

No. Codex and Claude Code are execution surfaces. ZYAL can become the portable runbook embedded in prompts or files and interpreted by a host. Codex has AGENTS.md, skills, sandboxing, approvals, and cloud tasks; Claude has tools, hooks, subagents, skills, MCP, and permissions. ([OpenAI Developers][39])

**Upgrade needed:** add `interop.adapters.codex`, `interop.adapters.claude_code`, `skills`, `hooks`, `permissions.v2`, and `sandbox` mapping.

---

## “Is ZYAL redundant with MCP?”

No. MCP exposes tools/resources/prompts. ZYAL governs when, how, and under what evidence/permission those tools can be used. MCP is a capability protocol; ZYAL is a control policy. ([Model Context Protocol][35])

**Upgrade needed:** treat MCP as a first-class capability gateway.

---

## “Is ZYAL redundant with A2A?”

No. A2A lets agents communicate. ZYAL should govern which agents may communicate, what evidence must accompany handoffs, and what permissions are delegated. ([Google Developers Blog][36])

**Upgrade needed:** add `a2a` gateway, handoff evidence, delegated permissions, and audit logs.

# 33. Critical design principle: ZYAL must remain declarative

The biggest risk is making ZYAL too powerful in the wrong way.

Do not make ZYAL a programming language.

Avoid:

```yaml
script: |
  while true:
    if model_says_done():
      promote()
```

Prefer:

```yaml
promotion:
  require_evidence:
    - tests_passed
    - affected_scope_known
    - rollback_known
```

ZYAL should be:

* declarative,
* typed,
* schema-validatable,
* previewable,
* host-enforced,
* non-Turing-complete,
* auditable,
* portable.

Any dynamic behavior should happen through host-recognized primitives: evals, conditions, gates, workflows, handlers, and bounded experiments.

# 34. The most important architectural change

The biggest single upgrade is this:

> Replace “daemon loop with optional incubator” with “durable task graph with bounded loops, governed memory, evidence bundles, and promotion gates.”

In v1, the unit is mostly:

```text
run → iteration → checkpoint
```

In v2, the unit should be:

```text
run
  → task graph
    → work package
      → plan
      → attempts
      → experiments
      → patch candidates
      → evidence bundles
      → review
      → checkpoint
      → promotion
      → delivery
      → memory updates
```

That gives ZYAL enough structure to manage serious projects.

# 36. Final thesis

ZYAL’s core idea is excellent: **the host, not the model, owns autonomy**.

But to become the default language for the world’s most ambitious coding projects, ZYAL needs to evolve from:

> “A YAML daemon loop with stop conditions and checkpoints”

into:

> **A portable, host-enforced autonomous software delivery contract.**

The killer features should be:

1. **Evidence-first promotion.**
2. **Typed, provenance-backed memory.**
3. **AST/LSP/build-graph code intelligence.**
4. **Experiment portfolios and patch tournaments.**
5. **Anti-vibe-coding audits.**
6. **Fine-grained capability permissions.**
7. **MCP/A2A governance.**
8. **Skill systems with evaluation-gated self-improvement.**
9. **Distributed durable runtime support.**
10. **Federated repo/task/merge orchestration.**

The most important product decision is this:

> ZYAL should be the neutral control language that makes every agent runtime safer, more observable, more evidence-driven, and more ambitious.

That is how ZYAL can avoid being “yet another agent YAML” and become the language that lets autonomous coding systems operate for hours, days, or months without becoming unbounded, opaque, or dangerous.
