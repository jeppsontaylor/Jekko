# ZYAL deep expansion: make it the host-enforced operating language for long-running coding agents

My conclusion: **ZYAL is not redundant.** It is targeting a real missing layer. OpenHands, LangGraph, Hermes, OpenClaw, Claude Code, Codex, Kiro, SWE-agent, Magentic-One, Aider, and MCP each solve important parts of long-running agentic work, but none of them are exactly a **prompt-portable, host-enforced, declarative daemon runbook language**. ZYAL’s opportunity is to become the neutral control plane that any chat surface can preview, validate, arm, monitor, pause, fork, replay, and promote.

The most important upgrade is to define ZYAL as a **policy-and-orchestration DSL**, not another agent framework. It should compile down to OpenCode, OpenHands, LangGraph, Claude Code, Codex, OpenClaw, or custom runtimes. ZYAL should describe what is allowed, how work is decomposed, what evidence is required, how memory is written, how experiments are scored, how tasks are promoted, and when humans must intervene.

The core architecture should become:

> **Prompt-native runbook → host parser → typed IR → policy engine → sandbox/workspace runtime → event-sourced ledger → evidence-gated delivery.**

That is the wedge that makes ZYAL bigger than “yet another agent config file.”

---

## 1. What the outside systems teach ZYAL

### OpenHands: event-sourced software-engineering agent runtime

OpenHands is very close to ZYAL’s world, but from the other direction. It provides a composable SDK, GUI, CLI, cloud/enterprise deployment, sandboxes, tools, event streams, REST/WebSocket services, and model-agnostic execution. The OpenHands SDK paper emphasizes extensible agents, custom tools, memory management, local-to-remote portability, sandboxed execution, lifecycle control, model routing, and security analysis. Its architecture also includes an event-sourced state model with deterministic replay, typed tool contracts, workspace abstractions, and context condensation. ([arXiv][1])

**ZYAL lesson:** ZYAL should not try to replace OpenHands. ZYAL should be able to **drive** OpenHands. Add explicit event-sourcing, replay, workspace, sandbox, typed tool, and condensation semantics so an ZYAL run can be implemented by OpenHands or by OpenCode’s own runtime.

---

### LangGraph / LangChain: durable graph runtime, interrupts, memory, human-in-loop

LangGraph’s strength is durable execution: checkpointing, persistence, pause/resume, fault tolerance, human-in-the-loop interrupts, thread-scoped memory, long-term memory, and time travel. LangGraph checkpoints every super-step, supports pending writes for recovery, and lets humans approve, edit, reject, or respond to tool calls through interrupt middleware. ([LangChain Docs][2])

**ZYAL lesson:** ZYAL should gain **graph-aware durability** without becoming a graph programming language. Add a `state`, `replay`, `interrupts`, `approval`, and `memory` vocabulary. ZYAL should be able to compile a daemon run into a LangGraph graph, but its own syntax should remain declarative and host-enforced.

---

### Hermes Agent: memory, self-improving skills, cron, messaging, subagents

Hermes has a strong “always-on personal agent” design: persistent memory, full-text search over prior sessions, user modeling, cron jobs, Telegram and other messaging channels, real terminal backends, isolated subagents, MCP, skills, trajectory export, and RL-oriented data generation. Its memory design separates bounded always-in-context memory from on-demand SQLite/FTS session search. Its skills system uses progressive disclosure, `SKILL.md`, references, templates, scripts, security scanning, quarantine, and a curator that archives or consolidates agent-created skills over time. ([Hermes Agent][3])

**ZYAL lesson:** ZYAL is missing a first-class **procedural memory and skills lifecycle**. Add `memory` lanes and `skills` blocks with progressive loading, provenance, review, quarantine, curation, rollback, and security scans. Otherwise ZYAL daemons will repeat the same mistakes and will not compound capability.

---

### OpenClaw and OpenClaw Code Agent: chat-launched sessions, worktree lifecycle, plan approval

OpenClaw Code Agent is highly relevant because it runs Claude Code and Codex as managed background sessions from chat, adds plan approval, wake routing, worktree isolation, merge/PR follow-through, suspend/resume/fork/interrupt/recover, goal loops, output visibility, cost statistics, and a shared control plane over multiple backends. ([GitHub][4])

OpenClaw’s security docs are also useful because they explicitly distinguish personal assistant trust boundaries from hostile multi-tenant isolation, warn that shared bots with tool access create delegated-authority risk, and recommend strict allowlists, sandboxing, local-only gateways, per-channel-peer DM scope, and minimal tools. ([OpenClaw][5])

**ZYAL lesson:** ZYAL should add **chat/session lifecycle semantics**: plan-first launch, same-session revisions, wake routing, worktree decision states, PR lifecycle, safe cleanup, and cost/output/status surfaces. ZYAL also needs explicit trust-boundary modeling because “available inside any chat prompt” is powerful and dangerous.

---

### MCP, ACP, AGENTS.md: interoperability standards

MCP is becoming the tool/data integration substrate. The MCP spec defines tools, resources, prompts, sampling, roots, elicitation, and security principles around user consent, data privacy, tool safety, and explicit consent before tool invocation. Its roadmap includes server cards, retry semantics, task primitives, audit trails, enterprise auth, gateways/proxies, and config portability. ([Model Context Protocol][6])

ACP standardizes communication between code editors and coding agents. AGENTS.md is an open, Markdown-based project-instruction format with build/test commands, code style, testing instructions, security considerations, nested monorepo instructions, and a nearest-file-wins precedence model. ([GitHub][7])

**ZYAL lesson:** ZYAL should become the **runbook layer above MCP, ACP, AGENTS.md, SKILL.md, and chat prompts**. It should import context from AGENTS.md and steering docs, export agent configs to ACP-compatible clients, bind MCP tools with least privilege, and generate portable run manifests.

---

### Claude Code, Codex, Kiro: hooks, subagents, sandboxing, specs

Claude Code’s documentation stresses permissions, allowlists, sandboxing, deterministic hooks, skills, and subagents with separate contexts and tool permissions. Its hook reference lists lifecycle events such as session start/end, pre/post tool use, permission requests, subagent start/stop, worktree create/remove, pre/post compact, and file changes. ([Claude][8])

Anthropic’s long-running harness research is directly relevant: compaction alone is not enough; agents need initializer sessions, structured progress artifacts, incremental progress, context resets, planner/generator/evaluator splits, and explicit handoff artifacts to avoid half-implemented work and premature completion. ([Anthropic][9])

Codex’s sandbox docs make the same structural point as ZYAL: sandbox boundaries and approval policies work together, with commands inheriting sandbox limits and approvals requested when crossing them. Codex subagents inherit sandbox policy, and approval requests can surface from inactive threads. ([OpenAI Developers][10])

Kiro’s specs formalize requirements, design, and tasks; its task execution builds dependency graphs and runs independent tasks concurrently. Kiro steering files define product, stack, and structure context, with inclusion modes to avoid always loading everything. ([Kiro][11])

**ZYAL lesson:** Add first-class **spec gates, hook events, task DAGs, dependency waves, context resets, and multi-agent evaluator loops**. Also add conditional context loading; loading all context all the time is a failure mode.

---

### SWE-agent, Magentic-One, Reflexion, Voyager, Aider, TheAuditor

SWE-agent’s key insight is Agent-Computer Interface design: custom file viewers, concise grep, edit-time syntax checking, and LM-centric feedback formats can materially improve repository-level coding. ([Swe Agent][12])

Magentic-One uses an orchestrator that plans, delegates to specialized agents, tracks progress, updates task/progress ledgers, replans when progress stalls, and recommends containers, logs, human oversight, limited internet, and prompt-injection caution. ([Microsoft GitHub][13])

Reflexion shows that agents can improve by writing linguistic reflections into episodic memory rather than changing model weights. Voyager shows the power of automatic curricula, executable skill libraries, iterative prompting, environment feedback, execution-error feedback, and self-verification. ([OpenReview][14])

Aider’s repo map shows how large-codebase context can be compressed through graph-ranked symbol maps within token budgets. TheAuditor is a useful anti-vibe-coding example: it indexes code into SQLite, supports data-flow/taint analysis, architectural hotspot detection, deterministic queries, and incremental reindexing after edits. ([Aider][15])

**ZYAL lesson:** Add **agent-computer-interface contracts**, repository intelligence graphs, deterministic codebase query tools, reflection memory, automatic curriculum, and skill compounding.

## 2. The biggest gaps in current ZYAL

ZYAL already has excellent bones: host-owned loop, sentinel arming, SQLite truth, stop conditions, guardrails, constraints, retry, hooks, fan-out, multi-agent workers, and an incubator. The missing pieces are mostly about **durability, memory, interoperability, typed evidence, massive-codebase orchestration, and anti-jank enforcement**.

### Gap 1: Arming is not safe enough for “any chat prompt”

Current examples include:

```text
ZYAL_ARM RUN_FOREVER id=...
```

That is fine for documentation, but dangerous if ZYAL is truly available inside any chat prompt. A model, webpage, issue comment, README, Slack message, or tool output could emit a syntactically valid block.

**Upgrade:** ZYAL arming must be origin-aware and host-bound.

```yaml
arming:
  mode: human_only
  require_preview_hash: true
  require_host_nonce: true
  accepted_origins:
    - trusted_user_message
    - signed_cli_input
    - signed_api_request
  rejected_origins:
    - assistant_message
    - tool_output
    - webpage_content
    - repository_file
    - issue_comment
  arm_sentinel:
    format: "ZYAL_ARM RUN_FOREVER id={{id}} preview_sha256={{sha}} nonce={{host_nonce}}"
```

The rule should be absolute: **assistant output may propose ZYAL, but cannot arm ZYAL.** Tool output, webpages, docs, GitHub issues, and dependency READMEs must be treated as untrusted content. The host must parse only trusted-origin text.

---

### Gap 2: SQLite truth exists, but event-sourcing/replay is underspecified

“SQLite is truth” is good. But the language should define event types, idempotency, replay, snapshots, pending writes, fork semantics, and deterministic/non-deterministic boundaries.

Add:

```yaml
state:
  storage: sqlite
  event_sourcing:
    enabled: true
    append_only: true
    schema_version: zyal.events.v1
    deterministic_replay: true
    snapshot_every: 25
    pending_writes: recover
  idempotency:
    require_for:
      - shell
      - git_commit
      - git_push
      - mcp_tool
      - network
      - external_api
    key_template: "{{run_id}}:{{task_id}}:{{iteration}}:{{action_id}}"
  replay:
    allow:
      - dry_run
      - audit
      - fork_from_checkpoint
    redact:
      - secrets
      - tool_outputs_matching: ["AKIA", "sk-", "BEGIN PRIVATE KEY"]
  forks:
    enabled: true
    max_depth: 3
    require_reason: true
```

Why this matters: if a run fails six hours in, the host should know exactly which side effects happened, which observations were seen, which checkpoint produced the current worktree, and whether replay would re-run a network call or just restore the observation.

### Gap 3: Permissions are too coarse

Current:

```yaml
permissions:
  shell: ask
  edit: allow
  git_commit: ask
```

This is not enough for 1B+ codebases or enterprise environments. ZYAL needs **capability leases**, not broad permission toggles.

Add:

```yaml
capabilities:
  leases:
    - id: local-read
      grants:
        filesystem:
          read: ["."]
          write: []
      expires: on_stop

    - id: workspace-edit
      grants:
        filesystem:
          read: ["."]
          write: ["src/**", "tests/**"]
        deny:
          - ".env*"
          - "**/secrets/**"
          - "migrations/**"
      requires:
        approval: plan_approved
      expires: on_checkpoint

    - id: ci-readonly
      grants:
        shell:
          allow:
            - "bun test*"
            - "pnpm test*"
            - "npm test*"
            - "npx tsc*"
            - "git diff*"
            - "git status*"
          deny:
            - "rm *"
            - "curl * | sh"
            - "git push*"
```

Then bind capabilities to agents and passes:

```yaml
agents:
  supervisor:
    agent: plan
    capabilities: [local-read, ci-readonly]
  workers:
    - id: builder
      count: 4
      agent: build
      isolation: git_worktree
      capabilities: [local-read, workspace-edit, ci-readonly]
```

This lets the host enforce least privilege by role, pass, task, file scope, and time.

### Gap 4: No first-class memory architecture

ZYAL has `context`, but not durable memory. That is a major gap. For long-running agents, **context management is not memory management**. A 1M-token window still does not solve stale facts, irrelevant distractions, hallucinated state, lost decisions, or repeated mistakes.

Add memory lanes:

```yaml
memory:
  strategy: layered
  prompt_snapshot:
    mode: frozen_per_session
    include:
      - objective
      - current_task_capsule
      - active_plan
      - nearest_AGENTS_md
      - relevant_skills
    max_tokens: 16000

  lanes:
    working:
      scope: run
      store: sqlite
      purpose: "Current plan, progress, blockers, next action"
      compact_every: 5
      max_tokens: 4000

    episodic:
      scope: run_and_project
      store: sqlite_fts
      write:
        - event_summary
        - failure_reflection
        - checkpoint_summary
        - human_feedback
      retrieve:
        query: task_similarity
        max_results: 8
      distill:
        on:
          - checkpoint
          - error
          - no_progress
        require_structured_output: true

    semantic:
      scope: project
      store: vector_sqlite
      write_policy: host_reviewed
      facts:
        require_source: true
        require_confidence: true
        expire_after: 90d
      conflict_resolution: prefer_newer_verified_source

    procedural:
      scope: project_or_user
      format: SKILL.md
      progressive_disclosure: true
      create_when:
        - tool_calls_gte: 5
        - repeated_success_gte: 2
        - human_marks_reusable: true
      require_review_before_activation: true
      security_scan: required

    codebase:
      scope: repo
      indexes:
        - symbols
        - imports
        - call_graph
        - ownership
        - build_targets
        - test_coverage
        - dependency_graph
        - migration_history
        - architectural_decisions
      update: incremental_on_diff
      query_tool: repo_graph
```

Key rule: **memory writes must be evidence-based.** The model can propose memory entries, but the host should validate source, scope, expiry, and taint.

### Gap 5: No skills lifecycle

Hermes and Claude Code both show that skills are becoming a major primitive, but skills are also a major supply-chain risk. Snyk’s ToxicSkills study scanned 3,984 agent skills and reported 13.4% with critical issues and 36.82% with at least one security issue; it also highlights that skills often inherit shell, filesystem, credentials, messaging, and persistent memory capabilities. ([Snyk][16])

ZYAL needs a skill block that is safe by default:

```yaml
skills:
  enabled: true
  formats:
    - agentskills.io
    - claude_SKILL_md
    - hermes_SKILL_md
  sources:
    local:
      paths:
        - ".opencode/skills"
        - ".claude/skills"
        - ".hermes/skills"
      trust: project
    registry:
      allow: []
      default: deny

  loading:
    mode: progressive_disclosure
    index_budget_tokens: 1000
    full_skill_budget_tokens: 4000
    auto_include:
      require_match_score_gte: 0.75
      max_per_turn: 3

  creation:
    allow_agent_created: true
    write_to: ".opencode/skills/generated"
    triggers:
      - completed_task_with_reusable_workflow
      - human_correction_repeated
      - repeated_failure_resolved
    require:
      - usage_example
      - verification_command
      - known_pitfalls
      - rollback_notes

  security:
    scan_on_install: true
    scan_on_update: true
    quarantine_on:
      - prompt_injection
      - suspicious_download
      - credential_request
      - curl_pipe_shell
      - hidden_unicode
      - remote_instruction_loading
    activation_requires:
      - host_scan_passed
      - human_approval_for_new_skill
      - no_untrusted_setup_scripts

  curator:
    enabled: true
    cadence: weekly
    archive_unused_after: 45d
    consolidate_similar: true
    snapshot_before_mutation: true
    rollback: true
```

This turns skills into **auditable procedural memory**, not random prompt files with shell access.

### Gap 6: No project/spec gate

ZYAL’s `job.objective` is too thin for ambitious work. Kiro’s spec model is a good reference: requirements, design, tasks, acceptance criteria, sequence diagrams, and task tracking. ([Kiro][11])

Add:

```yaml
spec:
  mode: required_before_edit
  artifacts:
    requirements:
      file: ".opencode/specs/{{run_id}}/requirements.md"
      require:
        - user_stories
        - acceptance_criteria
        - non_goals
        - unchanged_behaviors
    design:
      file: ".opencode/specs/{{run_id}}/design.md"
      require:
        - architecture
        - affected_components
        - data_flow
        - error_handling
        - security_considerations
        - rollback_plan
    tasks:
      file: ".opencode/specs/{{run_id}}/tasks.md"
      require:
        - task_dag
        - estimated_risk
        - verification_per_task
  approval:
    before_first_edit: ask
    allow_quick_plan_for:
      risk_score_lt: 0.25
      diff_budget_files_lte: 3
```

This directly attacks vibe coding. The agent must not just “start coding.” It must produce a testable contract.

### Gap 7: Fan-out is too flat for billion-line repos

Current `fan_out` is map-reduce over simple item lists. For 1B+ LOC, you need **task graphs, dependency waves, locks, ownership, risk, and merge trains**.

Add:

```yaml
task_graph:
  source:
    - issues:
        provider: github
        query: "label:zyal-migration state:open"
    - shell:
        command: "opencode repo-graph affected --json"
  decomposition:
    strategy: architecture_dag
    max_depth: 5
    require_leaf_tasks:
      - bounded_files
      - clear_acceptance
      - verification_command
      - rollback_plan
  dependencies:
    infer_from:
      - import_graph
      - build_graph
      - ownership
      - test_coverage
      - migration_order
  scheduling:
    waves: true
    max_parallel_per_wave: 20
    risk_order: low_to_high
    critical_path_first: true
  locks:
    mode: pessimistic
    by:
      - file
      - package
      - migration_table
      - api_contract
    timeout: 30m
    on_conflict: park_and_replan
  ownership:
    require_owner_review_for:
      - auth/**
      - payments/**
      - migrations/**
      - infra/**
```

This makes ZYAL suitable for large-scale migration work where independent shards can run concurrently but shared interfaces and risky modules are protected.

### Gap 8: Incubator is strong, but experiments need a richer search model

Your incubator is the best part of the current design. But it mixes “make task clearer” with “try competing solutions.” Those should be separate.

Add a top-level `experiments` block:

```yaml
experiments:
  enabled: true
  mode: n_version_search
  trigger_when:
    any:
      - risk_score_gte: 0.65
      - unknown_solution_space: true
      - repeated_attempts_gte: 2
      - performance_sensitive: true

  variants:
    count: 5
    isolation: git_worktree
    diversity:
      prompts:
        - minimal_patch
        - architectural_refactor
        - test_first
        - compatibility_layer
        - dependency_free
      model_temperature: varied
      disallow_shared_context_between_variants: true

  baseline:
    command: "git checkout {{base_ref}} && bun test --timeout 30000"
    capture:
      - test_results
      - perf_metrics
      - bundle_size
      - changed_files

  score:
    rubric:
      functional_correctness: 0.35
      test_quality: 0.20
      maintainability: 0.20
      security: 0.15
      diff_minimality: 0.10
    commands:
      - id: tests
        command: "bun test --timeout 30000 --reporter=json"
        assert: { exit_code: 0 }
      - id: typecheck
        command: "npx tsc --noEmit"
        assert: { exit_code: 0 }
      - id: security
        command: "opencode audit --format json"
        assert:
          json:
            path: "$.critical"
            equals: 0

  reduce:
    strategy: synthesize_best
    require_human_review_if:
      variants_disagree_on_architecture: true
      best_score_lt: 0.75
    promote:
      require:
        - passing_tests
        - no_critical_security_findings
        - reviewer_summary
        - rollback_plan
```

This gives ZYAL a disciplined version of “try five approaches in parallel,” without merging random jank.

### Gap 9: No anti-jank / anti-vibe quality contract

The user’s “jankurai” reference is exactly the right instinct: agentic coding needs **runtime jank detection**, not vibes. Current community tools are starting to position themselves around realtime audits for “vibe coding” failure modes, but ZYAL should make this structural rather than a plugin afterthought. ([LinkedIn][17])

Add:

```yaml
quality:
  anti_jank:
    enabled: true
    mode: block_promotion
    checks:
      - id: no_unexplained_large_diff
        type: diff_budget
        max_files_changed: 25
        max_added_lines: 1500
        on_violation: require_plan_revision

      - id: no_test_deletion
        type: file_diff
        deny_patterns:
          - "^-.*it\\("
          - "^-.*test\\("
          - "^-.*describe\\("
        unless: "replacement_test_added"
        on_violation: block

      - id: no_silent_behavior_change
        type: semantic
        require:
          - changed_behavior_list
          - unchanged_behavior_list
          - backwards_compatibility_notes

      - id: no_todo_escape_hatches
        type: diff_pattern
        deny_patterns:
          - "TODO: fix later"
          - "temporary workaround"
          - "disable lint"
          - "skip\\("
          - "only\\("
        on_violation: block

      - id: defensive_programming
        type: rubric
        require:
          - input_validation
          - error_handling
          - timeout_or_cancellation
          - idempotency_if_retried
          - logging_without_secrets

  jank_score:
    command: "jankurai audit --json . || true"
    threshold:
      promote_max: 0.15
      warn_max: 0.30
    evidence_path: ".opencode/daemon/{{run_id}}/jank.json"
```

A recent arXiv paper on vibe-coding safety reports a gap between functional correctness and secure solutions in agent-generated code, including an example where 61% of solutions from one evaluated setup were functionally correct but only 10.5% were secure. This supports making security and quality gates promotion requirements, not advisory suggestions. ([arXiv][18])

### Gap 10: Verification is too shell-command-centered

Shell checks are necessary but not sufficient. ZYAL should distinguish:

* **Stop checks**: “Can the daemon stop?”
* **Checkpoint checks**: “Can this change be committed?”
* **Promotion checks**: “Can this task move to ready?”
* **Delivery checks**: “Can this branch become a PR or release?”
* **Regression checks**: “Did we preserve behavior?”
* **Evidence checks**: “Can a human audit what happened?”

Add:

```yaml
verification:
  strategy: evidence_driven
  test_selection:
    mode: impacted
    graph: codebase.test_coverage
    fallback: full_suite
    require_full_suite_before_delivery: true

  layers:
    static:
      - command: "npx tsc --noEmit"
      - command: "bun run lint"
    unit:
      - command: "bun test --unit --timeout 30000"
    integration:
      - command: "bun test --integration --timeout 60000"
        run_when:
          touched:
            - "src/api/**"
            - "src/db/**"
    security:
      - command: "opencode audit --critical-only --json"
        assert:
          json: { path: "$.critical", equals: 0 }
    performance:
      - command: "bun run bench --json"
        compare_to: baseline
        max_regression_pct: 5
    migration:
      - command: "bun run migrate:dry-run"
        run_when:
          touched: ["migrations/**", "schema/**"]

  flaky_tests:
    detection: rerun_failed
    max_reruns: 2
    classify:
      command: "opencode test classify-flaky --json"
    do_not_ignore_without_human: true

  evidence:
    require_for_checkpoint:
      - commands_run
      - stdout_tail
      - exit_codes
      - files_changed
      - tests_added_or_updated
      - risk_assessment
      - rollback_plan
    store_artifacts: true
    redact_secrets: true
```

This makes “tests passed” an evidence object, not just a line in terminal output.

## 3. Proposed new top-level ZYAL keys

Current ZYAL has 21 optional top-level keys. I would add these, grouped by purpose.

### Safety and trust

```yaml
surface:        # where ZYAL can be parsed from
arming:         # human-only arming with nonce/hash/origin
policy:         # org/repo/user policy layering
capabilities:   # least-privilege leases
security:       # sandbox, secrets, network, taint, supply-chain
```

### State, memory, context

```yaml
state:          # event sourcing, replay, snapshots, forks
memory:         # working/episodic/semantic/procedural/codebase lanes
skills:         # procedural memory lifecycle and security
knowledge:      # repo/project/org context sources
models:         # model routing, cost, fallback, role models
```

### Work planning and execution

```yaml
spec:           # requirements/design/tasks gate
task_graph:     # DAG, waves, locks, ownership
workspaces:     # containers, worktrees, remote sandboxes
experiments:    # N-version prototypes and scoring
curriculum:     # long-horizon automatic task sequencing
```

### Quality, observability, delivery

```yaml
verification:   # layered checks and evidence
quality:        # anti-jank, maintainability, review rubric
observability:  # traces, metrics, dashboards, alerts
delivery:       # PR, merge queue, release, canary, rollback
interop:        # MCP/ACP/AGENTS/SKILL/export adapters
```

## 4. A more powerful ZYAL v2 skeleton

This is the shape I would aim for:

```yaml
<<<ZYAL v2:daemon id=ambitious-project>>>
version: v2
intent: daemon
confirm: RUN_FOREVER

job:
  name: Ambitious project daemon
  objective: "Complete a high-risk codebase migration with evidence-gated delivery."
  risk:
    - "Large blast radius"
    - "Potential flaky tests"
    - "Potential security regressions"

surface:
  parse_from:
    - trusted_user_message
    - signed_cli_input
    - signed_api_request
  reject_from:
    - assistant_message
    - tool_output
    - webpage_content
    - repository_file
    - issue_comment

arming:
  mode: human_only
```

## 5. How ZYAL should manage memory better

The key is to stop treating memory as “more context.” Memory should be a typed, queryable, evidence-backed system.

### Recommended memory lanes

| Lane              |             Scope | Purpose                                              | Write policy                  |
| ----------------- | ----------------: | ---------------------------------------------------- | ----------------------------- |
| Working memory    |          Run/task | Current plan, next action, blockers                  | Automatic, compacted          |
| Episodic memory   |       Run/project | What happened, failures, reflections, human feedback | Automatic but summarized      |
| Semantic memory   |       Project/org | Stable facts, decisions, conventions                 | Evidence-backed, reviewed     |
| Procedural memory |      User/project | Reusable workflows and skills                        | Reviewed and security-scanned |
| Codebase memory   |     Repo/monorepo | Symbols, build graph, tests, ownership, dependencies | Deterministic index           |
| Experiment memory |              Task | Variant hypotheses, scores, failures                 | Automatic, archived           |
| Delivery memory   | Branch/PR/release | Evidence, approvals, rollback plan                   | Immutable                     |

### Add memory write contracts

```yaml
memory:
  writes:
    require_schema: true
    default_ttl: 90d
    require_source_for:
      - semantic
      - codebase
      - delivery
    prohibit:
      - secrets
      - raw_chain_of_thought
      - unverified_tool_claims
    sanitize:
      - invisible_unicode
      - prompt_injection_phrases
      - secret_patterns
  reflection:
    enabled: true
    write_to: episodic
    trigger_on:
      - checkpoint_failed
      - no_progress
      - human_correction
      - failed_experiment
    schema:
      type: object
      required: [failure_mode, attempted_fix, evidence, next_strategy]
```

This borrows from Reflexion’s episodic feedback idea, but keeps it host-structured rather than letting the model dump uncontrolled memories.

## 7. Multi-pass intelligence: better than one agent looping forever

ZYAL already has incubator passes. Expand this into a general “pass algebra.”

### Proposed pass types

```yaml
passes:
  - scout              # discover facts without modifying
  - map                # build codebase/task graph
  - spec               # requirements/design/tasks
  - plan               # implementation plan
  - threat_model       # security/privacy/blast radius
  - test_design        # characterize expected behavior
  - prototype          # isolated implementation
  - implement          # production patch
  - critic             # adversarial review
  - repair             # fix issues found by critic/tests
  - synthesize         # merge experiment learnings
  - verify             # run evidence checks
  - compress           # summarize into task memory
  - promote_review     # host evidence review
  - delivery           # PR/merge/release
```

### Add pass contracts

```yaml
pass_contracts:
  scout:
    writes: scratch_only
    must_output:
      - relevant_files
      - unknowns
      - risk_notes

  spec:
    writes: scratch_only
    must_output:
      - requirements
      - design
      - task_dag

  implement:
    writes: bounded_worktree
    requires:
      - approved_spec
      - locked_workset
      - test_plan

  critic:
    writes: scratch_only
    context: diff_plus_spec_plus_tests
    must_output:
      - blocking_objections
      - non_blocking_objections
      - missing_tests
      - security_concerns

  promote_review:
    writes: scratch_only
    requires:
      - passing_verification
      - no_blocking_objections
      - rollback_plan
      - evidence_summary
```

This gives ZYAL a composable language for “do smarter work,” not just “loop until tests pass.”

## 9. How ZYAL should respond to overlap concerns

### “Is ZYAL redundant with LangGraph?”

No. LangGraph is a runtime/orchestration framework. ZYAL should be a **portable runbook and policy declaration** that can compile to a LangGraph graph. ZYAL should not expose arbitrary graph programming as YAML; it should expose bounded policies: loops, tasks, passes, approvals, memory, evidence, retries, and delivery.

### “Is ZYAL redundant with OpenHands?”

No. OpenHands is an agent SDK/runtime/workspace/sandbox system. ZYAL can be the **host-enforced configuration language** for OpenHands-like runs. OpenHands gives you tools and execution; ZYAL gives you the strict lifecycle contract: when to run, when to stop, what evidence is required, what tools are permitted, how memory is written, and who approves promotion.

### “Is ZYAL redundant with Hermes?”

No. Hermes is an always-on agent with persistent memory, skills, messaging, cron, and subagents. ZYAL is missing Hermes-like compounding capability. The right move is to import Hermes lessons into ZYAL: skills, memory lanes, curator, cron triggers, chat routing, and session search — but make them host-governed and security-scanned.

### “Is ZYAL redundant with OpenClaw / OpenClaw Code Agent?”

No. OpenClaw Code Agent is a chat-to-coding-session orchestrator with plan approval, lifecycle, worktrees, PR follow-through, and goal loops. ZYAL should absorb those concepts into syntax so OpenClaw, OpenCode, Codex, and Claude Code can all understand the same runbook. ZYAL should be the neutral language; OpenClaw can be one surface/runtime.

### “Is ZYAL redundant with MCP?”

No. MCP is a tool/data protocol. ZYAL should define which MCP servers/tools/resources are available, under what trust level, with which secrets, roots, network policy, approvals, and audit requirements. MCP tool descriptions and external resources should be treated as untrusted unless the host policy marks them trusted.

### “Is ZYAL redundant with AGENTS.md?”

No. AGENTS.md is project guidance. ZYAL is executable daemon policy. ZYAL should read AGENTS.md as context, but AGENTS.md should not be allowed to arm a daemon or override safety policy.

### “Is ZYAL redundant with Claude Code / Codex config?”

No. Claude Code and Codex configs are product/runtime-specific. ZYAL should compile into those configs where possible, but retain a higher-level, portable lifecycle model. The same ZYAL should run in OpenCode, Codex, Claude Code, OpenHands, or a LangGraph backend.

## 13. The ZYAL runtime should have a typed intermediate representation

To become a world-standard language, ZYAL needs a formal IR:

```yaml
ir:
  schema: zyal.ir.v1
  compile:
    validate_yaml: true
    expand_templates: true
    resolve_policy_layers: true
    normalize_durations: true
    normalize_paths: true
    compute_preview_hash: true
    produce_capability_graph: true
    produce_task_graph: true
    produce_risk_report: true
  reject_if:
    - unknown_top_level_key
    - ambiguous_permission
    - unbounded_loop_without_circuit_breaker
    - write_scope_without_sandbox
    - external_network_without_policy
    - memory_write_without_redaction
```

The host should show a preview like:

```text
ZYAL Preview
------------
Run ID: monorepo-api-migration
Policy result: valid
Loop: forever, circuit breaker 5 errors → pause
Max parallel workers: 20
Writable paths: services/{{task.service}}/**, libs/auth/**, tests/**
Denied paths: .env*, secrets/**, infra/prod/**
Network: denied by default
Secrets exposed to model: none
Human gates:
  - before first edit
  - new dependency
  - PR creation
  - merge
Estimated risk: high
Preview hash: sha256:...
```

Then the human arms that exact hash.

## 14. What to keep from current ZYAL

Keep these. They are strong:

1. **Sentinel-wrapped prompt-native syntax.** This is the key to “any chat prompt.”
2. **Preview before arming.** Strengthen it with origin, hash, and nonce.
3. **Host owns loop.** Non-negotiable.
4. **SQLite truth.** Expand to event sourcing and replay.
5. **Guardrails/constraints as runtime enforcement.** Keep structural safety.
6. **Incubator.** This is ZYAL’s differentiator; expand it.
7. **On-handlers max depth 1.** Excellent anti-loop invariant.
8. **Write scopes for passes.** Expand to capability leases and worksets.
9. **Stop conditions.** Add richer condition types, but keep host-evaluated stops.
10. **Human approval gates.** Make them typed and evidence-based.

## 16. The most important new safety invariants

I would add these to the spec:

1. **Assistant output cannot arm a daemon.** Only trusted human or signed host-origin input can arm.
2. **Preview hash must match the armed run.** Any change invalidates the arm.
3. **Untrusted content cannot modify policy.** Webpages, tool outputs, repo files, issues, and docs are tainted.
4. **All side effects require idempotency records.** Shell, git, network, MCP, and external API calls must be logged.
5. **No memory write without provenance.** Semantic/procedural memory needs source, timestamp, scope, and trust level.
6. **No skill activation without scan.** Skills are supply-chain artifacts.
7. **No promotion by model score alone.** Promotion requires host evidence.
8. **No large diff without plan revision.** Big changes need explicit spec/evidence.
9. **No test reduction without human approval.** Deleting/skipping tests blocks promotion by default.
10. **No write without bounded workset.** Every edit must map to a task, scope, and rollback.
11. **No external network without policy.** Network access is deny-by-default.
12. **No secret in model context.** Secrets are brokered to tools only.
13. **No multi-agent write collision.** File/package/API locks are host-enforced.
14. **No delivery without rollback plan.** PR/merge/release requires recovery path.
15. **No hidden daemon tools outside active pass.** Keep this from current ZYAL.

## 17. The final product vision

ZYAL should become the language that lets a human say:

> “Here is the objective. Here is the blast radius. Here are the tools, budgets, workspaces, memories, skills, tests, approval gates, and delivery rules. You may work for hours, but you may not hallucinate completion, hide uncertainty, leak secrets, edit outside scope, merge without evidence, or forget what happened.”

That is the difference between vibe coding and autonomous software engineering.

The winning ZYAL design is not “YAML for agents.” It is:

* **Kubernetes job control** for coding agents,
* **Temporal/LangGraph durability** without forcing users to write graphs,
* **OpenHands/OpenCode execution** with a portable control plane,
* **Hermes-style compounding memory** with security gates,
* **OpenClaw-style chat orchestration** with worktree lifecycle,
* **Kiro-style specs** with acceptance criteria,
* **SWE-agent/Aider-style agent-computer interfaces** for codebase context,
* **MCP/ACP interoperability** for tools and editors,
* **evidence-gated promotion** for production-quality work.

If ZYAL takes that path, it can plausibly become the default runbook language for ambitious coding projects: not because it replaces every agent framework, but because it gives all of them a common, host-enforced contract for long-running autonomous work.
