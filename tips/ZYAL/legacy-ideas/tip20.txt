# ZYAL expansion study: make it the portable control plane for ambitious coding agents

ZYAL is **not redundant** if it stays focused on what most agent frameworks do not own: a **host-enforced, prompt-embeddable, durable runbook language** for long-running coding work. LangChain/LangGraph, OpenHands, Hermes, OpenClaw, AutoGen, CrewAI, Codex, Claude Code, and OpenCode all expose pieces of the problem: agents, tools, sandboxes, memory, skills, graphs, approvals, event logs, and evaluation. ZYAL’s opportunity is to become the **portable safety-and-control layer** above them.

The strongest version of ZYAL is not “YAML for agents.” It is closer to **Kubernetes + GitHub Actions + safety policy + experimental design + memory protocol for coding agents**.

The core upgrade I recommend:

> ZYAL should evolve from a daemon loop definition into a **multi-plane agent control language**:
>
> 1. **Control plane** — lifecycle, budgets, gates, approvals, state machines.
> 2. **Execution plane** — sandboxes, worktrees, containers, shells, MCP tools, credentials.
> 3. **Knowledge plane** — memory, specs, codegraph, task capsules, lessons, skills.
> 4. **Assurance plane** — evals, evidence, tests, security, observability, provenance, rollback.

The outside evidence is clear: long-running agents fail less from “not enough prompting” and more from **state loss, context drift, missing harnesses, weak verification, approval fatigue, insecure tools, and unstructured memory**. Anthropic’s long-running-agent engineering notes emphasize harnesses, discrete session handoffs, structured artifacts, and context coherence; OpenHands emphasizes event-sourced state and deterministic replay; LangGraph emphasizes durable checkpointing, human-in-the-loop, and time travel; Hermes emphasizes persistent user memory and self-improving skills; ProjectGuard and SlopCodeBench show that coding agents drift and degrade across long-horizon feature work without external project-state tracking and quality gates. ([Anthropic][1])

---

## 1. What ZYAL already gets right

ZYAL’s strongest ideas are excellent:

1. **Host owns the loop.** This is the most important design choice. The model should never decide when a daemon is “done,” when it is allowed to commit, or when it may promote a hard task.

2. **Preview before arming.** The separate `ZYAL_ARM RUN_FOREVER` sentinel is a real safety primitive. It distinguishes “show me a runbook” from “execute this runbook.”

3. **SQLite as truth.** A durable event/task/pass ledger is the right foundation. OpenHands uses immutable, type-safe events and describes its event stream as both memory and integration point; ZYAL is already aligned with that direction. ([OpenHands Docs][2])

4. **Budgeted autonomy.** Circuit breakers, incubator pass budgets, retry policies, and stop conditions are the correct way to make “long running” not mean “unbounded.”

5. **Guardrails as runtime policy.** Pattern checks, constraints, permissions, and promotion gates must be host-enforced. OWASP’s LLM risk taxonomy includes prompt injection, insecure output handling, insecure plugin/tool design, and excessive agency; ZYAL’s structural safety approach is the correct response. ([OWASP][3])

6. **Incubator passes.** The `scout → ideas → strengthen → critic → synthesize → prototype → review → compress` lifecycle is one of the most promising parts of the spec. It turns hard work into a staged evidence pipeline instead of “vibe until it compiles.”

The gap is not that ZYAL is weak. The gap is that ZYAL currently describes a daemon loop, but the world’s hardest coding projects need a **distributed, replayable, memory-aware, spec-tracked, evidence-gated agent operating system**.

# 2. Major lesson from the ecosystem

## OpenHands: ZYAL needs event semantics, replay, and SDK compatibility

OpenHands is probably the closest architectural neighbor. Its Software Agent SDK paper describes a stateless-by-default agent architecture with a single source of truth, event-sourced state, deterministic replay, immutable config, typed tools, MCP integration, and workspace abstraction. ([arXiv][4])

ZYAL overlaps with OpenHands on durable state, workspace isolation, and event logs. But ZYAL is currently less explicit about **event schemas**, **replay semantics**, **time travel**, **idempotency**, and **deterministic rehydration**.

**Upgrade:** ZYAL should define a formal event model:

```yaml
state:
  backend:
    type: sqlite
    per_run_truth: true
    replicate_to:
      - type: postgres
        mode: async
      - type: object_store
        mode: append_only
  event_log:
    schema: zyal.events.v1
    append_only: true
    hash_chain: true
    redact_secrets: true
    include:
      - state_transition
      - model_message_summary
      - tool_call
      - tool_result_digest
      - file_diff
      - approval_decision
      - guardrail_result
      - checkpoint_result
      - memory_write
      - promotion_decision
  replay:
    deterministic: true
    tool_result_cache: true
    restore_workspace: true
    restore_memory_snapshot: true
    require_same_schema_version: true
```

This is not cosmetic. For a 1B+ line codebase, you need to ask: “Why did this daemon touch 47 files?” “What evidence allowed this promotion?” “Can I replay from the last safe checkpoint?” “Can I fork from iteration 83 and try a different approach?”

OpenHands has the right instinct: event streams are not logs only; they are **the agent’s durable memory and the system’s integration surface**. ZYAL should make that first-class. ([OpenHands Docs][2])

## LangGraph: ZYAL needs graph execution, checkpoints, and time travel

LangGraph’s relevant strength is not “agents.” It is **durable stateful workflows**: checkpoints, memory, human-in-the-loop, tracing, and the ability to resume or fork execution. LangGraph documents checkpoints at every step, human review, memory, fault tolerance, and time travel/forking from previous graph states. ([LangChain Docs][5])

ZYAL’s `loop`, `fan_out`, `on`, and `incubator` are useful, but they are not yet a general execution graph. Current ZYAL can express “forever loop with handlers” and “map-reduce fan-out,” but not enough of:

* DAG dependencies
* conditional branches
* rollback edges
* graph-level resource budgets
* forked experiments
* joining partial state
* replay from checkpoints
* human approval nodes
* scheduled or external triggers

**Upgrade:** add an optional `graph` layer that compiles to the daemon loop when simple, but supports richer orchestration when needed.

```yaml
graph:
  nodes:
    discover:
      type: task_discovery
      run: "gh issue list --json number,title,labels"
      emits: tasks

    classify:
      type: model
      agent: plan
      input: "{{discover.tasks}}"
      output_schema: task_classification.v1

    incubate_hard:
      type: incubator
      when: "{{classify.risk_score >= 0.7}}"

    fix_simple:
      type: agent_pass
      agent: build
      when: "{{classify.risk_score < 0.7}}"

    verify:
      type: evidence_gate
      require:
        - tests_pass
        - no_spec_drift
        - no_security_regression

    checkpoint:
      type: checkpoint

  edges:
    - discover -> classify
    - classify -> incubate_hard
    - classify -> fix_simple
    - incubate_hard -> verify
    - fix_simple -> verify
    - verify -> checkpoint

  checkpointing:
    every_node: true
    allow_time_travel: ask
    allow_fork: ask
```

The key is that ZYAL should remain readable as YAML, but the host should compile it into a typed state machine.

## Hermes Agent: ZYAL needs skill memory, user/team memory, and self-improvement — but governed

Hermes is relevant because it treats the agent as a persistent, learning entity. Its docs describe a closed learning loop, agent-curated memory, autonomous skill creation, skill self-improvement, full-text recall, user modeling, multiple terminal backends, many communication platforms, and cron scheduling. ([Hermes Agent][6])

Hermes also has a prompt builder that assembles persona/config files, memory files, skills, project instruction files, tool guidance, provider-specific instructions, prompt caching, and a context compressor. ([GitHub][7])

ZYAL should not copy Hermes wholesale, but it should learn from Hermes’ biggest insight: **long-lived agents need a managed learning loop**.

Current ZYAL has task capsules and pass outputs, but it lacks a full memory taxonomy:

```yaml
memory:
  layers:
    - id: run_state
      scope: run
      storage: sqlite
      retrieve: always
      write: host_only

    - id: task_capsule
      scope: task
      storage: sqlite
      retrieve: on_task
      write: host_summarized
      contains:
        - objective
        - constraints
        - attempted_approaches
        - failures
        - current_plan
        - open_questions
        - evidence

    - id: project_semantic_state
      scope: repository
      storage: sqlite
      retrieve: relevant
      write_policy: evidence_only
      schema:
        decisions: array
        invariants: array
        public_apis: array
        architectural_constraints: array
        known_risks: array

    - id: structural_codegraph
      scope: repository
      provider: lsp_tree_sitter_dependency_graph
      refresh: after_checkpoint
      contains:
        - symbols
        - imports
        - call_graph
        - ownership
        - tests_for_symbols
        - runtime_entrypoints

    - id: failure_bank
      scope: organization
      retrieve: similar_failure
      write: after_failed_checkpoint
      retention: 180d
      contains:
        - symptom
        - root_cause
        - failed_attempts
        - eventual_fix
        - prevention_rule

    - id: skills
      scope: team
      retrieve: by_task_type
      write: require_review
      storage: signed_skill_registry

  compaction:
    strategy: evidence_preserving
    preserve:
      - architectural_decisions
      - unresolved_bugs
      - implementation_details
      - failed_attempts
      - tests_added
      - risk_acceptances
      - changed_files
    discard:
      - raw_tool_output_after_indexed
      - duplicate_logs
      - stale_plan_versions
```

This should explicitly avoid storing hidden chain-of-thought. Store **decisions, evidence, summaries, task state, tests, risks, and artifacts** — not private reasoning transcripts.

Hermes’ skill self-improvement is powerful, but ZYAL should govern it with provenance, review, and sandboxing:

```yaml
skills:
  registries:
    - id: repo
      path: .opencode/skills
      trust: local_reviewed
    - id: org
      source: zyal-skill-registry://engineering
      trust: signed

  generate:
    from_successful_runs: ask
    require_human_review: true
    min_reuse_likelihood: 0.6

  improve:
    enabled: true
    source:
      - repeated_failures
      - high_quality_successes
      - human_feedback
    require_diff_review: true

  curate:
    schedule: weekly
    actions:
      - dedupe
      - prune_low_value
      - merge_overlapping
      - update_examples
      - revoke_insecure

  security:
    provenance_required: true
    signature_required: true
    sandbox_skill_execution: true
    deny_unreviewed_tool_grants: true
```

Hermes shows the value of persistent learning; ZYAL’s job is to make that learning **auditable, scoped, and revocable**.

## OpenClaw: ZYAL can learn from gateway/channel orchestration, but must avoid skill-marketplace risk

OpenClaw is relevant less as a coding-agent harness and more as a **personal AI gateway/control plane**. Public docs describe a gateway daemon, multi-device/channel operation, voice/listen modes, Canvas, and many integrations. ([GitHub][8])

But OpenClaw also highlights the danger of “skills” and marketplace-style extensions. Reporting on OpenClaw discussed malicious skills, file/script/shell access, credential theft, and markdown-based instructions that target both users and agents. ([The Verge][9])

ZYAL should add channel/gateway support, but very carefully:

```yaml
triggers:
  allowed_sources:
    - direct_user_chat
    - signed_file
    - github_issue
    - ci_failure
    - scheduled_cron
  reject_sources:
    - assistant_output
    - tool_output
    - retrieved_webpage
    - untrusted_markdown

  require_trusted_origin_for_arm: true
  require_hash_bound_arm: true

channels:
  notify:
    - type: slack
      events: [paused, approval_required, promoted, aborted]
      redaction: secrets
    - type: github_check
      events: [checkpoint_failed, evidence_ready]
```

For “available inside any chat prompt,” this is critical: ZYAL blocks must be parsed **only from trusted user-originated input**, never from model output, webpages, documents, tool output, or repo markdown. Otherwise an issue comment, dependency README, or malicious skill could smuggle an `ZYAL_ARM` block.

A stronger arming syntax should bind the arm action to the exact normalized body:

```text
ZYAL_ARM RUN_FOREVER id=my-script sha256=5a9... nonce=host-issued-123 approved_by=user
```

That turns arming into a host-verifiable action, not just a magic string.

## Codex and Claude Code: ZYAL needs first-class sandbox, network, secrets, and approval policy

Codex official docs describe sandboxing, approval modes, default network restrictions, isolated cloud containers, setup-phase network access, and removing secrets before the agent phase. ([OpenAI Developers][10]) Anthropic’s sandboxing guidance similarly emphasizes filesystem and network isolation to prevent exfiltration and escape. ([Anthropic][11])

ZYAL’s current `permissions` block is too coarse. `shell: ask` and `edit: allow` are not enough for serious autonomy.

Add a `workspace` and `capabilities` model:

```yaml
workspace:
  isolation:
    type: container
    image: ghcr.io/org/agent-runtime:stable
    ephemeral: true
    restore_from: last_green_checkpoint

  filesystem:
    read:
      - "."
      - "/tmp/zyal-readonly-docs"
    write:
      - "."
      - "/tmp/zyal-scratch"
    deny:
      - "~/.ssh"
      - "~/.aws"
      - "~/.config"
      - ".env"
      - ".env.*"
      - "/"
    max_diff_files: 80
    max_diff_bytes: 500000

  network:
    default: deny
    allow_domains:
      - registry.npmjs.org
      - api.github.com
      - objects.githubusercontent.com
    deny_private_networks: true
    log_dns: true

  secrets:
    setup_phase:
      allow:
        - GITHUB_TOKEN_READONLY
    agent_phase:
      allow: []
    redact_in_logs: true
    block_if_diff_contains_secret: true

resources:
  limits:
    wall_clock: 8h
    cpu: "8"
    memory: 16Gi
    disk: 50Gi
    max_tool_calls: 2000
    max_tokens: 2000000
    max_cost_usd: 100
```

Then replace coarse permissions with scoped capability grants:

```yaml
capabilities:
  grants:
    - id: read_repo
      tools: [read_file, list_dir, grep, lsp]
      paths: ["."]
      ttl: run

    - id: edit_source
      tools: [write_file, apply_patch]
      paths: ["src/**", "tests/**"]
      deny_paths: ["migrations/**", "infra/**"]
      ttl: iteration

    - id: run_safe_tests
      shell:
        allow:
          - "bun test*"
          - "npm test*"
          - "pnpm test*"
          - "pytest*"
        deny:
          - "rm -rf*"
          - "curl * | sh"
          - "git push*"
          - "sudo *"
      ttl: iteration

    - id: dependency_change
      require_approval: true
      applies_to:
        - "package.json"
        - "bun.lockb"
        - "pnpm-lock.yaml"
```

Claude Code’s auto-approval work also matters: Anthropic reported that users approved most prompts, creating approval fatigue, and used classifiers to automate low-risk decisions. ([Anthropic][12]) ZYAL should support risk-classified approvals:

```yaml
approvals:
  classifier:
    enabled: true
    model: host_policy
    labels:
      - safe_read
      - safe_test
      - source_edit
      - dependency_change
      - schema_migration
      - credential_access
      - destructive

  policy:
    safe_read: allow
    safe_test: allow
    source_edit: allow_if_in_scope
    dependency_change: ask
    schema_migration: require_two_person
    credential_access: deny
    destructive: deny

  evidence_required_for_ask:
    - intended_command
    - reason
    - affected_paths
    - rollback_plan
    - risk_label
```

# 3. The biggest missing concept: an evidence graph

ZYAL has stop conditions, checkpoint verification, guardrails, and readiness scores. But it needs a unified **evidence graph**.

Right now, `readiness_score` and `tests_identified_gte` are separate from `checkpoint.verify`, `guardrails`, `assertions`, `constraints`, and `stop`. For large projects, promotion should never be “score says 0.78.” Promotion should be: **which machine-verifiable evidence proves this task is safe to move forward?**

Add:

```yaml
evidence:
  artifacts:
    - id: test_report
      type: junit
      source: "reports/junit.xml"
      produced_by: "bun test --reporter=junit"
      required_for: [checkpoint, promotion]

    - id: typecheck_report
      type: shell_assertion
      command: "npx tsgo --noEmit"
      assert: { exit_code: 0 }
      required_for: [checkpoint]

    - id: security_report
      type: sarif
      source: "reports/security.sarif"
      required_for: [promotion]

    - id: spec_trace
      type: traceability_matrix
      required_for: [promotion]

    - id: rollback_plan
      type: markdown
      required_for: [human_approval, promotion]

  rules:
    checkpoint:
      all:
        - test_report.pass_rate_gte: 1.0
        - typecheck_report.exit_code: 0
        - changed_files_mapped_to_requirements: true
        - no_secret_leaks: true

    promotion:
      all:
        - checkpoint.passed: true
        - spec_trace.coverage_gte: 0.95
        - rollback_plan.exists: true
        - unresolved_critical_objections: 0
        - human_review_required_if_risk_gte: 0.7
```

This would let every stop condition, checkpoint, incubator promotion, and human approval reference named evidence rather than ad hoc shell snippets.

# 4. Reduce vibe coding: require specs, traceability, tests, and drift detection

The anti-vibe-coding lesson is extremely important. ProjectGuard’s paper argues that coding agents can develop “emergent specifications” that drift from the user’s actual requirements, and proposes an external project-state layer tracking semantic decisions and structural repo state. It reports large improvements in faithfulness and fewer severe failures. ([arXiv][13])

SlopCodeBench is even harsher: it evaluates agents repeatedly extending prior solutions over many checkpoints and reports that no tested agent completed any problem end-to-end, with quality degradation, erosion, and verbosity across checkpoints. ([Hugging Face][14])

Separately, research on vibe coding security found that agents could produce functional but insecure code; one result in the abstract says SWE-Agent with Claude 4 Sonnet reached 61% functional correctness but only 10.5% security, and vulnerability hints did not reliably mitigate the issue. ([arXiv][15])

So ZYAL should add a **spec-control layer**:

```yaml
spec:
  sources:
    - path: AGENTS.md
      type: agent_instructions
    - path: specs/**/*.md
      type: requirements
    - path: docs/architecture/**/*.md
      type: architecture
    - path: .github/copilot-instructions.md
      type: agent_instructions

  require_before_code:
    one_of:
      - existing_requirement_id
      - generated_requirement_reviewed

  artifacts:
    requirements:
      path: .opencode/spec/requirements.yaml
      schema: zyal.requirements.v1
    acceptance_tests:
      path: .opencode/spec/acceptance-tests.yaml
    adr:
      path: docs/adr/{{task_id}}.md
      required_if:
        - public_api_change
        - schema_migration
        - cross_service_change

  traceability:
    require:
      every_changed_file_to_requirement: true
      every_requirement_to_test_or_risk_acceptance: true
      every_public_api_change_to_adr: true
      every_migration_to_rollback: true

  drift:
    detect: true
    compare_against:
      - user_objective
      - requirements
      - project_semantic_state
      - AGENTS.md
    on_unmapped_change: block_checkpoint
    on_spec_conflict: pause
```

This is the direct antidote to “jankurai” or vibe-coded jank: **no untraceable change gets promoted**.

For projects that want a Jankurai-like audit, ZYAL should not hard-code one vendor, but it can expose an extensible anti-jank gate. Public posts describe Jankurai as auditing vibe-coding failure modes and integrating with coding agents, but ZYAL should treat this as one scanner among many. ([LinkedIn][16])

```yaml
quality:
  anti_jank:
    enabled: true
    scanners:
      - id: static_jank_patterns
        command: "zyal-jank-scan --json"
        parse: "$.score"
        require_gte: 0.9
      - id: security
        command: "semgrep --config auto --json"
        parse: "$.results"
        require_empty: true
      - id: dependency_audit
        command: "npm audit --json"
        allow:
          max_moderate: 0
          max_high: 0
          max_critical: 0

    patterns:
      deny:
        - hidden_global_state
        - swallowed_errors
        - missing_authz_check
        - unaudited_external_call
        - test_removed_without_replacement
        - config_secret_in_code
        - broad_try_catch
        - silent_fallback
```

# 5. Memory upgrade: from compaction to project intelligence

Current ZYAL has:

```yaml
context:
  strategy: hybrid
  compact_every: 10
  hard_clear_every: 20
  preserve:
    - objective
    - progress
```

That is a good start, but too shallow. Long-running agents need a **memory operating system**.

Anthropic’s context engineering guidance emphasizes preserving architectural decisions, unresolved bugs, and implementation details; clearing or summarizing tool results; using structured note-taking; and keeping persistent memory outside the context window. ([Anthropic][17]) Hermes similarly shows persistent memory, user modeling, skill recall, and context compression as part of the agent architecture. ([Hermes Agent][6])

ZYAL should split memory into at least seven layers:

| Layer                  | Purpose                                 | Write authority                     |
| ---------------------- | --------------------------------------- | ----------------------------------- |
| Run memory             | Current daemon lifecycle                | Host                                |
| Task capsule           | Objective, attempts, failures, evidence | Host summaries plus model proposals |
| Project semantic state | Decisions, invariants, APIs, risks      | Evidence-gated                      |
| Structural codegraph   | Symbols, ownership, dependency graph    | Host/indexer                        |
| Episodic memory        | Prior similar runs and fixes            | Host summarized                     |
| Skill memory           | Reusable procedures                     | Review-gated                        |
| Human/team preferences | Style, risk tolerance, conventions      | Human/team approved                 |

Recommended schema:

```yaml
memory:
  retrieval:
    strategy: hybrid
    methods:
      - keyword
      - vector
      - code_symbol
      - dependency_graph
      - recent_event
      - failure_similarity
    max_context_tokens: 24000
    admission_policy:
      require_relevance_score_gte: 0.55
      prefer:
        - current_task_capsule
        - project_invariants
        - nearby_code_symbols
        - failing_test_logs
      penalize:
        - stale_plans
        - old_tool_output
        - unrelated_skill_docs

  writes:
    require_schema: true
    allow_model_proposals: true
    host_validates:
      - references_existing_files
      - cites_evidence_event_ids
      - no_secret_values
      - no_unverified_claims

  compaction:
    every_iterations: 8
    hard_clear_every: 24
    output_schema:
      type: object
      required:
        - objective
        - current_state
        - decisions
        - failed_attempts
        - open_questions
        - evidence
        - next_actions
      properties:
        decisions:
          items:
            required: [decision, reason_summary, evidence_event_ids]
        failed_attempts:
          items:
            required: [attempt, failure_mode, avoid_next_time]
```

The host should store compacted summaries as **public, auditable state**, not hidden reasoning.

# 6. Add disjoint experiments and stateful synthesis

ZYAL’s `fan_out` is currently useful for parallel audit tasks, but ambitious projects need something more specific: **controlled disjoint experiments**.

A good coding agent should often try several radically different approaches in isolation:

* minimal patch
* test-first patch
* architecture-preserving patch
* refactor-heavy patch
* performance-optimized patch
* compatibility-preserving patch
* migration-safe patch

Then the host should score them, synthesize the best ideas, and reject the rest.

Add:

```yaml
experiments:
  enabled: true
  strategy: fork_compare_synthesize

  route_when:
    any:
      - risk_score_gte: 0.6
      - public_api_change: true
      - repeated_attempts_gte: 2
      - estimated_files_changed_gte: 10

  fork:
    from: last_green_checkpoint
    isolation: git_worktree
    count: 5
    naming: "exp/{{task_id}}/{{hypothesis_id}}"
    reset_on_finish: true

  hypotheses:
    require_diversity: true
    axes:
      - minimal_patch
      - test_first
      - architecture_preserving
      - performance
      - rollback_simplicity
    generated_by: plan
    reviewed_by: critic

  worker:
    agent: build
    timeout: 20m
    max_parallel: 3
    writes: isolated_worktree

  compare:
    score:
      tests_pass: 30
      spec_traceability: 20
      security: 15
      diff_minimality: 10
      maintainability: 10
      performance: 10
      rollback_simplicity: 5

    tie_breakers:
      - fewer_public_api_changes
      - smaller_diff
      - better_test_coverage

  synthesize:
    agent: plan
    input:
      - top_2_patches
      - failure_summaries
      - critic_objections
    output:
      - recommended_patch
      - rejected_ideas
      - merge_plan
      - rollback_plan

  merge:
    strategy: patch_stack
    require_clean_apply: true
    require_evidence_gate: true
    on_conflict: pause
```

This is a huge unlock. It converts “one agent tries one thing repeatedly” into “the system runs a bounded experiment, scores variants, and builds on the winner.”

LangGraph-style checkpointing and forking are especially relevant here, because experiments need to fork from a known state and preserve original history. ([LangChain Docs][18])

# 7. Add formal execution semantics

ZYAL currently describes many pieces, but it needs a formal lifecycle. Otherwise implementations will diverge.

Add:

```yaml
semantics:
  schema: zyal.v1.1
  execution_model: event_sourced_state_machine

  states:
    - previewed
    - armed
    - initializing
    - running
    - waiting_for_approval
    - incubating
    - checkpointing
    - paused
    - stopping
    - stopped
    - aborted
    - exhausted
    - failed

  event_ordering:
    deterministic_within_run: true
    handler_max_depth: 1
    handler_reentrancy: deny
    concurrent_events: serialize_by_task

  idempotency:
    required_for:
      - hooks
      - checkpoint
      - promotion
      - cleanup
    key_template: "{{run_id}}:{{task_id}}:{{action_id}}:{{iteration}}"

  template_rendering:
    allowed_variables:
      - run_id
      - task_id
      - iteration
      - checkpoint_id
      - evidence
    escape_shell_args: true
    deny_untrusted_template_values: true
```

This prevents bugs like:

* an `on` handler triggering itself indirectly
* duplicate commits after retry
* cleanup running twice
* race conditions between workers
* template injection into shell commands
* a paused daemon still mutating files

# 8. Upgrade permissions into least-privilege policy

Current ZYAL:

```yaml
permissions:
  shell: ask
  edit: allow
  git_commit: ask
  git_push: deny
```

This is too coarse for long-running autonomy.

OpenCode already has agent-specific permissions and command-level allow/deny examples, including denying `git push`. ([OpenCode][19]) OpenCode also distinguishes Plan agents with restricted editing/bash behavior and subagents such as read-only exploration agents. ([OpenCode][20])

ZYAL should support **permission scopes by agent, pass, path, command, risk, and state**:

```yaml
permissions:
  defaults:
    shell: deny
    edit: deny
    network: deny
    mcp: deny
    git_commit: ask
    git_push: deny

  by_agent:
    plan:
      read: allow
      edit: deny
      shell:
        allow:
          - "git status"
          - "git diff*"
          - "grep*"
          - "rg*"

    build:
      edit:
        allow_paths:
          - "src/**"
          - "tests/**"
        deny_paths:
          - "infra/**"
          - "migrations/**"
          - ".github/workflows/**"
      shell:
        allow:
          - "bun test*"
          - "bun run lint*"
          - "npx tsgo --noEmit*"
        ask:
          - "bun install*"
          - "npm install*"
        deny:
          - "git push*"
          - "rm -rf*"
          - "sudo*"
          - "curl * | sh"

  by_state:
    previewed:
      all_tools: deny
    running:
      use_agent_policies: true
    checkpointing:
      edit: deny
      shell:
        allow:
          - "bun test*"
          - "npx tsgo --noEmit*"
    paused:
      all_tools: deny
```

This becomes much more powerful than a simple `ask | allow | deny`.

# 9. MCP: add context admission, budgets, and tool trust

ZYAL already has `mcp.profiles`, but it should be stricter. MCP is an open protocol for connecting LLM apps to tools and context, and it supports hosts, clients, servers, tools, resources, prompts, and composable workflows. ([Model Context Protocol][21]) But MCP servers can also dump huge context or expose dangerous tools. OpenCode’s MCP docs explicitly warn that MCP servers add to context and that a GitHub MCP server can add enough tokens to exceed context limits. ([OpenCode][22])

Add:

```yaml
mcp:
  profiles:
    readonly_code:
      servers: ["filesystem", "lsp"]
      tools: ["read_file", "list_dir", "symbols", "references"]
      max_context_tokens: 8000
      trust: local

    github_readonly:
      servers: ["github"]
      tools: ["get_issue", "list_pull_requests", "get_file_contents"]
      resources: ["repo://org/project"]
      max_context_tokens: 6000
      trust: oauth_scoped
      rate_limit:
        requests_per_minute: 30

    github_write:
      servers: ["github"]
      tools: ["create_pull_request", "add_comment"]
      require_approval: true
      trust: oauth_scoped

  context_admission:
    default: summarize_before_insert
    reject_if:
      - tokens_gt: 12000
      - contains_secret_pattern: true
      - source_untrusted: true
    require_digest_for_large_results: true

  tool_policy:
    require_schema_validation: true
    require_idempotency_key_for_mutations: true
    mutation_tools_require_approval: true
```

This lets ZYAL benefit from MCP without becoming vulnerable to context flooding or tool overreach.

# 13. Add rollback and recovery as first-class features

Long-running agents should always know how to undo themselves.

```yaml
rollback:
  required_for:
    - checkpoint
    - promotion
  strategy:
    code: git_revert_patch
    migrations: require_down_migration
    config: restore_previous_config
    feature_flags: default_off

  artifacts:
    - id: rollback_plan
      path: .opencode/daemon/{{run_id}}/rollback/{{task_id}}.md
    - id: inverse_patch
      command: "git diff --binary > .opencode/daemon/{{run_id}}/rollback/{{task_id}}.patch"

  verify:
    - command: "git apply --check .opencode/daemon/{{run_id}}/rollback/{{task_id}}.patch"
      assert: { exit_code: 0 }

  on_failed_checkpoint:
    auto_restore: last_green_checkpoint
    preserve_failure_artifacts: true
```

For production-impacting changes, add:

```yaml
release:
  strategy: staged
  feature_flag_required: true
  canary:
    enabled: true
    metrics:
      - error_rate
      - latency_p95
      - conversion
    rollback_if:
      error_rate_delta_gt: 0.02
      latency_p95_delta_gt: 0.15
```

ZYAL should not only ask “did tests pass?” It should ask: “Can we safely undo this?”

# 23. Overlap and gap analysis by system

## OpenHands

**Overlap:** durable agent runtime, workspace abstraction, event logs, MCP, typed tools.
**ZYAL gap:** event schemas, replay, deterministic state restoration, runtime SDK compatibility.
**Recommendation:** ZYAL should compile into or wrap OpenHands-style runtimes rather than duplicate them. OpenHands’ event-sourced architecture is a strong implementation target for ZYAL’s state model. ([arXiv][4])

## LangChain / LangGraph

**Overlap:** long-running workflows, graph state, human-in-loop, persistence, tracing.
**ZYAL gap:** graph semantics, checkpoints, time travel, forked execution.
**Recommendation:** add `graph`, `checkpointing`, `fork`, and `experiments` so ZYAL can either compile to LangGraph or use LangGraph-like semantics. ([LangChain Docs][5])

## Hermes Agent

**Overlap:** persistent agent, memory, skills, task delegation, terminal backends, background scheduling.
**ZYAL gap:** skill lifecycle, self-improvement, memory taxonomy, user/team modeling.
**Recommendation:** add skill generation/curation, but make every skill auditable, signed, sandboxed, and review-gated. Hermes’ learning loop is powerful; ZYAL should govern it. ([Hermes Agent][6])

## OpenClaw

**Overlap:** gateway daemon, multi-channel personal assistant control plane.
**ZYAL gap:** channel triggers, gateway notifications, cross-device approvals.
**Risk:** marketplace/skill ecosystems create serious attack surfaces.
**Recommendation:** support trusted triggers and channels, but reject ZYAL arming from untrusted markdown, tool output, or retrieved content. ([GitHub][8])

## OpenCode

**Overlap:** terminal/IDE coding agent, Plan/Build agents, subagents, permissions, MCP.
**ZYAL gap:** persistent daemon lifecycle, durable runs, promotion gates, incubator, fleet execution.
**Recommendation:** ZYAL should be native to OpenCode but portable. It should map directly to OpenCode agents, permission rules, and MCP profiles. OpenCode already supports agents, subagents, granular permission overrides, and MCP; ZYAL can become the durable orchestration layer above those features. ([OpenCode][30])

## Codex / Claude Code

**Overlap:** coding agents, worktrees, approvals, sandboxing, skills, cloud execution.
**ZYAL gap:** provider-neutral sandbox declarations, secret lifecycle, approval classifiers, portable skill policy.
**Recommendation:** copy the safety shape, not the product specifics: filesystem isolation, network deny-by-default, setup-phase secrets, agent-phase secret removal, and scoped approvals. ([OpenAI Developers][10])

## AutoGen, Microsoft Agent Framework, CrewAI

**Overlap:** multi-agent coordination, workflows, role-based agents, tools, human participation. AutoGen introduced multi-agent conversations among LLM/tool/human agents, Microsoft Agent Framework includes workflows, checkpointing, human-in-loop, state, context providers, middleware, and MCP clients, and CrewAI provides role-based crews and event-driven flows. ([Microsoft][31])
**ZYAL gap:** typed multi-agent contracts, graph nodes, handoff schemas, remote-agent interoperability.
**Recommendation:** ZYAL should control crews/graphs rather than become another crew framework.

```yaml
capabilities:     # fine-grained grants beyond permissions
memory:           # layered memory, retrieval, compaction, write policy
spec:             # requirements, ADRs, traceability, drift
evidence:         # named evidence artifacts and gates
evals:            # offline/online evaluators, scorecards
experiments:      # fork/compare/synthesize
review:           # typed human approval workflows
rollback:         # undo plans, inverse patches, restore points
skills:           # generate, import, curate, sign, sandbox
observability:    # traces, metrics, costs, dashboards
interop:          # MCP, A2A, OpenHands, LangGraph, Codex, Claude, etc.
fleet:            # large-scale scheduling and resource control
supply_chain:     # dependency, SBOM, provenance, signing
repo_instructions:# AGENTS.md, CLAUDE.md, Copilot instructions import
policy_packs:     # reusable signed policy bundles
```

Existing keys should remain:

```yaml
loop
context
checkpoint
tasks
agents
mcp
permissions
ui
on
fan_out
guardrails
assertions
retry
hooks
constraints
incubator
stop
```

But some should become sugar over newer primitives:

* `permissions` becomes shorthand for `capabilities`.
* `fan_out` becomes shorthand for `experiments` or `graph`.
* `checkpoint.verify` becomes shorthand for `evidence.rules.checkpoint`.
* `context` becomes shorthand for `memory.compaction`.
* `guardrails`, `assertions`, and `constraints` feed into `evidence` and `evals`.

# 26. Example: disjoint experiments for a hard architecture change

```yaml
version: v1.1
intent: daemon
confirm: RUN_FOREVER

job:
  name: Migrate authentication middleware safely
  objective: Replace legacy auth middleware with policy-based authorization without changing external behavior.
  risk:
    - Security regression
    - Public API behavior drift
    - Cross-service breakage

spec:
  require_before_code:
    one_of:
      - existing_requirement_id
      - generated_requirement_reviewed
  artifacts:
    adr:
      path: docs/adr/auth-middleware-migration.md
      required_if: [authz_change]
  traceability:
    require:
      every_changed_file_to_requirement: true
      every_authz_change_to_security_test: true

incubator:
  enabled: true
  strategy: evidence_maturation
  route_when:
    any:
      - authz_change: true
      - risk_score_gte: 0.6

experiments:
  enabled: true
  strategy: fork_compare_synthesize
  fork:
    from: last_green_checkpoint
    isolation: git_worktree
    count: 4
  hypotheses:
    require_diversity: true
    axes:
      - minimal_patch
      - compatibility_wrapper
      - policy_engine_first
      - test_first
  worker:
    agent: build
    max_parallel: 2
    timeout: 30m
  compare:
    score:
      security_tests_pass: 30
      compatibility_tests_pass: 25
      diff_minimality: 10
      rollback_simplicity: 10
      maintainability: 15
      performance: 10
  synthesize:
    output:
      - recommended_patch
      - rejected_ideas
      - security_rationale
      - rollback_plan

evals:
  datasets:
    - id: auth_unit_tests
      command: "bun test tests/auth --timeout 60000"
      required: true
    - id: compatibility_flows
      command: "bun run test:e2e:auth"
      required: true
    - id: security_static
      command: "semgrep --config p/security-audit --json"
      required: true

review:
  gates:
    - id: security_review
      when:
        authz_change: true
      reviewers:
        - role: security
        - role: owner_team
      require:
        - evidence_pack
        - rollback_plan
        - experiment_comparison
      require_two_person: true

rollback:
  required_for: [promotion]
  strategy:
    code: git_revert_patch
    feature_flags: default_off

stop:
  definition_of_done:
    all:
      - evidence: auth_unit_tests
      - evidence: compatibility_flows
      - evidence: security_static
      - review: security_review
      - git_clean:
          allow_untracked: false
```

This is how ZYAL becomes more than “run tests until green.” It becomes a controlled research-and-delivery process.

# 28. Security hardening for prompt-embedded ZYAL

Because ZYAL is intended to appear inside arbitrary chat prompts, its parser must be hostile-environment safe.

Recommended rules:

```yaml
parser:
  accepted_sources:
    - trusted_user_message
    - signed_file
    - host_ui_editor
  rejected_sources:
    - assistant_message
    - model_tool_output
    - retrieved_web_content
    - repository_markdown
    - issue_comment_untrusted
    - email_body
    - slack_message_untrusted

  sentinels:
    require_matching_ids: true
    reject_code_fences: true
    require_normalized_hash_for_arm: true
    require_host_nonce_for_arm: true

  preview:
    required: true
    show:
      - normalized_yaml
      - permissions_diff
      - risk_summary
      - budget_summary
      - network_access
      - secrets_access
      - write_scope
      - approval_gates
```

This is one of the highest-priority changes. If ZYAL can be embedded in any prompt, then **prompt-injected ZYAL** becomes an obvious attack path unless parsing is origin-aware.

# 31. The “ZYAL compiler” architecture

To make ZYAL portable across OpenCode, Codex, Claude, OpenHands, and others, implement ZYAL as a compiler plus runtime adapter system.

```text
ZYAL source
   ↓ parse
Normalized ZYAL AST
   ↓ validate
Policy-expanded ZYAL IR
   ↓ plan
Execution graph/state machine
   ↓ bind
Runtime adapter:
  - OpenCode
  - OpenHands
  - LangGraph
  - Codex CLI
  - Claude Code
  - shell/container
  - A2A remote agent
   ↓ execute
Event log + evidence graph + memory
```

Key intermediate representation:

```yaml
ir:
  tasks:
    - id
    - objective
    - state
    - allowed_capabilities
    - required_evidence
  transitions:
    - from
    - to
    - guard
    - action
  policies:
    - permissions
    - sandbox
    - approvals
    - evals
  artifacts:
    - evidence
    - memory
    - checkpoints
```

This gives ZYAL a future beyond OpenCode. Any host can parse ZYAL and decide how much it supports.

# 33. The central design principle

ZYAL should not try to be LangChain, OpenHands, Hermes, OpenClaw, Codex, Claude Code, or OpenCode.

It should be the language that says:

* what the agent may do,
* where it may do it,
* how long it may try,
* what evidence it must produce,
* how memory is preserved,
* how experiments are compared,
* when humans must approve,
* how changes are rolled back,
* how the run is replayed,
* and how the system learns without becoming unsafe.

That is the gap in the ecosystem.

The current ZYAL spec already has the right philosophy: **host owns the loop, preview before arming, durable state, budgets, structural safety.** The v.next opportunity is to add **formal execution semantics, evidence graphs, memory layers, spec traceability, experiment orchestration, sandbox/credential policy, observability, skill governance, and fleet scheduling**.

If built this way, ZYAL can become the default control language for ambitious coding agents because it is not another agent framework. It is the **portable, host-enforced contract between humans, agents, tools, codebases, and production risk**.
