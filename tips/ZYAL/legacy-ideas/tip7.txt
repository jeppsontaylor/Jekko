## Executive verdict

ZYAL is **not redundant**. It is aiming at a different layer than OpenHands, LangGraph, Hermes, OpenClaw, Codex, or Claude Code. Those systems mostly provide agent runtimes, tools, gateways, memory systems, sandboxes, or workflow libraries. ZYAL can become the **host-enforced control-plane language** above them: the manifest that says what may run, under what budget, with what memory, under what evidence requirements, in which sandbox, with which rollback path, and when humans must approve.

That is a huge idea. The closest analogy is not “another agent framework.” It is closer to **Kubernetes manifest + Temporal workflow + CI policy + agent eval contract + audit log** for autonomous coding.

But ZYAL v1 is currently too loop-centric. It has excellent instincts around preview/arming, SQLite truth, checkpoints, incubator passes, guardrails, constraints, permissions, and host-owned stop conditions. To manage genuinely ambitious work — especially 1B+ LOC monorepos — ZYAL needs to evolve from a daemon runbook into an **agent operating contract** with first-class support for workflow graphs, reproducible environments, memory governance, skill libraries, task leases, evidence provenance, tracing, human decisions, codebase indexing, experiments, rollouts, identity, and cross-runtime portability.

The core upgrade I would make: **ZYAL should not describe “an agent loop.” ZYAL should describe a bounded, observable, replayable, evidence-gated engineering campaign.**

# 1. What the outside systems teach us

I anchored this in primary docs, repositories, and papers rather than social anecdotes. X, LinkedIn, and Quora can reveal sentiment, but they are not strong foundations for a runtime language. The most useful signal came from official docs and research.

## OpenHands: event-sourced, sandboxed agent runtime

OpenHands’ strongest lesson is that long-running agents need a real event system, not only logs. Its docs describe events as immutable, type-safe messages forming an append-only history, with observation/action pairs used as the agent’s memory and integration point. The agent loop is stateless and event-driven, with each step reading prior events, optionally condensing context, asking the LLM, executing actions, and appending results. Its runtime also emphasizes Docker-based sandboxing for security, consistency, resource control, isolation, and reproducibility. ([OpenHands Docs][1])

**ZYAL implication:** SQLite truth is right, but ZYAL needs a typed event schema, replay semantics, idempotency keys, spans, and event hash chains. “ledger.jsonl” should not merely be a mirror. It should be a projection of a typed event-sourced database.

## LangGraph / LangChain: durable workflow runtime, HITL, middleware, memory

LangGraph frames itself as infrastructure for long-running, stateful agents with durable execution, streaming, human-in-the-loop, memory, and production observability. Its durable execution docs stress checkpointing progress, pausing/resuming after failures or human approval, and making side effects deterministic or idempotent. LangChain’s agent middleware also exposes useful categories: summarization, human-in-the-loop, model/tool call limits, model fallback, PII handling, tool retries, context editing, filesystem, and subagents. ([LangChain Docs][2])

**ZYAL implication:** ZYAL should add a real `workflow` graph, not only `loop` plus `fan_out`. It should also make human decisions first-class, with approve/edit/reject/respond semantics and persisted resumability.

## Hermes Agent: persistent memory, self-improving skills, durable multi-agent board

Hermes is especially relevant because it pushes beyond one-shot coding assistance. Its docs emphasize a built-in learning loop that creates and improves skills from experience, uses persistent memory, searches past conversations, and supports model-agnostic operation. Its recent release notes describe a durable multi-agent Kanban board with heartbeats, task reclaim, zombie detection, retries, hallucination gates, checkpoints, and auto-resume after restart. ([GitHub][3])

Hermes’ memory docs are also directly useful: it distinguishes curated bounded memory from session search; uses SQLite FTS5; injects memory snapshots at session start; supports external memory providers; and includes security scanning for memory injection, exfiltration, and invisible Unicode. Its skills system uses progressive disclosure: a small skill list first, full content only when needed, and specific references at deeper levels. ([Hermes Agent][4])

**ZYAL implication:** ZYAL’s `context` block is far too small. ZYAL needs typed memory stores, retrieval policy, write policy, compression policy, skill registry, skill curation, memory provenance, memory redaction, and background memory jobs.

## OpenClaw: gateway, channels, identity, delegates, idempotency

OpenClaw’s docs are useful because they think about agents as persistent assistants connected to channels. It has a long-lived gateway, multi-channel inbox, role nodes, a canvas, JSON schema validation, events, pairing, auth, and idempotency keys for side-effecting sends and agent calls. Its security docs explicitly warn that the gateway is a personal-assistant trust model, not a hostile multi-tenant security boundary. Its runtime docs describe serialized queues, locks, plugin hooks, compaction, retry, streaming, and long default runtime timeouts. ([OpenClaw][5])

OpenClaw’s multi-agent docs also include isolated agents with separate workspaces, agent directories, session stores, bounded sanitized recall, and delegated agents with scoped authority. ([OpenClaw][6])

**ZYAL implication:** ZYAL is missing identity, channels, tenant boundaries, scoped delegates, idempotency, and queue/lock semantics. These matter if ZYAL is meant to run from any chat prompt, not only inside one local coding session.

## Codex and Claude Code: instruction files, permissions, hooks, subagents

Codex and Claude Code both show that coding agents already have strong local ergonomics: instruction files, MCP, approval modes, subagents, review agents, hooks, and worktree-based parallelism. Codex reads `AGENTS.md` files with precedence across global and project scopes, and the Codex CLI includes model switching, reviews from a separate Codex agent, subagents, web search, cloud tasks, scripting, MCP, and approval modes. ([OpenAI Developers][7])

Claude Code has project/user/managed settings, permissions, hooks, subagents, MCP, memory files, resume workflows, worktree-based parallel sessions, and plan-before-editing practices. Its hook docs are especially relevant because hooks are deterministic control points rather than relying on the model to remember rules. ([Claude API Docs][8])

**ZYAL implication:** ZYAL should not compete with these tools at the UX layer. It should compile down to them, supervise them, or use them as executors. ZYAL should become the portable policy/workflow layer that can drive OpenCode, Codex, Claude Code, OpenHands, LangGraph, and custom MCP tools.

## Research: agent-computer interfaces, reflection, skills, multi-agent SOPs, and “agentless” repair

SWE-agent shows that the agent-computer interface matters: its paper argues that custom commands for editing, navigation, and execution meaningfully improve software-engineering agent performance. Voyager demonstrates automatic curriculum, an ever-growing skill library, and iterative feedback/self-verification. Reflexion demonstrates verbal self-reflection stored in episodic memory. MetaGPT and ChatDev show the value of role specialization, SOPs, and explicit intermediate artifacts. Agentless is the most important counterweight: it shows that simpler localize-repair-rerank pipelines can compete with or beat more generic autonomous agents on software repair tasks. ([arXiv][9])

**ZYAL implication:** ZYAL must support both agentic and non-agentic workflows. The best coding system will not always be a chat loop. Sometimes it should be: localize → generate patches → rank → verify → promote. ZYAL needs deterministic pipelines, candidate pools, experiments, and reducers.

# 2. What ZYAL already gets very right

ZYAL’s strongest ideas are excellent:

| ZYAL idea                              | Why it is strong                                                                            |
| -------------------------------------- | ------------------------------------------------------------------------------------------- |
| **Host owns the loop**                 | Prevents the model from self-certifying completion or deciding to run forever.              |
| **Preview before arming**              | Makes execution a deliberate human act. This is essential for “any chat prompt” deployment. |
| **SQLite is truth**                    | Durable state beats hidden context, local markdown summaries, or model memory.              |
| **Stop conditions are host-evaluated** | The model cannot simply claim success.                                                      |
| **Checkpoint verification**            | Commits only after host-observed verification.                                              |
| **Guardrails and constraints**         | Runtime enforcement is stronger than prompt instruction.                                    |
| **Incubator passes**                   | This is the most novel part: hard tasks mature before touching production code.             |
| **Write scopes**                       | `scratch_only` and `isolated_worktree` are the right primitives.                            |
| **Circuit breakers**                   | Essential for preventing runaway loops.                                                     |
| **On-handlers max depth 1**            | Prevents handler cascades and accidental infinite loops.                                    |

The correct design direction is: **keep ZYAL declarative and host-enforced. Do not turn it into a scripting language.**

# 3. The most important missing pieces

## 3.1 ZYAL needs a top-level `budget`

Right now budget is scattered: loop circuit breakers, incubator budgets, worker timeouts. A serious daemon needs one global budget contract covering wall time, tokens, money, tool calls, process time, disk, network, retries, branch count, diff size, and risk.

Without this, “bounded by design” is incomplete.

Proposed syntax:

```yaml
budget:
  wall_time: 12h
  calendar_deadline: "2026-05-09T18:00:00-06:00"
  max_iterations: 200
  max_llm_calls: 500
  max_tool_calls: 2000
  max_tokens:
    input: 50_000_000
    output: 5_000_000
  max_cost_usd: 250
  max_shell_seconds: 7200
  max_parallel_workers: 8
  max_git_worktrees: 12
  max_changed_files: 120
  max_diff_lines:
    added: 12000
    deleted: 12000
  max_new_dependencies: 3
  max_network_requests: 200
  on_exhaust: pause
```

This should be enforced before every side effect, not just checked after an iteration.

## 3.2 ZYAL needs a `workflow` DAG, not only a loop

`loop` is useful for “keep fixing until tests pass.” It is insufficient for complex work. Ambitious coding projects need staged workflows: discover, localize, design, review, prototype, test, rank, merge, harden, benchmark, canary, PR, monitor.

LangGraph’s durable execution model is a clear signal here: long-running agents need persisted progress and resumable execution, not just repeated turns. ([LangChain Docs][2])

Proposed syntax:

```yaml
workflow:
  mode: durable_dag
  default_retry: checkpoint_safe
  nodes:
    - id: discover
      type: shell
      command: "bun test --reporter=json > .zyal/artifacts/test-failures.json"
      produces: ["artifact:test_failures"]

    - id: localize
      type: agent
      agent: scout
      reads: ["artifact:test_failures", "index:symbol_graph"]
      writes: scratch_only
      produces: ["artifact:localization_report"]

    - id: design
      type: agent
      agent: plan
      reads: ["artifact:localization_report"]
      writes: scratch_only
      produces: ["artifact:implementation_plan"]

    - id: plan_review
      type: approval
      requires: ["artifact:implementation_plan"]
      decisions: [approve, edit, reject, respond]
      on_reject: abort
      on_edit: design

    - id: patch_pool
      type: experiment_pool
      strategy: disjoint_candidates
      candidates: 5
      base: checkpoint:pre_patch
      worker:
        agent: build
        isolation: git_worktree
      produces: ["candidate_patch:*"]

    - id: rank
      type: reducer
      strategy: best_verified_patch
      ranking:
        - tests_pass
        - typecheck_pass
        - minimal_diff
        - low_blast_radius
        - reviewer_score
      produces: ["candidate_patch:best"]

    - id: promote
      type: checkpoint
      patch: "candidate_patch:best"

  edges:
    - discover -> localize
    - localize -> design
    - design -> plan_review
    - plan_review -> patch_pool
    - patch_pool -> rank
    - rank -> promote
```

This one addition would make ZYAL dramatically more powerful. The current `incubator` can remain as a specialized workflow preset.

## 3.3 ZYAL needs typed artifacts and evidence provenance

Current `readiness` fields like `tests_identified_gte` and `plan_reviewed_gte` are good, but they are too abstract. The host needs to know **what artifact proves the field**, who or what produced it, whether it is fresh, whether it was verified, and whether it is tied to a specific patch.

Proposed syntax:

```yaml
evidence:
  store: sqlite
  hash_artifacts: true
  require_provenance: true
  artifact_root: ".opencode/daemon/{{run_id}}/artifacts"

  types:
    implementation_plan:
      schema:
        type: object
        required: [objective, affected_files, risks, tests, rollback]
    test_report:
      schema:
        type: object
        required: [command, exit_code, passed, failed, duration_ms]
    patch_review:
      schema:
        type: object
        required: [summary, risks, approval, objections]
    benchmark_report:
      schema:
        type: object
        required: [metric, before, after, threshold]

  claims:
    tests_identified:
      proven_by:
        - artifact_type: implementation_plan
          json_path: "$.tests"
          assert: non_empty
    scope_bounded:
      proven_by:
        - artifact_type: implementation_plan
          json_path: "$.affected_files"
          assert:
            max_items: 25
    rollback_known:
      proven_by:
        - artifact_type: implementation_plan
          json_path: "$.rollback"
          assert: non_empty
```

Promotion should not depend on a score alone. It should depend on **claims bound to artifacts**.

## 3.4 ZYAL needs `approvals` as a first-class durable concept

`permissions: ask` is too coarse. Human approval should be a structured state machine: approve, reject, edit, respond, defer, escalate. LangChain’s human-in-the-loop middleware explicitly supports interrupting execution, persisting state, and resuming with decisions like approve/edit/reject/respond. ([LangChain Docs][10])

Proposed syntax:

```yaml
approvals:
  default_channel: tui
  persist: sqlite
  require_reviewer_identity: true
  decisions: [approve, edit, reject, respond, defer]

  gates:
    - id: arm_daemon
      when: before_start
      require:
        - human: true
        - runbook_hash_matches: true
        - trusted_channel: user

    - id: approve_plan
      when: before_node
      node: patch_pool
      prompt: "Approve implementation plan before code changes."
      evidence:
        - artifact: implementation_plan
      decisions: [approve, edit, reject, respond]
      on_edit: rerun_node: design
      on_reject: abort

    - id: approve_dependency
      when: dependency_added
      prompt: "New dependency requested."
      require_reason: true
      decisions: [approve, reject]

    - id: approve_push
      when: git_push
      decisions: [approve, reject]
      on_reject: keep_local_checkpoint
```

This turns approvals into durable workflow events rather than ad hoc CLI prompts.

## 3.5 ZYAL needs stronger arming semantics for “any chat prompt”

The current sentinel design is good but insufficient for hostile or noisy contexts. If ZYAL can appear inside any chat prompt, the host must distinguish trusted user intent from model-generated or quoted text.

The ARM line should include a host-generated nonce and a hash of the exact effective runbook. Otherwise, a model could accidentally emit or copy an arming line, or a prompt-injected document could contain one.

Proposed arming upgrade:

```yaml
arming:
  require_separate_message: true
  require_trusted_channel: user
  require_host_nonce: true
  require_effective_hash: true
  nonce_ttl: 10m
  reject_if_generated_by_assistant: true
  reject_if_inside_code_fence: true
  reject_if_quoted: true
  preview_must_show:
    - permissions
    - budget
    - network
    - filesystem_scope
    - secrets_scope
    - git_push_policy
    - max_cost_usd
```

Future ARM sentinel:

```text
ZYAL_ARM RUN_FOREVER id=my-script sha256=8f4d... nonce=host-2026-05-08-abc123
```

This preserves the “preview before arming” principle while making it robust enough for arbitrary chat surfaces.

## 3.6 ZYAL needs reproducible `environment` and sandbox contracts

OpenHands’ runtime docs emphasize sandboxing for security, reproducibility, consistency, isolation, and resource control. ([OpenHands Docs][11]) ZYAL needs to describe not just what commands run, but where they run.

Proposed syntax:

```yaml
environment:
  workspace:
    strategy: git_worktree
    base_ref: origin/main
    clean_before_start: true
    allow_dirty_start: false

  sandbox:
    backend: docker
    image: "ghcr.io/acme/monorepo-dev@sha256:..."
    user: "node"
    cpu: 8
    memory: 32GiB
    disk: 200GiB
    timeout: 12h
    mounts:
      - source: repo://
        target: /workspace
        mode: rw
      - source: cache://bun
        target: /home/node/.bun
        mode: rw

  services:
    - id: postgres
      image: "postgres:16"
      seed: "scripts/seed-test-db.sh"
      healthcheck: "pg_isready"
    - id: redis
      image: "redis:7"

  network:
    default: deny
    allow:
      - host: github.com
        ports: [443]
      - host: registry.npmjs.org
        ports: [443]
        until: after_install

  secrets:
    expose:
      - name: GITHUB_TOKEN
        to: [git_fetch, create_pr]
        mode: masked
      - name: SLACK_WEBHOOK
        to: [notify]
        mode: masked
    deny_to_llm: true
```

This is essential for 1B+ LOC work because the environment is often harder than the patch.

## 3.7 ZYAL needs memory governance, not only context compaction

Long-running agents fail when they forget important decisions, preserve irrelevant chatter, or let untrusted content poison memory. Hermes’ memory design is a useful reference: bounded curated memory, session search, injected snapshots, FTS5, external memory providers, and memory security checks. ([Hermes Agent][4]) LangChain also distinguishes short-term thread memory from long-term namespaced memory and separates semantic, episodic, and procedural memory. ([LangChain Docs][12])

Proposed syntax:

```yaml
memory:
  enabled: true
  store: sqlite
  security:
    scan_for_prompt_injection: true
    scan_for_secret_exfiltration: true
    reject_invisible_unicode: true
    require_source_provenance: true

  namespaces:
    - id: run
      type: episodic
      retention: until_run_archived
      writable_by: [host, agents]
      retrieval: always

    - id: repo_facts
      type: semantic
      retention: long_term
      writable_by: [host_indexer]
      retrieval: relevance
      max_tokens: 12000

    - id: decisions
      type: episodic
      retention: long_term
      writable_by: [approval_gate, promotion_review]
      retrieval: always_for_related_tasks

    - id: procedures
      type: procedural
      retention: long_term
      writable_by: [skill_curator]
      retrieval: progressive_disclosure

    - id: user_preferences
      type: profile
      retention: long_term
      writable_by: [human, memory_curator]
      retrieval: only_when_relevant

  retrieval:
    strategy: hybrid
    max_tokens: 20000
    include:
      - current_objective
      - active_task_capsule
      - relevant_decisions
      - repo_facts
      - loaded_skills
    exclude:
      - stale_failed_attempts
      - untrusted_tool_output

  write_policy:
    hot_path:
      allow: false
      reason: "Do not mutate long-term memory during every agent step."
    after_checkpoint:
      extract:
        - decisions
        - verified_facts
        - reusable_procedures
        - unresolved_risks
    after_failure:
      extract:
        - failed_hypothesis
        - error_signature
        - avoidance_rule

  compaction:
    strategy: evidence_preserving
    preserve:
      - approvals
      - checkpoints
      - failing_tests
      - root_cause
      - current_plan
      - rollback_plan
    summarize_tool_logs: true
    never_summarize:
      - diffs
      - test_failures
      - stack_traces
      - approval_decisions
```

The critical principle: **memory writes must be governed.** The model should not be able to silently add permanent beliefs.

## 3.8 ZYAL needs a skill library with progressive disclosure and curation

Hermes’ skills docs show a strong pattern: list skills cheaply, load full skill content only when needed, and load specific references at deeper levels. Its curator can mark skills stale, archive them, protect pinned skills, run dry-runs, use backups, and review patches. ([Hermes Agent][13])

ZYAL should include this directly:

```yaml
skills:
  enabled: true
  discovery:
    sources:
      - repo://.opencode/skills
      - org://engineering/skills
      - user://skills
    precedence: [repo, org, user]

  progressive_disclosure:
    level0_list_budget: 2000
    level1_load_on_trigger: true
    level2_reference_on_demand: true

  schema:
    required:
      - trigger
      - procedure
      - pitfalls
      - verification
      - rollback

  use_policy:
    require_trigger_match: true
    record_usage: true
    block_unverified_skills_for:
      - production_migration
      - security_fix
      - auth_change

  create_from_success:
    enabled: true
    when: after_promotion
    require:
      - repeated_pattern: true
      - tests_passed: true
      - human_approved: true
    writes: scratch_only
    approval_required: true

  curator:
    enabled: true
    schedule: weekly
    stale_after: 90d
    protect_pinned: true
    dry_run_first: true
    archive_unused: true
    backup_before_change: true
```

This is how ZYAL stops being a runbook language and becomes a system that compounds engineering knowledge.

## 3.9 ZYAL needs a durable task board with leases, heartbeats, and reclaim

Hermes’ latest multi-agent Kanban release is very relevant: durable board, heartbeat, reclaim, zombie detection, retries, incomplete-exit blocking, and hallucination gates. ([GitHub][14]) ZYAL’s `tasks` block is currently a discovery hook, not a robust task operating system.

Proposed syntax:

```yaml
task_graph:
  store: sqlite
  states:
    - backlog
    - ready
    - leased
    - running
    - blocked
    - reviewing
    - ready_to_promote
    - done
    - parked
    - failed

  dependencies:
    enforce: true
    allow_cycles: false

  lease:
    heartbeat_every: 30s
    stale_after: 5m
    reclaim_stale: true
    max_reclaims_per_task: 3
    zombie_detection: true

  retries:
    per_task_max: 3
    classify_failures:
      - env_failure
      - flaky_test
      - merge_conflict
      - model_error
      - unknown

  blocking:
    require_reason: true
    require_unblock_condition: true

  hallucination_gate:
    enabled: true
    task_must_reference_existing_files: true
    claimed_failures_must_match_logs: true
    claimed_completion_requires_evidence: true
```

For 1B+ LOC, this becomes non-negotiable. You cannot coordinate thousands of tasks with a simple task discovery shell command.

## 3.10 ZYAL needs codebase indexing and blast-radius analysis

A 1B+ LOC project cannot be loaded into context. It needs a structured index: symbols, ownership, dependencies, APIs, tests, packages, services, build graph, runtime topology, incidents, migrations, and release channels.

Proposed syntax:

```yaml
codebase_index:
  enabled: true
  store: sqlite
  update: incremental
  sources:
    - type: git
      ref: origin/main
    - type: build_graph
      command: "bazel query //..."
    - type: symbols
      command: "npx tsgo --noEmit --generateTrace .zyal/trace"
    - type: ownership
      files:
        - CODEOWNERS
        - .github/teams.yml
    - type: tests
      command: "scripts/list-tests --json"
    - type: api_contracts
      paths:
        - "openapi/**/*.yaml"
        - "proto/**/*.proto"

  indexes:
    - symbol_graph
    - dependency_graph
    - test_coverage_map
    - ownership_map
    - migration_map
    - service_topology
    - risk_history
    - flaky_test_registry

  retrieval:
    prefer_exact_symbol_match: true
    include_callers_and_callees: true
    include_owning_tests: true
    max_context_files: 80

  blast_radius:
    compute_before_edit: true
    block_if:
      affected_services_gt: 5
      public_api_changed: true
      migration_without_rollback: true
```

This is one of the biggest upgrades. It turns ZYAL from “let an agent run commands” into “let a host reason about the engineering system.”

# 4. Specific overlap and gap analysis

## ZYAL vs OpenHands

OpenHands is a runtime and SDK. It gives you sandboxed execution, event history, observations/actions, and agent loop infrastructure. ZYAL should not replicate that. ZYAL should define the **policy, workflow, and acceptance contract** that an OpenHands runtime can execute. OpenHands’ event system and sandboxing are stronger than ZYAL v1’s current durability model, but ZYAL’s preview/arming, host-owned stop conditions, checkpoint gating, and incubator concept are stronger as a declarative control surface. ([OpenHands Docs][1])

**Gap to close:** add typed events, replay, idempotency, sandbox descriptions, and runtime adapters.

```yaml
runtime:
  adapter: openhands
  event_model: typed_append_only
  sandbox: docker
  replay:
    enabled: true
    deterministic_until_side_effect: true
    side_effects_require_idempotency_key: true
```

## ZYAL vs LangChain / LangGraph

LangGraph is already a durable workflow runtime for stateful agents. ZYAL overlaps with it when ZYAL tries to define loops, retries, and handlers. The better relationship is: **ZYAL is the manifest; LangGraph can be one compiled execution target.** LangGraph has stronger graph semantics, durable execution, HITL, middleware, and memory primitives today. ZYAL has a more domain-specific coding-agent contract with git, checkpoints, guardrails, and incubators. ([LangChain Docs][2])

**Gap to close:** add `workflow: durable_dag`, human decisions, middleware-like policy slots, and memory namespaces.

```yaml
compile:
  targets:
    - id: local_opencode
      type: native
    - id: langgraph
      type: langgraph
      emit: ".zyal/generated/workflow.py"
      check_idempotency: true
```

## ZYAL vs Hermes

Hermes is far ahead on persistent memory, skills, self-improvement, and multi-agent task-board mechanics. ZYAL’s incubator has a stronger explicit promotion gate, but ZYAL lacks skill learning, memory providers, curated memory, task leases, heartbeat, reclaim, zombie detection, and auto-resume semantics. ([GitHub][3])

**Gap to close:** add `memory`, `skills`, `skill_curator`, and `task_graph`.

```yaml
self_improvement:
  enabled: true
  allowed_outputs:
    - skill_patch
    - memory_candidate
    - test_harness_improvement
  disallowed_outputs:
    - permission_expansion
    - budget_expansion
    - secret_request
  require_human_approval: true
```

The key safety line: **agents may propose self-improvements; the host approves and installs them.**

## ZYAL vs OpenClaw

OpenClaw is much more channel-aware. It thinks about gateways, user/channel routing, role nodes, idempotency keys, session stores, delegates, scoped authority, and cross-session recall. ZYAL v1 assumes a coding repo context, not a world where a daemon can be initiated from Slack, Telegram, WhatsApp, a TUI, an IDE, or a web app. ([OpenClaw][5])

**Gap to close:** add identity, channel, tenant, delegate, idempotency, and session-scope semantics.

```yaml
identity:
  run_owner: "user:alice"
  organization: "org:acme"
  agent_identity: "agent:zyal-daemon"
  delegate_authority:
    can_create_pr: true
    can_push_branch: false
    can_merge: false
```

## ZYAL vs Claude Code

Claude Code has strong interactive ergonomics: hooks, permissions, subagents, project settings, memory, MCP, plan mode, and worktree workflows. ZYAL should be the long-horizon supervisor around Claude Code-style sessions. Claude Code hooks show that deterministic lifecycle control is highly valuable; ZYAL should generalize that to multi-hour and multi-day campaigns. ([Claude API Docs][15])

**Gap to close:** ZYAL should import/export `CLAUDE.md`, `AGENTS.md`, MCP configs, hooks, and permission profiles.

```yaml
instruction_files:

    - zyal
    - managed_policy
    - repo

  conflict_policy: zyal_overrides
```

## ZYAL vs Codex

Codex already supports AGENTS.md, approvals, reviews by separate agent, subagents, MCP, cloud tasks, scripting, and web search. ZYAL can become the declarative layer that says: “run Codex as a worker under this budget, with this memory, this evidence contract, and these promotion gates.” ([OpenAI Developers][7])

**Gap to close:** add runtime adapters and capability negotiation.

```yaml
runtimes:
  executors:
    - id: opencode
      type: native
      capabilities: [edit, shell, git, mcp, subagents]

      type: cli

      type: cli
      command: "claude --print --output-format json"
      capabilities: [edit, shell, hooks, subagents, mcp]
  selection:
    default: opencode
    fallback: codex
```

## ZYAL vs Agentless

Agentless is the warning shot. Not every coding problem benefits from an open-ended agent loop. A localize-repair-rerank pipeline can be cheaper, more reproducible, and easier to evaluate. ([arXiv][16])

**Gap to close:** ZYAL needs non-chat workflow nodes.

```yaml
workflow:

      type: patch_rerank

      type: shell_matrix
      commands:
        - "bun test"
        - "bun run lint"
        - "npx tsgo --noEmit"
```

This is also a powerful anti-vibe-coding move.

# 6. Concrete ZYAL upgrades in detail

## 6.1 `state`: event sourcing, replay, and idempotency

Current ZYAL says SQLite is truth. Great. Now make it precise.

```yaml
state:
  store: sqlite
  path: ".opencode/daemon/state.sqlite"
  event_log:
    typed: true
    append_only: true
    hash_chain: true
    include_prev_hash: true
    redact_secrets: true

  tables:
    - runs
    - events
    - spans
    - tasks
    - leases
    - checkpoints
    - artifacts
    - approvals
    - memory_items
    - skills
    - budgets
    - experiments
    - rollouts

  idempotency:
    required_for:
      - git_commit
      - git_push
      - create_pr
      - send_message
      - run_migration
      - external_api_call
    key_format: "{{run_id}}:{{node_id}}:{{attempt}}:{{effect_hash}}"

  replay:
    enabled: true
    mode: audit
    can_replay_until:
      - first_external_side_effect
      - non_idempotent_tool_call
```

OpenHands’ immutable event history and OpenClaw’s idempotency-key design both point toward this. ([OpenHands Docs][1])

## 6.3 `models`: explicit routing, fallback, and evaluator models

## 6.4 `policy`: finer-grained capability control

`permissions.shell: ask` is too blunt. ZYAL needs path, command, tool, agent, risk, and time-scoped policies.

```yaml
policy:
  default: deny

  filesystem:
    read:
      allow:
        - "src/**"
        - "tests/**"
        - "package.json"
      deny:
        - ".env*"
        - "secrets/**"
    write:
      allow:
        - "src/**"
        - "tests/**"
      ask:
        - "package.json"
        - "migrations/**"
      deny:
        - ".github/workflows/deploy.yml"
        - "secrets/**"

  shell:
    allow:
      - command: "bun test*"
      - command: "bun run lint*"
      - command: "npx tsgo --noEmit*"
      - command: "git diff*"
      - command: "git status*"
    ask:
      - command: "bun install*"
      - command: "git commit*"
    deny:
      - pattern: "rm -rf /"
      - pattern: "git push --force"
      - pattern: "curl * | sh"
      - pattern: "DROP DATABASE"

  network:
    default: deny
    allow:
      - domain: "github.com"
      - domain: "registry.npmjs.org"
        only_during: install

  dependencies:
    new_dependency: ask
    version_major_upgrade: ask
    unpinned_dependency: deny
    unknown_license: deny

  migrations:
    require_rollback: true
    require_dry_run: true
    require_backup_plan: true
    production_credentials: deny

  agent_scopes:
    scout:
      filesystem_write: deny
      shell: readonly
    builder:
      filesystem_write: allow_scoped
      shell: allow_tests
    critic:
      filesystem_write: deny
      shell: allow_readonly
```

Claude Code and Codex both already expose permissions/approval concepts, but ZYAL should make them more composable and auditable. ([Claude][19])

## 6.5 `experiments`: disjoint patch pools and stateful improvement

This directly addresses your “multiple passes,” “disjointed experiments,” and “stateful building on current state” request.

Current incubator passes are linear. ZYAL should also support candidate pools:

```yaml
experiments:
  enabled: true
  store: sqlite
  default_isolation: git_worktree

  pools:
    - id: auth_fix_candidates
      trigger: task_risk_high
      base: checkpoint:latest_verified
      strategy: disjoint_patch_pool
      candidates: 6
      max_parallel: 3
      diversity:
        encourage_different_files: true
        encourage_different_root_causes: true
        reject_duplicate_patch_shape: true
      worker:
        agent: build
        timeout: 20m
        writes: isolated_worktree

      verify:
        commands:
          - "bun test tests/auth --timeout 30000"
          - "npx tsgo --noEmit"
          - "bun run lint"
        flaky_retry: 2

      rank:
        primary:
          - tests_pass
          - typecheck_pass
          - security_guardrails_pass
        secondary:
          - minimal_diff
          - low_blast_radius
          - no_new_dependency
          - reviewer_score
        tie_breaker: smallest_patch

      lineage:
        after_round:
          keep:
            - best_verified
            - most_promising_failed
          next_round_base: best_verified
          allow_stateful_refinement: true
          max_rounds: 3
```

This gives you two crucial modes:

1. **Disjoint experiments**: independent candidates from the same verified base.
2. **Stateful refinement**: later rounds build on the best verified candidate.

That is how you avoid a single model wandering into jank.

## 6.6 `anti_jank`: reducing vibe coding explicitly

“Vibe coding” becomes dangerous when the agent edits before understanding, cannot prove claims, changes too much, skips tests, hides uncertainty, or treats green tests as complete correctness.

Make anti-jank a formal policy:

```yaml
anti_jank:
  enabled: true

  require_before_edit:
    - failing_signal_identified
    - root_cause_hypothesis
    - affected_files_bounded
    - test_plan_written
    - rollback_plan_written

  prohibit:
    - skipping_tests
    - deleting_tests_without_approval
    - broad_refactor_without_task
    - dependency_add_without_reason
    - modifying_public_api_without_contract_update
    - touching_unrelated_files
    - speculative_large_rewrite

  patch_limits:
    max_files_without_approval: 12
    max_diff_lines_without_approval: 800
    require_plan_update_if_exceeded: true

  required_output:
    every_iteration:
      schema:
        type: object
        required:
          - hypothesis
          - files_changed
          - evidence
          - next_action
          - uncertainty
          - rollback
```

This is not just a style preference. It is a host-enforced way to prevent code churn.

## 6.7 `quality`: richer acceptance than “tests pass”

For serious work, “tests pass” is necessary but not sufficient.

```yaml
quality:
  gates:
    - id: unit_tests
      command: "bun test --timeout 30000"
      assert: { exit_code: 0 }

    - id: typecheck
      command: "npx tsgo --noEmit"
      assert: { exit_code: 0 }

    - id: lint
      command: "bun run lint"
      assert: { exit_code: 0 }

    - id: no_test_deletion
      command: "git diff --name-only --diff-filter=D | grep -E 'test|spec' && exit 1 || exit 0"
      assert: { exit_code: 0 }

    - id: public_api_contract
      command: "scripts/check-api-compat"
      assert: { exit_code: 0 }

    - id: security_scan
      command: "scripts/security-scan --changed"
      assert: { exit_code: 0 }

    - id: benchmark
      command: "scripts/bench --compare main --json .zyal/artifacts/bench.json"
      assert:
        json:
          path: "$.regression_percent"
          lte: 2

  flaky_tests:
    detect: true
    retry_failed: 2
    quarantine_requires_human: true

  mutation_testing:
    enabled_for:
      - security_critical
      - parser
      - auth
    command: "scripts/mutation-test --changed"
```

## 6.8 `rollout`: from local patch to production safety

ZYAL currently stops around commit/push. For ambitious projects, the daemon should understand staged rollout, even if the final deploy is human-gated.

```yaml
rollout:
  enabled: true
  stages:
    - id: local
      type: shell
      command: "bun test && npx tsgo --noEmit"

    - id: ci
      type: github_check
      wait_for:
        - build
        - unit-tests
        - integration-tests
        - security

    - id: preview_env
      type: deployment
      environment: preview
      require_approval: false
      verify:
        - command: "scripts/smoke-preview"

    - id: canary
      type: deployment
      environment: canary
      require_approval: true
      traffic_percent: 1
      monitor:
        duration: 30m
        metrics:
          - error_rate
          - latency_p95
          - rollback_events

    - id: production
      type: deployment
      environment: production
      require_approval: true

  rollback:
    automatic_on:
      - error_rate_gt: 2x_baseline
      - latency_p95_gt: 1.5x_baseline
    require_runbook: true
```

## 6.9 `recovery`: crash, rollback, resume, and cleanup

Long-running systems must assume interruption.

```yaml
recovery:
  resume:
    on_host_restart: true
    on_process_crash: true
    restore_from: latest_consistent_checkpoint
    verify_workspace_before_resume: true

  rollback:
    strategies:
      - git_reset_to_checkpoint
      - revert_commit
      - discard_worktree
      - restore_artifacts
    require_human_for:
      - production_migration
      - pushed_commit
      - external_side_effect

  cleanup:
    on_abort:
      archive_artifacts: true
      delete_unmerged_worktrees: ask
      preserve_logs: true
    on_success:
      delete_scratch: true
      archive_summary: true

  stuck_detection:
    no_new_events_for: 10m
    no_progress_iterations_gte: 5
    same_error_repeated_gte: 3
    on_stuck: incubate_current_task
```

```yaml
    dry_run_first: true
    stale_after: 90d
    protect_pinned: true
    archive_unused: true
    backup_before_change: true

memory:
  namespaces:
    - id: run_lessons
      type: episodic
      retention: long_term
      writable_by: [promotion_review]
    - id: procedures
      type: procedural
      retention: long_term
      writable_by: [skill_curator]
    - id: repo_facts
      type: semantic
      retention: long_term
      writable_by: [host_indexer]
  write_policy:
    after_checkpoint:
      extract:
        - reusable_procedures
        - errors_to_avoid
        - verified_repo_facts
    after_failure:
      extract:
        - failed_hypothesis
        - root_cause_if_known
  security:
    scan_for_prompt_injection: true
    scan_for_secret_exfiltration: true
    require_source_provenance: true

approvals:
  gates:
    - id: approve_new_skill
      when: skill_created
      decisions: [approve, edit, reject]
    - id: approve_memory_write
      when: long_term_memory_write
      decisions: [approve, reject]
      only_for:
        - user_preferences
        - security_sensitive_facts
```

This lets ZYAL accumulate knowledge without allowing the model to silently rewrite institutional memory.

# 8. Improvements to the existing ZYAL blocks

## `loop`

Current:

```yaml
loop:
  policy: forever
```

Upgrade:

```yaml
loop:
  policy: forever
  sleep: 5s
  max_iterations: from_budget
  progress_metric:
    any:
      - new_verified_artifact
      - reduced_failing_tests
      - completed_task
      - new_checkpoint
      - new_root_cause_evidence
  no_progress:
    after_iterations: 5
    action: route_to_incubator
  circuit_breaker:
    max_consecutive_errors: 5
    max_same_error: 3
    max_retries_per_node: 2
    on_trip: pause
```

Reason: “No progress” must be evidence-based, not vibes.

## `checkpoint`

Current checkpointing is git-oriented. Add evidence, artifact bundle, and rollback.

```yaml
checkpoint:
  when: after_verified_change
  noop_if_clean: true
  evidence_bundle:
    include:
      - diff
      - test_report
      - lint_report
      - typecheck_report
      - plan
      - rollback_plan
      - risk_review
  rollback:
    method: revert_commit
    verify_after_rollback:
      - command: "bun test --timeout 30000"
        assert: { exit_code: 0 }
  git:
    add: ["."]
    commit_message: "daemon: {{task_id}} {{summary}}"
    sign: true
    push: ask
```

## `tasks`

Current task discovery is shell-based. Add a real task graph.

```yaml
tasks:
  ledger: sqlite
  source:
    - github_issues
    - linear
    - local_todo
    - shell
  schema:
    required:
      - id
      - objective
      - acceptance
      - risk
      - owner
      - dependencies
  dedupe:
    strategy: semantic_and_file_overlap
  priority:
    score:
      - severity
      - unblock_count
      - estimated_risk
      - confidence
```

## `agents`

Current workers are simple. Add contracts.

```yaml
agents:
  supervisor:
    agent: plan
    responsibilities:
      - maintain_task_graph
      - assign_workers
      - review_evidence
    forbidden:
      - edit_files

  workers:
    - id: builder
      count: 4
      agent: build
      isolation: git_worktree
      contract:
        input_schema: task_capsule
        output_schema: patch_candidate
        must_produce:
          - changed_files
          - test_report
          - risk_summary
          - rollback_plan

    - id: critic
      count: 2
      agent: critic
      isolation: same_session
      contract:
        forbidden:
          - edit_files
        must_produce:
          - objections
          - severity
          - recommendation
```

## `mcp`

MCP is crucial, but ZYAL needs capability pinning and trust levels. MCP is an open protocol for connecting LLM applications to external tools and data sources, so ZYAL should treat MCP servers as capability providers with scoped trust, not just tool lists. ([Model Context Protocol][20])

```yaml
mcp:
  profiles:
    readonly_repo:
      trust: low
      servers: ["filesystem"]
      tools: ["read_file", "list_dir"]
      resources: ["repo://main"]
      side_effects: deny

    github_pr:
      trust: medium
      servers: ["github"]
      tools: ["create_pr", "list_checks"]
      side_effects: ask
      idempotency_required: true

    production_observability:
      trust: high
      servers: ["datadog"]
      tools: ["query_metrics"]
      side_effects: deny
      secrets:
        expose: ["DATADOG_READ_TOKEN"]
        deny_to_llm: true
```

MCP’s elicitation feature also matters: external tools can request user input mid-workflow. ZYAL should route those requests through the same durable approval system rather than letting servers interrupt informally. ([Model Context Protocol][21])

## `on`

Current `on` handlers are useful but need budgets, idempotency, and non-cascade guarantees per handler.

```yaml
on:
  - signal: no_progress
    count_gte: 3
    budget:
      max_fires: 2
      cooldown: 10m
    do:
      - checkpoint_scratch: true
      - route_to: incubator

  - signal: permission_denied
    do:
      - summarize_blocked_action: true
      - request_approval:
          gate: permission_escalation
    cascade: deny
```

## `guardrails`

Guardrails should be split into deterministic, model-based, and human-gated guardrails. LangChain’s docs make this distinction explicit: deterministic guardrails are rule/code based, while model-based guardrails use an LLM for judgment. ([LangChain Docs][22])

```yaml
guardrails:
  deterministic:
    - name: no-force-push
      scope: tool_input
      deny_patterns: ["git push --force", "git push -f"]
      action: block

  model_based:
    - name: suspicious-large-refactor
      scope: file_diff
      model: critic
      prompt: "Does this diff exceed the requested task scope?"
      on_fail: require_human_review

  pii:
    - name: redact-secrets
      strategies: [redact, block]
      patterns: [api_keys, tokens, private_keys]

  human_gated:
    - name: production-risk
      trigger:
        any:
          - path_matches: "migrations/**"
          - public_api_changed: true
          - dependency_added: true
      action: approval
```

## `incubator`

The incubator is the jewel of ZYAL. Upgrade it into a general “hard task maturation pipeline.”

```yaml
incubator:
  enabled: true
  strategy: generate_pool_strengthen

  route_when:
    any:
      - repeated_attempts_gte: 3
      - no_progress_iterations_gte: 5
      - risk_score_gte: 0.7
      - blast_radius_gt: 10
      - unknown_environment_failure: true

  hypothesis_tracking:
    enabled: true
    require_each_pass_to_update:
      - known_facts
      - hypotheses
      - disproven_hypotheses
      - open_questions
      - next_best_test

  passes:
    - id: scout
      type: scout
      context: blind
      writes: scratch_only
      output: artifact:scout_report

    - id: hypothesis_pool
      type: idea
      context: inherit
      writes: scratch_only
      count: 5
      diversity: maximize

    - id: adversarial_critic
      type: critic
      context: critic
      writes: scratch_only
      require:
        - list_failure_modes
        - identify_missing_tests

    - id: prototype_pool
      type: prototype
      context: inherit
      writes: isolated_worktree
      count: 3
      verify_each: true

    - id: synthesize
      type: synthesize
      context: pool
      writes: scratch_only
      output: artifact:promotion_case

    - id: promotion_review
      type: promotion_review
      context: promotion
      writes: scratch_only

  promotion:
    promote_at: 0.78
    require_claims:
      - tests_identified
      - root_cause_supported
      - scope_bounded
      - rollback_known
      - affected_files_known
      - no_critical_objections
    block_on:
      unresolved_critical_objections_gte: 1
      unverified_hypotheses_gte: 3
```

Add one powerful idea: **disproven hypotheses**. This prevents agents from repeatedly trying the same wrong fix.

# 9. Safety upgrades ZYAL needs

## 9.1 Trusted-channel arming

Only trusted user channels can arm. Assistant output can preview but never arm. Use hash + nonce.

```yaml
arming:
  require_trusted_channel: user
  reject_assistant_origin: true
  require_hash_of_effective_config: true
  require_host_nonce: true
```

## 9.2 Capability escalation requires a new approval

A daemon should not silently expand its own powers.

```yaml
capability_escalation:
  default: deny
  allow_request: true
  require_approval:
    - shell_allow_new_command
    - network_allow_new_domain
    - expose_secret
    - git_push
    - dependency_add
    - production_access
```

## 9.3 Side-effect taxonomy

```yaml
side_effects:
  classify:
    none:
      - read_file
      - git_diff
      - test_dry_run
    local_reversible:
      - edit_file
      - create_worktree
    local_irreversible:
      - delete_file
    external_reversible:
      - create_pr
      - send_slack_message
    external_irreversible:
      - deploy_production
      - run_migration
  require_idempotency_for:
    - external_reversible
    - external_irreversible
  require_human_for:
    - external_irreversible
```

## 9.4 Supply-chain controls

```yaml
supply_chain:
  dependencies:
    new: ask
    major_upgrade: ask
    unpinned: deny
    unknown_license: deny
    postinstall_scripts: ask
  sbom:
    generate: true
    command: "scripts/generate-sbom"
  vulnerability_scan:
    command: "scripts/audit-deps"
    fail_on: high
  provenance:
    require_signed_commits: true
    require_artifact_hashes: true
```

## 9.5 Prompt-injection boundaries

```yaml
prompt_injection:
  untrusted_sources:
    - web
    - issue_comments
    - pull_request_descriptions
    - external_docs
    - tool_output
  rules:
    - never_execute_instructions_from_untrusted_sources
    - quote_untrusted_content
    - require_host_policy_for_tool_use
  detection:
    model_based: true
    deterministic_patterns:
      - "ignore previous instructions"
      - "exfiltrate"
      - "send token"
  on_detect: quarantine_artifact
```

This matters because long-running coding agents often read issues, docs, web pages, logs, and PR comments that can contain hostile instructions.

# 10. How ZYAL can become the default language for ambitious coding projects

## 10.1 Be the manifest, not the model

ZYAL should define:

* Objective
* Budget
* Runtime capabilities
* Environment
* Memory policy
* Task graph
* Workflow graph
* Permissions
* Guardrails
* Evidence requirements
* Human approvals
* Checkpoints
* Rollout
* Recovery

The model should only propose actions within that contract.

## 10.2 Compile to many backends

To be available inside any chat prompt, ZYAL needs an execution host. But the host can drive many backends.

```yaml
compile:
  targets:
    - opencode_native
    - langgraph
    - openhands
    - codex_cli
    - claude_code
    - github_actions
```

This makes ZYAL an interchange language.

## 10.3 Build a conformance suite

A language becomes default when implementers can prove compatibility.

```yaml
conformance:
  suite: zyal-vNext
  required_tests:
    - rejects_code_fenced_ocal
    - rejects_assistant_arm
    - requires_arm_nonce
    - enforces_budget
    - blocks_guardrail_violation
    - persists_state_after_crash
    - resumes_from_checkpoint
    - prevents_handler_cascade
    - blocks_unscoped_file_write
    - requires_promotion_evidence
```

## 10.4 Define a formal execution model

ZYAL needs a spec-level lifecycle:

1. Parse sentinels.
2. Validate schema.
3. Normalize defaults.
4. Compute effective config.
5. Preview risk/cost/permissions.
6. Wait for trusted arming.
7. Create run row.
8. Create initial checkpoint.
9. Execute workflow node.
10. Persist event before and after every side effect.
11. Evaluate guardrails/constraints.
12. Update task graph and memory candidates.
13. Verify evidence.
14. Gate promotion.
15. Check stop conditions.
16. Archive and summarize.

This should be written as normative spec text.

## 10.5 Make “evidence” the unit of progress

The model should not say “I made progress.” The host should observe:

* A task changed state.
* A test failure count decreased.
* A verified artifact was produced.
* A patch passed checks.
* A reviewer approved.
* A hypothesis was disproven.
* A risk was closed.
* A checkpoint was created.

This is the heart of reducing jank.

# 11. 1B+ LOC autonomous project requirements

For billion-line projects, ZYAL needs to assume the following:

## Requirement 1: No whole-repo context

The system must retrieve from indexes, not stuff files into context.

Needed ZYAL features:

```yaml
codebase_index
memory.retrieval
task_graph.sharding
blast_radius
test_coverage_map
ownership_map
```

## Requirement 2: Work must be sharded and leased

Thousands of tasks need durable ownership, stale detection, reclaim, and merge strategy.

Needed ZYAL features:

```yaml
task_graph.lease
task_graph.dependencies
agents.workers
locks
merge_batches
conflict_policy
```

## Requirement 3: Every patch needs local and global evidence

Local package tests are not enough. Global checks, API compatibility, security, performance, and canary metrics matter.

Needed ZYAL features:

```yaml
quality.gates
rollout.stages
evidence.claims
promotion.require_claims
```

## Requirement 4: Merge strategy must be explicit

```yaml
merge:
  strategy: batched_trunk_integration
  batch_size: 20
  conflict_policy:
    same_file: isolate
    same_symbol: serialize
    public_api: require_arch_review
  rebase:
    before_each_batch: true
    autostash: true
  validation:
    after_each_batch:
      - affected_tests
      - typecheck
      - lint
    after_all_batches:
      - global_ci
```

## Requirement 5: Ownership and approval are unavoidable

```yaml
ownership:
  source: CODEOWNERS
  require_owner_approval_for:
    - public_api_changed
    - security_sensitive
    - package_boundary_changed
    - migration
```

## Requirement 6: Memory must be hierarchical

* Run memory: what happened in this daemon.
* Task memory: task capsule, failures, attempts.
* Repo memory: verified facts about the codebase.
* Org memory: procedures and policies.
* User memory: preferences.
* Skill memory: reusable procedures.

## Requirement 7: Evaluation must be continuous

```yaml
evals:
  regression_suite:
    schedule: after_checkpoint
    commands:
      - "scripts/run-agent-evals --changed"
  agent_quality:
    score:
      - accepted_patch_rate
      - rollback_rate
      - approval_rejection_rate
      - diff_minimality
      - repeated_error_rate
      - cost_per_merged_change
```

## Requirement 8: The system must know when not to code

Add a `decision` node:

```yaml
workflow:
  nodes:
    - id: decide_action
      type: decision
      options:
        - code_change
        - ask_human
        - write_design_doc
        - file_issue
        - park_task
        - request_missing_env
      require_reason: true
```

This prevents agents from forcing code changes when the real blocker is ambiguous requirements, missing credentials, flaky CI, or an underspecified product decision.

# 14. The sharpest critique of current ZYAL

The current ZYAL spec is excellent but has six structural weaknesses:

1. **It treats looping as primary.** The future should be durable workflows with loops as one node type.
2. **It under-specifies state.** “SQLite is truth” needs typed events, replay, idempotency, spans, and hash chains.
3. **It lacks memory governance.** Context compaction is not enough for multi-day work.
4. **It lacks environment reproducibility.** Serious work needs sandbox, services, secrets, network, and resource policy.
5. **It lacks human decision semantics.** `ask` is not enough; approvals need durable decisions and evidence bundles.
6. **It lacks scale primitives.** 1B+ LOC work needs codebase indexes, sharding, leases, ownership, blast-radius analysis, and batch merge strategy.

None of these invalidate ZYAL. They are exactly the upgrades that turn it into a serious control plane.

# 15. The “most powerful” version of ZYAL in one sentence

**ZYAL should be a portable, host-enforced, evidence-gated operating contract for autonomous engineering campaigns, with durable workflows, typed state, governed memory, reproducible sandboxes, scoped capabilities, task leases, experiment pools, human approvals, and rollout-aware promotion.**

That would make it much more than a YAML wrapper around agents. It would make it the language of record for controlled autonomous software engineering.
