I studied the ZYAL draft as a proposed **host-enforced agent operating contract**, not just a YAML runbook. That framing is the biggest strength: ZYAL already says the host owns lifecycle, arming, budgets, state, guardrails, evidence, approvals, rollback, and done-ness, with SQLite as source of truth and generated mirror files for inspection.

I read “LangChane” as **LangChain/LangGraph**, and “Hermis” as **Hermes Agent**. The key lesson from the ecosystem is: **ZYAL should not become another agent framework. It should become the portable control plane that safely composes existing agent frameworks, tool protocols, UI protocols, memory systems, and durable workflow engines.**

## Reference map: what ZYAL should absorb, not duplicate

LangGraph already covers durable execution, human-in-the-loop control, short- and long-term memory, tracing, and production deployment for long-running stateful agents. That overlaps heavily with ZYAL’s workflow, context, memory, and observability ambitions. ([LangChain Docs][1])

MCP is the dominant tool/data integration layer: it standardizes how hosts, clients, and servers expose context, tools, and capabilities over JSON-RPC. ZYAL’s `mcp`, `capabilities`, `security`, and `sandbox` blocks should be MCP-aware rather than inventing a parallel tool protocol. ([Model Context Protocol][2])

A2A is the relevant agent-to-agent protocol: it supports asynchronous `Task` processing, streaming status updates, artifacts, task lookup, and immediate progress feedback for complex work. ZYAL should use A2A for remote delegation rather than inventing its own cross-agent wire format. ([A2A Protocol][3])

AG-UI is the user-facing event protocol layer: it standardizes state, UI intents, user interactions, streaming, multimodal attachments, frontend tool calls, and shared state between agent backends and applications. ZYAL’s TUI/API/event stream should map cleanly to AG-UI. ([AG-UI][4])

OpenClaw and Hermes are closer to always-on personal-agent products than pure protocols. OpenClaw’s official repo describes a local, always-on personal assistant with a gateway control plane and many chat channels; Hermes positions itself as a self-hosted agent with persistent memory, auto-generated skills, scheduled automations, subagents, sandboxing, and multi-channel presence. ([GitHub][5])

Agent Protocol’s useful abstraction is **Runs, Threads, Store**: execute agents, organize multi-turn executions, and manage long-term memory. ZYAL should be able to export/import that shape. ([GitHub][6])

OpenAI’s Agents SDK docs emphasize the “server owns orchestration, tool execution, state, and approvals” path, which is exactly ZYAL’s philosophical lane; its tracing records LLM generations, tool calls, handoffs, guardrails, and custom events. ([OpenAI Developers][7])

For memory, Letta, Mem0, Zep, and Hermes are directly relevant. Letta distinguishes semantic archival memory queried on demand from pinned working memory; Mem0 separates conversation/session/user/org memory and has research claiming large latency and token-cost reductions versus full-context approaches; Zep builds temporal knowledge graphs that handle changing facts over time; Hermes uses bounded curated memory injected at session start. ([Letta Docs][8])

For reliability, ZYAL should borrow from durable workflow systems. Temporal’s saga pattern uses compensating steps for long-running transactions; Dapr workflows persist timers and retry state across restarts. ([Temporal][9])

For security, MCP’s own security guidance already calls out confused-deputy risks, per-client consent, token-passthrough as an anti-pattern, SSRF, session hijacking, local server compromise, and scope minimization. OWASP’s GenAI Security Project now explicitly covers generative AI, LLM, and agentic application risks, while NIST’s AI RMF is a voluntary framework for managing AI risks across design, deployment, and evaluation. ([Model Context Protocol][10])

## Gaps and potential issues I would fix first

The most important spec-level issue is **versioning clarity**. The sentinel says `ZYAL v1`, the body says `version: v1`, but the document includes v2 and v2.1 blocks. That creates parser, compatibility, and safety ambiguity. Use `version: v2.1`, `requires: [...]`, and explicit feature negotiation.

There are also several internal consistency issues: the document says “38 total top-level keys” but lists 40 by my count; “optional keys” includes required keys; `job.risk` is typed as `string[]` but examples use scalar `medium`; and `git_clean: {}` appears even though the earlier syntax says `allow_untracked` is required. These are small but important because ZYAL’s value depends on being machine-verifiable.

`permissions` and `capabilities` need strict precedence rules. My recommendation: `capabilities` should supersede `permissions`; `permissions` becomes a legacy shorthand compiled into `capabilities`; `command_floor` always wins.

`RUN_FOREVER` is rhetorically dangerous. The protocol is actually lease-bounded by budgets, approvals, and circuit breakers. I would rename the human arming phrase to something like `ZYAL_ARM LEASED_DAEMON`, while keeping `RUN_FOREVER` as a backwards-compatible alias.

SHA-256 evidence signing is not enough. A hash proves integrity, not authorship or authority. ZYAL needs canonicalization plus signatures, signer identity, signing purpose, timestamp, key rotation, and revocation.

Memory is currently promising but under-specified. Long-loop agents fail less from “not enough context” than from **bad memory lifecycle**: stale facts, poisoned tool output, lossy summaries, repeated failed plans, and retrieval overconfidence.

## Top 10 best ideas to dramatically upgrade ZYAL

### 1. Add a real **Memory OS**, not just memory stores

This is the biggest upgrade. ZYAL should treat memory as a governed, typed, versioned subsystem with lifecycle rules, not as “summaries saved somewhere.” Long loops need working memory, episodic memory, semantic memory, procedural memory, negative memory, organizational memory, and invalidation rules.

A stronger block would look like:

```yaml
memory:
  topology:
    working:
      scope: iteration
      retention: until_iteration_end
      inject: always
    episodic:
      scope: task
      indexed_by: [time, actor, requirement_id, artifact_hash]
      retention: until_archive
    semantic:
      scope: run
      backend: vector_graph
      retrieve: on_demand
    procedural:
      scope: global
      promotion_gate: skill_review
    negative:
      scope: task
      purpose: failed_hypotheses_and_dead_ends
      retrieve_before_retry: true

  lifecycle:
    promote_to_long_term:
      requires_evidence: true
      confidence_gte: 0.75
      source_must_be: [tool_result, test_result, human_note, approved_summary]
    invalidate:
      on_requirement_change: true
      on_test_failure: true
      on_source_file_change: true
    compact:
      method: extract_facts_open_questions_decisions
      preserve: [requirements, blockers, failed_attempts, invariants]
      verify_against_sources: true

  retrieval:
    budget_tokens: 3000
    require_source_refs: true
    include_negative_memory: true
    diversity: [recent, high_confidence, unresolved, contradictory]

  safety:
    quarantine_untrusted_tool_output: true
    memory_poison_scan: true
    two_phase_commit: true
```

This borrows the right lessons: Letta separates archival memory from working context, Mem0 layers memory and shows why full-context replay is expensive, Zep models temporal facts that change over time, and Hermes keeps memory bounded and curated. ([Letta Docs][8])

### 2. Make ZYAL the **interop envelope** for MCP, A2A, AG-UI, Agent Protocol, LangGraph, OpenClaw, and Hermes

ZYAL’s killer positioning should be: **one host-enforced contract that can run across existing agent ecosystems**.

Add an `interop` block:

```yaml
interop:
  mcp:
    import_servers_from: ".mcp.json"
    enforce_capabilities: true
    deny_unregistered_tools: true

  a2a:
    allowed_remote_agents:
      - card_url: "https://example.com/.well-known/agent.json"
        max_task_cost_usd: 2.00
        allowed_artifacts: [patch, report, test_results]

  ag_ui:
    expose_event_stream: true
    redact_raw_prompts: true
    expose: [state_delta, task_status, artifact_preview, approval_request]

  agent_protocol:
    export:
      runs: true
      threads: true
      store: true

  langgraph:
    import_state_graph: optional
    map_checkpoints_to: sqlite

  gateways:
    openclaw: { mode: external_trigger_only }
    hermes: { mode: delegated_subagent }
```

This prevents wheel reinvention. MCP already owns tool/data integration, A2A owns remote agent collaboration, AG-UI owns frontend/backend agent interaction, Agent Protocol owns a useful runs/threads/store shape, and LangGraph owns durable stateful agent graphs. ([Model Context Protocol][2])

### 3. Add **replayable witness packs** for every serious run

SQLite-as-truth is excellent, but ZYAL should go further: every run should be replayable, inspectable, and forensically auditable.

Add:

```yaml
witness:
  enabled: true
  capture:
    - canonical_zyal_hash
    - model_profile_and_version
    - prompt_inputs_redacted
    - tool_inputs_outputs
    - memory_deltas
    - approvals
    - environment_fingerprint
    - artifact_hashes
    - command_exit_codes
    - git_base_and_diff
  replay:
    modes: [exact_when_possible, deterministic_tools_only, audit_only]
  export:
    format: zyal-witness-v1
```

This is how ZYAL becomes enterprise-grade. When an agent makes a surprising change after 9 hours, the user should not just see a summary; they should get a signed causal record of what happened, why, with which tools, under which permissions, and based on which memories. OpenAI Agents tracing already records generations, tool calls, handoffs, guardrails, and custom events; OpenTelemetry is defining GenAI and agent spans; ZYAL should emit both. ([OpenAI GitHub][11])

### 4. Replace “objective text” with a **requirements ledger and evidence graph**

ZYAL has `job.objective`, `stop`, `evidence`, and `done`, but the missing layer is a typed requirements ledger. Agents drift when they optimize the nearest passing test instead of the actual user goal.

Add:

```yaml
requirements:
  source: github_issue
  items:
    - id: R1
      statement: "All existing tests pass without deleting or weakening tests."
      acceptance:
        - shell: "bun test --timeout 30000"
        - quality_check: no_test_deletion
      owner: user
      priority: must
    - id: R2
      statement: "Root cause is documented before production patch."
      acceptance:
        - evidence_exists: root_cause

drift:
  detect_requirement_changes: true
  model_may_propose_changes: true
  human_must_approve_changes: true

evidence_graph:
  require_edges:
    - from: failing_test
      to: root_cause
    - from: root_cause
      to: patch
    - from: patch
      to: verification
```

The upgrade is subtle but profound: ZYAL stops asking “did the loop stop?” and starts asking “which requirement has been satisfied by which evidence?” That makes “done” defensible.

### 5. Add **identity-bound capability leases**, data classification, and a real secrets broker

The v2.1 `capabilities` block is one of ZYAL’s strongest ideas, but it should become identity-aware. A run should have its own identity, short-lived credentials, audited scopes, data labels, and network egress limits.

Add:

```yaml
identity:
  run_principal: "zyal-runner:${run_id}"
  credential_ttl: 30m
  require_audience_bound_tokens: true
  prohibit_token_passthrough: true

data_policy:
  labels:
    - public
    - internal
    - secret
    - regulated
  default_label: internal
  tool_output_labeling: required
  secret_to_public_flow: block

secrets:
  broker: host
  expose_to_model: never
  expose_to_tools: by_capability_only
  redact_in_logs: true
  rotate_on_run_end: true
```

This directly addresses MCP’s known security problem space: confused-deputy failures, per-client consent, token-passthrough, SSRF, session hijacking, and scope minimization. ZYAL should make the safe pattern the default, not an optional best practice. ([Model Context Protocol][10])

### 6. Define a **skill supply-chain lifecycle**

Hermes explicitly emphasizes persistent memory and auto-generated skills; OpenClaw and similar assistants make reusable skills/extensions central to usefulness. ZYAL should support self-improving skills, but only through quarantine, tests, signatures, and human promotion. ([Hermes Agent][12])

Add:

```yaml
skills:
  registries:
    - id: trusted-local
      path: ".jekko/skills"
      trust: signed_only

  generated:
    allowed: true
    initial_state: quarantine
    writes: scratch_only
    require:
      - unit_tests
      - example_invocations
      - capability_manifest
      - security_review
      - human_approval

  promotion:
    from: quarantine
    to: trusted-local
    approval: skill_review
    sign_with: host_key

  revocation:
    on_failed_regression: disable
    on_security_violation: delete_and_block_hash
```

This turns “the agent learned a trick” into a governed artifact. Without this, long-running agents eventually accumulate unreviewed procedures that become a hidden attack surface.

### 7. Add an **eval-native improvement loop**

ZYAL already has `experiments`, but it should make evaluation a first-class protocol surface. The protocol should continuously measure whether memory, model routing, tools, prompts, skills, and workflows are improving or degrading the agent.

Add:

```yaml
evals:
  suites:
    - id: regression
      source: ".jekko/evals/regression.yml"
      run_before_promote: true
    - id: memory_quality
      checks:
        - stale_fact_rate_lt: 0.05
        - retrieval_precision_gte: 0.80
        - negative_memory_reuse_gte: 0.60
    - id: security
      adversarial:
        - prompt_injection
        - tool_poisoning
        - secret_exfiltration

  shadow:
    enabled: true
    compare_against: last_green_run
    no_write: true

  scorecard:
    publish_to: evidence_bundle
```

This is what makes ZYAL scientifically significant. It becomes not just a run-control format, but a way to produce measurable agent improvement over time. LangGraph and OpenAI’s Agents stack both emphasize tracing/debugging/evaluation as production needs, and ZYAL should specify the contract-level version of that. ([LangChain Docs][1])

### 8. Add **federated delegation** through A2A-style agent contracts

Multi-agent work inside one process is not enough. ZYAL should be able to lease work to remote agents, local assistants, specialized code agents, research agents, browser agents, security agents, or personal assistants—without surrendering host control.

Add:

```yaml
delegation:
  protocol: a2a
  registry:
    - id: security-reviewer
      agent_card: "https://sec.example/.well-known/agent.json"
      trust: approved_vendor
      allowed_inputs: [diff, dependency_list, threat_model]
      forbidden_inputs: [secrets, customer_data]
      max_cost_usd: 5.00
      timeout: 20m

  contracts:
    require_artifacts: [findings, severity, reproduction, recommended_patch]
    require_streaming_status: true
    allow_cancellation: true
    verify_returned_artifacts: true
```

A2A already models async tasks, task status updates, artifacts, and streaming progress. ZYAL should govern which agents can receive which data, what they must return, how much they may cost, and when their result counts as evidence. ([A2A Protocol][3])

### 9. Add durable workflow primitives: **leases, locks, idempotency, sagas, compensations**

ZYAL has loops, workflows, retries, rollbacks, and checkpoints. It still needs the boring distributed-systems machinery that makes long-running work safe.

Add:

```yaml
durability:
  idempotency:
    key_template: "{{repo}}:{{branch}}:{{task_id}}:{{git_sha}}"
    on_duplicate: attach_to_existing_run

  locks:
    - resource: "git:branch:main"
      mode: exclusive
      ttl: 30m
      renew: true

  leases:
    heartbeat_every: 30s
    expire_after: 2m
    on_expire: pause_and_release_locks

  compensation:
    required_for: [external_side_effect, deployment, migration]
    order: reverse
    verify_before_promote: true

  durable_timers:
    backend: sqlite
    survive_restart: true
```

This is the difference between “agent loop” and “durable autonomous operation.” Temporal’s sagas show why compensating actions matter for long-running transactions, and Dapr’s durable timers/retries show why timers and retry state must survive process restarts. ([Temporal][9])

### 10. Build an **ZYAL compiler** with static verification, policy profiles, and risk mapping

ZYAL needs a compiler, not only a parser. The compiler should reject unsafe or ambiguous runbooks before arming.

Add:

```yaml
policy:
  profile: enterprise_coding
  engines: [builtin, rego]
  require_static_checks:
    - every_loop_has_budget
    - every_workflow_has_terminal_state
    - every_write_scope_has_capability
    - every_remote_agent_has_data_policy
    - every_secret_has_broker
    - every_promotion_has_evidence
    - no_permission_capability_conflict
    - no_unreachable_approval_gate
    - rollback_required_for_high_risk_paths

risk_mapping:
  frameworks:
    - owasp_genai
    - nist_ai_rmf
  output: evidence_bundle
```

This would make ZYAL credible beyond developer demos. OWASP is explicitly focused on LLM and agentic application risks, and NIST’s AI RMF provides a recognized risk-management frame; ZYAL should be able to emit a “risk controls satisfied” report as part of every armed run. ([OWASP Foundation][13])

## My strongest strategic recommendation

Do **not** position ZYAL as “a better LangGraph,” “a better MCP,” or “a better OpenClaw/Hermes.” Position it as:

> **The host-enforced operating contract for long-running autonomous agents.**

That means ZYAL should own:

1. authority, arming, budgets, approvals, and lifecycle;
2. memory governance and compaction across long loops;
3. evidence, replay, and auditability;
4. capability leases and data boundaries;
5. interop with MCP, A2A, AG-UI, Agent Protocol, LangGraph, OpenAI Agents SDK, OpenClaw, Hermes, and future systems.

The three upgrades I would implement first are: **Memory OS**, **Interop Profiles**, and **Witness/Replayer + Compiler**. Those three would turn ZYAL from a powerful runbook into a serious protocol layer.
