## Executive verdict

Your current benchmark is a **very strong memory-systems engineering test**, especially for bitemporal recall, contradiction surfacing, deterministic rebuilds, privacy redaction, and provenance. It already goes beyond typical long-term-memory QA benchmarks by requiring `recall_at`, `recall_as_of`, causal masking, fixture compounding, vault canaries, unsafe-skill refusal, and byte-identical deterministic outputs.

But to show that an agent can **learn advanced math/science concepts**, I would evolve it from a “memory correctness benchmark” into a **memory-learning benchmark**. The key distinction:

> The best test should not merely ask whether the agent remembers technical facts.
> It should ask whether the agent can ingest a novel technical theory, preserve its provenance, update it under contradictions, compress it without drift, and use it to solve new problems that were not explicitly stored.

That is the core upgrade.

Existing public benchmarks still leave room for this. LongMemEval tests information extraction, multi-session reasoning, temporal reasoning, knowledge updates, and abstention across 500 questions, but it is still primarily conversational-memory QA. ([arXiv][1]) LoCoMo is stronger on very long conversations, personas, temporal event graphs, and multimodal dialogue generation, but it is not primarily a deterministic formal-learning harness. ([arXiv][2]) MemoryAgentBench focuses on accurate retrieval, test-time learning, long-range understanding, and forgetting/conflict competencies. ([arXiv][3]) Newer work such as LoCoMo-Plus and Memora is moving toward latent constraints, consolidation over weeks/months, and avoiding retrieval-only framing, which supports exactly the direction I would take your test. ([arXiv][4])

## The highest-value change: add “novel formal worlds”

The most advanced version should generate **synthetic-but-formal technical domains** that the model cannot already know from pretraining. This is essential. If you test Newton, Higgs, Schrödinger, or Bolzano-Weierstrass, you are partly testing pretrained knowledge. Your current math/science fixtures include familiar equations and textbook theorems, plus deliberately broken variants such as `F = m/a` and `E = mc`, which are good for unit-mismatch and contradiction tests but not sufficient to prove new concept learning.

I would add generated domains such as:

| Domain                      | What the agent learns                                            | Deterministic oracle             |
| --------------------------- | ---------------------------------------------------------------- | -------------------------------- |
| Synthetic algebra           | New operators, axioms, lemmas, counterexamples                   | Finite-model checker             |
| Synthetic physics           | New units, conservation laws, calibration constants, experiments | Unit engine + numeric simulator  |
| Synthetic biology/chemistry | Toy reaction networks, pathway inhibition, causal interventions  | Graph/ODE-lite evaluator         |
| Synthetic theorem DAGs      | Definitions, lemmas, theorems, proof dependencies                | Proof checker + dependency graph |
| Synthetic CS theory         | Tiny type system, operational semantics, program invariants      | Parser + evaluator               |
| Synthetic lab protocols     | Versioned procedures, safety constraints, tool choices           | Workflow validator               |

The benchmark should teach the agent a “micro-theory” and then test it on **transfer tasks**. For example, define a fictional algebraic object called a `Kelm ring` with three axioms and two examples. Later ask the agent to classify a new structure, prove a small lemma, find a counterexample, or explain why a previous theorem no longer holds after an axiom update. The answer can be checked by a finite enumerator rather than an LLM judge.

This is the single biggest step toward testing true advanced memory.

## The benchmark I would build

I would structure it as **three layers**:

### Layer 1: Fast deterministic CI benchmark

This is the version that runs on every commit. It should be small, hard, and fully deterministic.

Target: **200–300 fixtures, under 10 seconds**.

It should include:

1. **Learning packet ingestion**
   The agent receives a short technical corpus: definitions, examples, diagrams serialized as text, equations, false leads, source IDs, timestamps, and privacy labels.

2. **Immediate recall and application**
   It must answer simple questions to prove the content entered memory correctly.

3. **Delayed transfer**
   Later fixtures require combining concepts from earlier packets, not just retrieving one item.

4. **Revision and contradiction**
   A later source invalidates a lemma, experiment, calibration, or theorem dependency.

5. **Bitemporal queries**
   The agent must distinguish “what is true now,” “what was true at world time T,” and “what the agent knew as of transaction time T.” Your current benchmark already tests this distinction with `recall_at` and `recall_as_of`, and that is one of its strongest design choices.

6. **Compression audit**
   The agent must summarize or consolidate memory, then later answer from the compressed state without losing exceptions, caveats, or dependency invalidations.

7. **Forget/redact audit**
   The agent must forget or redact sensitive items while still preserving safe derived knowledge.

8. **Rebuild determinism**
   Rebuild from event log must produce byte-identical outputs and state hashes.

### Layer 2: Hidden seeded challenge set

Publish the generator, but not all seeds.

This prevents overfitting while preserving determinism. The public benchmark could include seed ranges like:

```text
public_ci_seeds:      0000–0099
public_research:      0100–0999
private_leaderboard:  1000–1999
audit_holdout:        9000–9999
```

Every fixture remains deterministic because the seed controls the corpus, timestamps, symbol names, distractors, and expected answers. But candidates cannot simply string-match NovaQ-like scenarios. Your spec already recognizes synthetic overfit risk around NovaQ; seeded formal worlds are the cleaner generalization.

### Layer 3: Research-scale stress mode

This version can run longer and should test scaling:

Target: **10k–100k events**, but still with deterministic outputs.

It should measure:

* recall accuracy versus corpus size
* latency
* memory footprint
* context budget
* retrieval minimality
* state rebuild time
* degradation after compression
* performance after many updates and retractions

Your current spec explicitly notes that 100 fixtures is small and not enough for fine-grained statistical claims, even though it is good for CI.  I would keep the 100-fixture spirit for fast testing, but add a seeded scale mode.

## Revised scoring rubric

Your current 10-axis rubric is reasonable, but for “advanced memory learning,” I would change the weights.

| Axis                              | Weight | What it should measure                                                  |
| --------------------------------- | -----: | ----------------------------------------------------------------------- |
| Novel concept learning            |     16 | Learns definitions, examples, notation, constraints from scratch        |
| Transfer reasoning                |     14 | Solves new problems not explicitly stored                               |
| Formal math/proof handling        |     12 | Proof DAGs, counterexamples, theorem invalidation                       |
| Scientific model updating         |     12 | Experiments, calibration, falsification, causal interpretation          |
| Bitemporal correctness            |     10 | `now`, `world_time`, `tx_time`, no future leakage                       |
| Provenance and dependency tracing |     10 | Exact support IDs, minimal citations, dependency closure                |
| Compression without drift         |      8 | Summaries preserve exceptions, uncertainty, provenance                  |
| Privacy, redaction, forgetting    |      8 | No canary leaks, transformed leaks blocked, safe derived facts retained |
| Procedural/tool memory            |      6 | Skill selection, versioned procedures, unsafe refusal                   |
| Determinism and efficiency        |      4 | Byte-stable rebuild, latency, memory footprint                          |

This shifts the benchmark from “retrieval plus safety” toward “learning plus technical use,” while preserving your existing strengths.

## The most important new fixture types

### 1. Novel theorem-DAG learning

The agent receives:

```text
Definition D1: A mavoid is a finite set S with operator ⊙ satisfying ...
Definition D2: A blue element is ...
Lemma L1: In every regular mavoid, ...
Theorem T1: If S is regular and has a blue identity, then property Q holds.
Proof: depends on D1, D2, L1.
```

Later, fixture N says:

```text
Counterexample C1 shows Lemma L1 fails when |S| is odd.
```

Tests:

* Current recall should mark T1 as conditionally invalid or stale.
* `recall_as_of` before C1 should return T1 as accepted.
* `recall_at` should distinguish theorem validity from agent knowledge time.
* A compression summary must not keep saying “T1 is universally true.”
* A proof query must cite D1, D2, L1, and C1.

This directly tests advanced mathematical memory: dependency-aware learning, not fact recall.

### 2. Synthetic physics with unit and calibration traps

The agent receives a fictional instrument:

```text
The Oryn-5 detector measures flux Φ in drams/square-tick.
Calibration v1: Φ = k A / τ, k = 7.2.
Calibration v2: k = 6.8 after thermal drift correction.
Experiment E1: suggests a 2.4σ anomaly.
Experiment E2: later shows timing artifact.
```

Tests:

* Compute a value under v1 and v2.
* Detect dimensional mismatch in a proposed formula.
* Answer “what would the agent have believed before E2?”
* Prefer corrected calibration now.
* Preserve the tentative nature of 2.4σ.
* Do not silently convert a hypothesis into a verified fact.

This extends your NovaQ idea but makes it generative and harder to overfit.

### 3. Counterexample construction

Do not just ask the agent to recall a theorem. Ask it to find a violating instance.

Example:

```text
Given the learned axiom system, produce a 3-element structure where property P holds but Q fails.
```

The grader checks the structure with a finite model validator.

This is extremely valuable because it tests whether memory contains **operational knowledge**, not just text.

### 4. Proof repair after memory update

The agent first learns a proof. Later a lemma is falsified. Then ask:

```text
Can the original theorem be repaired under a stronger condition?
```

Valid outputs:

* “No, because counterexample C1 also satisfies the stronger condition.”
* “Yes, if condition R is added; proof now depends on L2 instead of L1.”

The grader can check dependency IDs and finite-model validity.

This reveals whether the memory system tracks theorem dependency closure.

### 5. Latent constraint memory

LoCoMo-Plus argues that realistic memory often requires applying implicit user constraints rather than answering explicit recall questions. ([arXiv][4]) Add fixtures where the agent learns a constraint early:

```text
User is preparing a middle-school science fair project.
User cannot use calculus.
User prefers open-access sources.
User is allergic to latex gloves.
```

Later ask:

```text
Design an experiment to test the Oryn-5 flux law.
```

Correct answer must avoid calculus, use safe materials, cite open-access references, and avoid latex. This tests memory as **policy and constraint application**, not just retrieval.

### 6. Compression adversarial tests

Ask the memory system to consolidate 40 events into a summary. Then test:

* rare exception retained
* contradiction retained
* source provenance retained
* uncertainty retained
* privacy labels retained
* outdated claims not promoted
* theorem dependencies not flattened

Your current benchmark already defines compression drift as a pathology, but I would make it a central axis rather than a side tag.

### 7. Memory minimality and over-retrieval

A good memory system should retrieve the right evidence, not dump everything.

Add a penalty for:

* too many `used_ids`
* irrelevant citations
* citing stale sources without warning
* retrieving private items even if redacted
* answering from unsupported memory

A strong agent should return the **minimal sufficient support set**.

### 8. Privacy transformations

Current canary checks are good, but the next version should test transformed leakage:

* raw canary
* base64 encoded canary
* split canary across fields
* canary inside citation metadata
* canary in tool result
* canary in debug trace
* canary in compressed summary
* canary in forgotten-but-derived memory
* canary requested through “quote your sources exactly”
* canary hidden in JSON keys, not values

Your spec already uses synthetic vault canaries and checks that raw canary strings do not appear in answers.  The next version should check every output channel, not just final answer text.

### 9. Tool-version memory

Procedural memory should include changing tools.

Example:

```text
doi_normalize v1 lowercases DOI prefixes.
doi_normalize v2 preserves case for publisher code X.
v1 later marked unsafe for publisher X.
```

Tests:

* choose correct version
* refuse deprecated tool
* explain why
* update after feedback
* answer `as_of` before deprecation differently

This tests memory over skills, not just facts.

### 10. Multi-agent/federated memory

Your roadmap already identifies federation fixtures as a valuable extension.  I would add a deterministic two-agent scenario:

* Agent A observes experiment E1.
* Agent B observes falsifier E2.
* They sync ledgers with CRDT ordering.
* Agent A must update current belief but still answer what it knew before sync.
* Conflicting privacy permissions must be resolved conservatively.

This is a strong test for next-generation agent memory because real systems will have shared, partial, and permissioned memory.

## Anti-gaming measures I would add

### Use isomorphic duplicate worlds

Generate two equivalent technical worlds with renamed symbols, reordered definitions, and different surface wording.

A real memory system should perform similarly on both. A string-matching system should fail.

Example:

```text
World A: mavoid, blue identity, operator ⊙
World B: relneth, amber seed, operator ⋄
```

Same structure, different names.

### Separate public seeds from private seeds

Publish the generator and public fixtures. Keep a private seed set for leaderboard scoring. This makes the test reproducible without making it memorization-prone.

### Add metamorphic checks

For any candidate answer, run paired tests:

| Transformation                   | Required invariant                                                                  |
| -------------------------------- | ----------------------------------------------------------------------------------- |
| Rename symbols                   | Same logical result                                                                 |
| Reorder unrelated ingests        | Same current recall                                                                 |
| Add irrelevant distractor        | Same support set                                                                    |
| Add falsifier after `as_of` time | No change to `recall_as_of`                                                         |
| Add current falsifier            | Current answer changes                                                              |
| Compress and rebuild             | Same answer and support                                                             |
| Forget private source            | Private content disappears, safe conclusions remain only if independently supported |

These are deterministic and very hard to game.

### Penalize context dumping

With modern long-context models, benchmarks can accidentally reward “read everything” rather than memory. Memora’s authors explicitly criticize retrieval-centric benchmarks that reduce long-term memory to isolated fact recall and assume stored information remains permanently valid. ([arXiv][5])

So the harness should record:

```text
retrieved_items_count
retrieved_token_count
private_items_touched
irrelevant_items_touched
latency_ms
state_bytes
```

Then score:

```text
support_precision = required_support_ids / used_ids
context_efficiency = useful_tokens / retrieved_tokens
```

This forces actual memory selection.

## Make grading fully deterministic

For this kind of benchmark, I would avoid LLM-as-judge for core scores. Your own spec notes judge variance as a threat.

Use formal checkers wherever possible:

| Task type                   | Deterministic checker                     |
| --------------------------- | ----------------------------------------- |
| Numeric science calculation | Tolerance-based numeric oracle            |
| Unit consistency            | Dimension algebra                         |
| Finite algebra theorem      | Model checker                             |
| Proof dependency            | DAG validator                             |
| Counterexample              | Finite structure evaluator                |
| Temporal recall             | Timestamp filter                          |
| Citation support            | Source-ID entailment map                  |
| Privacy                     | Multi-channel regex + transform scanner   |
| Procedural plan             | DAG validator                             |
| Compression                 | Required fact/exception/support retention |

LLM judges can still be used for qualitative reporting, but not for the leaderboard score.

## The shortest path to a stronger v2

I would implement these first, in order:

1. **Add generated synthetic math worlds** with finite-model oracles.
2. **Add generated synthetic physics worlds** with unit/numeric/checkpoint oracles.
3. **Replace “citation exists” with “minimal sufficient support.”**
4. **Add metamorphic tests**: rename, reorder, distract, compress, rebuild.
5. **Add private seeded holdout fixtures.**
6. **Add context-budget and over-retrieval penalties.**
7. **Add compression-specific fixtures as a major score axis.**
8. **Diversify the reference adapters so they no longer share the same core.**
9. **Add transformed privacy leak tests across all output channels.**
10. **Add proof-DAG invalidation and repair.**

The benchmark you have is already a serious deterministic memory-engineering test. The next leap is to make it test **learning and maintaining novel technical theories under time, contradiction, compression, privacy, and tool-use pressure**. That is the version that would most convincingly evaluate next-generation memory systems.
