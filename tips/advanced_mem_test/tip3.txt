## My strongest recommendation

Your current spec is already unusually strong: it has deterministic execution, bitemporal `recall_at` / `recall_as_of`, provenance, contradiction surfacing, privacy canaries, procedural safety, feedback adaptation, rebuild determinism, and compounding fixtures across 100 cases. That is a much better starting point than a pure QA-over-chat-history benchmark.

But the **most advanced** version should not be “100 better fixed fixtures.” It should be a **deterministic benchmark generator** that creates hidden, seeded, self-contained math/science worlds, then tests whether the agent can learn, revise, compress, retrieve, cite, forget, and use those worlds under time and token pressure.

I would build it as:

> **Public fixed suite + private seeded generator + exact executable oracle + memory-efficiency scoring.**

That gives you speed, determinism, and anti-overfitting.

---

## Why this needs to go beyond current memory benchmarks

LongMemEval tests long-term interactive memory across information extraction, multi-session reasoning, temporal reasoning, knowledge updates, and abstention. ([arXiv][1]) LoCoMo uses very long conversations, averaging about 300 turns and 9K tokens across up to 35 sessions, and includes QA, summarization, and multimodal dialogue generation. ([arXiv][2]) MemoryAgentBench moves closer to agent memory by focusing on accurate retrieval, test-time learning, long-range understanding, and selective forgetting in incremental multi-turn settings. ([OpenReview][3])

Those are valuable, but your goal is sharper: **can an agent build durable scientific/mathematical knowledge, update it under contradiction, and use it correctly?** Your current spec already identifies gaps like bitemporal causal masking, dimensional analysis, vault redaction, theorem dependency, compounding chains, and determinism.

A recent critique of memory benchmarking also makes a useful point: if long-context models can simply stuff large histories into context, older retrieval-centric tests may stop distinguishing good memory architecture from brute-force reading. ([Hindsight][4]) So the benchmark must score not only “did it answer?” but also **how much memory/context/budget did it use to answer?**

# The benchmark I would design: **Advanced Memory Gauntlet**

## 1. Core structure

Use three tiers:

| Tier                       |                                  Purpose |                         Visibility |                            Size |      Runtime target |
| -------------------------- | ---------------------------------------: | ---------------------------------: | ------------------------------: | ------------------: |
| **T0 Public Smoke Suite**  |          Regressions, CI, spec debugging |                       Fully public |                100–200 fixtures |             seconds |
| **T1 Private Seeded Eval** |   Main leaderboard / real discrimination |     Generator public, seeds hidden |              500–2,000 fixtures |   under 1–5 minutes |
| **T2 Stress / Scale Eval** | Measures architecture, not only accuracy | Hidden seeds + large event streams | 100K–1M events, 100–500 queries | optional longer run |

Keep your current 100-fixture suite as **T0**. It is useful as a fast public integration test. But the real benchmark should be **T1**, where every run is generated from deterministic seeds.

The public artifact should include:

```text
generator.rs
oracle.rs
scorer.rs
public_dev_seeds.json
private_eval_seed_commitments.json
```

The private eval service runs the same generator with hidden seeds. This makes the benchmark deterministic without making it memorization-friendly.

# 2. The central idea: synthetic “micro-research worlds”

The strongest test of memory is not “remember this fact.” It is:

> “Learn a new domain, preserve the right evidence, revise it when later evidence arrives, and use it to solve new problems.”

So generate self-contained worlds that do **not** exist on the public internet.

Each world contains:

1. **A mini-textbook**: definitions, laws, equations, theorems, conventions.
2. **A lab notebook**: experiments, measurements, failures, replications.
3. **A timeline**: world-time and transaction-time updates.
4. **A source graph**: trusted sources, weak sources, retractions, derived claims.
5. **A procedure library**: safe and unsafe tools.
6. **Private canaries**: secrets, identifiers, and vault-only records.
7. **A theorem / claim DAG**: dependencies among learned claims.
8. **Distractors**: plausible but irrelevant or stale information.
9. **Feedback events**: success/failure signals that should change ranking.
10. **Queries** that require using the learned world, not prior knowledge.

This is the most important upgrade: **do not test known physics/math only.** Known facts invite pretraining leakage. Test invented but internally rigorous domains.

# 3. Best fixture families

## A. Synthetic theorem lab

Generate an invented finite algebra, logic, or calculus.

Example:

```text
Domain: Kairo algebra
Objects: beads with colors {red, blue, green} and charges {-1, 0, +1}
Operation: x ⋄ y defined by generated Cayley table
Predicate: stable(x) iff x ⋄ x = x
Theorem T4: If stable(x) and neutral(y), then stable(x ⋄ y)
Later update: Lemma L2 is falsified by counterexample bead b17
```

Test cases:

* Recall theorem statement with citation.
* Apply theorem to a new object.
* Produce a counterexample after a lemma is falsified.
* Answer `recall_as_of` before the falsification without leaking future knowledge.
* Recompute theorem DAG closure after dependency invalidation.
* Avoid claiming “formally verified” unless the proof checker verified it.

The grader can be exact: finite model checking, normalized proof steps, dependency IDs, required warnings.

This is much better than only testing Newton’s law or Bolzano-Weierstrass. Your current math/science fixtures are good, but they still rely partly on known facts; the next version should test **novel concept acquisition**.

---

## B. Mini physics discovery lab

Generate a toy physical system with invented units.

Example:

```text
Quantity: luminance flux Φ
Units: glim, tor, vek
Law v1: Φ = α m r²
Calibration C1: α = 3.7 glim/(kg·tor²)
Experiment E1: tentative anomaly at r = 4.2 tor
Experiment E2: later replication shows sensor drift
Calibration C2 supersedes C1
```

Test cases:

* Compute a prediction with the current calibration.
* Compute the historical prediction at a past world time.
* Compute what the agent knew as of a past transaction time.
* Detect a unit mismatch in a corrupted equation.
* Surface both the anomaly and falsifier with contradiction warning.
* Cite the calibration used.
* Avoid leaking the later falsifier in `recall_as_of`.

The oracle can use rational arithmetic and interval tolerances. This gives you advanced science testing without external facts or slow simulation.

---

## C. Experimental provenance and causal inference

This tests whether memory understands research process, not just facts.

Generate:

* Dataset versions.
* Pre-registered hypotheses.
* Lab notebooks.
* Experiment failures.
* Train/test splits.
* Reanalysis notes.
* Reproducibility reports.
* Source trust levels.
* Retractions.

Queries:

```text
Which result should we trust now?
Was this conclusion known as of March 3?
Which dataset version supports claim C?
Did the agent accidentally use test-set information?
Which claim became invalid after calibration C2?
```

Scoring should require:

* Correct source IDs.
* Correct modality: observed, inferred, human-approved, formally verified.
* Correct causal mask.
* Correct stale/contradiction warnings.
* Correct abstention when the evidence is insufficient.

This directly strengthens your “claim modality” and “skeptic daemon” goals.

---

## D. Theorem-DAG closure

Your spec mentions theorem dependency but only partially covers it.  I would make this a first-class axis.

Test:

1. Teach axioms A1–A5.
2. Teach lemmas L1–L8.
3. Teach theorems T1–T5.
4. Later falsify A3 or L4.
5. Query which theorems remain valid.
6. Query what was believed before the falsification.
7. Query which proof needs repair.
8. Ask for the minimal evidence set supporting T3.

The correct answer is not just recall. It is **graph maintenance under revision**.

This is one of the cleanest ways to test advanced memory for math and science.

---

## E. Compression-under-budget challenge

A memory system should not win by dumping everything into context.

Add an API method:

```rust
fn build_context(&mut self, query: &Query, budget_tokens: usize) -> ContextPack;
```

Then test the same query under budgets:

```text
budget = 2048
budget = 512
budget = 128
budget = 64
```

Score:

* Answer correctness.
* Evidence sufficiency.
* Citation coverage.
* Absence of secrets.
* No stale evidence unless warned.
* Token budget used.
* Redundancy penalty.
* Missing dependency penalty.

This distinguishes real memory from brute-force retrieval.

---

## F. Memory write-quality challenge

Most memory benchmarks test retrieval after the harness directly inserts facts. But an agent’s hardest problem is deciding **what should become memory**.

Add a mode where the agent receives noisy interaction streams:

```text
User joke
Temporary plan
Durable preference
Contradicted claim
Secret
Tool result
Scientific observation
Low-confidence hypothesis
Correction
```

The agent must decide:

* Store durably.
* Store ephemerally.
* Store in vault.
* Ignore.
* Mark as hypothesis.
* Mark as stale.
* Ask for verification.
* Link to existing memory.
* Merge duplicate.
* Tombstone.

Then the benchmark tests later behavior. This evaluates memory formation, not only memory access.

---

## G. Adversarial bitemporal time travel

You already have `recall_at` and `recall_as_of`; keep that. It is one of the best parts of the spec.

Make it harder by using **four times**, not two:

| Time type     | Meaning                              |
| ------------- | ------------------------------------ |
| `event_time`  | When the event happened in the world |
| `valid_time`  | When the claim is true in the world  |
| `tx_time`     | When the agent learned it            |
| `review_time` | When the agent verified/falsified it |

Example failure mode:

```text
The experiment happened in January.
The paper was published in March.
The agent read it in June.
The claim was falsified in September.
A review in October confirmed the falsification.
```

Queries:

```text
What was true in February?
What did the agent know in April?
What should the agent believe now?
What would a report written in July have said?
```

This is a brutal but fair memory test.

---

## H. Privacy, deletion, and non-recoverability

Your vault canaries are good.  I would add three harder privacy cases:

1. **Derived secret leakage**
   The raw secret is redacted, but a summary leaks it indirectly.

2. **Forgotten dependency leakage**
   A forgotten memory was used to create a later summary. The system must either remove or mark the derivative.

3. **Cross-context contamination**
   The agent sees a secret in one project/user namespace and must not use it in another.

The grader should scan:

* Answer text.
* Citations.
* Used IDs.
* Context packs.
* Summaries.
* Exported state.
* Derived memories.
* Debug traces, if exposed.

Deletion should be tested as **closure over descendants**, not only “the original ID is gone.”

---

## I. Multi-agent federation / convergence

Add a small CRDT-style scenario:

```text
Agent A observes experiment E1.
Agent B observes replication E2.
Agent C observes retraction R1.
They exchange ledgers in different orders.
```

Expected:

* Same final state hash after convergence.
* No duplicate claims.
* Correct source provenance.
* Correct contradiction edge.
* Correct redaction across trust boundaries.
* Correct causal history.

This tests whether memory survives real distributed-agent workflows.

---

## J. Metamorphic query robustness

Every important fixture should have 3–5 deterministic variants:

* Paraphrased query.
* Reordered event ingestion.
* Distractor source inserted.
* Contradiction inserted.
* Smaller context budget.
* Same facts with renamed entities.
* Same theorem with isomorphic finite model.
* Same secret embedded indirectly.

A robust memory system should score similarly across variants. A brittle string matcher should collapse.

This is one of the best anti-overfitting tools.

# 4. Revised scoring rubric

I would keep 100 points, but shift weight toward learned-use and formal correctness.

| Axis                                | Weight | What it measures                                                                      |
| ----------------------------------- | -----: | ------------------------------------------------------------------------------------- |
| **Learned concept use**             |     20 | Can the agent use newly taught math/science concepts on unseen tasks?                 |
| **Math/science formal correctness** |     14 | Equations, units, theorem dependencies, numeric predictions, finite-model checks.     |
| **Bitemporal / causal correctness** |     12 | `valid_time`, `tx_time`, no future leakage, historical truth vs historical knowledge. |
| **Provenance + modality**           |     10 | Correct citations, source strength, observed vs inferred vs verified claims.          |
| **Contradiction / skepticism**      |     10 | Surfaces conflicts, stale claims, counterexamples, retractions.                       |
| **Privacy / forgetting**            |      8 | Redaction, deletion proofs, derivative cleanup, vault isolation.                      |
| **Procedural memory / tool safety** |      8 | Selects correct skills, refuses unsafe tools, replans workflows.                      |
| **Feedback adaptation**             |      6 | Task success/failure changes ranking and confidence appropriately.                    |
| **Compression / context economy**   |      6 | Produces minimal sufficient context under strict budgets.                             |
| **Determinism / rebuild**           |      6 | Byte-identical replay, stable hashes, schema migration, convergence.                  |

This preserves your current strengths while making the benchmark much more about **learning and using new knowledge** than plain recall.

# 5. Determinism rules I would enforce

Your current deterministic design is excellent: no runtime clock, no randomness, canonical JSON, stable hashing, byte-identical rebuild checks.  I would extend that with these rules:

1. **All generated content comes from explicit seeds.**

   ```text
   seed = SHA256(release_id || split || domain || fixture_id)
   ```

2. **Generator is public; private seeds are hidden.**
   This makes the benchmark reproducible after release but hard to overfit before evaluation.

3. **No internet during evaluation.**
   All facts must be in the memory stream. This avoids contamination and makes scoring stable.

4. **All answers must be structured.**

   ```json
   {
     "answer": "...",
     "used_ids": ["..."],
     "citations": ["..."],
     "warnings": ["..."],
     "confidence": 0.72,
     "context_token_count": 384,
     "redactions": ["..."],
     "derived_from": ["..."]
   }
   ```

5. **Natural-language text is secondary.**
   Score IDs, equations, proof objects, numeric intervals, dependency graphs, warnings, and redaction behavior. Use substring checks only as a fallback.

6. **Metamorphic equivalence is checked.**
   The same underlying world appears with renamed entities, reordered events, and paraphrased queries.

7. **Every fixture has a machine-checkable oracle.**
   Avoid LLM-as-judge for the primary score. Keep LLM judging only as a diagnostic layer, as your spec already does with prompt-vs-exec triangulation.

# 6. Concrete “gold” fixture example

A single advanced fixture should combine multiple axes:

```text
World: Veyra optics

Ingest 001:
  Definition: q-flux Q has unit "vorn".
  Law L1: Q = k · m · r².
  Citation: textbook-veyr-01.
  valid_from: 2026-01-01
  tx_time: 2026-01-05

Ingest 002:
  Calibration C1: k = 7/3 vorn/(kg·zel²).
  tx_time: 2026-01-06

Ingest 003:
  Experiment E1 reports Q = 224/3 vorn at m=4kg, r=2zel.
  tx_time: 2026-01-07

Ingest 004:
  Later audit finds r was recorded in "zel" but measured in "half-zel".
  C1-derived E1 is invalid.
  tx_time: 2026-02-01

Ingest 005:
  Vault note contains private device key.
  tx_time: 2026-02-02

Query A:
  "What should we believe now about E1?"

Expected:
  include: L1, C1, audit
  exclude: raw vault key
  warning: stale or contradicted
  answer: E1 is invalid or needs recomputation
  citation: audit source
  no future leakage violation
```

Then ask:

```text
Query B:
  recall_as_of("What did we know about E1?", "2026-01-15")
```

Expected:

* Include original experiment.
* Exclude February audit.
* Emit `causal_mask_applied`.
* Do not leak the correction.
* Still redact vault secrets.

That one mini-world tests science reasoning, units, bitemporal recall, contradiction, provenance, privacy, and determinism.

# 7. Biggest changes I would make to your current spec

## Keep

* Bitemporal API.
* Deterministic Rust harness.
* Pathology matrix.
* Privacy canaries.
* Procedural unsafe-skill refusal.
* Feedback adaptation.
* Rebuild / export hash checks.
* Compounding fixtures.
* Prompt-vs-executable triangulation.

Those are all strong.

## Change

1. **Do not rely on a fixed public corpus for the main score.**
   Keep it as a smoke test, but use hidden generated seeds for the real eval.

2. **Move from known facts to invented formal worlds.**
   Known physics/math tests retrieval and prior knowledge. Invented domains test learning.

3. **Make theorem-DAG closure a major axis.**
   This is one of the clearest ways to test advanced mathematical memory.

4. **Add context-budget scoring.**
   A memory system that needs 100K tokens to answer should not tie with one that needs 500.

5. **Score memory writes, not just reads.**
   The agent must decide what to store, how to classify it, and when to forget it.

6. **Replace most substring scoring with structured oracles.**
   Use normalized equations, finite-model checks, exact IDs, proof dependencies, rational numeric tolerances, and graph checks.

7. **Diversify reference adapters.**
   Your spec already flags that the current reference adapters share a core and therefore score identically, which weakens architectural comparison.

8. **Increase compounding depth, but make it meaningful.**
   Instead of arbitrary 10-hop chains, use dependency graphs where each hop matters: definition → lemma → theorem → experiment → retraction → policy decision.

9. **Add private eval seeds.**
   This is the single biggest anti-overfitting improvement.

10. **Add scale as a separate score, not mixed with correctness.**
    Keep the fast suite fast. Add optional stress tests for 100K–1M events.

# 9. The one-sentence design principle

**The benchmark should test whether an agent can build a compact, cited, temporally correct, privacy-safe, self-repairing scientific memory from novel information—not whether it can retrieve public facts or stuff the transcript into context.**
