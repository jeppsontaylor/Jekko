I agree with your instinct: **OCAL should not just be “forever loop until clean.”** It should also provide a safe way for agents to recognize: “This task is too uncertain to implement directly; I need a bounded research/prototype/hardening lane where knowledge compounds across loops.”

The best addition is an **OCAL Hardening Lane**: a host-enforced workflow for hard tasks that lets multiple fresh agents plan, critique, prototype, preserve task-local memory, and only promote to implementation when evidence-based confidence crosses a threshold.

This should be added to the existing OCAL daemon plan as a first-class feature, not as a prompt convention. The uploaded plan already makes the right core decision: OpenCode runtime owns looping, SQLite is source of truth, normal session semantics remain unchanged, OCAL uses strict gated syntax, and generated mirrors under `.opencode/daemon/<runID>/` are inspection artifacts, not truth.

# 1. The important shift

The current OCAL plan solves:

> “Agents stop even though the job is not done.”

The new issue is different:

> “Agents keep working, but once easy tasks are gone, they may either avoid hard tasks, thrash on them, or make low-confidence changes.”

So OCAL needs two separate loops:

```text
Daemon loop:
  Keep the whole job alive until stop criteria pass.

Task loop:
  Decide whether each task is ready for direct implementation or needs hardening.

Hardening loop:
  Bound research, planning, critique, prototyping, and memory accumulation for difficult tasks.
```

The daemon loop can be infinite. The hardening loop should usually be **bounded**.

That distinction matters. You want the overall process to continue forever, but you do **not** want one hard algorithmic rework to burn forever in scratchpad mode.

# 2. What other communities already teach us

The broader agent ecosystem is converging around several patterns that OCAL should reuse rather than reinvent.

LangGraph is the strongest precedent for durable, stateful execution. Its persistence layer saves graph state as checkpoints, enabling human-in-the-loop workflows, memory, time travel debugging, and fault-tolerant execution; its durable execution docs explicitly frame workflows as resumable from persisted state. ([LangChain Docs][1])

LangGraph also cleanly separates short-term thread memory from long-term cross-session memory, which maps directly to OCAL’s need for “current attempt context” versus “task-local durable scratchpad.” ([LangChain Docs][2])

LangGraph’s interrupt model is highly relevant: it can pause execution, save graph state, and wait indefinitely for external input. OCAL should copy that semantic shape for `pause_on: low_confidence`, `pause_on: permission_denied`, and `pause_on: needs_human_decision`. ([LangChain Docs][3])

Google ADK has a direct `LoopAgent` concept: it repeatedly runs subagents for a max iteration count or until a termination condition is met, specifically for iterative refinement such as revising code. That is basically the runtime shape OCAL should expose for bounded hardening passes. ([Adk][4])

AutoGen’s current AgentChat team model supports patterns like round-robin teams, selector-based teams, and MagenticOne-style generalist teams for open-ended file/web tasks. That supports the idea that OCAL should distinguish “supervisor decides next worker” from “fixed pass sequence.” ([Microsoft GitHub][5])

CrewAI Flows are another precedent: they are structured, event-driven workflows that manage state and control execution; CrewAI also documents human-in-the-loop workflows as a core pattern. OCAL’s hardening lane should feel more like a small flow/state machine than an open-ended chat transcript. ([CrewAI Documentation][6])

OpenAI’s Agents SDK documents multi-agent orchestration through handoffs and agents-as-tools; the “manager calls specialists as tools” model is especially relevant for an OCAL supervisor spawning plan/review/prototype agents without losing central control. ([OpenAI Developers][7])

LangChain’s Deep Agents docs are almost exactly on point: deep agents combine planning, subagents, file systems for context management, and long-term memory; the subagent docs explicitly call out “context quarantine” as a reason to delegate work. OCAL’s hardening lane should formalize that same idea inside OpenCode. ([LangChain Docs][8])

OpenHands’ skills system shows the durable-instructions side of the problem: skills are specialized prompts that provide domain knowledge and task-specific workflows, with repository customization and keyword-triggered activation. OpenHands also advises keeping tasks small and committing/pushing frequently, which supports OCAL’s easy-task/direct-lane strategy. ([OpenHands Docs][9])

Cline’s `new_task` tool is a strong precedent for context reset with continuity: it packages current progress and context into a fresh task with a clean context window. OCAL hard-clear epochs and hardening attempts should follow that model, but with DB-backed state instead of only prompt handoff. ([Cline Documentation][10])

Roo Code’s Boomerang Tasks / Orchestrator mode show the same pattern from another coding-agent community: break complex workflows into delegated subtasks handled by specialized modes, then return results to the orchestrator. ([Roo Code Docs][11])

Aider’s repository map is also relevant: it gives broad codebase awareness without brute-forcing full context. OCAL should eventually support task-local “context packs” and repo summaries for hard tasks instead of making every fresh pass rediscover the whole repo. ([Aider][12])

Finally, raw model self-confidence should not be trusted as the gate. Recent calibration work continues to find that verbalized LLM confidence is often miscalibrated and can be overconfident on low-accuracy outputs. OCAL should use model confidence as one signal, but promotion must depend on evidence: tests, audits, reviewers, invariants, reproducible prototypes, and host checks. ([Nature][13])

# 3. Recommended plan update: add an OCAL Hardening Lane

Add this to the current OCAL implementation plan as **Phase 3.5: Hard Task Routing and Cross-Loop Collaboration**.

The core idea:

```text
Easy task:
  claim -> implement -> verify -> checkpoint -> done

Hard task:
  claim -> triage -> independent plans -> critique -> synthesis
       -> optional prototype/stub worktree
       -> evidence scoring
       -> promote to implementation only when confidence threshold passes
       -> otherwise mark blocked / defer / ask human / retry bounded
```

The hardening lane gives difficult tasks a place to accumulate useful work **without polluting the main daemon context** and without prematurely committing risky edits.

# 4. What counts as a “hard task”?

Do not let the model route everything into hardening. The runtime should require one of several gates.

A task enters hardening only if one of these is true:

```yaml
routing:
  hard_if:
    any:
      - host_risk_score_gte: 0.70
      - evidence_confidence_lte: 0.55
      - failed_attempts_gte: 2
      - changed_paths_match:
          - "packages/opencode/src/session/**"
          - "packages/opencode/src/server/**"
          - "packages/opencode/migration/**"
      - task_kind_in:
          - algorithm_rework
          - migration
          - concurrency
          - security
          - permission_model
          - multi_agent_runtime
      - agent_requests_hardening:
          require_uncertainty_report: true
```

And it should stay in the direct lane when the task is simple:

```yaml
routing:
  easy_if:
    all:
      - host_risk_score_lte: 0.35
      - changed_lines_lte: 80
      - has_direct_verification: true
      - failed_attempts_lte: 0
      - no_public_api_change: true
```

This prevents “analysis paralysis” where every task becomes a research project.

# 5. Evidence confidence, not vibes confidence

OCAL should introduce two different concepts:

```yaml
confidence:
  model_self_report:
    allowed: true
    weight: 0.10

  evidence_score:
    required_for_promotion: 0.78
```

The model can say, “I am only 40% confident,” but the runtime should compute an evidence score from host-verifiable facts.

Suggested evidence formula:

```text
evidence_score =
  0.25 * tests_signal
+ 0.20 * audit_signal
+ 0.15 * reviewer_consensus
+ 0.15 * invariant_quality
+ 0.10 * prototype_success
+ 0.10 * blast_radius_control
+ 0.05 * model_self_confidence
```

Example signals:

| Signal                  | High score means                                              |
| ----------------------- | ------------------------------------------------------------- |
| `tests_signal`          | Relevant tests exist and pass.                                |
| `audit_signal`          | `jankurai audit`, typecheck, lint, or equivalent checks pass. |
| `reviewer_consensus`    | Independent reviewer agents agree the approach is safe.       |
| `invariant_quality`     | The task has written invariants or acceptance criteria.       |
| `prototype_success`     | A prototype/stub demonstrates feasibility.                    |
| `blast_radius_control`  | Changes are isolated, small, or behind a feature path.        |
| `model_self_confidence` | Agent is not expressing major uncertainty. Low weight only.   |

This is the most important safety rule:

> A task is not promoted because the agent “feels confident.” It is promoted because there is enough evidence.

# 6. New runtime state machine

Add a per-task state machine under the existing daemon runtime.

```text
daemon_run
  └── daemon_task
        ├── discovered
        ├── triaged
        ├── direct_ready
        ├── implementing
        ├── verifying
        ├── done
        ├── blocked
        └── hardening
              ├── planning
              ├── adversarial_review
              ├── synthesis
              ├── prototype
              ├── evidence_review
              ├── promoted
              ├── deferred
              └── human_needed
```

The daemon continues forever. Each hard task has bounded hardening passes:

```text
max_hardening_rounds: 3
max_plans_per_round: 3
max_prototypes: 2
max_wall_time: 90m
max_tokens: configured
```

When hardening budget is exhausted, the task should not disappear. It should become:

```text
blocked:
  reason: insufficient_evidence
  artifacts:
    - best_plan
    - rejected_plans
    - open_questions
    - prototype_results
    - next_human_decision_needed
```

Then the daemon can move on to other work or pause if only blocked hard tasks remain.

# 8. New OCAL syntax: `routing`, `hardening`, `scratchpad`, `promotion`, `cleanup`

The cleanest addition is not to overload `loop`. Add a new top-level `hardening:` section.

## Minimal version

```yaml
routing:
  default: direct
  hard_if:
    any:
      - host_risk_score_gte: 0.70
      - evidence_confidence_lte: 0.55

      - agent_requests_hardening:
          require_uncertainty_report: true

    all:
      - host_risk_score_lte: 0.35

      - failed_attempts_lte: 0

  enabled: true
  max_concurrent_tasks: 1
  max_rounds_per_task: 3

    on_budget_exhausted: block_with_artifacts
```

That is the simple, reliable first version.

# 9. Full hardening syntax

```yaml
hardening:
  enabled: true

  classify:
    model_self_report: true
    host_signals:
      - touched_critical_paths
      - failed_attempts
      - verification_gap
      - public_api_change
      - migration_change
      - concurrency_change
      - security_or_permissions_change
      - large_diff
      - unclear_acceptance_criteria
    scoring:
      hard_threshold: 0.70
      easy_threshold: 0.35

  scratchpad:
    storage: sqlite
    mirror: ".opencode/daemon/${run.id}/tasks/${task.id}"
    write_policy: append_only
    preserve:
      - task_objective
      - constraints
      - hypotheses
      - rejected_approaches
      - accepted_invariants
      - open_questions
      - prototypes
      - evidence
      - decisions
    max_bytes_per_task: 250000
    max_attempts: 12

  passes:
    - id: independent_plans
      type: plan
      count: 3
      context: fresh
      permissions:
        read: allow
        edit: deny
        bash: allow_readonly
      must_not_read:
        - ".opencode/daemon/${run.id}/tasks/${task.id}/attempts/*"
      output:
        schema: plan_v1

    - id: critique_existing
      type: critique
      count: 2
      context: fresh
      must_read:
        - task
        - plans
        - constraints
      permissions:
        read: allow
        edit: deny
      output:
        schema: critique_v1

    - id: synthesize_plan
      type: synthesis
      count: 1
      context: compact
      must_read:
        - task
        - plans
        - critiques
        - evidence
      output:
        schema: accepted_plan_candidate_v1

    - id: prototype
      type: prototype
      count: 1
      isolation: git_worktree
      permissions:
        read: allow
        edit: allow
        bash: ask
        git_commit: deny
        git_push: deny
      output:
        schema: prototype_result_v1

    - id: adversarial_review
      type: review
      count: 2
      context: fresh
      permissions:
        read: allow
        edit: deny
        bash: allow_readonly
      output:
        schema: vote_v1

  confidence:
    evidence_score:
      promote_gte: 0.78
      block_lte: 0.35
    model_self_report:
      weight: 0.10
    require:
      - accepted_invariants
      - verification_plan
      - reviewer_consensus
      - no_unresolved_critical_questions

  promotion:
    when:
      all:
        - evidence_score_gte: 0.78
        - reviewer_accept_gte: 2
        - reviewer_reject_eq: 0
        - prototype_status_in: [passed, not_needed]
    action:
      route_to: implementation
      inject_context:
        - accepted_plan
        - invariants
        - verification_plan
        - known_risks
        - rejected_approaches
      permissions:
        edit: allow
        bash: ask
        git_commit: ask
        git_push: ask

  cleanup:
    on_promoted:
      keep:
        - ACCEPTED_PLAN.md
        - DECISIONS.md
        - EVIDENCE.jsonl
        - RISKS.md
      archive_rejected: true
      delete_temp_worktrees: true
      keep_db_history: true
    on_done:
      archive_task_dir: true
      delete_prototype_branches: ask
      keep_ledger: true
    on_blocked:
      keep_all_artifacts: true
      write_handoff: true
```

# 10. Pass types OCAL should support

These should be host-understood pass types, not just prose.

| Pass type          | Purpose                                                  | Default permissions                    | Default context                |
| ------------------ | -------------------------------------------------------- | -------------------------------------- | ------------------------------ |
| `triage`           | Determine task kind, risk, missing info.                 | Read-only.                             | Fresh.                         |
| `independent_plan` | Generate plan without contamination from other plans.    | Read-only.                             | Fresh, no prior attempts.      |
| `critique`         | Attack a plan, find risks and contradictions.            | Read-only.                             | Fresh plus selected artifacts. |
| `synthesis`        | Combine best ideas into one accepted plan candidate.     | Read-only.                             | Compact task memory.           |
| `prototype`        | Try feasibility in isolated worktree.                    | Edit in temp worktree only.            | Accepted plan candidate.       |
| `stub`             | Write non-final interfaces/scaffolding for exploration.  | Edit in temp worktree only.            | Prototype context.             |
| `test_design`      | Create tests or verification plan before implementation. | Usually read-only, optional temp edit. | Task + invariants.             |
| `implementation`   | Final real change.                                       | Edit allowed.                          | Accepted plan + evidence.      |
| `review`           | Validate implementation or plan.                         | Read-only.                             | Fresh plus diff/artifacts.     |
| `rebuttal`         | Argue why current solution is wrong.                     | Read-only.                             | Fresh plus accepted plan.      |
| `retrospective`    | Compress learnings after success/failure.                | Read/write scratchpad.                 | Full task memory.              |

# 11. How “fresh agent” versus “improve existing plan” should work

You correctly identified two different useful modes:

1. A fresh agent makes a new independent plan.
2. A fresh agent reads the existing plan and improves it.

Both should exist.

## Independent planning

Use when you want diversity and less anchoring.

```yaml
- id: independent_plans
  type: plan
  count: 3
  context: fresh
  must_not_read:
    - previous_plans
    - previous_critiques
  output:
    schema: plan_v1
```

## Improvement planning

Use after there is already enough accumulated knowledge.

```yaml
- id: improve_current_best
  type: plan_improvement
  count: 2
  context: fresh
  must_read:
    - accepted_plan_candidate
    - rejected_approaches
    - open_questions
    - evidence
  output:
    schema: plan_delta_v1
```

## Adversarial planning

Use when stakes are high.

```yaml
- id: disprove_plan
  type: rebuttal
  count: 2
  context: fresh
  instruction: |
    Assume the current plan is subtly wrong.
    Find the most likely failure mode.
    Do not propose implementation unless the plan survives your critique.
```

This is much better than one agent endlessly revising its own plan.

# 14. Important behavior: easy-first, hard-lane later

You described the natural progression exactly:

1. Agents grab high-confidence tasks first.
2. Easy tasks disappear.
3. Only low-confidence tasks remain.
4. The system needs to stop pretending the next low-confidence task is just another normal item.

Add this task selection policy:

```yaml
tasks:
  selection:
    policy: confidence_band
    bands:
      - name: direct_easy
        evidence_confidence_gte: 0.75
        max_risk: 0.35
      - name: direct_medium
        evidence_confidence_gte: 0.60
        max_risk: 0.60
      - name: hardening
        evidence_confidence_lt: 0.60
    easy_first: true
    hard_task_after_easy_exhausted: true
    avoid_starvation:
      enabled: true
      after_iterations: 10
```

This gives you sane behavior:

```text
While easy tasks exist:
  finish easy tasks directly

When only hard tasks remain:
  activate bounded hardening lane

If hardening promotes a task:
  implement it

If hardening cannot promote:
  write blocked handoff and continue / pause depending on policy
```

# 16. “Memory” rules for hard tasks

Task memory must be:

1. **Task-local**: memory belongs to a specific task, not the whole daemon.
2. **Append-only by default**: avoid agents rewriting history.
3. **Typed**: hypothesis, risk, evidence, decision, open question, rejected approach.
4. **Promotable**: only accepted artifacts flow into implementation prompt.
5. **Cleanable**: prototypes and rejected plans can be archived or deleted after task completion.
6. **DB-backed**: mirror files are inspection artifacts only.

Syntax:

```yaml
memory:
  scopes:
    daemon:
      purpose: global_run_state
      max_bytes: 500000
    task:
      purpose: hardening_scratchpad
      max_bytes: 250000
    attempt:
      purpose: per_agent_private_work
      max_bytes: 75000
      visible_to_other_attempts: false

  promotion_rules:
    task_to_implementation:
      include:
        - accepted_plan
        - decisions.active
        - risks.high
        - invariants.accepted
        - verification_plan
      exclude:
        - rejected_plans
        - private_chain_of_thought
        - raw_logs_over_limit
```

Important: never preserve private chain-of-thought. Preserve conclusions, evidence, decisions, and structured summaries.

# 17. Cleanup model

This is critical. Hardening will create lots of debris unless cleanup is first-class.

```yaml
cleanup:
  temp_worktrees:
    on_promoted: delete
    on_done: delete
    on_blocked: keep_for_debug
    require_clean_or_patch_saved: true

  scratchpad:
    on_done:
      archive: true
      compact_to:
        - ACCEPTED_PLAN.md
        - DECISIONS.md
        - EVIDENCE.jsonl
        - RETROSPECTIVE.md
      delete:
        - raw_tool_logs
        - failed_private_attempt_dirs
    retention:
      done: 14d
      blocked: 30d
      failed: 30d

  branches:
    prototype:
      delete: ask
    implementation:
      keep: true

  db:
    keep_history: true
    mark_archived: true
```

The rule should be:

```text
Source truth is never deleted.
Temporary worktrees and noisy mirrors can be cleaned after proof is archived.
```

# 20. A simpler syntax option for users

The full YAML is powerful but intimidating. Add a compact preset system.

```yaml
hardening:
  preset: difficult_code_task
  promote_when:
    evidence_score_gte: 0.78
  budget:
    rounds: 3
    plans: 3
    prototypes: 1
```

Preset expands to:

```yaml
passes:
  - independent_plans
  - critique
  - synthesis
  - prototype_if_needed
  - adversarial_review
```

Suggested presets:

| Preset                  | Expands to                                                    |
| ----------------------- | ------------------------------------------------------------- |
| `none`                  | Direct implementation only.                                   |
| `light_review`          | One plan, one reviewer.                                       |
| `difficult_code_task`   | Three plans, critique, synthesis, optional prototype, review. |
| `algorithm_rework`      | More invariants, tests-first, prototype required.             |
| `migration_or_schema`   | Extra rollback, migration review, generated-file checks.      |
| `security_sensitive`    | More reviewers, no direct push, human approval.               |
| `concurrency_sensitive` | Race/deadlock review, stress test requirements.               |
| `public_api_change`     | Compatibility and docs review required.                       |

Example:

```yaml
routing:
  default: direct
  hard_if:
    any:
      - failed_attempts_gte: 2
      - host_risk_score_gte: 0.70
      - agent_requests_hardening:
          require_uncertainty_report: true

hardening:
  preset: difficult_code_task
  max_concurrent_tasks: 1
  promote_when:
    evidence_score_gte: 0.78
  on_failure: block_with_handoff
```

This is probably the public-facing version most users should use.

# 22. Hard task lifecycle example

For a difficult parser security rework:

```text
1. Daemon sees parser task.
2. Runtime score: risk 0.82 because parser security + activation safety.
3. Task enters hardening.
4. Three fresh agents produce independent plans.
5. Two reviewers critique plans.
6. Supervisor synthesizes one accepted plan candidate.
7. Prototype agent tries parser changes in temp worktree.
8. Reviewer agents inspect prototype and tests.
9. Evidence score reaches 0.81.
10. Runtime promotes task to implementation.
11. Implementation agent receives only accepted plan, invariants, rejected approaches, verification plan.
12. Final reviewer checks diff.
13. Checkpoint commits/pushes if configured.
14. Cleanup archives hardening artifacts and deletes temp worktree.
```

The important part: the implementation agent does **not** receive all noisy scratchpad material. It receives the distilled accepted context.

# 23. The “bounded scratchpad loop” pattern

For truly hard algorithmic work, use this:

```yaml
hardening:
  preset: algorithm_rework
  max_rounds_per_task: 4

  passes:
    - id: define_invariants
      type: test_design
      count: 2
      context: fresh
      output:
        schema: invariants_v1

    - id: rival_algorithms
      type: independent_plan
      count: 4
      context: fresh
      output:
        schema: algorithm_plan_v1

    - id: compare_complexity
      type: critique
      count: 2
      focus:
        - correctness
        - complexity
        - edge_cases
        - migration_risk

    - id: prototype_best_two
      type: prototype
      count: 2
      isolation: git_worktree
      select_from: top_ranked_plans
      permissions:
        edit: allow
        git_commit: deny
        git_push: deny

    - id: choose
      type: synthesis
      count: 1
      output:
        schema: accepted_plan_candidate_v1

  promotion:
    when:
      all:
        - evidence_score_gte: 0.82
        - prototype_status_in: [passed]
        - invariant_coverage_gte: 0.70
        - reviewer_accept_gte: 2
```

This lets OCAL build up knowledge without getting trapped forever.

# 27. Final recommendation

Keep OCAL V1 focused, but include the **syntax and schema now** so the design does not paint itself into a corner.

Ship implementation in this order:

1. **OCAL parser/gate**
2. **Durable daemon loop**
3. **Direct task ledger**
4. **Hard-task routing metadata**
5. **Single-agent hardening lane**
6. **Multi-agent independent plan / critique / synthesis**
7. **Prototype worktrees**
8. **Evidence-confidence promotion**
9. **Cleanup/archive**
10. **Full swarm/MCP profiles**

The key design sentence:

> OCAL forever mode keeps the job alive; OCAL hardening mode keeps hard tasks honest.

And the key safety sentence:

> A hard task may accumulate memory, plans, critiques, and prototypes, but it may not touch the real implementation lane until the runtime promotes it based on evidence.
