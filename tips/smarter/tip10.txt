## My recommendation: add **ZYAL Workcells** for hard-task incubation

The current ZYAL daemon plan is strong for “keep looping until host stop checks pass.” I would not replace it. I would add one new concept:

> **Workcell:** a bounded, durable, per-task collaboration chamber where hard or low-confidence tasks can accumulate plans, critiques, experiments, proofs, failed attempts, and accepted decisions across multiple daemon iterations before the runtime allows real implementation or checkpointing.

This is the missing piece for your “all easy work is done, only scary work remains” case. The daemon should not just keep grabbing low-confidence tasks and smashing into them. It should promote them into a structured hard-task lifecycle:

```text
normal task queue
   │
   ├─ easy / medium → implement → verify → checkpoint
   │
   └─ hard / low-confidence → Workcell
                              ├─ independent plan passes
                              ├─ refinement passes
                              ├─ red-team / critic passes
                              ├─ spike / prototype in isolated scratch or worktree
                              ├─ synthesis into accepted plan
                              ├─ confidence gate
                              └─ implementation only after evidence threshold
```

The previous ZYAL plan already made the right foundational choices: strict activation syntax, host-enforced daemon runtime, SQLite as source of truth, mirrored `.jekko/daemon/<runID>/ledger.jsonl`/`STATE.md`, hard context clearing by epoch/child session rather than deleting history, and a staged rollout from serial loop to checkpoint/context to swarms/MCP. I would extend that plan, not fork it.

# 1. What not to do

Do **not** let the model decide, by vibes alone:

```yaml
if_confidence_low: do_more_research
if_confidence_high: implement
```

That will route too many things into the incubator, or too few, depending on model mood.

The robust version is:

```text
The model may report uncertainty.
The host computes routing from evidence.
The host enforces budgets and gates.
The host owns promotion to implementation.
```

A model self-rating can be one signal, but it should be low-weight. Better signals include prior failed attempts, number of files/subsystems touched, lack of tests, failed checks, unresolved reviewer blockers, high audit severity, dependency ambiguity, API uncertainty, risky git operations, and whether independent agents converge on the same plan.

# 2. Why this aligns with the broader agent ecosystem

There is no single mature standard that already gives Jekko a host-enforced “forever daemon plus hard-task incubation language.” The surrounding ecosystem has pieces of it:

LangGraph is probably the strongest reference for the runtime model: it persists graph state as checkpoints at execution boundaries, supports replay/time travel, and uses persistence to enable fault-tolerant and human-in-the-loop workflows. That maps directly to ZYAL iterations, hard-task passes, and resumable Workcells. ([LangChain Docs][1])

CrewAI Flows has a useful parallel: event-driven workflows with shared state, SQLite-backed persistence, human feedback pauses, and memory methods for storing/recalling facts across runs. The lesson is that hard work needs explicit persisted workflow state, not just prompt continuation. ([CrewAI Documentation][2])

OpenHands has relevant concepts too: microagents/skills for reusable repository context, sub-agent delegation for parallel independent work, and an append-only typed event system that serves as both memory and integration point. That supports the idea that ZYAL should keep an append-only Workcell event/artifact log and use specialist workers for bounded passes. ([OpenHands Docs][3])

Claude Code’s current docs are especially relevant. Claude Code subagents have separate context windows, tool restrictions, model selection, skills, persistent memory, and background execution; agent teams are experimental but explicitly designed for shared tasks, inter-agent messaging, and centralized management; hooks include `Stop`, `SubagentStop`, `PreCompact`, and lifecycle interception. ZYAL should learn from that, but keep the loop host-enforced instead of hook-only. ([Claude][4])

OpenAI’s Agents SDK has sessions for persistent memory, automatic history prepending/persistence, compaction wrappers, and handoffs for delegating to specialized agents. That supports ZYAL’s design direction: persistent session state, filtered handoffs, and specialist agents as runtime primitives. ([OpenAI GitHub][5])

Jekko itself already has the native ingredients: primary agents and subagents, permissions with `allow`/`ask`/`deny`, a `doom_loop` safety guard, MCP integration, and ACP support. But Jekko community issues also show current pain around agent teams, sequential subtask execution, and compaction/run termination, so ZYAL should avoid depending on fragile prompt continuation or current sequential task behavior. ([Jekko][6])

Protocol-wise, AG-UI, A2A, MCP, and ACP occupy adjacent layers: AG-UI for evented agent/frontend state, A2A for agent-to-agent tasks/artifacts, MCP for tools/data, and ACP for editor/agent integration. ZYAL should not try to replace those; it should compile its daemon/workcell events into Jekko-native events that could later be bridged to those protocols. ([AG-UI][7])

Prompt/programming-language projects like POML, LMQL, and DSPy are useful inspiration, but they are not the same product. POML structures prompts, LMQL provides LLM-programming constraints/control flow, and DSPy is “programming, not prompting” for modular AI systems. ZYAL’s niche should be **agent runtime control inside Jekko**, not general prompt formatting. ([GitHub][8])

# 3. The addition: ZYAL Workcell Runtime

## 3.1 New top-level ZYAL section

Add this to the ZYAL spec:

```yaml
hard_tasks:
  route:
    enabled: true
    when: easy_queue_empty
    score:
      hard_if_gte: 0.72
      low_confidence_if_lte: 0.55
    require_any_signal:
      - prior_failed_attempt
      - touches_multiple_subsystems
      - no_direct_test_oracle
      - reviewer_blocker
      - algorithmic_unknown
      - large_refactor
    never_route_if:
      - trivial_diff
      - docs_only
      - explicit_user_fast_path

  workcell:
    max_cycles: 5
    max_wall_time: 2h
    max_cost_usd: 8
    scratch:
      store: sqlite_and_mirror
      mirror_path: ".jekko/daemon/${run.id}/workcells/${task.id}/"
      write_policy: generated_only
      cleanup: summarize_and_archive

    passes:
      - id: classify
        agent: plan
        purpose: score_risk_and_choose_route

      - id: independent_plans
        fanout: 3
        agent: plan
        isolation: fresh_context
        share: none
        output: plan_candidate

      - id: synthesize
        agent: plan
        reads: [plan_candidate]
        output: accepted_plan_draft

      - id: red_team
        fanout: 2
        agent: plan
        reads: [accepted_plan_draft]
        output: blocker_report

      - id: spike
        agent: build
        isolation: git_worktree
        writes: scratch_only
        output: spike_result

      - id: promote
        agent: plan
        gate:
          confidence_gte: 0.82
          no_blockers: true
          evidence:
            - accepted_plan
            - rollback_plan
            - verification_plan
            - spike_result

    on_exhausted: pause
```

This is simple enough to read, but powerful enough to encode cross-loop collaboration.

## 3.2 New lifecycle state

Add task states:

```text
pending
claimed
running
implemented
verified
checkpointed
done

incubating
incubated_ready
incubated_blocked
incubated_exhausted
```

Add Workcell states:

```text
created
classifying
planning
critiquing
spiking
synthesizing
promotion_check
ready_for_implementation
blocked
exhausted
archived
```

The important distinction:

```text
A daemon run can be forever.
A hard-task workcell must be bounded.
```

Your forever daemon should not create infinite scratchpad churn. The daemon can run forever across tasks, but each Workcell gets an explicit pass/cost/time/attempt budget.

# 4. Hard-task routing: do not route everything

The router should only promote tasks into Workcells when one of these is true:

```yaml
hard_tasks:
  route:
    when: easy_queue_empty
```

or:

```yaml
hard_tasks:
  route:
    when:
      any:
        - prior_attempts_gte: 2
        - task_score_gte: 0.72
        - explicit_label: hard
```

or:

```yaml
hard_tasks:
  route:
    when:
      all:
        - model_confidence_lte: 0.55
        - evidence_missing: true
        - estimated_blast_radius_gte: medium
```

I would make `easy_queue_empty` the default because it matches your actual concern: let the daemon chew through high-confidence work first, then route the remaining scary items into a more careful process.

## 4.1 Risk score formula

Do not hide this in the prompt. Make it a host-side function with transparent evidence:

```ts
riskScore =
  0.18 * priorFailureScore +
  0.16 * blastRadiusScore +
  0.14 * subsystemCountScore +
  0.12 * testOracleWeaknessScore +
  0.10 * auditSeverityScore +
  0.10 * dependencyUncertaintyScore +
  0.08 * algorithmicNoveltyScore +
  0.07 * reviewerBlockerScore +
  0.03 * modelSelfUncertaintyScore +
  0.02 * userLabelScore
```

Then persist the explanation:

```json
{
  "task_id": "audit-parser-cap-rule",
  "risk_score": 0.81,
  "route": "workcell",
  "signals": {
    "prior_failures": 2,
    "touches_subsystems": ["parser", "session", "audit"],
    "test_oracle": "weak",
    "model_self_uncertainty": 0.64,
    "reviewer_blockers": 1
  }
}
```

This keeps the model honest and debuggable.

# 5. Confidence gate: evidence, not vibes

Use this pattern:

```yaml
confidence:
  source: evidence_weighted
  promote_if:
    gte: 0.82
  components:
    plan_consensus:
      weight: 0.20
      require: true
    critic_clearance:
      weight: 0.20
      require: true
    spike_success:
      weight: 0.20
      require: false
    verification_plan_quality:
      weight: 0.15
      require: true
    rollback_safety:
      weight: 0.10
      require: true
    test_oracle_strength:
      weight: 0.10
      require: false
    model_self_report:
      weight: 0.05
      require: false
```

The key rule:

```text
The model can claim confidence.
Only evidence can promote the task.
```

A task becomes implementable only if the host can point to:

```text
- accepted plan
- known risks
- critic blockers resolved or waived
- test/verification plan
- rollback plan
- optional spike result
```

## 6.3 Cleanup

Add:

```yaml
hard_tasks:
  workcell:
    scratch:
      cleanup:
        on_promoted: keep_summary_and_accepted_plan
        on_done: archive_raw_logs_after: 7d
        on_abort: keep_all
        on_satisfied: summarize_and_gc
```

Runtime cleanup behavior:

```text
On task done:
  keep:
    STATE.md
    accepted plan
    final summary
    evidence receipts
    commit SHA
  archive:
    raw worker logs
    stale candidate plans
    spike scratch files
  delete:
    temporary worktrees, unless configured otherwise
```

This prevents `.jekko/daemon` from becoming an infinite landfill.

# 7. Cross-loop collaboration patterns to support

## Pattern A: independent braid

Best for algorithmic or ambiguous tasks where anchoring is dangerous.

```yaml
- id: independent_plans
  fanout: 3
  isolation: fresh_context
  share: none
```

Each agent starts fresh, sees the task and minimal repo context, and proposes a solution independently. This avoids forcing everyone to inherit a bad first plan.

## Pattern B: relay refinement

Best when one promising plan exists but needs improvement.

```yaml
- id: relay_refine
  rounds: 3
  pass_previous: true
  instruction: improve_or_reject_previous_plan
```

Agent B reads Agent A’s plan and must either improve it or explicitly reject it with reasons. Agent C reads both.

## Pattern C: red-team pass

Best before implementation.

```yaml
- id: red_team
  fanout: 2
  objective: find_failure_modes
  output: blocker_report
```

The critic should not propose a giant new plan unless the current one is fundamentally flawed. It should produce blockers, missing tests, rollback issues, and “do not implement until…” items.

## Pattern D: spike/prototype

Best when the model cannot know whether the idea works without touching code.

```yaml
- id: spike
  isolation: git_worktree
  writes: scratch_only
  max_files: 5
  max_time: 20m
```

Spikes must not directly become the final implementation unless promoted. They are evidence producers.

## Pattern E: proof/test-design pass

Best for tasks where implementation looks easy but correctness is hard.

```yaml
- id: proof
  agent: plan
  objective: define invariants_and_tests
  output: verification_plan
```

## Pattern F: synthesis and lock

Best after several plans/critiques.

```yaml
- id: synthesize
  reads: [plan_candidate, blocker_report, spike_result]
  output: accepted_plan
  lock_after: true
```

After lock, implementation agents should not freestyle a new architecture unless they explicitly request reopening the Workcell.

# 10. Syntax library additions

## `hard_tasks.route`

Controls whether a task enters incubation.

```yaml
hard_tasks:
  route:
    enabled: true
    when: easy_queue_empty | always | after_failed_attempts | explicit_label
    score:
      hard_if_gte: 0.72
      low_confidence_if_lte: 0.55
```

## `hard_tasks.route.signals`

Defines scoring inputs.

```yaml
signals:
  prior_failed_attempt: { weight: 0.18 }
  touches_multiple_subsystems: { weight: 0.16 }
  weak_or_missing_tests: { weight: 0.12 }
  algorithmic_unknown: { weight: 0.12 }
  reviewer_blocker: { weight: 0.10 }
  model_self_uncertainty: { weight: 0.04 }
```

## `hard_tasks.workcell`

Creates durable task memory.

```yaml
workcell:
  max_cycles: 5
  max_wall_time: 2h
  max_cost_usd: 8
```

## `hard_tasks.workcell.scratch`

Controls where plans/prototypes/notes live.

```yaml
scratch:
  store: sqlite | sqlite_and_mirror
  mirror_path: ".jekko/daemon/${run.id}/workcells/${task.id}/"
  write_policy: generated_only | scratch_worktree | repo_allowed_after_gate
  cleanup:
    on_done: summarize_and_archive
```

## `passes[].isolation`

Controls context sharing.

```yaml
isolation: fresh_context | read_workcell | read_accepted_plan | full_daemon_context
```

Meanings:

```text
fresh_context:
  Worker sees task + pinned repo rules, but not prior candidate plans.

read_workcell:
  Worker sees all Workcell artifacts.

read_accepted_plan:
  Worker sees only the synthesized accepted plan.

full_daemon_context:
  Worker sees broader daemon context. Use sparingly.
```

## `passes[].share`

Controls anti-anchoring.

```yaml
share: none | prior_pass | all_artifacts | accepted_only
```

## `passes[].output`

Typed artifact contract.

```yaml
output:
  kind: plan_candidate
  schema:
    required:
      - summary
      - assumptions
      - implementation_steps
      - risks
      - tests
      - rollback
```

## `gate`

Promotion gate.

```yaml
gate:
  confidence_gte: 0.82
  no_unresolved_blockers: true
  require_artifacts:
    - accepted_plan
    - verification_plan
    - rollback_plan
```

## `on_exhausted`

Bounded failure behavior.

```yaml
on_exhausted:
  action: pause | create_followup_task | ask_user | downgrade_to_manual
  reason: "Could not reach confidence gate."
```

## `reopen`

Lets implementation reopen incubation if reality contradicts the plan.

```yaml
reopen:
  if:
    - verification_plan_invalid
    - implementation_discovers_new_architecture
    - red_team_blocker_after_implementation
  max_reopens: 2
```

## `cleanup`

Ensures scratchpads do not accumulate forever.

```yaml
cleanup:
  worktrees: delete_on_done
  raw_logs: archive_after_7d
  candidate_plans: keep_winner_and_summaries
  mirrors: keep_state_and_receipts
```

# 11. Runtime flow update

Add these phases to the daemon controller:

```text
created
armed
evaluating_stop
discovering_tasks
selecting_task
classifying_task

if easy:
  running_iteration
  checkpointing
  evaluating_stop

if hard:
  workcell_created
  workcell_pass_running
  workcell_synthesizing
  workcell_criticizing
  workcell_spiking
  workcell_promotion_check

if promoted:
  running_accepted_plan
  checkpointing
  verifying
  cleanup_workcell
  evaluating_stop

if exhausted:
  paused_hard_task
```

The central loop becomes:

```ts
while (daemon.active) {
  await evaluateStopChecks()

  const task = await selectNextTask()

  const route = await classifyTask(task)

  if (route === "easy") {
    await runImplementationIteration(task)
    await checkpointIfVerified(task)
    continue
  }

  const workcell = await getOrCreateWorkcell(task)
  const result = await runWorkcellCycle(workcell)

  if (result.status === "promoted") {
    await runAcceptedPlan(task, result.acceptedPlan)
    await checkpointIfVerified(task)
    await cleanupWorkcell(task)
    continue
  }

  if (result.status === "exhausted") {
    await pauseDaemon("Hard task incubation exhausted")
    return
  }
}
```

# 15. Concrete safety rules

These should be hard-coded defaults:

```text
1. A Workcell cannot run forever.
2. A Workcell cannot write to the main worktree before promotion.
3. A spike cannot be committed directly.
4. A model self-confidence score cannot promote a task alone.
5. Independent plan workers must not see each other’s outputs.
6. Red-team blockers must be resolved, waived by policy, or pause the daemon.
7. Failed Workcells become paused/blocked tasks, not infinite retries.
8. Scratch mirrors are generated artifacts; DB is source of truth.
9. Cleanup never deletes final summaries, accepted plans, evidence, or commit SHAs.
10. One writer at a time unless using isolated worktrees.
```

This also responds to the Jekko community concern that parallel task/subagent execution and agent teams need safe coordination, not uncontrolled same-worktree writes. ([GitHub][11])

# 18. The final conceptual model

ZYAL should have three nested control layers:

```text
Daemon
  Runs forever until host stop checks pass.

Task
  Work item selected from ledger/audit/git/tasklist.

Workcell
  Bounded hard-task incubator for tasks that are too risky to implement directly.
```

And three kinds of memory:

```text
Daemon memory:
  objective, stop criteria, global ledger, checkpoint history.

Task memory:
  attempts, evidence, status, verification, commit refs.

Workcell memory:
  plans, critiques, spikes, blockers, decisions, accepted plan.
```

That gives you the behavior you want:

```text
Easy tasks:
  fast path

Hard tasks:
  structured collaborative incubation

Impossible / under-specified tasks:
  bounded pause with evidence, not infinite failure

Forever loop:
  continues across tasks, compaction, assistant stops, and epochs

Workcell loop:
  bounded, evidence-gated, cleaned up when complete
```

The highest-leverage addition is therefore:

```yaml
hard_tasks:
  route: ...
  workcell: ...
  passes: ...
  confidence: ...
  cleanup: ...
```

That turns ZYAL from “an infinite coding loop” into a real autonomous engineering runtime.
