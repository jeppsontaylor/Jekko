Yes. The missing layer is a **bounded hard-task incubation system** inside ZYAL.

The parent daemon can still be **FOREVER**, but an individual hard task should not spin forever. It should enter a **bounded “incubator” lane** where multiple passes, fresh agents, refiner agents, critics, prototypes, and scratchpads compound knowledge until either:

1. the task becomes executable with enough evidence-backed confidence,
2. it is split into smaller tasks,
3. it is escalated to a human/reviewer, or
4. it exhausts a bounded pass/budget limit and returns to the task queue as blocked/incubated.

This is the main update I would make to the current ZYAL plan. The uploaded plan already correctly says ZYAL should be host-enforced, SQLite-backed, preview/armed, and not prompt-only; this addition should live inside that same durable runtime, not as a loose Markdown tracker or another prompt convention.

# Executive recommendation

Add a new ZYAL subsystem:

```yaml
hard_tasks:
  enabled: true
```

Public name ideas:

```text
ZYAL Hard Task Incubator
ZYAL Forge
ZYAL Quest Mode
ZYAL Research Lane
ZYAL Deepening Passes
```

I would choose **Hard Task Incubator** for docs and schema because it is descriptive and not too magical. In the TUI, it can be branded as **FORGE** or **QUEST** if you want it to feel powerful.

Core idea:

```text
Normal daemon loop:
  choose task → fix → validate → checkpoint → repeat forever

Hard-task path:
  detect high risk / low confidence / repeated failure
  → create bounded task workbench
  → run multiple structured passes
  → preserve useful memory
  → prototype safely
  → review/adversarial critique
  → either promote to execution or split/escalate/block
```

This is strongly aligned with existing agent communities. OpenHands has experimental critic-driven iterative refinement with success thresholds, max iterations, and follow-up prompts; its docs describe a critic that scores task success probability and triggers refinement until a threshold or max iteration is reached. ([OpenHands Docs][1]) OpenHands also has stuck detection for repeated action/observation cycles, repeated errors, monologues, and alternating ping-pong patterns, which maps directly to your “agent may struggle and make mistakes” concern. ([OpenHands Docs][2]) LangGraph’s core positioning is long-running, stateful agents with durable execution, memory, and human-in-the-loop, and its interrupt mechanism persists graph state before waiting for external input. ([GitHub][3]) AutoGen has explicit termination conditions evaluated after agent responses and group-chat managers that choose speakers until termination is reached. ([Microsoft GitHub][4]) Google ADK’s `LoopAgent` is deterministic and runs sub-agents until a max iteration count or termination mechanism, and its docs explicitly use writer/critic iterative improvement as an example. ([Adk][5])

The key lesson from all of those systems is: **do not let model self-confidence alone drive the workflow**. Use model confidence as one input, but host-side routing, validators, critics, state, budgets, and stop conditions must decide what happens.

# The most important design rule

## Parent loops may be infinite. Task incubation must be bounded.

Use:

```yaml
loop:
  policy: forever
```

for the daemon.

Use:

```yaml
hard_tasks:
  incubation:
    max_passes: 6
    max_runtime: 45m
    max_cost_usd: 8
```

for each hard task.

Why: if every hard task gets its own forever loop, one bad task can absorb the daemon forever. The parent forever loop should keep making progress across the repo; the hard-task incubator should create useful knowledge and then either execute, split, block, or requeue.

# Why this should not route every task

The failure mode you already see coming is real: if the syntax says “when confidence is low, incubate,” the model may route too much into incubation. That makes the system expensive and slow.

So route to incubation only when **host-computed risk** or **historical failure evidence** crosses a threshold.

Bad:

```yaml
route_when:
  confidence_below: 0.8
```

Better:

```yaml
route_when:
  any:
    - host_risk_score:
        gte: 0.72
    - repeated_failure:
        same_task: 2
    - stuck_detector:
        trips: true
    - touches_sensitive_area:
        any: [migration, auth, permission, payment, generated_contracts]
    - uncertainty:
        agent_confidence_lte: 0.55
        and_validator_coverage_lte: 0.5
```

The agent’s confidence should be one signal, not the authority. The runtime should compute a risk score from objective features:

```text
diff size
number of files touched
sensitive paths
schema/migration/generated zones
known failing tests
lack of tests
previous failed attempts
permission/security implications
unresolved design ambiguity
cross-package ownership
merge conflict likelihood
model self-confidence
reviewer/critic score
validator coverage
```

OpenHands’ critic model is a useful reference here because it treats the critic score as a trigger for iterative refinement, with thresholds and max iterations, instead of blindly trusting the main agent’s “done” signal. ([OpenHands Docs][1])

# Why this syntax works

It separates the concerns that should never be mixed:

```text
classify      = should this task be incubated?
incubation    = how do we study it?
workbench     = where does memory go?
passes        = what perspectives are required?
outcomes      = how does the host decide what next?
cleanup       = what survives after completion?
```

The most important bit is this:

```yaml
read_prior_workbench: false
```

versus:

```yaml
read_prior_workbench: true
```

That gives you both behaviors you mentioned:

1. **Fresh agents** that make a new plan without being anchored by the previous plan.
2. **Refiner agents** that are forced to read and improve the current best plan.

You want both. Fresh planners reduce anchoring. Refiners preserve accumulated insight.

OpenHands has a directly related “fork a conversation” feature where the fork deep-copies events and state while leaving the source immutable; that is essentially the safe version of “start fresh but do not destroy the original.” ([OpenHands Docs][6])

# Evidence confidence, not vibes confidence

Do not call the promotion field simply `confidence`. Call it:

```yaml
evidence_confidence_gte: 0.78
```

Because a model’s self-confidence is cheap. Evidence confidence should be computed from things like:

```text
Does a concrete plan exist?
Are acceptance tests identified?
Did a reviewer find no critical blocker?
Did prototype compile or produce useful output?
Are risky assumptions listed?
Are prior failed approaches recorded?
Is there a rollback plan?
Are touched files/path locks known?
Is the task small enough now?
```

Suggested calculation model:

```yaml
confidence_model:
  evidence_confidence:
    acceptance_tests_defined: 0.20
    prototype_success: 0.20
    reviewer_no_critical_blockers: 0.20
    implementation_plan_specific: 0.15
    rollback_plan_exists: 0.10
    prior_failures_explained: 0.10
    agent_self_confidence: 0.05
```

Notice `agent_self_confidence` is only 5%.

That is the difference between “the model feels good” and “the runtime has evidence that this task is ready.”

# The best “hard task” collaboration pattern

Use five roles, but do not always run all five. The runtime chooses based on risk/budget.

## 1. Triage agent

Purpose:

```text
Classify task risk, missing information, likely failure modes.
```

Output:

```yaml
risk_score: 0.81
confidence: 0.42
recommended_route: incubate
reasons:
  - touches session loop and compaction
  - tests not obvious
  - previous agent stopped twice
```

## 2. Fresh planner agents

Purpose:

```text
Make independent plans without seeing prior plans.
```

Why:

```text
Avoids anchoring.
Finds different decompositions.
Surfaces hidden assumptions.
```

Syntax:

```yaml
- id: fresh_plan_a
  type: fresh_plan
  read_prior_workbench: false

- id: fresh_plan_b
  type: fresh_plan
  read_prior_workbench: false
```

## 3. Refiner agent

Purpose:

```text
Read all plans and build the current best plan.
```

Syntax:

```yaml
- id: refine_best
  type: refine_plan
  read_prior_workbench: true
```

## 4. Prototype agent

Purpose:

```text
Try a risky idea in a throwaway worktree or scratch branch.
```

Syntax:

```yaml
- id: prototype
  type: prototype
  isolation: git_worktree
  commit: false
```

## 5. Reviewer / red-team agent

Purpose:

```text
Attack the plan before execution.
```

Syntax:

```yaml
- id: adversarial_review
  type: review
  focus: [regressions, unsafe_assumptions, missing_tests]
```

This is similar in spirit to OpenHands’ refactor/critique loop, where a primary agent does the work and a critique agent scores it before deciding whether to continue. ([OpenHands Docs][7]) It also lines up with AutoGen’s group-chat manager pattern, where a manager selects the next speaker and stops issuing turns when termination is reached. ([Microsoft GitHub][8])

# Three syntax styles to support

I would support one canonical syntax and two convenience syntaxes.

## 1. Canonical YAML ZYAL

Best for serious work.

```yaml

  enabled: true
  incubation:
    max_passes: 6
```

Why it works:

```text

easy to validate
easy to diff

natural fit for existing plan
```

Best for humans in TUI.

Why it works:

```text

safe because it opens preview instead of running
```

## 3. Library template

Best for repeatable workflows.

```text
dangerous defaults can be avoided
```

Do **not** add XML tags or loose inline commands. The Jekko community is already experimenting with auto-continue plugins and stop hooks; one Reddit plugin describes nudging the agent when it emits `TASK_STATUS: INCOMPLETE`, capped at 20 nudges, and an Jekko issue proposes a `session.stopping` hook but warns unconditional `stop=false` can run forever. ([Reddit][9]) This reinforces the ZYAL design: continuation should be explicit, armed, bounded where appropriate, and host-controlled.

# New parser gates for hard-task syntax

Existing gates remain:

```text
opening sentinel first non-whitespace content
column 1
matching close id
intent required
confirm required
trailing ZYAL_ARM required
reject code fences
reject block quotes
reject duplicate blocks
```

Add these gates:

```text
intent: incubate cannot use confirm: RUN_FOREVER

hard_tasks.incubation must have at least one terminal outcome
prototype pass must declare isolation
worker write tasks require git_worktree or file locks
```

Invalid examples:

```yaml
intent: incubate
```

Reject because task incubation must be bounded.

```yaml
hard_tasks:

    max_passes: infinite
```

Reject.

```yaml
passes:
  - type: prototype
    isolation: same_worktree
```

Reject unless paths are locked and write scope is exclusive.

# Hard-task routing algorithm

Pseudocode-level plan:

```text
select next task
load task history
compute static risk
compute historical failure risk
compute validator coverage
ask model for uncertainty only if needed
combine into risk/confidence score

if task is low risk and validators exist:
  execute fast path

else if repeated failure or high risk:
  incubate bounded

else:
  execute fast path but enable stricter reviewer/checkpoint
```

Important: do not ask the model “should this be hard?” for every task. That is expensive and noisy. First use host features. Ask the model only when the host score is near the threshold.

```yaml
classify:
  model_call:
    only_when_score_between: [0.55, 0.72]
```

# Example: hard task workbench output

The agent should not write random scratch notes. It should write structured entries.

```markdown
# Workbench: daemon-hard-clear-design

## Problem
Implement hard context clearing for daemon sessions without deleting DB history.

## Constraints
- Normal sessions must not change behavior.
- Generic compaction auto-continue must not own daemon continuation.
- Hard clear creates new daemon epoch/child session.
- Old DB messages remain readable.

## Known Risks
- Breaking `jekko run` plain idle exit.
- Duplicating session status semantics.
- Losing pinned stop criteria during epoch rotation.

## Failed Approaches
1. Modifying `runLoop()` to never break globally.
   Reason rejected: changes normal sessions and risks infinite-loop bugs.

## Current Best Plan
1. Keep `prompt()` semantics unchanged.
2. Add `loopResult()` terminal reason.
3. Daemon runner creates child session on hard clear.
4. Inject pinned daemon capsule into child session.

## Acceptance Tests
- Plain `jekko run` exits on idle.
- Daemon continues after assistant stop when stop checks fail.
- Hard clear creates epoch=2 and preserves run state.
- Old messages remain readable.

## Open Questions
- Exact API for child session creation.
```

This format is useful to humans and compresses well during context clearing.

# Cleanup policy

Hard-task scratchpads can become garbage if unmanaged. Make cleanup explicit.

```yaml
workbench:
  cleanup:
    on_completed:
      summarize_to: task_ledger
      keep:
        - WORKBENCH.md
        - BEST_PLAN.md
        - ACCEPTANCE_TESTS.md
        - final_review
      delete:
        - raw_tool_outputs
        - temporary_worktrees
        - prototype_branches
      after: immediate

    on_promoted:
      keep_until_task_done: true
      prune_raw_after: 7d

    on_blocked:
      keep_full_workbench: true

    on_aborted:
      keep_summary: true
      delete_worktrees: ask
```

Do not delete the SQLite history. Delete or archive only mirrors, worktrees, raw logs, and temporary artifacts.

# How this interacts with context clearing

The hard-task workbench should become part of the daemon context capsule.

When a task is incubating, each new pass gets:

```text
original job objective
current task statement
hard constraints
current workbench summary
attempt summaries
do_not_repeat list
open questions
promotion criteria
allowed tools/paths
```

Fresh-plan passes intentionally get less:

```yaml
read_prior_workbench: false
context:
  include:
    - original_task
    - constraints
    - relevant_files
  exclude:
    - prior_plans
    - prior_reviewer_opinions
```

Refiner/reviewer passes get more:

```yaml
read_prior_workbench: true
context:
  include:
    - all_attempt_summaries
    - failed_approaches
    - best_current_plan
    - prototype_results
    - reviewer_feedback
```

This lets you control anchoring versus continuity.

OpenHands’ condenser and persistence docs are relevant here: its condenser summarizes long histories when size limits are exceeded, and its persisted conversation state includes message history, execution status, iteration count, tool outputs, statistics, workspace context, activated skills, secrets, and custom agent state. ([OpenHands Docs][10]) ZYAL should follow that architecture but make the task workbench a first-class, queryable store rather than an accidental side effect of chat history.

# Multi-agent collaboration: better than `THIS_TRACKER.md`

Replace:

```text
Read THIS_TRACKER.md and update it.
```

with brokered task operations:

```text
daemon_lease_task
daemon_read_workbench
daemon_write_memory
daemon_submit_attempt
daemon_request_review
daemon_promote_task
daemon_split_task
daemon_mark_blocked
```

No worker directly owns the source of truth. They submit structured results. The supervisor/integrator decides.

## Worker protocol

```text
1. Worker leases task.
2. Worker receives scoped context.
3. Worker works in isolated worktree if writing.
4. Worker submits attempt summary + artifacts.
5. Reviewer scores attempt.
6. Integrator promotes/splits/blocks.
7. Lease is released.
```

## Why not same-worktree workers?

Because parallel write agents in the same worktree will eventually stomp each other. Use:

```yaml
workers:
  isolation: git_worktree
```

or:

```yaml
locks:
  paths:
    - "packages/jekko/src/session/**"
```

Claude Code’s subagent docs show a mature UI model where running subagents can be viewed/stopped and library/project/plugin subagents can be managed; that is the kind of UX Jekko should eventually mirror for ZYAL workers. ([Claude][11]) OpenAI’s Agents SDK docs also frame agents as applications that plan, call tools, collaborate across specialists, and keep enough state for multi-step work, with the application owning orchestration, tool execution, approvals, and state; that is exactly the separation ZYAL should preserve. ([OpenAI Developers][12])

# How this relates to existing standards

Do **not** market ZYAL as replacing existing standards.

Position it like this:

```text
AGENTS.md: repo instructions for agents
MCP: tool/resource/prompt protocol
POML/PDL: prompt composition languages
LangGraph/ADK/AutoGen/CrewAI: code-first orchestration frameworks
ZYAL: Jekko-native runbook syntax for host-enforced coding-agent loops
```

AGENTS.md is a simple open format for guiding coding agents and is used by many open-source projects; it is instruction context, not a daemon workflow language. ([Agents][13]) MCP is the tool/resource/prompt interoperability protocol; it standardizes connections to external tools and context, but it is not itself a task-incubation loop language. ([Model Context Protocol][14]) POML and PDL are prompt-organization/composition systems, useful references for structure and validation, but they do not solve Jekko-specific git checkpointing, worktree isolation, TUI arming, or forever daemon semantics. ([Microsoft GitHub][15])

ZYAL’s unique value is:

```text
host-enforced agentic coding runbooks inside Jekko
with explicit arming, stop checks, git policy, context clearing,
task ledger, hard-task incubation, and TUI visibility
```

# The cleanest updated plan statement

I would add this to the ZYAL plan:

> **Phase 2.5: Hard Task Incubator.** Add host-routed, bounded incubation for high-risk or low-confidence tasks. The daemon remains forever, but individual hard-task passes are bounded. Each hard task gets a durable SQLite-backed workbench with generated human-readable mirrors. The runtime can run fresh planners, plan refiners, prototype attempts in isolated worktrees, and adversarial reviewers. Promotion to implementation is controlled by evidence confidence, validators, and reviewer results, not model self-confidence alone. Completed task workbenches are summarized and cleaned; blocked task workbenches are preserved for human handoff.

# Final syntax recommendation

Use these public concepts:

```yaml
hard_tasks:
  enabled: true

incubation:
  max_passes: 6

workbench:
  source: sqlite
  mirror: ".jekko/daemon/{{run.id}}/tasks/{{task.id}}/"

passes:
  - type: fresh_plan
    read_prior_workbench: false

  - type: refine_plan
    read_prior_workbench: true

  - type: prototype
    isolation: git_worktree

  - type: adversarial_review

  - type: converge

promote_when:
  evidence_confidence_gte: 0.78
```

Those names are simple, readable, and map to known patterns:

```text
LoopAgent / iterative refinement       → incubation passes
critic / reviewer                      → adversarial_review
LangGraph persistence/checkpointing    → workbench + durable state
AutoGen termination conditions         → promote/block/stop predicates
Claude hooks / Jekko stopping hook  → host-enforced continuation gates
MCP                                    → tool/resource layer, not loop authority
```

The highest-leverage idea is the **task workbench**. Once every hard task has a durable structured memory space, agents can compound knowledge across context clears, fresh starts, subagents, prototypes, and reviews without relying on one fragile chat transcript or one giant tracker Markdown file.
