## Executive recommendation — the best agent-search stack in 2026

For safe, agent-friendly internet and publication search, **do not rely on one search engine**. The best architecture is a **router** that chooses between privacy-first web search, AI-native search, source-specific publication APIs, and a controlled knowledge store.

My top stack:

**1. Primary safe web search:** **Brave Search API**
Best all-around production choice for agents that need privacy-oriented, independent web search. Brave says its Search API is built on an independent index rather than Google/Bing, and its API supports web/news/images/videos plus LLM-oriented context features. ([Brave][1])

**2. Private/self-hosted metasearch:** **SearXNG**
Best “not tracked” option when you can self-host. SearXNG aggregates results from many search services, and its docs/GitHub describe users as neither tracked nor profiled. Public instances are convenient but should not be treated as private because you do not control the operator. ([GitHub][2])

**3. AI-native agent search:** **Tavily + Exa + Parallel/Linkup as optional specialists**
Tavily is excellent for RAG-style search/extract/crawl workflows; Exa is excellent for semantic/neural discovery and “find similar” research; Parallel and Linkup are strong newer options optimized for LLM-ready excerpts and enterprise/security workflows. ([Tavily][3])

**4. Reading/extraction layer:** **Firecrawl, Jina Reader, Crawl4AI, Trafilatura**
Search results are not enough. Agents should fetch, parse, and cite source pages. Firecrawl is the best hosted agent-ready crawling/extraction layer; Jina Reader is very useful for URL-to-Markdown and search-read flows; Crawl4AI and Trafilatura are strong self-hosted/open-source extraction choices. ([Firecrawl][4])

**5. Publications and scholarly research:** **OpenAlex + Crossref + Semantic Scholar + PubMed/NCBI + arXiv + CORE + Europe PMC + Unpaywall + Lens**
This is the best publication stack. Use OpenAlex for broad scholarly graph discovery, Crossref for DOI metadata, Semantic Scholar for paper/citation exploration and recommendations, PubMed/NCBI for biomedical literature, arXiv for preprints, CORE/Europe PMC for open-access full text, Unpaywall for legal OA copies by DOI, and Lens for patents plus scholarly records. ([OpenAlex][5])

**6. Knowledge management:** **LlamaIndex or Haystack + LangGraph + Qdrant/Weaviate/pgvector + eval/observability**
Use LlamaIndex or Haystack to ingest and query external knowledge, LangGraph for stateful multi-step agents, and a vector/hybrid database such as Qdrant, Weaviate, Milvus, or pgvector. Add Ragas, Phoenix, LangSmith, or TruLens for evaluation and tracing. ([GitHub][6])

# Best tools by category

## 1. Privacy-first web search

### 1. Brave Search API — best production default

**Use it for:** general web search, news search, current facts, agent search tools, fallback search, LLM context retrieval.

**Why it ranks first:** Brave has a real API, an independent index, production documentation, and strong privacy positioning. Its docs describe the API as privacy-first developer infrastructure built on Brave’s independent index, not Google or Bing. ([Brave][1]) Brave also offers an official MCP server for AI assistants, including web/local/place/image/video/news/LLM-context/AI-summary tools. ([GitHub][7])

**Limitations:** It is still a third-party API, so for highly sensitive queries you should review its enterprise privacy terms and data-retention behavior. It is not the same privacy posture as a fully self-hosted search stack.

**Verdict:** **Best default web-search API for agents.**

---

### 2. SearXNG — best self-hosted “not tracked” search

**Use it for:** private metasearch, internal agent tools, low-cost fallback search, open-source deployments.

SearXNG is a free metasearch engine that aggregates results from many search services/databases and says users are neither tracked nor profiled. ([GitHub][2]) Its private-instance documentation is important: public instances are run by unknown operators, while private instances can be run by a company, group, VPN, or single user. ([SearXNG Documentation][8])

**Strengths:** self-hostable, configurable engines, privacy control, JSON endpoint, strong agent community adoption.

**Limitations:** upstream engines may rate-limit or block; result quality varies by enabled engines; public instances are not automatically private; production deployments need caching, backoff, and monitoring.

**Verdict:** **Best privacy-first open-source option.**

---

### 3. Mojeek — best independent privacy search alternative

**Use it for:** independent-index fallback, privacy-oriented search, non-Google/Bing result diversity.

Mojeek describes itself as a no-tracking independent search engine. Its API returns publicly available page URLs, titles, and snippets. ([Mojeek][9])

**Strengths:** own crawler/index, privacy orientation, useful diversity against Brave/SearXNG.

**Limitations:** smaller index and less agent tooling than Brave, Tavily, or Exa; the API is closer to classic search results than full AI-ready retrieval.

**Verdict:** **Excellent secondary independent index.**

---

### 4. DuckDuckGo / DDGS — good privacy signal, weak official agent API

DuckDuckGo remains strong for human privacy. Its privacy policy says it does not save or share search/browsing history, and its help docs emphasize private search features. ([DuckDuckGo][10])

However, DuckDuckGo’s official Instant Answer API is **not** a full search-results API; it does not provide all links or full SERP coverage. ([Postman][11]) Libraries such as `ddgs`/`duckduckgo_search` can be useful for prototypes, but they are unofficial metasearch/scraping-style tools and can be fragile. ([GitHub][12])

**Verdict:** **Good for lightweight/prototype search; not the best production agent search backbone.**

---

### 5. Kagi — high-quality private search, but API access is constrained

Kagi is private/ad-free and has useful enrichment and summarization APIs, but its Search API is listed as invite-only beta. ([Kagi Help][13])

**Verdict:** **Promising, but not the safest default for agent infrastructure until Search API access is broadly available.**

# 2. Best AI-native search APIs for agents

## Tavily — best “search + scrape + RAG-ready” API

Tavily is specifically built for AI agents and RAG workflows. Its docs describe a Search API that aggregates sources, scores/ranks/filters them, and can return LLM-ready context with controls for content fields and token limits. ([Tavily][3]) Tavily also has Search, Extract, Crawl, and Map capabilities, plus an MCP server. ([Tavily Docs][14])

**Best for:** general AI agents, RAG pipelines, “search and return usable text,” coding agents, web research agents.

**Caution:** It is proprietary and vendor-mediated. Review privacy, retention, and enterprise terms for sensitive workloads. Tavily’s trust center states its systems and controls are audited under SOC 2 Type II. ([trust.tavily.com][15])

**Verdict:** **Best agent-first search/extraction API.**

## Exa — best semantic/neural discovery

Exa is built as a search engine for AIs. Its docs expose search, contents, answer, and research capabilities, and its search modes include neural, fast, auto, and deeper modes. ([Exa][16]) Exa is especially good when the agent needs to find conceptually similar pages, companies, papers, code, or obscure pages that keyword search may miss. ([Amazon Web Services, Inc.][17])

**Best for:** semantic discovery, “find me things like this,” deep research, company/people/code search, broad exploratory research.

**Caution:** Neural search can surface surprising results; pair with a strict source-quality/reranking layer.

**Verdict:** **Best semantic search layer.**

## Parallel — strong new agent-search API

Parallel’s Search API takes a natural-language objective and returns relevant excerpts optimized for LLMs, reducing the need for multiple keyword searches. It also publishes agent-readable docs via `llms.txt`/Markdown conventions. **Best for:** objective-based agent search, LLM-ready excerpts, newer agent workflows.

**Verdict:** **High-potential modern agent-search provider.**

## Linkup — strong enterprise/security-oriented search API

Linkup’s `/search` endpoint is positioned as context retrieval for web content, returning sources to ground LLM answers. ([Linkup][20]) Its docs mention Zero Data Retention, SOC 2 Type II, GDPR, SSO, and even “Bring Your Own Cloud” for high data-sovereignty requirements. ([Linkup][21])

**Best for:** enterprise agents, regulated workflows, sales/market/competitive intelligence, source-grounded LLM answers.

**Caution:** Validate coverage and latency on your actual query set.

**Verdict:** **Best emerging enterprise privacy/compliance search API.**

# 5. Best publication and scholarly search stack

For online publications, especially scholarly literature, agents should **not** rely on general web search. Use source-specific APIs.

## Core scholarly layer

| Need                       | Best tool                     | Why                                                                                                                          |
| -------------------------- | ----------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| Broad scholarly graph      | **OpenAlex**                  | Open catalog of hundreds of millions of scholarly works, linking works, authors, institutions, funders, etc. ([OpenAlex][5]) |
| DOI metadata               | **Crossref**                  | REST API for works, journals, members, funders, licenses, prefixes, and types. ([www.crossref.org][34])                      |
| Paper/citation exploration | **Semantic Scholar API**      | APIs for papers, authors, citations, venues, recommendations, datasets. ([Semantic Scholar][35])                             |
| Biomedical literature      | **PubMed / NCBI E-utilities** | Public API to Entrez databases including PubMed and PMC. ([NCBI][36])                                                        |
| Preprints                  | **arXiv API**                 | Programmatic access to arXiv search and metadata. ([arXiv][37])                                                              |
| Open-access full text      | **CORE**                      | Metadata and full-text/PDF access from open-access journals and repositories. ([CORE][38])                                   |
| Life-science full text     | **Europe PMC**                | RESTful service for publications and related information in Europe PMC. ([Europe PMC][39])                                   |
| Legal OA copies            | **Unpaywall**                 | Free programmatic access to the Unpaywall database for open-access locations by DOI. ([Unpaywall][40])                       |
| Patents + scholarly        | **Lens**                      | REST access to scholarly works and patents. ([Lens API][41])                                                                 |

## Best publication-search method

Use this pipeline:

1. **Query expansion:** generate keywords, synonyms, authors, DOI/title variants, MeSH terms for biomedical topics, and date ranges.
2. **Discovery:** search OpenAlex, Semantic Scholar, PubMed/arXiv/domain APIs.
3. **Metadata normalization:** resolve DOI, PMID, PMCID, arXiv ID, title, author, year, venue.
4. **Citation expansion:** collect references and cited-by links from OpenAlex/Semantic Scholar/Lens.
5. **Full-text acquisition:** use CORE, Europe PMC, PubMed Central, Unpaywall, and publisher APIs where legally available.
6. **Quality scoring:** rank by source type, venue, date, methodology, citations, retraction/status checks where available, and relevance.
7. **Chunking and indexing:** store abstracts, full text, tables, methods, claims, and citation relationships separately.
8. **Answering:** require citations to primary literature, not blog summaries, for scientific claims.

For rate-limit safety, follow each API’s published rules. NCBI, for example, documents 3 requests/second without an API key and 10 requests/second with a key. ([NLM Support Center][42]) Crossref recommends identifying clients with mailto/User-Agent, caching, and backing off when response times increase. ([www.crossref.org][43])

# 8. MCP: the best interface layer for agent tools

Use **Model Context Protocol** as the integration layer where possible. MCP is an open standard for connecting AI systems with tools and data sources; MCP servers expose tools that models can invoke, with schemas and controlled access. ([Anthropic][58])

Good MCP-ready search options include Brave, Tavily, Exa, SearXNG implementations, and emerging multi-provider web-search MCP servers. Brave has an official MCP server; Tavily and Exa have MCP servers; community SearXNG MCP servers are available. ([GitHub][7])

**Recommendation:** build a **unified internal Search MCP** with these tools behind it:

* `web_search_privacy`: Brave + SearXNG + Mojeek
* `web_search_ai`: Tavily + Exa + Parallel/Linkup
* `read_url`: Firecrawl/Jina/Crawl4AI/Trafilatura
* `scholarly_search`: OpenAlex + Crossref + Semantic Scholar + PubMed + arXiv
* `news_search`: GDELT + Media Cloud + AskNews/NewsAPI
* `verify_claim`: Fact Check API + source triangulation
* `store_knowledge`: vector/hybrid DB with provenance

# 10. Safety requirements for agent internet search

Agents are vulnerable because webpages can contain instructions, hidden text, malicious HTML, prompt injections, and poisoned content. OWASP defines prompt injection as manipulating model responses in ways that can bypass safety measures, and NIST’s 2025 adversarial ML taxonomy discusses adversarial attacks and mitigations across the AI lifecycle. ([OWASP Gen AI Security Project][63])

Your search agent should enforce these rules:

1. **Treat every webpage as untrusted data.**
   Never let page text instruct the agent to call tools, reveal secrets, change policies, or ignore the system prompt.

2. **Separate content from instructions.**
   Wrap retrieved text in a clearly labeled data block. The model should summarize it, not obey it.

3. **No direct action from search results.**
   Search results can inform an answer, but they should not trigger payments, messages, account changes, downloads, or code execution without a separate policy gate.

4. **Use allowlists for high-stakes domains.**
   For medicine, law, finance, security, and science, prefer official docs, primary publications, standards bodies, government sites, and peer-reviewed sources.

5. **Fetch pages before answering.**
   Do not rely only on snippets except for low-risk queries. Snippets are often incomplete or misleading.

6. **Triangulate current or high-stakes claims.**
   Require two or more independent sources for important claims, and prefer primary sources.

7. **Preserve provenance.**
   Store URL, title, author, publisher, publication date, retrieval time, content hash, extraction method, and citation metadata.

8. **Respect robots.txt, API terms, and rate limits.**
   The Robots Exclusion Protocol is not authorization, but well-behaved agents should honor it and use official APIs where possible. ([IETF Datatracker][64])

9. **Cache safely.**
   Cache public results to reduce repeated calls, but avoid storing sensitive user queries unless explicitly needed.

10. **Evaluate adversarial pages.**
    Test pages that contain hidden prompt injections, fake citations, contradictory evidence, and manipulated metadata.

# 11. Best implementation blueprint

## Tier A — privacy-first open-source stack

Use this when privacy/control matters most.

* **Search:** self-hosted SearXNG + Mojeek API + optional Brave Search API
* **Extraction:** Crawl4AI + Trafilatura + optional Jina Reader
* **Publications:** OpenAlex, Crossref, Semantic Scholar, PubMed/NCBI, arXiv, CORE, Europe PMC, Unpaywall
* **Storage:** pgvector or Qdrant
* **Orchestration:** Haystack or LlamaIndex + LangGraph
* **Evaluation:** Phoenix + Ragas or TruLens
* **Interface:** internal MCP server

This is the best stack for “not tracked” agent search.

---

## Tier B — best production SaaS stack

Use this when speed, quality, and reliability matter most.

* **Search:** Brave Search API
* **AI search:** Tavily + Exa
* **Optional enterprise search:** Linkup or Parallel
* **Extraction:** Firecrawl + Jina Reader
* **Publications:** OpenAlex/Crossref/Semantic Scholar/PubMed/arXiv/CORE/Unpaywall
* **Storage:** Qdrant or Weaviate
* **Orchestration:** LangGraph + LlamaIndex
* **Evaluation:** LangSmith + Phoenix/Ragas

This is the best stack for most real products.

---

## Tier C — regulated/enterprise stack

Use this when privacy, auditability, and compliance matter.

* **Search:** private SearXNG, Brave enterprise terms, Linkup ZDR/BYOC where appropriate
* **Extraction:** self-hosted Crawl4AI/Trafilatura or contracted Firecrawl with retention review
* **Storage:** pgvector/Qdrant/Weaviate in your VPC
* **Policies:** source allowlists, audit logs, no sensitive query retention, human approval for external actions

Use this to test quickly.

3. **Tavily** — best agent/RAG search plus extraction.
4. **Exa** — best semantic/neural discovery.
5. **Firecrawl** — best hosted page extraction/crawling.
6. **OpenAlex + Crossref + Semantic Scholar** — best scholarly metadata/citation base.
7. **PubMed/arXiv/CORE/Europe PMC/Unpaywall/Lens** — best domain-specific publication layer.

## What I would not use as the main backbone

* **DuckDuckGo official Instant Answer API** as the only search layer: private brand, but not a full search-results API. ([Postman][11])
* **Public SearXNG instances** for sensitive queries: convenient, but operator trust is unknown. ([SearXNG Documentation][8])
* **Google Scholar scraping** as a primary scholarly workflow: use OpenAlex, Crossref, Semantic Scholar, PubMed, arXiv, CORE, and Unpaywall instead.
* **Bing Search API classic:** retired; Azure Grounding with Bing is a different product with compliance caveats. ([Microsoft Learn][29])
* **Single-vendor search:** every provider has blind spots, cost changes, outage risk, and ranking bias.

## The ideal agent-search design

Build a **Search Router**:

1. Classify query: current web, academic, news, code, company, legal/medical/financial, internal knowledge, or high-risk.
2. Select tools:

   * Privacy/current web → Brave + SearXNG/Mojeek
   * AI-ready context → Tavily/Exa/Parallel/Linkup
   * Publications → OpenAlex/Crossref/Semantic Scholar/PubMed/arXiv/CORE/Unpaywall
   * News → GDELT/Media Cloud/AskNews/NewsAPI
3. Retrieve top results from at least two independent sources for important claims.
4. Fetch and extract source pages.
5. Deduplicate and normalize metadata.
6. Rerank with hybrid retrieval.
7. Store source-backed chunks with timestamps and citations.
8. Answer only from retrieved evidence.
9. Cite everything important.
10. Evaluate continuously for recall, groundedness, freshness, source quality, privacy leakage, and prompt-injection resistance.

The strongest practical answer is: **Brave + self-hosted SearXNG + Tavily + Exa + Firecrawl/Jina + OpenAlex/Crossref/Semantic Scholar/PubMed/arXiv/CORE/Unpaywall + LlamaIndex/Haystack + Qdrant/pgvector + LangGraph/MCP + Ragas/Phoenix.**
