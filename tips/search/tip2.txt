# Agent-safe search and publication-research stack — May 2026

Here’s the big conclusion: **do not give agents “one search engine.”** Give them a **search router** that chooses the safest and highest-quality source for the job.

My recommended production stack is:

**Brave Search API Enterprise ZDR + Mojeek/SearXNG privacy fallback + Tavily + Exa + Parallel or Perplexity Search + Firecrawl/Crawl4AI + OpenAlex/Semantic Scholar/Crossref/PubMed/arXiv/CORE/Unpaywall + LlamaIndex or LangGraph + Qdrant/pgvector + Zep/Graphiti or Mem0.**

The key distinction is that **privacy search for humans is not the same thing as agent-ready search**. DuckDuckGo is excellent for private human search, but its own Instant Answer API says it is **not** a full search-results API, so it should not be the primary agent search tool. ([DuckDuckGo][1])

There are two different “publication” use cases: **news/media publications** and **scholarly/scientific publications**.

| ---------------------------------------------- | ------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| **Media Cloud**                                | Researching media ecosystems and news flows            | Media Cloud is an open-source media research project with search/directory APIs and a news archive model built around sources, feeds, and stories. ([Media Cloud][18]) |

## 4. GitHub, X, LinkedIn, Reddit: best approach for agents

| Platform     | Best agent method                                                                                                                                                                                                                                                                                                        | What to avoid                                                                                                                    |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------- |
| **GitHub**   | Use GitHub REST/GraphQL APIs and the official GitHub MCP Server. The MCP server lets AI tools read repositories/code, manage issues/PRs, analyze workflows/security, and configure toolsets/read-only access. ([GitHub Docs][29])                                                                                        | Avoid unauthenticated scraping and uncontrolled write access. Use narrow scopes, tool allowlists, and read-only mode by default. |
| **X**        | Use the official X API for public conversation search. X provides recent search for the last 7 days and full-archive search back to 2006 for pay-per-use/enterprise customers. ([X Developer Platform][30])                                                                                                              | Avoid scraping timelines/search pages; use API access and respect pricing/terms.                                                 |
| **Reddit**   | Use Reddit Data API or PRAW for permitted Reddit access. Reddit requires OAuth for API clients and says commercial use, research beyond limits, or non-permitted use requires separate agreement; its terms also restrict AI training on user content without express permission from rightsholders. ([Reddit Help][31]) | Avoid bulk scraping or using Reddit content for model training/memory without rights review.                                     |
| **LinkedIn** | Use official LinkedIn API products only: Consumer, Marketing, Sales/Talent/Learning/Regulatory products, and approved Posts API use cases. ([LinkedIn Developer][32])                                                                                                                                                    | Do not treat LinkedIn as an open public search corpus. Broad scraping is high-risk and brittle.                                  |

**Social/dev recommendation:** For public internet agents, use **GitHub API/MCP**, **X API**, **Reddit API/PRAW under terms**, and **LinkedIn official APIs only**. For LinkedIn, build workflows around permitted org/member-owned data rather than broad scraping.

## 6. Best methods for agents searching the internet and publications

### A. Use a search router, not one search tool

Classify every query before searching:

1. **Sensitive/private query** → Brave Enterprise ZDR, Mojeek, or self-hosted SearXNG/Crawl4AI. Strip secrets, names, account IDs, and unnecessary user context.
2. **General web query** → Brave + Tavily or Exa.
3. **Deep multi-hop research** → Parallel + Exa + Firecrawl.
4. **News/media query** → GDELT/Media Cloud/NewsAPI/Event Registry + Brave/Tavily/You.com news.
5. **Scholarly query** → OpenAlex + Semantic Scholar + Crossref + PubMed/arXiv/CORE/Unpaywall.
6. **Social query** → official platform APIs only: X API, Reddit API, GitHub API/MCP, LinkedIn approved APIs.

### B. Search in stages

A strong agent search loop should look like this:

1. **Plan:** Identify topic, date range, required source type, language, jurisdiction, and freshness.
2. **Generate queries:** Produce exact-name queries, synonym queries, domain-filtered queries, and source-specific queries.
3. **Retrieve broadly:** Query at least two independent systems when correctness matters.
4. **Deduplicate:** Canonicalize URLs, DOIs, PMIDs, arXiv IDs, and publisher landing pages.
5. **Extract:** Prefer official APIs, RSS feeds, open full-text sources, and legal OA sources. Use Firecrawl/Crawl4AI/Jina/Docling only when official feeds are insufficient.
6. **Rerank:** Score by source authority, recency, primary-source status, citation count where relevant, and whether the page actually supports the claim.
7. **Verify:** Require at least one primary source for factual claims, and multiple independent sources for controversial or high-stakes claims.
8. **Store:** Save metadata, source URL/DOI, retrieval time, license, hash/snapshot, chunks, embeddings, and citation anchors.
9. **Update memory carefully:** Put only verified durable facts into long-term memory; keep volatile search observations temporary.

### C. Scholarly-publication workflow

For academic literature, agents should not just keyword-search and summarize the first PDF. A better flow is:

**OpenAlex broad search → Semantic Scholar relevance/citation expansion → Crossref DOI metadata validation → PubMed/arXiv/domain-specific search → Unpaywall/CORE/Europe PMC full-text lookup → PDF parsing → citation graph expansion → summary with provenance.**

For biomedical work, add **PubMed MeSH terms**, article-type filters, clinical-trial filters, and PMC full-text extraction. PubMed and NCBI APIs are the canonical route for biomedical search automation. ([PubMed][24])

### D. Security rules for web-reading agents

Treat every webpage, PDF, GitHub README, Reddit thread, and scraped article as **untrusted input**. OWASP explicitly calls out indirect prompt injection from websites and files as a risk for LLM applications. ([OWASP Gen AI Security Project][38])

Minimum controls:

* Search tools should be **read-only by default**.
* Retrieved text must never be allowed to issue tool instructions.
* Use separate channels for **data** versus **agent instructions**.
* Strip scripts, hidden text, forms, tracking pixels, and suspicious metadata before model ingestion.
* Never let a retrieved page trigger email, payment, code execution, credential use, browser actions, or write operations without explicit policy checks.
* Use allowlists for high-trust domains and blocklists for known spam/malware.
* Log retrieval provenance and keep source snapshots for auditability.
* Do not store third-party content unless your provider plan and the publisher’s rights permit it.

## 7. Recommended stacks

### Best balanced production stack

Use this for most serious agent systems:

* **Search:** Brave Search API Enterprise ZDR + Tavily + Exa
* **Deep research:** Parallel or Perplexity Search
* **Extraction:** Firecrawl hosted + Crawl4AI self-host fallback
* **Publications:** OpenAlex + Semantic Scholar + Crossref + PubMed + arXiv + CORE + Unpaywall
* **Docs/PDFs:** LlamaParse + Docling
* **Knowledge:** Qdrant or pgvector + Graphiti/Zep or Mem0
* **Agent framework:** LangGraph or LlamaIndex
* **Tooling:** MCP wrappers with read-only defaults

### Best privacy-first/self-hosted stack

Use this when queries may contain sensitive context:

* **Search:** Brave Enterprise ZDR, Mojeek, self-hosted SearXNG
* **Crawling:** Crawl4AI self-hosted
* **Parsing:** Docling local, Marker, or LlamaParse only for non-sensitive docs
* **Storage:** pgvector or Qdrant self-hosted
* **Memory:** Graphiti self-hosted or carefully scoped Mem0/Zep
* **Policy:** no external provider receives raw PII/secrets

### Best scholarly research stack

Use this for literature-review agents:

* **Discovery:** OpenAlex + Semantic Scholar
* **Metadata truth:** Crossref + DOI normalization
* **Biomedical:** PubMed + PMC + Europe PMC
* **Preprints:** arXiv
* **Full text:** Unpaywall + CORE + publisher OA pages
* **Parsing:** LlamaParse or Docling
* **Knowledge graph:** Graphiti/Zep for paper → author → institution → citation → claim relationships
* **Evaluation:** citation accuracy, source coverage, DOI correctness, retraction checks

### Best social/dev intelligence stack

Use this for agents studying developer and social signals:

* **GitHub:** GitHub REST/GraphQL + GitHub MCP Server
* **X:** official X API recent/full-archive search
* **Reddit:** Reddit Data API + PRAW, under Reddit terms
* **LinkedIn:** approved LinkedIn APIs only
* **Do not:** scrape private, login-gated, or terms-restricted platform content into agent memory without rights review

## Final ranking: what I would actually buy/build first

1. **Brave Search API Enterprise ZDR** — best privacy-first production search foundation.
2. **Tavily** — easiest agent/RAG search API to operationalize.
3. **Exa** — best AI-native semantic search and similar-document/company/person/code discovery.
4. **Firecrawl + Crawl4AI** — best pair for hosted plus self-hosted extraction.
5. **OpenAlex + Semantic Scholar + Crossref** — core scholarly search/metadata trio.
6. **PubMed/PMC + arXiv + Unpaywall/CORE/Europe PMC** — domain-specific scholarly/full-text layer.
7. **Parallel or Perplexity Search** — deep/multi-hop or raw-ranked search enhancement.
8. **Qdrant or pgvector** — durable retrieval memory.
9. **Graphiti/Zep or Mem0** — long-term memory and evolving knowledge.
10. **MCP everywhere** — the clean interface for giving agents controlled search, extraction, GitHub, database, and file access.

The strongest architecture is **privacy-aware routing + multi-source retrieval + controlled extraction + provenance-rich knowledge storage + prompt-injection hardening**. That gives agents useful internet access without turning them into uncontrolled scrapers or leaking sensitive queries into every search provider.
