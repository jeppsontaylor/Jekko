## My strongest recommendation

As of **May 11, 2026**, the best agent-search stack is **not a single “private Google replacement.”** The best stack is layered:

**Production default:**
**Brave Search API + Brave LLM Context API** for independent, agent-ready web search; **self-hosted SearXNG** for privacy-controlled metasearch; **Exa/Tavily/Firecrawl/Jina Reader** for agent-native search, extraction, and crawling; **OpenAlex + Semantic Scholar + Crossref + PubMed/Europe PMC + arXiv + CORE + Unpaywall** for publications; and **Qdrant + Tantivy/Postgres** for long-term knowledge management.

That mix gives you privacy, independent indexing, strong retrieval quality, source provenance, publication coverage, and Rust-friendly infrastructure.

## 2. My ranked “best stack” for your agent system

### Tier 1: Core production search

**1. Brave Search API + Brave LLM Context API**
This is my top production pick for agents that need web search without depending on Google/Bing scraping. Brave emphasizes its independent index, and the LLM Context API is especially interesting because it returns pre-extracted, compact context chunks rather than only links. ([Brave][1])

Use it for:

* General web discovery.
* RAG-ready web snippets.
* Fast fact-finding.
* Privacy-sensitive enterprise workflows with ZDR.
* Reducing dependence on search-result scraping.

**2. Self-hosted SearXNG**
This is the best “not tracked, like DuckDuckGo, but agent-controllable” option. Run your own instance, disable engines you do not trust, enable JSON output, rate-limit agents, and treat it as a privacy gateway. SearXNG is explicitly no-profile/no-track oriented and supports API-like search endpoints. ([SearxNG][3])

Use it for:

* Private metasearch.
* Cross-checking Brave/Exa/Tavily.
* Internal agents where you want infrastructure control.
* Query fanout without committing to one vendor.

**3. Mojeek**
Mojeek is a good independent-index alternative, especially when you want non-Big-Tech search with an actual API and no ads. It is not as agent-polished as Brave/Tavily/Exa, but it is valuable for diversity. ([Mojeek][6])

**4. Kagi Search API**
Kagi is attractive for quality and privacy posture, but I would treat it as a premium fallback rather than the default unless you already use Kagi internally. It says it does not log searches or tie them to accounts, but the API is paid/beta-gated. ([Kagi Help][7])

**5. DuckDuckGo**
Excellent human-private search, but not a great primary agent API. Use it for manual/human workflows or lightweight instant-answer tasks, not as the backbone of agent search. DuckDuckGo’s own privacy posture is strong, but the official API is not a full search API. ([DuckDuckGo][4])

## 3. Best agent-native search/extraction tools

### Tavily

Tavily is one of the cleanest “search for AI agents” APIs. It combines search, scraping, filtering, and extraction in a single call, which reduces glue code. ([Tavily Docs][8])

Best for:

* Autonomous research agents.
* “Find me the best sources and summarize evidence.”
* Rapid product iteration.
* Teams that want fewer moving parts.

### Exa

Exa is especially good for semantic discovery: “find pages like this,” research exploration, company/code/web search, and AI-native search patterns. Its docs explicitly frame it as a custom search engine built for AIs and agents. ([Exa][16])

Best for:

* Semantic search.
* Company/repo/source discovery.
* Research discovery where keyword search misses things.

### Firecrawl

Firecrawl is my top pick for extraction and crawl workflows. It supports search, scrape, interact, agent, crawl, map, and batch scrape; its agent endpoint can search/navigate/retrieve without pre-specified URLs. ([GitHub][9])

Best for:

* Pages that need JS rendering.
* Site crawling.
* Markdown extraction.
* Screenshots/HTML/structured extraction.
* Agent browsing with guardrails.

### Jina Reader

Jina Reader is excellent as a lightweight extraction layer. It can return title/content/timestamp JSON and has a search mode that returns results as readable LLM-friendly content. ([Jina AI][17])

Best for:

* Quick conversion of public pages to clean text.
* Lightweight agent fetches.
* Low-glue-code RAG ingestion.

## 4. Best methods for online publications

For publications, do **not** rely on general web search first. Use publication-native APIs first, then web search only for missing context.

### Recommended scholarly routing

| Task                                                         | Best API/tool                        |
| ------------------------------------------------------------ | ------------------------------------ |
| Broad scholarly discovery                                    | **OpenAlex**                         |
| Citation graph, paper recommendations, embeddings            | **Semantic Scholar**                 |
| DOI metadata and canonical title/author/publisher resolution | **Crossref**                         |
| Biomedical/life sciences                                     | **PubMed, PMC, Europe PMC**          |
| Preprints in physics/math/CS/etc.                            | **arXiv API**                        |
| Open-access full text                                        | **CORE, Unpaywall, Europe PMC, PMC** |
| Journal-level OA quality checks                              | **DOAJ**                             |
| Datasets, preprints, samples, research objects               | **DataCite**                         |

OpenAlex is particularly strong as the default scholarly index because it exposes works with fields like DOI, title, publication year/date, type, language, citation count, and retraction status. Semantic Scholar complements it with papers/authors/citations/venues, recommendations, and SPECTER2 embeddings. Crossref is the best canonical DOI metadata source, and PubMed/PMC/Europe PMC should be preferred for biomedical literature. ([OpenAlex Developers][10])

### Publication search method

For each research query:

1. Search **OpenAlex** for broad candidate papers.
2. Resolve DOI/title/author metadata through **Crossref**.
3. Pull citation context and related-paper recommendations from **Semantic Scholar**.
4. For biomedical topics, search **PubMed/PMC/Europe PMC** directly.
5. For preprints, query **arXiv**.
6. For full text, check **Unpaywall**, **CORE**, **Europe PMC**, or the publisher’s allowed OA version.
7. Store every evidence item with DOI/PMID/arXiv ID, URL, license, source API, query, rank, retrieved timestamp, and excerpt offsets.

For RAG over papers, preserve paper structure. Chunk by section, subsection, paragraph, table, figure caption, abstract, methods, and conclusion, rather than using blind token windows. Recent agentic RAG work emphasizes planning, tool use, reflection, adaptive retrieval, hybrid retrieval, reranking, and structure-aware indexing. ([arXiv][18])

## 5. Social/code/news sources: what to use and what to avoid

### GitHub

Use **GitHub REST**, **GitHub GraphQL**, and **GitHub MCP**. GitHub is the best source for open-source tool discovery, code examples, release velocity, issue quality, and dependency intelligence. GitHub’s newer issue search work also includes semantic/hybrid search types in GraphQL. ([GitHub Docs][11])

### Reddit

Use the official Reddit API and listing endpoints where possible. Reddit is useful for qualitative feedback, product complaints, real-world failure modes, and “what actually works” sentiment. Do not treat Reddit as authoritative evidence without verification. ([Reddit][12])

### Hacker News

Use the official HN Firebase API for near-real-time public data and Algolia’s HN Search API for search. This is great for developer tooling signals. ([GitHub][19])

### X

Use the official X API only. X has recent and full-archive search endpoints, but access, pricing, and policy constraints can change quickly. ([X Developer Platform][20])

### LinkedIn

Do **not** scrape LinkedIn. Use approved APIs/partner access only. LinkedIn’s terms explicitly restrict unauthorized bots, scraping, automated copying, and crawling. ([LinkedIn][13])

### News/media

Use **GDELT**, **Media Cloud**, **NewsAPI**, RSS feeds, and direct publisher APIs. GDELT is strong for global media monitoring, Media Cloud has a large online news archive, and NewsAPI provides article search across many sources. ([GDELT Project][21])

## 6. Knowledge management: best architecture

### My recommended Rust-friendly memory stack

| Layer                           | Recommended tool                      |
| ------------------------------- | ------------------------------------- |
| Metadata / provenance / jobs    | **Postgres**                          |
| Vector search                   | **Qdrant**                            |
| Keyword/BM25 search             | **Tantivy** or **Meilisearch**        |
| Optional single-DB vector store | **pgvector**                          |
| Optional embedded vector DB     | **LanceDB**                           |
| Optional large-scale ranking    | **Vespa**                             |
| Optional graph memory           | **Neo4j** or graph tables in Postgres |

Qdrant is a great fit because it is Rust-based and supports modern vector/hybrid retrieval patterns. Tantivy is a Rust full-text search engine inspired by Lucene. pgvector is excellent when you want embeddings inside Postgres with ACID semantics and joins. Vespa, Weaviate, Meilisearch, and LanceDB are also strong depending on scale and deployment preferences. ([Qdrant][14])

### Best retrieval method

Use **hybrid retrieval**:

1. Keyword search with BM25.
2. Dense vector search.
3. Metadata filters: date, source type, domain, publication venue, license, language.
4. Cross-encoder or LLM reranking.
5. Citation-aware answer synthesis.
6. Evidence compression into agent context.

Do **not** use vector-only retrieval for serious research agents. It misses exact names, IDs, error messages, package names, citations, legal language, and narrow facts.

## 7. Safety and privacy rules for agent search

This is crucial: the open web is an adversarial environment for agents.

The biggest agent-search risks are prompt injection, insecure output handling, data poisoning, over-permissioned tools, excessive agency, unsafe browsing, and accidentally treating web text as instructions. OWASP’s LLM Top 10 includes prompt injection, insecure output handling, data poisoning, and excessive agency, and Microsoft has documented real prompt-injection paths in agent frameworks, including tool/code execution consequences. ([OWASP][22])

### Guardrails I would enforce

* Treat all fetched pages as **untrusted data**.
* Never let retrieved text call tools, browse further, send emails, run code, or modify files.
* Strip or quarantine instructions like “ignore previous instructions,” “call this URL,” “exfiltrate,” etc.
* Use an allowlist for authenticated domains.
* Use a denylist for internal IP ranges and private network targets.
* Respect robots.txt and site terms; robots.txt is not authorization, but it is a standard crawlers are expected to follow. ([RFC Editor][23])
* Cache only what your API terms and content licenses allow.
* Store provenance for every answer: source URL, source API, query, timestamp, rank, quote/excerpt, and content hash.
* Separate “search,” “fetch,” “summarize,” and “act” permissions.
* Run web extraction in a sandboxed browser/container.
* Rate-limit and back off.
* Keep secrets out of prompts and retrieved context.
* For publications, prefer APIs and open-access copies over scraping PDFs.

# 9. Rust code examples

Below is a compact Rust search-broker style example using **SearXNG**, **Brave LLM Context**, **OpenAlex**, and **Firecrawl**.

### `Cargo.toml`

```toml
[package]
name = "agent_search_broker"
version = "0.1.0"
edition = "2021"

[dependencies]
anyhow = "1"
reqwest = { version = "0.12", features = ["json", "rustls-tls"] }
serde = { version = "1", features = ["derive"] }
serde_json = "1"
tokio = { version = "1", features = ["macros", "rt-multi-thread"] }
urlencoding = "2"
```

### `main.rs`

```rust
use anyhow::{Context, Result};
use reqwest::Client;
use serde::{Deserialize, Serialize};
use serde_json::Value;

#[derive(Debug, Clone, Serialize)]
pub struct SearchHit {
    pub title: String,
    pub url: String,
    pub snippet: Option<String>,
    pub source: String,
    pub score: Option<f64>,
}

#[derive(Debug, Deserialize)]
struct SearxngResponse {
    results: Vec<SearxngHit>,
}

#[derive(Debug, Deserialize)]
struct SearxngHit {
    title: Option<String>,
    url: String,
    content: Option<String>,
    score: Option<f64>,
    engine: Option<String>,
}

/// Search a self-hosted SearXNG instance.
///
/// Example instance:
///   https://searxng.yourdomain.example
///
/// Make sure JSON output is enabled in your SearXNG settings.
pub async fn searxng_search(
    client: &Client,
    instance: &str,
    query: &str,
    limit: usize,
) -> Result<Vec<SearchHit>> {
    let url = format!(
        "{}/search?q={}&format=json&language=en&categories=general&safesearch=1",
        instance.trim_end_matches('/'),
        urlencoding::encode(query)
    );

    let response = client
        .get(url)
        .send()
        .await
        .context("failed to call SearXNG")?
        .error_for_status()
        .context("SearXNG returned an error status")?
        .json::<SearxngResponse>()
        .await
        .context("failed to parse SearXNG response")?;

    Ok(response
        .results
        .into_iter()
        .take(limit)
        .map(|hit| SearchHit {
            title: hit.title.unwrap_or_else(|| "Untitled".to_string()),
            url: hit.url,
            snippet: hit.content,
            source: format!("searxng:{}", hit.engine.unwrap_or_else(|| "unknown".to_string())),
            score: hit.score,
        })
        .collect())
}

/// Brave LLM Context API.
///
/// This returns compact context suitable for RAG/agents.
/// Keep the response as serde_json::Value because providers may evolve response fields.
pub async fn brave_llm_context(
    client: &Client,
    brave_api_key: &str,
    query: &str,
) -> Result<Value> {
    let response = client
        .get("https://api.search.brave.com/res/v1/llm/context")
        .header("X-Subscription-Token", brave_api_key)
        .query(&[("q", query)])
        .send()
        .await
        .context("failed to call Brave LLM Context API")?
        .error_for_status()
        .context("Brave returned an error status")?
        .json::<Value>()
        .await
        .context("failed to parse Brave response")?;

    Ok(response)
}

/// Search OpenAlex works.
///
/// Good default for academic/publication discovery.
/// Uses `mailto` for polite API usage.
pub async fn openalex_works(
    client: &Client,
    query: &str,
    mailto: &str,
    limit: usize,
) -> Result<Vec<SearchHit>> {
    let per_page = limit.to_string();

    let response = client
        .get("https://api.openalex.org/works")
        .query(&[
            ("search", query),
            ("per-page", per_page.as_str()),
            ("filter", "is_oa:true"),
            ("sort", "cited_by_count:desc"),
            ("mailto", mailto),
        ])
        .send()
        .await
        .context("failed to call OpenAlex")?
        .error_for_status()
        .context("OpenAlex returned an error status")?
        .json::<Value>()
        .await
        .context("failed to parse OpenAlex response")?;

    let mut hits = Vec::new();

    if let Some(results) = response.get("results").and_then(|v| v.as_array()) {
        for work in results {
            let title = work
                .get("title")
                .and_then(|v| v.as_str())
                .unwrap_or("Untitled")
                .to_string();

            let url = work
                .get("doi")
                .and_then(|v| v.as_str())
                .or_else(|| work.get("id").and_then(|v| v.as_str()))
                .unwrap_or("")
                .to_string();

            let year = work
                .get("publication_year")
                .and_then(|v| v.as_i64())
                .map(|y| y.to_string());

            let cited_by = work
                .get("cited_by_count")
                .and_then(|v| v.as_i64())
                .map(|c| c.to_string());

            let snippet = Some(format!(
                "year={}, cited_by={}",
                year.unwrap_or_else(|| "unknown".to_string()),
                cited_by.unwrap_or_else(|| "unknown".to_string())
            ));

            hits.push(SearchHit {
                title,
                url,
                snippet,
                source: "openalex".to_string(),
                score: None,
            });
        }
    }

    Ok(hits)
}

/// Firecrawl scrape example.
///
/// Use after you already decided a URL is worth fetching.
/// Store markdown plus source metadata; do not let page text instruct the agent.
pub async fn firecrawl_scrape_markdown(
    client: &Client,
    firecrawl_api_key: &str,
    url: &str,
) -> Result<Value> {
    let body = serde_json::json!({
        "url": url,
        "formats": ["markdown"]
    });

    let response = client
        .post("https://api.firecrawl.dev/v2/scrape")
        .bearer_auth(firecrawl_api_key)
        .json(&body)
        .send()
        .await
        .context("failed to call Firecrawl")?
        .error_for_status()
        .context("Firecrawl returned an error status")?
        .json::<Value>()
        .await
        .context("failed to parse Firecrawl response")?;

    Ok(response)
}

#[tokio::main]
async fn main() -> Result<()> {
    let client = Client::builder()
        .user_agent("agent-search-broker/0.1 contact=research@example.com")
        .build()
        .context("failed to build HTTP client")?;

    let query = "agentic RAG hybrid retrieval citation aware";

    let searxng_instance =
        std::env::var("SEARXNG_INSTANCE").unwrap_or_else(|_| "https://searx.example.com".into());

    let brave_key = std::env::var("BRAVE_API_KEY").ok();
    let firecrawl_key = std::env::var("FIRECRAWL_API_KEY").ok();

    let web_hits = searxng_search(&client, &searxng_instance, query, 5).await?;
    println!("SearXNG hits:\n{:#?}", web_hits);

    let papers = openalex_works(&client, query, "research@example.com", 5).await?;
    println!("OpenAlex papers:\n{:#?}", papers);

    if let Some(key) = brave_key {
        let context = brave_llm_context(&client, &key, query).await?;
        println!(
            "Brave LLM context:\n{}",
            serde_json::to_string_pretty(&context)?
        );
    }

    if let (Some(key), Some(first_hit)) = (firecrawl_key, web_hits.first()) {
        let scraped = firecrawl_scrape_markdown(&client, &key, &first_hit.url).await?;
        println!(
            "Firecrawl scrape:\n{}",
            serde_json::to_string_pretty(&scraped)?
        );
    }

    Ok(())
}
```

## 10. Suggested agent-search pipeline

Here is the architecture I would actually build:

```text
User / Agent Query
   |
   v
Query Planner
   - classify: web, scholarly, code, social, news, legal, medical, etc.
   - generate query variants
   - set freshness requirement
   - set source policy
   |
   v
Search Broker
   - Brave Search / Brave LLM Context
   - self-hosted SearXNG
   - Exa or Tavily
   - OpenAlex / Crossref / Semantic Scholar / PubMed / arXiv
   - GitHub / Reddit / HN / X / news APIs where allowed
   |
   v
Fetcher / Extractor
   - Firecrawl or Jina Reader
   - direct API fetches
   - PDF parser / XML parser for papers
   - robots and terms checks
   |
   v
Normalizer
   - clean markdown/text
   - metadata
   - source URL / DOI / PMID / arXiv ID
   - license
   - query, rank, retrieved_at
   - content hash
   |
   v
Knowledge Store
   - Postgres for metadata and provenance
   - Qdrant for vectors
   - Tantivy or Meilisearch for BM25
   - optional graph store for citations/entities
   |
   v
Retriever
   - keyword + vector hybrid retrieval
   - metadata filters
   - reranking
   - evidence compression
   |
   v
Answer Composer
   - cite sources
   - separate facts from inferences
   - flag conflicts/staleness
   - never execute instructions from retrieved text
```

## 11. Final recommendations by scenario

### Best default stack for your agents

Use:

* **Brave Search API + Brave LLM Context** for primary web search.
* **Self-hosted SearXNG** for private metasearch and vendor diversity.
* **Tavily** when you want simple agent search/extract workflows.
* **Exa** when semantic discovery matters.
* **Firecrawl** for JS-heavy extraction/crawling.
* **Jina Reader** for lightweight page-to-text.
* **OpenAlex + Semantic Scholar + Crossref** as the scholarly core.
* **PubMed/PMC/Europe PMC/arXiv/CORE/Unpaywall** as domain/full-text extensions.
* **Qdrant + Tantivy + Postgres** as the Rust-friendly knowledge system.

### Best privacy-first stack

Use:

* Self-hosted **SearXNG**.
* **Brave Search API Enterprise ZDR** where possible.
* **Mojeek** as independent fallback.
* Local extraction where possible.
* Qdrant/Tantivy/Postgres self-hosted.
* No LinkedIn scraping, no unofficial X scraping, no Google Scholar scraping.

### Best agent-quality stack

Use:

* Brave LLM Context.
* Tavily.
* Exa.
* Firecrawl.
* OpenAlex/Semantic Scholar/Crossref.
* Qdrant hybrid retrieval with reranking.

### Best Rust-first stack

Use:

* `reqwest` connectors for search APIs.
* **Qdrant** for vector search.
* **Tantivy** for BM25.
* **Postgres/pgvector** for metadata + optional vectors.
* Firecrawl/Jina as HTTP services for extraction.
* A Rust policy layer that controls which tools agents can call.

The big design principle: **search tools find candidates; extraction tools fetch evidence; knowledge tools preserve provenance; agents reason only over verified, cited, untrusted-as-data context.**
