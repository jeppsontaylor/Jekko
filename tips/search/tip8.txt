Fantastic brief. The best answer is **not one search engine**; it is a governed **agent search broker** that routes each query to the right source, strips sensitive data, fetches only allowed content, extracts clean text, scores evidence, and writes provenance-rich memories.

## Privacy/search-engine reality check

For “not tracked like DuckDuckGo,” the critical nuance is this: **a third-party API can still see the query, API key, timestamp, and source IP.** The safest agent setup is therefore: redact sensitive query text, route through an org-owned backend, cache aggressively, use self-hosted SearXNG where possible, and prefer no-tracking providers such as Brave/Mojeek when you need a real web index. DuckDuckGo is great for humans and says it does not track searches or browsing history, but its public Instant Answer API is not a full web-results API, so I would not make it the primary agent search backend. ([DuckDuckGo][8])

Kagi is also worth watching: it has a strong privacy posture, a public Universal Summarizer API, public enrichment APIs, and a Search API, but its Search API is still closed beta/invite-only, so I would treat it as optional rather than core infrastructure. ([Kagi][9])

## What I would build

Build a **Search Broker** service with these adapters:

1. **Privacy-first web adapter:** Brave Search API, SearXNG, Mojeek.
2. **Agent search adapter:** Tavily, Exa, Linkup, Valyu.
3. **Publication adapter:** OpenAlex, Crossref, Semantic Scholar, arXiv, PubMed, Europe PMC, Unpaywall, CORE.
4. **News adapter:** GDELT, Media Cloud, publisher feeds, NewsAPI.ai/Event Registry if budget allows.
5. **Social/dev adapter:** GitHub, Reddit, X, LinkedIn approved APIs only.
6. **Extractor adapter:** Firecrawl/Crawl4AI/Jina/Trafilatura/GROBID.
7. **Memory adapter:** Qdrant or pgvector plus Tantivy/BM25 plus strict provenance metadata.

Do **not** let the LLM freely browse and click arbitrary pages with privileged tools. OWASP flags indirect prompt injection as a risk when LLMs ingest external sources like websites or files, and its prompt-injection guidance recommends isolating untrusted content from privileged tool use. ([OWASP Gen AI Security Project][10])

## Operating rules for safe agent search

Use this retrieval flow:

1. **Classify the query.** Is it current news, scholarly, code, social sentiment, legal/medical/financial, or general web?
2. **Redact before search.** Strip secrets, user identifiers, customer names, emails, internal URLs, auth tokens, and private context unless explicitly allowed.
3. **Prefer source-native APIs.** Use PubMed for biomedical, arXiv for preprints, GitHub for repositories, Reddit/X official APIs for social, and Crossref/OpenAlex for DOI metadata.
4. **Use search engines for discovery, not final truth.** Search results identify candidates; authoritative pages establish facts.
5. **Fetch content through an extractor.** Convert HTML/PDF into markdown/structured records; never execute site scripts for ordinary search.
6. **Respect robots.txt, terms, rate limits, and paywalls.** Robots.txt is standardized as the Robots Exclusion Protocol, but it is not access authorization; still, compliant agents should honor it and publisher terms. ([RFC Editor][11])
7. **Quarantine external content.** Treat retrieved pages as untrusted data, not instructions.
8. **Store provenance.** Every memory should include URL, title, author/publisher, retrieval time, provider, content hash, license/paywall status, and exact supporting span.
9. **Use hybrid retrieval.** Vector search alone is not enough; combine BM25/Tantivy, vectors/Qdrant, source filters, recency filters, and citation graph expansion.
10. **Revalidate stale knowledge.** Memories from news/social should expire quickly; scholarly and standards documents can have longer TTLs.

For docs-heavy sites, first check `/llms.txt` or `/llms-full.txt`; it is a proposed convention for giving LLMs a curated Markdown guide to a website. It is not universal, but it is increasingly useful for developer docs. ([llms-txt][12])

## Best tool picks by scenario

**Best privacy-first default:** Brave Search API + self-hosted SearXNG + Mojeek. Brave gives better coverage and an agent-friendly API; SearXNG gives self-hosted privacy control; Mojeek gives an independent no-tracking fallback. ([Brave][13])

**Best agent-native search pair:** Tavily + Exa. Tavily is excellent for production RAG/search/extract/crawl workflows and advertises security/privacy/content validation layers; Exa is excellent for semantic discovery, deep search, contents extraction, and AI-oriented indexes. ([Tavily][14])

**Best “one API across web + publications + proprietary-ish data” candidate:** Valyu. It is worth evaluating seriously because its docs describe web search plus PubMed, SEC filings, clinical trials, arXiv, patents, and financial data through one API. I would benchmark it before depending on it, because it is newer and more specialized than Brave/Tavily/Exa. ([Valyu][15])

**Best extraction/crawl layer:** Firecrawl for hosted/self-hosted agent scraping, Crawl4AI for open-source crawler workflows, Jina Reader for fast URL-to-markdown, and GROBID for academic PDFs. Firecrawl, Tavily, and Exa also have MCP servers, which is handy if your agents run through MCP-compatible clients. ([GitHub][16])

**Best publication architecture:** OpenAlex first, Crossref second, Semantic Scholar third, then domain APIs. For biomedical: PubMed/E-utilities and Europe PMC. For legal open access: Unpaywall/CORE. For PDFs: GROBID. PubMed’s E-utilities docs also note API-key requirements for higher request rates, so production agents should include rate-limit handling. ([OpenAlex Developers][17])

**Best social strategy:** use social as weak evidence and trend discovery, not as ground truth. GitHub is strong evidence for software artifacts; Reddit is useful for user reports; X is useful for real-time signals; LinkedIn is heavily restricted and should only be used through approved LinkedIn APIs and permissions. ([GitHub Docs][6])

## 1. Privacy-first web search broker: Brave + self-hosted SearXNG

```rust
use anyhow::{bail, Context, Result};
use reqwest::Client;
use serde::{Deserialize, Serialize};
use std::collections::HashSet;

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SearchResult {
    pub title: String,
    pub url: String,
    pub snippet: Option<String>,
    pub provider: String,
}

fn scrub_query(query: &str) -> Result<String> {
    let q = query.trim();

    // Extend this list with your org-specific secret patterns.
    let forbidden = [
        "sk-",
        "ghp_",
        "xoxb-",
        "-----BEGIN PRIVATE KEY-----",
        "Authorization:",
        "Bearer ",
    ];

    if forbidden.iter().any(|needle| q.contains(needle)) {
        bail!("Refusing to send a query that appears to contain a secret");
    }

    // Avoid accidentally sending huge private context blobs.
    Ok(q.chars().take(500).collect())
}

#[derive(Debug, Deserialize)]
struct BraveResponse {
    web: Option<BraveWeb>,
}

#[derive(Debug, Deserialize)]
struct BraveWeb {
    results: Vec<BraveHit>,
}

#[derive(Debug, Deserialize)]
struct BraveHit {
    title: String,
    url: String,
    description: Option<String>,
}

pub async fn brave_search(query: &str, api_key: &str) -> Result<Vec<SearchResult>> {
    let client = Client::new();

    let resp = client
        .get("https://api.search.brave.com/res/v1/web/search")
        .header("X-Subscription-Token", api_key)
        .header("Accept", "application/json")
        .query(&[
            ("q", query),
            ("count", "10"),
            ("safesearch", "strict"),
            ("text_decorations", "false"),
            ("spellcheck", "true"),
        ])
        .send()
        .await?
        .error_for_status()?
        .json::<BraveResponse>()
        .await?;

    Ok(resp
        .web
        .map(|w| {
            w.results
                .into_iter()
                .map(|r| SearchResult {
                    title: r.title,
                    url: r.url,
                    snippet: r.description,
                    provider: "brave".to_string(),
                })
                .collect()
        })
        .unwrap_or_default())
}

#[derive(Debug, Deserialize)]
struct SearxngResponse {
    results: Vec<SearxngHit>,
}

#[derive(Debug, Deserialize)]
struct SearxngHit {
    title: Option<String>,
    url: String,
    content: Option<String>,
}

pub async fn searxng_search(query: &str, base_url: &str) -> Result<Vec<SearchResult>> {
    let client = Client::new();
    let url = format!("{}/search", base_url.trim_end_matches('/'));

    let resp = client
        .get(url)
        .query(&[
            ("q", query),
            ("format", "json"),
            ("safesearch", "2"),
            ("language", "en-US"),
        ])
        .send()
        .await?
        .error_for_status()?
        .json::<SearxngResponse>()
        .await?;

    Ok(resp
        .results
        .into_iter()
        .map(|r| SearchResult {
            title: r.title.unwrap_or_else(|| r.url.clone()),
            url: r.url,
            snippet: r.content,
            provider: "searxng".to_string(),
        })
        .collect())
}

fn dedupe_by_url(results: Vec<SearchResult>) -> Vec<SearchResult> {
    let mut seen = HashSet::new();
    let mut out = Vec::new();

    for r in results {
        if seen.insert(r.url.clone()) {
            out.push(r);
        }
    }

    out
}

#[tokio::main]
async fn main() -> Result<()> {
    let query = scrub_query("best rust vector database for agent memory")?;
    let mut results = Vec::new();

    if let Ok(brave_key) = std::env::var("BRAVE_API_KEY") {
        results.extend(
            brave_search(&query, &brave_key)
                .await
                .context("Brave search failed")?,
        );
    }

    // Example: SEARXNG_URL="https://search.yourdomain.internal"
    // Ensure your SearXNG instance has JSON output enabled.
    if let Ok(searxng_url) = std::env::var("SEARXNG_URL") {
        results.extend(
            searxng_search(&query, &searxng_url)
                .await
                .context("SearXNG search failed")?,
        );
    }

    for r in dedupe_by_url(results).into_iter().take(10) {
        println!("[{}] {} — {}", r.provider, r.title, r.url);
    }

    Ok(())
}
```

## 2. Agent-native search calls: Tavily + Exa

```rust
use anyhow::Result;
use reqwest::Client;
use serde_json::{json, Value};

pub async fn tavily_agent_search(query: &str, api_key: &str) -> Result<Value> {
    let client = Client::new();

    let body = json!({
        "query": query,
        "search_depth": "advanced",
        "max_results": 8,
        "include_answer": false,
        "include_raw_content": "markdown",
        "topic": "general"
    });

    let resp = client
        .post("https://api.tavily.com/search")
        .bearer_auth(api_key)
        .json(&body)
        .send()
        .await?
        .error_for_status()?
        .json::<Value>()
        .await?;

    Ok(resp)
}

pub async fn exa_semantic_search(query: &str, api_key: &str) -> Result<Value> {
    let client = Client::new();

    let body = json!({
        "query": query,
        "type": "auto",
        "numResults": 8,
        "contents": {
            "text": true,
            "highlights": true
        }
    });

    let resp = client
        .post("https://api.exa.ai/search")
        .header("x-api-key", api_key)
        .header("Content-Type", "application/json")
        .json(&body)
        .send()
        .await?
        .error_for_status()?
        .json::<Value>()
        .await?;

    Ok(resp)
}
```

Use Tavily when you want dependable search/extract/crawl knobs. Use Exa when you want semantic discovery and “find the pages a keyword search would miss.”

## 4. Store searchable agent memory in Qdrant

```rust
use anyhow::Result;
use reqwest::Client;
use serde_json::{json, Value};

pub async fn create_qdrant_collection(
    qdrant_url: &str,
    collection: &str,
    vector_size: usize,
) -> Result<()> {
    let client = Client::new();

    let body = json!({
        "vectors": {
            "size": vector_size,
            "distance": "Cosine"
        }
    });

    client
        .put(format!(
            "{}/collections/{}",
            qdrant_url.trim_end_matches('/'),
            collection
        ))
        .json(&body)
        .send()
        .await?
        .error_for_status()?;

    Ok(())
}

pub async fn upsert_memory(
    qdrant_url: &str,
    collection: &str,
    id: u64,
    embedding: Vec<f32>,
    payload: Value,
) -> Result<()> {
    let client = Client::new();

    let body = json!({
        "points": [
            {
                "id": id,
                "vector": embedding,
                "payload": payload
            }
        ]
    });

    client
        .put(format!(
            "{}/collections/{}/points?wait=true",
            qdrant_url.trim_end_matches('/'),
            collection
        ))
        .json(&body)
        .send()
        .await?
        .error_for_status()?;

    Ok(())
}

#[tokio::main]
async fn main() -> Result<()> {
    // Match this to your embedding model dimension.
    create_qdrant_collection("http://localhost:6333", "agent_memory", 1536).await?;

    // Replace with your real embedding vector.
    let embedding = vec![0.0_f32; 1536];

    let payload = json!({
        "url": "https://example.com/article",
        "title": "Example Article",
        "provider": "brave",
        "retrieved_at": "2026-05-11T00:00:00Z",
        "content_hash": "sha256:...",
        "source_type": "web",
        "license": "unknown",
        "trusted_level": "unverified",
        "text": "Extracted passage or chunk goes here."
    });

    upsert_memory(
        "http://localhost:6333",
        "agent_memory",
        1,
        embedding,
        payload,
    )
    .await?;

    Ok(())
}
```

For production, pair this with a lexical index. My favorite Rust-native pairing is **Qdrant for vectors + Tantivy for BM25**, with Postgres storing canonical source records, crawl state, and provenance.

## Final build recommendation

For a Rust-friendly, safe, high-quality agent search system, I would ship:

**Core:** Brave Search API, self-hosted SearXNG, Mojeek, Tavily, Exa.
**Extraction:** Firecrawl or Crawl4AI, plus Jina Reader fallback, plus GROBID for scholarly PDFs.
**Publications:** OpenAlex, Crossref, Semantic Scholar, arXiv, PubMed/E-utilities, Europe PMC, Unpaywall, CORE.
**News/publications:** GDELT, Media Cloud, publisher RSS/sitemaps, Event Registry/NewsAPI.ai if budget allows.
**Social/dev:** GitHub API, Reddit API, X API, LinkedIn approved APIs only.
**Knowledge:** Qdrant + Tantivy + Postgres/pgvector, with provenance, TTLs, and trust scoring.
**Safety:** query redaction, source allow/block lists, robots/ToS compliance, untrusted-content quarantine, no autonomous login/form submission, and citations for every answer.
