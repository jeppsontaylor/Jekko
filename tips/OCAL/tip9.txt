# OCAL deep expansion: make it the agentic control plane, not another agent framework

My strongest conclusion: **OCAL is not redundant if it stays focused on host-enforced lifecycle, policy, evidence, memory, and promotion.** It becomes redundant only if it tries to become “LangGraph in YAML,” “OpenHands in YAML,” or “Claude/Codex/OpenCode config in YAML.” The winning move is bigger and cleaner:

**OCAL should become the portable run contract for autonomous coding work.**
Think: **Temporal/Kubernetes Jobs + CI/CD gates + policy-as-code + durable agent memory + multi-agent experiment orchestration**, expressed in a chat-pasteable, host-enforced language.

OpenCode already has configurable agents, permissions, plugins, snapshots, models, MCP, commands, and GitHub integration, so OCAL should not duplicate those one-for-one. OpenCode agents can be configured with prompts, models, and tool access; permissions determine whether actions run automatically, ask, or are blocked; snapshots already support undo/revert but can create indexing/disk costs in large repos; plugins hook into events; and GitHub comments can invoke OpenCode in Actions runners. ([OpenCode][1])

OCAL’s unique wedge is: **the model does not own the loop, the host does.** That is genuinely powerful. The missing pieces are mostly about making that host control plane richer, safer, more replayable, more portable, and more enterprise-scale.

I treated social/forum sources as weak signals and primary docs, papers, and open-source docs as stronger evidence. One OpenCode community issue is especially relevant: a reported subagent/task permission bypass showed that a restricted agent could delegate to a task/subagent with broader write permissions, which is exactly the kind of cross-agent escalation OCAL must make structurally impossible. ([GitHub][2])

---

# 1. The core thesis: OCAL should be the “agent runbook kernel”

OCAL should not merely say:

> “run forever until tests pass.”

It should say:

> “Here is the complete, typed, replayable, policy-checked operating envelope for autonomous software work: what agents may do, what evidence they must produce, how memory is written, how experiments are compared, how permissions propagate, how risks are scored, how humans approve irreversible steps, how state is replayed, and how work is promoted.”

That gives OCAL a sharp role above OpenCode, Codex, Claude Code, OpenHands, LangGraph, Hermes, OpenClaw, AutoGen, CrewAI, and deterministic CI tools.

A good mental model:

| Layer                 | Examples                                             | OCAL role                                        |
| --------------------- | ---------------------------------------------------- | ------------------------------------------------ |
| Coding agent          | OpenCode, Codex CLI, Claude Code, OpenHands          | OCAL invokes, constrains, and observes them      |
| Agent runtime / graph | LangGraph, Microsoft Agent Framework, AutoGen        | OCAL can compile to or interoperate with these   |
| Skills/tools          | MCP, shell, browser, repo tools, LSP, test runners   | OCAL defines capability contracts and grants     |
| Policy / safety       | permissions, sandbox, secrets, approvals, guardrails | OCAL makes safety transitive and auditable       |
| State / memory        | SQLite, event logs, vector stores, repo graphs       | OCAL defines durable memory lanes and provenance |
| Promotion             | commit, PR, merge, release, rollout                  | OCAL requires host evidence before promotion     |

The most important design principle I would add:

> **OCAL is a declarative contract over autonomous work, not a prompt. The host compiles OCAL into tool permissions, workflow topology, memory policy, sandbox policy, approval packets, and evidence gates.**

---

# 2. What OCAL already gets right

OCAL already has several excellent ideas:

1. **Sentinel-based preview/arming.** This is a big deal. It makes “show me the daemon” separate from “start the daemon.”

2. **Host-owned lifecycle.** The model does not decide when it is done. Stop conditions, budgets, circuit breakers, and promotion are host-evaluated.

3. **SQLite as truth.** Durable state is essential for multi-hour or multi-day agentic work.

4. **Guardrails, constraints, permissions, and checkpoints.** These are much better than asking the model to “be careful.”

5. **Incubator passes.** This is one of the strongest parts of OCAL. Hard tasks should not go directly from vague prompt to production patch.

6. **Scratch-only and isolated worktree write scopes.** This is foundational for safe experimentation.

7. **On-handlers with max-depth behavior.** Avoiding recursive handler loops is a mature systems decision.

The biggest opportunity is to push these ideas from “good v1 daemon YAML” into a **full autonomy control protocol**.

# 3. Where OCAL overlaps with existing systems — and how to avoid redundancy

## OpenHands

OpenHands is very close to the problem space: it has a software-agent SDK, REST APIs, sandboxed execution, event-sourced state, typed tools, MCP support, context condensation, lifecycle control, multi-LLM routing, security analysis, remote execution, pause/resume, and stuck detection. Its paper emphasizes production coding agents needing flexible agent design, reliable/secure execution, and human interaction; it also describes immutable config, event-sourced state with deterministic replay, sandboxing, REST/WebSocket interfaces, multi-LLM routing, security analysis, secrets handling, and stuck detection. ([arXiv][3])

**OCAL should borrow:**

* Event-sourced replay semantics.
* Typed tool calls.
* Security risk scoring per action.
* Remote/sandbox execution model.
* Stuck detection.
* Multi-LLM routing.
* Secrets handling and masking.

**OCAL should not try to become OpenHands.** Instead, OCAL should be able to declare:

```yaml
adapters:
  openhands:
    mode: agent_server
    endpoint: "${OPENHANDS_URL}"
    capabilities:
      require: [typed_tools, sandbox, event_stream, pause_resume]
```

OCAL becomes the policy/runbook layer. OpenHands can be one executor.

## LangChain / LangGraph

LangGraph is a serious durable-execution runtime for long-running, stateful agents. Its docs emphasize durable execution, checkpointers, thread identifiers, interruption/resume, replay semantics, and wrapping side effects so they are not repeated during replay. LangChain’s HITL middleware can pause around tool calls and support approve/edit/reject/respond decisions with persisted graph state. LangGraph memory also distinguishes short-term thread memory from long-term namespace-scoped memory, and it draws on semantic, episodic, and procedural memory categories. ([LangChain Docs][4])

**OCAL should borrow:**

* Explicit replay semantics.
* Idempotent side-effect wrappers.
* Thread/run/task identifiers.
* Graph nodes and edges.
* HITL decision packets.
* Memory namespaces.

**OCAL should not become only a graph runtime.** It should describe the run contract and optionally compile to LangGraph.

## Claude Code / Claude Agent SDK

Claude Code and the Agent SDK have a mature set of ideas: hooks configured at lifecycle points, scoped subagents with tool/model/permission settings, fine-grained allow/ask/deny permissions, checkpointing for file edits, OpenTelemetry tracing, todo tracking, and skills. Claude’s file checkpointing supports undo/recovery for built-in file modification tools, while noting limitations such as Bash changes not being tracked. ([Claude][5])

**OCAL should borrow:**

* Hook scopes.
* Subagent scopes.
* Skill metadata and packaging.
* Checkpoint/rewind semantics.
* OTel traces.
* Structured todos as task state.

**Key OCAL upgrade:** permissions must be **transitive**, not merely local to one agent.

## Hermes Agent

Hermes is interesting because it focuses on persistent memory, skills, cron, messaging gateways, subagents, delegation, terminal backends, trajectory generation, skill learning, and orchestration of tools like OpenCode/Codex. Its README describes built-in learning loops, skill creation from experience, memory nudges, past-conversation search, multiple providers, cron, delegation, Docker/SSH/Singularity/Modal/Daytona/Vercel terminal backends, trajectory compression, and RL-oriented workflows. Hermes also has skills for orchestrating OpenCode and Codex as long-running background coding workers. ([GitHub][6])

**OCAL should borrow:**

* First-class skill lifecycle.
* Skill improvement from successful runs.
* Background process attach/poll/kill.
* Terminal backend abstraction.
* Trajectory compression.
* Memory providers.

**OCAL must be stricter than Hermes:** self-improving skills must not auto-escalate. A skill learned from a run should enter a quarantined state, be scanned, versioned, permission-bounded, and promoted only after host evidence.

## OpenClaw

OpenClaw is more of an always-on assistant/gateway system than a coding daemon, but it is valuable for security thinking. It emphasizes local-first gateway behavior, multi-channel inboxes, pairing/unknown-sender controls, security audit tooling, trusted-operator boundaries, sandboxed tool execution, sandbox modes/scopes/backends, workspace read/write controls, network controls, and context-visibility risks. ([GitHub][7])

**OCAL should borrow:**

* Trust boundaries.
* Inbound message authorization.
* Context visibility controls.
* Security doctor/audit command.
* Sandbox backend matrix.
* “Who can steer this run?” policy.

This matters because you want OCAL blocks to be available inside **any chat prompt**. That means OCAL must distinguish:

* Trusted host config.
* Trusted human arming action.
* Untrusted issue text.
* Untrusted web content.
* Untrusted tool output.
* Untrusted model output.

Without that, prompt injection becomes a control-plane attack.

## AutoGen, Magentic-One, Microsoft Agent Framework, CrewAI

AutoGen has explicit termination conditions, state save/load, and human-in-the-loop patterns. Magentic-One uses an orchestrator that creates plans, delegates to specialized agents, tracks progress, and revises plans. Microsoft Agent Framework combines agents and graph workflows with checkpointing and HITL. CrewAI supports hierarchical manager agents, delegation, validation, event-driven flows, state, branching, and loops. ([Microsoft GitHub][8])

**OCAL should borrow:**

* Declarative graph orchestration.
* Manager/worker/critic roles.
* Dynamic re-planning.
* Stateful workflows.
* Task validation by manager agents.

**But OCAL should preserve host authority.** Manager agents can propose, not promote.

## SWE-agent, MASAI, Agentless

SWE-agent argues that the agent-computer interface matters: purpose-built interfaces for navigating, editing, searching, and testing software improve outcomes. MASAI decomposes software-engineering tasks into specialized subagents with separate objectives and strategies, reducing long trajectories and letting different strategies gather different repository information. Agentless is an important counterpoint: simple localization/repair/validation pipelines can outperform complex open-ended agents on some software tasks. ([arXiv][9])

**OCAL should borrow:**

* Specialized coding interfaces, not raw shell everywhere.
* Agentless deterministic lanes.
* Localization before editing.
* Patch candidates with validation.
* Avoiding long, meandering trajectories.

This is central to reducing “vibe coding.”

# 4. The biggest missing idea: transitive capability enforcement

OCAL’s current `permissions` section is too flat.

Today you have:

```yaml
permissions:
  shell: ask
  edit: allow
  git_commit: ask
  git_push: deny
  workers: allow
  mcp: ask
```

That is not enough for ambitious autonomous work. The OpenCode permission-escalation issue shows why: a restricted agent may still delegate to a subagent/task that has broader capabilities unless the host enforces permission inheritance across the call graph. ([GitHub][2])

OCAL needs **capability algebra**:

```yaml
capabilities:
  inheritance: monotonic_decrease   # child can never exceed parent
  default: deny_unlisted
  delegation:
    max_depth: 2
    require_parent_capabilities: true
    child_cannot_exceed_parent: true
    deny_if_ambiguous: true

  grants:
    plan:
      tools:
        read_file: allow
        search: allow
        edit: deny
        shell: ask
        task_delegate: allow
      paths:
        "src/**": read
        "docs/**": read
        "**": none

    build:
      tools:
        read_file: allow
        edit: allow
        apply_patch: allow
        shell: ask
      paths:
        "src/**": write
        "tests/**": write
        "package.json": ask
        ".github/**": ask
        "**/.env*": deny

    critic:
      tools:
        read_file: allow
        shell: ask
        edit: deny
      paths:
        "**": read
```

Add **proof-carrying tool calls**:

```yaml
tool_authorization:
  require_proof_packet: true
  packet_fields:
    - requesting_agent
    - parent_agent
    - effective_capabilities
    - tool
    - path
    - reason
    - risk_score
    - approval_id
  deny_when:
    - effective_capability_missing
    - child_exceeds_parent
    - path_scope_violation
    - approval_expired
```

This is non-negotiable for OCAL v2. Without transitive permissions, multi-agent autonomy becomes a privilege-escalation system.

# 5. Durable execution needs to become explicit, replayable, and idempotent

OCAL says SQLite is truth, but it does not yet define enough replay semantics.

LangGraph’s durable execution docs make a crucial point: durable systems need persistence/checkpointers, stable thread identifiers, and side effects wrapped so replay does not accidentally repeat non-deterministic or irreversible operations. ([LangChain Docs][4])

OCAL should add:

```yaml
durability:
  state_store:
    type: sqlite
    path: ".opencode/daemon/state.sqlite"
  event_sourcing:
    enabled: true
    append_only: true
    hash_chain: true
    include_tool_io_hashes: true
  replay:
    mode: deterministic
    side_effects:
      shell: record_and_replay
      file_write: replay_from_snapshot
      git_commit: never_repeat_without_same_event_id
      network: require_idempotency_key
    nondeterminism:
      llm_calls: record_model_request_response_hash
      time: freeze_per_iteration
      random: seed_per_run
  snapshots:
    before_tool_write: true
    before_checkpoint: true
    before_promotion: true
  resume:
    after_crash: from_last_committed_event
    require_state_integrity: true
```

OCAL should distinguish these states:

```yaml
state_machine:
  run:
    states:
      - previewed
      - armed
      - starting
      - running
      - pausing
      - paused
      - resuming
      - stopping
      - stopped
      - aborted
      - failed
  task:
    states:
      - discovered
      - scoped
      - incubating
      - ready
      - building
      - verifying
      - checkpointed
      - blocked
      - promoted
      - archived
  pass:
    states:
      - queued
      - running
      - completed
      - failed
      - exhausted
      - superseded
```

For 1B+ LOC projects, reproducibility is not cosmetic. You need to answer:

* Why did this agent edit this file?
* Which evidence justified promotion?
* Could the same run be replayed?
* Did a crash duplicate a side effect?
* Did a subagent inherit or escalate permissions?
* Which model/tool result influenced the final patch?

# 6. Memory should be a first-class OCAL object, not just context compaction

Current OCAL has `context`, but not a complete memory system.

Long-running agents need several memory lanes:

1. **Run memory** — what happened in this daemon run.
2. **Task memory** — decisions, failed attempts, hypotheses, constraints.
3. **Repo memory** — architecture, APIs, module boundaries, ownership.
4. **Org memory** — coding standards, deployment policies, compliance.
5. **Procedural memory** — reusable skills and playbooks.
6. **Episodic memory** — examples of past tasks and successful fixes.
7. **Semantic memory** — stable facts about the codebase and domain.

LangGraph’s memory docs distinguish short-term thread-scoped memory from long-term namespace-scoped memory, and semantic/episodic/procedural memory categories are a useful fit for OCAL. They also note that memory writes can happen on the hot path or in the background, each with tradeoffs. ([LangChain Docs][10])

OCAL should add:

```yaml
memory:
  enabled: true

  stores:
    run:
      type: sqlite
      scope: run
      retention: 90d
    task:
      type: sqlite
      scope: task
      retention: 365d
    repo_semantic:
      type: vector
      scope: repo
      embedding_model: small
      retention: persistent
    repo_graph:
      type: symbol_graph
      scope: repo
      source: indexer
    procedural:
      type: skill_registry
      scope: org
      retention: persistent

  lanes:
    semantic:
      writes:
        allowed_from: [synthesize, promotion_review, postmortem]
        require_provenance: true
        require_evidence: true
        approval: ask_for_org_scope
      retrieval:
        max_items: 12
        freshness_weight: 0.3
        confidence_threshold: 0.65

    episodic:
      writes:
        allowed_from: [after_checkpoint, on_exhaust, postmortem]
        include_failed_attempts: true
        summarize_only: true
      retrieval:
        similar_task_top_k: 5

    procedural:
      writes:
        allowed_from: [after_successful_run]
        status: quarantined
        require_security_scan: true
        require_human_promotion: true

  compaction:
    strategy: evidence_preserving
    never_drop:
      - requirements
      - constraints
      - failed_attempts
      - approval_decisions
      - changed_files
      - test_results
      - unresolved_risks

  forgetting:
    decay:
      stale_after: 180d
      lower_confidence_by: 0.2
    delete_when:
      - superseded_by_newer_architecture_decision
      - contains_secret
      - manually_revoked

  provenance:
    required: true
    fields:
      - source_event_id
      - source_file
      - source_commit
      - author_agent
      - confidence
      - evidence_hash
```

This lets OCAL avoid two bad extremes:

* **No memory:** every daemon repeats the same mistakes.
* **Uncontrolled memory:** stale hallucinated “facts” become permanent system behavior.

Memory should be **evidence-bound, scoped, decayed, revocable, and queryable**.

# 7. Add a repo intelligence graph for billion-line projects

For 1B+ LOC, a model cannot “read the repo.” It needs a host-owned index.

OCAL should add a `repo` section that declares the indexing and routing substrate:

```yaml
repo:
  scale: billion_loc

  index:
    symbol_graph:
      enabled: true
      languages: [typescript, go, python, java, rust, cpp]
      update: incremental
      include:
        - definitions
        - references
        - call_graph
        - inheritance
        - imports
        - exports
        - public_api
    dependency_graph:
      enabled: true
      package_managers: [npm, pnpm, cargo, go, maven, gradle, bazel]
    test_graph:
      enabled: true
      map_tests_to_code: true
      flaky_test_history: true
    ownership:
      sources:
        - CODEOWNERS
        - service_catalog
        - team_metadata
    architecture:
      adr_paths: ["docs/adr/**", "architecture/**"]
      forbidden_edges:
        - from: "payments/**"
          to: "experimental/**"

  retrieval:
    strategy: scoped_graph_then_semantic
    max_context_files: 40
    max_tokens_per_iteration: 120000
    require_localization_before_edit: true
    localization:
      min_confidence: 0.65
      fallback: incubate

  impact_analysis:
    before_edit: true
    compute:
      - affected_packages
      - affected_tests
      - affected_public_apis
      - migration_risk
      - owners
```

For massive projects, OCAL should make **localization a gate**:

```yaml
quality:
  localization_gate:
    required_before:
      - edit
      - shell_mutating_command
    evidence:
      - affected_files_known_gte: 1
      - relevant_symbols_identified_gte: 1
      - tests_mapped_gte: 1
    on_fail: incubate_current_task
```

This is one of the strongest anti-jank controls: **do not edit until the daemon can explain the affected files, symbols, tests, and ownership boundary.**

# 8. Replace “vibe coding” with evidence-driven patch contracts

The problem you called “vibe coding / jankurai” is real: agents make plausible edits without a tight hypothesis, reproduction, test, or rollback story.

OCAL needs an explicit **patch contract**.

```yaml
patch_contract:
  required: true

  before_edit:
    require:
      - problem_statement
      - hypothesis
      - affected_files
      - affected_symbols
      - test_strategy
      - rollback_plan
    block_if:
      - no_reproduction_path
      - affected_files_unknown
      - public_api_change_unreviewed

  during_edit:
    max_diff:
      files: 20
      lines: 800
      allow_exceed_with: approval
    forbidden:
      - delete_tests_without_approval
      - skip_tests_without_approval
      - broad_rewrite_without_design_review
      - dependency_upgrade_without_reason
    require_comments_for:
      - generated_code
      - compatibility_shims
      - migration_scripts

  after_edit:
    require:
      - tests_run
      - static_analysis_run
      - diff_summary
      - risk_summary
      - rollback_known
      - no_unexplained_deletions

  promotion:
    require_evidence_bundle: true
```

Add “claim requires evidence”:

```yaml
evidence:
  claim_policy: require_support
  claims:
    "tests pass":
      require:
        - command_result.exit_code: 0
        - command_result.timestamp
    "bug fixed":
      require:
        - reproduction_test_fails_before: true
        - reproduction_test_passes_after: true
    "no API break":
      require:
        - api_diff: clean
    "safe migration":
      require:
        - migration_dry_run: pass
        - rollback_plan: present
```

This shifts OCAL from “the agent says it fixed it” to “the host verified evidence.”

# 9. Upgrade the incubator into a full experiment engine

The existing incubator is excellent, but it should become a **portfolio of bounded experiments**.

Hard tasks often need disjoint attempts:

* A minimal patch attempt.
* A refactor attempt.
* A compatibility shim attempt.
* A dependency/config attempt.
* A test-only localization attempt.
* A “delete nothing, change one file” attempt.
* A “new abstraction” attempt.

OCAL should support this directly:

```yaml
experiments:
  enabled: true
  strategy: disjoint_worktrees
  objective: maximize_verified_progress
  max_parallel: 6
  max_rounds: 4

  lanes:
    - id: minimal_patch
      hypothesis: "Smallest code change can fix the failing behavior."
      agent: build
      isolation: git_worktree
      diff_budget: { files: 5, lines: 200 }

    - id: test_first
      hypothesis: "A failing reproduction test will localize the bug."
      agent: build
      isolation: git_worktree
      writes:
        allow: ["tests/**"]
      promotion: evidence_only

    - id: refactor_boundary
      hypothesis: "The failure is caused by unclear boundary logic."
      agent: build
      isolation: git_worktree
      diff_budget: { files: 12, lines: 700 }
      require_design_review: true

    - id: agentless_patch
      strategy: localize_repair_validate
      agent: none
      tools: [search, static_analysis, patch_generator]

  scoring:
    metrics:
      - tests_pass
      - reproduction_fixed
      - diff_size_inverse
      - risk_score_inverse
      - static_analysis_clean
      - reviewer_score
      - affected_api_count_inverse
    tie_breaker: smallest_safe_diff

  kill_rules:
    - no_new_evidence_after: 2 rounds
    - repeated_same_patch_gte: 2
    - diff_budget_exceeded: pause
    - violates_constraint: abort_lane

  reduce:
    strategy: best_verified_patch
    allow_patch_composition: ask
    require_human_for_merge_conflict: true
```

This is a major leap beyond “try again.” The daemon should **compete hypotheses**, not just repeat attempts.

# 10. Add stateful patch stacks and rewinds

Claude’s checkpointing docs are a useful reminder: undo/recovery should be explicit, but tool-specific; file checkpoints may not cover Bash side effects. ([Claude][11])

OCAL should model patch stacks:

```yaml
patch_stack:
  enabled: true
  base: origin/main

  snapshots:
    before_each_lane: true
    before_each_mutating_tool: true
    include:
      - file_tree
      - git_index
      - dependency_lockfiles
      - generated_artifacts_manifest
    exclude:
      - node_modules/**
      - dist/**
      - .cache/**

  rewind:
    allow:
      - lane
      - iteration
      - last_tool_call
    deny:
      - after_pushed_commit
      - after_external_side_effect_without_rollback

  side_effects:
    package_install:
      track_lockfile: true
      sandbox_required: true
    database_migration:
      require_dry_run: true
      require_rollback: true
      sandbox_required: true
    network_call:
      require_idempotency_key: true
      record_request_hash: true
```

This is crucial for “stateful building on current state.” You want the daemon to learn from failed attempts without permanently polluting the workspace.

# 11. Add explicit graph orchestration, not only `fan_out`

Current OCAL has `agents` and `fan_out`, but ambitious projects need task DAGs, conditional routing, and multi-pass state machines.

Add:

```yaml
graph:
  mode: host_orchestrated
  default_timeout: 30m

  nodes:
    localize:
      type: deterministic
      tool: repo_localizer
      outputs: [affected_files, affected_symbols, candidate_tests]

    plan:
      type: agent
      agent: plan
      inputs: [localize.outputs]
      outputs: [patch_plan, risk_register]

    red_team_plan:
      type: agent
      agent: critic
      inputs: [plan.outputs]
      outputs: [objections]

    implement:
      type: agent_pool
      agent: build
      count: 3
      isolation: git_worktree
      inputs: [plan.outputs, red_team_plan.outputs]
      outputs: [candidate_patch]

    verify:
      type: deterministic
      commands:
        - "bun test --timeout 30000"
        - "npx tsgo --noEmit"
      outputs: [test_report, type_report]

    review:
      type: agent
      agent: critic
      inputs: [candidate_patch, verify.outputs]
      outputs: [review_score, unresolved_objections]

    promote:
      type: host_gate
      require:
        - verify.test_report.exit_code == 0
        - review.unresolved_objections == 0
        - evidence.reproduction_fixed == true

  edges:
    - from: localize
      to: plan
    - from: plan
      to: red_team_plan
    - from: red_team_plan
      to: implement
    - from: implement
      to: verify
    - from: verify
      to: review
    - from: review
      to: promote

  routing:
    - if: "localize.confidence < 0.65"
      to: incubator
    - if: "review.unresolved_critical_objections > 0"
      to: implement
    - if: "verify.failed_tests_changed == false"
      to: experiment_lane:minimal_patch
```

This lets OCAL express Magentic-One/CrewAI/AutoGen-style orchestration while preserving host-owned gates.

# 12. Add HITL approval packets, not just `ask`

Current `ask` is underspecified. LangChain’s HITL pattern supports decisions like approve, edit, reject, or respond, and persists graph state while waiting. ([LangChain Docs][12])

OCAL should make approvals typed:

```yaml
approvals:
  persistence: sqlite
  require_signed_decisions: true
  default_expiry: 24h

  reviewers:
    security:
      match:
        - paths: ["auth/**", "crypto/**", "payments/**"]
        - risk_score_gte: 0.7
    db:
      match:
        - paths: ["migrations/**", "schema/**"]
    owner:
      source: CODEOWNERS

  decisions:
    allowed:
      - approve
      - approve_once
      - approve_for_scope
      - edit_request
      - reject
      - escalate
      - pause_run
      - abort_run

  packets:
    include:
      - requested_action
      - effective_capabilities
      - parent_agent
```

```yaml
      - user_requirements
      - non_goals
      - affected_files
      - affected_symbols
      - constraints
      - hypotheses
      - attempts
      - failed_attempts
      - evidence
      - unresolved_questions
      - rollback_plan
      - promotion_status

  update_policy:
    after_each_iteration: true
    after_failed_attempt: true
    after_checkpoint: true
    after_human_feedback: true

  compression:
    preserve:
      - failed_attempts
      - exact_error_signatures
      - commands_run
      - changed_files
      - approval_decisions
      - reasoning_summary
    drop:
      - verbose logs after hashing
      - duplicate tool output
```

This preserves useful reasoning summaries and evidence without storing hidden chain-of-thought.

# 36. Add learning loops, but only from verified outcomes

Research systems like Reflexion, Self-Refine, Voyager, MemoryBank, and CoALA all point toward memory, reflection, skills, and structured agent architectures as important for long-horizon agents. Reflexion stores verbal reflections in episodic memory; Self-Refine iteratively critiques/refines outputs; Voyager builds a skill library through lifelong exploration; MemoryBank explores long-term memory and updating; CoALA frames language agents around memory, actions, and decision-making. ([arXiv][18])

OCAL should support learning, but never from unverified guesses:

```yaml
learning:
  enabled: true
  source:
    only_from:
      - successful_checkpoint
      - promoted_task
      - human_reviewed_postmortem
    exclude:
      - failed_unverified_attempt
      - hallucinated_fact
      - secret_adjacent_context

  reflections:
    write_to: episodic_memory
    require:
      - what_worked
      - what_failed
      - reusable_pattern
      - evidence
      - scope_limitations

  skill_generation:
    enabled: true
    from:
      - repeated_successful_pattern_gte: 3
    initial_status: quarantined
    require:
      - security_scan
      - deterministic_test
      - human_approval

  postmortem:
    on_exhaust: true
    on_abort: true
    on_revert: true
    include:
      - root_cause
      - failed_hypotheses
      - missing_tools
      - missing_context
      - suggested_ocal_upgrade
```

This makes OCAL self-improving without becoming self-authorizing.

# 45. Concrete “OCAL v2” skeleton

This is not an armable block; it is a proposed shape.

```yaml
version: v2
intent: daemon
confirm: RUN_FOREVER

job:
  name: Enterprise autonomous migration
  objective: Migrate all packages from legacy auth middleware to AuthV3 without breaking public APIs.
  risk:
    - Auth changes are security-sensitive
    - Migration spans many packages
    - Tests may be flaky
    - Public API compatibility must be preserved

schema:
  unknown_keys: reject
  min_host_version: ">=1.0.0"

autonomy:
  level: guarded_autonomous

durability:
  event_sourcing:
    enabled: true
    append_only: true
    hash_chain: true
  replay:
    mode: deterministic
    side_effects:
      shell: record_and_replay
      git_commit: never_repeat_without_same_event_id
      network: require_idempotency_key

capabilities:
  inheritance: monotonic_decrease
  default: deny_unlisted
  delegation:
    child_cannot_exceed_parent: true
    require_parent_capabilities: true

workspace:
  isolation:
    default: container

      experiment_lane: git_worktree
      prototype_pass: git_worktree
  filesystem:
    deny_paths: ["**/.env*", "~/.ssh/**", "/var/run/docker.sock"]

    ask_write: ["package.json", "pnpm-lock.yaml", ".github/**"]

network:
  default: deny

    - host: "registry.npmjs.org"

secrets:
  registry:
    source: host

repo:
  index:
    symbol_graph: { enabled: true, update: incremental }
    dependency_graph: { enabled: true }
    test_graph: { enabled: true, map_tests_to_code: true }

      sources: [CODEOWNERS, service_catalog]
  retrieval:
    strategy: scoped_graph_then_semantic
    require_localization_before_edit: true

memory:
  enabled: true
  lanes:
    semantic:
      writes:
        allowed_from: [synthesize, promotion_review]
        require_provenance: true
    episodic:
      writes:
        allowed_from: [after_checkpoint, on_exhaust]
        include_failed_attempts: true
    procedural:
      writes:
        allowed_from: [after_successful_run]

        require_human_promotion: true

  mode: host_orchestrated

    localize:
      type: deterministic
      tool: repo_localizer

      agent: plan

      type: agent_pool

      type: agent
      agent: critic

      type: host_gate
  edges:
    - { from: localize, to: plan }
    - { from: plan, to: implement }
    - { from: implement, to: verify }
    - { from: verify, to: review }
    - { from: review, to: promote }

experiments:
  enabled: true
  strategy: disjoint_worktrees
  max_parallel: 6
  lanes:
    - id: minimal_patch
      diff_budget: { files: 5, lines: 200 }
    - id: compatibility_shim
      diff_budget: { files: 12, lines: 500 }
    - id: codemod
      strategy: deterministic
  reduce:
    strategy: best_verified_patch

patch_contract:
  required: true
  before_edit:
    require:
      - problem_statement
      - hypothesis
      - affected_files
      - affected_symbols
      - test_strategy
      - rollback_plan
  during_edit:
    forbidden:
      - delete_tests_without_approval
      - skip_tests_without_approval
      - broad_rewrite_without_design_review
  after_edit:
    require:
      - tests_run
      - static_analysis_run
      - risk_summary
      - rollback_known

evidence_bundle:
  required_for: [checkpoint, pr_create, promote]
  include:
    - objective
    - requirements
    - affected_files
    - affected_symbols
    - diff_summary
    - tests_before
    - tests_after
    - static_analysis
    - security_scan
    - rollback_plan

  require_signed_decisions: true

    security:

        - risk_score_gte: 0.7
  decisions:
    allowed: [approve, approve_once, edit_request, reject, escalate, pause_run, abort_run]

observability:
  traces:
    enabled: true

      - tokens_total

    - id: draft_pr
      type: pull_request
      require: [owner_review_requested]
```

# 50. The “killer features” that would make OCAL the default language

## 1. Transitive permission calculus

No agent, subagent, tool, MCP server, or skill can exceed the effective capability of its parent call chain.

## 2. Evidence-gated promotion

No score, model confidence, or agent self-report can promote code. Promotion requires host-verified evidence.

## 3. Durable replay

Every run is event-sourced, resumable, and replay-aware. Side effects are idempotent or require approval.

## 4. Memory with provenance

Memory is not “whatever the model remembers.” It is scoped, evidence-bound, decayed, revocable, and audited.

## 5. Experiment portfolios

Hard tasks spawn bounded, disjoint experiments that compete on evidence.

## 6. Repo graph for massive codebases

Symbol graph + dependency graph + test graph + ownership graph become the host-owned map.

## 7. Anti-jank patch contracts

Every edit must have a hypothesis, scope, test strategy, and rollback path.

## 8. Universal adapter model

OCAL can run through OpenCode, Codex, Claude Code, OpenHands, or LangGraph depending on host capability.

## 9. Policy-as-code integration

Enterprises can enforce their own governance without forking OCAL.

## 10. Security doctor

Before arming, OCAL explains risk, effective permissions, escalation paths, and missing host capabilities.

---

# Final recommendation

Do **not** position OCAL as “a YAML language for long-running agents.” That sounds small and overlaps too much.

Position it as:

> **OCAL is the host-enforced autonomy contract for software engineering agents. It defines lifecycle, permissions, memory, evidence, experiments, state, and promotion for autonomous coding work across OpenCode, Codex, Claude, OpenHands, and other agent runtimes.**

The current spec is a strong v1. The highest-impact upgrades are:

1. **Transitive permissions and delegation safety.**
2. **Durable event-sourced replay with idempotent side effects.**
3. **First-class memory lanes with provenance and decay.**
4. **Repo intelligence graph for huge codebases.**
5. **Experiment lanes and patch candidate scoring.**
6. **Anti-vibe patch contracts and evidence bundles.**
7. **Typed HITL approval packets.**
8. **Sandbox/network/secrets/taint policies.**
9. **Graph orchestration beyond fan-out.**
10. **Universal host adapters and capability negotiation.**

That combination would make OCAL much more than another agent config. It would be the missing control plane for ambitious autonomous software engineering.
