# OCAL deep expansion: from daemon runbook to **agent operating contract**

OCAL is **not redundant**. It is aimed at the most important missing layer in agentic coding: a host-enforced, portable contract that can be embedded in a chat prompt while still giving the host—not the model—authority over runtime, tools, writes, budgets, approvals, and evidence.

The strongest conclusion from studying OpenHands, LangGraph/LangChain, Hermes Agent, OpenClaw, Claude Code, Codex, MCP, Skills, A2A, durable-execution systems, and long-horizon coding benchmarks is this:

**OCAL should not try to become “another agent framework.” It should become the declarative control plane that can orchestrate any agent framework.**

That means OCAL’s winning niche is:

> **A portable, host-enforced delegation contract for long-running coding work, independent of which model, chat UI, coding agent, runtime, or worker framework is underneath.**

The current OCAL v1 already has a great core: sentinels, preview-before-arm, SQLite truth, bounded loops, checkpoints, permissions, guardrails, constraints, fan-out, incubator passes, and promotion gates. But for 1B+ LOC projects, it needs several major upgrades: stronger authority modeling, durable graph workflows, memory promotion, code intelligence, experiment tournaments, supply-chain trust, observability, fleet scheduling, explicit sandboxes, and cross-agent interoperability.

I could not literally exhaust every X, LinkedIn, Quora, Reddit, blog, repo, and paper in existence, but I weighted primary docs, repos, papers, and representative public discussions over noisy social commentary. That is the right evidence hierarchy for a language spec.

---

## 1. What other systems teach us

### OpenHands

OpenHands is closest to OCAL’s intended environment. It is an open-source coding-agent platform and SDK with sandboxed runtimes, evaluation harnesses, and composable agents. Its Software Agent SDK is explicitly aimed at one-off coding tasks, routine maintenance, major multi-agent refactors, ephemeral Docker/Kubernetes workspaces, and scaling to many agents. ([GitHub][1])

**What OCAL should borrow:**

* Strong runtime abstraction: Docker, Kubernetes, local, remote ephemeral workspaces.
* Explicit agent-controller-runtime separation.
* Evaluation harnesses as first-class citizens, not optional extras.
* Support for thousands of workers, but governed by host policy.

**Where OCAL is stronger:**

OpenHands provides runtime and SDK machinery. OCAL can provide the **portable governance contract** that says what a run is allowed to do, when it must stop, what evidence is required, and what gets persisted.

### LangChain / LangGraph

LangGraph’s strongest contribution is durable graph execution: checkpoints, state persistence, human-in-the-loop inspection, interruption, approval, and resumption. Its docs emphasize durable execution, short- and long-term memory, and human-in-loop workflows. ([LangChain Docs][2])

**What OCAL should borrow:**

* Graph-shaped workflows, not only loops.
* Checkpointed node state.
* Replay/time-travel debugging.
* Human approval as a state transition.
* Interrupt/resume semantics at node boundaries.

**Where OCAL is stronger:**

LangGraph is programmatic. OCAL can be **operator-facing YAML** that compiles to LangGraph, Temporal, Inngest, OpenHands, Codex, Claude Code, or native OpenCode execution.

### Hermes Agent

Hermes Agent is important because it focuses on a long-lived autonomous agent with persistent memory, automatic skill creation, scheduled automations, server deployment, delegation, and tool use. Its docs describe a self-improving agent that creates skills from experience, persists knowledge, searches past conversations, and runs on servers while accessible through messaging surfaces. ([GitHub][3])

Hermes also has an interesting `execute_code` model where the agent can write Python scripts that call Hermes tools programmatically, reducing multi-step workflows into fewer model turns. ([Hermes Agent][4])

**What OCAL should borrow:**

* Memory promotion from experience.
* Skill creation as a first-class lifecycle.
* Long-lived delegation surfaces.
* Worker delegation across tools like OpenCode, Claude Code, and Codex. ([GitHub][5])
* Programmatic tool composition, but host-gated.

**Where OCAL must be stricter:**

Self-improvement is powerful but dangerous. OCAL should support self-improving skills only through **host-reviewed skill promotion**, signed manifests, test evidence, rollback plans, and trust tiers.

### OpenClaw

OpenClaw is useful because it is building a personal assistant gateway: messaging surfaces, memory, cron, skills, standing orders, delegates, and plugin architecture. Its Gateway owns messaging surfaces such as WhatsApp, Telegram, Slack, Discord, Signal, iMessage, and WebChat, while clients connect through WebSocket and nodes declare roles/capabilities. ([OpenClaw][6])

OpenClaw also has host-owned cron, memory promotion, skills, standing orders, and delegate architecture. ([OpenClaw][7])

**What OCAL should borrow:**

* Standing orders.
* Gateway/control-plane concepts.
* Cron and scheduled autonomy.
* Memory workspaces.
* Skill discovery with filters.
* Delegate capability declarations.

**Where OCAL is different:**

OpenClaw is a broad personal-assistant/control gateway. OCAL should be a **coding-work authority language**: repo-scoped, evidence-gated, CI-aware, memory-aware, and safe for large software organizations.

### Claude Code and Codex

Claude Code and Codex show what production coding agents are converging on: sandboxing, permissions, hooks, subagents, skills, worktrees, background/parallel execution, and human approval. Claude Code supports custom subagents with their own context, tools, and permissions; deterministic hooks; sandboxing; memory via `CLAUDE.md`; and permission modes. ([Claude][8])

Codex similarly emphasizes isolated environments, sandboxed local execution, approval policies, cloud tasks, parallel work, worktrees, and long-running tasks across IDE, CLI, web, mobile, and CI/CD contexts. ([OpenAI][9])

**What OCAL should borrow:**

* Explicit sandbox semantics.
* Worktree-first parallelism.
* Approval policy as machine-readable config.
* Hooks and lifecycle gates.
* Subagent delegation.
* Multi-surface UX: terminal, IDE, web, mobile, CI.

**Where OCAL can win:**

Claude Code and Codex are products/runtimes. OCAL can become a **standard contract block** that any product can detect, preview, validate, and execute.

### MCP, Skills, AGENTS.md, A2A

MCP is becoming the standard tool/data connector layer; Anthropic describes it as an open standard for connecting AI apps to tools, data, and workflows. ([Model Context Protocol][10])

Agent Skills and Codex/Anthropic skill formats push toward portable, progressively disclosed capabilities. ([Microsoft Learn][11])

AGENTS.md is already a widely used repo instruction format for coding agents. ([Agents][12])

A2A is relevant for agent-to-agent communication and coordination across systems, complementary to MCP’s tool/data focus. ([Google Developers Blog][13])

**OCAL implication:**

OCAL should explicitly integrate with these formats, but treat them as **advisory or capability layers**, not authority layers. Authority stays in OCAL and the host.

### Security research: MCP and Skills are dangerous if blindly trusted

MCP and Skills introduce serious supply-chain and prompt-injection risks. OWASP’s MCP Top 10 includes tool poisoning, supply-chain attacks, rug pulls, schema poisoning, and tool shadowing. Research on MCP threat modeling identifies tool poisoning in metadata as a major vulnerability, and other security work proposes manifest signing, semantic vetting, and defensive heuristics. ([OWASP][14])

Skills have similar risks. Research on Agent Skills describes prompt-injection and sensitive-file exfiltration risks, while another survey reported vulnerabilities in community-contributed skills and proposed tiered governance. ([arXiv][15])

**OCAL implication:**

OCAL needs supply-chain policy for MCP servers, skills, plugins, agent manifests, and tool descriptors.

### Benchmarks say long-horizon coding is still weak

The benchmark story matters a lot. SWE-bench Verified has driven rapid coding-agent progress, but newer long-horizon benchmarks show current agents still struggle with project evolution, architecture, extension robustness, and end-to-end software development. SlopCodeBench reports that no evaluated agent solved full extension problems end-to-end and highlights structural erosion, verbosity growth, and brittle code extension. SWE-EVO reports a major gap between benchmark-style issue fixing and long-horizon release-note-driven software evolution. CodeClash argues that real software engineering is more goal-oriented than isolated issue solving. ([SWE-bench][16])

METR’s time-horizon framing is also relevant: it measures the length of task a human expert would take that an AI agent can complete with a given reliability, rather than merely how long the agent runs. ([METR][17])

**OCAL implication:**

OCAL cannot optimize only for “fix the test.” It must govern **architecture quality, long-term maintainability, regression resistance, memory correctness, and evidence accumulation**.

---

# 2. Biggest conclusion: OCAL needs to become an **Agent Operating Contract**

The current OCAL v1 is a strong daemon control language. But for 1B+ coding projects, it should evolve into something larger:

> **OCAL = declarative authority, evidence, memory, execution, and promotion contract for coding agents.**

That means every daemon run should answer:

1. **Who authorized this?**
2. **What exact authority was delegated?**
3. **What environment may it touch?**
4. **What memory may it read or write?**
5. **What tools may it call?**
6. **What code areas may it modify?**
7. **What evidence is required before promotion?**
8. **What is the rollback plan?**
9. **What telemetry proves what happened?**
10. **How does this compose with other agents and runtimes?**

OCAL v1 answers several of these. OCAL vNext should answer all of them.

# 3. What OCAL is currently missing

## Missing piece 1: explicit authority and standing orders

OCAL has `permissions`, but permissions are not the same as **authority**.

A permission says: “May this run use shell?”

Authority says:

* Who delegated this work?
* For which repositories?
* For which branches?
* For which file scopes?
* Under what triggers?
* For how long?
* With what escalation rules?
* With what legal/security constraints?
* With what approval group?

For ambitious coding projects, OCAL needs `authority`.

### Proposed syntax

```yaml
authority:
  delegated_by: "human:eng-lead@example.com"
  accountability:
    owner: "team:platform"
    reviewers: ["team:security", "team:infra"]
    escalation_channel: "slack:#agent-ops"

  standing_orders:
    - id: nightly-ci-repair
      description: Repair CI failures on non-production branches.
      triggers:
        cron: "0 2 * * *"
        webhooks: ["ci_failed", "buildkite_failed", "github_check_failed"]
      scope:
        repos:
          allow: ["github://acme/*"]
          deny: ["github://acme/prod-infra"]
        branches:
          allow_regex: "^(main|develop|release/.+|ocal/.+)$"
          write_prefix: "ocal/ci-fix/"
        files:
          deny:
            - ".github/workflows/prod-deploy.yml"
            - "infra/prod/**"
            - "**/*.pem"
            - "**/.env*"
      escalation:
        ask_on:
          - database_migration
          - public_api_change
          - auth_or_permissions_change
          - secrets_access
          - production_deploy
        auto_pause_on:
          - policy_conflict
          - owner_unavailable
          - unrecognized_repo
```

**Why this matters:** Hermes and OpenClaw show the importance of long-lived automations and standing orders, but OCAL should make them auditable and repo-safe. OpenClaw’s standing orders and host-owned cron are especially relevant here. ([OpenClaw][18])

## Missing piece 2: sandboxing as a first-class block

OCAL has tool permissions, but tool permission is not enough. A shell command can be allowed while still needing network isolation, filesystem boundaries, CPU/memory limits, secret isolation, and egress controls.

OpenHands, Codex, and Claude Code all treat sandboxing as central. OpenHands uses sandboxed Docker environments; Codex uses isolated cloud/local sandboxes and approval policies; Claude Code has filesystem/network sandboxing and permission modes. ([OpenHands Docs][19])

### Proposed syntax

```yaml
sandbox:
  default:
    runtime: docker                 # docker | kubernetes | firecracker | local | remote
    image: "ghcr.io/acme/ocal-node-bun:2026.05"
    user: nonroot
    ephemeral: true

    filesystem:
      read:
        allow:
          - "$REPO"
          - "$WORKTREE"
          - ".opencode/daemon"
        deny:
          - "~/.ssh/**"
          - "~/.aws/**"
          - "**/.env*"
          - "**/*secret*"
      write:
        allow:
          - "$WORKTREE/**"
          - ".opencode/daemon/**"
        deny:
          - "$REPO/.git/config"
          - "infra/prod/**"

    network:
      mode: restricted              # deny | restricted | allow
      allow_hosts:
        - "registry.npmjs.org"
        - "github.com"
      deny_hosts:
        - "169.254.169.254"
        - "*.internal.acme.com"

    resources:
      cpu: "4"
      memory: "8Gi"
      disk: "20Gi"
      timeout: "30m"

    process:
      deny_patterns:
        - "sudo"
        - "chmod 777"
        - "curl .* | sh"
        - "wget .* | sh"

  overrides:
    prototype:
      runtime: kubernetes
      ephemeral: true
      network:
        mode: restricted
    promotion_review:
      network:
        mode: deny
```

**Key distinction:** `permissions.shell: allow` decides whether shell may be requested. `sandbox.network.mode: restricted` decides what shell can actually reach. Both are needed.

## Missing piece 3: identity, secrets, and credential boundaries

OCAL currently has `permissions`, but for real organizations it needs identity-aware tool access:

* Which service account is used?
* What scopes does it have?
* Are credentials injected?
* Are secrets redacted?
* Can the model see secret values?
* Are tokens short-lived?
* Are secret accesses logged?
* Can the agent create credentials? Almost always no.

### Proposed syntax

```yaml
identity:
  run_as:
    type: service_account
    id: "ocal-ci-repairer"
    expires_after: 4h

  credentials:
    default: none

    allow:
      - id: github-readonly
        provider: vault
        scope: "github:read"
        expose_to_model: false
        mount_as: env
        env: GITHUB_TOKEN

      - id: npm-install-token
        provider: vault
        scope: "npm:read"
        expose_to_model: false
        mount_as: process_secret

    deny:
      - "prod-db-password"
      - "cloud-admin"
      - "signing-key"

  audit:
    log_secret_access: true
    redact_tool_output: true
    require_human_approval_for:
      - new_secret
      - write_secret
      - production_secret
```

**Critical point:** Most agent specs talk about tools. Enterprise deployments care about **identity**.

## Missing piece 4: memory as a typed lifecycle, not just context preservation

OCAL has `context.preserve`, SQLite state, task capsules, and incubator scratch. That is a good start. But long-running coding agents need a much richer memory system.

Modern agent systems distinguish short-term context, durable memory, project rules, episodic memory, semantic memory, skills, and user/team preferences. LangGraph emphasizes memory and checkpoints; Hermes emphasizes persistent memory and skill creation; OpenClaw has memory promotion; Claude Code has `CLAUDE.md` and auto memory; Codex Skills use progressive disclosure. ([LangChain Docs][20])

OCAL should treat memory writes as dangerous. A bad memory can poison future runs.

### Proposed syntax

```yaml
memory:
  stores:
    run:
      type: sqlite
      scope: run
      ttl: 30d

    task:
      type: sqlite
      scope: task
      ttl: 180d

    project:
      type: repo_files
      files:
        - "AGENTS.md"
        - "docs/architecture/**/*.md"
        - "docs/decisions/**/*.md"
      writable: ask

    org:
      type: mcp
      profile: readonly-knowledge
      writable: deny

    vector:
      type: sqlite_vec
      embeddings: "text-embedding-3-large"
      scope: project
      writable: host_promoted_only

    code_graph:
      type: scip
      refresh: incremental

  read_policy:
    max_tokens_per_iteration: 12000
    retrieval:
      strategy: hybrid
      include:
        - recent_failures
        - relevant_decisions
        - touched_symbols
        - test_history
        - owner_notes
      exclude:
        - stale_hypotheses
        - unresolved_scratch
        - low_confidence_memories

  write_policy:
    default: scratch_only
    promote:
      require_evidence: true
      require_category: true
      require_review_for:
        - project_rule
        - architecture_decision
        - skill
        - user_preference
      categories:
        - fact
        - decision
        - pitfall
        - invariant
        - workflow
        - test_mapping
        - skill_candidate
      evidence:
        min_sources: 1
        allowed_sources:
          - test_result
          - code_reference
          - human_feedback
          - successful_checkpoint
          - production_observation
      on_conflict: quarantine

  compaction:
    strategy: evidence_weighted
    compact_every: 8
    preserve:
      - active_hypothesis
      - failing_tests
      - touched_files
      - constraints
      - reviewer_feedback
    discard:
      - failed_unpromoted_hypotheses
      - redundant_tool_output
      - stale_plans
```

### Why this is a big deal

Today, many coding agents “remember” things by dumping summaries into context. That causes jank: stale assumptions, hallucinated facts, overwritten constraints, and forgotten reviewer feedback. OCAL should make memory a **typed, reviewable, evidence-backed database**.

### Add memory tests

```yaml
memory_tests:
  - name: no-unsupported-project-rules
    check:
      shell: "ocal memory lint --require-evidence project"
    on_fail: pause

  - name: stale-memory-quarantine
    check:
      shell: "ocal memory stale --older-than 180d --category fact"
    on_fail: warn

  - name: memory-conflict-detection
    check:
      shell: "ocal memory conflicts --scope project"
    on_fail: ask_human
```

## Missing piece 5: code intelligence and repo maps

For 1B+ LOC codebases, the agent cannot rely on grep plus vibes. It needs structured code intelligence:

* Symbol graph.
* Call graph.
* Dependency graph.
* Ownership graph.
* Test impact graph.
* Architecture decision records.
* API compatibility metadata.
* Security-sensitive paths.
* Build graph.
* Generated-code boundaries.
* Monorepo package boundaries.

Sourcegraph’s platform is built around indexed code search and code intelligence across repos; SCIP is a language-agnostic source-code indexing protocol; CodeQL models code as data for semantic analysis; Semgrep provides static analysis across many languages. ([Sourcegraph][21])

### Proposed syntax

```yaml
code_intelligence:
  repo_map:
    generate_on_start: true
    refresh: incremental
    include:
      - packages
      - services
      - public_apis
      - owners
      - generated_code
      - security_boundaries
      - dependency_graph
      - test_impact_graph

  indexes:
    - id: ripgrep
      type: lexical
      command: "rg"

    - id: scip
      type: symbol_graph
      command: "scip index"

    - id: codeql
      type: semantic_security
      databases:
        - "codeql-db"

    - id: semgrep
      type: static_analysis
      rulesets:
        - "p/security-audit"
        - "p/typescript"

    - id: build_graph
      type: build_system
      command: "bazel query //..."

    - id: test_impact
      type: test_mapping
      command: "tools/test-impact changed-files"

  retrieval:
    before_planning:
      include_symbols: true
      include_callers: true
      include_callees: true
      include_tests: true
      include_owners: true
      max_files: 200

  restrictions:
    generated_code:
      write: deny
      unless: "generator_updated"
    public_api:
      require:
        - compatibility_review
        - changelog_entry
        - owner_approval
```

### Key upgrade

OCAL should require agents to produce changes from a **code-intelligence substrate**, not only from chat context.

## Missing piece 6: graph workflows, not just loops

OCAL’s `loop` is useful, but complex coding projects need explicit DAG/state-machine execution:

* Discover.
* Triage.
* Plan.
* Spawn experiments.
* Evaluate.
* Review.
* Merge.
* Validate.
* Promote.
* Monitor.
* Roll back if needed.

LangGraph’s graph/checkpoint model and durable-execution systems like Temporal and Inngest show why this matters: long-running workflows need checkpointed state, human waits, retries, durable timers, audit trails, and resumption after delay. ([LangChain Docs][2])

### Proposed syntax

```yaml
workflow:
  type: graph
  start: discover

  nodes:
    discover:
      kind: shell
      command: "ocal tasks discover"
      next: triage

    triage:
      kind: agent
      agent: plan
      output_schema: triage_result
      next:
        if:
          - when: "$.risk_score >= 0.7"
            goto: incubate
          - when: "$.ready_to_fix == true"
            goto: implement
          - otherwise: park

    incubate:
      kind: incubator
      profile: hard-task
      next:
        if:
          - when: "$.promotion.ready == true"
            goto: implement
          - otherwise: park

    implement:
      kind: experiment_tournament
      profile: patch-tournament
      next: verify

    verify:
      kind: verification
      checks: ["unit", "typecheck", "lint", "security"]
      next:
        if:
          - when: "$.checks.all_passed == true"
            goto: review
          - otherwise: implement

    review:
      kind: review
      reviewers: ["critic", "security", "maintainer-sim"]
      next:
        if:
          - when: "$.review.approved == true"
            goto: checkpoint
          - otherwise: implement

    checkpoint:
      kind: checkpoint
      next: promote

    promote:
      kind: human_gate
      require: ["evidence_bundle", "rollback_plan", "owner_approval"]
      next: stop

    park:
      kind: task_state
      state: parked
      next: stop

    stop:
      kind: terminal
```

### Why this matters

Loops are fine for “fix tests.” Graphs are necessary for “modernize the billing system without breaking 90 downstream services.”

## Missing piece 7: experiment tournaments and disjointed explorations

The current incubator has passes, but OCAL needs an explicit mechanism for **parallel disjoint experiments**:

* Generate diverse hypotheses.
* Implement each in isolated worktrees.
* Score them independently.
* Compare against baselines.
* Merge only the best evidence-backed candidate.
* Preserve failed attempts as memory.
* Prevent all workers from converging on the same mediocre patch.

Research patterns like Tree of Thoughts, Reflexion, Self-Refine, Voyager, and ReAct all point toward iterative exploration, feedback, self-critique, tool-grounding, and skill accumulation. ([arXiv][22])

### Proposed syntax

```yaml
experiments:
  - id: patch-tournament
    strategy: hypothesis_tournament

    generate:
      agent: plan
      count: 8
      diversity: high
      require:
        - hypothesis
        - affected_files
        - expected_tests
        - rollback_plan
        - risk_score
      avoid_duplicate_plans: true

    workspace:
      isolation: git_worktree
      branch_prefix: "ocal/exp/{{task_id}}/"
      inherit_memory:
        include: ["task", "project"]
        exclude: ["other_experiment_private_notes"]

    execute:
      agent: build
      max_parallel: 4
      timeout: 30m
      writes: isolated_worktree
      stop_on:
        - no_progress_iterations_gte: 3
        - diff_size_gt: "5000 lines"

    evaluate:
      baseline:
        capture_before: true
      checks:
        - id: tests
          command: "bun test --timeout 60000"
          weight: 0.35
        - id: typecheck
          command: "npx tsgo --noEmit"
          weight: 0.20
        - id: lint
          command: "bun run lint"
          weight: 0.10
        - id: complexity
          command: "ocal metrics complexity --diff"
          weight: 0.10
          prefer: lower
        - id: diff_minimality
          command: "ocal metrics diff-minimality"
          weight: 0.10
          prefer: higher
        - id: security
          command: "semgrep scan --config p/security-audit"
          weight: 0.10
        - id: reviewer_score
          agent: critic
          weight: 0.05

    reduce:
      strategy: best_evidence
      require_pairwise_review: true
      block_if:
        - all_scores_lt: 0.72
        - unresolved_critical_objections_gte: 1
      keep:
        winner: true
        losers: summarize_and_archive
      promote:
        into: ready_queue
```

This is one of the biggest OCAL upgrades. “Multiple passes” should not mean “the same agent keeps trying.” It should mean **diverse controlled experiments with evidence-based reduction**.

## Missing piece 8: anti-vibe-coding / anti-jank controls

The user used “vibe coding” and “jankurai.” This is exactly where OCAL can shine.

Most agents optimize for the immediate test. That can produce:

* Bloated code.
* Overfitted fixes.
* Removed edge cases.
* Skipped tests.
* Hidden behavior changes.
* Complexity creep.
* Fragile abstractions.
* Verbose duplicate implementations.
* Misplaced code.
* Architectural drift.

SlopCodeBench is especially relevant because it evaluates whether agents can extend their own code and reports structural erosion and verbosity growth. ([arXiv][23])

### Proposed syntax

```yaml
quality:
  anti_jank:
    enabled: true

    metrics:
      - id: diff_size
        command: "ocal metrics diff-size"
        max_delta: "+800 lines"

      - id: duplication
        command: "ocal metrics duplication --diff"
        max_delta: "+3%"

      - id: cyclomatic_complexity
        command: "ocal metrics complexity --diff"
        max_delta: "+5%"

      - id: dependency_fanout
        command: "ocal metrics dependency-fanout --diff"
        max_delta: "+2"

      - id: public_api_surface
        command: "ocal metrics public-api-diff"
        max_delta: "0"
        unless: "api_change_approved"

      - id: test_deletion
        command: "ocal metrics deleted-tests"
        invariant: equals_zero

      - id: skipped_tests
        command: "rg \"skip\\(|\\.only\\(\" test src packages || true"
        invariant: equals_zero

    on_violation:
      default: incubate_current_task
      severe: pause

  architecture:
    require_design_review_when:
      - touched_files_gte: 20
      - public_api_changed: true
      - dependency_boundary_crossed: true
      - risk_score_gte: 0.7

    fitness_functions:
      - name: no-layering-violations
        command: "npm run arch:check"
        on_fail: block_promotion

      - name: no-circular-dependencies
        command: "npm run deps:cycles"
        on_fail: block_promotion

      - name: no-generated-code-edits
        command: "ocal check generated-code"
        on_fail: block
```

### Important principle

OCAL should make jank measurable. If quality cannot be measured, the agent will eventually optimize for superficial success.

## Missing piece 9: continuous evals and golden tasks

OCAL has verification commands, but it needs persistent eval suites:

* Golden tasks.
* Regression scenarios.
* Architecture checks.
* Performance baselines.
* Security scenarios.
* “Can this code be extended?” tests.
* Long-horizon project benchmarks.

OpenHands has evaluation tooling around software-agent behavior, and agent observability/evaluation platforms increasingly center traces, datasets, and feedback loops. ([GitHub][24])

### Proposed syntax

```yaml
evals:
  suites:
    - id: ci
      type: shell
      commands:
        - "bun run lint"
        - "npx tsgo --noEmit"
        - "bun test --timeout 60000"

    - id: golden_tasks
      type: replay
      dataset: ".ocal/evals/golden-tasks.jsonl"
      max_regression_rate: 0.0

    - id: extension_robustness
      type: synthetic_followup
      generate_followups: 5
      require_pass_rate_gte: 0.8

    - id: architecture
      type: fitness
      commands:
        - "npm run arch:check"
        - "npm run deps:cycles"

    - id: performance
      type: benchmark
      command: "npm run bench"
      compare_to: baseline
      max_slowdown: "5%"

  run:
    before_checkpoint: ["ci"]
    before_promotion: ["ci", "architecture", "golden_tasks"]
    nightly: ["extension_robustness", "performance"]

  regression_policy:
    on_any_regression: block_promotion
    on_flaky:
      retry: 2
      then: ask_human
```

### Why this matters

The real goal is not “the current test passes.” The real goal is “the project remains healthy after this change and after future changes.”

## Missing piece 10: observability, tracing, and replay

OCAL’s SQLite state is a great foundation. But large organizations need:

* Trace spans for every LLM call, tool call, memory read/write, permission decision, test run, checkpoint, and human approval.
* Cost attribution.
* Token attribution.
* Failure clustering.
* Replay from any checkpoint.
* Diff provenance.
* Evidence bundles.
* OpenTelemetry export.
* Local mirror plus central observability.

LangSmith, OpenTelemetry, Laminar, and AgentOps all point to the importance of tracing, evaluation, monitoring, and debugging non-deterministic agent behavior. ([LangChain][25])

### Proposed syntax

```yaml
observability:
  trace:
    enabled: true
    format: opentelemetry
    export:
      - sqlite
      - "otlp://localhost:4317"
    spans:
      - run
      - task
      - iteration
      - llm_call
      - tool_call
      - permission_decision
      - sandbox_event
      - memory_read
      - memory_write
      - test_run
      - diff
      - checkpoint
      - human_approval
      - promotion
      - rollback

  cost:
    track:
      - model_tokens
      - wall_time
      - tool_runtime
      - compute_cost
      - external_api_cost
    budgets:
      per_run: "$50"
      per_task: "$5"
      per_experiment: "$1"

  replay:
    enabled: true
    retain_inputs: true
    retain_tool_outputs: true
    redact_secrets: true
    deterministic_shell_replay: best_effort

  evidence_bundle:
    on_checkpoint: true
    include:
      - diff
      - tests
      - metrics
      - reviewer_feedback
      - memory_writes
      - policy_decisions
      - rollback_plan
```

### CLI/API upgrades

```sh
opencode daemon trace <runID>
opencode daemon replay <runID> --from iteration:12
opencode daemon evidence <runID> <taskID>
opencode daemon cost <runID>
opencode daemon policy-decisions <runID>
```

## Missing piece 11: supply-chain security for MCP, Skills, plugins, and workers

OCAL currently has MCP profiles, but it needs trust policies. MCP servers, Skills, plugins, and worker agents should not be treated as neutral. Tool metadata and skill instructions can contain malicious or misleading content. OWASP and security research around MCP and Skills make this an urgent requirement. ([OWASP][14])

### Proposed syntax

```yaml
supply_chain:
  default_trust: quarantine

  mcp:
    require_manifest: true
    require_signature: true
    allow_unsigned: deny
    semantic_vetting:
      enabled: true
      checks:
        - tool_description_prompt_injection
        - hidden_instruction_patterns
        - credential_exfiltration_patterns
        - overbroad_permissions
    servers:
      - id: github
        trust: verified
        tools:
          allow: ["read_issue", "list_prs", "create_pr"]
          deny: ["delete_repo", "modify_branch_protection"]

      - id: filesystem
        trust: local
        tools:
          allow: ["read_file", "write_file", "list_dir"]
        scope:
          root: "$WORKTREE"

  skills:
    discovery:
      paths:
        - ".opencode/skills"
        - ".agent/skills"
      allow_marketplace: ask

    trust_tiers:
      verified:
        require_signature: true
        allow_scripts: true
      local:
        require_review: true
        allow_scripts: ask
      community:
        allow_scripts: deny
        allow_network: deny

    promotion:
      skill_candidate_from_memory: allow
      require:
        - tests
        - human_review
        - manifest_signature
        - rollback_plan

  plugins:
    require_sbom: true
    require_lockfile: true
    vulnerability_scan:
      command: "npm audit --production"
      on_fail: block
```

### Key rule

A tool’s natural-language description must never be treated as authority. It is untrusted input.

## Missing piece 12: interoperability adapters

OCAL should be available inside any chat prompt: OpenCode, Codex, Claude, etc. That means OCAL needs a portability layer.

OpenCode already has permissions, tools, AGENTS.md, MCP, custom tools, and many providers. ([OpenCode][26])

Claude Code and Codex have overlapping features—permissions, hooks, skills, subagents, sandboxes, worktrees—but different syntax and runtime semantics. ([Claude][27])

### Proposed syntax

```yaml
interop:
  host:
    preferred: opencode
    fallback:
      - claude_code
      - codex
      - openhands
      - langgraph

  adapters:
    opencode:
      permissions: native
      mcp: native
      agents: native
      sqlite: native

    claude_code:
      map_permissions_to: ".claude/settings.json"
      map_memory_to: "CLAUDE.md"
      map_hooks_to: "hooks"
      map_subagents_to: ".claude/agents"

    codex:
      map_permissions_to: "codex-policy"
      map_sandbox_to: "codex-sandbox"
      map_skills_to: "skills"

    openhands:
      map_workers_to: "openhands-sdk"
      map_sandbox_to: "openhands-runtime"
      map_evals_to: "openhands-evaluation"

    langgraph:
      compile_workflow_to: "StateGraph"
      compile_checkpoints_to: "checkpointer"

  prompt_embedding:
    detect_sentinels: true
    reject_code_fenced_blocks: true
    preview_without_arm: true
    require_arm_from_human: true
```

### Important design rule

OCAL should be **host-intercepted**, not model-interpreted. The chat model may see the block, but the host validates and executes it.

## Missing piece 13: worker contracts and agent cards

OCAL has `agents` and `workers`, but ambitious projects need machine-readable worker capability declarations:

* What tools can this worker use?
* What languages does it know?
* What contexts does it receive?
* What outputs must it produce?
* Is it allowed to write?
* Does it have network?
* Is it trusted to review?
* Can it spawn children?
* What happens if it diverges?

This connects to A2A-style agent coordination and OpenClaw’s node capability model. ([Google Developers Blog][13])

### Proposed syntax

```yaml
workers:
  registry:
    - id: typescript-builder
      kind: coding_agent
      provider: opencode
      model: "strong-coder"
      capabilities:
        languages: ["typescript", "javascript"]
        tools: ["shell", "edit", "mcp.filesystem"]
        writes: ["isolated_worktree"]
      limits:
        max_runtime: 30m
        max_tokens: 200000
        max_diff_lines: 2000
      output_schema: worker_patch_result

    - id: security-reviewer
      kind: review_agent
      provider: claude_code
      capabilities:
        tools: ["read", "semgrep", "codeql"]
        writes: ["scratch_only"]
      output_schema: security_review

  coordination:
    protocol: a2a_like
    message_log: sqlite
    allow_worker_to_worker_messages: true
    require_supervisor_visibility: true
    prevent_collusion:
      independent_reviews: true
      blind_review: true
```

## Missing piece 14: concurrency control for very large codebases

At 1B+ LOC, multiple agents will touch overlapping files, services, APIs, tests, and dependencies. OCAL needs scheduling, locks, leases, and conflict policy.

### Proposed syntax

```yaml
scheduler:
  mode: fleet
  max_active_runs: 200
  max_active_tasks_per_repo: 20
  max_active_tasks_per_service: 5

  queues:
    - id: critical-ci
      priority: 100
      concurrency: 20

    - id: refactors
      priority: 40
      concurrency: 5

    - id: experiments
      priority: 20
      concurrency: 50

  leases:
    enabled: true
    ttl: 30m
    lock_granularity:
      - file
      - package
      - service
      - public_api
    conflict_policy:
      same_file: block
      same_package: warn
      public_api: require_owner_approval

  rate_limits:
    model_calls_per_minute: 1000
    github_api_per_hour: 4000
    ci_jobs_per_hour: 500

  fairness:
    prevent_repo_starvation: true
    max_queue_age: 24h
```

## Missing piece 15: rollout, deployment, and post-merge monitoring

OCAL currently stops at checkpoint/promotion. Real coding projects need post-merge validation:

* Create PR.
* Merge behind feature flag.
* Canary.
* Monitor.
* Roll back.
* Learn from production telemetry.

### Proposed syntax

```yaml
rollout:
  target: pull_request

  pr:
    create: ask
    title: "ocal: {{task.name}}"
    include_evidence_bundle: true
    require_reviewers:
      - code_owner
      - security_if_needed
      - human_owner

  merge:
    strategy: squash
    require:
      - ci_green
      - code_owner_approval
      - no_policy_violations
      - rollback_plan

  deploy:
    mode: canary
    environments:
      - staging
      - production
    require_human_for: production

  monitor:
    duration: 2h
    signals:
      - error_rate
      - latency_p95
      - crash_rate
      - support_tickets
      - custom_business_metric
    on_regression:
      - rollback
      - pause_related_runs
      - open_incident
      - write_memory:
          category: pitfall
          require_review: true
```

# 4. Proposed OCAL vNext top-level keys

OCAL v1 has a solid set of keys. I would add these top-level blocks:

| New block           | Purpose                                                            |
| ------------------- | ------------------------------------------------------------------ |
| `authority`         | Who delegated what, scope, standing orders, escalation             |
| `sandbox`           | Runtime isolation, filesystem/network/process/resource limits      |
| `identity`          | Service accounts, credentials, secret exposure, audit              |
| `memory`            | Typed stores, promotion rules, evidence, compaction                |
| `memory_tests`      | Validate memory quality and conflicts                              |
| `code_intelligence` | Symbol graph, call graph, test impact, static analysis             |
| `workflow`          | Durable graph/state-machine execution                              |
| `experiments`       | Hypothesis tournaments and disjointed worktrees                    |
| `quality`           | Anti-jank metrics, architecture fitness functions                  |
| `evals`             | Golden tasks, extension robustness, performance, regression suites |
| `observability`     | Traces, costs, replay, evidence bundles                            |
| `supply_chain`      | MCP/skills/plugin trust, signing, SBOM, vetting                    |
| `interop`           | Mapping to OpenCode, Codex, Claude, OpenHands, LangGraph           |
| `workers`           | Worker registry, capability contracts, A2A-like coordination       |
| `scheduler`         | Fleet concurrency, leases, queues, rate limits                     |
| `rollout`           | PR, merge, canary, monitor, rollback                               |
| `risk`              | Formal blast-radius and safety classification                      |
| `governance`        | Compliance, audit, retention, policy-as-code                       |
| `models`            | Model routing, budget, fallback, confidence caps                   |
| `datasets`          | Task/eval datasets and generated regression cases                  |
| `artifacts`         | Durable storage of diffs, patches, logs, generated docs            |

# 6. Direct response to overlap concerns

## “Is OCAL redundant with LangChain or LangGraph?”

No. LangGraph is a programmatic runtime for graph-shaped agent execution with durable state, checkpoints, and human-in-the-loop semantics. OCAL should be a declarative contract that can compile into a LangGraph workflow or run natively in OpenCode. ([LangChain Docs][2])

**Recommendation:** Add `workflow.type: graph` and optional `interop.adapters.langgraph.compile_workflow_to: StateGraph`.

## “Is OCAL redundant with OpenHands?”

No. OpenHands provides agent/runtime/SDK/evaluation infrastructure. OCAL can orchestrate OpenHands workers and define the safety/evidence contract around them. OpenHands is especially strong on sandboxed runtime and evaluation infrastructure. ([GitHub][28])

**Recommendation:** Treat OpenHands as a worker/runtime backend.

```yaml
workers:
  registry:
    - id: openhands-builder
      provider: openhands
      runtime: docker
      writes: isolated_worktree
```

## “Is OCAL redundant with Hermes?”

No. Hermes is a long-lived self-improving agent with memory, skills, messaging, automation, and delegation. OCAL should borrow Hermes’s learning and skill-generation ideas, but make them host-gated and evidence-backed. ([GitHub][3])

**Recommendation:** Add memory promotion and skill promotion blocks. Do not let the agent silently rewrite its own operating procedures.

## “Is OCAL redundant with OpenClaw?”

No. OpenClaw is closer to a gateway/personal assistant platform with channels, cron, standing orders, memory, skills, and delegate architecture. OCAL should adopt standing orders and gateway concepts, but specialize them for safe coding autonomy. ([OpenClaw][6])

**Recommendation:** Add `authority.standing_orders`, `control_plane.channels`, and `scheduler`.

## “Is OCAL redundant with Claude Code or Codex?”

No. Claude Code and Codex are powerful coding-agent products with permissions, hooks, sandboxing, worktrees, subagents, skills, and parallel tasks. OCAL can become the portable layer that maps to both. ([Claude][8])

**Recommendation:** OCAL should define adapters, not compete feature-by-feature.

## “Is OCAL redundant with MCP?”

No. MCP is a tool/data connector protocol. OCAL is an authority and lifecycle contract. MCP tells an agent what tools exist; OCAL tells the host which tools may be used, under what trust level, with what scopes, and what evidence is needed afterward. ([Model Context Protocol][10])

**Recommendation:** OCAL should treat MCP metadata as untrusted until vetted.

# 7. The most important OCAL design principle

The decisive principle should be:

> **Advisory inputs are not authority.**

These are advisory:

* AGENTS.md
* CLAUDE.md
* Skills
* MCP tool descriptions
* model-generated plans
* memory summaries
* chat instructions
* worker claims
* self-reported confidence

These are authority:

* OCAL `authority`
* OCAL `permissions`
* OCAL `sandbox`
* OCAL `identity`
* OCAL `supply_chain`
* OCAL `constraints`
* OCAL `checkpoint`
* OCAL `promotion`
* host policy
* human approvals
* signed manifests

This distinction is how OCAL avoids becoming another prompt trick.

# 8. How OCAL should manage memory better

The memory system should have five layers.

## Layer 1: transient context

What the model sees right now.

```yaml
context:
  strategy: hybrid
  compact_every: 8
  preserve:
    - objective
    - active_hypothesis
    - constraints
    - failing_tests
    - reviewer_feedback
```

## Layer 2: run memory

Facts from this run only.

```yaml
memory:
  stores:
    run:
      type: sqlite
      scope: run
      ttl: 30d
```

## Layer 3: task memory

Durable history for one task.

```yaml
memory:
  stores:
    task:
      type: sqlite
      scope: task
      ttl: 180d
```

## Layer 4: project memory

Repo-specific durable knowledge.

```yaml
memory:
  stores:
    project:
      type: repo_files
      files:
        - "AGENTS.md"
        - "docs/agent-memory/**/*.md"
      writable: ask
```

## Layer 5: org memory

Policies, architecture standards, security rules, ownership, historical incidents.

```yaml
memory:
  stores:
    org:
      type: mcp
      profile: readonly-knowledge
      writable: deny
```

## Memory promotion should be evidence-based

```yaml
memory:
  write_policy:
    default: scratch_only
    promote:
      require_evidence: true
      require_review_for:
        - project_rule
        - architecture_decision
        - skill
      on_conflict: quarantine
```

## Never let scratch become truth automatically

Scratch is cheap. Truth is expensive.

That one rule will prevent a huge amount of agentic jank.

# 9. How OCAL reduces vibe coding

OCAL can reduce vibe coding by requiring every meaningful change to have:

1. A hypothesis.
2. A bounded scope.
3. A test mapping.
4. A rollback plan.
5. A quality budget.
6. A reviewer pass.
7. Evidence before checkpoint.
8. Memory writes only after evidence.

A model can still be creative. But it cannot silently turn creativity into production code.

### Add a required change contract

```yaml
change_contract:
  required_for:
    - edit
    - checkpoint
    - promotion

  schema:
    type: object
    required:
      - hypothesis
      - intended_behavior_change
      - affected_files
      - tests_to_run
      - rollback_plan
      - risk_score
      - non_goals
    properties:
      hypothesis: { type: string }
      intended_behavior_change: { type: string }
      affected_files: { type: array, items: { type: string } }
      tests_to_run: { type: array, items: { type: string } }
      rollback_plan: { type: string }
      risk_score: { type: number, minimum: 0, maximum: 1 }
      non_goals: { type: array, items: { type: string } }

  on_missing: pause
```

This is a beautiful OCAL feature because it does not need to understand the model’s inner reasoning. It only requires inspectable commitments.

# 11. OCAL should support “stateful building on current state”

The user specifically asked about stateful building. This is important.

There are two opposite needs:

1. **Disjoint experiments**: explore independently to avoid convergence.
2. **Stateful building**: once a good path exists, let agents build incrementally on it.

OCAL should support both.

### Proposed syntax

```yaml
development_mode:
  phases:
    - id: exploration
      strategy: disjoint_experiments
      max_parallel: 8
      share_memory_between_experiments: false
      end_when:
        - best_score_gte: 0.78

    - id: consolidation
      strategy: stateful_build
      base: best_experiment
      agents:
        - builder
        - critic
        - test-writer
      share_memory_between_agents: true
      require_iteration_contract: true
      end_when:
        - tests_pass
        - quality_pass
        - reviewer_approved

    - id: hardening
      strategy: adversarial_review
      agents:
        - security-reviewer
        - performance-reviewer
        - maintainer-reviewer
      writes: scratch_only
```

This is a major improvement over simple “fan out” because it separates **divergent search** from **convergent refinement**.

Different tasks need different models. A cheap model can grep, summarize, or classify. A stronger model should handle architecture and promotion decisions.

```yaml
models:

    default: "balanced-coder"

    by_task:
      scout: "cheap-fast"

      security_review: "security-specialist"

    fallback:
      on_rate_limit: "balanced-coder"
      on_context_overflow: "long-context"

  budgets:
    per_run:
      max_tokens: 5000000
      max_cost: "$100"

      max_tokens: 500000
      max_cost: "$10"

  confidence:
    model_confidence_cap: 0.6
    require_evidence_over_confidence: true
```

OCAL already has `model_confidence_cap` inside incubator readiness. Make that global.

# 14. The OCAL runtime should have three interpreters

To avoid ambiguity, OCAL should separate three runtime roles.

## 1. Parser/validator

* Validates sentinels.
* Validates YAML schema.
* Rejects code-fenced execution blocks.
* Checks IDs match.
* Checks required fields.
* Checks unknown keys.
* Produces preview.

## 2. Policy engine

* Evaluates permissions.
* Evaluates sandbox policy.
* Evaluates identity/secrets.
* Evaluates supply-chain trust.
* Evaluates guardrails.
* Evaluates authority.

## 3. Orchestrator

* Runs workflow nodes.
* Spawns workers.
* Maintains SQLite state.
* Emits telemetry.
* Handles retries.
* Handles approvals.
* Performs checkpoints.
* Stops or pauses.

This separation will make OCAL safer and easier to port.

# 16. The strongest final form of OCAL

The most powerful version of OCAL is not “YAML that tells an agent what to do.”

It is:

> **A host-enforced operating contract for autonomous software work.**

It should make these guarantees:

1. The model cannot arm itself.
2. The model cannot expand its authority.
3. The model cannot silently persist false memory.
4. The model cannot promote work without evidence.
5. The model cannot bypass sandbox/identity policy.
6. The model cannot trust arbitrary MCP/Skill/plugin metadata.
7. The model cannot merge experiments by vibes.
8. The model cannot hide cost, traces, or tool decisions.
9. The model cannot erase failed hypotheses.
10. The model cannot convert scratch into truth without review.

That is how OCAL becomes more than a runbook.

That is how it becomes the default language for ambitious autonomous coding projects.
