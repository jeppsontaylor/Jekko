## Bottom line

OCAL is **not redundant** if it becomes the portable **agent operations language**: a host-enforced contract for lifecycle, permissions, memory, evidence, experiments, approvals, and rollback. It *is* redundant if it only becomes another YAML wrapper around “run an agent until tests pass.”

The biggest upgrade is to shift OCAL from:

> “daemon loop + stop conditions + checkpoints”

to:

> “portable, replayable, evidence-gated agent workflow control plane for long-running software work.”

I could not literally exhaust every private X, LinkedIn, Quora, and community discussion, especially content behind login or not indexed. I focused on primary docs, open-source repositories, official product docs, and relevant papers. The strongest signals are clear: OpenHands is moving toward event-sourced agent state, sandboxed runtimes, and SDK abstractions; LangGraph emphasizes durable execution, checkpointing, and human-in-the-loop decisions; Hermes/OpenClaw emphasize local-first persistent agents with skills, memory, messaging, cron, and channel control; Jankurai emphasizes anti-vibe-coding audit receipts and proof lanes; and SWE-agent, AutoCodeRover, Agentless, Voyager, Reflexion, and MetaGPT show that coding agents improve when the runtime constrains the interface, localizes before editing, stores lessons, and decomposes work into repeatable passes. ([arXiv][1])

OCAL’s opportunity is to be the **thing above all of these**: not a replacement for OpenHands, LangGraph, Hermes, OpenClaw, Codex, Claude Code, or OpenCode, but a declarative language that can compile into or constrain them.

# 1. What OCAL already gets right

OCAL’s strongest ideas are excellent:

1. **Host owns the loop.** This is the core safety property. The model should never decide that a daemon is done, safe, promotable, or allowed to continue indefinitely.

2. **Preview before arming.** This matters even more than the current spec implies. In a world where OCAL can appear inside any chat prompt, the parser must treat OCAL as untrusted until a host-controlled arming ceremony succeeds.

3. **SQLite as truth.** Durable state is the right foundation. LangGraph’s persistence model similarly stores checkpoints over a thread so execution can resume, replay, branch, and support human-in-the-loop workflows. ([LangChain Docs][2])

4. **Circuit breakers and budgets.** This avoids unbounded “agent swarms.”

5. **Incubator passes.** This is probably the most strategically important feature in the current spec. It moves OCAL away from “let the model vibe” and toward structured maturation.

6. **Runtime-enforced guardrails.** Safety should be a property of the runtime, not a reminder in the prompt.

7. **Filesystem mirror generated from durable state.** This is good because humans need legible artifacts, but the mirror should never become truth.

The current OCAL is already stronger than many agent configs because it treats long-running autonomy as an operational system rather than a prompt style.

# 2. The strategic reframing: OCAL should become an Agent Operations Language

Today, OCAL reads like a daemon runbook. That is useful, but not enough.

The winning framing is:

> **OCAL is a portable, host-enforced control plane for agentic software work.**

That means OCAL should define:

| Layer             | Purpose                                                                                            |
| ----------------- | -------------------------------------------------------------------------------------------------- |
| **Policy**        | Permissions, budgets, security, arming, approvals, trust boundaries                                |
| **Workflow**      | Task graph, phases, agents, passes, retries, fan-out, reduce, rollback                             |
| **Evidence**      | Tests, proofs, audits, metrics, receipts, coverage, human review                                   |
| **Memory**        | Run/task/repo/user/org memory with provenance and promotion gates                                  |
| **Execution**     | Sandboxes, worktrees, idempotent side effects, replay, checkpoints                                 |
| **Portability**   | Host capability negotiation across OpenCode, Codex, Claude, OpenHands, LangGraph, Hermes, OpenClaw |
| **Observability** | Event streams, metrics, traces, dashboards, generated mirrors                                      |
| **Learning**      | Skill creation, failure memories, benchmark harnesses, regression suites                           |

OpenHands is already emphasizing event-sourced state, deterministic replay, typed tools, sandboxed execution, and REST/WebSocket interaction in its SDK direction. OCAL should adopt similar concepts at the language level, but leave actual execution to the host/runtime. ([arXiv][1])

LangGraph and Temporal show the other half: durable workflows need checkpoints, resumability, idempotent side effects, and explicit human intervention points. OCAL should not merely say “retry” or “pause”; it should define replay-safe action boundaries and approval decisions. ([LangChain Docs][3])

# 3. Biggest missing concept: evidence

OCAL currently has `stop`, `checkpoint.verify`, `guardrails`, `constraints`, and `incubator.readiness`. These are good, but they are scattered.

The language needs a first-class **evidence model**.

Right now, OCAL can say:

```yaml
checkpoint:
  verify:
    - command: "bun test"
```

But ambitious coding projects need to answer:

* What evidence proves this change is correct?
* What evidence proves it is safe?
* What evidence proves the agent did not delete, skip, mock, or bypass the hard part?
* What evidence proves the change affects the intended files?
* What evidence proves no security boundary was crossed?
* What evidence is required before commit, before push, before PR, before deploy?
* Which evidence is machine-generated, model-generated, or human-reviewed?
* Which evidence is stale after another file changes?
* Which evidence can be replayed?

This is where Jankurai is highly relevant. Jankurai positions itself as an anti-vibe-coding audit system that maps many failure modes to controls and produces receipts rather than trusting model claims. Its docs emphasize proof loops, audits, context packs, owner maps, generated zones, security checks, and “receipts” before merge. ([GitHub][4])

OCAL should absorb that idea.

## Proposed addition: `evidence`

```yaml
evidence:
  policy: no_proof_no_checkpoint

  providers:
    unit_tests:
      type: shell
      command: "bun test --timeout 30000 --reporter=json"
      timeout: 60s
      parse: json
      emits:
        - tests_passed
        - tests_failed
        - duration_ms

    typecheck:
      type: shell
      command: "npx tsgo --noEmit --pretty false"
      timeout: 60s
      emits:
        - type_errors

    lint:
      type: shell
      command: "bun run lint --format json"
      timeout: 60s

    jankurai:
      type: shell
      command: "jankurai audit . --json"
      timeout: 120s
      optional: true
      emits:
        - vibe_risk_score
        - missing_receipts
        - proof_lane_status

  proof_lanes:
    - id: changed_behavior_has_tests
      applies_when:
        file_diff:
          any_path_matches: ["src/**", "packages/**"]
      require:
        - provider: unit_tests
          assert:
            exit_code: 0
        - artifact: test_selection_report
          assert:
            covers_changed_symbols: true

    - id: no_silent_test_deletion
      require:
        - provider: test_inventory
          invariant: gte_baseline

    - id: type_safety
      require:
        - provider: typecheck
          assert:
            exit_code: 0

    - id: security_smoke
      applies_when:
        diff_contains_any:
          - "auth"
          - "token"
          - "password"
          - "secret"
          - "permission"
      require:
        - provider: security_scan
          assert:
            critical_findings: 0

  gates:
    checkpoint:
      require_lanes:
        - changed_behavior_has_tests
        - no_silent_test_deletion
        - type_safety

    promotion:
      require_lanes:
        - changed_behavior_has_tests
        - no_silent_test_deletion
        - type_safety
        - security_smoke
      require_human_review_when:
        - risk_score_gte: 0.6
        - files_changed_gte: 20
        - touches_paths: ["infra/**", "migrations/**", "auth/**"]
```

This would make OCAL much more powerful than a normal agent config. The daemon would not merely “try to pass tests.” It would maintain a **proof ledger**.

# 4. Second biggest missing concept: trust boundaries

OCAL has permissions and guardrails, but it does not yet have a full trust model.

This is dangerous because OCAL is intended to be available inside any chat prompt. That means OCAL itself can appear in:

* user prompts,
* assistant responses,
* issue comments,
* pull request descriptions,
* Slack messages,
* Discord messages,
* README files,
* AGENTS.md files,
* web pages,
* tool outputs,
* malicious dependency docs,
* generated artifacts.

Modern coding agents are exposed to prompt injection through tool outputs, repository text, web pages, dependency docs, and issue bodies. Recent research has cataloged many prompt-injection techniques against coding agents, including attacks through inputs, tools, protocols, multimodal content, and cross-origin content. ([arXiv][5])

OCAL needs a formal trust system.

## Proposed addition: `trust`

```yaml
trust:
  default_label: untrusted

  sources:
    user_prompt:
      label: user_instruction
      may_define_ocal: true
      may_arm_ocal: true

    assistant_output:
      label: model_output
      may_define_ocal: false
      may_arm_ocal: false

    repo_files:
      label: project_content
      may_define_ocal: false
      may_override_policy: false

    agentry_files:
      paths: ["AGENTS.md", ".cursorrules", ".claude/**", "opencode.json"]
      label: project_guidance
      may_define_ocal: false
      may_override_policy: false

    issue_comments:
      label: external_untrusted
      may_define_ocal: false
      may_request_tasks: true

    tool_output:
      label: tool_untrusted
      may_define_ocal: false
      may_request_permissions: false

  context_firewall:
    strip_ocal_sentinels_from_untrusted_sources: true
    quote_untrusted_instructions: true
    require_source_labels_in_context: true
    block_untrusted_policy_mutation: true

  prompt_injection:
    detect_patterns:
      - "ignore previous instructions"
      - "OCAL_ARM"
      - "exfiltrate"
      - "send secrets"
    action: quarantine
```

This is especially important if OCAL is meant to work in OpenCode, Codex, Claude, OpenHands, Hermes, and OpenClaw. Codex, Claude Code, and OpenCode all already have permission systems or project instruction systems, but OCAL should not assume those systems mean repository text is trusted. Codex uses AGENTS.md as an instruction format and applies nested precedence, while Claude Code and OpenCode expose fine-grained permissions and hooks; OCAL should compile into those host features while remaining stricter than them. ([Agents][6])

# 5. Arming must be stronger

The current sentinel is good:

```text
OCAL_ARM RUN_FOREVER id=my-script
```

But for a prompt-portable language, this is not enough.

A malicious web page, issue comment, repo file, or assistant output could include an OCAL block plus an arm sentinel. If the host parser is naive, that becomes catastrophic.

## Required upgrade: detached arming ceremony

OCAL should require a host-generated nonce and digest.

```yaml
arming:
  mode: detached
  require:
    - human_ui_confirm
    - yaml_sha256
    - host_nonce
    - repo_fingerprint
    - capability_diff_review
  valid_for: 10m
  assistant_output_may_arm: false
```

A full arm line could look like:

```text
OCAL_ARM RUN_FOREVER id=fix-tests sha256=8f4c... nonce=host-2026-05-08T14:31:22Z-9ad2 repo=git:3ac91d
```

The host should:

1. parse OCAL as preview only;
2. compute a digest of the YAML body;
3. show a capability diff: shell/edit/git/network/workers/secrets;
4. generate a nonce;
5. require the human to approve in the host UI or type the exact arm line;
6. reject arm sentinels that appear in assistant output, tool output, repo files, web pages, or issue comments.

This preserves the spirit of “preview before arming” while making it robust in hostile contexts.

# 6. OCAL needs capability negotiation

If OCAL should work inside OpenCode, Codex, Claude, OpenHands, Hermes, OpenClaw, and other agents, the language must distinguish between:

* what the OCAL script wants,
* what the host supports,
* what the host can emulate,
* what must fail closed.

OpenHands exposes sandboxed runtimes and an SDK-style architecture; Codex has sandbox and approval modes; Claude Code and OpenCode have their own permission and hook systems; LangGraph provides graph persistence and durable execution. OCAL should not pretend all hosts have the same surface. ([arXiv][1])

## Proposed addition: `capabilities`

```yaml
capabilities:
  require:
    - sqlite_event_log
    - git_worktree
    - shell.exec
    - file.edit
    - approvals.human
    - checkpoints.replayable

  prefer:
    - sandbox.docker
    - ast.symbol_index
    - coverage.map_changed_symbols
    - open_telemetry
    - mcp
    - model_routing

  forbid_emulation:
    - isolated_worktree
    - secret_redaction
    - permission_deny

  on_missing:
    required: abort
    preferred: warn
```

## Proposed addition: `adapters`

```yaml
adapters:
  opencode:
    permissions:
      shell: ask
      edit: allow
      git_push: deny
    agents:
      plan: plan
      build: build
      explore: explore

  codex:
    sandbox_mode: workspace-write
    approval_policy: on-request
    agentry:
      read_agents_md: true
      max_instruction_bytes: 32768

  claude_code:
    permission_mode: default
    hooks:
      pre_tool_use: enforce_guardrails
      post_tool_use: collect_receipts

  openhands:
    runtime: docker
    event_stream: sdk
    state_model: event_sourced

  langgraph:
    checkpointing: required
    thread_id: "{{run.id}}"
    durable_execution: true

  hermes:
    memory: project_scoped
    skills: allow_read_only
    cron: deny_unless_triggered

  openclaw:
    messaging_gateway: allowed
    inbound_dm_trust: untrusted
    pairing_required: true
```

This makes OCAL portable without making it vague.

# 7. OCAL needs a real workflow graph

The current `loop`, `tasks`, `agents`, `fan_out`, and `incubator` blocks are useful, but they are still imperative-ish. Ambitious coding projects need explicit workflows:

* understand,
* localize,
* plan,
* experiment,
* build,
* verify,
* critique,
* merge,
* document,
* checkpoint,
* promote,
* rollback.

LangGraph’s model of graph steps, checkpoints, and human interrupts is relevant here. It stores checkpoints at execution boundaries and supports human review by pausing while preserving graph state. ([LangChain Docs][2])

MetaGPT is also relevant because it shows the value of encoding standardized operating procedures into multi-agent workflows rather than letting an agent improvise every time. ([OpenReview][7])

## Proposed addition: `workflow`

```yaml
workflow:
  mode: graph

  nodes:
    - id: intake
      type: analyze_request
      agent: plan
      writes: scratch_only
      emits:
        - task_card
        - risk_profile
        - acceptance_criteria

    - id: localize
      type: code_localization
      agent: explore
      writes: scratch_only
      inputs: [task_card]
      tools:
        - symbol_search
        - call_graph
        - test_map
        - blame
      emits:
        - suspect_files
        - suspect_symbols
        - relevant_tests

    - id: propose
      type: candidate_generation
      agent: plan
      count: 3
      writes: scratch_only
      inputs: [suspect_files, acceptance_criteria]
      emits:
        - candidate_plans

    - id: experiment
      type: disjoint_experiments
      agent: build
      isolation: git_worktree
      count_from: candidate_plans
      inputs: [candidate_plans]
      emits:
        - patches
        - experiment_reports

    - id: evaluate
      type: score_candidates
      inputs: [patches, experiment_reports]
      evidence:
        require_lanes:
          - type_safety
          - changed_behavior_has_tests
      emits:
        - ranked_candidates

    - id: merge_best
      type: apply_patch
      strategy: best_score
      inputs: [ranked_candidates]
      writes: repo
      gate:
        require_human_when:
          - files_changed_gte: 20
          - risk_score_gte: 0.6

    - id: verify
      type: proof
      evidence:
        require_lanes:
          - type_safety
          - changed_behavior_has_tests
          - no_silent_test_deletion

    - id: checkpoint
      type: git_checkpoint
      when: verified_change

  edges:
    - from: intake
      to: localize
    - from: localize
      to: propose
    - from: propose
      to: experiment
    - from: experiment
      to: evaluate
    - from: evaluate
      to: merge_best
    - from: merge_best
      to: verify
    - from: verify
      to: checkpoint
```

This should coexist with simple loops. OCAL v1 can remain simple. OCAL v2 should support both:

```yaml
loop:
  policy: forever
```

and:

```yaml
workflow:
  mode: graph
```

# 8. OCAL needs better memory

The current `context` block handles context window management, but not true memory.

Hermes emphasizes persistent and session search, a learning loop that creates skills from experience, scheduled automations, subagents, and terminal backends. OpenClaw similarly emphasizes local-first operation, persistent memory, messaging gateways, tools, apps, and skills. ([GitHub][8])

OpenHands also distinguishes always-on project instructions from on-demand skills, where on-demand loading helps preserve context. ([OpenHands Docs][9])

OCAL needs memory layers with provenance, scope, TTL, promotion rules, and deletion.

## Proposed addition: `memory`

```yaml
memory:
  layers:
    - id: run
      scope: run
      storage: sqlite
      ttl: until_run_archived
      write:
        by_model: allow
        require_provenance: true

    - id: task
      scope: task
      storage: sqlite
      ttl: 90d
      write:
        by_model: allow
        require_evidence: true

    - id: repo
      scope: repository
      storage: ".opencode/memory/repo.sqlite"
      ttl: 365d
      write:
        by_model: propose_only
        approval: maintainer
      read:
        retrieval: semantic_and_symbolic
        max_items: 20

    - id: user
      scope: user
      storage: host_user_memory
      write:
        by_model: deny
        approval: explicit_user

    - id: org
      scope: organization
      storage: org_policy_store
      write:
        by_model: deny
        approval: admin

  records:
    schema:
      type: object
      required:
        - claim
        - source_event
        - confidence
        - scope
        - expires_at
      properties:
        claim: { type: string }
        source_event: { type: string }
        evidence_refs: { type: array, items: { type: string } }
        confidence: { type: number, minimum: 0, maximum: 1 }
        scope: { enum: [run, task, repo, user, org] }
        expires_at: { type: string }

  retrieval:
    include:
      - current_task_capsule
      - relevant_repo_memories
      - recent_failed_attempts
      - accepted_design_decisions
    exclude:
      - unreviewed_model_claims
      - quarantined_untrusted_text
      - expired_records

  promotion:
    run_to_task:
      require:
        - evidence_ref
        - not_quarantined
    task_to_repo:
      require:
        - human_review
        - repeated_success_gte: 2
        - no_open_objections
```

This is crucial. Otherwise, “memory” becomes a way to persist hallucinations.

## Memory should store lessons, not hidden chain-of-thought

OCAL should never store hidden reasoning as memory. It should store:

* decisions,
* assumptions,
* evidence,
* failed attempts,
* file maps,
* architectural constraints,
* test strategies,
* known flaky tests,
* rollback plans,
* accepted patterns,
* rejected patterns,
* skill candidates,
* human feedback.

A good memory record looks like:

```yaml
- claim: "Auth middleware rejects expired refresh tokens before DB lookup."
  source_event: "event:verify:2026-05-08T18:20:31Z"
  evidence_refs:
    - "test:auth-refresh-expiry"
    - "diff:src/auth/middleware.ts"
  confidence: 0.92
  scope: repo
  expires_at: "2027-05-08T00:00:00Z"
```

A bad memory record looks like:

```yaml
- claim: "I think the auth system is probably fine."
```

OCAL should reject the second one.

# 9. OCAL needs skill lifecycle management

Hermes and OpenHands both point toward a world where agents have skills. Voyager’s paper is also relevant: it uses an automatic curriculum, a skill library of executable code, and environment feedback/self-verification to improve over time. ([GitHub][8])

OCAL currently has `mcp.profiles`, but MCP tools are not the same as skills.

A skill is a reusable procedure with:

* name,
* purpose,
* activation criteria,
* tool permissions,
* required evidence,
* examples,
* version,
* provenance,
* tests,
* rollback/deactivation path.

## Proposed addition: `skills`

```yaml
skills:
  registry:
    sources:
      - ".opencode/skills"
      - "repo://skills"
      - "org://approved-skills"
    trust:
      repo_local: review_required
      org_approved: trusted
      generated: quarantined

  use:
    policy: allow_reviewed_only
    activation:
      max_loaded: 5
      strategy: relevance_then_recent_success
    require:
      - skill_has_tests
      - permission_scope_declared
      - no_secret_access_unless_approved

  generate:
    enabled: true
    from:
      - repeated_successful_task
      - repeated_failed_then_fixed_task
      - human_request
    writes: scratch_only
    promote:
      require:
        - human_review
        - test_command_passes
        - usage_example
        - permissions_minimized
        - no_untrusted_instructions
      to: repo_registry

  deactivate:
    when:
      - failure_rate_gte: 0.3
      - security_objection
      - stale_for: 180d
    action: quarantine
```

Generated skills should never become trusted automatically. The model can propose them; the host and human must promote them.

# 10. OCAL needs disjointed experiments as a first-class primitive

The current `incubator.prototype` and `fan_out` are close, but ambitious projects need something more explicit:

* multiple hypotheses,
* isolated worktrees,
* divergent patches,
* shared evaluation,
* patch lineage,
* merge strategy,
* losing experiment archive,
* extracted lessons.

This is how OCAL reduces “jankurai” and vibe coding: instead of one agent stumbling forward, it runs disciplined experiments and chooses the best by evidence.

## Proposed addition: `experiments`

```yaml
experiments:
  enabled: true
  strategy: disjoint_worktrees

  hypotheses:
    source: model_and_static_analysis
    min: 2
    max: 5
    require_distinct_files_or_strategy: true
    reject_near_duplicates: true

  worktrees:
    naming: ".opencode/daemon/{{run.id}}/experiments/{{task.id}}/{{experiment.id}}"
    base: current_checkpoint
    cleanup: archive_losers
    keep_winner: true

  each:
    agent: build
    timeout: 20m
    write_scope: isolated_worktree
    permissions:
      shell: ask
      edit: allow
      git_commit: deny
      network: deny

  scoring:
    metrics:
      - id: tests
        weight: 0.35
        provider: unit_tests
      - id: typecheck
        weight: 0.20
        provider: typecheck
      - id: diff_minimality
        weight: 0.15
        measure: changed_lines_inverse
      - id: localization_fit
        weight: 0.15
        measure: touches_suspect_symbols
      - id: review_quality
        weight: 0.15
        provider: critic_review

  reduce:
    strategy: best_score_then_human
    require_margin_gte: 0.12
    if_no_clear_winner: synthesize_or_pause

  memory:
    archive_losing_hypotheses: true
    extract_lessons: true
```

This is a major differentiator. LangChain and AutoGen can orchestrate agents, but OCAL can make the experiment pattern declarative, auditable, and host-enforced.

# 11. OCAL needs deterministic replay and side-effect discipline

SQLite truth is good, but not enough. A daemon that edits files, runs shell commands, calls APIs, pushes branches, starts servers, and hits MCP tools needs replay semantics.

LangGraph’s durable execution docs explicitly call out persistence and the need to wrap non-deterministic operations and side effects in tasks to support resumption and durability. ([LangChain Docs][3])

OpenHands SDK similarly emphasizes event-sourced state, deterministic replay, immutable config, and sandboxed execution. ([arXiv][1])

## Proposed addition: `execution`

```yaml
execution:
  state_model: event_sourced

  replay:
    mode: deterministic_where_possible
    capture:
      - model_id
      - model_parameters
      - prompt_hash
      - context_refs
      - tool_name
      - tool_args_hash
      - tool_result_hash
      - shell_command
      - shell_exit_code
      - stdout_digest
      - stderr_digest
      - git_head
      - git_tree_hash
      - environment_digest
      - permission_decision
      - approval_decision
    side_effects:
      require_idempotency_key: true
      record_before_after: true
      replay_policy: do_not_reexecute_without_approval

  actions:
    shell:
      idempotency: command_hash_plus_cwd_plus_env
      capture_output_limit: 1mb
      redact_secrets: true

    git_commit:
      idempotency: tree_hash
      require_clean_index_after: true

    network:
      default: deny
      require_receipt: true
```

This gives OCAL a proper operational semantics.

# 12. OCAL needs typed task lifecycle, not just discovery

The `tasks` block currently supports a ledger and shell discovery. That is too small.

Large projects need tasks as durable entities with:

* state,
* owner,
* acceptance criteria,
* risk,
* dependency graph,
* affected files,
* related tests,
* readiness score,
* blockers,
* evidence,
* memory,
* rollback plan,
* artifact refs.

## Proposed addition: `task_schema`

```yaml
tasks:
  ledger: sqlite

  schema:
    required:
      - id
      - title
      - objective
      - acceptance
      - state
      - risk
    properties:
      state:
        enum:
          - discovered
          - triaged
          - incubating
          - ready
          - active
          - verifying
          - blocked
          - parked
          - done
      risk:
        type: number
        minimum: 0
        maximum: 1
      acceptance:
        type: array
        items: { type: string }
      dependencies:
        type: array
        items: { type: string }
      affected_files:
        type: array
        items: { type: string }
      rollback:
        type: string

  states:
    discovered:
      next: [triaged, parked]
    triaged:
      next: [incubating, ready, blocked]
    incubating:
      next: [ready, parked, blocked]
    ready:
      next: [active]
    active:
      next: [verifying, blocked]
    verifying:
      next: [done, active, blocked]
    blocked:
      next: [triaged, parked]
    parked:
      next: [triaged]
    done:
      terminal: true
```

The current incubator is strong, but it should operate over typed task states.

# 13. OCAL needs code-intelligence primitives

A key lesson from SWE-agent is that the agent-computer interface matters: agents perform better when the environment gives them useful commands for inspecting, navigating, editing, and testing code. ([arXiv][10])

AutoCodeRover is relevant because it uses program-structure information, code search, and fault localization instead of relying only on free-form agent exploration. ([arXiv][11])

Agentless is also important because it shows that a simple localization-and-repair pipeline can be competitive without letting the LLM autonomously decide every tool call. ([arXiv][12])

OCAL should therefore add a code-intelligence tool class.

## Proposed addition: `code_intelligence`

```yaml
code_intelligence:
  enabled: true

  indexes:
    - type: symbols
      command: "opencode index symbols --json"
      refresh: on_file_change
    - type: call_graph
      command: "opencode index callgraph --json"
    - type: test_map
      command: "opencode index tests --json"
    - type: ownership
      command: "opencode index owners --json"
    - type: dependency_graph
      command: "opencode index deps --json"

  localization:
    strategy: hybrid
    use:
      - stack_traces
      - failing_tests
      - symbol_search
      - call_graph
      - blame
      - recent_changes
    output_schema:
      required:
        - suspect_files
        - suspect_symbols
        - relevant_tests
        - confidence

  patching:
    prefer:
      - ast_patch
      - minimal_diff
    forbid:
      - broad_rewrite_without_plan
      - deleting_tests_without_approval
```

This would make OCAL much more suitable for serious coding work than generic shell-only agents.

# 14. OCAL needs model routing and ensemble policy

OCAL has agents like `plan` and `build`, but does not describe model selection, cost controls, fallback, or ensemble use.

Long-running projects should route models by phase:

* cheap model for indexing and summarization,
* strong reasoning model for design review,
* code-specialized model for patching,
* fast model for lint fixes,
* ensemble for high-risk promotion review.

## Proposed addition: `models`

```yaml
models:
  budget:
    max_usd_per_run: 20
    max_tokens_per_run: 2_000_000
    max_tokens_per_task: 250_000

  routes:
    scout:
      prefer: cheap_fast
      fallback: standard
    plan:
      prefer: reasoning_strong
      fallback: standard
    build:
      prefer: code_strong
      fallback: code_fast
    critic:
      prefer: reasoning_strong
      ensemble:
        count: 2
        diversity: provider_or_temperature
    summarize:
      prefer: cheap_fast

  degradation:
    on_budget_80_percent:

      disable_parallel_ideas: true

      pause: true

```

This is important because “ambitious coding projects” need cost and quality control.

# 15. OCAL needs richer approvals

The current `ask | allow | deny` model is not enough. LangGraph’s human-in-the-loop middleware supports decision types like approve, edit, reject, and response, with execution paused and state persisted. ([LangChain Docs][13])

Temporal’s human-in-loop pattern is also relevant: workflows can pause for risky actions, wait efficiently, keep an audit trail, and resume after receiving a signal. ([Temporal Docs][14])

OCAL should have structured approvals.

## Proposed addition: `approvals`

```yaml
approvals:
  decisions:
    - approve
    - edit
    - reject
    - respond
    - defer

  reviewers:
    maintainer:
      source: local_user
    security:
      source: group
      required_when:
        - touches_paths: ["auth/**", "infra/**"]
        - secret_access_requested: true
    product:
      source: group
      required_when:
        - changes_public_api: true

  gates:
    shell_risky:
      when:
        command_matches_any:
          - "rm -rf"
          - "curl .* | sh"
          - "chmod -R"
      require:
        reviewer: maintainer
        show:
          - command
          - cwd
          - env_diff
          - risk_reason

    checkpoint:
      when:
        risk_score_gte: 0.6
      require:
        reviewer: maintainer
        show:
          - diff
          - evidence_summary
          - rollback_plan

    promotion:
      require:
        - reviewer: maintainer
        - reviewer: security
          when:
            touches_paths: ["auth/**", "secrets/**", "infra/**"]

  timeout:
    default: pause
```

This turns approval into an auditable object, not a transient prompt.

# 18. OCAL needs rollback as a first-class object

A daemon that can edit code for hours must also be able to undo or isolate changes.

rollback:
  strategy: checkpointed_git

      - lockfiles
      - generated_files_manifest

# 21. OCAL should support “agentless mode”

This sounds paradoxical, but it is important.

Agentless showed that a two-phase localization-and-repair process can perform strongly without a fully autonomous agent deciding future actions. ([arXiv][12])

OCAL should be able to say:

```yaml
mode:
  autonomy: constrained
  strategy: localize_then_repair
```

Example:

```yaml
repair_strategy:
  mode: agentless_pipeline

  localize:
    levels:
      - files
      - functions
      - lines
    candidates_per_level: 5
    require_evidence:
      - stack_trace_match
      - symbol_match
      - test_failure_relevance

  repair:
    patches_per_location: 3
    minimize_diff: true
    forbid_new_architecture: true

  filter:
    run:
      - "bun test {{relevant_tests}}"
      - "npx tsgo --noEmit"
    rank_by:
      - tests_pass
      - minimal_diff
      - touched_suspect_symbols
```

This is useful for CI failures, bug fixes, and regressions where full autonomy adds risk.

# 22. OCAL should distinguish “maintain” from “stop”

Some examples use:

```yaml
stop:
  all:
    - git_clean:
        allow_untracked: false
```

That can be ambiguous. Does the daemon stop immediately if the repo is clean? Or is it meant to maintain cleanliness after doing work?

OCAL should split:

```yaml
termination:
  when:
    all:
      - objective_satisfied
      - git_clean:
          allow_untracked: false

maintenance:
  invariants:
    - git_clean_between_tasks
    - no_uncommitted_verified_changes
```

Or:

```yaml
run_mode:
  type: maintain_until_cancelled
  maintain:
    - ci_green
    - git_clean
  wake_on:
    - check_failed
    - new_task
    - schedule
```

This matters for daemon semantics.

# 30. Direct comparison with the systems you named

## OCAL vs. OpenHands

OpenHands is an execution platform and SDK for agents: sandboxed runtimes, event streams, multi-agent coordination, tools, and benchmarks. Its SDK direction includes event-sourced state, deterministic replay, immutable config, typed tools/MCP, and secure execution. ([arXiv][19])

OCAL should not compete with OpenHands as an execution platform. OCAL should become a **declarative run contract** that OpenHands can execute.

**Overlap:**

* sandboxing,
* event logs,
* agent runtime,
* multi-agent flows,
* tools/MCP,
* pause/resume.

**OCAL gaps relative to OpenHands:**

* typed event model,
* deterministic replay,
* portable runtime abstraction,
* local/remote runtime support,
* stronger SDK integration,
* explicit event component architecture.

**Recommended move:**

```yaml
adapters:
  openhands:
    runtime: docker
    state_model: event_sourced
    event_stream: sdk
    action_executor: sandbox
```

OCAL should be able to say: “Run this policy using OpenHands.”

---

## OCAL vs. LangChain / LangGraph

LangChain is broader application infrastructure; LangGraph is closer to durable agent workflows. LangGraph supports persistence through checkpoints, thread IDs, human-in-the-loop interruptions, and durable execution patterns. ([LangChain Docs][2])

OCAL’s current loop is simpler than LangGraph’s graph model.

**Overlap:**

* durable execution,
* retries,
* human-in-the-loop,
* workflow nodes,
* persistence,
* agent orchestration.

**OCAL gaps relative to LangGraph:**

* explicit DAG/graph semantics,
* checkpoint namespaces,
* thread/time-travel model,
* side-effect wrapping,
* resumable interrupts,
* typed state transitions.

**Recommended move:**

OCAL should support both:

```yaml
loop:
  policy: forever
```

and:

```yaml
workflow:
  mode: graph
```

OCAL should compile to LangGraph for hosts that support it, but preserve its own stricter safety/evidence semantics.

---

## OCAL vs. Hermes

Hermes is important because it is pushing toward a self-improving local agent with memory, skills, scheduled automations, messaging gateway, subagents, and multiple terminal backends. ([GitHub][8])

OCAL currently has almost none of this longitudinal “agent life” infrastructure.

**Overlap:**

* long-running work,
* subagents,
* memory-ish context,
* skills/MCP adjacent features,
* scheduled operations,
* command surface.

**OCAL gaps relative to Hermes:**

* skill creation and promotion lifecycle,
* persistent/session memory,
* messaging gateway policy,
* cron/scheduled automations,
* local user preferences,
* migration/import from other agent settings,
* command interface.

**Recommended move:**

OCAL should not become Hermes. Instead, OCAL should govern Hermes runs:

```yaml
adapters:
  hermes:
    memory: project_scoped
    skills: reviewed_only
    messaging_gateway: allowed
    scheduled_automation: require_trigger_policy
```

And OCAL should import the best ideas: memory layers, skill lifecycle, and scheduled triggers.

---

## OCAL vs. OpenClaw

OpenClaw emphasizes a local-first gateway agent, multichannel inbox, multi-agent routing, voice/talk, canvas, tools, apps, onboarding, skills, and sandboxed non-main sessions. It also treats inbound DMs as untrusted unless pairing/allowlist policy is configured. ([GitHub][17])

**Overlap:**

* long-running local agent control,
* tools,
* skills,
* channels,
* sandboxed sessions,
* status/trace commands.

**OCAL gaps relative to OpenClaw:**

* channel trust and pairing,
* messaging controls,
* user-facing command grammar,
* local app/browser/system access policy,
* canvas/artifact workspace,
* session classes.

**Recommended move:**

OCAL should add `triggers`, `channels`, and `trust` so OpenClaw can use OCAL as a strict job/run policy.

---

## OCAL vs. Codex / Claude Code / OpenCode

These are host environments. They expose permissions, project instructions, hooks, agents/subagents, and sandboxing in different ways. Codex uses AGENTS.md as project guidance and has sandbox/approval settings; Claude Code and OpenCode both expose permission controls; OpenCode has built-in agent modes such as build, plan, general, and explore. ([Agents][6])

**Overlap:**

* permissions,
* hooks,
* subagents,
* project instructions,
* sandboxing,
* plan/build modes.

**OCAL gaps:**

* cross-host capability negotiation,
* adapter mapping,
* fail-closed policy when a host cannot enforce a constraint,
* policy compilation.

**Recommended move:**

OCAL should become the portable layer above these systems.

---

## OCAL vs. Jankurai

Jankurai is very aligned with OCAL’s safety goals. It is not an agent; it is an anti-vibe-coding standard/audit CLI with controls, receipts, proof loops, context packs, generated zones, owner maps, and risk mapping. ([GitHub][4])

**Overlap:**

* guardrails,
* proof,
* constraints,
* auditability,
* anti-jank safety.

**OCAL gaps:**

* proof lanes,
* explicit receipts,
* owner maps,
* generated zones,
* daily proof loop,
* no-proof-no-merge policy,
* risk-control taxonomy.

**Recommended move:**

Make Jankurai-like proof lanes first-class in OCAL, and optionally allow Jankurai as an evidence provider.

      require_lanes:
        - no_test_deletion
        - changed_behavior_has_tests

        - lint_clean

workflow:
  mode: graph
  nodes:
    - id: localize
      type: code_localization

      writes: scratch_only

permissions:
  shell: ask

# 37. OCAL should have formal schema and compatibility levels

The spec currently lists syntax, but it should become machine-validatable.

```yaml
schema:
  version: ocal.schema.v1
  compatibility:
    min_host_version: "0.8.0"
    features:
      - evidence
      - trust
      - workflow_graph
      - experiments
      - deterministic_replay
  unknown_keys: reject
  deprecated_keys: warn
```

Rules I would enforce:

1. Unknown top-level keys fail unless explicitly allowed.
2. Feature use requires capability support.
3. All shell commands require timeout.
4. All loops require a circuit breaker.
5. All write scopes must be explicit.
6. All promotions require evidence.
7. All model-generated memory starts quarantined.
8. All network access defaults to deny.
9. All secret access is tool-scoped, never model-context-scoped.
10. All complete OCAL blocks require detached arming.

# 38. What OCAL should remove or tighten

## Tighten `permissions.shell: allow`

For serious daemons, `shell: allow` should be rare. Prefer:

```yaml
permissions:
  shell:
    default: ask
    allow:
      - "bun test*"
      - "bun run lint*"
      - "npx tsgo --noEmit*"
    deny:
      - "rm -rf*"
      - "curl * | sh"
      - "git push --force*"
```

OpenCode already supports granular permission rules by tool and pattern, so OCAL should adopt that level of specificity. ([OpenCode][20])

## Tighten `git.add: ["."]`

This is convenient but dangerous. Prefer:

```yaml
checkpoint:
  git:
    add:
      mode: changed_files_from_task
      exclude:
        - ".env*"
        - "node_modules/**"
        - ".opencode/daemon/**"
```

## Tighten stop conditions

Require at least one objective/evidence-based condition unless `run_mode` is `maintain_until_cancelled`.

## Tighten guardrail regexes

Secret detection should not be just `"sk-"` and `"AKIA"`. Use a provider-based scanner plus entropy checks.

```yaml
guardrails:
  file_diff:
    - name: secret-scan
      provider: host.secret_scanner
      action: block
```

# 40. How OCAL becomes the default language for ambitious coding projects

OCAL wins if it becomes the standard answer to this question:

> “How do I safely delegate a software objective to agents across hours or days while preserving human control, evidence, memory, rollback, and portability?”

To get there, the language needs a few non-negotiable design principles.

## Principle 1: OCAL is policy, not a prompt

A model can propose OCAL. A model cannot arm OCAL. A model cannot weaken OCAL. A model cannot promote work without evidence.

## Principle 2: no proof, no checkpoint

Tests passing is often not enough. OCAL should require proof lanes appropriate to the change.

## Principle 3: memory is evidence-linked

No unreviewed model claim should become durable repo memory.

## Principle 4: experiments beat vibes

For hard tasks, OCAL should branch, test, compare, synthesize, and archive.

## Principle 5: agent autonomy is a variable

Some tasks need a graph of agents. Some need a deterministic agentless localization/repair pipeline. OCAL should support both.

## Principle 6: fail closed on host limitations

If Claude, Codex, OpenCode, OpenHands, Hermes, or OpenClaw cannot enforce a required constraint, OCAL should not silently degrade.

## Principle 7: every side effect is recorded

Shell, git, network, secrets, memory, skill creation, approvals, and rollbacks must produce durable events.

## Principle 8: external text is untrusted

Repo files, issue comments, web pages, tool output, Slack messages, and assistant responses cannot define or arm daemon policy.

# 42. Final recommendation

Do **not** position OCAL as “OpenCode’s daemon YAML.”

Position it as:

> **OCAL: the portable, host-enforced AgentOps language for long-running software work.**

Its signature capabilities should be:

1. **Detached arming with digest/nonce.**
2. **Capability negotiation across hosts.**
3. **Trust-labeled context firewall.**
4. **Evidence/proof lanes.**
5. **Typed event-sourced state.**
6. **Graph workflows plus simple loops.**
7. **Disjoint experiment worktrees.**
8. **Memory with provenance and promotion gates.**
9. **Skill lifecycle with quarantine and review.**
10. **Code-intelligence-driven localization.**
11. **No-proof-no-checkpoint policy.**
12. **Rollback-first operations.**
13. **Channel-safe triggers for persistent agents.**
14. **Fail-closed adapters for OpenCode, Codex, Claude, OpenHands, LangGraph, Hermes, and OpenClaw.**

The current OCAL is a strong seed. The upgraded OCAL should be the thing that prevents “vibe coding” from becoming “vibe infrastructure.”
