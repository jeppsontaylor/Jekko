## Executive verdict

OCAL is **not redundant** if you position it correctly. It should not try to become LangGraph, OpenHands, Claude Code, Codex, Hermes, or OpenClaw. It should become the **portable, host-enforced autonomy contract** that those systems can execute, compile, or partially honor.

The strongest version of OCAL is:

> **A declarative, chat-embeddable, cryptographically previewable, host-enforced operating contract for long-running coding agents.**

That makes it closer to **Kubernetes Job + Temporal workflow + policy-as-code + eval harness + agent memory ledger** than to a prompt format.

Your current OCAL already has unusually strong bones: host-owned loop, arming sentinel, SQLite truth, circuit breakers, checkpoints, guardrails, incubator passes, promotion gates, worktree isolation, and structured output. The main missing piece is that OCAL still reads like a **daemon loop DSL**. To become the default language for ambitious coding projects, it needs to evolve into a **full agentic software-delivery control plane**.

I would enhance OCAL rather than attack it. The right criticism is not “this overlaps with OpenHands or LangGraph.” The right criticism is:

> OCAL v1 controls loops, but the world’s hardest coding projects require controlled **state evolution**, **evidence-backed engineering**, **memory formation**, **workflow graphs**, **experiments**, **tool/skill learning**, **security boundaries**, **human governance**, and **cross-runtime portability**.

Below is the expansion I would make.

---

# 1. What OCAL should learn from the ecosystem

## OpenHands

OpenHands is closest to OCAL on the “autonomous software engineer” axis. Its SDK defines core components such as agents, LLMs, conversations, tools, workspaces, events, and security policies, while the OpenHands repo exposes CLI, GUI, cloud, enterprise, and REST/API surfaces on top of that SDK. ([OpenHands Docs][1])

The OpenHands paper frames production-ready agents around flexibility, security/reliability, user interfaces, local-to-remote execution, REST/WebSocket control, sandboxes, lifecycle control, multi-LLM routing, and security analysis. ([arXiv][2])

**Implication for OCAL:** OCAL should not reimplement “agent framework internals.” It should become the **declarative contract** that can drive OpenHands-like workspaces, event streams, sandboxes, security policies, and lifecycle controls.

Missing from OCAL today:

```yaml
runtime:
  engine: openhands
  workspace:
    type: docker
    image: ghcr.io/org/project-dev:latest
  api:
    event_stream: websocket
    lifecycle_control: true
  security_policy:
    profile: repo_write_no_network
```

OCAL should be able to say: “Run this work as an OpenHands-compatible agent session, but enforce these OCAL gates, memory rules, budgets, and approval conditions.”

## LangGraph / LangChain

LangGraph’s major advantage is **durable execution**. Its docs emphasize saving progress at key points, resuming after interruptions, checkpointing, deterministic/idempotent workflows, and wrapping non-deterministic or side-effecting operations in tasks. ([LangChain Docs][3]) LangGraph also provides orchestration, streaming, human-in-the-loop support, persistence, memory, and tracing through LangSmith. ([LangChain Docs][4])

LangChain’s memory model distinguishes runtime memory from background memory and treats long-term memories as structured JSON documents in namespaces/keys that can be searched across namespaces. ([LangChain Docs][5])

**Implication for OCAL:** OCAL needs a **workflow graph** and a **memory graph**, not just `loop`, `on`, and `incubator`.

Current OCAL has:

```yaml
loop:
  policy: forever
```

But ambitious projects need:

```yaml
graph:
  nodes:
    intake:
      kind: host
    map_codebase:
      kind: agent
      agent: scout
    design:
      kind: agent
      agent: plan
    implement:
      kind: agent
      agent: build
    verify:
      kind: shell_suite
    review:
      kind: agent
      agent: critic
    promote:
      kind: gate
  edges:
    - from: intake
      to: map_codebase
    - from: map_codebase
      to: design
    - from: design
      to: implement
      when: evidence.design_reviewed == true
    - from: implement
      to: verify
    - from: verify
      to: review
      when: evidence.tests_passed == true
    - from: review
      to: promote
      when: evidence.critical_objections == 0
```

OCAL should become a **graph-first language** with loops as a special case.

## Codex / OpenAI agent infrastructure

OpenAI’s Codex product is explicitly framed as a cloud software-engineering agent that can work on multiple tasks in parallel, write features, fix bugs, answer codebase questions, and propose PRs in separate sandboxed environments. ([OpenAI][6]) The Codex app presents multiple agents as separate threads/projects and supports long-running tasks, parallel execution, and worktrees. ([OpenAI][7])

OpenAI’s Symphony project is especially relevant: it treats an issue tracker as an always-on agent orchestrator, uses isolated workspaces, bounded concurrency, authoritative issue-tracker state, deterministic per-issue workspaces, and stops active runs when an issue becomes ineligible. ([OpenAI][8]) Codex Max is described as trained for long-horizon work across multiple context windows with compaction and capable of independent work for hours. ([OpenAI][9])

Codex App Server exposes bidirectional JSON-RPC, long-lived processes, event streams, approvals, sessions, JSONL-over-stdio operation, and server-held source-of-truth state so tabs can reconnect. ([OpenAI][10]) Codex subagents can spawn specialized agents in parallel and collect results. ([OpenAI Developers][11])

**Implication for OCAL:** OCAL should add **issue-queue orchestration**, **JSON-RPC agent runtime adapters**, **approval envelopes**, and **multi-agent worktree state lineage**.

OCAL should be able to control a whole repo portfolio, not only one daemon:

```yaml
portfolio:
  sources:
    - type: github_issues
      repo: org/repo
      query: "label:ocal-ready state:open"
  dispatch:
    max_active_runs: 6
    max_active_per_repo: 2
    priority:
      - label: security
        weight: 100
      - label: release-blocker
        weight: 80
  eligibility:
    require:
      - issue_has_acceptance_criteria
      - branch_is_current
    stop_when:
      - issue_closed
      - label_removed: ocal-ready
      - human_comment_contains: "/ocal pause"
```

This is where OCAL can become much more powerful than a single-run language.

## Claude Code

Claude Code has strong primitives that OCAL should absorb: subagents with YAML front matter, tool permissions, model selection, persistent memory directories, project/user/org/plugin scopes, hooks, max turns, skills, background tasks, and isolation settings. ([Claude][12]) Claude Code hooks are deterministic lifecycle commands that can format, block, inject context, and enforce rules instead of relying only on model behavior. ([Claude][13]) Its permissions model supports allow/ask/deny precedence and multiple modes, including dangerous bypass only for isolated environments. ([Claude][14]) Claude’s sandboxing docs emphasize OS-level filesystem/network isolation, reducing prompt fatigue and enabling autonomy. ([Claude][15])

Claude’s memory docs also reveal a key weakness OCAL should solve: memory is useful, but context inclusion is not the same as enforcement. Claude Code uses fresh session context plus memory files such as `CLAUDE.md`, but that is still contextual rather than a hard policy boundary. ([Claude][16]) Anthropic has also described checkpoints, subagents, hooks, and background tasks as part of making Claude Code work more autonomously. ([Anthropic][17])

**Implication for OCAL:** OCAL should import/export Claude-like subagents, hooks, and skills, but upgrade them into **host-enforced contracts**.

Example:

```yaml
adapters:
  claude_code:
    import:
      subagents: ".claude/agents/*.md"
      skills: ".claude/skills/**/SKILL.md"
      memory: ["CLAUDE.md", ".claude/memory/*.md"]
    map_permissions:
      allow: capabilities
      deny: guardrails.input
      ask: approvals
    export:
      subagents: true
      hooks: true
```

OCAL should become the thing that says: “Claude Code may have hooks, but OCAL decides whether a checkpoint can promote.”

## Hermes Agent

Hermes is highly relevant because it pushes the “always-on agent” model further than most coding tools. Its repo describes a self-improving learning loop that creates and improves skills from experience, persists knowledge, searches conversations, supports multiple providers, operates across Telegram/Discord/Slack/WhatsApp/Signal/CLI, has active memory, scheduler support, delegation/parallelization, terminal backends, and trajectory compression. ([GitHub][18]) Its site describes it as an open-source server-resident agent that remembers, gets more capable, works across channels, uses cron, runs isolated subagents, and has sandbox backends. ([Hermes Agent][19]) Recent Hermes release notes mention background process auto-notifications, live model switching, inactivity-based timeouts based on actual tool activity, approval buttons on Slack/Telegram, MCP OAuth 2.1 PKCE, OSV malware scanning, structured logs, config validation, and security hardening. ([GitHub][20])

**Implication for OCAL:** Hermes exposes one of OCAL’s biggest missing features: **skill learning**.

OCAL currently has passes and scratch memory, but not a controlled lifecycle for:

1. discovering repeated patterns,
2. generating reusable skills,
3. testing skills,
4. promoting skills,
5. versioning skills,
6. revoking bad skills.

OCAL should add:

```yaml
skills:
  format: agent-skills
  discover:
    - ".opencode/skills"
    - ".claude/skills"
    - ".agent-skills"
  generate:
    allow: true
    writes: scratch_only
    from_events:
      - repeated_successful_tool_sequence
      - repeated_human_instruction
      - repeated_failure_fixed
    require:
      - unit_test_for_skill
      - static_scan_clean
      - human_approval
  promote:
    to: repo_registry
    after_successful_uses_gte: 3
    max_scope: repo
  revoke:
    when:
      - causes_failed_checkpoint
      - violates_guardrail
      - human_blocks
```

OCAL should borrow Hermes’s self-improvement, but constrain it harder. Hermes-style learning is powerful; OCAL should make it auditable.

## OpenClaw

OpenClaw is more of a general personal-assistant/gateway system. Its repo describes a local, always-on personal assistant that answers through supported channels, and its site markets it as an assistant that can clear inboxes, send emails, manage calendar tasks, and act through WhatsApp/Telegram/chat. ([GitHub][21])

**Implication for OCAL:** OCAL should learn from OpenClaw’s **channel-native control surface**, but not inherit its broad personal-assistant authority.

OCAL should support multi-channel status and approval:

```yaml
notifications:
  channels:
    - tui
    - slack
    - github_check
    - email
  events:
    checkpoint_failed:
      notify: slack
    approval_required:
      notify: [tui, slack]
    daemon_paused:
      notify: [tui, email]

approvals:
  gates:
    promote:
      approvers:
        - role: maintainer
      quorum: 1
      channels: [tui, github_check]
    expose_secret:
      approvers:
        - role: repo_admin
      quorum: 1
      channels: [tui]
```

But OCAL should remain **repo-scoped by default**, not personal-life-scoped.

## Microsoft Agent Framework, AutoGen, CrewAI, AutoGPT

Microsoft Agent Framework explicitly distinguishes when to use a workflow versus an agent and combines AutoGen/Semantic Kernel ideas such as session state, type safety, middleware, telemetry, graph-based workflows, robust state, long-running workflows, and human-in-the-loop support. ([Microsoft Learn][22]) CrewAI’s flows emphasize structured event-driven workflows with state, conditionals, loops, branching, memory, knowledge, observability, guardrails, structured outputs, persistence, resume, and hierarchical/hybrid processes. ([CrewAI Documentation][23]) AutoGen uses an event-driven programming model with teams, GraphFlow, memory, and logging. ([Microsoft GitHub][24]) AutoGPT’s platform emphasizes workflow management via blocks, deployment controls, monitoring, analytics, and continuous agent operation. ([GitHub][25])

**Implication for OCAL:** OCAL should not only describe “agents.” It should explicitly distinguish:

```yaml
execution:
  prefer_deterministic_workflow_for:
    - parsing
    - linting
    - test running
    - dependency scanning
    - issue classification
  use_agent_for:
    - ambiguous_design
    - code_synthesis
    - critique
    - migration_strategy
```

A mature OCAL runtime should prefer shell, scripts, typed workflows, and deterministic tools whenever possible, and use models only where judgment is needed.

That is one of the best ways to reduce vibe coding.

## Recent research and benchmark lessons

Recent agentic-coding research points toward three major upgrades for OCAL.

First, OPENDEV emphasizes event-driven system reminders, conditional prompt composition, lazy MCP tools, multi-layer safety architecture, adaptive context compaction, and experience-driven memory pipelines. ([arXiv][26]) That directly supports adding **dynamic context assembly** and **active reminders** to OCAL.

Second, Live-SWE-agent shows the value of self-evolving agents that create custom tools on the fly for software-engineering tasks, improving solve rates without changing the underlying agent loop. ([arXiv][27]) That supports adding a **controlled tool/skill generation lifecycle**.

Third, SWE-Bench Pro and similar benchmarks show that real-world software tasks are long-horizon, professional, multi-repo/multi-language, and complex. ([OpenReview][28]) This means OCAL must treat engineering as a lifecycle: requirements, design, experiments, implementation, verification, review, promotion, rollback, and memory consolidation.

There is also a strong anti-vibe-coding signal. One 2025 arXiv benchmark on feature implementation reported low success rates for natural-language feature implementation tasks, and a 2025/2026 security study found that functional success does not imply secure code. ([arXiv][29]) ZORO’s 2026 “active rules” work argues that passive rule files are insufficient and that agents need anchored, proof-backed rule following. ([arXiv][30])

**Implication for OCAL:** OCAL should evolve from “run until tests pass” to **prove the change is acceptable**.

# 2. The biggest thing OCAL is missing: evidence as a first-class object

Your spec already has stop conditions, guardrails, constraints, assertions, and readiness scores. But they are distributed across different sections. OCAL needs a central concept:

> **Evidence is the only thing that can satisfy a gate. Model claims are never evidence unless explicitly marked as weak evidence.**

Add a top-level `evidence` block.

```yaml
evidence:
  store: sqlite
  mirror: true
  trust_levels:
    host_verified: 1.0
    shell_verified: 0.95
    static_analysis: 0.9
    test_result: 0.9
    human_approval: 1.0
    model_claim: 0.2
  fields:
    tests_passed:
      source:
        shell:
          command: "bun test --timeout 30000 --reporter=json"
          timeout: 60s
          assert:
            exit_code: 0
      trust: test_result

    typecheck_passed:
      source:
        shell:
          command: "npx tsgo --noEmit"
          timeout: 60s
          assert:
            exit_code: 0
      trust: static_analysis

    no_tests_removed:
      source:
        shell:
          command: "git diff -- tests | ! grep -E 'describe\\.skip|it\\.skip|test\\.skip|\\.only\\('"
          timeout: 10s
          assert:
            exit_code: 0
      trust: shell_verified

    affected_files_known:
      source:
        shell:
          command: "git diff --name-only | jq -R -s 'split(\"\\n\")[:-1]'"
          timeout: 5s
          assert:
            json:
              type: array
              minItems: 1
      trust: shell_verified

  required_for:
    checkpoint:
      - tests_passed
      - typecheck_passed
      - no_tests_removed
      - affected_files_known
    promote:
      - tests_passed
      - typecheck_passed
      - no_tests_removed
      - human_or_critic_reviewed
      - rollback_plan_known
```

This would make OCAL stronger than most agent frameworks because it would separate:

* what the model believes,
* what the host observed,
* what a deterministic tool verified,
* what a human approved,
* what can actually promote code.

This is the single most important upgrade.

# 3. Add active rules, not passive instructions

OCAL should treat repo rules like executable policy, not context.

Today, tools like Claude Code and OpenCode rely partly on memory/rules files such as `CLAUDE.md`, `AGENTS.md`, or project instructions. Those are valuable, but they are still easy for an agent to forget, misprioritize, or misinterpret. Claude’s docs explicitly frame hooks as deterministic lifecycle controls that can enforce behavior rather than relying only on model memory. ([Claude][13]) OpenCode also has permission/rule surfaces, including permission controls and AGENTS.md-style repository instructions. ([OpenCode][31])

OCAL should add `active_rules`:

```yaml
active_rules:
  mode: enforce
  sources:
    - path: "AGENTS.md"
      trust: repo_instruction
      compile_to: rules
    - path: "docs/architecture.md"
      trust: repo_doc
      compile_to: rules
    - inline:
        - id: no-test-skipping
          severity: critical
          rule: "Do not skip, delete, or weaken tests to make CI pass."
          applies_to: [file_diff, shell_command]
          evidence_required:
            - no_tests_removed
          on_violation: block_checkpoint

        - id: no-silent-api-contract-change
          severity: high
          rule: "Do not change public API behavior without adding or updating contract tests."
          applies_to: [file_diff]
          detector:
            shell:
              command: "git diff --name-only | grep -E 'src/api|packages/.*/public' || true"
          evidence_required:
            - contract_tests_updated
          on_violation: pause

        - id: no-unreviewed-migration
          severity: critical
          rule: "Database migrations require rollback plan and dry-run evidence."
          applies_to: [file_diff]
          detector:
            shell:
              command: "git diff --name-only | grep -E 'migration|schema|prisma|sql' || true"
          evidence_required:
            - migration_dry_run_passed
            - rollback_plan_known
          on_violation: block_promotion
```

This is how OCAL can directly attack “vibe coding.” It forces the agent to connect every risky action to proof.

# 4. Add anti-vibe / anti-jank gates

“Vibe coding” is often described as using natural-language intent and letting AI produce executable code; criticism centers on insufficient review, hidden security issues, and maintainability risk. Public discussion around tools such as Jankurai claims value in realtime auditing for known AI-coding failure modes across tools like Claude, Codex, Cursor, and OpenCode, but those public claims should be treated as weak signals unless backed by reproducible evaluations. ([IBM][32])

OCAL should not depend on any one anti-vibe tool. Instead, it should define a general `quality_gates` interface:

```yaml
quality_gates:
  anti_vibe:
    enabled: true
    fail_closed: true
    checks:
      - id: changed-code-has-tests
        type: diff_coverage
        require:
          changed_production_files_have_related_tests: true
        on_fail: block_checkpoint

      - id: no-unexplained-large-diff
        type: diff_size
        max_changed_lines_without_design_review: 300
        on_fail: require_design_review

      - id: no-test-weakening
        type: shell
        command: "git diff | ! grep -E 'skip\\(|\\.only\\(|TODO: disable|return true'"
        assert:
          exit_code: 0
        on_fail: block_checkpoint

      - id: dependency-risk
        type: shell
        command: "npm audit --json"
        assert:
          json:
            path: "$.metadata.vulnerabilities.critical"
            equals: 0
        on_fail: pause

      - id: external_audit
        type: external
        command: "jankurai audit --json ."
        optional: true
        on_fail: warn
        parse:
          score_path: "$.score"
          require_gte: 0.85
```

The important design principle:

> Anti-vibe checks should be pluggable, but OCAL’s core should define the evidence contract.

# 5. Add graph-first execution

`loop`, `fan_out`, `on`, and `incubator` are useful, but they are not enough for large projects. A large coding project needs a state machine.

Add a top-level `graph`.

```yaml
graph:
  start: intake
  nodes:
    intake:
      type: host
      objective: "Normalize task, collect acceptance criteria, classify risk."
      outputs:
        task_capsule:
          schema: task_capsule_v1

    repo_map:
      type: deterministic
      run:
        - "git status --short"
        - "find src tests packages -maxdepth 3 -type f | sed 's#^#file:#'"
      outputs:
        repo_snapshot:
          schema: repo_snapshot_v1

    requirements:
      type: agent
      agent: plan
      context:
        include:
          - task_capsule
          - repo_snapshot
          - active_rules
      outputs:
        requirements_matrix:
          schema: requirements_matrix_v1
      writes: scratch_only

    design:
      type: agent
      agent: architect
      outputs:
        design_doc:
          schema: design_doc_v1
      gate:
        require:
          - requirements_matrix_complete
          - scope_bounded

    experiment:
      type: experiment
      experiment: implementation_branches

    implement:
      type: agent
      agent: build
      writes: isolated_worktree
      input_ref: experiment.selected_ref

    verify:
      type: deterministic
      run:
        - "bun run lint"
        - "npx tsgo --noEmit"
        - "bun test --timeout 60000"
      outputs:
        verification_report:
          schema: verification_report_v1

    critique:
      type: agent
      agent: critic
      writes: scratch_only
      context:
        include:
          - design_doc
          - verification_report
          - git_diff
          - active_rules

    promote:
      type: gate
      require:
        - tests_passed
        - typecheck_passed
        - no_tests_removed
        - critic_no_critical_objections
        - rollback_plan_known

  edges:
    - from: intake
      to: repo_map
    - from: repo_map
      to: requirements
    - from: requirements
      to: design
    - from: design
      to: experiment
      when: evidence.scope_bounded == true
    - from: experiment
      to: implement
    - from: implement
      to: verify
    - from: verify
      to: critique
    - from: critique
      to: promote
      when: evidence.critic_no_critical_objections == true
    - from: critique
      to: design
      when: evidence.critical_objections_resolved == false
      max_cycles: 2
```

This creates an actual lifecycle, not a forever loop with handlers.

# 6. Add state lineage and branch genealogy

For ambitious projects, the runtime must know not only “what happened,” but **which state evolved from which state**.

OCAL should track:

* base commit,
* worktree ID,
* branch ID,
* patch ID,
* experiment ID,
* selected lineage,
* abandoned lineage,
* merged lineage,
* checkpoint lineage,
* memory lineage.

Add:

```yaml
lineage:
  enabled: true
  refs:
    base:
      type: git_ref
      value: "origin/main"
    current:
      type: worktree_ref
      value: "{{worktree.id}}"
  track:
    - git_commit
    - worktree
    - patch
    - experiment_branch
    - memory_snapshot
    - skill_version
  on_abandon:
    summarize_diff: true
    preserve_lessons: true
    delete_worktree: true
```

This matters because “disjointed experiments” are only useful if the runtime can later answer:

* Which branch discovered the real bug?
* Which branch had the smallest diff?
* Which branch introduced security risk?
* Which branch had a useful partial fix?
* Which lessons should survive after the worktree is deleted?

# 7. Add first-class experiments

Your `fan_out` supports map-reduce, but ambitious coding needs **hypothesis-driven experimentation**.

Current fan-out:

```yaml
fan_out:
  strategy: map_reduce
```

Proposed:

```yaml
experiments:
  implementation_branches:
    strategy: disjoint_hypotheses
    isolation: git_worktree
    max_parallel: 4
    shared_context:
      - objective
      - active_rules
      - requirements_matrix
      - repo_map
    disallow_shared_context:
      - other_branch_code
      - other_branch_reasoning
    branches:
      - id: minimal_patch
        hypothesis: "A small targeted fix can satisfy the tests without refactor."
        agent: build
        budget:
          max_tool_calls: 40
          max_changed_lines: 200

      - id: refactor_boundary
        hypothesis: "The failure is caused by bad module boundaries; refactor the boundary."
        agent: build
        budget:
          max_tool_calls: 70
          max_changed_lines: 600

      - id: test_oracle_first
        hypothesis: "A failing test oracle should be written before implementation."
        agent: test
        budget:
          max_tool_calls: 50

      - id: static_analysis
        hypothesis: "Type/lint/static analysis will reveal the cause faster than implementation."
        agent: scout
        budget:
          max_tool_calls: 30

    scoring:
      metrics:
        tests_passed:
          weight: 0.35
        typecheck_passed:
          weight: 0.15
        diff_minimality:
          weight: 0.15
        maintainability_review:
          weight: 0.15
        security_risk:
          weight: -0.2
        migration_risk:
          weight: -0.1
      select:
        strategy: pareto_then_critic
        require:
          - tests_passed
          - typecheck_passed
          - no_critical_guardrail_violations

    reduce:
      strategy: merge_or_select
      agent: critic
      output:
        selected_ref: true
        rejected_refs_summary: true
        reusable_findings: true
```

This is how OCAL can support both:

* **disjointed experiments** that avoid groupthink,
* **stateful building** once a winning branch is selected.

# 8. Add memory as a first-class subsystem

Your current `context` block handles compaction cadence. That is not enough.

Long-running agents need multiple memory types:

1. **Working memory** — current task state.
2. **Episodic memory** — what happened in prior iterations.
3. **Semantic memory** — stable facts about the repo/domain.
4. **Procedural memory** — reusable skills and workflows.
5. **Failure memory** — what was tried and why it failed.
6. **Decision memory** — approved design decisions and tradeoffs.
7. **Evidence memory** — verified facts from commands, tests, diffs.
8. **Human preference memory** — maintainer preferences and corrections.
9. **Risk memory** — known sharp edges.
10. **Artifact memory** — logs, diffs, screenshots, benchmark reports.

OpenHands’ context condenser work describes the practical problem: growing context causes slower responses, higher cost, and reduced effectiveness, so the system needs bounded conversation state that preserves useful information. ([OpenHands][33]) LangChain’s docs distinguish runtime memory from background memory, which is important because not every memory operation should block the agent’s active loop. ([LangChain Docs][5])

Add:

```yaml
memory:
  store: sqlite
  mirror: true
  embedding_index:
    enabled: true
    provider: host_default
    namespaces:
      - repo_facts
      - task_episodes
      - failures
      - decisions
      - skills
      - human_feedback

  namespaces:
    repo_facts:
      schema:
        type: object
        required: [fact, source, confidence, valid_until]
      write_policy:
        allowed_sources: [shell_verified, human_approval, static_analysis]
        model_claims_require_confirmation: true
      retention: 180d

    task_episodes:
      schema:
        type: object
        required: [task_id, summary, changed_files, evidence_ids]
      write_policy:
        on:
          - checkpoint
          - failure
          - promotion
          - abort
      retention: 365d

    failures:
      schema:
        type: object
```

# 10. Upgrade permissions into capabilities

Your `permissions` block is good but too coarse for multi-hour autonomy.

Current:

```yaml
permissions:
  shell: ask
  edit: allow
  git_commit: ask
```

Ambitious coding agents need **capability leases**:

* filesystem read/write by path,
* shell command allowlists,
* network egress by host,
* package install controls,
* secret access controls,
* browser access,
* MCP server/tool scopes,
* token/cost limits,
* approval TTL,
* revocation,
* auditability.

MCP’s authorization spec emphasizes OAuth 2.1-style authorization, bearer-token handling, audience validation, secure token storage, and PKCE for public clients. ([Model Context Protocol][34]) This matters because once agents use MCP servers, permissions are no longer just “can run shell?” They become identity, token, resource, and tool-bound.

Add:

```yaml
capabilities:
  default: deny

  filesystem:
    read:
      allow:
        - "repo/**"
        - ".opencode/daemon/**"
      deny:
        - "**/.env*"
        - "**/secrets/**"
        - "~/.ssh/**"
    write:
      allow:
        - "repo/src/**"
        - "repo/tests/**"
        - "repo/docs/**"
        - ".opencode/daemon/**"
      deny:
        - "repo/.git/**"
        - "repo/package-lock.json"
      require_approval_for:
        - "repo/migrations/**"
        - "repo/infra/**"

  shell:
    mode: structured_allowlist
    allow:
      - command: "bun"
        args_regex: "^(test|run lint|install)$"
      - command: "npx"
        args_regex: "^(tsgo --noEmit|eslint .*)$"
      - command: "git"
        args_regex: "^(status|diff|add|commit|log|show|branch|worktree).*$"
    deny_patterns:
      - "rm\\s+-rf\\s+/"
      - "curl\\s+.*\\|\\s*sh"
      - "git\\s+push\\s+--force"
      - "chmod\\s+777"
    max_runtime: 120s
    max_output_bytes: 200000

  network:
    egress: deny
    allow_hosts:
      - "registry.npmjs.org"
      - "api.github.com"
    require_approval_for:
      - "*"

  secrets:
    expose: deny
    broker:
      enabled: true
      require_approval: true
      ttl: 5m
      mount_as: ephemeral_env
      redact_in_logs: true

  mcp:
    default: ask
    servers:
      filesystem:
        tools:
          read_file: allow
          write_file: allow
          list_dir: allow
      github:
        tools:
          list_issues: allow
          create_pr: ask
          merge_pr: deny

  leases:
    max_duration: 30m
    renew_requires_host_check: true
    revoke_on:
      - guardrail_violation
      - permission_denied
      - circuit_breaker_trip
```

This would make OCAL feel much more like a real autonomy runtime than a runbook.

# 51. Proposed expanded OCAL example: “Ambitious project daemon”

This is not an armable block; it is a schema sketch showing how OCAL could evolve.

```yaml
version: v1.1
intent: daemon
confirm: RUN_FOREVER

schema:
  id: ocal.dev/v1.1
  strict: true
  unknown_keys: reject
  compatibility:
    platforms: [opencode, codex_app_server, claude_code, openhands, langgraph]

job:
  name: Ambitious project implementation
  objective: Implement the requested feature safely, with tests, review, rollback, and evidence-backed promotion.
  risk:
    - Large codebase context may cause incomplete understanding
    - Agent may overfit to visible tests
    - Multi-worker experiments may create conflicting patches
    - Dependency or migration changes may create supply-chain risk

arm:
  mode: explicit_human
  bind_to:
    block_hash: sha256
    normalized_yaml: true
  source:
    require_user_message: true
    reject_assistant_message: true
    reject_tool_output: true
  preview:
    required: true
    show:
      - effective_capabilities
      - graph
      - budgets
      - approval_gates
      - write_paths
      - network_hosts
      - stop_conditions

budgets:
  wall_clock: 6h
  active_tool_time: 90m
  max_model_calls: 200
  max_shell_commands: 300
  max_changed_lines: 2000
  max_total_cost_usd: 75
  on_exhaust:
    action: pause
    require_summary: true

capabilities:
  default: deny
  filesystem:
    read:
      allow: ["repo/**", ".opencode/daemon/**"]
      deny: ["**/.env*", "**/secrets/**", "~/.ssh/**"]
    write:
      allow: ["repo/src/**", "repo/tests/**", "repo/docs/**", ".opencode/daemon/**"]
      require_approval_for: ["repo/migrations/**", "repo/infra/**", "repo/package.json"]
  shell:
    mode: structured_allowlist
    allow:
      - command: "bun"
        args_regex: "^(test|run lint|install)$"
      - command: "npx"
        args_regex: "^(tsgo --noEmit|eslint .*)$"
      - command: "git"
        args_regex: "^(status|diff|add|commit|log|show|branch|worktree).*$"
    deny_patterns:
      - "git\\s+push\\s+--force"
      - "curl\\s+.*\\|\\s*sh"
      - "rm\\s+-rf\\s+/"
  network:
    egress: deny
    allow_hosts: ["registry.npmjs.org", "api.github.com"]
  secrets:
    expose: deny

trust_zones:
  host_policy:
    trust: highest
  user_instruction:
    trust: high
  repo_files:
    trust: medium
    can_define_policy: false
  tool_output:
    trust: low
    instructions_are_data: true
  web_content:
    trust: low
    instructions_are_data: true

memory:
  store: sqlite
  mirror: true
  namespaces:
    repo_facts:
      write_policy:
        allowed_sources: [shell_verified, human_approval, static_analysis]
        model_claims_require_confirmation: true
    task_episodes:
      write_policy:
        on: [checkpoint, failure, promotion, abort]
    failures:
      retrieval_priority: high
    decisions:
      write_policy:
        require_human_or_gate: true
    skills:
      write_policy:
        require_skill_promotion: true
  retrieval:
    strategy: hybrid
    max_tokens: 6000
    require_citations: true
  consolidation:
    background: true
    after: [checkpoint, task_promoted, daemon_paused]

active_rules:
  mode: enforce
  sources:
    - path: "AGENTS.md"
      compile_to: rules
    - path: "docs/architecture.md"
      compile_to: rules
  rules:
    - id: no-test-skipping
      severity: critical
      rule: "Do not skip, delete, or weaken tests to make CI pass."
      evidence_required: [no_tests_removed]
      on_violation: block_checkpoint
    - id: migration-requires-rollback
      severity: critical
      detector:
        shell:
          command: "git diff --name-only | grep -E 'migration|schema|\\.sql|prisma' || true"
      evidence_required: [rollback_plan_known, migration_dry_run_passed]
      on_violation: block_promotion

evidence:
  store: sqlite
  fields:
    tests_passed:
      source:
        shell:
          command: "bun test --timeout 60000"
          timeout: 90s
          assert: { exit_code: 0 }
    typecheck_passed:
      source:
        shell:
          command: "npx tsgo --noEmit"
          timeout: 60s
          assert: { exit_code: 0 }
    lint_passed:
      source:
        shell:
          command: "bun run lint"
          timeout: 60s
          assert: { exit_code: 0 }
    no_tests_removed:
      source:
        shell:
          command: "git diff -- tests | ! grep -E 'describe\\.skip|it\\.skip|test\\.skip|\\.only\\('"
          timeout: 10s
          assert: { exit_code: 0 }
    changed_files_known:
      source:
        shell:
          command: "git diff --name-only"
          timeout: 5s
          assert:
            stdout_regex: ".+"
  required_for:
    checkpoint:
      - tests_passed
      - typecheck_passed
      - lint_passed
      - no_tests_removed
      - changed_files_known
    promote:
      - tests_passed
      - typecheck_passed
      - lint_passed
      - no_tests_removed
      - critic_no_critical_objections
      - rollback_plan_known

graph:
  start: intake
  nodes:
    intake:
      type: host
      outputs: [task_capsule]

    repo_map:
      type: deterministic
      run:
        - "git status --short"
        - "git ls-files"

    requirements:
      type: agent
      agent: plan
      writes: scratch_only
      outputs: [requirements_matrix]

    design:
      type: agent
      agent: architect
      writes: scratch_only
      outputs: [design_doc]

    experiment:
      type: experiment
      experiment: implementation_branches

    implement:
      type: agent
      agent: build
      writes: isolated_worktree
      input_ref: "{{experiments.implementation_branches.selected_ref}}"

    verify:
      type: deterministic
      require_evidence:
        - tests_passed
        - typecheck_passed
        - lint_passed

    critique:
      type: agent
      agent: critic
      writes: scratch_only
      outputs: [critique_report]

    checkpoint:
      type: checkpoint
```

# 52. How OCAL should handle overlap with major systems

## OCAL vs OpenHands

OpenHands is an execution environment and SDK. OCAL should be the declarative run contract that can target OpenHands. OpenHands has workspaces, tools, agents, events, security policies, REST/WebSocket surfaces, and local/remote execution. ([OpenHands Docs][1])

**Resolution:** OCAL should compile to OpenHands sessions rather than compete with OpenHands.

---

## OCAL vs LangGraph

LangGraph is a graph runtime with durable execution, checkpointing, persistence, HITL, memory, and traces. ([LangChain Docs][3])

**Resolution:** OCAL should borrow graph semantics and compile to LangGraph where useful, but OCAL’s unique contribution is **chat-embedded host policy**, **arming**, **capabilities**, **evidence gates**, and **coding-specific promotion semantics**.

---

## OCAL vs Claude Code

Claude Code has subagents, hooks, permissions, memory files, sandboxing, background tasks, and checkpoints. ([Claude][12])

**Resolution:** OCAL should import/export Claude Code settings but define the stronger control layer. Claude Code hooks can run commands; OCAL should decide whether those results satisfy promotion evidence.

---

## OCAL vs Codex / Symphony

Codex provides cloud agents, parallel tasks, worktrees, App Server event streams, approvals, and long-running software-engineering workflows. ([OpenAI][6]) Symphony shows issue-tracker-orchestrated autonomy with isolated workspaces, bounded concurrency, authoritative issue state, and deterministic per-issue workspaces. ([OpenAI][8])

**Resolution:** OCAL should adopt issue-queue orchestration and App-Server-style event streaming. OCAL’s advantage is being portable across hosts and explicit about policy/evidence.

---

## OCAL vs Hermes

Hermes has memory, skills, cross-channel access, scheduling, isolated subagents, trajectory compression, live model switching, approvals, and security hardening. ([GitHub][18])

**Resolution:** OCAL should borrow Hermes’s self-improvement loop, but make every generated skill gated, tested, scoped, versioned, and revocable.

---

## OCAL vs OpenClaw

OpenClaw is a local, always-on personal assistant across user channels and personal workflows. ([GitHub][21])

**Resolution:** OCAL should borrow multi-channel approval/status UX but stay repo/task-scoped. Do not let OCAL become a broad personal-assistant permission surface by default.

---

## OCAL vs CrewAI / AutoGen / Microsoft Agent Framework / AutoGPT

These systems emphasize workflows, agent teams, event-driven orchestration, state, monitoring, deployment, graph flow, and human-in-the-loop operation. ([Microsoft Learn][22])

**Resolution:** OCAL should not become another Python framework. It should be the declarative policy/workflow layer that can map into these runtimes.

# 53. Recommended OCAL roadmap

## v1.1 — Make current OCAL production-safe

Add:

* strict schema validation,
* normalized IR,
* content-hash arming,
* evidence block,
* active rules,
* capability model,
* trust zones,
* richer budgets,
* observability event schema,
* typed artifacts,
* memory namespaces,
* adapter declarations.

This keeps v1 recognizable but makes it enforceable.

## v1.2 — Make OCAL graph-first

Add:

* `graph`,
* typed nodes,
* deterministic nodes,
* agent nodes,
* gate nodes,
* experiment nodes,
* state lineage,
* replay,
* idempotency,
* branch genealogy.

This makes OCAL suitable for complex projects.

## v1.3 — Make OCAL anti-vibe by default

Add:

* requirements matrix,
* traceability,
* active rules,
* proof obligations,
* test-oracle passes,
* adversarial critics,
* anti-vibe quality gates,
* rollback gates,
* risk scoring.

This makes OCAL trusted by maintainers.

## v1.4 — Make OCAL self-improving safely

Add:

* skill generation,
* skill tests,
* skill promotion,
* skill revocation,
* failure-memory extraction,
* human-feedback memory,
* background memory consolidation.

This gives OCAL Hermes/Live-SWE-agent-style learning without uncontrolled mutation.

## v2.0 — Make OCAL portable infrastructure

Add:

* runtime adapters,
* issue-queue orchestration,
* portfolio scheduling,
* remote event API,
* RBAC,
* org policies,
* MCP auth profiles,
* audit/replay,
* portable skill registry,
* cross-host compatibility.

This is where OCAL becomes a standard.

# 54. The design mantra I would use

OCAL should be built around these principles:

1. **The model proposes; the host disposes.**
2. **No gate is satisfied by vibes.**
3. **Evidence beats confidence.**
4. **Memory must have provenance.**
5. **Skills must be tested before promotion.**
6. **Context is assembled, not dumped.**
7. **Experiments should be isolated, scored, and reducible.**
8. **Every side effect needs a capability lease.**
9. **Every long run needs replay and recovery.**
10. **Every promotion needs rollback awareness.**
11. **Every external system is an adapter, not the source of truth.**
12. **Any chat may carry OCAL; only the host may execute it.**

# 55. The single most powerful upgrade

If you only add one concept, add this:

```yaml
evidence:
  required_for:
    checkpoint: [...]
    promote: [...]
```

That one block changes OCAL from:

> “An agent loop with guardrails”

into:

> “A host-enforced engineering proof system.”

That is how OCAL becomes more than a daemon language. It becomes the default control language for serious autonomous software work.
