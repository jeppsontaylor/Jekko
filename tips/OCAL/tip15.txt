## My conclusion

OCAL should **not** be killed as redundant. It is closest to something the ecosystem still lacks: a **portable, host-enforced runbook language for long-running coding agents**. LangGraph, OpenHands, Hermes, OpenClaw, Claude Code, Codex, Jules, Devin, and GitHub Agent HQ all expose parts of this picture, but none are exactly “a pasteable, declarative, human-armable, host-owned control contract that can run across agent backends.”

The key upgrade is to stop thinking of OCAL as “daemon YAML” and make it the **agentic engineering control plane**:

> **OCAL = systemd + Temporal durable workflows + GitHub branch protection + Spec Kit + sandbox policy + multi-agent Kanban + replayable event log + memory/skills governance.**

OCAL already has strong primitives: host-owned lifecycle, explicit ARM sentinel, budgets, circuit breakers, guardrails, SQLite truth, write scopes, incubator passes, promotion gates, and hidden daemon tools. The missing pieces are mostly around **portable execution semantics, workflow graphs, replay/fork, memory, skills, multi-agent ownership, experiment selection, traceability, anti-vibe gates, protocol interop, and stronger arming security**.

I prioritized primary docs, repos, and papers. I also looked at social signals around “jankurai” and vibe-coding audits, but I would treat X/LinkedIn/Reddit/Quora-like sources as weak signals unless backed by code, docs, or benchmarks. I could not reliably verify Quora-style claims, so I would not base OCAL design decisions on them.

---

## 1. Competitive map: what OCAL should borrow, not clone

### OpenHands

OpenHands is highly relevant because it already emphasizes event-sourced state, typed events, replayability, persistence, and sandboxed execution. Its SDK describes stateless agents composed around event-sourced state, deterministic replay, immutable configuration, typed tools, MCP, and workspace abstraction. Its runtime also uses sandboxed Docker execution for security, isolation, reproducibility, and resource control. ([arXiv][1])

**OCAL overlap:** durable state, sandboxing, tool/event control.

**OCAL gap:** OCAL says “SQLite is truth,” but it does not yet define a **canonical event schema**, deterministic replay semantics, fork semantics, side-effect idempotency, or how tool observations are normalized across backends.

**Upgrade:** OCAL should define a durable state machine and event log as part of the language, not just as an implementation note.

---

### LangGraph / LangChain

LangGraph’s strongest ideas are durable execution, checkpointers, interrupts, time travel, forks, and explicit memory categories. Its docs distinguish semantic memory, episodic memory, and procedural memory, and its durable execution model warns that side-effectful or nondeterministic operations must be wrapped so they are not repeated incorrectly during replay. LangGraph also supports replay and fork from checkpoints, plus human-in-the-loop decisions such as approve, edit, reject, and respond. ([LangChain Docs][2])

**OCAL overlap:** loops, checkpoints, stop conditions, on-handlers.

**OCAL gap:** OCAL is still mostly linear-loop oriented. It needs **graph workflows**, fork/time-travel, typed interrupts, and durable side-effect semantics.

**Upgrade:** add `workflow`, `state`, `replay`, `forks`, and `approvals` sections.

---

### Hermes Agent

Hermes is important because it is aggressively pushing self-improvement, persistent memory, skill creation, provider-agnostic operation, gateway integrations, cron, MCP, parallel delegation, and durable multi-agent Kanban. Its current release notes mention heartbeat, reclaim, zombie detection, retries, hallucination recovery, task ownership enforcement, checkpoint upgrades, and auto-resume after restart. ([Hermes Agent][3])

**OCAL overlap:** daemon mode, tasks, workers, MCP, persistent state.

**OCAL gap:** OCAL has no explicit **skill lifecycle**, no durable task board with leases/heartbeats, no worker zombie handling, no agent-owned procedural memory governance, and no “goal lock” mechanism.

**Upgrade:** add `skills`, `board`, `worker_health`, `leases`, `goal_lock`, and `retrospective` blocks.

---

### OpenClaw

OpenClaw contributes several excellent ideas: a gateway as source of truth, append-only transcripts, context hygiene, session routing, scheduled tasks, plugins, tools, channels, markdown-visible memory, and “memory wiki” structures with claims, evidence, contradiction/freshness tracking, dashboards, and digests. It also describes pre-compaction memory flush and optional background consolidation. ([OpenClaw][4])

**OCAL overlap:** durable mirror, SQLite, hooks, tasks, daemon state.

**OCAL gap:** OCAL’s memory is mostly implied. It needs first-class memory governance: what can be remembered, why, with what evidence, when it expires, and how contradictions are resolved.

**Upgrade:** add `memory` with semantic, episodic, procedural, and claims-based memory stores.

---

### Claude Code

Claude Code’s ecosystem has CLAUDE.md, skills, subagents, hooks, MCP, and plugins. Anthropic’s own context-engineering writing emphasizes that context is finite and that high-signal curation matters more than dumping everything into the prompt. Anthropic’s long-running-agent work also highlights that multi-day coding workflows need test oracles, persistent memory, and orchestration. ([Claude][5])

**OCAL overlap:** hooks, agents, MCP, context handling.

**OCAL gap:** Claude-style memory files and instructions are mostly advisory context. OCAL should be **enforced policy**, not another instruction file.

**Upgrade:** support Claude/Codex/OpenHands/OpenClaw/Hermes as backends, but keep OCAL as the host-verified contract above them.

---

### Codex / OpenAI Agents SDK

Codex emphasizes OS-enforced sandboxing, approvals, network restrictions, and reduced approval fatigue within controlled boundaries. The OpenAI Agents SDK exposes guardrails, tracing, handoffs, tools, human review, and observability. ([OpenAI Developers][6])

**OCAL overlap:** permissions, guardrails, workers, checkpoints.

**OCAL gap:** OCAL needs more explicit **approval leases**, trace export, sandbox/network policy, and handoff semantics.

**Upgrade:** add `security.sandbox`, `security.network`, `approvals`, `observability.trace`, and `handoffs`.

---

### Devin, Jules, GitHub Agent HQ

Devin, Jules, and GitHub Agent HQ are all converging on autonomous cloud coding tasks, visible plans, GitHub integration, branches, PRs, review loops, and multi-agent control surfaces. Jules specifically emphasizes parallel cloud VM tasks and visible workflow; GitHub Agent HQ is explicitly about managing multiple coding agents from one place. ([docs.devin.ai][7])

**OCAL overlap:** daemon work, parallel workers, checkpoints.

**OCAL gap:** OCAL does not yet have a backend adapter model, PR lifecycle model, or “agent fleet” abstraction.

**Upgrade:** make OCAL portable across OpenCode, Codex, Claude Code, OpenHands, Hermes, Jules-like cloud workers, and future agents.

## 2. The biggest missing thing: OCAL needs a portable execution contract

Right now OCAL is excellent as an OpenCode daemon language. To become the default language for ambitious coding projects, it needs to be **host-portable**.

That means any host should be able to parse:

```yaml
runtime:
  portability: required
  host_must_enforce: true
  assistant_may_not_arm: true
  assistant_may_not_stop: true
  unsupported_capability: pause_with_report
```

OCAL should support capability negotiation:

```yaml
capabilities:
  require:
    - sandbox.worktree
    - git.diff
    - shell.timeout
    - approvals.human
    - events.append_only
    - memory.sqlite
  prefer:
    - lsp.symbol_index
    - mcp.oauth
    - trace.otel
    - workers.git_worktree
  on_missing:
    required: refuse_to_arm
    preferred: degrade_with_warning
  report:
    write: ".opencode/daemon/{{runID}}/CAPABILITIES.json"
```

This is critical for “available inside ANY chat prompt.” If a user pastes OCAL into Claude, Codex, OpenCode, OpenHands, or a custom IDE agent, the host should either:

1. enforce it,
2. preview what it can enforce,
3. degrade safely, or
4. refuse to arm.

No host should silently treat OCAL as a prompt.

## 3. Strengthen arming: add body hash, user-origin proof, and preview binding

Your current sentinel design is good, but the most dangerous failure mode is **assistant-generated arming** or **prompt-injected arming**.

The host should only accept `OCAL_ARM` when all of these are true:

1. The ARM line came from a human/user-originated message, not an assistant/tool/web/file output.
2. The ARM line references a canonical hash of the previewed body.
3. The preview shown to the user is byte-for-byte the script being armed.
4. The host has emitted a capabilities report.
5. The host has emitted a risk summary.

Proposed upgrade:

```yaml
arming:
  require_user_origin: true
  require_preview_ack: true
  require_body_digest: true
  canonicalization: yaml_c14n_v1
  assistant_generated_arm: reject
  tool_output_arm: reject
  stale_preview: reject
  preview_expires_after: 30m
```

Future ARM sentinel:

```text
OCAL_ARM RUN_FOREVER id=my-script sha256=8f14... preview=prev_01J...
```

This prevents a malicious assistant, web page, repo file, MCP tool, or copied code block from smuggling an executable daemon command.

## 4. Add a workflow graph, not just loops

The current `loop` is useful, but ambitious projects are not just loops. They are staged workflows:

1. clarify
2. ground in repo
3. write spec
4. design
5. plan
6. implement
7. test
8. review
9. harden
10. merge
11. retrospect

Spec Kit is relevant here because its value is not merely “make a spec”; it forces agents through requirements, planning, and task decomposition before coding. Spec-driven agent studies also suggest phase-level context grounding improves output quality. ([Microsoft Developer][8])

Add:

```yaml
workflow:
  type: state_machine
  default_timeout: 30m

  phases:
    - id: clarify
      objective: Resolve ambiguous requirements before touching code.
      writes: none
      permissions:
        edit: deny
        shell: allow
      complete_when:
        all:
          - artifact_exists: "SPEC_DRAFT.md"
          - questions_open_lte: 0

    - id: ground
      objective: Prove understanding of the current codebase.
      requires: [clarify]
      writes: scratch_only
      success:
        evidence:
          file_citations_gte: 5
          test_files_identified_gte: 1
          symbols_mapped_gte: 3
          risky_assumptions_lte: 2

    - id: plan
      objective: Produce implementation plan and rollback path.
      requires: [ground]
      writes: scratch_only
      gate:
        type: human_review
        require_decision: approve
      outputs:
        - "PLAN.md"
        - "RISK_REGISTER.md"
        - "ROLLBACK.md"

    - id: implement
      objective: Apply minimal code changes.
      requires: [plan]
      writes: repo
      permissions:
        edit: allow
        shell: ask
      complete_when:
        all:
          - tests_added_or_mapped: true
          - diff_size_lte: 2000

    - id: verify
      objective: Run host-defined test oracles.
      requires: [implement]
      writes: none
      complete_when:
        all:
          - shell:
              command: "bun test --timeout 60000"
              timeout: 90s
              assert: { exit_code: 0 }

    - id: review
      objective: Independent critique before checkpoint.
      requires: [verify]
      agent: critic
      writes: scratch_only
      complete_when:
        all:
          - artifact_exists: "REVIEW.md"
          - unresolved_critical_objections_lte: 0
```

This would make OCAL much harder to misuse. A host can enforce that no edits happen until `ground` and `plan` are complete.

## 5. Add anti-vibe-coding gates

“Vibe coding” is not “using AI to code.” The dangerous version is accepting generated code without understanding, tests, review, or security analysis. Recent security research on AI-generated code found agents can be functionally correct but insecure, which supports making security gates structural, not optional. Trajectory research also suggests agent behavior can be improved by diagnosing search/read/edit patterns and adding feedback. ([arXiv][9])

OCAL should add an explicit anti-vibe section:

```yaml
anti_vibe:
  enabled: true

  no_edit_until:
    - repo_grounding_passed
    - acceptance_criteria_defined
    - tests_identified
    - rollback_known

  require_understanding:
    cite_existing_files_gte: 5
    cite_existing_tests_gte: 1
    explain_public_api_changes: true
    explain_data_model_changes: true
    list_assumptions: true
    unresolved_assumptions_lte: 2

  patch_shape:
    max_files_changed: 20
    max_diff_lines: 2000
    prefer_small_commits: true
    require_change_manifest: true

  audits:
    - name: security-review
      type: host_check
      command: "bun run security:audit --json"
      timeout: 120s
      assert:
        exit_code: 0
    - name: architecture-drift
      type: host_check
      command: "bun run arch:check --json"
      timeout: 60s
      on_fail: block_checkpoint

  model_output:
    forbid:
      - "trust me"
      - "probably works"
      - "not tested"
    require:
      - files_changed
      - tests_run
      - risks
      - rollback
```

For Jankurai-style tools, do **not** hardcode one vendor/tool into OCAL. Add an external audit plugin interface:

```yaml
audit_plugins:
  - id: vibe-failure-mode-audit
    command: "jankurai audit --format=json"
    timeout: 120s
    optional: true
    assert:
      json:
        "$.critical_findings": 0
    on_fail: block_checkpoint
```

I found public social claims around Jankurai-style “anti-vibe” audits, but I would treat them as ecosystem signals rather than core design dependencies. The right OCAL move is to support **pluggable failure-mode audits** generically. ([LinkedIn][10])

## 6. Add first-class spec, plan, and traceability

OCAL should not just say “objective.” It should require a durable chain:

```text
Requirement → Acceptance criterion → Design decision → Task → Code diff → Test → Review finding → Commit
```

Add:

```yaml
spec:
  source: ".specs/{{job.slug}}/spec.md"
  mode: required_before_edit

  require:
    acceptance_criteria_gte: 3
    non_goals_gte: 1
    user_stories_gte: 1
    explicit_edge_cases: true
    security_considerations: true
    rollback_strategy: true

  traceability:
    enabled: true
    map_requirements_to:
      - plan_items
      - tasks
      - files_changed
      - tests
      - review_findings
    write_manifest: ".opencode/daemon/{{runID}}/TRACEABILITY.json"

  drift:
    compare:
      - spec
      - plan
      - diff
      - tests
      - final_summary
    on_violation: block_checkpoint
```

Then require every checkpoint to include:

```yaml
change_manifest:
  required: true
  schema:
    type: object
    required:
      - requirements_satisfied
      - files_changed
      - tests_added
      - tests_run
      - risks
      - rollback
    properties:
      requirements_satisfied:
        type: array
        items: { type: string }
      files_changed:
        type: array
        items: { type: string }
      tests_added:
        type: array
        items: { type: string }
      tests_run:
        type: array
        items: { type: string }
      risks:
        type: array
        items: { type: string }
      rollback:
        type: string
```

This is how OCAL beats vibe coding: it makes “what changed and why” host-checkable.

## 7. Make SQLite truth event-sourced, replayable, and forkable

The spec currently says SQLite is truth. Good. But OCAL should go further: define the **event model** and replay semantics.

OpenHands explicitly uses append-only events as memory/integration points, and LangGraph’s durable execution/time-travel model makes checkpoint replay and forks first-class. OCAL should combine these. ([OpenHands Docs][11])

Add:

```yaml
state:
  backend: sqlite
  event_sourcing: true
  event_schema: ocal_event_v1
  mirror: ".opencode/daemon/{{runID}}"

  replay:
    deterministic: best_effort
    require_idempotency_keys: true
    side_effect_policy: record_and_skip_on_replay
    nondeterminism:
      shell: record_stdout_stderr_exit
      llm: record_model_provider_prompt_hash_response_hash
      mcp: record_request_response
      time: freeze_per_iteration
      network: record_or_disallow

  forks:
    enabled: true
    fork_from:
      - checkpoint
      - iteration
      - phase
      - event
    preserve_lineage: true
    compare_forks: true

  snapshots:
    every_iterations: 5
    before_phase:
      - implement
      - checkpoint
      - promote
```

Define event types:

```yaml
events:
  required_fields:
    - event_id
    - run_id
    - timestamp
    - actor
    - phase
    - iteration
    - parent_event_id
    - type
    - payload_hash

  types:
    - run_started
    - phase_started
    - phase_completed
    - llm_requested
    - llm_responded
    - tool_requested
    - tool_approved
    - tool_denied
    - tool_observed
    - file_diff_created
    - test_started
    - test_completed
    - memory_written
    - skill_used
    - worker_heartbeat
    - checkpoint_created
    - checkpoint_failed
    - human_decision
    - run_paused
    - run_aborted
    - run_completed
```

This is not cosmetic. It enables:

* deterministic-ish replay,
* postmortems,
* audits,
* trajectory diagnostics,
* cross-agent comparison,
* forked experiments,
* rollback,
* evidence-based promotion.

## 8. Add time travel and disjointed experiments

Your incubator is promising, but the next level is **parallel hypothesis testing from the same state**.

Current OCAL can run passes and workers. It should also be able to say:

> “From this checkpoint, create five isolated attempts with deliberately different strategies, score them objectively, merge the winner, and preserve the lessons from the losers.”

Add:

```yaml
experiments:
  enabled: true
  objective: Explore competing implementation strategies before committing.

  fork_from: "phase:plan.completed"
  isolation: git_worktree
  max_parallel: 5
  max_total: 8
  timeout_per_variant: 30m

  diversity:
    require_distinct_strategies: true
    strategies:
      - minimal_patch
      - test_first
      - adapter_layer
      - refactor_core
      - compatibility_shim

  variants:
    count: 5
    agent: build
    context:
      include:
        - SPEC.md
        - PLAN.md
        - RISK_REGISTER.md
      exclude:
        - other_variants_private_notes

  score:
    weights:
      tests_pass: 0.35
      acceptance_criteria: 0.25
      security_audit: 0.15
      diff_minimality: 0.10
      maintainability_review: 0.10
      performance: 0.05

    checks:
      - id: tests
        command: "bun test --timeout 60000"
        assert: { exit_code: 0 }
      - id: lint
        command: "bun run lint"
        assert: { exit_code: 0 }
      - id: security
        command: "bun run security:audit --json"
        assert:
          json:
            "$.critical": 0

  reduce:
    strategy: best_score_then_critic_review
    critic_agent: review
    require_human_approval_if:
      score_delta_lt: 0.1
      files_changed_gt: 30
      security_findings_gt: 0

  loser_handling:
    summarize_lessons: true
    archive_worktrees: true
    memory_write:
      only_if_evidence_backed: true
      store: episodic
```

This is how OCAL can outperform single-pass coding agents. It lets the host create **controlled creative diversity** while keeping merge authority centralized.

## 9. Add stateful continuity between passes

One failure mode in multi-pass systems is that each pass behaves like a new intern who does not know what happened. OCAL needs “stateful building on current state” as an explicit invariant.

Add:

```yaml
continuity:
  enabled: true

  before_each_pass:
    require_read:
      - ".opencode/daemon/{{runID}}/STATE.md"
      - ".opencode/daemon/{{runID}}/CHANGE_MANIFEST.json"
      - ".opencode/daemon/{{runID}}/open_decisions.json"

  pass_contract:
    must_reference_previous_decisions: true
    must_not_repeat_rejected_strategy: true
    must_update_open_questions: true

  stale_context:
    detect_if_git_head_changed: true
    detect_if_spec_changed: true
    detect_if_tests_changed: true
    on_stale: re-ground
```

The pass should not just “try again.” It should explain what changed since the last attempt and what it learned.

## 10. Upgrade memory: semantic, episodic, procedural, and claim-based

OCAL’s current `context` is useful but insufficient. Ambitious projects need memory that is:

* typed,
* scoped,
* evidence-backed,
* fresh,
* contradiction-aware,
* compressible,
* recallable,
* safe to write,
* safe to forget.

LangGraph’s memory taxonomy and OpenClaw’s visible markdown/memory-wiki approach are both useful here. Hermes adds the procedural-memory angle: the agent can create or update skills based on experience, but this must be governed. ([LangChain Docs][12])

Add:

```yaml
memory:
  enabled: true

  stores:
    semantic:
      purpose: Stable facts about the repo, architecture, APIs, and domain.
      backend: sqlite_hybrid
      namespace: "{{repo}}/semantic"
      require_provenance: true
      ttl: 90d
      contradiction_policy: flag_and_review

    episodic:
      purpose: What happened during this run and prior related runs.
      backend: sqlite
      namespace: "{{repo}}/episodes"
      retention: 180d
      summarize_after: 20_events

    procedural:
      purpose: Reusable workflows, debugging recipes, and coding practices.
      backend: skills
      namespace: "{{repo}}/skills"
      auto_write: scratch_only
      promote: human_approve

    claims:
      purpose: Evidence-backed claims with freshness and contradiction tracking.
      backend: sqlite
      schema:
        required: [claim, evidence, confidence, freshness, owner]
      require_evidence_count_gte: 1

  retrieval:
    max_tokens: 4000
    prefer_recent: true
    require_citations: true
    query_expansion: true
    include_negative_lessons: true

  compaction:
    pre_flush: true
    compact_every: 10
    preserve:
      - objective
      - current_phase
      - open_questions
      - irreversible_decisions
      - failing_tests
      - accepted_plan
      - rejected_strategies

  consolidation:
    enabled: true
    schedule: "after_checkpoint"
    promote_when:
      repeated_use_gte: 2
      evidence_count_gte: 2
      contradiction_check: pass
    write_digest: ".opencode/daemon/{{runID}}/MEMORY_DIGEST.md"

  privacy:
    secrets_to_memory: deny
    user_private_data: require_scope
    redact_patterns:
      - "sk-[A-Za-z0-9_-]+"
      - "AKIA[A-Z0-9]{16}"
```

Also add a memory write contract:

```yaml
memory_write_policy:
  require:
    - why_this_matters
    - evidence_event_ids
    - expiration_or_review_date
    - scope
  forbid:
    - unverified_assumptions
    - tool_output_without_source
    - secrets
    - personal_data_without_scope
```

This prevents “memory pollution,” where the model writes guesses into durable memory.

## 11. Add skill lifecycle governance

Hermes, Claude Code, OpenHands, and OpenClaw all point toward skills as modular procedural memory. But automatically generated skills are risky: they can encode bad assumptions, unsafe shell commands, stale APIs, or repo-specific hacks.

OCAL should govern skills with the same seriousness as code.

```yaml
skills:
  enabled: true

  sources:
    - id: repo
      path: ".agents/skills"
      trust: repo
      mode: read_write_with_review

    - id: shared
      path: "~/.config/opencode/skills"
      trust: user
      mode: read_only

    - id: marketplace
      trust: external
      mode: install_requires_scan

  use:
    auto_select: true
    max_active: 5
    require_skill_manifest: true
    log_skill_usage: true

  create:
    enabled: true
    writes: scratch_only
    trigger:
      any:
        - repeated_failure_resolved
        - complex_task_completed
        - user_correction_received
    require:
      - examples
      - when_to_use
      - when_not_to_use
      - risks
      - validation_command
    promote:
      gate: human_review
      run_validation: true

  update:
    allowed: true
    writes: scratch_only
    require_diff_review: true

  security:
    scan_shell_commands: true
    deny_network_by_default: true
    deny_secret_access: true
    require_signature_for_external: ask
```

Add a generated skill format:

```yaml
skill_contract:
  required_files:
    - SKILL.md
    - examples.md
    - validation.yaml
  required_sections:
    - Purpose
    - When to use
    - When not to use
    - Procedure
    - Validation
    - Failure modes
    - Security notes
```

This turns “the agent learned something” into a reviewable artifact.

## 12. Add durable Kanban and worker leases

The current `agents` block is too thin for serious multi-agent work. Hermes’s durable Kanban/heartbeat/reclaim/zombie detection is a strong signal here. OCAL needs a **task board**.

```yaml
board:
  enabled: true
  backend: sqlite

  columns:
    - backlog
    - ready
    - in_progress
    - blocked
    - review
    - done
    - parked

  task_schema:
    required:
      - id
      - title
      - acceptance_criteria
      - owner
      - status
      - evidence
      - retry_count
      - risk_score

  ownership:
    lease_duration: 15m
    heartbeat_every: 60s
    reclaim_after_missed_heartbeats: 3
    destructive_tools_require_owner: true

  zombie_detection:
    enabled: true
    no_event_for: 10m
    no_heartbeat_for: 3m
    on_zombie:
      - reclaim_task: true
      - summarize_worker_state: true
      - pause_if_dirty_worktree: true

  retries:
    max_per_task: 3
    backoff: exponential
    on_exhausted: park_with_summary

  completion:
    require:
      - acceptance_criteria_satisfied
      - tests_passed
      - change_manifest
      - reviewer_approved
```

Then upgrade agents:

```yaml
agents:
  supervisor:
    agent: plan
    responsibilities:
      - decompose
      - assign
      - review
      - merge

  workers:
    - id: explorer
      count: 2
      agent: scout
      isolation: read_only
      can_claim: [backlog, ready]
      writes: scratch_only

    - id: builder
      count: 3
      agent: build
      isolation: git_worktree
      can_claim: [ready]
      writes: isolated_worktree

    - id: critic
      count: 1
      agent: review
      isolation: read_only
      can_claim: [review]
```

## 33. Addressing direct overlap concerns

### “Is OCAL redundant with LangGraph?”

No. LangGraph is a framework/runtime for building graph-based agents. OCAL should be a **declarative policy and runbook language** that can compile to LangGraph-like durable workflows or run inside OpenCode. OCAL should import LangGraph’s durable execution, interrupts, memory taxonomy, and time-travel concepts, but it should remain host-facing and pasteable.

### “Is OCAL redundant with OpenHands?”

No. OpenHands is an agent platform/SDK/runtime with event sourcing and sandboxing. OCAL can target OpenHands as a backend. The overlap is useful: it validates OCAL’s emphasis on durable state and host control. OCAL should adopt OpenHands-style event rigor and sandbox expectations.

### “Is OCAL redundant with Hermes?”

No. Hermes is an agent product with memory, skills, gateways, cron, and durable multi-agent Kanban. OCAL should absolutely borrow Hermes’s skill self-improvement and worker-health ideas, but OCAL’s differentiator is **host-enforced declarative execution** that can run across multiple agents.

### “Is OCAL redundant with OpenClaw?”

No. OpenClaw is broader assistant infrastructure: gateway, channels, plugins, sessions, markdown memory, cron, memory wiki. OCAL should borrow gateway/session/memory-wiki ideas. OCAL should remain focused on **coding-project execution contracts**.

### “Is OCAL redundant with Claude Code or Codex?”

No. Claude Code and Codex are coding agents. OCAL should be the control language that tells a host how those agents may be used. Claude/Codex instructions are often advisory; OCAL must be enforced.

### “Is OCAL redundant with Devin/Jules/GitHub Agent HQ?”

No. Those are managed agent products and hubs. OCAL should be the portable runbook that can drive such agents, compare them, gate them, and archive their evidence.

## 35. P0/P1/P2 roadmap

### P0 — make OCAL safe and precise

These are must-haves before OCAL becomes widely portable:

1. **Canonical parser and schema**

   * reject duplicate YAML keys,
   * reject unknown keys by default,
   * reject unsafe YAML features,
   * canonicalize and hash body.

2. **Secure arming**

   * user-origin only,
   * preview hash required,
   * assistant/tool/file-origin ARM rejected.

3. **Capability negotiation**

   * required vs preferred features,
   * safe refusal if host cannot enforce.

4. **Canonical event schema**

   * append-only events,
   * SQLite tables,
   * replay boundaries,
   * idempotency keys.

5. **Workflow phases**

   * no edit before grounding/plan,
   * phase-specific permissions,
   * human gates.

6. **Anti-vibe gates**

   * require tests,
   * require evidence,
   * require rollback,
   * require change manifest.

7. **Security model**

   * sandbox,
   * network allowlist,
   * secrets policy,
   * prompt-injection handling.

---

1. Memory stores:

   * semantic,

## 36. The core philosophy to preserve

The temptation will be to add everything until OCAL becomes another prompt DSL. Do not do that.

The rule should be:

> Every OCAL feature must compile into one of: host permission, host check, durable event, typed artifact, bounded budget, sandbox policy, human gate, or replayable state transition.

If a feature is merely “tell the model to behave,” it does not belong in OCAL. It belongs in a prompt, skill, or agent profile.

OCAL’s unfair advantage is enforcement.

The strongest possible positioning is:

> **OCAL turns AI coding from vibe coding into audited, replayable, evidence-gated engineering.**

That is the path to making it the default language for the world’s most ambitious coding projects.
